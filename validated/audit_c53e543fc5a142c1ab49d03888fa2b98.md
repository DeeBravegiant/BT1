### Title
Stale Former-Participant Votes Count Toward TEE Code-Hash Whitelisting Threshold - (File: `crates/contract/src/tee/proposal.rs`, `crates/contract/src/lib.rs`)

### Summary

`CodeHashesVotes::count_votes()` tallies **all** stored votes for a `NodeImageHash` without filtering by the current participant set. The `vote_code_hash` entry point compares this unfiltered count against the current governance threshold. Votes cast by participants who were subsequently removed via resharing persist in storage and continue to count, allowing a malicious Docker image hash to reach the whitelisting threshold with fewer current-participant votes than the protocol requires.

### Finding Description

In `vote_code_hash` (`crates/contract/src/lib.rs`, line 1407–1431), after authenticating the caller and recording the vote, the contract checks:

```rust
let votes = self.tee_state.vote(code_hash, &participant);
if votes >= self.threshold()?.value() {
    self.tee_state.whitelist_tee_proposal(code_hash, tee_upgrade_deadline_duration);
}
``` [1](#0-0) 

`self.tee_state.vote(code_hash, &participant)` delegates to `CodeHashesVotes::vote()`, which inserts the new vote and then calls `count_votes()`:

```rust
fn count_votes(&self, proposal: &NodeImageHash) -> u64 {
    self.proposal_by_account
        .values()
        .filter(|&prop| prop == proposal)
        .count() as u64
}
``` [2](#0-1) 

`count_votes` iterates **all** entries in `proposal_by_account` — a `BTreeMap<AuthenticatedParticipantId, NodeImageHash>` — without checking whether each voter is still a member of the current participant set. The threshold used for comparison (`self.threshold()?.value()`) is derived from the **current** `ThresholdParameters`, creating a mismatch: the denominator (threshold) reflects the post-resharing participant set, but the numerator (vote count) includes votes from the pre-resharing participant set.

The struct `CodeHashesVotes` does expose a `get_remaining_votes` helper that filters by current participants: [3](#0-2) 

However, this helper is **not** used inside `vote_code_hash` when computing the count that is compared against the threshold. The same pattern applies to `LauncherHashVotes::vote()` / `count_votes()` used in `vote_add_launcher_hash`. [4](#0-3) 

### Impact Explanation

The TEE code-hash allowlist controls which Docker images MPC nodes are permitted to run. A whitelisted malicious image could exfiltrate key shares from the TEE, enabling unauthorized threshold signatures or secret recovery. This maps to the **Critical** allowed impact: *unauthorized access to MPC key shares or signing capability that materially enables forgery or secret recovery*.

### Likelihood Explanation

A Byzantine participant (strictly below the signing threshold, so unable to forge signatures alone) votes for a malicious `NodeImageHash` before being removed from the participant set via a resharing event. After resharing, their vote persists. The attacker then needs only `(new_threshold − 1)` current participants to vote for the same hash — one fewer than the protocol requires — for the hash to be whitelisted. In a scenario where resharing reduces the participant set (e.g., 5→3 participants, threshold 3→2), two Byzantine former participants' votes plus a single deceived current participant suffice to whitelist a malicious image, whereas the protocol intends to require two current-participant votes.

### Recommendation

Replace the raw `count_votes` call inside `vote_code_hash` (and `vote_add_launcher_hash`) with a participant-filtered count, analogous to how `vote_update` and `TeeVerifierVotes::vote` already handle this:

```rust
// vote_update (correct pattern, lib.rs ~1363):
let valid_votes_count = running_state
    .parameters.participants().participants().iter()
    .filter(|(account_id, _, _)| {
        self.proposed_updates.vote_by_participant
            .get(account_id).is_some_and(|voted_id| *voted_id == id)
    })
    .count();
``` [5](#0-4) 

For `vote_code_hash`, the fix is to use `get_remaining_votes(current_participants).count_votes(code_hash)` (or an equivalent inline filter) rather than the unfiltered `CodeHashesVotes::vote()` return value when performing the threshold comparison. Additionally, ensure `get_remaining_votes` (or equivalent cleanup) is called atomically during every resharing transition to purge stale votes before any subsequent threshold check.

### Proof of Concept

1. Network has 5 participants `{P1, P2, P3, P4, P5}`, governance threshold = 3.
2. Byzantine participants P1 and P2 call `vote_code_hash(malicious_hash)`. Both votes are stored in `proposal_by_account`.
3. A resharing is approved (legitimately, by 3 honest participants). P1 and P2 are removed. New set: `{P3, P4, P5}`, new threshold = 2 (60% of 3, rounded up).
4. If `get_remaining_votes` is not called during resharing, P1 and P2's votes remain in `proposal_by_account`.
5. P3 (deceived or colluding) calls `vote_code_hash(malicious_hash)`. `count_votes` returns 3 (P1 + P2 + P3). `3 >= threshold(2)` → `whitelist_tee_proposal` is called.
6. `malicious_hash` is now an allowed Docker image. MPC nodes that restart will pull and run the malicious image, exposing key shares.

The protocol intended to require 2 current-participant votes; only 1 current-participant vote was needed because 2 stale votes from removed participants were counted.

    pub fn get_remaining_votes(&self, participants: &Participants) -> Self {
        let remaining = self
            .proposal_by_account
            .iter()
            .filter(|(participant_id, _)| {
                participants.is_participant_given_participant_id(&participant_id.get())
            })
            .map(|(participant_id, vote)| (participant_id.clone(), *vote))
            .collect();
        CodeHashesVotes {
            proposal_by_account: remaining,
        }
    }
