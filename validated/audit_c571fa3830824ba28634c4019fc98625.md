Audit Report

## Title
Reward Tokens Permanently Stranded When `totalKernelStaked` Drops to Zero Mid-Period - (File: contracts/KERNEL/KernelDepositPool.sol)

## Summary
`KernelDepositPool` implements a Synthetix-style staking rewards mechanism where `rewardPerToken()` halts accumulation when `totalKernelStaked == 0`. If all stakers call `initiateWithdrawal` during an active reward window, the remaining reward tokens for that period are never attributed to any user and become permanently unrecoverable, as the contract contains no rescue or sweep function for `rewardsToken`.

## Finding Description
The `rewardPerToken()` function at L408–414 returns the frozen `rewardPerTokenStored` whenever `totalKernelStaked == 0`, halting all reward accumulation. The `updateReward` modifier at L232–242 snaps `updatedAt = lastTimeRewardApplicable()` on every state-changing call, so when the last staker calls `initiateWithdrawal` (L320–327), `totalKernelStaked` is decremented to zero and `updatedAt` is set to the current timestamp. All reward tokens that would have accrued from that point to `finishAt` are never credited to any `rewards[user]` mapping entry.

When the period ends and a new staker deposits, the admin can call `notifyRewardAmount`. The function computes `receivedAmount = balanceAfter - balanceBefore` (L573–577), which captures only the newly transferred tokens. The stranded balance already sitting in the contract is not included in `receivedAmount` and therefore not folded into the new `rewardRate` (L579–584). Those tokens remain in the contract balance indefinitely.

The contract has no admin sweep, no `Recoverable` inheritance, and no function that can transfer `rewardsToken` out except `getReward()`, which only pays `rewards[msg.sender]`. The `notifyRewardAmount` guard `if (totalKernelStaked == 0) revert NoStakedTokens()` (L570) only prevents starting a new period with zero stakers; it does not prevent the mid-period zero-supply scenario and does not recover stranded tokens.

The contract's own NatSpec at L18–22 explicitly acknowledges this risk and states the mitigation is purely operational (ensuring tokens are always staked), with no on-chain enforcement.

## Impact Explanation
Reward tokens accruing during any zero-supply interval within an active reward period are permanently frozen inside `KernelDepositPool` with no on-chain recovery path. This matches the allowed impact: **Medium — Permanent freezing of unclaimed yield**.

## Likelihood Explanation
Low. Requires all stakers to call `initiateWithdrawal` during an active reward period. This is a realistic scenario in a low-participation pool, during a market stress event, or when a single large staker holds the entire stake. Each staker calling `initiateWithdrawal` is a normal, permissionless user action. The contract's own NatSpec treats this as a known operational risk with no on-chain safeguard.

## Recommendation
Add an admin-only rescue function that transfers only the excess `rewardsToken` balance — i.e., `rewardsToken.balanceOf(address(this))` minus the sum of all pending `rewards[user]` entries (tracked via a `totalPendingRewards` accumulator) — to prevent recovery of legitimately owed user rewards while allowing stranded tokens to be reclaimed.

## Proof of Concept
1. Admin calls `notifyRewardAmount(1_000e18)` with `totalKernelStaked = 100e18`. `rewardRate = 1_000e18 / duration`, `finishAt = block.timestamp + duration`.
2. Halfway through the period, all stakers call `initiateWithdrawal`. The final call sets `totalKernelStaked = 0` and snaps `updatedAt = block.timestamp` via the `updateReward` modifier.
3. For the remaining half of the period, `rewardPerToken()` returns the frozen `rewardPerTokenStored`. Approximately `500e18` reward tokens accrue to no address.
4. After `finishAt`, a new staker deposits 1 wei. Admin calls `notifyRewardAmount(X)`. `receivedAmount = X` (only the newly transferred amount). New `rewardRate = X / duration`. The `500e18` stranded tokens are not included.
5. `rewardsToken.balanceOf(address(this))` exceeds the sum of all `rewards[user]` by `~500e18` permanently. No function in the contract can transfer this excess out.

Foundry invariant test plan: assert that `rewardsToken.balanceOf(address(this)) == sum(rewards[user] for all users) + (rewardRate * (finishAt - block.timestamp))` at all times; the invariant breaks after `totalKernelStaked` reaches zero mid-period.
