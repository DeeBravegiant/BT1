### Title
Missing Payload-Hash Binding in `respond_verify_foreign_tx` Enables Cross-Request Signature Replay - (File: `crates/contract/src/lib.rs`)

### Summary
`respond_verify_foreign_tx` verifies that the submitted signature is valid over `response.payload_hash` under the root public key, but does **not** verify that `response.payload_hash` equals the canonical hash of the stored request. A single malicious MPC node (below threshold) can replay a valid threshold signature obtained from a prior legitimate foreign-tx verification to resolve a completely different pending request with a forged payload hash, delivering a fabricated attestation to the waiting caller.

### Finding Description
In `respond_verify_foreign_tx` the contract performs exactly two checks:

1. The signature is valid over `response.payload_hash` under the root public key.
2. The request key exists in `pending_verify_foreign_tx_requests`. [1](#0-0) 

What is **absent** is any check that:

```
response.payload_hash == SHA-256(borsh(ForeignTxSignPayload { request: stored_request, values: … }))
```

The canonical payload type that the MPC network actually signs is `ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 { request, values })`, whose hash is computed by `compute_msg_hash`. [2](#0-1) 

The node-side signing path hard-codes a zero tweak, meaning the signature is produced under the **root** key, not a derived key. [3](#0-2) 

Because the contract never binds `response.payload_hash` to the stored request, any previously produced root-key signature over any hash `H` can be submitted as the response to any pending `VerifyForeignTransactionRequest`. The contract will accept it as long as the ECDSA check passes and the request key is present in the map.

**Attack path:**

1. Attacker (a single malicious MPC node, below threshold) observes an on-chain `respond_verify_foreign_tx` call for request R1 that carries `payload_hash = H1` and a valid threshold signature `S1`.
2. Attacker submits (or waits for) a `verify_foreign_transaction` request for a different transaction R2, creating a pending entry.
3. Attacker races legitimate nodes and calls `respond_verify_foreign_tx(R2, { payload_hash: H1, signature: S1 })`.
4. The contract verifies `S1` over `H1` under the root key — valid. It finds R2 in the pending map — present. It resolves the yield and delivers `{ payload_hash: H1, signature: S1 }` to R2's caller.
5. R2's caller receives a response whose `payload_hash` is the hash of R1's payload, not R2's. [4](#0-3) 

### Impact Explanation
The caller of R2 receives a structurally valid but semantically forged attestation. In the primary use case — the Omnibridge inbound flow where a bridge contract releases funds upon receiving a verified foreign-chain event — the attacker can substitute the attestation of one transaction for another. A bridge contract that trusts `payload_hash` without independently recomputing it from the expected `(request, values)` pair will release funds against the wrong foreign-chain event, constituting invalid bridge execution. This matches the **High** impact category: forged foreign-chain verification causing invalid bridge execution.

### Likelihood Explanation
Exploitation requires only a single attested MPC participant acting maliciously — strictly below the signing threshold. No key forgery is needed; the attacker reuses a legitimately produced threshold signature that is already public on-chain. The race condition against honest nodes is the only practical constraint, and a well-positioned node can win it. The `ForeignChainSignatureVerifier::verify_signature` helper in the SDK does perform the missing check, but the contract itself does not enforce it, leaving callers who do not use the SDK unprotected. [5](#0-4) 

### Recommendation
Include the extracted values in `VerifyForeignTransactionResponse` and have `respond_verify_foreign_tx` recompute the expected hash on-chain:

```rust
let expected_hash = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: stored_request.request.clone(),
    values: response.values.clone(),
}).compute_msg_hash()?;
require!(expected_hash == response.payload_hash, "payload hash mismatch");
```

This binds the signature to the exact request stored on-chain, eliminating cross-request replay regardless of caller behaviour.

### Proof of Concept

```
// Step 1 – obtain a legitimate signature for R1
// (observed on-chain after honest nodes process R1)
let H1 = payload_hash_of_R1;
let S1 = threshold_signature_over_H1;   // already public on-chain

// Step 2 – create a pending request for a different transaction R2
contract.verify_foreign_transaction(R2_args);   // attacker or any user

// Step 3 – replay S1 as the response to R2
contract.respond_verify_foreign_tx(
    R2_request,
    VerifyForeignTransactionResponse { payload_hash: H1, signature: S1 },
);
// Contract checks: verify_ecdsa(S1, H1, root_pk) == Ok  ✓
//                  pending_map.contains(R2)              ✓
// Missing check:   H1 == hash(ForeignTxSignPayload{R2, values})  ✗ (not performed)

// Step 4 – R2's caller receives { payload_hash: H1, signature: S1 }
// If the bridge contract does not recompute the expected hash,
// it treats this as a valid attestation of R2 and releases funds.
```
