Audit Report

## Title
Zero-Amount Fee Transfer in `MerkleDistributor.claim()` Freezes Unclaimed Yield for Revert-on-Zero Tokens - (File: contracts/utils/MerkleDistributor/MerkleDistributor.sol)

## Summary
`MerkleDistributor.claim()` unconditionally calls `IERC20(token).safeTransfer(protocolTreasury, fee)` at line 144 without checking whether `fee > 0`. When `feeInBPS` is zero — a valid and explicitly permitted configuration — this produces a zero-value transfer that causes any revert-on-zero ERC20 token to revert the entire `claim()` call. Because `claim()` is the sole mechanism for users to receive their allocated tokens, this permanently freezes unclaimed yield for all claimants under this configuration.

## Finding Description
In `MerkleDistributor.claim()`:

```solidity
uint256 fee = (claimableAmount * feeInBPS) / 10_000;   // = 0 when feeInBPS == 0
uint256 amountToSend = claimableAmount - fee;

IERC20(token).safeTransfer(account, amountToSend);

// Send the fee to the protocol treasury
IERC20(token).safeTransfer(protocolTreasury, fee);     // ← unconditional, no zero-check
``` [1](#0-0) 

`feeInBPS = 0` is a valid state: `initialize()` and `setFeeInBPS()` both only reject values **strictly greater than** `MAX_FEE_IN_BPS` (1000), so zero passes both guards. [2](#0-1) [3](#0-2) 

The `token` address is also freely configurable via `setToken()` with no restriction on token type, making a revert-on-zero token a realistic deployment choice. [4](#0-3) 

The sibling contracts `KernelMerkleDistributor` and `KernelTop100MerkleDistributor` both correctly guard this transfer:

```solidity
if (fee > 0) {
    kernel.safeTransfer(protocolTreasury, fee);
}
``` [5](#0-4) [6](#0-5) 

`MerkleDistributor` is missing this guard entirely.

## Impact Explanation
When `feeInBPS == 0` and the distributed token reverts on zero-value transfers, every call to `claim()` reverts. Since `claim()` is the only function through which users can receive their merkle-allocated tokens, all unclaimed yield is permanently frozen for as long as this condition holds. This matches the allowed impact: **Medium — Permanent freezing of unclaimed yield**.

## Likelihood Explanation
- `feeInBPS = 0` is a valid, operationally reasonable configuration (no protocol fee) reachable at initialization or via `setFeeInBPS(0)` by the owner.
- The `token` is configurable and unrestricted; revert-on-zero tokens (e.g., LEND, BNB) are a documented and well-known ERC20 class.
- No attacker action is required; the freeze is automatic and affects all claimants simultaneously.
- The condition is fully reproducible and requires no external oracle, governance capture, or social engineering.

## Recommendation
Add a zero-amount guard before the fee transfer, consistent with the pattern already used in `KernelMerkleDistributor` and `KernelTop100MerkleDistributor`:

```solidity
uint256 fee = (claimableAmount * feeInBPS) / 10_000;
uint256 amountToSend = claimableAmount - fee;

IERC20(token).safeTransfer(account, amountToSend);

if (fee > 0) {
    IERC20(token).safeTransfer(protocolTreasury, fee);
}
```

## Proof of Concept
1. Deploy `MerkleDistributor` with a token that reverts on zero-value transfers and `feeInBPS = 0` (or call `setFeeInBPS(0)` post-deployment).
2. Set a valid merkle root via `setMerkleRoot()` and fund the contract with the token.
3. Any user calls `claim(index, account, cumulativeAmount, merkleProof)` with a valid proof.
4. Execution reaches line 138: `fee = (claimableAmount * 0) / 10_000 = 0`.
5. Line 144 executes `IERC20(token).safeTransfer(protocolTreasury, 0)`.
6. The token reverts on the zero-value transfer; the entire `claim()` call reverts.
7. The user's state is rolled back (no tokens received, no claim recorded), and the yield remains frozen.

**Foundry test sketch:**
```solidity
function test_claimRevertsOnZeroFeeWithRevertOnZeroToken() public {
    // Deploy a mock ERC20 that reverts on zero-value transfers
    RevertOnZeroToken tok = new RevertOnZeroToken();
    distributor.setToken(address(tok));
    distributor.setFeeInBPS(0);
    // ... set merkle root, fund contract, build valid proof ...
    vm.expectRevert();
    distributor.claim(index, user, cumulativeAmount, proof);
}
```
