### Title
`respond_verify_foreign_tx` Does Not Validate `payload_hash` Against the Queued Request, Enabling Cross-Request Replay by a Single Byzantine Participant - (File: `crates/contract/src/lib.rs`)

---

### Summary

The `respond_verify_foreign_tx` function verifies that the submitted signature is cryptographically valid over `response.payload_hash`, but never checks that `response.payload_hash` is the hash of `ForeignTxSignPayload { request: <the_queued_request>, values: <actual_extracted_values> }`. A single Byzantine attested participant can replay a valid `(payload_hash, signature)` pair from any previously accepted foreign-tx response and use it to resolve an entirely different pending request, delivering a forged attestation to the waiting caller.

---

### Finding Description

`respond_verify_foreign_tx` in `crates/contract/src/lib.rs` (lines 691–754) performs two checks before resolving a pending yield:

1. The caller is an attested participant.
2. The ECDSA signature in `response` is valid over `response.payload_hash` under the domain's root public key. [1](#0-0) 

What it does **not** check is whether `response.payload_hash` is the hash of `ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 { request: <queued_request>, values: … })` — i.e., whether the hash encodes the same `ForeignChainRpcRequest` (tx_id, extractors, finality, etc.) that was originally queued under the `request` key.

The pending-request map is keyed on `VerifyForeignTransactionRequest`: [2](#0-1) 

The lookup uses the caller-supplied `request` argument, not the `payload_hash`. Therefore, an attacker can supply:

- `request = tx_A` — a legitimately pending request they want to hijack.
- `response = (payload_hash_B, signature_B)` — a `(hash, signature)` pair from any previously accepted `respond_verify_foreign_tx` call for a completely different transaction `tx_B`.

Because `signature_B` is a valid MPC signature over `payload_hash_B`, step 2 passes. The contract then resolves all yields queued under `tx_A` with the fraudulent `(payload_hash_B, signature_B)` response.

The correct design — as implemented for regular `respond` — is to derive the expected payload from the stored request and verify the signature against that derived value: [3](#0-2) 

For foreign-tx responses the analogous check would require the contract to receive the extracted `values` alongside the `payload_hash` and recompute `SHA-256(borsh(ForeignTxSignPayload { request, values }))` to confirm consistency. The `ForeignTxSignPayload::compute_msg_hash` helper already exists for this purpose: [4](#0-3) 

The off-chain SDK (`ForeignChainSignatureVerifier::verify_signature`) does perform this check, but it is caller-side and optional: [5](#0-4) 

The MPC contract itself never enforces it.

---

### Impact Explanation

A single Byzantine attested participant resolves a pending `verify_foreign_transaction` request with a `payload_hash` that encodes a **different** foreign-chain transaction (different `tx_id`, different extracted values, or both). The waiting NEAR contract receives a `VerifyForeignTransactionResponse` whose `payload_hash` and signature are internally consistent (the signature is genuinely valid over the hash) but attest to the wrong on-chain event.

Any bridge or omnibridge contract that does not independently reconstruct and compare the expected `payload_hash` — which requires knowing the extracted values in advance — will accept this forged attestation as proof that the target foreign-chain transaction finalized. This enables invalid bridge execution: funds can be released on NEAR for a foreign-chain deposit that never occurred, or for a deposit that was already consumed by a prior request.

This matches the allowed impact: **High — forged foreign-chain verification that causes invalid bridge execution or double-spend conditions**.

---

### Likelihood Explanation

The attacker must be a single attested participant (below signing threshold). No threshold collusion is required. The attack steps are:

1. Observe any previously accepted `respond_verify_foreign_tx` transaction on-chain and extract `(payload_hash_B, signature_B)`.
2. Wait for a target `verify_foreign_transaction` request (`tx_A`) to appear in the pending map.
3. Call `respond_verify_foreign_tx(request = tx_A, response = (payload_hash_B, signature_B))`.

All inputs are publicly observable on-chain. The call requires only that the attacker holds a valid TEE attestation, which is a prerequisite for any participant node. No key material beyond the attacker's own node credentials is needed.

---

### Recommendation

Require the responding node to include the extracted `values` in the response alongside the `payload_hash`. The contract should then recompute:

```
expected_hash = SHA-256(borsh(ForeignTxSignPayload::V1 { request: <queued_request>, values: <provided_values> }))
```

and assert `expected_hash == response.payload_hash` before accepting the signature. This mirrors the pattern already used in `respond` for regular signature requests, where the payload is derived from the stored request rather than trusted from the caller.

---

### Proof of Concept

**Vulnerable code** — `respond_verify_foreign_tx` accepts any `payload_hash` as long as the signature over it is valid: [6](#0-5) 

**Attack path**:

1. Honest request for Bitcoin tx `[0xAA; 32]` is queued via `verify_foreign_transaction`.
2. A separate honest request for Bitcoin tx `[0xBB; 32]` is processed; the MPC network produces `(payload_hash_BB, sig_BB)` and a legitimate node calls `respond_verify_foreign_tx(request=tx_BB, response=(payload_hash_BB, sig_BB))`. This is accepted and visible on-chain.
3. The Byzantine node calls `respond_verify_foreign_tx(request=tx_AA, response=(payload_hash_BB, sig_BB))`.
4. The contract verifies `sig_BB` over `payload_hash_BB` — valid. It resolves the yield for `tx_AA` with `(payload_hash_BB, sig_BB)`.
5. The caller of `verify_foreign_transaction` for `tx_AA` receives a response whose `payload_hash` encodes `tx_BB`'s block hash, not `tx_AA`'s. Any bridge contract that does not independently verify the `payload_hash` will treat this as a valid attestation of `tx_AA`.

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }
