### Title
`respond_verify_foreign_tx` Accepts Caller-Supplied `payload_hash` Without Binding It to the Submitted `request` — Cross-Request Signature Replay by a Single Byzantine Attested Participant - (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_verify_foreign_tx` verifies that the submitted signature is valid over `response.payload_hash`, but it never checks that `response.payload_hash` was actually derived from the `request` argument used as the pending-yield lookup key. A single Byzantine attested participant can take a legitimately produced threshold signature for `request_A` (publicly visible on-chain) and use it to resolve a completely different pending `request_B`, delivering a forged verification response to the caller of `request_B`.

---

### Finding Description

**The asymmetry between `respond` and `respond_verify_foreign_tx`**

In the regular signing flow, `respond` extracts the payload hash directly from the stored `SignatureRequest`:

```rust
// crates/contract/src/lib.rs:600
let payload_hash = request.payload.as_ecdsa().expect("Payload is not ECDSA");
```

The hash is bound to the request at submission time and stored in contract state. The contract cannot be given a different hash.

In `respond_verify_foreign_tx`, the payload hash comes entirely from the caller-supplied `response`:

```rust
// crates/contract/src/lib.rs:726
let payload_hash: [u8; 32] = response.payload_hash.0;

// Check the signature is correct against the root public key
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,
)
.is_ok()
```

The contract only checks: *"is this signature valid over this hash under the root key?"* It never checks: *"does this hash commit to the `request` I was given?"*

**Why the contract cannot recompute the hash**

The signed payload is `SHA-256(borsh(ForeignTxSignPayload::V1 { request, values }))`. The `values` field contains the extracted foreign-chain data (block hashes, logs, etc.) that only the MPC nodes observe at query time. The contract has no access to `values`, so it cannot recompute the expected hash. This is acknowledged in the docs:

> "Only the hash is included in the response to stay within NEAR's promise data limits."

This design gap means the contract's signature check is decoupled from the request identity.

**Attack path**

1. A legitimate `verify_foreign_transaction(request_A)` is processed. The MPC network produces a valid threshold signature `sig_A` over `H_A = SHA-256(borsh(ForeignTxSignPayload{request_A, values_A}))`. An honest node calls `respond_verify_foreign_tx(request_A, {H_A, sig_A})`, which is recorded on-chain.

2. A single Byzantine attested participant submits `verify_foreign_transaction(request_B)` — for example, a Bitcoin `tx_id` that does not exist or has not reached the required confirmations.

3. The Byzantine participant calls `respond_verify_foreign_tx(request_B, {payload_hash: H_A, signature: sig_A})`, reusing the legitimately produced response from step 1.

4. The contract checks:
   - Caller is an attested participant → **passes** (Byzantine node is registered)
   - `sig_A` is valid over `H_A` under the root public key → **passes** (signature is genuine)
   - `request_B` exists in `pending_verify_foreign_tx_requests` → **passes** (it was submitted in step 2)

5. `resolve_yields_for` drains all pending yields for `request_B` and delivers `{H_A, sig_A}` to every caller waiting on `request_B`.

6. The caller of `verify_foreign_transaction(request_B)` receives a `VerifyForeignTransactionResponse` whose `payload_hash` commits to `request_A`'s data, not `request_B`'s.

**The SDK verifier does catch this — but only if used**

`ForeignChainSignatureVerifier::verify_signature` in `crates/near-mpc-sdk/src/foreign_chain.rs` recomputes the expected hash from the caller's own request and expected values, then asserts `expected_payload_hash == response.payload_hash`. A caller using this helper would detect the mismatch. However:

- The contract itself provides no such guarantee.
- Callers that verify only the ECDSA signature (which is valid) and not the payload binding would be deceived.
- The `VerifyForeignTransactionResponse` type carries no indication that the hash was computed for a different request.

---

### Impact Explanation

A single Byzantine attested participant (one compromised TEE node, below the signing threshold) can deliver a forged foreign-chain verification response to any caller waiting on a pending `verify_foreign_transaction` request. The caller receives a cryptographically valid threshold signature, but one that was produced for a different transaction. Any bridge contract or application that:

- Calls `verify_foreign_transaction` to gate a cross-chain asset release or mint, and
- Verifies only the ECDSA signature validity (not the payload-hash binding to the specific request)

would incorrectly conclude that the MPC network verified their specific foreign-chain transaction, enabling unauthorized bridge execution or double-spend conditions.

**Impact class:** High — Cross-chain replay / forged foreign-chain verification causing invalid bridge execution.

---

### Likelihood Explanation

- Requires a single Byzantine attested participant (one compromised or malicious MPC node, strictly below the signing threshold).
- The reused signature is publicly observable on-chain after any legitimate `respond_verify_foreign_tx` call; no additional cryptographic capability is needed.
- The attacker controls the timing: they can submit `request_B` and immediately replay a previously observed `{H_A, sig_A}` before honest nodes respond.
- The vulnerability is reachable from any unprivileged NEAR account (to submit `request_B`) combined with a single Byzantine node (to call `respond_verify_foreign_tx` with the replayed response).

---

### Recommendation

1. **Bind the payload hash to the request at the contract level.** Since the contract cannot recompute the full `ForeignTxSignPayload` hash (it lacks `values`), the MPC nodes should include a commitment to the request identity in the response. One approach: require the response to include a Borsh-serialized prefix of the payload (the `request` portion only), and have the contract verify that this prefix matches the stored request. Alternatively, store a per-request nonce or request-hash in the pending map and require the response to commit to it.

2. **Alternatively, include the request hash in the signed payload at the node level.** The `build_signature_request` function in `crates/node/src/providers/verify_foreign_tx/sign.rs` constructs the `SignatureRequest` from the `ForeignTxSignPayload`. The contract could store `SHA-256(borsh(request))` at submission time and require that `response.payload_hash` is consistent with it (e.g., by requiring the first 32 bytes of the Borsh-encoded payload to match the stored request hash prefix).

3. **Document the caller-side verification requirement as a security invariant**, not merely a convenience. The `ForeignChainSignatureVerifier` check must be treated as mandatory for any security-sensitive consumer of `verify_foreign_transaction`.

### Proof of Concept

```
// Step 1: Observe a legitimate response on-chain for request_A
// request_A = Bitcoin { tx_id: [0x01; 32], confirmations: 6, extractors: [BlockHash] }
// H_A = SHA-256(borsh(ForeignTxSignPayload::V1 { request: request_A, values: [BlockHash([0xAB;32])] }))
// sig_A = valid threshold ECDSA signature over H_A under root key

// Step 2: Submit a fake request_B
verify_foreign_transaction({
    domain_id: <foreign_tx_domain>,
    payload_version: V1,
    request: Bitcoin { tx_id: [0x02; 32], confirmations: 6, extractors: [BlockHash] }
})
// → pending yield queued for request_B

// Step 3: Byzantine attested participant replays sig_A for request_B
respond_verify_foreign_tx(
    request = { domain_id: <foreign_tx_domain>, payload_version: V1,
                request: Bitcoin { tx_id: [0x02; 32], ... } },   // request_B
    response = { payload_hash: H_A, signature: sig_A }            // response for request_A
)

// Contract checks:
//   assert_caller_is_attested_participant_and_protocol_active() → OK (Byzantine node is registered)
//   verify_ecdsa_signature(sig_A, H_A, root_pk) → OK (sig_A is genuine)
//   resolve_yields_for(request_B, serialize({H_A, sig_A})) → drains pending yield for request_B

// Step 4: Caller of request_B receives {payload_hash: H_A, signature: sig_A}
// H_A commits to request_A (tx_id=[0x01;32]), not request_B (tx_id=[0x02;32])
// If caller only checks signature validity → incorrectly concludes MPC verified tx_id=[0x02;32]
```

```rust
pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
```
