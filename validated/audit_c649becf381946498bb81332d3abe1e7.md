Audit Report

## Title
rsETH Price Reset to 1e18 on Zero Supply Ignores Residual sfrxETH Holdings, Enabling First-Minter Yield Theft — (`contracts/LRTOracle.sol`)

## Summary

`LRTOracle._updateRsETHPrice` unconditionally resets `rsETHPrice` to `1e18` when `rsethSupply == 0`, without inspecting actual protocol holdings. If sfrxETH remains in `LRTUnstakingVault` (which is included in `getTotalAssetDeposits`) when supply reaches zero, the first subsequent depositor receives rsETH priced at `1e18` while the true backing ratio is higher, allowing them to claim all protocol holdings — including residual sfrxETH they did not deposit — upon withdrawal.

## Finding Description

**Root cause — `LRTOracle._updateRsETHPrice` (lines 218–222):**

```solidity
if (rsethSupply == 0) {
    rsETHPrice = 1 ether;
    highestRsethPrice = 1 ether;
    return;
}
```

The branch returns before calling `_getTotalEthInProtocol()`, so any sfrxETH still tracked by the deposit pool is silently ignored.

**How residual sfrxETH accumulates while rsETH supply drains to zero:**

`LRTWithdrawalManager.initiateWithdrawal` locks `expectedAssetAmount` at the rsETHPrice prevailing at request time. `_unlockWithdrawalRequests` then computes `payoutAmount = _calculatePayoutAmount(request, rsETHPrice, assetPrice)`:

```solidity
uint256 currentReturn = (request.rsETHUnstaked * rsETHPrice) / assetPrice;
return (request.expectedAssetAmount < currentReturn) ? request.expectedAssetAmount : currentReturn;
```

If `rsETHPrice` was stale (lower) at initiation time, `expectedAssetAmount` is lower than the current fair value. At unlock time with an updated (higher) `rsETHPrice`, `currentReturn > expectedAssetAmount`, so `payoutAmount = expectedAssetAmount`. The difference — accumulated sfrxETH yield — is **not** transferred out of the unstaking vault. After all rsETH is burned, this residual sfrxETH remains in the vault.

**Residual is counted by `getTotalAssetDeposits`:**

`LRTDepositPool.getAssetDistributionData` explicitly includes:
```solidity
assetLyingUnstakingVault = IERC20(asset).balanceOf(lrtUnstakingVault);
```
and `getTotalAssetDeposits` sums all components including `assetLyingUnstakingVault`. So `_getTotalEthInProtocol()` — which is bypassed by the zero-supply branch — would have correctly accounted for this residual.

**First-minter exploitation:**

`getRsETHAmountToMint` divides by the stored `rsETHPrice`:
```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

With `rsETHPrice = 1e18` (reset by the zero-supply branch) and `sfrxETH.pricePerShare() = 1.1e18`, a depositor of `D` sfrxETH receives `D × 1.1e18 / 1e18 = 1.1D` rsETH. Their rsETH now represents all protocol holdings — their `D` sfrxETH plus the residual `R` sfrxETH. After `updateRSETHPrice` is called, `rsETHPrice = (D + R) × 1.1e18 / 1.1D`. The depositor can withdraw `(D + R)` sfrxETH, extracting `R` sfrxETH they did not deposit.

No existing guards prevent this: `depositAsset` only checks `whenNotPaused` and `onlySupportedERC20Token`; `updateRSETHPrice` is `public whenNotPaused`; the `pricePercentageLimit` check is located after the zero-supply early return and does not apply here.

## Impact Explanation

**High — Theft of unclaimed yield.** The attacker extracts sfrxETH yield that accumulated in the unstaking vault during normal protocol operation and was not distributed to withdrawing users due to the minimum-payout logic. The stolen amount equals the residual sfrxETH left in the vault after the final `unlockQueue` call. This is a direct, quantifiable theft of protocol-held yield by an unprivileged external caller.

## Likelihood Explanation

The precondition — `rsethSupply` reaching exactly zero while sfrxETH remains in the unstaking vault and the protocol is unpaused — requires a complete protocol wind-down. sfrxETH yield accrues continuously, `rsETHPrice` is only updated on explicit calls, and the minimum-payout logic in `_calculatePayoutAmount` structurally produces a residual whenever `assetPrice` rises between initiation and unlock. A complete wind-down (migration, emergency shutdown) is a realistic lifecycle event. The attack requires no special privileges and is executable by any external caller.

**Likelihood: Low.**

## Recommendation

Replace the unconditional early return with a branch that still computes the actual backing ratio when protocol holdings are non-zero:

```solidity
if (rsethSupply == 0) {
    uint256 totalETH = _getTotalEthInProtocol();
    if (totalETH == 0) {
        rsETHPrice = 1 ether;
        highestRsethPrice = 1 ether;
    }
    // If totalETH > 0 with zero supply, price is undefined;
    // revert or leave rsETHPrice unchanged to block new mints
    // until the residual is swept or redistributed by governance.
    return;
}
```

Additionally, add a guard in `depositAsset` / `depositETH` that reverts when `rsethSupply == 0` but `_getTotalEthInProtocol() > 0`, preventing any mint until the residual is explicitly resolved.

## Proof of Concept

```
State setup (local fork / unit test):
  sfrxETH.pricePerShare = 1.1e18
  Protocol holds 10 sfrxETH in LRTUnstakingVault (residual after wind-down)
  rsETH.totalSupply() == 0

Step 1: call LRTOracle.updateRSETHPrice()
  → zero-supply branch fires: rsETHPrice = 1e18, highestRsethPrice = 1e18

Step 2: attacker calls LRTDepositPool.depositAsset(sfrxETH, 1e18, 0, "")
  → rsethAmountToMint = 1e18 * 1.1e18 / 1e18 = 1.1e18 rsETH minted
  → protocol now holds 11 sfrxETH total, rsETHSupply = 1.1e18

Step 3: call LRTOracle.updateRSETHPrice()
  → totalETHInProtocol = 11 * 1.1e18 = 12.1e18
  → newRsETHPrice = 12.1e18 / 1.1e18 = 11e18

Step 4: attacker initiates withdrawal for 1.1e18 rsETH
  → expectedAssetAmount = 1.1e18 * 11e18 / 1.1e18 = 11 sfrxETH

Assert: attacker deposited 1 sfrxETH, withdraws 11 sfrxETH.
Invariant broken: first minter extracted 10 sfrxETH of residual yield they did not deposit.
