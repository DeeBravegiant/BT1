### Title
Sub-Transaction Execution Units Excluded from `maxTxExUnits`/`maxBlockExUnits` Validation in Dijkstra Era — (File: `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Utxo.hs`)

---

### Summary

In the Dijkstra era, a top-level transaction may embed an ordered set of sub-transactions, each carrying its own redeemers and Plutus scripts. The `totExUnits` function — used by both the per-transaction (`maxTxExUnits`) and per-block (`maxBlockExUnits`) resource-limit checks — only sums the redeemers attached to the **top-level** transaction's witness set. Sub-transaction redeemers are never included. A transaction author can therefore declare near-zero ExUnits at the top level while packing arbitrarily expensive Plutus scripts into sub-transactions, bypassing both protocol-parameter resource caps.

---

### Finding Description

**`totExUnits` only reads top-level witnesses**

`totExUnits` is defined as:

```haskell
totExUnits tx = foldMap snd $ tx ^. witsTxL . rdmrsTxWitsL . unRedeemersL
``` [1](#0-0) 

For a Dijkstra top-level transaction (`DijkstraTx`), `tx ^. witsTxL` resolves to `dtWits` — the top-level witness set. Sub-transactions are stored in `dtBody ^. subTransactionsTxBodyL` (an `OMap` of `Tx SubTx era`), each carrying its own `dstWits` field with independent redeemers. `totExUnits` never traverses this map. [2](#0-1) 

**Per-transaction limit check uses `totExUnits`**

The Dijkstra UTXO rule applies the same Alonzo check:

```haskell
{- totExunits tx ≤ maxTxExUnits pp -}
runTest $ Alonzo.validateExUnitsTooBigUTxO pp tx
``` [3](#0-2) 

`validateExUnitsTooBigUTxO` calls `totExUnits tx`, which, as shown above, ignores sub-transaction redeemers entirely. [4](#0-3) 

**Per-block limit check also uses `totExUnits`**

The BBODY rule sums ExUnits across all top-level transactions in a block:

```haskell
validateExUnits txs ppMax =
  let txTotal = foldMap totExUnits txs
``` [5](#0-4) 

Because `totExUnits` ignores sub-transactions, the block-level cap `maxBlockExUnits` is equally blind to sub-transaction script costs.

**Sub-transactions are fully executed**

Sub-transactions are processed through the SUBLEDGERS rule before the top-level UTXO/UTXOW rules run:

```haskell
LedgerState utxoStateAfterSubLedgers certStateAfterSubLedgers <-
  trans @(EraRule "SUBLEDGERS" era) $ TRC (...)
``` [6](#0-5) 

Their Plutus scripts are collected and evaluated via `mkDijkstraStAnnSubTx`, which builds a full `dsastPlutusScriptsWithContext` for each sub-transaction: [7](#0-6) 

The scripts are therefore **executed** but their declared ExUnits are **never validated** against `maxTxExUnits` or counted toward `maxBlockExUnits`.

---

### Impact Explanation

The `maxTxExUnits` and `maxBlockExUnits` protocol parameters exist to bound the CPU and memory a node must spend validating a single transaction and a full block, respectively. By hiding script execution cost inside sub-transactions, an attacker can force every honest node to perform arbitrarily large amounts of Plutus evaluation while the block passes all resource-limit checks. This constitutes attacker-controlled transactions exceeding intended validation limits — matching the **Medium** allowed impact: *"Attacker-controlled transactions … exceed intended validation limits."*

---

### Likelihood Explanation

The Dijkstra era is the current production tip of the era chain. Any unprivileged user can submit a top-level transaction containing sub-transactions. No special role, key, or governance majority is required. The only prerequisite is that a block producer includes the transaction, which is the normal mempool flow. Likelihood is **Medium** — the feature is new and the attack requires crafting a valid nested transaction, but no privileged access is needed.

---

### Recommendation

Extend `totExUnits` (or introduce a Dijkstra-specific variant) to recursively sum ExUnits across all sub-transactions:

```haskell
totExUnitsDijkstra tx =
  totExUnits tx
    <> foldMap totExUnits (OMap.elems $ tx ^. bodyTxL . subTransactionsTxBodyL)
```

Use this aggregate in both `validateExUnitsTooBigUTxO` (UTXO rule) and `validateExUnits` (BBODY rule) for the Dijkstra era. Alternatively, apply a per-sub-transaction `maxTxExUnits` check inside the SUBLEDGERS rule and add sub-transaction totals to the block-level accumulator.

---

### Proof of Concept

1. Set `maxTxExUnits = ExUnits { exUnitsMem = 1000, exUnitsSteps = 1000 }` and `maxBlockExUnits = ExUnits { exUnitsMem = 5000, exUnitsSteps = 5000 }`.
2. Craft a Dijkstra top-level transaction with **zero** redeemers in `dtWits` (so `totExUnits tx = ExUnits 0 0`).
3. Embed N sub-transactions, each containing a Plutus script redeemer declaring `ExUnits { exUnitsMem = 4000, exUnitsSteps = 4000 }` (each individually below `maxTxExUnits`, but collectively far above `maxBlockExUnits`).
4. Submit the transaction. The UTXO rule passes (`0 ≤ maxTxExUnits`). The BBODY rule passes (`0 ≤ maxBlockExUnits`). All sub-transaction scripts are executed, consuming `N × 4000` mem/steps of actual node resources — unbounded by any protocol parameter.
