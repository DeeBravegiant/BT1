### Title
Staking Reward Withdrawal Permitted When Delegated DRep Has Expired — (`File: eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Ledger.hs`)

---

### Summary
The `validateWithdrawalsDelegated` function in the Conway LEDGER rule checks only whether a staking key has *any* DRep delegation entry, but does not verify whether the delegated DRep is still active (non-expired). As a result, a staking key whose delegated DRep has expired can still successfully withdraw rewards, bypassing the governance-participation requirement that the formal Agda specification mandates must block such withdrawals.

---

### Finding Description

`validateWithdrawalsDelegated` is invoked during transaction processing in the Conway LEDGER rule to enforce that every key-hash withdrawal address is delegated to a DRep:

```haskell
isNotDRepDelegated keyHash = isNothing $ do
    accountState <- lookupAccountState (KeyHashObj keyHash) accounts
    accountState ^. dRepDelegationAccountStateL
``` [1](#0-0) 

The predicate returns `True` (i.e., "not delegated") only when `dRepDelegationAccountStateL` is `Nothing`. It does **not** consult `vsDRepsL` to check whether the referenced DRep credential is still registered and within its activity window. DRep expiry is tracked separately in `DRepState.drepExpiry` and the `vsNumDormantEpochsL` counter:

```haskell
data DRepState = DRepState
  { drepExpiry :: !EpochNo
  , ...
  }
``` [2](#0-1) 

The actual expiry check used elsewhere in the codebase is:

```haskell
binOpEpochNo (+) (vState ^. vsNumDormantEpochsL) (drep' ^. drepExpiryL) < currentEpoch
``` [3](#0-2) 

This check is absent from `validateWithdrawalsDelegated`. Consequently, once a DRep's activity period lapses, the staking key's `dRepDelegationAccountStateL` field still holds the (now-expired) DRep credential, and the withdrawal check passes.

The production test suite explicitly confirms this behaviour and records the divergence from the formal specification:

```haskell
-- https://github.com/IntersectMBO/formal-ledger-specifications/issues/635
-- TODO: Re-enable after issue is resolved, by removing this override
disableInConformanceIt "Withdraw from a key delegated to a DRep that expired after delegation" $ do
    ...
    isDRepExpired drep `shouldReturn` True
    submitTx_ $ mkBasicTx $ mkBasicTxBody
        & withdrawalsTxBodyL .~ Withdrawals [(ra, balance)]
    -- succeeds in Haskell; Agda spec says it must fail
``` [4](#0-3) 

The `disableInConformanceIt` wrapper marks tests that the Haskell implementation passes but the Agda formal specification rejects, confirming a deterministic rule divergence.

---

### Impact Explanation

The Conway governance design requires that every staking key performing a reward withdrawal must be actively delegated to a live DRep. This is the on-chain mechanism that ties reward access to governance participation. When a DRep expires, the formal specification treats the delegation as no longer satisfying this requirement and mandates that withdrawals be rejected until the key re-delegates to an active DRep.

Because `validateWithdrawalsDelegated` omits the expiry check, a transaction author can withdraw accumulated rewards while their delegated DRep is expired, circumventing the governance-participation gate. This constitutes **withdrawals outside design parameters** as defined by the formal specification, matching the Medium impact tier: *"Attacker-controlled transactions … modify … withdrawals … outside design parameters."*

---

### Likelihood Explanation

DRep expiry is a normal, expected lifecycle event governed by `ppDRepActivity`. Any staking key whose DRep becomes inactive (through natural expiry after `ppDRepActivity` epochs without a governance proposal, or through the dormant-epoch mechanism) is immediately in a state where this bypass is reachable. No special privilege, key compromise, or governance majority is required. Any ordinary transaction sender holding a staking key in this state can craft a withdrawal transaction that the ledger accepts but the formal specification would reject.

---

### Recommendation

Extend `validateWithdrawalsDelegated` (or a companion check in the same LEDGER rule) to verify that the DRep referenced by `dRepDelegationAccountStateL` is not expired at the time of the withdrawal. The expiry predicate already exists as `isDRepExpired` / `vsActualDRepExpiry` in the codebase and should be reused:

```haskell
isNotActiveDRepDelegated keyHash = ...
    dRep <- accountState ^. dRepDelegationAccountStateL
    case dRep of
      DRepCredential cred ->
        case Map.lookup cred (vState ^. vsDRepsL) of
          Nothing -> True   -- unregistered
          Just ds  -> binOpEpochNo (+) numDormant (drepExpiry ds) < currentEpoch
      _ -> False  -- AlwaysAbstain / AlwaysNoConfidence are never "expired"
```

This aligns the Haskell implementation with the Agda formal specification and closes the tracked divergence at https://github.com/IntersectMBO/formal-ledger-specifications/issues/635.

### Proof of Concept

1. Register a staking credential `cred` and a DRep `drep`.
2. Set `ppDRepActivityL = EpochInterval 1` and `ppGovActionLifetimeL = EpochInterval 4`.
3. Submit and expire a governance proposal to generate a reward for `cred`.
4. Submit a governance action to trigger DRep activity tracking, then pass 4 epochs so `isDRepExpired drep` returns `True`.
5. Delegate `cred` to `drep` (now expired) via `DelegTxCert cred (DelegVote (DRepCredential drep))`.
6. Submit a withdrawal transaction for `cred`'s full reward balance.
7. **Observed**: transaction is accepted by the ledger.
8. **Expected per formal spec**: transaction must be rejected because `drep` is expired.
