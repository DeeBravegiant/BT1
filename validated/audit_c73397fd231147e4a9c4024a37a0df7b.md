### Title
`psVRFKeyHashes` Over-Removal via `Map.withoutKeys` on Dangling VRF Keys Allows Duplicate VRF Registration After v11 - (File: `eras/shelley/impl/src/Cardano/Ledger/Shelley/Rules/PoolReap.hs`)

---

### Summary

In `poolReapTransition`, when future pool params are applied at the epoch boundary, the cleanup of "dangling" VRF key hashes uses `Map.withoutKeys` (complete removal) instead of a reference-count decrement. If a VRF key is shared by multiple pools (possible for pools registered before protocol version 11), re-registering one of those pools with a new VRF key causes the shared VRF key to be entirely removed from `psVRFKeyHashes` at the epoch boundary, even though other pools still hold that key. This breaks the VRF uniqueness invariant introduced in v11, allowing a new pool to register with the now-"free" VRF key and creating duplicate VRF keys in the active pool set.

---

### Finding Description

`PState` contains `psVRFKeyHashes :: Map (VRFVerKeyHash StakePoolVRF) (NonZero Word64)`, a reference-counted map introduced at protocol version 11 to enforce VRF key uniqueness across stake pools. [1](#0-0) 

At the v11 hard fork, `populateVRFKeyHashes` initialises this map from both `psStakePools` and `psFutureStakePoolParams`, correctly counting every occurrence of each VRF key, including duplicates that existed before v11. [2](#0-1) 

When a pool re-registers with a new VRF key after v11, `poolTransition` adds the new VRF key to `psVRFKeyHashes` but deliberately leaves the old key's count unchanged (the old key is still live in `psStakePools` until the epoch boundary). [3](#0-2) 

At the epoch boundary, `poolReapTransition` computes `danglingVRFKeyHashes` — the set of current VRF keys that are being replaced by future params — and removes them from `psVRFKeyHashes` using `Map.withoutKeys`: [4](#0-3) [5](#0-4) 

`Map.withoutKeys danglingVRFKeyHashes` **completely removes** the entry for a VRF key regardless of its reference count. If `psVRFKeyHashes[VRF_A] = 2` (two pools share VRF_A from before v11) and one pool re-registers with VRF_B, `danglingVRFKeyHashes = {VRF_A}` and the entire entry for VRF_A is deleted, even though the second pool still holds VRF_A.

Contrast this with `removeVRFKeyHashOccurrences`, which correctly decrements the count and only removes the entry when it reaches zero: [6](#0-5) 

After the incorrect removal, the VRF uniqueness check in `poolTransition` sees `Map.notMember VRF_A psVRFKeyHashes = True` and permits a new pool to register with VRF_A: [7](#0-6) 

This creates two active pools sharing VRF_A — a state that v11 was specifically designed to prevent.

---

### Impact Explanation

Two pools sharing the same VRF key are elected for identical slots (their VRF outputs are identical). Both produce valid blocks for those slots. Honest nodes that receive the two competing blocks in different orders may follow different chains, producing deterministic disagreement in ledger state evaluation. This matches:

**High. Deterministic disagreement between honest nodes from ledger rule evaluation.**

The VRF uniqueness invariant is a ledger-level safety property. Its violation is an invalid ledger state transition accepted by all honest nodes, which is the root cause of the divergence.

---

### Likelihood Explanation

The precondition is that two pools share a VRF key from before v11 (permitted under earlier protocol versions). After the v11 hard fork, `psVRFKeyHashes` is populated with the correct counts for all pre-existing pools, including duplicates. Any pool operator controlling one of the duplicate-VRF pools can trigger the bug by submitting a single re-registration certificate with a fresh VRF key, waiting one epoch, and then registering a new pool with the original shared VRF key. No privileged access, governance majority, or key compromise is required. The entry path is a standard `RegPool` certificate submitted by an unprivileged transaction sender.

---

### Recommendation

Replace `Map.withoutKeys danglingVRFKeyHashes` with a reference-count decrement using the existing `removeVRFKeyHashOccurrences` helper, converting the `Set` to a list:

```haskell
-- current (buggy):
& certPStateL . psVRFKeyHashesL
  %~ ( removeVRFKeyHashOccurrences retiredVRFKeyHashes
         . (`Map.withoutKeys` danglingVRFKeyHashes)
     )

-- fixed:
& certPStateL . psVRFKeyHashesL
  %~ ( removeVRFKeyHashOccurrences retiredVRFKeyHashes
         . removeVRFKeyHashOccurrences (Set.toList danglingVRFKeyHashes)
     )
```

This mirrors the correct treatment of `retiredVRFKeyHashes` and preserves the invariant that `psVRFKeyHashes[v]` equals the number of active or future-staged pools holding VRF key `v`.

---

### Proof of Concept

1. **Before v11**: Submit two `RegPool` certificates for `pool1` and `pool2`, both with `sppVrf = VRF_A`. This is accepted because v11 duplicate-VRF enforcement is not yet active.
2. **Enact v11 hard fork**: `populateVRFKeyHashes` sets `psVRFKeyHashes = {VRF_A: 2}`.
3. **After v11**: Submit a `RegPool` certificate for `pool1` with `sppVrf = VRF_B` (fresh key). Validation passes: `VRF_B ∉ psVRFKeyHashes`. State: `psVRFKeyHashes = {VRF_A: 2, VRF_B: 1}`, `psFutureStakePoolParams[pool1] = {vrf: VRF_B}`.
4. **Pass epoch**: `poolReapTransition` computes `danglingVRFKeyHashes = {VRF_A}` (pool1 is changing VRF). `Map.withoutKeys {VRF_A}` removes VRF_A entirely. State: `psVRFKeyHashes = {VRF_B: 1}`. `pool2` still holds VRF_A in `psStakePools`.
5. **Register pool3 with VRF_A**: Validation: `Map.notMember VRF_A psVRFKeyHashes = True`. Certificate accepted. `psVRFKeyHashes = {VRF_A: 1, VRF_B: 1}`.
6. **Result**: `pool2` and `pool3` both hold VRF_A. Both are elected for the same slots. Both produce valid competing blocks, enabling chain divergence between honest nodes. [4](#0-3) [5](#0-4) [2](#0-1)

    removeVRFKeyHashOccurrences ::
      [VRFVerKeyHash StakePoolVRF] ->
      Map (VRFVerKeyHash StakePoolVRF) (NonZero Word64) ->
      Map (VRFVerKeyHash StakePoolVRF) (NonZero Word64)
    removeVRFKeyHashOccurrences vrfs vrfsMap = F.foldl' (flip removeVRFKeyHashOccurrence) vrfsMap vrfs
    removeVRFKeyHashOccurrence =
      -- Removes the key from the map if the value drops to 0
      Map.update (mapNonZero (\n -> n - 1))
