### Title
Single Byzantine Participant Can Freeze All Signing Operations via `verify_tee()` — (File: `crates/contract/src/lib.rs`)

---

### Summary

The `MpcContract` contains an `accept_requests` boolean flag that is a direct analog to `stETH.isStopped()`. When set to `false`, every user-facing request method (`sign`, `request_app_private_key`, `verify_foreign_transaction`) panics immediately, and responses to already-queued pending requests are also rejected. A single Byzantine participant strictly below the signing threshold can deliberately engineer the conditions that cause any participant's call to `verify_tee()` to flip this flag to `false`, freezing the entire signing service.

---

### Finding Description

`MpcContract` stores a field `accept_requests: bool` (initialized `true`). Every user-facing request is gated through `check_request_preconditions()`: [1](#0-0) 

```rust
// 4. Refuse the request if the contract is not currently accepting requests
if !self.accept_requests {
    env::panic_str(&TeeError::TeeValidationFailed.to_string())
}
```

The flag is set to `false` inside `verify_tee()`: [2](#0-1) 

```rust
// kicking out participants would break the threshold relation
self.accept_requests = false;
return Ok(false);
```

`verify_tee()` is callable by **any single participant** (enforced only by `voter_or_panic()`): [3](#0-2) 

The freeze also blocks responses to already-queued `verify_foreign_transaction` requests: [4](#0-3) 

```rust
if !self.accept_requests {
    return Err(TeeError::TeeValidationFailed.into());
}
```

**Attack path:**

1. Byzantine participant **P** (one node, below threshold) deliberately allows their TEE attestation to expire or become invalid — no key material or privileged access is required.
2. The network is in a configuration where removing P would violate the threshold relation (e.g., a `t-of-n` network where `n - 1 < t`). This is a realistic steady-state for a tightly-configured production cluster.
3. **Any** participant (including P itself) calls `verify_tee()`. The function calls `reverify_and_cleanup_participants()`, detects P's invalid attestation, determines that kicking P out would break the threshold, and sets `self.accept_requests = false`.
4. From this point: all new `sign` / `request_app_private_key` / `verify_foreign_transaction` calls panic with `TeeValidationFailed`; all already-queued `verify_foreign_transaction` responses are rejected; pending requests time out after ~200 blocks.

The contract's own test confirms this behavior: [5](#0-4) 

Recovery requires **manual operator intervention** — there is no automatic or threshold-gated path back to `accept_requests = true` once the threshold-breaking condition is present.

---

### Impact Explanation

- **Request-lifecycle freeze**: All pending and future signing requests are blocked. Callers whose `sign` yields are already open cannot receive a response; those requests time out and the 1-yoctonear deposit is returned, but the cross-chain operation they were authorizing is permanently lost.
- **Bridge / foreign-chain flow breakage**: `verify_foreign_transaction` responses are explicitly rejected when `accept_requests = false`, meaning any bridge flow that depends on MPC-verified foreign-chain proofs is halted.
- **No automatic recovery**: The flag is only reset to `true` by a subsequent successful `verify_tee()` call, which requires the degraded participant to be removed — impossible if removal itself breaks the threshold.

This matches the allowed Medium impact: *"request-lifecycle, participant-state, or contract execution-flow manipulation that breaks production safety/accounting invariants."*

---

### Likelihood Explanation

- Requires one Byzantine participant to deliberately let their TEE attestation lapse — a passive action requiring no special capability.
- The threshold-breaking condition (`n - 1 < t`) is the normal operating point for many production MPC clusters (e.g., 3-of-3 or 4-of-5 with one node already degraded).
- `verify_tee()` is a routine maintenance call that any participant can submit at any time; the attacker does not need to be the one who calls it.
- No collusion above one participant is required.

---

### Recommendation

1. **Decouple new-request gating from response gating.** Pending requests that were accepted before the TEE degradation event should still be fulfillable. Remove the `accept_requests` check from `respond_verify_foreign_tx` (and the equivalent `respond` path for signatures) so that in-flight work can complete.

2. **Require threshold votes to enter the frozen state.** Rather than a single `verify_tee()` call setting `accept_requests = false`, require a threshold of participants to independently confirm the degraded state before freezing.

3. **Add a time-bounded grace period.** Before setting `accept_requests = false`, emit an on-chain warning and allow a governance window (e.g., N blocks) during which the degraded participant can renew their attestation, preventing a single lapsed attestation from immediately freezing the network.

---

### Proof of Concept

```
Setup: 3-of-3 MPC cluster (threshold = 3, participants = [A, B, C])

Step 1: Participant C (Byzantine) deliberately does not renew their TEE attestation.
        C's attestation expires on-chain.

Step 2: Participant A (honest) calls verify_tee() as part of routine maintenance.

Step 3: reverify_and_cleanup_participants() detects C's expired attestation.
        Remaining valid participants = {A, B} = 2, but threshold = 3.
        validate_governance_against_reconstruction(2, threshold=3, ...) returns Err.
        → self.accept_requests = false

Step 4: User calls sign(...) with 1 yoctonear deposit.
        check_request_preconditions() hits:
            if !self.accept_requests { env::panic_str("TEE validation failed") }
        Transaction reverts. Signing is frozen.

Step 5: Any already-queued verify_foreign_transaction response attempt:
            if !self.accept_requests { return Err(TeeValidationFailed) }
        Pending bridge proofs cannot be delivered. They time out after ~200 blocks.

Recovery: Requires manual operator intervention to add a new participant or
          restructure the threshold — no automatic path exists.
``` [6](#0-5) [7](#0-6) [8](#0-7)

    /// Common preconditions enforced on every user-facing request method (`sign`,
    /// `request_app_private_key`, `verify_foreign_transaction`):
    ///
    /// 1. The target domain exists and its purpose matches `expected_purpose`.
    /// 2. The caller attached enough prepaid gas to perform the yield/resume flow.
    /// 3. The caller attached at least `minimum_deposit` (excess is refunded).
    /// 4. The contract is currently accepting user requests.
    ///
    /// Returns the validated domain config and the caller's account id.
    fn check_request_preconditions(
        &self,
        domain_id: DomainId,
        expected_purpose: DomainPurpose,
        minimum_gas: Gas,
        minimum_deposit: NearToken,
    ) -> (DomainConfig, AccountId) {
        // 1. Look up the domain and check its purpose.
        let domains = match self.protocol_state.domain_registry() {
            Ok(domains) => domains,
            Err(err) => env::panic_str(&err.to_string()),
        };
        let Some(domain_config) = domains.get_domain_by_domain_id(domain_id) else {
            env::panic_str(
                &InvalidParameters::DomainNotFound {
                    provided: domain_id,
                }
                .to_string(),
            );
        };
        if domain_config.purpose != expected_purpose {
            env::panic_str(
                &InvalidParameters::WrongDomainPurpose {
                    domain_id: domain_config.id,
                    expected: expected_purpose,
                    actual: domain_config.purpose,
                }
                .to_string(),
            );
        }
        let domain_config = domain_config.clone();

        // 2. Make sure the call will not run out of gas doing yield/resume logic.
        let prepaid_gas = env::prepaid_gas();
        if prepaid_gas < minimum_gas {
            env::panic_str(
                &InvalidParameters::InsufficientGas {
                    provided: prepaid_gas.as_gas(),
                    required: minimum_gas.as_gas(),
                }
                .to_string(),
            );
        }

        // 3. Require the minimum deposit and refund any excess.
        let predecessor = env::predecessor_account_id();
        require_deposit(minimum_deposit, &predecessor);

        // 4. Refuse the request if the contract is not currently accepting requests
        //    (e.g. because TEE validation has failed).
        if !self.accept_requests {
            env::panic_str(&TeeError::TeeValidationFailed.to_string())
        }

        (domain_config, predecessor)
    }

    pub fn verify_tee(&mut self) -> Result<bool, Error> {
        log!("verify_tee: signer={}", env::signer_account_id());
        // Caller must be a participant (node or operator).
        self.voter_or_panic();
        let ProtocolContractState::Running(running_state) = &mut self.protocol_state else {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        };
        let current_params = running_state.parameters.clone();

        let tee_upgrade_deadline_duration =
            Duration::from_secs(self.config.tee_upgrade_deadline_duration_seconds);

        match self.tee_state.reverify_and_cleanup_participants(
            current_params.participants(),
            tee_upgrade_deadline_duration,
        ) {
            TeeValidationResult::Full => {
                self.accept_requests = true;
                log!("All participants have an accepted Tee status");
                Ok(true)
            }
            TeeValidationResult::Partial {
                participants_with_valid_attestation,
            } => {
                let remaining = participants_with_valid_attestation.len();
                // Defense in depth: the surviving participant set must keep the full
                // threshold relation intact — the GovernanceThreshold must still sit
                // within its bounds for the smaller set (in particular it must not
                // exceed the remaining participant count or the upper cap) and must
                // remain at least every domain's ReconstructionThreshold (the kickout
                // keeps the existing per-domain thresholds). Otherwise we refuse and
                // wait for manual intervention.
                let max_reconstruction_threshold =
                    max_reconstruction_threshold(running_state.domains.domains());
                if let Err(err) = ThresholdParameters::validate_governance_against_reconstruction(
                    u64::try_from(remaining).expect("participant count fits in u64"),
                    current_params.threshold(),
                    max_reconstruction_threshold,
                ) {
                    log!(
                        "Kicking out participants with an invalid TEE status would break the threshold relation ({:?}); {} participants remain with a valid TEE status. This requires manual intervention. We will not accept new signature requests as a safety precaution.",
                        err,
                        remaining,
                    );
                    self.accept_requests = false;
                    return Ok(false);
                }

                // here, we set it to true, because at this point, we have at least `threshold`
                // number of participants with an accepted Tee status.
                self.accept_requests = true;

                // do we want to adjust the threshold?
                //let n_participants_new = new_participants.len();
                //let new_threshold = (3 * n_participants_new + 4) / 5; // minimum 60%
                //let new_threshold = new_threshold.max(2); // but also minimum 2
                let new_threshold = usize::try_from(current_params.threshold().value())
                    .expect("threshold value fits in usize");

                let threshold_parameters = ThresholdParameters::new(
                    participants_with_valid_attestation,
                    Threshold::new(new_threshold as u64),
                )
                .expect("Require valid threshold parameters"); // this should never happen.
                current_params.validate_incoming_proposal(&threshold_parameters)?;
                // This resharing only changes the participant set, so the
                // per-domain reconstruction-threshold updates map is empty.
                let proposed_parameters =
                    ProposedThresholdParameters::new(threshold_parameters, BTreeMap::new());
                let res = running_state.transition_to_resharing_no_checks(&proposed_parameters);
                if let Some(resharing) = res {
                    self.protocol_state = ProtocolContractState::Resharing(resharing);
                }

                Ok(true)
            }
        }
    }
