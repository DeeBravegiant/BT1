Audit Report

## Title
Fee-Before-Price Ordering in `_updateRsETHPrice` Allows Block Stuffing to Freeze `rsETHPrice` via `DailyFeeMintLimitExceeded` - (File: contracts/LRTOracle.sol)

## Summary

In `_updateRsETHPrice()`, the daily fee mint limit check at line 303 executes before `rsETHPrice` is committed at line 313. When `updateRSETHPrice()` is not called for ≥2 consecutive days (e.g., via block stuffing), the accumulated multi-period fee exceeds `maxFeeMintAmountPerDay`, causing `_checkAndUpdateDailyFeeMintLimit` to revert unconditionally and leaving `rsETHPrice` permanently stale until the manager raises the daily cap.

## Finding Description

The execution path in `_updateRsETHPrice()` is confirmed by the source:

- **Lines 244–246**: `rewardAmount = totalETHInProtocol - previousTVL` accumulates **all** rewards since the last successful price update, not one day's worth.
- **Line 301**: `rsethAmountToMintAsProtocolFee = protocolFeeInETH.divWad(newRsETHPrice)` scales linearly with the number of missed periods.
- **Line 303**: `_checkAndUpdateDailyFeeMintLimit(rsethAmountToMintAsProtocolFee)` fires **before** the price is committed.
- **Line 313**: `rsETHPrice = newRsETHPrice` is **never reached** if line 303 reverts.

Inside `_checkAndUpdateDailyFeeMintLimit()` (lines 197–210):

```solidity
if (block.timestamp >= feePeriodStartTime + 1 days) {
    currentPeriodMintedFeeAmount = 0;          // resets correctly
    feePeriodStartTime = getCurrentPeriodStartTime();
}
if (currentPeriodMintedFeeAmount + feeAmount > maxFeeMintAmountPerDay) {
    revert DailyFeeMintLimitExceeded(...);     // feeAmount is N-days worth
}
```

The period reset correctly zeroes `currentPeriodMintedFeeAmount`, but `feeAmount` passed in is the **entire multi-period fee**. If the oracle was not updated for N ≥ 2 days, `feeAmount ≈ N × dailyFee`. When `maxFeeMintAmountPerDay` is calibrated to one day's expected fee, the check reverts unconditionally on every subsequent call. Both `updateRSETHPrice()` (public) and `updateRSETHPriceAsManager()` (manager-only) call the same `_updateRsETHPrice()` internal function, so the manager path is equally blocked until `setMaxFeeMintAmountPerDay()` is called first.

## Impact Explanation

`rsETHPrice` storage is never updated (line 313 is unreachable while the revert condition holds). `LRTDepositPool.getRsETHAmountToMint()` at line 520 divides by `lrtOracle.rsETHPrice()` — a stale price means all deposit minting uses an incorrect exchange rate. The freeze persists until the manager calls `setMaxFeeMintAmountPerDay()` with a sufficiently higher value, which is an out-of-band admin action. This constitutes a **Low — Block stuffing** impact, which is explicitly within the allowed impact scope.

## Likelihood Explanation

Block stuffing for 2+ consecutive days on Ethereum mainnet is extremely capital-intensive (filling every block at the gas limit for 48+ hours). However, the precondition only requires **2 missed periods** — a single day of stuffing suffices if `maxFeeMintAmountPerDay` is set tightly to one day's expected fee. The same freeze can also be triggered by an unintentional keeper outage with no attacker required, making the code path reachable in normal operational failure scenarios. `maxFeeMintAmountPerDay` is set by the manager and can be calibrated to exactly one day's expected fee, making the off-by-one condition easy to hit.

## Recommendation

Move `rsETHPrice = newRsETHPrice` **before** the fee check, or restructure `_checkAndUpdateDailyFeeMintLimit` to cap the minted fee at `maxFeeMintAmountPerDay` rather than reverting. A non-reverting cap ensures the price is always updated even when the fee exceeds the daily limit:

```solidity
// Option A: always commit price first, then cap fee
rsETHPrice = newRsETHPrice;
uint256 cappedFee = _capAndUpdateDailyFeeMintLimit(rsethAmountToMintAsProtocolFee);
// mint cappedFee instead of full fee
```

Alternatively, split the fee accumulation across multiple virtual periods proportionally to elapsed time.
