Audit Report

## Title
`_withdrawFromAave` Principal Cap Causes `completeWithdrawal` to Revert When Accrued Interest Bridges the Gap — (`contracts/LRTWithdrawalManager.sol`)

## Summary
`_withdrawFromAave` caps withdrawals at `min(aaveAWETH.balanceOf, totalETHDepositedToAave)` to reserve accrued interest for the treasury. When interest has accrued, a user whose `expectedAssetAmount` falls in the range `(totalETHDepositedToAave, aaveAWETH.balanceOf]` will have their withdrawal silently capped short, causing the post-withdrawal balance check in `_processWithdrawalCompletion` to revert with `InsufficientLiquidityForWithdrawal` even though the full aWETH balance would cover the request. The user's rsETH has already been burned and they cannot reclaim it.

## Finding Description
In `_withdrawFromAave` (L905–921), the withdrawal ceiling is computed as:

```solidity
uint256 withdrawablePrincipal = aaveBalance < totalETHDepositedToAave
    ? aaveBalance
    : totalETHDepositedToAave;
withdrawnAmount = amount > withdrawablePrincipal ? withdrawablePrincipal : amount;
```

When `aaveBalance > totalETHDepositedToAave` (i.e., interest has accrued), `withdrawablePrincipal = totalETHDepositedToAave`. Any `amount` exceeding `totalETHDepositedToAave` is silently capped.

In `_processWithdrawalCompletion` (L719–732), after calling `_withdrawFromAave(amountNeeded)`, the contract checks:

```solidity
uint256 balanceAfter = address(this).balance;
if (balanceAfter < request.expectedAssetAmount) {
    revert InsufficientLiquidityForWithdrawal();
}
```

If `request.expectedAssetAmount > totalETHDepositedToAave` but `<= aaveAWETH.balanceOf`, the withdrawal is capped, `balanceAfter < request.expectedAssetAmount`, and the call reverts. The user's rsETH was burned at `unlockQueue` time; they cannot reclaim it and cannot complete the withdrawal. Calling `collectInterestToTreasury()` before `completeWithdrawal` does not resolve the issue — it reduces `aaveBalance` to `totalETHDepositedToAave`, which is still less than `request.expectedAssetAmount`.

## Impact Explanation
**Low — Contract fails to deliver promised returns, but doesn't lose value.** The contract holds sufficient aWETH to honour the request but refuses to use the interest portion, blocking the user's ETH redemption. No funds are permanently destroyed, but the user is unable to receive the ETH they are owed on demand.

## Likelihood Explanation
No attacker action is required. Interest accrues continuously and passively on any ETH deposited to Aave. Any user whose `expectedAssetAmount` (set at initiation time using the rsETH price) falls in the range `(totalETHDepositedToAave, aaveAWETH.balanceOf]` will be affected. This is reachable by any unprivileged user through the normal `completeWithdrawal` public call path.

## Recommendation
In `_withdrawFromAave`, when called from the withdrawal completion path, use `aaveBalance` as the ceiling instead of `totalETHDepositedToAave`, and floor `totalETHDepositedToAave` at zero when interest is consumed. Alternatively, pass a flag to `_withdrawFromAave` indicating whether interest may be used, and adjust `totalETHDepositedToAave` accordingly (e.g., `totalETHDepositedToAave = withdrawnAmount > totalETHDepositedToAave ? 0 : totalETHDepositedToAave - withdrawnAmount`). The interest-reservation logic should be relaxed when user redemptions require it.

## Proof of Concept
```
State:
  totalETHDepositedToAave = 100 ETH
  aaveAWETH.balanceOf(withdrawalManager) = 105 ETH  (5 ETH interest accrued)
  address(withdrawalManager).balance = 0 ETH
  request.expectedAssetAmount = 102 ETH

Call: completeWithdrawal(ETH_TOKEN, "")
  → _processWithdrawalCompletion (L719–732)
      contractBalance = 0 < 102  → enter Aave branch
      amountNeeded = 102
      _withdrawFromAave(102) (L905–921):
          aaveBalance = 105
          withdrawablePrincipal = min(105, 100) = 100   ← capped at principal
          withdrawnAmount = min(102, 100) = 100
          withdraws 100 ETH; totalETHDepositedToAave = 0
      balanceAfter = 100 < 102
      → revert InsufficientLiquidityForWithdrawal        ← user withdrawal frozen

Invariant violated:
  aaveAWETH.balanceOf (105) >= request.expectedAssetAmount (102)
  yet completeWithdrawal reverts.
```

Foundry fork test plan: deploy against a mainnet fork with Aave v3, deposit 100 ETH via `_depositToAave`, fast-forward blocks to accrue ~5 ETH interest, initiate a withdrawal for 102 ETH worth of rsETH, unlock the queue, then call `completeWithdrawal` and assert it reverts with `InsufficientLiquidityForWithdrawal` despite `aaveAWETH.balanceOf >= 102 ETH`.
