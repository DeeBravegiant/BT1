### Title
Missing Cross-Sub-Transaction Uniqueness Check Allows Same Deposit Refund to Be Claimed Multiple Times - (`eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/SubLedger.hs`, `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Utxo.hs`)

---

### Summary

In the Dijkstra era, a top-level transaction can embed multiple sub-transactions via `subTransactionsTxBodyL`. There is no cross-sub-transaction check preventing two or more sub-transactions from each including an `UnRegDepositTxCert` for the **same** staking credential. Because the `SUBUTXOW` rule for each sub-transaction computes value conservation using the cert state **before** that sub-transaction's own certificates are applied, and because the `consumed` calculation includes the deposit refund based solely on the presence of the cert in the tx body (not on whether the credential is still registered), an attacker can construct a batch that claims the same deposit refund multiple times, creating ADA out of thin air.

The developers have already identified this scenario as a known open issue: the test `xit "Multiple subtransactions cannot get the same refund"` is explicitly disabled with `error "TODO: predicate failure not yet implemented"`, confirming no enforcement exists yet.

---

### Finding Description

**Root cause — missing batch-level refund uniqueness check**

The `SUBLEDGERS` rule processes sub-transactions sequentially via `foldM`, threading the `LedgerState` (including `CertState`) through each step: [1](#0-0) 

Inside each `SUBLEDGER` invocation, `SUBENTITIES` is called with the **input** `certState` (the state as it arrived for this sub-transaction), and `SUBUTXOW` is also called with that same **input** `certState`: [2](#0-1) 

The value-conservation check inside `SUBUTXOW` → `SUBUTXO` calls `validateValueNotConservedUTxO`, which invokes `consumed`. The `consumed` function adds a deposit refund for every `UnRegDepositTxCert` found in the tx body, **regardless of whether the credential is still registered in the cert state at that point**: [3](#0-2) 

Compare this with `validateBatchWithdrawals`, which **does** perform a cross-sub-transaction aggregate check for withdrawals: [4](#0-3) 

No equivalent `validateBatchRefunds` function exists. There is no check that the union of all `UnRegDepositTxCert` credentials across all sub-transactions is a set of distinct credentials.

**Developer acknowledgement**

The test file explicitly marks this scenario as a known open issue: [5](#0-4) 

The `xit` marker disables the test, and `error "TODO: predicate failure not yet implemented"` confirms no rejection predicate has been implemented.

---

### Impact Explanation

**Critical — Direct creation of ADA through an invalid ledger state transition.**

An attacker who controls a staking credential with a registered deposit can construct a top-level Dijkstra transaction containing N sub-transactions, each bearing `UnRegDepositTxCert stakingCred deposit`. If the batch is accepted, the deposit is refunded N times while only one deposit existed. This inflates the circulating ADA supply by `(N-1) × deposit`, constituting a direct invalid ledger state transition that creates ADA out of thin air.

---

### Likelihood Explanation

**High.** The attack requires only:
1. A registered staking credential (any user can register one).
2. Constructing a valid top-level Dijkstra transaction with multiple sub-transactions — an operation available to any unprivileged transaction sender.

No privileged role, governance majority, or leaked key is required. The Dijkstra era is the current development target, so this path will be reachable on mainnet when the era activates.

---

### Recommendation

Add a batch-level uniqueness check analogous to `validateBatchWithdrawals`. Before processing sub-transactions, collect all `UnRegDepositTxCert` credentials across the top-level tx body and all sub-transaction bodies, and reject the batch if any credential appears more than once. Concretely:

```haskell
validateBatchRefunds :: DijkstraEraTxBody era => Tx TopTx era -> Test (DijkstraUtxoPredFailure era)
validateBatchRefunds tx =
  let allUnregCreds = concatMap unregCreds $
        (tx ^. bodyTxL) : OMap.elems (tx ^. bodyTxL . subTransactionsTxBodyL)
      duplicates = findDuplicates allUnregCreds
  in failureOnNonEmpty duplicates DuplicateDepositRefundInBatch
```

This mirrors the existing `validateBatchWithdrawals` pattern and closes the gap. [4](#0-3) 

---

### Proof of Concept

1. Register staking credential `C` with deposit `D` ADA.
2. Construct sub-tx₁ with `certsTxBodyL = [UnRegDepositTxCert C D]` and outputs totalling `D` ADA more than inputs (claiming the refund).
3. Construct sub-tx₂ identically to sub-tx₁ (same cert, same output claiming `D` ADA).
4. Construct a top-level Dijkstra tx with `subTransactionsTxBodyL = OMap.fromFoldable [subTx₁, subTx₂]`.
5. Submit. `SUBLEDGERS` processes sub-tx₁ (credential unregistered, `D` ADA refunded), then sub-tx₂. Because no cross-sub-transaction refund uniqueness check exists, sub-tx₂ also passes value conservation (the `consumed` calculation counts the cert-body refund again) and `D` ADA is credited a second time.
6. Net effect: `D` ADA created from nothing; the attacker's outputs contain `2D` ADA while only `D` was deposited.

The developers' own disabled test at `eras/dijkstra/impl/testlib/Test/Cardano/Ledger/Dijkstra/Imp/CertSpec.hs:53–75` describes exactly this scenario and confirms no predicate failure is currently raised. [5](#0-4)

-- | For each account, the total withdrawals across the entire batch should not exceed the original account balance.
-- Unregistered accounts are treated as having 0 balance.
validateBatchWithdrawals ::
  ( EraTx era
  , EraAccounts era
  , DijkstraEraTxBody era
  ) =>
  Accounts era ->
  Tx TopTx era ->
  Test (DijkstraUtxoPredFailure era)
validateBatchWithdrawals accounts tx =
  let allWithdrawals =
        Map.unionsWith (<>) $
          unWithdrawals (tx ^. bodyTxL . withdrawalsTxBodyL)
            : [ unWithdrawals $ subTx ^. bodyTxL . withdrawalsTxBodyL
              | subTx <- OMap.elems $ tx ^. bodyTxL . subTransactionsTxBodyL
              ]
      badWithdrawals =
        Map.mapMaybeWithKey
          ( \acctAddr withdrawn ->
              let balance = getAccountBalance acctAddr
               in if withdrawn > balance
                    then Just Mismatch {mismatchSupplied = withdrawn, mismatchExpected = balance}
                    else Nothing
          )
          allWithdrawals
   in failureOnNonEmptyMap badWithdrawals WithdrawalsExceedAccountBalance
