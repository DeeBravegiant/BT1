### Title
Solana `log_metadata` Accepts Uninitialized Metaplex PDA, Permanently Registering Token on NEAR with Empty Metadata — (`solana/programs/bridge_token_factory/src/instructions/user/log_metadata.rs`)

---

### Summary

`parse_metadata_account` silently returns empty strings when the Metaplex PDA is not yet owned by the Metaplex program. No downstream guard rejects an empty `name`/`symbol` before the Wormhole message is posted. NEAR's `deploy_token_callback` and `deploy_token_internal` also perform no such check. Once the token slot is occupied (even with empty metadata), the `TokenExists` guard permanently blocks any corrective re-registration.

---

### Finding Description

**Step 1 — Solana: empty strings returned without error**

In `parse_metadata_account`, the ownership branch is:

```rust
if metadata.owner == &MetaplexID {
    // deserialize and return real name/symbol
} else {
    Ok((String::default(), String::default()))   // ← silent empty-string return
}
``` [1](#0-0) 

When the Metaplex PDA exists on-chain but is still owned by the system program (not yet initialized), the `else` branch fires and `("", "")` is returned. The function does not error.

**Step 2 — Solana: no guard before `post_message`**

`process()` takes whatever `(name, symbol)` is returned and immediately serializes it into a `LogMetadataPayload` and posts it via Wormhole — no non-empty assertion anywhere:

```rust
let payload = LogMetadataPayload {
    token: self.mint.key(),
    name: name.trim_end_matches('\0').to_string(),   // "" after trim
    symbol: symbol.trim_end_matches('\0').to_string(), // ""
    decimals: self.mint.decimals,
}
.serialize_for_near(())?;
self.common.post_message(payload)?;
``` [2](#0-1) 

**Step 3 — NEAR: `deploy_token_callback` passes empty metadata straight through**

`deploy_token_callback` extracts `metadata.name` and `metadata.symbol` from the verified VAA and forwards them to `deploy_token_internal` with no emptiness check: [3](#0-2) 

**Step 4 — NEAR: `deploy_token_internal` permanently occupies the token slot**

The token ID is derived deterministically from the mint address. `deployed_tokens.insert` is guarded by `require!(..., BridgeError::TokenExists)`, so the slot is permanently taken after the first successful call: [4](#0-3) 

The deployer then creates the NEAR token contract with empty `name`/`symbol` — no validation in the deployer either: [5](#0-4) 

**Step 5 — No recovery path**

There is no `update_metadata` or equivalent permissionless or DAO function for Solana-originated tokens in the NEAR bridge contract. Once the slot is taken, the correct metadata can never be registered.

**Contrast with NEAR's own `log_metadata`**

NEAR's native `log_metadata_callback` explicitly rejects empty strings:

```rust
require!(
    !metadata.name.is_empty() && !metadata.symbol.is_empty(),
    BridgeError::InvalidMetadata.as_ref()
);
``` [6](#0-5) 

The Solana path has no equivalent guard.

---

### Impact Explanation

An unprivileged attacker permanently registers any classic SPL token on NEAR with empty `name` and `symbol` before the legitimate Metaplex metadata is created. All subsequent `deploy_token` calls for the same mint fail with `TokenExists`. The NEAR-side token contract is deployed with empty metadata and cannot be corrected. This is a permanent, irreversible asset-identity corruption: every wallet, DEX, and bridge UI that queries the NEAR token will see blank name/symbol, and the legitimate token identity can never be established on NEAR for that mint.

---

### Likelihood Explanation

The precondition — a classic SPL mint whose Metaplex PDA has not yet been initialized — is common during token launches (mint is created first, metadata is added in a separate transaction). The attack window is the gap between mint creation and metadata initialization. The call is fully permissionless, requires no special accounts, and costs only the Wormhole message fee. The attacker only needs to monitor the mempool or chain for new mint accounts.

---

### Recommendation

Add a non-empty guard in `process()` immediately after `(name, symbol)` is resolved, mirroring the check already present in NEAR's `log_metadata_callback`:

```rust
require!(
    !name.is_empty() && !symbol.is_empty(),
    ErrorCode::InvalidTokenMetadata
);
```

This should be placed after line 128 (after the `else` branch of the token-program check) and before the `LogMetadataPayload` is constructed. [7](#0-6) 

Optionally, also add the same guard inside `deploy_token_callback` on NEAR as a defense-in-depth layer.

---

### Proof of Concept

1. Create a classic SPL mint (token program, not Token-2022). Do **not** initialize the Metaplex metadata PDA yet.
2. Call `log_metadata` on the Solana bridge program, passing the correct (but uninitialized) Metaplex PDA as `metadata`.
3. `parse_metadata_account` sees `metadata.owner == system_program`, returns `("", "")`.
4. A Wormhole VAA is emitted with `name = ""`, `symbol = ""`, `decimals = X`.
5. Submit the VAA to NEAR's `deploy_token`. `deploy_token_callback` calls `deploy_token_internal` with empty metadata.
6. The NEAR token contract is created with empty name/symbol; `deployed_tokens.insert` succeeds.
7. Now initialize the Metaplex metadata PDA with the real name/symbol.
8. Call `log_metadata` again (legitimate path). A new VAA is emitted with correct metadata.
9. Submit the new VAA to NEAR's `deploy_token` → **panics with `ERR_TOKEN_EXISTS`**.
10. The token is permanently registered on NEAR with empty name/symbol; no recovery path exists.

The existing test fixture in `test_log_metadata.rs` already sets up the exact precondition (correct Metaplex PDA address, owned by system program) and confirms the instruction **succeeds** rather than reverting — directly demonstrating the missing guard. [8](#0-7)
