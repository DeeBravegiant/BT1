Audit Report

## Title
Excess `msg.value` Permanently Locked in `MultiChainRateProvider.updateRate()` — No ETH Withdrawal Path - (File: contracts/cross-chain/MultiChainRateProvider.sol)

## Summary
`MultiChainRateProvider.updateRate()` is `payable` and forwards exactly the on-chain `estimatedFee` to each LayerZero receiver in a loop, but never refunds the difference between `msg.value` and the sum of fees consumed. The contract defines no `receive()` fallback, no withdrawal function, and no sweep mechanism, so any excess ETH sent by callers is permanently locked. Both concrete implementations (`RSETHMultiChainRateProvider`, `AGETHMultiChainRateProvider`) inherit this abstract contract and add no recovery path.

## Finding Description
In `MultiChainRateProvider.updateRate()` (L108–137), for each registered `RateReceiver` the function calls `ILayerZeroEndpoint.estimateFees()` on-chain and forwards exactly that amount:

```solidity
(uint256 estimatedFee,) = ILayerZeroEndpoint(layerZeroEndpoint)
    .estimateFees(dstChainId, address(this), _payload, false, bytes(""));

ILayerZeroEndpoint(layerZeroEndpoint).send{ value: estimatedFee }(
    dstChainId, remoteAndLocalAddresses, _payload, payable(msg.sender), address(0x0), bytes("")
);
```

After the loop completes, `address(this).balance` retains any ETH not consumed by the fee sum. A grep over `MultiChainRateProvider.sol` for `receive()`, `withdraw`, `sweep`, and `rescue` returns zero matches — there is no mechanism to recover this balance. `RSETHMultiChainRateProvider` and `AGETHMultiChainRateProvider` are thin wrappers that only implement `getLatestRate()` and add no recovery function.

By contrast, `CrossChainRateProvider.updateRate()` (L96) passes `msg.value` directly to the single `send()` call, so LayerZero's endpoint handles any refund natively and no ETH accumulates in that contract.

Because `estimateFees()` is a view call whose result can differ from the fee actually charged at execution time (due to block-to-block oracle/gas price changes), callers must routinely send a buffer above the estimated total to avoid reverts. This buffer is the ETH that accumulates and cannot be recovered.

## Impact Explanation
Any ETH sent in excess of `Σ estimatedFee_i` is permanently frozen in the contract. There is no upgrade mechanism, no owner sweep, and no refund path. This directly matches the **Critical — Permanent freezing of funds** impact class. The funds belong to the caller (not the protocol), but the contract's design makes their loss inevitable for any caller who sends a safety buffer, and the loss is irreversible.

## Likelihood Explanation
`updateRate()` has no access control; any address may call it. Callers are directed by the contract's own NatSpec to consult LayerZero fee estimation docs and supply ETH. Because on-chain fee estimates can shift between the estimation call and the `send()` call, any prudent caller will include a buffer. With multiple receivers registered, the buffer grows proportionally. This is a recurring, realistic scenario — not a one-time edge case — and every call with a non-zero buffer permanently locks that excess.

## Recommendation
After the loop in `updateRate()`, refund any remaining ETH to `msg.sender`:

```solidity
// after the for-loop
uint256 remaining = address(this).balance;
if (remaining > 0) {
    (bool ok,) = payable(msg.sender).call{ value: remaining }("");
    require(ok, "ETH refund failed");
}
```

Alternatively, add an `onlyOwner` ETH recovery function, or restructure the function to accept an exact per-receiver fee array from the caller and revert if `msg.value` does not match the sum exactly.

## Proof of Concept
1. Deploy `RSETHMultiChainRateProvider` with two registered receivers on a fork.
2. Call `estimateTotalFee()` off-chain; suppose it returns `0.010 ETH`.
3. Call `updateRate{ value: 0.011 ether }()` from any EOA.
4. The loop sends `estimatedFee` (≈0.005 ETH) to each receiver; total consumed ≈ 0.010 ETH.
5. Assert `address(multiChainRateProvider).balance == 0.001 ether` after the call.
6. Attempt every function in the contract — none allows withdrawal. The 0.001 ETH is permanently locked.
7. Repeat on subsequent calls; balance accumulates monotonically with no recovery path.
