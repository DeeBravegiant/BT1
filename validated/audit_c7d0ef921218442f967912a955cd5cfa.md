### Title
`max_l2_gas_amount` Admission Limit Not Enforced for Declare Transactions in Gateway Stateless Validator - (File: `crates/apollo_gateway/src/stateless_transaction_validator.rs`)

### Summary
The `StatelessTransactionValidator` enforces a per-transaction `max_l2_gas_amount` cap (1,210,000,000) for Invoke and DeployAccount transactions, but explicitly skips this check for Declare transactions. This is a direct analog of the external bug: one component implies a limit, but a code path for a different transaction type bypasses it entirely.

### Finding Description
In `validate_resource_bounds()`, the `max_l2_gas_amount` check is guarded by a type branch that silently no-ops for Declare:

```rust
// TODO(Arni): Consider adding a validation for max_l2_gas_amount for declare.
if let RpcTransaction::Declare(_) = tx {
    // ← no check at all
} else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh { ... });
}
``` [1](#0-0) 

The default `max_l2_gas_amount` is 1,210,000,000: [2](#0-1) 

For Invoke and DeployAccount, any `l2_gas.max_amount` above this value is rejected at the gateway: [3](#0-2) 

But the test `valid_l2_gas_amount_on_declare` explicitly confirms that a Declare transaction with `max_amount = 200` (above a configured limit of 100) passes validation: [4](#0-3) 

This means a Declare transaction with `l2_gas.max_amount` up to `u64::MAX` (18,446,744,073,709,551,615) passes the stateless validator unconditionally.

### Impact Explanation
The `l2_gas.max_amount` field directly controls the per-transaction execution step limit in `EntryPointExecutionContext::max_steps()`: [5](#0-4) 

For a Declare transaction with `max_amount = u64::MAX`, the computed `tx_upper_bound` is `u64::MAX / l2_gas_per_step`, which is astronomically large. The actual step limit is then `min(tx_upper_bound, invoke_tx_max_n_steps)`. While `invoke_tx_max_n_steps` (a `u32`) caps the absolute worst case, a Declare transaction with `max_amount` between `max_l2_gas_amount` (1.21 B) and `invoke_tx_max_n_steps * l2_gas_per_step` can execute more steps than any Invoke or DeployAccount transaction is permitted to claim at admission. The gateway's admission invariant — that no admitted transaction may claim more than `max_l2_gas_amount` L2 gas — is broken for the Declare type.

Additionally, the fee pre-validation check (`minimal_gas_amount ≤ max_amount`) trivially passes for any large `max_amount`, so the stateful balance check becomes the only backstop. If a user has sufficient balance to cover `max_amount × min_gas_price`, the transaction is admitted to the mempool and sequenced.

The bouncer enforces block-level `sierra_gas` (5,000,000,000) but does not enforce a per-transaction `max_amount` ceiling at admission: [6](#0-5) 

### Likelihood Explanation
Any unprivileged user can craft a Declare V3 transaction with an arbitrarily large `l2_gas.max_amount`. The gateway's stateless validator is the first and only admission gate that is supposed to enforce this limit. No privileged access is required. The TODO comment in the source confirms the gap is known but unresolved.

### Recommendation
Remove the Declare-type exception in `validate_resource_bounds()` and apply the `max_l2_gas_amount` check uniformly to all transaction types, including Declare:

```rust
// Apply to all types, not just non-Declare.
if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh {
        gas_amount: resource_bounds.l2_gas.max_amount,
        max_gas_amount: self.config.max_l2_gas_amount,
    });
}
``` [7](#0-6) 

### Proof of Concept
1. Construct a valid Declare V3 transaction with `l2_gas.max_amount = 5_000_000_000` (above the 1,210,000,000 gateway limit but within the block bouncer's `sierra_gas` cap) and `l2_gas.max_price_per_unit = min_gas_price`.
2. Submit it to the gateway's `add_transaction` endpoint.
3. The stateless validator accepts it (the Declare branch is a no-op for the `max_l2_gas_amount` check).
4. The stateful validator checks balance: `5_000_000_000 × 8_000_000_000 = 4 × 10^19` STRK units. A funded account passes.
5. The transaction enters the mempool and is sequenced. During execution, `max_steps()` computes `tx_upper_bound = 5_000_000_000 / l2_gas_per_step`, which exceeds what any Invoke/DeployAccount transaction is permitted to claim at admission.
6. An equivalent Invoke transaction with the same `max_amount` is rejected at step 3 with `MaxGasAmountTooHigh`. [8](#0-7) [9](#0-8)

    fn validate_resource_bounds(
        &self,
        tx: &RpcTransaction,
    ) -> StatelessTransactionValidatorResult<()> {
        if !self.config.validate_resource_bounds {
            return Ok(());
        }

        let resource_bounds = *tx.resource_bounds();
        // The resource bounds should be positive even without the tip.
        if ValidResourceBounds::AllResources(resource_bounds).max_possible_fee(Tip::ZERO) == Fee(0)
        {
            return Err(StatelessTransactionValidatorError::ZeroResourceBounds { resource_bounds });
        }

        if resource_bounds.l2_gas.max_price_per_unit.0 < self.config.min_gas_price {
            return Err(StatelessTransactionValidatorError::MaxGasPriceTooLow {
                gas_price: resource_bounds.l2_gas.max_price_per_unit,
                min_gas_price: self.config.min_gas_price,
            });
        }

        // TODO(Arni): Consider adding a validation for max_l2_gas_amount for declare.
        if let RpcTransaction::Declare(_) = tx {
        } else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
            return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh {
                gas_amount: resource_bounds.l2_gas.max_amount,
                max_gas_amount: self.config.max_l2_gas_amount,
            });
        }

        Ok(())
    }

    pub fn try_update<S: StateReader>(
        &mut self,
        state_reader: &S,
        tx_state_changes_keys: &StateChangesKeys,
        tx_execution_summary: &ExecutionSummary,
        tx_builtin_counters: &CairoPrimitiveCounterMap,
        tx_resources: &TransactionResources,
        versioned_constants: &VersionedConstants,
        receipt_l2_gas: GasAmount,
    ) -> TransactionExecutorResult<()> {
        // The countings here should be linear in the transactional state changes and execution info
        // rather than the cumulative state attributes.
        let marginal_state_changes_keys =
            tx_state_changes_keys.difference(&self.state_changes_keys);
        let marginal_executed_class_hashes = tx_execution_summary
            .executed_class_hashes
            .difference(&self.get_executed_class_hashes())
            .cloned()
            .collect();
        let n_marginal_visited_storage_entries = tx_execution_summary
            .visited_storage_entries
            .difference(&self.visited_storage_entries)
            .count();
        let tx_weights = get_tx_weights(
            state_reader,
            &marginal_executed_class_hashes,
            n_marginal_visited_storage_entries,
            tx_resources,
            &marginal_state_changes_keys,
            versioned_constants,
            tx_builtin_counters,
            &self.bouncer_config,
            receipt_l2_gas,
        )?;

        let tx_bouncer_weights = tx_weights.bouncer_weights;

        // Check if the transaction can fit the current block available capacity.
        let err_msg = format!(
            "Addition overflow. Transaction weights: {tx_bouncer_weights:?}, block weights: {:?}.",
            self.get_bouncer_weights()
        );
        let next_accumulated_weights =
            self.get_bouncer_weights().checked_add(tx_bouncer_weights).expect(&err_msg);
        if !self.bouncer_config.has_room(next_accumulated_weights) {
            let exceeded_weights =
                self.bouncer_config.get_exceeded_weights(next_accumulated_weights);
            log::debug!(
                "Transaction cannot be added to the current block, block capacity reached; \
                 transaction weights: {:?}, block weights: {:?}. Block max capacity reached on \
                 fields: {}",
                tx_weights.bouncer_weights,
                self.get_bouncer_weights(),
                exceeded_weights
            );
            record_exceeded_bouncer_resources(&exceeded_weights);
            Err(TransactionExecutorError::BlockFull)?
        }

        self.update(tx_weights, tx_execution_summary, &marginal_state_changes_keys);

        Ok(())
