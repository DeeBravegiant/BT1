### Title
Untrusted Peer Can Trigger `perform_atomic_rollback(0)` via Fabricated Short Chain, Wiping All Wallet State — (`chia/wallet/wallet_node.py`)

### Summary

An untrusted peer can force a complete wallet state rollback to height 0 by serving a fabricated chain of `HeaderBlock`s with height below `WEIGHT_PROOF_RECENT_BLOCKS`. Two guards that would normally prevent this — the weight proof requirement and the `LONG_SYNC_THRESHOLD` cap — are both bypassed by this height condition. The rollback is committed to the database before `add_block` is called, so even when the fabricated blocks subsequently fail VDF validation, the wallet state erasure persists.

---

### Finding Description

**Routing bypass — no weight proof required**

In `new_peak_from_untrusted`: [1](#0-0) 

When `new_peak_hb.height < self.constants.WEIGHT_PROOF_RECENT_BLOCKS`, the code goes directly to `sync_from_untrusted_close_to_peak` without ever calling `long_sync_from_untrusted` or `fetch_and_update_weight_proof`. No weight proof is validated.

**LONG_SYNC_THRESHOLD cap also bypassed**

Inside `wallet_short_sync_backtrack`, the cap that would redirect to long sync requires all three conditions: [2](#0-1) 

The third condition (`header_block.height >= self.constants.WEIGHT_PROOF_RECENT_BLOCKS`) is false when the attacker's peak height is below `WEIGHT_PROOF_RECENT_BLOCKS`. The cap never fires, so the backtrack loop runs all the way to height 0 without limit.

**Chain continuity check is satisfiable by the attacker** [3](#0-2) 

`header_hash` is a deterministic hash of block content. The attacker controls the peer and can pre-compute a chain of fabricated `HeaderBlock`s where each block's `header_hash` equals the `prev_header_hash` of the next block. This check passes trivially.

**Rollback committed before `add_block` is called** [4](#0-3) 

When the backtrack reaches height 0 and `peak_hb is not None` (wallet has existing state), `should_skip_rollback = False` and `perform_atomic_rollback(0)` is called. Inside `perform_atomic_rollback`, the rollback is wrapped in `async with self.wallet_state_manager.db_wrapper.writer()`, which commits the transaction to the database before returning: [5](#0-4) 

Only after this committed rollback does the code attempt `add_block`: [6](#0-5) 

**`WalletBlockchain.add_block` validates VDF proofs — but too late** [7](#0-6) 

The fabricated blocks fail `validate_finished_header_block` (invalid VDF proofs, invalid PoSpace), returning `INVALID_BLOCK`. `wallet_short_sync_backtrack` raises `ValueError`. The exception propagates up and is caught in `_process_new_subscriptions`, which closes the peer. But the rollback is already durably committed.

**Timestamp check is satisfiable** [8](#0-7) 

`get_timestamp_for_height_from_peer` fetches a block from the peer and reads `foliage_transaction_block.timestamp`. It does not validate VDF proofs or foliage signatures. The attacker can serve a fabricated `HeaderBlock` with any recent timestamp to pass this check.

**The regression test confirms the rollback is intentional when peak exists** [9](#0-8) 

This test explicitly documents that `perform_atomic_rollback(0)` is called when `peak_hb is not None` and the chain reaches genesis — the exact condition the attacker triggers.

---

### Impact Explanation

`perform_atomic_rollback(0)` calls `reorg_rollback(0)`, which deletes all coin records, transaction history, and wallet sync state back to height 0. The wallet's `_finished_sync_up_to` is set to 0. All sub-wallet state (DL, NFT, DID, pool) is erased. The wallet must perform a full resync. During the attack window, the user cannot see coins or create spends. This is a durable, database-committed state corruption triggered by an unprivileged peer with no keys or admin access required.

---

### Likelihood Explanation

Any wallet not connected to a trusted local full node (the common case for most users) accepts untrusted peer connections. The attacker only needs to:
1. Connect as an untrusted peer
2. Send a `NewPeakWallet` with `height < WEIGHT_PROOF_RECENT_BLOCKS` and `weight > current_peak.weight`
3. Serve a pre-computed chain of fabricated `HeaderBlock`s with consistent `prev_header_hash` links and a recent timestamp

No cryptographic secrets, no mining, no special privileges are required. The fabricated chain construction is straightforward since `header_hash` is a deterministic hash of attacker-controlled content.

---

### Recommendation

The rollback in `wallet_short_sync_backtrack` must not be committed before the incoming blocks are validated. Options:

1. **Validate blocks before rolling back**: Run `validate_finished_header_block` on all blocks in the chain before calling `perform_atomic_rollback`. Only roll back if all blocks pass validation.
2. **Require weight proof for any rollback from an untrusted peer**: Before calling `wallet_short_sync_backtrack` for an untrusted peer with existing state, require a validated weight proof showing the new chain is heavier.
3. **Wrap rollback and add_block in a single atomic transaction**: If `add_block` fails, roll back the database transaction that included the `reorg_rollback`.

### Proof of Concept

```python
# Attacker-controlled peer serving fabricated chain
# Pre-compute chain: block_0 (height=0), block_1 (height=1), ..., block_N (height=N)
# where N < WEIGHT_PROOF_RECENT_BLOCKS (e.g., N = 300 on mainnet where WEIGHT_PROOF_RECENT_BLOCKS=1000)
# Each block_i.prev_header_hash == block_{i-1}.header_hash (computed from content)
# block_N.foliage_transaction_block.timestamp = int(time.time())  # recent timestamp
# block_N.weight > wallet_current_peak.weight

# 1. Connect untrusted peer to wallet
# 2. Send NewPeakWallet(height=N, weight=LARGE, header_hash=block_N.header_hash)
# 3. When wallet requests block_N: serve block_N (passes hash/weight/height check)
# 4. When wallet requests timestamp: serve block_N (recent timestamp, passes is_timestamp_in_sync)
# 5. wallet_short_sync_backtrack begins backtracking
# 6. For each request for block at height h: serve block_h (header_hash matches prev_header_hash check)
# 7. Loop reaches height 0, peak_hb is not None → should_skip_rollback = False
# 8. perform_atomic_rollback(0) committed to DB ← wallet state wiped
# 9. add_block(block_0) → INVALID_BLOCK (bad VDF) → ValueError raised
# 10. Exception caught in _process_new_subscriptions, peer closed
# 11. Wallet state is durably erased; wallet must full-resync

# Assert: wallet._finished_sync_up_to == 0, all coin records gone
```

    async def perform_atomic_rollback(self, fork_height: int, cache: PeerRequestCache | None = None) -> None:
        self.log.info(f"perform_atomic_rollback to {fork_height}")
        # this is to start a write transaction
        async with self.wallet_state_manager.db_wrapper.writer():
            try:
                removed_wallet_ids = await self.wallet_state_manager.reorg_rollback(fork_height)
                await self.wallet_state_manager.blockchain.set_finished_sync_up_to(fork_height, in_rollback=True)
                if cache is None:
                    self.rollback_request_caches(fork_height)
                else:
                    cache.clear_after_height(fork_height)
            except Exception as e:
                tb = traceback.format_exc()
                self.log.error(f"Exception while perform_atomic_rollback: {e} {tb}")
                raise
            else:
                await self.wallet_state_manager.blockchain.clean_block_records()

                for wallet_id in removed_wallet_ids:
                    self.wallet_state_manager.wallets.pop(wallet_id)
