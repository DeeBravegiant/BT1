Audit Report

## Title
Unbounded Gas Growth in `updateRSETHPrice()` Due to O(Assets × NDCs × Withdrawals) `getQueuedWithdrawals` Call Multiplication — (`contracts/LRTOracle.sol`, `contracts/LRTDepositPool.sol`, `contracts/NodeDelegator.sol`)

## Summary

`updateRSETHPrice()` (public, no role guard) calls `_getTotalEthInProtocol()`, which iterates over every supported asset and per asset calls `getAssetUnstaking()` on every NDC. `getAssetUnstaking()` invokes `getQueuedWithdrawals(ndc)` — loading the full EigenLayer withdrawal queue for that NDC — and iterates over every withdrawal and strategy. Because `_addNewSupportedAsset()` has no upper bound on `supportedAssetList.length`, each new asset multiplies the number of `getQueuedWithdrawals` calls by the NDC count, creating O(A × N × W × S) gas growth that can exceed the block gas limit and permanently disable price updates.

## Finding Description

**Verified call chain:**

1. `LRTOracle.updateRSETHPrice()` (L87–89) is `public whenNotPaused` with no role restriction. It calls `_updateRsETHPrice()` → `_getTotalEthInProtocol()` (L231).

2. `_getTotalEthInProtocol()` (L331–349) iterates over `lrtConfig.getSupportedAssetList()` — an unbounded array — and for each asset calls `ILRTDepositPool.getTotalAssetDeposits(asset)`.

3. `getTotalAssetDeposits()` (L385–397) calls `getAssetDistributionData(asset)`, which at L446–456 loops over every NDC in `nodeDelegatorQueue` and calls `INodeDelegator(ndc).getAssetUnstaking(asset)` per NDC.

4. `NodeDelegator.getAssetUnstaking()` (L405–427) calls `_getDelegationManager().getQueuedWithdrawals(address(this))`, loading **all** queued withdrawal structs for that NDC from EigenLayer storage, then iterates over every withdrawal and every strategy within it.

5. `LRTConfig._addNewSupportedAsset()` (L106–118) performs no length check on `supportedAssetList` before pushing.

**Result:** `getQueuedWithdrawals` is called **A × N** times per `updateRSETHPrice()` invocation. Each call loads the full withdrawal queue for that NDC. Total withdrawal struct iterations = A × Σ(W_i per NDC).

**Existing mitigations and their limits:**

| Parameter | Cap | Covers |
|---|---|---|
| `maxNodeDelegatorLimit` | 10 default (admin-adjustable) | NDC count |
| `maxUncompletedWithdrawalCount` | ≤ 80 global | Total withdrawal count |
| `supportedAssetList.length` | **None** | Asset count multiplier |

The protocol's own comment at `LRTUnstakingVault.sol` L151–153 acknowledges the gas concern ("120 is the max number of uncompleted withdrawals that allows us to still perform update rsETH price") but only accounts for the withdrawal count dimension. It does not account for the asset-count multiplier: each new supported asset multiplies the number of `getQueuedWithdrawals` calls by N.

**Conservative gas estimate (warm storage):**
- Each `getQueuedWithdrawals` call with 8 withdrawals × 2 strategies ≈ 30,000–50,000 gas
- 20 assets × 10 NDCs = 200 calls → ~8M gas (within limit)
- 40 assets × 10 NDCs = 400 calls → ~16M gas (approaching limit)
- 50 assets × 10 NDCs = 500 calls → ~20M+ gas (exceeds practical limit)
- If `maxNodeDelegatorLimit` is raised to 20 and 40 assets are added: 800 calls → ~32M gas → **exceeds 30M block gas limit**

Both `updateRSETHPrice()` (public) and `updateRSETHPriceAsManager()` (manager-only) call the same `_updateRsETHPrice()` → `_getTotalEthInProtocol()` path, so neither escapes the gas ceiling once it is hit.

## Impact Explanation

If `updateRSETHPrice()` exceeds the block gas limit, the rsETH price can never be updated. This permanently disables deposits (`getRsETHAmountToMint` uses `rsETHPrice()` at L520) and breaks the core accounting invariant of the protocol. The impact matches **Medium — Unbounded gas consumption**, with escalation risk to temporary-to-permanent freezing of the price update mechanism.

## Likelihood Explanation

Likelihood is low-to-medium. The precondition is legitimate governance growth: TIME_LOCK_ROLE adding assets over time (a normal protocol operation, not an attacker exploit). The existing caps on NDCs (10) and withdrawals (80) provide meaningful mitigation at current asset counts (~2–5 assets). However, there is no hard cap on assets, and gas grows linearly with each addition. A protocol that grows to 30–50 supported assets with a raised NDC limit would be at risk without any code change. The protocol's own comment at `LRTUnstakingVault.sol` L151–153 confirms the developers are aware of the gas sensitivity but have not addressed the asset-count dimension.

## Recommendation

1. **Add a hard cap on `supportedAssetList.length`** in `_addNewSupportedAsset()`: `require(supportedAssetList.length < MAX_ASSETS, "AssetLimitReached")`.
2. **Cache `getQueuedWithdrawals` results per NDC** across asset iterations. Restructure `_getTotalEthInProtocol()` or `getAssetDistributionData()` to call `getQueuedWithdrawals` once per NDC and filter by asset in memory, reducing calls from A×N to N.
3. **Document the combined cap invariant**: `assets × NDCs × avg_withdrawals_per_NDC × avg_strategies_per_withdrawal < gas_budget / gas_per_iteration`.

    function getAssetUnstaking(address asset) external view returns (uint256 amount) {
        (IDelegationManager.Withdrawal[] memory queuedWithdrawals, uint256[][] memory withdrawalShares) =
            _getDelegationManager().getQueuedWithdrawals(address(this));

        for (uint256 withdrawalIndex = 0; withdrawalIndex < queuedWithdrawals.length; withdrawalIndex++) {
            IDelegationManager.Withdrawal memory withdrawal = queuedWithdrawals[withdrawalIndex];

            for (uint256 strategyIndex = 0; strategyIndex < withdrawal.strategies.length; strategyIndex++) {
                IStrategy strategy = withdrawal.strategies[strategyIndex];

                address strategyAsset = address(strategy) == address(lrtConfig.beaconChainETHStrategy())
                    ? LRTConstants.ETH_TOKEN
                    : address(strategy.underlyingToken());

                if (strategyAsset != asset) continue;

                uint256 sharesToUnstake = withdrawalShares[withdrawalIndex][strategyIndex];
                amount += strategyAsset == LRTConstants.ETH_TOKEN
                    ? sharesToUnstake
                    : strategy.sharesToUnderlyingView(sharesToUnstake);
            }
        }
    }
