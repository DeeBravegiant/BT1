Audit Report

## Title
`_scaleWeightsToShares` Arithmetic Overflow DoS's `addLiquidityWeighted` When `maxAmountToken0/1 = type(uint256).max` and Pool Requires Small Token Amount — (`metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol`)

## Summary

`MetricOmmPoolLiquidityAdder._scaleWeightsToShares` computes `Math.mulDiv(max0, WAD, need0)` where `WAD = 1e18`. When a caller passes `maxAmountToken0 = type(uint256).max` (the idiomatic "no limit" sentinel) and the probe returns any `need0 < 1e18` (e.g., `need0 = 1` for a 1-share deposit into a fresh bin), the 512-bit intermediate product `type(uint256).max × 1e18` produces a quotient that exceeds `uint256`, causing OpenZeppelin's `Math.mulDiv` to revert unconditionally. This permanently DoS's `addLiquidityWeighted` for the most common usage pattern.

## Finding Description

The two `addLiquidityWeighted` overloads implement a probe-then-scale pattern:

1. A dry-run `addLiquidity` call is made with `KIND_PROBE` callback data.
2. The callback immediately reverts with `LiquidityProbe(amount0Delta, amount1Delta)`, carrying the exact token amounts the pool would require.
3. `_scaleWeightsToShares` uses those amounts (`need0`, `need1`) to scale the weight vector to fit within the caller's budget.

The probe and callback flow is confirmed at: [1](#0-0) [2](#0-1) 

The vulnerable scale factor computation is: [3](#0-2) 

The guard `need0 == 0 ? type(uint256).max` correctly handles the "pool needs no token0" case, but there is **no guard** for the case where `need0 > 0` and `max0 × WAD / need0 > type(uint256).max`.

OpenZeppelin's `Math.mulDiv` uses 512-bit intermediate arithmetic but reverts when the final quotient exceeds `uint256` (via `require(denominator > prod1)`). The overflow condition is: `max0 × WAD > type(uint256).max × need0`, i.e., `max0 / need0 > type(uint256).max / WAD`. For `max0 = type(uint256).max`, this simplifies to **`need0 < WAD = 1e18`**.

**When does `need0 < 1e18` occur?**

`need0` is `amount0Added` from `LiquidityLib.addLiquidity`. For a fresh bin above the current price with `initialScaledToken0PerShareE18 = 1e18` and `sharesToAdd = 1`: [4](#0-3) 

`amount0Scaled = ceil(1e18 × 1 / 1e18) = 1`. After `_deltasScaledToExternal` with `TOKEN_0_SCALE_MULTIPLIER = 1` (18-decimal token): [5](#0-4) 

`need0 = ceil(1 / 1) = 1 ≪ WAD = 1e18` → overflow and revert.

This is not a contrived edge case. Any liquidity addition requiring fewer than `1e18` native units of token0 (i.e., less than 1 full token for 18-decimal assets, or less than ~1 million USDC for 6-decimal assets) triggers the revert when `max0 = type(uint256).max`.

## Impact Explanation

`addLiquidityWeighted` is the primary periphery entry point for budget-constrained liquidity addition. The function is completely unusable when `type(uint256).max` is passed as the max cap and the pool requires a small token amount — which is the normal case for any small-share deposit. This constitutes broken core liquidity flow: users cannot add liquidity through the weighted path for the most common usage pattern. The test suite itself uses `type(uint256).max` as the "no limit" cap for `addLiquidityExactShares`, confirming this is the expected idiom: [6](#0-5) [7](#0-6) 

## Likelihood Explanation

- `type(uint256).max` as a "no limit" cap is standard DeFi practice and is used throughout the project's own test suite.
- `need0 < 1e18` is the **common case** for any deposit of less than 1 token0 (18-decimal) or less than ~1 million USDC (6-decimal). Small liquidity additions are routine.
- No special pool state, privileged role, or malicious setup is required — any unprivileged LP caller triggers this on any legitimate pool with a small-share deposit.

## Recommendation

Treat an overflowing ratio as "unconstrained" (i.e., return `type(uint256).max`) rather than reverting. Apply a saturating check before the `mulDiv`:

```diff
- uint256 scaleWad0 = need0 == 0 ? type(uint256).max : Math.mulDiv(max0, WAD, need0);
- uint256 scaleWad1 = need1 == 0 ? type(uint256).max : Math.mulDiv(max1, WAD, need1);
+ uint256 scaleWad0 = (need0 == 0 || max0 >= type(uint256).max / WAD)
+     ? type(uint256).max
+     : Math.mulDiv(max0, WAD, need0);
+ uint256 scaleWad1 = (need1 == 0 || max1 >= type(uint256).max / WAD)
+     ? type(uint256).max
+     : Math.mulDiv(max1, WAD, need1);
```

The condition `max0 >= type(uint256).max / WAD` (≈ `1.15 × 10^59`) is a safe proxy for "the ratio would overflow"; at that scale the caller is effectively unconstrained and returning `type(uint256).max` is semantically correct.

  {

    uint256 scaleWad1 = need1 == 0 ? type(uint256).max : Math.mulDiv(max1, WAD, need1);
