Audit Report

## Title
`MetricOmmPoolLiquidityAdder` Accepts Unvalidated Pool Address, Allowing Malicious Pool to Drain Caller Tokens via Callback — (`metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol`)

## Summary

`MetricOmmPoolLiquidityAdder` has no factory reference and performs no pool validation in any of its four public entry points. A malicious contract passed as `pool` satisfies every callback guard and can pull up to `maxAmount0`/`maxAmount1` of any token from the caller's wallet by re-entering `metricOmmModifyLiquidityCallback` with attacker-controlled token addresses returned from `getImmutables()`.

## Finding Description

The constructor accepts only `weth` — no factory is stored: [1](#0-0) 

The contract self-documents the gap: [2](#0-1) 

All four public entry points (`addLiquidityExactShares` ×2, `addLiquidityWeighted` ×2) pass the caller-supplied `pool` directly into `_addLiquidity` without any registry check: [3](#0-2) [4](#0-3) 

`_addLiquidity` stores the attacker-supplied address as the **expected** callback caller in transient storage, then calls into it: [5](#0-4) 

The malicious pool's `addLiquidity` immediately re-enters `metricOmmModifyLiquidityCallback` encoding `KIND_PAY`. Every guard in the callback passes:

- `expectedPool != address(0)` — set to the malicious pool
- `msg.sender == expectedPool` — malicious pool is the caller
- `amount0Delta <= max0 && amount1Delta <= max1` — attacker sets deltas ≤ caps [6](#0-5) 

After passing all guards, the callback calls `getImmutables()` on `msg.sender` (the malicious pool) to obtain `token0`/`token1`, then transfers the victim's tokens to the malicious pool: [7](#0-6) 

**Contrast with `MetricOmmSwapRouterBase`**, which stores an immutable `FACTORY` and calls `_requireFactoryPool(pool)` before setting any callback context, and re-validates `FACTORY.isPool(caller)` inside the callback: [8](#0-7) [9](#0-8) [10](#0-9) 

`MetricOmmPoolLiquidityAdder` has no equivalent protection.

## Impact Explanation

Any user who has approved `MetricOmmPoolLiquidityAdder` for token spending and is tricked into calling any of the four public entry points with a malicious pool address loses up to `maxAmount0` of token0 and `maxAmount1` of token1. If the user passes `type(uint256).max` for both caps (a common pattern), their entire approved balance is drained in a single transaction. This is a direct loss of user principal with no protocol-side precondition.

## Likelihood Explanation

The attack requires only that a victim has approved the adder and is induced to call it with a malicious pool address — achievable via a malicious frontend, phishing, or a compromised integration. Users familiar with `MetricOmmSimpleRouter`'s factory-validation model will reasonably expect the liquidity adder to provide the same guarantee. The `addLiquidityWeighted` path is additionally dangerous because the probe phase calls the malicious pool before the pay context is set, allowing attacker-controlled `need0`/`need1` values to manipulate share scaling: [11](#0-10) 

## Recommendation

Add an immutable `FACTORY` reference to `MetricOmmPoolLiquidityAdder` (mirroring `MetricOmmSwapRouterBase`) and call `_requireFactoryPool(pool)` at the top of every public entry point, before any external call to `pool`:

```solidity
IMetricOmmPoolFactory internal immutable FACTORY;

constructor(address weth, address factory) PeripheryPayments(weth) {
    FACTORY = IMetricOmmPoolFactory(factory);
}

function _requireFactoryPool(address pool) internal view {
    if (!FACTORY.isPool(pool)) revert InvalidPool(pool);
}
```

For `addLiquidityWeighted`, the check must also precede the probe call at line 106/139 to prevent attacker-controlled `need0`/`need1` from influencing share scaling.

## Proof of Concept

1. Deploy `MaliciousPool` with attacker-controlled `token0`, `token1`, and `stealAmount = victimBalance`.
2. Victim approves `MetricOmmPoolLiquidityAdder` for `token0`.
3. Victim (or attacker via phishing) calls `addLiquidityExactShares(address(malicious), victim, 0, deltas, victimBalance, 0, "")`.
4. `_addLiquidity` stores `malicious` as `expectedPool` in transient storage and calls `malicious.addLiquidity(...)`.
5. `MaliciousPool.addLiquidity` immediately calls `adder.metricOmmModifyLiquidityCallback(victimBalance, 0, abi.encode(uint8(1)))`.
6. All guards pass; callback calls `malicious.getImmutables()` → returns attacker's `token0`/`token1`; calls `pay(token0, victim, malicious, victimBalance)`.
7. `victimBalance` of `token0` is transferred from victim to `MaliciousPool`. Assert passes.
