### Title
Unauthenticated On-Chain R-CAT Spend Triggers Attacker-Controlled `convert_to_revocable`, Permanently Mutating Victim's CATWallet with Attacker's Revocation Key — (`chia/wallet/wallet_state_manager.py`, `chia/wallet/cat_wallet/r_cat_wallet.py`)

### Summary

An unprivileged attacker who owns any amount of a DEFAULT_CATS token can craft a valid on-chain R-CAT spend that causes a victim's `CATWallet` (with an empty lineage store) to be permanently converted to an `RCATWallet` embedding the attacker's `hidden_puzzle_hash`. All future coins sent to the victim's wallet will be spendable by the attacker via the revocation layer's hidden puzzle path.

### Finding Description

The vulnerability lives in `handle_cat` in `WalletStateManager` and `convert_to_revocable` in `RCATWallet`. The code path has no ownership or authorization check before mutating the victim's wallet type.

**Step 1 — Hint check (the only "ownership" gate):** [1](#0-0) 

`hinted_coin.hint` is attacker-controlled data embedded in the coin spend. The attacker sets it to any of the victim's known p2 puzzle hashes (which are public on-chain). This causes `derivation_record` to be non-`None`, passing the only gate that could reject the coin as "not ours."

**Step 2 — Puzzle hash mismatch triggers special-CAT branch:** [2](#0-1) 

The attacker's coin has puzzle hash `CAT(tail, revocation_layer(attacker_hidden_ph, victim_p2_ph))`, which does not match the plain `CAT(tail, victim_p2_ph)` constructed at line 1256. `match_revocation_layer` is called on the attacker-controlled `coin_spend.puzzle_reveal`, returning the attacker's `hidden_puzzle_hash`. [3](#0-2) 

**Step 3 — Existing CATWallet found and `convert_to_revocable` called with attacker hash:** [4](#0-3) 

No signature check, no ownership proof, no verification that the coin is actually addressed to the victim. The attacker's `revocation_layer_match[0]` (their `hidden_puzzle_hash`) is passed directly.

**Step 4 — `convert_to_revocable`: the only guard is `lineage_store.is_empty()`:** [5](#0-4) 

If the victim's lineage store is empty (no confirmed CAT coins yet), the function:
- Overwrites the wallet type to `RCAT` with the attacker's `hidden_puzzle_hash` baked in
- Calls `puzzle_store.delete_wallet` — erasing all derivation records for that wallet
- Calls `create_more_puzzle_hashes` — regenerating them with `revocation_layer(attacker_hidden_ph, victim_p2_ph)` as the inner puzzle for every future address [6](#0-5) 

Every future puzzle hash generated by the wallet now embeds the attacker's `hidden_puzzle_hash` in the revocation layer.

### Impact Explanation

After the conversion, every coin sent to the victim's RCAT wallet has puzzle `CAT(tail, revocation_layer(attacker_hidden_ph, victim_p2_ph))`. The revocation layer allows the holder of the hidden puzzle preimage to spend the coin unilaterally, bypassing the victim's key entirely. This is unauthorized coin control over all future CAT receipts — a High-severity impact under the allowed scope.

### Likelihood Explanation

- Victim precondition: has a `CATWallet` for a `DEFAULT_CATS` asset_id with an empty lineage store. This is the default state for any wallet that was auto-created for a DEFAULT_CAT but has not yet received any coins of that type.
- Attacker cost: must own any nonzero amount of the target CAT (purchasable on any DEX).
- Victim's puzzle hashes are public (subscribed to by the full node, visible on-chain).
- No cryptographic barrier; the attack is a single crafted spend bundle.

### Recommendation

In `handle_cat`, before calling `convert_to_revocable`, verify that the coin's actual puzzle hash matches what the victim's wallet would generate for the detected wallet type and the detected `hidden_puzzle_hash`. Specifically, reconstruct `CAT(tail, revocation_layer(revocation_layer_match[0], our_inner_puzzle))` and assert it equals `coin_state.coin.puzzle_hash`. This ensures the coin is genuinely addressed to the victim under the claimed revocation key, not merely hinted at them.

Additionally, the `lineage_store.is_empty()` guard in `convert_to_revocable` is insufficient as a sole defense — it only protects wallets that have already received coins.

    @classmethod
    async def convert_to_revocable(
        cls,
        cat_wallet: CATWallet,
        hidden_puzzle_hash: bytes32,
    ) -> bool:
        if not await cat_wallet.lineage_store.is_empty():
            cat_wallet.log.error("Received a revocable CAT to a CAT wallet that already has CATs")
            return False
        replace_self = cls()
        replace_self.standard_wallet = cat_wallet.standard_wallet
        replace_self.log = logging.getLogger(cat_wallet.get_name())
        replace_self.log.info(f"Converting CAT wallet {cat_wallet.id()} to R-CAT wallet")
        replace_self.wallet_state_manager = cat_wallet.wallet_state_manager
        replace_self.lineage_store = cat_wallet.lineage_store
        replace_self.cat_info = cls.wallet_info_type(
            cat_wallet.cat_info.limitations_program_hash, None, hidden_puzzle_hash
        )
        await cat_wallet.wallet_state_manager.user_store.update_wallet(
            WalletInfo(
                cat_wallet.id(), cat_wallet.get_name(), uint8(cls.wallet_type.value), bytes(replace_self.cat_info).hex()
            )
        )
        updated_wallet_info = await cat_wallet.wallet_state_manager.user_store.get_wallet_by_id(cat_wallet.id())
        assert updated_wallet_info is not None
        replace_self.wallet_info = updated_wallet_info

        cat_wallet.wallet_state_manager.wallets[cat_wallet.id()] = replace_self
        await cat_wallet.wallet_state_manager.puzzle_store.delete_wallet(cat_wallet.id())
        result = await cat_wallet.wallet_state_manager.create_more_puzzle_hashes()
        await result.commit(cat_wallet.wallet_state_manager)
        return True
