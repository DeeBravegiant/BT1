### Title
Caller-Controlled `EvmFinality::Latest` Accepted Without Minimum Finality Enforcement Enables Reorg-Based Forged Foreign-Chain Attestation — (File: `crates/foreign-chain-inspector/src/evm/inspector.rs`, `crates/contract/src/lib.rs`)

---

### Summary

The `verify_foreign_transaction` contract method accepts `EvmFinality::Latest` in an `EvmRpcRequest` without any minimum finality enforcement. An unprivileged caller can request an MPC attestation for an EVM transaction (Polygon, BNB, Base, Arbitrum, HyperEVM, Abstract) using the `Latest` tag, which provides zero reorg protection. The MPC network will sign a payload attesting to data from a block that may be reorganized away, enabling a caller to present a valid MPC signature for a transaction that no longer exists on the canonical chain.

---

### Finding Description

`EvmFinality` is a caller-supplied enum with three variants: `Latest`, `Safe`, and `Finalized`. [1](#0-0) 

The `verify_foreign_transaction` entry point in the contract performs only four precondition checks: domain existence, gas, deposit, and chain availability. It performs **no validation of the `finality` field** inside the `EvmRpcRequest`. [2](#0-1) 

The `EvmInspector::verify_finality_level` function maps `EthereumFinality::Latest` directly to `FinalityTag::Latest` and queries `eth_getBlockByNumber("latest")`. The only check performed is that the latest block number is ≥ the transaction's block number — a condition satisfied by any included transaction. [3](#0-2) 

The subsequent `verify_block_is_canonical` call is a **point-in-time** check: it verifies the block hash at the transaction's height matches the receipt at the moment of the RPC call. It provides no guarantee of future canonicality. [4](#0-3) 

If the check passes and the MPC network signs the payload, the resulting `ForeignTxSignPayloadV1 { request, values }` is a permanent, valid MPC attestation. A subsequent reorg that removes the attested block does not invalidate the already-issued signature. [5](#0-4) 

The same absence of minimum enforcement applies to Bitcoin's `BlockConfirmations`: the field is a plain `u64` wrapper with no lower-bound validation anywhere in the contract or inspector stack. [6](#0-5) 

---

### Impact Explanation

**High — Forged foreign-chain verification / light-client-style verification bypass causing invalid bridge execution.**

The MPC network's `verify_foreign_transaction` flow is the trust anchor for the Omnibridge inbound path (foreign chain → NEAR). When the MPC cluster signs a `ForeignTxSignPayloadV1`, downstream bridge contracts treat that signature as a definitive attestation that the foreign transaction finalized. If the attested block is later reorganized away, the signature remains cryptographically valid and can be submitted to a bridge contract to release NEAR-side funds for a deposit that never became canonical on the foreign chain. This is a forged foreign-chain verification enabling double-spend / invalid bridge execution.

---

### Likelihood Explanation

**High.** The attack path requires only an unprivileged `verify_foreign_transaction` call with `finality: EvmFinality::Latest`. No special role, key material, or collusion is needed. Polygon — one of the explicitly supported and tested chains — experiences multiple reorgs per day with depth exceeding 3 blocks (as documented in the external report and on Polygonscan). The `Latest` tag provides zero confirmation depth, making any included transaction immediately eligible for MPC attestation before any reorg protection window has elapsed.

---

### Recommendation

1. **Reject `EvmFinality::Latest` at the contract level.** Add a validation step inside `verify_foreign_transaction` (or a dedicated `validate_evm_request` helper) that panics if `finality == EvmFinality::Latest` for any production EVM chain.

2. **Enforce per-chain minimum finality in the contract's `ChainEntry` configuration.** Store a `minimum_finality: EvmFinality` field alongside each whitelisted chain's `ProviderConfig`. Reject requests whose `finality` is weaker than the on-chain minimum. This allows governance to set chain-appropriate floors (e.g., `Finalized` for Polygon, `Safe` for Ethereum mainnet).

3. **Enforce a minimum `BlockConfirmations` for Bitcoin.** Add a lower-bound check (e.g., ≥ 6) either in the contract or in the inspector, analogous to the EVM finality floor.

4. **Consider rejecting `EvmFinality::Safe` for Polygon.** Polygon's `safe` tag corresponds to only ~32 blocks, which is still within the range of historically observed reorgs. `Finalized` (checkpoint-based) is the appropriate minimum for Polygon.

---

### Proof of Concept

**Attacker-controlled entry path:**

```rust
// Any unprivileged account can call this with Latest finality
let request = VerifyForeignTransactionRequestArgs {
    request: ForeignChainRpcRequest::Polygon(EvmRpcRequest {
        tx_id: EvmTxId(deposit_tx_hash),
        extractors: vec![EvmExtractor::BlockHash, EvmExtractor::Log { log_index: 0 }],
        finality: EvmFinality::Latest,   // <-- zero reorg protection
    }),
    domain_id: foreign_tx_domain_id,
    payload_version: ForeignTxPayloadVersion::V1,
};
contract.verify_foreign_transaction(request);
```

**Execution path through the vulnerable code:**

1. `verify_foreign_transaction` (contract) — no finality validation, enqueues the request. [7](#0-6) 

2. MPC node picks up the request, calls `EvmInspector::extract` with `EthereumFinality::Latest`. [8](#0-7) 

3. `verify_finality_level` queries `eth_getBlockByNumber("latest")` — passes for any included transaction. [3](#0-2) 

4. `verify_block_is_canonical` passes at the moment of the RPC call. [9](#0-8) 

5. MPC signs `ForeignTxSignPayloadV1 { request: Polygon(Latest, tx_id, ...), values: [BlockHash, Log] }`.

6. A Polygon reorg removes the attested block. The MPC signature remains valid.

7. Attacker submits the signature to the bridge contract on NEAR, claiming funds for a deposit that no longer exists on the canonical Polygon chain.

    async fn verify_block_is_canonical(
        &self,
        receipt_block_number: U64,
        receipt_block_hash: H256,
    ) -> Result<(), ForeignChainInspectionError> {
        let args = GetBlockByNumberArgs::new(
            BlockNumberOrTag::Number(receipt_block_number),
            ReturnFullTransactionObjects::from(false),
        );
        let canonical: GetBlockByNumberResponse = self
            .client
            .request(GET_BLOCK_BY_NUMBER_METHOD, &args)
            .await?;

        let hash_matches = canonical.hash == receipt_block_hash;
        let height_matches = canonical.number == receipt_block_number;
        if !hash_matches || !height_matches {
            return Err(ForeignChainInspectionError::NonCanonicalBlock {
                block_number: receipt_block_number.as_u64(),
                receipt_hash: receipt_block_hash.into(),
                canonical_hash: canonical.hash.into(),
            });
        }
        Ok(())
    }
