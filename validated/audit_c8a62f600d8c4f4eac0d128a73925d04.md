Audit Report

## Title
Missing Staleness Check in `CrossChainRateReceiver.getRate()` Enables Mispriced wrsETH Minting — (`contracts/cross-chain/CrossChainRateReceiver.sol`)

## Summary

`CrossChainRateReceiver` stores both `rate` and `lastUpdated` but `getRate()` returns the raw stored rate with no age validation. All three L2 deposit pools (`RSETHPoolV3`, `RSETHPoolV2`, `RSETHPoolNoWrapper`) consume this value directly to compute minted wrsETH amounts. When the LayerZero relay is delayed, the stored rate diverges from the true L1 rsETH price, causing every deposit to be priced at the wrong rate. Since rsETH is a yield-bearing token whose rate monotonically increases, the dominant real-world scenario is a stale rate below the true rate, allowing depositors to mint excess wrsETH at the expense of existing holders' accrued yield.

## Finding Description

`CrossChainRateReceiver.lzReceive()` is the sole write path for the oracle state:

```solidity
// contracts/cross-chain/CrossChainRateReceiver.sol L95-97
rate = _rate;
lastUpdated = block.timestamp;
```

The public read function exposes the raw stored value with no age validation:

```solidity
// contracts/cross-chain/CrossChainRateReceiver.sol L103-105
function getRate() external view returns (uint256) {
    return rate;
}
```

`lastUpdated` is stored but never read anywhere in the codebase — confirmed by a full grep across all contract files, which found `lastUpdated` only in `CrossChainRateReceiver.sol` and the provider contracts, never in any validation logic.

All three L2 pools delegate pricing through the same `IOracle` interface:

- `RSETHPoolV3.getRate()` → `IOracle(rsETHOracle).getRate()` (L235-237)
- `RSETHPoolV2.getRate()` → `IOracle(rsETHOracle).getRate()` (L201-203)
- `RSETHPoolNoWrapper.getRate()` → `IOracle(rsETHOracle).getRate()` (L220-222)

Each pool's minting formula divides by this rate:

```solidity
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
```

(`RSETHPoolV3` L307, `RSETHPoolV2` L233, `RSETHPoolNoWrapper` L285)

There is no guard anywhere in the deposit path that checks `block.timestamp - lastUpdated < maxStaleness`. The protocol is aware of the staleness pattern — `ChainlinkOracleForRSETHPoolCollateral.getRate()` explicitly checks `answeredInRound < roundID` and `timestamp == 0` — but this protection was not applied to `CrossChainRateReceiver`.

**Exploit path (Scenario A — stale rate < true rate):**
1. `lzReceive` was last called at time T with rate = 1.00e18.
2. Time passes; rsETH accrues yield; true rate rises to 1.05e18.
3. LayerZero relay is delayed (bridge congestion, fee spike, operator inattention).
4. Depositor calls `pool.deposit{value: 1 ether}("ref")`.
5. Pool calls `getRate()` → returns stale 1.00e18.
6. `rsETHAmount = 1e18 * 1e18 / 1e18 = 1e18 wrsETH` minted.
7. At the true rate, depositor should receive only `1e18 * 1e18 / 1.05e18 ≈ 0.952e18 wrsETH`.
8. The excess 0.048e18 wrsETH represents yield already accrued by existing holders — permanently diluted.

## Impact Explanation

**High: Theft of unclaimed yield.**

rsETH is a yield-bearing token whose exchange rate increases monotonically as staking rewards accrue. The dominant real-world scenario is therefore Scenario A: the stored L2 rate lags behind the true L1 rate. Every deposit during a staleness window mints excess wrsETH, diluting the yield that existing wrsETH holders have already earned but not yet claimed. The magnitude scales with both the staleness duration and the deposit volume. This is a direct, concrete transfer of value from existing holders to new depositors — not a hypothetical loss.

Scenario B (stale rate > true rate) is also reachable (e.g., if a rate correction occurs on L1) and maps to **Medium: Permanent freezing of unclaimed yield** for the depositor who receives fewer tokens than entitled.

## Likelihood Explanation

`updateRate()` on `CrossChainRateProvider` is permissionless — any caller willing to pay the LayerZero messaging fee can trigger it. However, it is not automated on-chain; it requires off-chain initiation. Bridge congestion, LZ fee spikes, or operator inattention are realistic operational conditions that can leave the L2 rate stale for hours or days. rsETH's rate increases continuously (approximately in line with ETH staking APR, ~3-4% annually), so even a 24-hour staleness window produces measurable mispricing. The condition is repeatable and requires no special attacker capability — any depositor benefits from and perpetuates the mispricing.

## Recommendation

Add a configurable `maxStaleness` parameter and revert in `getRate()` if the stored rate is too old:

```solidity
uint256 public maxStaleness; // e.g. 24 hours

function getRate() external view returns (uint256) {
    require(block.timestamp - lastUpdated <= maxStaleness, "Rate is stale");
    return rate;
}
```

This causes deposits to revert rather than execute at a stale price, giving operators time to refresh the rate before funds are mispriced. `maxStaleness` should be owner-configurable and initialized to a value consistent with the expected update frequency.

## Proof of Concept

```solidity
// Foundry fork test — Scenario A (stale rate < true rate → theft of yield)
function test_staleRateMintExcessWrsETH() external {
    // Simulate: lzReceive was called 2 days ago with rate = 1.00e18
    // True current rsETH price has since risen to 1.05e18 (yield accrual)
    vm.store(
        address(rateReceiver),
        bytes32(uint256(0)), // rate slot
        bytes32(uint256(1.00e18))
    );
    vm.store(
        address(rateReceiver),
        bytes32(uint256(1)), // lastUpdated slot
        bytes32(block.timestamp - 2 days)
    );

    uint256 depositAmount = 1 ether;
    vm.deal(user, depositAmount);

    uint256 existingHolderBalanceBefore = wrsETH.balanceOf(existingHolder);

    vm.prank(user);
    pool.deposit{value: depositAmount}("ref");

    uint256 wrsETHReceived = wrsETH.balanceOf(user);
    // At stale 1.00e18: user gets 1.000e18 wrsETH
    // At true 1.05e18: user should get ~0.952e18 wrsETH
    // Excess ~0.048e18 wrsETH dilutes existing holders' unclaimed yield
    assertGt(wrsETHReceived, 0.952e18, "User received more wrsETH than entitled at true rate");

    // Existing holder's proportional claim on underlying ETH is now reduced
    // because total wrsETH supply increased by more than the deposited ETH warranted
}
```
