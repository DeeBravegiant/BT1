### Title
Unvalidated `ProposalInit.builder` Allows Malicious Proposer to Redirect All Block Fees to Arbitrary Address - (`crates/apollo_consensus_orchestrator/src/utils.rs`)

### Summary

`ProposalInit.builder` is a proposer-supplied field that is directly used as `sequencer_address` in `BlockInfo` for block execution. The validator's `is_proposal_init_valid` function never checks this field against any expected value. A malicious proposer can set `builder` to any address, causing all transaction fees in the block to be transferred to an attacker-controlled address instead of the legitimate sequencer.

### Finding Description

`ProposalInit` carries two identity fields: `proposer` (the consensus identity, validated against the committee) and `builder` (the block-building address, never validated).

In `convert_to_sn_api_block_info`, the `builder` field is directly mapped to `sequencer_address`:

```rust
// crates/apollo_consensus_orchestrator/src/utils.rs:329-332
Ok(starknet_api::block::BlockInfo {
    ...
    sequencer_address: init.builder,   // ← proposer-supplied, never validated
    ...
})
``` [1](#0-0) 

This `BlockInfo` is passed to the batcher via `ValidateBlockInput` in `initiate_validation`:

```rust
// crates/apollo_consensus_orchestrator/src/validate_proposal.rs:455-466
let input = ValidateBlockInput {
    proposal_id,
    deadline: ...,
    retrospective_block_hash: ...,
    block_info: convert_to_sn_api_block_info(init)?,
};
batcher.validate_block(input.clone()).await...
``` [2](#0-1) 

The validator's `is_proposal_init_valid` checks height, `l1_da_mode`, `l2_gas_price_fri`, timestamp bounds, `starknet_version`, `version_constant_commitment`, L1 gas prices, and `fee_proposal`. The `builder` field is absent from `ProposalInitValidation` and is never checked:

```rust
// crates/apollo_consensus_orchestrator/src/validate_proposal.rs:75-85
pub(crate) struct ProposalInitValidation {
    pub height: BlockNumber,
    pub block_timestamp_window_seconds: u64,
    pub previous_proposal_init: Option<PreviousProposalInitInfo>,
    pub l1_da_mode: L1DataAvailabilityMode,
    pub l2_gas_price_fri: GasPrice,
    pub starknet_version: StarknetVersion,
    pub fee_actual: Option<GasPrice>,
    // ← no expected_builder field
}
``` [3](#0-2) 

The `proposer` field IS validated in the consensus manager against the committee-derived expected proposer:

```rust
// crates/apollo_consensus/src/manager.rs:860-866
if proposer != init.proposer {
    warn!("Invalid proposer for height {} and round {}: expected {:?}, got {:?}", ...);
    return Ok(VecDeque::new());
}
``` [4](#0-3) 

But `builder` has no equivalent guard. The proposer node sets `builder` from its own configuration in `initiate_build`:

```rust
// crates/apollo_consensus_orchestrator/src/build_proposal.rs:169-174
let init = ProposalInit {
    ...
    proposer: args.build_param.proposer,
    builder: args.builder_address,   // ← from node config, unconstrained on wire
    ...
};
``` [5](#0-4) 

Because both the proposer and validator use the same `init.builder` when computing the block commitment (via `convert_to_sn_api_block_info`), consensus reaches agreement on a block whose `sequencer_address` is attacker-controlled. The `ProposalFinMismatch` check does not catch this because both sides derive the commitment from the same (unvalidated) `builder`. [6](#0-5) 

### Impact Explanation

`sequencer_address` is the recipient of all fee transfers in the block. In `execute_fee_transfer`, the fee token's `transfer` entry point is called with `sequencer_address` as the recipient:

```rust
// crates/blockifier/src/transaction/account_transaction.rs:571-573
calldata: calldata![
    *block_context.block_info.sequencer_address.0.key(), // Recipient.
    ...
],
``` [7](#0-6) 

A malicious proposer sets `builder` to an attacker-controlled address. Every transaction fee in the block is transferred to that address. The state diff and block commitment reflect this, so the theft is finalized on-chain. This matches the **Critical** impact: "Incorrect fee, gas, bouncer, resource accounting, refund, balance, or L1 gas price effect with economic impact."

Secondary impact: if `builder` is set to match a specific transaction's `sender_address`, `is_sequencer_the_sender()` returns `true` for that transaction, altering its fee-transfer execution path (sequential vs. concurrent), producing a wrong `fee_transfer_call_info` in the execution result. [8](#0-7) 

### Likelihood Explanation

Any validator node that is selected as the proposer for a round can exploit this. Validator selection is deterministic and rotates across the committee. The attack requires no special capability beyond being the current-round proposer. The malicious `builder` value passes all existing validation checks and produces a valid `ProposalCommitment` that other validators accept.

### Recommendation

Add `expected_builder: ContractAddress` to `ProposalInitValidation`, populated from the node's own configured builder address (the same source used in `initiate_build`). In `is_proposal_init_valid`, enforce:

```rust
if init_proposed.builder != proposal_init_validation.expected_builder {
    return Err(ValidateProposalError::InvalidProposalInit(
        init_proposed.clone(),
        proposal_init_validation.clone(),
        format!("builder mismatch: expected={:?}, got={:?}",
            proposal_init_validation.expected_builder, init_proposed.builder),
    ));
}
```

Alternatively, derive the expected `builder` from the validated `proposer` field via a proposer-to-builder registry, ensuring the two identities are cryptographically bound.

### Proof of Concept

1. Attacker controls a validator node that is selected as proposer for height H, round R.
2. Attacker modifies `builder_address` in their node config (or patches `initiate_build`) to set `builder` = `ATTACKER_WALLET`.
3. The node broadcasts `ProposalInit { proposer: LEGITIMATE_PROPOSER_ID, builder: ATTACKER_WALLET, ... }`.
4. `handle_proposal` in the consensus manager validates `proposer == expected_proposer` ✓ — passes.
5. `is_proposal_init_valid` checks height, gas prices, timestamp, version — none check `builder` ✓ — passes.
6. `convert_to_sn_api_block_info` sets `sequencer_address = ATTACKER_WALLET`.
7. All transactions execute with `ATTACKER_WALLET` as fee recipient; `execute_fee_transfer` transfers fees to `ATTACKER_WALLET`.
8. Both proposer and validator compute the same `partial_block_hash` (both used `ATTACKER_WALLET`), so `ProposalFinMismatch` is not triggered.
9. Consensus commits the block; all fees for height H are permanently credited to `ATTACKER_WALLET`.
