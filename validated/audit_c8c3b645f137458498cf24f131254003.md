Audit Report

## Title
Asymmetric `priceLimitX64 = 0` Handling Silently Blocks All `zeroForOne = false` Swaps — (`metric-core/contracts/MetricOmmPool.sol`)

## Summary
When `priceLimitX64 = 0` is passed to `swap()` with `zeroForOne = false`, both `_swapToken1ForToken0SpecifiedInput` and `_swapToken1ForToken0SpecifiedOutput` immediately return `(0, 0, 0)` / `(0, 0, 0, 0)` due to an initial guard that is always true for any positive price. The same sentinel value `0` works correctly as "no lower bound" for `zeroForOne = true` swaps. The transaction does not revert; it silently executes a no-op, delivering zero tokens to the recipient.

## Finding Description
The root cause is an asymmetric initial guard in the four internal swap routing functions.

For `zeroForOne = false` (buying token0 with token1), `_swapToken1ForToken0SpecifiedInput` contains:

```solidity
if (params.priceLimitX64 <= initialPriceX64) {
    return (0, 0, 0);
}
``` [1](#0-0) 

`_swapToken1ForToken0SpecifiedOutput` has the identical guard:

```solidity
if (params.priceLimitX64 <= initialPriceX64) {
    return (0, 0, 0, 0);
}
``` [2](#0-1) 

With `priceLimitX64 = 0` and any positive `initialPriceX64` (which is always the case for a live pool), `0 <= initialPriceX64` is **always true**, so both functions return immediately with zero amounts before touching any bin.

For `zeroForOne = true` (selling token0), `_swapToken0ForToken1SpecifiedInput` has the mirror guard:

```solidity
if (params.priceLimitX64 >= initialPriceX64) {
    return (0, 0, 0);
}
``` [3](#0-2) 

With `priceLimitX64 = 0`, `0 >= initialPriceX64` is **false** for any positive price, so the swap proceeds. Inside the loop, the per-bin guard `if (params.priceLimitX64 != 0 && ...)` at line 1157 further confirms `0` is treated as "no lower bound" for this direction. [4](#0-3) 

Back in `swap()`, when `_executeSwap` returns `(0, 0, 0)`, the code proceeds to call the swap callback with `(0, 0, callbackData)` and emits a `Swap` event with zero deltas — the transaction succeeds silently with no token movement. [5](#0-4) 

No validation of `priceLimitX64` against direction exists in `swap()` itself. [6](#0-5) 

## Impact Explanation
This constitutes broken core pool functionality. Any caller — including a periphery router or direct integrator — that uses `priceLimitX64 = 0` as the natural "no upper bound" sentinel for `zeroForOne = false` swaps (mirroring how `0` works for `zeroForOne = true`) will silently receive `(0, 0)` from `swap()`. The user receives zero output tokens and zero input is pulled. In a multi-hop router, downstream logic operating on zero deltas will produce incorrect results. The failure is silent (no revert), making it undetectable without explicit output validation.

## Likelihood Explanation
`priceLimitX64 = 0` is the natural "unlimited" sentinel for `zeroForOne = true` and is nowhere documented as invalid for `zeroForOne = false`. Any router, aggregator, or direct caller that symmetrically uses `0` to mean "no price limit" for both directions will silently receive zero output on every `zeroForOne = false` swap. No special privilege is required; this is triggered by any unprivileged caller passing a standard, reasonable input.

## Recommendation
Add a direction-aware guard in `swap()` or in the routing functions. The cleanest fix is to treat `priceLimitX64 = 0` as `type(uint128).max` when `zeroForOne = false` (meaning "no upper bound"), mirroring how `0` is already treated as "no lower bound" for `zeroForOne = true`. Alternatively, add an explicit revert:

```solidity
require(priceLimitX64 != 0 || zeroForOne, InvalidPriceLimit());
```

and document that callers must pass `type(uint128).max` for "no upper bound" on `zeroForOne = false` swaps.

## Proof of Concept
1. Deploy a pool with token0/token1 and add liquidity to at least one bin.
2. Call `pool.swap(recipient, false, 1e18, 0, callbackData, "")` — `zeroForOne = false`, `priceLimitX64 = 0`.
3. `_executeSwap` routes to `_swapToken1ForToken0SpecifiedInput`; `initialPriceX64 > 0` so `0 <= initialPriceX64` is true → returns `(0, 0, 0)`.
4. `swap()` calls the callback with `(0, 0, callbackData)`, emits `Swap(... 0, 0 ...)`, and returns `(0, 0)`.
5. No token0 is transferred to `recipient`; no token1 is pulled from the caller.
6. Repeat with `zeroForOne = true, priceLimitX64 = 0`: the swap executes normally across all bins, confirming the asymmetry.
