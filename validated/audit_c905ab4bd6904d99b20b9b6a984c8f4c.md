### Title
Fee Under-Pricing for Batches with Subtransaction Reference Scripts — (`eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs`)

### Summary
The minimum fee for a Dijkstra batch is computed using only the top-level transaction's reference scripts, while the total reference script size limit is enforced across the entire batch (top-level + all subtransactions). Subtransactions pay no fees at all. An attacker can construct a batch where subtransactions reference scripts up to `ppMaxRefScriptSizePerTx` bytes, paying only the top-level tx's (near-zero) ref script fee, while forcing nodes to deserialize the full batch's ref scripts.

### Finding Description
`getMinFeeTxUtxo = getConwayMinFeeTxUtxo` in `DijkstraEra` calls `txNonDistinctRefScriptsSize utxo tx`, which only inspects the top-level tx's `inputsTxBodyL ∪ referenceInputsTxBodyL`. It does not traverse `subTransactionsTxBodyL`. Meanwhile, `validateAllRefScriptSize` calls `batchNonDistinctRefScriptsSize`, which explicitly sums `txNonDistinctRefScriptsSize` over the top-level tx and every subtransaction. Subtransactions have no fee field enforced (`updateUTxOStateNoFees`), so the only fee covering the entire batch's ref script deserialization cost is the top-level tx's fee — which is computed ignoring subtransaction ref scripts entirely.

### Impact Explanation
**Medium** — fees are modified outside design parameters. An attacker can submit batches where the paid fee is strictly less than the fee that would be required if subtransaction ref scripts were included in the calculation. The gap scales with `ppMaxRefScriptSizePerTx` and the tiered `ppMinFeeRefScriptCostPerByte` schedule. This enables systematic under-priced resource consumption (ref script deserialization) per batch submitted.

The claimed impact of **High — permanent freezing of funds requiring a hard fork** is **not reachable** through this path. The total ref script size is bounded by `ppMaxRefScriptSizePerTx`, so the resource cost per batch is bounded. No funds are frozen; no ledger state divergence occurs; no hard fork is required to recover. The correct impact is Medium (fee manipulation outside design parameters).

### Likelihood Explanation
Straightforward to exploit: any transaction author can embed subtransactions with large reference script inputs. No privileged access, governance majority, or key compromise is required. The batch passes all existing validation checks.

### Recommendation
Replace `getConwayMinFeeTxUtxo` in the `DijkstraEra` `EraUTxO` instance with a Dijkstra-specific implementation that passes `batchNonDistinctRefScriptsSize utxo tx` (instead of `txNonDistinctRefScriptsSize utxo tx`) to `getMinFeeTx`, so the minimum fee accounts for all ref scripts in the batch. [2](#0-1) 

### Proof of Concept
1. Set `ppMinFeeRefScriptCostPerByte` to a non-zero value.
2. Construct a top-level tx with no reference inputs.
3. Embed N subtransactions each referencing UTxO entries whose scripts total just under `ppMaxRefScriptSizePerTx` bytes.
4. Compute `getMinFeeTxUtxo pp tx utxo` — it returns a fee with zero ref-script component.
5. Compute `batchNonDistinctRefScriptsSize utxo tx` — it returns the full subtransaction ref script size.
6. Assert: `getMinFeeTxUtxo pp tx utxo < getMinFeeTx pp tx (batchNonDistinctRefScriptsSize utxo tx)`.
7. Submit the batch with the lower fee — it is accepted by the ledger.
