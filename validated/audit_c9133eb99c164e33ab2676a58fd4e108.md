### Title
Unguarded `init` Function Allows Any Caller to Front-Run MPC Contract Initialization with Malicious Participant Set - (File: crates/contract/src/lib.rs)

### Summary
The `MpcContract::init` function lacks the `#[private]` access modifier present on `init_running`. Any NEAR account can call `init` during the window between contract deployment and the legitimate initialization transaction, injecting an attacker-controlled `ThresholdParameters` (participant set and threshold). With threshold=1 and themselves as the sole participant, an attacker gains full unilateral control of the MPC signing network.

### Finding Description
The `init` function is the primary initialization entry point for the MPC contract. It is decorated with `#[init]` (which only prevents re-initialization after state is written) but **not** `#[private]` (which would restrict callers to the contract account itself).

```rust
#[handle_result]
#[init]
pub fn init(
    parameters: dtos::ThresholdParameters,
    init_config: Option<dtos::InitConfig>,
) -> Result<Self, Error> {
``` [1](#0-0) 

By contrast, `init_running` — the migration-path initializer — correctly carries both `#[init]` and `#[private]`:

```rust
#[private]
#[init]
#[handle_result]
pub fn init_running(...)
``` [2](#0-1) 

The ABI snapshot confirms this asymmetry: `init` lists only `["init"]` as a modifier, while `init_running` lists `["init", "private"]`. [3](#0-2) 

The deployment procedure is a two-step process: first `deploy_contract`, then a separate `init_contract` transaction. The gap between these two on-chain events is the attack window. [4](#0-3) 

Inside `init`, the contract calls `TeeState::with_mocked_participant_attestations(initial_participants)`, which grants all supplied participants mock-valid TEE attestations without any real TEE verification. This means an attacker-supplied participant set is immediately treated as fully attested. [5](#0-4) 

### Impact Explanation
An attacker who successfully front-runs `init` with `ThresholdParameters { participants: [attacker], threshold: 1 }`:

1. Becomes the sole recognized participant with a mock-valid attestation.
2. Can immediately call `vote_add_domains` to add signing domains (ECDSA, EdDSA, CKD).
3. Can call `start_keygen_instance` (as leader, since they are the only participant) and `vote_pk` to complete distributed key generation — with themselves as the only voter, threshold=1 is immediately met.
4. Gains the ability to call `respond` and deliver signatures for any `sign` request, issuing threshold signatures for arbitrary foreign-chain transactions.

This satisfies the **Critical** impact class: *Unauthorized transaction execution, threshold signature issuance, or confidential key derivation output without the required participant authorization.*

The legitimate deployer's subsequent `init` call will fail with "contract already initialized," leaving the attacker in permanent control of the MPC network.

### Likelihood Explanation
- The attack window is the block gap between the `deploy` and `init` transactions, which are always separate on-chain events per the documented deployment procedure.
- An attacker passively monitoring the NEAR blockchain for new deployments of the MPC contract WASM (identifiable by code hash) can detect the deployment and submit a front-running `init` transaction with higher gas in the same or next block.
- No privileged access, leaked keys, or threshold collusion is required — a single unprivileged NEAR account suffices.
- The `#[init]` guard only prevents a second initialization; it provides zero caller restriction.

### Recommendation
Add `#[private]` to `init`, mirroring the protection already applied to `init_running`:

```rust
#[handle_result]
#[init]
#[private]          // ← add this
pub fn init(
    parameters: dtos::ThresholdParameters,
    init_config: Option<dtos::InitConfig>,
) -> Result<Self, Error> {
``` [1](#0-0) 

In NEAR SDK, `#[private]` restricts the method so that `predecessor_account_id == current_account_id`, meaning only the contract account itself (the deployer) can call it. This is the same protection already in place for `init_running`. [2](#0-1) 

### Proof of Concept
1. Deployer broadcasts: `deploy mpc-contract.near use-file mpc_contract.wasm without-init-call`.
2. Attacker observes the deployment receipt on-chain.
3. Attacker broadcasts (with higher gas priority):
   ```json
   {
     "method": "init",
     "args": {
       "parameters": {
         "threshold": 1,
         "participants": {
           "next_id": 1,
           "participants": [["attacker.near", 0, {"tls_public_key": "<attacker_ed25519_pk>", "url": "https://attacker.example"}]]
         }
       }
     }
   }
   ```
4. Attacker's `init` executes first; `TeeState::with_mocked_participant_attestations` grants the attacker a valid mock attestation.
5. Deployer's `init` fails: contract state already exists.
6. Attacker calls `vote_add_domains([{id:0, purpose:Sign, protocol:CaitSith, ...}])` — passes immediately (threshold=1, attacker is the only voter).
7. Attacker calls `start_keygen_instance` and `vote_pk` with an arbitrary Secp256k1 public key whose private key the attacker knows — passes immediately.
8. Contract transitions to `Running` with attacker's key. Attacker can now call `respond` to fulfill any user `sign` request with a valid signature, issuing unauthorized threshold signatures over foreign-chain transactions. [6](#0-5)
