### Title
Validator Accepts Arbitrary `fee_proposal_fri` During Startup Window When `fee_actual` Is `None` — (File: `crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

---

### Summary

In `is_proposal_init_valid`, the `fee_proposal_fri` bounds check is gated on `fee_actual` being `Some`. During the startup window (first `fee_proposal_window_size` blocks), `fee_actual` is always `None`, so the bounds check is silently skipped. A malicious proposer can inject any `fee_proposal_fri` value into the accepted `ProposalInit`. Those values are recorded verbatim into `fee_proposals_window`, and after the window fills, `compute_fee_actual` returns a median derived from the injected values, permanently raising the L2 gas price floor for all subsequent blocks.

---

### Finding Description

`is_proposal_init_valid` enforces the `fee_proposal_fri` range only when both `fee_actual` and `fee_proposal_fri` are `Some`:

```rust
// Validate fee_proposal is within the configured margin of fee_actual.
// During initiation (fee_actual is None, <window_size blocks), bounds are not enforced.
if let (Some(fee_actual), Some(fee_proposal)) =
    (proposal_init_validation.fee_actual, init_proposed.fee_proposal_fri)
{
    let (lower_bound, upper_bound) = fee_proposal_bounds(
        fee_actual,
        VersionedConstants::latest_constants().fee_proposal_margin_ppt,
    );
    if fee_proposal.0 < lower_bound || fee_proposal.0 > upper_bound {
        return Err(...);
    }
}
``` [1](#0-0) 

When `fee_actual` is `None` the entire block is skipped. `fee_actual` is `None` for every height below `fee_proposal_window_size` because `compute_fee_actual` returns `None` when the window is not yet full:

```rust
let Some(start) = height.0.checked_sub(window_size) else {
    warn!("Cannot compute fee_actual for height {height}: height is below window_size ...");
    return None;
};
``` [2](#0-1) 

The proposer's own fallback when `fee_actual` is `None` is well-defined — it freezes at `self.l2_gas_price`:

```rust
let Some(fee_actual) = fee_actual else {
    warn!("fee_actual unavailable, freezing fee_proposal at l2_gas_price");
    return self.l2_gas_price;
};
``` [3](#0-2) 

The validator already independently verifies `l2_gas_price_fri` against its own state:

```rust
if !(init_proposed.height == proposal_init_validation.height
    && init_proposed.l1_da_mode == proposal_init_validation.l1_da_mode
    && init_proposed.l2_gas_price_fri == proposal_init_validation.l2_gas_price_fri)
``` [4](#0-3) 

So the validator has the correct reference value (`l2_gas_price_fri`) but does not use it to bound `fee_proposal_fri` during the startup window. A malicious proposer can set `fee_proposal_fri` to any value — including `u128::MAX` — and the validator will accept it.

The accepted value is then committed to the block header and recorded into `fee_proposals_window`:

```rust
self.record_fee_proposal(height, init.fee_proposal_fri);
``` [5](#0-4) 

After `window_size` blocks, `compute_fee_actual` returns the median of those recorded values. That median is then used as a hard floor for the L2 gas price in every subsequent block:

```rust
let effective_min = match fee_actual {
    Some(fa) => GasPrice(max(config_min.0, fa.0)),
    None => config_min,
};
calculate_next_base_gas_price(current_l2_gas_price, l2_gas_used, gas_target, effective_min)
``` [6](#0-5) 

---

### Impact Explanation

A malicious proposer who controls a majority of blocks during the startup window can set `fee_proposal_fri` to an extreme value on each of those blocks. Once the window is full, `fee_actual` equals the median of those extreme values. `calculate_next_l2_gas_price_for_fin` then uses `fee_actual` as the minimum gas price floor for every subsequent block. The L2 gas price ratchets upward toward that floor (at most `1/333` per block due to `MIN_GAS_PRICE_INCREASE_DENOMINATOR`), eventually causing all transactions to be overcharged or to fail entirely. This is an incorrect fee/gas effect with direct economic impact on all users of the network.

---

### Likelihood Explanation

The vulnerability window is the first `fee_proposal_window_size` blocks of the network's life (or after a fresh genesis). A validator who is selected as proposer for a majority of those blocks — possible if they hold a large stake fraction — can execute the attack without any off-chain coordination. No special privilege beyond being a validator is required.

---

### Recommendation

When `fee_actual` is `None`, the validator should verify that `init_proposed.fee_proposal_fri` equals `proposal_init_validation.l2_gas_price_fri` (the same reference value already checked for `l2_gas_price_fri`), or is within a small absolute margin of it. Concretely, replace the silent skip with:

```rust
if fee_actual is None && fee_proposal_required {
    // proposer must freeze at l2_gas_price when window is incomplete
    if fee_proposal != l2_gas_price_fri {
        return Err(InvalidProposalInit(..., "fee_proposal must equal l2_gas_price during startup"));
    }
}
```

---

### Proof of Concept

1. Network starts at height 0; `fee_proposals_window` is empty; `compute_fee_actual` returns `None` for all heights `< window_size`.
2. Malicious validator is selected as proposer for blocks 0 … `window_size − 1`.
3. For each block, proposer sets `fee_proposal_fri = Some(GasPrice(u128::MAX))` in `ProposalInit` instead of the correct fallback `l2_gas_price`.
4. Validator calls `is_proposal_init_valid`:
   - `fee_proposal_required = true` (V0_14_3+, `Some(_)` arm passes).
   - `fee_actual = None` → the `if let (Some(fee_actual), Some(fee_proposal))` pattern does not match → bounds check is skipped entirely.
5. Proposal is accepted; `record_fee_proposal(height, Some(GasPrice(u128::MAX)))` stores the extreme value.
6. After `window_size` blocks, `compute_fee_actual` returns `Some(GasPrice(u128::MAX))` (median of all-`u128::MAX` window).
7. `calculate_next_l2_gas_price_for_fin` sets `effective_min = u128::MAX`; the L2 gas price ratchets toward `u128::MAX` at `+0.3%` per block.
8. Within ~230 blocks the L2 gas price doubles; within ~1600 blocks it reaches `u128::MAX`, making every transaction unaffordable and effectively halting the chain.
