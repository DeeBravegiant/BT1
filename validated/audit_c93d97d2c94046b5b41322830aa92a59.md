Audit Report

## Title
Token Decimal Precision Not Normalized in `viewSwapRsETHAmountAndFee` Causes Drastically Undercalculated rsETH Minted for Non-18-Decimal Token Deposits - (File: contracts/pools/RSETHPoolV3.sol)

## Summary
The token-deposit overload of `viewSwapRsETHAmountAndFee(uint256 amount, address token)` in all five L2 pool contracts multiplies the raw token amount (in native decimals) directly by a 1e18-precision oracle rate without first normalizing to 18 decimals. For any supported token with fewer than 18 decimals, the computed rsETH output is `10^(18 - tokenDecimals)` times smaller than correct, causing a near-total loss of deposited value for the user. The bug is present identically across `RSETHPoolV3`, `RSETHPoolNoWrapper`, `RSETHPoolV3ExternalBridge`, `RSETHPoolV3WithNativeChainBridge`, and `RSETHPool`.

## Finding Description
The ETH-deposit overload correctly normalizes the amount to 18-decimal precision: [1](#0-0) 

```solidity
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
```

The token-deposit overload omits this normalization: [2](#0-1) 

```solidity
uint256 tokenToETHRate = IOracle(supportedTokenOracle[token]).getRate();
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
```

`ChainlinkOracleForRSETHPoolCollateral.getRate()` always returns a value normalized to 1e18 precision, regardless of the underlying Chainlink feed's native decimals: [3](#0-2) 

So `tokenToETHRate` is always 1e18-scaled, but `amountAfterFee` is in the token's native decimals. For a 6-decimal token (e.g., USDC), `amountAfterFee` is 1e12 times smaller than the equivalent 18-decimal value, making the final `rsETHAmount` 1e12 times smaller than correct.

The `addSupportedToken` function performs no decimal check on the token being added: [4](#0-3) 

The same flawed calculation is present in all other pool contracts: [5](#0-4) [6](#0-5) [7](#0-6) [8](#0-7) 

## Impact Explanation
**Critical — Direct theft of user funds.** A user depositing a non-18-decimal token via the public `deposit(address token, uint256 amount, string referralId)` function transfers their full token balance to the pool (which is then bridged to L1 and permanently absorbed), but receives rsETH/wrsETH worth approximately `10^(18 - tokenDecimals)` times less than fair value. For a 6-decimal token the shortfall is a factor of 1e12 (~100% loss). The user holds almost no rsETH and has no mechanism to recover the deposited tokens. This is a permanent, near-total loss of deposited principal, matching the Critical impact class of "Direct theft of any user funds."

## Likelihood Explanation
The currently deployed supported token is wstETH (18 decimals), for which the formula is coincidentally correct. However, `addSupportedToken` accepts any ERC-20 and oracle pair with no decimal validation. The protocol's architecture explicitly anticipates future token expansion (multiple `reinitialize` versions already add tokens). Any legitimate governance action to add a common LST or stablecoin with fewer than 18 decimals (e.g., USDC, USDT, WBTC) immediately activates the bug for every subsequent depositor of that token. The admin is not the attacker; they act in good faith, and the bug silently harms all users who deposit the newly supported token. Likelihood is Medium.

## Recommendation
Normalize `amountAfterFee` to 18-decimal precision before applying the rate ratio in all five pool contracts:

```solidity
uint8 tokenDecimals = IERC20Metadata(token).decimals();
uint256 amountNormalized = amountAfterFee * 1e18 / (10 ** uint256(tokenDecimals));
rsETHAmount = amountNormalized * tokenToETHRate / rsETHToETHrate;
```

Alternatively, enforce an 18-decimal requirement inside `addSupportedToken` and revert for any token with fewer decimals.

## Proof of Concept
1. Admin calls `addSupportedToken(usdc, chainlinkOracleForUSDC)` — USDC has 6 decimals; oracle returns `4e14` (USDC/ETH at $2500/ETH, normalized to 1e18).
2. User calls `RSETHPoolV3.deposit(usdc, 1_000e6, "")` — depositing 1000 USDC (≈ 0.4 ETH fair value, correct rsETH ≈ `3.8e17`).
3. `viewSwapRsETHAmountAndFee` computes:
   - `amountAfterFee ≈ 1_000e6` (assuming 0 fee for simplicity)
   - `tokenToETHRate = 4e14`
   - `rsETHToETHrate = 1.05e18`
   - `rsETHAmount = 1_000e6 * 4e14 / 1.05e18 ≈ 380_952`
4. User receives `380_952` wrsETH (≈ `3.8e-13` rsETH) instead of `≈ 3.8e17` wrsETH — a loss of ~1e12×.
5. The 1000 USDC is bridged to L1 and absorbed. The user's loss is ~100% of deposited value.

**Foundry fork test plan:** Deploy a local fork, mock a 6-decimal ERC-20 and a `ChainlinkOracleForRSETHPoolCollateral` pointing to a mock Chainlink feed, call `addSupportedToken`, then call `deposit` and assert `rsETHAmount < correctAmount / 1e11`.
