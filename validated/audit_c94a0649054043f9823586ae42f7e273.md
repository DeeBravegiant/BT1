### Title
Gateway Admits Signature-Bypassed Invoke Transactions via Overly Broad `skip_stateful_validations` Proxy Check — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function in the Apollo Gateway bypasses the blockifier's `__validate__` entry-point (i.e., account signature verification) for any Invoke V3 transaction whose nonce equals `1` and whose sender address appears in `account_tx_in_pool_or_recent_block`. The proxy check is too broad: it returns `true` for **any** transaction type in the mempool for that address, not exclusively a `deploy_account`. An unprivileged attacker who observes a victim's `deploy_account` in the mempool can immediately submit an Invoke with nonce `1` and an **arbitrary/invalid signature** from the same address. The gateway admits it without running `__validate__`, violating the invariant that every admitted transaction must pass account-level signature verification.

---

### Finding Description

`skip_stateful_validations` is the sequencer-native analog of the external report's shared-nonce isolation failure. In the external report, a shared nonce space lets one permissioned user invalidate another user's signed message. Here, a shared "account presence" signal (`account_tx_in_pool_or_recent_block`) lets any observer inject a signature-less Invoke into the mempool under a victim's undeployed address.

**Root cause — the proxy check is not type-specific.** [1](#0-0) 

The function checks:

```rust
if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
    return mempool_client
        .account_tx_in_pool_or_recent_block(tx.sender_address())
        .await
        ...
}
```

The comment claims this is sufficient because the account "either has a `deploy_account` transaction or transactions with future nonces that passed validations." The second branch is circular: the only way a future-nonce Invoke can be in the mempool for an undeployed account is if `skip_validate` was already triggered for it. The first branch is the real guard, but the implementation never checks the *type* of the transaction that caused the address to appear in the mempool.

**`account_tx_in_pool_or_recent_block` is address-only, not type-specific.** [2](#0-1) 

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
```

`contains_account` in `MempoolState` checks only whether the address key exists in `staged` or `committed` maps: [3](#0-2) 

No transaction-type filter is applied anywhere in this path.

**When `skip_validate = true`, `run_validate_entry_point` sets `validate = false`.** [4](#0-3) 

```rust
let strict_nonce_check = false;
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
```

With `validate = false`, the blockifier's `StatefulValidator::perform_validations` short-circuits before calling `__validate__`: [5](#0-4) 

```rust
ApiTransaction::Invoke(_) => {
    tx.perform_pre_validation_stage(self.state(), &tx_context)?;
    if !tx.execution_flags.validate {
        return Ok(());
    }
    ...
}
```

The signature is never checked at admission time.

**`validate_by_mempool` (called before `skip_stateful_validations`) does not check signatures.** [6](#0-5) 

`ValidationArgs` carries only `address`, `account_nonce`, `tx_hash`, `tx_nonce`, `tip`, and `max_l2_gas_price`: [7](#0-6) 

No signature field is present; the mempool's `validate_tx` only enforces nonce ordering and fee escalation.

---

### Impact Explanation

An attacker can submit an Invoke transaction with nonce `1` and a completely invalid (or empty) signature for any undeployed account whose `deploy_account` is currently in the mempool. The gateway admits the transaction without running the account's `__validate__` entry point. The transaction enters the mempool and can be included in a block. During block execution the blockifier **does** call `__validate__`, which fails, causing the transaction to be rejected. The concrete effects are:

- **Mempool admission of invalid transactions** — the invariant "every admitted transaction has passed account signature verification" is broken.
- **Block-space consumption** — the rejected Invoke occupies a slot in the block, displacing valid transactions.
- **Mempool slot squatting** — the attacker's Invoke occupies the `(address, nonce=1)` slot, preventing the legitimate user from submitting their own nonce-1 Invoke (the mempool would return `DuplicateNonce`).

The last point is the direct analog of the external report: the attacker front-runs the victim's nonce-1 slot, making the victim's legitimate Invoke unsubmittable until the attacker's transaction is evicted or committed.

This matches the required impact: **High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

---

### Likelihood Explanation

**Low–Medium.** The attacker must:
1. Observe a `deploy_account` transaction for a target address in the mempool (publicly visible).
2. Construct an Invoke with nonce `1` from that address with any signature.
3. Submit it before the `deploy_account` is committed (a window of one block time).

No privileged access is required. The attack is fully unprivileged and requires only standard RPC access.

---

### Recommendation

Replace the address-only proxy check with a type-specific check. The mempool should expose a method such as `has_deploy_account_in_pool_or_recent_block(address)` that inspects the transaction type stored for the address. Alternatively, the gateway can track the `deploy_account` transaction hash returned during the initial submission and pass it explicitly (as the `native_blockifier` `PyValidator` already does via `deploy_account_tx_hash: Option<TransactionHash>`): [8](#0-7) 

The Apollo Gateway's `skip_stateful_validations` should require a confirmed `deploy_account` hash, not merely address presence.

### Proof of Concept

```
State: address X has on-chain nonce = 0 (undeployed).

Step 1 — Victim submits deploy_account:
  POST /add_transaction
  { type: DEPLOY_ACCOUNT, sender: X, nonce: 0, signature: <valid> }
  → Gateway admits; mempool now contains deploy_account for X.
  → account_tx_in_pool_or_recent_block(X) == true.

Step 2 — Attacker submits Invoke with invalid signature:
  POST /add_transaction
  { type: INVOKE, sender: X, nonce: 1, signature: [0x0, 0x0] }

  Gateway flow:
    get_nonce_from_state(X)          → Nonce(0)          ✓
    validate_nonce(nonce=1, acct=0)  → 0 ≤ 1 ≤ max_gap  ✓
    validate_by_mempool(nonce=1)     → no duplicate      ✓
    skip_stateful_validations:
      nonce == 1 && acct_nonce == 0  → true
      account_tx_in_pool_or_recent_block(X) → true  ← deploy_account present
      returns true  (skip __validate__)
    run_validate_entry_point(skip=true) → validate=false → returns Ok(())

  → Attacker's Invoke admitted to mempool with invalid signature.

Step 3 — Victim attempts to submit their own Invoke (nonce=1):
  POST /add_transaction
  { type: INVOKE, sender: X, nonce: 1, signature: <valid> }
  → validate_by_mempool → MempoolError::DuplicateNonce  ← BLOCKED

Step 4 — Block is built:
  deploy_account(X) executes → X deployed, nonce becomes 1.
  Invoke(X, nonce=1, sig=[0,0]) executes → __validate__ fails → REJECTED.
  Victim's valid Invoke was never admitted.
```

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}

    pub fn should_run_stateful_validations(
        &mut self,
        account_tx: &AccountTransaction,
        deploy_account_tx_hash: Option<TransactionHash>,
    ) -> StatefulValidatorResult<bool> {
        if account_tx.tx_type() != TransactionType::InvokeFunction {
            return Ok(true);
        }
        let tx_info = account_tx.create_tx_info();
        let nonce = self.stateful_validator.get_nonce(tx_info.sender_address())?;

        let deploy_account_not_processed =
            deploy_account_tx_hash.is_some() && nonce == Nonce(Felt::ZERO);
        let tx_nonce = tx_info.nonce();
        let is_post_deploy_nonce = Nonce(Felt::ONE) <= tx_nonce;
        let nonce_small_enough_to_qualify_for_validation_skip =
            tx_nonce <= self.max_nonce_for_validation_skip;

        let skip_validate = deploy_account_not_processed
            && is_post_deploy_nonce
            && nonce_small_enough_to_qualify_for_validation_skip;

        Ok(!skip_validate)
    }
