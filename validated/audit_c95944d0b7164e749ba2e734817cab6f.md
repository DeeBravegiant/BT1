### Title
Caller-Agnostic `VerifyForeignTransactionRequest` Key Enables Queue-Saturation to Block Legitimate Bridge Inflows - (File: `crates/contract/src/lib.rs`)

### Summary
The `verify_foreign_transaction` endpoint stores pending requests under a key that does not include the caller's account ID. Any unprivileged account can submit 128 identical requests for a known foreign-chain transaction ID, saturating the bounded fan-out queue (`MAX_PENDING_REQUEST_FAN_OUT = 128`) and causing all subsequent legitimate submissions of the same request to be rejected with `PendingRequestQueueFull`. This breaks the bridge's request-lifecycle invariants by permanently blocking a specific foreign-tx verification until the attacker's queued yields time out.

---

### Finding Description

**Root cause — caller identity is absent from the `VerifyForeignTransactionRequest` key.**

`sign()` derives a per-caller key by hashing `(predecessor_id, path)` into a `Tweak`:

```rust
// crates/near-mpc-crypto-types/src/sign.rs:118-125
impl SignatureRequest {
    pub fn new(domain: DomainId, payload: Payload, predecessor_id: &AccountId, path: &str) -> Self {
        let tweak = crate::kdf::derive_tweak(predecessor_id, path);
        SignatureRequest { domain_id: domain, tweak, payload }
    }
}
```

Two different callers submitting the same `(payload, path, domain_id)` produce **different** `SignatureRequest` keys because the tweak encodes the caller. Front-running is structurally impossible for `sign()`. [1](#0-0) 

`verify_foreign_transaction()` does the opposite. The conversion helper `args_into_verify_foreign_tx_request` simply drops the predecessor:

```rust
// crates/contract/src/dto_mapping.rs:840-848
pub fn args_into_verify_foreign_tx_request(
    args: dtos::VerifyForeignTransactionRequestArgs,
) -> dtos::VerifyForeignTransactionRequest {
    dtos::VerifyForeignTransactionRequest {
        domain_id: args.domain_id,
        request: args.request,
        payload_version: args.payload_version,
    }
}
``` [2](#0-1) 

The resulting `VerifyForeignTransactionRequest` struct contains no caller field:

```rust
// crates/near-mpc-contract-interface/src/types/foreign_chain.rs:124-128
pub struct VerifyForeignTransactionRequest {
    pub request: ForeignChainRpcRequest,
    pub domain_id: DomainId,
    pub payload_version: ForeignTxPayloadVersion,
}
``` [3](#0-2) 

This struct is used directly as the map key in `pending_verify_foreign_tx_requests`. Every caller who submits the same `(ForeignChainRpcRequest, DomainId, ForeignTxPayloadVersion)` tuple is enqueued under the **same** key. [4](#0-3) 

**The fan-out queue has a hard cap of 128 entries per key:**

```rust
// crates/contract/src/pending_requests.rs:37,51-57
pub const MAX_PENDING_REQUEST_FAN_OUT: u8 = 128;

if queue.len() >= usize::from(MAX_PENDING_REQUEST_FAN_OUT) {
    env::panic_str(
        &RequestError::PendingRequestQueueFull { limit: MAX_PENDING_REQUEST_FAN_OUT }.to_string(),
    );
}
``` [5](#0-4) 

Once the queue for a given `VerifyForeignTransactionRequest` key reaches 128, every subsequent `verify_foreign_transaction` call for that same foreign-tx is rejected with `PendingRequestQueueFull`. The NEAR yield-resume promise for the rejected call never receives a response; the caller's transaction fails.

The test comment at line 3242 of `lib.rs` acknowledges the design: *"a different account would today be blocked from receiving a response by alice's submission"* — but the test only exercises the two-caller fan-out case, not the queue-saturation case. [6](#0-5) 

---

### Impact Explanation

**Medium — request-lifecycle manipulation that breaks bridge inflow accounting invariants.**

`verify_foreign_transaction` is the on-chain entry point for the Omnibridge inbound flow: a bridge contract calls it to obtain an MPC attestation that a specific foreign-chain transaction finalized, then uses that attestation to release funds on NEAR. If the queue for a specific foreign-tx ID is saturated before the bridge contract submits its request, the bridge contract's call fails. The bridge user's funds remain locked on the foreign chain until the attacker's 128 queued yields time out (NEAR yield timeout period). If the attacker re-saturates the queue on each retry, the inflow is permanently blocked without any threshold-level collusion.

The attacker does not receive the victim's funds; the impact is targeted denial of a specific bridge inflow, breaking the bridge's request-lifecycle and accounting invariants.

---

### Likelihood Explanation

**Medium.** The attack requires:
1. Observing a specific foreign-chain transaction ID before the bridge contract processes it — trivially achievable by monitoring the foreign chain.
2. Submitting 128 `verify_foreign_transaction` calls with 1 yoctonear deposit each — total cost ≈ 128 yoctonear + minimal gas, essentially free.
3. No privileged access, no TEE compromise, no threshold collusion.

The attacker-controlled entry path is the public `verify_foreign_transaction` endpoint, callable by any NEAR account. [7](#0-6) 

---

### Recommendation

1. **Include the caller in the request key.** Derive a per-caller component (analogous to the `Tweak` used in `sign()`) and embed it in `VerifyForeignTransactionRequest`. This makes each caller's queue slot independent, eliminating cross-caller saturation.

2. **Alternatively, enforce a per-caller sub-limit.** Track how many yields each `predecessor_account_id` has queued for a given request key and reject submissions that exceed a per-caller cap (e.g., 1–2), while keeping the global fan-out cap for legitimate deduplication.

---

### Proof of Concept

```
// Setup: Bitcoin tx_id [0xbb; 32] is about to be verified by a bridge contract.

// Step 1: Attacker observes the pending Bitcoin tx on-chain.

// Step 2: Attacker submits 128 identical verify_foreign_transaction calls
//         (each with 1 yoctonear deposit, ~10 Tgas):
for _ in 0..128 {
    attacker.call(mpc_contract, "verify_foreign_transaction")
        .args_json({ "request": {
            "domain_id": FOREIGN_TX_DOMAIN_ID,
            "payload_version": "V1",
            "request": { "Bitcoin": { "tx_id": [0xbb;32], "confirmations": 1,
                                      "extractors": ["BlockHash"] } }
        }})
        .deposit(1 yoctonear)
        .transact_async();
}

// Step 3: Queue for VerifyForeignTransactionRequest { Bitcoin([0xbb;32]), ... } is now full (128).

// Step 4: Bridge contract submits its legitimate request:
bridge_contract.call(mpc_contract, "verify_foreign_transaction")
    .args_json({ "request": { /* same parameters */ } })
    .deposit(1 yoctonear)
    .transact();
// → FAILS: "Pending-request queue is full (limit: 128)"

// Step 5: Bridge user's inflow is blocked. Funds remain locked on Bitcoin
//         until the attacker's 128 yields time out.
```

pub const MAX_PENDING_REQUEST_FAN_OUT: u8 = 128;

/// Append a yield index to the pending-request fan-out queue for `request`.
///
/// Panics with `RequestError::PendingRequestQueueFull` if the resulting queue would
/// exceed `MAX_PENDING_REQUEST_FAN_OUT`.
pub(crate) fn push_pending_yield<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: K,
    data_id: CryptoHash,
) where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let queue = requests.entry(request).or_default();
    if queue.len() >= usize::from(MAX_PENDING_REQUEST_FAN_OUT) {
        env::panic_str(
            &RequestError::PendingRequestQueueFull {
                limit: MAX_PENDING_REQUEST_FAN_OUT,
            }
            .to_string(),
        );
    }
    queue.push(YieldIndex { data_id });
