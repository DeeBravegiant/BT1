Audit Report

## Title
ETH Deposit Limit Check Omits Incoming Amount, Allowing Cap Bypass - (`contracts/LRTDepositPool.sol`)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` applies structurally different comparisons for ETH versus ERC-20 assets: the ETH branch omits the incoming `amount` from the comparison, so the cap is never enforced against the size of a new ETH deposit. Any unprivileged depositor can push ETH deposits arbitrarily beyond the configured limit in a single transaction.

## Finding Description
At [1](#0-0)  the ETH branch evaluates `totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)` while the ERC-20 branch evaluates `totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)`. The ETH branch returns `false` (limit not exceeded) whenever the running total has not yet crossed the cap, regardless of how large the incoming deposit is.

`_beforeDeposit` at [2](#0-1)  relies solely on this check before minting rsETH; there is no secondary guard. The public entry point `depositETH` at [3](#0-2)  passes `msg.value` directly to `_beforeDeposit` with no additional limit enforcement. The view helper `getAssetCurrentLimit` at [4](#0-3)  confirms the intended semantics: remaining headroom is `limit - totalAssetDeposits`, meaning a deposit of `amount` should be rejected when `totalAssetDeposits + amount > limit`.

## Impact Explanation
The ETH deposit cap (`depositLimitByAsset[ETH_TOKEN]`) is a governance-set risk-management parameter. Because the check ignores the incoming amount, a single depositor can send a deposit many multiples of the remaining headroom and the transaction succeeds: rsETH is minted for the full amount and the recorded total exceeds the configured cap. No funds are stolen, but the protocol's deposit-limit invariant is broken for every ETH depositor.

**Impact: Low** — Contract fails to deliver promised returns (deposit cap is not enforced for ETH); no direct fund loss or freeze.

## Likelihood Explanation
`depositETH` is public and permissionless. No special role, precondition, or prior state manipulation is required. Any depositor who observes `getTotalAssetDeposits(ETH_TOKEN) < depositLimitByAsset[ETH_TOKEN]` can bypass the cap in a single transaction. The condition is deterministically reachable whenever the ETH deposit limit has not already been exceeded.

**Likelihood: High**

## Recommendation
Add `amount` to the ETH branch to mirror the ERC-20 branch:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

## Proof of Concept
Assume `depositLimitByAsset[ETH_TOKEN] = 100 ether` and `getTotalAssetDeposits(ETH_TOKEN) = 99 ether`.

1. Attacker calls `depositETH{value: 1000 ether}(0, "")`.
2. `_beforeDeposit` calls `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 1000 ether)`.
3. ETH branch evaluates `99 ether > 100 ether` → `false` → limit not exceeded.
4. `_beforeDeposit` returns rsETH amount for 1000 ETH; `_mintRsETH` mints it.
5. `getTotalAssetDeposits(ETH_TOKEN)` is now 1099 ether — 10× the configured cap — with no revert.

**Foundry test plan:** Deploy with `depositLimitByAsset[ETH_TOKEN] = 100 ether`, seed 99 ETH of prior deposits, call `depositETH{value: 1000 ether}`, assert the call does not revert and `getTotalAssetDeposits(ETH_TOKEN) > 100 ether`.
