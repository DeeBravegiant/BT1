Audit Report

## Title
Cross-Token Withdrawal Drains Bridger Collateral and Permanently Freezes Depositor's Assets - (File: `contracts/L2/RsETHTokenWrapper.sol`)

## Summary

`_withdraw` enforces only that the requested asset is in `allowedTokens`, but never verifies that the caller is redeeming the same token they deposited. Because wrsETH is a single fungible token whose `totalSupply` aggregates deposits of all allowed tokens, any wrsETH holder can burn their balance to claim whichever allowed token they choose. When the bridger has deposited tokenB as collateral for already-circulating wrsETH, an attacker who holds wrsETH originally minted against tokenA can drain the bridger's tokenB, leaving tokenA permanently irrecoverable.

## Finding Description

`_withdraw` performs exactly one asset-level check before burning and transferring:

```solidity
// contracts/L2/RsETHTokenWrapper.sol line 121
if (!allowedTokens[_asset]) revert TokenNotAllowed();
```

There is no per-user or per-mint record of which token backed each wrsETH unit. `_deposit` mints wrsETH 1:1 for any allowed token:

```solidity
// line 134-141
ERC20Upgradeable(_asset).safeTransferFrom(msg.sender, address(this), _amount);
_mint(_to, _amount);
```

`maxAmountToDepositBridgerAsset` computes headroom per-asset independently:

```solidity
// line 109
return wrsETHSupply - balanceOfAssetInWrapper;
```

This allows the bridger to legitimately deposit N tokenB as long as `totalSupply() - tokenB.balanceOf(wrapper) >= N`, regardless of how much tokenA is already held. Once both tokens are present, the invariant `sum(tokenX.balanceOf(wrapper)) == totalSupply()` can be broken by any holder calling `withdraw` with the non-deposited token.

## Impact Explanation

Two Critical impacts are simultaneously triggered:

1. **Direct theft of user funds**: The bridger's tokenB collateral is transferred to the attacker.
2. **Permanent freezing of funds**: After the attack `totalSupply() == 0`, so no wrsETH exists to redeem the locked tokenA. There is no admin rescue path in the contract.

Both match allowed Critical impacts in the scope.

## Likelihood Explanation

The precondition — two tokens in `allowedTokens` — is the explicit purpose of `addAllowedToken` (TIMELOCK_ROLE) and `reinitialize` (DEFAULT_ADMIN_ROLE). The bridger depositing collateral for a second token is the documented normal operational flow (`depositBridgerAssets` is described as the mechanism to "collateralize already minted wrsETH on L2"). No privileged role compromise, leaked key, or external protocol failure is required; any wrsETH holder can execute step 3 unilaterally. The attack is repeatable as long as the bridger continues to post collateral.

## Recommendation

Maintain per-token share accounting. The minimal fix is a `mapping(address user => mapping(address token => uint256 shares))` updated on every `_deposit` and `_withdraw`, enforcing that a user can only redeem the token they deposited. Alternatively, issue separate wrapper tokens per allowed asset rather than a single fungible wrsETH. A lighter mitigation is to store a `depositedToken` per wrsETH mint and enforce it on burn, but this breaks fungibility. The core invariant to enforce is: `tokenX.balanceOf(wrapper) >= shares minted against tokenX` at all times.

## Proof of Concept

```
State: tokenA and tokenB both in allowedTokens; totalSupply = 0.

Step 1 — user calls deposit(tokenA, 100):
  tokenA.balanceOf(wrapper) = 100
  totalSupply()             = 100

Step 2 — bridger calls depositBridgerAssets(tokenB, 100):
  maxAmountToDepositBridgerAsset(tokenB)
    = totalSupply() - tokenB.balanceOf(wrapper)
    = 100 - 0 = 100  ✓ (no revert)
  tokenB.balanceOf(wrapper) = 100
  totalSupply()             = 100   (unchanged)

Step 3 — user calls withdraw(tokenB, 100):
  allowedTokens[tokenB] == true  ✓ (only guard, line 121)
  burns 100 wrsETH from user
  transfers 100 tokenB to user

Final state:
  tokenA.balanceOf(wrapper) = 100   ← permanently frozen
  tokenB.balanceOf(wrapper) = 0     ← drained
  totalSupply()             = 0     ← no wrsETH left to redeem tokenA
```

Foundry invariant test to confirm: assert `sum(token.balanceOf(wrapper) for token in allowedTokens) >= totalSupply()` after any sequence of `deposit`/`withdraw`/`depositBridgerAssets` calls with two allowed tokens. This invariant will be violated after the sequence above.
