Audit Report

## Title
Missing Caller Authorization in `claim` Allows Forced Fee Extraction on Any User's Unclaimed Yield - (File: contracts/utils/MerkleDistributor/MerkleDistributor.sol)

## Summary
The `claim` function in `MerkleDistributor.sol` accepts an arbitrary `account` parameter but never validates that `msg.sender == account`. Any external caller with a valid merkle proof can trigger a claim on behalf of any user, forcing the protocol fee deduction on the victim's unclaimed rewards without their consent. The fee â€” up to 10% â€” is permanently diverted to `protocolTreasury`, reducing the user's yield below their full entitlement.

## Finding Description
In `MerkleDistributor.sol`, the `claim` function at lines 97â€“147 performs merkle proof verification, fee calculation, and token transfer, but contains no check that the caller is the beneficiary: [1](#0-0) 

The fee is computed and sent to `protocolTreasury` unconditionally: [2](#0-1) 

The claim is then permanently recorded against `account`: [3](#0-2) 

By contrast, `KernelMerkleDistributor.sol`'s `_processClaim` explicitly enforces caller identity before any state change: [4](#0-3) 

Merkle proofs are published off-chain for users to self-claim, making them publicly accessible. An attacker can obtain any user's proof and call `claim(index, victimAddress, cumulativeAmount, proof)` directly. The contract will verify the proof (which is valid), deduct the fee, send the remainder to `victimAddress`, and mark the claim as processed â€” all without the victim's involvement.

## Impact Explanation
This is **Theft of unclaimed yield (High)**. The victim's claimable yield is permanently reduced by `fee = (claimableAmount * feeInBPS) / 10_000`, up to 10% of the claimable amount, which is diverted to `protocolTreasury`. The victim has no recourse: the claim is marked processed at the current `cumulativeAmount`, and the fee cannot be recovered. If the fee is later reduced or waived by the owner, the victim is denied the benefit of claiming under the lower fee â€” a concrete, irreversible loss of yield. [5](#0-4) 

## Likelihood Explanation
The attack requires no capital, no special role, and no victim mistake. The only precondition is possession of a valid merkle proof for the target user, which is routinely published publicly so users can self-claim. Any unprivileged external caller can execute this against any unclaimed account at any time, repeatedly across all users, as long as the contract is unpaused.

## Recommendation
Add a caller validation check at the top of `claim` in `MerkleDistributor.sol`, consistent with `KernelMerkleDistributor.sol`:

```solidity
if (account != msg.sender) {
    revert Unauthorized();
}
```

The `Unauthorized` error should also be added to the `IMerkleDistributor` interface in `MerkleDistributor.sol`, mirroring the pattern in `KernelMerkleDistributor.sol`. [6](#0-5) 

## Proof of Concept
1. Deploy `MerkleDistributor` with `feeInBPS = 1000` (10%).
2. Set a merkle root that includes Alice's entry `(index=1, alice, cumulativeAmount=100e18)`.
3. Obtain Alice's merkle proof from the publicly distributed off-chain data.
4. Attacker (any EOA) calls `claim(1, alice, 100e18, aliceProof)`.
5. Contract transfers `90e18` tokens to Alice and `10e18` to `protocolTreasury`.
6. `userClaims[alice].cumulativeAmount` is set to `100e18`; the claim is permanently processed.
7. If the owner subsequently calls `setFeeInBPS(0)`, Alice cannot reclaim the 10e18 already extracted â€” the loss is permanent.

Foundry test sketch:
```solidity
function test_forcedClaimExtractsFee() public {
    // setup: merkle root with alice entry, feeInBPS = 1000
    vm.prank(attacker);
    distributor.claim(1, alice, 100e18, aliceProof);
    assertEq(token.balanceOf(alice), 90e18);
    assertEq(token.balanceOf(protocolTreasury), 10e18);
    // alice's claim is now marked processed; she cannot reclaim the 10e18
    vm.expectRevert(IMerkleDistributor.AlreadyClaimed.selector);
    vm.prank(alice);
    distributor.claim(1, alice, 100e18, aliceProof);
}
```
