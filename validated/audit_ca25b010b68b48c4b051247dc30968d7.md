### Title
Stale `drepDelegs` in Bootstrap Phase Causes Incorrect Forward-Delegation Clearing on DRep Unregistration — (File: `eras/conway/impl/src/Cardano/Ledger/Conway/Rules/GovCert.hs`)

---

### Summary

During the Conway bootstrap phase (`pvMajor < 10`), `processDelegationInternal` in `Deleg.hs` intentionally omits removing a stake credential from the *old* DRep's `drepDelegs` reverse-delegation set when the credential re-delegates to a new DRep. This leaves a stale entry in the old DRep's `drepDelegs`. When that old DRep is later unregistered via `ConwayUnRegDRep`, the `clearDRepDelegations` helper iterates over the stale `drepDelegs` set and unconditionally sets `dRepDelegationAccountStateL .~ Nothing` for every credential listed — including credentials that have already moved their delegation to a different DRep. The result is that a live, valid DRep delegation is silently destroyed, reducing the target DRep's effective voting power outside the design parameters of the governance system.

---

### Finding Description

**Root cause — `processDelegationInternal` (`Deleg.hs` lines 363–365):**

```haskell
| isNothing mAccountState || preserveIncorrectDelegation ->
    certVStateL . vsDRepsL
      %~ Map.adjust (drepDelegsL %~ Set.insert stakeCred) dRepCred
```

When `preserveIncorrectDelegation = True` (i.e., `pvMajor pv < natVersion @10`), re-delegation from DRep A → DRep B only *inserts* `stakeCred` into DRep B's `drepDelegs`. It never calls `unDelegReDelegDRep`, so DRep A's `drepDelegs` retains `stakeCred` as a stale entry. The forward delegation in the account (`dRepDelegationAccountStateL`) is correctly updated to `Just (DRepCredential dRepBCred)`.

**Consequence — `ConwayUnRegDRep` (`GovCert.hs` lines 244–254):**

```haskell
let
  certState' =
    certState & certVStateL . vsDRepsL %~ Map.delete cred
  clearDRepDelegations delegs accountsMap =
    foldr (Map.adjust (dRepDelegationAccountStateL .~ Nothing)) accountsMap delegs
pure $
  case mDRepState of
    Nothing -> certState'
    Just dRepState ->
      certState'
        & certDStateL . accountsL . accountsMapL
          %~ clearDRepDelegations (drepDelegs dRepState)
```

`mDRepState` is captured before deletion (correct), but `drepDelegs dRepState` is the *stale* set that still contains `stakeCred`. `clearDRepDelegations` therefore sets `dRepDelegationAccountStateL .~ Nothing` for `stakeCred`, destroying its live delegation to DRep B. The reverse entry in DRep B's `drepDelegs` is left intact, producing an inconsistent ledger state:

| State field | Expected | Actual after attack |
|---|---|---|
| `stakeCred` account → DRep | `Just (DRepCredential dRepBCred)` | `Nothing` |
| DRep B `drepDelegs` | contains `stakeCred` | still contains `stakeCred` (stale) |

**Test confirmation (`DelegSpec.hs` lines 313–323):**

```haskell
impAnn "Check that unregistration of previous delegation does not affect current delegation" $ do
  unRegisterDRep drepCred
  ifBootstrap
    ( do
        accounts <- getsNES $ nesEsL . esLStateL . lsCertStateL . certDStateL . accountsL
        expectNothingExpr (lookupDRepDelegation cred accounts)   -- forward delegation wiped
        expecteReverseDRepDelegation cred drepCred2 True         -- reverse delegation stale
    )
    (expectDelegatedVote cred (DRepCredential drepCred2))
```

The test explicitly confirms that in the bootstrap phase, unregistering the *old* DRep wipes the credential's forward delegation to the *new* DRep.

---

### Impact Explanation

Voting power for DRep ratification is computed from the forward delegation stored in each account's `dRepDelegationAccountStateL`. Once that field is set to `Nothing`, the stake credential's ADA weight is excluded from DRep B's tally. An attacker who engineers the sequence below can silently reduce any target DRep's ratification weight:

1. Attacker registers DRep A (permissionless, deposit returned on unregistration).
2. Attacker's own stake credentials delegate to DRep
