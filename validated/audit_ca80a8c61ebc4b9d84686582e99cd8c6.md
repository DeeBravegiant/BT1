Audit Report

## Title
ETH Deposit Limit Bypass Due to Missing `+ amount` in Limit Check - (File: contracts/LRTDepositPool.sol)

## Summary
`LRTDepositPool._checkIfDepositAmountExceedesCurrentLimit()` omits the incoming `amount` from the ETH branch comparison, while the LST branch correctly includes it. This allows any depositor to push total ETH deposits past the configured `depositLimitByAsset` cap whenever deposits are at or near the limit, as the check evaluates only the pre-deposit total rather than the post-deposit total.

## Finding Description
In `_checkIfDepositAmountExceedesCurrentLimit` ( [1](#0-0) ), the ETH branch at line 679 evaluates `totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)` without adding `amount`, while the LST branch at line 681 correctly evaluates `totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)`.

This function is called from `_beforeDeposit` ( [2](#0-1) ), which reverts with `MaximumDepositLimitReached` only when the function returns `true`. Because the ETH branch never accounts for the incoming deposit, when `totalAssetDeposits == depositLimit` the check returns `false` and the deposit proceeds. The public `depositETH` function ( [3](#0-2) ) is callable by any user with no privilege requirements, making this reachable by any depositor.

Existing guards (`whenNotPaused`, `nonReentrant`, `onlySupportedAsset`) do not address the asymmetric comparison logic.

## Impact Explanation
**Low. Contract fails to deliver promised returns, but doesn't lose value.** The `depositLimitByAsset` cap is a protocol-level risk management parameter. When the ETH deposit total is at or above the limit, the cap is silently bypassed: rsETH is minted and ETH is staked into EigenLayer beyond the intended exposure ceiling. No direct fund theft or freezing occurs, but the protocol fails to enforce its stated deposit constraint.

## Likelihood Explanation
**Medium.** No special privileges are required — any external caller can invoke `depositETH`. The deposit limit is an actively managed on-chain parameter. Any depositor (or bot) monitoring the pool can trigger this condition whenever `totalAssetDeposits >= depositLimit`, which is a normal operational state. The condition is trivially detectable and repeatable.

## Recommendation
Add `+ amount` to the ETH branch to match the LST branch:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

## Proof of Concept
1. Admin sets `depositLimitByAsset[ETH_TOKEN] = 100 ether` via `LRTConfig.updateAssetDepositLimit`.
2. Prior deposits bring `getTotalAssetDeposits(ETH_TOKEN)` to exactly `100 ether`.
3. Any user calls `depositETH{value: 1 ether}(0)`.
4. `_beforeDeposit` calls `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 1 ether)`.
5. Check evaluates `100 ether > 100 ether` → `false` → no revert.
6. `_mintRsETH` mints rsETH; total ETH deposits become `101 ether`, exceeding the cap.
7. The same bypass applies for any subsequent deposit as long as `totalAssetDeposits >= depositLimit`.

**Foundry test plan:** Deploy `LRTDepositPool` on a local fork, set `depositLimitByAsset[ETH_TOKEN] = 100 ether`, simulate prior deposits to reach exactly `100 ether`, then call `depositETH{value: 1 ether}` and assert it does not revert and that `getTotalAssetDeposits(ETH_TOKEN)` returns `101 ether`.

        uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
        if (asset == LRTConstants.ETH_TOKEN) {
            return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));
        }
