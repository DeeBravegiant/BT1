Audit Report

## Title
Staking Rewards Permanently Frozen When `totalKernelStaked` Drops to Zero During Active Reward Period - (File: `contracts/KERNEL/KernelDepositPool.sol`)

## Summary
`KernelDepositPool.rewardPerToken()` returns the frozen `rewardPerTokenStored` without accumulation whenever `totalKernelStaked == 0`. Because `initiateWithdrawal()` immediately decrements `totalKernelStaked` while the `updateReward` modifier unconditionally advances `updatedAt`, any reward tokens accruing during the zero-staking window are silently skipped and permanently locked in the contract. No admin recovery or sweep function exists, and the `notifyRewardAmount` delta-transfer pattern cannot reclaim the stranded balance.

## Finding Description
Three code paths interact to produce the freeze:

1. **`rewardPerToken()` early-return** (`L408–414`): when `totalKernelStaked == 0` the function returns the cached `rewardPerTokenStored` unchanged, so elapsed time generates no new reward accumulation.

2. **`updateReward` modifier unconditionally advances `updatedAt`** (`L232–242`): `updatedAt = lastTimeRewardApplicable()` executes regardless of whether `totalKernelStaked` is zero. Every subsequent interaction (stake, getReward, etc.) silently consumes the time window without distributing rewards.

3. **`initiateWithdrawal()` immediately decrements `totalKernelStaked`** (`L325–326`): the decrement happens before the mandatory `withdrawalDelay` elapses, so the contract enters the zero-staking state while the reward period (`finishAt`) is still in the future.

The `notifyRewardAmount` guard (`L570`) only prevents *starting* a new period with zero stakers; it does not prevent stakers from exiting mid-period. There is no `rescue`, `sweep`, or admin recovery function anywhere in the contract. The `notifyRewardAmount` delta pattern (`balanceAfter - balanceBefore`, `L573–577`) counts only newly deposited tokens, so the frozen balance is invisible to any future reward period.

The contract's own NatSpec (`L18–22`) explicitly acknowledges this behavior and relies on a purely operational mitigation ("ensuring there are always some tokens staked"), which has no on-chain enforcement.

## Impact Explanation
**Medium — Permanent freezing of unclaimed yield.**

Reward tokens transferred via `notifyRewardAmount` and allocated to an active period are irrecoverably locked whenever `totalKernelStaked` reaches zero mid-period. The tokens cannot be redistributed, recovered by the admin, or recaptured in a future `notifyRewardAmount` call. This matches the allowed impact class "Medium. Permanent freezing of unclaimed yield."

## Likelihood Explanation
The scenario requires only a single unprivileged actor: the last remaining staker calling `initiateWithdrawal`. This can occur naturally (all other stakers exit first) or be deliberately engineered. The `withdrawalDelay` (up to `MAX_WITHDRAWAL_DELAY = 30 days`, `L35`) creates a guaranteed multi-day window during which `totalKernelStaked == 0` while the reward period is still active, maximising the frozen amount. No special privileges, flash loans, or external dependencies are required.

## Recommendation
Do not advance `updatedAt` when `totalKernelStaked == 0`, so the skipped interval is replayed once staking resumes — matching the canonical Synthetix `StakingRewards` pattern:

```solidity
modifier updateReward(address _account) {
    rewardPerTokenStored = rewardPerToken();
    if (totalKernelStaked > 0) {          // only advance cursor when stakers exist
        updatedAt = lastTimeRewardApplicable();
    }
    if (_account != address(0)) {
        rewards[_account] = earned(_account);
        userRewardPerTokenPaid[_account] = rewardPerTokenStored;
    }
    _;
}
```

Alternatively, extend `finishAt` by the zero-staking dead-time duration each time `totalKernelStaked` transitions from non-zero to zero, so the full reward budget is eventually distributed.

## Proof of Concept
1. Admin calls `notifyRewardAmount(1_000e18)` with `duration = 10 days`. `rewardRate ≈ 1157 wei/s`, `finishAt = now + 10 days`.
2. Alice is the sole staker: `balanceOf[Alice] = 1_000e18`, `totalKernelStaked = 1_000e18`.
3. After **5 days**, Alice calls `initiateWithdrawal(1_000e18)`:
   - `updateReward(Alice)` snapshots her 5-day earnings correctly into `rewards[Alice]`.
   - `totalKernelStaked` becomes **0** (`L326`).
   - `updatedAt` is advanced to `block.timestamp` (`L234`).
4. For the remaining **5 days** until `finishAt`, every call to `rewardPerToken()` hits the early-return at `L409–411`. Any interaction (e.g., a new staker calling `stake`) triggers `updateReward`, which advances `updatedAt` further without accumulating rewards.
5. Alice calls `claimWithdrawal` and `getReward` — she correctly receives only her 5-day share (~500e18).
6. The **5-day reward tranche** (~500e18 tokens) remains permanently locked in `KernelDepositPool` with no mechanism to recover it.

Foundry test outline:
```solidity
function test_rewardsFrozenWhenTotalStakedZero() public {
    vm.prank(admin);
    pool.notifyRewardAmount(1_000e18); // duration = 10 days

    // Alice is sole staker
    vm.prank(alice);
    pool.stake(1_000e18);

    vm.warp(block.timestamp + 5 days);

    vm.prank(alice);
    pool.initiateWithdrawal(1_000e18); // totalKernelStaked → 0

    vm.warp(block.timestamp + 5 days); // reward period ends

    vm.prank(alice);
    pool.getReward();

    // ~500e18 reward tokens remain stranded in the contract
    uint256 stranded = rewardsToken.balanceOf(address(pool));
    assertGt(stranded, 499e18);
}
```
