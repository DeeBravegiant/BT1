Audit Report

## Title
Stale Cached Rate Returned Without Staleness Check Enables Over-Minting of wrsETH - (File: `contracts/cross-chain/CrossChainRateReceiver.sol`)

## Summary
`CrossChainRateReceiver` caches the rsETH/ETH exchange rate received via LayerZero and records the update time in `lastUpdated`, but `getRate()` returns the cached `rate` unconditionally without consulting `lastUpdated`. Because `RSETHPoolV3` uses this rate to compute how much `wrsETH` to mint per ETH deposited, any period in which the L2 cache lags behind L1 appreciation allows depositors to receive more `wrsETH` than the current rsETH value warrants, diluting existing holders' accrued yield.

## Finding Description
`CrossChainRateReceiver` stores two state variables: `rate` (the cached exchange rate) and `lastUpdated` (the timestamp of the last LayerZero update). [1](#0-0) 

`lzReceive` updates both fields atomically when a LayerZero message arrives from the L1 rate provider: [2](#0-1) 

However, the public `getRate()` function returns `rate` with no reference to `lastUpdated`: [3](#0-2) 

`RSETHPoolV3` sets `rsETHOracle` to a `RSETHRateReceiver` (the concrete subclass of `CrossChainRateReceiver`) and calls `getRate()` inside `viewSwapRsETHAmountAndFee` to compute the minting amount: [4](#0-3) 

The minting formula `rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate` means a stale-low rate (rsETH has appreciated on L1 but the L2 cache is old) produces a larger `rsETHAmount` than the depositor is entitled to. The `deposit()` function mints this inflated amount directly to the caller: [5](#0-4) 

The L1 `MultiChainRateProvider.updateRate()` is manually triggered and requires ETH payment for LayerZero gas fees, with no on-chain enforcement of any update frequency: [6](#0-5) 

There is no staleness guard anywhere in the deposit flow â€” not in `CrossChainRateReceiver.getRate()`, not in `RSETHPoolV3.viewSwapRsETHAmountAndFee()`, and not in `RSETHPoolV3.deposit()`. The `dailyMintLimit` check limits total daily volume but does not prevent exploitation within that limit. [7](#0-6) 

## Impact Explanation
When the L2 cached rate is stale-low, every depositor calling `deposit()` on `RSETHPoolV3` receives more `wrsETH` than the current rsETH value warrants. The excess `wrsETH` supply dilutes the share of accrued yield belonging to existing `wrsETH` holders. This constitutes **theft of unclaimed yield** (High impact), matching the allowed impact scope exactly.

## Likelihood Explanation
- `updateRate()` requires a manual call and ETH payment; it is not automatic and has no on-chain deadline.
- LayerZero message delivery can be delayed during network congestion.
- During any period of rsETH appreciation on L1 (e.g., after a reward distribution updates `rsETHPrice`), the L2 rate will lag until `updateRate()` is manually called.
- Any unprivileged depositor can observe the stale rate on-chain by reading `CrossChainRateReceiver.rate` and `lastUpdated`, then immediately exploit it by calling `deposit()`.
- The attack is repeatable up to the `dailyMintLimit` and requires no special privileges or capital beyond the deposit amount.

## Recommendation
Add a configurable `maxStaleness` parameter to `CrossChainRateReceiver` and revert in `getRate()` if the cached rate is too old:

```solidity
uint256 public maxStaleness; // e.g., 24 hours

function getRate() external view returns (uint256) {
    require(block.timestamp - lastUpdated <= maxStaleness, "Rate is stale");
    return rate;
}
```

`maxStaleness` should be set by the owner and should reflect the expected update cadence with a reasonable buffer. This ensures that `RSETHPoolV3` and any other consumer will revert rather than silently use an outdated rate.

## Proof of Concept
1. L1 `LRTOracle.rsETHPrice` is updated to `1.05e18` (rsETH has appreciated 5%).
2. `MultiChainRateProvider.updateRate()` has not been called; `CrossChainRateReceiver.rate` still holds `1.00e18` and `lastUpdated` is stale.
3. Attacker calls `RSETHPoolV3.deposit{value: 1 ether}("")`.
4. `deposit()` calls `viewSwapRsETHAmountAndFee(1e18)`, which calls `IOracle(rsETHOracle).getRate()` â†’ returns `1.00e18` (stale).
5. `rsETHAmount = 1e18 * 1e18 / 1.00e18 = 1.000e18` wrsETH minted.
6. Correct amount at current rate: `1e18 * 1e18 / 1.05e18 â‰ˆ 0.952e18` wrsETH.
7. Attacker receives ~5% excess wrsETH, diluting existing holders' unclaimed yield.

**Foundry fork test plan:**
- Fork an L2 where `RSETHRateReceiver` is deployed.
- Warp `block.timestamp` forward past any reasonable update window without calling `updateRate()`.
- Manually set `CrossChainRateReceiver.rate` to a value lower than the current L1 rate (simulating stale cache).
- Call `RSETHPoolV3.deposit{value: 1 ether}("")` as an unprivileged address.
- Assert that the minted `wrsETH` amount exceeds `1e18 * 1e18 / currentL1Rate`, confirming over-minting. [8](#0-7) [9](#0-8)
