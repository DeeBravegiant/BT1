### Title
Missing State Diff Hash Verification in P2P Sync Allows Malicious Peer to Inject Corrupted State — (`crates/apollo_p2p_sync/src/client/state_diff.rs`)

### Summary

`parse_data_for_block` in the P2P sync state diff client assembles a `ThinStateDiff` from peer-supplied `StateDiffChunk` messages and validates only that the total chunk length matches `state_diff_length` from the stored header. It never calls `calculate_state_diff_hash` to verify the assembled diff against the `state_diff_commitment` stored in that same header. A malicious peer can therefore inject an arbitrary `ThinStateDiff` that passes all existing checks and is committed permanently to storage via `append_state_diff`.

### Finding Description

**Phase 1 — Header injection (no cryptographic barrier):**

`HeaderStreamBuilder::parse_data_for_block` accepts a `SignedBlockHeader` from a peer with only two checks:
- `block_number` matches the expected value
- `signatures.len() == ALLOWED_SIGNATURES_LENGTH` (count only, no cryptographic verification) [1](#0-0) 

The `block_hash` is not recomputed, and the signature bytes are not verified against any known sequencer public key. A malicious peer can therefore supply any `state_diff_commitment` and `state_diff_length=N` in the header, which is then stored verbatim.

**Phase 2 — State diff injection (no hash cross-check):**

`StateDiffStreamBuilder::parse_data_for_block` reads `target_state_diff_len` from the stored header and collects chunks until `current_state_diff_len == target_state_diff_len`. After assembly it calls only `validate_deprecated_declared_classes_non_conflicting`. It never reads `state_diff_commitment` from the header and never calls `calculate_state_diff_hash`. [2](#0-1) 

**Phase 3 — Permanent storage write:**

The assembled `ThinStateDiff` is written directly to storage: [3](#0-2) 

**Why the committer's `verify_state_diff_hash` does not help:**

The committer's hash check exists in `commit_or_load` and is gated on `self.config.verify_state_diff_hash`: [4](#0-3) 

This check is in a completely separate service (`apollo_committer`) used by the batcher/sequencer path. The P2P sync path calls `append_state_diff` on `apollo_storage` directly and never touches the committer. The `verify_state_diff_hash = true` deployment config is therefore irrelevant to this attack surface. [5](#0-4) 

### Impact Explanation

A syncing node that accepts a malicious peer's header and state diff chunks will permanently store a `ThinStateDiff` whose Poseidon hash does not match the `state_diff_commitment` in the header. This corrupts the node's view of contract storage, nonces, deployed contracts, and declared classes for that block. All subsequent RPC state queries (`get_storage_at`, `get_nonce`, `get_class_hash_at`) will return wrong values, and any execution that reads state from this node will produce wrong results. The stored `state_diff_commitment` will appear valid while the actual state is silently wrong.

### Likelihood Explanation

Any peer that can establish a P2P connection to the victim node can execute this attack. No validator key, sequencer key, or privileged credential is required. The attacker only needs to:
1. Connect as a peer
2. Send a `SignedBlockHeader` with the target block number, `state_diff_length=N`, and any garbage signature byte (length check only)
3. Send exactly N length-units of crafted `StateDiffChunk` messages

The crafted chunks pass all existing checks (`ConflictingStateDiffParts`, `EmptyStateDiffPart`, `WrongStateDiffLength`, `validate_deprecated_declared_classes_non_conflicting`) as long as they are internally consistent and sum to N.

### Recommendation

In `parse_data_for_block` in `crates/apollo_p2p_sync/src/client/state_diff.rs`, after the assembled `ThinStateDiff` passes length and conflict checks, read `state_diff_commitment` from the stored header and verify:

```rust
let stored_commitment = storage_reader
    .begin_ro_txn()?
    .get_block_header(block_number)?
    .expect("...")
    .state_diff_commitment
    .ok_or(...)?;

let computed = calculate_state_diff_hash(&result);
if computed != stored_commitment {
    return Err(ParseDataError::BadPeer(BadPeerError::StateDiffHashMismatch { ... }));
}
```

This mirrors the check already present in `Committer::commit_or_load` and closes the gap between the two code paths. [4](#0-3) 

### Proof of Concept

1. Start a node running `P2pSyncClient`.
2. Act as a peer; send a `SignedBlockHeader` for block 0 with `state_diff_length=1` and `state_diff_commitment=<hash of real diff>`, with a single garbage signature byte.
3. Send one `StateDiffChunk::ContractDiff` with a different contract address/class hash (length=1, so the length check passes).
4. Observe that `append_state_diff` is called with the crafted diff.
5. Query `get_state_diff(BlockNumber(0))` from storage and assert it does not match the `state_diff_commitment` stored in the header — confirming the corrupted state was accepted without error.
