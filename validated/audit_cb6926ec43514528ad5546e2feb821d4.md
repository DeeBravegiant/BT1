**Key code path:**

1. `determine_coin_type` fetches the parent coin spend via `fetch_coin_spend_for_coin_state` → `fetch_coin_spend`
2. `fetch_coin_spend` verifies `puzzle_reveal` against the parent coin's `puzzle_hash`, but **does not verify the solution**

**The missing check:**

In `puzzle_solution_received`, the `coin` parameter is the actual on-chain child coin. The function computes `child_coin` from `compute_additions(data.parent_coin_spend)` — running the puzzle with the peer-supplied solution — and stores `child_coin` without ever asserting `child_coin.name() == coin.name()` or `child_coin.puzzle_hash == coin.puzzle_hash`. **Attack feasibility:**

- `fetch_coin_spend` checks `puzzle.get_tree_hash() != coin.puzzle_hash` but places **no constraint on the solution**. [3](#0-2) 
- `get_metadata_and_phs` runs `unft.p2_puzzle.run(unft.get_innermost_solution(solution))`. For a standard NFT, `p2_puzzle` is `puzzle_for_pk(owner_pubkey)` — a delegated-puzzle wrapper. - The attacker sets the memo (`condition.at("rrrff")`) to a victim p2_puzzle_hash they know (e.g., from a hint seen on-chain). The wallet looks up a derivation record for it, finds one, and proceeds. [5](#0-4)

- The wallet then calls `add_coin(child_coin, ...)` storing a phantom `NFTCoinInfo` for a coin that does not exist on-chain (or has a different puzzle_hash than the actual child coin). [7](#0-6) 

**Why the existing guards do not block this:**

- `validate_received_state_from_peer` validates the actual child coin's existence in the blockchain — it does not validate the parent coin's solution. [8](#0-7) 
- `UncurriedNFT.uncurry` is applied to the puzzle_reveal (verified), not the solution. [9](#0-8)

### Title
Malicious Peer Can Inject Phantom NFT Record via Crafted Parent Coin Spend Solution — (`chia/wallet/nft_wallet/nft_wallet.py`)

### Summary
`puzzle_solution_received` trusts the solution returned by a peer without verifying that the child coin derived from `compute_additions` matches the actual on-chain coin. A malicious peer can supply a crafted solution to cause the wallet to store a phantom NFT it cannot spend.

### Finding Description
`fetch_coin_spend` validates the puzzle reveal against the parent coin's puzzle hash but imposes no constraint on the solution. `puzzle_solution_received` uses the peer-supplied solution to:
1. Extract `p2_puzzle_hash` via `get_metadata_and_phs` (runs the p2 puzzle with attacker-controlled innermost solution)
2. Reconstruct `child_puzzle` and `child_puzzle_hash` from the victim's derivation record
3. Find a matching coin in `compute_additions(data.parent_coin_spend)` (also runs with the crafted solution)
4. Store that coin as an owned NFT

There is no assertion that `child_coin.name() == coin.name()` (the actual on-chain coin passed to `coin_added`).

### Impact Explanation
The wallet stores a phantom `NFTCoinInfo` record for a coin that does not exist on-chain (or has a different puzzle hash). The wallet believes it owns an NFT it cannot spend. If the wallet attempts to include this phantom NFT in an offer settlement, the offer will fail or produce an invalid spend bundle. This is wallet state corruption with direct security impact on offer/trade settlement state — a **High** severity finding.

### Likelihood Explanation
Any peer connected to the wallet can execute this attack. The attacker needs only: (a) a real NFT coin on-chain (any NFT), (b) knowledge of one of the victim's p2 puzzle hashes (observable from on-chain hints), and (c) the ability to serve a crafted solution. No private key material is required.

### Recommendation
In `puzzle_solution_received`, after finding `child_coin` via `compute_additions`, assert it matches the actual on-chain coin:

```python
if child_coin.name() != coin.name():
    raise ValueError(
        f"Child coin from solution {child_coin.name()} does not match "
        f"on-chain coin {coin.name()}"
    )
```

Additionally, `fetch_coin_spend` should ideally validate the solution against the block's removals/additions root, or at minimum the wallet should cross-check the computed additions against the known on-chain child coin state.

### Proof of Concept
1. Connect a malicious peer to the victim wallet.
2. Identify a victim p2_puzzle_hash `V` from the wallet's subscribed puzzle hashes (observable via hints on-chain).
3. Locate any real NFT parent coin `P` on-chain with puzzle_hash `H_P`.
4. Compute `child_puzzle_hash = create_full_puzzle(singleton_id, metadata, metadata_updater_hash, puzzle_for_pk(victim_pubkey))`.
5. Craft solution `S'`: the innermost delegated puzzle outputs `CREATE_COIN(nft_state_layer(..., puzzle_for_pk(victim_pubkey)).get_tree_hash(), 1, [V])`.
6. When the wallet requests `RequestPuzzleSolution` for `P`, return the real puzzle_reveal with `S'`.
7. Observe that the wallet stores a phantom `NFTCoinInfo` for a coin with puzzle_hash = `child_puzzle_hash` that does not exist on-chain.
8. Confirm the wallet reports owning this NFT but any spend attempt fails.

```python
def get_metadata_and_phs(unft: UncurriedNFT, solution: SerializedProgram) -> tuple[Program, bytes32]:
    conditions = unft.p2_puzzle.run(unft.get_innermost_solution(Program.from_serialized(solution)))
    metadata = unft.metadata
    puzhash_for_derivation: bytes32 | None = None
    for condition in conditions.as_iter():
        if condition.list_len() < 2:
            # invalid condition
            continue
        condition_code = condition.first().as_int()
        log.debug("Checking condition code: %r", condition_code)
        if condition_code == -24:
            # metadata update
            metadata = update_metadata(metadata, condition)
            metadata = Program.to(metadata)
        elif condition_code == 51:
            atom = condition.rest().rest().first().as_int()

            if atom == 1:
                # destination puzhash
                if puzhash_for_derivation is not None:
                    # ignore duplicated create coin conditions
                    continue
                memo = bytes32(condition.at("rrrff").as_atom())
                puzhash_for_derivation = memo
                log.debug("Got back puzhash from solution: %s", puzhash_for_derivation)
    assert puzhash_for_derivation
    return metadata, puzhash_for_derivation
```

```python
        self,

        peer: WSChiaConnection,

        """
        Returns True if the coin_state is valid and included in the blockchain proved by the weight proof.
        """
        if peer.closed:
            return False
        # Only use the cache if we are talking about states before the fork point. If we are evaluating something
        # in a reorg, we cannot use the cache, since we don't know if it's actually in the new chain after the reorg.
        if can_use_peer_request_cache(coin_state, peer_request_cache, fork_height):
            return True

        spent_height: uint32 | None = None if coin_state.spent_height is None else uint32(coin_state.spent_height)
        confirmed_height: uint32 | None = (
            None if coin_state.created_height is None else uint32(coin_state.created_height)
        )
        current = await self.wallet_state_manager.coin_store.get_coin_record(coin_state.coin.name())
        # if remote state is same as current local state we skip validation

        current_spent_height = None
        if current is not None and current.spent_block_height != 0:
            current_spent_height = current.spent_block_height

            current is not None
            and current_spent_height == spent_height
            and current.confirmed_block_height == confirmed_height

            peer_request_cache.add_to_states_validated(coin_state)
            return True

        # If coin was removed from the blockchain
        if confirmed_height is None:
```
