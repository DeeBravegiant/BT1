### Title
`propose_update` Storage Deposits Permanently Locked in Contract When `do_update` Clears All Proposals — (`File: crates/contract/src/update.rs`)

---

### Summary

The `propose_update` function requires participants to attach a NEAR token deposit proportional to the size of the proposed contract binary or config. When `vote_update` reaches threshold and calls `do_update`, **all** pending proposals are cleared from storage, but the deposits paid by every proposer are never refunded. These tokens are permanently locked in the MPC contract with no withdrawal mechanism.

---

### Finding Description

**Step 1 — Deposit collection in `propose_update`:**

`propose_update` in `lib.rs` requires the caller to attach at least `ProposedUpdates::required_deposit(&update)` in NEAR tokens. Only the *excess* above the required amount is refunded; the required portion stays in the contract's balance. [1](#0-0) 

**Step 2 — Deposit calculation:**

`required_deposit` is computed as `storage_byte_cost × bytes_used`, where `bytes_used` includes the full contract binary length plus overhead for 128 participant vote slots. For a typical WASM binary (~500 KB–1 MB), this amounts to tens to hundreds of NEAR tokens per proposal. [2](#0-1) 

**Step 3 — `do_update` clears all proposals with no refund:**

When the vote threshold is reached, `do_update` removes the winning entry, then calls `self.entries.clear()` and `self.vote_by_participant.clear()`, wiping every other pending proposal. No refund is issued to any proposer — not to the winner, and not to the losers whose proposals were discarded. [3](#0-2) 

**Step 4 — No proposer identity is tracked:**

`UpdateEntry` stores only the `update` payload and `bytes_used`; there is no field recording which account submitted the proposal. This means even a future fix would need to add proposer tracking. [4](#0-3) 

**Step 5 — No withdrawal path exists:**

There is no contract method to withdraw accumulated deposits. The freed storage reduces the contract's storage obligation, making those NEAR tokens available in the contract's balance — but they are inaccessible to the original proposers. [5](#0-4) 

---

### Impact Explanation

Every successful contract upgrade permanently confiscates the storage deposits of all proposers. For a 1 MB contract binary, `required_deposit` is approximately `1,000,000 bytes × 10^-5 NEAR/byte = ~10 NEAR` per proposal. With multiple concurrent proposals (the codebase explicitly supports this), the total locked amount scales linearly. The tokens accumulate in the contract balance with no governance-approved withdrawal path, constituting a permanent, quantifiable loss of participant funds.

This matches the allowed impact: **Medium — Balance/accounting invariant broken without relying on network-level DoS or operator misconfiguration.**

---

### Likelihood Explanation

This loss occurs on **every successful `vote_update` execution**, which is a routine, expected operation during normal MPC network upgrades. No adversarial action is required — the bug is triggered by the standard upgrade governance flow. Any participant who has ever called `propose_update` and had their proposal cleared (either because a different proposal won, or because they were the winner) loses their deposit.

---

### Recommendation

1. Add a `proposer: AccountId` field to `UpdateEntry` to record who submitted each proposal.
2. In `do_update`, before calling `self.entries.clear()`, iterate over all remaining entries and issue `Promise::new(entry.proposer).transfer(required_deposit(&entry.update))` for each.
3. Also refund the winning proposer after the entry is removed at line 196.

```rust
pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
    let entry = self.entries.remove(id)?;
    // Refund winning proposer
    Promise::new(entry.proposer.clone())
        .transfer(required_deposit(entry.bytes_used));

    // Refund all other proposers before clearing
    for (_, other_entry) in self.entries.iter() {
        Promise::new(other_entry.proposer.clone())
            .transfer(required_deposit(other_entry.bytes_used));
    }

    self.entries.clear();
    self.vote_by_participant.clear();
    // ... rest of update logic
}
```

### Proof of Concept

1. Participant A calls `propose_update` with a 500 KB contract binary, attaching the required ~5 NEAR deposit. The excess is refunded; 5 NEAR stays in the contract.
2. Participant B calls `propose_update` with a different 500 KB binary, attaching ~5 NEAR.
3. Threshold participants vote for B's proposal via `vote_update`.
4. `vote_update` calls `do_update` (lib.rs:1383), which removes B's entry, then calls `self.entries.clear()` (clearing A's entry too) and `self.vote_by_participant.clear()`.
5. Neither A nor B receives any refund. Both have permanently lost ~5 NEAR each. The contract balance increases by ~10 NEAR with no withdrawal mechanism. [6](#0-5) [5](#0-4)

fn bytes_used(update: &Update) -> u128 {
    let mut bytes_used = std::mem::size_of::<UpdateEntry>() as u128;

    // Assume a high max of 128 participant votes per update entry.
    bytes_used += 128 * std::mem::size_of::<AccountId>() as u128;

    match update {
        Update::Contract(code) => {
            bytes_used += code.len() as u128;
        }
        Update::Config(config) => {
            let bytes = serde_json::to_vec(&config).unwrap();
            bytes_used += bytes.len() as u128;
        }
    }

    bytes_used
}

fn required_deposit(bytes_used: u128) -> NearToken {
    env::storage_byte_cost().saturating_mul(bytes_used)
}
