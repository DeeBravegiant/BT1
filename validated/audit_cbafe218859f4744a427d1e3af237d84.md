### Title
Receipt Size Hard Limit Bypassed via `promise_return` and Value-Return Paths - (`runtime/runtime/src/lib.rs`)

### Summary

The `max_receipt_size` hard limit (4 MiB) can be bypassed by an unprivileged user deploying a contract that either (a) uses `promise_return` to redirect a near-maximum-size receipt's output, or (b) returns a value close to `max_length_returned_data`. In both cases, the runtime mutates or constructs receipts **after** size validation, producing receipts that exceed `max_receipt_size` and enter the chain. This is an explicitly acknowledged, unfixed bug in the production codebase.

### Finding Description

`validate_receipt` in `runtime/runtime/src/verifier.rs` enforces the receipt size limit only in `ValidateReceiptMode::NewReceipt` mode: [1](#0-0) 

However, in `runtime/runtime/src/lib.rs`, after a function call executes, the runtime performs two post-validation mutations that can push a receipt over the limit:

**Path 1 — `promise_return`:** When a contract returns `ReturnData::ReceiptIndex`, the runtime appends the parent receipt's `output_data_receivers` directly onto the child receipt's list, **after** that child receipt was already validated: [2](#0-1) 

A contract can craft a receipt whose size equals exactly `max_receipt_size`, pass validation, then have `output_data_receivers` appended, pushing it over the limit.

**Path 2 — value return:** When a contract returns `ReturnData::Value`, the runtime wraps the raw data in a `DataReceipt`. The VM enforces `max_length_returned_data` (4 MiB) on the payload, but `validate_data_receipt` only checks the payload length, not the total serialized receipt size: [3](#0-2) 

A payload at the `max_length_returned_data` limit produces a data receipt whose total Borsh-serialized size exceeds `max_receipt_size` once headers are included.

The codebase explicitly acknowledges this bug in two places: [4](#0-3) [5](#0-4) 

Two integration tests confirm that oversized receipts are produced and propagate through the chain: [6](#0-5) [7](#0-6) 

### Impact Explanation

`max_receipt_size` is a **hard limit** whose purpose is to keep `ChunkStateWitness` under 17 MiB, as documented: [8](#0-7) 

Oversized receipts that bypass this limit can inflate `ChunkStateWitness` beyond the design budget. Chunk validators must re-execute receipts to verify chunk application; if they encounter receipts that the chunk producer accepted but that violate the hard limit, the validators' enforcement logic diverges from the producer's behavior. The `ExistingReceipt` mode is a workaround that tolerates these receipts rather than rejecting them, meaning the broken invariant persists in the live chain state. The congestion control layer also has a workaround that clamps oversized receipt sizes to `max_receipt_size` for accounting purposes, masking the true size: [9](#0-8) 

The corrupted protocol value is the `max_receipt_size` hard limit enforcement, which is supposed to bound the state witness size but is silently violated.

### Likelihood Explanation

Any unprivileged user can deploy a WASM contract (within `max_contract_size`) that uses standard NEAR promise APIs (`promise_create`, `promise_then`, `promise_return`, `value_return`) to trigger either path. No special privileges are required. The entry path is a normal signed transaction calling a user-deployed contract.

### Recommendation

1. Re-validate the total serialized receipt size **after** `output_data_receivers` are appended in the `promise_return` path, and reject the receipt if it exceeds `max_receipt_size`.
2. In `validate_data_receipt`, check the total serialized size of the data receipt (not just the payload length) against `max_receipt_size`.
3. Remove the `ExistingReceipt` workaround once the root cause is fixed.

### Proof of Concept

The nearcore test suite itself demonstrates the bug end-to-end. `test_max_receipt_size_promise_return` and `test_max_receipt_size_value_return` both call `assert_oversized_receipt_occurred`, which scans the chain and asserts that a receipt exceeding `max_receipt_size` was actually included and processed: [10](#0-9)

/// Assert that there was an incoming receipt with size above max_receipt_size
fn assert_oversized_receipt_occurred(node: &TestLoopNode<'_>) {
    let client = node.client();
    let chain = &client.chain;
    let epoch_manager = &*client.epoch_manager;

    let tip = chain.head().unwrap();
    let epoch_id = epoch_manager.get_epoch_id(&tip.last_block_hash).unwrap();
    let protocol_version = epoch_manager.get_epoch_protocol_version(&epoch_id).unwrap();
    let runtime_config = client.runtime_adapter.get_runtime_config(protocol_version);
    let max_receipt_size = runtime_config.wasm_config.limit_config.max_receipt_size;

    let mut block = chain.get_block(&tip.last_block_hash).unwrap();

    // Go over all blocks down to genesis looking for a receipt above max_receipt_size.
    loop {
        if block.header().is_genesis() {
            panic!("Didn't find receipt with size above max_receipt_size!");
        }
        let prev_block = chain.get_block(block.header().prev_hash()).unwrap();

        let shard_layout = epoch_manager
            .get_shard_layout(&epoch_manager.get_epoch_id(block.hash()).unwrap())
            .unwrap();

        let oversized = if ProtocolFeature::Spice.enabled(protocol_version) {
            // With spice chunks are executed asynchronously and their produced receipts are
            // persisted as receipt proofs keyed by the block in which the chunk was applied,
            // rather than as incoming receipts on the following block.
            shard_layout.shard_ids().any(|shard_id| {
                chain
                    .chain_store()
                    .iter_receipt_proofs_for_shard(block.hash(), shard_id)
                    .iter()
                    .flat_map(|proof| proof.0.iter())
                    .any(|receipt| receipt_is_oversized(receipt, max_receipt_size))
            })
        } else {
            block.chunks().iter_new().any(|new_chunk| {
                let shard_id = new_chunk.shard_id();
                let prev_shard_index = epoch_manager
                    .get_prev_shard_id_from_prev_hash(block.header().prev_hash(), shard_id)
                    .unwrap()
                    .2;
                let prev_height_included =
                    prev_block.chunks().get(prev_shard_index).unwrap().height_included();
                let incoming_receipts_proofs = get_incoming_receipts_for_shard(
                    &chain.chain_store,
                    epoch_manager,
                    shard_id,
                    &shard_layout,
                    *block.hash(),
                    prev_height_included,
                    ReceiptFilter::TargetShard,
                )
                .unwrap();
                incoming_receipts_proofs
                    .iter()
                    .flat_map(|response| response.1.iter())
                    .flat_map(|proof| proof.0.iter())
                    .any(|receipt| receipt_is_oversized(receipt, max_receipt_size))
            })
        };

        if oversized {
            return;
        }

        block = prev_block;
    }
}

```markdown
## State witness size limits

Some limits were introduced to keep the size of `ChunkStateWitness` reasonable.
`ChunkStateWitness` contains all the incoming transactions and receipts that will be processed during chunk application and in theory a single receipt could be tens of megabytes in size. Distributing a `ChunkStateWitness` this large would be troublesome, so we limit the size and number of transactions, receipts, etc. The limits aim to keep the total uncompressed size of `ChunkStateWitness` under 17MiB.

There are two types of size limits:

* Hard limit - the size must be below this limit, anything else is considered invalid
* Soft limit - things are added until the limit is exceeded, after that things stop being added. The last added thing is allowed to slightly exceed the limit.

The limits are:

* `max_transaction_size = 1.5 MiB`
  * All transactions must be below 1.5 MiB, otherwise they'll be considered invalid and rejected.
  * Previously was 4MiB, now reduced to 1.5MiB
* `max_receipt_size - 4 MiB`:
  * All receipts must be below 4 MiB, otherwise they'll be considered invalid and rejected.
  * Previously there was no limit on receipt size. Set to 4MiB, might be reduced to 1.5MiB in the future to match the transaction limit.
```
