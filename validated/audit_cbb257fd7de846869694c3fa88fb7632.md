Audit Report

## Title
Missing Circuit Breaker in `CrossChainRateReceiver.lzReceive` Allows Depressed Rate to Over-Mint rsETH on L2 — (`contracts/cross-chain/CrossChainRateReceiver.sol`)

## Summary
`CrossChainRateReceiver.lzReceive` unconditionally writes any authenticated rate payload into storage with no monotonicity guard or downside circuit breaker. Because `CrossChainRateProvider.updateRate()` is permissionless, an attacker can time a cross-chain push to coincide with a transient L1 price dip, locking a depressed rate into the L2 receiver. All L2 pools then mint excess rsETH against that depressed rate, creating unbacked supply that causes protocol insolvency when the rate corrects.

## Finding Description

**L2 receiver — no protection:**
`CrossChainRateReceiver.lzReceive` performs only source-chain and source-address authentication, then writes the payload rate directly with no check against the previous `rate`, no `highestRate` watermark, and no pause trigger: [1](#0-0) 

**L1 circuit breaker — present but bypassable:**
`LRTOracle._updateRsETHPrice` pauses the protocol when `pricePercentageLimit > 0` and the drop exceeds the limit. However, `pricePercentageLimit` defaults to `0`, making the condition `pricePercentageLimit > 0 && ...` always false — the circuit breaker never fires in the default configuration: [2](#0-1) 

Even when `pricePercentageLimit` is set, drops within the limit are pushed cross-chain without any L2-side check.

**Rate provider is permissionless:**
`CrossChainRateProvider.updateRate()` carries only a `nonReentrant` modifier — no role restriction. Any EOA can call it at any time to push the current `LRTOracle.rsETHPrice` cross-chain: [3](#0-2) 

`RSETHRateProvider.getLatestRate()` reads the stored `rsETHPrice` directly from `LRTOracle`: [4](#0-3) 

`LRTOracle.updateRSETHPrice()` is also `public whenNotPaused` — permissionless when unpaused: [5](#0-4) 

**L2 pools use the rate directly for minting:**
`RSETHPoolV3.viewSwapRsETHAmountAndFee` computes `rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate`. A depressed `rsETHToETHrate` inflates `rsETHAmount` proportionally: [6](#0-5) 

`RSETHPoolV3.deposit` calls this and mints directly: [7](#0-6) 

The `dailyMintLimit` check in `limitDailyMint` is computed using the already-depressed rate, so the attacker can mint up to `dailyMintLimit` rsETH tokens — each backed by far less ETH than their face value implies: [8](#0-7) 

**Complete exploit path:**
1. Observe (or wait for) a natural dip in underlying asset prices that causes `_getTotalEthInProtocol() / rsethSupply` to drop (e.g., stETH depeg).
2. Call `LRTOracle.updateRSETHPrice()` on L1 to capture the dip into `rsETHPrice`. With `pricePercentageLimit == 0`, no pause fires regardless of drop magnitude.
3. Call `RSETHRateProvider.updateRate{value: lzFee}()` — permissionless — to push the depressed rate cross-chain via LayerZero.
4. LayerZero delivers the message; `RSETHRateReceiver.lzReceive` writes `rate = depressedRate` with no check.
5. Attacker calls `RSETHPoolV3.deposit{value: N ether}("")` on L2. `rsETHAmount = N * 1e18 / depressedRate` — inflated proportionally to the rate depression.
6. When the rate corrects (next legitimate update), attacker redeems inflated rsETH for more ETH than deposited, draining the pool.

## Impact Explanation
If `rate` is pushed as `0.5e18` while the true backing rate is `1.05e18`, every 1 ETH deposited mints `1e18 / 0.5e18 = 2` rsETH instead of `≈0.952` rsETH — a ~2.1× over-issuance. The excess rsETH is unbacked. When the rate corrects, existing holders can redeem more ETH than was deposited, draining the pool. This constitutes **Critical — Protocol insolvency**.

## Likelihood Explanation
- `updateRate()` requires no role — any EOA paying the LayerZero fee can call it.
- `updateRSETHPrice()` is public when unpaused — the attacker can sequence both calls in the same block.
- With `pricePercentageLimit == 0` (the default until explicitly set by admin), arbitrarily large drops are pushed cross-chain without any L1 pause.
- Even with `pricePercentageLimit` set, drops within the limit bypass the L1 circuit breaker entirely, and the L2 has no independent check.
- The attack is repeatable each day up to `dailyMintLimit`, and the `dailyMintLimit` check itself uses the depressed rate, so the ETH-denominated cap is effectively inflated.

## Recommendation
1. Track `highestRate` in `CrossChainRateReceiver`; update it only when the new rate exceeds it.
2. Compute the percentage drop from `highestRate` inside `lzReceive`; if it exceeds a configurable `rateDropLimit`, revert or ignore the update and emit an alert.
3. Expose an owner-only `setRateDropLimit` and `pause`/`unpause` pair on the receiver.
4. Restrict `CrossChainRateProvider.updateRate()` to a trusted keeper role to prevent adversarial timing of rate pushes.
5. Ensure `pricePercentageLimit` is set to a non-zero value during deployment so the L1 circuit breaker is active.
