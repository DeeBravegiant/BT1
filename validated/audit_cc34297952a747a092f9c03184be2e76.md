Audit Report

## Title
Stale `rsETHPrice` Used in Deposit Minting Without Prior `updateRSETHPrice()` Call ‚ÄĒ (File: contracts/LRTDepositPool.sol)

## Summary
`LRTDepositPool.depositAsset()` and `depositETH()` compute the rsETH mint amount using `lrtOracle.rsETHPrice()`, a cached state variable in `LRTOracle` that is only refreshed by explicit calls to `updateRSETHPrice()`. Neither deposit entry point nor the internal `_beforeDeposit()` / `getRsETHAmountToMint()` chain triggers a price refresh before reading the cached value. When EigenLayer rewards or LST rebases have accrued since the last keeper update, a depositor receives more rsETH than their deposit warrants, extracting value from existing holders' unclaimed yield.

## Finding Description
`LRTOracle` stores `rsETHPrice` as a persistent state variable updated only by `updateRSETHPrice()` (public) or `updateRSETHPriceAsManager()` (manager-only):

```solidity
// LRTOracle.sol L28
uint256 public override rsETHPrice;
```

The deposit flow reads this cached value without refreshing it:

- `depositAsset()` / `depositETH()` call `_beforeDeposit()` (a `private view` function ‚ÄĒ cannot mutate state).
- `_beforeDeposit()` calls `getRsETHAmountToMint()` (a `public view` function ‚ÄĒ also cannot mutate state).
- `getRsETHAmountToMint()` divides by `lrtOracle.rsETHPrice()` directly.

```solidity
// LRTDepositPool.sol L520
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

Because `_beforeDeposit()` is `private view`, it structurally cannot call `updateRSETHPrice()`. The calling chain never invokes a price refresh. Between keeper updates, as protocol TVL grows (EigenLayer rewards, LST rebases), the true rsETH price rises above the stored `rsETHPrice`. Any deposit during this window uses the stale lower price:

```
rsethAmountToMint = depositValue / staleRsETHPrice  >  depositValue / trueRsETHPrice
```

The excess rsETH minted represents a claim on TVL that was earned by prior holders, not by the new depositor. When `updateRSETHPrice()` is eventually called, the new price is computed over a larger rsETH supply, permanently diluting existing holders' share of the accrued rewards.

The `pricePercentageLimit` guard in `_updateRsETHPrice()` can additionally delay keeper updates when price movement exceeds the configured threshold, extending the exploitable stale window further.

## Impact Explanation
**High ‚ÄĒ Theft of unclaimed yield.**

Existing rsETH holders accumulate yield as protocol TVL grows relative to rsETH supply. A depositor who mints at a stale-low price receives a larger share of the pool than their deposit justifies. The difference is extracted from the unrealised yield belonging to prior holders. The magnitude scales with (a) time elapsed since the last `updateRSETHPrice()` call and (b) deposit size. In a protocol managing hundreds of millions in TVL with daily EigenLayer rewards, even a few hours of stale price represents a meaningful extractable amount per large deposit, and the attack is repeatable every update cycle.

## Likelihood Explanation
**Medium.**

`updateRSETHPrice()` is called by an off-chain keeper on a periodic schedule, not on every block. The stale window is always non-zero. Any unprivileged external user can observe the last update timestamp on-chain, estimate accrued rewards, and time their deposit to maximise the stale-price advantage ‚ÄĒ no special access or victim cooperation is required. The `pricePercentageLimit` guard can further extend the exploitable window when price movement is large.

## Recommendation
Call `updateRSETHPrice()` (or an internal equivalent) at the start of `depositAsset()` and `depositETH()` before the rsETH mint amount is computed:

```solidity
function depositAsset(
    address asset,
    uint256 depositAmount,
    uint256 minRSETHAmountExpected,
    string calldata referralId
) external nonReentrant whenNotPaused onlySupportedERC20Token(asset) {
+   ILRTOracle(lrtConfig.getContract(LRTConstants.LRT_ORACLE)).updateRSETHPrice();
    uint256 rsethAmountToMint = _beforeDeposit(asset, depositAmount, minRSETHAmountExpected);
    ...
}
```

Apply the same change to `depositETH()`. This ensures the mint calculation always uses the freshest possible rsETH price, eliminating the stale-price dilution vector.

## Proof of Concept
1. Assume `rsETHPrice` was last updated 6 hours ago at `1.010 ETH/rsETH`. Since then, EigenLayer rewards have pushed the true price to `1.011 ETH/rsETH` (~0.1% increase).
2. Attacker calls `depositAsset(stETH, 1000e18, 0, "")` without calling `updateRSETHPrice()` first.
3. `getRsETHAmountToMint` computes: `1000e18 * 1e18 / 1.010e18 ‚Čą 990.099 rsETH` (stale price).
4. True fair amount: `1000e18 * 1e18 / 1.011e18 ‚Čą 989.12 rsETH`.
5. Attacker receives ‚Čą0.979 excess rsETH extracted from existing holders' accrued yield.
6. Attacker waits for keeper to call `updateRSETHPrice()` (or calls it themselves), then redeems at the updated price, capturing the difference.

**Foundry fork test plan:**
- Fork mainnet at a block where `rsETHPrice` is known to be stale (e.g., shortly after a reward accrual event).
- Record `rsETHPrice` before deposit.
- Call `depositAsset()` as an unprivileged address.
- Call `updateRSETHPrice()`.
- Assert that the attacker's rsETH balance, when redeemed at the new price, exceeds the deposited value ‚ÄĒ and that existing holders' ETH-per-rsETH ratio decreased relative to a scenario where `updateRSETHPrice()` was called before the deposit.
