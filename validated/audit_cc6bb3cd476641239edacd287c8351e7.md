### Title
`max_receipt_size` Invariant Bypassed via Post-Validation `output_data_receivers` Mutation - (`runtime/runtime/src/lib.rs`)

### Summary

The `max_receipt_size` hard limit (4 MiB) can be exceeded by an unprivileged user through a specific promise-return pattern. The size check is performed at receipt creation time, but the runtime subsequently mutates the receipt by appending `output_data_receivers` without re-validating the size. This is the exact same invariant-bypass class as the LooksRare report: a limit is enforced in one code path but a later mutation in a different code path silently violates it.

### Finding Description

`validate_receipt()` in `runtime/runtime/src/verifier.rs` enforces `max_receipt_size` only when called with `ValidateReceiptMode::NewReceipt`: [1](#0-0) 

After this validation passes, the runtime processes `promise_return()` results in `runtime/runtime/src/lib.rs`. When a contract executing receipt A calls `promise_return(C)`, the runtime appends A's `output_data_receivers` directly into C's receipt struct **after** C has already passed size validation: [2](#0-1) 

There is no re-validation of C's size after this mutation. If C was crafted to be exactly at `max_receipt_size` before the append, the appended `output_data_receivers` push it over the limit.

A second bypass path exists via `value_return()`: returning a value of size `max_length_returned_data` (4 MiB) causes the runtime to wrap it in a `DataReceipt`. The `validate_data_receipt()` function only checks the payload length against `max_length_returned_data`, not the total serialized receipt size: [3](#0-2) 

The `DataReceipt` envelope overhead causes the total receipt to exceed `max_receipt_size`.

The codebase explicitly acknowledges this bug in the `ValidateReceiptMode` enum documentation: [4](#0-3) 

And in `congestion_control.rs`, a workaround clamps the oversized receipt's size for outgoing-limit accounting, but the receipt itself remains oversized in state: [5](#0-4) 

The bug is confirmed and demonstrated by two integration tests: [6](#0-5) [7](#0-6) 

### Impact Explanation

The `max_receipt_size` limit is a **hard limit** designed to keep `ChunkStateWitness` under 17 MiB for stateless validation. An oversized receipt (e.g., 4 MiB + overhead) that enters state and is forwarded between shards can cause the `ChunkStateWitness` to exceed its intended bound. Chunk validators distributing and processing witnesses that exceed the expected size limit may diverge in their validation results, potentially causing chunk endorsement failures or consensus instability. The concrete corrupted protocol value is the receipt stored in the trie and forwarded as part of `source_receipt_proofs`, which violates the hard invariant documented in `docs/misc/state_witness_size_limits.md`: [8](#0-7) 

### Likelihood Explanation

Any unprivileged user who can deploy a contract and submit transactions can trigger this. The `promise_return` path requires crafting a receipt whose args fill exactly `max_receipt_size` minus the base receipt overhead, then chaining it with a `.then()` callback. The `value_return` path is even simpler: return a value of exactly `max_length_returned_data` bytes from a promise that has a `.then()` continuation. Both patterns are reachable via standard public RPC transactions with no special privileges.

### Recommendation

After the `output_data_receivers` mutation at `runtime/runtime/src/lib.rs` lines 1029–1035, re-validate the modified receipt's serialized size against `max_receipt_size`. If the size is exceeded, treat it as a receipt validation error (fail the action) rather than silently producing an oversized receipt. Similarly, `validate_data_receipt()` should check the total serialized `DataReceipt` size, not just the payload length.

### Proof of Concept

The existing test `test_max_receipt_size_promise_return` in `test-loop-tests/src/tests/max_receipt_size.rs` already demonstrates the bug end-to-end:

1. Deploy a contract.
2. Call `max_receipt_size_promise_return_method1` which creates a promise chain `[A -then-> B]`. During A's execution, it creates receipt C with args sized to exactly `max_receipt_size - base_overhead`, then calls `promise_return(C)`.
3. The runtime appends B's `data_id` as an `output_data_receiver` into C, pushing C's serialized size above `max_receipt_size`.
4. `assert_oversized_receipt_occurred()` confirms an incoming receipt with `size > max_receipt_size` is present in the chain. [9](#0-8)

fn validate_data_receipt(
    limit_config: &LimitConfig,
    receipt: &DataReceipt,
) -> Result<(), ReceiptValidationError> {
    let data_len = receipt.data.as_ref().map(|data| data.len()).unwrap_or(0);
    if data_len as u64 > limit_config.max_length_returned_data {
        return Err(ReceiptValidationError::ReturnedValueLengthExceeded {
            length: data_len as u64,
            limit: limit_config.max_length_returned_data,
        });
    }
    Ok(())
}

fn test_max_receipt_size_promise_return() {
    init_test_logger();

    let account = create_account_id("account0");
    let account_signer = create_user_test_signer(&account);
    let mut env = TestLoopBuilder::new()
        .enable_rpc()
        .add_user_account(&account, Balance::from_near(10_000))
        .build();

    // Deploy the test contract
    let deploy_contract_tx = SignedTransaction::deploy_contract(
        101,
        &account,
        near_test_contracts::rs_contract().into(),
        &account_signer,
        env.rpc_node().head().last_block_hash,
    );
    env.rpc_runner().run_tx(deploy_contract_tx, Duration::seconds(5));

    // User calls a contract method
    // Contract method creates a DAG with two promises: [A -then-> B]
    // When promise A is executed, it creates a third promise - `C` and does a `promise_return`.
    // The DAG changes to: [C ->then-> B]
    // The receipt for promise C is a maximum size receipt.
    // Adding the `output_data_receivers` to C's receipt makes it go over the size limit.
    let base_receipt_template = Receipt::V0(ReceiptV0 {
        predecessor_id: account.clone(),
        receiver_id: account.clone(),
        receipt_id: CryptoHash::default(),
        receipt: ReceiptEnum::Action(ActionReceipt {
            signer_id: account.clone(),
            signer_public_key: account_signer.public_key().into(),
            gas_price: Balance::ZERO,
            output_data_receivers: vec![],
            input_data_ids: vec![],
            actions: vec![Action::FunctionCall(Box::new(FunctionCallAction {
                method_name: "noop".into(),
                args: vec![],
                gas: Gas::ZERO,
                deposit: Balance::ZERO,
            }))],
        }),
    });
    let base_receipt_template = action_receipt_v1_to_latest(&base_receipt_template);
    let base_receipt_size = borsh::object_length(&base_receipt_template).unwrap();
    let max_receipt_size = 4_194_304;
    let args_size = max_receipt_size - base_receipt_size;

    // Call the contract
    let large_receipt_tx = SignedTransaction::call(
        102,
        account.clone(),
        account.clone(),
        &account_signer,
        Balance::ZERO,
        "max_receipt_size_promise_return_method1".into(),
        format!("{{\"args_size\": {}}}", args_size).into(),
        Gas::from_teragas(300),
        env.rpc_node().head().last_block_hash,
    );
    env.rpc_runner().run_tx(large_receipt_tx, Duration::seconds(5));

    // Make sure that the last promise in the DAG was called
    let assert_test_completed = SignedTransaction::call(
        103,
        account.clone(),
        account,
        &account_signer,
        Balance::ZERO,
        "assert_test_completed".into(),
        "".into(),
        Gas::from_teragas(300),
        env.rpc_node().head().last_block_hash,
    );
    env.rpc_runner().run_tx(assert_test_completed, Duration::seconds(5));

    assert_oversized_receipt_occurred(&env.validator());
}
