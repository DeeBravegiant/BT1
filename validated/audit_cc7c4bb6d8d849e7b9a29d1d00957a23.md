Audit Report

## Title
`MultiChainRateProvider.updateRate()` Drains Stranded ETH via Unchecked `msg.value` in Payable Loop - (File: contracts/cross-chain/MultiChainRateProvider.sol)

## Summary
`MultiChainRateProvider.updateRate()` is an unrestricted `payable` function that loops over all registered `rateReceivers`, computing and forwarding a separate `estimatedFee` to LayerZero per iteration. There is no check that `msg.value` covers the sum of all per-iteration fees, and no excess ETH is refunded. Any ETH stranded in the contract from a prior overpaying caller can be fully consumed by a subsequent caller who sends `msg.value = 0`, constituting direct theft of user funds at-rest.

## Finding Description
`updateRate()` is declared `external payable nonReentrant` with no access control: [1](#0-0) 

Inside the loop, each iteration independently calls `estimateFees` and immediately forwards that exact amount to the LayerZero endpoint: [2](#0-1) 

The total ETH consumed equals `Σ estimatedFee_i` across all receivers. There is no pre-loop `require(msg.value >= totalFee)` guard and no post-loop refund of `address(this).balance`. The contract has no `receive()` or `fallback()` function, so ETH can only enter via `updateRate()`. When a caller sends more ETH than the sum of fees (a realistic outcome since fees must be estimated off-chain), the surplus is permanently stranded. Any subsequent caller can invoke `updateRate()` with `msg.value = 0`; the EVM satisfies each `send{ value: estimatedFee }` call from the contract's stored balance, draining it entirely.

By contrast, `CrossChainRateProvider.updateRate()` avoids this by forwarding the full `msg.value` in a single call with no loop: [3](#0-2) 

The helper `estimateTotalFee()` exists but is a view function that is never called inside `updateRate()` to enforce a minimum: [4](#0-3) 

## Impact Explanation
ETH overpaid by a legitimate caller becomes user funds at-rest inside the contract. Any unprivileged external account can call `updateRate()` with zero ETH and have those stranded funds consumed to pay their LayerZero messaging fees. This is **direct theft of user funds at-rest**, mapping to **Critical** impact under the allowed scope ("Direct theft of any user funds, whether at-rest or in-motion, other than unclaimed yield").

## Likelihood Explanation
The precondition — ETH accumulating in the contract — is realistic and low-effort to trigger:
1. `updateRate()` is public and payable; any caller must estimate the total fee off-chain.
2. The on-chain `estimateTotalFee()` view and the actual fees at execution time can diverge (gas price changes, receiver count changes between estimate and call), making overpayment a natural occurrence.
3. Once any residue exists, the drain requires a single zero-ETH call from any EOA or contract, with no special privileges, no front-running dependency, and no victim interaction.
4. The attack is repeatable: every overpayment event creates a new drainable residue.

## Recommendation
1. Compute the total required fee before the loop and enforce a minimum: accumulate `estimatedFee` values in a `totalFee` variable, then `require(msg.value >= totalFee, "insufficient fee")` before sending any of them.
2. After the loop, refund any excess: `if (msg.value > totalFee) payable(msg.sender).transfer(msg.value - totalFee)`.
3. Alternatively, restrict `updateRate()` to a trusted role (`onlyOwner` or a dedicated keeper role) so only authorized callers can trigger cross-chain sends, eliminating the zero-cost drain vector.

## Proof of Concept
```
// Setup: MultiChainRateProvider deployed with 2 receivers,
//        each estimatedFee = 0.01 ETH at call time.

// Step 1 – Alice calls updateRate() with 0.025 ETH (overestimates by 0.005 ETH)
//   → loop iteration 0: estimatedFee = 0.01 ETH → sent to LZ endpoint
//   → loop iteration 1: estimatedFee = 0.01 ETH → sent to LZ endpoint
//   → 0.005 ETH remains in contract; no refund path exists
//   → address(MultiChainRateProvider).balance == 0.005 ETH

// Step 2 – Bob calls updateRate() with msg.value = 0
//   → loop iteration 0: estimatedFee = 0.01 ETH → paid from contract balance
//   → loop iteration 1: estimatedFee = 0.01 ETH → paid from contract balance
//   → call succeeds; Bob paid 0 ETH for a full cross-chain rate update
//   → Alice's 0.005 ETH is consumed; address(MultiChainRateProvider).balance == 0

// Foundry fork test outline:
// 1. Deploy a mock LZ endpoint that returns a fixed estimatedFee and accepts send().
// 2. Deploy a concrete MultiChainRateProvider with 2 receivers.

// 4. assertEq(address(provider).balance, 0.005 ether);
// 5. vm.prank(bob); provider.updateRate{value: 0}();
// 6. assertEq(address(provider).balance, 0); // Bob drained Alice's overpayment
```
