### Title
Missing `confirmations > 0` Validation in `verify_foreign_transaction` Allows Signing of Unconfirmed Bitcoin Transactions - (File: `crates/contract/src/lib.rs`, `crates/foreign-chain-inspector/src/bitcoin/inspector.rs`)

---

### Summary

The `verify_foreign_transaction` contract entry point accepts a `BitcoinRpcRequest` with `confirmations: BlockConfirmations(0)`. The contract performs no lower-bound check on this field. The node-side inspector then evaluates `0 <= transaction_block_confirmation`, which is trivially true for any `u64`, bypassing the confirmation-count guard entirely. The MPC network will sign a payload attesting to an unconfirmed (or zero-confirmation) Bitcoin transaction, enabling bridge double-spend conditions.

---

### Finding Description

`BlockConfirmations` is a `u64` newtype with a JSON-schema minimum of `0.0`:

```json
"BlockConfirmations": {
  "type": "integer",
  "format": "uint64",
  "minimum": 0.0
}
``` [1](#0-0) 

The `verify_foreign_transaction` contract method calls `check_request_preconditions` (which validates domain existence, gas, deposit, and TEE state) and then checks only whether the chain is whitelisted. It performs **no validation** that `confirmations > 0`: [2](#0-1) 

On the node side, `BitcoinInspector::extract()` enforces the confirmation threshold with:

```rust
let enough_block_confirmations =
    block_confirmations_threshold <= transaction_block_confirmation;
if !enough_block_confirmations {
    return Err(...NotEnoughBlockConfirmations...);
}
``` [3](#0-2) 

When `block_confirmations_threshold = BlockConfirmations(0)`, the expression `0 <= x` is always `true` for any `u64` value of `x`, including `0` (an unconfirmed mempool transaction). The guard is completely bypassed.

This is the direct analog of the reported pattern: the *existence* of the `confirmations` field is checked (it is a required field in the request), but its *value* is never validated to be `> 0` at the contract boundary.

---

### Impact Explanation

An attacker submits `verify_foreign_transaction` with a `BitcoinRpcRequest` containing `confirmations: 0` for a transaction that is still in the Bitcoin mempool (0 on-chain confirmations). Every MPC node's `BitcoinInspector` passes the confirmation check trivially. The threshold-of-nodes signs a `ForeignTxSignPayload` that encodes the unconfirmed transaction. The attacker delivers this signature to a bridge contract on NEAR to claim inbound funds. The underlying Bitcoin transaction can then be double-spent or reorganized away, leaving the bridge with a signed attestation for a transaction that never finalized.

This matches: **High — forged foreign-chain verification / light-client-style verification bypass that causes invalid bridge execution or double-spend conditions.**

---

### Likelihood Explanation

Any unprivileged NEAR account can call `verify_foreign_transaction` with an arbitrary `confirmations` value. No special role, key, or collusion is required. The attacker only needs to broadcast a Bitcoin transaction and immediately submit the request with `confirmations: 0` before the transaction confirms. The attack is straightforward and requires no cryptographic capability beyond a standard NEAR account.

---

### Recommendation

Add an explicit lower-bound check at the contract level before enqueuing the request:

```rust
// In verify_foreign_transaction, before enqueue_yield_request:
if let dtos::ForeignChainRpcRequest::Bitcoin(ref btc_req) = request.request {
    if btc_req.confirmations.0 == 0 {
        env::panic_str(
            &InvalidParameters::MalformedPayload {
                reason: "Bitcoin confirmations must be > 0".to_string(),
            }
            .to_string(),
        );
    }
}
```

Alternatively, enforce this at the `BlockConfirmations` type level by changing the JSON-schema `minimum` to `1.0` and adding a constructor that rejects zero. This mirrors the recommendation in the original report: check that the value is `> 0`, not merely that the field exists.

---

### Proof of Concept

1. Broadcast a Bitcoin transaction `T` (it enters the mempool with 0 confirmations).
2. Call `verify_foreign_transaction` on the MPC contract with:
   ```json
   {
     "request": {
       "domain_id": <foreign_tx_domain_id>,
       "payload_version": "V1",
       "request": {
         "Bitcoin": {
           "tx_id": "<T's txid>",
           "confirmations": 0,
           "extractors": ["BlockHash"]
         }
       }
     }
   }
   ```
   with `1 yoctoNEAR` attached.
3. The contract enqueues the request (no `confirmations > 0` check).
4. Each MPC node calls `BitcoinInspector::extract(tx_id, BlockConfirmations(0), ...)`. The check `0 <= rpc_response.confirmations` (where `rpc_response.confirmations = 0` for a mempool tx) evaluates to `true`. The inspector proceeds and returns extracted values.
5. The MPC network produces a threshold signature over the `ForeignTxSignPayload` encoding `T`.
6. The attacker delivers the signature to a bridge contract to claim NEAR-side funds.
7. The attacker double-spends `T` on Bitcoin (e.g., via RBF), invalidating the bridge deposit while retaining the NEAR-side payout. [4](#0-3) [5](#0-4)

    pub fn verify_foreign_transaction(&mut self, request: VerifyForeignTransactionRequestArgs) {
        log!(
            "verify_foreign_transaction: predecessor={:?}, request={:?}",
            env::predecessor_account_id(),
            request
        );

        self.check_request_preconditions(
            request.domain_id,
            DomainPurpose::ForeignTx,
            Gas::from_tgas(self.config.sign_call_gas_attachment_requirement_tera_gas),
            MINIMUM_SIGN_REQUEST_DEPOSIT,
        );

        let requested_chain = request.request.chain();
        let supported_chains = self.get_supported_foreign_chains();
        if !supported_chains.contains(&requested_chain) {
            env::panic_str(
                &InvalidParameters::ForeignChainNotSupported {
                    requested: requested_chain,
                }
                .to_string(),
            );
        }

        let callback_gas = Gas::from_tgas(
            self.config
                .return_signature_and_clean_state_on_success_call_tera_gas,
        );

        let request = args_into_verify_foreign_tx_request(request);
        let callback_args = serde_json::to_vec(&(&request,)).unwrap();
        self.enqueue_yield_request(

            callback_args,
            callback_gas,
            move |this, id| this.add_verify_foreign_tx_request(request, id),
        );
    }

    async fn extract(
        &self,
        transaction: BitcoinTransactionHash,
        block_confirmations_threshold: BlockConfirmations,
        extractors: Vec<BitcoinExtractor>,
    ) -> Result<Vec<BitcoinExtractedValue>, ForeignChainInspectionError> {
        let request_parameters = GetRawTransactionArgs {
            transaction_hash: TransportBitcoinTransactionHash::from(*transaction),
            verbose: VERBOSE_RESPONSE,
        };

        // TODO(#1978): add retry mechanism if the error from the request is transient
        let rpc_response: GetRawTransactionVerboseResponse = self
            .client
            .request(GET_RAW_TRANSACTION_METHOD, &request_parameters)
            .await?;

        let transaction_block_confirmation = rpc_response.confirmations.into();
        let enough_block_confirmations =
            block_confirmations_threshold <= transaction_block_confirmation;

        if !enough_block_confirmations {
            return Err(ForeignChainInspectionError::NotEnoughBlockConfirmations {
                expected: block_confirmations_threshold,
                got: transaction_block_confirmation,
            });
        }

        self.verify_block_is_canonical(rpc_response.blockhash)
            .await?;

        let extracted_values = extractors
            .iter()
            .map(|extractor| extractor.extract_value(&rpc_response))
            .collect();

        Ok(extracted_values)
    }
