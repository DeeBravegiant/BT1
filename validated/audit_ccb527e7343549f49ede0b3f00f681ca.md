Audit Report

## Title
Missing `msg.sender` Validation in `MerkleDistributor.claim()` Enables Forced Fee-Bearing Claims on Behalf of Any Account - (File: contracts/utils/MerkleDistributor/MerkleDistributor.sol)

## Summary
`MerkleDistributor.claim()` accepts an arbitrary `account` parameter without verifying that `msg.sender == account`. Any caller possessing a valid merkle proof for a victim can force a claim on the victim's behalf. When `feeInBPS > 0`, this permanently extracts up to 10% of the victim's claimable allocation as a fee to `protocolTreasury`, an amount the victim would have retained had they waited for the fee to be reduced or zeroed.

## Finding Description
`MerkleDistributor.claim()` at lines 97–147 of `contracts/utils/MerkleDistributor/MerkleDistributor.sol` performs three checks before executing a claim: (1) a non-zero merkle root, (2) a valid index range, and (3) an `isClaimed` guard. It never checks that `msg.sender == account`. The merkle leaf is constructed solely from `(index, account, cumulativeAmount)` — no caller identity is bound into the proof:

```solidity
bytes32 node = keccak256(abi.encodePacked(index, account, cumulativeAmount));
```

After proof verification, the fee is unconditionally deducted and the remainder sent to `account`:

```solidity
uint256 fee = (claimableAmount * feeInBPS) / 10_000;
uint256 amountToSend = claimableAmount - fee;
IERC20(token).safeTransfer(account, amountToSend);
IERC20(token).safeTransfer(protocolTreasury, fee);
```

`userClaims[account].cumulativeAmount` is then updated to `cumulativeAmount`, permanently consuming the claim slot at the current fee rate. The victim cannot re-claim the fee-bearing portion.

The sister contract `KernelMerkleDistributor._processClaim()` (lines 311–313) explicitly blocks this with:

```solidity
if (account != msg.sender) {
    revert Unauthorized();
}
```

This confirms the omission in `MerkleDistributor` is unintentional. The `feeInBPS` ceiling is `MAX_FEE_IN_BPS = 1000` (10%), so the maximum extractable loss per victim is 10% of their claimable allocation.

**Exploit path:**
1. Owner sets `feeInBPS` to any value above 0 (e.g., 1000 = 10%) — a normal operational state.
2. Victim holds unclaimed tokens and is deferring their claim waiting for the fee to drop.
3. Attacker obtains the victim's merkle proof from the publicly published proof file (standard for airdrop deployments).
4. Attacker calls `claim(index, victim, cumulativeAmount, victimProof)` from any EOA.
5. Contract verifies the proof, deducts the fee, sends `amountToSend` to victim, sends `fee` to `protocolTreasury`.
6. Victim's claim slot is permanently consumed at the high fee rate; the fee amount is irreversibly lost.

## Impact Explanation
**High — Theft of unclaimed yield.** The fee deducted from `claimableAmount` represents yield the victim was entitled to receive in full (had they claimed when `feeInBPS == 0`). By forcing the claim at a moment when `feeInBPS > 0`, the attacker causes the victim to permanently forfeit up to 10% of their claimable allocation. The loss is irreversible: `userClaims[account].cumulativeAmount` is updated to `cumulativeAmount`, so the fee-bearing portion cannot be reclaimed. This is a direct, concrete, permanent reduction in the victim's token holdings — squarely within the "Theft of unclaimed yield" impact class.

## Likelihood Explanation
**Medium.** `feeInBPS > 0` is a normal operational state explicitly supported by the contract. Merkle proofs for airdrop/distribution contracts are routinely published off-chain (JSON/IPFS) so users can self-claim; an attacker reads them without any privileged access. The attacker requires no special role, no capital, and no complex setup — only a public function call. The attack is repeatable across all victims whose proofs are published and who have not yet claimed. The only limiting factor is that `feeInBPS` must be non-zero at the time of the forced claim.

## Recommendation
Add the same caller restriction present in `KernelMerkleDistributor._processClaim()` to `MerkleDistributor.claim()`, immediately after the `isClaimed` guard:

```solidity
if (account != msg.sender) {
    revert Unauthorized();
}
```

Additionally, add `nonReentrant` to `claim()` as a defence-in-depth measure, consistent with `KernelMerkleDistributor`.

## Proof of Concept
```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.27;

// Foundry fork test — preconditions:
//   feeInBPS = 1000 (10%)
//   victim has 1000e18 tokens claimable at currentIndex
//   victimProof is read from the published proof file

function test_forcedClaimExtractsFee() public {
    uint256 victimBalanceBefore = token.balanceOf(victim);
    uint256 treasuryBalanceBefore = token.balanceOf(protocolTreasury);

    // Attacker (not victim) calls claim on victim's behalf
    vm.prank(attacker);
    merkleDistributor.claim(currentIndex, victim, 1000e18, victimProof);

    // Victim receives only 900e18 (10% fee deducted)
    assertEq(token.balanceOf(victim) - victimBalanceBefore, 900e18);
    // Treasury receives 100e18 (the stolen yield)
    assertEq(token.balanceOf(protocolTreasury) - treasuryBalanceBefore, 100e18);
    // Victim's claim slot is permanently consumed — cannot reclaim the 100e18
    assertTrue(merkleDistributor.isClaimed(currentIndex, victim));
}
```
