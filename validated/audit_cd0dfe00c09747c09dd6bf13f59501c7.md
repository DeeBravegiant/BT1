Audit Report

## Title
Zero wrsETH Minted on Sub-Threshold ETH Deposits Due to Floor-Division Truncation — (File: contracts/pools/RSETHPoolV3.sol)

## Summary
`viewSwapRsETHAmountAndFee` computes `rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate` using Solidity floor division. When `amountAfterFee * 1e18 < rsETHToETHrate`, the result truncates to zero. No post-computation zero-check exists before `wrsETH.mint(msg.sender, rsETHAmount)` is called, so a depositor sending a non-zero but sub-threshold ETH amount receives zero wrsETH while their ETH is retained by the pool.

## Finding Description
In `RSETHPoolV3.deposit` (ETH overload), the only input guard is `if (amount == 0) revert InvalidAmount()` at line 256. After that, `viewSwapRsETHAmountAndFee(amount)` at line 307 computes:

```solidity
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
```

When `amountAfterFee * 1e18 < rsETHToETHrate` (e.g., `amount = 1 wei`, `rsETHToETHrate = 1.05e18`), the result is `0`. The function then executes `wrsETH.mint(msg.sender, 0)` at line 262 — OpenZeppelin's `_mint` with amount zero is a no-op and does not revert. The `limitDailyMint` modifier (lines 96–125) also computes `rsETHAmount` via the same function; when it is zero, `dailyMintAmount += 0` passes silently and provides no protection. The depositor's ETH is absorbed into the pool's balance with no corresponding share minted.

The identical pattern is present in:
- `RSETHPoolV3ExternalBridge.deposit` / `viewSwapRsETHAmountAndFee` (lines 366–384, 418–427)
- `RSETHPoolV3WithNativeChainBridge.deposit` / `viewSwapRsETHAmountAndFee` (lines 282–301, 335–344)
- `RSETHPoolV2.deposit` / `viewSwapRsETHAmountAndFee` (lines 207–219, 225–234)
- Token-deposit overloads using `rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate` (line 334 in V3)

`LRTDepositPool._beforeDeposit` has a `minRSETHAmountExpected` guard at line 667, but callers passing `0` as the minimum bypass it silently.

## Impact Explanation
A depositor who sends a non-zero ETH amount smaller than `rsETHToETHrate / 1e18` wei receives zero wrsETH. Their ETH is retained by the pool with no share minted and no refund path. This constitutes the contract failing to deliver its promised return (wrsETH in exchange for ETH). The ETH is not destroyed — it accrues to the pool's bridgeable balance — so the protocol does not lose value, but the individual depositor does. This matches **Low — Contract fails to deliver promised returns, but doesn't lose value** from the allowed impact scope.

## Likelihood Explanation
The rsETH/ETH rate starts at `1e18` and grows monotonically. Once it exceeds `1e18` (which occurs naturally as restaking yield accrues), any deposit of exactly 1 wei produces `rsETHAmount = 0`. The condition is reachable by any unprivileged external caller at any time after the protocol has been live. The affected deposit amount per call is at most a few wei, so normal user deposits (which are orders of magnitude larger) are entirely unaffected. Likelihood is **Low** — the condition is always reachable but only affects dust-level deposits.

## Recommendation
Add a zero-check on the computed rsETH amount immediately after `viewSwapRsETHAmountAndFee` returns, in every `deposit` function across all pool variants:

```solidity
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
if (rsETHAmount == 0) revert InvalidAmount();
```

Apply the same fix to the token-deposit overloads. For `LRTDepositPool._beforeDeposit`, enforce `minRSETHAmountExpected > 0` or add an explicit `if (rsethAmountToMint == 0) revert` after line 665.

## Proof of Concept
1. Deploy `RSETHPoolV3` with `rsETHOracle` returning `1.05e18` (realistic post-yield rate) and `feeBps = 0`.
2. Call `deposit{value: 1}("")` from any EOA.
3. `viewSwapRsETHAmountAndFee(1)` computes: `fee = 0`, `amountAfterFee = 1`, `rsETHAmount = 1 * 1e18 / 1.05e18 = 0`.
4. `limitDailyMint` modifier: `dailyMintAmount + 0 > dailyMintLimit` → false; `dailyMintAmount += 0` → no change; modifier passes.
5. `wrsETH.mint(msg.sender, 0)` executes — zero tokens minted, transaction succeeds.
6. Caller's wrsETH balance: 0. Pool ETH balance: increased by 1 wei. No refund path exists.

Foundry fuzz test plan: fuzz `amount` in `[1, rsETHToETHrate / 1e18 - 1]` and assert `wrsETH.balanceOf(depositor) > 0` after each `deposit` call — the assertion will fail for all values in that range once the rate exceeds `1e18`. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
