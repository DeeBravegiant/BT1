Audit Report

## Title
Lack of Slippage Protection in `LRTWithdrawalManager.initiateWithdrawal()` Allows Users to Receive Fewer Assets Than Expected - (File: contracts/LRTWithdrawalManager.sol)

## Summary
`initiateWithdrawal()` accepts no `minOutputAmount` parameter, so users cannot set a floor on the assets they receive. The final payout is recalculated at `unlockQueue()` time using live oracle prices that may have moved adversely during the up-to-8-day delay, and no cancellation path exists once rsETH is transferred in.

## Finding Description
When `initiateWithdrawal()` is called, rsETH is immediately pulled from the user and `expectedAssetAmount` is computed from current oracle prices and stored: [1](#0-0) 

At `unlockQueue()` time, `_calculatePayoutAmount()` resolves the final disbursement as the **minimum** of the stored `expectedAssetAmount` and the freshly computed `currentReturn`: [2](#0-1) 

This is asymmetric: price improvement is capped at the original estimate, while price deterioration silently reduces the payout. The function signature provides no `minOutputAmount` guard: [3](#0-2) 

A grep for `cancel` in `LRTWithdrawalManager.sol` returns no matches, confirming there is no cancellation path. The withdrawal delay is set to 57,600 blocks (~8 days): [4](#0-3) 

By contrast, `LRTDepositPool._beforeDeposit()` reverts if the minted rsETH falls below `minRSETHAmountExpected`, providing the equivalent protection on the deposit side: [5](#0-4) 

The withdrawal path has no equivalent guard.

## Impact Explanation
**Low — Contract fails to deliver promised returns, but doesn't lose value.** A user who initiates a withdrawal expecting ~100 stETH may receive materially less if rsETH depreciates during the 8-day window (e.g., due to a slashing event or LST depeg). The user receives the current fair value of their rsETH at unlock time — not an inflated or stolen amount — but they had no ability to set a floor and cannot cancel once committed. This matches the allowed impact: "Contract fails to deliver promised returns, but doesn't lose value."

## Likelihood Explanation
**Medium.** The default `withdrawalDelayBlocks = 8 days / 12 seconds` creates a window long enough for meaningful rsETH/asset price divergence. The oracle prices (`lrtOracle.rsETHPrice()` and `lrtOracle.getAssetPrice(asset)`) reflect real EigenLayer restaking valuations that can shift due to slashing events, LST depegs, or protocol losses. Any unprivileged rsETH holder can trigger this path with no special preconditions. [4](#0-3) 

## Recommendation
Add a `minExpectedAssetAmount` parameter to `initiateWithdrawal()` and revert if `expectedAssetAmount < minExpectedAssetAmount`, mirroring the pattern already used in `LRTDepositPool._beforeDeposit()`. Optionally, implement a `cancelWithdrawal()` function that returns the locked rsETH to the user before `unlockQueue()` processes their request, allowing users to exit if unlock-time prices are unfavorable.

## Proof of Concept
1. rsETH/ETH oracle price = 1.05. Alice calls `initiateWithdrawal(stETH, 100e18, "")`. `expectedAssetAmount` is stored as ~100 stETH. rsETH is transferred to the contract.
2. Over the next 8 days a slashing event reduces rsETH/ETH to 0.90. Alice has no `cancelWithdrawal()` to call.
3. Operator calls `unlockQueue(stETH, ...)`. `_calculatePayoutAmount` computes `currentReturn = 100e18 * 0.90e18 / stETHPrice ≈ 85.7 stETH`. Since `85.7 < 100`, `request.expectedAssetAmount` is overwritten to `85.7 stETH`.
4. Alice calls `completeWithdrawal(stETH, "")` and receives ~85.7 stETH — ~14.3% less than she expected — with no recourse.

**Foundry fork test plan:** Fork mainnet, deploy with live oracle addresses, call `initiateWithdrawal` with a known rsETH amount, `vm.roll` forward by `withdrawalDelayBlocks`, mock `lrtOracle.rsETHPrice()` to return a lower value, call `unlockQueue`, then call `completeWithdrawal` and assert the received amount is less than the originally stored `expectedAssetAmount`.
