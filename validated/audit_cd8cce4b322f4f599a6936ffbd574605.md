Audit Report

## Title
ETH Deposit Limit Bypass Due to Missing `amount` in Limit Check - (File: contracts/LRTDepositPool.sol)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` omits the incoming `amount` from the ETH branch's cap comparison, checking only whether the running total already exceeds the limit rather than whether the new deposit would push it over. Any unprivileged depositor can therefore bypass the configured `depositLimitByAsset(ETH_TOKEN)` cap in a single transaction, while the identical ERC-20 branch correctly includes `amount`.

## Finding Description
In `contracts/LRTDepositPool.sol` at lines 676–682:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)); // amount MISSING
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // correct
}
```

The ETH branch at line 679 evaluates `totalAssetDeposits > limit`, which only returns `true` when the cap is already exceeded. It never evaluates whether the incoming `amount` would push the total over the cap. The ERC-20 branch at line 681 correctly evaluates `totalAssetDeposits + amount > limit`.

The exploit path is:
1. `depositETH{value: X}(...)` is called by any user — it is `public payable nonReentrant`.
2. `_beforeDeposit` calls `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, msg.value)` at line 661.
3. While `totalAssetDeposits <= depositLimitByAsset(ETH_TOKEN)`, the check returns `false` regardless of `msg.value`.
4. `_mintRsETH` mints rsETH for the full `msg.value`, pushing total ETH deposits arbitrarily above the cap.

No existing guard compensates for this omission. The `nonReentrant` and `whenNotPaused` modifiers are irrelevant to the accounting check.

## Impact Explanation
The ETH deposit cap is a protocol-level risk control bounding EigenLayer exposure. Bypassing it breaks the invariant `getTotalAssetDeposits(ETH) ≤ depositLimitByAsset(ETH)`, allowing rsETH to be minted against more ETH collateral than the protocol intends to accept. This maps to **Low: Contract fails to deliver promised returns** (the cap guarantee is violated) and potentially **Medium: Temporary freezing of funds** if the resulting over-exposure triggers downstream liquidity or withdrawal constraints.

## Likelihood Explanation
The entry point `depositETH` is public and permissionless. No special role, governance action, or precondition beyond holding ETH is required. The condition `totalAssetDeposits <= depositLimitByAsset(ETH_TOKEN)` is the normal operating state of the protocol. Any depositor can trigger this in a single transaction, and it is repeatable.

## Recommendation
Add `amount` to the ETH branch to mirror the ERC-20 branch:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

## Proof of Concept
1. Deploy/fork with `depositLimitByAsset(ETH_TOKEN) = 1000 ether` and seed `999 ether` already deposited.
2. Call `depositETH{value: 500 ether}(0, "")` from any EOA.
3. `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 500 ether)` evaluates `999 ether > 1000 ether` → `false` → no revert.
4. rsETH is minted for 500 ETH; total ETH deposits become 1499 ETH, 49.9% above the cap.
5. Confirm: an equivalent ERC-20 deposit evaluates `999 + 500 > 1000` → `true` → `MaximumDepositLimitReached` revert, demonstrating the asymmetry.

Foundry invariant test: assert `getTotalAssetDeposits(ETH_TOKEN) <= lrtConfig.depositLimitByAsset(ETH_TOKEN)` after arbitrary sequences of `depositETH` calls — this invariant will be violated with the current code.
