Audit Report

## Title
Missing Chainlink Staleness Validation Enables Oracle Rate Abuse - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary
`ChainlinkPriceOracle.getAssetPrice()` discards all staleness indicators from `latestRoundData()`, accepting prices that may be hours old. This stale price propagates into both the rsETH mint-amount calculation and the `rsETHPrice` update, enabling an unprivileged depositor to mint excess rsETH by exploiting the timing gap between a stale oracle and the true market price, diluting existing holders' yield.

## Finding Description
`ChainlinkPriceOracle.getAssetPrice()` destructures the Chainlink response discarding `updatedAt` and `answeredInRound`:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
``` [1](#0-0) 

No `block.timestamp - updatedAt > heartbeat` or `answeredInRound < roundId` check is performed. The stale price propagates through two paths:

**Path 1 — rsETHPrice update:** `LRTOracle.updateRSETHPrice()` is public and permissionless. It calls `_getTotalEthInProtocol()`, which calls `getAssetPrice(asset)` for every supported LST: [2](#0-1) [3](#0-2) 

The resulting TVL is divided by `rsethSupply` to set `rsETHPrice`: [4](#0-3) 

**Path 2 — rsETH mint amount:** `LRTDepositPool.getRsETHAmountToMint()` uses both the live oracle price and the stored `rsETHPrice`: [5](#0-4) 

**Correct attack direction (PoC in submission has the direction inverted):** The exploit is profitable when the oracle is stale at a price *above* the true market price (e.g., after a slashing event or market drop). Algebraically, for a deposit of amount `A` of stETH with stale price `P` and true price `P' < P`, with `E` ETH and `S` stETH in protocol:

- Stale rsethToMint = `A·P·R / (E + S·P)`
- True rsethToMint = `A·P'·R / (E + S·P')`

Since `P > P'`, the stale mint is larger — the depositor receives excess rsETH backed by assets worth less than credited, extracting value from existing holders.

**Existing guard is insufficient:** `_updateRsETHPrice()` contains a `pricePercentageLimit` guard that reverts if `newRsETHPrice` exceeds `highestRsethPrice` by more than the configured limit: [6](#0-5) 

This guard is bypassed entirely when `pricePercentageLimit == 0` (its default unset state), and even when set, it compares against the all-time high rather than the previous price, leaving intra-heartbeat drift fully exploitable. `ChainlinkOracleForRSETHPoolCollateral` checks `answeredInRound < roundID` but still omits the time-based guard: [7](#0-6) 

## Impact Explanation
**High — Theft of unclaimed yield.** When a supported LST's Chainlink feed is stale at a price above the true market value, `_getTotalEthInProtocol()` overstates TVL, `rsETHPrice` is set above the true rate, and a depositor minting rsETH receives more shares than their deposited assets warrant. The excess rsETH represents a direct transfer of accrued yield from existing holders to the attacker. No privileged access is required.

## Likelihood Explanation
Chainlink feeds have documented heartbeat intervals (e.g., 24 h for stETH/ETH on mainnet) and can lag during L2 sequencer downtime or extreme network congestion. Both `updateRSETHPrice()` and `depositAsset()` are reachable by any external caller. The full heartbeat window of price drift is exploitable on every call. The attack is repeatable as long as the oracle remains stale.

## Recommendation
Add a configurable `maxStaleness` per feed and revert if the price is stale or the round is incomplete:

```solidity
(uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
    priceFeed.latestRoundData();

require(answeredInRound >= roundId, "Stale round");
require(updatedAt != 0, "Incomplete round");
require(block.timestamp - updatedAt <= maxStaleness[asset], "Stale price");
```

Apply the equivalent time-based guard to `ChainlinkOracleForRSETHPoolCollateral.getRate()`.

## Proof of Concept
1. Deploy a fork of mainnet. Warp time forward 25 hours without triggering a Chainlink stETH/ETH feed update (simulating sequencer downtime or congestion). True stETH price has dropped 0.5% since the last oracle update.
2. Call `LRTOracle.updateRSETHPrice()` as an unprivileged EOA. `_getTotalEthInProtocol()` uses the stale (high) stETH price → `rsETHPrice` is set ~0.5% above the true rate.
3. Immediately call `LRTDepositPool.depositAsset(stETH, largeAmount, 0)`. `getRsETHAmountToMint` uses the stale (high) asset price in the numerator and the inflated `rsETHPrice` in the denominator; the net effect mints ~0.5% excess rsETH relative to the true exchange rate.
4. Warp time to allow the oracle to update. Call `updateRSETHPrice()` again; `rsETHPrice` corrects downward. The attacker's rsETH is now worth more per unit than the corrected rate warrants, extracting yield from existing holders.
5. A Foundry fork test can assert `rsethMinted > trueRsethMinted` by comparing the mint output against a reference calculation using the true price.
