Audit Report

## Title
Unbounded Nested Loops in `updateRSETHPrice()` and Deposit Path Can Cause Gas Exhaustion and Temporary Fund Freeze - (File: contracts/LRTOracle.sol, contracts/LRTDepositPool.sol, contracts/NodeDelegator.sol)

## Summary

`LRTOracle.updateRSETHPrice()` is publicly callable and internally executes nested loops over `supportedAssetList` and `nodeDelegatorQueue`, with each NDC iteration invoking `getAssetUnstaking()` which itself loops over all queued EigenLayer withdrawals and their strategies. Neither array has an on-chain hard cap. As the protocol grows, this function can exceed the block gas limit, permanently preventing price updates. Additionally, the deposit path independently traverses the same NDC loop per asset, meaning deposits can also be frozen once the NDC queue grows sufficiently large.

## Finding Description

`LRTOracle._getTotalEthInProtocol()` iterates over every supported asset in `supportedAssetList` and for each calls `ILRTDepositPool.getTotalAssetDeposits(asset)`:

```
LRTOracle.sol L336-348: for each asset → getTotalAssetDeposits(asset)
LRTDepositPool.sol L446-456: for each NDC → getAssetBalance(asset) + getAssetUnstaking(asset)
NodeDelegator.sol L409-427: for each queued withdrawal → for each strategy → external call
```

The total gas scales as `|supportedAssets| × |nodeDelegatorQueue| × |queuedWithdrawals| × |strategies_per_withdrawal|`, each step involving external SLOAD-heavy calls or cross-contract reads.

Neither bound is enforced on-chain:
- `_addNewSupportedAsset()` in `LRTConfig.sol` (L106-118) has no maximum length check on `supportedAssetList`.
- `updateMaxNodeDelegatorLimit()` in `LRTDepositPool.sol` (L290-297) only enforces `maxNodeDelegatorLimit_ >= nodeDelegatorQueue.length`, with no upper ceiling.

The deposit path is independently vulnerable: `depositAsset()` / `depositETH()` → `_beforeDeposit()` → `_checkIfDepositAmountExceedesCurrentLimit()` → `getTotalAssetDeposits(asset)` → `getAssetDistributionData(asset)` → the same NDC loop with `getAssetUnstaking()` per NDC. This path loops over all NDCs for a single asset, meaning it can OOG independently of the oracle update.

Existing guards are insufficient: `whenNotPaused` only prevents calls when the contract is paused; it does not bound loop iterations. The `maxNodeDelegatorLimit` check in `addNodeDelegatorContractToQueue()` enforces the current limit but that limit itself is unbounded.

## Impact Explanation

**Medium. Unbounded gas consumption / Temporary freezing of funds.**

Once the combined loop depth exceeds the block gas limit (~30M gas on Ethereum mainnet), `updateRSETHPrice()` reverts on every call, permanently staling `rsETHPrice`. More critically, the deposit path (`depositAsset` / `depositETH`) independently traverses the NDC loop per asset; if this sub-loop also exceeds the gas limit, all deposits revert, constituting a temporary (escalating to permanent) freeze of user funds. Withdrawals that depend on `rsETHPrice` will execute at a stale price, degrading protocol accounting. No single malicious action is required — the freeze emerges from legitimate protocol growth.

## Likelihood Explanation

The protocol is designed to scale: more LSTs are added as supported assets, more NDCs are deployed to handle restaking capacity, and each NDC accumulates queued EigenLayer withdrawals over time. `updateRSETHPrice()` is callable by any unprivileged external user (`public whenNotPaused`), so any user can trigger the revert once the gas threshold is crossed. No attacker capability beyond a standard EOA is required. The condition is irreversible without admin intervention (removing NDCs or assets), which itself requires all balances to be zero.

## Recommendation

1. Enforce an on-chain maximum in `_addNewSupportedAsset()` (e.g., `require(supportedAssetList.length < MAX_SUPPORTED_ASSETS)`).
2. Enforce an on-chain hard cap in `updateMaxNodeDelegatorLimit()` (e.g., `require(maxNodeDelegatorLimit_ <= MAX_NDC_COUNT)`).
3. Consider caching per-asset TVL snapshots updated lazily (push model) rather than recomputing the full sum on every price update or deposit.
4. Paginate `_getTotalEthInProtocol()` or split it into per-asset keeper-driven updates.
5. Add a gas-cost benchmark test that fails if the combined loop exceeds a safe fraction of the block gas limit.

## Proof of Concept

1. Deploy protocol; add 10 supported LST assets via `addNewSupportedAsset()`.
2. Admin calls `updateMaxNodeDelegatorLimit(20)` and adds 20 NDCs via `addNodeDelegatorContractToQueue()`.
3. Each NDC accumulates 10 queued EigenLayer withdrawals with 3 strategies each (achievable through normal protocol operation).
4. Any EOA calls `updateRSETHPrice()`.
5. `_getTotalEthInProtocol()` executes: 10 assets × 20 NDCs × (1 `getAssetBalance` + 1 `getAssetUnstaking` with 10×3 inner iterations + external `getQueuedWithdrawals` call per NDC per asset) = thousands of external calls.
6. Transaction reverts with out-of-gas; `rsETHPrice` is never updated.
7. Subsequently, any call to `depositAsset()` for any asset also traverses 20 NDCs × `getAssetUnstaking()` per NDC, reverting with out-of-gas and freezing all deposits.

Foundry test plan: fork mainnet, deploy protocol with the above configuration, call `updateRSETHPrice()` with a gas limit of 30M and assert revert; then call `depositAsset()` and assert revert, confirming fund freeze.
