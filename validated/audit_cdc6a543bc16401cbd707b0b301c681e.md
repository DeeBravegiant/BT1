Audit Report

## Title
Missing Slippage Protection in `deposit()` Functions Allows Users to Receive Fewer rsETH Than Expected - (File: contracts/pools/RSETHPoolV3.sol)

## Summary
All L2 pool `deposit()` functions lack a `minRSETHAmountExpected` parameter, leaving users with no on-chain protection against receiving fewer rsETH than observed when constructing the transaction. The rsETH amount is computed at execution time using the live oracle rate, which can increase between transaction submission and inclusion. The L1 `LRTDepositPool` already enforces this guard, confirming the protocol recognizes the need for it.

## Finding Description
In `RSETHPoolV3.deposit(string)` (lines 246–265), the rsETH output is computed entirely from the live oracle rate at execution time via `viewSwapRsETHAmountAndFee(amount)` (line 258), with no caller-supplied floor. The same pattern appears in `RSETHPoolV3.deposit(address,uint256,string)` (lines 271–293, line 286), `RSETHPoolV3ExternalBridge.deposit(string)` (lines 366–384, line 377), and `RSETHPoolNoWrapper.deposit(string)` (lines 231–244, line 237). The rate computation in `viewSwapRsETHAmountAndFee` (lines 299–308) divides by `getRate()`, the live cross-chain oracle rate. Because `rsETHToETHrate` only increases over time (staking yield accrual), any rate update applied between transaction submission and execution strictly reduces the rsETH minted to the user. By contrast, `LRTDepositPool.depositETH()` (lines 76–93) accepts `minRSETHAmountExpected` and enforces it in `_beforeDeposit` (lines 665–669) with `if (rsethAmountToMint < minRSETHAmountExpected) revert MinimumAmountToReceiveNotMet()`. No equivalent check exists in any L2 pool variant. The existing guards (`nonReentrant`, `whenNotPaused`, `limitDailyMint`, `amount == 0`) do not address rate-change exposure.

## Impact Explanation
**Low.** A user who calls `viewSwapRsETHAmountAndFee` to preview their output and then submits a deposit will silently receive fewer rsETH than observed if the oracle rate updates before inclusion. The deposited ETH is correctly accounted for and the protocol does not lose value, but the user's token position is worse than intended with no revert and no recourse. This matches the allowed impact: "Contract fails to deliver promised returns, but doesn't lose value."

## Likelihood Explanation
**Medium.** The L2 oracle rate is updated on a regular schedule by a cross-chain rate provider. Any deposit transaction pending in the mempool when a rate update lands executes at the new higher rate, yielding fewer rsETH. This is a routine operating condition affecting every L2 pool variant (`RSETHPoolV3`, `RSETHPoolV3ExternalBridge`, `RSETHPoolWithNativeChainBridge`, `RSETHPool`, `RSETHPoolNoWrapper`) on every chain where they are deployed. No special attacker capability is required; the rate update is a normal protocol event.

## Recommendation
Add a `minRSETHAmountExpected` parameter to all `deposit()` overloads in every L2 pool contract, mirroring the pattern in `LRTDepositPool._beforeDeposit()`. After computing `rsETHAmount` via `viewSwapRsETHAmountAndFee`, add:
```solidity
if (rsETHAmount < minRSETHAmountExpected) revert MinimumAmountToReceiveNotMet();
```
before minting or transferring rsETH. This gives callers an on-chain guarantee consistent with the L1 interface.

## Proof of Concept
1. Call `RSETHPoolV3.viewSwapRsETHAmountAndFee(1 ether)` — observe expected output `X` rsETH at current rate `R`.
2. Submit `RSETHPoolV3.deposit{value: 1 ether}(referralId)`.
3. Before inclusion, the cross-chain rate provider updates `rsETHToETHrate` from `R` to `R' > R`.
4. Transaction executes: `rsETHAmount = (1 ether - fee) * 1e18 / R'` — strictly less than `X`.
5. `wrsETH.mint(msg.sender, rsETHAmount)` succeeds with no revert; user receives fewer rsETH than previewed with no recourse.

Foundry fork test plan: fork the target L2 at a block just before a known rate-provider update transaction; call `viewSwapRsETHAmountAndFee` to record `X`; roll forward one block (applying the rate update); call `deposit{value: 1 ether}` and assert `rsETHAmount < X` with no revert.
