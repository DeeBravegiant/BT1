### Title
Unvalidated `builder` Field in `ProposalInit` Allows Malicious Proposer to Redirect Sequencer Fees and Corrupt `sequencer_address` in Executed Blocks - (`crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

---

### Summary

`ProposalInit.builder` is accepted from the network and passed directly as `sequencer_address` into the blockifier's `BlockInfo` without any validation in `is_proposal_init_valid`. Every other economically significant field in `ProposalInit` is checked against a locally-derived reference value; `builder` is the sole exception. A malicious-but-legitimately-selected proposer can set `builder` to an arbitrary address, causing all transaction fees for that block to be credited to the attacker-controlled address, the `get_sequencer_address` syscall to return the wrong value for every transaction in the block, and the committed block header to carry the wrong `sequencer_address`.

---

### Finding Description

`ProposalInit` carries a `builder` field defined as "Address of the one who builds/sequences the block": [1](#0-0) 

`convert_to_sn_api_block_info` maps this field directly to `sequencer_address` in the `BlockInfo` that is handed to the blockifier for execution: [2](#0-1) 

`is_proposal_init_valid` validates `height`, `l1_da_mode`, `l2_gas_price_fri`, all four L1 gas prices (within margin), `starknet_version`, `version_constant_commitment`, `fee_proposal_fri`, and `timestamp`. It does **not** validate `builder`: [3](#0-2) 

`ProposalInitValidation` — the struct that carries the locally-derived reference values — has no `builder` field, so there is no expected value to compare against: [4](#0-3) 

The `ProposalFinMismatch` guard at the end of `validate_proposal` does **not** protect against this: both the proposer and the validator derive `batcher_block_commitment` from the same `builder`-tainted `BlockInfo`, so they compute identical `partial_block_hash` values and the check passes: [5](#0-4) 

The `proposer` field is separately validated at the consensus-manager layer against the committee: [6](#0-5) 

`builder` is a distinct field and receives no analogous check anywhere in the pipeline.

---

### Impact Explanation

**Wrong state / wrong execution result (Critical).**

1. **Fee theft.** `sequencer_address` is the account that receives transaction fees. With an attacker-controlled `builder`, every fee paid in the block is credited to the attacker's address rather than the legitimate sequencer. This is a direct, per-block economic loss.

2. **Wrong `get_sequencer_address` syscall result.** Contracts that call `get_sequencer_address` during execution receive the attacker-supplied address. Any contract logic that branches on the sequencer address (e.g., fee-rebate contracts, access-control checks) will execute incorrectly, producing wrong state and wrong events.

3. **Wrong committed block header.** `sequencer_address` is part of the block header that feeds into the partial block hash and ultimately the on-chain state commitment. The committed block carries a permanently wrong `sequencer_address`.

---

### Likelihood Explanation

Any validator that is legitimately selected as proposer for a round can exploit this. The attack requires only setting one field to a different address — no cryptographic break, no race condition, no special network position. In a BFT network with `n` validators, each validator is selected as proposer roughly `1/n` of the time, so a single compromised or malicious validator can steal fees from every block it proposes. The attack is silent: no error is logged, no metric is incremented, and the proposal passes all existing checks.

---

### Recommendation

Add `builder` to `ProposalInitValidation` with the node's locally-configured builder address, and add a check in `is_proposal_init_valid`:

```rust
// In ProposalInitValidation:
pub builder: ContractAddress,

// In is_proposal_init_valid:
if init_proposed.builder != proposal_init_validation.builder {
    return Err(ValidateProposalError::InvalidProposalInit(
        init_proposed.clone(),
        proposal_init_validation.clone(),
        format!(
            "builder mismatch: expected={:?}, proposed={:?}",
            proposal_init_validation.builder, init_proposed.builder
        ),
    ));
}
```

The expected `builder` value should be derived from the node's own configuration (the same source used in `initiate_build`): [7](#0-6) 

---

### Proof of Concept

1. Malicious validator `V` is legitimately selected as proposer for height `H`, round `R`.
2. `V` constructs `ProposalInit` with `builder = attacker_address` (any address `V` controls).
3. `V` calls `propose_block` on its own batcher with `block_info.sequencer_address = attacker_address`.
4. The batcher executes all transactions; fees are credited to `attacker_address`; `get_sequencer_address` returns `attacker_address`.
5. The batcher returns `partial_block_hash` computed over the block with `sequencer_address = attacker_address`.
6. `V` sends `ProposalInit(builder=attacker_address)` + transaction batches + `ProposalFin(proposal_commitment = Poseidon(partial_block_hash, fee_proposal_fri))` to all validators.
7. Each validator calls `is_proposal_init_valid` — no `builder` check — passes.
8. Each validator calls `initiate_validation` → `convert_to_sn_api_block_info` → `sequencer_address = attacker_address`.
9. Each validator's batcher executes the same transactions with `sequencer_address = attacker_address`, producing the same `partial_block_hash`.
10. `batcher_block_commitment == received_fin.proposal_commitment` — `ProposalFinMismatch` check passes.
11. `valid_proposals.insert_proposal` stores the proposal; consensus votes proceed; the block is finalized and committed with `sequencer_address = attacker_address` and all fees in `attacker_address`. [8](#0-7) [9](#0-8)
