### Title
Hardcoded Compile-Time EVM `CHAIN_ID` in Wallet Contract Enables Cross-Chain Replay After Hard Fork - (File: runtime/near-wallet-contract/implementation/wallet-contract/src/internal.rs)

### Summary
The ETH implicit account wallet contract bakes the EVM chain ID (397 for mainnet, 398 for testnet) into the WASM binary as a compile-time constant. The chain-ID check in `validate_tx_relayer_data` compares incoming Ethereum transactions against this frozen constant. If NEAR ever hard-forks and assigns a new EVM chain ID, the deployed WASM continues to accept transactions signed under the old chain ID, enabling cross-chain replay attacks that corrupt account balances, nonces, and access keys.

### Finding Description
`CHAIN_ID` is declared as a compile-time constant by reading a source file at build time:

```rust
// internal.rs line 20
pub const CHAIN_ID: u64 = std::include!("CHAIN_ID");  // evaluates to 397 on mainnet
```

The value `397` is frozen into the WASM binary at the moment the contract is compiled. The sole runtime guard against cross-chain replay is:

```rust
// internal.rs lines 328-330
if tx.chain_id != Some(CHAIN_ID) {
    return Err(Error::Relayer(RelayerError::InvalidChainId));
}
```

Because `CHAIN_ID` is a constant embedded in the binary rather than a value read from the chain environment at call time, it cannot change without recompiling and redeploying the WASM. The wallet contract is deployed as a global contract whose hash is hardcoded in `runtime/near-wallet-contract/src/lib.rs`; upgrading it requires a coordinated protocol-level action. In the window between a hard fork that changes the EVM chain ID and a successful upgrade of the global contract, every ETH implicit account on the network is exposed.

The nonce field (`WalletContract::nonce`) provides same-chain replay protection but offers no cross-chain protection: immediately after a fork both chains share identical state, so the nonce on the new chain equals the nonce on the old chain, and a transaction signed pre-fork with the old chain ID satisfies both the nonce check and the (now-stale) chain-ID check on the new chain.

### Impact Explanation
After a hard fork that changes the EVM chain ID, any signed Ethereum transaction that was valid on the pre-fork chain can be submitted to `rlp_execute` on the post-fork chain by any unprivileged caller. The wallet contract will accept it because `tx.chain_id == 397 == CHAIN_ID`. Depending on the transaction payload, the attacker can:

- **Drain balances**: replay a `Transfer` action to move NEAR out of the victim's ETH implicit account.
- **Hijack access keys**: replay an `AddKey` action to install an attacker-controlled key, or a `DeleteKey` action to remove the owner's key.
- **Execute arbitrary function calls**: replay a `FunctionCall` action against any contract the victim previously authorized.

The corrupted protocol values are the victim's account balance (trie state), the account's nonce stored in `WalletContract::nonce` (contract storage), and the account's access-key set (trie state).

### Likelihood Explanation
The precondition is a hard fork that changes the EVM chain ID registered for NEAR mainnet (currently 397 on chainlist.org). Hard forks are uncommon but not unprecedented in blockchain ecosystems; the Golom report's judge explicitly noted "hard forks can and do happen." The NEAR ecosystem has already assigned distinct EVM chain IDs for mainnet (397) and testnet (398), demonstrating that the mapping is not immutable. The exploit itself requires no special privileges: any account can call `rlp_execute` on any ETH implicit account.

### Recommendation
Replace the compile-time constant with a runtime lookup. The NEAR WASM host exposes a `chain_id` function (available via `near_sdk::env`) that returns the current NEAR chain ID string (e.g., `"mainnet"`). The wallet contract should map this string to the expected EVM chain ID at call time rather than at compile time:

```rust
fn expected_evm_chain_id() -> u64 {
    match env::chain_id().as_str() {
        "mainnet" => 397,
        "testnet" => 398,
        _ => /* localnet/sandbox fallback */ 398,
    }
}
```

This ensures that if the EVM chain ID is ever updated via a protocol upgrade, the mapping can be updated in the same upgrade, and no stale constant remains in the binary.

### Proof of Concept

1. User signs an Ethereum transaction on NEAR mainnet (pre-fork): `chain_id=397`, `nonce=5`, `to=<target>`, action = transfer 10 NEAR to attacker.
2. NEAR hard-forks; the new chain assigns EVM chain ID 399. Both chains share identical state at the fork point, so the victim's wallet contract nonce is still 5 on the new chain.
3. Attacker submits the pre-fork signed transaction to `rlp_execute` on the new chain.
4. `validate_tx_relayer_data` evaluates `tx.chain_id == Some(397) == Some(CHAIN_ID)` — passes.
5. Nonce check: `5 == 5` — passes.
6. The transfer executes; 10 NEAR leaves the victim's account on the new chain.

#[near_bindgen]
impl WalletContract {
    /// Return the nonce value currently stored in the contract.
    /// Following the Ethereum protocol, only transactions with nonce equal
    /// to the current value will be accepted.
    /// Additionally, the Ethereum protocol requires the nonce of an account increment
    /// by 1 each time a transaction with the correct nonce and a valid signature
    /// is submitted (even if that transaction eventually fails). In this way, each
    /// nonce value can only be used once (hence the name "nonce") and thus transaction
    /// replay is prevented.
    pub fn get_nonce(&self) -> U64 {
        U64(self.nonce)
    }

    /// This is the main entry point into this contract. It accepts an RLP-encoded
    /// Ethereum transaction signed by the private key associated with the address
    /// for the account where this contract is deployed. RLP is a binary format,
    /// so the argument is actually passed as a base64-encoded string.
    /// The Ethereum transaction represents a Near action the owner of the address
    /// wants to perform. This method decodes that action from the Ethereum transaction
    /// and crates a promise to perform that action.
    /// Actions on Near are sent to a particular account ID where they are supposed to
    /// be executed (for example, a `FunctionCall` action is sent to the contract
    /// which will execute the method). In the Ethereum transaction only the address
    /// of the target can be specified because it does not have a notion of named accounts
    /// like Near has. The `target` field of this method gives the actual account ID
    /// that the action will be sent to. The `target` must itself be an eth-implicit
    /// account and match the `to` address of the Ethereum transaction; or `target`
    /// must hash to the address given in the `to` field of the Ethereum transaction.
    /// The output of this function is an `ExecuteResponse` which gives the output
    /// of the Near action or an error message if there was a problem during the execution.
    #[payable]
    pub fn rlp_execute(
        &mut self,
        target: AccountId,
        tx_bytes_b64: String,
    ) -> PromiseOrValue<ExecuteResponse> {
        // To ensure user actions are executed in the desired order,
        // having multiple transactions in flight at the same time is
        // not allowed.
        if self.has_in_flight_tx {
            return PromiseOrValue::Value(ExecuteResponse {
                success: false,
                success_value: None,
                error: Some(
                    "Error: transaction already in progress, please try again later.".into(),
                ),
            });
        }
        let current_account_id = env::current_account_id();
        let predecessor_account_id = env::predecessor_account_id();
        let result = inner_rlp_execute(
            current_account_id.clone(),
            predecessor_account_id,
            target,
            tx_bytes_b64,
            &mut self.nonce,
        );

        match result {
            Ok(promise) => {
                self.has_in_flight_tx = true;
                PromiseOrValue::Promise(promise)
            }
            Err(Error::Relayer(_)) if env::signer_account_id() == current_account_id => {
                let promise = create_ban_relayer_promise(current_account_id);
                self.has_in_flight_tx = true;
                PromiseOrValue::Promise(promise)
            }
            Err(e) => PromiseOrValue::Value(e.into()),
        }
    }
