### Title
Presignature Reuse via Host-Level Disk Rollback Enables ECDSA Key Recovery — (File: `crates/node/src/providers/ecdsa/sign.rs`, `docs/securing-mpc-with-tee-design-doc.md`)

---

### Summary

The NEAR MPC node persists presignatures and Beaver triples to RocksDB and deletes them upon consumption. No rollback-proof record of consumption exists. A host-level adversary — explicitly within the TEE threat model — can replace the node's disk with a prior snapshot, restoring a previously consumed presignature. Reusing the same presignature nonce across two distinct signing sessions allows standard ECDSA nonce-reuse key recovery, yielding the full MPC secret key share and enabling unauthorized signature issuance for any foreign-chain transaction.

---

### Finding Description

**Vulnerability class:** Replay / idempotency failure — successful consumption of a one-time cryptographic asset is not permanently recorded in a tamper-proof store, so the operation can be replayed.

**Root cause — presignature consumption without rollback protection:**

The signing pipeline is:

1. The leader node calls `presignature_store.take_owned()`, which **deletes** the presignature from RocksDB.
2. The leader broadcasts the presignature ID to followers, who call `take_unowned(id)` to retrieve their shares.
3. The signing protocol runs and the resulting signature is submitted on-chain via `respond`. [1](#0-0) 

The deletion in step 1 is a local RocksDB write. The disk is encrypted with a key derived from CVM measurements, but the host OS can take a snapshot of the encrypted disk at any time and restore it later — the CVM measurements do not change between snapshots, so the same sealing key decrypts both the live disk and the snapshot.

The project's own threat model explicitly acknowledges this as an **open, unmitigated risk**: [2](#0-1) [3](#0-2) 

The cryptographic documentation reinforces the severity: reusing a presignature nonce is explicitly prohibited and leads to key recovery: [4](#0-3) [5](#0-4) 

**Attack path (no threshold collusion required):**

1. The host OS snapshots Node A's encrypted disk at time T₀ (presignature P is present in RocksDB).
2. A user submits a `sign` request; Node A consumes P (deletes it from RocksDB) and produces signature σ₁ over message h₁.
3. The host OS restores Node A's disk to the T₀ snapshot — P is back in RocksDB.
4. A second `sign` request arrives (any user, any message h₂ ≠ h₁); Node A consumes P again and produces σ₂ over h₂.
5. The attacker now holds (σ₁, h₁) and (σ₂, h₂) sharing the same nonce k (from P). Standard ECDSA nonce-reuse recovery yields the secret key share x:

   `x = (s₁·h₂ − s₂·h₁) / (r·(s₂ − s₁)) mod q`

6. With x the attacker can forge threshold signatures for arbitrary foreign-chain transactions without any further MPC participation.

The asset-generation documentation confirms that each presignature is intended to be consumed **exactly once** and that the owner's queue management is the sole enforcement mechanism: [6](#0-5) [7](#0-6) 

There is no monotonic counter, on-chain nonce, or TPM-sealed usage log that would survive a disk rollback and prevent the restored presignature from being consumed a second time.

---

### Impact Explanation

**Severity: Critical.**

Successful exploitation yields the ECDSA secret key share of the compromised node. Because the MPC scheme requires only a threshold of shares to reconstruct the signing key, an attacker who compromises a threshold-many nodes via this technique (or combines this with other share-extraction methods) can forge signatures for any foreign-chain transaction — Bitcoin, Ethereum, Solana, etc. — controlled by the MPC network. This constitutes:

- Unauthorized access to MPC key shares and signing capability.
- Bypass of threshold-signature requirements (key recovery eliminates the need for live MPC participation).
- Potential theft of all funds held at MPC-controlled foreign-chain addresses. [8](#0-7) 

---

### Likelihood Explanation

**Likelihood: Medium.**

The attacker must be the host OS of at least one MPC node — a role explicitly within the TEE threat model ("A privileged attacker with host-level (root) access can interrupt execution … and control scheduling"). The host does not need to break TDX hardware isolation; it only needs filesystem-level access to snapshot and restore the encrypted disk, which is a standard OS capability. No timing precision is required: the host can snapshot the disk at any idle moment, then restore it after observing that a signing session completed (detectable from on-chain events). The attack requires no collusion among MPC participants and no knowledge of cryptographic secrets. [9](#0-8) 

---

### Recommendation

1. **Monotonic usage counter:** Before consuming a presignature, atomically increment a TPM-backed or TEE-sealed monotonic counter and record `(presignature_id, counter_value)` in a log that cannot be rolled back. On startup, verify the counter has not regressed; refuse to consume any presignature whose counter value is ≤ the last recorded value.

2. **On-chain consumption acknowledgment:** After consuming a presignature, submit a lightweight on-chain transaction recording the presignature ID. On restart, cross-check local RocksDB against on-chain records and discard any presignature that was already acknowledged on-chain.

3. **Epoch-scoped presignature IDs with sequence numbers:** Assign globally monotone sequence numbers to presignatures within an epoch and enforce that the node never processes a sequence number it has already processed, using a sealed, append-only log.

4. **Short-lived presignatures:** Reduce the window of exposure by limiting the lifetime of stored presignatures and generating them on-demand rather than buffering large pools. [7](#0-6)

### Proof of Concept

**Setup:** Two signing sessions using the same presignature P = (R, k, σ) where R = k·G.

**Session 1 (before rollback):**
- Message hash h₁, signature (r, s₁) where r = x-coord(R), s₁ = k⁻¹(h₁ + r·x) mod q.

**Session 2 (after disk rollback restores P):**
- Message hash h₂ ≠ h₁, signature (r, s₂) where s₂ = k⁻¹(h₂ + r·x) mod q.

**Key recovery:**
```
s₁ - s₂ = k⁻¹(h₁ - h₂) mod q
k = (h₁ - h₂)(s₁ - s₂)⁻¹ mod q
x = (s₁·k - h₁)·r⁻¹ mod q
```

The recovered x is the node's secret key share. This is the standard ECDSA nonce-reuse attack, confirmed applicable by the project's own security documentation: [10](#0-9)
