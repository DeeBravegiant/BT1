### Title
Proposer Deposits Permanently Locked in Contract After `propose_update` Clears All Entries — (`File: crates/contract/src/update.rs`, `crates/contract/src/lib.rs`)

---

### Summary

`propose_update` collects a storage-staking deposit from each proposer proportional to the size of their proposed update. When any update reaches threshold and `do_update` executes, it clears **all** pending entries and votes — freeing the storage — but never refunds the deposits to any proposer. Because `UpdateEntry` does not record the proposer's account ID or the deposit amount, there is no recovery path. Every proposer's deposit is permanently locked in the contract balance.

---

### Finding Description

`propose_update` in `crates/contract/src/lib.rs` computes a required deposit based on storage cost and keeps exactly that amount:

```rust
let attached = env::attached_deposit();
let required = ProposedUpdates::required_deposit(&update);
// excess refunded, but `required` is kept permanently
``` [1](#0-0) 

The `UpdateEntry` struct stored in `ProposedUpdates::entries` records only the update payload and `bytes_used` — **not** the proposer's account ID or the deposit amount:

```rust
pub(crate) struct UpdateEntry {
    pub(super) update: Update,
    pub(super) bytes_used: u128,
}
``` [2](#0-1) 

When `do_update` is called (triggered by `vote_update` reaching threshold), it removes the winning entry and then calls `self.entries.clear()` and `self.vote_by_participant.clear()`, wiping every other pending proposal:

```rust
pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
    let entry = self.entries.remove(id)?;
    // Clear all entries as they might be no longer valid
    self.entries.clear();
    self.vote_by_participant.clear();
    ...
}
``` [3](#0-2) 

Freeing the storage returns the staked NEAR to the **contract's** balance (NEAR runtime behavior), but since no proposer identity or deposit amount is recorded, the contract cannot and does not return any deposit to any proposer — winning or losing. There is no `withdraw_update_deposit` or equivalent function anywhere in the contract.

---

### Impact Explanation

**Medium.** Every successful `vote_update` execution permanently locks the deposits of all concurrent proposers inside the contract with no recovery path. For a contract-code update, `bytes_used` includes the full WASM binary plus a 128-participant vote overhead:

```rust
fn bytes_used(update: &Update) -> u128 {
    let mut bytes_used = std::mem::size_of::<UpdateEntry>() as u128;
    bytes_used += 128 * std::mem::size_of::<AccountId>() as u128;
    match update {
        Update::Contract(code) => { bytes_used += code.len() as u128; }
        ...
    }
    bytes_used
}
``` [4](#0-3) 

A typical MPC contract WASM binary is hundreds of kilobytes. At NEAR's storage cost (~10 NEAR/MB), each proposer loses several NEAR tokens per proposal. With multiple concurrent proposals (the codebase explicitly supports and tests this), multiple participants each lose their full deposit on every upgrade cycle. This breaks the production accounting invariant that a deposit paid for storage is returned when that storage is freed.

---

### Likelihood Explanation

**High.** This occurs in normal operation on every successful contract or config upgrade. The sandbox tests confirm multiple simultaneous proposals are expected and that `do_update` clears all of them: [5](#0-4) 

No attacker action is required — the loss is triggered by the honest threshold-vote flow itself.

---

### Recommendation

1. Extend `UpdateEntry` to record the proposer's `AccountId` and the `deposit` amount paid.
2. In `do_update`, before calling `self.entries.clear()`, iterate over all remaining entries and schedule a `Promise::new(entry.proposer).transfer(entry.deposit)` for each one (including the winning entry's proposer if the deposit should be returned on execution).
3. Alternatively, track deposits in a separate `LookupMap<UpdateId, (AccountId, NearToken)>` and drain it alongside `entries.clear()`.

---

### Proof of Concept

1. Participant A calls `propose_update` with a 500 KB WASM binary, attaching ~5 NEAR (required deposit). Entry stored, deposit kept.
2. Participant B calls `propose_update` with a different WASM binary, attaching ~5 NEAR. Entry stored, deposit kept.
3. Threshold participants vote for B's proposal. `vote_update` calls `do_update(&B_id, ...)`.
4. `do_update` removes B's entry, then calls `self.entries.clear()` (removes A's entry) and `self.vote_by_participant.clear()`.
5. Both A's ~5 NEAR and B's ~5 NEAR remain in the contract balance. Neither proposer receives a refund. No function exists to retrieve them. [6](#0-5) [7](#0-6)

    #[handle_result]
    pub fn propose_update(
        &mut self,
        #[serializer(borsh)] args: ProposeUpdateArgs,
    ) -> Result<UpdateId, Error> {
        // Only voters can propose updates:
        let proposer = self.voter_or_panic();
        let update: Update = args.try_into()?;

        let attached = env::attached_deposit();
        let required = ProposedUpdates::required_deposit(&update);
        if attached < required {
            return Err(InvalidParameters::InsufficientDeposit {
                attached: attached.as_yoctonear(),
                required: required.as_yoctonear(),
            }
            .into());
        }

        let id = self.proposed_updates.propose(update);

        log!(
            "propose_update: signer={}, id={:?}",
            env::signer_account_id(),
            id,
        );

        // Refund the difference if the proposer attached more than required.
        if let Some(diff) = attached.checked_sub(required)
            && diff > NearToken::from_yoctonear(0)
        {
            Promise::new(proposer).transfer(diff).detach();
        }

        Ok(id)
    }

    pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
        let entry = self.entries.remove(id)?;

        // Clear all entries as they might be no longer valid
        self.entries.clear();
        self.vote_by_participant.clear();

        let mut promise = Promise::new(env::current_account_id());
        match entry.update {
            Update::Contract(code) => {
                // deploy contract then do a `migrate` call to migrate state.
                promise = promise.deploy_contract(code).function_call(
                    method_names::MIGRATE,
                    Vec::new(),
                    NearToken::from_near(0),
                    gas,
                );
            }
            Update::Config(config) => {
                // If we vote for a new config, we should use
                // the value `contract_upgrade_deposit_tera_gas` from the config
                // as the new gas value
                let new_config_gas_value = Gas::from_tgas(config.contract_upgrade_deposit_tera_gas);
                promise = promise.function_call(
                    method_names::UPDATE_CONFIG,
                    serde_json::to_vec(&(&config,)).unwrap(),
                    NearToken::from_near(0),
                    new_config_gas_value,
                );
            }
        }
        Some(promise)
    }

fn bytes_used(update: &Update) -> u128 {
    let mut bytes_used = std::mem::size_of::<UpdateEntry>() as u128;

    // Assume a high max of 128 participant votes per update entry.
    bytes_used += 128 * std::mem::size_of::<AccountId>() as u128;

    match update {
        Update::Contract(code) => {
            bytes_used += code.len() as u128;
        }
        Update::Config(config) => {
            let bytes = serde_json::to_vec(&config).unwrap();
            bytes_used += bytes.len() as u128;
        }
    }

    bytes_used
}
