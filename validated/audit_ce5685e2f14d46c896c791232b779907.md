Audit Report

## Title
ETH Deposit Limit Bypass via Missing `amount` in Limit Check — (File: `contracts/LRTDepositPool.sol`)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` omits the incoming `amount` from the ETH branch of its cap check, evaluating only whether the pre-call total already exceeds the limit rather than whether the new deposit would push it over. Any unprivileged depositor can therefore send an arbitrarily large ETH amount in a single transaction and bypass the admin-configured deposit cap entirely, as long as the running total has not yet crossed it.

## Finding Description
In `_checkIfDepositAmountExceedesCurrentLimit` (L676–682), the ETH branch reads:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)); // amount absent
}
return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // amount present
``` [1](#0-0) 

The `amount` parameter (which is `msg.value` forwarded from `depositETH`) is silently discarded for ETH. The predicate becomes "has the limit *already* been exceeded?" rather than "would *this* deposit exceed the limit?"

The call chain is: `depositETH` (L87) → `_beforeDeposit` (L661) → `_checkIfDepositAmountExceedesCurrentLimit`. The guard returns `false` whenever `totalAssetDeposits ≤ limit`, regardless of how large `msg.value` is, so `_beforeDeposit` does not revert and the full deposit is accepted. [2](#0-1) [3](#0-2) 

The ERC20 path (`depositAsset`, L111) correctly includes `depositAmount` in the comparison and is unaffected. [4](#0-3) 

## Impact Explanation
The deposit limit is a risk-management parameter set by the admin to cap protocol exposure to EigenLayer strategy capacity. Because the ETH check is broken, a single depositor can mint rsETH against an arbitrarily large ETH deposit in one transaction. Excess ETH that cannot be deployed to EigenLayer strategies sits idle in the deposit pool earning no yield, diluting returns for all existing rsETH holders.

**Impact: Low — Contract fails to deliver promised returns, but does not lose value.**

## Likelihood Explanation
`depositETH` is `external payable` with no role restriction. No flash loan, multi-step setup, or privileged access is required. The condition is trivially met whenever the protocol is below its cap, which is the normal operating state. Any depositor with sufficient ETH can trigger this in a single transaction.

## Recommendation
Include `amount` in the ETH branch, consistent with the ERC20 branch:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
``` [5](#0-4) 

## Proof of Concept
1. Admin sets `depositLimitByAsset[ETH_TOKEN] = 100 ether`.
2. `getTotalAssetDeposits(ETH_TOKEN)` returns `0`.
3. Attacker calls `depositETH{value: 10_000 ether}(0, "")`.
4. `_checkIfDepositAmountExceedesCurrentLimit` evaluates `0 > 100 ether` → `false` → no revert.
5. `_mintRsETH` mints rsETH for the full 10,000 ETH; protocol now holds 100× the intended cap.
6. All subsequent ETH deposits are blocked (total now exceeds limit), but the excess is already accepted and rsETH minted.

**Foundry test plan:**
```solidity
function test_ethDepositLimitBypass() public {
    vm.prank(admin);
    lrtConfig.setDepositLimitByAsset(LRTConstants.ETH_TOKEN, 100 ether);

    address attacker = makeAddr("attacker");
    vm.deal(attacker, 10_000 ether);

    vm.prank(attacker);
    depositPool.depositETH{value: 10_000 ether}(0, "");

    assertGt(depositPool.getTotalAssetDeposits(LRTConstants.ETH_TOKEN), 100 ether);
}
```
