### Title
Caller-Controlled `EvmFinality::Latest` Accepted Without Restriction Enables Reorg-Based Double-Spend Attestation — (File: `crates/foreign-chain-inspector/src/evm/inspector.rs`, `crates/contract/src/lib.rs`)

---

### Summary

The `verify_foreign_transaction` contract method and the EVM inspector accept `EvmFinality::Latest` (a spot/unfinalized block tip) as a valid finality level without any minimum-finality enforcement. This is the direct analog of the reported `slot0()` spot-price vulnerability: just as `slot0()` returns a manipulatable instantaneous price, `FinalityTag::Latest` returns the current chain tip, which is subject to reorg. An unprivileged caller can obtain a threshold-signed MPC attestation for a transaction that is subsequently reorged away, enabling a double-spend against any bridge contract that trusts the MPC network as its finality oracle.

---

### Finding Description

`EvmFinality` is a caller-controlled enum with three variants: `Latest`, `Safe`, and `Finalized`. [1](#0-0) 

The contract's `verify_foreign_transaction` entry point performs no validation of the `finality` field — it only checks that the requested chain is in the supported set and enqueues the request: [2](#0-1) 

The node's `execute_foreign_chain_request` passes the caller-supplied finality directly to the EVM inspector without any floor check: [3](#0-2) 

Inside `EvmInspector::verify_finality_level`, `EthereumFinality::Latest` maps to `FinalityTag::Latest`, which resolves to the current chain tip — a value that can be reorged: [4](#0-3) 

The signed payload (`ForeignTxSignPayloadV1`) embeds the full `ForeignChainRpcRequest`, including the caller-supplied `finality` field: [5](#0-4) 

Because the `finality` field is part of the signed message, the MPC network produces a cryptographically valid threshold signature over `(request{finality=Latest}, observed_values)`. The signature is not "forged" — it is a genuine threshold attestation — but it attests to a state that may be invalidated by a reorg.

---

### Impact Explanation

**High — forged foreign-chain verification enabling double-spend / invalid bridge execution.**

The primary stated use case of `verify_foreign_transaction` is the Omnibridge inbound flow: "Chain Signatures are required to attest that a foreign transaction finalized successfully." [6](#0-5) 

A bridge contract that relies on the MPC network as its finality oracle — and does not independently re-validate the `finality` field embedded in the signed payload — will accept a `Latest`-finality attestation as proof of settlement. If the underlying EVM transaction is subsequently reorged, the attacker holds a valid MPC threshold signature attesting to a transaction that never finalized, and can use it to claim bridge funds on NEAR while the foreign-chain funds are returned by the reorg.

---

### Likelihood Explanation

**Medium.** The attack requires:
1. An EVM chain with non-negligible reorg probability (Polygon, BNB, Abstract, Base, Arbitrum, HyperEVM — all supported chains).
2. The attacker to submit `verify_foreign_transaction` with `finality: Latest` immediately after broadcasting the foreign-chain transaction.
3. A reorg to occur before the bridge contract's settlement window closes.

Polygon in particular has a documented history of deep reorgs (51+ blocks in 2023). The attacker controls the timing of step 1 and 2; step 3 is probabilistic but achievable, especially if the attacker can influence block production on the foreign chain.

---

### Recommendation

Enforce a minimum finality level at the contract layer. Reject `EvmFinality::Latest` (and optionally `EvmFinality::Safe` for high-value bridges) in `verify_foreign_transaction`:

```rust
// In verify_foreign_transaction, before enqueue_yield_request:
if let ForeignChainRpcRequest::Evm(ref req) = request.request {
    if req.finality == EvmFinality::Latest {
        env::panic_str("EvmFinality::Latest is not permitted; use Safe or Finalized");
    }
}
```

Alternatively, enforce the minimum at the node layer in `execute_foreign_chain_request` so that even if the contract is not upgraded, nodes refuse to sign `Latest`-finality observations. The contract-layer check is preferable because it provides an on-chain, auditable guarantee.

---

### Proof of Concept

1. Attacker broadcasts a deposit transaction `T` on Polygon (a supported EVM chain).
2. `T` appears in the latest block `B` (not yet safe/finalized).
3. Attacker immediately calls `verify_foreign_transaction` with:
   ```json
   {
     "request": { "Polygon": { "tx_id": "<T>", "finality": "Latest", "extractors": ["BlockHash"] } },
     "domain_id": <ForeignTx domain>,
     "payload_version": "V1"
   }
   ```
4. MPC nodes call `eth_getTransactionReceipt` → receipt found in block `B`. `verify_finality_level` with `FinalityTag::Latest` passes because `B` is the current tip. `verify_block_is_canonical` passes because `B` is canonical at this instant. [7](#0-6) 
5. Threshold of nodes independently verify and sign. The MPC contract returns a valid threshold signature over `SHA-256(borsh(ForeignTxSignPayload{request{finality=Latest}, values=[BlockHash(B)]}))`.
6. Attacker submits the signature to the NEAR bridge contract, claiming the deposit. Bridge contract verifies the MPC signature (valid) and releases NEAR-side funds.
7. A reorg on Polygon removes block `B`. Transaction `T` is evicted from the canonical chain. Attacker's Polygon funds are returned (or the attacker never actually sent real value if they controlled the block producer).
8. Attacker has NEAR-side funds and the Polygon-side funds, constituting a double-spend enabled by the MPC network's acceptance of a spot/unfinalized block attestation. [8](#0-7) [9](#0-8)

    async fn verify_finality_level(
        &self,
        receipt_block_number: U64,
        finality: EthereumFinality,
    ) -> Result<(), ForeignChainInspectionError> {
        let finality_tag = match finality {
            EthereumFinality::Finalized => FinalityTag::Finalized,
            EthereumFinality::Safe => FinalityTag::Safe,
            EthereumFinality::Latest => FinalityTag::Latest,
        };
        let args = GetBlockByNumberArgs::new(
            BlockNumberOrTag::Tag(finality_tag),
            ReturnFullTransactionObjects::from(false),
        );
        let head: GetBlockByNumberResponse = self
            .client
            .request(GET_BLOCK_BY_NUMBER_METHOD, &args)
            .await?;

        if head.number < receipt_block_number {
            return Err(ForeignChainInspectionError::NotFinalized);
        }
        Ok(())
    }
