### Title
Wrong Transaction Hash in DeclareV1 Fee Estimation and Simulation Due to `ClassHash::default()` — (`crates/apollo_rpc/src/v0_8/api/mod.rs`)

### Summary

When an unprivileged RPC client submits a `BroadcastedDeclareV1Transaction` to `starknet_estimateFee` or `starknet_simulateTransactions`, the conversion to `ExecutableTransactionInput` explicitly sets `class_hash: ClassHash::default()` (i.e., `0x0`). Because the Starknet V1 declare transaction hash formula includes the class hash, `calc_tx_hash` computes a hash over `class_hash=0` instead of the real class hash derived from the submitted contract class. The blockifier then executes the transaction with this wrong hash, so the `get_tx_info` syscall inside `__validate__` returns the wrong `transaction_hash`, and the returned `FeeEstimation` / `SimulatedTransaction` corresponds to a different transaction than the one the client submitted.

### Finding Description

In `TryFrom<BroadcastedDeclareTransaction> for ExecutableTransactionInput`: [1](#0-0) 

The `DeclareTransactionV0V1` struct is constructed with `class_hash: ClassHash::default()`. The comment acknowledges this is a workaround because the blockifier's API requires the field but "doesn't need" it. However, the field is used by `calc_tx_hash`: [2](#0-1) 

`calc_tx_hash` calls `get_transaction_hash(tx, chain_id, &TransactionOptions { only_query })` on the `DeclareTransaction::V1(tx)` where `tx.class_hash == 0`. The Starknet V1 declare hash formula is:

```
h("declare", 1, sender_address, h(class_hash), max_fee, chain_id, nonce)
```

With `class_hash=0`, the computed hash is `h(0)` instead of `h(real_class_hash)`. This wrong hash is then passed to `to_blockifier_tx` and used as the authoritative `tx_hash` for execution: [3](#0-2) 

### Impact Explanation

- **`starknet_estimateFee`**: The fee is estimated for a transaction whose hash is `h(..., h(0), ...)` rather than `h(..., h(real_class_hash), ...)`. Any account whose `__validate__` checks the transaction hash (standard practice) will see the wrong value, causing the estimate to either fail spuriously or succeed with incorrect fee data.
- **`starknet_simulateTransactions`**: The returned `SimulatedTransaction` trace contains a transaction hash that does not match the transaction the client submitted, making the simulation result misleading or incorrect.

This is a concrete, reachable wrong-value return from a public RPC endpoint, fitting the **High** impact category: *"RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value."*

### Likelihood Explanation

Any unprivileged client calling `starknet_estimateFee` or `starknet_simulateTransactions` with a `BroadcastedDeclareV1Transaction` triggers this path unconditionally. No special privileges or race conditions are required. The code path is deterministic.

### Recommendation

Compute the real class hash from the submitted contract class before constructing `DeclareTransactionV0V1`. For deprecated (Cairo 0) classes, the class hash is the Pedersen hash of the contract class definition. Replace:

```rust
class_hash: ClassHash::default(),
```

with:

```rust
class_hash: compute_deprecated_class_hash(&sn_api_contract_class)
    .map_err(internal_server_error)?,
```

using the appropriate hash function from `starknet_api` or `blockifier`. This ensures `calc_tx_hash` produces the correct transaction hash for simulation and fee estimation.

### Proof of Concept

1. Construct a `BroadcastedDeclareV1Transaction` with a known Cairo 0 contract class whose real class hash is `H_real`.
2. Call `starknet_estimateFee` (or `starknet_simulateTransactions`) via the RPC.
3. Observe that the transaction hash in the trace equals `h("declare", 1, sender, h(0), max_fee, chain_id, nonce)`.
4. Compute the expected hash using `H_real` and assert they differ.
5. If the account's `__validate__` checks `get_tx_info().transaction_hash`, the validation will fail or behave incorrectly, producing a wrong fee estimate.

    fn calc_tx_hash(self, chain_id: &ChainId) -> ExecutionResult<(Self, TransactionHash)> {
        match self.apply_on_transaction(|tx, only_query| {
            get_transaction_hash(tx, chain_id, &TransactionOptions { only_query })
        }) {
            (original_tx, Ok(tx_hash)) => Ok((original_tx, tx_hash)),
            (_, Err(err)) => Err(ExecutionError::TransactionHashCalculationFailed(err)),
