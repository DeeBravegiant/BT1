Audit Report

## Title
Uninitialized `maxNumberOfWithdrawalsPerUser` Blocks All Withdrawal Initiations — (File: contracts/KERNEL/KernelDepositPool.sol)

## Summary
`maxNumberOfWithdrawalsPerUser` is declared as a storage variable at line 108 but is never assigned in `initialize()`. It defaults to `0`, causing the guard in `initiateWithdrawal()` to always revert with `WithdrawalLimitReached` because `array.length >= 0` is unconditionally true. Every staker's withdrawal is blocked until an admin separately calls `setMaxNumberOfWithdrawalsPerUser()`.

## Finding Description
`maxNumberOfWithdrawalsPerUser` is declared but never set in `initialize()`: [1](#0-0) [2](#0-1) 

The guard in `initiateWithdrawal()` is: [3](#0-2) 

With `maxNumberOfWithdrawalsPerUser == 0`, the expression `userWithdrawalIds[msg.sender].length >= 0` evaluates to `true` for every caller (a `uint256` array length is always `≥ 0`), causing an unconditional revert. The setter explicitly rejects `0` as invalid, confirming it is not a valid operational value: [4](#0-3) 

## Impact Explanation
Any user who has staked KERNEL tokens via `stake()` or `stakeFor()` cannot initiate a withdrawal of their principal. Their staked balance is locked in the contract until an admin calls `setMaxNumberOfWithdrawalsPerUser()`. This is a **Medium. Temporary freezing of funds** — the funds are not permanently lost, but are inaccessible for an indeterminate window between deployment and the admin's corrective call.

## Likelihood Explanation
The broken state is the default state of every fresh deployment. No attacker action is required — any staker who calls `initiateWithdrawal()` before the admin configures the parameter will have their transaction revert. The exposure window is the entire period between contract deployment and the admin's separate configuration call, which is not guaranteed to happen promptly or at all.

## Recommendation
Initialize `maxNumberOfWithdrawalsPerUser` to a safe non-zero default inside `initialize()`:

```solidity
function initialize(address _admin, address _kernelToken, address _rewardToken) external initializer {
    // ... existing setup ...
    maxNumberOfWithdrawalsPerUser = MAX_WITHDRAWALS_PER_USER; // 100
}
```

This ensures the contract is immediately operational after deployment without requiring a separate admin transaction.

## Proof of Concept
1. Admin deploys and calls `initialize(admin, kernelToken, rewardToken)`. `maxNumberOfWithdrawalsPerUser` is `0`.
2. Alice calls `stake(100e18)`. `balanceOf[Alice] = 100e18`.
3. Alice calls `initiateWithdrawal(100e18)`.
4. Check: `userWithdrawalIds[Alice].length >= maxNumberOfWithdrawalsPerUser` → `0 >= 0` → `true`.
5. Transaction reverts with `WithdrawalLimitReached`.
6. Alice's 100e18 KERNEL tokens remain locked. No public function exists to recover them until the admin calls `setMaxNumberOfWithdrawalsPerUser(n)` for `n > 0`.

Foundry test sketch:
```solidity
function test_withdrawalBlockedOnFreshDeploy() public {
    // deploy + initialize (no setMaxNumberOfWithdrawalsPerUser call)
    pool.stake(100e18);
    vm.expectRevert(KernelDepositPool.WithdrawalLimitReached.selector);
    pool.initiateWithdrawal(100e18);
}
```
