Audit Report

## Title
Stale `rsETHPrice` Allows Deposits at Below-Fair-Value Rate, Diluting Existing Holders' Unclaimed Yield - (File: contracts/LRTOracle.sol)

## Summary
`LRTOracle.updateRSETHPrice()` carries no access control beyond `whenNotPaused`, and `LRTDepositPool.getRsETHAmountToMint()` divides by the stored `rsETHPrice` rather than a freshly computed price. Between keeper invocations, rewards accrue in EigenLayer strategies and the stored price becomes stale (lower than fair value). Any depositor during this window receives more rsETH than they are entitled to, diluting the unclaimed yield of all existing rsETH holders.

## Finding Description

**Root cause 1 — `updateRSETHPrice()` is permissionlessly callable:**

`LRTOracle.sol` line 87 exposes `updateRSETHPrice()` as `public whenNotPaused` with no role check. Any EOA or contract may call it at will. [1](#0-0) 

**Root cause 2 — deposit minting reads the stored stale price:**

`getRsETHAmountToMint()` divides by `lrtOracle.rsETHPrice()`, which returns the last written storage value. No price refresh occurs before or during deposit. [2](#0-1) 

`_beforeDeposit()` calls `getRsETHAmountToMint()` without triggering a price update, and `depositETH()` calls `_beforeDeposit()` directly. [3](#0-2) [4](#0-3) 

**How staleness arises:**

`_getTotalEthInProtocol()` sums live on-chain balances (EigenLayer pod shares, NDC balances, vault balances), so `totalETHInProtocol` grows continuously as rewards accrue. The stored `rsETHPrice` only advances when `_updateRsETHPrice()` is executed. Between two consecutive keeper calls the stored price is lower than the fair price `totalETHInProtocol / rsethSupply`. [5](#0-4) 

**Why the `pricePercentageLimit` guard does not protect deposits:**

The threshold check at lines 252–266 only fires inside `_updateRsETHPrice()` and only blocks the *price-update* call when the increase exceeds the configured limit. It has no effect on the deposit mint calculation whatsoever. Normal daily accrual (e.g., 0.05 % per day) falls well below any reasonable limit and passes freely. [6](#0-5) 

**Exploit flow:**

1. Rewards accrue; stored `rsETHPrice` = 1.00 ETH, fair price = 1.10 ETH (TVL 1 100 ETH, supply 1 000 rsETH).
2. Attacker calls `depositETH{value: 100 ETH}(0, "")`. Mint calculation: `(100e18 × 1e18) / 1.00e18 = 100 rsETH`. Fair amount ≈ 90.9 rsETH. Attacker receives **9.1 excess rsETH**.
3. Attacker (or anyone) calls `updateRSETHPrice()`. New price = 1 200 / 1 100 ≈ 1.0909 ETH.
4. Attacker sells 100 rsETH on a DEX at ≈ 1.0909 ETH → proceeds ≈ 109.09 ETH, cost 100 ETH, **profit ≈ 9.09 ETH**.
5. Original 1 000 rsETH holders now hold claims worth 1 000 × 1.0909 = 1 090.9 ETH instead of the 1 100 ETH they were entitled to — a loss of ≈ 9.1 ETH transferred to the attacker.

Note: the fee-minting "compounding" sub-claim in the submission is incorrect — `previousTVL` increases by exactly the deposited amount (attacker's 100 rsETH × 1.00 = 100 ETH), and `totalETHInProtocol` also increases by 100 ETH, so `rewardAmount` is unchanged. The primary dilution mechanism is sufficient on its own.

## Impact Explanation

Existing rsETH holders suffer concrete, quantifiable dilution of their unclaimed yield. The attacker extracts value that belongs to prior depositors by minting rsETH at a below-fair-value price and immediately liquidating on a secondary market (Curve, Uniswap, etc.) without waiting for the 8-day L1 withdrawal delay. This matches the allowed impact: **High — Theft of unclaimed yield**.

## Likelihood Explanation

- The attack requires only a standard ETH deposit followed by a public function call — no privileged role, no flash loan, no special setup.
- A staleness window exists between every pair of consecutive keeper invocations. Keeper failures, gas spikes, or deliberate attacker delay (withholding the `updateRSETHPrice()` call) all extend the window.
- The `pricePercentageLimit` guard does not impede the attack for normal daily accrual rates.
- The attack is repeatable every keeper cycle.

**Likelihood: Medium.**

## Recommendation

1. **Preferred fix:** Call `_updateRsETHPrice()` (or an equivalent fresh-price computation) inside `getRsETHAmountToMint()` or `_beforeDeposit()` so every mint uses a live price. Because `_beforeDeposit` is already `private view`, the price-update logic would need to be extracted into a separate internal write step called before the view calculation.
2. **Alternative:** Restrict `updateRSETHPrice()` to a keeper role and require the keeper to atomically update the price in the same transaction as (or immediately before) any deposit, or use a time-weighted price to smooth stale-price arbitrage.
3. **Minimum mitigation:** Add a maximum staleness check in `getRsETHAmountToMint()` that reverts if `rsETHPrice` has not been updated within an acceptable window (e.g., 1 hour), forcing depositors to trigger a price refresh first.

## Proof of Concept

Foundry fork test outline:

```solidity
function test_stalePrice_dilution() public fork {
    // 1. Simulate reward accrual: advance EigenLayer pod shares so
    //    _getTotalEthInProtocol() returns 1100 ETH while rsETHPrice == 1e18
    //    and rsETH supply == 1000e18.

    // 2. Attacker deposits 100 ETH via LRTDepositPool.depositETH{value:100e18}(0,"")
    uint256 attackerRsETH = rsETH.balanceOf(attacker);
    assertGt(attackerRsETH, 100e18); // receives > 100 rsETH at stale price

    // 3. Anyone calls LRTOracle.updateRSETHPrice()
    lrtOracle.updateRSETHPrice();
    uint256 newPrice = lrtOracle.rsETHPrice(); // ≈ 1.0909e18

    // 4. Verify existing holders are diluted
    uint256 holderClaim = existingSupply * newPrice / 1e18;
    assertLt(holderClaim, 1100e18); // less than the 1100 ETH they were owed

    // 5. Attacker sells rsETH on DEX at newPrice, profit > 0
}
```

The call sequence is fully permissionless: `LRTDepositPool.depositETH()` → `_beforeDeposit()` → `getRsETHAmountToMint()` → reads stale `rsETHPrice`; then `LRTOracle.updateRSETHPrice()` callable by anyone. [1](#0-0) [7](#0-6)
