Audit Report

## Title
Stale `rsETHPrice` Cache Allows Depositors to Extract Accrued Yield Before Price Update - (File: contracts/LRTOracle.sol)

## Summary
`LRTOracle.rsETHPrice` is a cached state variable updated only when `updateRSETHPrice()` is explicitly called. Because `depositAsset`/`depositETH` mint rsETH using this stale cached price as the denominator while `getAssetPrice()` reads live Chainlink values, an attacker can deposit at the artificially low stale price to receive excess rsETH, then call the public `updateRSETHPrice()` themselves, and initiate a withdrawal at the now-correct higher price — extracting yield that belongs to existing rsETH holders.

## Finding Description

**Root cause:** `rsETHPrice` is a stored state variable updated only on explicit calls to `updateRSETHPrice()`, which is `public whenNotPaused` with no access control. [1](#0-0) 

The mint formula in `getRsETHAmountToMint()` divides by the stale cached `rsETHPrice`: [2](#0-1) 

`getAssetPrice(asset)` reads a live Chainlink price, while `rsETHPrice` is the last stored value. As EigenLayer rewards accrue between `updateRSETHPrice()` calls, the true rsETH value rises above `rsETHPrice`. During this window the denominator is artificially low, so depositors receive more rsETH than fair value.

`initiateWithdrawal()` calls `getExpectedAssetAmount()`, which also reads `rsETHPrice`: [3](#0-2) 

The `expectedAssetAmount` is stored in the `WithdrawalRequest` at initiation time: [4](#0-3) 

During `unlockQueue()`, `_calculatePayoutAmount()` returns `min(request.expectedAssetAmount, currentReturn)`: [5](#0-4) 

This means the payout is capped at the `expectedAssetAmount` locked in at initiation — which the attacker set after triggering the price update. The min() check does not protect against the attack; it only prevents the attacker from benefiting from further price increases after initiation.

**Why the `pricePercentageLimit` guard is insufficient:** The guard reverts only if the price increase exceeds the configured threshold and the caller is not a manager. When the accrued yield fits within the threshold (e.g., a normal daily reward cycle), any unprivileged caller can invoke `updateRSETHPrice()` freely. [6](#0-5) 

## Impact Explanation
**High — Theft of unclaimed yield.** The attacker's excess rsETH represents a larger proportional claim on the protocol TVL than was deposited. The profit is extracted directly from yield that should have accrued to existing rsETH holders. The attack is repeatable every reward cycle and requires no special permissions.

## Likelihood Explanation
**Medium.** EigenLayer rewards accrue continuously, so a staleness window exists between every pair of `updateRSETHPrice()` calls. No special permissions are required: `updateRSETHPrice()` is public, deposits are open, and the attacker controls the timing of both the deposit and the price update. The `pricePercentageLimit` guard limits per-call profit but does not prevent the attack when the price increase is within the configured threshold.

## Recommendation
Call `_updateRsETHPrice()` (or an equivalent internal price refresh) atomically at the start of `depositAsset()` and `depositETH()` before computing `rsethAmountToMint`. This ensures every deposit uses the current, reward-inclusive price and eliminates the staleness window entirely.

## Proof of Concept

**Initial state:**
- `rsETHPrice = 1.000e18` (stale; true value is `1.010e18` due to accrued EigenLayer rewards)
- stETH/ETH Chainlink price = `1.000e18`
- `pricePercentageLimit` allows a 1% increase (the 1% reward fits within the limit)

**Step 1 — Attacker deposits 100 stETH at stale price** (`depositAsset`):
```
rsethAmountToMint = (100e18 * 1.000e18) / 1.000e18 = 100e18 rsETH
Fair amount       = (100e18 * 1.000e18) / 1.010e18 ≈  99.01e18 rsETH
Excess rsETH      ≈ 0.99e18 rsETH
```

**Step 2 — Attacker calls `updateRSETHPrice()`** (public, no access control):
- `rsETHPrice` updates to `1.010e18`

**Step 3 — Attacker calls `initiateWithdrawal(stETH, 100e18)`**:
```
expectedAssetAmount = getExpectedAssetAmount(stETH, 100e18)
                    = 100e18 * 1.010e18 / 1.000e18 = 101e18 stETH
```
This value is stored in `WithdrawalRequest.expectedAssetAmount`.

**Step 4 — Operator calls `unlockQueue()`** (normal protocol operation):
```
currentReturn = (100e18 * 1.010e18) / 1.000e18 = 101e18 stETH
payoutAmount  = min(101e18, 101e18) = 101e18 stETH
```
`request.expectedAssetAmount` is set to `101e18 stETH`.

**Step 5 — After `withdrawalDelayBlocks`, attacker calls `completeWithdrawal()`**:
- Attacker receives **101 stETH**
- Net profit: **1 stETH** extracted from existing holders' accrued yield

**Foundry test plan:** Deploy protocol with mock Chainlink oracle returning `1.000e18` for stETH. Seed EigenLayer mock to reflect `1.010e18` true rsETH value without updating `rsETHPrice`. Execute the five-step sequence above in a single test. Assert attacker's stETH balance increased by ~1 stETH and that existing holder's proportional claim on TVL decreased by the corresponding amount.
