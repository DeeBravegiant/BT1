Audit Report

## Title
Protocol Fee rsETH Dilution Rate Varies With `updateRSETHPrice()` Call Frequency - (File: contracts/LRTOracle.sol)

## Summary
In `_updateRsETHPrice()`, the ETH-denominated fee is always `protocolFeeInBPS` of incremental reward, but the rsETH amount minted is `protocolFeeInETH / newRsETHPrice`. Because `newRsETHPrice` is lower when the function is called more frequently (less reward has compounded per call), more rsETH is minted per ETH of fee on each invocation. For the same total reward, higher call frequency produces strictly greater cumulative rsETH dilution than a single call, meaning the effective fee rate is not fixed at `protocolFeeInBPS` and existing holders receive fewer net rewards than the nominal parameter implies.

## Finding Description
The fee conversion at line 301 uses `newRsETHPrice`, which is computed at line 250 as `(totalETHInProtocol - protocolFeeInETH) / rsethSupply`. When `updateRSETHPrice()` is called more frequently, each call sees a smaller `rewardAmount`, a smaller `protocolFeeInETH`, and therefore a lower `newRsETHPrice` (less reward has compounded into the price). Dividing the ETH fee by this lower price yields a larger rsETH mint. Across N calls covering the same total reward as one call, the cumulative rsETH minted is strictly larger because the average price used as the denominator is lower. The daily fee mint limit (`_checkAndUpdateDailyFeeMintLimit`) caps total rsETH minted per day but does not fix the per-unit-reward accounting: within the daily cap, more frequent calls still extract more rsETH dilution per ETH of reward than the nominal `protocolFeeInBPS` rate implies. `updateRSETHPrice()` carries no access control beyond `whenNotPaused`, so any external caller can trigger this at will.

## Impact Explanation
Existing rsETH holders are diluted by the treasury fee mint at a rate that exceeds `protocolFeeInBPS` when the function is called at high cadence. No funds are directly stolen, but holders receive fewer net rewards than the protocol's nominal fee parameter promises. This matches **Low — Contract fails to deliver promised returns, but doesn't lose value**.

## Likelihood Explanation
`updateRSETHPrice()` is a public, permissionless function callable by any address. Any depositor, rsETH holder, or bot can call it at arbitrary frequency. No privileged role, special balance, or external condition is required. The condition is always reachable on a live deployment.

## Recommendation
Convert the fee to rsETH using the **pre-fee price** (`totalETHInProtocol / rsethSupply`) rather than the post-fee price `newRsETHPrice`. This makes the rsETH mint amount independent of call cadence: `rsethFee = protocolFeeInETH / (totalETHInProtocol / rsethSupply)`, which simplifies to `rsethFee = protocolFeeInETH * rsethSupply / totalETHInProtocol` — a fixed share of supply growth regardless of how often the function is called. Alternatively, adopt a time-weighted continuous fee model (analogous to EIP-4626 streaming fees) so the effective rate is strictly a function of elapsed time and reward, not call frequency.

## Proof of Concept
Using `rsethSupply = 100`, `rsETHPrice = 1.0`, `protocolFeeInBPS = 1000` (10%), TVL grows 100 → 120 ETH:

**Single call:**
- `rewardAmount = 20`, `protocolFeeInETH = 2`
- `newRsETHPrice = (120 − 2) / 100 = 1.18`
- `rsethFee = 2 / 1.18 ≈ 1.695 rsETH` minted

**Two calls (100→110, then 110→120):**
- Call 1: `protocolFeeInETH = 1`, `newRsETHPrice = 109/100 = 1.09`, `rsethFee₁ = 1/1.09 ≈ 0.917`; supply → 100.917
- Call 2: `previousTVL = 100.917 × 1.09 ≈ 110`, `protocolFeeInETH = 1`, `newRsETHPrice = 119/100.917 ≈ 1.1792`, `rsethFee₂ = 1/1.1792 ≈ 0.848`; supply → 101.765

Two calls mint ≈ **1.765 rsETH** vs. one call minting ≈ **1.695 rsETH** — ~4% more dilution for identical total reward. A Foundry fuzz test parameterizing call count N over a fixed TVL delta would confirm the monotonic relationship between call frequency and cumulative rsETH minted, bounded only by the daily limit.
