Audit Report

## Title
Wrong Guard Condition for ETH Deposit Limit Allows Limit to Be Exceeded - (File: contracts/LRTDepositPool.sol)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` omits the incoming `amount` from the ETH branch comparison, checking only whether the current total already exceeds the cap rather than whether the incoming deposit would exceed it. As a result, any depositor can push the ETH total above the configured deposit limit by up to one full deposit in a single transaction.

## Finding Description
The function at [1](#0-0)  contains two branches. For ERC-20 assets the check is `totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)`, which correctly accounts for the incoming deposit. For ETH the check is `totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)`, silently dropping `amount`. This means the guard only reverts when the cap is **already** breached, not when the deposit **would** breach it.

The guard is invoked unconditionally via `_beforeDeposit` before every ETH deposit: [2](#0-1) 

`depositETH` passes `msg.value` as the deposit amount and proceeds to mint rsETH for the full value regardless: [3](#0-2) 

No other guard exists between the limit check and the mint, so the full `msg.value` is accepted and rsETH is minted even when the deposit pushes the ETH total above the cap.

## Impact Explanation
The ETH deposit cap (`depositLimitByAsset[ETH_TOKEN]`) is the protocol's primary mechanism for bounding total ETH exposure. The bypass allows rsETH to be minted against more ETH than governance has authorised, violating the protocol's promised deposit ceiling. This matches the allowed impact: **Low — Contract fails to deliver promised returns, but doesn't lose value.** Deposited ETH is not stolen; the protocol simply accepts more than the configured limit.

## Likelihood Explanation
The entry path is fully permissionless. Any address can call `depositETH` with a `msg.value` that would push `getTotalAssetDeposits(ETH_TOKEN)` above the limit. No special role, front-running, or external dependency is required. The condition is reachable whenever `totalAssetDeposits` is within one deposit of the cap, which is a normal operating state. **Likelihood: High.**

## Recommendation
Remove the ETH-specific branch and apply the same logic used for ERC-20 tokens:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

The special-casing of `ETH_TOKEN` is unnecessary and is the root cause of the bug.

## Proof of Concept
Assume `depositLimitByAsset[ETH_TOKEN] = 100 ether` and `getTotalAssetDeposits(ETH_TOKEN) = 99 ether`.

1. Attacker calls `depositETH{value: 10 ether}(0, "")`.
2. `_beforeDeposit` calls `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 10 ether)`.
3. ETH branch evaluates `99 ether > 100 ether` → `false` → no revert.
4. `_mintRsETH` mints rsETH for the full 10 ETH.
5. `getTotalAssetDeposits(ETH_TOKEN)` is now 109 ETH — 9 ETH above the configured cap.

**ERC-20 path (correct):** `99 + 10 = 109 > 100` → reverts with `MaximumDepositLimitReached`.

A Foundry test can reproduce this by deploying the pool with a mock config returning `depositLimitByAsset = 100 ether`, seeding 99 ETH of prior deposits, then asserting that `depositETH{value: 10 ether}` succeeds and `getTotalAssetDeposits` returns 109 ETH.
