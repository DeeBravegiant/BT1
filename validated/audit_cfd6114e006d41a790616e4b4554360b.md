### Title
Caller-Agnostic `verify_foreign_transaction` Queue Key Allows Any Unprivileged Account to Saturate the Fan-Out Cap and Block Foreign-Chain Bridge Verification - (File: `crates/contract/src/lib.rs`, `crates/near-mpc-contract-interface/src/types/foreign_chain.rs`)

---

### Summary

`verify_foreign_transaction()` stores pending yields under a request key that contains **no caller identity**. Any unprivileged account can submit 128 identical requests for a known foreign-chain transaction ID, saturating the `MAX_PENDING_REQUEST_FAN_OUT = 128` cap and causing every subsequent submission for that transaction to revert with `PendingRequestQueueFull`. This blocks the Omnibridge inbound flow — the primary production use-case of this endpoint — for as long as the attacker continuously re-fills the queue.

---

### Finding Description

The `VerifyForeignTransactionRequest` struct used as the map key in `pending_verify_foreign_tx_requests` contains only `(request, domain_id, payload_version)` — **no caller account ID**:

```rust
pub struct VerifyForeignTransactionRequest {
    pub request: ForeignChainRpcRequest,
    pub domain_id: DomainId,
    pub payload_version: ForeignTxPayloadVersion,
}
``` [1](#0-0) 

The conversion from args to request in `args_into_verify_foreign_tx_request()` confirms no predecessor account is included:

```rust
pub fn args_into_verify_foreign_tx_request(
    args: dtos::VerifyForeignTransactionRequestArgs,
) -> dtos::VerifyForeignTransactionRequest {
    dtos::VerifyForeignTransactionRequest {
        domain_id: args.domain_id,
        request: args.request,
        payload_version: args.payload_version,
    }
}
``` [2](#0-1) 

This is the **opposite** of `sign()`, where `SignatureRequest::new()` derives a `tweak` from `(predecessor_id, path)`, making each caller's queue entry unique: [3](#0-2) 

The contract's own test explicitly acknowledges the caller-agnostic nature and its consequence:

```rust
// And: caller bob submits the identical request — a different account would today
// be blocked from receiving a response by alice's submission.
``` [4](#0-3) 

The fan-out queue is capped at `MAX_PENDING_REQUEST_FAN_OUT = 128`. When the cap is reached, `push_pending_yield` panics:

```rust
if queue.len() >= usize::from(MAX_PENDING_REQUEST_FAN_OUT) {
    env::panic_str(
        &RequestError::PendingRequestQueueFull {
            limit: MAX_PENDING_REQUEST_FAN_OUT,
        }
        .to_string(),
    );
}
``` [5](#0-4) 

`verify_foreign_transaction()` has no caller restriction — only a 1 yoctoNEAR deposit and gas check: [6](#0-5) 

Each queued yield times out after `REQUEST_EXPIRATION_BLOCKS = 200` blocks (~200 seconds on NEAR). The attacker can continuously re-fill the queue as slots drain, sustaining the DoS indefinitely at negligible cost. [7](#0-6) 

---

### Impact Explanation

The primary production use-case of `verify_foreign_transaction()` is the **Omnibridge inbound flow** (foreign chain → NEAR), where the MPC network must attest that a specific foreign-chain transaction finalized before releasing bridged assets on NEAR.

An attacker who knows a target transaction ID (observable on-chain or predictable from bridge protocol) can:
1. Saturate the 128-slot queue for that transaction at a cost of 128 yoctoNEAR (~$0).
2. Block any bridge service or user from submitting the same verification request.
3. Sustain the block indefinitely by re-filling as slots time out.

This breaks the request-lifecycle safety invariant: a legitimate user who paid a deposit and submitted a valid foreign-chain transaction cannot get it verified, stalling the bridge inbound flow. This maps to **Medium** impact: request-lifecycle and contract execution-flow manipulation that breaks production safety/accounting invariants without relying on network-level DoS or operator misconfiguration.

---

### Likelihood Explanation

- **No privilege required**: any NEAR account can call `verify_foreign_transaction()`.
- **Negligible cost**: 128 yoctoNEAR per saturation cycle (~200 blocks), effectively free.
- **Target is observable**: foreign-chain transaction IDs are public; bridge inbound flows are predictable.
- **Sustained attack is trivial**: the attacker simply re-submits as slots expire.
- **Likelihood: Medium** — requires knowledge of the target transaction ID, but that is publicly available for any bridge deposit.

---

### Recommendation

Include the caller's `predecessor_account_id` in the `VerifyForeignTransactionRequest` key, analogous to how `SignatureRequest` incorporates a caller-derived tweak and `CKDRequest` incorporates the predecessor account ID. This makes each caller's queue entry independent, preventing one account from saturating another's slot.

Alternatively, if the caller-agnostic fan-out is intentional (so multiple bridge services can share a single MPC response for the same transaction), enforce a **per-caller sub-limit** within the shared queue, or require a higher deposit that makes saturation economically infeasible.

---

### Proof of Concept

```rust
// Attacker (Alice) saturates the queue for a known Bitcoin tx that Bob's bridge needs to verify.
// Cost: 128 yoctoNEAR total.

let target_tx_id = [0xAB; 32]; // publicly observable Bitcoin tx ID
let request_args = VerifyForeignTransactionRequestArgs {
    domain_id: foreign_tx_domain_id,
    payload_version: ForeignTxPayloadVersion::V1,
    request: ForeignChainRpcRequest::Bitcoin(BitcoinRpcRequest {
        tx_id: target_tx_id.into(),
        confirmations: 1.into(),
        extractors: vec![BitcoinExtractor::BlockHash],
    }),
};

// Alice submits 128 identical requests, filling the caller-agnostic queue.
for _ in 0..128 {
    alice.call(contract.id(), "verify_foreign_transaction")
        .args_json(json!({ "request": request_args }))
        .deposit(NearToken::from_yoctonear(1))
        .max_gas()
        .transact()
        .await?;
}

// Bob's bridge service now tries to verify the same transaction.
// This reverts with: "Pending-request queue is full for this request key (limit: 128)."
let result = bob.call(contract.id(), "verify_foreign_transaction")
    .args_json(json!({ "request": request_args }))
    .deposit(NearToken::from_yoctonear(1))
    .max_gas()
    .transact()
    .await?
    .into_result();

assert!(result.is_err()); // PendingRequestQueueFull
// Bob must wait ~200 blocks per slot (~200 seconds) for the queue to drain.
// Alice re-fills as slots expire, sustaining the block indefinitely.
```
