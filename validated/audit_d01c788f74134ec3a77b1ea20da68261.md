### Title
Missing `message` Field in Solana `FinalizeTransferPayload::serialize_for_near` Causes Permanent Fund Lock - (File: `solana/programs/bridge_token_factory/src/state/message/finalize_transfer.rs`)

### Summary
The Solana `FinalizeTransferPayload::serialize_for_near` omits the `message` field from its Borsh serialization. NEAR's `TransferMessagePayload::encode_hashable()` includes the `message` bytes whenever they are non-empty. Any transfer to Solana whose signed payload carries a non-empty `message` will produce a hash mismatch, causing `verify_signature` to permanently reject the settlement and locking the user's tokens in the NEAR bridge forever.

### Finding Description

NEAR's canonical signing path in `sign_transfer` builds a `TransferMessagePayload` and calls `encode_hashable()`: [1](#0-0) 

When `message` is non-empty, `borsh::to_vec(self)` is used — the full struct including the `message: Vec<u8>` field. The Borsh layout appends a 4-byte little-endian length prefix followed by the raw bytes.

On the Solana side, `FinalizeTransferPayload` has no `message` field at all: [2](#0-1) 

And `serialize_for_near` never writes any message bytes: [3](#0-2) 

The signature verifier hashes exactly this serialized buffer: [4](#0-3) 

When `message` is non-empty, the NEAR-signed hash covers `[... fee_recipient_bytes | len(message) | message_bytes]`, while Solana hashes `[... fee_recipient_bytes]`. The two hashes diverge, `secp256k1_recover` returns a different public key, and `require!(signer.0 == *derived_near_bridge_address, ...)` panics — permanently blocking finalization.

Compare with EVM, which correctly handles the conditional message encoding: [5](#0-4) 

And Starknet, which also handles it correctly: [6](#0-5) 

Solana is the only destination chain whose payload serializer has no equivalent branch.

The user-controlled entry point is `InitTransferMsg::msg`: [7](#0-6) 

NEAR extracts the destination-chain hook payload from this field in `sign_transfer`: [8](#0-7) 

If `DestinationChainMsg::destination_msg()` returns non-empty bytes for a Solana-bound transfer, the signed payload includes the message and Solana can never verify it.

### Impact Explanation

Tokens are burned or locked on the source chain at `initTransfer` time. If the MPC-signed payload cannot be verified on Solana, `finalize_transfer` is permanently blocked. There is no recovery path: the nonce is never marked complete, but the source-side lock is irrecoverable. This matches the **irreversible fund lock / permanently unclaimable user value** impact class.

### Likelihood Explanation

Any unprivileged user who initiates a transfer to a Solana recipient and supplies a non-empty `msg` that produces a non-empty `destination_msg()` triggers the condition. The `msg` field is fully user-controlled, bounded only by `MAX_INIT_TRANSFER_MSG_LEN = 2048` bytes. No privileged role or key compromise is required.

### Recommendation

Add the `message` field to `FinalizeTransferPayload` and serialize it in `serialize_for_near`, mirroring the EVM and Starknet implementations:

```rust
// After fee_recipient serialization:
match &self.message {
    None => {}
    Some(msg) => {
        // Borsh Vec<u8>: 4-byte LE length + raw bytes
        (msg.len() as u32).serialize(&mut writer)?;
        writer.write_all(msg)?;
    }
}
```

Alternatively, if Solana intentionally does not support the `message` hook, NEAR's `sign_transfer` must assert that `message` is empty when the destination chain is Solana, and reject the signing request otherwise, so funds are never committed to an unverifiable payload.

### Proof of Concept

1. User calls `ft_on_transfer` on NEAR with `InitTransferMsg { recipient: OmniAddress::Sol(...), msg: Some("<valid DestinationChainMsg JSON with non-empty destination_msg>"), ... }`.
2. NEAR stores the transfer, relayer calls `sign_transfer`.
3. `sign_transfer` extracts a non-empty `message` bytes vector and calls `encode_hashable()`, which serializes the full `TransferMessagePayload` including the message bytes. MPC signs this hash.
4. Relayer submits `FinalizeTransferPayload` + signature to Solana `finalize_transfer`.
5. `FinalizeTransferPayload::serialize_for_near` produces a buffer without the message bytes.
6. `keccak::hash(serialized)` differs from the MPC-signed hash.
7. `secp256k1_recover` returns a wrong key; `require!(signer.0 == *derived_near_bridge_address)` panics.
8. The transaction reverts. The user's tokens remain locked in the NEAR bridge with no recovery path.
