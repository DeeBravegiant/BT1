**The core issue:** `execute_signing_instructions` in `chia/wallet/wallet.py` accepts a `PathHint` with an arbitrary BIP32 path and derives + signs with the resulting key, with **no restriction on which path is allowed**.

The only guard present is a fingerprint match check: [1](#0-0)

There is **no validation of what `path` contains**. The `PathHint` type itself only carries `root_fingerprint` and `path`: [2](#0-1) 

And the key derivation paths for farmer and pool keys are: [3](#0-2) 


Wallet keys use path type `2` (`[12381, 8444, 2, ...]`), but the code will equally derive and sign with type `0` (farmer) or `1` (pool) — or any other path — without complaint.

### Title
Unrestricted BIP32 Path in `execute_signing_instructions` Allows Signing with Farmer/Pool Keys — (`chia/wallet/wallet.py`)

### Summary
`execute_signing_instructions` accepts caller-supplied `PathHint` values and derives child keys from the master private key along any BIP32 path. There is no guard restricting derivation to the wallet key namespace (`[12381, 8444, 2, ...]`). A caller with wallet RPC access can obtain valid signatures from the farmer key (`[12381, 8444, 0, 0]`), pool key (`[12381, 8444, 1, 0]`), singleton owner keys (`[12381, 8444, 5, n]`), or pooling authentication keys (`[12381, 8444, 6, ...]`).

### Finding Description
In `chia/wallet/wallet.py`, `execute_signing_instructions` iterates over `path_hints` and for each hint whose `root_fingerprint` matches the wallet's master pubkey fingerprint, it calls `_derive_path(master_sk, path)` with the **caller-controlled** `path` list. [4](#0-3) 

The derived private key is unconditionally inserted into `sk_lookup`. When a `SigningTarget` is later processed whose `fingerprint` matches the derived key, the wallet signs the caller-supplied `message` with that key: [5](#0-4) 

No check exists to verify that the derived path falls within the wallet key namespace. The `PathHint` dataclass imposes no path constraints: [2](#0-1) 

### Impact Explanation
- **Pool reward diversion:** The singleton owner key (`[12381, 8444, 5, n]`) controls pool membership for plot NFTs. Signing an attacker-chosen message with this key can authorize a pool switch, redirecting farming rewards.
- **Farmer key compromise:** The farmer key (`[12381, 8444, 0, 0]`) is used to sign blocks and pool partial proofs. Arbitrary message signing with it enables impersonation in pool protocols.
- **Pool authentication key abuse:** Keys at `[12381, 8444, 6, ...]` authenticate the farmer to pools; signing attacker messages with these enables pool-level impersonation.

### Likelihood Explanation
The wallet RPC is localhost-only and requires SSL mutual authentication. Any local process running as the same OS user (or with access to the SSL certs in `~/.chia/mainnet/config/ssl/`) can call this endpoint. The attack requires no special privileges beyond wallet RPC access, which is the normal access level for any Chia application or script running locally.

### Recommendation
Add a path allowlist or prefix check before deriving keys in `execute_signing_instructions`. Only paths beginning with `[12381, 8444, 2, ...]` (wallet keys) should be accepted. All other path prefixes should be rejected with an error, regardless of whether the `root_fingerprint` matches.

```python
WALLET_PATH_PREFIX = [12381, 8444, 2]

for path_hint in signing_instructions.key_hints.path_hints:
    ...
    path = [int(step) for step in path_hint.path]
    if path[:3] != WALLET_PATH_PREFIX:
        raise ValueError(f"PathHint path {path} is outside the wallet key namespace")
    ...
```
