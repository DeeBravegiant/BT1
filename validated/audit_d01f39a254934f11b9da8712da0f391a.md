Audit Report

## Title
Stale Cross-Chain agETH Rate Enables Over-Minting via Token Deposit, Stealing Yield from Existing Holders — (`contracts/agETH/AGETHPoolV3.sol`)

## Summary

`AGETHPoolV3.deposit(token, amount, referralId)` mints agETH using the ratio `tokenToETHRate / agETHToETHrate`. The `agETHToETHrate` is sourced from `AGETHRateReceiver`, a `CrossChainRateReceiver` that stores the last LayerZero-delivered rate with no staleness enforcement. When this rate is stale-low relative to the live token oracle, the minting formula over-issues agETH, diluting existing holders' accrued yield.

## Finding Description

`AGETHPoolV3.viewSwapAgETHAmountAndFee(amount, token)` at L184–194 computes:

```solidity
uint256 agETHToETHrate = getRate();                                          // L188
uint256 tokenToETHRate = IOracle(supportedTokenOracle[token]).getRate();     // L191
agETHAmount = amountAfterFee * tokenToETHRate / agETHToETHrate;              // L194
```

`getRate()` at L104–106 delegates to `IOracle(agETHOracle).getRate()`, where `agETHOracle` is an `AGETHRateReceiver` — a concrete `CrossChainRateReceiver`. That contract's `getRate()` at L103–105 unconditionally returns the stored `rate`:

```solidity
function getRate() external view returns (uint256) {
    return rate;
}
```

`rate` is updated only on `lzReceive` (L95–97), which sets `lastUpdated = block.timestamp`. However, `lastUpdated` is **never read** in `getRate()` or anywhere in the deposit path. There is no maximum-age check, no circuit breaker, and no revert path for a stale rate. The `tokenToETHRate` for the deposited LST is fetched from a separate, live on-chain oracle (L191), creating an asymmetry: one rate is live, the other can be arbitrarily old.

Exploit path:
1. The cross-chain LayerZero message is delayed (naturally or due to congestion), leaving `rate` stale-low.
2. Attacker calls `deposit(token, amount, referralId)` permissionlessly.
3. `agETHToETHrate` is stale-low; `tokenToETHRate` is live and higher.
4. `agETHAmount = amountAfterFee * tokenToETHRate / agETHToETHrate` is inflated.
5. Attacker receives excess agETH shares representing yield accrued by existing holders.

## Impact Explanation

agETH is yield-bearing: its ETH value rises over time. Existing holders' accrued-but-unclaimed yield is embedded in the rising agETH/ETH rate. When an attacker deposits at a stale-low `agETHToETHrate`, they receive more agETH shares than the deposited value warrants. Upon the rate correcting, those excess shares represent a direct transfer of unclaimed yield from existing holders to the attacker. This matches the allowed impact: **High — Theft of unclaimed yield**.

Numeric example:
- True agETH/ETH rate: `1.05e18`; stale stored rate: `1.00e18`
- wstETH/ETH rate (live): `1.05e18`; deposit: `1e18` wstETH
- Correct agETH minted: `1.05e18 * 1.05e18 / 1.05e18 = 1.05e18`
- Actual agETH minted: `1.05e18 * 1.05e18 / 1.00e18 = 1.1025e18`
- Excess: `~0.0525e18` agETH stolen from existing holders' yield

## Likelihood Explanation

Cross-chain LayerZero messages experience natural delays (minutes to hours) and can be further delayed by network congestion or relayer downtime. The `lastUpdated` field in `CrossChainRateReceiver` confirms the protocol is aware of the time dimension but provides no enforcement. `deposit(token, amount, referralId)` is permissionless — no admin access, no oracle manipulation, and no front-running is required. Any user can exploit this during any staleness window, which occurs routinely in normal cross-chain operation.

## Recommendation

Add a staleness guard in `CrossChainRateReceiver.getRate()`:

```solidity
uint256 public maxRateAge = 24 hours; // configurable by owner

function getRate() external view returns (uint256) {
    require(block.timestamp - lastUpdated <= maxRateAge, "Rate is stale");
    return rate;
}
```

This blocks deposits when the cross-chain rate has not been refreshed within an acceptable window, preventing over-minting.

## Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.27;

// Foundry fuzz test
// Invariant: agETHAmount * agETHToETHrate <= amountAfterFee * tokenToETHRate
// (attacker should not receive more ETH-value in agETH than deposited)

function testStaleRateOverMint(uint256 agETHToETHrate) public {
    // Stale-low range: rate has not updated while true rate grew to 1.05e18
    agETHToETHrate = bound(agETHToETHrate, 1e18, 1.05e18 - 1);

    uint256 tokenToETHRate = 1.05e18; // live wstETH/ETH oracle
    uint256 amountAfterFee = 1e18;

    // Mirrors AGETHPoolV3.viewSwapAgETHAmountAndFee L194
    uint256 agETHAmount = amountAfterFee * tokenToETHRate / agETHToETHrate;

    // Invariant fails: attacker receives more ETH-value in agETH than deposited
    assertLe(
        agETHAmount * agETHToETHrate,
        amountAfterFee * tokenToETHRate,
        "Over-minting: attacker steals yield"
    );
}
```

The assertion fails for all `agETHToETHrate < tokenToETHRate` due to integer division, confirming over-minting. A fork test can be constructed by deploying `AGETHPoolV3` with a mock `AGETHRateReceiver` whose `rate` is set to `1.00e18` (stale) while the wstETH oracle returns `1.05e18`, then calling `deposit` and verifying the minted agETH exceeds the fair share.
