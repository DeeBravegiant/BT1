### Title
`respond_verify_foreign_tx` Accepts `payload_hash` Not Bound to the Submitted `request` — Forged Foreign-Chain Verification via Cross-Request Signature Replay - (File: `crates/contract/src/lib.rs`)

### Summary

`respond_verify_foreign_tx` verifies only that `response.signature` is a valid ECDSA signature over `response.payload_hash` using the MPC root public key. It never recomputes the expected `payload_hash` from the `request` parameters and checks they match. A single Byzantine attested participant (below signing threshold) who learns a valid `(payload_hash, signature)` pair from one MPC signing session can replay it against a *different* pending foreign-tx request, resolving that request with forged verification data.

### Finding Description

In `crates/contract/src/lib.rs` at `respond_verify_foreign_tx` (lines 691–753), the contract performs two independent checks:

1. The caller is an attested participant.
2. `response.signature` is a valid ECDSA signature over `response.payload_hash` under the root public key.

It then calls `pending_requests::resolve_yields_for(&mut self.pending_verify_foreign_tx_requests, &request, ...)` to drain all yields queued under `request`. [1](#0-0) 

The missing check is: **the contract never recomputes the canonical `payload_hash` from `request` and asserts it equals `response.payload_hash`**. The canonical hash is defined as:

```rust
ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: request.request.clone(),
    values: extracted_values,   // ← never verified on-chain
}).compute_msg_hash()
``` [2](#0-1) 

The `values` (extracted foreign-chain data) are entirely absent from the on-chain `VerifyForeignTransactionRequest` stored in the pending map. The contract has no way to reconstruct the expected hash without them, and it never tries. [3](#0-2) 

### Impact Explanation

A Byzantine attested participant (single node, strictly below threshold) who participates in the MPC signing session for **Request A** learns the tuple `(payload_hash_A, sig_A)` before the leader submits it on-chain. They can immediately call:

```
respond_verify_foreign_tx(request = B, response = {payload_hash_A, sig_A})
```

where **Request B** is any other pending foreign-tx request. The contract accepts this call because:
- `sig_A` is a valid signature over `payload_hash_A` under the root key ✓
- Request B exists in `pending_verify_foreign_tx_requests` ✓

Request B is resolved and all its queued yields are drained with the forged response `{payload_hash_A, sig_A}`. The bridge contract that submitted Request B receives a valid MPC signature, but over the *wrong* payload hash — one that encodes the extracted values of a completely different foreign transaction. If the bridge contract does not call `ForeignChainSignatureVerifier::verify_signature` from the SDK (or implements its own looser check), it will accept this as proof that transaction B was verified, enabling invalid bridge execution or double-spend. [4](#0-3) 

Even when the bridge does call `verify_signature`, the pending request is permanently consumed: the legitimate response for Request B can never be delivered, breaking the request lifecycle.

### Likelihood Explanation

- Requires only **one** Byzantine attested participant — strictly below the signing threshold. No collusion is needed.
- The participant learns `(payload_hash, sig)` during the MPC protocol execution, before the leader broadcasts the `respond_verify_foreign_tx` transaction. Frontrunning on NEAR is straightforward (submit with higher gas priority or race the leader).
- Any bridge contract that omits the `verify_signature` SDK call — or checks only that a valid MPC signature exists without checking *what* was signed — is immediately exploitable for fund theft.
- The `near-mpc-sdk` provides `verify_signature` as a helper but does not enforce its use; the MPC contract itself imposes no binding between `response.payload_hash` and `request`. [5](#0-4) 

### Recommendation

Inside `respond_verify_foreign_tx`, after retrieving the pending request, recompute the expected payload hash from the `request` fields and assert it equals `response.payload_hash` before accepting the response. Because `extracted_values` are not stored on-chain, the contract must either:

1. **Store the expected `payload_hash` at request submission time** (computed by the submitter and committed on-chain), then verify `response.payload_hash == stored_hash`; or
2. **Require the node to include the `extracted_values` in the response** and recompute the hash on-chain inside `respond_verify_foreign_tx`.

Option 1 is cheaper. The submitter computes `ForeignTxSignPayload::V1{request, expected_values}.compute_msg_hash()` off-chain and passes it as an additional field in `VerifyForeignTransactionRequestArgs`; the contract stores it alongside the yield and checks it in `respond_verify_foreign_tx`.

### Proof of Concept

```
// Setup: two pending requests exist
// Request A: Bitcoin tx_id = [0xAA; 32], extractors = [BlockHash]
// Request B: Bitcoin tx_id = [0xBB; 32], extractors = [BlockHash]

// Step 1: Byzantine participant joins MPC signing session for Request A.
// The session produces: payload_hash_A = hash(ForeignTxSignPayloadV1{request_A, [BlockHash(0xCC..)]})
//                       sig_A = ECDSA_root_key(payload_hash_A)

// Step 2: Before the leader submits respond_verify_foreign_tx(A, {payload_hash_A, sig_A}),
// the Byzantine participant submits:
contract.respond_verify_foreign_tx(
    request = VerifyForeignTransactionRequest { request: bitcoin_tx_BB, domain_id, payload_version },
    response = VerifyForeignTransactionResponse { payload_hash: payload_hash_A, signature: sig_A },
)

// Result:
// - Contract checks: verify_ecdsa_signature(sig_A, payload_hash_A, root_pk) → OK ✓
// - Contract checks: pending_verify_foreign_tx_requests.get(&request_B) → Some(...) ✓
// - Contract resolves Request B with {payload_hash_A, sig_A}
// - Bridge contract for Request B receives a valid MPC signature over tx_id=0xAA data
// - If bridge omits verify_signature check → accepts forged proof → releases funds
// - Request B's yield is consumed; legitimate response can never be delivered
``` [6](#0-5)

impl ForeignChainSignatureVerifier {
    pub fn verify_signature(
        self,
        response: &VerifyForeignTransactionResponse,
        // TODO(#2232): don't use interface API types for public keys
        public_key: &PublicKey,
    ) -> Result<(), VerifyForeignChainError> {
        let expected_payload = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
            request: self.request,
            values: self.expected_extracted_values,
        });

        let expected_payload_hash = expected_payload
            .compute_msg_hash()
            .map_err(|_| VerifyForeignChainError::FailedToComputeMsgHash)?;

        let payload_is_correct = expected_payload_hash == response.payload_hash;

        if !payload_is_correct {
            return Err(VerifyForeignChainError::IncorrectPayloadSigned {
                got: response.payload_hash.clone(),
                expected: expected_payload_hash,
            });
        }
        let verification_result = match (public_key, &response.signature) {
            (
                PublicKey::Secp256k1(secp256k1_public_key),
                SignatureResponse::Secp256k1(k256_signature),
            ) => near_mpc_signature_verifier::verify_ecdsa_signature(
                k256_signature,
                &expected_payload_hash,
                secp256k1_public_key,
            ),
            (PublicKey::Ed25519(ed25519_public_key), SignatureResponse::Ed25519 { signature }) => {
                near_mpc_signature_verifier::verify_eddsa_signature(
                    signature,
                    expected_payload_hash.as_slice(),
                    ed25519_public_key,
                )
            }
            // TODO(#2234): improve types so these errors can't happen
            (PublicKey::Bls12381(_bls12381_g2_public_key), _) => {
                return Err(VerifyForeignChainError::UnexpectedSignatureScheme);
            }
            _ => return Err(VerifyForeignChainError::UnexpectedSignatureScheme),
        };

        verification_result.map_err(|_| VerifyForeignChainError::SignatureVerificationFailed)
    }
