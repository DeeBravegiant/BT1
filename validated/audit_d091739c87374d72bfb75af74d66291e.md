Audit Report

## Title
Stale `rsETHPrice` in `instantWithdrawal()` and `initiateWithdrawal()` Causes Permanent Loss of Accrued Yield — (File: `contracts/LRTWithdrawalManager.sol`)

## Summary
`LRTOracle.rsETHPrice` is a cached storage variable updated only when `updateRSETHPrice()` is explicitly called. Neither `instantWithdrawal()` nor `initiateWithdrawal()` refreshes this price before computing the asset payout via `getExpectedAssetAmount()`. Any user who withdraws while the price is stale receives fewer assets than the current rsETH/asset exchange rate entitles them to; the shortfall permanently accrues to remaining rsETH holders at the withdrawing user's expense.

## Finding Description
`LRTOracle.rsETHPrice` is a plain storage variable set only inside `_updateRsETHPrice()`, which is invoked via the public `updateRSETHPrice()` or the manager-gated `updateRSETHPriceAsManager()`. [1](#0-0) [2](#0-1) 

`getExpectedAssetAmount()` reads this cached value directly without triggering a refresh: [3](#0-2) 

**`instantWithdrawal()` path:** The function computes `assetAmountUnlocked` from the stale price, immediately burns the user's rsETH, and transfers the understated asset amount. No call to `updateRSETHPrice()` precedes this sequence. [4](#0-3) 

The rsETH is burned at a price lower than the true current price; the shortfall stays in the protocol and dilutes the withdrawing user's share.

**`initiateWithdrawal()` path:** `expectedAssetAmount` is computed with the stale price and stored as the ceiling for the user's payout: [5](#0-4) 

Later, `_calculatePayoutAmount()` enforces `min(expectedAssetAmount, currentReturn)`: [6](#0-5) 

Because restaking rewards cause `rsETHPrice` to rise monotonically over time, `currentReturn` at unlock time will normally exceed the stale `expectedAssetAmount`, so the user is permanently capped at the lower stale figure. The rsETH burned in `unlockQueue()` is worth more than the assets disbursed; the surplus accrues to remaining rsETH holders. [7](#0-6) 

No existing guard in either withdrawal function calls or enforces a price refresh. `updateRSETHPrice()` is public but is never invoked automatically within the withdrawal flow.

## Impact Explanation
**High — Theft of unclaimed yield.** In both paths the withdrawing user's accrued yield is not returned to them; it is redistributed to remaining rsETH holders via the inflated per-share value that results from burning rsETH at a below-market price. The loss is permanent and proportional to the staleness of the price and the size of the withdrawal. This matches the allowed impact "High. Theft of unclaimed yield."

## Likelihood Explanation
`updateRSETHPrice()` is not called automatically within any withdrawal path. The price can be stale for hours or days between operator-triggered or bot-triggered updates. Any user who calls `instantWithdrawal()` or `initiateWithdrawal()` without first calling `updateRSETHPrice()` is affected. Because the protocol does not document or enforce this prerequisite, the vast majority of users will not perform this step, making the impact routine rather than edge-case. No special privileges or attacker role is required; a normal withdrawer triggers the loss simply by using the protocol as presented.

## Recommendation
At the start of both `instantWithdrawal()` and `initiateWithdrawal()`, before `getExpectedAssetAmount()` is invoked, call `ILRTOracle(lrtConfig.getContract(LRTConstants.LRT_ORACLE)).updateRSETHPrice()` (or an equivalent internal helper) so that the price used to compute the payout reflects all accrued rewards up to the moment of withdrawal. This ensures the user's rsETH is valued at the current exchange rate before it is burned or committed.

## Proof of Concept
1. `rsETHPrice` was last updated 12 hours ago at `1.05e18` (1.05 ETH per rsETH). Restaking rewards have since pushed the true price to `1.06e18`, but `updateRSETHPrice()` has not been called.
2. Alice calls `instantWithdrawal(ETH, 100e18, "")` without first calling `updateRSETHPrice()`.
3. `getExpectedAssetAmount` returns `100 * 1.05e18 / 1e18 = 105 ETH` (stale price).
4. Alice's 100 rsETH is burned; she receives `105 ETH` (minus fee).
5. Had `updateRSETHPrice()` been called first, she would have received `106 ETH`.
6. Alice permanently loses 1 ETH of accrued yield; the shortfall remains in the protocol, benefiting all other rsETH holders.

For `initiateWithdrawal()`: the stale `expectedAssetAmount = 105 ETH` is stored. When `unlockQueue()` is later called with the updated price of `1.06e18`, `_calculatePayoutAmount` returns `min(105, 106) = 105 ETH`, permanently locking Alice out of her 1 ETH of yield.

**Foundry fork test outline:**
```solidity
// 1. Fork mainnet at a block where rsETHPrice is known.
// 2. Advance time by 12 hours without calling updateRSETHPrice().
// 3. Record lrtOracle.rsETHPrice() (stale) vs. the value that _updateRsETHPrice() would compute.
// 4. Call instantWithdrawal() as Alice; record assets received.
// 5. Reset state; call updateRSETHPrice() then instantWithdrawal() as Alice; record assets received.
// 6. Assert step-5 assets > step-4 assets, confirming yield loss.
```
