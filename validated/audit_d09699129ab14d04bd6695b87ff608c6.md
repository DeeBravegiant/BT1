### Title
Stale Vote Accumulation in `vote_code_hash` Enables TEE Code Hash Whitelisting Below Threshold — (File: `crates/contract/src/tee/proposal.rs`, `crates/contract/src/lib.rs`)

---

### Summary

`CodeHashesVotes::vote()` counts **all** stored votes for a hash without filtering by current participants. After a resharing, stale votes from removed participants are supposed to be purged by the `clean_tee_status` detached promise. If that promise fails, stale votes persist. A single current participant then voting for the same hash can push the raw count over the governance threshold, whitelisting a TEE code hash that was never approved by the required number of current participants. The same flaw exists in `LauncherHashVotes::vote()` and `MeasurementVotes::vote()`.

---

### Finding Description

`CodeHashesVotes::vote()` tallies every entry in `proposal_by_account` that matches the hash, with no participant-membership filter:

```rust
fn count_votes(&self, proposal: &NodeImageHash) -> u64 {
    self.proposal_by_account
        .values()
        .filter(|&prop| prop == proposal)
        .count() as u64          // ← counts ALL stored votes, including stale ones
}
``` [1](#0-0) 

`vote_code_hash` uses this raw count directly against the governance threshold:

```rust
let votes = self.tee_state.vote(code_hash, &participant);
if votes >= self.threshold()?.value() {
    self.tee_state.whitelist_tee_proposal(code_hash, tee_upgrade_deadline_duration);
}
``` [2](#0-1) 

After a resharing completes, `vote_reshared` spawns `clean_tee_status` as a **detached** promise to purge stale votes:

```rust
Promise::new(env::current_account_id())
    .function_call(
        method_names::CLEAN_TEE_STATUS.to_string(),
        ...
        Gas::from_tgas(self.config.clean_tee_status_tera_gas),
    )
    .detach();   // ← failure is silent; stale votes survive
``` [3](#0-2) 

`clean_tee_status` itself calls `clean_non_participant_votes`, which is the only mechanism that removes stale votes: [4](#0-3) 

The developers already recognised this exact failure mode for `vote_update` and added an explicit in-line participant filter with the comment *"This ensures correctness even if the cleanup promise in `MpcContract::vote_reshared()` fails"*:

```rust
let valid_votes_count = running_state
    .parameters.participants().participants().iter()
    .filter(|(account_id, _, _)| {
        self.proposed_updates.vote_by_participant
            .get(account_id)
            .is_some_and(|voted_id| *voted_id == id)
    })
    .count();
``` [5](#0-4) 

No equivalent guard exists in `vote_code_hash`, `vote_add_launcher_hash`, or `vote_add_measurement`. The asymmetry is the root cause.

---

### Impact Explanation

If a malicious TEE image hash is whitelisted, nodes running that image can submit attestations that pass on-chain verification, join the MPC participant set, and receive key-share material during the next DKG or resharing. This constitutes unauthorized access to MPC key shares and signing capability — a Critical-class outcome under the allowed impact scope. The immediate on-chain effect (whitelisting) is a Medium-class accounting invariant break; the downstream consequence (key-share exposure) escalates it.

---

### Likelihood Explanation

The attack requires:
1. T−1 colluding participants (strictly below the governance threshold) voting for a malicious hash before being removed.
2. A resharing that removes those participants (requires threshold agreement from honest nodes, but can be triggered legitimately, e.g., adding new nodes).
3. `clean_tee_status` failing silently — possible if `clean_tee_status_tera_gas` is under-provisioned for the participant count, or if any panic occurs inside the function.
4. One current participant voting for the same hash (could be tricked, or could be a colluding node that survived the resharing).

With N=5, T=3: two malicious participants vote pre-resharing; after cleanup failure, one honest participant's vote reaches the threshold of 3. The required collusion drops from T to 1 honest vote plus T−1 stale votes.

---

### Recommendation

Apply the same in-line participant filter that `vote_update` already uses. In `vote_code_hash` (and analogously in `vote_add_launcher_hash` / `vote_add_measurement`), replace the raw `votes` count with a count restricted to current participants:

```rust
let current_participants = threshold_parameters.participants();
let votes = self.tee_state
    .votes
    .proposal_by_account
    .iter()
    .filter(|(p, h)| {
        *h == code_hash
        && current_participants.is_participant_given_participant_id(&p.get())
    })
    .count() as u64;
```

This makes the threshold check resilient to `clean_tee_status` failures, matching the defensive pattern already present in `vote_update`.

---

### Proof of Concept

**Setup**: N=5 participants {P1, P2, P3, P4, P5}, governance threshold T=3.

1. P1 and P2 (malicious, colluding) call `vote_code_hash(malicious_hash)`. `CodeHashesVotes` now holds 2 votes for `malicious_hash`. Threshold not yet reached (2 < 3).

2. A legitimate resharing removes P1 and P2. New participant set: {P3, P4, P5}.

3. `vote_reshared` fires `clean_tee_status` as a detached promise. The promise fails silently (e.g., `clean_tee_status_tera_gas` is set to a value insufficient for the participant-map iteration). P1 and P2's votes remain in `CodeHashesVotes`.

4. P3 (honest but deceived) calls `vote_code_hash(malicious_hash)`. `CodeHashesVotes::count_votes` returns 3 (P1_stale + P2_stale + P3). The check `3 >= threshold(3)` passes.

5. `whitelist_tee_proposal(malicious_hash, ...)` executes. The malicious hash is now in the allowed list.

6. Nodes running the malicious image submit attestations that pass `submit_participant_info` verification, join the MPC network, and receive key-share material in the next resharing. [6](#0-5) [7](#0-6) [8](#0-7)

    pub fn vote_code_hash(&mut self, code_hash: NodeImageHash) -> Result<(), Error> {
        log!(

            env::signer_account_id(),
            code_hash,
        );
        self.voter_or_panic();

        let threshold_parameters = self.protocol_state.threshold_parameters_or_panic();

        let participant = AuthenticatedParticipantId::new(threshold_parameters.participants())?;

        let tee_upgrade_deadline_duration =

        if votes >= self.threshold()?.value() {
            self.tee_state
                .whitelist_tee_proposal(code_hash, tee_upgrade_deadline_duration);
        }

        Ok(())
    }
