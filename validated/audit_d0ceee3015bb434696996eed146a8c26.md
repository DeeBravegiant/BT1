### Title
`respond_verify_foreign_tx` Does Not Bind `response.payload_hash` to the Submitted Request — Cross-Request Signature Replay by a Single Byzantine Participant - (File: crates/contract/src/lib.rs)

### Summary

The `respond_verify_foreign_tx` contract method verifies that the submitted signature is cryptographically valid over `response.payload_hash`, but it never verifies that `response.payload_hash` is actually the hash of `ForeignTxSignPayload { request: <the pending request>, values: <observed values> }`. Any single attested participant can replay a valid `(payload_hash, signature)` pair from a previously completed foreign-tx request as the response to a completely different pending request. The contract accepts it, drains the pending queue, and delivers the forged response to every waiting caller.

---

### Finding Description

`respond_verify_foreign_tx` enforces two checks before resolving pending yields:

1. The caller is an attested participant (`assert_caller_is_attested_participant_and_protocol_active`).
2. The signature is valid over `response.payload_hash` under the domain's root public key. [1](#0-0) 

What is **never checked** is whether `response.payload_hash` is the SHA-256 of `borsh(ForeignTxSignPayload { request: request.request, values: <anything> })`. The `request` argument is used only as a lookup key into `pending_verify_foreign_tx_requests`; its content is never incorporated into the hash verification. [2](#0-1) 

The signed payload is defined as:

```
msg_hash = SHA-256(borsh(ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: ForeignChainRpcRequest,   // ← includes tx_id, chain, extractors
    values: Vec<ExtractedValue>,       // ← the observed on-chain data
})))
``` [3](#0-2) 

Because `values` are not part of the on-chain request and are not included in `VerifyForeignTransactionResponse`, the contract has no way to reconstruct the expected hash from the `request` alone — and it does not attempt to do so. [4](#0-3) 

Compare with the regular `respond` method, where the payload is embedded in the `SignatureRequest` key itself, so the contract always verifies the signature over the exact payload the user submitted: [5](#0-4) 

No such binding exists for `respond_verify_foreign_tx`.

---

### Impact Explanation

A single Byzantine attested participant can execute the following cross-request replay:

1. Observe any previously completed `respond_verify_foreign_tx` call on-chain for `request_A`, obtaining the public `(payload_hash_A, sig_A)` pair. `payload_hash_A = SHA-256(borsh(ForeignTxSignPayload { request_A, values_A }))`.
2. When `request_B` (a different foreign-chain transaction) is pending, submit `respond_verify_foreign_tx(request_B, { payload_hash: payload_hash_A, signature: sig_A })`.
3. The contract finds `request_B` in `pending_verify_foreign_tx_requests` ✓, verifies `sig_A` over `payload_hash_A` ✓, and resolves all queued yields for `request_B` with `{ payload_hash_A, sig_A }`.

Every caller waiting on `request_B` receives a `VerifyForeignTransactionResponse` whose `payload_hash` is the hash of `request_A`'s observation, not `request_B`'s. The pending entry for `request_B` is permanently drained; no correct response can ever be delivered.

Bridge contracts (e.g., Omnibridge inbound flow) that do not independently recompute the expected `payload_hash` from known extracted values — which is the common case when the bridge does not know the block hash in advance — cannot detect the forgery and will process an invalid bridge execution. Even contracts that do use `ForeignChainSignatureVerifier::verify_signature` from the SDK will receive a mismatched hash and fail, causing a permanent DoS for that request. [6](#0-5) 

---

### Likelihood Explanation

- **Attacker role**: any single attested MPC participant — no threshold collusion required.
- **Source of the replay material**: all `respond_verify_foreign_tx` calls are on-chain NEAR transactions; `payload_hash` and `signature` are public. Any participant (or external observer who bribes/compromises one participant) can collect them.
- **Race window**: the honest leader must query a foreign-chain RPC provider before it can respond (5-second timeout per node). The attacker can submit the forged response immediately after seeing the new `verify_foreign_transaction` receipt, winning the race.
- **No special setup**: the attacker does not need to be the elected leader for `request_B`; `respond_verify_foreign_tx` is callable by any attested participant. [7](#0-6) 

---

### Recommendation

Include the full `ForeignTxSignPayload` (or at minimum `Vec<ExtractedValue>`) in `VerifyForeignTransactionResponse`. In `respond_verify_foreign_tx`, recompute the expected hash on-chain:

```rust
let expected_hash = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: request.request.clone(),
    values: response.values.clone(),   // new field
}).compute_msg_hash()?;

if expected_hash != response.payload_hash {
    return Err(RespondError::PayloadHashMismatch.into());
}
```

Then verify the signature over `expected_hash`. This binds the signed observation to the specific pending request and closes the replay vector entirely.

### Proof of Concept

```
// Step 1 – collect a valid (payload_hash_A, sig_A) from any past on-chain respond call.
payload_hash_A = SHA-256(borsh(ForeignTxSignPayload { request_A, values_A }))
sig_A          = valid ECDSA signature over payload_hash_A under domain root key

// Step 2 – user B submits a new request.
verify_foreign_transaction(request_B)   // request_B.request ≠ request_A.request

// Step 3 – attacker (any attested participant) races the honest leader.
respond_verify_foreign_tx(
    request  = request_B,                          // exists in pending map ✓
    response = { payload_hash: payload_hash_A,     // sig valid ✓, binding NOT checked ✗
                 signature:    sig_A }
)

// Step 4 – contract resolves request_B's yields with the forged response.
// All callers of verify_foreign_transaction(request_B) receive
// { payload_hash_A, sig_A } — a signed attestation of request_A's observation,
// not request_B's.  The pending entry is permanently consumed.
```

The root cause is at `crates/contract/src/lib.rs` lines 718–747: the contract verifies `sig` over `response.payload_hash` but never asserts `response.payload_hash == SHA-256(borsh(ForeignTxSignPayload { request.request, … }))`. [8](#0-7) [9](#0-8)

    pub fn compute_msg_hash(&self) -> std::io::Result<Hash256> {

        Ok(Hash256(hasher.finalize().into()))
