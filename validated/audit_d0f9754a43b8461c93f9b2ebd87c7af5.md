Audit Report

## Title
Unprotected `_withdrawFromAave` Call in `_processWithdrawalCompletion` Causes DoS on ETH Withdrawals - (File: `contracts/LRTWithdrawalManager.sol`)

## Summary
When Aave integration is enabled, `_processWithdrawalCompletion` calls `_withdrawFromAave` with no try/catch protection. If Aave's pool is paused or WETH liquidity is exhausted, `aaveWETHGateway.withdrawETH` reverts, causing every user's `completeWithdrawal` call to revert. This temporarily freezes ETH withdrawals for all users who have already burned their rsETH via `initiateWithdrawal`. The asymmetry is confirmed: the deposit path in `unlockQueue` wraps its Aave call in a try/catch, but the withdrawal path does not.

## Finding Description
In `_processWithdrawalCompletion` (line 699), when `isAaveIntegrationEnabled` is true and the contract's ETH balance is below the user's expected withdrawal amount, `_withdrawFromAave` is called at line 724 with no error handling: [1](#0-0) 

`_withdrawFromAave` calls `aaveWETHGateway.withdrawETH` directly at line 917: [2](#0-1) 

If Aave's pool is paused or WETH utilization is near 100%, this call reverts and propagates up through `_processWithdrawalCompletion` to `completeWithdrawal`, blocking all ETH withdrawal completions. Because `unlockQueue` deposits unlocked ETH into Aave (line 311), the contract's own ETH balance is structurally kept low, making the Aave withdrawal path the critical dependency for every `completeWithdrawal` call.

The deposit path already has the correct protection: [3](#0-2) 

No equivalent guard exists for the withdrawal path.

## Impact Explanation
Users who have already initiated ETH withdrawals (burning rsETH via `initiateWithdrawal`) cannot complete their withdrawal via `completeWithdrawal` while Aave is paused or illiquid. Their ETH is temporarily frozen in the protocol until Aave's state normalizes. This matches **Medium — Temporary freezing of funds**.

## Likelihood Explanation
Aave v3 has a guardian-controlled global pause that has been exercised on mainnet. During market stress, WETH utilization can spike to near 100%, making `withdrawETH` revert. When `isAaveIntegrationEnabled` is true, the `unlockQueue` flow structurally deposits idle ETH into Aave, making the contract dependent on Aave for every subsequent ETH `completeWithdrawal`. Any normal user who has queued a withdrawal is affected. Likelihood is **Medium**.

## Recommendation
Wrap the `_withdrawFromAave` call in `_processWithdrawalCompletion` in a try/catch, mirroring the pattern already used in `unlockQueue`:

```solidity
try this.withdrawFromAaveExternal(amountNeeded) {
} catch {
    revert AaveWithdrawalFailed();
}
```

This surfaces a clear revert reason so users and operators know to retry once Aave is operational, rather than having the call silently propagate an opaque Aave revert.

## Proof of Concept
1. `isAaveIntegrationEnabled` is `true`; operator has called `unlockQueue`, depositing idle ETH into Aave via the try/catch-protected path at line 311.
2. User calls `initiateWithdrawal(ETH_TOKEN, rsETHAmount, ...)`, burning rsETH; request is queued.
3. Operator calls `unlockQueue`; ETH is redeemed from `LRTUnstakingVault` and deposited to Aave.
4. Aave's pool is paused (guardian action) or WETH utilization reaches ~100%.
5. User calls `completeWithdrawal(ETH_TOKEN, ...)`.
6. `_processWithdrawalCompletion` detects `address(this).balance < request.expectedAssetAmount` (line 722).
7. `_withdrawFromAave(amountNeeded)` is called (line 724) → `aaveWETHGateway.withdrawETH(...)` at line 917 reverts.
8. Revert propagates; user cannot retrieve their ETH until Aave resumes normal operation.

A Foundry fork test can reproduce this by forking mainnet, enabling Aave integration, depositing ETH to Aave, calling `vm.prank(aaveGuardian); aavePool.setPoolPause(true)`, then asserting that `completeWithdrawal` reverts.

    function _withdrawFromAave(uint256 amount) internal returns (uint256 withdrawnAmount) {
        if (amount == 0) return 0;

        uint256 aaveBalance = aaveAWETH.balanceOf(address(this));
        if (aaveBalance == 0) revert InsufficientAaveBalance();

        // Only withdraw up to the principal amount (don't use accrued interest for user withdrawals)
        uint256 withdrawablePrincipal = aaveBalance < totalETHDepositedToAave ? aaveBalance : totalETHDepositedToAave;

        withdrawnAmount = amount > withdrawablePrincipal ? withdrawablePrincipal : amount;
        if (withdrawnAmount == 0) return 0;

        aaveWETHGateway.withdrawETH(aavePool, withdrawnAmount, address(this));
        totalETHDepositedToAave -= withdrawnAmount;

        emit ETHWithdrawnFromAave(withdrawnAmount, totalETHDepositedToAave);
    }
