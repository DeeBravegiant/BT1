### Title
Single Byzantine Participant Can Deliver Malicious Confidential Key via Unverified `respond_ckd` for `AppPublicKey` Variant - (File: `crates/contract/src/lib.rs`)

---

### Summary

The `respond_ckd` function in the MPC contract performs **no cryptographic verification** of the CKD response when the request uses the `AppPublicKey` (privately verifiable) variant. A single Byzantine participant — strictly below the signing threshold — can race to submit an arbitrary malicious `CKDResponse` before honest nodes, and the contract will unconditionally accept and deliver it to the user. This bypasses the threshold security model for confidential key derivation.

---

### Finding Description

The `respond_ckd` function handles two variants of `CKDAppPublicKey`:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}   // ← NO CHECK
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV` (publicly verifiable), the contract calls `ckd_output_check()` to verify the response against the MPC public key before accepting it. For `AppPublicKey` (privately verifiable), the branch is a **no-op** — the response is accepted unconditionally and immediately delivered to the user via `resolve_yields_for`. [2](#0-1) 

Contrast this with the `respond` function for signature requests, which always verifies the signature cryptographically before accepting it:

```rust
if !signature_is_valid {
    return Err(RespondError::InvalidSignature.into());
}
``` [3](#0-2) 

The `respond_ckd` function requires only **one** attested participant to call it — there is no threshold voting for responses. A single Byzantine participant can therefore:

1. Monitor the contract for pending CKD requests using the `AppPublicKey` variant.
2. Call `respond_ckd` with an arbitrary `CKDResponse` (wrong `big_y`, wrong `big_c`) before honest nodes submit the correct output.
3. The contract passes the `assert_caller_is_attested_participant_and_protocol_active()` check, finds no verification branch for `AppPublicKey`, and calls `resolve_yields_for` — delivering the malicious key to the user. [4](#0-3) 

Honest participants who subsequently call `respond_ckd` with the correct output are rejected because the yield slot is already resolved.

---

### Impact Explanation

The user receives a malicious confidential key. Because the `AppPublicKey` variant is privately verifiable (only the app can verify after decryption), the attack is undetectable on-chain. The delivered `(big_y, big_c)` pair is attacker-controlled, meaning:

- The user's derived secret is wrong or known to the attacker.
- Any funds or cryptographic material protected by that key are at risk.
- The correct key is permanently lost for that request (the yield is consumed).

This constitutes **unauthorized confidential key derivation output without the required threshold participant authorization** — matching the Critical impact tier.

---

### Likelihood Explanation

- The attacker must be a single attested participant, strictly below the signing threshold — explicitly listed as a valid attacker profile in the objective.
- `AppPublicKey` (privately verifiable) is a documented, production-supported variant per the CKD design.
- The attack is a simple race: submit a malicious `respond_ckd` call before honest nodes. No threshold collusion, no leaked keys, no operator access required.
- The window is bounded only by NEAR block time; a participant running a modified node can submit immediately upon seeing the request event.

---

### Recommendation

1. **Mandate `AppPublicKeyPV`** for all production CKD requests, since it enables on-chain verification via `ckd_output_check`. Reject `AppPublicKey` variant at the contract level.
2. **Or** introduce threshold voting for CKD responses (analogous to how threshold signatures require all nodes to agree before a valid signature can be produced), so a single Byzantine participant cannot unilaterally resolve a CKD request.
3. **Or** add a commitment scheme (e.g., require the response to include a BLS proof verifiable against the MPC public key) that allows the contract to reject malformed responses for the `AppPublicKey` variant.

---

### Proof of Concept

```
1. User calls request_app_private_key({
       derivation_path: "m/0",
       app_public_key: AppPublicKey(<user_bls_pk>),   // privately verifiable
       domain_id: 0
   }) with 1 yoctonear deposit.

2. Byzantine participant (attested, participant ID < threshold) observes the
   pending CKD request on-chain.

3. Byzantine participant calls respond_ckd(
       request = <the pending CKDRequest>,
       response = CKDResponse {
           big_y: [0xAA; 48],   // arbitrary wrong encrypted key
           big_c: [0xBB; 48],   // arbitrary wrong commitment
       }
   ).

4. Contract execution path:
   - assert_caller_is_signer()                          → passes (signer == predecessor)
   - is_running_or_resharing()                          → passes
   - accept_requests                                    → passes
   - assert_caller_is_attested_participant_and_protocol_active() → passes
   - match AppPublicKey(_) => {}                        → NO VERIFICATION
   - resolve_yields_for(...)                            → delivers malicious key to user

5. User receives CKDResponse { big_y: [0xAA;48], big_c: [0xBB;48] }.
   Honest nodes' subsequent respond_ckd calls are rejected (yield already resolved).
   User's derived secret is attacker-controlled.

    #[handle_result]
    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }
