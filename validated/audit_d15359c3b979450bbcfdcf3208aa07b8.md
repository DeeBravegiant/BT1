Audit Report

## Title
ETH Temporarily Frozen in L1VaultV2 via Permissionless `receive()` Inflating Balance Past Deposit Limit — (`contracts/L1VaultV2.sol`)

## Summary

`L1VaultV2.receive()` accepts ETH from any caller with no access control. `depositETHForL1VaultETH()` always deposits `address(this).balance` in full. An attacker can send 1 wei to the vault when the remaining ETH deposit headroom in `LRTDepositPool` is less than the vault's current balance, causing the subsequent `depositETH` call to revert with `MaximumDepositLimitReached`. Because the vault has no partial-deposit path and no ETH rescue function, all vault ETH is frozen until an admin raises the deposit cap.

## Finding Description

**Permissionless ETH entry point.** `L1VaultV2.receive()` has no access control: [1](#0-0) 

Any address can call `address(vault).transfer(1 wei)`.

**Full-balance deposit with no partial option.** `depositETHForL1VaultETH()` unconditionally uses `address(this).balance`: [2](#0-1) 

There is no `amount` parameter; the manager cannot choose to deposit less than the full balance.

**ETH limit check includes the incoming ETH.** `_checkIfDepositAmountExceedesCurrentLimit` for ETH reads `getTotalAssetDeposits(ETH)`, which calls `getETHDistributionData()`. That function reads `address(this).balance` of the deposit pool *after* the ETH has already arrived (since `depositETH` is `payable`): [3](#0-2) 

The ETH-specific branch of the limit check is: [4](#0-3) 

For ETH, `amount` is not added separately — it is already reflected in `totalAssetDeposits` via `address(this).balance`. So the effective check is `(priorSystemETH + vaultBalance) > depositLimit`. If the attacker's dust tip pushes `vaultBalance` over the remaining headroom, this returns `true`.

**Revert propagates, ETH is stuck.** [5](#0-4) 

The vault has no partial-deposit path and no ETH withdrawal/rescue function, so all ETH (including legitimately bridged L2 ETH) is frozen until an admin increases `depositLimitByAsset(ETH)`.

## Impact Explanation

This is a **temporary freeze of funds (Medium)**. All ETH sitting in `L1VaultV2` — including ETH legitimately bridged from L2 — cannot be deposited into `LRTDepositPool` and therefore cannot be converted to rsETH and bridged back. The freeze persists until an admin governance action raises the deposit cap. This matches the allowed impact: *Medium — Temporary freezing of funds*.

## Likelihood Explanation

- The attacker needs only to send 1 wei to the vault when the remaining ETH deposit headroom is less than the vault's current balance.
- No role, approval, or special access is required.
- The condition (deposit limit nearly exhausted) is a normal operational state for a live protocol.
- Cost to the attacker is negligible (1 wei + gas).
- The attack is repeatable: every time the admin raises the limit, the attacker can re-trigger it.

## Recommendation

1. **Add an `amount` parameter** to `depositETHForL1VaultETH()` so the manager can deposit exactly the amount that fits within the remaining limit, rather than always using `address(this).balance`.
2. **Add an admin-only ETH rescue/sweep function** to allow recovery of ETH that cannot be deposited due to limit constraints.
3. Alternatively, inside `depositETHForL1VaultETH()`, cap `balanceOfETH` to `lrtDepositPool.getAssetCurrentLimit(ETH_IDENTIFIER)` before calling `depositETH`.

## Proof of Concept

```
Preconditions:
  depositLimit = 100 ETH
  currentSystemETH = 99 ETH  (1 ETH headroom)
  vault.balance = 1 ETH      (legitimate L2 bridge deposit)

Attack:
  attacker.sendValue(vault, 1 wei);   // vault.balance = 1 ETH + 1 wei

Manager action:
  vault.depositETHForL1VaultETH();
  // balanceOfETH = 1 ETH + 1 wei
  // depositPool.depositETH{value: 1 ETH + 1 wei}(...)
  // address(depositPool).balance now = 99 ETH + 1 ETH + 1 wei = 100 ETH + 1 wei
  // totalAssetDeposits (ETH) = 100 ETH + 1 wei
  // _checkIfDepositAmountExceedesCurrentLimit: 100 ETH + 1 wei > 100 ETH → true
  // revert MaximumDepositLimitReached

Result: depositETHForL1VaultETH() reverts; vault.balance unchanged at 1 ETH + 1 wei.
        All vault ETH is frozen until admin raises depositLimitByAsset(ETH).
```

A Foundry fork test can confirm this by: (1) forking mainnet, (2) setting `depositLimitByAsset(ETH)` to `currentSystemETH + vaultBalance - 1 wei`, (3) calling `vault.transfer(1 wei)` from an attacker EOA, (4) calling `depositETHForL1VaultETH()` from the manager role, and asserting the call reverts with `MaximumDepositLimitReached` while `vault.balance` remains unchanged.
