### Title
Blacklisted USDC/USDT EVM Recipient Permanently Locks Bridged Funds - (File: evm/src/omni-bridge/contracts/OmniBridge.sol)

### Summary
When a user bridges a blacklistable ERC-20 token (USDC, USDT) from NEAR back to Ethereum and the destination EVM address is or becomes blacklisted, the `finTransfer` function in `OmniBridge.sol` will always revert. Because the recipient is embedded in an MPC-signed payload that cannot be altered, and no admin ERC-20 rescue path exists in the EVM contract, the EVM-side tokens are permanently locked. The user's NEAR-side wrapped tokens are already burned at `init_transfer` time, so the loss is irreversible.

### Finding Description

**Bug class:** State-machine / asset-accounting — a token transfer to a blacklisted address causes the settlement function to revert permanently, with no alternative redemption path.

**EVM settlement path (`OmniBridge.sol::finTransfer`):**

The nonce is marked used before the token transfer is attempted:

```solidity
completedTransfers[payload.destinationNonce] = true;   // line 287
```

Then, for a native ERC-20 (e.g., USDC locked in the bridge when the user originally bridged EVM→NEAR), the transfer is:

```solidity
IERC20(payload.tokenAddress).safeTransfer(
    payload.recipient,
    payload.amount
);   // lines 351-354
```

If `payload.recipient` is on USDC's or USDT's blacklist, `safeTransfer` reverts. Because Solidity reverts roll back all state changes, `completedTransfers` is also rolled back, so the nonce is never consumed. The relayer can retry indefinitely, but every attempt reverts for the same reason.

**Why the recipient cannot be changed:**

`sign_transfer` on the NEAR side constructs the `TransferMessagePayload` with the recipient taken directly from the stored `TransferMessage`:

```rust
let transfer_payload = TransferMessagePayload {
    // ...
    recipient: transfer_message.recipient,   // fixed at init_transfer time
    // ...
};
```

The MPC signs this exact payload. There is no mechanism to re-sign with a different recipient without modifying the stored `TransferMessage`, which no public function permits.

**Why NEAR-side tokens are already gone:**

`init_transfer_internal` burns deployed tokens (NEAR-side wrapped USDC) immediately and returns `U128(0)` to the `ft_transfer_call` caller, consuming all tokens:

```rust
self.burn_tokens_if_needed(token_id.clone(), transfer_message.amount);
// ...
U128(0)   // all tokens consumed
```

**No ERC-20 rescue function exists in `OmniBridge.sol`:**

The contract exposes no admin function to recover stuck ERC-20 tokens. The only egress for locked ERC-20s is `finTransfer`, which permanently fails for a blacklisted recipient.

### Impact Explanation

**Critical — Irreversible fund lock.**

- The user's NEAR-side wrapped USDC is burned at `init_transfer` time and cannot be recovered.
- The EVM-side USDC remains locked in `OmniBridge.sol` with no permissionless or admin-accessible ERC-20 withdrawal path.
- The `TransferMessage` persists in `pending_transfers` on NEAR indefinitely; `sign_transfer` can be called repeatedly but every resulting `finTransfer` call on EVM reverts.
- The only theoretical mitigation requires a UUPS upgrade of the EVM contract by the admin, which is a privileged out-of-band action not guaranteed to be available.

### Likelihood Explanation

**Low-Medium.** USDC and USDT both maintain active blacklists. A user can be blacklisted by Circle or Tether at any time, including after a bridge transfer is initiated but before `finTransfer` is executed on EVM. Additionally, a user may inadvertently specify a blacklisted address as the EVM recipient. The bridge imposes no pre-flight check on whether the EVM recipient is blacklisted before burning NEAR-side tokens.

### Recommendation

1. **Pre-flight check (NEAR side):** Before burning NEAR-side tokens in `init_transfer_internal`, verify (or at minimum warn) that the destination token is not a known blacklistable asset when the recipient is an EVM address.
2. **Recipient override / cancellation (NEAR side):** Add a DAO-accessible or user-accessible function to update the recipient in a `pending_transfers` entry and re-sign, or to cancel the transfer and re-mint the NEAR-side tokens.
3. **ERC-20 rescue (EVM side):** Add an admin-only `rescueERC20` function in `OmniBridge.sol` that can redirect stuck tokens to an alternative address when `finTransfer` is permanently blocked.
4. **Token allowlist (EVM side):** Restrict `finTransfer` to a set of pre-approved, non-blacklistable tokens, analogous to the fix applied in the referenced Irrigation Protocol commit.

### Proof of Concept

1. Alice holds 1,000 USDC on NEAR (a deployed bridge token, originally bridged from Ethereum).
2. Alice calls `ft_transfer_call` on the NEAR USDC contract, routing through the bridge with `recipient = OmniAddress::Eth(alice_eth_address)`.
3. `init_transfer_internal` burns Alice's 1,000 NEAR-USDC immediately and stores the `TransferMessage` in `pending_transfers`.
4. Circle blacklists `alice_eth_address` (e.g., due to a compliance action).
5. Relayer calls `sign_transfer` on NEAR → MPC signs a `TransferMessagePayload` with `recipient = alice_eth_address`.
6. Relayer calls `finTransfer` on `OmniBridge.sol` with the signed payload.
7. `completedTransfers[nonce] = true` is set, then `USDC.safeTransfer(alice_eth_address, 1000e6)` reverts due to the blacklist. The entire transaction reverts.
8. Steps 6–7 can be repeated indefinitely; every attempt reverts.
9. Alice's 1,000 NEAR-USDC is burned (gone). The 1,000 EVM-USDC is locked in `OmniBridge.sol` with no recovery path. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
