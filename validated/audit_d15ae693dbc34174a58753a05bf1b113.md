### Title
Sub-Transaction ExUnits Excluded from Fee Calculation and Block Resource Limit Enforcement â€” (`eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Utxo.hs`, `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Bbody.hs`)

---

### Summary

In the Dijkstra era, a top-level transaction may embed sub-transactions (`dtbrSubTransactions`), each carrying its own Plutus redeemers and ExUnits. The `totExUnits` function only folds over the **top-level** transaction's redeemer map. Consequently, sub-transaction ExUnits are invisible to (1) the minimum-fee calculation, (2) the per-transaction ExUnits limit check, and (3) the per-block ExUnits limit check. An unprivileged sender can craft a batch whose sub-transactions declare arbitrarily large ExUnits, consuming script-execution resources while paying a fee computed as if no scripts run, and without being constrained by `maxTxExUnits` or `maxBlockExUnits`.

---

### Finding Description

**Root cause â€” `totExUnits` ignores sub-transaction redeemers**

`totExUnits` is defined as a fold over the top-level transaction's redeemer map only:

```haskell
totExUnits tx = foldMap snd $ tx ^. witsTxL . rdmrsTxWitsL . unRedeemersL
``` [1](#0-0) 

Sub-transactions live in `dtbrSubTransactions :: !(OMap TxId (Tx SubTx era))` inside the top-level body and carry their own `witsTxL . rdmrsTxWitsL`. Those redeemers are never reached by `totExUnits`. [2](#0-1) 

**Sub-transactions provably carry redeemers that trigger script execution**

`validateBatchCollateral` explicitly checks whether *any* sub-transaction has redeemers and, if so, enforces collateral on the top-level transaction:

```haskell
hasAnyRedeemers t =
  hasRedeemers t || any hasRedeemers (t ^. bodyTxL . subTransactionsTxBodyL)
hasRedeemers = not . null . (^. witsTxL . rdmrsTxWitsL . unRedeemersL)
``` [3](#0-2) 

The collateral guard exists precisely because sub-transaction Plutus scripts are executed. Yet the three resource-accounting paths all use `totExUnits`, which sees none of those ExUnits.

**Gap 1 â€” Minimum fee does not include sub-transaction script fees**

`alonzoMinFeeTx` (the base used by `getConwayMinFeeTx`, which Dijkstra inherits) computes the script-fee component as:

```haskell
txscriptfee (pp ^. ppPricesL) allExunits
  where allExunits = totExUnits tx
``` [4](#0-3) 

Sub-transaction ExUnits contribute zero to `allExunits`, so the fee charged for their script execution is zero regardless of how large those ExUnits are.

**Gap 2 â€” Per-transaction ExUnits limit does not cover sub-transactions**

`dijkstraUtxoTransition` enforces:

```haskell
{- totExunits tx â‰¤ maxTxExUnits pp -}
runTest $ Alonzo.validateExUnitsTooBigUTxO pp tx
``` [5](#0-4) 

`validateExUnitsTooBigUTxO` compares `totExUnits tx` against `maxTxExUnits`:

```haskell
totalExUnits = totExUnits tx
``` [6](#0-5) 

A top-level transaction with zero top-level redeemers but N sub-transactions each declaring `maxTxExUnits` passes this check unconditionally.

**Gap 3 â€” Per-block ExUnits limit does not cover sub-transactions**

`dijkstraBbodyTransition` calls:

```haskell
Alonzo.validateExUnits @era txs $ pp ^. ppMaxBlockExUnitsL
``` [7](#0-6) 

`validateExUnits` accumulates `foldMap totExUnits txs`:

```haskell
let txTotal = foldMap totExUnits txs
``` [8](#0-7) 

Because `totExUnits` ignores sub-transaction redeemers, a block filled with batched transactions whose sub-transactions each carry `maxTxExUnits` of ExUnits reports a block total of zero, passing the block limit check.

**Collateral is also under-sized**

`validateBatchCollateral` validates collateral against the top-level `txfee`:

```haskell
Babbage.validateTotalCollateral pp (tx ^. bodyTxL) utxoCollateral
``` [9](#0-8) 

Since `txfee` was computed without sub-transaction script fees, the collateral percentage is applied to an under-stated fee, leaving the protocol under-compensated even in the script-failure path.

---

### Impact Explanation

An unprivileged transaction sender can submit Dijkstra-era batches in which sub-transactions declare arbitrarily large ExUnits. The ledger accepts these transactions while:

- Charging a fee that omits the script-execution component for all sub-transaction scripts (fees modified outside design parameters).
- Allowing a single batch to consume multiples of `maxTxExUnits` in script execution without triggering `ExUnitsTooBigUTxO` (per-transaction validation limit exceeded).
- Allowing a block to contain effectively unbounded total script execution work without triggering `TooManyExUnits` (per-block validation limit exceeded).

This matches the **Medium** allowed impact: *attacker-controlled transactions exceed intended validation limits or modify fees outside design parameters*.

---

### Likelihood Explanation

Any user who can submit a valid Dijkstra-era transaction can exploit this. No privileged role, governance majority, or key compromise is required. The only prerequisite is the ability to construct a top-level transaction with embedded sub-transactions that carry Plutus redeemers â€” a standard operation in the Dijkstra era. The attack is directly analogous to the Optimism zero-gas-limit deposit: the resource-metering path is bypassed by placing the resource consumption one level below where the check is applied.

---

### Recommendation

1. Introduce a `batchTotExUnits` function that folds over both the top-level transaction's redeemers and all sub-transaction redeemers:
   ```haskell
   batchTotExUnits tx =
     totExUnits tx
       <> foldMap totExUnits (tx ^. bodyTxL . subTransactionsTxBodyL)
   ```
2. Replace `totExUnits tx` with `batchTotExUnits tx` in:
   - `alonzoMinFeeTx` (or the Dijkstra override of `getMinFeeTx`) so the fee covers sub-transaction script execution.
   - `validateExUnitsTooBigUTxO` when called from `dijkstraUtxoTransition` so the per-transaction limit is enforced over the whole batch.
   - `validateExUnits` in `dijkstraBbodyTransition` so the per-block limit accounts for all script work in the block.

---

### Proof of Concept

1. Construct a Dijkstra top-level transaction with **no** top-level redeemers (`totExUnits tx = ExUnits 0 0`).
2. Embed K sub-transactions, each with a Plutus redeemer declaring `ExUnits maxTxExUnitsMem maxTxExUnitsSteps`.
3. Set `txfee` to the value returned by `getMinFeeTxUtxo` â€” which, because `totExUnits tx = 0`, omits all script fees.
4. Provide collateral sized to `collateralPercent % * txfee` (also under-stated).
5. Submit the transaction.

**Expected (correct) behaviour**: rejected with `ExUnitsTooBigUTxO` and `FeeTooSmallUTxO`.

**Actual behaviour**: accepted. `validateExUnitsTooBigUTxO` sees `ExUnits 0 0 â‰¤ maxTxExUnits` and passes; `feesOK` sees `minfee = size_fee + 0` and passes; `validateExUnits` in BBODY accumulates zero across all top-level transactions and passes. Sub-transaction Plutus scripts are executed (collateral guard fires on sub-transaction redeemers per `validateBatchCollateral`) but the protocol receives no compensation for that execution work.
