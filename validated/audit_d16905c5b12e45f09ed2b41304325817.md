Audit Report

## Title
Reward Tokens Permanently Locked When `totalKernelStaked` Drops to Zero Mid-Period - (File: contracts/KERNEL/KernelDepositPool.sol)

## Summary
`KernelDepositPool` uses a Synthetix-style reward accounting model where `rewardPerToken()` halts accumulation when `totalKernelStaked == 0`. Any unprivileged staker can call `initiateWithdrawal` for their full balance, immediately zeroing `totalKernelStaked` and freezing reward accumulation for the remainder of the active period. The stranded reward tokens have no on-chain recovery path.

## Finding Description
`rewardPerToken()` returns the frozen `rewardPerTokenStored` whenever `totalKernelStaked == 0`:

```solidity
// contracts/KERNEL/KernelDepositPool.sol L408-414
function rewardPerToken() public view returns (uint256) {
    if (totalKernelStaked == 0) {
        return rewardPerTokenStored;   // accumulation halts
    }
    return rewardPerTokenStored + (rewardRate * (lastTimeRewardApplicable() - updatedAt) * DECIMAL_PRECISION)
        / totalKernelStaked;
}
```

`initiateWithdrawal` decrements `totalKernelStaked` immediately at call time — before the `withdrawalDelay` elapses — meaning the accounting damage occurs the moment the withdrawal is initiated, not when KERNEL tokens are transferred:

```solidity
// contracts/KERNEL/KernelDepositPool.sol L325-326
balanceOf[msg.sender] -= _amount;
totalKernelStaked -= _amount;   // can reach 0 immediately
```

The guard in `notifyRewardAmount` only prevents *starting* a new reward period with zero stakers; it does not prevent the supply from draining to zero during an active period, and it also blocks the admin from starting a new period to "absorb" the stranded tokens:

```solidity
// contracts/KERNEL/KernelDepositPool.sol L570
if (totalKernelStaked == 0) revert NoStakedTokens();
```

The contract's own NatSpec (L18–23) explicitly acknowledges this limitation and relies entirely on an off-chain operational assumption ("ensuring there are always some tokens staked … for the entire duration of the reward period"), with no on-chain enforcement. `KernelDepositPool` does not inherit `Recoverable` and has no admin function to sweep stranded `rewardsToken` balances. The four admin functions (`setRewardsDuration`, `notifyRewardAmount`, `setWithdrawalDelay`, `setMaxNumberOfWithdrawalsPerUser`) provide no recovery path.

## Impact Explanation
**Medium — Permanent freezing of unclaimed yield.**

`rewardsToken` pre-funded via `notifyRewardAmount` for an active reward period becomes permanently unclaimable whenever `totalKernelStaked` reaches zero mid-period. The tokens remain in the contract balance but are never credited to any user and cannot be withdrawn by any role. This is a direct, irreversible loss of yield owed to stakers, matching the allowed impact "Medium. Permanent freezing of unclaimed yield."

## Likelihood Explanation
The scenario is reachable by any unprivileged staker with no special permissions, front-running, or external conditions. If a single user holds the entire staked supply (common in early protocol stages or after mass withdrawals), they can call `initiateWithdrawal` for their full balance at any time, immediately dropping `totalKernelStaked` to zero. The `withdrawalDelay` does not prevent the accounting damage — it only delays the KERNEL token transfer, while `totalKernelStaked` is decremented immediately at `initiateWithdrawal` time.

## Recommendation
Add an admin-only `recoverStrandedRewards` function (or inherit `Recoverable`) that allows the admin to sweep `rewardsToken` balance in excess of what is owed to current stakers (`totalKernelStaked * rewardPerToken() / DECIMAL_PRECISION`). Alternatively, enforce the invariant on-chain by reverting `initiateWithdrawal` when it would bring `totalKernelStaked` to zero while a reward period is active (`block.timestamp < finishAt`), or by tracking and crediting "unallocated" reward tokens to a treasury address whenever `totalKernelStaked` is zero.

## Proof of Concept
1. Admin calls `notifyRewardAmount(1_000e18)` with `duration = 7 days`. `rewardRate = 1_000e18 / 7 days`. `finishAt = now + 7 days`.
2. Alice is the only staker: `balanceOf[Alice] = 100e18`, `totalKernelStaked = 100e18`.
3. After 1 day, Alice calls `initiateWithdrawal(100e18)`. `totalKernelStaked` becomes `0` immediately.
4. For the remaining 6 days, `rewardPerToken()` returns the frozen `rewardPerTokenStored`. No rewards accumulate.
5. `rewardRate * 6 days ≈ 857e18` reward tokens are stuck in the contract with no claimant.
6. Admin cannot call `notifyRewardAmount` to start a new period (reverts with `NoStakedTokens`). No recovery function exists.

**Foundry test plan:**
```solidity
function test_rewardsLockedOnZeroStake() public {
    // Setup: fund reward, Alice stakes
    vm.prank(admin);
    pool.notifyRewardAmount(1_000e18);
    // Warp 1 day
    vm.warp(block.timestamp + 1 days);
    // Alice withdraws all
    vm.prank(alice);
    pool.initiateWithdrawal(100e18);
    assertEq(pool.totalKernelStaked(), 0);
    // Warp to end of period
    vm.warp(block.timestamp + 6 days);
    // Verify rewardPerTokenStored is frozen (no new accumulation)
    uint256 rptBefore = pool.rewardPerTokenStored();
    vm.warp(block.timestamp + 1);
    assertEq(pool.rewardPerToken(), rptBefore);
    // Verify ~857e18 tokens are stranded in contract
    assertGt(rewardsToken.balanceOf(address(pool)), 800e18);
}
```
