Audit Report

## Title
Systematic Round-Trip Loss Due to Double Truncation in Deposit and Withdrawal Conversions - (`contracts/LRTDepositPool.sol` / `contracts/LRTWithdrawalManager.sol`)

## Summary
Both the deposit minting path and the withdrawal redemption path apply integer division that truncates toward zero. Because truncation is applied in both directions, a user who deposits an asset and later withdraws the resulting rsETH will receive strictly fewer underlying tokens than they deposited whenever the division is inexact. The truncated remainder silently accrues to the protocol's TVL, benefiting remaining rsETH holders at the expense of the withdrawing user.

## Finding Description
**Deposit path** — `LRTDepositPool.getRsETHAmountToMint` (line 520):
```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```
The user receives `floor(amount × assetPrice / rsETHPrice)` rsETH. Any fractional remainder is discarded.

**Withdrawal path** — `LRTWithdrawalManager.getExpectedAssetAmount` (line 593):
```solidity
underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset);
```
The user receives `floor(rsETHAmount × rsETHPrice / assetPrice)` of the underlying asset. A second truncation is applied.

**Payout at unlock** — `LRTWithdrawalManager._calculatePayoutAmount` (line 833):
```solidity
uint256 currentReturn = (request.rsETHUnstaked * rsETHPrice) / assetPrice;
```
A third truncation is applied at unlock time, and the function returns the minimum of `expectedAssetAmount` and `currentReturn`, so the user can never recover the truncated amount.

The `minRSETHAmountExpected` slippage guard on deposit (line 667–668) only protects against the first truncation if the user computes the minimum correctly; it provides no protection against the second and third truncations applied at withdrawal time.

**Root cause**: Both conversion directions truncate, so the composition `withdraw(deposit(X))` yields `floor(floor(X × p) / p) ≤ X − 1` for any non-exact ratio `p = assetPrice / rsETHPrice`. The truncated dust inflates `totalETHInProtocol` without a corresponding rsETH liability, permanently benefiting remaining holders.

## Impact Explanation
**Low — Contract fails to deliver promised returns, but does not lose value.**

Every user who completes a full deposit→withdraw round-trip receives strictly fewer underlying tokens than they deposited whenever the division is inexact. At normal deposit sizes (≥ 1e18 wei), the per-operation loss is bounded at 1–2 wei, but it is systematic and irreversible. The truncated value accrues to the protocol TVL, inflating the rsETH price for remaining holders. No funds are stolen outright, but the protocol does not deliver the full round-trip value it implicitly promises through its price-based conversion model.

## Likelihood Explanation
**High.** The truncation fires on every deposit/withdrawal cycle where `amount × assetPrice` is not exactly divisible by `rsETHPrice`. Because `rsETHPrice` is a continuously updated WAD-precision value (`(totalETHInProtocol − protocolFeeInETH).divWad(rsethSupply)`, line 250 of `LRTOracle.sol`), exact divisibility is essentially never guaranteed in practice. Any unprivileged user who calls `depositAsset` followed by `initiateWithdrawal` and `completeWithdrawal` triggers the loss with no special preconditions.

## Recommendation
- **Short term**: Change `getExpectedAssetAmount` and `_calculatePayoutAmount` to use ceiling division (e.g., OpenZeppelin `Math.mulDiv(amount, rsETHPrice, assetPrice, Math.Rounding.Up)`) for the asset-out calculation. This ensures the protocol never over-promises and the user receives at least as much as the rsETH is worth.
- **Short term**: Document the rounding behaviour clearly so integrators and users can account for the ≤ 1 wei per operation loss.
- **Long term**: Add invariant fuzz tests asserting `getExpectedAssetAmount(asset, getRsETHAmountToMint(asset, X)) >= X − 1` for all valid inputs.

## Proof of Concept
1. Alice calls `LRTDepositPool.depositAsset(stETH, 5, 0, "")` when `assetPrice = 1e18`, `rsETHPrice = 3e18`.
2. `getRsETHAmountToMint` returns `floor(5 × 1e18 / 3e18) = 1`. Alice receives 1 rsETH.
3. Alice calls `LRTWithdrawalManager.initiateWithdrawal(stETH, 1)`.
4. `getExpectedAssetAmount` returns `floor(1 × 3e18 / 1e18) = 3`. Request records `expectedAssetAmount = 3`.
5. After the delay, `completeWithdrawal` calls `_calculatePayoutAmount`: `currentReturn = floor(1 × 3e18 / 1e18) = 3`. Transfers 3 wei to Alice.
6. Alice deposited 5 wei and received 3 wei — a net loss of 2 wei with no price movement.

At production scale (1e18 wei deposit, same prices): `floor(floor(1e18 × 1e18 / 3e18) × 3e18 / 1e18) = 999999999999999999` — a loss of 1 wei, systematic on every round-trip.

**Foundry fuzz test plan**:
```solidity
function testFuzz_roundTripLoss(uint256 amount, uint256 assetPrice, uint256 rsETHPrice) public {
    vm.assume(amount > 0 && assetPrice > 0 && rsETHPrice > 0);
    uint256 rsETH = (amount * assetPrice) / rsETHPrice;
    vm.assume(rsETH > 0);
    uint256 returned = (rsETH * rsETHPrice) / assetPrice;
    assertLe(returned, amount); // always true — user never gets back more
    // assert loss is bounded
    assertGe(returned, amount - 2); // fails for small amounts like 5 wei
}
