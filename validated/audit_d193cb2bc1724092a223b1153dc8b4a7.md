Audit Report

## Title
Pool Admin Bypasses Factory Fee Cap via Uncapped `setPoolBinAdditionalFees` — (File: metric-core/contracts/MetricOmmPoolFactory.sol)

## Summary
`MetricOmmPoolFactory` enforces `maxAdminSpreadFeeE6` to cap pool-admin spread revenue, and `setPoolAdminFees` correctly validates against this cap. However, `setPoolBinAdditionalFees` forwards `addFeeBuyE6`/`addFeeSellE6` to the pool with no cap validation, allowing a pool admin to set per-bin fees up to the `uint16` maximum (65,535 E6 = 6.5535%) regardless of the factory-enforced cap, including when `maxAdminSpreadFeeE6 = 0`.

## Finding Description
The factory defines `HARD_MAX_SPREAD_FEE_E6 = 200_000` and exposes `maxAdminSpreadFeeE6` as the runtime cap on pool-admin spread fees. [1](#0-0) 

`setPoolAdminFees` enforces this cap correctly: [2](#0-1) 

`setPoolBinAdditionalFees` performs no such check and passes values directly to the pool: [3](#0-2) 

`MetricOmmPool.setBinAdditionalFees` only validates the bin index, not the fee magnitudes, and writes the unchecked values directly into `BinState`: [4](#0-3) 

`addFeeBuyE6` and `addFeeSellE6` are `uint16` fields, accepting values up to 65,535: [5](#0-4) 

These per-bin fees are applied on top of the global spread fee during every swap through the affected bin, fully bypassing the factory owner's cap intent.

## Impact Explanation
This is a confirmed admin-boundary break: the pool admin exceeds the factory-enforced `maxAdminSpreadFeeE6` cap via an unchecked code path. Every trader swapping through a bin with elevated `addFeeBuyE6`/`addFeeSellE6` suffers a direct, real-token loss of up to 6.5535% per swap. If the factory owner sets `maxAdminSpreadFeeE6 = 0` to eliminate admin spread entirely, the pool admin can still extract up to 6.5535% per bin. This constitutes a High-severity direct loss of user principal.

## Likelihood Explanation
The pool admin is explicitly semi-trusted — the cap system exists because the pool admin is not fully trusted. Exploitation requires a single transaction with no timelock, governance step, or special precondition beyond pool admin role. It is immediately repeatable on any bin after pool creation.

## Recommendation
Add cap validation in `setPoolBinAdditionalFees` before forwarding to the pool:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (addFeeBuyE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

## Proof of Concept
1. Factory owner deploys factory with `maxAdminSpreadFeeE6 = 0`.
2. Pool admin calls `factory.setPoolAdminFees(pool, 0, 0)` — passes cap check at L414-415.
3. Pool admin calls `factory.setPoolBinAdditionalFees(pool, 0, 65_535, 65_535)` — no cap check at L450-457, call succeeds.
4. `MetricOmmPool.setBinAdditionalFees` writes `addFeeBuyE6 = 65_535` and `addFeeSellE6 = 65_535` into `_binStates[0]` at L471-472.
5. Every trader swapping through bin 0 pays an extra 6.5535% fee routed to the admin, violating the factory owner's zero-admin-spread policy.
