### Title
Deposit Refund Reuse Across Sub-Transactions in a Batch Drains the Deposit Pot - (`eras/dijkstra/impl/testlib/Test/Cardano/Ledger/Dijkstra/Imp/CertSpec.hs`)

---

### Summary

The Dijkstra era introduces sub-transactions: a top-level transaction can embed a batch of sub-transactions (`sub_transactions` field). An unprivileged transaction author can craft a top-level transaction containing multiple sub-transactions that each include an `UnRegDepositTxCert` for the **same** staking credential. Because no cross-sub-transaction predicate prevents this, each sub-transaction independently claims the same deposit refund, decrementing `utxosDeposited` multiple times for a single deposit. This creates ADA out of thin air by draining the deposit pot beyond its actual obligation.

The developers have already identified this scenario and left a pending test (`xit`) with `error "TODO: predicate failure not yet implemented"`, confirming the protection is absent.

---

### Finding Description

**Vulnerability class:** Funds/accounting bug — value reuse in a loop (batch of sub-transactions).

**Analog to external report:** Just as `msg.value` persists across every iteration of `batch()` → `execute()` delegatecalls, a deposit refund amount embedded in an `UnRegDepositTxCert` certificate is independently consumed by each sub-transaction in a Dijkstra batch without any cross-sub-transaction deduplication guard.

**Root cause — step 1: Dijkstra refund calculation reads from the certificate, not the ledger state.**

In `dijkstraTotalRefundsTxCerts`, the refund for a sub-transaction is computed by summing the deposit amounts declared inside the certificates themselves, with no lookup into the current ledger state:

```haskell
dijkstraTotalRefundsTxCerts = foldMap' $ \case
  UnRegDepositTxCert _ deposit -> deposit
  UnRegDRepTxCert _ deposit -> deposit
  _ -> zero
``` [1](#0-0) 

**Root cause — step 2: `updateUTxOStateNoFees` decrements `utxosDeposited` for every sub-transaction independently.**

Each sub-transaction is processed by `dijkstraSubUtxoTransition`, which calls `Shelley.updateUTxOStateNoFees`. That function computes `totalRefunds = certsTotalRefundsTxBody pp certState txBody` and applies `utxosDeposited <> depositChange` to the running UTxO state: [2](#0-1) 

The `certState` passed into `SubUtxoEnv` is the **original** pre-batch cert state (fixed as an environment, not updated between sub-transactions at the UTxO level): [3](#0-2) 

**Root cause — step 3: No cross-sub-transaction predicate prevents the same credential from being deregistered in multiple sub-transactions.**

The SUBUTXO transition rule applies `updateUTxOStateNoFees` for each sub-transaction without any guard that checks whether a prior sub-transaction in the same batch already claimed the refund for the same credential: [4](#0-3) 

**Root cause — step 4: The developers explicitly acknowledge the missing predicate.**

The test file contains a pending test (`xit`) titled `"Multiple subtransactions cannot get the same refund"`. It constructs exactly this attack — two sub-transactions each containing `UnRegDepositTxCert stakingCred keyDeposit` for the same credential — and calls `submitFailingTx` with `error "TODO: predicate failure not yet implemented"`, confirming the protection does not yet exist: [5](#0-4) 

---

### Impact Explanation

**Impact: Critical — Direct creation of ADA through an invalid ledger state transition.**

For each sub-transaction that claims the refund, the preservation-of-value check passes (the sub-transaction's outputs include the claimed refund amount, so consumed = produced for that sub-transaction). However, `utxosDeposited` is decremented once per sub-transaction. If N sub-transactions all claim the same deposit of amount `D`, the deposit pot is reduced by `N × D` while only `D` was ever deposited. The extra `(N-1) × D` lovelace materialises in UTxO outputs, constituting direct ADA creation from an invalid ledger state transition.

This violates the fundamental preservation-of-ADA invariant that the entire ledger is built upon. [6](#0-5) 

---

### Likelihood Explanation

**Likelihood: High.**

- No privileged role is required. Any transaction author can submit a top-level Dijkstra transaction.
- The attack requires only: (a) one registered staking credential, (b) a top-level transaction with two or more sub-transactions each containing `UnRegDepositTxCert` for that credential.
- The developers have already identified and documented the scenario as a known gap (`xit` test), confirming it is reachable and not yet blocked.
- The Dijkstra era is the production target for this codebase.

---

### Recommendation

1. **Implement the missing cross-sub-transaction deduplication predicate.** Before processing sub-transactions, collect all credentials being deregistered across the entire batch (top-level + all sub-transactions). Reject the batch if any credential appears in more than one deregistration certificate across the batch.

2. **Enable and complete the pending test.** Replace the `xit` with `it` and supply the correct `PredicateFailure` constructor once the guard is implemented.

3. **Apply the same deduplication logic to DRep deregistrations** (`UnRegDRepTxCert`) and any other certificate type that produces a deposit refund, since `dijkstraTotalRefundsTxCerts` handles both. [7](#0-6) 

---

### Proof of Concept

```
1. Register staking credential C with deposit D.

2. Construct subTx1:
     inputs  = { utxo_input_1 }
     certs   = [ UnRegDepositTxCert C D ]
     outputs = [ addr1 ← value1 + D ]   -- claims refund

3. Construct subTx2:
     inputs  = { utxo_input_2 }          -- different UTxO input
     certs   = [ UnRegDepositTxCert C D ]
     outputs = [ addr2 ← value2 + D ]   -- claims same refund again

4. Construct topTx:
     sub_transactions = OMap.fromFoldable [subTx1, subTx2]

5. Submit topTx.

Expected (correct) result: transaction rejected — C cannot be deregistered twice.
Actual (current) result:   transaction accepted — deposit pot decremented by 2D,
                            but only D was ever deposited. Net ADA created: D lovelace.
```

This is structurally identical to the external report's PoC: a batch of N calls each reusing the same value, with the proxy/deposit pot drained by N × value instead of 1 × value. [5](#0-4) [8](#0-7)
