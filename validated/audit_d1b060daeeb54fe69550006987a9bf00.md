### Title
No Minimum Finality/Confirmation Threshold Enforced for Foreign Chain Verification Requests — (File: `crates/contract/src/lib.rs`)

---

### Summary

The `verify_foreign_transaction` contract method accepts user-supplied finality parameters (`BlockConfirmations` for Bitcoin, `EvmFinality` for EVM chains) without enforcing any protocol-level minimum. An unprivileged caller can set `confirmations: 0` (Bitcoin) or `finality: Latest` (EVM), causing MPC nodes to sign attestations for transactions that have not reached sufficient finality. This is the direct analog of the Pyth confidence-interval omission: just as that protocol accepted prices without validating the accompanying uncertainty metadata, this protocol accepts foreign-chain verification requests without validating the accompanying finality-quality metadata, enabling forged foreign-chain verification and double-spend conditions in bridge flows.

---

### Finding Description

`verify_foreign_transaction` in `crates/contract/src/lib.rs` accepts a `VerifyForeignTransactionRequestArgs` containing chain-specific finality parameters and passes them through to MPC nodes without any protocol-level minimum check. [1](#0-0) 

The only validations performed are: correct domain purpose, sufficient gas/deposit, and that the chain is in the supported set. No minimum `confirmations` or minimum `finality` level is enforced.

The `BlockConfirmations` type is a plain `u64` newtype with no minimum constraint: [2](#0-1) 

The ABI schema confirms `"minimum": 0.0` is the only constraint, meaning `confirmations: 0` is a valid on-chain input.

The Bitcoin inspector uses the user-supplied threshold directly with a `<=` comparison: [3](#0-2) 

With `block_confirmations_threshold = 0`, the expression `0 <= transaction_block_confirmation` is always `true` for any confirmed transaction (u64 ≥ 0), so the finality gate is completely bypassed. The inspector then proceeds to the canonical-chain check and, if that passes, signs the payload.

For EVM chains, the `verify_finality_level` function maps the user-supplied `EthereumFinality::Latest` to `FinalityTag::Latest` and queries the latest block head: [4](#0-3) 

A transaction in the latest block satisfies `head.number >= receipt_block_number`, so the check passes even though the latest block is not finalized and can be reorged.

The signed payload (`ForeignTxSignPayload`) commits to the full request including the user-supplied `confirmations` value and the observed `ExtractedValue`s (e.g., `BlockHash`): [5](#0-4) 

Once signed, this payload is a permanent on-chain attestation that the transaction existed in a specific block at the time of signing — regardless of whether that block is later reorged.

---

### Impact Explanation

A bridge contract consuming `VerifyForeignTransactionResponse` signatures from the MPC network can be exploited via a double-spend:

1. Attacker submits Bitcoin transaction T to a bridge (locking BTC).
2. Attacker calls `verify_foreign_transaction` with `confirmations: 0` for T.
3. MPC nodes observe T in a block with 1 confirmation; `0 <= 1` passes; canonical-chain check passes; MPC signs the payload attesting `T → BlockHash B`.
4. Attacker uses the MPC signature to claim bridge funds on NEAR.
5. A Bitcoin reorg removes T from the canonical chain (block B is orphaned).
6. Attacker double-spends the BTC on the Bitcoin side.

The MPC signature is irrevocable — it permanently attests to a transaction that no longer exists on the canonical chain. This directly satisfies the **High** impact criterion: *"forged foreign-chain verification … that causes invalid bridge execution or double-spend conditions."*

---

### Likelihood Explanation

- **Entry path**: Any unprivileged account can call `verify_foreign_transaction` with `confirmations: 0`. No special role, key, or operator access is required.
- **Feasibility**: Bitcoin 1-block reorgs occur regularly (several per year on mainnet); shallow reorgs are well within attacker capability, especially with selfish-mining or hash-power rental.
- **No threshold collusion required**: MPC nodes behave honestly — they faithfully apply the user-supplied threshold. The vulnerability is in the contract's failure to enforce a meaningful minimum, not in node misbehavior.

---

### Recommendation

Enforce a protocol-level minimum finality requirement inside `verify_foreign_transaction` before enqueuing the request:

- **Bitcoin**: Require `confirmations >= N` (e.g., `N = 1` as an absolute floor, with a governance-configurable recommended minimum of 6).
- **EVM chains**: Disallow `EvmFinality::Latest`; require at least `EvmFinality::Safe` or `EvmFinality::Finalized`.
- **Starknet / Aptos / TON**: Apply equivalent chain-appropriate finality floors.

These minimums should be stored in the contract's `Config` (alongside `sign_call_gas_attachment_requirement_tera_gas`) and enforced before the request is enqueued, so that no MPC node ever signs an attestation for a transaction that has not reached the protocol's minimum finality bar.

### Proof of Concept

```json
// Call verify_foreign_transaction with confirmations: 0
{
  "request": {
    "request": {
      "Bitcoin": {
        "tx_id": "<32-byte tx id of a 1-confirmation Bitcoin tx>",
        "confirmations": 0,
        "extractors": ["BlockHash"]
      }
    },
    "domain_id": <foreign_tx_domain_id>,
    "payload_version": 1
  }
}
```

1. Contract accepts the request — no minimum-confirmation check exists in `verify_foreign_transaction`. [6](#0-5) 

2. MPC nodes call `getrawtransaction`; the response has `confirmations: 1`; the inspector evaluates `0 <= 1 → true`; the canonical-chain check passes. [7](#0-6) 

3. MPC signs `ForeignTxSignPayload{request: Bitcoin{tx_id, confirmations: 0, …}, values: [BlockHash(B)]}`.

4. Attacker receives a valid `VerifyForeignTransactionResponse` with a threshold signature attesting the transaction was in block B.

5. Bitcoin reorg orphans block B; attacker double-spends BTC while retaining the NEAR-side claim backed by the MPC signature.

    pub fn verify_foreign_transaction(&mut self, request: VerifyForeignTransactionRequestArgs) {
        log!(
            "verify_foreign_transaction: predecessor={:?}, request={:?}",
            env::predecessor_account_id(),
            request
        );

        self.check_request_preconditions(
            request.domain_id,
            DomainPurpose::ForeignTx,
            Gas::from_tgas(self.config.sign_call_gas_attachment_requirement_tera_gas),
            MINIMUM_SIGN_REQUEST_DEPOSIT,
        );

        let requested_chain = request.request.chain();
        let supported_chains = self.get_supported_foreign_chains();
        if !supported_chains.contains(&requested_chain) {
            env::panic_str(
                &InvalidParameters::ForeignChainNotSupported {
                    requested: requested_chain,
                }
                .to_string(),
            );
        }

        let callback_gas = Gas::from_tgas(
            self.config
                .return_signature_and_clean_state_on_success_call_tera_gas,
        );

        let request = args_into_verify_foreign_tx_request(request);
        let callback_args = serde_json::to_vec(&(&request,)).unwrap();
        self.enqueue_yield_request(
            method_names::RETURN_VERIFY_FOREIGN_TX_AND_CLEAN_STATE_ON_SUCCESS,
            callback_args,
            callback_gas,
            move |this, id| this.add_verify_foreign_tx_request(request, id),
        );
    }
