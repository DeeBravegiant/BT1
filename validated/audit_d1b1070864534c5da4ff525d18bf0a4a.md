Audit Report

## Title
Missing Staleness Check on Chainlink Price Feed Allows Stale Asset Prices to Corrupt rsETH Exchange Rate - (File: `contracts/oracles/ChainlinkPriceOracle.sol`)

## Summary
`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards the `updatedAt` return value, accepting arbitrarily stale prices without validation. These prices flow directly into `LRTOracle._getTotalEthInProtocol()` and ultimately set `rsETHPrice`, the global exchange rate used for all deposits and withdrawals. Because `updateRSETHPrice()` is permissionless, any caller can lock in a stale-price-derived exchange rate at will.

## Finding Description
In `ChainlinkPriceOracle.getAssetPrice()` (L52), `latestRoundData()` is called and only `price` is used; `updatedAt` is silently dropped:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();   // updatedAt discarded
``` [1](#0-0) 

This price is consumed by `_getTotalEthInProtocol()` (L339), which sums the ETH-denominated value of every supported LST: [2](#0-1) 

`totalETHInProtocol` is then used to compute `newRsETHPrice` (L250), which is written to the global `rsETHPrice` state variable (L313): [3](#0-2) [4](#0-3) 

`updateRSETHPrice()` is `public whenNotPaused` with no role restriction, callable by any address: [5](#0-4) 

**Why existing guards are insufficient**: The `pricePercentageLimit` check (L252–266) only reverts or pauses when `newRsETHPrice` deviates from `highestRsethPrice` beyond the configured threshold. A moderately stale price that has not moved far from the historical peak passes this check silently, allowing the corrupted exchange rate to be committed. [6](#0-5) 

## Impact Explanation
**Stale price inflated above actual**: `totalETHInProtocol` is overstated → `rsETHPrice` is set above true value → any user who deposits immediately after the stale update receives more rsETH than their assets are worth, directly diluting existing rsETH holders. This constitutes **direct theft of user funds (Critical)** when the stale deviation is large enough to produce meaningful dilution yet small enough to stay within `pricePercentageLimit`. At minimum, when the deviation is small, the protocol **fails to deliver promised returns (Low)** to depositors and withdrawers who transact at the incorrect rate.

**Stale price deflated below actual**: `rsETHPrice` is understated → depositors receive fewer rsETH tokens than deserved and withdrawers receive less ETH than owed — **contract fails to deliver promised returns (Low)**, with potential **temporary freezing** if the downside guard triggers a pause.

## Likelihood Explanation
Chainlink ETH-denominated LST feeds (stETH/ETH, ETHx/ETH) have a 24-hour heartbeat, meaning a feed can be up to 24 hours stale before Chainlink's own circuit breaker fires. During extreme market volatility or network congestion, feeds can lag significantly. Because `updateRSETHPrice()` requires no privilege, an attacker can monitor on-chain `updatedAt` values and deliberately call the function during a known stale window to lock in the incorrect exchange rate before the feed recovers. This is repeatable and requires no victim cooperation.

## Recommendation
Add a configurable `maxStaleness` parameter per asset and validate `updatedAt` in `ChainlinkPriceOracle.getAssetPrice()`:

```solidity
(, int256 price,, uint256 updatedAt,) = priceFeed.latestRoundData();
if (block.timestamp - updatedAt > maxStaleness[asset]) revert StalePriceFeed();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

Set `maxStaleness` per asset to match the Chainlink feed's documented heartbeat plus a small buffer (e.g., heartbeat + 1 hour). Additionally, validate that `price > 0` to guard against a zero/negative answer.

## Proof of Concept
1. A Chainlink feed for a supported LST (e.g., stETH/ETH) goes stale — `updatedAt` is 12 hours old while the actual LST price has dropped 5%.
2. An attacker calls `LRTOracle.updateRSETHPrice()` (no role required, `public whenNotPaused`).
3. `_getTotalEthInProtocol()` calls `ChainlinkPriceOracle.getAssetPrice(stETH)` → `latestRoundData()` returns the stale (inflated) price; no staleness check exists.
4. `totalETHInProtocol` is overstated by ~5% of the stETH TVL.
5. `newRsETHPrice` is set higher than the true value; if the 5% deviation is within `pricePercentageLimit`, the guard does not trigger.
6. The attacker (or any user) immediately deposits and mints rsETH at the inflated rate, receiving more rsETH than their deposit is worth.
7. When the feed recovers and `updateRSETHPrice()` is called again, `rsETHPrice` corrects downward, diluting all existing holders proportionally.

**Foundry fork test outline**:
- Fork mainnet at a block where a supported LST Chainlink feed is near its heartbeat boundary.
- Mock `latestRoundData()` to return a stale `updatedAt` (e.g., `block.timestamp - 23 hours`) with a price 3% above current.
- Call `updateRSETHPrice()` as an unprivileged EOA; assert `rsETHPrice` is set to the inflated value.
- Deposit as attacker; record rsETH minted.
- Advance time, call `updateRSETHPrice()` with the correct price; assert `rsETHPrice` decreases and existing holder balances are diluted.
