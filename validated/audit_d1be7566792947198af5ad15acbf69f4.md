### Title
Farmer Unconditionally Accepts `farmer_reward_address_override` from Harvester Without Enforcing CHIP-22 Fee Quality Check — (`chia/farmer/farmer_api.py`, `chia/farmer/farmer.py`)

---

### Summary

The CHIP-22 fee-quality convention is supposed to gate when a third-party harvester may redirect the farmer reward to its own address. The farmer's enforcement path is split across two messages: the fee-quality check fires on `NewProofOfSpace` but only emits log warnings; the actual reward-address substitution fires on `RespondSignatures` with **no check at all**. A malicious third-party harvester can therefore redirect the 0.25 XCH farmer reward to any arbitrary puzzle hash on every block it wins, with zero enforcement from the farmer.

---

### Finding Description

**Step 1 — Fee-quality check is advisory only.**

`notify_farmer_reward_taken_by_harvester_as_fee` is called when `NewProofOfSpace` arrives carrying a non-`None` `farmer_reward_address_override`: [1](#0-0) 

Inside that function the fee-quality comparison is performed, but both the passing and failing branches only call `self.log.info` / `self.log.warning`. The function signature is `-> None`; it raises nothing and returns no sentinel that the caller could act on: [2](#0-1) 

When `fee_info` is omitted entirely the farmer also only logs a warning and continues: [3](#0-2) 

**Step 2 — Override is accepted unconditionally in `_process_respond_signatures`.**

When `RespondSignatures` arrives, `_process_respond_signatures` substitutes the override address into `DeclareProofOfSpace` with no fee-quality check whatsoever: [4](#0-3) 

The resulting `DeclareProofOfSpace` carries the attacker-controlled puzzle hash as `farmer_reward_address`, which the full node uses to create the 0.25 XCH farmer reward coin: [5](#0-4) 

**Structural asymmetry (analog to the external report).**

| Path | Guard applied | Effect of guard failure |
|---|---|---|
| Normal farmer reward | `farmer_target` from config | N/A — no override possible |
| `farmer_reward_address_override` via `NewProofOfSpace` | `notify_farmer_reward_taken_by_harvester_as_fee` | Log warning only — does not block |
| `farmer_reward_address_override` via `RespondSignatures` | **None** | Override accepted unconditionally |

The fee-quality check is applied to the `NewProofOfSpace` message but the reward address is consumed from the later `RespondSignatures` message. Even if both messages carry the same override value, the check on the first message has no enforcement effect on the second.

The `RespondSignatures` protocol message definition confirms both fields exist independently: [6](#0-5) 

---

### Impact Explanation

A malicious third-party harvester (CHIP-22 scenario) that holds winning plots can redirect the 0.25 XCH farmer reward to any puzzle hash on every block it wins. The farmer's configured `farmer_target` is silently bypassed. Over a farming session this constitutes a continuous, compounding payout redirection of XCH from the farmer to the attacker. This falls squarely within the allowed High impact category: *"Bypass of … authorization that enables … payout redirection."*

---

### Likelihood Explanation

The attacker must operate a third-party harvester that is already connected to the victim farmer and must hold plots that win blocks. Both conditions are realistic in the CHIP-22 third-party harvesting model, which is the exact scenario the fee-quality convention was designed to govern. No cryptographic break, key leak, or host compromise is required — the attacker simply sets `farmer_reward_address_override` to their own puzzle hash in every `RespondSignatures` response and omits or falsifies `fee_info`.

---

### Recommendation

`_process_respond_signatures` must enforce the fee-quality check before substituting the override address. Concretely:

1. Re-derive `fee_quality` from `pospace.proof` and `response.challenge_hash` (both already available in `_process_respond_signatures`).
2. Require that `response.farmer_reward_address_override` is accompanied by a `fee_info` field (sourced from the original `NewProofOfSpace` stored alongside the proof-of-space cache entry).
3. If `fee_quality > fee_info.applied_fee_threshold` or `fee_info` is absent, discard the override and use `self.farmer.farmer_target` instead (or return `None` to drop the block submission entirely).

Alternatively, store the result of the fee-quality check when `NewProofOfSpace` is processed and gate the override acceptance in `_process_respond_signatures` on that stored result.

---

### Proof of Concept

A malicious harvester implementation overrides `request_signatures` to inject an attacker-controlled address:

```python
# Malicious harvester: intercept RespondSignatures and inject override
async def request_signatures(self, request: RequestSignatures) -> Message:
    # Produce a legitimate signature response
    response: RespondSignatures = await real_request_signatures(request)
    # Unconditionally redirect farmer reward — no fee_info needed
    malicious_response = dataclasses.replace(
        response,
        farmer_reward_address_override=ATTACKER_PUZZLE_HASH,
        # fee_info omitted entirely — farmer only logs a warning
    )
    return make_msg(ProtocolMessageTypes.respond_signatures, malicious_response)
```

The farmer receives this response, enters `_process_respond_signatures`, hits line 917 (`response.farmer_reward_address_override is not None` → `True`), and emits `DeclareProofOfSpace` with `farmer_reward_address = ATTACKER_PUZZLE_HASH`. The full node creates the 0.25 XCH farmer reward coin to the attacker's address. The farmer's `farmer_target` is never consulted.

    def notify_farmer_reward_taken_by_harvester_as_fee(
        self, sp: farmer_protocol.NewSignagePoint, proof_of_space: harvester_protocol.NewProofOfSpace
    ) -> None:
        """
        Apply a fee quality convention (see CHIP-22: https://github.com/Chia-Network/chips/pull/88)
        given the proof and signage point. This will be tested against the fee threshold reported
        by the harvester (if any), and logged.
        """
        assert proof_of_space.farmer_reward_address_override is not None

        challenge_str = str(sp.challenge_hash)

        ph_prefix = self.config["network_overrides"]["config"][self.config["selected_network"]]["address_prefix"]
        farmer_reward_puzzle_hash = encode_puzzle_hash(proof_of_space.farmer_reward_address_override, ph_prefix)

        self.log.info(
            f"Farmer reward for challenge '{challenge_str}' "
            + f"taken by harvester for reward address '{farmer_reward_puzzle_hash}'"
        )

        fee_quality = calculate_harvester_fee_quality(proof_of_space.proof.proof, sp.challenge_hash)
        fee_quality_rate = float(fee_quality) / float(0xFFFFFFFF) * 100.0

        if proof_of_space.fee_info is not None:
            fee_threshold = proof_of_space.fee_info.applied_fee_threshold
            fee_threshold_rate = float(fee_threshold) / float(0xFFFFFFFF) * 100.0

            if fee_quality <= fee_threshold:
                self.log.info(
                    f"Fee threshold passed for challenge '{challenge_str}': "
                    + f"{fee_quality_rate:.3f}%/{fee_threshold_rate:.3f}% ({fee_quality}/{fee_threshold})"
                )
            else:
                self.log.warning(
                    f"Invalid fee threshold for challenge '{challenge_str}': "
                    + f"{fee_quality_rate:.3f}%/{fee_threshold_rate:.3f}% ({fee_quality}/{fee_threshold})"
                )
                self.log.warning(
                    "Harvester illegitimately took a fee reward that "
                    + "did not belong to it or it incorrectly applied the fee convention."
                )
        else:
            self.log.warning(
                "Harvester illegitimately took reward by failing to provide its fee rate "
                + f"for challenge '{challenge_str}'. "
                + f"Fee quality was {fee_quality_rate:.3f}% ({fee_quality} or 0x{fee_quality:08x})"
            )
