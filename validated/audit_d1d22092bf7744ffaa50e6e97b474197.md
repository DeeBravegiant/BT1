### Title
Hardcoded `address-map.near` Registrar Account ID in Wallet Contract Breaks EOA Base-Token Transfers on Testnet/Localnet — (`runtime/near-wallet-contract/implementation/wallet-contract/src/ADDRESS_REGISTRAR_ACCOUNT_ID`)

---

### Summary

The ETH wallet contract bakes `address-map.near` (a mainnet-only account) as a compile-time constant into every network variant of the WASM (mainnet, testnet, localnet). The build system correctly parameterises `CHAIN_ID` per network but never updates `ADDRESS_REGISTRAR_ACCOUNT_ID`. On testnet and localnet the registrar cross-contract call always fails, permanently breaking the EOA base-token-transfer path that requires an address check.

---

### Finding Description

`runtime/near-wallet-contract/implementation/wallet-contract/src/ADDRESS_REGISTRAR_ACCOUNT_ID` contains the single string `address-map.near`. [1](#0-0) 

This file is embedded at compile time:

```rust
const ADDRESS_REGISTRAR_ACCOUNT_ID: &str = std::include_str!("ADDRESS_REGISTRAR_ACCOUNT_ID");
``` [2](#0-1) 

`build.rs` builds three separate WASM artifacts — mainnet, testnet, localnet — and correctly swaps `CHAIN_ID` before each build: [3](#0-2) 

However, `ADDRESS_REGISTRAR_ACCOUNT_ID` is **never touched** by `build_contract`. There is no analogous read/write/restore sequence for it. All three WASM binaries therefore embed `address-map.near` regardless of the target network.

At runtime, when `inner_rlp_execute` determines that an EOA base-token transfer targets another eth-implicit account and an address check is required, it calls:

```rust
let account_id = ADDRESS_REGISTRAR_ACCOUNT_ID
    .trim()
    .parse()
    .unwrap_or_else(|_| env::panic_str("Invalid address registrar"));
ext_registrar::ext(account_id).with_static_gas(REGISTRAR_LOOKUP_GAS)
``` [4](#0-3) 

On testnet/localnet `address-map.near` does not exist, so the promise always returns `PromiseResult::Failed`. The callback then returns:

```rust
PromiseResult::Failed => {
    return PromiseOrValue::Value(ExecuteResponse {
        success: false, ...
        error: Some("Call to Address Registrar contract failed".into()),
    });
}
``` [5](#0-4) 

Critically, the nonce was **not** incremented before the registrar call (the increment is intentionally deferred for this code path), so the user's nonce is never consumed and the transfer can never succeed — every retry hits the same dead registrar.

---

### Impact Explanation

On testnet and localnet, any eth-implicit account user who submits a valid ETH-signed base-token transfer to another eth-implicit account that triggers the address-check branch receives a permanent `"Call to Address Registrar contract failed"` outcome. The intended NEAR transfer is never executed. The corrupted protocol result is the `ExecuteResponse` outcome of `rlp_execute`: it is always `success: false` for this class of transactions on non-mainnet networks, making the ETH-emulation transfer feature entirely non-functional there.

---

### Likelihood Explanation

Any unprivileged user on testnet can trigger this by calling `rlp_execute` with a valid RLP-encoded ETH transaction whose `to` field is another eth-implicit account address. No special privileges are required. The CHAIN_ID validation (`tx.chain_id != Some(CHAIN_ID)`) correctly rejects mainnet-signed transactions on testnet, so users must sign with chain ID 398 — a normal testnet operation. [6](#0-5) 

---

### Recommendation

Apply the same read/write/restore pattern used for `CHAIN_ID` to `ADDRESS_REGISTRAR_ACCOUNT_ID` in `build_contract`. Pass a per-network registrar account ID string (e.g. `address-map.near` for mainnet, `address-map.testnet` for testnet, a local account for localnet) and write it to the source file before each Docker build, then restore it afterward — exactly mirroring lines 65–73 of `build.rs`. [7](#0-6) 

---

### Proof of Concept

1. Deploy the testnet wallet contract WASM (built with `CHAIN_ID=398` but `ADDRESS_REGISTRAR_ACCOUNT_ID=address-map.near`) to a testnet eth-implicit account `0xAAA...`.
2. Sign an ETH transaction with chain ID 398, `from=0xAAA...`, `to=0xBBB...` (another eth-implicit account), zero calldata (base-token transfer).
3. Call `rlp_execute(target="0xBBB...testnet", tx_bytes_b64=<base64 of signed tx>)`.
4. The contract resolves `ADDRESS_REGISTRAR_ACCOUNT_ID` → `address-map.near`, schedules a cross-contract call to that non-existent testnet account.
5. The callback fires with `PromiseResult::Failed`; `rlp_execute` returns `{success: false, error: "Call to Address Registrar contract failed"}`.
6. The user's nonce is unchanged; every subsequent attempt produces the same result.

```rust
fn build_contract(
    dir: &str,
    contract_name: &str,
    output: &str,
    chain_id: u64,
) -> anyhow::Result<()> {
    let wasm_target_path = format!("./res/{}.wasm", output);
    if Path::new(&wasm_target_path).exists() {
        // Skip building if an artifact is already present
        return Ok(());
    }

    let absolute_dir = Path::new(dir).canonicalize()?;

    let chain_id_path = absolute_dir.join("wallet-contract/src/CHAIN_ID");
    let chain_id_content = std::fs::read(&chain_id_path).context("Failed to read CHAIN_ID file")?;

    // Update the chain id before building
    std::fs::write(&chain_id_path, chain_id.to_string().into_bytes())?;
    docker_build(absolute_dir.to_str().expect("path should be valid UTF-8"))?;

    // Restore chain id file to original value after building
    std::fs::write(&chain_id_path, chain_id_content)?;

    let build_artifact_path =
        format!("target/wasm32-unknown-unknown/release/{}.wasm", contract_name);
    let src = absolute_dir.join(build_artifact_path);

    std::fs::copy(&src, &wasm_target_path)
        .with_context(|| format!("Failed to copy `{}` to `{}`", src.display(), wasm_target_path))?;
    Ok(())
}
```
