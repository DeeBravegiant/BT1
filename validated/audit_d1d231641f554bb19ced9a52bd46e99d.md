Audit Report

## Title
`assetsCommitted` Decremented by Stale Original Amount Instead of Actual Payout, Creating Phantom Withdrawal Capacity — (File: `contracts/LRTWithdrawalManager.sol`)

## Summary
In `LRTWithdrawalManager._unlockWithdrawalRequests`, `assetsCommitted[asset]` is decremented by the pre-update `request.expectedAssetAmount` (line 802) before it is overwritten with `payoutAmount` (line 804). When the rsETH/asset price has declined since the request was initiated, `payoutAmount < originalAmount`, so `assetsCommitted` is reduced by more than the vault actually disburses. This inflates `getAvailableAssetAmount`, creating phantom withdrawal capacity that allows new users to queue withdrawals against assets that do not exist, permanently locking their rsETH in the contract.

## Finding Description
`initiateWithdrawal` records `expectedAssetAmount` at the current rsETH price and adds it to `assetsCommitted`:

```solidity
// LRTWithdrawalManager.sol L173
assetsCommitted[asset] += expectedAssetAmount;
``` [1](#0-0) 

When the operator calls `unlockQueue`, `_unlockWithdrawalRequests` computes `payoutAmount = min(originalAmount, currentReturn)` via `_calculatePayoutAmount`:

```solidity
// L833-834
uint256 currentReturn = (request.rsETHUnstaked * rsETHPrice) / assetPrice;
return (request.expectedAssetAmount < currentReturn) ? request.expectedAssetAmount : currentReturn;
``` [2](#0-1) 

Then the critical ordering error occurs:

```solidity
// L802-804
assetsCommitted[asset] -= request.expectedAssetAmount;  // uses OLD (larger) value
request.expectedAssetAmount = payoutAmount;             // then overwrites
``` [3](#0-2) 

When `currentReturn < originalAmount` (price drop), `assetsCommitted` is reduced by `originalAmount` but the vault only releases `payoutAmount`. The vault balance (`assetLyingUnstakingVault`) falls by `payoutAmount`, while `assetsCommitted` falls by `originalAmount`.

`getAvailableAssetAmount` computes:

```solidity
// L601-602
uint256 totalAssets = lrtDepositPool.getTotalAssetDeposits(asset);
availableAssetAmount = totalAssets > assetsCommitted[asset] ? totalAssets - assetsCommitted[asset] : 0;
``` [4](#0-3) 

`getTotalAssetDeposits` includes `assetLyingUnstakingVault`, so after one unlock cycle with a price drop, `getAvailableAssetAmount` is inflated by exactly `originalAmount - payoutAmount`. [5](#0-4) 

The guard in `initiateWithdrawal` at line 170 passes against this phantom capacity:

```solidity
if (expectedAssetAmount > getAvailableAssetAmount(asset)) revert ExceedAmountToWithdraw();
``` [6](#0-5) 

New users' rsETH is transferred in and their requests are recorded, but the vault holds no assets to back them. When the operator later calls `unlockQueue`, the `availableAssetAmount < payoutAmount` check at line 800 causes the loop to break immediately, leaving those requests permanently unprocessable. [7](#0-6) 

## Impact Explanation
**Medium — Temporary (potentially permanent) freezing of funds.** Users who initiate withdrawals against phantom capacity have their rsETH transferred into `LRTWithdrawalManager` with no path to completion. The rsETH is not burned until `unlockQueue` succeeds, so it cannot be recovered by the user. If the vault shortfall is never externally covered, the freeze is permanent, escalating to a Critical impact. The shortfall per cycle is exactly `originalAmount - payoutAmount`, and it compounds across every unlock cycle where a price decline occurred.

## Likelihood Explanation
**Medium.** No special permissions or oracle manipulation are required. Any market event causing the rsETH/LST rate to decline between `initiateWithdrawal` and the operator's `unlockQueue` call triggers the discrepancy. This is ordinary price movement. The phantom capacity is then exploitable by any unprivileged user calling `initiateWithdrawal`. The effect compounds across every affected unlock cycle.

## Recommendation
Decrement `assetsCommitted` by `payoutAmount` (the actual disbursement) rather than the stale `request.expectedAssetAmount`:

```solidity
// Before (wrong):
assetsCommitted[asset] -= request.expectedAssetAmount;
request.expectedAssetAmount = payoutAmount;

// After (correct):
assetsCommitted[asset] -= payoutAmount;
request.expectedAssetAmount = payoutAmount;
```

This ensures `assetsCommitted` always reflects the true outstanding obligation, keeping `getAvailableAssetAmount` accurate.

## Proof of Concept
1. rsETH/stETH rate at request time: 1 rsETH = 1.05 stETH. User calls `initiateWithdrawal(stETH, 1e18 rsETH)`. `expectedAssetAmount = 1.05e18`. `assetsCommitted[stETH] += 1.05e18`.
2. rsETH price drops. At unlock time: 1 rsETH = 1.00 stETH. `_calculatePayoutAmount` returns `min(1.05e18, 1.00e18) = 1.00e18`.
3. `_unlockWithdrawalRequests` executes: `assetsCommitted[stETH] -= 1.05e18` (old value); vault redeems `1.00e18` stETH → `totalAssets` drops by `1.00e18`.
4. `getAvailableAssetAmount` = `(totalAssets - 1.00e18) - (assetsCommitted - 1.05e18)` = original + **0.05e18 phantom capacity**.
5. A second user calls `initiateWithdrawal(stETH, X rsETH)` where `expectedAssetAmount ≤ 0.05e18`. The guard at line 170 passes. Their rsETH is transferred in.
6. The vault holds 0 stETH for this asset. `unlockQueue` hits `availableAssetAmount < payoutAmount` at line 800 and breaks. The second user's rsETH is frozen in the contract indefinitely.
