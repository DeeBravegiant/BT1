Audit Report

## Title
Permanent Freezing of Unclaimed Yield When `totalKernelStaked` Drops to Zero Mid-Period — (`contracts/KERNEL/KernelDepositPool.sol`)

## Summary

`rewardPerToken()` returns the frozen `rewardPerTokenStored` whenever `totalKernelStaked == 0`, halting all reward accumulation. Because `initiateWithdrawal` contains no guard preventing it from driving `totalKernelStaked` to zero during an active reward period, a sole staker (or coordinated group holding 100% of stake) can exit mid-period and permanently strand the remaining `(finishAt - block.timestamp) * rewardRate` reward tokens in the contract. No admin recovery function exists.

## Finding Description

**Root cause — `rewardPerToken()` short-circuits on zero supply:** [1](#0-0) 

When `totalKernelStaked == 0`, the function returns `rewardPerTokenStored` unchanged. Any elapsed time between `updatedAt` and `lastTimeRewardApplicable()` is silently discarded — those rewards are never credited to anyone.

**`initiateWithdrawal` decrements `totalKernelStaked` with no active-period guard:** [2](#0-1) 

The `updateReward(msg.sender)` modifier correctly snapshots the exiting staker's accrued rewards before the decrement, but nothing prevents the resulting `totalKernelStaked` from reaching zero while `block.timestamp < finishAt`.

**`notifyRewardAmount` only guards the start of a period, not mid-period exits:** [3](#0-2) 

This check is bypassed entirely by `initiateWithdrawal`.

**No recovery path exists.** The contract's last function is `setMaxNumberOfWithdrawalsPerUser` at L610–620; there is no `recoverERC20`, `sweep`, or equivalent admin escape hatch. [4](#0-3) 

**The NatSpec explicitly acknowledges this design gap and defers to an off-chain operational control:** [5](#0-4) 

That operational control ("ensuring there are always some tokens staked for the entire duration") is not enforced by any on-chain invariant.

**Exploit flow:**
1. Admin calls `notifyRewardAmount` while `totalKernelStaked > 0` — period starts, `rewardRate` set.
2. Sole staker (or coordinated group at 100%) calls `initiateWithdrawal(fullBalance)` at time `T` where `T < finishAt`.
3. `updateReward` snapshots rewards up to `T`; then `totalKernelStaked` becomes 0.
4. All subsequent calls to `rewardPerToken()` return the frozen `rewardPerTokenStored`.
5. Rewards accruing from `T` to `finishAt` — equal to `(finishAt - T) * rewardRate` — are never distributed and cannot be recovered.
6. After `withdrawalDelay`, the attacker reclaims their full principal via `claimWithdrawal` at zero net cost.

## Impact Explanation

**Medium — Permanent freezing of unclaimed yield.** The `rewardsToken` balance corresponding to the remainder of the active period is permanently locked in `KernelDepositPool`. No staker can earn it (all `balanceOf` values are zero), and no admin function can retrieve it. This matches the allowed impact: *Medium. Permanent freezing of unclaimed yield.*

## Likelihood Explanation

The precondition — a single address or coordinated group holding 100% of `totalKernelStaked` — is realistic during early deployment (a single bootstrap staker before others join) or in a low-participation environment. No privileged role is required; `initiateWithdrawal` is a public function callable by any staker. The attacker bears no economic cost because their principal is returned after `withdrawalDelay`. The scenario is repeatable across every new reward period.

## Recommendation

1. **Add a `recoverERC20` admin function** that allows recovery of excess reward tokens — i.e., `rewardsToken.balanceOf(address(this))` minus the sum of all outstanding `rewards[user]` values — so stranded rewards can be rescued and re-notified.
2. **Alternatively**, enforce on-chain that `totalKernelStaked` cannot reach zero during an active period by reverting in `initiateWithdrawal` when `block.timestamp < finishAt && totalKernelStaked - _amount == 0`.
3. The NatSpec operational note (L18–23) is not a sufficient security control and should not be relied upon as a mitigation.

## Proof of Concept

```solidity
// Setup: admin starts a 7-day reward period with sole staker
pool.setRewardsDuration(7 days);
rewardsToken.approve(address(pool), 1000e18);
pool.notifyRewardAmount(1000e18);
// rewardRate = 1000e18 / 7 days, finishAt = block.timestamp + 7 days

// Advance time to mid-period (e.g., day 3)
vm.warp(block.timestamp + 3 days);

// Sole staker exits — updateReward credits rewards up to day 3, then totalKernelStaked = 0
pool.initiateWithdrawal(attackerFullBalance);

// rewardPerToken() is now frozen for all future calls
assertEq(pool.rewardPerToken(), pool.rewardPerTokenStored());

// Advance past finishAt
vm.warp(pool.finishAt() + 1);

// Remaining 4 days of rewards are permanently locked
uint256 locked = 4 days * pool.rewardRate();
assertGt(rewardsToken.balanceOf(address(pool)), 0);
// No function exists to recover `locked` tokens
```
