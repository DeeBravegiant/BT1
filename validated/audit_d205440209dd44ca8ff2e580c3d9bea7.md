### Title
Proposer's Storage Deposit Permanently Locked When `do_update` Clears All Pending Entries - (`crates/contract/src/update.rs`)

---

### Summary

When a participant calls `propose_update`, they must attach a NEAR deposit to cover on-chain storage costs for the update entry. When any update reaches the voting threshold and `do_update` is triggered, it clears **all** pending update entries and votes — including proposals from other participants — but never refunds any of the deposits paid by those proposers. These NEAR tokens are permanently locked in the contract with no recovery path.

---

### Finding Description

In `propose_update` (`crates/contract/src/lib.rs`), the contract collects a deposit from the proposer sized to cover the storage cost of the new `UpdateEntry`:

```rust
let attached = env::attached_deposit();
let required = ProposedUpdates::required_deposit(&update);
// ...
let id = self.proposed_updates.propose(update);
// Only excess above `required` is refunded:
if let Some(diff) = attached.checked_sub(required)
    && diff > NearToken::from_yoctonear(0)
{
    Promise::new(proposer).transfer(diff).detach();
}
```

The `required` portion stays in the contract to stake the storage used by the entry.

When `vote_update` reaches threshold, it calls `do_update` (`crates/contract/src/update.rs`):

```rust
pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
    let entry = self.entries.remove(id)?;

    // Clear all entries as they might be no longer valid
    self.entries.clear();
    self.vote_by_participant.clear();
    // ... build and return Promise, no deposit refunds
}
```

`do_update` removes the executed entry **and** clears every other pending entry. Freeing this storage returns the staked NEAR to the contract's own balance — but the original proposers' deposits are never returned to them. The `UpdateEntry` struct even tracks `bytes_used` (the exact storage amount paid for), yet this field is never used to compute a refund.

There is no other contract method that refunds these deposits. `remove_update_vote` only removes a vote record, not the proposal entry, and issues no refund.

---

### Impact Explanation

Every successful contract upgrade permanently locks the storage deposits of **all** proposers whose entries were cleared by `do_update`, not just the winning proposer. For a large contract binary (up to ~1.5 MB), `required_deposit` can reach tens of NEAR (storage cost ≈ 1 NEAR per 100 KB). Multiple concurrent proposals — a normal governance scenario — multiply the locked amount. The funds accumulate in the contract balance with no withdrawal mechanism, breaking the accounting invariant that a deposit paid for storage should be returned when that storage is freed.

**Impact: Medium** — balance/accounting invariant broken; real NEAR tokens are permanently locked per the allowed scope ("Balance … manipulation that breaks production safety/accounting invariants").

---

### Likelihood Explanation

**Likelihood: High** — this triggers on every successful contract upgrade, which is a routine governance operation. The MPC contract has already been upgraded multiple times on mainnet (`v1.signer`). Any participant who proposed an update that was not the one ultimately executed loses their deposit unconditionally.

---

### Recommendation

In `do_update`, before clearing entries, iterate over all remaining entries and schedule a `Promise::transfer` back to each proposer for the deposit they paid. The `bytes_used` field already stored in each `UpdateEntry` provides the exact amount to refund (`env::storage_byte_cost() * bytes_used`). The proposer's account ID must be stored in `UpdateEntry` at proposal time to enable this refund.

```rust
// Before clearing, refund all non-executed entries
for (_, entry) in self.entries.iter() {
    let refund = env::storage_byte_cost().saturating_mul(entry.bytes_used);
    Promise::new(entry.proposer.clone()).transfer(refund).detach();
}
self.entries.clear();
```

---

### Proof of Concept

1. Participant A calls `propose_update` with a 10 NEAR deposit (large contract binary). The deposit is accepted; excess is refunded; 10 NEAR remains in the contract.
2. Participant B calls `propose_update` with a 10 NEAR deposit for a different update. Same result.
3. Participants vote for B's update until threshold is reached; `vote_update` calls `do_update(&B_id, gas)`.
4. `do_update` removes B's entry, then calls `self.entries.clear()` — removing A's entry too.
5. Both A's and B's 10 NEAR deposits remain in the contract balance. Neither is refunded.
6. No contract method exists to recover these funds.

    pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
        let entry = self.entries.remove(id)?;

        // Clear all entries as they might be no longer valid
        self.entries.clear();
        self.vote_by_participant.clear();

        let mut promise = Promise::new(env::current_account_id());
        match entry.update {
            Update::Contract(code) => {
                // deploy contract then do a `migrate` call to migrate state.
                promise = promise.deploy_contract(code).function_call(
                    method_names::MIGRATE,
                    Vec::new(),
                    NearToken::from_near(0),
                    gas,
                );
            }
            Update::Config(config) => {
                // If we vote for a new config, we should use
                // the value `contract_upgrade_deposit_tera_gas` from the config
                // as the new gas value
                let new_config_gas_value = Gas::from_tgas(config.contract_upgrade_deposit_tera_gas);
                promise = promise.function_call(
                    method_names::UPDATE_CONFIG,
                    serde_json::to_vec(&(&config,)).unwrap(),
                    NearToken::from_near(0),
                    new_config_gas_value,
                );
            }
        }
        Some(promise)
    }

fn bytes_used(update: &Update) -> u128 {
    let mut bytes_used = std::mem::size_of::<UpdateEntry>() as u128;

    // Assume a high max of 128 participant votes per update entry.
    bytes_used += 128 * std::mem::size_of::<AccountId>() as u128;

    match update {
        Update::Contract(code) => {
            bytes_used += code.len() as u128;
        }
        Update::Config(config) => {
            let bytes = serde_json::to_vec(&config).unwrap();
            bytes_used += bytes.len() as u128;
        }
    }

    bytes_used
}

fn required_deposit(bytes_used: u128) -> NearToken {
    env::storage_byte_cost().saturating_mul(bytes_used)
}
