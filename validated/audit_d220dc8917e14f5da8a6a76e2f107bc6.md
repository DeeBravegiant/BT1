Audit Report

## Title
`unlockQueue` and `completeWithdrawal` Blocked by `whenNotPaused`, Temporarily Freezing In-Flight rsETH - (File: contracts/LRTWithdrawalManager.sol)

## Summary
Users who call `initiateWithdrawal` transfer their rsETH into `LRTWithdrawalManager` before the queue is processed. Both `unlockQueue` (the operator step that burns rsETH and redeems assets) and `completeWithdrawal` (the user step that claims assets) carry `whenNotPaused` modifiers. When the contract is legitimately paused in response to an emergency, all in-flight withdrawal requests are frozen with no recovery path: the queue cannot be drained, the user cannot claim, and no cancellation function exists.

## Finding Description
The withdrawal lifecycle is two-step. In step 1, `initiateWithdrawal` (lines 150–178) pulls `rsETHUnstaked` from the user via `safeTransferFrom` at line 166 and records a `WithdrawalRequest`. The rsETH is now held by the contract. In step 2a, `unlockQueue` (lines 268–320) carries `whenNotPaused` at line 279; it burns the held rsETH and redeems the underlying asset from the unstaking vault. In step 2b, `completeWithdrawal` (line 183) also carries `whenNotPaused`; it transfers the redeemed asset to the user.

When `PAUSER_ROLE` calls `pause()` (line 347), both step 2a and step 2b revert with `Pausable: paused`. A search of the contract reveals no `cancelWithdrawal`, `rescindWithdrawal`, or equivalent function that would return the user's rsETH from a pending request. The rsETH deposited at step 1 is stranded in the contract for the entire pause duration. The only exit is for `LRT_ADMIN` to call `unpause()` (line 352), which is entirely at the admin's discretion and timing.

Existing guards do not mitigate this: `initiateWithdrawal` itself is also `whenNotPaused`, so no new requests can be created during a pause, but requests already in the queue before the pause are fully trapped.

## Impact Explanation
**Medium — Temporary freezing of funds.** Every user with a pending withdrawal request at the moment of a pause has their rsETH locked in `LRTWithdrawalManager` with no self-help path. The freeze duration is bounded only by how long the admin chooses to keep the contract paused. The impact is concrete and directly matches the allowed scope entry "Temporary freezing of funds."

## Likelihood Explanation
Pausing is a routine emergency response available to any address holding `PAUSER_ROLE`. The protocol explicitly provisions this role for security incidents. Any security incident that triggers a pause while withdrawal requests are pending — a realistic and foreseeable combination requiring no attacker action — activates the freeze. The pauser acts in good faith; no collusion or malicious intent is required. Given that the withdrawal queue processes requests with an 8-day delay (`withdrawalDelayBlocks` initialized to `8 days / 12 seconds` at line 94), there will almost always be in-flight requests when a pause occurs.

## Recommendation
Remove `whenNotPaused` from `unlockQueue` so the queue can be drained even during a pause, allowing users to subsequently call `completeWithdrawal` once unpaused (or remove `whenNotPaused` from `completeWithdrawal` as well for already-unlocked requests). Alternatively, add a `cancelWithdrawal(address asset)` function that is callable regardless of pause state, returns the user's rsETH from their oldest pending (locked) request, decrements `assetsCommitted`, and removes the request from `userAssociatedNonces`.

## Proof of Concept
1. Alice calls `initiateWithdrawal(stETH, 1e18, "ref")`. `LRTWithdrawalManager` pulls `1e18` rsETH from Alice at line 166. A `WithdrawalRequest` is recorded; Alice's rsETH is now held by the contract.
2. A security incident occurs; `PAUSER_ROLE` calls `pause()`.
3. Operator attempts `unlockQueue(stETH, ...)` → reverts at the `whenNotPaused` check (line 279).
4. Alice attempts `completeWithdrawal(stETH, "ref")` → reverts at the `whenNotPaused` check (line 183).
5. No `cancelWithdrawal` function exists. Alice's `1e18` rsETH remains locked in `LRTWithdrawalManager` for the entire pause duration.

**Foundry test sketch:**
```solidity
function test_pauseFreezesInFlightWithdrawal() public {
    // Alice initiates withdrawal
    vm.prank(alice);
    withdrawalManager.initiateWithdrawal(stETH, 1e18, "ref");

    // Emergency pause
    vm.prank(pauser);
    withdrawalManager.pause();

    // unlockQueue reverts
    vm.prank(operator);
    vm.expectRevert("Pausable: paused");
    withdrawalManager.unlockQueue(stETH, 1, minAssetPrice, minRsEthPrice, maxAssetPrice, maxRsEthPrice);

    // completeWithdrawal reverts
    vm.prank(alice);
    vm.expectRevert("Pausable: paused");
    withdrawalManager.completeWithdrawal(stETH, "ref");

    // Alice's rsETH is confirmed held by the contract with no recovery path
    assertEq(rsETH.balanceOf(address(withdrawalManager)), 1e18);
}
```

```text
    /// asset. Will be finalised by calling `completeWithdrawal` after the manager unlocked the request and the delay
    /// has past. There is an edge case were the user withdraws last underlying asset and that asset gets slashed.
    function initiateWithdrawal(
        address asset,
        uint256 rsETHUnstaked,
        string calldata referralId
    )
        external
        override
        nonReentrant
        whenNotPaused
        onlySupportedAsset(asset)
        onlySupportedStrategy(asset)
    {
        if (rsETHUnstaked == 0 || rsETHUnstaked < minRsEthAmountToWithdraw[asset]) {
            revert InvalidAmountToWithdraw();
        }

        IERC20(lrtConfig.rsETH()).safeTransferFrom(msg.sender, address(this), rsETHUnstaked);

        uint256 expectedAssetAmount = getExpectedAssetAmount(asset, rsETHUnstaked);

        if (expectedAssetAmount > getAvailableAssetAmount(asset)) revert ExceedAmountToWithdraw();

        // preventing over-withdrawal.
        assetsCommitted[asset] += expectedAssetAmount;

        _addUserWithdrawalRequest(asset, rsETHUnstaked, expectedAssetAmount);

        emit ReferralIdEmitted(referralId);
```
