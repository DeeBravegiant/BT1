Audit Report

## Title
ETH Deposit Limit Bypass Due to Missing `amount` in Limit Check - (File: contracts/LRTDepositPool.sol)

## Summary
`LRTDepositPool._checkIfDepositAmountExceedesCurrentLimit()` omits the incoming `amount` from the ETH branch of its limit check, evaluating only the pre-deposit total against the cap. Any unprivileged depositor can therefore push ETH deposits beyond `depositLimitByAsset[ETH_TOKEN]` in a single transaction, while the equivalent ERC-20 path correctly includes `amount`. The protocol's primary ETH deposit cap is silently unenforceable.

## Finding Description
In `contracts/LRTDepositPool.sol` at lines 676–682, the guard function reads:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));   // amount omitted
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // correct
}
```

The ETH branch returns `true` only when the **pre-deposit** total already exceeds the cap. The `amount` parameter is never added. As long as `totalAssetDeposits ≤ depositLimit` at call time, the check passes regardless of `msg.value`.

The call chain is:
- `depositETH` (line 87) → `_beforeDeposit` (line 661) → `_checkIfDepositAmountExceedesCurrentLimit`

`depositETH` is `external payable` with no role restriction, so any caller can reach this path. `_beforeDeposit` at line 661 is the sole enforcement point; no secondary cap check exists.

## Impact Explanation
This matches the allowed impact: **Low — Contract fails to deliver promised returns, but doesn't lose value.**

The `depositLimitByAsset` cap is the protocol's stated risk-management ceiling for ETH intake. Bypassing it means:
1. The protocol accepts more ETH than its risk parameters allow.
2. `getRsETHAmountToMint` mints rsETH proportional to the full over-limit deposit, growing rsETH supply beyond the intended ceiling.
3. After the breach, `getAssetCurrentLimit(ETH_TOKEN)` permanently returns `0` (line 404–405), falsely signalling the pool is full even though the limit was already violated — the invariant `getTotalAssetDeposits(ETH) ≤ depositLimitByAsset(ETH)` is silently broken.

No funds are stolen or frozen; the protocol simply fails to enforce its own deposit cap for ETH.

## Likelihood Explanation
`depositETH` is public and payable, requiring no special role or privilege. Any depositor who observes `totalAssetDeposits` at or near the limit can send a single large ETH deposit to breach it. No front-running, governance access, or external dependency is required. The condition is trivially observable on-chain and repeatable.

## Recommendation
Remove the ETH-specific branch and apply the same `+ amount` pattern uniformly:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

This ensures the post-deposit total is always validated against the cap regardless of asset type.

## Proof of Concept
**Preconditions:**
- `depositLimitByAsset[ETH_TOKEN]` = 1000 ETH
- `getTotalAssetDeposits(ETH_TOKEN)` = 999 ETH

**Steps:**
1. Attacker calls `depositETH{value: 500 ether}(0, "")`.
2. `_beforeDeposit` calls `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 500e18)`.
3. ETH branch evaluates `999e18 > 1000e18` → `false` → limit not exceeded, no revert.
4. `getRsETHAmountToMint` mints rsETH for the full 500 ETH.
5. `getTotalAssetDeposits(ETH_TOKEN)` is now 1499 ETH — 499 ETH above the cap.
6. `getAssetCurrentLimit(ETH_TOKEN)` returns `0` permanently.

For ERC-20, the same scenario correctly reverts: `999e18 + 500e18 > 1000e18` → `true` → `MaximumDepositLimitReached`.

**Foundry test plan:** Deploy `LRTDepositPool` with `depositLimitByAsset[ETH_TOKEN]` = 1000 ether, seed `totalAssetDeposits` to 999 ether, call `depositETH{value: 500 ether}`, assert the call succeeds and `getTotalAssetDeposits` returns 1499 ether — confirming the limit was not enforced.
