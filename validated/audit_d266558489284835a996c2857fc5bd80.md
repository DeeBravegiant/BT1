Audit Report

## Title
`purgeStateGuardRole` Does Not Clear `pendingStateGuard`, Allowing a Stale Nominee to Seize the Oracle Guard Role — (File: smart-contracts-poc/contracts/oracles/providers/OracleBase.sol)

## Summary

`purgeStateGuardRole` deletes `stateGuard[feedId]` but leaves `pendingStateGuard[feedId]` intact. A pending nominee set before the abdication can immediately call `acceptStateGuardRole` and install themselves as the new `stateGuard`, gaining unauthorized control over `setPriceGuard` for that feed. This allows the attacker to disable the price band, causing any oracle price — including manipulated or extreme values — to reach pool swap execution.

## Finding Description

In `smart-contracts-poc/contracts/oracles/providers/OracleBase.sol`, `purgeStateGuardRole` only clears the active guard:

```solidity
// L120-124
function purgeStateGuardRole(bytes32 feedId) external checkRole(feedId) {
    delete stateGuard[feedId];   // ← clears active guard
    // pendingStateGuard[feedId] is NOT cleared
    emit StateGuardDeleted(feedId);
}
``` [1](#0-0) 

`acceptStateGuardRole` has no dependency on `stateGuard`; it only checks `pendingStateGuard`:

```solidity
// L111-118
function acceptStateGuardRole(bytes32 feedId) external {
    require(pendingStateGuard[feedId] == msg.sender, InvalidGuard(msg.sender));
    delete pendingStateGuard[feedId];
    stateGuard[feedId] = msg.sender;
    emit StateGuardUpdated(feedId, msg.sender);
}
``` [2](#0-1) 

After `purgeStateGuardRole` executes, `stateGuard[feedId]` is `address(0)` and `checkRole` falls back to `ADMIN_ROLE`. However, the stale nominee can immediately call `acceptStateGuardRole` and install themselves as the new `stateGuard`, overriding ADMIN's restored authority before ADMIN can call `purgePendingStateGuardRole`. The existence of `purgePendingStateGuardRole` as a separate function confirms the protocol intended pending state to be clearable, but `purgeStateGuardRole` never invokes it. [3](#0-2) 

## Impact Explanation

`stateGuard` is the sole authority for `setPriceGuard` on a feed once set (ADMIN is locked out by `checkRole`): [4](#0-3) 

An attacker who seizes `stateGuard` can call `setPriceGuard(feedId, 1, type(uint128).max)`, effectively disabling the price band. The `priceGuard` is enforced in both `PriceProvider._getBidAndAskPrice()` and `AnchoredPriceProvider._readLeg()`: [5](#0-4) [6](#0-5) 

With `guardMin=1` and `guardMax=type(uint128).max`, the guard check `mid < guardMin || mid > guardMax` never triggers for any valid oracle price, removing the circuit-breaker entirely. Any oracle price — including stale or extreme values — flows into `MetricOmmPool` swap execution. This satisfies the allowed impact gate: **bad-price execution** (unbounded bid/ask quote reaches a pool swap) and **oracle role check bypassed by an unprivileged path**.

## Likelihood Explanation

The trigger requires two sequential actions by the current `stateGuard` (a non-trusted role — only factory owner and oracle admin are trusted):

1. `setStateGuardRole(feedId, nominee)` — nominating a pending guard (a legitimate earlier proposal that was later reconsidered).
2. `purgeStateGuardRole(feedId)` — intending to fully abdicate and return control to ADMIN.

This is a realistic operational sequence: a guard decides to step down after having proposed a successor. The guard's reasonable assumption is that purging their role clears all pending state. The attacker can also front-run any ADMIN `purgePendingStateGuardRole` call in the same block as the abdication, since after `purgeStateGuardRole` the `checkRole` modifier requires ADMIN_ROLE to call `purgePendingStateGuardRole`, while `acceptStateGuardRole` is permissionless for the nominee.

## Recommendation

Clear `pendingStateGuard` inside `purgeStateGuardRole`:

```solidity
function purgeStateGuardRole(bytes32 feedId) external checkRole(feedId) {
    delete stateGuard[feedId];
+   delete pendingStateGuard[feedId];   // prevent stale nominee from claiming
    emit StateGuardDeleted(feedId);
}
```

## Proof of Concept

```
1. ADMIN calls setStateGuardRole(feedId, guard1)
2. guard1 calls acceptStateGuardRole(feedId)
   → stateGuard[feedId] = guard1

3. guard1 calls setStateGuardRole(feedId, attacker)
   → pendingStateGuard[feedId] = attacker

4. guard1 decides to abdicate:
   guard1 calls purgeStateGuardRole(feedId)
   → stateGuard[feedId] = address(0)
   → pendingStateGuard[feedId] = attacker  ← NOT cleared

5. attacker calls acceptStateGuardRole(feedId)
   → pendingStateGuard[feedId] == attacker ✓
   → stateGuard[feedId] = attacker         ← unauthorized seizure

6. attacker calls setPriceGuard(feedId, 1, type(uint128).max)
   → price guard disabled; any oracle price passes PriceProvider/AnchoredPriceProvider guard check
   → pool swaps execute at unbounded prices
```
