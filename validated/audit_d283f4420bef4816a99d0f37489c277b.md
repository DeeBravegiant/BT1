### Title
Negative `collAdaBalance` via unchecked collateral return coin in `IsValid False` branch causes ADA creation and fee-pot decrement — (`eras/babbage/impl/src/Cardano/Ledger/Babbage/Collateral.hs`, `eras/babbage/impl/src/Cardano/Ledger/Babbage/Rules/Utxo.hs`)

### Summary

A transaction with no Plutus scripts, empty redeemers, `isValid = False`, and a `collateralReturnTxBodyL` coin exceeding the sum of all collateral UTxO inputs passes all ledger validation checks. The `IsValid False` branch in `updateUTxOStateByTxValidity` then coerces the resulting negative `DeltaCoin` from `collAdaBalance` directly into a negative `Coin` and adds it to `utxosFees`, decrementing the fee pot. Simultaneously, the collateral return output — containing more ADA than the collateral inputs — is inserted into the UTxO, creating ADA out of thin air.

### Finding Description

**Step 1 — `feesOK` skips `validateTotalCollateral` when redeemers are empty.** [1](#0-0) 

```haskell
-- Part 2: (txrdmrs tx ≠ ∅ ⇒ validateCollateral)
unless (null $ tx ^. witsTxL . rdmrsTxWitsL . unRedeemersL) $
  validateTotalCollateral pp txBody utxoCollateral
```

When `rdmrsTxWitsL` is empty, `validateTotalCollateral` — which enforces that collateral balance ≥ required collateral percentage and that `collAdaBalance` equals `totalCollateralTxBodyL` — is entirely skipped. This is by design for non-script transactions, but it leaves the collateral return coin completely unchecked.

**Step 2 — `collAdaBalance` returns a negative `DeltaCoin` when collateral return coin exceeds collateral input sum.** [2](#0-1) 

```haskell
collAdaBalance txBody utxoCollateral = toDeltaCoin $
  case txBody ^. collateralReturnTxBodyL of
    SNothing -> colbal
    SJust txOut -> colbal <-> (txOut ^. coinTxOutL @era)
  where
    colbal = sumAllCoin utxoCollateral
```

`Coin` is `newtype Coin = Coin Integer`. The `<->` operator is integer subtraction with no underflow guard. If `txOut ^. coinTxOutL > colbal`, the result is a negative `Coin`, which `toDeltaCoin` wraps into a negative `DeltaCoin`. No assertion or guard prevents this.

**Step 3 — `updateUTxOStateByTxValidity` coerces the negative `DeltaCoin` into `Coin` and adds it to `utxosFees`.** [3](#0-2) 

```haskell
IsValid False ->
  let !(utxoKeep, utxoDel) = extractKeys (unUTxO utxo) (txBody ^. collateralInputsTxBodyL)
      UTxO collouts = collOuts txBody
      DeltaCoin collateralFees = collAdaBalance txBody utxoDel -- NEW to Babbage
   in pure $!
        utxoState
          { utxosUtxo = UTxO (Map.union utxoKeep collouts)
          , utxosFees = utxosFees utxoState <> Coin collateralFees -- NEW to Babbage
          ...
          }
```

`DeltaCoin collateralFees = collAdaBalance txBody utxoDel` pattern-matches the `DeltaCoin` newtype to extract the raw `Integer`. `Coin collateralFees` then wraps that same `Integer` into a `Coin` — no sign check, no assertion. `utxosFees utxoState <> Coin collateralFees` is integer addition; with a negative `collateralFees`, `utxosFees` is decremented.

**Step 4 — `babbageEvalScriptsTxInvalid` succeeds with no Plutus scripts.** [4](#0-3) 

With no Plutus scripts, `plutusScriptsWithContextStAnnTx` returns `Right []`. The `?!:` check passes. `Alonzo.when2Phase` is a no-op (it only activates when 2-phase scripts are present), so no `ValidationTagMismatch` is raised for `isValid = False`. The `IsValid False` branch in `utxosTransition` completes successfully. [5](#0-4) 

**Step 5 — `validateValueNotConservedUTxO` does not cover collateral.** [6](#0-5) 

`validateValueNotConservedUTxO` checks `inputsTxBodyL` (regular inputs) against `outputsTxBodyL` (regular outputs). Collateral inputs and the collateral return output are not part of this balance check. A collateral return coin exceeding collateral input sum is invisible to value conservation.

### Correction to the Claim's Stated Precondition

The claim states "at least one Plutus script address input but zero redeemers." This specific path does **not** work: `plutusScriptsWithContextStAnnTx` would fail with `CollectErrors` when a Plutus spending input has no corresponding redeemer, causing `babbageEvalScriptsTxInvalid` to fail before reaching `updateUTxOStateByTxValidity`.

The correct and sufficient precondition is: **no Plutus scripts at all** (pure key-hash inputs/outputs), empty redeemers, `isValid = False`, and `collateralReturnTxBodyL` coin > `sumAllCoin utxoCollateral`. All other steps in the claimed call sequence are confirmed correct.

### Impact Explanation

Two simultaneous invariant violations occur in a single accepted transaction:

1. **ADA creation**: The collateral return output (coin = X + 1) is inserted into the UTxO while only X ADA was removed from collateral inputs. One lovelace is created per transaction; this is unbounded and repeatable.
2. **Fee pot decrement**: `utxosFees` is decremented by 1 lovelace per transaction. Repeated application drives `utxosFees` negative, corrupting the fee accounting used for treasury and reserve distributions.

Both effects are permanent ledger state changes. Honest nodes following the ledger rules would accept the block, causing all nodes to converge on the corrupted state. Recovery requires a hard fork.

**Impact: Critical** — Direct creation of ADA through an invalid ledger state transition; honest nodes accept an invalid block causing permanent ledger divergence.

### Likelihood Explanation

The transaction requires no privileged access. The `isValid` field is part of the serialized `Tx` submitted to the mempool; any transaction sender can set it to `False`. With no Plutus scripts, `when2Phase` is a no-op and no `ValidationTagMismatch` is raised, so any block producer following the ledger rules would include the transaction. The crafted transaction passes all `babbageUtxoValidation` checks. The exploit is locally testable with a unit test.

### Recommendation

1. **In `updateUTxOStateByTxValidity`**: Assert `collateralFees >= 0` before constructing `Coin collateralFees`, or use a safe conversion that rejects negative values.
2. **In `collAdaBalance`**: Return `Maybe DeltaCoin` or enforce a precondition that `coinTxOutL ≤ sumAllCoin utxoCollateral`, failing the transaction if violated.
3. **In `feesOK`**: Consider whether `validateTotalCollateral` should also run when `isValid = False` is set and collateral inputs are present, regardless of redeemer count.
4. **Invariant assertion**: Add an STS assertion that `utxosFees` is non-decreasing across any accepted transaction.
