Audit Report

## Title
Single Merkle Root Overwrite Enables Block Stuffing to Temporarily Invalidate Pending Claims — (`contracts/utils/MerkleDistributor/MerkleBlastPointsDistributor.sol`)

## Summary

`MerkleBlastPointsDistributor` stores only a single `currentMerkleRoot`. When `setMerkleRoot()` is called, it atomically overwrites the root and increments `currentIndex`. A user whose `claim()` transaction is pending in the mempool will have their proof verified against the new root, permanently invalidating the old proof. An attacker can use block stuffing on Blast L2 to guarantee this ordering, forcing the user to obtain a new off-chain proof before they can claim.

## Finding Description

`setMerkleRoot()` overwrites the single stored root and advances `currentIndex`:

```solidity
// L145-148
currentMerkleRoot = _merkleRootToSet;
currentMerkleRootIndex++;
currentIndex++;
```

No historical root is retained. `claim()` validates exclusively against `currentMerkleRoot`:

```solidity
// L112-113
if (!MerkleProofUpgradeable.verify(merkleProof, currentMerkleRoot, node)) {
    revert InvalidMerkleProof();
}
```

The index guard at L101–103 only rejects `index > currentIndex`, so an old index `N` remains numerically valid after `currentIndex` advances to `N+1`. This creates the race: the index check passes, but the proof check fails because the root it was generated against no longer exists.

**Attack sequence:**
1. User generates valid proof `π` for `(index=N, account, amounts)` against root `R1` (`currentIndex = N`).
2. User broadcasts `claim(N, account, ..., π)`.
3. Attacker fills blocks with high-gas spam, preventing inclusion of the user's transaction.
4. Owner's `setMerkleRoot(R2)` is included: `currentMerkleRoot = R2`, `currentIndex = N+1`.
5. Attacker stops stuffing; user's transaction is included.
6. `N > N+1` → false → index check passes.
7. `verify(π, R2, node)` → reverts `InvalidMerkleProof`.

The user's proof `π` is permanently unusable. They must obtain a new off-chain proof for index `N+1` against `R2` before they can claim.

## Impact Explanation

**Low — Block stuffing.** The attacker uses block stuffing to guarantee that the root update lands before the user's claim, forcing the user to wait for the next off-chain proof generation cycle before their Blast Points/Gold can be claimed. The "permanent freezing of unclaimed yield" escalation claimed in the report requires the owner to intentionally omit the user from the new Merkle tree — an admin action that is out of scope per the program rules. Without that assumption, the impact is temporary: the user's yield is inaccessible until a new proof is issued, which is a concrete but recoverable harm matching **Low — Block stuffing**.

## Likelihood Explanation

Blast is an L2 with low gas costs, making block stuffing economically feasible. The owner calls `setMerkleRoot()` on a predictable periodic schedule observable on-chain, so the attacker can time the stuffing window precisely. No privileged access is required by the attacker; the owner's root update is routine operation.

## Recommendation

Store a mapping of historical roots indexed by `currentMerkleRootIndex` and allow `claim()` to accept a `rootIndex` parameter, verifying the proof against `historicalRoots[rootIndex]`. Alternatively, enforce a time-lock or grace period between `setMerkleRoot()` calls during which the old root remains valid for claims.

## Proof of Concept

The submitted Foundry test correctly demonstrates the behavior: after `setMerkleRoot(root2)` replaces `currentMerkleRoot`, a `claim()` call with a proof built against `root1` reverts with `InvalidMerkleProof`. The index check (`1 > 2` → false) passes, confirming the race condition is reachable by any unprivileged caller whose transaction is delayed by block stuffing.
