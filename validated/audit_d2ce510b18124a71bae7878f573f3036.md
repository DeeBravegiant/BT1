### Title
Unguarded `assert` in `solution_to_pool_state` Crashes Pool Wallet on Malformed Inner Solution — (`chia/pools/pool_puzzles.py`)

---

### Summary

A bare `assert num_args in {2, 3}` at `pool_puzzles.py:413` inside `solution_to_pool_state` is called from both `get_current_state` and `apply_state_transition` with no surrounding exception handling. A CoinSpend whose inner_solution has a length outside `{2, 3}` (e.g., 4 elements) triggers an unhandled `AssertionError` that propagates uncaught through the pool wallet, crashing the wallet process and — if the spend is committed to the DB before the crash — permanently preventing the pool wallet from loading.

---

### Finding Description

**Root cause — unguarded assert:** [1](#0-0) 

`inner_solution` is extracted from the full solution and its Python-list length is checked with a bare `assert`. Any value outside `{2, 3}` raises `AssertionError` with no recovery path.

**`get_current_state` — no exception handling:** [2](#0-1) 

The `while pool_state is None` loop calls `solution_to_pool_state` directly. An `AssertionError` propagates uncaught to every caller of `get_current_state`.

**`apply_state_transition` — stores first, then crashes:** [3](#0-2) 

The spend is written to the DB at line 286 via `add_spend`, and only *then* does the loop at lines 291–300 call `solution_to_pool_state` on the newly stored spend. If the outer DB transaction is committed before the exception propagates, the malformed spend is persisted.

**`add_spend` — no solution validation:** [4](#0-3) 

The only guards are: duplicate check, height ordering, and parent coin name chain check. There is zero validation of the solution structure.

**`create_from_db` — no re-validation on load:** [5](#0-4) 

On wallet restart, `create_from_db` simply reconstructs the object from DB without re-validating stored spends. The first call to `get_current_state` after restart will hit the assert.

---

### Impact Explanation

- **Immediate crash**: A malicious peer that delivers a CoinSpend with the correct singleton coin name (public on-chain) but a 4-element inner_solution causes `apply_state_transition` to crash with `AssertionError` at line 294 during the `solution_to_pool_state` loop.
- **Permanent unloadability**: If the DB transaction is committed before the exception propagates, the malformed spend is persisted. On every subsequent wallet restart, `create_from_db` → `get_current_state` → `solution_to_pool_state` → `assert` fails, making the pool wallet permanently unloadable.
- **All pool operations fail**: `get_current_state` is called by `update_pool_config`, `generate_travel_transactions`, `rewind`, and RPC handlers — all become unreachable.

This maps to: **High — Corruption of pool membership state / permanent inability to process pool wallet operations.**

---

### Likelihood Explanation

The coin name check in `apply_state_transition` is the only guard: [6](#0-5) 

The singleton coin ID is public on-chain. A malicious peer connected to the wallet can provide a fabricated `CoinSpend` with the correct coin name but a malformed solution (4-element inner_solution). The wallet requests `CoinSpend` data from peers during sync and does not independently verify solution structure before calling `apply_state_transition`.

---

### Recommendation

1. Replace the bare `assert` in `solution_to_pool_state` with a proper guard that returns `None` or raises a typed exception:
   ```python
   if num_args not in {2, 3}:
       log.error(f"Unexpected inner_solution length {num_args}")
       return None
   ```
2. Wrap the `solution_to_pool_state` call in `get_current_state` and `apply_state_transition` with a `try/except Exception` that logs and returns a safe value.
3. Validate solution structure in `add_spend` or `apply_state_transition` before persisting.

---

        parent of the first spend must already be present in the DB. Note that this is not committed to the DB

            # find the most recent transition in wallet_id
            rows = list(
                await conn.execute_fetchall(
                    "SELECT transition_index, height, coin_spend "
                    "FROM pool_state_transitions "
                    "WHERE wallet_id=? "
                    "ORDER BY transition_index DESC "
                    "LIMIT 1",
                    (wallet_id,),
                )
            )
            serialized_spend = bytes(spend)
            if len(rows) == 0:
                transition_index = 0
            else:
                existing = list(
                    await conn.execute_fetchall(
                        "SELECT COUNT(*) FROM pool_state_transitions WHERE wallet_id=? AND height=? AND coin_spend=?",
                        (wallet_id, height, serialized_spend),
                    )
                )
                if existing[0][0] != 0:
                    # we already have this transition in the DB
                    return

                if height < row[1]:

                (

                    wallet_id,
                    height,
