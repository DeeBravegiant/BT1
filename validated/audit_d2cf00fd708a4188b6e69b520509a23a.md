Audit Report

## Title
Stale rsETH Price Window After Large EigenLayer Checkpoint Allows Theft of Unclaimed Yield — (`contracts/LRTOracle.sol`)

## Summary

When a completed EigenLayer checkpoint pushes `newRsETHPrice` above `highestRsethPrice * (1 + pricePercentageLimit)`, the public `updateRSETHPrice()` reverts for non-manager callers without pausing the deposit pool. The stored `rsETHPrice` remains at the pre-checkpoint (lower) value. Any depositor can mint rsETH at this stale price and, once the manager calls `updateRSETHPriceAsManager()`, redeem for more ETH than deposited, extracting yield that belongs to existing holders.

## Finding Description

`_updateRsETHPrice()` in `LRTOracle.sol` contains an upside circuit-breaker at lines 252–266. When `newRsETHPrice - highestRsethPrice > pricePercentageLimit.mulWad(highestRsethPrice)`, the function reverts with `PriceAboveDailyThreshold` for any caller lacking the `MANAGER` role. Because this is a revert, no state is written: `rsETHPrice` stays at its previous lower value and the deposit pool is not paused.

Contrast this with the downside protection at lines 270–282, which explicitly calls `lrtDepositPool.pause()`, `withdrawalManager.pause()`, and `_pause()` before returning. No equivalent pause is triggered on the upside breach path.

`LRTDepositPool.getRsETHAmountToMint()` at line 520 divides by `lrtOracle.rsETHPrice()`, which is a plain storage read of the last written value. `_beforeDeposit()` at lines 648–669 performs no freshness check on this price. Deposits therefore proceed at the stale, lower price throughout the window between the checkpoint completing and the manager calling `updateRSETHPriceAsManager()`.

The price jump originates from `getEffectivePodShares()` at lines 556–562 of `NodeDelegator.sol`, which returns `stakedButUnverifiedNativeETH + withdrawableShare`. After a checkpoint proof is submitted, EigenLayer's `podOwnerDepositShares` increases immediately, raising `withdrawableShare` and therefore `_getTotalEthInProtocol()` and `newRsETHPrice`.

Exploit flow:
1. EigenLayer checkpoint completes; `newRsETHPrice` is computed to be >1% above `highestRsethPrice`.
2. Any call to `updateRSETHPrice()` reverts; `rsETHPrice` is not updated; deposit pool remains open.
3. Attacker calls `depositETH{value: X}(0, "")` and receives `X / rsETHPrice_old` rsETH — more than the true post-checkpoint rate justifies.
4. Manager calls `updateRSETHPriceAsManager()`; `rsETHPrice` advances to `rsETHPrice_new > rsETHPrice_old`.
5. Attacker's rsETH is now redeemable for `(X / rsETHPrice_old) * rsETHPrice_new > X` ETH; the surplus is extracted from yield owed to pre-existing holders.

## Impact Explanation

**High — Theft of unclaimed yield.** The consensus rewards credited by the checkpoint represent yield accrued to existing rsETH holders. By depositing at the stale price, the attacker dilutes those holders' claim on that yield. The attacker's profit is `(X / P_old) * (P_new - P_old)` ETH, sourced entirely from yield that was not yet reflected in the price. This is a direct, quantifiable extraction of unclaimed yield by an unprivileged external caller.

## Likelihood Explanation

- `pricePercentageLimit` is documented at 1% (`1e16`); this is the intended operational value, not a misconfiguration.
- EigenLayer checkpoints accumulate consensus rewards over time. Infrequent checkpointing (days or weeks) routinely produces single-step price jumps exceeding 1%.
- The trigger condition is trivially observable on-chain: any call to `updateRSETHPrice()` that reverts with `PriceAboveDailyThreshold` signals the window is open.
- The attacker requires no special role, no front-running, and no collusion. The only requirement is that the manager has not yet responded.
- Even a short manager response latency (minutes) is sufficient for the attack to execute.

## Recommendation

1. **Freshness gate on deposits**: Record `block.timestamp` or `block.number` of the last successful `rsETHPrice` write in `_updateRsETHPrice()`. In `_beforeDeposit()`, revert if the price is older than a configurable threshold (e.g., 1 hour).
2. **Auto-pause on blocked update**: When `isPriceIncreaseOffLimit` is true and the caller is not a manager, pause the deposit pool (and optionally the withdrawal manager) before reverting, mirroring the existing downside-protection logic at lines 277–281 of `LRTOracle.sol`.
3. **Incremental price cap**: Instead of reverting, cap the public update to `highestRsethPrice * (1 + pricePercentageLimit)`, allowing the price to advance in increments without creating a stale window.

## Proof of Concept

```solidity
// Foundry fork test
// Preconditions:
//   pricePercentageLimit = 1e16 (1%)
//   Validators active; checkpoint not run for 2+ weeks
//   newRsETHPrice after checkpoint is ~3% above highestRsethPrice

// Step 1: Operator completes EigenLayer checkpoint proof
// getEffectivePodShares() now returns a higher value; newRsETHPrice > highestRsethPrice * 1.01

// Step 2: Confirm public update is blocked; rsETHPrice unchanged
uint256 stalePriceBefore = lrtOracle.rsETHPrice();
vm.expectRevert(ILRTOracle.PriceAboveDailyThreshold.selector);
lrtOracle.updateRSETHPrice();
assertEq(lrtOracle.rsETHPrice(), stalePriceBefore); // price not updated

// Step 3: Attacker deposits at stale price; deposit pool is NOT paused
vm.prank(attacker);
lrtDepositPool.depositETH{value: 100 ether}(0, "");
uint256 attackerRsETH = rsETH.balanceOf(attacker);
// attackerRsETH = 100e18 / stalePriceBefore  (inflated)

// Step 4: Manager updates price to true value
vm.prank(manager);
lrtOracle.updateRSETHPriceAsManager();
uint256 newPrice = lrtOracle.rsETHPrice(); // newPrice > stalePriceBefore

// Step 5: Assert attacker profit
uint256 attackerEthValue = attackerRsETH * newPrice / 1e18;
assertGt(attackerEthValue, 100 ether); // attacker extracted yield from existing holders
```
