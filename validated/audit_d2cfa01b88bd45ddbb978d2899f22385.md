Audit Report

## Title
Stale Cross-Chain Rate in `AGETHPoolV3` Allows Minting Excess agETH During Rate Lag Window — (`contracts/cross-chain/CrossChainRateReceiver.sol`, `contracts/agETH/AGETHPoolV3.sol`)

## Summary
`CrossChainRateReceiver.getRate()` returns the last stored rate unconditionally with no staleness check, despite tracking a `lastUpdated` timestamp. `AGETHPoolV3.deposit()` uses this rate directly to mint agETH with no freshness guard or slippage floor, meaning any depositor can exploit the structural lag window between L1 yield accrual and L2 LayerZero delivery to mint more agETH than the true rate warrants, extracting yield that belongs to existing holders.

## Finding Description
`CrossChainRateReceiver` stores both `rate` and `lastUpdated` on every `lzReceive` call, but `getRate()` returns the raw stored value with no staleness enforcement:

```solidity
// CrossChainRateReceiver.sol L95-104
rate = _rate;
lastUpdated = block.timestamp;
...
function getRate() external view returns (uint256) {
    return rate;
}
```

`AGETHPoolV3.deposit()` calls `viewSwapAgETHAmountAndFee`, which divides the deposit by the oracle rate obtained via `getRate() → IOracle(agETHOracle).getRate()` — resolving to `AGETHRateReceiver.getRate()` (the stale value above):

```solidity
// AGETHPoolV3.sol L160-168
fee = amount * feeBps / 10_000;
uint256 amountAfterFee = amount - fee;
uint256 agETHToETHrate = getRate();
agETHAmount = amountAfterFee * 1e18 / agETHToETHrate;
```

There is no `minAgETHAmountExpected` parameter, no `lastUpdated` staleness check, and no circuit-breaker on the deposit path. The rate update path (`MultiChainRateProvider.updateRate()`) dispatches a LayerZero message that takes minutes to hours to deliver, creating a structural lag window during which `R_stale < R_true`. The attacker does not need to front-run any specific transaction — they only need to deposit while the L2 rate lags the L1 rate, a condition that is always true between update cycles.

The protocol already demonstrates awareness of this pattern: `ChainlinkOracleForRSETHPoolCollateral.getRate()` explicitly checks `answeredInRound < roundID` and `timestamp == 0` to reject stale prices, but no equivalent guard exists for the cross-chain rate path.

## Impact Explanation
When `R_stale < R_true`:
- `agETHMinted = amountAfterFee * 1e18 / R_stale` (inflated)
- `agETHFair = amountAfterFee * 1e18 / R_true` (correct)
- `excess = agETHMinted - agETHFair > 0`

The attacker receives `excess` agETH for free. Because agETH is redeemable on L1 at the true rate, the attacker can bridge the excess agETH to L1 and redeem it for ETH, extracting the yield delta that should have accrued to existing holders. The pool is left with a collateral shortfall equal to `excess * R_true`, borne by existing agETH holders.

**Impact: High — Theft of unclaimed yield.**

## Likelihood Explanation
- The rate lag is structural and continuous; it exists between every update cycle.
- `updateRate()` is permissionless, but LayerZero delivery latency (typically minutes, potentially hours under congestion) always creates a window.
- No special role, key, or governance action is required. Any EOA can call `deposit()`.
- The attacker can monitor the L1 agETH price oracle off-chain and deposit on L2 whenever the lag exceeds the `feeBps` threshold.
- A 24-hour update gap with 0.5% yield accrual on a 100 ETH deposit yields ~0.5 ETH profit, well above typical gas costs.

## Recommendation
1. **Add a staleness check in `CrossChainRateReceiver.getRate()`**: define a `MAX_RATE_AGE` constant and revert if `block.timestamp - lastUpdated > MAX_RATE_AGE`.
2. **Add a `minAgETHAmountExpected` parameter to `AGETHPoolV3.deposit()`** so depositors can set a slippage floor, protecting against rate manipulation in either direction.
3. **Consider a deposit pause** triggered automatically when `lastUpdated` exceeds a threshold, mirroring the staleness pattern already used in `ChainlinkOracleForRSETHPoolCollateral`.

## Proof of Concept
```solidity
// Fork test (L2 fork, e.g. Arbitrum)
// Precondition: AGETHRateReceiver.rate = 1.05e18 (stale), true L1 rate = 1.06e18

function testStaleRateExploit() external {
    uint256 deposit = 100 ether;
    deal(attacker, deposit);

    // 1. Deposit at stale rate — no staleness check prevents this
    vm.prank(attacker);
    pool.deposit{value: deposit}("ref");
    // agETHReceived ≈ 100e18 * 1e18 / 1.05e18 ≈ 95.238e18

    uint256 agETHReceived = agETH.balanceOf(attacker);

    // 2. Simulate LZ message delivering the true rate
    vm.prank(layerZeroEndpoint);
    rateReceiver.lzReceive(srcChainId, srcAddressEncoded, 0, abi.encode(1.06e18));

    // 3. Assert attacker's agETH is worth more than deposited
    uint256 trueRate = rateReceiver.getRate(); // 1.06e18
    uint256 ethValue = agETHReceived * trueRate / 1e18;
    // ethValue ≈ 95.238e18 * 1.06e18 / 1e18 ≈ 100.952e18

    assertGt(ethValue, deposit); // ~0.952 ETH extracted from existing holders
}
```

The excess represents yield stolen from existing agETH holders, confirmable on unmodified production code with a local fork test against the deployed contracts.
