### Title
Wrong Receiver ID in `validate_delegate_action` Blocks `DeterministicStateInit` Inside Meta-Transactions - (File: runtime/runtime/src/action_validation.rs)

### Summary

`validate_delegate_action` passes the **outer transaction's receiver** (`receiver`) instead of the **inner delegate action's receiver** (`delegate_action.receiver_id()`) when validating inner actions. Because `validate_deterministic_state_init` checks that the receiver matches the derived deterministic account ID, any legitimate `DeterministicStateInit` action nested inside a meta-transaction is unconditionally rejected before `FixDelegatedDeterministicStateInit` activates.

### Finding Description

In `validate_delegate_action`, the `inner_receiver` variable is set to the outer transaction's `receiver` argument on pre-fix protocol versions:

```rust
let inner_receiver =
    if ProtocolFeature::FixDelegatedDeterministicStateInit.enabled(current_protocol_version) {
        delegate_action.receiver_id()   // correct
    } else {
        receiver                        // wrong: outer tx receiver, not delegate receiver
    };
validate_actions_with_mode(limit_config, &actions, inner_receiver, ...)?;
```

`validate_deterministic_state_init` then calls `derive_near_deterministic_account_id` and compares the result against `inner_receiver`. Because `receiver` (the outer transaction's `receiver_id`) is almost never equal to `delegate_action.receiver_id()`, the check always fails for any legitimate `DeterministicStateInit` placed inside a `Delegate` or `DelegateV2` action.

The code comment confirms this: *"This is a bug fixed with `FixDelegatedDeterministicStateInit` that validated against the wrong id. This makes it impossible to initialize deterministic accounts from meta transactions."*

The same wrong-context pattern is the root cause in the external report: the base helper (`TapiocaOmnichainEngineHelper`) is always instantiated instead of the token-specific extended helper, so the `_sanitizeMsgType` allowlist is always evaluated against the wrong set of permitted types, blocking a legitimate cross-chain flow.

### Impact Explanation

Any user who submits a meta-transaction (`SignedDelegateAction` / `DelegateV2`) whose inner action list contains a `DeterministicStateInit` receives an `InvalidDeterministicStateInitReceiver` error at the initial tx-validation stage. The transaction is rejected before it reaches the mempool or any chunk. The user flow — initializing a deterministic account via a relayer-sponsored meta-transaction — is completely unavailable on protocol versions below `FixDelegatedDeterministicStateInit`. The corrupted protocol value is the **validity decision** for this class of transactions: they are incorrectly classified as invalid.

### Likelihood Explanation

Any unprivileged user who attempts to use `DeterministicStateInit` inside a delegate action triggers the bug. No special privileges are required; the entry path is a standard signed transaction submitted through the public RPC. The feature (`DeterministicStateInit`) is a production action type, and meta-transactions are a documented user-facing feature.

### Recommendation

Ensure `delegate_action.receiver_id()` is always used as the receiver for inner-action validation, removing the pre-fix branch entirely once `FixDelegatedDeterministicStateInit` is universally activated. Until then, the protocol-version gate must be the only path, and the buggy `receiver` branch should be removed as soon as the feature version is stabilized.

### Proof of Concept

The existing test `test_deterministic_state_init_meta_tx_receiver_check_pre_fix` in `test-loop-tests/src/tests/deterministic_account_id.rs` demonstrates the bug: on `fix_version - 1`, a meta-transaction carrying a `DeterministicStateInit` inner action passes initial tx validation (because the wrong receiver is used) but then fails at receipt processing with `InvalidDeterministicStateInitReceiver`. A legitimate user constructing the same transaction with the correct `delegate_action.receiver_id()` would also fail at the initial validation stage, because `validate_delegate_action` evaluates the check against `receiver` (the outer tx receiver) rather than the delegate's own receiver. fn validate_delegate_action(
    limit_config: &LimitConfig,
    delegate_action: VersionedDelegateActionRef<'_>,
    receiver: &AccountId,
    current_protocol_version: ProtocolVersion,
    mode: ValidateReceiptMode,
) -> Result<(), ActionsValidationError> {
    let actions = delegate_action.get_actions();
    let inner_receiver =
        if ProtocolFeature::FixDelegatedDeterministicStateInit.enabled(current_protocol_version) {
            // This is the correct receiver id to use for the check.
            delegate_action.receiver_id()
        } else {
            // This is a bug fixed with `FixDelegatedDeterministicStateInit` that
            // validated against the wrong id. This makes it impossible to
            // initialize deterministic accounts from meta transactions.
            // The bug cannot be abused, if someone crafts a state init that passes
            // validation here, it will fail when it is checked as incoming receipt.
            receiver
        };
    validate_actions_with_mode(
        limit_config,
        &actions,
        inner_receiver,
        current_protocol_version,
        mode,
    )?;
    Ok(())
