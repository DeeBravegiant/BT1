### Title
Unprivileged Actor Can Register Any Stake Credential Without Owner's Consent, Blocking Atomic Register-and-Delegate and Temporarily Freezing Reward Withdrawals — (File: `eras/conway/impl/src/Cardano/Ledger/Conway/TxCert.hs`)

---

### Summary

In the Conway era, `ConwayRegCert cred SNothing` (stake credential registration without an explicit deposit) requires **no witness from the credential owner**. Any unprivileged transaction sender can register any stake credential on behalf of another address without their consent. This is the direct Cardano analog of the `depositFor` front-running vulnerability: an attacker can front-run a victim's `ConwayRegDelegCert` (atomic register-and-delegate) transaction by registering the credential first, causing the victim's transaction to fail with `StakeKeyRegisteredDELEG`. In post-bootstrap Conway, this temporarily prevents the victim from withdrawing rewards (which require DRep delegation) until they submit additional corrective transactions.

---

### Finding Description

**Root cause — no witness required for `ConwayRegCert _ SNothing`:**

`getVKeyWitnessConwayTxCert` and `getScriptWitnessConwayTxCert` both return `Nothing` for the `ConwayRegCert _ SNothing` constructor, meaning the ledger demands no authorization from the credential owner:

```haskell
-- eras/conway/impl/src/Cardano/Ledger/Conway/TxCert.hs
getVKeyWitnessConwayTxCert = \case
  ConwayTxCertDeleg delegCert ->
    case delegCert of
      ConwayRegCert _ SNothing -> Nothing          -- ← no witness required
      ConwayRegCert cred (SJust _) -> credKeyHashWitness cred
      ...
```

The code comment acknowledges this is a transitional design:

> *"We preserve the old behavior of not requiring a witness for staking credential registration, but only during the transitional period of Conway era and only for staking credential registration certificates without a deposit. Future eras will require a witness for registration certificates, because the one without a deposit will be removed."*

**DELEG rule processes the certificate unconditionally:**

In `conwayDelegTransition`, the `ConwayRegCert stakeCred SNothing` branch only checks that the credential is not already registered, then registers it — no ownership check:

```haskell
-- eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Deleg.hs
ConwayRegCert stakeCred sMayDeposit -> do
  forM_ sMayDeposit checkDepositAgainstPParams
  checkStakeKeyNotRegistered stakeCred          -- only uniqueness check
  pure $
    certState
      & certDStateL . accountsL
        %~ registerConwayAccount stakeCred ppKeyDepositCompact Nothing
```

The deposit (`ppKeyDepositCompact`) is charged to the **attacker's** UTxO, not the victim's. The victim's credential is registered with no delegation (`Nothing`).

**Attack path — front-running `ConwayRegDelegCert`:**

A victim intends to submit `ConwayRegDelegCert stakeCred delegatee deposit` (atomic register + delegate to a DRep/pool, which *does* require a witness). An attacker observing the mempool submits `ConwayRegCert stakeCred SNothing` first (no witness needed, only transaction fees + deposit). When the victim's transaction is processed, `checkStakeKeyNotRegistered` fires `StakeKeyRegisteredDELEG`, and the victim's transaction is rejected.

The victim's credential is now registered but with **no DRep delegation**. In post-bootstrap Conway, reward withdrawals require DRep delegation (`ConwayWdrlNotDelegatedToDRep`). The victim's accumulated rewards are therefore unwithdrawable until they submit a separate `ConwayDelegCert` transaction.

---

### Impact Explanation

**Impact: Medium** — Attacker-controlled certificates modify the victim's staking state outside design parameters, temporarily blocking reward withdrawals.

1. The victim's atomic `ConwayRegDelegCert` transaction is rejected; the victim loses the transaction fee and must resubmit.
2. The victim's stake credential is registered without DRep delegation. In post-bootstrap Conway, `ConwayWdrlNotDelegatedToDRep` prevents reward withdrawals until the victim submits a corrective `ConwayDelegCert`.
3. The attacker pays the key deposit (`ppKeyDeposit`) but the victim recovers it upon unregistration — making this a net-loss griefing attack, viable against high-value targets.

This fits the allowed Medium impact: *"Attacker-controlled transactions… modify fees, deposits, refunds, rewards, treasury donations, or withdrawals outside design parameters."*

---

### Likelihood Explanation

**Likelihood: Medium.**

- The attack requires only a standard transaction with a `ConwayRegCert cred SNothing` certificate — no special privileges, keys, or majority.
- The attacker must observe the victim's pending transaction in the mempool and submit their own transaction before the victim's is included. Block producers can reorder transactions within a block, and a higher-fee transaction from the attacker can achieve priority.
- The economic cost (paying the key deposit, which the victim recovers) limits casual attacks but does not prevent targeted griefing of large delegators or DRep candidates.
- The attack surface is active for the entire Conway transitional period until the no-deposit registration variant is removed.

---

### Recommendation

1. **Require a witness for all stake credential registrations in Conway**, including `ConwayRegCert cred SNothing`. The code already acknowledges this is the intended future direction. Accelerating this change closes the attack surface immediately.
2. As an interim measure, require that `ConwayRegCert` always carry an explicit deposit (`SJust`), making the no-witness path unreachable.
3. Alternatively, validate that the transaction submitter is the credential owner for registration certificates, consistent with how deregistration and delegation are handled.

### Proof of Concept

```
-- Attacker observes victim's pending transaction:
--   ConwayRegDelegCert victimCred (DelegVote drepCred) deposit
--   (requires victimCred's witness)

-- Attacker submits first (no witness needed):
Tx {
  certs = [ ConwayRegCert victimCred SNothing ]
  -- attacker pays ppKeyDeposit from their own UTxO
}

-- Victim's transaction now fails:
-- DELEG: checkStakeKeyNotRegistered victimCred
--   → StakeKeyRegisteredDELEG victimCred

-- Result:
-- victimCred is registered with no DRep delegation.
-- In post-bootstrap Conway, victim cannot withdraw rewards:
--   ConwayWdrlNotDelegatedToDRep [victimKeyHash]
-- Victim must submit an additional ConwayDelegCert to fix delegation,
-- incurring extra fees and a delay of at least one slot.
```

-- `getVKeyWitnessConwayTxCert` we preserve the old behavior of not requiring a witness

-- era and only for staking credential registration certificates without a deposit. Future

getScriptWitnessConwayTxCert ::
  ConwayTxCert era ->
  Maybe ScriptHash
getScriptWitnessConwayTxCert = \case
  ConwayTxCertDeleg delegCert ->
    case delegCert of
      ConwayRegCert _ SNothing -> Nothing
      ConwayRegCert cred (SJust _) -> credScriptHash cred
      ConwayUnRegCert cred _ -> credScriptHash cred
      ConwayDelegCert cred _ -> credScriptHash cred
      ConwayRegDelegCert cred _ _ -> credScriptHash cred
  -- PoolIds can't be Scripts
  ConwayTxCertPool {} -> Nothing
  ConwayTxCertGov govCert -> govWitness govCert
  where
    govWitness :: ConwayGovCert -> Maybe ScriptHash
    govWitness = \case
      ConwayAuthCommitteeHotKey coldCred _hotCred -> credScriptHash coldCred
      ConwayResignCommitteeColdKey coldCred _ -> credScriptHash coldCred
      ConwayRegDRep cred _ _ -> credScriptHash cred
      ConwayUnRegDRep cred _ -> credScriptHash cred
      ConwayUpdateDRep cred _ -> credScriptHash cred
