Audit Report

## Title
No Price Staleness Check in `ChainlinkPriceOracle.getAssetPrice()` Allows Stale Oracle Price to Mint Incorrect rsETH Amounts - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary
`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards all return values except `answer`, performing no staleness validation. A stale inflated price flows directly into rsETH mint calculations, allowing an attacker to deposit LST assets at an outdated high price and receive more rsETH than their assets are worth, diluting all existing rsETH holders.

## Finding Description
In `contracts/oracles/ChainlinkPriceOracle.sol` at line 52, `getAssetPrice()` executes:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
``` [1](#0-0) 

All five return values — `roundId`, `answer`, `startedAt`, `updatedAt`, `answeredInRound` — are available, but only `answer` is used. There is no check that `answeredInRound >= roundId`, `updatedAt != 0`, or that `updatedAt` falls within an acceptable heartbeat window.

This price is consumed by the public deposit path: `depositAsset()`/`depositETH()` → `_beforeDeposit()` → `getRsETHAmountToMint()` → `lrtOracle.getAssetPrice()` → `ChainlinkPriceOracle.getAssetPrice()`. [2](#0-1) [3](#0-2) 

The `minRSETHAmountExpected` slippage guard in `_beforeDeposit()` does not protect against this — it only ensures the caller receives at least their expected amount; it does not prevent over-minting when the oracle price is stale and inflated.

The codebase already demonstrates the correct pattern in `contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol`:

```solidity
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
``` [4](#0-3) 

`ChainlinkPriceOracle` — the oracle used for the core deposit path — has none of these guards.

## Impact Explanation
**Critical — Protocol insolvency / Direct theft of user funds.**

When a Chainlink LST/ETH feed is stale with an inflated price (e.g., the asset has crashed but the feed has not updated), an attacker deposits LST assets at the stale high price and receives rsETH valued at more ETH than was actually deposited. The rsETH total supply grows relative to actual backing ETH, permanently diluting all existing rsETH holders. Repeated exploitation drains the backing ratio, leading to protocol insolvency. This constitutes direct theft from existing rsETH holders whose shares are devalued.

## Likelihood Explanation
Chainlink feeds can go stale during L1/L2 network congestion, sequencer outages on L2 deployments, or when a feed approaches deprecation. The staleness window can last minutes to hours. The attack requires no special role — any address can call `depositAsset()` or `depositETH()` permissionlessly. The attacker only needs to observe that the on-chain `updatedAt` timestamp is old and that the stale price is above the true market price, both of which are publicly readable on-chain. The attack is repeatable for as long as the feed remains stale.

## Recommendation
Apply staleness checks in `ChainlinkPriceOracle.getAssetPrice()`, consistent with the pattern already used in `ChainlinkOracleForRSETHPoolCollateral`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);

    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) = priceFeed.latestRoundData();

    require(answeredInRound >= roundId, "Stale price");
    require(updatedAt != 0, "Round not complete");
    require(price > 0, "Invalid price");

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

Optionally, add a configurable per-feed heartbeat check: `require(block.timestamp - updatedAt <= maxStaleness[asset], "Price too old")`.

## Proof of Concept
**Exploit sequence (fork test against mainnet/L2 deployment):**

1. Identify a Chainlink LST/ETH feed registered in `ChainlinkPriceOracle` where `updatedAt` is older than the feed's documented heartbeat (publicly readable via `latestRoundData()`).
2. Confirm the stale price is above the current true market price (e.g., stale price = 1.05e18, true price = 0.95e18).
3. Call `LRTDepositPool.depositAsset(lstToken, 100e18, minRsETH, "")`.
4. `_beforeDeposit()` calls `getRsETHAmountToMint(lstToken, 100e18)`.
5. `getRsETHAmountToMint()` calls `lrtOracle.getAssetPrice(lstToken)` → `ChainlinkPriceOracle.getAssetPrice()`.
6. `(, int256 price,,,) = priceFeed.latestRoundData()` returns stale `1.05e18` with no revert.
7. `rsethAmountToMint = (100e18 * 1.05e18) / rsETHPrice` — attacker receives rsETH backed by 105 ETH, having deposited only ~95 ETH worth of assets.
8. Existing rsETH holders are diluted by the 10 ETH discrepancy per 100 LST deposited.

**Foundry fork test plan:** Fork mainnet, mock or warp `block.timestamp` past the feed heartbeat without updating the feed, call `depositAsset()`, assert that `rsETHPrice()` (total ETH backing / rsETH supply) has decreased relative to pre-deposit value, confirming dilution.
