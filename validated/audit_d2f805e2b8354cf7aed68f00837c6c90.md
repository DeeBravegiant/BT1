Audit Report

## Title
Unbounded Withdrawal Queue Growth Enables Temporary Freezing of Legitimate User Withdrawals - (File: `contracts/LRTWithdrawalManager.sol`)

## Summary

Any rsETH holder can flood the global withdrawal queue with minimum-sized entries before legitimate users queue their requests. Because `nextLockedNonce[asset]` advances strictly sequentially and `completeWithdrawal` enforces that a user's nonce must be strictly less than `nextLockedNonce[asset]`, every attacker-inserted entry must be individually processed by the operator before any later legitimate user's withdrawal can be finalized. This causes a concrete, code-enforced temporary freeze of legitimate users' funds.

## Finding Description

**Root cause â€” shared sequential nonce with no per-user isolation:**

`_addUserWithdrawalRequest` assigns the current `nextUnusedNonce[asset]` to the caller and increments it globally:

```solidity
// LRTWithdrawalManager.sol L756-757
userAssociatedNonces[asset][msg.sender].pushBack(nextUnusedNonce_);
nextUnusedNonce[asset] = nextUnusedNonce_ + 1;
```

`initiateWithdrawal` is callable by any rsETH holder with no per-address cap on pending requests. The only floor is `minRsEthAmountToWithdraw[asset]`, which is admin-settable and can be zero or negligible.

**Sequential unlock bottleneck:**

`_unlockWithdrawalRequests` always starts from `nextLockedNonce[asset]` and advances upward:

```solidity
// LRTWithdrawalManager.sol L790-814
while (nextLockedNonce_ < firstExcludedIndex) {
    ...
    unchecked { nextLockedNonce_++; }
}
nextLockedNonce[asset] = nextLockedNonce_;
```

The operator passes `firstExcludedIndex` as an upper bound only â€” there is no mechanism to set a lower bound or skip entries. The loop always begins at the current `nextLockedNonce[asset]`.

**Completion gate:**

`_processWithdrawalCompletion` enforces that a user's nonce must be strictly less than `nextLockedNonce[asset]`:

```solidity
// LRTWithdrawalManager.sol L707
if (usersFirstWithdrawalRequestNonce >= nextLockedNonce[asset]) revert WithdrawalLocked();
```

**Attack path:**

1. Attacker deposits ETH/LST into `LRTDepositPool` to obtain rsETH.
2. Attacker calls `initiateWithdrawal` N times with `minRsEthAmountToWithdraw[asset]` rsETH each, inserting N entries at nonces 0â€¦N-1.
3. Legitimate user calls `initiateWithdrawal`; their nonce is N.
4. Operator calls `unlockQueue` â€” the while loop must traverse all N attacker entries before `nextLockedNonce` reaches N.
5. Until all N attacker entries are processed (potentially across many transactions due to block gas limits), the legitimate user's `completeWithdrawal` reverts with `WithdrawalLocked`.

**Secondary capacity exhaustion:**

Each attacker request commits `expectedAssetAmount` to `assetsCommitted[asset]`. `getAvailableAssetAmount` returns `totalAssets - assetsCommitted[asset]`, so if the attacker's aggregate committed amount approaches total deposits, new `initiateWithdrawal` calls revert with `ExceedAmountToWithdraw`:

```solidity
// LRTWithdrawalManager.sol L170
if (expectedAssetAmount > getAvailableAssetAmount(asset)) revert ExceedAmountToWithdraw();
```

**Existing guards are insufficient:**

- `minRsEthAmountToWithdraw` is admin-settable and may be zero or very small; even at a non-zero value, the attacker recovers their rsETH after the delay, making the cost purely gas.
- `nonReentrant` and `whenNotPaused` do not limit per-address queue depth.
- The operator's `unlockQueue` cannot skip attacker entries â€” `firstExcludedIndex` only caps the upper bound of the iteration.

## Impact Explanation

**Medium â€” Temporary freezing of funds.** Legitimate users who have already queued withdrawals receive a code-enforced `WithdrawalLocked` revert on `completeWithdrawal` until the operator has processed all preceding attacker entries. With thousands of attacker entries and block gas limits, this delay can span many blocks or days. The secondary capacity-exhaustion effect can additionally prevent new withdrawal initiations, compounding the freeze.

## Likelihood Explanation

Any rsETH holder can execute this attack. rsETH is freely obtainable by depositing ETH or supported LSTs into `LRTDepositPool`. The attacker's cost is gas for N `initiateWithdrawal` calls plus temporary lock-up of rsETH, which is fully recovered after the withdrawal delay. With a small or zero `minRsEthAmountToWithdraw`, the rsETH cost per entry is minimal, making the attack low-cost relative to its impact and repeatable.

## Recommendation

1. **Enforce a meaningful minimum per withdrawal request** â€” set `minRsEthAmountToWithdraw` to a value that makes flooding economically prohibitive.
2. **Add a per-address cap on pending withdrawal requests** â€” reject `initiateWithdrawal` if `userAssociatedNonces[asset][msg.sender].length()` exceeds a protocol-defined maximum.
3. **Decouple per-user completion from the global unlock nonce** â€” maintain per-user unlock state so processing one user's requests does not block another's.
4. **Allow `unlockQueue` to skip entries** â€” permit the operator to advance `nextLockedNonce` past entries that have zero payout or belong to a known spammer, without requiring sequential processing.
