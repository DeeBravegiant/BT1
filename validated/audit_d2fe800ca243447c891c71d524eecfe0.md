### Title
Unverified CKD Response for `AppPublicKey` Variant Allows Byzantine Participant to Deliver Arbitrary Key Material - (File: crates/contract/src/lib.rs)

---

### Summary

The `respond_ckd` function in the MPC contract performs **no cryptographic verification** of the CKD response when the `AppPublicKey` (privately verifiable, legacy) variant is used. A single Byzantine attested participant — strictly below the signing threshold — can race to call `respond_ckd` with an arbitrary `(big_y, big_c)` pair, consuming the pending CKD request and delivering attacker-controlled key material to the requesting application.

---

### Finding Description

In `respond_ckd`, the contract branches on the request's `app_public_key` variant:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← NO VERIFICATION
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV`, the contract runs `ckd_output_check`, which verifies the pairing equation `e(big_c, G2) = e(big_y, app_pk2) · e(H(pk, app_id), mpc_pk)` on-chain. [2](#0-1) 

For `AppPublicKey` (the default/legacy variant used in all README examples and the `ckd-example-cli`), the branch is an empty no-op. Any `(big_y, big_c)` pair is accepted unconditionally and delivered to the requesting app via `resolve_yields_for`. [3](#0-2) 

The only guards on `respond_ckd` are that the caller is the signer account and is an attested participant — there is no requirement that the caller be the designated coordinator, and no check that the response is cryptographically consistent with the MPC master key and the app's `app_id`. [4](#0-3) 

The `AppPublicKey` variant is the primary usage path. The CKD design document explicitly states that for this variant, "Verification is performed by the app after decryption" — but the contract does not enforce this, and many integrations may omit the off-chain check. [5](#0-4) 

---

### Impact Explanation

**Vulnerability class:** Request-lifecycle / confidential-key-derivation output corruption.

A Byzantine attested participant (one node, below the signing threshold) can:

1. Observe a pending `AppPublicKey` CKD request on-chain. The `app_public_key = A = a·G1` is visible in the request.
2. Construct an arbitrary pair: choose scalar `y` and known G1 point `K`, set `big_y = y·G1`, `big_c = A·y + K`.
3. Call `respond_ckd` before the legitimate coordinator. The contract accepts the response with no verification.
4. The yield is resolved; the app receives `(big_y, big_c)`.
5. The app decrypts: `secret = big_c − big_y·a = A·y + K − y·a·G1 = K`. The attacker knows `K`, so they know the app's derived secret.

If the app performs the intended off-chain BLS verification (`e(secret, G2) = e(H(pk, app_id), mpc_pk)`), it detects the forgery — but the request is already consumed. The app must resubmit, and the attacker can repeat the attack indefinitely (persistent DoS of the CKD service for that app).

If the app omits the off-chain verification (a realistic scenario for integrations that treat the MPC contract as a trusted oracle), it uses an attacker-controlled key for all subsequent operations (encryption, signing), constituting a confidential key derivation output substitution.

This breaks the core safety invariant of the CKD feature: that the derived key is uniquely determined by `(app_id, mpc_master_key)` and is unknown to any MPC participant.

---

### Likelihood Explanation

- The `AppPublicKey` variant is the default and legacy path; it is the variant shown in all documentation examples and the `ckd-example-cli`.
- Any single attested participant can call `respond_ckd` — there is no coordinator-only restriction in the contract.
- The attack requires no threshold collusion: one Byzantine node suffices.
- The `app_public_key` is public on-chain, giving the attacker all inputs needed to craft a plausible-looking but attacker-controlled response.

---

### Recommendation

1. **Deprecate `AppPublicKey` in favor of `AppPublicKeyPV`** for all new integrations. The `AppPublicKeyPV` variant already has correct on-chain verification via `ckd_output_check`.
2. **Add on-chain verification for `AppPublicKey`**: since the G2 component is absent, the contract cannot run the full pairing check. However, it can verify that `big_s = big_c − big_y·(app_public_key)` satisfies `e(big_s, G2) = e(H(pk, app_id), mpc_pk)` — but this requires knowing `a`, which only the app knows. The correct fix is to require `AppPublicKeyPV` for all new requests.
3. **Document clearly** that `AppPublicKey` responses are unverified on-chain and that apps MUST perform the off-chain BLS verification before using the derived key.

---

### Proof of Concept

```
1. Alice submits request_app_private_key with AppPublicKey = A (visible on-chain).
2. Byzantine participant Bob (attested, below threshold) observes the pending request.
3. Bob chooses y = 1, K = G1 (known point).
   Sets big_y = G1, big_c = A + G1.
4. Bob calls respond_ckd(request, {big_y, big_c}).
   Contract: AppPublicKey branch → no verification → resolve_yields_for succeeds.
5. Alice receives {big_y=G1, big_c=A+G1}.
6. Alice decrypts: secret = (A + G1) − 1·a·G1 = a·G1 + G1 − a·G1 = G1.
7. Bob knows secret = G1 (the generator). Alice's "confidential" derived key is public.
8. If Alice skips off-chain verification, she uses G1 as her key for all operations.
   If Alice verifies, the check fails and she must resubmit — Bob repeats step 3.
```

pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}
