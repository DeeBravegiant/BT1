Audit Report

## Title
ETH Deposit Limit Bypass Due to Missing `amount` in Limit Check - (File: contracts/LRTDepositPool.sol)

## Summary
`LRTDepositPool._checkIfDepositAmountExceedesCurrentLimit` omits the incoming `amount` from the ETH branch of its limit comparison, checking only whether pre-deposit totals already exceed the cap. The ERC20 branch correctly includes `amount`. This allows any depositor to push ETH deposits arbitrarily above `depositLimitByAsset(ETH_TOKEN)` in a single transaction, minting rsETH beyond the protocol's configured TVL cap.

## Finding Description
In `contracts/LRTDepositPool.sol` at L676–682:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)); // ← amount omitted
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // ← correct
}
``` [1](#0-0) 

The ETH branch evaluates the pre-deposit state only. As long as `totalAssetDeposits <= depositLimitByAsset(ETH_TOKEN)` before the call, the check returns `false` regardless of how large `amount` is, and `_beforeDeposit` does not revert. [2](#0-1) 

The full call path:
1. User calls `depositETH{value: X}(...)` — `msg.value` is passed as `depositAmount`.
2. `_beforeDeposit(ETH_TOKEN, msg.value, ...)` invokes `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, msg.value)`.
3. The ETH branch ignores `msg.value`; returns `false` whenever pre-deposit total ≤ limit.
4. `_mintRsETH` mints rsETH for the full deposit with no cap enforcement. [3](#0-2) 

No other guard in `depositETH` or `_beforeDeposit` enforces the deposit limit for ETH.

## Impact Explanation
The `depositLimitByAsset` cap is a governance-set protocol invariant bounding per-asset TVL exposure. The ETH path fails to enforce it: total ETH deposits can exceed the configured limit by an unbounded amount in a single transaction, with rsETH minted for the excess. This is a concrete instance of **Low — Contract fails to deliver promised returns**: the deposit ceiling invariant is broken for ETH while it is correctly enforced for all ERC20 assets.

## Likelihood Explanation
Any unprivileged ETH depositor can trigger this with no special role, front-running, or external dependency. The only precondition is that `totalAssetDeposits(ETH_TOKEN) <= depositLimitByAsset(ETH_TOKEN)` at the time of the call — i.e., the protocol has not already exceeded its own limit, which is the normal operating state. The condition is always reachable for any active depositor and is repeatable across blocks.

## Recommendation
Remove the ETH-specific branch and apply the same `+ amount` logic uniformly:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
``` [1](#0-0) 

## Proof of Concept
**Setup:** `depositLimitByAsset(ETH_TOKEN) = 100 ether`, `getTotalAssetDeposits(ETH_TOKEN) = 99.9 ether`.

1. Attacker calls `depositETH{value: 50 ether}(0, "")`.
2. `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 50 ether)` evaluates `99.9 ether > 100 ether` → `false`.
3. `_beforeDeposit` does not revert; `_mintRsETH` mints rsETH for 50 ETH.
4. Post-deposit: `getTotalAssetDeposits(ETH_TOKEN) = 149.9 ether` — 49.9 ETH above the limit.

**Contrast (ERC20):** Same conditions with an ERC20 asset: step 2 evaluates `99.9 + 50 > 100` → `true`, reverting with `MaximumDepositLimitReached`.

**Foundry test plan:** Deploy `LRTDepositPool` with a mock `lrtConfig` returning `depositLimitByAsset = 100 ether`. Pre-seed `getTotalAssetDeposits` to return `99.9 ether`. Call `depositETH{value: 50 ether}`. Assert the call succeeds and `getTotalAssetDeposits` returns `149.9 ether`, confirming the limit was not enforced. Then apply the fix and assert the same call reverts with `MaximumDepositLimitReached`.
