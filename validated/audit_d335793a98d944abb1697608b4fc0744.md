### Title
`RequireGuard (KeyHashObj kh)` Native Script Does Not Verify Key Signature — Only Checks Self-Declared Guards Field - (File: `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Scripts.hs`)

---

### Summary

In the Dijkstra era, the `evalDijkstraNativeScript` evaluator handles the `RequireGuard (KeyHashObj kh)` constructor by checking only that `kh` is present in the transaction body's `guardsTxBodyL` field — a field the transaction submitter controls freely. It never verifies that `kh` actually signed the transaction. Any unprivileged sender can therefore satisfy a `RequireGuard (KeyHashObj kh)` script by self-declaring the target key hash in their own transaction body, spending UTxOs they do not own.

---

### Finding Description

**Vulnerability class:** Insufficient return-value / authorization check — the evaluator calls a membership test against a self-supplied set instead of verifying the cryptographic authorization that the constructor semantically implies.

**Root cause — `evalDijkstraNativeScript`:**

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Scripts.hs  line 575
RequireGuard cred -> cred `OSet.member` guards
```

`guards` is bound from the transaction body itself:

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Tx.hs  lines 325-328
validateDijkstraNativeScript tx =
  evalDijkstraNativeScript vhks (tx ^. bodyTxL . vldtTxBodyL) (tx ^. bodyTxL . guardsTxBodyL)
  where
    vhks = Set.map witVKeyHash (tx ^. witsTxL . addrTxWitsL)
```

`vhks` — the set of keys that **actually signed** the transaction — is passed to `evalDijkstraNativeScript` as the first argument and is correctly used for `RequireSignature`:

```haskell
RequireSignature hash -> hash `Set.member` keyHashes   -- keyHashes = vhks
```

But for `RequireGuard (KeyHashObj kh)`, the evaluator ignores `vhks` entirely and instead checks membership in `guards = tx ^. bodyTxL . guardsTxBodyL`. The `guardsTxBodyL` field is authored and signed by the **transaction submitter**, not by the holder of `kh`. Any submitter can write any key hash into that field.

**No compensating check exists in the UTXOW rule.** The Dijkstra UTXOW transition (`dijkstraUtxowTransition`) calls:

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Utxow.hs  lines 276-277
{- witsVKeyNeeded utxo tx genDelegs ⊆ witsKeyHashes -}
runTest $ Shelley.validateNeededWitnesses witsKeyHashes certState originalUtxo txBody
```

`getWitsVKeyNeeded` delegates to `getConwayWitsVKeyNeeded`, which does not include key-hash guards. The `getDijkstraScriptsNeeded` function adds only **script-credential** guards to `scriptsNeeded`:

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs  lines 173-176
guardingScriptsNeeded = AlonzoScriptsNeeded $
  catMaybes $
    zipAsIxItem (txb ^. guardsTxBodyL) $
      \(AsIxItem idx cred) -> (\sh -> (GuardingPurpose (AsIxItem idx sh), sh)) <$> credScriptHash cred
```

`credScriptHash` returns `Nothing` for `KeyHashObj`, so key-hash guards are silently excluded from `scriptsNeeded` and from the witness-needed set. There is no path that enforces a signature from the guarded key.

The test suite confirms the behavior — the test `"Spending inputs locked by script requiring a keyhash guard"` passes by adding the key hash to `guardsTxBodyL` with no corresponding signature:

```haskell
-- eras/dijkstra/impl/testlib/Test/Cardano/Ledger/Dijkstra/Imp/UtxowSpec.hs  line 38
submitTx_ $ tx & bodyTxL . guardsTxBodyL .~ [guardKeyHash]
```

The `impDijkstraSatisfyNativeScript` helper also documents the gap explicitly:

```haskell
-- eras/dijkstra/impl/testlib/Test/Cardano/Ledger/Dijkstra/ImpTest.hs  lines 153-156
-- TODO: actual satisfy the native scripts by updating the transaction's guards
ns@(RequireGuard _)
  | evalDijkstraNativeScript mempty vi guards ns -> pure $ Just mempty
  | otherwise -> pure Nothing
```

`Just mempty` means zero additional signing keys are required — confirming that no signature enforcement exists.

---

### Impact Explanation

**Critical — Direct loss of ADA or native assets through an invalid ledger state transition.**

Any UTxO whose payment credential is a `RequireGuard (KeyHashObj kh)` native script can be spent by an arbitrary transaction submitter who does not hold the private key for `kh`. The submitter simply includes `kh` in the `guardsTxBodyL` field of their transaction body. The ledger accepts the transaction as phase-1 valid, removes the UTxO from the set, and credits the attacker's output. The legitimate owner of `kh` loses their funds with no recourse short of a hard fork.

---

### Likelihood Explanation

**High.** The attack requires no special privilege, no governance majority, no leaked key, and no off-chain coordination. Any transaction submitter who can observe the chain, identify a UTxO locked by a `RequireGuard (KeyHashObj kh)` script, and construct a valid transaction body (trivially possible with any wallet or CLI tool) can execute the exploit. The only prerequisite is that the Dijkstra era is active on the network.

---

### Recommendation

In `evalDijkstraNativeScript`, the `RequireGuard (KeyHashObj kh)` branch must verify that `kh` is in the set of keys that actually signed the transaction (`keyHashes`/`vhks`), not in the self-declared `guards` field:

```haskell
-- Current (vulnerable):
RequireGuard cred -> cred `OSet.member` guards

-- Corrected for KeyHashObj:
RequireGuard (KeyHashObj kh) -> kh `Set.member` keyHashes
RequireGuard (ScriptHashObj _) -> cred `OSet.member` guards  -- script path unchanged
```

Alternatively, `getDijkstraScriptsNeeded` and `getConwayWitsVKeyNeeded` (or its Dijkstra override) must be extended to include key-hash guards in the required-witness set, so that `validateNeededWitnesses` enforces the signature independently of the script evaluator.

---

### Proof of Concept

1. Alice locks 1000 ADA in a UTxO with payment script `RequireGuard (KeyHashObj aliceKH)`.
2. Attacker Bob, who does not hold Alice's private key, constructs a transaction:
   - `inputs = {aliceUTxO}`
   - `outputs = {bobAddress: 1000 ADA}`
   - `guardsTxBodyL = [aliceKH]`  ← self-declared, no signature from Alice
   - `witsTxL = {bobSignature}`   ← only Bob's own signature (for fee inputs)
3. The UTXOW rule calls `validateFailedBabbageScripts`, which calls `validateNativeScript tx aliceScript`.
4. `validateDijkstraNativeScript` calls `evalDijkstraNativeScript vhks vi guards aliceScript`.
5. `evalDijkstraNativeScript` evaluates `RequireGuard (KeyHashObj aliceKH)` as `aliceKH `OSet.member` guards` → `True` (because Bob listed it).
6. The script passes. The UTXOW rule finds no missing witnesses. The transaction is accepted. Alice's 1000 ADA is transferred to Bob.

```haskell
evalDijkstraNativeScript ::
  (DijkstraEraScript era, NativeScript era ~ DijkstraNativeScript era) =>
  Set.Set (KeyHash Witness) ->
  ValidityInterval ->
  OSet (Credential Guard) ->
  NativeScript era ->
  Bool
evalDijkstraNativeScript keyHashes (ValidityInterval txStart txExp) guards = go
  where
    -- the evaluation will stop as soon as it reaches the required number of valid scripts
    isValidMOf n SSeq.Empty = n <= 0
    isValidMOf n (ts SSeq.:<| tss) =
      n <= 0 || if go ts then isValidMOf (n - 1) tss else isValidMOf n tss
    go = \case
      RequireTimeStart lockStart -> lockStart `lteNegInfty` txStart
      RequireTimeExpire lockExp -> txExp `ltePosInfty` lockExp
      RequireSignature hash -> hash `Set.member` keyHashes
      RequireAllOf xs -> all go xs
      RequireAnyOf xs -> any go xs
      RequireMOf m xs -> isValidMOf m xs
      RequireGuard cred -> cred `OSet.member` guards
      _ -> error "Impossible: All NativeScripts should have been accounted for"
```
