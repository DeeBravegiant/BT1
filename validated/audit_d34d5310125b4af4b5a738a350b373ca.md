### Title
Malicious Third-Party Harvester Can Unconditionally Redirect Farmer Block Rewards via Unvalidated `farmer_reward_address_override` in `RespondSignatures` — (`File: chia/farmer/farmer_api.py`)

### Summary
The CHIP-22 fee convention allows a third-party harvester to redirect the farmer's block reward to an arbitrary address by setting `farmer_reward_address_override`. The farmer's only guard — `notify_farmer_reward_taken_by_harvester_as_fee()` — is a **logging-only** function that never blocks the override. Furthermore, when the override arrives a second time inside `RespondSignatures`, `_process_respond_signatures()` accepts it with **zero validation**. A malicious harvester can therefore divert every farmer block reward (1.75 XCH) to an attacker-controlled puzzle hash.

### Finding Description

**CHIP-22 design**: A connected harvester may set `farmer_reward_address_override` in `NewProofOfSpace` to claim the farmer reward as a fee. The farmer is expected to verify the "fee quality" (a deterministic hash of the proof and challenge) against the harvester-supplied `applied_fee_threshold`.

**Guard is logging-only, never enforcing**

In `FarmerAPI.new_proof_of_space()`, when `farmer_reward_address_override` is non-`None`, the farmer calls:

```python
if new_proof_of_space.farmer_reward_address_override is not None:
    self.farmer.notify_farmer_reward_taken_by_harvester_as_fee(sp, new_proof_of_space)
``` [1](#0-0) 

Inside `notify_farmer_reward_taken_by_harvester_as_fee()`, when the fee quality check fails (i.e., `fee_quality > fee_threshold`, meaning the harvester is not entitled to the reward), the farmer only emits a `log.warning` and **returns normally** — it never raises, never returns an error, and never prevents the override from being used:

```python
else:
    self.log.warning(
        "Harvester illegitimately took a fee reward that "
        + "did not belong to it or it incorrectly applied the fee convention."
    )
``` [2](#0-1) 

**Second override in `RespondSignatures` has no check at all**

After the farmer sends `RequestSignatures` to the harvester, the harvester's `RespondSignatures` message also carries a `farmer_reward_address_override` field. In `_process_respond_signatures()`, this field is accepted and used unconditionally:

```python
farmer_reward_address = self.farmer.farmer_target
if response.farmer_reward_address_override is not None:
    farmer_reward_address = response.farmer_reward_address_override
    include_source_signature_data = True
``` [3](#0-2) 

There is no fee quality check, no threshold comparison, and no rejection path. The resulting `farmer_reward_address` is placed directly into `DeclareProofOfSpace` and broadcast to the full node:

```python
return farmer_protocol.DeclareProofOfSpace(
    ...
    farmer_reward_address,
    ...
)
``` [4](#0-3) 

The `RespondSignatures` protocol message definition confirms the field exists and is optional (`bytes32 | None`): [5](#0-4) 

### Impact Explanation

Every time the farmer wins a block, the 1.75 XCH farmer reward is sent to whatever puzzle hash is in `DeclareProofOfSpace.farmer_reward_puzzle_hash`. A malicious harvester can set this to any address — including one it controls — on every winning proof. The farmer receives no reward and has no on-chain recourse. This constitutes **unauthorized reward diversion affecting XCH**, matching the Critical impact tier.

### Likelihood Explanation

Any operator of a third-party harvester that connects to a victim farmer can exploit this. The CHIP-22 feature is explicitly designed to allow third-party harvesters, so the attack surface is intentional and reachable without any key compromise. The attacker only needs a valid proof of space (which the harvester legitimately produces) and the ability to inject an arbitrary `farmer_reward_address_override` in both `NewProofOfSpace` and `RespondSignatures`.

### Recommendation

1. **Enforce, not just log**: In `new_proof_of_space()`, after calling `notify_farmer_reward_taken_by_harvester_as_fee()`, check the return value or re-evaluate the fee quality inline. If the quality check fails, **drop the proof** (return early) rather than continuing.

2. **Validate in `_process_respond_signatures()`**: Before accepting `response.farmer_reward_address_override`, re-derive the fee quality from the stored proof and the signage point challenge, and verify it passes the threshold. If it does not, ignore the override and use `self.farmer.farmer_target`.

3. **Cross-check consistency**: Ensure the `farmer_reward_address_override` in `RespondSignatures` matches the one originally declared in `NewProofOfSpace`, preventing a harvester from switching addresses between the two messages.

### Proof of Concept

1. Attacker operates a harvester that connects to a victim farmer.
2. When the harvester finds a valid proof of space, it sends `NewProofOfSpace` with:
   - `farmer_reward_address_override` = attacker's puzzle hash
   - `fee_info.applied_fee_threshold` = `0xFFFFFFFF` (maximum, so fee quality always "passes" the log check — or omit `fee_info` entirely; the farmer only logs a warning either way)
3. The farmer calls `notify_farmer_reward_taken_by_harvester_as_fee()`, which logs a warning but does not block.
4. The farmer sends `RequestSignatures` to the harvester.
5. The harvester responds with `RespondSignatures` containing `farmer_reward_address_override` = attacker's puzzle hash.
6. `_process_respond_signatures()` sets `farmer_reward_address = response.farmer_reward_address_override` with no validation.
7. The farmer broadcasts `DeclareProofOfSpace` with the attacker's puzzle hash as `farmer_reward_puzzle_hash`.
8. The full node creates the block; the 1.75 XCH farmer reward coin is created at the attacker's address. [1](#0-0) [6](#0-5) [7](#0-6)

    def notify_farmer_reward_taken_by_harvester_as_fee(
        self, sp: farmer_protocol.NewSignagePoint, proof_of_space: harvester_protocol.NewProofOfSpace
    ) -> None:
        """
        Apply a fee quality convention (see CHIP-22: https://github.com/Chia-Network/chips/pull/88)
        given the proof and signage point. This will be tested against the fee threshold reported
        by the harvester (if any), and logged.
        """
        assert proof_of_space.farmer_reward_address_override is not None

        challenge_str = str(sp.challenge_hash)

        ph_prefix = self.config["network_overrides"]["config"][self.config["selected_network"]]["address_prefix"]
        farmer_reward_puzzle_hash = encode_puzzle_hash(proof_of_space.farmer_reward_address_override, ph_prefix)

        self.log.info(
            f"Farmer reward for challenge '{challenge_str}' "
            + f"taken by harvester for reward address '{farmer_reward_puzzle_hash}'"
        )

        fee_quality = calculate_harvester_fee_quality(proof_of_space.proof.proof, sp.challenge_hash)
        fee_quality_rate = float(fee_quality) / float(0xFFFFFFFF) * 100.0

        if proof_of_space.fee_info is not None:
            fee_threshold = proof_of_space.fee_info.applied_fee_threshold
            fee_threshold_rate = float(fee_threshold) / float(0xFFFFFFFF) * 100.0

            if fee_quality <= fee_threshold:
                self.log.info(
                    f"Fee threshold passed for challenge '{challenge_str}': "
                    + f"{fee_quality_rate:.3f}%/{fee_threshold_rate:.3f}% ({fee_quality}/{fee_threshold})"
                )
            else:
                self.log.warning(
                    f"Invalid fee threshold for challenge '{challenge_str}': "
                    + f"{fee_quality_rate:.3f}%/{fee_threshold_rate:.3f}% ({fee_quality}/{fee_threshold})"
                )
                self.log.warning(
                    "Harvester illegitimately took a fee reward that "
                    + "did not belong to it or it incorrectly applied the fee convention."
                )
        else:
            self.log.warning(
                "Harvester illegitimately took reward by failing to provide its fee rate "
                + f"for challenge '{challenge_str}'. "
                + f"Fee quality was {fee_quality_rate:.3f}% ({fee_quality} or 0x{fee_quality:08x})"
            )
