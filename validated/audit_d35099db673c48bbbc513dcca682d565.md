Audit Report

## Title
Beacon-Chain Slash on Unverified Validator Permanently Overcounts TVL, Inflating rsETH Price and Diluting New Depositors — (`contracts/NodeDelegator.sol`)

## Summary
`stakedButUnverifiedNativeETH` is incremented by exactly 32 ETH on every `stake32Eth()` call and decremented by exactly 32 ETH in `verifyWithdrawalCredentials()`, with no adjustment for beacon-chain slashing. Because `getEffectivePodShares()` adds this counter directly to the EigenLayer `withdrawableShare`, a slashed-but-unverified validator causes the protocol to overcount TVL by the slash amount for the entire window between staking and credential verification. Any `updateRSETHPrice()` call during this window stores an inflated `rsETHPrice`, causing new depositors to receive fewer rsETH tokens than their ETH warrants; when the price corrects after verification, those depositors bear the loss.

## Finding Description

**Staking path** — `stake32Eth()` unconditionally increments `stakedButUnverifiedNativeETH` by 32 ETH and sends 32 ETH to the beacon deposit contract: [1](#0-0) 

**Credential verification path** — `verifyWithdrawalCredentials()` decrements `stakedButUnverifiedNativeETH` by exactly `validatorFields.length * 32 ether`, regardless of the validator's actual beacon-chain balance at proof time: [2](#0-1) 

**TVL accounting** — `getEffectivePodShares()` returns `stakedButUnverifiedNativeETH + withdrawableShare`. For an unverified validator, `withdrawableShare = 0` (EigenLayer has not yet awarded any deposit shares), so the entire 32 ETH contribution comes from the unverified counter: [3](#0-2) 

**Price propagation** — `getETHDistributionData()` sums `getEffectivePodShares()` across all NDCs into `ethStakedInEigenLayer`: [4](#0-3) 

`_getTotalEthInProtocol()` → `getTotalAssetDeposits()` → `getAssetDistributionData()` → `getETHDistributionData()` feeds into `_updateRsETHPrice()`, which divides total ETH by `rsethSupply`: [5](#0-4) 

**Minting** — every deposit divides by the stored `rsETHPrice`: [6](#0-5) 

**The gap** — if a beacon-chain slash reduces a validator's balance from 32 ETH to 30 ETH before `verifyWithdrawalCredentials()` is called:

| Quantity | Expected | Actual (post-slash, pre-verify) |
|---|---|---|
| `stakedButUnverifiedNativeETH` | 30 ETH | 32 ETH |
| `withdrawableShare` | 0 | 0 |
| `getEffectivePodShares()` | 30 ETH | **32 ETH** |

`updateRSETHPrice()` is public and permissionless: [7](#0-6) 

Any caller can invoke it while the discrepancy exists, locking in an inflated `rsETHPrice`. The existing `pricePercentageLimit` upside guard does not apply here because the slash does not cause a price *increase* relative to `highestRsethPrice` — it merely prevents the price from decreasing as it should. The downside protection only triggers *after* verification when the price corrects, by which point the harm to depositors has already occurred. [8](#0-7) 

## Impact Explanation

New depositors who mint rsETH while `rsETHPrice` is inflated by the unaccounted slash receive fewer rsETH tokens than their ETH contribution warrants. When `verifyWithdrawalCredentials()` is eventually called, `stakedButUnverifiedNativeETH` is decremented by 32 ETH while EigenLayer awards shares only for the validator's proven effective balance (e.g., 30 ETH), causing `rsETHPrice` to drop. Those depositors' rsETH is then redeemable for less ETH than they deposited; the difference is redistributed to pre-existing holders through the price mechanism. This constitutes direct theft of depositor funds — **Critical** impact under the allowed scope.

## Likelihood Explanation

Beacon-chain slashing is a real, recurring event on mainnet. There is always a non-trivial window between `stake32Eth()` and `verifyWithdrawalCredentials()` during which the validator is active on the beacon chain but unverified in EigenLayer. `updateRSETHPrice()` is called regularly by bots and keepers, so the inflated price will be stored during this window without any attacker action. No privileged role or special capability is required to trigger the harm — any depositor who calls `depositETH()` during the window is automatically affected. The slash amount ranges from ~1 ETH (initial slash) to 16 ETH (correlation penalties), scaling the impact accordingly.

## Recommendation

Decrement `stakedButUnverifiedNativeETH` when a checkpoint proof demonstrates that a validator's balance has dropped below 32 ETH, or cap the per-validator contribution to `getEffectivePodShares()` at the checkpoint-proven balance rather than the fixed 32 ETH. Concretely, integrate with EigenLayer's `startCheckpoint` / `verifyCheckpointProofs` flow to update `stakedButUnverifiedNativeETH` to reflect actual beacon-chain balances before including unverified validators in TVL. Alternatively, exclude unverified validators from TVL entirely and only count them after `verifyWithdrawalCredentials()` succeeds.

## Proof of Concept

```solidity
// Foundry fork test (Holesky or mainnet fork)
// 1. Deploy/use existing NDC; call stake32Eth() for validator V
//    → stakedButUnverifiedNativeETH = 32e18
// 2. Simulate beacon-chain slash: validator V's balance drops to 30 ETH
//    (verifyWithdrawalCredentials() has NOT been called)
// 3. Call updateRSETHPrice()
//    → rsETHPrice computed using getEffectivePodShares() = 32e18 (inflated by 2e18)
// 4. New depositor calls depositETH{value: 1 ether}(0, "")
//    → receives rsethMinted = 1e18 * 1e18 / rsETHPrice (fewer tokens than correct price would give)
// 5. Call verifyWithdrawalCredentials() + startCheckpoint() + verifyCheckpointProofs()
//    → stakedButUnverifiedNativeETH -= 32e18; withdrawableShare += 30e18
// 6. Call updateRSETHPrice()
//    → rsETHPrice drops to reflect actual 30e18
// 7. Assert: rsethMinted * newRsETHPrice / 1e18 < 1 ether
//    (depositor's rsETH redeemable value is less than 1 ETH deposited)
```
