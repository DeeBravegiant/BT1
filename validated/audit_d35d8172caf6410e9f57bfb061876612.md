### Title
Missing Caller Deposit Refund in Intermediate Callback Failure Paths — (`File: runtime/near-wallet-contract/implementation/wallet-contract/src/lib.rs`)

### Summary

The `WalletContract`'s `rlp_execute` entry point is `#[payable]` and correctly tracks an external caller's attached NEAR deposit via `CallerDeposit`. The terminal callback `rlp_execute_callback` properly refunds this deposit on failure. However, two intermediate callbacks — `nep_141_storage_balance_callback` and `address_check_callback` — contain multiple early-return failure paths that return `PromiseOrValue::Value(...)` without issuing any refund promise for the caller's deposit. Any NEAR attached by an external caller in these flows is permanently absorbed into the wallet contract's balance.

### Finding Description

`rlp_execute` is marked `#[payable]` and captures the caller's attached deposit into a `CallerDeposit` struct at the start of `inner_rlp_execute`:

```rust
let context = ExecutionContext::new(..., env::attached_deposit())?;
let caller_deposit = CallerDeposit::new(&context);
``` [1](#0-0) 

`CallerDeposit::new` records the deposit only for external (non-self) callers with a non-zero deposit: [2](#0-1) 

The terminal callback `rlp_execute_callback` correctly refunds this deposit on a failed promise: [3](#0-2) 

However, `nep_141_storage_balance_callback` has three early-return paths that drop `caller_deposit` without refunding:

1. **`PromiseResult::Failed`** (line 204–210): returns immediately with an error response, no refund.
2. **JSON parse failure** (line 213–219): returns immediately, no refund.
3. **Action is not `FunctionCall`** (line 245–253): returns immediately, no refund. [4](#0-3) 

Similarly, `address_check_callback` has early-return paths that drop `caller_deposit`:

1. **`PromiseResult::Failed`** (line 142–148): returns immediately, no refund.
2. **JSON parse failure** (line 151–157): returns immediately, no refund.
3. **`maybe_account_id.is_some()` and signer ≠ current account** (line 167–173): returns immediately, no refund. [5](#0-4) 

In all these paths, the NEAR deposit that was attached to `rlp_execute` has already been credited to the wallet contract's balance. Because no refund promise is created, the tokens remain permanently in the wallet contract.

### Impact Explanation

An external caller (e.g., a relayer) who attaches NEAR to `rlp_execute` for an ERC-20 emulation transfer or an EOA base-token transfer to another ETH-implicit account permanently loses their attached deposit if the intermediate callback encounters any of the above failure conditions. The corrupted value is the caller's NEAR account balance — the attached deposit is irreversibly transferred to the wallet contract with no mechanism to recover it.

### Likelihood Explanation

The `nep_141_storage_balance_callback` failure path is reachable whenever a NEP-141 token contract's `storage_balance_of` call panics or runs out of gas. The `address_check_callback` failure path is reachable whenever the address registrar contract call fails. Both are realistic in production (e.g., a buggy or malicious token contract, gas exhaustion, or a temporarily unavailable registrar). Any relayer who attaches NEAR to fund a function call deposit in these flows is exposed.

### Recommendation

Each early-return failure branch in `nep_141_storage_balance_callback` and `address_check_callback` should issue a refund promise before returning, mirroring the pattern already used in `rlp_execute_callback`:

```rust
if let Some(CallerDeposit { account_id, yocto_near }) = caller_deposit {
    let refund_promise = env::promise_batch_create(&account_id);
    env::promise_batch_action_transfer(
        refund_promise,
        NearToken::from_yoctonear(yocto_near.into()),
    );
}
```

This should be applied to every `return PromiseOrValue::Value(...)` path in both callbacks where `caller_deposit` is `Some`.

### Proof of Concept

1. Deploy a NEP-141 token contract whose `storage_balance_of` method panics unconditionally.
2. An ETH-implicit account owner signs an ERC-20 transfer Ethereum transaction targeting this token contract.
3. An external relayer calls `rlp_execute` on the wallet contract with an attached deposit of, say, 3 NEAR (to fund the function call).
4. `inner_rlp_execute` creates `CallerDeposit { account_id: relayer, yocto_near: 3e24 }` and dispatches `storage_balance_of` → `nep_141_storage_balance_callback`.
5. `storage_balance_of` panics → `env::promise_result(0)` is `PromiseResult::Failed`.
6. `nep_141_storage_balance_callback` hits the early-return at line 204–210, returning an error `ExecuteResponse` with no refund promise.
7. The relayer's 3 NEAR is permanently absorbed into the wallet contract's balance. [6](#0-5) [7](#0-6)

                ),
            });
        }
        let current_account_id = env::current_account_id();
        let predecessor_account_id = env::predecessor_account_id();

            predecessor_account_id,
            target,
            tx_bytes_b64,
            &mut self.nonce,
        );

            Ok(promise) => {
                self.has_in_flight_tx = true;
                PromiseOrValue::Promise(promise)

                PromiseOrValue::Promise(promise)

impl CallerDeposit {
    pub fn new(context: &ExecutionContext) -> Option<Self> {
        // Only track for external (non-self) callers
