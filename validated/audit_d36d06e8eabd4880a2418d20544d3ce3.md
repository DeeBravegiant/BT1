Audit Report

## Title
Missing Slippage Protection in L2 Pool Deposit Functions - (`contracts/pools/RSETHPoolV3.sol`, `contracts/pools/RSETHPool.sol`, `contracts/pools/RSETHPoolNoWrapper.sol`, `contracts/pools/RSETHPoolV3ExternalBridge.sol`)

## Summary
All four L2 pool contracts expose `deposit()` functions that compute output amounts from a live oracle rate but accept no caller-supplied minimum output parameter. If the oracle rate increases between transaction submission and execution, users silently receive fewer rsETH/wrsETH than they previewed off-chain, with no on-chain revert and no recourse. The L1 counterpart (`LRTDepositPool._beforeDeposit`) already enforces this check, confirming the protection was intentionally designed but omitted from all L2 paths.

## Finding Description
In `RSETHPoolV3.deposit(string)` and `RSETHPoolV3.deposit(address,uint256,string)`, the output amount is computed exclusively from `viewSwapRsETHAmountAndFee`, which calls `getRate()` — a live oracle read — at execution time:

```solidity
// RSETHPoolV3.sol L299-308
function viewSwapRsETHAmountAndFee(uint256 amount) public view returns (uint256 rsETHAmount, uint256 fee) {
    fee = amount * feeBps / 10_000;
    uint256 amountAfterFee = amount - fee;
    uint256 rsETHToETHrate = getRate();          // live oracle read
    rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
}
```

Neither `deposit` overload accepts a `minRsETHAmountExpected` argument, and no minimum-output check exists before `wrsETH.mint(msg.sender, rsETHAmount)` is called. The identical pattern is present in `RSETHPool.sol` (L265-305), `RSETHPoolNoWrapper.sol` (L231-271), and `RSETHPoolV3ExternalBridge.sol` (L366-412).

By contrast, the L1 path in `LRTDepositPool._beforeDeposit` (L648-669) explicitly enforces:
```solidity
if (rsethAmountToMint < minRSETHAmountExpected) {
    revert MinimumAmountToReceiveNotMet();
}
```

Exploit flow:
1. User calls `viewSwapRsETHAmountAndFee(1 ether)` off-chain; oracle rate is `1.05e18`, expected output ≈ `0.952 wrsETH`.
2. User submits `deposit{value: 1 ether}("ref")`.
3. Protocol updates oracle to `1.10e18` before the tx is mined (normal protocol operation, no adversary needed).
4. `deposit()` executes: `rsETHAmount = 1e18 * 1e18 / 1.10e18 ≈ 0.909 wrsETH`.
5. User receives ≈ `0.909 wrsETH` — a ~4.5% shortfall — with no revert and no recourse. Input ETH is already consumed.

No existing guard prevents this: `nonReentrant` prevents reentrancy, `whenNotPaused` prevents execution when paused, and `limitDailyMint` caps total daily volume — none of these protect the individual user's expected output.

## Impact Explanation
The contract fails to deliver the output amount a user reasonably expects based on the protocol-provided preview function. The user's input is consumed and cannot be recovered, but the user does receive some rsETH — value is not destroyed, only reduced relative to expectation. This maps precisely to **Low — Contract fails to deliver promised returns, but doesn't lose value**.

## Likelihood Explanation
No adversarial actor is required. The rsETH oracle rate is updated periodically by the protocol as part of normal operation. On L2 networks with short block times (e.g., Arbitrum, Optimism, Base), a rate update can land in the same block or the block immediately preceding a user's deposit. Any user depositing on any of the four affected contracts is exposed on every deposit. The condition is triggered by ordinary protocol maintenance, not by an attacker.

## Recommendation
Add a `minRsETHAmountExpected` parameter to every `deposit()` overload in all four L2 pool contracts, mirroring `LRTDepositPool._beforeDeposit`. After computing `rsETHAmount`, revert if it falls below the caller-supplied minimum:

```solidity
function deposit(string memory referralId, uint256 minRsETHAmountExpected) external payable ... {
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(msg.value);
    if (rsETHAmount < minRsETHAmountExpected) revert MinimumAmountToReceiveNotMet();
    feeEarnedInETH += fee;
    wrsETH.mint(msg.sender, rsETHAmount);
    emit SwapOccurred(msg.sender, rsETHAmount, fee, referralId);
}
```

Apply the same pattern to the token overload and to all four contracts.

## Proof of Concept
Foundry fork test outline:

```solidity
function test_slippage_no_protection() public {
    // 1. Fork L2 with RSETHPoolV3 deployed
    // 2. Record oracle rate R1 = getRate()
    // 3. Compute expected = viewSwapRsETHAmountAndFee(1 ether).rsETHAmount
    // 4. Simulate oracle update: rate R2 = R1 * 110 / 100 (10% increase)
    //    (call setRate or warp + poke depending on oracle implementation)
    // 5. Call pool.deposit{value: 1 ether}("ref") as user
    // 6. Assert wrsETH.balanceOf(user) < expected
    // 7. Assert no revert occurred
    // Demonstrates: user receives fewer tokens than previewed, silently
}
```

The test requires no privileged role — only a standard EOA depositing ETH. The same test applies to the token overload and to `RSETHPool`, `RSETHPoolNoWrapper`, and `RSETHPoolV3ExternalBridge` by substituting the contract address.
