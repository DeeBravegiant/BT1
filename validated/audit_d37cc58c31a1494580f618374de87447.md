### Title
Unbounded Growth of `interested_coins` Store Enables External-Party Wallet Sync DoS - (`chia/wallet/wallet_interested_store.py`, `chia/wallet/wallet_state_manager.py`)

---

### Summary

The wallet's `WalletInterestedStore.interested_coins` table grows without bound because coin IDs are added whenever a coin is received but are only removed for `CoinType.CLAWBACK` coins when spent. For all other coin types (standard, CAT, pool wallet singleton, DL wallet singleton), spent coin IDs accumulate permanently. An unprivileged attacker can deliberately inflate this list by sending many CAT transactions to a victim's wallet. During every subsequent wallet sync, `get_coin_ids_to_subscribe()` fetches the entire bloated list and attempts to subscribe to all entries at the full node, exhausting the per-peer subscription quota and preventing the wallet from registering subscriptions for its legitimate active coins (offers, singletons, clawback coins).

---

### Finding Description

**Problem 1 — Coin IDs are added but almost never removed.**

`add_interested_coin_ids()` is called in multiple places:

- Line 1906 of `wallet_state_manager.py`: every newly received unspent coin
- Line 2006: every CAT change coin
- Line 2003: every clawback child coin
- Line 2118: every new pool wallet singleton
- Line 862 of `data_layer_wallet.py`: every new DL singleton [1](#0-0) 

`remove_interested_coin_id()` is called in exactly **one** place — only for `CoinType.CLAWBACK` coins when spent: [2](#0-1) 

For every other coin type, the coin ID remains in `interested_coins` forever, even after the coin is spent and is no longer relevant. [3](#0-2) 

**Problem 2 — The entire list is fetched and subscribed on every sync.**

`get_coin_ids_to_subscribe()` returns every entry in `interested_coins` unconditionally: [4](#0-3) 

During `long_sync()`, this full list is iterated in batches of 1000 and sent to the full node via `subscribe_to_coin_updates`: [5](#0-4) 

**Problem 3 — External parties can inflate the list via CAT transfers.**

The spam filter in `filter_spam()` only applies to standard wallet transactions: [6](#0-5) 

CAT transactions are not filtered. An attacker who knows the victim has a CAT wallet for a given asset ID (e.g., a popular CAT like USDS) can send many tiny CAT transactions to the victim. Each transaction creates a new CAT coin; the CAT change coin is added to `interested_coins` at line 2006 and never removed after it is spent.

**Problem 4 — The full node enforces a per-peer subscription limit shared across puzzle hashes and coin IDs.**

`add_coin_subscriptions()` enforces a cap on the total subscription count (puzzle hashes + coin IDs) per peer: [7](#0-6) 

`register_for_coin_updates` truncates the coin ID list to `max_subscriptions` before passing it to the subscription set: [8](#0-7) 

If the bloated `interested_coins` list exhausts the subscription quota, the full node silently drops the excess entries. Legitimate active coin subscriptions (active offers, current pool wallet singletons, current DL singletons) that appear later in the list may never be registered.

---

### Impact Explanation

If the subscription quota is exhausted by stale spent coin IDs:

- **Offer settlement**: The wallet will not receive `CoinState` updates for offer settlement coins, causing active offers to appear perpetually pending and preventing the wallet from detecting when an offer is taken or cancelled.
- **Pool wallet**: The wallet will not receive updates for the current pool singleton coin, breaking pool reward tracking and pool state transitions.
- **Data Layer**: The wallet will not receive updates for the current DL singleton, corrupting Data Layer root/store sync state.

These match the allowed High impact: *"Permanent or long-lived inability for honest wallets to process valid spend bundles, sync updates, pool actions, or Data Layer updates"* and *"Corruption of offer/trade settlement state, wallet sync state, pool membership state, or Data Layer root/store state with direct security impact."*

---

### Likelihood Explanation

- The attacker only needs to know one CAT asset ID the victim holds (e.g., a publicly known popular CAT).
- No privileged access is required; sending CAT transactions is a normal on-chain operation.
- Each transaction costs a small fee, making a large-scale attack expensive but not infeasible for a motivated attacker targeting a high-value wallet.
- The effect is persistent across wallet restarts because `interested_coins` is a persisted SQLite table.

---

### Recommendation

1. **Remove coin IDs from `interested_coins` when the coin is confirmed spent**, not only for CLAWBACK coins. In `_add_coin_states`, after `coin_store.set_spent()` is called for any coin type, call `interested_store.remove_interested_coin_id(coin_name)`.

2. **Periodically prune `interested_coins`** of entries whose coin IDs are confirmed spent in the local `coin_store`.

3. **Apply a spam filter to CAT coin states** analogous to the existing XCH spam filter, to limit the rate at which external parties can force new entries into `interested_coins`.

---

### Proof of Concept

1. Victim has a CAT wallet for asset ID `A` (e.g., USDS).
2. Attacker sends 10,000 transactions of 1 mojo of CAT `A` to the victim's wallet address.
3. Each transaction is processed by `_add_coin_states`; the CAT change coin is added to `interested_coins` via `add_interested_coin_ids` (line 2006).
4. Victim spends those CAT coins normally. The spent coin IDs remain in `interested_coins` (only CLAWBACK coins are removed at line 2059).
5. On the next wallet sync, `get_coin_ids_to_subscribe()` returns all 10,000+ stale coin IDs plus the victim's legitimate active coin IDs.
6. `long_sync()` sends all of them to the full node in batches of 1000. The full node's per-peer subscription quota is consumed by stale entries.
7. The victim's active offer coin IDs and pool wallet singleton coin IDs, appearing later in the list, are silently dropped by `add_coin_subscriptions` once the quota is reached.
8. The victim's wallet no longer receives `CoinState` updates for active offers or pool wallet singletons, causing offer settlement failures and pool sync corruption.

```python
    async def get_interested_coin_ids(self) -> list[bytes32]:
        async with self.db_wrapper.writer_maybe_transaction() as conn:
            cursor = await conn.execute("SELECT coin_name FROM interested_coins")
            rows_hex = await cursor.fetchall()
        return [bytes32(bytes.fromhex(row[0])) for row in rows_hex]

    async def add_interested_coin_id(self, coin_id: bytes32) -> None:
        async with self.db_wrapper.writer_maybe_transaction() as conn:
            cursor = await conn.execute("INSERT OR REPLACE INTO interested_coins VALUES (?)", (coin_id.hex(),))
            await cursor.close()

    async def remove_interested_coin_id(self, coin_id: bytes32) -> None:
        async with self.db_wrapper.writer_maybe_transaction() as conn:
            cursor = await conn.execute("DELETE FROM interested_coins WHERE coin_name=?", (coin_id.hex(),))
            await cursor.close()
```

```python
    def add_coin_subscriptions(self, peer_id: bytes32, coin_ids: list[bytes32], max_items: int) -> set[bytes32]:
        """
        Adds subscriptions until max_items is reached. Filters out duplicates and returns all additions.
        """

        subscription_count = self.peer_subscription_count(peer_id)
        added: set[bytes32] = set()

        def limit_reached() -> set[bytes32]:
            log.info(
                "Peer %s attempted to exceed the subscription limit while adding coin subscriptions.",
                peer_id,
            )
            return added

        # If the subscription limit is reached, bail.
        if subscription_count >= max_items:
            return limit_reached()

        # Decrement this counter to know if we've hit the subscription limit.
        subscriptions_left = max_items - subscription_count

        for coin_id in coin_ids:
            if not self._coin_subscriptions.add_subscription(peer_id, coin_id):
                continue

            subscriptions_left -= 1
            added.add(coin_id)

            if subscriptions_left == 0:
                return limit_reached()

        return added
```
