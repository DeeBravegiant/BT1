Audit Report

## Title
rsETH Inflation Attack via ETH Donation to `LRTDepositPool` Enables Zero-Share Minting - (File: `contracts/LRTDepositPool.sol`)

## Summary
An attacker who is the first depositor can donate ETH directly to `LRTDepositPool` and call the permissionless `updateRSETHPrice()` to inflate `rsETHPrice` by an unbounded factor, since `pricePercentageLimit` defaults to `0`. Any subsequent depositor who passes `minRSETHAmountExpected = 0` will have their ETH accepted by the protocol while receiving 0 rsETH in return, permanently freezing their funds with no withdrawal path.

## Finding Description

**Weakness 1 — Direct ETH donation inflates TVL.**

`LRTDepositPool` exposes a bare `receive()`: [1](#0-0) 

`getETHDistributionData()` reads the raw contract balance: [2](#0-1) 

Any ETH sent directly to the contract is immediately counted in TVL and therefore in `totalETHInProtocol` used by `_updateRsETHPrice()`.

**Weakness 2 — `updateRSETHPrice()` is public and uncapped when `pricePercentageLimit == 0`.**

`updateRSETHPrice()` is callable by anyone: [3](#0-2) 

`pricePercentageLimit` is a storage variable that initializes to `0`: [4](#0-3) 

The only guard against a large price jump short-circuits to `false` when `pricePercentageLimit == 0`: [5](#0-4) 

With `pricePercentageLimit == 0`, `isPriceIncreaseOffLimit` is always `false`, so the price can be inflated by any arbitrary multiple in a single call.

**Weakness 3 — `_beforeDeposit` does not reject a zero-rsETH mint.**

The slippage check only compares against `minRSETHAmountExpected`: [6](#0-5) 

There is no `require(rsethAmountToMint > 0)`. When `minRSETHAmountExpected = 0`, the check `0 < 0` is `false` and the deposit proceeds. `RSETH.mint(victim, 0)` does not revert: the `checkDailyMintLimit(0)` modifier evaluates `0 > maxMintAmountPerDay` as `false` regardless of the limit value, and OpenZeppelin `_mint` with amount `0` is a no-op: [7](#0-6) [8](#0-7) 

**Mint formula:**

After inflation, `rsETHPrice = (1 + X) * 1e18`. For a victim deposit `Y` where `Y < 1 + X`: [9](#0-8) 

`rsethAmountToMint = (Y * 1e18) / ((1 + X) * 1e18) = Y / (1 + X) = 0` by integer truncation.

**Note on `rsethSupply == 0` guard:** `_updateRsETHPrice()` returns early with `rsETHPrice = 1 ether` when supply is zero: [10](#0-9) 

The PoC correctly accounts for this — the attacker first deposits 1 wei to bring supply to 1 before donating and inflating.

## Impact Explanation

**Critical — Permanent freezing of user funds.** The victim's ETH is transferred into `LRTDepositPool` as `msg.value` before any check can revert it, but 0 rsETH is minted. With no rsETH balance, the victim has no mechanism to initiate a withdrawal. Their ETH is permanently locked in the protocol. The attacker, holding 100% of rsETH supply (1 wei), can eventually redeem it for the entire pool balance (their donation + victim's deposit), constituting direct theft of user funds.

## Likelihood Explanation

**Medium.** The attacker must act when rsETH supply is negligibly small (e.g., at protocol launch or after a reset). On Ethereum mainnet, front-running the first legitimate deposit is straightforward. The victim must pass `minRSETHAmountExpected = 0`, which is the default for many integrations and scripts that omit slippage protection. The critical enabler — `pricePercentageLimit == 0` — is the contract's default state at deployment and requires no special conditions.

## Recommendation

1. **Enforce a non-zero mint amount.** Add `require(rsethAmountToMint > 0, "ZeroMint")` inside `_beforeDeposit` before the slippage check. [6](#0-5) 

2. **Initialize `pricePercentageLimit` to a safe non-zero value** (e.g., `1e16` for 1%) during deployment so that a single `updateRSETHPrice()` call cannot inflate the price by an unbounded factor. [4](#0-3) 

3. **Seed the pool at deployment.** Mint a small amount of rsETH to a dead address (e.g., `address(0xdead)`) during initialization so that `rsethSupply` is never 0 for an attacker-controlled first deposit.

## Proof of Concept

```
State: rsETHPrice = 1e18, pricePercentageLimit = 0, rsETH totalSupply = 0

Step 1 — Attacker deposits 1 wei ETH:
  depositETH{value: 1}(minRSETHAmountExpected=0, "")
  rsethAmountToMint = (1 * 1e18) / 1e18 = 1
  → Attacker receives 1 wei rsETH. rsETH totalSupply = 1.

Step 2 — Attacker donates X ETH directly to LRTDepositPool:
  (bool ok,) = address(lrtDepositPool).call{value: X}("");
  → LRTDepositPool.balance = 1 + X

Step 3 — Attacker calls LRTOracle.updateRSETHPrice():
  totalETHInProtocol = (1 + X).mulWad(1e18) = 1 + X
  newRsETHPrice = (1 + X).divWad(1) = (1 + X) * 1e18
  pricePercentageLimit == 0 → isPriceIncreaseOffLimit = false → no revert
  → rsETHPrice = (1 + X) * 1e18

Step 4 — Victim deposits Y ETH (Y ≤ X) with minRSETHAmountExpected = 0:
  rsethAmountToMint = (Y * 1e18) / ((1 + X) * 1e18) = Y / (1 + X) = 0
  0 < 0 → no revert
  RSETH.mint(victim, 0) → no revert (checkDailyMintLimit(0) passes, _mint(to,0) is no-op)
  → Victim's Y ETH is in the pool; victim has 0 rsETH and no withdrawal path.

Step 5 — Attacker redeems 1 wei rsETH:
  Attacker's 1 wei rsETH = 100% of supply.
  Attacker recovers 1 + X + Y ETH.
  Net attacker profit: Y ETH (victim's entire deposit).
```
