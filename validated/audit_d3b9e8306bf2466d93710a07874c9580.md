### Title
Unvalidated `farmer_reward_address_override` in `RespondSignatures` Enables Malicious Harvester to Divert Farmer Block Rewards - (File: chia/farmer/farmer_api.py)

### Summary

A malicious harvester can permanently redirect a farmer's XCH block reward to an arbitrary puzzle hash by setting `farmer_reward_address_override` in a `RespondSignatures` message. The farmer never validates this field against the override (if any) declared in the earlier `NewProofOfSpace` message, and the fee-quality logging guard is bypassed entirely when the override is absent from `NewProofOfSpace` but present in `RespondSignatures`.

### Finding Description

The CHIP-22 harvester-fee protocol introduces `farmer_reward_address_override` as an optional field in two harvester protocol messages:

1. `NewProofOfSpace` — sent by the harvester to the farmer when a valid proof is found.
2. `RespondSignatures` — sent by the harvester in reply to the farmer's `RequestSignatures`.

Both fields are `bytes32 | None` and carry an arbitrary puzzle hash. [1](#0-0) 

When the farmer receives `NewProofOfSpace`, it calls `notify_farmer_reward_taken_by_harvester_as_fee` only if `farmer_reward_address_override is not None`. That function logs the override and checks a fee-quality threshold, but **does not reject the proof or the override** — it is advisory only. [2](#0-1) [3](#0-2) 

Later, in `_process_respond_signatures`, the farmer unconditionally replaces its configured `farmer_target` with whatever `response.farmer_reward_address_override` contains, **with no check that this value matches the override (or absence of override) declared in the original `NewProofOfSpace`**: [4](#0-3) 

This `farmer_reward_address` is then placed into `DeclareProofOfSpace.farmer_puzzle_hash`: [5](#0-4) 

The full node accepts `request.farmer_puzzle_hash` verbatim and embeds it into the unfinished block's `FoliageBlockData`: [6](#0-5) 

The farmer then signs the foliage block data hash (which commits to this puzzle hash), and the block is accepted by consensus. The farmer's 1/8 block reward is permanently paid to the attacker-controlled address.

### Impact Explanation

A connected malicious harvester can divert the farmer's XCH block reward (currently ~0.25 XCH per block) to any arbitrary address on every block the harvester helps win. The reward coin is created on-chain at the attacker's puzzle hash and is immediately spendable by the attacker. The farmer receives nothing. This is an irreversible, on-chain accounting change affecting XCH.

### Likelihood Explanation

Any third-party harvester service (the explicit use-case of CHIP-22) can execute this attack. The harvester is already a connected, protocol-speaking peer. No leaked keys, admin access, or cryptographic break is required. The attack is silent: by sending `farmer_reward_address_override = None` in `NewProofOfSpace`, the harvester bypasses even the advisory fee-quality log, so the farmer sees no warning.

### Recommendation

In `_process_respond_signatures`, validate that `response.farmer_reward_address_override` matches the override stored when the corresponding `NewProofOfSpace` was processed. Concretely:

- When storing a proof in `self.farmer.proofs_of_space`, also record the `farmer_reward_address_override` from `NewProofOfSpace`.
- In `_process_respond_signatures`, reject (return `None`) if `response.farmer_reward_address_override` differs from the stored value for that proof.
- Additionally, enforce the fee-quality threshold as a hard rejection rather than a log-only warning.

### Proof of Concept

1. Malicious harvester connects to farmer as a normal harvester peer.
2. Harvester finds a valid proof and sends `NewProofOfSpace` with `farmer_reward_address_override = None` — no fee-quality log is triggered.
3. Farmer sends `RequestSignatures` back to the harvester.
4. Harvester replies with `RespondSignatures` where `farmer_reward_address_override = <attacker_puzzle_hash>`.
5. `_process_respond_signatures` (farmer_api.py:917–918) replaces `farmer_reward_address` with the attacker's puzzle hash without any cross-check.
6. Farmer broadcasts `DeclareProofOfSpace` with `farmer_puzzle_hash = <attacker_puzzle_hash>` to the full node.
7. Full node builds the unfinished block with `farmer_reward_puzzle_hash = <attacker_puzzle_hash>` (full_node_api.py:1069).
8. Farmer signs the foliage block data committing to the attacker's address; block passes consensus validation.
9. The farmer's reward coin is created at the attacker's address and is permanently lost to the farmer.

    def notify_farmer_reward_taken_by_harvester_as_fee(
        self, sp: farmer_protocol.NewSignagePoint, proof_of_space: harvester_protocol.NewProofOfSpace
    ) -> None:
        """
        Apply a fee quality convention (see CHIP-22: https://github.com/Chia-Network/chips/pull/88)
        given the proof and signage point. This will be tested against the fee threshold reported
        by the harvester (if any), and logged.
        """
        assert proof_of_space.farmer_reward_address_override is not None

        challenge_str = str(sp.challenge_hash)

        ph_prefix = self.config["network_overrides"]["config"][self.config["selected_network"]]["address_prefix"]
        farmer_reward_puzzle_hash = encode_puzzle_hash(proof_of_space.farmer_reward_address_override, ph_prefix)

        self.log.info(
            f"Farmer reward for challenge '{challenge_str}' "
            + f"taken by harvester for reward address '{farmer_reward_puzzle_hash}'"
        )

        fee_quality = calculate_harvester_fee_quality(proof_of_space.proof.proof, sp.challenge_hash)
        fee_quality_rate = float(fee_quality) / float(0xFFFFFFFF) * 100.0

        if proof_of_space.fee_info is not None:
            fee_threshold = proof_of_space.fee_info.applied_fee_threshold
            fee_threshold_rate = float(fee_threshold) / float(0xFFFFFFFF) * 100.0

            if fee_quality <= fee_threshold:
                self.log.info(
                    f"Fee threshold passed for challenge '{challenge_str}': "
                    + f"{fee_quality_rate:.3f}%/{fee_threshold_rate:.3f}% ({fee_quality}/{fee_threshold})"
                )
            else:
                self.log.warning(
                    f"Invalid fee threshold for challenge '{challenge_str}': "
                    + f"{fee_quality_rate:.3f}%/{fee_threshold_rate:.3f}% ({fee_quality}/{fee_threshold})"
                )
                self.log.warning(
                    "Harvester illegitimately took a fee reward that "
                    + "did not belong to it or it incorrectly applied the fee convention."
                )
        else:
            self.log.warning(
                "Harvester illegitimately took reward by failing to provide its fee rate "
                + f"for challenge '{challenge_str}'. "
                + f"Fee quality was {fee_quality_rate:.3f}% ({fee_quality} or 0x{fee_quality:08x})"
            )
