Audit Report

## Title
Missing Chainlink Oracle Staleness and Validity Checks Allow Stale Prices to Corrupt rsETH Exchange Rate - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary
`ChainlinkPriceOracle.getAssetPrice` calls `latestRoundData()` but discards all return values except `price`, performing no staleness check on `updatedAt`, no positive-price validation, and no round-completeness check. A stale or manipulated Chainlink price propagates through `LRTOracle._updateRsETHPrice()` and corrupts `rsETHPrice`, enabling depositors to mint excess rsETH at the expense of existing holders. The protocol's circuit breaker provides only partial mitigation and is entirely absent when `pricePercentageLimit == 0` (its default uninitialized value).

## Finding Description

`ChainlinkPriceOracle.getAssetPrice` at lines 49–55 silently discards `roundId`, `startedAt`, `updatedAt`, and `answeredInRound`:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
``` [1](#0-0) 

Three validations are absent:
1. **No staleness check** — `updatedAt` is never compared to `block.timestamp`.
2. **No positive-price check** — `int256 price` is cast directly to `uint256`; a zero price returns `0`, a negative price wraps to near-`type(uint256).max`.
3. **No `answeredInRound >= roundId` check** — round completeness is never verified.

This price flows through `LRTOracle.getAssetPrice` → `_getTotalEthInProtocol` → `_updateRsETHPrice`, which sets the global `rsETHPrice` storage variable: [2](#0-1) [3](#0-2) 

`updateRSETHPrice()` is a public, permissionless function callable by any external account: [4](#0-3) 

**Why the circuit breaker is insufficient:**

`_updateRsETHPrice` contains a downside circuit breaker that pauses the protocol if `newRsETHPrice` drops more than `pricePercentageLimit` below `highestRsethPrice`: [5](#0-4) 

This fails to protect in two concrete cases:
- **`pricePercentageLimit == 0` (default)**: The guard condition `pricePercentageLimit > 0 && ...` is always `false`, so the circuit breaker never fires. The stale price is committed unconditionally.
- **Small staleness within the threshold**: A stale price that is, e.g., 1–2% below the true price passes the circuit breaker if `pricePercentageLimit` is set to 5%. The stale price is committed and `rsETHPrice` is corrupted.

In both cases, `rsETHPrice = newRsETHPrice` is executed with the stale value, and the corrupted rate is then consumed by `RSETHPoolV3.viewSwapRsETHAmountAndFee` via `getRate()`: [6](#0-5) 

## Impact Explanation

**Impact: High — Theft of unclaimed yield.**

When `rsETHPrice` is set below its true value, the `deposit()` function in `RSETHPoolV3` computes `rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate` using the depressed rate, minting excess wrsETH to the depositor. This excess comes at the direct expense of existing rsETH/wrsETH holders whose backing is diluted — constituting theft of unclaimed yield. The attack is repeatable every time a Chainlink feed is stale and the deviation is within the circuit breaker threshold (or the threshold is unset). [7](#0-6) 

## Likelihood Explanation

- Chainlink LST/ETH feeds (stETH/ETH, rETH/ETH, etc.) have heartbeat intervals of up to 24 hours; staleness within that window is a normal operating condition, not an edge case.
- `updateRSETHPrice()` is public with no role restriction, so an attacker can deliberately time the call to coincide with a known stale window.
- `pricePercentageLimit` is a `uint256` storage variable initialized to `0` by default; unless the admin has explicitly called `setPricePercentageLimit`, the circuit breaker is entirely disabled.
- The attack requires no special privileges, no victim mistakes, and no external protocol compromise — only a public function call timed to a stale feed. [8](#0-7) 

## Recommendation

Add staleness, positive-price, and round-completeness guards in `getAssetPrice`. Use per-feed staleness thresholds stored in a mapping, since heartbeat intervals differ across feeds:

```solidity
mapping(address asset => uint256 maxStaleness) public assetMaxStaleness;

function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);

    (uint80 roundId, int256 price, , uint256 updatedAt, uint80 answeredInRound)
        = priceFeed.latestRoundData();

    require(price > 0, "ChainlinkPriceOracle: non-positive price");
    uint256 maxStaleness = assetMaxStaleness[asset];
    require(maxStaleness > 0, "ChainlinkPriceOracle: staleness not configured");
    require(updatedAt != 0 && block.timestamp - updatedAt <= maxStaleness,
            "ChainlinkPriceOracle: stale price");
    require(answeredInRound >= roundId, "ChainlinkPriceOracle: incomplete round");

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

Additionally, ensure `pricePercentageLimit` is set to a non-zero value during deployment/initialization so the circuit breaker in `_updateRsETHPrice` is active.

## Proof of Concept

**Preconditions:**
- `pricePercentageLimit == 0` (default, circuit breaker disabled), OR stale deviation is within the configured threshold.
- stETH/ETH Chainlink feed heartbeat is 24 h; last update was 23 h 50 min ago at price `0.97e18` (true market: `1.00e18`).

**Steps:**
1. Attacker calls `LRTOracle.updateRSETHPrice()` (public, no role required).
2. `getAssetPrice(stETH)` returns `0.97e18` — no staleness check fires.
3. `_getTotalEthInProtocol()` underestimates total ETH by ~3% of stETH TVL.
4. `newRsETHPrice` is set ~3% below its true value; with `pricePercentageLimit == 0`, the circuit breaker does not fire and `rsETHPrice = newRsETHPrice` is committed.
5. Attacker calls `RSETHPoolV3.deposit{value: 10 ether}(...)` on the L2 pool.
6. `viewSwapRsETHAmountAndFee` computes `rsETHAmount = 10e18 * 1e18 / depressedRate`, minting ~3% excess wrsETH.
7. Attacker redeems wrsETH for rsETH, extracting value from existing holders.
