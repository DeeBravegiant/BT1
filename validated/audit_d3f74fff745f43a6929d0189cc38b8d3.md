### Title
Broken Pigeonhole Invariant in `set_state_header`: Unbound `from_shard_id` Allows Receipt Substitution, Diverging State Root â€” (`chain/chain/src/state_sync/adapter.rs`)

### Summary

`ChainStateSyncAdapter::set_state_header` accepts a peer-supplied `ShardStateSyncResponseHeader` whose receipt-proof completeness is validated using a Pigeonhole argument. The argument is unsound: `from_shard_id` is never cryptographically bound to the Merkle-proof position in the block's outgoing-receipts tree. A malicious peer can omit shard A's receipts, duplicate shard C's receipts under `from_shard_id=A`, and pass all four checks (4c count, 4d uniqueness, 4e leaf proof, 4f root proof). `set_state_finalize` then applies the wrong receipt set, producing a state root that diverges from every honest node.

### Finding Description

**Validation logic (steps 4câ€“4f):**

```
4c: receipt_proofs.len() == block_header.chunks_included()   // count check
4d: all from_shard_id values are distinct                    // uniqueness check
4e: verify_path(root, proof, hash(ReceiptList(shard_id, receipts)))
4f: verify_path(block_outgoing_root, block_proof, root)
```

`root` and `block_proof` come from the peer-supplied `RootProof` struct. [2](#0-1) 

`verify_path` is position-agnostic â€” it only checks that `compute_root_from_path(path, item_hash) == root`. [1](#0-0) 

The block's outgoing-receipts Merkle tree is built from all chunks' `prev_outgoing_receipts_root` values. A valid `block_proof` for shard C's root at position C passes step 4f regardless of what `from_shard_id` claims. [7](#0-6) 

**Concrete attack (block with 3 included chunks: shards A, B, C):**

| j | from_shard_id | receipts | root | block_proof | 4d | 4e | 4f |
|---|---|---|---|---|---|---|---|
| 0 | B | R_B | root_B | proof_B | âś“ | âś“ | âś“ |
| 1 | C | R_C | root_C | proof_C | âś“ | âś“ | âś“ |
| 2 | **A** (fake) | **R_C** | **root_C** | **proof_C** | âś“ (A not seen) | âś“ (same as j=1) | âś“ (same as j=1) |

All checks pass. Shard A's receipts R_A are silently omitted; shard C's receipts R_C are applied twice.

The header is then stored to `DBCol::StateHeaders`. [8](#0-7) 

`set_state_finalize` reads `incoming_receipts_proofs` from the stored header, calls `collect_receipts_from_response` (which ignores `from_shard_id` entirely), and passes the wrong receipt list to `apply_chunk`. [9](#0-8) 

No post-application check compares `apply_result.new_root` against any expected value â€” the corrupted root is simply saved as the new chunk extra.

### Impact Explanation

The syncing node ends up with a state root that differs from every honest node's state root for the same shard. Depending on the omitted receipts, this can cause:
- Incorrect account balances, nonces, or contract storage
- Invalid blocks produced by the synced validator (rejected by the network)
- Inability to validate incoming blocks (consensus failure for the affected node)

Because state sync runs at every epoch boundary for nodes that are not tracking a shard, a single malicious peer serving state headers can corrupt any node that syncs from it.

### Likelihood Explanation

State sync headers are requested from arbitrary peers; there is no requirement that the responding peer be a validator or trusted node. Any peer that can respond to a `StateRequestHeader` message can serve a crafted header. The attack requires knowledge of the block's Merkle tree structure (publicly available from block data) and the ability to construct a valid `ShardStateSyncResponseHeader` with the substituted proof â€” no cryptographic secret is needed.

### Recommendation

Replace the position-agnostic `verify_path` in step 4f with `verify_path_with_index`, passing the shard index derived from `from_shard_id` as the expected position. This cryptographically binds each `from_shard_id` to its Merkle-tree position, making the Pigeonhole argument sound:

```rust
// Derive the expected index from from_shard_id (using the epoch's shard layout)
let expected_index = shard_layout.get_shard_index(*from_shard_id)?;
let num_shards = block.chunks().len() as u64;
if !verify_path_with_index(
    *block_header.prev_chunk_outgoing_receipts_root(),
    block_proof,
    root,
    expected_index as u64,
    num_shards,
) { ... }
```

`verify_path_with_index` already exists in the codebase and checks both the path and the position. [10](#0-9) 

Additionally, validate that each `from_shard_id` is a member of the epoch's shard layout, and that the set of `from_shard_id` values equals the set of included shards.

### Proof of Concept

1. Observe a block B with `chunks_included = 3` (shards A, B, C). Obtain `root_C`, `proof_C` (leaf proof for target shard T in shard C's outgoing receipts tree), and `block_proof_C` (Merkle proof that `root_C` is in B's `prev_chunk_outgoing_receipts_root`).

2. Construct a `ShardStateSyncResponseHeaderV2` where `incoming_receipts_proofs[0]` contains three `ReceiptProof` entries:
   - `(R_B, ShardProof{from_shard_id=B, proof=proof_B})` with `RootProof(root_B, block_proof_B)`
   - `(R_C, ShardProof{from_shard_id=C, proof=proof_C})` with `RootProof(root_C, block_proof_C)`
   - `(R_C, ShardProof{from_shard_id=A, proof=proof_C})` with `RootProof(root_C, block_proof_C)`

3. Serve this header to a syncing node via the state sync peer protocol.

4. Assert `set_state_header` returns `Ok(())` â€” all four checks pass.

5. Trigger `set_state_finalize`. Assert the resulting `ChunkExtra::state_root` differs from the reference state root computed by an honest node that applied `[R_A, R_B, R_C]`.
