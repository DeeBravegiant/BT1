Audit Report

## Title
Stale `rsETHPrice` Enables Yield Dilution and Theft from Existing rsETH Holders - (File: `contracts/LRTOracle.sol`)

## Summary
`LRTOracle.updateRSETHPrice()` is public and permissionless, but `rsETHPrice` is only updated when explicitly called. Between updates, accrued staking rewards cause the true per-share value to exceed the stored price. An attacker can deposit at the stale (undervalued) price, receive excess rsETH, trigger or wait for a price update, then initiate withdrawal at the corrected price — capturing a proportional share of the yield that rightfully belongs to existing holders.

## Finding Description

**Root cause:** `rsETHPrice` is a lazily-updated state variable. `LRTDepositPool.getRsETHAmountToMint()` divides by this stale value directly:

```solidity
// LRTDepositPool.sol L520
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

When staking rewards accrue (via `receiveFromRewardReceiver()` or EigenLayer restaking), the actual TVL rises above `rsethSupply × rsETHPrice`. Any deposit during this window mints more rsETH than the depositor's ETH is worth at the true price.

**Exploit path:**

1. Rewards accrue, making actual TVL > `rsethSupply × rsETHPrice`. This is publicly observable on-chain.
2. Attacker calls `depositETH{value: X}()`. `_beforeDeposit()` calls `getRsETHAmountToMint()`, which divides by the stale (lower) `rsETHPrice`, minting excess rsETH.
3. Attacker calls `updateRSETHPrice()`. `_updateRsETHPrice()` recomputes price over the now-larger supply (attacker's deposit included), diluting the reward increase across all holders. If the price increase exceeds `pricePercentageLimit`, the attacker waits for the protocol keeper — this is a delay, not a prevention.
4. Attacker calls `initiateWithdrawal()`. `getExpectedAssetAmount()` uses the updated (higher) `rsETHPrice`, locking in the inflated ETH amount.
5. After `withdrawalDelayBlocks`, attacker calls `completeWithdrawal()` and receives the locked amount.

**Why existing guards fail:**

- `pricePercentageLimit` (L252–266 of `LRTOracle.sol`) only prevents a non-manager from calling `updateRSETHPrice()` if the price increase exceeds the configured threshold. It does not prevent the deposit at the stale price, and the attacker can simply wait for the keeper if the threshold is exceeded.
- `minRSETHAmountExpected` in `depositETH()` is a minimum slippage guard for the depositor — it does not protect existing holders and the attacker actively benefits from receiving more rsETH.
- `_calculatePayoutAmount()` (L824–835 of `LRTWithdrawalManager.sol`) returns `min(expectedAssetAmount, currentReturn)`, capping the payout at the amount locked at `initiateWithdrawal()` time. Since the attacker initiates withdrawal after the price update, `expectedAssetAmount` is already set to the higher value.

## Impact Explanation

**High — Theft of unclaimed yield.**

Existing rsETH holders accumulate yield as protocol TVL grows. When an attacker deposits at a stale price, the subsequent price update is computed over a larger rsETH supply, permanently reducing the per-share yield credit for all pre-existing holders. The attacker exits at the updated price, capturing the stolen yield. The loss scales linearly with the attacker's capital relative to the existing supply.

Concrete example:
- Existing supply: 1,000 rsETH; stale price: 1.000 ETH; actual TVL: 1,010 ETH (10 ETH rewards accrued).
- Attacker deposits 1,000 ETH → receives 1,000 rsETH (at stale 1.000 rate).
- Price update: new TVL = 2,010 ETH, new supply = 2,000 rsETH → new price = **1.005 ETH/rsETH**.
- Attacker withdraws 1,000 rsETH → receives **1,005 ETH**. Profit: **5 ETH**.
- Existing holders: 1,000 rsETH now worth 1,005 ETH instead of the 1,010 ETH owed. **Loss: 5 ETH** (half of all accrued rewards stolen).

## Likelihood Explanation

**Medium.** The attack is fully permissionless — no privileged access, no front-running, no victim mistake required. The attacker only needs to monitor on-chain state for reward accrual (publicly readable via `getTotalAssetDeposits()` vs. `rsethSupply × rsETHPrice`), supply sufficient ETH capital, and wait through the `withdrawalDelayBlocks` queue (default ~8 days). At 5% APY, locking 1,000 ETH for 8 days costs ~1.1 ETH in opportunity cost; stealing 5 ETH yields ~3.9 ETH net profit. The attack becomes more attractive the longer `updateRSETHPrice()` goes uncalled and the larger the attacker's capital. It is repeatable on every reward accrual cycle.

## Recommendation

Call `_updateRsETHPrice()` inside `_beforeDeposit()` before computing `getRsETHAmountToMint()`. This ensures every deposit uses the current exchange rate, eliminating the stale-price window entirely:

```solidity
function _beforeDeposit(address asset, uint256 depositAmount, uint256 minRSETHAmountExpected)
    private
    returns (uint256 rsethAmountToMint)  // remove `view`
{
    ILRTOracle(lrtConfig.getContract(LRTConstants.LRT_ORACLE)).updateRSETHPrice();
    // ... existing checks ...
}
```

Alternatively, enforce a staleness bound on `rsETHPrice` (e.g., revert if last update was more than N blocks ago) so deposits cannot proceed against a stale price.

## Proof of Concept

**Foundry fork test outline:**

```solidity
// Setup: fork mainnet, advance time so rewards accrue
// Assert: getTotalAssetDeposits(ETH) > rsethSupply * rsETHPrice (stale gap exists)

// Step 1: Attacker deposits 1000 ETH at stale price
uint256 rsethBefore = rsETH.balanceOf(attacker);
depositPool.depositETH{value: 1000 ether}(0, "");
uint256 rsethMinted = rsETH.balanceOf(attacker) - rsethBefore;
// Assert rsethMinted > 1000 ether (received more than fair share)

// Step 2: Update price (or wait for keeper)
lrtOracle.updateRSETHPrice();
uint256 newPrice = lrtOracle.rsETHPrice();
// Assert newPrice > 1 ether (price increased, but less than it would have without attacker)

// Step 3: Initiate withdrawal at updated price
rsETH.approve(address(withdrawalManager), rsethMinted);
withdrawalManager.initiateWithdrawal(ETH_TOKEN, rsethMinted, "");
// expectedAssetAmount locked at newPrice * rsethMinted

// Step 4: Advance blocks past withdrawalDelayBlocks, operator calls unlockQueue()
vm.roll(block.number + withdrawalManager.withdrawalDelayBlocks() + 1);
// operator: withdrawalManager.unlockQueue(ETH_TOKEN, ...)

// Step 5: Complete withdrawal
withdrawalManager.completeWithdrawal(ETH_TOKEN, "");
// Assert attacker ETH balance > 1000 ether (profit from stolen yield)
// Assert existing holders' rsETH worth less than pre-attack entitlement
```
