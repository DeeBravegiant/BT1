### Title
Deposits from superseded `propose_update` proposals are permanently locked when `do_update` clears all entries — (`File: crates/contract/src/update.rs`)

---

### Summary

`propose_update` requires each proposer to attach a NEAR deposit sized to cover storage staking for their update entry. When any update reaches threshold and `do_update` executes, it bulk-clears **all** pending entries and votes. Because `UpdateEntry` never records the proposer's account ID or the deposit amount, and `do_update` issues no refund transfers, every deposit paid for a non-executed proposal is permanently locked inside the contract with no recovery path.

---

### Finding Description

`propose_update` in `crates/contract/src/lib.rs` collects a storage-staking deposit from each proposer:

```rust
let attached = env::attached_deposit();
let required = ProposedUpdates::required_deposit(&update);
// excess refunded, but the exact `required` amount is kept
``` [1](#0-0) 

The deposit is sized by `bytes_used`, which for a full contract binary can be several NEAR tokens. The exact required amount stays in the contract; only the excess is returned to the proposer. [2](#0-1) 

`UpdateEntry`, the struct stored per proposal, holds only the update payload and `bytes_used`. It does **not** record the proposer's `AccountId` or the deposit amount:

```rust
pub(crate) struct UpdateEntry {
    pub(super) update: Update,
    pub(super) bytes_used: u128,
    // ← no proposer field, no deposit field
}
``` [3](#0-2) 

When `vote_update` reaches threshold it calls `do_update`, which bulk-clears every pending entry:

```rust
pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
    let entry = self.entries.remove(id)?;
    // Clear all entries as they might be no longer valid
    self.entries.clear();
    self.vote_by_participant.clear();
    // ... no refund loop
}
``` [4](#0-3) 

Because the proposer identity and deposit amount were never persisted, `do_update` cannot issue refunds even in principle. The NEAR tokens from every cleared proposal remain in the contract's balance with no withdrawal mechanism.

---

### Impact Explanation

Every participant whose proposal is cleared by a competing update permanently loses their storage-staking deposit. For a full contract binary (which can be hundreds of kilobytes), `required_deposit` scales linearly with `bytes_used` and can amount to several NEAR tokens per proposal. With multiple competing proposals in flight — a normal governance scenario — the total locked amount grows with each governance cycle. The funds are not accessible to any account: the contract has no owner-withdraw function, and the freed storage staking is absorbed into the contract's balance without being returned.

This matches the **Medium** allowed impact: *"Balance, request-lifecycle, participant-state, or contract execution-flow manipulation that breaks production safety/accounting invariants without relying on network-level DoS or operator misconfiguration."*

---

### Likelihood Explanation

Competing proposals are a routine governance event. Any time two participants independently propose different contract upgrades or config changes before one reaches threshold, the loser's deposit is lost. The scenario requires no adversarial intent — it arises naturally from concurrent governance activity. The only precondition is that the caller is a current participant/voter, which is the normal operational role.

---

### Recommendation

Store the proposer's `AccountId` and the deposit amount inside `UpdateEntry`:

```rust
pub(crate) struct UpdateEntry {
    pub(super) update: Update,
    pub(super) bytes_used: u128,
    pub(super) proposer: AccountId,   // ← add
    pub(super) deposit: NearToken,    // ← add
}
```

In `do_update`, before calling `self.entries.clear()`, iterate over all remaining entries and schedule a `Promise::new(entry.proposer).transfer(entry.deposit)` for each one. This mirrors the pattern already used in `propose_update` for excess-deposit refunds.

---

### Proof of Concept

1. Participant A calls `propose_update` with a 500 KB contract binary. `required_deposit` ≈ `500_000 * storage_byte_cost` ≈ 5 NEAR. A pays 5 NEAR; the contract keeps it. [5](#0-4) 

2. Participant B calls `propose_update` with a different binary of similar size, paying another 5 NEAR.

3. Participants vote for B's proposal. When the threshold is reached, `vote_update` calls `do_update`: [6](#0-5) 

4. `do_update` removes B's entry, then calls `self.entries.clear()` — silently dropping A's entry with no refund: [7](#0-6) 

5. A's 5 NEAR is permanently locked. `UpdateEntry` never stored A's account ID or deposit, so no recovery is possible: [3](#0-2)

    pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
        let entry = self.entries.remove(id)?;

        // Clear all entries as they might be no longer valid
        self.entries.clear();
        self.vote_by_participant.clear();

        let mut promise = Promise::new(env::current_account_id());
        match entry.update {
            Update::Contract(code) => {
                // deploy contract then do a `migrate` call to migrate state.
                promise = promise.deploy_contract(code).function_call(
                    method_names::MIGRATE,
                    Vec::new(),
                    NearToken::from_near(0),
                    gas,
                );
            }
            Update::Config(config) => {
                // If we vote for a new config, we should use
                // the value `contract_upgrade_deposit_tera_gas` from the config
                // as the new gas value
                let new_config_gas_value = Gas::from_tgas(config.contract_upgrade_deposit_tera_gas);
                promise = promise.function_call(
                    method_names::UPDATE_CONFIG,
                    serde_json::to_vec(&(&config,)).unwrap(),
                    NearToken::from_near(0),
                    new_config_gas_value,
                );
            }
        }
        Some(promise)
    }

fn bytes_used(update: &Update) -> u128 {
    let mut bytes_used = std::mem::size_of::<UpdateEntry>() as u128;

    // Assume a high max of 128 participant votes per update entry.
    bytes_used += 128 * std::mem::size_of::<AccountId>() as u128;

    match update {
        Update::Contract(code) => {
            bytes_used += code.len() as u128;
        }
        Update::Config(config) => {
            let bytes = serde_json::to_vec(&config).unwrap();
            bytes_used += bytes.len() as u128;
        }
    }

    bytes_used
}

fn required_deposit(bytes_used: u128) -> NearToken {
    env::storage_byte_cost().saturating_mul(bytes_used)
}
