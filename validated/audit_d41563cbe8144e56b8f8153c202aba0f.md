### Title
Missing CKD Output Validity Check for `AppPublicKey` Variant in `respond_ckd()` Allows Single Malicious Participant to Forge Confidential Key Derivation Output - (File: `crates/contract/src/lib.rs`)

---

### Summary

The `respond_ckd()` function applies a cryptographic validity check (`ckd_output_check`) only for the `AppPublicKeyPV` variant of CKD requests, but omits this check entirely for the `AppPublicKey` variant. This asymmetric validation — directly analogous to the M-02 pattern — allows a single malicious attested participant to submit an arbitrary forged `CKDResponse` for `AppPublicKey` requests, delivering incorrect confidential key derivation output to users without the required threshold of honest participants agreeing.

---

### Finding Description

The `respond_ckd()` function at `crates/contract/src/lib.rs:653–689` handles two variants of CKD requests via a `match` on `request.app_public_key`:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}   // ← NO OUTPUT CHECK
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For the `AppPublicKeyPV` arm, `ckd_output_check` cryptographically verifies that the submitted `CKDResponse` (`big_y`, `big_c`) is consistent with the app's public key and the MPC network's root BLS key. For the `AppPublicKey` arm, the body is empty — any `CKDResponse` with arbitrary `big_y` and `big_c` values is unconditionally accepted.

Compare this with the `respond()` function for ECDSA/EdDSA signatures, which always verifies the submitted signature against the derived public key and payload hash before resolving any pending yield:

```rust
if !signature_is_valid {
    return Err(RespondError::InvalidSignature.into());
}
pending_requests::resolve_yields_for(...)
``` [2](#0-1) 

The `respond_ckd()` function for `AppPublicKey` resolves all pending yields immediately after the participant check, with no equivalent cryptographic gate: [3](#0-2) 

This is the exact asymmetric-validation pattern from M-02: one code path (`AppPublicKeyPV`) enforces the safety invariant; the parallel path (`AppPublicKey`) silently omits it.

---

### Impact Explanation

A single malicious attested participant can:

1. Observe a pending `AppPublicKey` CKD request in the contract.
2. Call `respond_ckd()` with a fabricated `CKDResponse` containing arbitrary BLS12-381 points (`big_y`, `big_c`).
3. The contract passes all checks (caller is attested participant, protocol is running, `accept_requests` is true) and reaches `resolve_yields_for`, which resolves **all** pending yields for that request with the forged response.
4. Honest nodes that subsequently attempt to respond find no pending yields remaining and are silently ignored.
5. The user's application receives a forged confidential key derivation output.

Because CKD output is used by TEE-based applications to derive private keys, a forged output means the application derives a key that the attacker chose — enabling the attacker to know or control the derived key material. This constitutes **confidential key derivation output produced without the required threshold of participant authorization**, matching the Critical impact tier.

---

### Likelihood Explanation

The attacker must be a single attested participant — strictly below the signing threshold. No collusion is required. The attack window is the time between a user submitting `request_app_private_key` and the first honest node calling `respond_ckd`. A racing malicious node that monitors the NEAR indexer can reliably win this race by submitting its forged response immediately upon seeing the request event, before the honest MPC nodes complete their threshold computation. [4](#0-3) 

---

### Recommendation

Add the same cryptographic output validity check for the `AppPublicKey` variant. If a contract-level check is not feasible for `AppPublicKey` (because the app key lacks the verifiable structure of `AppPublicKeyPV`), the contract should at minimum verify that the submitted response is consistent with the stored BLS root public key using whatever relation is computable on-chain, or reject `AppPublicKey` responses that cannot be verified. The fix mirrors the existing `AppPublicKeyPV` guard:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(app_pk) => {
        if !ckd_output_check_basic(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
```

### Proof of Concept

1. User calls `request_app_private_key` with `CKDAppPublicKey::AppPublicKey(some_bls_key)`. A pending yield is enqueued in `pending_ckd_requests`.
2. Malicious attested participant (account `evil.near`, with a valid TEE attestation on file) calls `respond_ckd(ckd_request, CKDResponse { big_y: [0u8;48].into(), big_c: [0u8;48].into() })`.
3. `respond_ckd` passes: signer check ✓, protocol running ✓, `accept_requests` ✓, attested participant ✓, `AppPublicKey` arm → empty body ✓.
4. `resolve_yields_for` resolves all pending yields with the forged response.
5. The user's callback receives `big_y = [0;48]`, `big_c = [0;48]` — a completely attacker-controlled CKD output — and derives a key from it.
6. Honest nodes subsequently attempt `respond_ckd` but find no pending yields; their correct responses are discarded. [4](#0-3)

    #[handle_result]
    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }
