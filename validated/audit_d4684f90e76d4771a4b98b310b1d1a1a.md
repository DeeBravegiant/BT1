Audit Report

## Title
Merkle Leaf Hash Excludes Contract Address, Enabling Cross-Instance Proof Replay for KERNEL Token Claims - (File: contracts/KERNEL/KernelMerkleDistributor.sol)

## Summary
`KernelMerkleDistributor._processClaim()` constructs the Merkle leaf as `keccak256(abi.encodePacked(index, account, cumulativeAmount))`, omitting `address(this)` and `block.chainid`. Because the duplicate-claim guard (`userClaims`) is per-contract storage, a proof submitted on one distributor instance is cryptographically valid on any other instance sharing the same Merkle root, allowing a user to claim the same allocation multiple times across instances. The same structural flaw exists in `KernelTop100MerkleDistributor._verifyClaimProof()`.

## Finding Description
In `KernelMerkleDistributor._processClaim()` (line 320):
```solidity
bytes32 node = keccak256(abi.encodePacked(index, account, cumulativeAmount));
```
The leaf does not bind to the deploying contract's address or chain ID. The replay guard is:
```solidity
mapping(address user => UserClaim userClaim) public userClaims;
```
This mapping is local to each contract instance. When `setMerkleRoot` is called with the same root `R` on two instances (e.g., Ethereum and Arbitrum), a user's `(index, account, cumulativeAmount, merkleProof)` tuple — made public by their first on-chain claim — passes `MerkleProofUpgradeable.verify` on the second instance identically, because the leaf hash is identical. `isClaimed` on the second instance returns `false` (fresh storage), so the `AlreadyClaimed` guard does not fire. The user receives tokens from both instances.

The same issue exists in `KernelTop100MerkleDistributor._verifyClaimProof()` (line 293):
```solidity
bytes32 leaf = keccak256(abi.encodePacked(user, amount));
```
No contract address or chain ID is included, making proofs portable across any instance sharing the same `merkleRoot`.

## Impact Explanation
**High — Theft of unclaimed yield.** A user can claim their full KERNEL allocation from every distributor instance that shares the same Merkle root, multiplying their yield extraction by the number of instances. Tokens held by each distributor contract beyond the attacker's legitimate share are drained. This directly matches the allowed impact class "Theft of unclaimed yield."

## Likelihood Explanation
**Low.** Exploitation requires the admin to set the same Merkle root on at least two `KernelMerkleDistributor` instances simultaneously. For a multi-chain protocol distributing the same reward epoch across chains, this is a realistic and operationally natural pattern. The window exists whenever the off-chain tooling does not embed the target contract address into each leaf. The exploit is repeatable for every instance sharing the root and requires no special attacker capability beyond submitting a public transaction.

## Recommendation
Include `address(this)` (and optionally `block.chainid`) in the leaf encoding in both contracts:

```solidity
// KernelMerkleDistributor._processClaim() — fixed
bytes32 node = keccak256(abi.encodePacked(address(this), index, account, cumulativeAmount));

// KernelTop100MerkleDistributor._verifyClaimProof() — fixed
bytes32 leaf = keccak256(abi.encodePacked(address(this), user, amount));
```

The off-chain Merkle tree generation must be updated to include the target contract address when building leaves, so that roots generated for one instance are cryptographically invalid on any other.

## Proof of Concept
1. Admin deploys `KernelMerkleDistributor` at `0xAAA` on Ethereum and `0xBBB` on Arbitrum.
2. Off-chain system generates Merkle root `R` from leaves including `keccak256(abi.encodePacked(1, alice, 1000e18))` and calls `setMerkleRoot(R)` on both `0xAAA` and `0xBBB`.
3. Alice calls `claim(1, alice, 1000e18, proof)` on `0xAAA`. Transaction is mined; `proof` is now public on-chain.
4. Alice calls `claim(1, alice, 1000e18, proof)` on `0xBBB` with identical arguments.
5. `_processClaim` on `0xBBB` recomputes `node = keccak256(abi.encodePacked(1, alice, 1000e18))` — identical to the leaf in `R` — and `MerkleProofUpgradeable.verify` returns `true`.
6. `isClaimed(1, alice)` on `0xBBB` returns `false` (fresh storage on Arbitrum).
7. Alice receives 1000e18 KERNEL on Arbitrum in addition to what she already received on Ethereum.

**Foundry fork test plan:** Deploy two instances of `KernelMerkleDistributor` (or fork Ethereum + Arbitrum), call `setMerkleRoot(R)` on both, fund both with KERNEL, have Alice claim on instance A, then replay the identical calldata on instance B, and assert Alice's KERNEL balance equals `2 * allocation`.
