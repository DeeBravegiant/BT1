### Title
`ProposalInit.builder` Is Accepted Without Validation, Allowing a Malicious Proposer to Inject an Arbitrary `sequencer_address` into Block Execution and the Block Hash Commitment — (`crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

### Summary

`ProposalInit` carries a `builder` field ("Address of the one who builds/sequences the block"). Every other security-sensitive field in `ProposalInit` — `proposer`, `height`, `l1_da_mode`, `l2_gas_price_fri`, all four L1 gas prices, `starknet_version`, `version_constant_commitment`, and `fee_proposal_fri` — is checked by `is_proposal_init_valid`. The `builder` field is never checked. `convert_to_sn_api_block_info` maps `init.builder` directly to `sequencer_address` in `BlockInfo`, which is then used as the sequencer address during execution (fee charging, `get_sequencer_address` syscall) and is hashed into `PartialBlockHashComponents` → `ProposalCommitment`. A consensus proposer can set `builder` to any address, have the proposal accepted by all validators, and cause the committed block to carry the wrong sequencer address.

### Finding Description

**Root cause — missing validation in `is_proposal_init_valid`**

`is_proposal_init_valid` validates every field of `ProposalInit` that has a security impact, but `builder` is absent from all checks: [1](#0-0) 

The validated set is `{height, l1_da_mode, l2_gas_price_fri, starknet_version, version_constant_commitment, timestamp, l1_gas_price_{fri,wei}, l1_data_gas_price_{fri,wei}, fee_proposal_fri}`. `builder` is not in it.

**Propagation — `builder` becomes `sequencer_address`**

After `is_proposal_init_valid` passes, `initiate_validation` calls `convert_to_sn_api_block_info(init)`: [2](#0-1) 

`init.builder` is assigned verbatim to `sequencer_address` in the `BlockInfo` that is forwarded to the batcher for both the proposer path (`propose_block`) and the validator path (`validate_block`). [3](#0-2) 

**Effect on block hash commitment**

The batcher builds `PartialBlockHashComponents` from the `BlockInfo`: [4](#0-3) 

`sequencer` (= `init.builder`) is then chained into the Poseidon block hash: [5](#0-4) 

Because both the proposer and the validator derive `sequencer_address` from the same `init.builder` wire value, their independently computed `ProposalCommitment` values agree — the `ProposalFinMismatch` guard does not fire — and the block is committed with the attacker-chosen sequencer address baked into the hash. [6](#0-5) 

**Effect on execution**

During block execution the blockifier exposes `sequencer_address` through the `get_sequencer_address` syscall and uses it as the fee-collection target. With an attacker-controlled `builder`, every transaction in the block sees the wrong sequencer address and fees are transferred to the attacker's address.

### Impact Explanation

| Impact category | Concrete effect |
|---|---|
| **Critical — wrong execution result** | `get_sequencer_address` syscall returns the attacker-chosen address for every transaction in the block. |
| **Critical — incorrect fee / balance** | Transaction fees are credited to the attacker's address instead of the legitimate sequencer. |
| **Critical — wrong block hash / state commitment** | `sequencer_address` is a direct input to `calculate_block_hash`; the committed block hash is permanently wrong. |

### Likelihood Explanation

Any consensus validator that wins a proposal round can trigger this without any off-chain coordination. The proposer simply sets `builder` to an arbitrary address in the `ProposalInit` it broadcasts. All honest validators will accept the proposal because `is_proposal_init_valid` does not check `builder`. The attack requires only a single malicious (or compromised) proposer and is reachable in every round that proposer leads.

### Recommendation

Add a check for `builder` inside `is_proposal_init_valid`, analogous to how `proposer` is validated against the committee in `handle_proposal`. The validator should compare `init_proposed.builder` against its own locally-configured `builder_address` (the same value the honest proposer would have used) and reject any proposal where they differ:

```rust
// In is_proposal_init_valid, after the existing checks:
if init_proposed.builder != proposal_init_validation.expected_builder {
    return Err(ValidateProposalError::InvalidProposalInit(
        init_proposed.clone(),
        proposal_init_validation.clone(),
        format!(
            "builder mismatch: expected={:?}, proposed={:?}",
            proposal_init_validation.expected_builder, init_proposed.builder
        ),
    ));
}
```

`ProposalInitValidation` should be extended with an `expected_builder: ContractAddress` field populated from the node's own configuration, mirroring the pattern already used for `height`, `l1_da_mode`, and `l2_gas_price_fri`. [7](#0-6) 

### Proof of Concept

1. A malicious proposer wins round `R` at height `H`.
2. It constructs `ProposalInit { builder: attacker_address, … }` with all other fields set to valid values.
3. It broadcasts the proposal stream to all validators.
4. Each validator calls `is_proposal_init_valid` — passes, because `builder` is never checked.
5. Each validator calls `convert_to_sn_api_block_info(init)` → `sequencer_address = attacker_address`.
6. Each validator's batcher executes the block with `sequencer_address = attacker_address`:
   - All `get_sequencer_address` syscalls return `attacker_address`.
   - All fee transfers go to `attacker_address`.
7. Each validator's batcher computes `PartialBlockHashComponents { sequencer: attacker_address, … }` and returns a `ProposalCommitment` that includes `attacker_address`.
8. The proposer's `ProposalFin.proposal_commitment` was computed with the same `attacker_address`, so `built_block == received_fin.proposal_commitment` — the `ProposalFinMismatch` guard passes.
9. Consensus reaches decision; the block is committed with `attacker_address` as sequencer in the block hash and in all execution receipts.
