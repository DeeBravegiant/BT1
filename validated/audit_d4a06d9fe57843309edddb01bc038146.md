Audit Report

## Title
Aave Pool Illiquidity Temporarily Blocks All ETH Withdrawal Completions With No Operator Escape Hatch — (`contracts/LRTWithdrawalManager.sol`)

## Summary
When the Aave v3 WETH pool reaches 100% utilization, `_withdrawFromAave` reverts because `aaveWETHGateway.withdrawETH` cannot source liquidity. This revert propagates uncaught through `_processWithdrawalCompletion`, blocking every `completeWithdrawal` call for ETH. Critically, both operator recovery paths — `emergencyWithdrawFromAave` and `setAaveIntegrationEnabled(false)` — also invoke `_withdrawFromAave` internally and fail under the same condition, leaving no on-chain path to unblock users until Aave utilization self-corrects.

## Finding Description
**Root cause:** `_withdrawFromAave` calls `aaveWETHGateway.withdrawETH` with no try/catch or fallback. When Aave WETH utilization is 100%, this external call reverts and the revert propagates up the entire call stack.

**Call path:**

`completeWithdrawal` → `_processWithdrawalCompletion` (L699) → `_withdrawFromAave` (L724) → `aaveWETHGateway.withdrawETH` (L917)

In `_processWithdrawalCompletion`, when `isAaveIntegrationEnabled && asset == ETH_TOKEN` and the contract's native ETH balance is insufficient, `_withdrawFromAave(amountNeeded)` is called with no error handling: [1](#0-0) 

Inside `_withdrawFromAave`, the external call to Aave has no try/catch: [2](#0-1) 

Because the entire transaction reverts, the `popFront()` at L705 and `delete` at L712 are also rolled back, so the user's withdrawal request is preserved and they can retry — but only after Aave liquidity is restored.

**rsETH is burned before this point:** During `unlockQueue`, rsETH is burned at L305 before the user can call `completeWithdrawal`. The user has no rsETH and no ETH during the freeze period. [3](#0-2) 

**Operator escape hatches also fail:**

`emergencyWithdrawFromAave` calls `_withdrawFromAave` directly — same revert path: [4](#0-3) 

`setAaveIntegrationEnabled(false)` calls `_withdrawFromAave(aaveBalance)` **before** setting the `isAaveIntegrationEnabled` flag, so the flag cannot be flipped to false while Aave is illiquid: [5](#0-4) 

## Impact Explanation
All unlocked ETH withdrawal requests are blocked for the duration of Aave WETH pool illiquidity. Users whose rsETH has already been burned (post-`unlockQueue`) hold neither rsETH nor ETH during this period. No user-accessible or operator-accessible on-chain path exists to unblock withdrawals while Aave is illiquid. This matches **Medium — Temporary freezing of funds**. The freeze is bounded by Aave's interest rate model (rates spike sharply at high utilization, incentivizing borrower repayment), making it temporary rather than permanent.

## Likelihood Explanation
Aave WETH at 100% utilization is a realistic, historically observed market condition (e.g., during periods of high ETH borrowing demand or market stress). The condition is not attacker-controlled; it is a normal market state. No special privileges or victim mistakes are required — any user with an unlocked ETH withdrawal request is affected automatically.

## Recommendation
1. Wrap `aaveWETHGateway.withdrawETH` in a try/catch inside `_withdrawFromAave`. On failure, revert with a descriptive error **without** popping the user's nonce from the queue (the current revert already achieves this via transaction rollback, but the error should be surfaced clearly).
2. Restructure `setAaveIntegrationEnabled(false)` to set the `isAaveIntegrationEnabled = false` flag **before** attempting to withdraw from Aave, or wrap the `_withdrawFromAave` call in try/catch so the integration can be disabled even when Aave is illiquid.
3. Consider adding a direct admin override that sets `isAaveIntegrationEnabled = false` without attempting any Aave withdrawal, as a last-resort escape hatch.
