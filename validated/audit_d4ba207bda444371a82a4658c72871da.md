Audit Report

## Title
Aave Pool Illiquidity Permanently Blocks ETH Withdrawal Completion With No Operator Escape Hatch — (`contracts/LRTWithdrawalManager.sol`)

## Summary

When `isAaveIntegrationEnabled` is `true` and the Aave v3 WETH pool reaches 100% utilization, every call to `completeWithdrawal` for ETH reverts unconditionally because `_withdrawFromAave` is called without any error handling. All three operator recovery paths (`emergencyWithdrawFromAave`, `setAaveIntegrationEnabled(false)`, `configureAaveIntegration`) also call `_withdrawFromAave` before taking any corrective action, so none of them can unblock the queue while Aave remains illiquid.

## Finding Description

In `_processWithdrawalCompletion`, when `isAaveIntegrationEnabled && asset == ETH_TOKEN` and the contract's native balance is insufficient, `_withdrawFromAave(amountNeeded)` is called with no `try/catch`:

```solidity
// L719-731
if (isAaveIntegrationEnabled && asset == LRTConstants.ETH_TOKEN) {
    uint256 contractBalance = address(this).balance;
    if (contractBalance < request.expectedAssetAmount) {
        uint256 amountNeeded = request.expectedAssetAmount - contractBalance;
        _withdrawFromAave(amountNeeded);   // ← no error handling
        ...
    }
}
```

Inside `_withdrawFromAave`, the sole external call is:

```solidity
// L917
aaveWETHGateway.withdrawETH(aavePool, withdrawnAmount, address(this));
```

When Aave's WETH pool is at 100% utilization, this call reverts, and the revert propagates all the way out of `completeWithdrawal`/`completeWithdrawalForUser`.

**Contrast with the deposit path**, which explicitly tolerates Aave failures via `try/catch` at L311-316 — the withdrawal path has no equivalent safety net.

**All three operator recovery paths are also blocked:**

1. `emergencyWithdrawFromAave` (L560) calls `_withdrawFromAave(amount)` directly — reverts.
2. `setAaveIntegrationEnabled(false)` (L495) calls `_withdrawFromAave(aaveBalance)` *before* setting `isAaveIntegrationEnabled = enabled` at L503 — the flag is never flipped.
3. `configureAaveIntegration` (L447) calls `_withdrawFromAave(aaveBalance)` before updating addresses — reverts before reconfiguration.

The result: once Aave WETH utilization hits 100%, no user can complete an ETH withdrawal and no operator can disable the integration to restore the direct-balance path.

## Impact Explanation

**Temporary freezing of funds (Medium).** All unlocked ETH withdrawal requests cannot be completed for the duration of Aave pool illiquidity. Users' rsETH has already been burned at `unlockQueue` time; they hold valid, unlocked claims but cannot redeem them. The freeze persists until external borrowers repay enough WETH to restore pool liquidity, which can last hours to days.

## Likelihood Explanation

No attacker action is required. The preconditions are:
- `isAaveIntegrationEnabled == true` — an intentional operator configuration.
- `address(this).balance < request.expectedAssetAmount` — the normal steady state when Aave integration is active, since idle ETH is deposited to Aave via `unlockQueue`.
- Aave WETH pool liquidity < `amountNeeded` — a realistic market condition (Aave v3 WETH utilization has historically reached near-100% during periods of high ETH borrowing demand).

This is a passive market-condition trigger requiring no privileged access or attacker coordination.

## Recommendation

1. Wrap the `_withdrawFromAave` call inside `_processWithdrawalCompletion` in a `try/catch` via an external self-call (mirroring the deposit pattern at L311), falling back to the contract's native ETH balance when Aave is unavailable.
2. Decouple `setAaveIntegrationEnabled(false)` from the withdrawal step: set `isAaveIntegrationEnabled = false` *first*, then attempt the Aave withdrawal in a separate, non-reverting operation, so operators can always disable the integration regardless of Aave pool state.
3. Apply the same decoupling to `configureAaveIntegration` and `emergencyWithdrawFromAave`.

// Fork test — Aave v3 mainnet, block where WETH utilization ≈ 100%
// 1. Deploy/configure LRTWithdrawalManager with Aave integration enabled.
// 2. Operator calls unlockQueue(ETH, ...) → ETH deposited to Aave.

// 3. User calls initiateWithdrawal(ETH, rsETHAmount, "").
// 4. Operator calls unlockQueue to unlock the request.
// 5. Warp past withdrawalDelayBlocks.
// 6. Mock Aave WETH gateway to revert (or fork at block where WETH liquidity == 0):
vm.mockCallRevert(
    aaveWETHGateway,
    abi.encodeWithSelector(IWrappedTokenGatewayV3.withdrawETH.selector),
    "UNDERLYING_BALANCE_ZERO"
);
// 7. completeWithdrawal reverts:
vm.expectRevert();
withdrawalManager.completeWithdrawal(ETH_TOKEN, "");

// 8. emergencyWithdrawFromAave also reverts:
vm.expectRevert();
