Audit Report

## Title
Unconditional `safeTransfer(protocolTreasury, 0)` Permanently Freezes Claims When `feeInBPS == 0` — (`contracts/utils/MerkleDistributor/MerkleDistributor.sol`)

## Summary
`MerkleDistributor.claim()` unconditionally calls `IERC20(token).safeTransfer(protocolTreasury, fee)` even when `fee` evaluates to zero. Because `feeInBPS = 0` is an explicitly valid configuration (only an upper-bound check exists), and because a class of ERC-20 tokens reverts on zero-value transfers, every `claim()` call will revert whenever this combination is active, permanently preventing all users from claiming their merkle-distributed yield.

## Finding Description
In `MerkleDistributor.sol`, `feeInBPS` is validated only against an upper bound at initialization and in `setFeeInBPS()`: [1](#0-0) [2](#0-1) 

This makes `feeInBPS = 0` a valid, intended state. Inside `claim()`, the fee is computed and then transferred unconditionally: [3](#0-2) 

When `feeInBPS == 0`, `fee = 0`, and `safeTransfer(protocolTreasury, 0)` is issued. ERC-20 tokens that revert on zero-value transfers (a known, documented class) will cause this call to revert. The state update recording the claim (lines 134–135) occurs before the transfer, but because Solidity reverts the entire transaction, the claim is never persisted — and since the revert is deterministic, the user can never successfully claim.

The sibling contract `KernelTop100MerkleDistributor.sol` already applies the correct guard: [4](#0-3) 

`MerkleDistributor.sol` lacks this guard entirely.

## Impact Explanation
**Medium — Permanent freezing of unclaimed yield.**

All users of a `MerkleDistributor` instance where `feeInBPS == 0` and `token` is a zero-transfer-reverting ERC-20 are completely unable to claim their rewards. The contract holds their tokens but every `claim()` call reverts deterministically. No user-side action can circumvent this; only an admin upgrade or fee change can unblock claims, but setting `feeInBPS` back to non-zero would incorrectly charge users a fee they were never supposed to pay.

## Likelihood Explanation
- `feeInBPS = 0` is explicitly supported — no lower-bound guard exists, and the owner can call `setFeeInBPS(0)` at any time.
- `token` is configurable to any ERC-20 via `setToken()`, and zero-transfer-reverting tokens are a known, deployed class.
- Any unprivileged user calling `claim()` triggers the revert — no special role, timing, or front-running is required.
- The revert is deterministic and repeatable on every call, making the freeze permanent without an upgrade.

## Recommendation
Guard the fee transfer with a zero-check, mirroring the pattern already used in `KernelTop100MerkleDistributor.sol`:

```solidity
// In MerkleDistributor.claim()
uint256 fee = (claimableAmount * feeInBPS) / 10_000;
uint256 amountToSend = claimableAmount - fee;

IERC20(token).safeTransfer(account, amountToSend);

if (fee > 0) {
    IERC20(token).safeTransfer(protocolTreasury, fee);
}
```

## Proof of Concept
1. Deploy `MerkleDistributor` with a token whose `transfer()` reverts on zero-value input and `feeInBPS = 0`.
2. Fund the contract and set a valid merkle root with a leaf for `alice` with `cumulativeAmount = 100e6`.
3. Alice calls `claim(1, alice, 100e6, proof)`.
4. Line 138: `fee = (100e6 * 0) / 10_000 = 0`; line 139: `amountToSend = 100e6`.
5. Line 141: `safeTransfer(alice, 100e6)` succeeds.
6. Line 144: `safeTransfer(protocolTreasury, 0)` reverts.
7. Entire transaction reverts. Alice's claim state is never written. Every subsequent call produces the same revert. Alice's tokens are permanently frozen in the contract. [5](#0-4)
