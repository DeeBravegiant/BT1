Audit Report

## Title
Permanent ETH Freeze for Contract Withdrawers Without Fallback in `LRTWithdrawalManager` - (File: contracts/LRTWithdrawalManager.sol)

## Summary
A contract address without a `receive`/`fallback` function can call `initiateWithdrawal` for ETH, transferring rsETH into the manager. When the operator later calls `unlockQueue`, the rsETH is irreversibly burned and ETH is moved into the withdrawal manager. Every subsequent call to `completeWithdrawal` reverts at `_transferAsset` because the ETH push to the non-payable contract fails, leaving the ETH permanently frozen with no recovery path.

## Finding Description
The withdrawal lifecycle spans three separate transactions:

**Step 1 — `initiateWithdrawal`** (L166): rsETH is transferred from `msg.sender` into the withdrawal manager via `safeTransferFrom`. No check is performed to verify that `msg.sender` can receive ETH. [1](#0-0) 

**Step 2 — `unlockQueue`** (L305–307): In a separate operator transaction, rsETH held by the manager is burned via `burnFrom` and the corresponding ETH is pulled from `LRTUnstakingVault` via `redeem`. Both operations are irreversible. [2](#0-1) 

**Step 3 — `completeWithdrawal` / `completeWithdrawalForUser`** (L183–203): Both call `_processWithdrawalCompletion(asset, user, referralId)` with the original non-payable `user` address. [3](#0-2) 

Inside `_processWithdrawalCompletion`, the nonce is popped (L705), the request is deleted (L712), `unlockedWithdrawalsCount` is decremented (L717), and then `_transferAsset` is called (L734). If `_transferAsset` reverts, the entire transaction reverts — restoring the request, the nonce, and the count — but the rsETH burned in Step 2 is already gone. [4](#0-3) 

`_transferAsset` uses a low-level call and reverts with `EthTransferFailed` when the recipient has no `receive`/`fallback`: [5](#0-4) 

**Sweep is permanently blocked**: `sweepRemainingAssets` checks `hasUnlockedWithdrawals(asset)` and reverts with `PendingWithdrawalsExist` as long as any unlocked withdrawal exists. Because the revert in Step 3 restores `unlockedWithdrawalsCount`, the stuck request keeps this gate permanently closed. [6](#0-5) 

There is no `cancelWithdrawal`, no alternative recipient parameter, and no admin rescue function in the contract.

## Impact Explanation
**Critical — Permanent freezing of funds.** The user's rsETH is burned (Step 2) and the corresponding ETH is irrecoverably locked in `LRTWithdrawalManager`. Every call to complete the withdrawal reverts identically. `sweepRemainingAssets` is permanently gated. There is no mechanism to redirect the ETH to a different address or recover it by any party.

## Likelihood Explanation
**Medium.** DeFi vaults, multisig wallets (e.g., Gnosis Safe), and DAO treasuries routinely hold rsETH and interact with withdrawal systems without implementing `receive`/`fallback`. The protocol accepts rsETH from any address at `initiateWithdrawal` time with no check that the caller can receive ETH. The failure is silent at initiation and only manifests after the rsETH is already burned in a separate operator transaction, giving the user no opportunity to intervene.

## Recommendation
1. **Preferred**: Add an optional `recipient` parameter to `initiateWithdrawal` so the caller can designate a separate EOA or payable address to receive ETH at completion time.
2. **Alternative**: At `initiateWithdrawal` time, when `asset == ETH_TOKEN`, attempt a zero-value call to `msg.sender` and revert if it fails, preventing non-payable contracts from entering the queue.
3. **Fallback**: Replace the ETH push in `_transferAsset` with a pull-payment pattern — credit the user's claimable balance and expose a `claimETH(address recipient)` function so the user can specify a payable destination.

## Proof of Concept
1. Deploy `VaultContract` — a contract with no `receive`/`fallback` that holds rsETH.
2. `VaultContract` calls `initiateWithdrawal(ETH_TOKEN, 1e18, "")`. rsETH is transferred to `LRTWithdrawalManager` at L166.
3. Operator calls `unlockQueue(ETH_TOKEN, ...)`. rsETH is burned at L305; ETH is moved from `LRTUnstakingVault` to `LRTWithdrawalManager` at L307.
4. `VaultContract` (or anyone) calls `completeWithdrawal(ETH_TOKEN, "")`. `_transferAsset` executes `payable(VaultContract).call{value: amount}("")` — returns `success = false` — reverts with `EthTransferFailed` at L879.
5. The withdrawal request is restored by the revert; rsETH burned in Step 3 is gone. Step 4 reverts identically on every future attempt.
6. `sweepRemainingAssets(ETH_TOKEN)` reverts with `PendingWithdrawalsExist` at L403 because `unlockedWithdrawalsCount` was restored by the revert. ETH is permanently frozen.

```text
    function _processWithdrawalCompletion(address asset, address user, string calldata referralId) internal {
        if (userAssociatedNonces[asset][user].empty()) {
            revert NoWithdrawalRequests(user, asset);
        }

        // Retrieve and remove the oldest withdrawal request for the user.
        uint256 usersFirstWithdrawalRequestNonce = userAssociatedNonces[asset][user].popFront();
        // Ensure the request is already unlocked.
        if (usersFirstWithdrawalRequestNonce >= nextLockedNonce[asset]) revert WithdrawalLocked();

        bytes32 requestId = getRequestId(asset, usersFirstWithdrawalRequestNonce);
        WithdrawalRequest memory request = withdrawalRequests[requestId];

        delete withdrawalRequests[requestId];

        // Check that the withdrawal delay has passed since the request's initiation.
        if (block.number < request.withdrawalStartBlock + withdrawalDelayBlocks) revert WithdrawalDelayNotPassed();

        unlockedWithdrawalsCount[asset]--;

        // If Aave integration is enabled and asset is ETH, withdraw from Aave if needed
        if (isAaveIntegrationEnabled && asset == LRTConstants.ETH_TOKEN) {
            uint256 contractBalance = address(this).balance;
            if (contractBalance < request.expectedAssetAmount) {
                uint256 amountNeeded = request.expectedAssetAmount - contractBalance;
                _withdrawFromAave(amountNeeded);

                // Verify we have sufficient balance after withdrawal
                uint256 balanceAfter = address(this).balance;
                if (balanceAfter < request.expectedAssetAmount) {
                    revert InsufficientLiquidityForWithdrawal();
                }
            }
        }

        _transferAsset(asset, user, request.expectedAssetAmount);

        emit ReferralIdEmitted(referralId);
        emit AssetWithdrawalFinalized(user, asset, request.rsETHUnstaked, request.expectedAssetAmount);
    }
```
