### Title
Truncated 160-bit Keccak256 Hash in `derive_near_deterministic_account_id` Enables Birthday-Attack Account Squatting - (File: core/primitives/src/utils.rs)

### Summary

The NEAR deterministic account ID derivation (`NEP-616`) uses only the last 20 bytes (160 bits) of a keccak256 hash over the borsh-encoded `DeterministicAccountStateInit`. An unprivileged attacker can mount a meet-in-the-middle birthday attack (~2^80 hashes) to find a collision between a malicious `state_init` they control and a legitimate `state_init` a target application will deploy. Because `action_deterministic_state_init` silently skips re-initialization when the account already has a contract, the attacker can pre-occupy the collided address with their own malicious global contract, permanently preventing the legitimate contract from ever being deployed there and gaining control over all funds sent to that account.

### Finding Description

**Root cause — truncated hash space:**

`derive_near_deterministic_account_id` in `core/primitives/src/utils.rs` computes:

```rust
let hash = sha3::Keccak256::digest(&data);
format!("0s{}", hex::encode(&hash[12..32])).parse().unwrap()
```

Only bytes `[12..32]` (20 bytes = 160 bits) are used as the account ID. The full 256-bit keccak256 output is discarded. This is identical to the Ethereum address derivation and is known to be vulnerable to birthday attacks at the 2^80 level (see EIP-3607).

**Attacker-controlled input space:**

`DeterministicAccountStateInit::V1` contains two fields the attacker fully controls in their own `state_init_attacker`:
- `code: GlobalContractIdentifier` — the attacker can deploy any global contract via `DeployGlobalContractAction` and reference it by `CodeHash` or `AccountId`.
- `data: BTreeMap<Vec<u8>, Vec<u8>>` — arbitrary key-value pairs with no semantic constraint beyond per-entry size limits.

By varying `data`, the attacker can generate an unbounded number of distinct `state_init_attacker` values, each producing a different 160-bit account ID.

**Silent no-op on re-initialization:**

In `runtime/runtime/src/deterministic_account_id.rs`, `action_deterministic_state_init` checks:

```rust
if account.contract().is_none() {
    // `uninit` -> `active` account state transition
    deploy_deterministic_account(...)?;
}
```

If the account already has a contract, the function silently succeeds without deploying the legitimate code. The deposit is refunded and no error is raised. This is by design (idempotent lazy initialization), but it is the mechanism that makes the collision exploitable.

**Attack steps:**

1. **Attacker deploys a malicious global contract** (e.g., one that calls `promise_batch_action_add_key_with_full_access` to add a full-access key to itself on first call).

2. **Birthday search (offline):**
   - Generate 2^80 `state_init_attacker` values (varying `data`) → store their 160-bit IDs in a hash table (set A).
   - Generate 2^80 candidate `state_init_target` values that a target application is likely to use (e.g., varying the `owner` field in sharded-contract-style deployments) → store their IDs in set B.
   - With ~2^80 entries in each set over a 2^160 space, the expected number of collisions is ≈1 (birthday bound).

3. **Pre-occupy the collided address:** Submit a `DeterministicStateInitAction` with `state_init_attacker` to the collided account ID. The action passes `validate_deterministic_state_init` because `derive_near_deterministic_account_id(state_init_attacker) == collided_id`. The malicious contract is deployed.

4. **Malicious code executes:** On first call, the malicious contract adds a full-access key for the attacker.

5. **Legitimate deployment silently fails:** When the target application submits its `DeterministicStateInitAction` with `state_init_target`, `account.contract().is_none()` is `false`, so the block at line 38–48 is skipped. The action succeeds with no error, but the legitimate contract is never deployed.

6. **Attacker drains the account:** The attacker uses their full-access key to transfer all funds.

### Impact Explanation

The corrupted protocol value is the **account contract code and storage state** at the collided deterministic account ID, and consequently the **account balance** of any funds deposited to it. Any application relying on `DeterministicStateInitAction` for trustless contract deployment (e.g., sharded contracts per NEP-616) can have its accounts silently hijacked. All NEAR tokens sent to the hijacked account are at the attacker's disposal.

### Likelihood Explanation

The attack requires ~2^80 keccak256 evaluations. As noted in EIP-3607 and referenced literature, the Bitcoin network sustains ~6.5×10^20 hashes/second, making 2^80 ≈ 1.2×10^24 hashes achievable in roughly 30 minutes at that scale. A fraction of that hashrate (e.g., a dedicated GPU cluster) can find a collision in days to weeks. The attacker can wait until the target account accumulates significant funds before executing step 5. The attack is unprivileged: it requires only the ability to submit signed transactions to the NEAR RPC.

### Recommendation

1. **Use the full 256-bit hash output** for the account ID encoding. Replace `hash[12..32]` with `hash[0..32]` and update the `0s` prefix format to accommodate 32 bytes. This raises the birthday bound to 2^128, which is computationally infeasible.
2. **Alternatively, include a domain separator and additional entropy** (e.g., a protocol-version tag or chain ID) in the hashed data to increase effective entropy.
3. **Treat re-initialization as an error** (or at minimum emit a warning event) when the existing contract does not match the submitted `state_init`, so applications can detect squatting.

### Proof of Concept

**Concrete trigger path:**

```
Attacker tx 1: DeployGlobalContractAction { code: <malicious_wasm>, deploy_mode: CodeHash }
  → global contract hash H_evil stored on-chain

Attacker offline: brute-force data_evil such that
  keccak256(borsh({ code: CodeHash(H_evil), data: data_evil }))[12..32]
  == keccak256(borsh({ code: AccountId("sharded.near"), data: {OWNER_KEY: victim_account} }))[12..32]
  → collided_id = "0s<20-byte-hex>"

Attacker tx 2: DeterministicStateInitAction {
    state_init: { code: CodeHash(H_evil), data: data_evil },
    deposit: 0
  } sent to receiver = collided_id
  → validate_deterministic_state_init passes (derived_id == collided_id)
  → malicious contract deployed at collided_id

Victim tx: DeterministicStateInitAction {
    state_init: { code: AccountId("sharded.near"), data: {OWNER_KEY: victim_account} },
    deposit: <storage_balance>
  } sent to receiver = collided_id
  → account.contract().is_none() == false → deployment skipped, tx succeeds silently
  → malicious contract remains; deposit refunded

Attacker tx 3: FunctionCall to collided_id → malicious code adds full-access key
Attacker tx 4: Transfer all funds out
```
