Audit Report

## Title
Unbounded Withdrawal Queue Flooding via Dust `initiateWithdrawal` Calls Causes Unbounded Gas Consumption in `_unlockWithdrawalRequests` - (File: `contracts/LRTWithdrawalManager.sol`)

## Summary
`LRTWithdrawalManager.initiateWithdrawal` accepts any non-zero rsETH amount because `minRsEthAmountToWithdraw` is a plain mapping that defaults to `0`, making the minimum-amount guard vacuously ineffective. An unprivileged attacker can flood the global per-asset withdrawal queue with arbitrarily many dust entries at the cost of gas only. Because `_unlockWithdrawalRequests` processes entries in strict FIFO order with no skip mechanism, a sufficiently large dust queue causes unbounded gas consumption per `unlockQueue` call and delays legitimate withdrawals queued after the dust entries.

## Finding Description
`minRsEthAmountToWithdraw` is declared as a plain `mapping(address => uint256)` with no initialization, so its default value is `0` for every asset.

The guard in `initiateWithdrawal` at line 162 is:
```solidity
if (rsETHUnstaked == 0 || rsETHUnstaked < minRsEthAmountToWithdraw[asset]) {
    revert InvalidAmountToWithdraw();
}
```
When `minRsEthAmountToWithdraw[asset] == 0`, the sub-condition `rsETHUnstaked < 0` is vacuously false for `uint256`, so any `rsETHUnstaked > 0` passes. Each successful call pushes a new entry into the global queue and increments `nextUnusedNonce[asset]` without bound via `_addUserWithdrawalRequest` (lines 744–757).

The operator-facing `_unlockWithdrawalRequests` (lines 790–814) iterates from `nextLockedNonce[asset]` to `firstExcludedIndex` in a `while` loop. For each dust entry, `payoutAmount` is near-zero, so neither break condition (`withdrawalDelayBlocks` check or `availableAssetAmount < payoutAmount`) fires — the loop processes every dust entry sequentially. The operator controls `firstExcludedIndex` to cap gas per call, but cannot skip over unprocessed dust entries; legitimate requests queued after the dust entries are blocked until all preceding dust entries are cleared.

`minAmountToDeposit` in `LRTDepositPool` also defaults to `0` (line 657), meaning depositing 1 wei of ETH yields a non-zero rsETH balance, removing any economic barrier to obtaining dust rsETH.

## Impact Explanation
**Medium — Unbounded gas consumption; temporary freezing of legitimate withdrawal requests.**

Each dust entry requires at minimum one `SLOAD` for `withdrawalRequests[requestId]`, one `SSTORE` for `assetsCommitted[asset]`, one `SSTORE` for `request.expectedAssetAmount`, and one `SSTORE` for `unlockedWithdrawalsCount[asset]`. With thousands of dust entries ahead of legitimate requests, the gas cost of each `unlockQueue` call grows proportionally to the number of dust entries that must be cleared. If the queue depth exceeds what can be processed within the Ethereum block gas limit (~30 M gas), legitimate withdrawal requests queued after the dust entries cannot be unlocked in any single transaction, temporarily freezing those funds until the dust is cleared across many operator calls. This matches the allowed impact: **Medium — Unbounded gas consumption** and **Medium — Temporary freezing of funds**.

## Likelihood Explanation
**Medium.** The attacker requires no special privileges — only a non-zero rsETH balance, which is obtainable by depositing 1 wei of ETH given `minAmountToDeposit == 0`. The attacker's net cost is gas only: rsETH is transferred into the contract on `initiateWithdrawal` and returned as the underlying asset when the dust request is eventually processed. The attack is repeatable across multiple addresses and assets, and no protocol-level barrier prevents it at default configuration.

## Recommendation
1. **Enforce a non-zero minimum withdrawal amount per asset at initialization.** Set `minRsEthAmountToWithdraw[asset]` to a meaningful floor (e.g., `0.001 ether` worth of rsETH) for every supported asset when the asset is added, rather than relying on a post-deployment admin call.
2. **Enforce a non-zero minimum deposit amount.** Set `minAmountToDeposit` to a meaningful floor at initialization to raise the cost of obtaining dust rsETH.
3. **Cap the number of pending withdrawal requests per user per asset.** Add a per-user cap on the `userAssociatedNonces` deque length in `_addUserWithdrawalRequest` to limit how many requests a single address can queue simultaneously.

## Proof of Concept
```solidity
// Preconditions (defaults, no admin action needed):
//   minRsEthAmountToWithdraw[ETH_TOKEN] == 0  (mapping default)
//   minAmountToDeposit == 0                    (mapping default)

// Step 1: Attacker deposits 1 wei ETH → receives tiny rsETH
lrtDepositPool.depositETH{value: 1}(0, "");

// Step 2: Attacker approves withdrawal manager
rsETH.approve(address(withdrawalManager), type(uint256).max);

// Step 3: Repeat N times across multiple addresses — each call costs only gas
for (uint i = 0; i < N; i++) {
    withdrawalManager.initiateWithdrawal(ETH_TOKEN, 1, "spam");
}

// Result: nextUnusedNonce[ETH_TOKEN] == N
// Operator's _unlockWithdrawalRequests must iterate through all N dust entries
// (each requiring SLOAD + multiple SSTOREs) before reaching any legitimate
// withdrawal request queued after the spam.
// Gas cost of unlockQueue scales as O(N); at sufficient N, legitimate
// withdrawals cannot be unlocked within a single block.
```

A Foundry fuzz test parameterizing `N` and measuring gas consumption of `unlockQueue` across increasing queue depths would reproduce the O(N) gas scaling and demonstrate the block-gas-limit threshold at which legitimate withdrawals become temporarily frozen.
