Audit Report

## Title
TVL Inflation via Direct Token Donation Blocks `updateRSETHPrice()`, Causing Stale rsETH Price - (`contracts/LRTOracle.sol`, `contracts/LRTDepositPool.sol`)

## Summary

`LRTDepositPool` exposes an open `receive()` function and uses raw `balanceOf`/`address.balance` reads throughout `getAssetDistributionData()` and `getETHDistributionData()`. Any unprivileged caller can donate ETH or LST tokens directly to tracked contracts to inflate `totalETHInProtocol`, which flows into `_updateRsETHPrice()` and can trigger `PriceAboveDailyThreshold` (blocking non-manager callers) or `DailyFeeMintLimitExceeded` (blocking all callers). While the price is stale, new depositors receive more rsETH than they should, permanently diluting existing holders' share of protocol TVL.

## Finding Description

`LRTDepositPool.receive()` at line 58 accepts ETH from any caller with no restrictions:

```solidity
receive() external payable { }
```

`getAssetDistributionData()` (lines 444–461) and `getETHDistributionData()` (lines 480–496) use raw balance reads:

```solidity
assetLyingInDepositPool = IERC20(asset).balanceOf(address(this));
assetLyingInNDCs += IERC20(asset).balanceOf(nodeDelegatorQueue[i]);
assetLyingUnstakingVault = IERC20(asset).balanceOf(lrtUnstakingVault);

ethLyingInDepositPool = address(this).balance;
ethLyingInNDCs += nodeDelegatorQueue[i].balance;
ethLyingInUnstakingVault = lrtUnstakingVault.balance;
```

These feed into `getTotalAssetDeposits()` → `_getTotalEthInProtocol()` (LRTOracle.sol lines 331–349) → `_updateRsETHPrice()`.

In `_updateRsETHPrice()` (lines 252–266), if the inflated `newRsETHPrice` exceeds `highestRsethPrice` by more than `pricePercentageLimit`, non-manager callers revert with `PriceAboveDailyThreshold`. Separately, the inflated TVL increases `protocolFeeInETH`, and if the resulting `rsethAmountToMintAsProtocolFee` exceeds `maxFeeMintAmountPerDay`, `_checkAndUpdateDailyFeeMintLimit()` (lines 197–210) reverts with `DailyFeeMintLimitExceeded`. Since both `updateRSETHPrice()` (line 88) and `updateRSETHPriceAsManager()` (line 95) call `_updateRsETHPrice()` internally, the `DailyFeeMintLimitExceeded` path blocks even the manager.

While `rsETHPrice` is stale, `getRsETHAmountToMint()` (line 520) divides by the stale lower price:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

New depositors receive more rsETH than they should, permanently diluting existing holders.

## Impact Explanation

The impact is **theft of unclaimed yield (High)**. The yield accrued to existing rsETH holders (reflected in the price appreciation from `highestRsethPrice` to the true `newRsETHPrice`) is permanently redistributed to new depositors who mint rsETH at the stale lower price. These rsETH tokens are already minted and cannot be reclaimed. The dilution is permanent for each deposit that occurs during the stale-price window, even after the manager eventually updates the price.

## Likelihood Explanation

Any unprivileged external caller can execute the attack by sending ETH directly to `LRTDepositPool` via a plain ETH transfer. The required donation is proportional to `TVL × pricePercentageLimit` (e.g., ~11 ETH on a 1000 ETH TVL with a 1% limit). The attacker permanently loses the donated ETH but partially recovers value if they hold rsETH. The attack is repeatable after each manager price update. The `DailyFeeMintLimitExceeded` path requires a larger donation to inflate the fee beyond the daily cap but is also reachable. **Likelihood: Low-Medium.**

## Recommendation

Replace raw `balanceOf`/`address.balance` reads with internal accounting variables updated only through controlled deposit and transfer functions. This prevents untracked donations from affecting `totalETHInProtocol`. Alternatively, restrict `LRTDepositPool.receive()` to only accept ETH from known protocol contracts (reward receivers, node delegators, converters), rejecting arbitrary ETH transfers.

## Proof of Concept

1. Protocol state: TVL = 1000 ETH, rsETH supply = 950, `rsETHPrice` = ~1.052e18, `highestRsethPrice` = 1.052e18, `pricePercentageLimit` = 1e16 (1%).
2. Attacker executes: `(bool ok,) = address(lrtDepositPool).call{value: 11 ether}("");`
3. `getETHDistributionData()` now returns `ethLyingInDepositPool` inflated by 11 ETH; `_getTotalEthInProtocol()` returns 1011 ETH.
4. `newRsETHPrice` = 1011e18 / 950 ≈ 1.0642e18; `priceDifference` = 0.0122e18 > 1% × 1.052e18 = 0.01052e18 → `isPriceIncreaseOffLimit = true`.
5. Any non-manager call to `updateRSETHPrice()` reverts with `PriceAboveDailyThreshold`; `rsETHPrice` remains at 1.052e18.
6. A new depositor deposits 10 ETH and receives `10e18 / 1.052e18 ≈ 9.506` rsETH instead of the correct `10e18 / 1.0642e18 ≈ 9.397` rsETH — ~1.2% excess rsETH minted at the expense of existing holders.
7. For the `DailyFeeMintLimitExceeded` path: a larger donation inflates `rewardAmount`, causing `rsethAmountToMintAsProtocolFee` to exceed `maxFeeMintAmountPerDay`, blocking even `updateRSETHPriceAsManager()` until the manager calls `setMaxFeeMintAmountPerDay()`.
