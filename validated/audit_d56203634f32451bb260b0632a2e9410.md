Audit Report

## Title
Donation Attack on `LRTDepositPool` Inflates `rsETHPrice` via Raw `balanceOf()` Accounting, Causing Depositors to Receive Zero rsETH - (File: contracts/LRTDepositPool.sol, contracts/LRTOracle.sol)

## Summary
`LRTDepositPool.getAssetDistributionData()` computes TVL using raw `IERC20.balanceOf(address(this))` and `address(this).balance`, both of which are inflatable by direct token/ETH transfers. Because `updateRSETHPrice()` is callable by any unprivileged account and the price-increase guard is disabled when `pricePercentageLimit == 0` (the Solidity default), an attacker can donate assets, call `updateRSETHPrice()` to write an arbitrarily inflated price into storage, and cause all subsequent depositors to receive zero rsETH while their assets are permanently absorbed into the pool.

## Finding Description

**Root cause 1 ŌĆö raw balance accounting:**

`getAssetDistributionData()` reads the live token balance directly:

```solidity
assetLyingInDepositPool = IERC20(asset).balanceOf(address(this));
``` [1](#0-0) 

For ETH the same pattern applies:

```solidity
ethLyingInDepositPool = address(this).balance;
``` [2](#0-1) 

The contract accepts arbitrary ETH via an open `receive()`:

```solidity
receive() external payable { }
``` [3](#0-2) 

Any direct ERC-20 transfer or ETH send therefore inflates `getTotalAssetDeposits()` and, by extension, `_getTotalEthInProtocol()`.

**Root cause 2 ŌĆö public price update with no cap when `pricePercentageLimit == 0`:**

`updateRSETHPrice()` carries no role restriction:

```solidity
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
``` [4](#0-3) 

Inside `_updateRsETHPrice()`, the only guard against an unbounded price increase is:

```solidity
bool isPriceIncreaseOffLimit =
    pricePercentageLimit > 0 && priceDifference > pricePercentageLimit.mulWad(highestRsethPrice);
``` [5](#0-4) 

`pricePercentageLimit` is a plain `uint256` storage variable that is never set in `initialize()`, so it defaults to zero. When zero, the condition short-circuits to `false` unconditionally, and the new price ŌĆö however large ŌĆö is written to storage:

```solidity
uint256 newRsETHPrice = (totalETHInProtocol - protocolFeeInETH).divWad(rsethSupply);
``` [6](#0-5) 

**Root cause 3 ŌĆö minting divides by the stale stored price:**

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [7](#0-6) 

`lrtOracle.rsETHPrice()` is the value last written by `updateRSETHPrice()`. After a donation attack inflates it, every subsequent depositor is divided by the inflated denominator, yielding zero.

**Why the `minRSETHAmountExpected` guard fails:**

```solidity
rsethAmountToMint = getRsETHAmountToMint(asset, depositAmount);
if (rsethAmountToMint < minRSETHAmountExpected) {
    revert MinimumAmountToReceiveNotMet();
}
``` [8](#0-7) 

When `minRSETHAmountExpected == 0` (the default many front-ends and users pass), `0 < 0` is false and the transaction succeeds, minting zero rsETH to the victim.

**Fee-minting interaction (additional precondition):** `_updateRsETHPrice()` calls `_checkAndUpdateDailyFeeMintLimit()` with the computed fee amount. If `protocolFeeInBPS > 0` and `maxFeeMintAmountPerDay == 0` (both defaults), the call reverts when a non-zero fee is computed, blocking the attack. The attack therefore executes cleanly when any of the following hold: (a) `protocolFeeInBPS == 0`, (b) `maxFeeMintAmountPerDay` has been set to a non-zero value, or (c) the deposit pool or withdrawal manager is paused (setting `protocolFeeInETH = 0` regardless of fee BPS). All three are realistic at protocol launch or during operational windows. [9](#0-8) 

## Impact Explanation

**Critical ŌĆö Direct theft of user funds.** The victim's deposited assets (ETH or ERC-20) are transferred into `LRTDepositPool` and zero rsETH is minted in return. The attacker's single wei of rsETH, being the only outstanding supply, represents a claim on the entire pool including the victim's funds. This is direct, permanent loss of principal ŌĆö not merely unclaimed yield ŌĆö and falls squarely within the Critical impact class.

## Likelihood Explanation

- `pricePercentageLimit` is zero by default and requires an explicit admin call to `setPricePercentageLimit()` to activate the guard. Any deployment window before that call is exploitable.
- `updateRSETHPrice()` is callable by any EOA or contract with no role restriction.
- The donation vector (ERC-20 `transfer()` or ETH `send()`) requires no special permissions.
- The attack is front-runnable: the attacker observes a pending deposit, donates, updates the price, and the victim's transaction executes against the inflated price.
- `minRSETHAmountExpected == 0` is the common default, removing the last user-side protection.

Likelihood: **Medium** (requires `pricePercentageLimit == 0` and a compatible fee configuration, both of which are the default state at deployment).

## Recommendation

1. **Replace raw `balanceOf()` with an internal accounting variable** incremented only through the official deposit path, making donations invisible to the price oracle.
2. **Require `pricePercentageLimit > 0` in `initialize()`** or enforce a hard-coded maximum price-increase cap per oracle update.
3. **Restrict `updateRSETHPrice()` to an authorized role** (e.g., `MANAGER` or a keeper) so arbitrary callers cannot trigger a price update immediately after a donation.
4. **Enforce a protocol-level minimum `minRSETHAmountExpected`** (e.g., revert if `rsethAmountToMint == 0`) to guarantee depositors always receive at least some rsETH.

## Proof of Concept

Assume `pricePercentageLimit == 0` (default), `protocolFeeInBPS == 0`, stETH price Ōēł 1 ETH, and the protocol has just launched.

| Step | Action | rsETH supply | `totalETHInProtocol` | `rsETHPrice` (stored) |
|------|--------|-------------|----------------------|----------------------|
| 1 | Attacker calls `depositAsset(stETH, 1 wei, 0, "")` | 1 wei | 1 wei | 1e18 |
| 2 | Attacker transfers 1 000 stETH directly to `LRTDepositPool` | 1 wei | Ōēł1 000e18 | 1e18 (stale) |
| 3 | Attacker calls `updateRSETHPrice()` | 1 wei | Ōēł1 000e18 | Ōēł1 000e18 ├Ś 1e18 / 1 = 1 000e36 |
| 4 | Victim calls `depositAsset(stETH, 1e18, 0, "")` | 1 wei | ŌĆö | ŌĆö |

At step 4, `getRsETHAmountToMint` computes:

```
rsethAmountToMint = (1e18 ├Ś 1e18) / (1 000e36)
                  = 1e36 / 1e39
                  = 0   ŌåÉ integer truncation
```

`0 < 0` is false, so `_beforeDeposit` does not revert. The victim's 1 stETH is transferred into `LRTDepositPool` and the victim receives zero rsETH. The attacker's 1 wei rsETH now represents a claim on Ōēł1 001 stETH.

**Foundry test plan:**
1. Deploy `LRTConfig`, `LRTOracle`, `LRTDepositPool`, and a mock stETH with price oracle returning 1e18.
2. Attacker deposits 1 wei stETH; assert rsETH supply == 1.
3. Attacker directly transfers 1 000e18 stETH to `LRTDepositPool`.
4. Attacker calls `updateRSETHPrice()`; assert `rsETHPrice > 1e36`.
5. Victim calls `depositAsset(stETH, 1e18, 0, "")`.
6. Assert victim's rsETH balance == 0 and `LRTDepositPool` stETH balance increased by 1e18.
