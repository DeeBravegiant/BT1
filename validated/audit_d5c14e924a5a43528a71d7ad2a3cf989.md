### Title
Incorrect VRF Key Hash Removal in `poolReapTransition` Deletes Shared Entries Instead of Decrementing Reference Count — (File: `eras/shelley/impl/src/Cardano/Ledger/Shelley/Rules/PoolReap.hs`)

---

### Summary

In `poolReapTransition`, the `danglingVRFKeyHashes` set is removed from `psVRFKeyHashes` using `Map.withoutKeys` (complete deletion). However, `psVRFKeyHashes` is a reference-count map (`Map (VRFVerKeyHash StakePoolVRF) (NonZero Word64)`). When a VRF key is shared by multiple pools — a state that is valid and explicitly handled for pools registered before the v11 hard fork — complete deletion instead of decrement corrupts the map. A subsequent pool registration certificate can then bypass the VRF uniqueness check, registering a pool with a VRF key that is still actively used by another pool.

---

### Finding Description

`psVRFKeyHashes` is a reference-count map introduced at the v11 hard fork. `populateVRFKeyHashes` (called during the hard fork transition) correctly counts all VRF keys across both `psStakePoolsL` and `psFutureStakePoolParamsL`, so a VRF key shared by N pools gets count N. [1](#0-0) 

During `poolReapTransition`, two separate removal paths exist for `psVRFKeyHashes`:

1. **Retired pools** — handled correctly via `removeVRFKeyHashOccurrences`, which decrements the count and only deletes the entry when it reaches zero: [2](#0-1) 

2. **Dangling VRF keys** (old VRF keys of pools that re-registered with a new VRF key in the same epoch) — handled **incorrectly** via `Map.withoutKeys`, which unconditionally deletes the entire entry regardless of its count: [3](#0-2) [4](#0-3) 

The `danglingVRFKeyHashes` set is built by finding pools that appear in both `psStakePoolsL` and `psFutureStakePoolParamsL` with differing VRF keys, then collecting the old (current) VRF key:

```haskell
danglingVRFKeyHashes =
  Set.fromList $ Map.elems $
    Map.merge Map.dropMissing Map.dropMissing
      ( Map.zipWithMaybeMatched $ \_ sps sppF ->
          if sps ^. spsVrfL /= sppF ^. sppVrfL then Just (sps ^. spsVrfL) else Nothing
      )
      (ps0 ^. psStakePoolsL)
      (ps0 ^. psFutureStakePoolParamsL)
```

If the old VRF key has a reference count of 2 (because another pool also uses it, registered before v11 when duplicates were permitted), `Map.withoutKeys` deletes the entry entirely. The other pool's VRF key is now absent from `psVRFKeyHashes`, making it appear unregistered.

The POOL rule's uniqueness check is: [5](#0-4) 

Because `psVRFKeyHashes` no longer contains the VRF key, `Map.notMember sppVrf psVRFKeyHashes` returns `True`, and a new pool registration with that VRF key succeeds — bypassing the uniqueness invariant.

---

### Impact Explanation

An attacker can submit a `RegPoolTxCert` pool registration certificate with a VRF key that is still actively used by another pool, bypassing the `VRFKeyHashAlreadyRegistered` check. This exceeds the intended validation limit introduced in protocol version 11. Two pools sharing a VRF key produce identical VRF outputs for the same slot, corrupting the leader-election lottery and allowing a pool operator to gain an unfair advantage in block production.

This matches: **Medium — attacker-controlled certificates exceed intended validation limits.**

---

### Likelihood Explanation

The precondition — pools sharing a VRF key — was explicitly permitted before v11 and is handled by `populateVRFKeyHashes`. The v11 hard fork is recent; pools registered before it with duplicate VRF keys remain in the ledger state. Pool re-registration (to update cost, margin, or metadata) is a routine operation. Any pool operator who re-registers after v11 with a new VRF key, while sharing the old VRF key with another pool, silently corrupts `psVRFKeyHashes`. The window for exploitation then opens until the next epoch boundary restores consistency (which it does not — the map stays corrupted until the remaining pool also retires or re-registers).

---

### Recommendation

Replace the `Map.withoutKeys danglingVRFKeyHashes` call with `removeVRFKeyHashOccurrences (Set.toList danglingVRFKeyHashes)`, consistent with how retired pool VRF keys are handled. This decrements the reference count rather than unconditionally deleting the entry, preserving correctness when multiple pools share a VRF key.

```haskell
-- Before (incorrect):
& certPStateL . psVRFKeyHashesL
  %~ ( removeVRFKeyHashOccurrences retiredVRFKeyHashes
         . (`Map.withoutKeys` danglingVRFKeyHashes)
     )

-- After (correct):
& certPStateL . psVRFKeyHashesL
  %~ ( removeVRFKeyHashOccurrences retiredVRFKeyHashes
         . removeVRFKeyHashOccurrences (Set.toList danglingVRFKeyHashes)
     )
```

---

### Proof of Concept

1. Before v11: register pools `kh1` and `kh2` both with VRF key `V`. This is permitted pre-v11.
2. Enact the v11 hard fork. `populateVRFKeyHashes` sets `psVRFKeyHashes = {V → 2}`.
3. Pool `kh1` re-registers with a fresh VRF key `V'`. The POOL rule inserts `V' → 1` into `psVRFKeyHashes`. `danglingVRFKeyHashes` will contain `V` at the next epoch boundary.
4. Pass an epoch. `poolReapTransition` executes `Map.withoutKeys {V}`, deleting `V` entirely. `psVRFKeyHashes = {V' → 1}`. Pool `kh2` still uses `V`, but `V` is absent from the map.
5. Register a new pool `kh3` with VRF key `V`. The check `Map.notMember V psVRFKeyHashes` passes (returns `True`). `kh3` is registered with `V`, duplicating `kh2`'s VRF key in violation of the v11 invariant. [6](#0-5) [7](#0-6)

populateVRFKeyHashes :: PState era -> PState era
populateVRFKeyHashes pState =
  pState
    & psVRFKeyHashesL
      %~ accumulateVRFKeyHashes (pState ^. psStakePoolsL) (^. spsVrfL)
        . accumulateVRFKeyHashes (pState ^. psFutureStakePoolParamsL) (^. sppVrfL)
  where
    accumulateVRFKeyHashes ::
      Map (KeyHash StakePool) a ->
      (a -> VRFVerKeyHash StakePoolVRF) ->
      Map (VRFVerKeyHash StakePoolVRF) (NonZero Word64) ->
      Map (VRFVerKeyHash StakePoolVRF) (NonZero Word64)
    accumulateVRFKeyHashes spMap getVrf acc =
      Map.foldr' (addVRFKeyHashOccurrence . getVrf) acc spMap
    addVRFKeyHashOccurrence vrfKeyHash =
      Map.insertWith combine vrfKeyHash (knownNonZeroBounded @1)
      where
        -- Saturates at maxBound: if (+1) would overflow to 0, keep existing value
        combine _ oldVal = fromMaybe oldVal $ mapNonZero (+ 1) oldVal
