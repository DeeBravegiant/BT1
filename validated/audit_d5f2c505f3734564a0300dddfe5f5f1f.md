### Title
Missing Minimum Confirmation Threshold Validation in `verify_foreign_transaction` Allows MPC Attestation of Unfinalized Bitcoin Transactions — (File: `crates/near-mpc-contract-interface/src/types/foreign_chain.rs`, `crates/contract/src/lib.rs`)

### Summary

The `verify_foreign_transaction` entry point accepts a `BitcoinRpcRequest` whose `confirmations` field is a plain `u64` with no minimum enforced anywhere in the contract or the node-side inspector. An unprivileged caller can submit `BlockConfirmations(0)` or `BlockConfirmations(1)`, causing the MPC network to produce a threshold signature attesting to a Bitcoin transaction that has not reached any meaningful finality. A bridge contract (e.g., Omnibridge) that consumes this signature without independently re-checking the embedded confirmation count would release NEAR-side assets for a Bitcoin transaction that may subsequently be reorganized out of the canonical chain.

### Finding Description

`BlockConfirmations` is defined as a transparent `u64` newtype with no invariant: [1](#0-0) 

The ABI schema explicitly permits `minimum: 0.0`: [2](#0-1) 

`verify_foreign_transaction` in the contract performs only two checks before enqueuing the request: (1) domain purpose is `ForeignTx`, and (2) the requested chain is in the supported set. It does **not** inspect or bound the `confirmations` value: [3](#0-2) 

On the node side, the Bitcoin inspector enforces the caller-supplied threshold with a simple `<=` comparison: [4](#0-3) 

With `block_confirmations_threshold = 0`, the condition `0 <= rpc_response.confirmations` is always `true` for any `u64`, so the check is vacuous. With `block_confirmations_threshold = 1`, a single-confirmation Bitcoin transaction (far below the conventional 6-confirmation finality bar) passes. The canonical-block check that follows still runs, but it only verifies that the block containing the transaction is on the current best chain at query time — it does not protect against near-tip reorganizations.

The signed payload encodes the caller-supplied `confirmations` value as part of `ForeignChainRpcRequest`, so the value is visible to downstream verifiers. However, the MPC network's design intent is to be the trusted finality oracle:

> *"Omnibridge inbound flow (foreign chain → NEAR) where Chain Signatures are required to attest that a foreign transaction **finalized successfully**."* [5](#0-4) 

By accepting `confirmations: 0` or `confirmations: 1` without rejection, the contract contradicts this invariant and shifts the entire finality-enforcement burden to every downstream consumer.

### Impact Explanation

A bridge contract that trusts the MPC signature as proof of Bitcoin finality — without re-reading the embedded `confirmations` field — will release NEAR-side assets for a Bitcoin transaction that can still be reorganized. This constitutes invalid bridge execution and direct loss of funds controlled by the bridge. Even a bridge that does re-read the field is vulnerable if it does not enforce a protocol-level minimum, because the MPC contract provides no such floor.

This matches the allowed impact: **High — forged foreign-chain verification that causes invalid bridge execution or double-spend conditions.**

### Likelihood Explanation

The entry point `verify_foreign_transaction` is a public, payable function callable by any NEAR account. No privileged role is required. The only cost is the `MINIMUM_SIGN_REQUEST_DEPOSIT`. A malicious or buggy bridge service can trivially submit `confirmations: 0` or `confirmations: 1` and receive a valid threshold signature from the MPC network.

### Recommendation

1. **Enforce a protocol-level minimum in the contract.** Reject `BitcoinRpcRequest` with `confirmations` below a configurable floor (e.g., `MIN_BITCOIN_CONFIRMATIONS = 1`, with the recommended production value being 6) inside `verify_foreign_transaction`, analogously to how `ChainEntry` validation rejects `quorum == 0`: [6](#0-5) 

2. **Add a typed `NonZeroU64` or bounded newtype** for `BlockConfirmations` so the constraint is enforced at the type level and cannot be bypassed by future callers.

3. **Document the minimum** in the ABI schema (`minimum: 1` instead of `minimum: 0`) so SDK consumers and bridge developers are aware of the floor.

### Proof of Concept

```rust
// Any unprivileged NEAR account can call:
contract.verify_foreign_transaction(VerifyForeignTransactionRequestArgs {
    domain_id: foreign_tx_domain_id,
    payload_version: ForeignTxPayloadVersion::V1,
    request: ForeignChainRpcRequest::Bitcoin(BitcoinRpcRequest {
        tx_id: BitcoinTxId(attacker_controlled_tx_hash),
        confirmations: BlockConfirmations(0), // ← no validation rejects this
        extractors: vec![BitcoinExtractor::BlockHash],
    }),
});
// The MPC network queries the Bitcoin RPC, finds confirmations >= 0 (always true),
// passes the canonical-block check, and produces a threshold ECDSA signature
// over SHA-256(borsh({ request, values: [block_hash] })).
// A bridge contract that does not re-check the embedded confirmations count
// will accept this signature as proof of finality and release NEAR-side assets.
```

    pub fn verify_foreign_transaction(&mut self, request: VerifyForeignTransactionRequestArgs) {
        log!(
            "verify_foreign_transaction: predecessor={:?}, request={:?}",
            env::predecessor_account_id(),
            request
        );

        self.check_request_preconditions(
            request.domain_id,
            DomainPurpose::ForeignTx,
            Gas::from_tgas(self.config.sign_call_gas_attachment_requirement_tera_gas),
            MINIMUM_SIGN_REQUEST_DEPOSIT,
        );

        let requested_chain = request.request.chain();
        let supported_chains = self.get_supported_foreign_chains();
        if !supported_chains.contains(&requested_chain) {
            env::panic_str(
                &InvalidParameters::ForeignChainNotSupported {
                    requested: requested_chain,
                }
                .to_string(),
            );
        }

        let callback_gas = Gas::from_tgas(
            self.config
                .return_signature_and_clean_state_on_success_call_tera_gas,
        );

        let request = args_into_verify_foreign_tx_request(request);
        let callback_args = serde_json::to_vec(&(&request,)).unwrap();
        self.enqueue_yield_request(
            method_names::RETURN_VERIFY_FOREIGN_TX_AND_CLEAN_STATE_ON_SUCCESS,
            callback_args,
            callback_gas,
            move |this, id| this.add_verify_foreign_tx_request(request, id),
        );
    }
