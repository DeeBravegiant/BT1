### Title
Dijkstra Batch Sub-Transaction Deposit Refund Replay — Multiple Sub-Transactions Can Claim the Same Stake Credential Deposit Refund - (File: eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/SubEntities.hs)

---

### Summary

The Dijkstra era introduces batch transactions containing multiple sub-transactions (`dtbrSubTransactions`). The `validateBatchWithdrawals` function in `Utxo.hs` correctly aggregates withdrawals across all sub-transactions and validates them against the original account balance. However, no equivalent batch-level guard exists for deposit refunds claimed via stake credential deregistration certificates (`UnRegDepositTxCert`). Multiple sub-transactions within the same batch can each include a deregistration certificate for the same staking credential, potentially claiming the same deposit refund multiple times from the deposit pot.

---

### Finding Description

**Vulnerability class:** Funds/accounting bug — deposit refund replay within a Dijkstra batch transaction.

The Dijkstra era's `DijkstraTxBodyRaw` includes a `dtbrSubTransactions` field holding an ordered map of sub-transactions keyed by `TxId`. [1](#0-0) 

The top-level `UTXO` rule calls `validateBatchWithdrawals` to aggregate all withdrawals across the top-level transaction and every sub-transaction, checking the total against the **original** (pre-batch) account balance: [2](#0-1) 

This batch-level guard correctly prevents a sub-transaction from withdrawing funds that another sub-transaction in the same batch already withdrew.

However, **no equivalent batch-level guard exists for deposit refunds** from `UnRegDepositTxCert` certificates. Deposit refunds are processed per-sub-transaction inside `dijkstraSubEntitiesTransition`: [3](#0-2) 

Each sub-transaction's certificates are processed independently via `trans @(EraRule "SUBCERTS" era)`. There is no aggregation step that checks whether the same staking credential's deposit has already been claimed by a sibling sub-transaction in the same batch.

The developers explicitly acknowledge this gap. The test `"Multiple subtransactions cannot get the same refund"` is marked `xit` (disabled/pending) with the comment:

```haskell
submitFailingTx tx . NE.singleton $ error "TODO: predicate failure not yet implemented"
``` [4](#0-3) 

The `error "TODO: predicate failure not yet implemented"` confirms that the specific check preventing this case has not been implemented. The test is disabled precisely because the protection is absent.

The asymmetry is stark: `validateBatchWithdrawals` (production code) guards withdrawals at the batch level using the original `accounts` snapshot, but the analogous guard for deposit refunds — which would need to check that the same credential's deposit is not claimed more than once across all sub-transactions — does not exist.

---

### Impact Explanation

**Critical — Direct creation of ADA through an invalid ledger state transition.**

An attacker who controls a registered staking credential with deposit `D` can craft a top-level Dijkstra transaction containing `N` sub-transactions, each carrying `UnRegDepositTxCert stakingCred D`. If the batch deposit refund check is absent, each sub-transaction independently claims `D` from the deposit pot, yielding `N × D` in refunds while only `D` was ever deposited. The deposit pot is drained by `(N-1) × D` beyond what was contributed, constituting direct creation of ADA from nothing — an invalid ledger state transition.

---

### Likelihood Explanation

**Medium.** The Dijkstra era is the newest era and sub-transactions are a new feature. An attacker must:
1. Register a staking credential (paying the deposit `D`).
2. Construct a valid Dijkstra top-level transaction with multiple sub-transactions each containing `UnRegDepositTxCert` for the same credential.
3. Submit the transaction.

No privileged keys, governance majority, or leaked secrets are required. The attacker only needs to be a normal transaction sender. The deposit `D` is a known protocol parameter, making the attack fully deterministic and repeatable.

---

### Recommendation

Implement a batch-level deposit refund guard analogous to `validateBatchWithdrawals`. Before processing sub-transaction certificates, aggregate all `UnRegDepositTxCert` credentials across the entire batch (top-level and all sub-transactions) and verify that no credential appears more than once. This mirrors the existing pattern:

```haskell
-- analogous to validateBatchWithdrawals
validateBatchDepositRefunds :: ... -> Tx TopTx era -> Test (...)
validateBatchDepositRefunds tx =
  let allDeregCreds = concatMap deregCredsOf (topTxCerts ++ allSubTxCerts)
      duplicates    = findDuplicates allDeregCreds
  in failureOnNonEmpty duplicates DuplicateDepositRefundInBatch
```

The fix should be applied in `dijkstraUtxoTransition` (or the `SUBLEDGERS` rule) before sub-transaction certificate processing begins, using the original cert state snapshot — exactly as `validateBatchWithdrawals` uses the original `accounts` snapshot.

---

### Proof of Concept

1. Alice registers staking credential `C`, paying deposit `D = ppKeyDepositL`.
2. Alice constructs sub-tx₁ with `certsTxBodyL = [UnRegDepositTxCert C D]` and a unique UTxO input `i₁`.
3. Alice constructs sub-tx₂ with `certsTxBodyL = [UnRegDepositTxCert C D]` and a unique UTxO input `i₂`.
4. Alice wraps both in a top-level Dijkstra transaction: `subTransactionsTxBodyL = OMap.fromFoldable [subTx1, subTx2]`.
5. If the batch deposit refund check is absent, both sub-transactions independently claim `D`, yielding `2D` in refunds from a deposit pot that only holds `D` for credential `C`.
6. The deposit pot is debited `2D`, creating `D` ADA from nothing.

-- Unregistered accounts are treated as having 0 balance.
validateBatchWithdrawals ::
  ( EraTx era
  , EraAccounts era
  , DijkstraEraTxBody era
  ) =>
  Accounts era ->
  Tx TopTx era ->
  Test (DijkstraUtxoPredFailure era)
validateBatchWithdrawals accounts tx =
  let allWithdrawals =
        Map.unionsWith (<>) $
          unWithdrawals (tx ^. bodyTxL . withdrawalsTxBodyL)
            : [ unWithdrawals $ subTx ^. bodyTxL . withdrawalsTxBodyL
              | subTx <- OMap.elems $ tx ^. bodyTxL . subTransactionsTxBodyL
              ]
      badWithdrawals =
        Map.mapMaybeWithKey
          ( \acctAddr withdrawn ->
              let balance = getAccountBalance acctAddr
               in if withdrawn > balance
                    then Just Mismatch {mismatchSupplied = withdrawn, mismatchExpected = balance}
                    else Nothing
          )
          allWithdrawals
   in failureOnNonEmptyMap badWithdrawals WithdrawalsExceedAccountBalance
  where
    getAccountBalance (AccountAddress _ (AccountId cred)) =
      case lookupAccountState cred accounts of
        Nothing -> mempty -- unregistered account, 0 balance
        Just accountState -> fromCompact $ accountState ^. balanceAccountStateL

dijkstraSubEntitiesTransition = do
  TRC (subCertsEnv, certState, certificates) <- judgmentContext
  let tx = certsTx subCertsEnv
      pp = certsPParams subCertsEnv
      curEpoch = certsCurrentEpoch subCertsEnv
      withdrawals = tx ^. bodyTxL . withdrawalsTxBodyL
      accounts = certState ^. certDStateL . accountsL

  runTest $ Conway.validateWithdrawalsDelegated accounts tx

  network <- liftSTS $ asks networkId
  let (missingWithdrawals, exceededWithdrawals) =
        case withdrawalsThatExceedAccountBalance withdrawals network accounts of
          Nothing -> (Map.empty, Map.empty)
          Just (missing, exceeded) -> (unWithdrawals missing, exceeded)
  failOnNonEmptyMap missingWithdrawals $
    injectFailure . SubWithdrawalsMissingAccounts . Withdrawals . NEM.toMap
  failOnNonEmptyMap exceededWithdrawals $ injectFailure . SubWithdrawalAmountsExceedAccountBalances

  let certStateBeforeSubCerts =
        certState
          & Conway.updateDormantDRepExpiries tx curEpoch
          & Conway.updateVotingDRepExpiries tx curEpoch (pp ^. ppDRepActivityL)
          & certDStateL . accountsL %~ applyWithdrawals withdrawals
  certStateAfterSubCerts <-
    trans @(EraRule "SUBCERTS" era) $ TRC (subCertsEnv, certStateBeforeSubCerts, certificates)

  let directDeposits = tx ^. bodyTxL . directDepositsTxBodyL
      accountsAfterSubCerts = certStateAfterSubCerts ^. certDStateL . accountsL
  failOnJust (directDepositsMissingAccounts directDeposits accountsAfterSubCerts) $
    injectFailure . SubDirectDepositsToMissingAccounts

  pure $ certStateAfterSubCerts & certDStateL . accountsL %~ applyDirectDeposits directDeposits
