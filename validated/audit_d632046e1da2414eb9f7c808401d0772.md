Audit Report

## Title
`ChainlinkPriceOracle.getAssetPrice()` Accepts Stale Chainlink Prices Without Freshness Validation, Enabling Excess rsETH Minting - (`contracts/oracles/ChainlinkPriceOracle.sol`)

## Summary

`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but silently discards the `updatedAt` timestamp and `answeredInRound` fields, accepting any price regardless of age. Because `LRTOracle.updateRSETHPrice()` is a permissionless public function, an attacker can trigger an rsETH price update during a stale feed window, causing the protocol to undervalue its TVL and mint excess rsETH to the next depositor, diluting all existing holders' proportional claim on the underlying ETH.

## Finding Description

`ChainlinkPriceOracle.getAssetPrice()` extracts only the raw price from `latestRoundData()`:

```solidity
// contracts/oracles/ChainlinkPriceOracle.sol L52
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
``` [1](#0-0) 

The five return values `(roundId, answer, startedAt, updatedAt, answeredInRound)` are fully decoded by the interface, but positions 3, 4, and 5 are thrown away. No maximum staleness window is enforced and no round-completeness check is performed.

By contrast, `ChainlinkOracleForRSETHPoolCollateral.getRate()` — used in the L2 pool path — does validate `answeredInRound < roundID` and `timestamp == 0`, demonstrating the protocol is aware of the pattern: [2](#0-1) 

The stale price propagates through `LRTOracle._getTotalEthInProtocol()`, which calls `getAssetPrice()` for every supported LST and sums their ETH values: [3](#0-2) 

The resulting `newRsETHPrice` is stored and used to determine how many rsETH tokens to mint per deposited asset: [4](#0-3) [5](#0-4) 

`updateRSETHPrice()` is a public, permissionless function with no access control: [6](#0-5) 

The existing `pricePercentageLimit` downside guard is insufficient: it defaults to `0` (disabled), and even when set, it only triggers a protocol pause for large price drops — it does not prevent the stale price from being committed and used for minting within the allowed band. [7](#0-6) 

## Impact Explanation

LSTs such as stETH and ETHx are monotonically appreciating — their ETH price only increases over time. When the Chainlink feed for such an asset is stale, the reported price is lower than the true current price. This causes `_getTotalEthInProtocol()` to underestimate the protocol's TVL, which lowers `newRsETHPrice`. A lower rsETH price means each unit of deposited asset mints more rsETH than it should. The excess rsETH dilutes all existing holders — their proportional claim on the underlying ETH is reduced. This constitutes **theft of unclaimed yield** from existing rsETH holders.

**Impact classification: High — Theft of unclaimed yield.**

## Likelihood Explanation

Chainlink price feeds become stale during network congestion (gas spikes preventing keeper updates), oracle node outages, or rapid price movements that outpace the heartbeat interval. For LST/ETH feeds, the heartbeat is typically 24 hours with a 0.5% deviation threshold — a stale feed is a realistic, historically observed condition. The attack requires no special privileges: any user can call `updateRSETHPrice()` followed by `depositAsset()` in the same transaction. The attack is repeatable whenever the feed is stale.

**Likelihood classification: Medium — requires an external condition (stale feed) that is realistic and has occurred on mainnet.**

## Recommendation

Add staleness and round-completeness checks to `ChainlinkPriceOracle.getAssetPrice()`, mirroring the pattern already used in `ChainlinkOracleForRSETHPoolCollateral`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);

    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
        priceFeed.latestRoundData();

    if (answeredInRound < roundId) revert StalePrice();
    if (updatedAt == 0) revert IncompleteRound();
    if (block.timestamp - updatedAt > MAX_STALENESS) revert StalePrice();
    if (price <= 0) revert InvalidPrice();

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

`MAX_STALENESS` should be set per-asset based on the Chainlink feed's documented heartbeat interval (e.g., 86 400 s + buffer for LST/ETH feeds).

## Proof of Concept

1. Chainlink's stETH/ETH feed becomes stale (`updatedAt` is 25 hours old; last reported price is 1.050 ETH/stETH; true current price is 1.052 ETH/stETH).
2. Attacker calls `LRTOracle.updateRSETHPrice()`. Inside `_updateRsETHPrice()`, `_getTotalEthInProtocol()` calls `ChainlinkPriceOracle.getAssetPrice(stETH)`, which returns the stale 1.050 value with no revert. The computed `newRsETHPrice` is lower than it should be (e.g., 1.049 ETH/rsETH instead of 1.051 ETH/rsETH). This value is stored in `rsETHPrice`.
3. Attacker calls `LRTDepositPool.depositAsset(stETH, amount, ...)`. The mint calculation uses the artificially low `rsETHPrice`, so the attacker receives more rsETH than the deposited stETH is worth at the true current rate.
4. When the Chainlink feed updates and `updateRSETHPrice()` is called again with the correct price, the attacker's excess rsETH now represents a larger-than-deserved claim on the protocol's TVL, at the expense of all prior holders.

**Foundry fork test outline:**
- Fork mainnet at a block where the stETH/ETH Chainlink feed is within its normal heartbeat.
- `vm.warp(block.timestamp + 25 hours)` to simulate staleness without the feed updating.
- Record `rsETHPrice` before and after calling `updateRSETHPrice()`.
- Assert that `rsETHPrice` is lower than the value computed using the live (non-stale) price.
- Call `depositAsset` and assert the minted rsETH amount exceeds what would be minted at the correct price. [1](#0-0) [6](#0-5) [8](#0-7)
