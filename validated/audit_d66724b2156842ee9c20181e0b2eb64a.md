### Title
Payable `submit_participant_info` Silently Absorbs Attached Deposit When Caller Is an Existing Participant Re-Submitting Attestation — (`File: crates/contract/src/lib.rs`)

---

### Summary

`submit_participant_info` is marked `#[payable]` and therefore accepts any attached NEAR deposit. However, the deposit-refund logic is gated behind a conditional that is `false` when an **existing participant** re-submits their attestation (e.g., during a TEE upgrade). In that path the function succeeds, the state is updated, and the entire attached deposit is silently absorbed by the contract with no refund issued.

---

### Finding Description

The function `submit_participant_info` in `crates/contract/src/lib.rs` is decorated with `#[payable]`, meaning callers may attach arbitrary NEAR tokens: [1](#0-0) 

After updating the TEE state, the code computes two boolean flags: [2](#0-1) 

The deposit-check-and-refund block is only entered when `attestation_storage_must_be_paid_by_caller` is `true`: [3](#0-2) 

When the condition evaluates to `false` — i.e., when `is_new_attestation == false` **AND** `caller_is_not_participant == false` (the caller is an existing participant doing a re-submission) — the entire block is skipped. No deposit is checked, and **no refund is issued**. Any NEAR attached to the call is permanently retained by the contract.

This is structurally identical to the WormholeFacet bug: a `payable` entry point accepts native tokens but contains a code path where those tokens are neither forwarded nor returned, causing them to be permanently locked.

---

### Impact Explanation

Any NEAR tokens attached by an existing participant when re-submitting their attestation are permanently frozen inside the MPC contract. The contract's balance grows by the attached amount, but the funds are unrecoverable — there is no withdrawal or sweep mechanism for such absorbed deposits. This breaks the production accounting invariant that deposits are only retained to cover storage costs and that any excess is refunded.

This maps to the **Medium** allowed impact: *"Balance, request-lifecycle, participant-state, or contract execution-flow manipulation that breaks production safety/accounting invariants."*

---

### Likelihood Explanation

The re-submission path is exercised routinely during TEE software upgrades, when every existing participant must call `submit_participant_info` again with a fresh attestation quote. The `#[payable]` attribute gives no indication to callers that attaching NEAR is harmful in this path. A participant following documentation or tooling that attaches a small deposit "just in case" (mirroring the pattern required by `sign` and `request_app_private_key`) will silently lose those funds. The trigger requires no special privilege beyond being a current participant.

---

### Recommendation

Add an unconditional refund of any attached deposit when `attestation_storage_must_be_paid_by_caller` is `false`. The simplest fix is to move the refund outside the conditional, or to add an `else` branch:

```rust
} else {
    // Caller is an existing participant re-submitting; no storage cost applies.
    // Refund the entire attached deposit.
    let attached = env::attached_deposit();
    if attached > NearToken::from_yoctonear(0) {
        Promise::new(account_id).transfer(attached).detach();
    }
}
```

Alternatively, remove `#[payable]` from `submit_participant_info` entirely and assert `env::attached_deposit() == NearToken::from_yoctonear(0)` at the top of the function, then handle storage payment through a separate dedicated flow. This mirrors the recommendation in the WormholeFacet report: remove the `payable` keyword if the deposit is not always needed.

---

### Proof of Concept

1. Participant `alice.near` is already registered in the MPC network (`voter_account()` succeeds → `caller_is_not_participant = false`).
2. A TEE upgrade is announced; `alice.near` must re-submit her attestation.
3. `alice.near` calls `submit_participant_info(new_attestation, tls_key)` and attaches `1 NEAR` (e.g., following tooling that mirrors the `sign` deposit pattern).
4. `add_participant` returns `ParticipantInsertion::UpdatedExistingParticipant` → `is_new_attestation = false`.
5. `attestation_storage_must_be_paid_by_caller = false || false = false`.
6. The `if` block is skipped; no refund promise is created.
7. The call returns `Ok(())`. `alice.near`'s balance is reduced by `1 NEAR`; the contract balance increases by `1 NEAR`. The funds are permanently locked. [4](#0-3)
