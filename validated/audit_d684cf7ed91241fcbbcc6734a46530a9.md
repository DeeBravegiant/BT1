Audit Report

## Title
`updatedAt` Advances Unconditionally When `totalKernelStaked == 0`, Permanently Freezing Unclaimed Yield — (File: `contracts/KERNEL/KernelDepositPool.sol`)

## Summary
The `updateReward` modifier unconditionally writes `updatedAt = lastTimeRewardApplicable()` even when `totalKernelStaked == 0`, while `rewardPerToken()` short-circuits and returns `rewardPerTokenStored` unchanged in that state. Any elapsed time during a zero-staking window is silently consumed: `rewardRate × elapsed` tokens are never credited to anyone and can never be recovered, because `updatedAt` has already advanced past that interval. The contract's own NatSpec (lines 14–22) acknowledges this behavior but relies entirely on off-chain operational discipline rather than on-chain enforcement.

## Finding Description
The `updateReward` modifier at lines 232–242 performs two unconditional state writes:

```solidity
rewardPerTokenStored = rewardPerToken();   // (A) no-op when totalKernelStaked == 0
updatedAt = lastTimeRewardApplicable();    // (B) always advances
```

`rewardPerToken()` at lines 408–414 returns `rewardPerTokenStored` unchanged when `totalKernelStaked == 0`, so step (A) distributes nothing. Step (B) still advances `updatedAt` to `block.timestamp` (or `finishAt`). The interval `[old_updatedAt, block.timestamp]` is consumed: the next `rewardPerToken()` call computes `(lastTimeRewardApplicable() - updatedAt)` starting from the new, advanced `updatedAt`, permanently skipping the zero-staking window.

`initiateWithdrawal` at lines 320–338 is itself guarded by `updateReward` and decrements `totalKernelStaked` (line 326), which can reach zero when the last staker withdraws. After that, every subsequent call to any `updateReward`-gated function (`stake`, `initiateWithdrawal`, `getReward`, `notifyRewardAmount`) silently advances `updatedAt` without distributing rewards.

`notifyRewardAmount` (line 570) guards against *starting* a reward period with zero stakers, but provides no protection against stakers withdrawing *during* an active reward period, which is the actual attack surface.

## Impact Explanation
**Medium — Permanent freezing of unclaimed yield.** Reward tokens corresponding to any zero-staking interval are permanently locked inside the contract. There is no admin recovery function, no sweep mechanism, and no way to re-distribute those tokens. The `updatedAt` cursor is monotonically advancing and cannot be rewound. This matches the allowed impact "Medium. Permanent freezing of unclaimed yield."

## Likelihood Explanation
Any staker who holds the entire staked supply can reduce `totalKernelStaked` to zero by calling `initiateWithdrawal` for their full balance — a normal, permissionless user action. This can occur organically (all stakers exit) or deliberately. Once at zero, a low-cost griefing loop is available to any address holding KERNEL tokens: `stake(1 wei)` then `initiateWithdrawal(1 wei)` across successive blocks advances `updatedAt` toward `finishAt`, destroying the remaining reward budget. No privileged role is required after the reward period has started.

## Recommendation
Guard the `updatedAt` write so it only advances when tokens are actually staked:

```solidity
modifier updateReward(address _account) {
    rewardPerTokenStored = rewardPerToken();
    if (totalKernelStaked > 0) {
        updatedAt = lastTimeRewardApplicable();
    }
    if (_account != address(0)) {
        rewards[_account] = earned(_account);
        userRewardPerTokenPaid[_account] = rewardPerTokenStored;
    }
    _;
}
```

This ensures `updatedAt` only advances when there are stakers to absorb rewards, preventing silent time-window consumption.

## Proof of Concept
```
Setup
─────
• rewardRate  = 1e18 tokens/second
• duration    = 1000 seconds
• Alice stakes 100 KERNEL at T=0
• Admin calls notifyRewardAmount() at T=0
  → finishAt = 1000, updatedAt = 0

Step 1 — Alice withdraws at T=100
  initiateWithdrawal(100) triggers updateReward(Alice):
    rewardPerToken() = 1e18  →  rewardPerTokenStored = 1e18
    updatedAt = 100
    rewards[Alice] = 100 tokens  ✓
  totalKernelStaked = 0

Step 2 — Bob stakes 1 wei at T=200
  stake(1) triggers updateReward(Bob):
    totalKernelStaked == 0  →  rewardPerToken() returns 1e18 (no change)
    updatedAt = 200  ← silently consumes T=100..200 (100 tokens locked)
    rewards[Bob] = 0

Step 3 — Bob claims at T=300
  getReward() triggers updateReward(Bob):
    rewardPerToken() = 1e18 + (1e18 * 100 * 1e18) / 1 = 101e18
    rewards[Bob] = 100 tokens  (only T=200..300 credited)

Result
──────
• Alice earned 100 tokens (T=0..100)    ✓
• Bob earned   100 tokens (T=200..300)  ✓
• 100 tokens for T=100..200 permanently locked — updatedAt jumped
  from 100→200 with totalKernelStaked==0

Foundry invariant test plan:
  invariant: sum of all earned() + sum of all rewards[] + rewardsToken.balanceOf(contract)
             == initial reward deposit at all times
  This invariant will break as soon as any zero-staking window is consumed.
```
