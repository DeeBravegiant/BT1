### Title
Single-Participant Forgery of Legacy CKD Response Bypasses Threshold Consensus — (`File: crates/contract/src/lib.rs`)

---

### Summary

The `respond_ckd` contract method applies a cryptographic validity check on the `(big_y, big_c)` output **only** for the `AppPublicKeyPV` (publicly verifiable) variant. For the legacy `AppPublicKey` variant — which is still the documented default and actively used — **no cryptographic verification is performed**. A single attested participant can call `respond_ckd` with an arbitrary forged `(big_y, big_c)` pair, and the contract will accept and deliver it to the user as the legitimate threshold CKD output, completely bypassing the MPC threshold requirement.

---

### Finding Description

In `crates/contract/src/lib.rs`, the `respond_ckd` function contains the following branch:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}   // ← NO VERIFICATION
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For the `AppPublicKeyPV` variant, `ckd_output_check` verifies the BLS pairing equation `e(big_c, g2) = e(big_y, app_pk2) · e(H(pk, app_id), mpc_pk)`, which cryptographically proves the output is a valid threshold computation over the MPC master secret key. [2](#0-1) 

For the `AppPublicKey` (legacy) variant, the match arm is **empty** — the response `(big_y, big_c)` is accepted unconditionally and immediately passed to `resolve_yields_for`, which delivers it to the waiting user. [3](#0-2) 

The only gate before this point is `assert_caller_is_attested_participant_and_protocol_active`, which verifies the caller is a single attested participant — not that a threshold of participants agreed on the output. [4](#0-3) 

The `AppPublicKey` variant is the documented legacy default format, explicitly described in the README and still accepted by the contract: [5](#0-4) 

---

### Impact Explanation

The CKD protocol is designed so that the user decrypts the output `(big_y, big_c)` using their private scalar `a` (where `app_public_key = a·G1`) to recover the MPC-derived BLS signature `sig = big_c + (-a)·big_y = msk·H(pk, app_id)`. This signature is then used to derive the confidential app-specific private key via HKDF.

A forging attacker who controls one attested participant node can:

1. Set `big_y = 0` (the G1 identity point) and `big_c = P` for any chosen point `P`. The user then computes `sig = P + (-a)·0 = P`. Since the attacker chose `P`, they know the resulting `sig` and therefore the derived key `s = HKDF(P)`.
2. Alternatively, set both to arbitrary garbage, causing the user's app to silently derive a wrong key with no on-chain indication of failure.

This constitutes **unauthorized confidential key derivation output without the required participant authorization**: a single Byzantine participant below the signing threshold can unilaterally determine the confidential key delivered to any user who submits a legacy `AppPublicKey` CKD request, completely breaking the threshold and confidentiality guarantees of the CKD protocol.

---

### Likelihood Explanation

- The `AppPublicKey` (legacy) variant is still the primary documented format and is used in production (e.g., the e2e test `ckd_response__passes_cryptographic_verification` uses it).
- Any single attested participant — a Byzantine node strictly below the signing threshold — can exploit this. No collusion is required.
- The attacker only needs to race the honest coordinator's `respond_ckd` call. Since the contract resolves the yield on the **first** valid `respond_ckd` call and removes the pending request, the attacker who submits first wins.
- The attack is silent: the user receives a response that appears structurally valid.

---

### Recommendation

Apply the same `ckd_output_check` pairing verification to the `AppPublicKey` variant. Since the legacy variant provides only a single G1 point `pk1` (not `pk2`), the existing `ckd_output_check` cannot be used directly. The fix requires one of:

1. **Deprecate and remove** the `AppPublicKey` legacy variant, requiring all callers to use `AppPublicKeyPV`.
2. **Add an equivalent check** for the legacy variant. Because `pk2` is unavailable, the contract cannot verify the pairing equation on-chain. The alternative is to require the responding node to include a BLS proof-of-knowledge or to require the coordinator to submit a threshold-aggregated proof alongside the response.

The `AppPublicKeyPV` path is already secure. The gap is exclusively in the `AppPublicKey` arm.

---

### Proof of Concept

1. User submits `request_app_private_key` with `app_public_key = AppPublicKey(a·G1)` for some private scalar `a`.
2. Attacker (one attested participant) calls `respond_ckd` with:
   - `request` matching the pending request key
   - `response = CKDResponse { big_y: G1_IDENTITY, big_c: ATTACKER_CHOSEN_POINT }`
3. The contract's `respond_ckd` hits the `AppPublicKey(_) => {}` arm — no check — and calls `resolve_yields_for`, delivering the forged response to the user.
4. The user computes `sig = ATTACKER_CHOSEN_POINT + (-a)·0 = ATTACKER_CHOSEN_POINT`.
5. The attacker, who chose `ATTACKER_CHOSEN_POINT`, knows `sig` and therefore `s = HKDF(sig)` — the user's "confidential" app private key.

The existing unit test `respond_ckd__should_succeed_when_response_is_valid_and_request_exists` inadvertently demonstrates this: it passes `big_y = [1u8; 48]` and `big_c = [2u8; 48]` (not a valid CKD output) with an `AppPublicKey` request, and the contract accepts it without complaint. [6](#0-5)

/// Check that `e(big_c, g2) = e(big_y, app_pk2) . e(hash_point, public_key)`.
///
/// Point validation is fully delegated to the host, as in
/// [`app_public_key_check`].
pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}

```markdown
- `app_public_key`: the ephemeral public key for the CKD request. Two formats are supported:
  - **Privately verifiable** (legacy): a single G1 point, e.g. `"bls12381g1:<base58>"` or `{"AppPublicKey": "bls12381g1:<base58>"}`.
  - **Publicly verifiable**: a pair of points `(pk1, pk2) = (a·G1, a·G2)`, passed as `{"AppPublicKeyPV": {"pk1": "bls12381g1:<base58>", "pk2": "bls12381g2:<base58>"}}`. This allows anyone to verify the encrypted result on-chain without the app's secret key.
- `domain_id` (integer): identifies the master key to use for deriving the ckd, and must correspond to bls12381.
