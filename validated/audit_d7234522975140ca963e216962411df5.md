Audit Report

## Title
Stale `ethValueInWithdrawal` Snapshot Causes Persistent rsETH Price Mispricing After Oracle Update — (`contracts/LRTConverter.sol`)

## Summary

`LRTConverter.ethValueInWithdrawal` is recorded at the oracle price prevailing at the time of `transferAssetFromDepositPool()` and is never revalued when the oracle price changes. `LRTDepositPool.getETHDistributionData()` reads this stale figure verbatim as `ethLyingInConverter`, which feeds into `LRTOracle._getTotalEthInProtocol()` and therefore into `rsETHPrice`. Any oracle price movement after the transfer creates a persistent divergence: when stETH yield accrues and the oracle price rises, `rsETHPrice` is understated, causing new depositors to receive more rsETH than the true backing warrants and diluting existing holders.

## Finding Description

**Step 1 — Snapshot written at transfer time.**
In `LRTConverter.transferAssetFromDepositPool()`, the ETH-equivalent value of the transferred LST is recorded at the current oracle price:

```solidity
// LRTConverter.sol L140
ethValueInWithdrawal += (_amount * lrtOracle.getAssetPrice(_asset)) / 1e18;
IERC20(_asset).safeTransferFrom(lrtDepositPoolAddress, address(this), _amount);
```

This value is never refreshed in place. The only mutations are additive (another transfer in), subtractive at current price (transfer back), or subtractive by actual ETH amount (ETH claim). None revalue the existing balance.

**Step 2 — Stale value read verbatim by deposit pool.**
`LRTDepositPool.getETHDistributionData()` reads `ethValueInWithdrawal` directly:

```solidity
// LRTDepositPool.sol L498-499
address lrtConverter = lrtConfig.getContract(LRTConstants.LRT_CONVERTER);
ethLyingInConverter = ILRTConverter(lrtConverter).ethValueInWithdrawal();
```

For non-ETH assets, `assetLyingInConverter` is explicitly set to `0` (L460), meaning all converter holdings are accounted for exclusively through this stale ETH snapshot.

**Step 3 — Stale value propagates into rsETH price.**
`LRTOracle._getTotalEthInProtocol()` calls `getTotalAssetDeposits(asset)` for every supported asset (L341), which for ETH calls `getETHDistributionData()` and includes the stale `ethLyingInConverter`. The result is used to compute `newRsETHPrice` (L250):

```solidity
// LRTOracle.sol L250
uint256 newRsETHPrice = (totalETHInProtocol - protocolFeeInETH).divWad(rsethSupply);
```

**Step 4 — Public trigger.**
`updateRSETHPrice()` is unrestricted (`public whenNotPaused`, L87). Any caller can commit the stale `totalETHInProtocol` into `rsETHPrice`. Deposits then use this stale price:

```solidity
// LRTDepositPool.sol L520
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

A lower `rsETHPrice` means more rsETH minted per unit deposited.

**Existing guards are insufficient.** The `pricePercentageLimit` downside guard only triggers a pause if the price drop exceeds the configured threshold; for continuous small stETH yield accrual (~0.01%/day), this threshold is never reached. The `highestRsethPrice` ceiling guard only applies to upward price movements.

## Impact Explanation

**Allowed impact: Low — Contract fails to deliver promised returns, but doesn't lose value.**

When stETH oracle price rises after a transfer to the converter (the chronic case, since stETH continuously accrues yield), `ethValueInWithdrawal` understates the true ETH value of assets held in the converter. This understates `rsETHPrice`. New depositors receive more rsETH than the true backing warrants, diluting the claims of existing rsETH holders. The dilution is proportional to the converter's share of TVL multiplied by the oracle price drift since the last asset movement. No funds leave the protocol, but existing holders receive less yield than they are owed.

## Likelihood Explanation

No attacker action is required. The preconditions are:
1. An operator calls `transferAssetFromDepositPool()` — a routine, expected operation.
2. The stETH oracle price increases — guaranteed over any non-trivial time window due to yield accrual.
3. Any caller (including the affected depositor) calls `updateRSETHPrice()` — a public, permissionless function.

All three conditions are met in normal protocol operation. The divergence is continuous and observable by any caller of `getETHDistributionData()` or `getTotalAssetDeposits(LRTConstants.ETH_TOKEN)`.

## Recommendation

Replace the stored snapshot with a live revaluation. The preferred fix is to remove `ethValueInWithdrawal` and replace the `getETHDistributionData()` read with a view function on `LRTConverter` that iterates over held LST balances and prices them at the current oracle rate:

```solidity
// In LRTConverter (new view function)
function getLiveEthValueInWithdrawal() external view returns (uint256 liveValue) {
    address[] memory supportedAssets = lrtConfig.getSupportedAssetList();
    ILRTOracle lrtOracle = ILRTOracle(lrtConfig.getContract(LRTConstants.LRT_ORACLE));
    for (uint256 i; i < supportedAssets.length; ++i) {
        address asset = supportedAssets[i];
        if (asset == LRTConstants.ETH_TOKEN) continue;
        uint256 balance = IERC20(asset).balanceOf(address(this));
        liveValue += (balance * lrtOracle.getAssetPrice(asset)) / 1e18;
    }
    liveValue += address(this).balance; // include any ETH pending send
}
```

Then update `getETHDistributionData()` to call `getLiveEthValueInWithdrawal()` instead of reading `ethValueInWithdrawal`. This eliminates the stale snapshot entirely.
