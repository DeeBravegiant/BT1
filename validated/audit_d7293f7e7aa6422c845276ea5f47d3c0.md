### Title
Missing Cryptographic Output Verification in `respond_ckd` for Legacy `AppPublicKey` Variant Allows Single Byzantine Participant to Deliver Fabricated CKD Response - (File: `crates/contract/src/lib.rs`)

### Summary

`respond_ckd` skips all cryptographic verification of the CKD response when the request uses the legacy `AppPublicKey` variant. A single Byzantine attested participant (strictly below the signing threshold) can call `respond_ckd` with an arbitrary fabricated `CKDResponse`, which the contract accepts and delivers to the waiting user without any check. This breaks the production safety invariant that CKD outputs must reflect genuine threshold computation.

### Finding Description

The three `respond*` functions in the Node API implement different levels of output verification:

- `respond` (ECDSA/EdDSA signatures): always verifies the submitted signature cryptographically against the derived public key before resolving the pending yield.
- `respond_verify_foreign_tx`: always verifies the submitted signature against the root public key.
- `respond_ckd`: **conditionally** verifies — only when the request uses `AppPublicKeyPV`. When the request uses the legacy `AppPublicKey` variant, the verification arm is an empty no-op `{}`. [1](#0-0) 

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← no check at all
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
```

After this match, `resolve_yields_for` is called unconditionally, resuming the user's yield-resume promise with whatever `response` was supplied. [2](#0-1) 

By contrast, `respond` always performs cryptographic verification before resolving: [3](#0-2) 

The `AppPublicKey` variant is still a supported, documented production path: [4](#0-3) 

### Impact Explanation

A single Byzantine attested participant (one node, below the signing threshold) can:

1. Observe a pending `request_app_private_key` call that uses the legacy `AppPublicKey` variant.
2. Immediately call `respond_ckd` with arbitrary `big_y` / `big_c` BLS12-381 G1 points — no valid threshold computation required.
3. The contract accepts the fabricated response, removes the pending request from state, and delivers the wrong encrypted key to the user.

The user receives a confidential key that does not correspond to any genuine MPC-derived secret. If the user has already committed funds to an address derived from the expected key (e.g., a Bitcoin or Ethereum address), those funds become permanently inaccessible. This breaks the production safety invariant that CKD outputs must be the product of genuine threshold computation.

This maps to the allowed impact: **Medium — request-lifecycle manipulation that breaks production safety/accounting invariants without relying on network-level DoS or operator misconfiguration.**

### Likelihood Explanation

- The `AppPublicKey` (legacy) variant is still accepted by the contract and documented in the README.
- Only **one** attested participant needs to act maliciously — strictly below the signing threshold.
- The attack is a single on-chain transaction with no off-chain coordination required.
- The attacker only needs to race the honest nodes to call `respond_ckd` first; once the yield is resolved, the pending request is removed and cannot be retried under the same request key. [5](#0-4) 

### Recommendation

Apply the same cryptographic output verification to the `AppPublicKey` variant that is already applied to `AppPublicKeyPV`. If on-chain verification is not possible for the legacy variant (because it requires the user's private key to decrypt), the contract should either:

1. Deprecate and reject the `AppPublicKey` variant entirely, requiring all new requests to use `AppPublicKeyPV` (which is publicly verifiable on-chain), or
2. Require that `respond_ckd` for `AppPublicKey` requests be accompanied by a threshold-signed attestation (e.g., a multi-sig proof from ≥ threshold participants) rather than accepting a single participant's unverified response.

The fix mirrors the remediation in the referenced report: add the missing check analogous to what `AppPublicKeyPV` already does.

### Proof of Concept

1. User calls `request_app_private_key` with `AppPublicKey` (legacy G1 point) and `domain_id` pointing to a BLS12-381 CKD domain. A yield-resume promise is created and stored in `pending_ckd_requests`.
2. A single Byzantine attested participant calls `respond_ckd` with the correct `CKDRequest` (observable from chain state) and a fabricated `CKDResponse { big_y: [0u8; 48], big_c: [0u8; 48] }`.
3. The contract passes `assert_caller_is_attested_participant_and_protocol_active()`, enters the `AppPublicKey(_) => {}` arm (no check), and calls `resolve_yields_for`, resuming the user's promise with the fabricated bytes.
4. The user's transaction completes with a wrong confidential key. Any funds committed to an address derived from the expected key are permanently inaccessible. [6](#0-5)

    #[handle_result]
    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }
