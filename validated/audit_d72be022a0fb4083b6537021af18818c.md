### Title
Caller Deposit Permanently Lost in `WalletContract` When Intermediate Callbacks Return Early Without Refund - (File: runtime/near-wallet-contract/implementation/wallet-contract/src/lib.rs)

### Summary
The `WalletContract` (ETH implicit account wallet) tracks an external caller's attached deposit via `CallerDeposit` and is supposed to refund it if the cross-contract call fails. However, in `nep_141_storage_balance_callback` and `address_check_callback`, several early-return error paths drop the `caller_deposit` without issuing a refund transfer, permanently absorbing the caller's NEAR into the wallet contract's balance with no recovery path.

### Finding Description

`CallerDeposit` is constructed in `inner_rlp_execute` to record any non-zero deposit attached by an external (non-self) caller: [1](#0-0) 

It is passed through the promise chain and is only refunded in `rlp_execute_callback` when `PromiseResult::Failed`: [2](#0-1) 

However, `nep_141_storage_balance_callback` has two early-return paths that silently drop `caller_deposit` without refunding:

1. When `storage_balance_of` returns `PromiseResult::Failed` (lines 204–209)
2. When the successful response cannot be deserialized (lines 213–219) [3](#0-2) 

Similarly, `address_check_callback` has three early-return paths that drop `caller_deposit`:

1. When the registrar `lookup` call returns `PromiseResult::Failed` (lines 142–148)
2. When the response cannot be deserialized (lines 151–158)
3. When `maybe_account_id.is_some()` and the signer is not the wallet owner (lines 167–173) [4](#0-3) 

In all these paths, `self.has_in_flight_tx` is reset to `false` and a `Value(ExecuteResponse { success: false, ... })` is returned — but no transfer promise is created to return the caller's deposit. The deposit, already credited to the wallet contract's account balance at the time `rlp_execute` was called, is permanently retained by the wallet contract.

### Impact Explanation

The corrupted protocol value is the **caller's NEAR balance**: it is permanently reduced by the attached deposit amount, with no on-chain recovery mechanism. The wallet contract's balance increases by the same amount. The caller (an unprivileged external account) has no way to reclaim the funds — there is no `withdraw` or `recover_deposit` method on `WalletContract`.

### Likelihood Explanation

The `nep_141_storage_balance_callback` path is reachable whenever:
- An external caller attaches a non-zero deposit to `rlp_execute` for an ERC-20 transfer
- The target NEP-141 token contract's `storage_balance_of` panics, runs out of gas, or returns non-JSON data

A malicious token contract can deliberately trigger this to steal the caller's deposit. Even a legitimate but buggy token contract can cause accidental loss. The `address_check_callback` path is reachable when the address registrar is unavailable or returns unexpected data.

### Recommendation

All early-return error paths in `nep_141_storage_balance_callback` and `address_check_callback` must issue a refund transfer to `caller_deposit.account_id` before returning, mirroring the pattern already used in `rlp_execute_callback`:

```rust
if let Some(CallerDeposit { account_id, yocto_near }) = caller_deposit {
    let refund_promise = env::promise_batch_create(&account_id);
    env::promise_batch_action_transfer(
        refund_promise,
        NearToken::from_yoctonear(yocto_near.into()),
    );
}
```

This pattern is already correctly applied in `rlp_execute_callback` but is missing from the intermediate callbacks.

### Proof of Concept

1. Deploy a malicious NEP-141 token contract whose `storage_balance_of` method always panics.
2. An external caller (e.g., a relayer) calls `rlp_execute` on a victim's ETH implicit account wallet, attaching 5 NEAR as deposit, targeting the malicious token contract for an ERC-20 transfer.
3. `inner_rlp_execute` creates `CallerDeposit { account_id: caller, yocto_near: 5e24 }` and schedules `storage_balance_of` → `nep_141_storage_balance_callback`.
4. The malicious token's `storage_balance_of` panics → `PromiseResult::Failed`.
5. `nep_141_storage_balance_callback` hits the early return at line 204–209, returning `PromiseOrValue::Value(ExecuteResponse { success: false, ... })` without any refund transfer.
6. The caller's 5 NEAR is permanently absorbed into the wallet contract's balance. The caller's account balance is reduced by 5 NEAR with no recovery path. [5](#0-4) [6](#0-5)
