Audit Report

## Title
Missing Withdrawal Cancellation Mechanism Permanently Freezes User rsETH When Vault Assets Are Insufficient — (`contracts/LRTWithdrawalManager.sol`)

## Summary

`LRTWithdrawalManager.initiateWithdrawal` irrevocably transfers a user's rsETH into the contract and increments `assetsCommitted[asset]`. If the `LRTUnstakingVault` never accumulates enough of the requested asset to satisfy the queued payout — a scenario the contract's own NatSpec acknowledges as possible due to EigenLayer slashing — the user's rsETH is permanently locked with no on-chain recovery path. No `cancelWithdrawal`, admin escape hatch, or timeout-based refund function exists anywhere in the contract.

## Finding Description

**Step 1 — rsETH is surrendered on `initiateWithdrawal`.** [1](#0-0) 

The user's rsETH is pulled into the contract and `assetsCommitted[asset]` is incremented by the oracle-priced expected asset amount. The NatSpec on this function explicitly acknowledges the slashing edge case but provides no remedy: [2](#0-1) 

**Step 2 — Unlock is gated on vault balance.**
`unlockQueue` derives `totalAvailableAssets` from `unstakingVault.balanceOf(asset)` (the vault's actual token balance), not from `getAvailableAssetAmount`: [3](#0-2) 

Inside `_unlockWithdrawalRequests`, if the vault balance is less than the computed `payoutAmount`, the loop breaks and `nextLockedNonce[asset]` is not advanced: [4](#0-3) 

`assetsCommitted[asset]` is only decremented for successfully unlocked requests: [5](#0-4) 

**Step 3 — Stuck `assetsCommitted` blocks all future withdrawals.**
`getAvailableAssetAmount` computes `totalAssets - assetsCommitted[asset]`, returning 0 when the committed amount equals or exceeds total deposits: [6](#0-5) 

Every new `initiateWithdrawal` call reverts if `expectedAssetAmount > getAvailableAssetAmount(asset)`, so the stuck committed amount cascades to block all subsequent withdrawers for that asset.

**Step 4 — No recovery path exists.**
A search of the entire contract confirms there is no `cancelWithdrawal`, no admin function to forcibly return rsETH, and no timeout-based escape hatch. The rsETH held in the contract can only leave via `unlockQueue` → `burnFrom` (for successfully unlocked requests) or via a future contract upgrade.

## Impact Explanation

**Critical — Permanent freezing of funds.**

A user who calls `initiateWithdrawal` surrenders their rsETH to the contract. If the `LRTUnstakingVault` never holds enough of the requested asset to satisfy the queued payout, the rsETH remains locked indefinitely. Because `assetsCommitted[asset]` is never decremented for the stuck request, subsequent users also find `getAvailableAssetAmount` returning zero, compounding the freeze across the entire withdrawal queue for that asset. There is no on-chain mechanism for the user or any admin to recover the locked rsETH.

## Likelihood Explanation

EigenLayer slashing is a documented, first-class protocol risk — not a theoretical edge case. A partial slash of a NodeDelegator's restaked position directly reduces `getTotalAssetDeposits(asset)` while leaving `assetsCommitted[asset]` unchanged, immediately making `getAvailableAssetAmount` return zero. Any user who had already called `initiateWithdrawal` before the slash event is placed in a state where their withdrawal can never be unlocked unless the vault is recapitalized to the exact required amount — and no such recapitalization mechanism exists in the contract. The scenario is reachable by any ordinary withdrawer without any privileged action; the user only needs to call the public `initiateWithdrawal` function before a slashing event occurs.

## Recommendation

1. **Add a `cancelWithdrawal` function** that allows a user to reclaim their rsETH after a configurable timeout (e.g., after `withdrawalDelayBlocks` have elapsed and the request is still locked at `nextLockedNonce[asset]`), decrementing `assetsCommitted[asset]` accordingly and returning the rsETH to the user.
2. **Add an admin/operator escape hatch** to forcibly cancel a specific locked request and return rsETH to the user, for use in emergency scenarios such as post-slashing recovery.
3. **Emit a dedicated event** inside `_unlockWithdrawalRequests` when a request cannot be unlocked due to insufficient assets, enabling off-chain monitoring to alert operators promptly.

## Proof of Concept

1. User calls `initiateWithdrawal(stETH, rsETHAmount)`. rsETH is transferred to `LRTWithdrawalManager`; `assetsCommitted[stETH] += expectedAssetAmount`.
2. A NodeDelegator is slashed on EigenLayer; `getTotalAssetDeposits(stETH)` drops below `assetsCommitted[stETH]`. `getAvailableAssetAmount(stETH)` now returns `0`.
3. The `LRTUnstakingVault` accumulates some stETH from partial unstaking, but less than `payoutAmount` for the queued request.
4. Operator calls `unlockQueue(stETH, ...)`. Inside `_unlockWithdrawalRequests`, `availableAssetAmount < payoutAmount` → `break`. `nextLockedNonce[asset]` is not advanced; the request remains locked.
5. `getAvailableAssetAmount(stETH)` returns `0`. No new withdrawal can be initiated for stETH.
6. The user has no function to call to recover their rsETH. The rsETH is permanently frozen in `LRTWithdrawalManager`.

**Foundry test plan:** Deploy `LRTWithdrawalManager` and `LRTUnstakingVault` on a mainnet fork. Call `initiateWithdrawal` for a user. Mock `getTotalAssetDeposits` to return a value below `assetsCommitted[asset]` (simulating slashing). Call `unlockQueue` with a vault balance less than `payoutAmount`. Assert `nextLockedNonce[asset]` is unchanged and the user's rsETH balance in the contract is non-zero. Assert no public function exists that allows the user to recover their rsETH.
