### Title
Deposit Refund in Failed Meta-Transaction Inner Receipt Goes to Delegate Sender (Alice) Instead of Relayer Who Paid - (`runtime/runtime/src/actions.rs`)

### Summary
When a meta-transaction's inner action receipt fails on the receiver's shard, the attached deposit is refunded to the delegate sender (`Alice`), not to the relayer who actually paid for it. An unprivileged user can deliberately craft a `DelegateAction` with a large deposit to a function call that will fail, causing the relayer to lose the deposit while the user gains it.

### Finding Description

In `apply_delegate_action`, the inner receipt spawned from a `DelegateAction` is constructed with `predecessor_id: sender_id.clone()` — i.e., Alice's account ID — not the relayer's: [1](#0-0) 

The deposit refund logic in `generate_refund_receipts` sends the deposit refund to `receipt.balance_refund_receiver()`, which resolves to the receipt's `predecessor_id`: [2](#0-1) 

The relayer paid for the deposit (it is deducted from the relayer's balance when the outer transaction is processed), but when the inner receipt fails on Bob's shard, the deposit is refunded to Alice (the predecessor of the inner receipt), not the relayer. The code comment in `apply_delegate_action` acknowledges this explicitly: [3](#0-2) 

The protocol documentation also acknowledges this as a known limitation with a concrete financial incentive for abuse: [4](#0-3) 

### Impact Explanation

The relayer's NEAR balance is reduced by the full deposit amount, and Alice's balance is increased by the same amount. This is a concrete, on-chain balance transfer from the relayer to Alice. The deposit amount is bounded only by the relayer's balance and the relayer's willingness to submit the transaction. The corrupted protocol value is the `amount` field of the balance refund receipt sent to Alice instead of the relayer.

### Likelihood Explanation

Alice can craft a `DelegateAction` that passes all pre-submission checks (valid signature, valid nonce, valid receiver) but targets a function call that will fail at execution time on Bob's shard (e.g., calling a non-existent method, or relying on contract state that Alice controls and can change after the relayer validates but before execution). The relayer cannot always predict execution failure. This is a realistic, low-cost attack requiring only a signed `DelegateAction` from an unprivileged user.

### Recommendation

When constructing the inner receipt in `apply_delegate_action`, set a `refund_to` field pointing to the relayer (the `action_receipt.signer_id()`) so that deposit refunds from failed inner receipts are returned to the party who paid. The `ActionReceiptV2` struct already has a `refund_to: Option<AccountId>` field for this purpose: [5](#0-4) 

The fix would be to emit a `Receipt::V2` (or equivalent) with `refund_to: Some(relayer_id)` where `relayer_id = action_receipt.signer_id()`.

### Proof of Concept

1. Alice creates a `DelegateAction` with `receiver_id = bob.near`, inner action = `FunctionCall { method_name: "nonexistent", deposit: 100 NEAR, ... }`, signed with her key.
2. Alice sends the `SignedDelegateAction` to a relayer off-chain.
3. The relayer wraps it in a transaction (relayer is signer, pays 100 NEAR deposit + gas), submits to chain.
4. On Alice's shard, `apply_delegate_action` creates an inner receipt with `predecessor_id = alice.near`, `signer_id = relayer`.
5. On Bob's shard, the function call fails (method does not exist). `generate_refund_receipts` emits a balance refund receipt to `alice.near` (the predecessor) for 100 NEAR.
6. Alice receives 100 NEAR; the relayer loses 100 NEAR.

The relayer paid for the deposit but Alice receives the refund — the same broken invariant as the Astaria report where the liquidation initiator received the NFT collateral instead of the lenders who provided the funds.

```markdown
## Balance refunds in meta transactions

Unlike gas refunds, the protocol sends balance refunds to the predecessor
(a.k.a. sender) of the receipt. This makes sense, as we deposit the attached
balance to the receiver, who has to explicitly reattach a new balance to new
receipts they might spawn.

In the world of meta transactions, this assumption is also challenged. If an
inner action requires an attached balance (for example a transfer action) then
this balance is taken from the relayer.

The relayer can see what the cost will be before submitting the meta transaction
and agrees to pay for it, so nothing wrong so far. But what if the transaction
fails execution on Bob's shard? At this point, the predecessor is `Alice` and
therefore she receives the token balance refunded, not the relayer. This is
something relayer implementations must be aware of since there is a financial
incentive for Alice to submit meta transactions that have high balances attached
but will fail on Bob's shard.
```
