Audit Report

## Title
Exact-input token1→token0 swap silently consumes entire trader input and returns zero token0 when bin `token0BalanceScaled` is dust-small — (`metric-core/contracts/libraries/SwapMath.sol`)

## Summary

In `buyToken0InBinSpecifiedIn`, when a bin's `token0BalanceScaled` (T) is small enough that `T × amountSpecifiedRemainingScaled < MAX_POS_BIN`, the analytical target-position refinement path sets `totalIn1Scaled = amountSpecifiedRemainingScaled` while leaving `out0Scaled = 0`. The swap loop then deducts the trader's entire input from `state.amountSpecifiedRemainingScaled` and credits zero to `state.amountCalculatedScaled`, so the trader pays in full and receives nothing.

## Finding Description

**Entry condition:** The `else` branch at line 579 is taken when `totalIn1Scaled > state.amountSpecifiedRemainingScaled` — i.e., the cost to drain the entire bin exceeds the trader's input. This is reachable when a bin has a dust `token0BalanceScaled` (T = 1) but a wide price range where `avgPriceX64 > I × 2^64` while `lowerPriceX64 ≤ I × 2^64` (so the minimum-input guard at line 550 does not fire).

**Arithmetic trace** (currBinPos = 0, T = 1, I = 1e18, MAX_POS_BIN = `type(uint104).max` ≈ 2^104):

**1.** `computeAnalyticalTargetPosForBuyToken0` returns `targetPos = 1` because the quadratic rounds `deltaPos` to 0 and the minimum-progress guard fires: [1](#0-0) 

**2.** `out0Scaled` at line 591 rounds to zero:
`(T × 1) / MAX_POS_BIN = 1 / 2^104 = 0` [2](#0-1) 

**3.** `totalIn1Scaled = 0` (ceilDiv of 0 is 0). The first refinement block at line 598 fires because `0 < I` and `targetPos (1) < maxFinalBinPos`: [3](#0-2) 

- `totalIn1Scaled` is set to 1 (line 599)
- `delta = 1`, `scaledDelta = ceilDiv(1 × 1e18, 1) = 1e18`
- `scaledTarget = 1e18 < MAX_POS_BIN`, so `targetPos = 1e18`

**4.** `out0Scaled` is recomputed at line 612: `(1 × 1e18) / MAX_POS_BIN = 0` (since 1e18 < 2^104). `totalIn1Scaled = 0` again. [4](#0-3) 

**5.** Second guard at line 620 fires (`0 < I` and `targetPos (1e18) < maxFinalBinPos`): sets `totalIn1Scaled = I = 1e18`. [5](#0-4) 

**6.** Line 624 does NOT fire (`totalIn1Scaled == amountSpecifiedRemainingScaled`, not `>`). Final accounting: [6](#0-5) 

`state.amountSpecifiedRemainingScaled -= 1e18 → 0`; `state.amountCalculatedScaled += 0 → 0`.

**Existing guards are insufficient:**

- The `nonEmptyBin` guard only skips bins where `token0BalanceScaled == 0`; a bin with T = 1 passes it. [7](#0-6) 

- The minimum-input guard at line 550 checks `(I << 64) < startingPriceX64` (a price-floor check against the *starting* price), not against the average price. A bin where `lowerPriceX64 ≤ I × 2^64 < avgPriceX64` bypasses this guard entirely. [8](#0-7) 

The swap loop terminates because `amountSpecifiedRemainingScaled = 0`, so no further bins are visited, and the pool returns `(amountInScaled, 0, protocolFeeAmountScaled)`. [9](#0-8) 

## Impact Explanation

Direct loss of user principal. The trader's entire token1 input is consumed by the pool; the LP in the affected bin receives token1 for free with no token0 leaving the bin. The corrupted value is `state.amountCalculatedScaled` (remains 0 while `amountSpecifiedRemainingScaled` is fully decremented). This satisfies the contest's Critical/High direct loss of user principal gate.

## Likelihood Explanation

The condition requires: (a) a bin with dust `token0BalanceScaled` (T such that `T × I < MAX_POS_BIN`), and (b) a wide enough price range that `avgPriceX64 > I × 2^64` while `lowerPriceX64 ≤ I × 2^64`. Both conditions are reachable without privileged access: an attacker can add a tiny amount of liquidity to a wide-range bin at a high price tier, then front-run a victim's swap. No malicious token or non-standard ERC20 behavior is required — only a public `swap` call on a pool with such a bin.

## Recommendation

In `buyToken0InBinSpecifiedIn`, after computing `out0Scaled` (both initially at line 591 and after each refinement step at line 612), add an early-exit guard:

```solidity
if (out0Scaled == 0) {
    return (currBinPos, 0, 0, 0, 0);
}
```

This prevents the refinement loop from incorrectly charging input against a zero-output move and mirrors the existing guard in `buyToken0InBinSpecifiedOut` which already returns early when output rounds to zero.

## Proof of Concept

```solidity
// Setup: T=1, currBinPos=0, I=1e18
// lowerPriceX64 just below 1e18*2^64, upperPriceX64 >> 1e18*2^64 (wide bin)
// avgPriceX64 > 1e18*2^64 → totalIn1Scaled (cost to drain bin) > I → else branch taken

BinState memory binState = BinState({
    token0BalanceScaled: 1,   // dust balance
    token1BalanceScaled: 0,
    lengthE6: 1,
    addFeeBuyE6: 0,
    addFeeSellE6: 0
});

SwapMath.SwapState memory state = SwapMath.SwapState({
    amountSpecifiedRemainingScaled: 1e18,
    amountCalculatedScaled: 0,
    protocolFeeAmountScaled: 0,
    feeExclusiveInputScaled: 0
});

// lowerPriceX64 = 1e18 * 2^64 - 1 (guard does not fire)
// upperPriceX64 = type(uint128).max (wide bin, avgPrice >> lowerPrice)
(uint256 finalBinPos, uint256 out0Scaled,,,) = SwapMath.buyToken0InBinSpecifiedIn(
    binState, 0, state, 0,
    1e18 * (1 << 64) - 1,   // lowerPriceX64
    type(uint128).max,        // upperPriceX64
    type(uint128).max,        // priceLimitX64
    0
);

assert(out0Scaled == 0);                              // trader receives nothing
assert(state.amountSpecifiedRemainingScaled == 0);    // entire input consumed
assert(state.amountCalculatedScaled == 0);            // zero credited to trader
```
