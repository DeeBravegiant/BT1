Audit Report

## Title
`updateRSETHPrice()` Permanently Blocked When `maxFeeMintAmountPerDay` Is Zero and Protocol Fees Accrue - (File: `contracts/LRTOracle.sol`)

## Summary
`LRTOracle.initialize()` never sets `maxFeeMintAmountPerDay`, leaving it at its default value of `0`. When `protocolFeeInBPS > 0` and TVL grows, `_updateRsETHPrice()` computes a non-zero `rsethAmountToMintAsProtocolFee` and passes it to `_checkAndUpdateDailyFeeMintLimit()`, which unconditionally reverts because `0 + feeAmount > 0` is always true. The assignment `rsETHPrice = newRsETHPrice` is never reached, freezing the stored price and blocking all yield accounting until a manager explicitly calls `setMaxFeeMintAmountPerDay()`.

## Finding Description
`initialize()` sets only `lrtConfig` and emits an event; `maxFeeMintAmountPerDay` is never initialized and defaults to `0`.

When `protocolFeeInBPS > 0` and `totalETHInProtocol > previousTVL`, the fee path at lines 244–247 produces `protocolFeeInETH > 0`. The branch at line 299 then computes `rsethAmountToMintAsProtocolFee = protocolFeeInETH.divWad(newRsETHPrice)` (always `> 0`) and calls `_checkAndUpdateDailyFeeMintLimit(rsethAmountToMintAsProtocolFee)` at line 303.

Inside `_checkAndUpdateDailyFeeMintLimit()`, the guard at lines 205–207 evaluates `currentPeriodMintedFeeAmount + feeAmount > maxFeeMintAmountPerDay`, which reduces to `feeAmount > 0` — always `true` — and reverts with `DailyFeeMintLimitExceeded`. Execution never reaches `rsETHPrice = newRsETHPrice` at line 313.

The `else` branch at line 310 (`_checkAndUpdateDailyFeeMintLimit(0)`) passes `0`, so `0 > 0` is false and does not revert; the bug is isolated to the fee-accrual path. No existing guard skips the limit check when `maxFeeMintAmountPerDay == 0`.

## Impact Explanation
`rsETHPrice` is frozen at its last stored value. Every subsequent call to `updateRSETHPrice()` by any public caller reverts for as long as `maxFeeMintAmountPerDay == 0` and fees continue to accrue. Accrued yield is permanently inaccessible to the price mechanism until a manager intervenes. This matches **Medium — Permanent freezing of unclaimed yield**: the yield exists in the protocol but cannot be reflected in the exchange rate, denying it to rsETH holders indefinitely in the default post-deployment state.

## Likelihood Explanation
The condition is the default state after `initialize()`. Any deployment where the admin has configured `protocolFeeInBPS > 0` but the manager has not yet called `setMaxFeeMintAmountPerDay()` is immediately affected as soon as EigenLayer rewards cause TVL to grow. No attacker action is required; any EOA calling the public `updateRSETHPrice()` triggers the revert. The `reinitialize()` function also does not set `maxFeeMintAmountPerDay`, so upgrades to v2 carry the same exposure.

## Recommendation
Decouple the fee-mint step from the price-update step. When `maxFeeMintAmountPerDay == 0` or the daily limit is exhausted, skip the fee mint but still commit the new price:

```solidity
if (protocolFeeInETH > 0) {
    uint256 rsethAmountToMintAsProtocolFee = protocolFeeInETH.divWad(newRsETHPrice);
    if (maxFeeMintAmountPerDay > 0 &&
        currentPeriodMintedFeeAmount + rsethAmountToMintAsProtocolFee <= maxFeeMintAmountPerDay) {
        _checkAndUpdateDailyFeeMintLimit(rsethAmountToMintAsProtocolFee);
        // mint ...
    }
    // else: skip fee mint silently, price update still proceeds
} else {
    _checkAndUpdateDailyFeeMintLimit(0);
}
rsETHPrice = newRsETHPrice;
```

Alternatively, require `maxFeeMintAmountPerDay > 0` in `initialize()` / `reinitialize()` before fee collection can be enabled, or add an explicit `if (maxFeeMintAmountPerDay == 0) return;` guard at the top of `_checkAndUpdateDailyFeeMintLimit()`.

## Proof of Concept
1. Deploy `LRTOracle` and call `initialize(lrtConfigAddr)`. `maxFeeMintAmountPerDay` is `0`.
2. Configure `LRTConfig` with `protocolFeeInBPS = 500`.
3. Allow EigenLayer rewards to accrue so `totalETHInProtocol > rsethSupply * rsETHPrice`.
4. Call `LRTOracle.updateRSETHPrice()` from any EOA.
5. `_updateRsETHPrice()` computes `protocolFeeInETH > 0`, then `rsethAmountToMintAsProtocolFee > 0`.
6. `_checkAndUpdateDailyFeeMintLimit(rsethAmountToMintAsProtocolFee)` evaluates `0 + rsethAmountToMintAsProtocolFee > 0` → `true` → reverts with `DailyFeeMintLimitExceeded(rsethAmountToMintAsProtocolFee, 0)`.
7. `rsETHPrice` is never updated. All subsequent deposits use the stale price, and accrued yield remains frozen.
