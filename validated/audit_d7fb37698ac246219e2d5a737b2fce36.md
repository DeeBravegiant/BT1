### Title
Stale `parameters_votes` Persists After Resharing Cancellation, Enabling Resharing with Fewer Votes Than Required - (File: `crates/contract/src/state/resharing.rs`)

---

### Summary

When resharing is cancelled via `vote_cancel_resharing`, the returned `previous_running_state` retains stale `parameters_votes` that were accumulated during the resharing re-proposal phase (`vote_new_parameters` on `ResharingContractState`). This is directly analogous to H-01: a state-transition operation updates one field (`previously_cancelled_resharing_epoch_id`) but omits resetting a related field (`parameters_votes`), leaving the two in an inconsistent state. The stale votes persist into the new Running state and count toward the threshold for triggering a subsequent resharing, breaking the invariant that **all** proposed participants must vote in the current epoch.

---

### Finding Description

**Root cause — missing reset in `vote_cancel_resharing`:**

During resharing, participants may call `vote_new_parameters` on the `ResharingContractState`. This internally calls `self.previous_running_state.process_new_parameters_proposal(proposal)`, which mutates `self.previous_running_state.parameters_votes` in place: [1](#0-0) 

If the re-proposal does not reach unanimity (fewer than all proposed participants vote), `self.previous_running_state.parameters_votes` now holds partial stale votes for the re-proposal.

When resharing is subsequently cancelled, `vote_cancel_resharing` clones `self.previous_running_state` and sets only `previously_cancelled_resharing_epoch_id`, without resetting `parameters_votes`: [2](#0-1) 

Compare this with the correct path: every other place that constructs a `RunningContractState` uses `RunningContractState::new()`, which explicitly resets `parameters_votes` to `ThresholdParametersVotes::default()`: [3](#0-2) 

**How stale votes count toward the next resharing:**

Back in Running state, a call to `vote_new_parameters` reaches `process_new_parameters_proposal`, whose final vote-count line is: [4](#0-3) 

`ThresholdParametersVotes::vote` counts **all** stored votes for the proposal, with no epoch-ID filter: [5](#0-4) 

The epoch-ID guard in `vote_new_parameters` only validates the *caller-supplied* `prospective_epoch_id` argument; it does not invalidate votes already stored in `parameters_votes`. Therefore, stale votes from the cancelled resharing's re-proposal phase are counted as if they were cast in the current epoch.

---

### Impact Explanation

The contract's governance invariant is that resharing requires **all** `new_num_participants` proposed participants to vote for the same proposal in the current epoch: [4](#0-3) 

Stale votes break this invariant. If `k` participants voted for re-proposal `P'` during resharing, and resharing is then cancelled, only `N' − k` additional participants need to vote for `P'` in the subsequent Running state to trigger a new resharing. This allows resharing to be initiated without the explicit, epoch-scoped consent of all proposed participants, violating the production safety invariant for participant-state transitions.

This maps to the allowed impact: **Medium — participant-state manipulation that breaks production safety/accounting invariants**.

---

### Likelihood Explanation

The sequence requires: (1) resharing is in progress, (2) at least one participant calls `vote_new_parameters` on the resharing state with a re-proposal that does not reach unanimity, (3) resharing is cancelled, (4) the same proposal is reused. Steps 2–3 are realistic operational scenarios (e.g., a participant set change is proposed mid-resharing, then abandoned). The attacker must be a legitimate participant, but no threshold-or-above collusion is required — a single participant who voted during step 2 benefits from the stale vote.

---

### Recommendation

In `vote_cancel_resharing`, construct the returned Running state via `RunningContractState::new()` (which resets `parameters_votes` to default) rather than cloning `previous_running_state` directly:

```rust
// Current (buggy):
let mut previous_running_state = self.previous_running_state.clone();
previous_running_state.previously_cancelled_resharing_epoch_id = Some(prospective_epoch_id);
Some(previous_running_state)

// Fixed:
let mut fresh = RunningContractState::new(
    self.previous_running_state.domains.clone(),
    self.previous_running_state.keyset.clone(),
    self.previous_running_state.parameters.clone(),
    self.previous_running_state.add_domains_votes.clone(),
);
fresh.previously_cancelled_resharing_epoch_id = Some(prospective_epoch_id);
Some(fresh)
```

This mirrors the pattern used in every other state-transition path and ensures `parameters_votes` is always epoch-scoped.

### Proof of Concept

**Starting state:**
```
Running: participants = {A, B, C, D, E}, threshold = 3, epoch = E
parameters_votes = {} (empty)
```

**Step 1 — Resharing starts** (all 5 vote for proposal P):
```
ResharingContractState created.
previous_running_state.parameters_votes = {} (reset by RunningContractState::new)
```

**Step 2 — Re-proposal during resharing** (A, B, C, D vote for P', E does not):
```
previous_running_state.parameters_votes = {A→P', B→P', C→P', D→P'}
```

**Step 3 — Resharing cancelled** (A, B, C vote to cancel):
```
Running state returned = previous_running_state (cloned)
previously_cancelled_resharing_epoch_id = Some(E+1)
parameters_votes = {A→P', B→P', C→P', D→P'}  ← stale votes NOT cleared
```

**Step 4 — New resharing attempt** (only E votes for P' with epoch E+2):
```
process_new_parameters_proposal counts: A, B, C, D (stale) + E (new) = 5 = N'
→ resharing triggered with only 1 new vote instead of 5
``` [2](#0-1) [3](#0-2) [4](#0-3)

    pub fn vote_cancel_resharing(&mut self) -> Result<Option<RunningContractState>, Error> {
        let previous_running_participants = self.previous_running_state.parameters.participants();
        let authenticated_candidate = AuthenticatedAccountId::new(previous_running_participants)?;
        self.cancellation_requests.insert(authenticated_candidate);

        let cancellation_votes_count = self.cancellation_requests.len() as u64;
        let previous_running_threshold = self.previous_running_state.parameters.threshold();

        let threshold_cancellation_votes_reached: bool =
            cancellation_votes_count >= previous_running_threshold.value();

        let running_state = if threshold_cancellation_votes_reached {
            let mut previous_running_state = self.previous_running_state.clone();
            let prospective_epoch_id = self.prospective_epoch_id();
            previous_running_state.previously_cancelled_resharing_epoch_id =
                Some(prospective_epoch_id);

            Some(previous_running_state)
        } else {
            None
        };

        Ok(running_state)
    }
