### Title
Unchecked `sub_epoch_n` index in `_validate_sub_epoch_segments()` causes unhandled `IndexError` on crafted weight proof — (File: `chia/full_node/weight_proof.py`)

### Summary
In `_validate_sub_epoch_segments()`, the loop iterates over `segments_by_sub_epoch.items()` and unconditionally accesses `summaries[sub_epoch_n]` without verifying that `sub_epoch_n` is within the bounds of the `summaries` list. A malicious peer can craft a weight proof containing a segment whose `sub_epoch_n` field equals or exceeds `len(summaries)`, triggering an unhandled Python `IndexError` that propagates out of weight proof validation entirely.

The `sub_epoch_n` values originate from `SubEpochChallengeSegment.sub_epoch_n` fields inside the peer-supplied weight proof bytes, deserialized at line 959:

```python
sub_epoch_segments: SubEpochSegments = SubEpochSegments.from_bytes(weight_proof_bytes)
```

The `summaries` list has a fixed length determined by the validated `sub_epochs` data. There is no guard that checks `sub_epoch_n < len(summaries)` before the access at line 985. If `sub_epoch_n >= len(summaries)`, Python raises an `IndexError`.

The preceding guard `validate_sub_epoch_sampling()` (called at lines 592 and 1733) only verifies that every *sampled* sub-epoch index has a corresponding segment present. It does not reject segments whose `sub_epoch_n` exceeds `len(summaries) - 1`. A crafted proof that satisfies all sampled-sub-epoch requirements but also includes one extra segment with an out-of-range `sub_epoch_n` passes `validate_sub_epoch_sampling` and then crashes `_validate_sub_epoch_segments`.

Neither `validate_weight_proof_single_proc` (line 596) nor `validate_weight_proof_inner` (line 1749) wraps the call to `_validate_sub_epoch_segments` in a `try/except`, so the `IndexError` propagates unhandled.

### Impact Explanation
An unprivileged peer that can send a weight proof to a syncing node can trigger an unhandled `IndexError` in the weight proof validation path. Depending on how the calling sync code handles the exception, this can:

- Crash the full node process entirely if the exception reaches the top-level event loop unhandled.
- Permanently prevent the node from completing sync if every reachable peer sends the crafted proof, since the validation exception is not converted to a graceful `False` return.

This maps to the **High** impact category: *"Permanent or long-lived inability for honest nodes… to process valid blocks, sync updates… under normal network assumptions."*

### Likelihood Explanation
Any peer that participates in the Chia network can send a weight proof to a syncing node. Constructing a proof with a valid set of sampled segments plus one extra segment with an arbitrary `sub_epoch_n` value requires only knowledge of the weight proof serialization format, which is public. No keys, admin access, or cryptographic breaks are required.

### Recommendation
Add an explicit bounds check before accessing `summaries[sub_epoch_n]` in `_validate_sub_epoch_segments()`:

```python
if sub_epoch_n >= len(summaries):
    log.error(f"sub_epoch_n {sub_epoch_n} out of range for summaries (len={len(summaries)})")
    return None
```

This should be inserted immediately after the `max_segments` check (after line 972) so that any out-of-range `sub_epoch_n` causes a clean `None` return rather than an unhandled exception. A similar defensive check should be added for `sub_epoch_n - 1` at line 983.

### Proof of Concept

1. Obtain a legitimate weight proof `wp` from a syncing node.
2. Deserialize `wp.sub_epoch_segments` and append one extra `SubEpochChallengeSegment` with `sub_epoch_n = len(wp.sub_epochs)` (i.e., one past the last valid index) and any otherwise-valid segment body.
3. Re-serialize and send the crafted `WeightProof` to a target node during its sync handshake.
4. The target node calls `validate_sub_epoch_sampling` → passes (all sampled sub-epochs are present).
5. The target node calls `_validate_sub_epoch_segments` → at line 985, `summaries[sub_epoch_n]` raises `IndexError: list index out of range`.
6. The exception propagates through `validate_weight_proof_single_proc` / `validate_weight_proof_inner` with no `try/except`, crashing or aborting the sync process.

def validate_sub_epoch_sampling(
    rng: random.Random, sub_epoch_weight_list: list[uint128], weight_proof: WeightProof
) -> bool:
    tip = weight_proof.recent_chain_data[-1]
    weight_to_check = _get_weights_for_sampling(rng, tip.weight, weight_proof.recent_chain_data)
    sampled_sub_epochs: dict[int, bool] = {}
    for idx in range(1, len(sub_epoch_weight_list)):
        if _sample_sub_epoch(sub_epoch_weight_list[idx - 1], sub_epoch_weight_list[idx], weight_to_check):
            sampled_sub_epochs[idx - 1] = True
            if len(sampled_sub_epochs) == WeightProofHandler.MAX_SAMPLES:
                break
    curr_sub_epoch_n = -1
    for sub_epoch_segment in weight_proof.sub_epoch_segments:
        if curr_sub_epoch_n < sub_epoch_segment.sub_epoch_n:
            if sub_epoch_segment.sub_epoch_n in sampled_sub_epochs:
                del sampled_sub_epochs[sub_epoch_segment.sub_epoch_n]
        curr_sub_epoch_n = sub_epoch_segment.sub_epoch_n
    if len(sampled_sub_epochs) > 0:
        return False
    return True
