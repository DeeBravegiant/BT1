### Title
Harvester-Controlled `farmer_reward_address_override` Accepted Without Threshold Enforcement, Enabling Unconditional Farmer Reward Diversion — (`chia/farmer/farmer_api.py`)

### Summary
A connected harvester can unconditionally redirect the farmer's block reward (XCH) to any arbitrary puzzle hash by setting `farmer_reward_address_override` in the `RespondSignatures` protocol message. The farmer never rejects this override — it only logs a warning when the self-reported fee threshold is not met — making the enforcement entirely honor-based and bypassable by any malicious harvester.

### Finding Description

CHIP-22 introduced `farmer_reward_address_override` as a mechanism for third-party harvesters to take a fee by redirecting the farmer's block reward to themselves, subject to a proof-quality threshold. The harvester sends a `NewProofOfSpace` message with `farmer_reward_address_override` set, and later a `RespondSignatures` message with the same field.

In `_process_respond_signatures` in `chia/farmer/farmer_api.py`, the farmer unconditionally substitutes the override for its own configured reward address:

```python
farmer_reward_address = self.farmer.farmer_target
if response.farmer_reward_address_override is not None:
    farmer_reward_address = response.farmer_reward_address_override   # line 918
    include_source_signature_data = True
``` [1](#0-0) 

This `farmer_reward_address` is then passed directly into `DeclareProofOfSpace` and broadcast to the full node, which includes it in the block's foliage as the farmer reward destination:

```python
return farmer_protocol.DeclareProofOfSpace(
    ...
    farmer_reward_address,   # line 929 — attacker-controlled
    ...
)
``` [2](#0-1) 

The only "guard" is `notify_farmer_reward_taken_by_harvester_as_fee`, which logs a warning when the fee quality does not meet the harvester-supplied threshold, but **never rejects the override or prevents the block from being declared with the attacker's address**:

```python
else:
    self.log.warning(
        "Harvester illegitimately took reward by failing to provide its fee rate ..."
    )
``` [3](#0-2) 

Furthermore, the `applied_fee_threshold` inside `fee_info` is entirely self-reported by the harvester. A malicious harvester can set it to `0` (always passes) or omit `fee_info` entirely. In all cases the farmer still uses the override address.

The `NewProofOfSpace` and `RespondSignatures` streamable types confirm `farmer_reward_address_override` is an optional `bytes32` with no protocol-level restriction: [4](#0-3) 

### Impact Explanation

**Critical — unauthorized diversion of XCH farmer block rewards.**

When a farmer wins a block, the farmer reward coin (currently 0.25 XCH) is created at the puzzle hash specified in `DeclareProofOfSpace.farmer_reward_puzzle_hash`. A malicious harvester can set this to any address it controls. The farmer has no on-chain or protocol-level mechanism to detect or prevent this. The reward is permanently lost to the farmer.

### Likelihood Explanation

Any harvester connected to the farmer — including third-party remote harvesting services, which CHIP-22 explicitly targets — can exploit this. The farmer actively connects to harvesters and trusts their protocol messages. No key compromise, admin access, or cryptographic break is required. The attacker only needs to be an accepted harvester peer.

### Recommendation

The farmer should enforce the fee-quality threshold locally before accepting `farmer_reward_address_override`:

1. Compute `fee_quality = calculate_harvester_fee_quality(proof, sp.challenge_hash)` on the farmer side (the farmer already has both values).
2. If `fee_info` is `None` or `fee_quality > fee_info.applied_fee_threshold`, **reject the override** (do not substitute `farmer_reward_address_override` for `self.farmer.farmer_target`) rather than merely logging a warning.
3. Optionally, allow the farmer to configure a maximum acceptable fee threshold so it can cap how much of its reward a harvester may claim.

### Proof of Concept

1. Operator runs a harvesting service and accepts a farmer's harvester connection.
2. When the farmer's plots win a block, the harvesting service sends `RespondSignatures` with:
   - `farmer_reward_address_override = <attacker_puzzle_hash>`
   - `fee_info = ProofOfSpaceFeeInfo(applied_fee_threshold=0)` (always passes the log check)
3. The farmer's `_process_respond_signatures` substitutes the attacker's address at line 918 and broadcasts `DeclareProofOfSpace` with `farmer_reward_address = <attacker_puzzle_hash>`.
4. The full node includes the block; the farmer reward coin is created at the attacker's address. The farmer receives nothing.

The farmer logs `"Fee threshold passed"` (because `fee_quality <= 0` is never true for a valid proof, but `applied_fee_threshold=0xFFFFFFFF` would always pass), or alternatively the attacker omits `fee_info` and the farmer logs a warning — in both cases the reward is already diverted with no recourse.
