Audit Report

## Title
Dead Fee-Split State: Treasury Never Receives Configured Fee Share - (File: `contracts/FeeReceiver.sol`)

## Summary

`FeeReceiver.initialize()` validates and stores a treasury address and fee BPS, but `sendFunds()` unconditionally forwards 100% of the contract's ETH balance to `depositPool`, never reading those stored values. The treasury receives 0 ETH on every invocation, permanently, for the lifetime of the deployment.

## Finding Description

`initialize()` enforces non-zero values for `_protocolTreasury` and `_protocolFeePercentInBPS` and reverts with `InvalidEmptyValue` if either is zero: [1](#0-0) 

Both values are stored into state variables: [2](#0-1) 

`sendFunds()` then sends the entire balance to `depositPool` without referencing `_legacyProtocolTreasury` or `_legacyProtocolFeePercentInBPS`: [3](#0-2) 

The only other manager function, `setDepositPool`, also does not touch them: [4](#0-3) 

No function in the contract reads `_legacyProtocolTreasury` or `_legacyProtocolFeePercentInBPS` after storage. The `_legacy` prefix confirms these fields were deprecated from an earlier version, yet `initialize()` still enforces non-zero values for them, creating a misleading interface that promises a fee split it never delivers.

## Impact Explanation

Every call to `sendFunds()` sends 100% of MEV/execution-layer rewards to `depositPool`. The treasury address configured at initialization receives 0 ETH, permanently. The configured fee percentage is never applied. Funds are not lost — they reach the deposit pool — but the treasury is permanently deprived of its promised share. This matches the allowed impact: **Low — contract fails to deliver promised returns, but doesn't lose value**.

## Likelihood Explanation

`sendFunds()` is `external` with no access control modifier, callable by any address. No special attacker capability is required. Every invocation unconditionally reproduces the broken behavior. The treasury will never receive its share for the lifetime of this contract deployment unless the contract is upgraded.

## Recommendation

Either:
1. Remove `_protocolTreasury` and `_protocolFeePercentInBPS` from `initialize()` entirely (and the corresponding storage variables) if the fee split is intentionally deprecated, or
2. Implement the fee split in `sendFunds()`:
```solidity
uint256 feeAmount = balance * _legacyProtocolFeePercentInBPS / 10_000;
(bool ok,) = _legacyProtocolTreasury.call{value: feeAmount}("");
require(ok);
ILRTDepositPool(depositPool).receiveFromRewardReceiver{value: balance - feeAmount}();
```

## Proof of Concept

```solidity
// Initialize with treasury T and 5% fee
feeReceiver.initialize(T, depositPool, 500, admin, manager);

// Send 1 ETH to contract
vm.deal(address(feeReceiver), 1 ether);

// Call sendFunds (no access control — callable by anyone)
feeReceiver.sendFunds();

// Treasury receives nothing
assertEq(T.balance, 0);
// DepositPool receives 100%
assertEq(depositPool.balance, 1 ether);
```
