Audit Report

## Title
Missing `whenNotPaused` Modifier on `addLiquidity` and `removeLiquidity` Allows Liquidity Operations on a Paused Pool - (File: metric-core/contracts/MetricOmmPool.sol)

## Summary
`MetricOmmPool` defines a `whenNotPaused` modifier backed by `_checkNotPaused()` and correctly applies it to `swap`, but `addLiquidity` and `removeLiquidity` carry only `nonReentrant` with no pause check. This allows both functions to execute against pool state — moving real token balances and mutating share accounting — even when `pauseLevel != 0`.

## Finding Description
The `whenNotPaused` modifier is defined at [1](#0-0)  and enforced by `_checkNotPaused()` at [2](#0-1) .

`swap` correctly carries both guards: [3](#0-2) 

`addLiquidity` carries only `nonReentrant`, with no pause check: [4](#0-3) 

`removeLiquidity` likewise carries only `nonReentrant`: [5](#0-4) 

Both functions mutate `binTotals`, `_binTotalShares`, and `_positionBinShares` and move real token balances via `LiquidityLib.addLiquidity`/`removeLiquidity` while the pool is supposed to be frozen. [6](#0-5) [7](#0-6) 

`pauseLevel` is set via `setPause`, an `onlyFactory` call that accepts levels 0–2: [8](#0-7) 

## Impact Explanation
When the pool is paused (e.g., oracle compromise, exploit response, emergency upgrade), the intended invariant is that no state-mutating pool operations proceed. Because `addLiquidity` and `removeLiquidity` bypass the pause check:

1. **`addLiquidity` while paused**: An LP can deposit tokens into a pool whose `curBinIdx`/`curPosInBin` are stale or corrupted. Shares are minted against those bad values; when the pool is unpaused and swaps resume at the bad price, the deposited tokens are immediately at risk, harming honest LPs.
2. **`removeLiquidity` while paused**: A protocol-level pause (level 2) intended to freeze all operations during an emergency upgrade or insolvency investigation can be bypassed. LPs can drain positions ahead of a recovery action, leaving the pool insolvent for remaining participants.

Both paths constitute direct loss of user principal and pool insolvency, meeting Critical/High severity under Sherlock thresholds.

## Likelihood Explanation
Any unprivileged LP or attacker can call `addLiquidity` or `removeLiquidity` at any time after a pause is set, because the pause check is simply absent. A mempool observer can front-run a `setPause` transaction. Even without front-running, the functions remain callable in the same block as or after the pause, with no preconditions beyond holding a position (for `removeLiquidity`) or tokens (for `addLiquidity`). The condition is trivially reachable and repeatable.

## Recommendation
Add `whenNotPaused` to both functions, mirroring the pattern already used by `swap`:

```solidity
function addLiquidity(...) external whenNotPaused nonReentrant(PoolActions.ADD_LIQUIDITY) ...

function removeLiquidity(...) external whenNotPaused nonReentrant(PoolActions.REMOVE_LIQUIDITY) ...
```

If emergency LP withdrawal during a pause is a deliberate design choice, document it explicitly and restrict it to `pauseLevel == 1` only, while blocking both operations at `pauseLevel == 2`.

## Proof of Concept
```solidity
// 1. Factory pauses the pool (e.g., oracle compromise detected)
factory.setPause(address(pool), 2);

// 2. swap correctly reverts:
pool.swap(...); // reverts PoolPaused ✓

// 3. addLiquidity proceeds without revert:
pool.addLiquidity(attacker, salt, deltas, callbackData, ""); // succeeds ✗

// 4. removeLiquidity also proceeds:
pool.removeLiquidity(lp, salt, deltas, ""); // succeeds ✗

// 5. When pool is unpaused, attacker's shares are backed by tokens
//    deposited at a compromised bin position; swaps drain honest LPs.
```
