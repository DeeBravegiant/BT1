Audit Report

## Title
`totalKernelStaked == 0` Gap Permanently Locks Reward Tokens — (`contracts/KERNEL/KernelDepositPool.sol`)

## Summary

`KernelDepositPool` uses a Synthetix-style staking rewards model where `rewardPerToken()` freezes accumulation when `totalKernelStaked == 0`, but the `updateReward` modifier unconditionally advances `updatedAt` to the current timestamp. Any rewards that should have accrued during a zero-staked gap are permanently unrecoverable. An unprivileged last staker can deliberately trigger this by calling `initiateWithdrawal()` for their full balance during an active reward period, with no code-level guard preventing it.

## Finding Description

`rewardPerToken()` (L408–414) short-circuits and returns the frozen `rewardPerTokenStored` when `totalKernelStaked == 0`:

```solidity
function rewardPerToken() public view returns (uint256) {
    if (totalKernelStaked == 0) {
        return rewardPerTokenStored;   // accumulation frozen
    }
    return rewardPerTokenStored + (rewardRate * (lastTimeRewardApplicable() - updatedAt) * DECIMAL_PRECISION)
        / totalKernelStaked;
}
```

The `updateReward` modifier (L232–242) always advances `updatedAt` regardless:

```solidity
modifier updateReward(address _account) {
    rewardPerTokenStored = rewardPerToken();   // frozen if totalKernelStaked == 0
    updatedAt = lastTimeRewardApplicable();    // always advances
    ...
}
```

`initiateWithdrawal()` (L320–338) applies `updateReward` before decrementing `totalKernelStaked`. The modifier fires while `totalKernelStaked` is still nonzero (correctly snapshotting the attacker's earned rewards up to that point), then the function body sets `totalKernelStaked = 0`. From this moment:

- Any subsequent call that triggers `updateReward` (e.g., the next `stake()`) will find `totalKernelStaked == 0`, freeze `rewardPerTokenStored`, and advance `updatedAt` to the current time.
- The entire `rewardRate × gap_duration` of tokens that should have accrued during the zero-staked window are silently skipped and permanently stranded in the contract.
- There is no admin recovery function in the contract to reclaim these tokens.

The NatSpec at L18–22 explicitly acknowledges this behavior but relies entirely on an off-chain operational assumption ("ensuring there are always some tokens staked") with zero code-level enforcement.

## Impact Explanation

Reward tokens accrued during the zero-staked window are permanently locked in the contract. No staker — present or future — can ever claim them, and no admin function exists to recover them. This is **permanent freezing of unclaimed yield**, matching the allowed Medium impact class. The magnitude scales with `rewardRate × gap_duration`, which can represent up to 30 days of the full reward allocation.

## Likelihood Explanation

The attacker must be the last (or only) staker when a reward period is active. This is achievable: the attacker stakes any nonzero amount before `notifyRewardAmount` is called (required because `notifyRewardAmount` reverts on `totalKernelStaked == 0` at L570), waits for the admin to start the reward period, then immediately calls `initiateWithdrawal()` for their full balance. The only cost is gas and the withdrawal delay (up to 30 days). The attack is repeatable each reward period and requires no special privilege.

## Recommendation

1. **Do not advance `updatedAt` when `totalKernelStaked == 0`**: In the `updateReward` modifier, only set `updatedAt = lastTimeRewardApplicable()` when `totalKernelStaked > 0`, so the zero-staked gap is not silently consumed.
2. **Enforce a minimum stake floor in code**: Revert `initiateWithdrawal` if it would bring `totalKernelStaked` to zero while a reward period is active (`block.timestamp < finishAt`).
3. **Add a recovery function**: Allow the admin to reclaim stranded reward tokens after a reward period ends if they were never distributed.

## Proof of Concept

```
Setup: rewardRate = 1e18 tokens/sec, duration = 30 days, withdrawalDelay = 30 days

T0: Attacker stakes 1 wei KERNEL → totalKernelStaked = 1
T0: Admin calls notifyRewardAmount(X) → finishAt = T0 + 30d, updatedAt = T0, rewardRate = X/30d

T0+1s: Attacker calls initiateWithdrawal(1):
  - updateReward fires (totalKernelStaked still 1):
      rewardPerTokenStored += rewardRate * 1s * 1e18 / 1  (tiny amount, attacker earns it)
      updatedAt = T0+1s
  - Function body: totalKernelStaked = 0

T0+1s to T0+30d: No one stakes. totalKernelStaked == 0.
  Any call to updateReward during this window advances updatedAt but freezes rewardPerTokenStored.

T0+30d: New staker calls stake(1e18):
  - updateReward fires (totalKernelStaked still 0):
      rewardPerToken() returns rewardPerTokenStored (frozen)
      updatedAt = lastTimeRewardApplicable() = finishAt = T0+30d
  - Function body: totalKernelStaked = 1e18

New staker calls getReward():
  - rewardPerToken(): lastTimeRewardApplicable() - updatedAt = finishAt - finishAt = 0
  - earned = 0

Result: ~30 days × rewardRate tokens permanently locked in contract.
         Attacker reclaims their 1 wei KERNEL after the withdrawal delay at no net cost.
```

Foundry test plan: deploy contract, stake 1 wei, call `notifyRewardAmount`, call `initiateWithdrawal(1)`, `vm.warp` to `finishAt`, stake again, call `getReward`, assert earned == 0 and `rewardsToken.balanceOf(contract) ≈ full reward allocation`.
