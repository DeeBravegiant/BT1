### Title
Dijkstra-Era `ppuWellFormed` Ignores Validity Bounds for New Reference-Script Fee Parameters — (File: `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/PParams.hs`)

---

### Summary

The `ppuWellFormed` implementation for `DijkstraEra` does not validate the four new Dijkstra-specific protocol parameters (`maxRefScriptSizePerBlock`, `maxRefScriptSizePerTx`, `refScriptCostStride`, `refScriptCostMultiplier`) that directly govern reference-script fee computation and block/transaction size limits. An unprivileged governance proposal author can submit a `ParameterChange` governance action that sets these parameters to zero (or to values that break the fee formula), and the ledger will accept and enact it, permanently disabling the DDoS-deterrent fee mechanism and the reference-script size caps.

---

### Finding Description

The `ConwayEraPParams` type class exposes `ppuWellFormed :: ProtVer -> PParamsUpdate era -> Bool`, which is the sole ledger-level gate that rejects malformed `ParameterChange` governance proposals before they enter the proposal queue. The check is invoked in `actionWellFormed` inside `conwayGovTransition`:

```haskell
actionWellFormed pv ga = failureUnless isWellFormed $ MalformedProposal ga
  where
    isWellFormed = case ga of
      ParameterChange _ ppd _ -> ppuWellFormed pv ppd
      _ -> True
```

The `DijkstraEra` instance of `ppuWellFormed` (lines 540–557) checks only the parameters inherited from Conway:

```haskell
instance ConwayEraPParams DijkstraEra where
  ppuWellFormed _pv ppu =
    and
      [ isValid (/= 0) ppuMaxBBSizeL
      , isValid (/= 0) ppuMaxTxSizeL
      , ...
      , isValid (/= 0) ppuNOptL
      ]
```

The four new Dijkstra parameters are **completely absent** from this list:

| Parameter | Lens | Semantic requirement |
|---|---|---|
| `dppMaxRefScriptSizePerBlock` | `ppuMaxRefScriptSizePerBlockL` | must be `> 0` (zero disables block-level cap) |
| `dppMaxRefScriptSizePerTx` | `ppuMaxRefScriptSizePerTxL` | must be `> 0` (zero disables tx-level cap) |
| `dppRefScriptCostStride` | `ppuRefScriptCostStrideL` | already `NonZero Word32` by type, but `> maxRefScriptSizePerTx` is not checked |
| `dppRefScriptCostMultiplier` | `ppuRefScriptCostMultiplierL` | `PositiveInterval` by type, but `< 1` (deflationary) is not checked |

The analogy to the Pyth report is exact: just as the Solidity contract used a price value while ignoring the confidence interval that signals whether the price is trustworthy, the Dijkstra ledger uses these parameters in fee and size-limit calculations while ignoring whether the governance-proposed values are within the bounds that make those calculations meaningful.

---

### Impact Explanation

**Critical / High — unauthorized modification of fees and resource limits outside design parameters.**

1. **Zero `maxRefScriptSizePerBlock` / `maxRefScriptSizePerTx`**: Setting either to `0` means the size-limit check `totalRefScriptSize <= pp ^. ppMaxRefScriptSizePerTxG` passes for every transaction (0 ≥ 0 is vacuously true or the comparison collapses), re-opening the DDoS vector that ADR-9 was specifically designed to close. Transactions with arbitrarily large reference scripts would be accepted, allowing an attacker to flood the network with cheap-to-submit but expensive-to-validate transactions.

2. **`refScriptCostMultiplier < 1`**: A multiplier below 1.0 makes each successive pricing tier *cheaper* than the previous one, inverting the exponential deterrent. An attacker could use very large reference scripts at near-zero marginal cost, again enabling the DDoS attack.

3. **`refScriptCostStride` larger than `maxRefScriptSizePerTx`**: The tiered fee formula never advances to a second tier, making the entire fee linear at the base rate regardless of script size — equivalent to the pre-Conway vulnerability.

All three scenarios result in fees and/or resource limits being set outside their design parameters by an attacker-controlled governance proposal, matching the "Medium" impact class: *attacker-controlled proposals modify fees, deposits, or withdrawals outside design parameters*. The size-limit bypass additionally risks a permanent ledger-level DDoS condition, which could rise to "High" (permanent freezing of honest node operation requiring a hard fork to restore).

---

### Likelihood Explanation

The entry path requires submitting a `ParameterChange` governance action, which is available to any ADA holder who can pay the `govActionDeposit`. No privileged key is needed. The `ppuWellFormed` check is the only ledger-enforced gate; there is no secondary validation of these four parameters anywhere in the transition rules. Once ratified and enacted, the parameters take effect immediately at the epoch boundary via `dijkstraApplyPPUpdates`, with no further sanity check.

---

### Recommendation

Extend the `ppuWellFormed` check in `DijkstraEra` to include the four new parameters:

```haskell
instance ConwayEraPParams DijkstraEra where
  ppuWellFormed _pv ppu =
    and
      [ ...existing checks...
      , isValid (/= 0) ppuMaxRefScriptSizePerBlockL
      , isValid (/= 0) ppuMaxRefScriptSizePerTxL
      -- refScriptCostStride is NonZero Word32 so zero is impossible by type,
      -- but add a cross-parameter check:
      , isValid (\s -> maybe True (\tx -> unNonZero s <= tx)
                         (strictMaybeToMaybe $ ppu ^. ppuMaxRefScriptSizePerTxL))
                ppuRefScriptCostStrideL
      -- multiplier must be >= 1 to preserve the deterrent property:
      , isValid (>= minBound {- i.e. 1.0 -}) ppuRefScriptCostMultiplierL
      ]
```

At minimum, `maxRefScriptSizePerBlock` and `maxRefScriptSizePerTx` must be non-zero, mirroring the pattern already applied to `ppuMaxBBSizeL`, `ppuMaxTxSizeL`, etc.

---

### Proof of Concept

1. Construct a `ParameterChange` governance action with a `PParamsUpdate DijkstraEra` that sets `ppuMaxRefScriptSizePerTxL = SJust 0`.
2. Submit it on-chain (requires only `govActionDeposit` ADA).
3. `actionWellFormed` calls `ppuWellFormed` — the check at lines 540–557 does **not** include `ppuMaxRefScriptSizePerTxL`, so the proposal passes as well-formed.
4. After ratification and enactment, `dijkstraApplyPPUpdates` sets `dppMaxRefScriptSizePerTx = THKD 0`.
5. The reference-script size check in the UTXO rule now allows transactions with arbitrarily large reference scripts, re-enabling the DDoS attack vector documented in ADR-9.
