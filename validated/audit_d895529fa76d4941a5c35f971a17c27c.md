Based on my investigation of the full call chain, here is my assessment:

### Title
DID Wallet Permanently Fails to Track Coins with Non-UTF-8 Metadata Due to Missing Exception Handling in `did_program_to_metadata` — (File: `chia/wallet/did_wallet/did_wallet_puzzles.py`)

---

### Summary

`did_program_to_metadata` decodes on-chain metadata bytes using `str(key, "utf-8")` and `str(val, "utf-8")` with no exception handling. An unprivileged on-chain actor can embed non-UTF-8 bytes in a DID's metadata field and transfer that DID to a victim wallet. Every code path that processes the resulting coin — `create_new_did_wallet_from_coin_spend`, `coin_added`, and `find_lost_did` — calls `did_program_to_metadata` without a surrounding `try/except`, so a `UnicodeDecodeError` propagates uncaught, leaving the DID wallet in a permanently broken sync state.

---

### Finding Description

**Root cause — `did_program_to_metadata`:** [1](#0-0) 

```python
def did_program_to_metadata(program: Program) -> dict[str, str]:
    metadata = {}
    for key, val in program.as_python():
        metadata[str(key, "utf-8")] = str(val, "utf-8")   # ← no try/except
    return metadata
```

`str(bytes, "utf-8")` raises `UnicodeDecodeError` on any byte sequence that is not valid UTF-8 (e.g., `b'\xff\xfe'`). DID metadata is a curried argument in the on-chain puzzle; any spender of a DID singleton can freely set it to arbitrary bytes.

**Call site 1 — `DIDWallet.coin_added` (existing DID wallet receives a new coin):** [2](#0-1) 

No `try/except` wraps the call. The exception aborts `coin_added` after `coin_store.add_coin_record` has already run (line 3475 in `wallet_state_manager.py`), leaving the coin record in the store but the DID wallet info stale/inconsistent. [3](#0-2) 

**Call site 2 — `create_new_did_wallet_from_coin_spend` (wallet creation on DID receipt):** [4](#0-3) 

No `try/except`. The wallet creation throws before `add_new_wallet` completes, so the DID is never registered.

**Call site 3 — `find_lost_did` (wallet resync / recovery):** [5](#0-4) 

Same pattern — `did_program_to_metadata` is called bare, so resync of any DID with non-UTF-8 metadata permanently fails.

---

### Impact Explanation

An attacker who owns any DID can:

1. Craft a DID spend whose new inner puzzle curries a metadata program containing non-UTF-8 bytes (e.g., `(list (c "key" (q . #xff #xfe)))`).
2. Transfer that DID to the victim's wallet address.

The victim's wallet will attempt to process the coin via `handle_did` → `create_new_did_wallet_from_coin_spend` → `did_program_to_metadata`, raising `UnicodeDecodeError`. The DID coin is never tracked. If the victim already holds the DID and the attacker is the previous owner who set the metadata before transfer, the same failure occurs in `coin_added`. In the resync path (`find_lost_did`), the wallet can never recover the DID.

This constitutes **corruption of wallet sync state with direct security impact**: the DID singleton (a valuable on-chain asset) becomes permanently invisible to the victim's wallet, and any DID-gated operations (NFT ownership, VC issuance, offer settlement) are blocked.

---

### Likelihood Explanation

- No special privilege is required; any DID owner can set arbitrary metadata bytes in a spend.
- The Chialisp DID puzzle imposes no UTF-8 constraint on metadata atoms.
- The attack is fully on-chain and requires no access to the victim's keys.
- The victim cannot defend against it; the malicious metadata arrives embedded in the puzzle reveal of the parent coin spend.

---

### Recommendation

Wrap the decoding in `did_program_to_metadata` with a `try/except` and fall back gracefully (e.g., hex-encode non-decodable bytes or skip the entry and log a warning):

```python
def did_program_to_metadata(program: Program) -> dict[str, str]:
    metadata = {}
    for key, val in program.as_python():
        try:
            k = str(key, "utf-8")
            v = str(val, "utf-8")
        except (UnicodeDecodeError, TypeError):
            k = key.hex() if isinstance(key, bytes) else repr(key)
            v = val.hex() if isinstance(val, bytes) else repr(val)
        metadata[k] = v
    return metadata
```

All three call sites (`coin_added`, `create_new_did_wallet_from_coin_spend`, `find_lost_did`) will then handle malformed metadata without raising.
