Now I have enough context to write the full finding. The core bug is in `validate_proposal.rs` lines 238–247: `insert_proposal` mutates `valid_proposals` **before** the `ProposalFin` commitment comparison, so a rejected proposal is permanently stored in the cache.

---

### Title
`valid_proposals` mutated before `ProposalFin` commitment check in `validate_proposal` — rejected proposal persists in cache, enabling node panic on re-validation — (File: `crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

---

### Summary

In `validate_proposal`, the validator calls `valid_proposals.insert_proposal(...)` to cache the executed proposal **before** it checks whether the batcher-computed commitment (`built_block`) matches the proposer-supplied `received_fin.proposal_commitment`. When those two values differ, the function returns `Err(ProposalFinMismatch)` and drops the `fin_sender` without notifying consensus — but the proposal is already permanently stored in `valid_proposals` under the batcher's commitment. Because `insert_proposal` contains an unconditional `assert!(previous.is_none(), ...)`, any subsequent attempt to insert a proposal for the same `(height, round)` — which can be triggered by a second proposal from the same malicious proposer after the first validation fails and `pending_validation_rounds` is cleared — causes an unrecoverable panic and crashes the validator node.

---

### Finding Description

**Vulnerable code** — `crates/apollo_consensus_orchestrator/src/validate_proposal.rs`, lines 238–247:

```rust
// Update valid_proposals before sending fin to avoid a race condition
// with `repropose` being called before `valid_proposals` is updated.
let mut valid_proposals = args.valid_proposals.lock().unwrap();
valid_proposals.insert_proposal(args.init, content, &args.proposal_id, finished_info);  // ← state mutated

// TODO(matan): Switch to signature validation.
if built_block != received_fin.proposal_commitment {          // ← check comes AFTER
    CONSENSUS_PROPOSAL_FIN_MISMATCH.increment(1);
    return Err(ValidateProposalError::ProposalFinMismatch);   // ← proposal already stored
}
``` [1](#0-0) 

`insert_proposal` computes the stored commitment from `finished_info.proposal_commitment.partial_block_hash` (the batcher's result, i.e. `built_block`), not from `received_fin.proposal_commitment` (the proposer's claim):

```rust
pub(crate) fn insert_proposal(...) {
    let proposal_commitment = proposal_commitment_from(
        finished_info.proposal_commitment.partial_block_hash,
        init.fee_proposal_fri,
    );
    // ...
    assert!(
        previous.is_none(),
        "Overwriting existing proposal for height {height} round {round}; ..."
    );
}
``` [2](#0-1) 

When `ProposalFinMismatch` is returned, `validate_and_send` propagates the error and drops `fin_sender` without sending:

```rust
async fn validate_and_send(
    args: ProposalValidateArguments,
    fin_sender: oneshot::Sender<ProposalCommitment>,
) -> Result<ProposalCommitment, ValidateProposalError> {
    let proposal_commitment = validate_proposal(args).await?;  // error propagates here
    fin_sender.send(proposal_commitment)...
``` [3](#0-2) 

Consensus receives `Err(Canceled)` from `fin_receiver`, calls `handle_finished_validation(None, round, ...)`, which removes the round from `pending_validation_rounds`:

```rust
fn handle_finished_validation(...) {
    self.pending_validation_rounds.remove(&round);
    self.state_machine.handle_event(StateMachineEvent::FinishedValidation(
        proposal_id,  // None — invalid
        round,
        valid_round,
    ))
}
``` [4](#0-3) 

The duplicate-validation guard in `handle_proposal` checks `pending_validation_rounds` and `has_proposal_for_round`. Once `pending_validation_rounds` no longer contains the round (cleared after the failed validation), and if the state machine does not record an entry for rounds with `None` commitments, a second proposal for the same `(height, round)` passes the guard and triggers a second `validate_proposal` call. That second call reaches `insert_proposal` for the same `(height, round)`, hitting the `assert!(previous.is_none(), ...)` and **panicking the process**. [5](#0-4) 

---

### Impact Explanation

A malicious proposer (any node that is scheduled as proposer for a round) can crash any honest validator node:

1. Send a proposal with syntactically valid transactions and a `ProposalFin` whose `proposal_commitment` differs from what the batcher will compute.
2. The validator executes the transactions, stores the proposal in `valid_proposals`, then rejects it (`ProposalFinMismatch`). The round is cleared from `pending_validation_rounds`.
3. The proposer immediately re-sends a proposal for the same `(height, round)`.
4. The validator attempts a second validation; `insert_proposal` hits `assert!(previous.is_none())` → **process panic / node crash**.

This matches the **High** impact category: *Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing* — here the admission-to-consensus bridge permanently corrupts its own cache state on a rejected proposal, enabling a crash of the validator.

---

### Likelihood Explanation

Any node that is elected proposer for at least one round can trigger this. No special privilege beyond being a scheduled proposer is required. The attack requires only two network messages (one malformed proposal, one re-send) and is deterministic.

---

### Recommendation

Move the `ProposalFin` comparison **before** `insert_proposal`, so the cache is only mutated when the proposal is fully valid:

```rust
// Check FIRST, then store.
if built_block != received_fin.proposal_commitment {
    CONSENSUS_PROPOSAL_FIN_MISMATCH.increment(1);
    return Err(ValidateProposalError::ProposalFinMismatch);
}

let mut valid_proposals = args.valid_proposals.lock().unwrap();
valid_proposals.insert_proposal(args.init, content, &args.proposal_id, finished_info);
Ok(built_block)
```

The comment "Update valid_proposals before sending fin to avoid a race condition with `repropose`" is still satisfied because `repropose` can only be called after consensus receives the commitment via `fin_sender`, which is sent only after `insert_proposal` in the corrected order.

### Proof of Concept

```
Round R, height H:

1. Attacker (proposer for round R) sends:
     ProposalPart::Transactions([tx1, tx2, tx3])
     ProposalPart::Fin {
         proposal_commitment: WRONG_HASH,   // ≠ batcher's computed hash
         executed_transaction_count: 3,
     }

2. Honest validator:
     - batcher executes tx1..tx3, computes CORRECT_HASH
     - insert_proposal(height=H, round=R, commitment=CORRECT_HASH)  ← stored
     - built_block(CORRECT_HASH) != received_fin(WRONG_HASH) → ProposalFinMismatch
     - fin_sender dropped; consensus sees Canceled
     - handle_finished_validation(None, R, ...) → pending_validation_rounds.remove(R)

3. Attacker immediately re-sends any proposal for (H, R).

4. Honest validator:
     - handle_proposal: pending_validation_rounds does not contain R → passes guard
     - validate_proposal called again for (H, R)
     - insert_proposal(height=H, round=R, ...) hits:
         assert!(previous.is_none(), "Overwriting existing proposal ...")
     → PANIC → node crash
```
