### Title
Race Condition Between `fast_fin_transfer_to_near_callback` and `fin_transfer_callback` Enables Double-Credit to Recipient — (`near/omni-bridge/src/lib.rs`)

---

### Summary

`fast_fin_transfer_to_near_callback` does not re-check `finalised_transfers` before calling `add_fast_transfer` and disbursing tokens. Because NEAR's async model introduces a multi-block window between `fast_fin_transfer` (which checks `finalised_transfers`) and its callback, a concurrent `fin_transfer_callback` can finalize the same transfer ID and send tokens to the recipient first. When the callback then runs, it finds neither `fast_transfers` nor `finalised_transfers` blocking it, and sends tokens to the recipient a second time.

---

### Finding Description

**Call flow for `fast_fin_transfer` (NEAR recipient path):**

`fast_fin_transfer` (line 782) checks `is_unified_transfer_finalised` — passes because `finalised_transfers` is empty for this ID. It then builds a promise chain:

```
check_or_pay_ft_storage  →  fast_fin_transfer_to_near_callback
```

At this point, neither `fast_transfers` nor `finalised_transfers` contains the transfer ID. The callback receipt is queued but not yet executed. [1](#0-0) [2](#0-1) 

**Concurrent `fin_transfer_callback` → `process_fin_transfer_to_near`:**

Before the callback receipt is processed (at least 2 blocks later), a second trusted relayer submits a `fin_transfer` proof for the same transfer ID. `fin_transfer_callback` calls `process_fin_transfer_to_near`, which:

1. Calls `add_fin_transfer` — **succeeds**, because `finalised_transfers` does not yet contain the ID.
2. Calls `get_fast_transfer_status` — **returns `None`**, because `fast_transfers` does not yet contain the ID.
3. Since no fast transfer is found, sends tokens directly to the original recipient (**disbursement #1**). [3](#0-2) [4](#0-3) [5](#0-4) 

**`fast_fin_transfer_to_near_callback` runs with no blocking guard:**

The callback has no check against `finalised_transfers`. It calls `add_fast_transfer`, which only checks `fast_transfers` for uniqueness — that map is still empty for this ID (the `fin_transfer` path never writes to it). The insert succeeds, and the callback sends tokens to the recipient again (**disbursement #2**). [6](#0-5) [7](#0-6) 

The `add_fin_transfer` uniqueness guard only prevents a second `fin_transfer` call; it does not prevent the already-queued `fast_fin_transfer_to_near_callback` from completing. [8](#0-7) 

---

### Impact Explanation

The recipient receives tokens twice for a single origin-chain transfer:

- **Disbursement #1**: bridge reserves (unlocked/minted) sent by `process_fin_transfer_to_near`.
- **Disbursement #2**: relayer's deposited tokens sent by `fast_fin_transfer_to_near_callback`.

The relayer's deposited tokens are consumed without reimbursement, and the bridge's locked-token accounting is decremented once while two token disbursements occur. This breaks the 1:1 backing invariant and constitutes unbacked supply / double-credit.

---

### Likelihood Explanation

Both `fast_fin_transfer` (via `ft_on_transfer`, line 760) and `fin_transfer` (line 675–677, `#[trusted_relayer]`) require a trusted relayer. The race window is real and spans at least two NEAR block delays (the `check_or_pay_ft_storage` cross-contract call). This can occur with two honest relayers acting independently and concurrently on the same transfer, or with a single malicious trusted relayer deliberately sequencing the calls. The protocol does not serialize these two paths. [9](#0-8) [10](#0-9) 

---

### Recommendation

Add a `finalised_transfers` check at the start of `fast_fin_transfer_to_near_callback`, before `add_fast_transfer` and before any token disbursement. If the transfer ID is already present in `finalised_transfers`, the callback should abort (panic or return the relayer's tokens). Concretely:

```rust
// At the top of fast_fin_transfer_to_near_callback, after the storage check:
if self.is_unified_transfer_finalised(&fast_transfer.transfer_id) {
    // Transfer already finalized via fin_transfer; abort to avoid double-credit.
    env::panic_str(BridgeError::TransferAlreadyFinalised.to_string().as_str());
}
```

This mirrors the guard already present in `fast_fin_transfer` itself (line 782) but must be repeated in the callback because state can change between the two execution points. [1](#0-0) 

---

### Proof of Concept

Using NEAR workspaces:

1. Register two trusted relayers (A and B) and a recipient account.
2. Relayer A calls `ft_transfer_call` on the token contract, triggering `ft_on_transfer` → `fast_fin_transfer` for `transfer_id = (Eth, nonce=5)` with a NEAR recipient. The `check_or_pay_ft_storage` receipt is now pending.
3. In the same or next block (before `fast_fin_transfer_to_near_callback` executes), Relayer B calls `fin_transfer` with a valid proof for `(Eth, nonce=5)`. The `fin_transfer_callback` receipt is queued.
4. Arrange receipt ordering so `fin_transfer_callback` executes first:
   - `add_fin_transfer` succeeds → `finalised_transfers` now contains `(Eth, nonce=5)`.
   - `get_fast_transfer_status` returns `None` → tokens sent to recipient (disbursement #1).
5. `fast_fin_transfer_to_near_callback` executes next:
   - `add_fast_transfer` succeeds (only checks `fast_transfers`, which is empty for this ID).
   - Tokens sent to recipient again (disbursement #2).
6. Assert recipient balance = 2× expected amount. Assert bridge locked-token counter decremented only once.

        &mut self,
        #[serializer(borsh)] fast_transfer: &FastTransfer,
        #[serializer(borsh)] storage_payer: AccountId,
        #[serializer(borsh)] relayer_id: AccountId,

        require!(
            Self::check_storage_balance_result(0),

        let OmniAddress::Near(recipient) = fast_transfer.recipient.clone() else {
            env::panic_str(BridgeError::InvalidState.to_string().as_str())

            &fast_transfer.msg,
        )

                .with_static_gas(RESOLVE_FAST_TRANSFER_GAS)
                .resolve_fast_transfer(

                    &fast_transfer.id(),
                    amount_without_fee,
                    !fast_transfer.msg.is_empty(),

        )
