Audit Report

## Title
ETH Deposit Limit Bypass Due to Missing Amount Inclusion in Limit Check - (File: contracts/LRTDepositPool.sol)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` applies asymmetric validation for ETH versus LST assets. The ETH branch omits the incoming deposit `amount` from the cap comparison, so when cumulative ETH deposits exactly equal the configured limit, any subsequent ETH deposit passes the check and is accepted, pushing total protocol ETH exposure above the cap by an arbitrary amount in a single transaction.

## Finding Description
In `contracts/LRTDepositPool.sol` at lines 676–682, the function `_checkIfDepositAmountExceedesCurrentLimit` has two branches:

- **LST branch (line 681):** `return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));` — correctly includes the incoming amount.
- **ETH branch (line 679):** `return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));` — silently drops `amount`.

When `totalAssetDeposits == depositLimit`, the ETH branch evaluates `depositLimit > depositLimit` → `false`, so `_beforeDeposit` (line 661) does not revert with `MaximumDepositLimitReached`. The call proceeds through `depositETH` (line 87), which passes `msg.value` to `_beforeDeposit` and then calls `_mintRsETH`, minting rsETH and accepting the ETH with no upper bound on `msg.value`. The deposit limit is therefore unenforceable for ETH once the cumulative total reaches the cap.

## Impact Explanation
The deposit limit is a protocol risk-management parameter. Because the ETH branch omits the incoming amount, a depositor can push total ETH held by the protocol above the configured cap by an arbitrary amount in a single call. No funds are stolen or frozen, but the contract fails to deliver its promised deposit cap enforcement. This matches the allowed impact: **Low — Contract fails to deliver promised returns, but doesn't lose value.**

## Likelihood Explanation
Any unprivileged external caller can trigger this by calling `depositETH` at the moment `totalAssetDeposits == depositLimit`. This condition arises naturally as deposits accumulate toward the cap. No special privileges, front-running, oracle manipulation, or admin collusion is required. The condition is repeatable: after the first bypass, `totalAssetDeposits > depositLimit`, so the ETH branch will revert on subsequent calls — but the damage (overshoot) from the single bypass transaction is already done and unbounded in size.

## Recommendation
Remove the ETH-specific branch and unify the logic to always include the incoming amount:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(
    address asset,
    uint256 amount
) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

## Proof of Concept
1. Admin sets the ETH deposit limit to 1 000 ETH via `lrtConfig`.
2. Cumulative ETH deposits reach exactly 1 000 ETH (`getTotalAssetDeposits(ETH_TOKEN) == 1000 ether`).
3. Attacker calls `depositETH{value: 500 ether}(0, "")`.
4. `_checkIfDepositAmountExceedesCurrentLimit` evaluates `1000 ether > 1000 ether` → `false`; `_beforeDeposit` does not revert.
5. `_mintRsETH` mints rsETH for the attacker; 500 ETH is accepted by the contract.
6. Total ETH in the protocol becomes 1 500 ETH — 50% above the intended cap — in a single unprivileged transaction.
