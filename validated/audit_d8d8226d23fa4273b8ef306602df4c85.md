Audit Report

## Title
ETH Deposit Limit Check Omits Incoming Amount, Allowing Limit Bypass - (File: contracts/LRTDepositPool.sol)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` applies an asymmetric comparison for ETH vs. LST assets. The ETH branch evaluates `totalAssetDeposits > limit` (no incoming amount), while the LST branch correctly evaluates `totalAssetDeposits + amount > limit`. When `totalAssetDeposits == limit`, the ETH branch returns `false`, allowing the deposit to proceed and push the total above the configured cap.

## Finding Description
In `contracts/LRTDepositPool.sol` at lines 676–682:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));   // ← missing + amount
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

The ETH branch omits `amount` from the comparison. When `totalAssetDeposits == depositLimitByAsset(ETH)`, the strict `>` evaluates to `false`, so `_beforeDeposit` (lines 648–670) does not revert with `MaximumDepositLimitReached`. The deposit proceeds through `depositETH` (lines 76–93), which is unrestricted (`external payable nonReentrant whenNotPaused`), minting rsETH and accepting ETH beyond the configured cap. The LST path is unaffected because it includes `+ amount`.

## Impact Explanation
The `depositLimitByAsset` cap is the protocol's stated mechanism to bound exposure to a single asset class. The ETH branch fails to enforce this invariant when the running total exactly equals the limit, allowing an unbounded single deposit to exceed the cap. This matches the allowed impact: **Low — contract fails to deliver promised returns, but doesn't lose value.** No direct theft or fund freeze occurs; the excess ETH is forwarded to NodeDelegators and staked in EigenLayer, permanently increasing protocol exposure beyond the configured bound.

## Likelihood Explanation
The precondition `totalAssetDeposits == depositLimitByAsset(ETH)` is most likely when an admin sets the limit to exactly the current total (e.g., to halt new ETH deposits), or when deposits naturally land on the limit. Both states are observable on-chain. Any unprivileged user monitoring the chain can immediately submit a large `depositETH` call in the same block, before the limit is raised or state changes. The condition is transient but predictable and repeatable.

## Recommendation
Add `amount` to the ETH branch to match the LST branch:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

## Proof of Concept
1. Admin calls `lrtConfig.setDepositLimitByAsset(ETH_TOKEN, 1000 ether)`.
2. Through normal deposits, `getTotalAssetDeposits(ETH_TOKEN)` reaches exactly `1000 ether`.
3. Attacker calls `depositETH{value: 500 ether}(0, "")`.
4. `_checkIfDepositAmountExceedesCurrentLimit` evaluates `1000 ether > 1000 ether` → `false` → no revert.
5. `_mintRsETH` mints rsETH for 500 ETH; `getTotalAssetDeposits(ETH_TOKEN)` becomes `1500 ether`.
6. The cap is now exceeded by 50%; subsequent deposits correctly revert, but the damage is permanent.

**Foundry test sketch:**
```solidity
function test_ethLimitBypass() public {
    // set limit to current balance
    uint256 current = lrtDepositPool.getTotalAssetDeposits(LRTConstants.ETH_TOKEN);
    lrtConfig.setDepositLimitByAsset(LRTConstants.ETH_TOKEN, current);
    // attacker deposits 1 wei above limit — should revert but doesn't
    vm.deal(attacker, 1 ether);
    vm.prank(attacker);
    lrtDepositPool.depositETH{value: 1 ether}(0, "");
    assertGt(lrtDepositPool.getTotalAssetDeposits(LRTConstants.ETH_TOKEN), current);
}
```
