### Title
Node Explicitly Bails on Ethereum and Solana `verify_foreign_transaction` Requests Despite Contract Accepting Them - (File: crates/node/src/providers/verify_foreign_tx/sign.rs)

### Summary

The `ForeignChain` enum and `ForeignChainsConfig` both include `Ethereum` and `Solana` as configurable chains. Nodes can register these chains as supported on-chain. The contract's `verify_foreign_transaction` gate accepts requests for any chain in the supported set. However, the node's `execute_foreign_chain_request` function unconditionally bails on `Ethereum` and `Solana` requests with hardcoded `bail!` statements. This creates a permanent mismatch: the contract accepts and enqueues requests for these chains, consumes a presignature, and the request permanently times out because every node will fail to produce a signature share.

### Finding Description

The `ForeignChain` enum defines 12 chains including `Ethereum`, `Solana`, and `Ton`. [1](#0-0) 

The node configuration struct `ForeignChainsConfig` exposes `solana` and `ethereum` as optional fields, and `all_configured_chains()` maps them to their `ForeignChain` DTO identifiers: [2](#0-1) 

This means operators can legitimately configure Solana and Ethereum in their node YAML, and nodes will register those chains on-chain via `register_available_foreign_chain_config`. Once all active participants register a chain, `get_supported_foreign_chains()` includes it, and the contract's `verify_foreign_transaction` gate passes: [3](#0-2) 

However, in the node's `execute_foreign_chain_request`, both `Ethereum` and `Solana` are hardcoded to unconditionally fail: [4](#0-3) 

`Ton` also bails: [5](#0-4) 

Furthermore, `ForeignChainInspectors` has no `ethereum`, `solana`, or `ton` fields at all — there is no inspector implementation for these chains: [6](#0-5) 

Note that `Ton` is not in `all_configured_chains()`, so it can never be registered and the contract will always reject Ton requests at the gate — that path is safe. The exploitable gap is specifically `Ethereum` and `Solana`, which **can** be registered but **cannot** be executed.

### Impact Explanation

When `Ethereum` or `Solana` is in the supported-chains set (because operators configured those fields), any unprivileged caller can submit a `verify_foreign_transaction` request for those chains. The contract accepts the request, deducts the deposit, and consumes a presignature from the `ForeignTx` domain. Every node then calls `execute_foreign_chain_request`, hits the `bail!`, and produces no signature share. The request permanently times out. The user's deposit is lost and the presignature is wasted. This breaks the request-lifecycle and accounting invariants of the `ForeignTx` domain: the contract's supported-chain gate is supposed to guarantee that accepted requests can be fulfilled, but it does not.

This maps to the **Medium** allowed impact: *balance, request-lifecycle, participant-state, or contract execution-flow manipulation that breaks production safety/accounting invariants*.

### Likelihood Explanation

Likelihood is **medium**. The `ForeignChainsConfig` struct exposes `solana` and `ethereum` fields with full YAML deserialization support, and the localnet/testnet config templates and documentation reference Solana as a valid chain. An operator who follows the configuration documentation and enables Solana or Ethereum will unknowingly put those chains into the supported set. Any user who then calls `verify_foreign_transaction` for those chains — including a malicious user who reads the on-chain supported-chains list — will trigger the permanent failure path.

### Recommendation

Either:
1. Remove `solana` and `ethereum` from `ForeignChainsConfig::all_configured_chains()` so nodes can never register them as supported until the inspector implementations exist; or
2. Implement the missing `SolanaInspector` and `EthereumInspector` in `ForeignChainInspectors` and wire them into `execute_foreign_chain_request`.

The `bail!` stubs should never be reachable for a chain that the contract's gate has already admitted. A compile-time or startup-time assertion that every chain in `all_configured_chains()` has a corresponding match arm in `execute_foreign_chain_request` (other than an unconditional bail) would prevent this class of regression.

### Proof of Concept

1. Operator configures node YAML with a `foreign_chains.solana` section (valid per `ForeignChainsConfig`).
2. All `n` active participants do the same; each calls `register_available_foreign_chain_config` on startup.
3. `get_supported_foreign_chains()` now includes `ForeignChain::Solana`.
4. Unprivileged caller submits:
   ```json
   verify_foreign_transaction({
     "domain_id": <foreign_tx_domain>,
     "payload_version": 1,
     "request": { "Solana": { "tx_id": "...", "finality": "Finalized", "extractors": [...] } }
   })
   ```
5. Contract passes the chain-support gate at line 535 of `lib.rs` and enqueues the request, consuming a presignature.
6. Every node reaches `execute_foreign_chain_request`, matches `ForeignChainRpcRequest::Solana`, and executes `bail!("ForeignChainRpcRequest::Solana is unsupported")` at line 129 of `sign.rs`.
7. No node produces a signature share; the request times out; the user's deposit is not returned. [7](#0-6) [8](#0-7) [3](#0-2)
