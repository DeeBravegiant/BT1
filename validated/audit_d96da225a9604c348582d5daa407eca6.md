### Title
Gateway Stateful Validator Checks Committed-State Balance Without Accounting for Pending Mempool Fee Obligations — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The gateway's `run_validate_entry_point` invokes `verify_can_pay_committed_bounds`, which reads the account's fee-token balance from the **latest committed block state only**. It does not subtract the maximum fees already reserved by pending transactions from the same account sitting in the mempool. This is the direct Sequencer analog of the StakingRewards `notifyRewardAmount` bug: just as that contract checked `balanceOf(rewardsToken)` without subtracting already-owed rewards, the gateway checks `get_fee_token_balance()` without subtracting fees already committed to pending mempool transactions. The result is that the gateway admits multiple transactions whose combined `max_possible_fee` exceeds the account's actual balance, causing later transactions to fail at execution time with "Insufficient fee token balance."

---

### Finding Description

In `StatefulTransactionValidator::run_validate_entry_point()`, the blockifier's `StatefulValidator::validate()` is called, which internally calls `perform_pre_validation_stage()`:

```rust
// crates/blockifier/src/transaction/account_transaction.rs

    &self,
    state: &mut S,
    tx_context: &TransactionContext,
) -> TransactionPreValidationResult<()> {
    ...
    if self.execution_flags.charge_fee {
        self.check_fee_bounds(tx_context)?;
        verify_can_pay_committed_bounds(state, tx_context).map_err(Box::new)?;

    ...

`verify_can_pay_committed_bounds` reads the balance from the committed state and checks only whether it covers the **current transaction's** `max_possible_fee`:

The state reader is instantiated from the **latest committed block** only:

The mempool's `validate_tx` — the only other admission gate — checks only nonce validity and fee escalation, **not balance**:

```rust
// crates/apollo_mempool/src/mempool.rs
pub fn validate_tx(&mut self, args: ValidationArgs) -> MempoolResult<()> {
    let tx_reference = (&args).into();
    self.validate_incoming_tx(tx_reference, args.account_nonce)?;
    self.validate_fee_escalation(tx_reference)?;
    Ok(())
}
``` [4](#0-3) 

The `MempoolState` tracks nonces (via `staged`/`committed` maps) but has no analogous structure for pending fee obligations: [5](#0-4) 

There is therefore **no guard** that prevents the gateway from accepting a second (or Nth) transaction from the same account when the cumulative `max_possible_fee` of all pending transactions already exceeds the account's balance.

---

### Impact Explanation

An account with balance `B` can submit `N` transactions each with `max_fee = B`. Every individual gateway check passes (`B >= B`). All `N` transactions enter the mempool. When the block builder executes them sequentially, only the first transaction can pay its fee; every subsequent transaction fails at post-execution with `FeeCheckError::InsufficientFeeTokenBalance`, reverts, and still consumes a nonce and a small revert fee. This:

- Causes the gateway to **admit invalid transactions** (matching the High impact: "Mempool/gateway/RPC admission accepts invalid transactions before sequencing").
- Wastes block capacity on transactions that are guaranteed to revert.
- Allows an unprivileged actor to fill the mempool with fee-insolvent transactions at low cost (only the first transaction's fee is actually paid).

---

### Likelihood Explanation

The trigger requires no privilege: any account holder can submit multiple transactions with `max_fee` close to their full balance. It can occur accidentally (a user submitting a batch without realising the total fees exceed their balance) or deliberately (an attacker flooding the mempool with fee-insolvent transactions to crowd out legitimate ones). The nonce-gap window (`max_allowed_nonce_gap`) in the gateway's `validate_nonce` bounds the number of concurrent pending transactions per account, but does not prevent the balance-exhaustion scenario within that window. [6](#0-5) 

---

### Recommendation

Before admitting a new transaction, the gateway (or the mempool's `validate_tx`) should query the mempool for the total `max_possible_fee` already reserved by pending transactions from the same sender and subtract that amount from the committed balance before calling `verify_can_pay_committed_bounds`. Concretely:

```
effective_balance = committed_balance - sum(max_possible_fee(tx) for tx in mempool[sender])
require effective_balance >= max_possible_fee(incoming_tx)
```

This mirrors the recommended fix for the StakingRewards bug (using `transferFrom` to track actual token flow) by making the admission check account for already-committed obligations rather than only the raw committed-state balance.

---

### Proof of Concept

1. Alice holds 100 STRK in the latest committed block.
2. Alice submits **TX1** with `max_fee = 80 STRK`. Gateway calls `verify_can_pay_committed_bounds`: committed balance (100) ≥ 80 → **accepted**, TX1 enters mempool.
3. Alice submits **TX2** with `max_fee = 80 STRK`. Gateway again calls `verify_can_pay_committed_bounds` against the same committed state: balance (100) ≥ 80 → **accepted**, TX2 enters mempool. The 80 STRK already reserved by TX1 is invisible to this check.
4. Block builder executes TX1: fee ≈ 80 STRK deducted; Alice's balance ≈ 20 STRK.
5. Block builder executes TX2: `PostExecutionReport::new` calls `check_can_pay_fee`; balance (20) < fee (80) → `FeeCheckError::InsufficientFeeTokenBalance`; TX2 **reverts** — but it was already admitted and consumed a nonce. [7](#0-6) [8](#0-7) [9](#0-8)

/// Returns the current fee balance and a boolean indicating whether the balance covers the fee.
pub fn get_balance_and_if_covers_fee(
    state: &mut dyn StateReader,
    tx_context: &TransactionContext,
    fee: Fee,
) -> TransactionFeeResult<(Felt, Felt, bool)> {
    let tx_info = &tx_context.tx_info;
    let (balance_low, balance_high) =
        state.get_fee_token_balance(tx_info.sender_address(), tx_context.fee_token_address())?;
    // TODO(Dori,1/10/2023): If/when fees can be more than 128 bit integers, this should be updated.
    let has_sufficient_balance = balance_high > Felt::ZERO || balance_low >= Felt::from(fee.0);
    if !has_sufficient_balance {
        debug!(
            "Fee token balance check: sender_address={}, tx_hash={}, balance={}, expected fee={}",
            tx_info.sender_address(),
            tx_info.transaction_hash(),
            balance_low,
            fee.0
        );
    }
    Ok((balance_low, balance_high, has_sufficient_balance))
}

pub fn verify_can_pay_committed_bounds(
    state: &mut dyn StateReader,
    tx_context: &TransactionContext,
) -> TransactionFeeResult<()> {
    let tx_info = &tx_context.tx_info;
    let committed_fee = tx_context.max_possible_fee();
    let (balance_low, balance_high, can_pay) =
        get_balance_and_if_covers_fee(state, tx_context, committed_fee)?;
    if can_pay {
        Ok(())
    } else {
        Err(match tx_info {
            TransactionInfo::Current(context) => match &context.resource_bounds {
                L1Gas(l1_gas) => TransactionFeeError::GasBoundsExceedBalance {
                    resource: Resource::L1Gas,
                    max_amount: l1_gas.max_amount,
                    max_price: l1_gas.max_price_per_unit,
                    balance: balance_to_big_uint(&balance_low, &balance_high),
                },
                AllResources(bounds) => TransactionFeeError::ResourcesBoundsExceedBalance {
                    bounds: *bounds,
                    balance: balance_to_big_uint(&balance_low, &balance_high),
                },
            },
            TransactionInfo::Deprecated(context) => TransactionFeeError::MaxFeeExceedsBalance {
                max_fee: context.max_fee,
                balance: balance_to_big_uint(&balance_low, &balance_high),
            },
        })
    }
}

    #[sequencer_latency_histogram(GATEWAY_VALIDATE_TX_LATENCY, true)]
    async fn run_validate_entry_point(
        &mut self,
        executable_tx: &ExecutableTransaction,
        skip_validate: bool,
    ) -> StatefulTransactionValidatorResult<()> {
        let only_query = false;
        let charge_fee = enforce_fee(executable_tx, only_query);
        let strict_nonce_check = false;
        let execution_flags =
            ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };

        let account_tx = AccountTransaction { tx: executable_tx.clone(), execution_flags };

        // Build block context.
        let mut versioned_constants = VersionedConstants::get_versioned_constants(
            self.config.versioned_constants_overrides.clone(),
        );
        // The validation of a transaction is not affected by the casm hash migration.
        versioned_constants.disable_casm_hash_migration();

        let mut block_info = self.gateway_fixed_block_state_reader.get_block_info().await?;
        block_info.block_number = block_info.block_number.unchecked_next();
        let block_context = BlockContext::new(
            block_info,
            self.chain_info.clone(),
            versioned_constants,
            BouncerConfig::max(),
        );

        // Move state into the blocking task and run CPU-heavy validation.
        let state_reader_and_contract_manager = self.take_state_reader_and_contract_manager();

        let cur_span = Span::current();
        #[allow(clippy::result_large_err)]
        tokio::task::spawn_blocking(move || {
            cur_span.in_scope(|| {
                let state = CachedState::new(state_reader_and_contract_manager);
                let mut blockifier_validator = StatefulValidator::create(state, block_context);
                blockifier_validator.validate(account_tx)
            })
        })
        .await
        .map_err(|e| StarknetError {
            code: StarknetErrorCode::UnknownErrorCode(
                "StarknetErrorCode.InternalError".to_string(),
            ),
            message: format!("Blocking task join error when running the validate entry point: {e}"),
        })?
        .map_err(|e| StarknetError {
            code: StarknetErrorCode::KnownErrorCode(KnownStarknetErrorCode::ValidateFailure),
            message: e.to_string(),
        })?;
        Ok(())
    }
