Audit Report

## Title
Multiplicative Gas Exhaustion in Public `updateRSETHPrice()` via Per-Asset-Per-NDC `getAssetUnstaking()` Calls - (File: `contracts/NodeDelegator.sol`)

## Summary
`LRTOracle.updateRSETHPrice()` is publicly callable and triggers a chain of nested loops that invoke `NodeDelegator.getAssetUnstaking()` once per supported asset per NDC. Each invocation performs an external `getQueuedWithdrawals()` call and then iterates over all returned withdrawals with additional external calls per strategy. The total gas cost scales multiplicatively with `assets × NDCs × withdrawals × strategies`, and the existing `maxUncompletedWithdrawalCount` cap of 80 does not prevent exhaustion of the 30M block gas limit under normal high-activity protocol conditions.

## Finding Description
The confirmed call chain is:

1. `LRTOracle.updateRSETHPrice()` — `public`, guarded only by `whenNotPaused` [1](#0-0) 
2. → `_getTotalEthInProtocol()` — loops over every supported asset, calling `getTotalAssetDeposits(asset)` per asset [2](#0-1) 
3. → `LRTDepositPool.getAssetDistributionData()` — loops over every NDC in `nodeDelegatorQueue`, calling `getAssetUnstaking(asset)` per NDC [3](#0-2) 
4. → `NodeDelegator.getAssetUnstaking()` — calls `getQueuedWithdrawals(address(this))` (one external call), then iterates over all returned withdrawals with a nested loop calling `strategy.underlyingToken()` and `strategy.sharesToUnderlyingView()` (two external calls per strategy per withdrawal) [4](#0-3) 

The critical inefficiency is that `getQueuedWithdrawals()` is called once per asset per NDC, meaning the same withdrawal data for a given NDC is fetched and decoded `supportedAssets.length` times. With `maxNodeDelegatorLimit = 10` [5](#0-4)  and 5 supported assets, this produces 50 separate `getQueuedWithdrawals()` calls, each returning up to 80 withdrawals.

The protocol's own comment in `LRTUnstakingVault.setMaxUncompletedWithdrawalCount()` acknowledges the gas concern and caps `maxUncompletedWithdrawalCount` at 80, noting "120 is the max number of uncompleted withdrawals that allows us to still perform update rsETH price." [6](#0-5)  However, this comment appears to reason about a single `getAssetUnstaking()` call rather than the full `assets × NDCs` multiplication. With 50 invocations each fetching 80 withdrawals with 2 strategies each, the external call count reaches approximately 50 × (1 + 80 × 2) = 8,050, yielding ~16.9M gas in external call overhead alone before accounting for ABI decoding of large `Withdrawal[]` arrays, memory allocation, and other execution costs — sufficient to approach or exceed the 30M block gas limit.

## Impact Explanation
If `updateRSETHPrice()` reverts out-of-gas, the stored `rsETHPrice` becomes stale. All price-dependent protocol paths — `initiateWithdrawal`, `instantWithdrawal`, `depositAsset`, `depositETH`, `unlockQueue` — read `rsETHPrice` directly, causing incorrect rsETH minting amounts and incorrect withdrawal payouts. This constitutes a temporary freezing of correct protocol operation, matching the allowed impact **Medium — Temporary freezing of funds** and **Medium — Unbounded gas consumption**.

## Likelihood Explanation
No attacker action is required. Normal protocol operation — operators calling `initiateUnstaking()` — fills the withdrawal queue. Any unprivileged EOA calling `updateRSETHPrice()` when the protocol is near capacity triggers the OOG revert. The condition is repeatable and self-sustaining: as long as the withdrawal queue remains full (which is the normal operating state during active unstaking), the public price update path remains broken. The privileged fallback `updateRSETHPriceAsManager()` exists but requires manual manager intervention for every price update cycle.

## Recommendation
1. **Cache `getQueuedWithdrawals()` once per NDC** and compute all asset amounts in a single pass, eliminating the per-asset redundant external call.
2. **Decouple unstaking amounts from the live price-update path**: maintain a running `assetUnstaking` tally updated on `initiateUnstaking` / `completeUnstaking` events (analogous to `stakedButUnverifiedNativeETH`), and read the stored value in `getAssetDistributionData()` instead of calling `getAssetUnstaking()` live.
3. **Revise the `maxUncompletedWithdrawalCount` cap** to account for the full `assets × NDCs` multiplier, not just a single invocation count.

## Proof of Concept
1. Deploy protocol with 5 supported assets, 10 NDCs (`maxNodeDelegatorLimit = 10`), `maxUncompletedWithdrawalCount = 80`.
2. Operators call `initiateUnstaking()` until each NDC holds 80 queued withdrawals with 2 strategies each.
3. Any unprivileged EOA calls `LRTOracle.updateRSETHPrice()`.
4. Execution: `updateRSETHPrice` → `_getTotalEthInProtocol` → 5× `getTotalAssetDeposits` → 5×10 = 50× `getAssetUnstaking` → 50× `getQueuedWithdrawals` (external) + 50×80×2 = 8,000 strategy external calls.
5. Transaction reverts out-of-gas; `rsETHPrice` is not updated.
6. All subsequent `depositAsset`, `initiateWithdrawal`, and `unlockQueue` calls operate on a stale price until a manager manually calls `updateRSETHPriceAsManager()`.

```text
    function getAssetUnstaking(address asset) external view returns (uint256 amount) {
        (IDelegationManager.Withdrawal[] memory queuedWithdrawals, uint256[][] memory withdrawalShares) =
            _getDelegationManager().getQueuedWithdrawals(address(this));

        for (uint256 withdrawalIndex = 0; withdrawalIndex < queuedWithdrawals.length; withdrawalIndex++) {
            IDelegationManager.Withdrawal memory withdrawal = queuedWithdrawals[withdrawalIndex];

            for (uint256 strategyIndex = 0; strategyIndex < withdrawal.strategies.length; strategyIndex++) {
                IStrategy strategy = withdrawal.strategies[strategyIndex];

                address strategyAsset = address(strategy) == address(lrtConfig.beaconChainETHStrategy())
                    ? LRTConstants.ETH_TOKEN
                    : address(strategy.underlyingToken());

                if (strategyAsset != asset) continue;

                uint256 sharesToUnstake = withdrawalShares[withdrawalIndex][strategyIndex];
                amount += strategyAsset == LRTConstants.ETH_TOKEN
                    ? sharesToUnstake
                    : strategy.sharesToUnderlyingView(sharesToUnstake);
            }
        }
    }
```
