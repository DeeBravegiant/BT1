Audit Report

## Title
Stale Chainlink Price Data Accepted Without Staleness Validation in `ChainlinkPriceOracle`, Enabling Incorrect rsETH Price Computation - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary
`ChainlinkPriceOracle.getAssetPrice()` discards the `updatedAt` timestamp and `answeredInRound` fields returned by `latestRoundData()`, accepting arbitrarily stale prices without reversion. This stale price flows through `LRTOracle._getTotalEthInProtocol()` into `_updateRsETHPrice()`, which sets the canonical `rsETHPrice` used to mint rsETH for depositors. Because `updateRSETHPrice()` is public and permissionless, any caller can lock in a deflated `rsETHPrice` during a Chainlink staleness window, allowing subsequent depositors to receive excess rsETH at the expense of all existing holders.

## Finding Description
`ChainlinkPriceOracle.getAssetPrice()` at line 52 reads only the `price` field:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

The `updatedAt` and `answeredInRound` return values are silently discarded. No check of the form `require(block.timestamp - updatedAt <= MAX_STALENESS)` or `require(answeredInRound >= roundId)` exists.

By contrast, `ChainlinkOracleForRSETHPoolCollateral.getRate()` in the same repository performs both checks at lines 30–31:

```solidity
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
```

The call chain is:
1. `updateRSETHPrice()` (public, no access control, line 87) → `_updateRsETHPrice()` (line 214)
2. `_updateRsETHPrice()` calls `_getTotalEthInProtocol()` (line 231)
3. `_getTotalEthInProtocol()` calls `getAssetPrice(asset)` for every supported LST (line 339), which delegates to `ChainlinkPriceOracle.getAssetPrice()`
4. The stale price is multiplied by `totalAssetAmt` and summed into `totalETHInProtocol` (line 343)
5. `newRsETHPrice` is computed as `(totalETHInProtocol - protocolFeeInETH).divWad(rsethSupply)` (line 250) and stored in `rsETHPrice` (line 313)

The downside-protection guard at lines 270–281 is insufficient: it only triggers a pause when `pricePercentageLimit > 0` and the price drop exceeds that limit relative to `highestRsethPrice`. If `pricePercentageLimit` is zero (its default unset value), no pause occurs at all. Even when set, a staleness-induced underestimation smaller than the configured threshold passes through silently.

## Impact Explanation
**Theft of unclaimed yield (High).** When a Chainlink LST/ETH feed is stale with a price below the true current value (the normal case for appreciating LSTs at the end of a 24-hour heartbeat window), an attacker calls the public `updateRSETHPrice()` to lock in a deflated `rsETHPrice`. The attacker then immediately calls `LRTDepositPool.depositAsset()`, receiving more rsETH than the true exchange rate entitles them to. When the Chainlink feed resumes and `rsETHPrice` is corrected upward, the attacker's excess rsETH represents a direct dilution of all pre-existing rsETH holders' proportional claim on the protocol TVL — constituting theft of unclaimed yield accrued by those holders.

## Likelihood Explanation
Chainlink LST/ETH feeds (e.g., stETH/ETH) operate with 24-hour heartbeat intervals and 0.5% deviation thresholds. During low-volatility periods, feeds routinely go the full heartbeat without an update. LSTs continuously accrue staking yield, so the last reported price is structurally below the true value at the end of any heartbeat window. No privileged access is required: `updateRSETHPrice()` is callable by any EOA or contract. The attacker needs only to monitor the Chainlink feed's `updatedAt` timestamp off-chain and call `updateRSETHPrice()` followed by `depositAsset()` in the same block or transaction sequence. This is repeatable every heartbeat cycle.

## Recommendation
Add staleness and round-completeness checks to `ChainlinkPriceOracle.getAssetPrice()`, mirroring the pattern already used in `ChainlinkOracleForRSETHPoolCollateral.getRate()`:

```solidity
(uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
    priceFeed.latestRoundData();
require(answeredInRound >= roundId, "Stale price");
require(updatedAt != 0, "Incomplete round");
require(block.timestamp - updatedAt <= maxStaleness[asset], "Price too old");
require(price > 0, "Invalid price");
```

Introduce a per-asset configurable `maxStaleness` mapping (set by `LRTManager`) to accommodate different Chainlink feed heartbeat intervals across supported LSTs.

## Proof of Concept
**Foundry fork test outline:**

1. Fork mainnet at a block where the stETH/ETH Chainlink feed's `updatedAt` is near its 24-hour heartbeat limit.
2. Deploy/use the existing `ChainlinkPriceOracle` with the stETH feed registered.
3. Warp `block.timestamp` forward by 23+ hours (within the heartbeat window, so the feed does not revert, but the price is stale).
4. Record `rsETHPrice` before the attack.
5. Call `LRTOracle.updateRSETHPrice()` from an unprivileged address — observe it succeeds and sets a deflated `rsETHPrice`.
6. Call `LRTDepositPool.depositAsset(stETH, largeAmount)` from the attacker — observe the attacker receives rsETH minted at the deflated price.
7. Warp forward until the Chainlink feed updates; call `updateRSETHPrice()` again — observe `rsETHPrice` corrects upward.
8. Assert that the attacker's rsETH balance, valued at the corrected price, exceeds the ETH value they deposited, with the difference sourced from dilution of pre-existing holders' share.
