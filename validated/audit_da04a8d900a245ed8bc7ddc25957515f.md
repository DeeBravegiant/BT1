Audit Report

## Title
Unvalidated Chainlink Price in `ChainlinkPriceOracle.getAssetPrice()` Enables Corrupted rsETH Price and Excess Minting - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary
`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards `roundId`, `updatedAt`, and `answeredInRound`, performing no staleness, round-completeness, or zero-price validation. The unvalidated price flows through `LRTOracle._getTotalEthInProtocol()` into `_updateRsETHPrice()`, which is triggered by the public `updateRSETHPrice()`. When a Chainlink feed returns a stale or zero price, the stored `rsETHPrice` is set below its true value, allowing an attacker to mint excess rsETH at the expense of existing holders.

## Finding Description

`ChainlinkPriceOracle.getAssetPrice()` fetches the price with no guards:

```solidity
// contracts/oracles/ChainlinkPriceOracle.sol:52-54
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
``` [1](#0-0) 

The sister contract `ChainlinkOracleForRSETHPoolCollateral.getRate()` explicitly enforces all three checks that are missing here:

```solidity
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
``` [2](#0-1) 

The unvalidated price is consumed by `_getTotalEthInProtocol()`: [3](#0-2) 

Which feeds `_updateRsETHPrice()` to compute and store `rsETHPrice`: [4](#0-3) 

`updateRSETHPrice()` is public and permissionless: [5](#0-4) 

The corrupted `rsETHPrice` is then used to compute rsETH minted per deposit: [6](#0-5) 

**Partial mitigation analysis — `pricePercentageLimit`:** `_updateRsETHPrice()` contains downside protection: if `newRsETHPrice` drops below `highestRsethPrice` by more than `pricePercentageLimit`, the protocol pauses instead of storing the bad price. [7](#0-6) 

However, this mitigation fails in two realistic scenarios:
1. `pricePercentageLimit` is initialized to `0` (its default value), which disables the check entirely (`pricePercentageLimit > 0` is false).
2. Even when set, a stale price that deviates within the configured threshold passes through silently and corrupts `rsETHPrice`. [8](#0-7) 

## Impact Explanation

When a Chainlink LST/ETH feed returns a stale or zero price, `_getTotalEthInProtocol()` underestimates the protocol's TVL. The resulting `rsETHPrice` is stored below its true value. Because `rsethAmountToMint` is inversely proportional to `rsETHPrice`, a depositor acting immediately after the corrupted price update receives more rsETH than their deposit is worth. When the oracle recovers and `rsETHPrice` is corrected upward, the attacker's rsETH is worth more than deposited — existing holders are diluted.

**Impact: High — Theft of unclaimed yield from existing rsETH holders.**

## Likelihood Explanation

Chainlink feeds go stale during network congestion, oracle downtime, or circuit-breaker events — these are documented, recurring occurrences. `updateRSETHPrice()` requires no privilege. An attacker can monitor feeds off-chain, wait for a stale or anomalous reading, call `updateRSETHPrice()` to lock in the bad price, then immediately call `depositAsset()` to capture inflated rsETH. No privileged access, no flash loan, and no oracle manipulation by the attacker is required — only observation and two public calls.

## Recommendation

Add staleness, round-completeness, and non-zero/non-negative checks to `ChainlinkPriceOracle.getAssetPrice()`, mirroring the pattern already used in `ChainlinkOracleForRSETHPoolCollateral`:

```solidity
(uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
    priceFeed.latestRoundData();

if (answeredInRound < roundId) revert StalePrice();
if (updatedAt == 0) revert IncompleteRound();
if (price <= 0) revert InvalidPrice();
if (block.timestamp - updatedAt > MAX_STALENESS) revert StalePrice();

return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

Additionally, ensure `pricePercentageLimit` is set to a non-zero value at initialization, and consider adding a per-asset zero-price guard inside `_getTotalEthInProtocol()` as defense-in-depth.

## Proof of Concept

1. Deploy a mock Chainlink aggregator for a supported LST (e.g., stETH) that returns `updatedAt = block.timestamp - 25 hours` and `answeredInRound < roundId` (simulating a stale feed), with a price 5% below the true value.
2. Confirm `pricePercentageLimit` is 0 (default) or set it to 10% so the 5% drop passes through.
3. Call `LRTOracle.updateRSETHPrice()` as an unprivileged EOA — it succeeds and stores the depressed `rsETHPrice`.
4. Call `LRTDepositPool.depositAsset(stETH, largeAmount, 0, "")` — `getRsETHAmountToMint` divides by the artificially low `rsETHPrice`, minting excess rsETH to the attacker.
5. Restore the mock feed to the correct price and call `updateRSETHPrice()` again — `rsETHPrice` corrects upward.
6. Verify the attacker's rsETH balance is worth more ETH than deposited, at the expense of existing holders (their share of TVL is diluted).
