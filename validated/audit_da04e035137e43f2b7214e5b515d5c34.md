Audit Report

## Title
Daily Mint Limit Exhaustion DoS — Any Depositor Can Block All Pool Deposits for 24 Hours - (File: contracts/pools/RSETHPoolV2.sol, RSETHPoolV2ExternalBridge.sol, RSETHPoolV3.sol, RSETHPoolV3ExternalBridge.sol, RSETHPoolV3WithNativeChainBridge.sol)

## Summary
The `limitDailyMint` modifier in all five L2 pool contracts enforces a shared global `dailyMintAmount` counter against `dailyMintLimit` with no per-user sub-limit. Any single depositor can exhaust the entire daily quota in one transaction, causing all subsequent `deposit()` calls from any address to revert with `DailyMintLimitExceeded` for up to 24 hours. The attacker's principal is fully preserved as wrsETH; the only cost is the `feeBps` fee.

## Finding Description
All five pool contracts implement an identical `limitDailyMint` modifier pattern. In `RSETHPoolV2.sol` (L72–94), the modifier computes `rsETHAmount` for the incoming deposit, resets `dailyMintAmount` to zero if a new calendar day has begun, then checks and increments the global accumulator:

```solidity
if (dailyMintAmount + rsETHAmount > dailyMintLimit) {
    revert DailyMintLimitExceeded();
}
dailyMintAmount += rsETHAmount;
```

The same pattern appears verbatim in `RSETHPoolV2ExternalBridge.sol` (L104–126), `RSETHPoolV3.sol` (L96–125), `RSETHPoolV3ExternalBridge.sol` (L130–159), and `RSETHPoolV3WithNativeChainBridge.sol` (L108–137). The `deposit()` functions in all contracts are public and gated only by `whenNotPaused` and `nonReentrant` — no role check, no per-user cap, no minimum deposit size enforced by the rate-limit logic.

**Exploit flow:**
1. At the start of a new day (`currentDay > lastMintDay`), `dailyMintAmount` resets to 0.
2. Attacker calls `deposit{value: X}()` where X is sized so that `rsETHAmount ≈ dailyMintLimit`.
3. `dailyMintAmount` reaches `dailyMintLimit` in a single transaction.
4. Every subsequent `deposit()` call from any address reverts with `DailyMintLimitExceeded` until the next day.
5. Attacker holds wrsETH worth ~X ETH minus the fee; principal is intact.

No existing guard prevents this: `nonReentrant` only blocks re-entrancy within a single call, `whenNotPaused` requires admin action to trigger, and there is no per-address accounting anywhere in the modifier.

## Impact Explanation
**Medium — Temporary freezing of funds (deposit path).** The L2 pool `deposit()` functions are the sole on-ramp for users to obtain wrsETH on each L2 chain. Exhausting `dailyMintLimit` blocks every user from depositing for up to 24 hours per attack cycle. The freeze is repeatable indefinitely. This maps directly to the allowed impact "Medium. Temporary freezing of funds."

## Likelihood Explanation
**Medium.** The attacker must hold capital equal to the `dailyMintLimit` in ETH (or a supported token). On L2 chains, gas costs are negligible. The recurring cost is only `feeBps` of the deployed capital per day (e.g., at 10 bps, ~3.65% annualized). The attacker retains wrsETH, so net economic loss is only the fee. A motivated competitor or adversary with sufficient capital can sustain this indefinitely. No special privileges, no oracle manipulation, and no victim mistakes are required — a single public `deposit()` call suffices.

## Recommendation
1. **Per-address sub-limits**: Track `dailyMintAmountPerUser[msg.sender]` and enforce a per-depositor cap within each daily window so no single address can consume the entire `dailyMintLimit`.
2. **Raise `feeBps` floor**: Ensure the fee is set high enough that exhausting the daily limit is economically unattractive.
3. **Finer-grained windows**: Consider hourly reset windows to reduce the maximum DoS duration from 24 hours to 1 hour.
4. **On-chain monitoring / circuit breaker**: Emit an event when a single deposit consumes more than a configurable fraction (e.g., 50%) of the remaining daily limit, enabling rapid admin response.

## Proof of Concept
**Foundry fork test outline** (any L2 where the pool is deployed):

```solidity
function test_dailyMintExhaustion() public {
    // 1. Warp to start of a new day so dailyMintAmount resets
    vm.warp(pool.startTimestamp() + pool.getCurrentDay() * 1 days + 1 days);

    uint256 limit = pool.dailyMintLimit();
    // 2. Compute ETH needed so rsETHAmount == dailyMintLimit
    uint256 rate = pool.getRate();          // rsETH/ETH rate (1e18-scaled)
    uint256 feeBps = pool.feeBps();
    // amountAfterFee * 1e18 / rate == limit  =>  amount = limit * rate / 1e18 / (1 - feeBps/10000)
    uint256 depositAmount = limit * rate / 1e18 * 10_000 / (10_000 - feeBps) + 1;

    address eve = makeAddr("eve");
    vm.deal(eve, depositAmount);

    // 3. Eve exhausts the daily limit in one call
    vm.prank(eve);
    pool.deposit{value: depositAmount}("ref");

    // 4. Any subsequent deposit reverts
    address alice = makeAddr("alice");
    vm.deal(alice, 1 ether);
    vm.prank(alice);
    vm.expectRevert(RSETHPoolV2.DailyMintLimitExceeded.selector);
    pool.deposit{value: 1 ether}("ref");
}
```

This test is reproducible against all five pool variants by substituting the contract type and selector. The same pattern applies to token deposits in V3 pools via `deposit(address token, uint256 amount, string referralId)`.
