### Title
User-Controlled Zero-Confirmation Threshold Enables MPC Signature for Unconfirmed Bitcoin Transactions - (File: `crates/foreign-chain-inspector/src/bitcoin/inspector.rs`)

### Summary

The `verify_foreign_transaction` flow accepts a user-supplied `BlockConfirmations` value with no minimum enforced at the contract or node layer. An unprivileged caller can submit `confirmations: 0`, causing the MPC network to issue a valid threshold signature attesting to a Bitcoin transaction that has not yet been included in any block. A bridge contract that trusts the MPC attestation without re-checking the confirmation count in the signed payload can be drained via a double-spend: the attacker claims NEAR-side funds using the MPC signature, then the unconfirmed Bitcoin transaction is dropped or reorganized away.

### Finding Description

`BlockConfirmations` is defined as a plain `u64` wrapper with no lower bound: [1](#0-0) 

The JSON schema confirms `minimum: 0.0`, meaning `0` is a valid wire value: [2](#0-1) 

The contract's `verify_foreign_transaction` method performs no validation of the `confirmations` field — it only checks that the domain exists, gas is sufficient, a 1-yoctonear deposit is attached, and the chain is in the supported set: [3](#0-2) 

The request is stored verbatim and forwarded to the MPC nodes. The node's `execute_foreign_chain_request` passes the user-supplied value directly to the inspector without any floor check: [4](#0-3) 

The Bitcoin inspector's confirmation check is:

```rust
let enough_block_confirmations =
    block_confirmations_threshold <= transaction_block_confirmation;
``` [5](#0-4) 

When `block_confirmations_threshold = 0`, the condition `0 <= N` is trivially true for any `N ≥ 0`, including a mempool transaction whose RPC-reported confirmation count is `0`. The inspector proceeds to extract values and the MPC network signs the resulting payload.

The signed payload (`ForeignTxSignPayloadV1`) embeds the original `ForeignChainRpcRequest` including the attacker-chosen `confirmations: 0`: [6](#0-5) 

A bridge contract that trusts the MPC attestation and does not independently re-validate the `confirmations` field in the signed payload will accept this as proof of finality.

### Impact Explanation

**High — Cross-chain verification bypass enabling double-spend conditions.**

The MPC network's `verify_foreign_transaction` feature is explicitly designed for bridge inbound flows (e.g., Omnibridge: foreign chain → NEAR). Its security guarantee is that the MPC network independently verified the transaction reached a sufficient finality level before signing. With `confirmations: 0`, that guarantee is voided: the MPC signature attests to a transaction that is only in the mempool. The attacker can:

1. Broadcast a Bitcoin transaction sending funds to the bridge deposit address.
2. Immediately call `verify_foreign_transaction` with `confirmations: 0`.
3. Receive a valid MPC threshold signature.
4. Submit the signature to the NEAR bridge contract to claim NEAR-side tokens.
5. RBF-replace or simply wait for the Bitcoin transaction to be dropped from the mempool (or reorganized out of a block if it briefly confirmed).
6. Retain both the NEAR-side tokens and the original Bitcoin funds.

This is a direct loss of funds controlled by the bridge contract that relies on the MPC foreign-chain verification flow.

### Likelihood Explanation

**Medium-High.** The attack requires only a 1-yoctonear deposit and standard NEAR transaction gas. No privileged access, no threshold collusion, and no TEE bypass is needed. The only prerequisite is that a bridge contract consuming `verify_foreign_transaction` responses does not independently re-check the `confirmations` value embedded in the signed payload — a reasonable assumption given the MPC network is positioned as the trusted verifier. The `BlockConfirmations(0)` value is accepted by the contract's JSON deserializer without error.

### Recommendation

Enforce a protocol-level minimum confirmation count at the contract layer before the request is stored. For example:

```rust
// In verify_foreign_transaction, before enqueue_yield_request:
if let ForeignChainRpcRequest::Bitcoin(ref btc_req) = request.request {
    const MIN_BITCOIN_CONFIRMATIONS: u64 = 1; // or a configurable governance param
    if btc_req.confirmations.0 < MIN_BITCOIN_CONFIRMATIONS {
        env::panic_str("Bitcoin confirmations below protocol minimum");
    }
}
```

A stronger approach is to make the minimum a governance-controlled parameter stored in `Config` (alongside `sign_call_gas_attachment_requirement_tera_gas`), so participants can vote to raise it without a contract upgrade. The node layer should add a redundant floor check as defense-in-depth, but the contract is the authoritative enforcement point because nodes are not required to be honest individually.

### Proof of Concept

1. Deploy the MPC contract and register Bitcoin as a supported foreign chain.
2. Broadcast a Bitcoin transaction `T` from address `A` to the bridge deposit address. Do **not** wait for any confirmation.
3. Call `verify_foreign_transaction` with:
   ```json
   {
     "request": {
       "domain_id": 0,
       "payload_version": "V1",
       "request": {
         "Bitcoin": {
           "tx_id": "<hex of T>",
           "confirmations": 0,
           "extractors": ["BlockHash"]
         }
       }
     }
   }
   ```
   with `deposit: 1 yoctonear`.
4. The contract stores the request. MPC nodes call `getrawtransaction` on `T`; the RPC returns `confirmations: 0` (mempool). The inspector evaluates `0 <= 0 → true` and proceeds to extract the block hash (which may be absent for a mempool tx, but the check itself passes).
5. The MPC network signs `ForeignTxSignPayloadV1 { request: Bitcoin { confirmations: 0, ... }, values: [...] }`.
6. The `respond_verify_foreign_tx` callback resolves the yield and returns the `VerifyForeignTransactionResponse` to the caller.
7. The caller submits this response to the NEAR bridge contract, which verifies the MPC signature and releases NEAR-side tokens.
8. The attacker RBF-replaces transaction `T` on Bitcoin, recovering the original Bitcoin funds.
9. Net result: attacker holds both the NEAR-side tokens and the original Bitcoin. [7](#0-6) [8](#0-7) [9](#0-8)

        log!(
            "verify_foreign_transaction: predecessor={:?}, request={:?}",
            env::predecessor_account_id(),
            request
        );

        self.check_request_preconditions(
            request.domain_id,
            DomainPurpose::ForeignTx,
            Gas::from_tgas(self.config.sign_call_gas_attachment_requirement_tera_gas),
            MINIMUM_SIGN_REQUEST_DEPOSIT,
        );

        let requested_chain = request.request.chain();
        let supported_chains = self.get_supported_foreign_chains();
        if !supported_chains.contains(&requested_chain) {
            env::panic_str(
                &InvalidParameters::ForeignChainNotSupported {
                    requested: requested_chain,
                }
                .to_string(),
            );
        }

        let callback_gas = Gas::from_tgas(
            self.config
                .return_signature_and_clean_state_on_success_call_tera_gas,
        );

        let request = args_into_verify_foreign_tx_request(request);
        let callback_args = serde_json::to_vec(&(&request,)).unwrap();
        self.enqueue_yield_request(
            method_names::RETURN_VERIFY_FOREIGN_TX_AND_CLEAN_STATE_ON_SUCCESS,
            callback_args,
            callback_gas,
            move |this, id| this.add_verify_foreign_tx_request(request, id),
        );
