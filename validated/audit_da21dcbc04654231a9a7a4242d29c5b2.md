Audit Report

## Title
Unvalidated Chainlink `latestRoundData()` Return Values in `ChainlinkPriceOracle.getAssetPrice()` Allow Stale/Invalid Price to Corrupt rsETH Exchange Rate - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary
`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` and silently discards `roundId`, `updatedAt`, and `answeredInRound`, accepting any returned `price` — including stale, zero, or incomplete-round values — without validation. Because `LRTOracle.updateRSETHPrice()` is `public whenNotPaused`, any unprivileged caller can trigger a price update during a stale window, permanently writing a corrupted `rsETHPrice` to storage. All subsequent deposits use that corrupted rate, enabling over-minting (protocol insolvency) or under-minting (theft of depositor yield).

## Finding Description
`ChainlinkPriceOracle.getAssetPrice()` (L49–55) discards three of the five return values from `latestRoundData()`:

```solidity
// contracts/oracles/ChainlinkPriceOracle.sol L52
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

No check is made that `answeredInRound >= roundId` (round completeness), `updatedAt > 0` (non-incomplete round), or `price > 0` (valid price). The same codebase already implements all three guards in `ChainlinkOracleForRSETHPoolCollateral.getRate()` (L30–32):

```solidity
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
```

The unguarded oracle feeds directly into the publicly callable `updateRSETHPrice()` → `_updateRsETHPrice()` → `_getTotalEthInProtocol()` → `getAssetPrice(asset)` (LRTOracle.sol L339). The computed `totalETHInProtocol` is used to derive and **persist** `rsETHPrice` to storage (LRTOracle.sol L250, L313). This stored price is then used in every deposit calculation (LRTDepositPool.sol L520):

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

The `pricePercentageLimit` guard in `_updateRsETHPrice()` (L252–281) is a partial mitigation only: it defaults to `0` (no protection), and even when set, only catches deviations exceeding the configured threshold — small stale deviations within the threshold are written unconditionally.

## Impact Explanation
**Scenario A — stale price lower than actual:** `rsETHPrice` is written too low. When the feed recovers, `getAssetPrice(asset)` returns the correct higher value while `rsETHPrice` remains stale-low. Deposits compute `amount * correct_high_price / stale_low_rsETHPrice`, minting excess rsETH and diluting existing holders — **protocol insolvency (Critical)**.

**Scenario B — stale price higher than actual:** `rsETHPrice` is written too high. Depositors receive fewer rsETH than entitled — **theft of depositor yield (High)**.

**Scenario C — `price == 0` (deprecated/broken feed, `pricePercentageLimit == 0`):** `totalETHInProtocol` collapses to near-zero, `rsETHPrice` is written near-zero, and all subsequent deposits mint a massive rsETH amount — **protocol insolvency (Critical)**.

## Likelihood Explanation
`updateRSETHPrice()` requires no privileges and is callable by any EOA or contract at any time. Chainlink LST/ETH feeds have documented heartbeat intervals (e.g., 24 h), creating a predictable stale window every cycle. An attacker can monitor on-chain round data and call `updateRSETHPrice()` at the moment a round is stale but before the next heartbeat fires. On L2 deployments, sequencer downtime can extend stale windows for hours. The attack is repeatable every heartbeat cycle and requires no privileged access. Likelihood: **Medium**.

## Recommendation
Mirror the validation already present in `ChainlinkOracleForRSETHPoolCollateral`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);

    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
        priceFeed.latestRoundData();

    if (answeredInRound < roundId) revert StalePrice();
    if (updatedAt == 0)            revert IncompleteRound();
    if (price <= 0)                revert InvalidPrice();

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

Additionally, add a configurable per-feed `maxStaleness` threshold: `if (block.timestamp - updatedAt > maxStaleness) revert StalePrice();`.

## Proof of Concept
1. Assume the stETH/ETH Chainlink feed has a 24-hour heartbeat. Its last update was 23 h 59 m ago (price = 0.998 ETH). True current price is 1.002 ETH (not yet reflected). `pricePercentageLimit` is either 0 or set above 0.4%.
2. Attacker calls `LRTOracle.updateRSETHPrice()` (public, no access control).
3. `_getTotalEthInProtocol()` calls `ChainlinkPriceOracle.getAssetPrice(stETH)` → returns stale 0.998 ETH with no revert.
4. `newRsETHPrice` is computed ~0.4% below fair value and written to `rsETHPrice` storage (LRTOracle.sol L313). The `pricePercentageLimit` check does not trigger if the limit is 0 or above 0.4%.
5. Chainlink heartbeat fires; feed updates to 1.002 ETH.
6. Attacker calls `LRTDepositPool.depositAsset(stETH, largeAmount, 0, "")`.
7. `getRsETHAmountToMint` computes: `largeAmount * 1.002e18 / stale_low_rsETHPrice` → attacker receives ~0.8% excess rsETH relative to fair value.
8. Attacker redeems via the withdrawal path, extracting value from existing holders.

**Foundry fork test plan:** Fork mainnet, set `pricePercentageLimit = 0`, mock the stETH/ETH Chainlink aggregator to return a stale round (`answeredInRound < roundId`, `updatedAt = block.timestamp - 25 hours`), call `updateRSETHPrice()`, assert `rsETHPrice` is written with the stale value, then mock the feed to return the recovered price and assert that a subsequent deposit mints more rsETH than the fair-value amount.
