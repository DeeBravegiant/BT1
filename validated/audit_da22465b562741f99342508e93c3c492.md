Audit Report

## Title
Missing Staleness Check on Cross-Chain Rate Enables Over-Minting of wrsETH — (`contracts/cross-chain/CrossChainRateReceiver.sol`)

## Summary

`CrossChainRateReceiver` records `lastUpdated` on every `lzReceive` call but never reads it in `getRate()`. `RSETHPoolV3.deposit()` consumes this rate unconditionally. If LayerZero message delivery halts (e.g., relayer gas funding lapses), the stale, lower rate persists indefinitely, causing every subsequent deposit to mint more `wrsETH` than the depositor's ETH contribution warrants, permanently diluting existing holders' accrued yield.

## Finding Description

`CrossChainRateReceiver` stores both `rate` and `lastUpdated`: [1](#0-0) 

`lzReceive` writes both on every message: [2](#0-1) 

But `getRate()` returns `rate` with no reference to `lastUpdated`: [3](#0-2) 

`RSETHPoolV3.getRate()` delegates directly to this oracle: [4](#0-3) 

`viewSwapRsETHAmountAndFee` uses this rate to compute the mint amount: [5](#0-4) 

`deposit()` calls `viewSwapRsETHAmountAndFee` and mints the result: [6](#0-5) 

The entire call chain — `deposit` → `viewSwapRsETHAmountAndFee` → `getRate` → `CrossChainRateReceiver.getRate` — never reads `lastUpdated` or compares it against any maximum staleness threshold. The `whenNotPaused` modifier and `dailyMintLimit` guard exist but neither checks rate freshness. The `paused` flag requires privileged admin action and provides no automatic circuit-breaker.

## Impact Explanation

rsETH is a yield-bearing token whose ETH price increases monotonically as staking rewards accrue. A stale rate is therefore always lower than the true current rate. Because the mint formula is `rsETHAmount = amountAfterFee * 1e18 / staleRate`, a lower `staleRate` produces a larger `rsETHAmount`. Every depositor during the stale window receives more `wrsETH` than their ETH contribution warrants. This inflates total `wrsETH` supply without a corresponding increase in backing rsETH, permanently diluting the yield that existing holders have already accrued but not yet claimed — **Theft of unclaimed yield (High)**.

## Likelihood Explanation

LayerZero relayer liveness depends on continuous gas funding by the operator. A funding lapse, relayer misconfiguration, or sustained network congestion on the destination chain can halt message delivery for hours or days without any on-chain signal. No attacker action is required to trigger the condition; it arises from ordinary operational failure. Any unprivileged depositor calling `deposit()` during the stale window exploits the mispricing automatically and repeatedly.

## Recommendation

Add a configurable `maxStaleness` parameter and revert in `getRate()` if the rate is too old:

```solidity
uint256 public maxStaleness; // e.g. 24 hours

function getRate() external view returns (uint256) {
    require(block.timestamp - lastUpdated <= maxStaleness, "Rate is stale");
    return rate;
}
```

This ensures that if the LayerZero relayer stops, deposits revert rather than proceeding with a mispriced rate. `maxStaleness` should be settable by the owner and initialized to a value consistent with the expected update frequency.
