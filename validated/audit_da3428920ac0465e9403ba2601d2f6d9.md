### Title
Unrestricted Direct Deposits Enable Griefing of Legacy-Mode Withdrawal Transactions - (File: eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Entities.hs)

### Summary

In the Dijkstra era, any transaction sender can include `DirectDeposits` in their transaction body to add ADA to any registered staking account without authorization from the account holder. When a victim's transaction uses legacy Plutus scripts (PlutusV1/V2), the `ENTITIES` rule enforces that the withdrawal amount must **exactly equal** the account balance (`IncompleteWithdrawals` check). An attacker can front-run the victim's withdrawal transaction by depositing a tiny amount (e.g., 1 lovelace) to the target account, causing the victim's transaction to fail because the specified withdrawal amount no longer matches the updated balance.

### Finding Description

The Dijkstra era introduces two interacting features whose combination creates a griefing path:

**Feature 1 — Unrestricted Direct Deposits:**
Any transaction can include a `dtbrDirectDeposits :: !DirectDeposits` field (a `Map AccountAddress Coin`) in its body. The `dijkstraEntitiesTransition` rule applies these deposits unconditionally to any registered account, with no authorization check from the target account holder. The only validation is `directDepositsMissingAccounts`, which merely confirms the target account is registered:

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Entities.hs:211-216
let directDeposits = tx ^. bodyTxL . directDepositsTxBodyL
    accountsAfterCerts = certStateAfterCerts ^. certDStateL . accountsL
failOnJust (directDepositsMissingAccounts directDeposits accountsAfterCerts) $
  injectFailure . DirectDepositsToMissingAccounts
pure $ certStateAfterCerts & certDStateL . accountsL %~ applyDirectDeposits directDeposits
```

`applyDirectDeposits` simply adds the deposited amount to the account balance:

```haskell
-- libs/cardano-ledger-core/src/Cardano/Ledger/State/Account.hs:290-298
applyDirectDeposits (DirectDeposits dd) =
  updateAccountBalances
    (\amount account -> addCompactCoin amount (account ^. balanceAccountStateL))
    dd
```

**Feature 2 — Dual Withdrawal Validation Modes:**
The `validateWithdrawals` function in the `ENTITIES` rule operates in two modes controlled by `eePlutusLegacyMode`:

- **Legacy mode** (triggered when the transaction spends a PlutusV1/V2 script): calls `withdrawalsThatDoNotDrainAccounts`, which requires `withdrawalAmount == fromCompact (account ^. balanceAccountStateL)` — an exact match.
- **Non-legacy mode**: calls `withdrawalsThatExceedAccountBalance`, which only requires `withdrawalAmount <= balance`.

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Entities.hs:225-243
validateWithdrawals legacyMode network withdrawals accounts = do
  missingWithdrawals <-
    if legacyMode
      then do
        let (missingWithdrawals, incompleteWithdrawals) =
              case withdrawalsThatDoNotDrainAccounts withdrawals network accounts of
                Nothing -> (Map.empty, Map.empty)
                Just (missing, incomplete) -> (unWithdrawals missing, incomplete)
        failOnNonEmptyMap incompleteWithdrawals IncompleteWithdrawals
        ...
```

The exact-drain check is confirmed in `withdrawalsThatDoNotDrainAccounts`:

```haskell
-- libs/cardano-ledger-core/src/Cardano/Ledger/State/Account.hs:206-210
withdrawalsThatDoNotDrainAccounts =
  categorizeWithdrawals
    ( \withdrawalAmount account ->
        withdrawalAmount == fromCompact (account ^. balanceAccountStateL)
    )
```

**The Interaction:**
Within `dijkstraEntitiesTransition`, withdrawals are validated and applied *before* direct deposits are applied (lines 201–216). This means that within a single transaction, direct deposits do not affect withdrawal validation. However, across transactions (within the same block or across blocks), an attacker's direct deposit processed before the victim's transaction permanently changes the victim's account balance, invalidating any pending legacy-mode withdrawal that specifies the old exact balance.

### Impact Explanation

This is a **Medium** impact finding. An attacker-controlled transaction modifies the victim's reward/account balance (the source of withdrawals) outside the victim's control, causing the victim's withdrawal to fail with `IncompleteWithdrawals`. This falls under: *"Attacker-controlled transactions... modify... rewards... or withdrawals outside design parameters."*

Specifically:
- Any user whose withdrawal transaction uses a legacy Plutus script (PlutusV1/V2) and specifies the exact current balance is vulnerable.
- The attacker can continuously re-grief the victim each time they resubmit, as long as the attacker is willing to spend 1 lovelace per attack.
- For DApps whose Plutus script logic is hardcoded to withdraw a specific amount (e.g., computed off-chain at transaction construction time), this can render the script permanently non-executable without redeployment.

Funds are not directly stolen, but withdrawals are blocked outside design parameters.

### Likelihood Explanation

The attack is cheap (1 lovelace per grief), requires no special privilege, and is executable by any transaction sender. The attacker only needs to know the victim's staking account address (which is public on-chain) and observe a pending legacy-mode withdrawal transaction in the mempool. The attack can be repeated indefinitely at minimal cost.

### Recommendation

1. **Require authorization for direct deposits**: Add a witness requirement so that direct deposits to an account require a signature from the account's staking credential (or a script witness), analogous to how withdrawals require the staking key witness.
2. **Alternatively, relax legacy-mode withdrawal validation**: Change the legacy-mode check from exact-drain (`==`) to at-most-drain (`<=`), consistent with the non-legacy mode. This would eliminate the exact-balance dependency that makes the attack possible.

### Proof of Concept

1. Alice has a registered staking account with balance `B = 5_000_000` lovelace, delegated to a DRep.
2. Alice constructs a Dijkstra transaction spending a PlutusV2 script UTxO, with `withdrawalsTxBodyL = [(aliceAccountAddress, Coin 5_000_000)]`. This triggers legacy mode.
3. Alice's transaction is broadcast to the mempool.
4. An attacker observes Alice's transaction and submits a transaction with `directDepositsTxBodyL = DirectDeposits [(aliceAccountAddress, Coin 1)]`, depositing 1 lovelace to Alice's account.
5. The attacker's transaction is included in a block before Alice's (or in the same block at an earlier position).
6. Alice's account balance is now `5_000_001` lovelace.
7. Alice's transaction is processed: `validateWithdrawals legacyMode` calls `withdrawalsThatDoNotDrainAccounts`, finds `5_000_000 ≠ 5_000_001`, and fails with `IncompleteWithdrawals`.
8. Alice must reconstruct and resubmit her transaction. The attacker can repeat step 4 each time Alice resubmits.
