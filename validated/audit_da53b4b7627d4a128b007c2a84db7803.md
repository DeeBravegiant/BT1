Audit Report

## Title
Missing Chainlink Staleness Validation Enables rsETH Over-Minting via Deflated `rsETHPrice` - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary
`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards `updatedAt` and `answeredInRound`, accepting any stale price without validation. Because `updateRSETHPrice()` is an unrestricted public function, an unprivileged attacker can force a deflated `rsETHPrice` into storage whenever any supported LST feed goes stale, then deposit a fresh-feed asset to receive more rsETH than the deposited value warrants, diluting all existing rsETH holders.

## Finding Description
`ChainlinkPriceOracle.getAssetPrice()` binds only the second return value of `latestRoundData()`:

```solidity
// contracts/oracles/ChainlinkPriceOracle.sol L52
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
``` [1](#0-0) 

`updatedAt` (slot 4) and `answeredInRound` (slot 5) are never read, so there is no heartbeat check and no round-completeness check.

This price feeds `LRTOracle._getTotalEthInProtocol()`, which sums `assetAmount × assetPrice` across all supported LSTs: [2](#0-1) 

That total is used in `_updateRsETHPrice()` to compute and store `rsETHPrice`: [3](#0-2) 

`updateRSETHPrice()` is callable by any address: [4](#0-3) 

The stored `rsETHPrice` is then used as the denominator when computing rsETH to mint: [5](#0-4) 

**Exploit flow:**
1. A supported LST feed (e.g. stETH/ETH) goes stale at a price below the true market price.
2. Attacker calls `updateRSETHPrice()` — no role required. `_getTotalEthInProtocol()` uses the stale low price, deflating `rsETHPrice` written to storage.
3. Attacker calls `depositAsset(cbETH, amount, ...)`. The numerator `lrtOracle.getAssetPrice(cbETH)` is a live call returning the correct cbETH price; the denominator `lrtOracle.rsETHPrice()` is the deflated stored value. The ratio yields more rsETH than the deposited value warrants.
4. When the stale feed recovers and `rsETHPrice` is updated back to its true value, the attacker's excess rsETH is worth more ETH than deposited, at the expense of all prior holders.

**Existing guard analysis:** `_updateRsETHPrice()` contains a downside protection check that auto-pauses the protocol if `newRsETHPrice` drops more than `pricePercentageLimit` below `highestRsethPrice`: [6](#0-5) 

This is a partial mitigation only. It does not prevent the attack when: (a) `pricePercentageLimit` is 0 (unset), in which case `isPriceDecreaseOffLimit` is always `false`; or (b) the staleness-induced price deviation is smaller than the configured limit (the PoC uses ~0.2%, well within any typical 1% threshold). The upside threshold check only reverts on price *increases*, not decreases, so it provides no protection here. [7](#0-6) 

By contrast, `ChainlinkOracleForRSETHPoolCollateral.getRate()` — a different oracle wrapper in the same repo — already performs `answeredInRound < roundID`, `timestamp == 0`, and `ethPrice <= 0` checks, confirming the team is aware of the pattern: [8](#0-7) 

## Impact Explanation
Existing rsETH holders suffer dilution: their proportional claim on the underlying ETH pool shrinks without compensation. This is direct, at-rest fund loss for every holder at the time of the exploit. The magnitude scales with the stale asset's TVL share and the degree of price deviation before the feed recovers. This matches **Critical — direct theft of user funds (at-rest)** in the allowed impact scope.

## Likelihood Explanation
Chainlink feeds have historically gone stale during periods of network congestion or node operator issues. The protocol supports multiple LST assets, each with its own feed; the probability that at least one feed lags during a high-congestion period is non-trivial. The attacker requires no special role, no front-running, and no oracle manipulation — only the ability to observe on-chain that one feed's `updatedAt` is old and to call two public functions (`updateRSETHPrice` then `depositAsset`). The attack is repeatable as long as the feed remains stale. **Likelihood: Low-to-Medium.**

## Recommendation
1. **Add staleness and round-completeness checks in `ChainlinkPriceOracle.getAssetPrice()`**, mirroring the pattern already present in `ChainlinkOracleForRSETHPoolCollateral`, and additionally enforce a configurable heartbeat window:

```solidity
(uint80 roundId, int256 price, , uint256 updatedAt, uint80 answeredInRound)
    = priceFeed.latestRoundData();

require(answeredInRound >= roundId,  "Stale price: incomplete round");
require(updatedAt != 0,              "Stale price: round not complete");
require(block.timestamp - updatedAt <= maxStaleness[asset], "Stale price: heartbeat exceeded");
require(price > 0,                   "Invalid price");
```

2. **Store a per-feed `maxStaleness` value** (e.g. 3600 s for a 1-hour heartbeat feed) set by the LRT manager alongside `updatePriceFeedFor`.

3. **Apply the same fix to `RSETHPriceFeed.latestRoundData()`**, which forwards the raw ETH/USD answer without staleness validation: [9](#0-8) 

## Proof of Concept
**Setup:** Protocol holds 1 000 stETH (stETH/ETH feed = 0.999 stale, real = 1.001) and 1 000 cbETH (cbETH/ETH feed = 1.05, fresh). rsETH supply = 2 049 rsETH. Real `rsETHPrice` ≈ 1.001 ETH.

**Step 1 — Feed goes stale.** stETH/ETH Chainlink feed stops updating. Last reported price: 0.999.

**Step 2 — Attacker calls `updateRSETHPrice()` (public, no role).**
- `_getTotalEthInProtocol()`: `1000 × 0.999 + 1000 × 1.05 = 2049 ETH` (stale stETH price used).
- Real total: `1000 × 1.001 + 1000 × 1.05 = 2051 ETH`.
- `rsETHPrice` written as `2049 / 2049 = 1.000 ETH` instead of real `≈ 1.001 ETH`.

**Step 3 — Attacker calls `depositAsset(cbETH, 100e18, ...)`.**
- `rsethAmountToMint = (100 × 1.05) / 1.000 = 105 rsETH`
- Correct: `(100 × 1.05) / 1.001 ≈ 104.895 rsETH`
- Attacker receives ~0.105 rsETH excess per 100 cbETH deposited.

**Step 4 — Feed recovers.** `updateRSETHPrice()` is called with correct prices. `rsETHPrice` rises back to ~1.001. The attacker's 105 rsETH is now worth more ETH than deposited, at the expense of all prior holders.

**Foundry fork test plan:**
1. Fork mainnet at a block where a supported LST Chainlink feed has a known stale round.
2. Deploy/use the existing `ChainlinkPriceOracle` pointing to that feed.
3. Call `updateRSETHPrice()` and record `rsETHPrice`.
4. Call `depositAsset` with a fresh-feed asset and record rsETH minted.
5. Advance time past the staleness window, call `updateRSETHPrice()` again, and assert the attacker's rsETH balance represents more ETH value than deposited.
