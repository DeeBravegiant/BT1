### Title
`validateWithdrawalsDelegated` Does Not Check DRep Expiry Before Permitting Reward Withdrawals - (`File: eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Ledger.hs`)

---

### Summary

The Conway ledger's withdrawal validation function `validateWithdrawalsDelegated` only checks that a staking key has *some* DRep delegation record, but never checks whether the delegated DRep is expired. Because `checkDRepRegistered` in the `DELEG` rule also omits an expiry check, an unprivileged user can delegate to an already-expired DRep and then withdraw accumulated rewards, bypassing the governance-participation requirement that the delegation check is designed to enforce.

---

### Finding Description

Conway requires that a staking key be delegated to a DRep before its reward account can be drained via a withdrawal. This is enforced by `validateWithdrawalsDelegated`:

```haskell
isNotDRepDelegated keyHash = isNothing $ do
  accountState <- lookupAccountState (KeyHashObj keyHash) accounts
  accountState ^. dRepDelegationAccountStateL
```

The predicate returns `True` (i.e., "not delegated") only when `dRepDelegationAccountStateL` is `Nothing`. It does **not** inspect the expiry field of the referenced DRep. An expired DRep is still present in `vsDRepsL` and still has a non-`Nothing` delegation record, so the check passes. [1](#0-0) 

The upstream delegation certificate rule (`checkDRepRegistered` in `conwayDelegTransition`) has the same gap: it only verifies `Map.member` against `vsDRepsL`, which contains expired DReps, and never calls any expiry predicate. [2](#0-1) 

By contrast, the ratification rule `dRepAcceptedRatio` explicitly excludes expired DReps from governance vote counting:

```haskell
| reCurrentEpoch > drepExpiry drepState -> (yes, tot) -- drep is expired, so we don't consider it
``` [3](#0-2) 

So the system correctly excludes expired DReps from governance power, but the withdrawal gate does not mirror this exclusion. The existing test suite confirms the current behavior explicitly allows withdrawals when delegated to an expired DRep: [4](#0-3) 

A second test case — "Withdraw from a key delegated to a DRep that expired *after* delegation" — is explicitly disabled with `disableInConformanceIt` and references formal-spec issue #635, confirming the formal specification disagrees with the current implementation on this point. [5](#0-4) 

---

### Impact Explanation

The governance-participation requirement for withdrawals is a core Conway design parameter: ADA holders must delegate to an active DRep to withdraw rewards, ensuring that circulating stake participates in on-chain governance. Allowing withdrawals while delegated to an expired DRep (which contributes zero voting power) nullifies this requirement. Any staking key holder can drain their reward account without contributing to governance, modifying withdrawals outside the intended design parameters.

This matches the **Medium** allowed impact: *"Attacker-controlled transactions… modify… withdrawals outside design parameters."*

---

### Likelihood Explanation

The attack requires no privileged access, no key compromise, and no coordination. Any staking key holder can:
1. Delegate to any registered DRep (including one that is already expired, since `checkDRepRegistered` only checks `Map.member`).
2. Wait for the DRep's expiry epoch to pass (or choose a DRep that is already past its `drepExpiry`).
3. Submit a withdrawal transaction — `validateWithdrawalsDelegated` passes because the delegation record is non-`Nothing`.

The `ppDRepActivityL` parameter controls how quickly DReps expire; with a short activity window, many DReps in the wild will be expired at any given time, making suitable targets easy to find.

---

### Recommendation

`validateWithdrawalsDelegated` should additionally verify that the DRep referenced by `dRepDelegationAccountStateL` is not expired. Concretely, after resolving the delegation to a `DRepCredential cred`, look up `cred` in `vsDRepsL` and compare `drepExpiry` against `currentEpoch` (accounting for `vsNumDormantEpochsL` via `vsActualDRepExpiry`). If the DRep is expired, the withdrawal should be rejected with the same `ConwayWdrlNotDelegatedToDRep` failure (or a new, more descriptive failure variant).

Similarly, `checkDRepRegistered` in `conwayDelegTransition` should reject delegation certificates that target an expired DRep, consistent with the ratification rule's treatment of expired DReps as having no governance standing.

---

### Proof of Concept

The existing test at `LedgerSpec.hs:148–173` is a direct proof of concept. The sequence is:

1. Set `ppDRepActivityL = EpochInterval 1` so DReps expire quickly.
2. Register a DRep and let it expire (`passNEpochs 4`; `isDRepExpired drep \`shouldReturn\` True`).
3. Delegate a fresh staking key to the expired DRep via `delegateToDRep cred (Coin 1_000_000) (DRepCredential drep)` — this succeeds because `checkDRepRegistered` only checks `Map.member`.
4. Submit a withdrawal transaction — `submitTx_` succeeds, confirming `validateWithdrawalsDelegated` does not reject the expired-DRep delegation. [4](#0-3)
