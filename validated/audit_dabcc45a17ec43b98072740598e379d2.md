Audit Report

## Title
Missing Staleness and Validity Checks in `ChainlinkPriceOracle.getAssetPrice()` Enables Over-Minting of rsETH via Stale Oracle Price - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary
`ChainlinkPriceOracle.getAssetPrice()` discards every return value from `latestRoundData()` except the raw price, performing no staleness, incomplete-round, or non-positive-price validation. The sibling oracle `ChainlinkOracleForRSETHPoolCollateral.getRate()` enforces all three guards on the identical Chainlink call. Because `ChainlinkPriceOracle` feeds directly into `LRTDepositPool.getRsETHAmountToMint()`, an attacker can deposit an LST while its Chainlink feed is stale at an inflated price, receiving more rsETH than the deposited collateral is worth and diluting the proportional TVL claim of all existing rsETH holders.

## Finding Description

`ChainlinkPriceOracle.getAssetPrice()` reads only the `price` field from `latestRoundData()`:

```solidity
// contracts/oracles/ChainlinkPriceOracle.sol  line 52
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

`roundId`, `startedAt`, `updatedAt`, and `answeredInRound` are silently discarded. [1](#0-0) 

`ChainlinkOracleForRSETHPoolCollateral.getRate()` — used by the RSETHPool family — performs three explicit guards on the same call:

```solidity
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0)            revert IncompleteRound();
if (ethPrice <= 0)             revert InvalidPrice();
``` [2](#0-1) 

The deposit mint calculation in `LRTDepositPool.getRsETHAmountToMint()` is:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [3](#0-2) 

`lrtOracle.getAssetPrice(asset)` delegates directly to `ChainlinkPriceOracle.getAssetPrice()`. [4](#0-3) 

**Exploit flow:**
1. A Chainlink LST/ETH feed (e.g., stETH/ETH) stops updating; `answeredInRound < roundId` becomes true and the last reported price is above the current market rate (e.g., 1.05 ETH/stETH vs. real 0.98 ETH/stETH).
2. `ChainlinkOracleForRSETHPoolCollateral.getRate()` reverts with `StalePrice()` — RSETHPool deposits are blocked.
3. `ChainlinkPriceOracle.getAssetPrice(stETH)` returns 1.05e18 — no revert.
4. Attacker calls `LRTDepositPool.depositAsset(stETH, amount, 0, "")`.
5. `getRsETHAmountToMint()` computes `amount * 1.05e18 / rsETHPrice`, minting ~7% more rsETH than the deposited collateral is worth at the real rate.
6. Attacker holds excess rsETH; existing holders' proportional claim on protocol TVL is diluted.

**Why existing mitigations are insufficient:**

`LRTOracle._updateRsETHPrice()` contains a `pricePercentageLimit` guard that reverts if the computed new rsETH price exceeds the highest historical price by more than the configured threshold. [5](#0-4)  However, this guard applies only when `updateRSETHPrice()` is called — it does not protect the deposit path. The attacker's excess rsETH is minted during `depositAsset()`, before any price update is triggered. Additionally, `pricePercentageLimit` defaults to 0 (no limit) until explicitly set by an admin.

## Impact Explanation

**High — Theft of unclaimed yield.**

An attacker who deposits during a stale-price window receives rsETH backed by less collateral than the minted amount implies. The excess rsETH supply dilutes the proportional ETH claim of all existing rsETH holders, reducing the yield and value accrual they would otherwise receive. The attack is repeatable for every stale window across every supported LST feed. No privileged access is required; any unprivileged depositor can execute it.

## Likelihood Explanation

Chainlink feeds go stale during network congestion, sequencer downtime, or feed deprecation — all documented, real-world events. The `answeredInRound < roundId` condition is a published Chainlink staleness indicator. An attacker needs only to monitor the on-chain state of the relevant Chainlink aggregator and call `depositAsset()` during the stale window. No special permissions, flash loans, or governance capture are required. The attack is executable by any EOA.

## Recommendation

Add the same guards present in `ChainlinkOracleForRSETHPoolCollateral` to `ChainlinkPriceOracle.getAssetPrice()`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);

    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
        priceFeed.latestRoundData();

    if (answeredInRound < roundId) revert StalePrice();
    if (updatedAt == 0)            revert IncompleteRound();
    if (price <= 0)                revert InvalidPrice();

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

Additionally, consider a per-feed `block.timestamp - updatedAt > heartbeat` check, as `answeredInRound < roundId` is a deprecated Chainlink staleness signal in newer aggregator versions.

## Proof of Concept

**Foundry fork test outline:**

1. Fork mainnet at a block where the stETH/ETH Chainlink feed is live.
2. Deploy or reference the existing `ChainlinkPriceOracle` and `LRTDepositPool`.
3. Use `vm.mockCall` to make `AggregatorV3Interface.latestRoundData()` return `(roundId=10, price=1.05e8, startedAt=T, updatedAt=T, answeredInRound=9)` — simulating a stale round where `answeredInRound < roundId`.
4. Record `rsETHPrice` and the attacker's rsETH balance before deposit.
5. Call `LRTDepositPool.depositAsset(stETH, 100e18, 0, "")` as the attacker.
6. Assert that `rsethAmountToMint > 100e18 * realRate / rsETHPrice` — i.e., the attacker received more rsETH than the real collateral value warrants.
7. Confirm that calling `ChainlinkOracleForRSETHPoolCollateral.getRate()` with the same mock reverts with `StalePrice()`, demonstrating the asymmetry.
