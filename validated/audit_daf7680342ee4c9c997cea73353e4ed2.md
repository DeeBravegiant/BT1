### Title
Caller-Controlled `finality` Parameter in `verify_foreign_transaction` Enables Weak-Finality Verification and Double-Spend - (File: `crates/contract/src/lib.rs`)

---

### Summary

Any unprivileged NEAR account can call `verify_foreign_transaction` and supply a caller-controlled `finality` field (e.g., `EvmFinality::Latest`) or a minimal `confirmations` value (e.g., `BlockConfirmations(1)` for Bitcoin). The contract performs no minimum-finality validation before enqueuing the request. MPC nodes then verify and sign a payload attesting to a transaction that may be reorganized out of the foreign chain, enabling a double-spend against any bridge that relies on the returned signature.

---

### Finding Description

`verify_foreign_transaction` is a public, payable contract method callable by any NEAR account for a 1 yoctonear deposit. [1](#0-0) 

The caller supplies a `VerifyForeignTransactionRequestArgs` that includes a `ForeignChainRpcRequest`. For EVM chains this struct carries a caller-controlled `finality: EvmFinality` field; for Bitcoin it carries a caller-controlled `confirmations: BlockConfirmations` field. [2](#0-1) 

The `EvmFinality` enum explicitly exposes a `Latest` variant (the weakest possible finality, representing the chain tip before any safety or finality checkpoint): [3](#0-2) 

The contract's only validation is:
1. Domain exists and has `ForeignTx` purpose.
2. The chain enum variant is in the supported-chains set. [4](#0-3) 

There is **no validation** of the `finality` level or the `confirmations` count. The raw caller-supplied values are forwarded verbatim to the MPC nodes via the yield/resume queue: [5](#0-4) 

MPC nodes then query the foreign chain using whatever finality the caller specified and, if the transaction is found, sign the payload and call `respond_verify_foreign_tx`. [6](#0-5) 

---

### Impact Explanation

The MPC network produces a threshold signature over `ForeignTxSignPayloadV1 { request, values }`, where `request` includes the caller-supplied `finality` field. A bridge contract that trusts this signature has no way to distinguish a `Finalized`-level attestation from a `Latest`-level one — both produce a valid MPC signature. An attacker who obtains a signature over a `Latest`-finality transaction can redeem bridge funds on NEAR before the Ethereum transaction is reorganized, resulting in a direct loss of funds from the bridge. This matches the allowed impact: **High — forged foreign-chain verification / double-spend condition**.

---

### Likelihood Explanation

The attack requires only a 1 yoctonear deposit and a standard NEAR transaction. No privileged role, key material, or threshold collusion is needed. The attacker only needs to submit a transaction on the foreign chain, immediately call `verify_foreign_transaction` with `EvmFinality::Latest`, and wait for the MPC signature before the block is reorganized. On chains with probabilistic finality (e.g., Ethereum pre-checkpoint, Polygon, BNB) or during periods of network instability, reorganizations of 1–2 blocks are realistic. The attack is directly profitable for any bridge with non-trivial TVL.

---

### Recommendation

1. **Enforce a minimum finality level on-chain**: The contract should reject requests with `EvmFinality::Latest` (and optionally `EvmFinality::Safe`) for bridge-critical domains, or require participants to vote on a per-chain minimum finality policy stored in the contract (analogous to how `ForeignChainRpcWhitelist` governs RPC providers).
2. **Enforce a minimum confirmation count for Bitcoin**: Reject `BlockConfirmations` values below a governance-set floor (e.g., 6).
3. **Include finality level in the signed payload hash**: Already done via `ForeignTxSignPayloadV1 { request, values }`, but bridge contracts must be taught to verify the `finality` field in the request before accepting the signature.

---

### Proof of Concept

```
1. Attacker controls address A on Ethereum and address B on NEAR.
2. Attacker sends 10 ETH from A to the bridge deposit address on Ethereum
   (tx_id = 0xdeadbeef...), which lands in the latest block N.
3. Immediately (before block N+1), attacker calls:
     verify_foreign_transaction({
       domain_id: <foreign_tx_domain>,
       payload_version: V1,
       request: ForeignChainRpcRequest::Ethereum(EvmRpcRequest {
         tx_id: 0xdeadbeef...,
         extractors: [BlockHash],
         finality: EvmFinality::Latest,   // <-- attacker-controlled weak finality
       })
     })
   with 1 yoctonear deposit from address B.
4. MPC nodes query eth_getTransactionReceipt against the "latest" tag,
   find the transaction, extract the block hash, and sign the payload.
5. Attacker uses the returned MPC signature to claim 10 NEAR-wrapped ETH
   from the bridge contract on NEAR.
6. Attacker then bribes a miner / uses a private mempool to reorganize
   block N, removing the deposit transaction.
7. The bridge has paid out 10 NEAR-wrapped ETH but the Ethereum deposit
   no longer exists — double-spend complete.
```

The root cause is that `verify_foreign_transaction` in `crates/contract/src/lib.rs` passes the caller-supplied `finality` field through to MPC nodes without any minimum-finality enforcement, directly analogous to `AutoPxGmx.compound` passing caller-supplied `fee` and `amountOutMinimum` to `SWAP_ROUTER.exactInputSingle` without validation.     #[payable]
    pub fn verify_foreign_transaction(&mut self, request: VerifyForeignTransactionRequestArgs) {
        log!(
            "verify_foreign_transaction: predecessor={:?}, request={:?}",
            env::predecessor_account_id(),
            request
        );

        self.check_request_preconditions(
            request.domain_id,
            DomainPurpose::ForeignTx,
            Gas::from_tgas(self.config.sign_call_gas_attachment_requirement_tera_gas),
            MINIMUM_SIGN_REQUEST_DEPOSIT,
        );

        let requested_chain = request.request.chain();
        let supported_chains = self.get_supported_foreign_chains();
        if !supported_chains.contains(&requested_chain) {
            env::panic_str(
                &InvalidParameters::ForeignChainNotSupported {
                    requested: requested_chain,
                }
                .to_string(),
            );
        }

        let callback_gas = Gas::from_tgas(
            self.config
                .return_signature_and_clean_state_on_success_call_tera_gas,
        );

        let request = args_into_verify_foreign_tx_request(request);
        let callback_args = serde_json::to_vec(&(&request,)).unwrap();
        self.enqueue_yield_request(
            method_names::RETURN_VERIFY_FOREIGN_TX_AND_CLEAN_STATE_ON_SUCCESS,
            callback_args,
            callback_gas,
            move |this, id| this.add_verify_foreign_tx_request(request, id),
        );
    }

* **Provider selection**: The request does **not** specify an RPC URL. Nodes deterministically select an allowed provider from the on-chain foreign-chain configurations.
* **Extractor-driven calls**: Each extractor implicitly defines which RPC method(s) it requires. Some extractors require more than one call. For the initial set:

  * **BlockHash (Ethereum)**: `eth_getTransactionReceipt` for `blockHash`, plus `eth_getBlockByNumber` for the finality-head and canonical-chain checks.
  * **BlockHash (Bitcoin)**: `getrawtransaction` (verbose) for the containing `blockhash` and confirmation count, then `getblockheader` + `getblockhash` for the canonical-chain defense-in-depth check.
  * **BlockHash (Starknet)**: `starknet_getTransactionReceipt` for `block_hash` + `finality_status`, then `starknet_getBlockWithTxHashes` for the canonical-chain defense-in-depth check.
  * **SolanaProgramIdIndex / SolanaDataHash**: `getTransaction` to access `transaction.message` + `meta` and instruction data.
* **Shared fetches**: When multiple extractors require the same underlying data, nodes may perform the RPC call once and share the result across extractors.

To keep behavior predictable and auditable, each extractor family must have a fixed, well-specified set of RPC methods it may invoke, with strict timeouts and response-size limits.
