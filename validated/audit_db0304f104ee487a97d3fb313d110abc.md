Audit Report

## Title
ETH Dust Deposits Yield Zero rsETH With No Recovery Path - (File: contracts/pools/RSETHPoolNoWrapper.sol)

## Summary
The `deposit(string)` function in `RSETHPoolNoWrapper` (and analogous functions in `RSETHPool`, `RSETHPoolV3`, `RSETHPoolV3ExternalBridge`, and `RSETHPoolV3WithNativeChainBridge`) checks only that `msg.value != 0`, but does not verify that the computed `rsETHAmount` is non-zero. When a dust ETH amount is deposited whose value truncates to zero via integer division in `viewSwapRsETHAmountAndFee`, the contract accepts the ETH, transfers 0 rsETH to the caller, and retains the ETH permanently with no user-accessible recovery path.

## Finding Description
In `RSETHPoolNoWrapper.deposit()` (L231–244), the only input guard is `if (amount == 0) revert InvalidAmount()`. After calling `viewSwapRsETHAmountAndFee(amount)`, no check is made on the returned `rsETHAmount` before executing `rsETH.safeTransfer(msg.sender, rsETHAmount)`.

In `viewSwapRsETHAmountAndFee` (L277–286):
```solidity
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
```
When `amountAfterFee < rsETHToETHrate / 1e18` (i.e., the deposit is smaller than the rsETH/ETH exchange rate expressed in wei), Solidity integer division truncates to zero. Since rsETH trades above 1 ETH (`rsETHToETHrate > 1e18`), any deposit of 1 wei produces `rsETHAmount = 0`. ERC-20's `transfer(address, 0)` is valid and does not revert, so the call succeeds silently.

The deposited ETH is absorbed into `address(this).balance` with no per-user accounting. All ETH-egress functions — `bridgeAssetsViaNativeBridge()`, `bridgeAssets()`, and `withdrawFees()` — are restricted to `BRIDGER_ROLE` (L382, L431, L450). No user-callable refund or withdrawal function exists. The same pattern is confirmed in `RSETHPool` (L265–278), `RSETHPoolV3` (L246–265), and the other pool variants.

## Impact Explanation
A depositor who sends a dust ETH amount receives 0 rsETH while their ETH is permanently retained by the pool and eventually swept to L1 as undifferentiated liquidity. The depositor has no mechanism to reclaim it. This matches **Low: Contract fails to deliver promised returns** — the contract accepts ETH and promises rsETH in return, but silently delivers nothing. The per-incident financial magnitude is negligible (1–2 wei), consistent with the Low tier.

## Likelihood Explanation
Any unprivileged caller can trigger this by sending 1 wei to `deposit()`. No special state, timing, or privilege is required. The only guard (`amount == 0`) does not prevent 1-wei deposits. Because rsETH consistently trades above 1 ETH, the zero-truncation condition is always reachable for 1-wei deposits. The scenario is unlikely to occur accidentally at scale but is trivially reproducible on demand.

## Recommendation
Add a post-computation zero check immediately after `viewSwapRsETHAmountAndFee` in all affected `deposit()` functions:

```solidity
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
if (rsETHAmount == 0) revert InvalidAmount();
```

Alternatively, enforce a `minAmountToDeposit` threshold (as `LRTDepositPool` does) to reject dust inputs before they reach the rate computation.

## Proof of Concept
1. Deploy or fork with `rsETHToETHrate = 1.05e18`, `feeBps = 0`, `isEthDepositEnabled = true`.
2. Call `RSETHPoolNoWrapper.deposit{value: 1}("")` from any EOA.
3. `viewSwapRsETHAmountAndFee(1)` returns `fee = 0`, `rsETHAmount = 1 * 1e18 / 1.05e18 = 0`.
4. `rsETH.safeTransfer(msg.sender, 0)` succeeds; caller receives 0 rsETH.
5. `address(this).balance` increases by 1 wei; no per-user record is created.
6. Confirm no user-callable function can recover the 1 wei; `withdrawFees` and `bridgeAssets` revert for non-`BRIDGER_ROLE` callers.
