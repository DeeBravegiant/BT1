### Title
Stale Votes from Removed Participants Counted in Unanimity-Required TEE Governance Operations — (`crates/contract/src/lib.rs`, `crates/contract/src/tee/tee_state.rs`)

---

### Summary

After a resharing, the post-resharing cleanup promise (`clean_tee_status`) is spawned as a **detached** promise that can fail silently. If it fails, stale votes from removed participants remain in `launcher_votes` and `measurement_votes`. The functions `vote_remove_launcher_hash` and `vote_remove_os_measurement` count **all** stored votes without filtering for current participants when checking the unanimity condition, unlike `vote_update` which has an explicit defense against this exact scenario. This allows the unanimity condition to be satisfied with fewer current-participant votes than required.

---

### Finding Description

`vote_update` explicitly guards against stale votes with an inline participant filter: [1](#0-0) 

The comment at line 1362 reads: *"This ensures correctness even if the cleanup promise in `MpcContract::vote_reshared()` fails."*

No equivalent guard exists in `vote_remove_launcher_hash`: [2](#0-1) 

The `votes` value returned by `tee_state.vote_launcher(action, &participant)` is produced by `MeasurementVotes::count_votes` (or the equivalent in `LauncherHashVotes`), which counts **every** entry in `vote_by_account` matching the action with no participant-membership filter: [3](#0-2) 

The cleanup that would remove stale votes is a detached promise spawned from `vote_reshared`: [4](#0-3) 

Because it is detached, an out-of-gas failure leaves stale votes in `TeeState.launcher_votes` and `TeeState.measurement_votes` indefinitely: [5](#0-4) 

The same structural gap exists in `vote_remove_os_measurement`, which also relies on `MeasurementVotes::count_votes` without filtering.

---

### Impact Explanation

`vote_remove_launcher_hash` and `vote_remove_os_measurement` require **unanimity** (all current participants) precisely because removing a launcher hash or OS measurement invalidates attestations of nodes still running that configuration. If stale votes from removed participants are counted:

- The unanimity condition `votes >= total_participants` can be satisfied without all current participants voting (stale count inflates `votes` while `total_participants` shrinks after resharing).
- A launcher hash still in use by a current participant can be removed without that participant's consent.
- Nodes whose launcher hash is removed can no longer submit valid attestations, causing them to be evicted from the participant set by `verify_tee`.
- If enough nodes are evicted, the participant set drops below the signing threshold, freezing the MPC network's ability to produce signatures.

This breaks the production safety invariant that unanimous consent of all current participants is required before invalidating any active launcher configuration.

---

### Likelihood Explanation

A Byzantine participant (below the signing threshold) can construct the following sequence without threshold-level collusion:

1. **N=5, T=3.** Participants P1–P4 vote to remove a launcher hash; P5 refuses (4/5 votes, unanimity not met).
2. P3 and P4's TEE attestations expire naturally (7-day window).
3. P1 calls `verify_tee`, which unilaterally triggers a resharing removing P3 and P4 → new set {P1, P2, P5}, N=3.
4. The detached `clean_tee_status` promise fails (e.g., insufficient `clean_tee_status_tera_gas` config, or a transient out-of-gas condition). Stale votes from P3 and P4 remain.
5. P1 calls `vote_remove_launcher_hash` again. `votes` = 4 (P1, P2, P3-stale, P4-stale); `total_participants` = 3. Condition `4 >= 3` is true → launcher hash removed.
6. P5 never voted; the hash it is running is now removed.

The `verify_tee` path requires only a single participant call and naturally-expired attestations — no threshold collusion is needed. The cleanup promise failing is a realistic edge case documented in the codebase itself (the `vote_update` comment acknowledges it).

---

### Recommendation

1. **Add an inline participant filter** in `vote_remove_launcher_hash` and `vote_remove_os_measurement`, mirroring the defense already present in `vote_update`: count only votes whose voter is a member of the current participant set before comparing against `total_participants`.

2. **Apply the same filter** to `vote_code_hash` and `vote_add_launcher_hash` (threshold-based), which have the same structural gap for the threshold check.

3. **Increase `clean_tee_status_tera_gas`** to a value that reliably covers the worst-case cleanup cost, reducing the probability of the detached promise failing.

---

### Proof of Concept

```
State: N=5, T=3, participants = {P1, P2, P3, P4, P5}
Launcher hash H is in the allowed set; P5 is running H.

Step 1: P1, P2, P3, P4 each call vote_remove_launcher_hash(H).
        launcher_votes = {P1:Remove, P2:Remove, P3:Remove, P4:Remove}
        votes=4, total_participants=5 → 4 < 5, not removed.

Step 2: P3 and P4's attestations expire (7-day window elapses).

Step 3: P1 calls verify_tee().
        → resharing triggered, new set = {P1, P2, P5}, N=3.
        → detached clean_tee_status promise spawned.

Step 4: clean_tee_status fails (out-of-gas).
        launcher_votes still = {P1:Remove, P2:Remove, P3:Remove, P4:Remove}

Step 5: P1 calls vote_remove_launcher_hash(H).
        P1's vote replaces its old entry (same value).
        count_votes(Remove) = 4  (P1, P2, P3-stale, P4-stale)
        total_participants   = 3
        4 >= 3 → H is removed from allowed_launcher_images.

Result: P5 never voted. H is removed. P5's next submit_participant_info
        fails attestation. verify_tee evicts P5. Participant set = {P1, P2},
        below threshold T=3 → MPC network cannot sign → funds frozen.
```

    /// Vote to remove a launcher image hash from the allowed set. Requires ALL participants
    /// to vote for removal, since this invalidates attestations of nodes running that launcher.
    #[handle_result]
    pub fn vote_remove_launcher_hash(
        &mut self,
        launcher_hash: LauncherImageHash,
    ) -> Result<(), Error> {
        log!(
            "vote_remove_launcher_hash: signer={}, launcher_hash={:?}",
            env::signer_account_id(),
            launcher_hash,
        );
        self.voter_or_panic();

        let threshold_parameters = self.protocol_state.threshold_parameters_or_panic();

        let participant = AuthenticatedParticipantId::new(threshold_parameters.participants())?;
        let action = LauncherVoteAction::Remove(launcher_hash);
        let votes = self.tee_state.vote_launcher(action, &participant);

        // Removal requires ALL participants to vote
        let total_participants = threshold_parameters.participants().len() as u64;
        if votes >= total_participants {
            let removed = self.tee_state.remove_launcher_image(&launcher_hash);
            log!("launcher hash remove result: {}", removed);
        }

        Ok(())
    }
