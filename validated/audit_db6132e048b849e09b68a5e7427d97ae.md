Audit Report

## Title
Rewards Permanently Frozen When `totalKernelStaked` Drops to Zero Mid-Period — (File: `contracts/KERNEL/KernelDepositPool.sol`)

## Summary

`KernelDepositPool` implements a Synthetix-style staking rewards mechanism where `rewardPerToken()` short-circuits to return the stored accumulator when `totalKernelStaked == 0`, but the `updateReward` modifier unconditionally advances `updatedAt` to `lastTimeRewardApplicable()`. When the last staker exits mid-period via `initiateWithdrawal`, the time elapsed during the zero-staking window is silently consumed on the next interaction, permanently locking the corresponding reward tokens in the contract with no recovery path.

## Finding Description

`rewardPerToken()` at lines 408–414 returns `rewardPerTokenStored` unchanged when `totalKernelStaked == 0`, freezing the accumulator. However, the `updateReward` modifier at lines 232–242 unconditionally executes `updatedAt = lastTimeRewardApplicable()` regardless of whether `totalKernelStaked` is zero. This means that when the next interaction occurs after a zero-staking gap, `updatedAt` is advanced past the entire gap, and the formula `rewardRate * (lastTimeRewardApplicable() - updatedAt)` in `rewardPerToken()` evaluates over zero elapsed time — the gap is erased.

The exploit path:
1. Admin calls `notifyRewardAmount` with stakers present; `rewardRate > 0`, `finishAt = now + duration`, `updatedAt = now`.
2. The last staker calls `initiateWithdrawal`. The `updateReward` modifier runs first (while `totalKernelStaked > 0`), correctly snapshots `rewardPerTokenStored`, sets `updatedAt = now`, then `totalKernelStaked -= _amount` → `totalKernelStaked == 0` (line 326).
3. Time passes. `rewardRate` is non-zero, `finishAt` is in the future, but no accumulation occurs.
4. A new staker calls `stake()`. `updateReward` fires before `totalKernelStaked` is incremented, so `totalKernelStaked` is still 0. `rewardPerToken()` returns `rewardPerTokenStored` unchanged, and `updatedAt` advances to `lastTimeRewardApplicable()`, consuming the entire gap.
5. `rewardRate × elapsed_gap` worth of reward tokens are permanently unallocatable.

The only existing guard — `notifyRewardAmount` reverting when `totalKernelStaked == 0` (line 570) — only prevents starting a new period with no stakers; it does not prevent stakers from exiting mid-period. There is no `sweep`, `rescueTokens`, or equivalent admin function anywhere in the contract. The contract's own NatSpec at lines 18–22 explicitly acknowledges this limitation but relies solely on operational controls.

## Impact Explanation

**Medium — Permanent freezing of unclaimed yield.** Reward tokens transferred into the contract via `notifyRewardAmount` that correspond to the zero-staking interval are permanently irrecoverable. The `getReward()` path at lines 381–390 only transfers `rewards[msg.sender]`, which is only credited via `earned()` → `rewardPerToken()`, and that accumulator never advanced during the zero-staking window. No admin rescue path exists. This matches the allowed impact: *Medium. Permanent freezing of unclaimed yield.*

## Likelihood Explanation

Any unprivileged staker holding the entire staked supply can unilaterally trigger this by calling the public `initiateWithdrawal` function. `initiateWithdrawal` immediately decrements `totalKernelStaked` at line 326 — the withdrawal delay only governs when underlying KERNEL tokens are transferred, not when the accounting drops to zero. The scenario is realistic whenever a dominant staker exits or the pool is small. No attacker coordination or privileged access is required.

## Recommendation

In the `updateReward` modifier, only advance `updatedAt` when `totalKernelStaked > 0`:

```solidity
modifier updateReward(address _account) {
    rewardPerTokenStored = rewardPerToken();
    if (totalKernelStaked > 0) {
        updatedAt = lastTimeRewardApplicable();
    }
    if (_account != address(0)) {
        rewards[_account] = earned(_account);
        userRewardPerTokenPaid[_account] = rewardPerTokenStored;
    }
    _;
}
```

This ensures the time gap during which no one is staked is not silently consumed — the accumulator will catch up correctly once a new staker arrives. Additionally, add an admin token-rescue function for reward tokens that are provably unallocatable after `finishAt` has passed.

## Proof of Concept

```solidity
// 1. Admin sets up reward period with Alice staked
kernelDepositPool.notifyRewardAmount(1_000e18); // 1000 tokens, 30-day duration
// rewardRate = 1000e18 / 30 days

// 2. Alice is the only staker; she initiates withdrawal after 10 days
// updateReward: rewardPerTokenStored advances for 10 days, updatedAt = now
// Then: totalKernelStaked -= aliceAmount → totalKernelStaked == 0
alice.initiateWithdrawal(aliceStake);

// 3. 20 days pass. totalKernelStaked == 0 the entire time.
// ~666e18 tokens accrue but are never credited.

// 4. Bob stakes
bob.stake(1e18);
// updateReward fires: totalKernelStaked still 0 at modifier entry
//   rewardPerToken() → returns rewardPerTokenStored (no accumulation)
//   updatedAt = lastTimeRewardApplicable() = finishAt
// totalKernelStaked += 1e18

// 5. Alice claims ~333e18 (her 10-day share). Bob claims 0.
// ~666e18 reward tokens permanently stuck in the contract.
uint256 stuck = rewardsToken.balanceOf(address(kernelDepositPool))
              - kernelDepositPool.rewards(alice)
              - kernelDepositPool.rewards(bob);
// stuck ≈ 666e18
```
