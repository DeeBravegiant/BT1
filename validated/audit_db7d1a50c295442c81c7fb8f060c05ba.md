Audit Report

## Title
Zero rsETH Minted for Dust Deposits Causes Permanent Loss of Depositor Funds - (File: contracts/pools/RSETHPoolV3.sol)

## Summary
All L2 pool deposit functions guard only against `amount == 0` but never verify that the computed `rsETHAmount` is non-zero before proceeding. Integer division in `viewSwapRsETHAmountAndFee` truncates `rsETHAmount` to zero for dust deposits, causing the depositor's ETH to be permanently absorbed into the pool while they receive nothing in return.

## Finding Description
In `RSETHPoolV3.deposit()`, the only input guard is:

```solidity
if (amount == 0) revert InvalidAmount();
```

`viewSwapRsETHAmountAndFee` then computes:

```solidity
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
```

With `rsETHToETHrate ≈ 1.05e18`, any deposit where `amountAfterFee < rsETHToETHrate / 1e18` (i.e., 1 wei with zero fee) produces `rsETHAmount = 0`. The `limitDailyMint` modifier does not block this: `dailyMintAmount + 0 > dailyMintLimit` evaluates false, so execution continues. `wrsETH.mint(msg.sender, 0)` is then called — a no-op — while the deposited ETH is retained in the pool balance and eventually bridged to L1. The same truncation applies in `RSETHPoolV3ExternalBridge.deposit()` and `RSETHPoolNoWrapper.deposit()` (which calls `rsETH.safeTransfer(msg.sender, 0)`, also a no-op).

## Impact Explanation
A depositor who sends a dust ETH amount (e.g., 1 wei) receives 0 rsETH/wrsETH. Their funds are permanently transferred to the pool and redistributed to existing rsETH holders through the bridging and L1 deposit flow. The depositor suffers a complete loss of their deposit with no recourse. This matches the **Low** impact category: *"Contract fails to deliver promised returns, but doesn't lose value"* — the protocol retains the value while the individual depositor is silently shortchanged.

## Likelihood Explanation
Any unprivileged user can trigger this by calling `deposit()` with `msg.value = 1`. No special permissions or preconditions are required. While accidental dust deposits are unlikely in normal usage, the path is fully reachable. A malicious actor could also exploit this to donate ETH to the pool at negligible cost, inflating the rsETH price for existing holders.

## Recommendation
Add an explicit post-computation guard in each `deposit()` function immediately after computing `rsETHAmount`:

```solidity
if (rsETHAmount == 0) revert InvalidAmount();
```

This applies to `RSETHPoolV3.deposit()`, `RSETHPoolV3ExternalBridge.deposit()`, `RSETHPoolNoWrapper.deposit()`, and their token-deposit overloads. Alternatively, enforce a protocol-level minimum deposit amount analogous to `LRTDepositPool.minAmountToDeposit`.

## Proof of Concept
1. `rsETHToETHrate = RSETHPoolV3.getRate()` returns `1.05e18`.
2. Caller invokes `RSETHPoolV3.deposit{value: 1}("")`.
3. `limitDailyMint` modifier calls `viewSwapRsETHAmountAndFee(1)`:
   - `fee = 1 * feeBps / 10_000 = 0`
   - `amountAfterFee = 1`
   - `rsETHAmount = 1 * 1e18 / 1.05e18 = 0`
4. Modifier check: `dailyMintAmount + 0 > dailyMintLimit` → false, execution continues.
5. `amount == 0` check passes (amount is 1).
6. `wrsETH.mint(msg.sender, 0)` — no tokens issued.
7. Pool retains 1 wei ETH; depositor receives nothing.

Foundry fuzz test plan:
```solidity
function testFuzz_dustDepositReceivesZeroRsETH(uint256 dustAmount) public {
    dustAmount = bound(dustAmount, 1, rsETHToETHrate / 1e18); // amounts that truncate to 0
    uint256 balanceBefore = wrsETH.balanceOf(user);
    vm.prank(user);
    pool.deposit{value: dustAmount}("");
    assertEq(wrsETH.balanceOf(user), balanceBefore); // user received nothing
}
```
