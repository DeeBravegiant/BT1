### Title
Silent permanent drop of a co-staged valid transaction when its predecessor is rejected in FIFO mode — (`crates/apollo_mempool/src/fifo_transaction_queue.rs`)

### Summary
When two transactions from the same address (nonce N and nonce N+1) are both staged (popped from the FIFO queue via `get_txs`), and `commit_block` reports nonce N as rejected with no committed nonces for that address, `collect_txs_to_rewind` marks the address as not-to-rewind (because the lowest staged nonce is rejected), then `staged_txs.clear()` silently discards nonce N+1. No code path in FIFO mode ever re-enqueues a pool transaction that is not already in the queue, so nonce N+1 is permanently excluded from sequencing.

### Finding Description
`collect_txs_to_rewind` uses the minimum staged nonce as a proxy for the entire address: if that nonce is rejected, the whole address is excluded from rewind. This is correct for the rejected tx itself, but incorrect for any higher-nonce txs that were co-staged. After `rewind_txs` calls `staged_txs.clear()`, those higher-nonce txs vanish from the staging area. `remove_rejected_txs` then calls `tx_queue.remove_by_address`, which is a no-op for staged txs (already cleared). The pool still holds nonce N+1, but the queue does not, and FIFO mode has no re-enqueue path from pool to queue (the fee-mode gap-closing path at `mempool.rs:670` is explicitly skipped with `if !self.is_fifo()`).

### Impact Explanation
A valid, properly signed transaction submitted through the public RPC/gateway is permanently excluded from block proposals. The sequencer's FIFO queue will never return it via `get_txs`, and it will eventually be evicted (the address enters `accounts_with_gap`) or expire via TTL. This corrupts the proposal's transaction set by omitting a transaction that should have been sequenced.

**Impact category:** High — valid transaction rejected before sequencing.

### Likelihood Explanation
The trigger requires two transactions from the same address to share the same `timestamp` + `block_number` metadata (so both are popped in one `get_txs` call), and the lower-nonce one must fail execution. In Echonet/replay mode, execution failures are a normal occurrence (reverts, insufficient balance, etc.), and multiple txs from the same address in the same source block are common. No privileged access is required; any unprivileged sender on the source chain can produce this state.

### Recommendation
In `collect_txs_to_rewind`, when the lowest staged nonce for an address is rejected and there are higher-nonce staged txs, those higher-nonce txs should still be rewound (pushed back to the front of the queue). The fix is to separate the "skip the rejected tx itself" decision from the "rewind remaining txs" decision: mark the address for rewind but exclude the rejected tx from the rewound set, rather than excluding the entire address.

Alternatively, after `staged_txs.clear()`, iterate over the cleared staged txs and re-insert any non-rejected, non-committed txs back into the queue.

fn test_rewind_preserves_timestamp_order(mut mempool: Mempool) {
    let input1 = add_tx_input!(tx_hash: 1, address: "0x1", tx_nonce: 0, account_nonce: 0);
    let input2 = add_tx_input!(tx_hash: 2, address: "0x1", tx_nonce: 1, account_nonce: 0);
    let input3 = add_tx_input!(tx_hash: 3, address: "0x2", tx_nonce: 0, account_nonce: 0);
    let input4 = add_tx_input!(tx_hash: 4, address: "0x3", tx_nonce: 0, account_nonce: 0);

    mempool.update_tx_block_metadata(tx_hash!(1), tx_metadata(1000, 100));
    mempool.update_tx_block_metadata(tx_hash!(2), tx_metadata(1000, 100));
    mempool.update_tx_block_metadata(tx_hash!(3), tx_metadata(1000, 100));
    mempool.update_tx_block_metadata(tx_hash!(4), tx_metadata(1001, 101));

    // Add tx1, tx2, tx3 (timestamp 1000); tx4 (timestamp 1001) is added after the rewind.
    for input in [&input1, &input2, &input3] {
        add_tx(&mut mempool, input);
    }

    assert_eq!(mempool.resolve_batch_timestamp(), 1000);
    // Fetch tx1 and tx2; leave tx3 in queue.
    get_txs_and_assert_expected(&mut mempool, 2, &[input1.tx, input2.tx.clone()]);

    // Commit only tx1; tx2 is rewound to the front.
    commit_block(&mut mempool, [("0x1", 1)], []);

    // Add tx4 (timestamp 1001) after the rewind.
    add_tx(&mut mempool, &input4);

    // Rewound tx2 is at the front → timestamp must still be 1000, not 1001.
    assert_eq!(mempool.resolve_batch_timestamp(), 1000);
    get_txs_and_assert_expected(&mut mempool, 10, &[input2.tx, input3.tx]);

    assert_eq!(mempool.resolve_batch_timestamp(), 1001);
    get_txs_and_assert_expected(&mut mempool, 1, &[input4.tx]);
}
