### Title
Sub-Transaction ExUnits Excluded from Block-Level and Per-Tx ExUnits Checks in Dijkstra Era â€” (`eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Bbody.hs`)

---

### Summary

In the Dijkstra era, `dijkstraBbodyTransition` delegates the block-level ExUnits check to `Alonzo.validateExUnits`, which calls `foldMap totExUnits txs` over the sequence of top-level transactions. `totExUnits` reads only the top-level transaction's own witness-set redeemers. Sub-transactions (`dtbrSubTransactions`) carry their own witness sets with their own redeemers and ExUnits budgets, which are never summed into this total. The same gap exists in the per-tx check (`Alonzo.validateExUnitsTooBigUTxO`) inside `dijkstraUtxoTransition`. The `SUBUTXO` rule performs no ExUnits check at all. Sub-transaction Plutus scripts are nonetheless executed during the top-level tx's UTXOS phase, consuming real CPU/memory resources against the budgets declared in sub-transaction redeemers.

### Impact Explanation

An unprivileged transaction author can craft a top-level transaction with zero top-level redeemers (`totExUnits = 0`) and embed sub-transactions whose redeemers declare arbitrarily large ExUnits budgets. Both the per-tx check (`ppMaxTxExUnitsL`) and the block-level check (`ppMaxBlockExUnitsL`) pass because neither sees sub-transaction redeemers. The sub-transaction Plutus scripts are still executed with those large budgets during the UTXOS phase. A block containing such transactions can cause nodes to spend far more CPU/memory on script evaluation than the protocol parameters are intended to allow, exceeding intended validation limits.

Additionally, because `alonzoMinFeeTx` also uses `totExUnits tx` for the script-fee component of the minimum fee, sub-transaction script execution costs are not reflected in the required fee, allowing the attacker to underpay for sub-transaction script execution.

**Impact category**: Medium â€” attacker-controlled transactions exceed intended validation limits (`ppMaxTxExUnitsL`, `ppMaxBlockExUnitsL`).

---

### Likelihood Explanation

The attack path is fully unprivileged: any transaction author can construct a `Tx TopTx DijkstraEra` with `dtbrSubTransactions` containing sub-transactions with large redeemer ExUnits. The transaction passes mempool validation (per-tx ExUnits check) and block-body validation (block-level ExUnits check). No governance action, key compromise, or privileged role is required. The Dijkstra era is not yet deployed on mainnet, but the code is present in the repository and the gap is structural.

---

### Recommendation

1. Extend `totExUnits` (or introduce a Dijkstra-specific variant) to recursively sum ExUnits from all sub-transaction redeemers when called on a `Tx TopTx DijkstraEra`.
2. Apply the per-tx ExUnits check (`validateExUnitsTooBigUTxO`) to each sub-transaction individually inside the SUBUTXO rule, or aggregate sub-tx ExUnits into the top-level total before the block-level check.
3. Ensure the minimum fee calculation (`alonzoMinFeeTx` / `dijkstraMinFeeTx`) also includes sub-transaction redeemer ExUnits.
