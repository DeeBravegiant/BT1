Audit Report

## Title
Unprivileged Depositor Can Exhaust Daily Mint Limit to DoS All Deposits — (`contracts/pools/RSETHPoolV3.sol`, `contracts/pools/RSETHPoolV3ExternalBridge.sol`, `contracts/pools/RSETHPoolV3WithNativeChainBridge.sol`)

## Summary
The `limitDailyMint` modifier in all three L2 pool variants tracks cumulative minted rsETH in `dailyMintAmount`, which is only ever incremented on deposit and reset once per calendar day. Any unprivileged depositor with sufficient capital can consume the entire `dailyMintLimit` in a single transaction, causing all subsequent deposits within that 24-hour window to revert with `DailyMintLimitExceeded`. The attacker retains the wrsETH received (a yield-bearing asset), so the attack carries no net financial cost and can be repeated every day indefinitely.

## Finding Description
The `limitDailyMint` modifier is applied to every public `deposit` entry point across all three contracts:

```solidity
modifier limitDailyMint(uint256 amount, address token) {
    ...
    if (currentDay > lastMintDay) {
        lastMintDay = currentDay;
        dailyMintAmount = 0;          // only reset at day boundary
    }
    if (dailyMintAmount + rsETHAmount > dailyMintLimit) {
        revert DailyMintLimitExceeded();
    }
    dailyMintAmount += rsETHAmount;   // only ever increases
    _;
}
```

Confirmed in:
- `RSETHPoolV3.sol` lines 96–125 [1](#0-0) 
- `RSETHPoolV3ExternalBridge.sol` lines 130–159 [2](#0-1) 
- `RSETHPoolV3WithNativeChainBridge.sol` lines 108–137 [3](#0-2) 

Both public deposit functions in each contract are gated by this modifier:
- `deposit(string)` (ETH) [4](#0-3) 
- `deposit(address,uint256,string)` (ERC20) [5](#0-4) 

There is no code path that decrements `dailyMintAmount`. The only function that conceptually reverses a deposit, `swapAssetToPremintedRsETH`, is gated behind `onlyRole(OPERATOR_ROLE)` and does not touch `dailyMintAmount` at all. [6](#0-5) 

`setDailyMintLimit` (admin-only) can raise the ceiling but does not reset `dailyMintAmount`, so it does not prevent the attack from being re-executed after the limit is raised. [7](#0-6) 

**Exploit flow:**
1. Attacker observes `dailyMintLimit = L` and `dailyMintAmount = 0` (start of day).
2. Attacker calls `pool.deposit{value: L_in_ETH}("ref")` — `limitDailyMint` sets `dailyMintAmount = L`.
3. Attacker receives wrsETH worth ≈ L ETH; no economic loss.
4. Every subsequent `deposit` call by any user reverts with `DailyMintLimitExceeded` for the rest of the 24-hour window.
5. At the next day boundary, `dailyMintAmount` resets to 0 and the attacker repeats.

## Impact Explanation
**Medium — Temporary freezing of funds.** All deposit paths are blocked for up to 24 hours across every affected L2 pool. Users who need to enter the protocol during that window are completely denied service. The freeze is self-resolving at the next day boundary but can be repeated every day at zero cost to the attacker. This matches the allowed impact: *Medium. Temporary freezing of funds.*

## Likelihood Explanation
**Medium.** Any address with sufficient capital to reach `dailyMintLimit` can execute this attack. The attacker retains full economic value (wrsETH ≈ ETH in value), so the only barrier is capital availability, not willingness to lose funds. On chains where the daily limit is set conservatively to limit bridge risk, the capital requirement may be modest. The attack is repeatable every 24 hours indefinitely with no cost.

## Recommendation
1. **Decrement `dailyMintAmount` on reverse swaps.** `swapAssetToPremintedRsETH` should reduce `dailyMintAmount` by the corresponding rsETH amount when called.
2. **Rate-limit per depositor** rather than (or in addition to) globally, so a single actor cannot exhaust the shared limit.
3. **Allow the admin to reset `dailyMintAmount`** in an emergency without waiting for the next calendar day.
4. **Consider a per-transaction cap** (maximum single deposit size) to raise the cost of limit exhaustion without removing the global safety rail.

## Proof of Concept
```solidity
// Foundry fork test outline (same pattern for all three pool variants)
function test_dailyMintDoS() public {
    // Setup: pool deployed, dailyMintLimit = 100 ether worth of rsETH
    uint256 limit = pool.dailyMintLimit();

    // Step 1: Attacker deposits exactly the daily limit
    vm.deal(attacker, limit);
    vm.prank(attacker);
    pool.deposit{value: limit}("ref");

    // Step 2: dailyMintAmount == dailyMintLimit
    assertEq(pool.dailyMintAmount(), pool.dailyMintLimit());

    // Step 3: Any subsequent deposit reverts
    vm.deal(victim, 1 ether);
    vm.prank(victim);
    vm.expectRevert(RSETHPoolV3.DailyMintLimitExceeded.selector);
    pool.deposit{value: 1 ether}("ref");

    // Step 4: Attacker holds wrsETH — no economic loss
    assertGt(wrsETH.balanceOf(attacker), 0);

    // Step 5: Warp to next day — limit resets, attack can repeat
    vm.warp(block.timestamp + 1 days);
    assertEq(pool.remainingDailyMintLimit(), pool.dailyMintLimit());
}
```
