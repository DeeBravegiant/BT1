### Title
Unauthenticated CRCAT Spend Triggers Irreversible CATWallet-to-CRCATWallet Conversion with Attacker-Controlled Parameters — (`chia/wallet/wallet_state_manager.py`)

### Summary

An unprivileged attacker who holds any amount of a victim's CAT TAIL can submit a crafted spend bundle that permanently and irreversibly converts the victim's `CATWallet` into a `CRCATWallet` configured with attacker-controlled `authorized_providers` and `proofs_checker`, without any user confirmation. After conversion, the victim's existing plain-CAT coins cannot be spent through the wallet (the CRCAT wallet requires a VC from the attacker's authorized providers), effectively locking the victim's funds.

---

### Finding Description

The vulnerability lives in `handle_cat` in `chia/wallet/wallet_state_manager.py`. When the wallet receives a `CoinState` for a coin whose parent spend is a CAT spend, it calls `CRCAT.get_next_from_coin_spend` to detect whether the output is a CRCAT. If it is, and if the victim has a plain `CATWallet` for the same TAIL, `convert_to_cr` is called unconditionally with parameters extracted directly from the attacker's spend.

**Concrete attack steps:**

1. **Attacker acquires any amount of the victim's CAT TAIL** (e.g., buys on a DEX).

2. **Attacker crafts a plain-CAT spend** whose inner puzzle outputs:
   - A `CREATE_COIN` condition with puzzle hash = `construct_cat_puzzle(CAT_MOD, asset_id, construct_cr_layer(attacker_providers, attacker_proofs_checker, victim_inner_puzzle_hash))` and hint = `victim_inner_puzzle_hash`
   - A `REMARK` condition `(1 victim_inner_puzzle_hash attacker_providers attacker_proofs_checker)` (opcode 1), which is the signal `CRCAT.get_next_from_coin_spend` uses to parse CRCAT parameters from a non-CRCAT parent spend.

   This spend is entirely valid on-chain — it is a standard CAT spend, no VC required.

3. **Victim's wallet receives the `CoinState`** for the output coin and calls `handle_cat`.

**Trace through `handle_cat`** (`wallet_state_manager.py` lines 1228–1312):

- **Line 1248**: `derivation_record` is found because the hint equals `victim_inner_puzzle_hash`, which is in the victim's puzzle store. [1](#0-0) 

- **Line 1258**: `cat_puzzle.get_tree_hash() != coin_state.coin.puzzle_hash` is `True` (the coin is a CRCAT, not a plain CAT), so the special-type branch is entered. [2](#0-1) 

- **Line 1268**: `CRCAT.get_next_from_coin_spend(coin_spend)` parses the REMARK condition from the plain-CAT parent spend and returns a `CRCAT` object with `inner_puzzle_hash = victim_inner_puzzle_hash`, `authorized_providers = attacker_providers`, `proofs_checker = attacker_proofs_checker`. [3](#0-2) 

- **Lines 1280–1289**: The ownership guard checks whether `crcat.inner_puzzle_hash` is in the victim's derivation records. Because the attacker set `inner_puzzle_hash = victim_inner_puzzle_hash`, this check **passes** and execution continues. [4](#0-3) 

- **Lines 1292–1295**: No existing `CRCATWallet` for this TAIL is found (victim only has a `CATWallet`). [5](#0-4) 

- **Lines 1299–1310**: The existing `CATWallet` for this TAIL is found, and `CRCATWallet.convert_to_cr` is called with the attacker's `authorized_providers` and `proofs_checker` — **no user confirmation, no authorization check**. [6](#0-5) 

**`convert_to_cr` effect** (`cr_cat_wallet.py` lines 169–193): The method overwrites the wallet's database record with `WalletType.CRCAT` and the attacker's parameters, and replaces the in-memory wallet object. This is permanent — there is no rollback or reverse conversion. [7](#0-6) 

---

### Impact Explanation

After conversion, the victim's wallet is a `CRCATWallet` configured with attacker-controlled `authorized_providers` and `proofs_checker`. Spending CRCAT coins through this wallet requires a VC issued by one of the `authorized_providers` with the required `proofs_checker` flags. Since the attacker controls these parameters, the victim cannot obtain a valid VC, and their existing plain-CAT coins (which are not CRCATs at all) cannot be spent through the wallet interface. The wallet type corruption is permanent and irreversible within the wallet software.

This satisfies the **High** impact category: corruption of puzzle ownership / wallet sync state with direct security impact, and long-lived inability to spend valid coins.

---

### Likelihood Explanation

- The attacker only needs to hold any nonzero amount of the victim's CAT TAIL (purchasable on any DEX).
- The crafted spend is a valid on-chain transaction requiring no special privileges.
- The victim's wallet processes it automatically upon sync with no user interaction.
- The attack is silent — the victim sees only a wallet-type change notification (`"converted cat wallet to cr"`).

---

### Recommendation

1. **Require explicit user authorization** before calling `convert_to_cr`. The conversion should only be triggered by a user-initiated action (e.g., an RPC call), not automatically during coin sync.
2. **Validate that the received CRCAT coin's puzzle hash matches what the wallet would construct** using the victim's own known `authorized_providers` and `proofs_checker` (if any), rather than blindly trusting parameters extracted from an attacker-supplied spend.
3. At minimum, **log a warning and skip conversion** if no existing CRCAT configuration is known for this TAIL, rather than adopting attacker-supplied parameters.

### Proof of Concept

```python
# Pseudocode test plan
# 1. Setup: victim wallet has CATWallet for TAIL X
# 2. Attacker acquires 1 mojo of TAIL X
# 3. Attacker crafts inner puzzle that outputs:
#    - CREATE_COIN with puzzle_hash = CRCAT(attacker_providers, attacker_checker, victim_ph).puzzle_hash
#      and hint = victim_ph, amount = 1
#    - REMARK (1 victim_ph attacker_providers attacker_checker)
# 4. Attacker submits the CAT spend bundle (no VC needed)
# 5. Block is farmed; victim wallet syncs
# 6. Assert: victim's wallet for TAIL X is now WalletType.CRCAT
# 7. Assert: wallet's authorized_providers == attacker_providers
# 8. Assert: victim cannot spend their original CAT coins (wallet rejects, demands VC)
```

The root cause is that `handle_cat` at line 1306 calls `convert_to_cr` with parameters sourced entirely from an untrusted peer-supplied coin spend, with no user confirmation and no verification that the parameters match any previously user-configured CRCAT policy. [8](#0-7) [7](#0-6)

    @classmethod
    async def convert_to_cr(
        cls,
        cat_wallet: CATWallet,
        authorized_providers: list[bytes32],
        proofs_checker: ProofsChecker,
    ) -> None:
        replace_self = cls()
        replace_self.standard_wallet = cat_wallet.standard_wallet
        replace_self.log = logging.getLogger(cat_wallet.get_name())
        replace_self.log.info(f"Converting CAT wallet {cat_wallet.id()} to CR-CAT wallet")
        replace_self.wallet_state_manager = cat_wallet.wallet_state_manager
        replace_self.info = cls.wallet_info_type(
            cat_wallet.cat_info.limitations_program_hash, None, authorized_providers, proofs_checker
        )
        await cat_wallet.wallet_state_manager.user_store.update_wallet(
            WalletInfo(
                cat_wallet.id(), cat_wallet.get_name(), uint8(WalletType.CRCAT.value), bytes(replace_self.info).hex()
            )
        )
        updated_wallet_info = await cat_wallet.wallet_state_manager.user_store.get_wallet_by_id(cat_wallet.id())
        assert updated_wallet_info is not None
        replace_self.wallet_info = updated_wallet_info

        cat_wallet.wallet_state_manager.wallets[cat_wallet.id()] = replace_self
