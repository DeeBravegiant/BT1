### Title
Missing NEAR Chain Binding in `ForeignTxSignPayload` Enables Cross-Fork Replay of Foreign-Transaction Verification Responses - (File: `crates/near-mpc-contract-interface/src/types/foreign_chain.rs`)

---

### Summary

The `ForeignTxSignPayloadV1` structure, whose hash is the only thing MPC nodes sign when responding to a `verify_foreign_transaction` request, contains no NEAR-chain-specific context (no chain ID, no contract account ID, no NEAR block hash). If NEAR undergoes a hard fork while a `verify_foreign_transaction` request is pending, a valid `VerifyForeignTransactionResponse` produced on one fork can be replayed verbatim on the other fork, causing the bridge to attest the same foreign-chain transaction twice and enabling a double-spend.

---

### Finding Description

When a user calls `verify_foreign_transaction`, the contract stores a pending yield and MPC nodes eventually call `respond_verify_foreign_tx` with a `VerifyForeignTransactionResponse` containing:

```
payload_hash = SHA-256(borsh(ForeignTxSignPayload))
signature    = ECDSA over payload_hash
```

The signed payload is defined as:

```rust
pub struct ForeignTxSignPayloadV1 {
    pub request: ForeignChainRpcRequest,   // foreign-chain tx ID + extractors
    pub values:  Vec<ExtractedValue>,      // values extracted from the foreign chain
}
``` [1](#0-0) 

The hash is computed as:

```rust
pub fn compute_msg_hash(&self) -> std::io::Result<Hash256> {
    let mut hasher = sha2::Sha256::new();
    borsh::BorshSerialize::serialize(self, &mut hasher)?;
    Ok(Hash256(hasher.finalize().into()))
}
``` [2](#0-1) 

Neither the payload struct nor the hash includes any NEAR-specific binding: no NEAR chain ID, no contract account ID, no NEAR block height. The `respond_verify_foreign_tx` function verifies only that the signature is valid against the MPC public key and that the caller is an attested participant: [3](#0-2) 

Because the signed material is entirely foreign-chain-scoped, a `VerifyForeignTransactionResponse` that is valid on NEAR fork A is cryptographically identical and equally valid on NEAR fork B, provided the same pending request exists there.

---

### Impact Explanation

**High — Cross-chain replay causing invalid bridge execution / double-spend.**

After a NEAR hard fork, both forks share the same pre-fork contract state, including all pending `verify_foreign_transaction` yields. A response submitted on fork A (e.g., attesting that a Bitcoin or Ethereum deposit transaction finalized) can be copied and submitted on fork B. The contract on fork B will accept it — the signature verifies correctly against the same MPC public key — and will resolve the pending yield, delivering a second attestation for the same foreign-chain event. Any bridge contract consuming this attestation (e.g., Omnibridge) would credit the deposit a second time, constituting a direct double-spend of bridged funds.

---

### Likelihood Explanation

**Low.** NEAR hard forks are rare. However, the Ethereum PoS transition (the Omni bridge incident cited in the original report) demonstrates that this class of event does occur on production networks. The attack requires no privileged access: any on-chain observer of fork A can copy the `respond_verify_foreign_tx` transaction and replay it on fork B. No threshold collusion, no TEE bypass, and no key material is needed — the attacker only needs to observe a finalized NEAR transaction.

---

### Recommendation

Bind the signed payload to the NEAR chain context by including the NEAR chain ID (or the MPC contract's account ID) in `ForeignTxSignPayloadV1`. For example:

```rust
pub struct ForeignTxSignPayloadV1 {
    pub near_contract_id: AccountId,       // e.g. env::current_account_id()
    pub request:          ForeignChainRpcRequest,
    pub values:           Vec<ExtractedValue>,
}
```

The `near_contract_id` (or an equivalent NEAR-chain discriminator) must be populated at request-creation time and included in the Borsh serialization before hashing. This ensures a signature produced on one NEAR deployment or fork is cryptographically invalid on any other.

---

### Proof of Concept

1. User calls `verify_foreign_transaction` on NEAR mainnet with a Bitcoin deposit tx. The contract stores a pending yield at block `N`.
2. At block `N+k`, NEAR undergoes a hard fork producing chains A and B. Both forks retain the identical pending yield (same contract storage).
3. MPC nodes operating on fork A compute the response: `payload_hash = SHA-256(borsh({request, values}))` and produce a valid ECDSA signature. They call `respond_verify_foreign_tx` on fork A; the yield resolves and the bridge credits the deposit on fork A.
4. An attacker copies the `respond_verify_foreign_tx` call data from fork A and submits it on fork B.
5. `respond_verify_foreign_tx` on fork B:
   - Checks the caller is an attested participant — passes (same participant set pre-fork).
   - Verifies the signature against `payload_hash` — passes (same MPC key, same payload, no NEAR context in the hash).
   - Resolves the pending yield on fork B.
6. The bridge on fork B credits the same deposit a second time. Double-spend achieved with no threshold collusion and no key material beyond what is already public on fork A's chain. [4](#0-3) [5](#0-4)

    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }
