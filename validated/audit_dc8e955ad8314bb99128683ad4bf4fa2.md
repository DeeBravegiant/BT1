Audit Report

## Title
`highestRsethPrice` One-Way Ratchet Enables Repeated Auto-Pause Griefing After Price Drop - (File: `contracts/LRTOracle.sol`)

## Summary

`LRTOracle` maintains `highestRsethPrice` as a monotonically increasing watermark with no admin reset path. After any price drop exceeding `pricePercentageLimit` (e.g., from an EigenLayer slashing event), the downside-protection logic auto-pauses `LRTDepositPool`, `LRTWithdrawalManager`, and `LRTOracle`. Because `highestRsethPrice` is permanently anchored to the old peak, every subsequent call to the public `updateRSETHPrice()` re-triggers the auto-pause after each admin unpause, keeping user deposits and withdrawals frozen indefinitely.

## Finding Description

`_updateRsETHPrice()` updates `highestRsethPrice` only upward:

```solidity
// update highest price if new price exceeds it
if (newRsETHPrice > highestRsethPrice) {
    highestRsethPrice = newRsETHPrice;
}
``` [1](#0-0) 

There is no corresponding function to decrease or reset `highestRsethPrice`. The downside-protection block compares the current price against this permanent peak:

```solidity
if (newRsETHPrice < highestRsethPrice) {
    uint256 diff = highestRsethPrice - newRsETHPrice;
    bool isPriceDecreaseOffLimit =
        pricePercentageLimit > 0 && diff > pricePercentageLimit.mulWad(highestRsethPrice);

    if (isPriceDecreaseOffLimit) {
        if (!lrtDepositPool.paused()) lrtDepositPool.pause();
        if (!withdrawalManager.paused()) withdrawalManager.pause();
        _pause();
        return;
    }
``` [2](#0-1) 

The public entry point is callable by any EOA when the oracle is not paused:

```solidity
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
``` [3](#0-2) 

The `updateRSETHPriceAsManager()` bypass does not help: when called while paused, `_updateRsETHPrice()` still hits the `isPriceDecreaseOffLimit` branch, calls `_pause()` (which returns early because `paused == true`), and returns without updating `highestRsethPrice` or `rsETHPrice`. [4](#0-3) 

**Exploit loop:**
1. EigenLayer slashing drops `newRsETHPrice` more than `pricePercentageLimit` below `highestRsethPrice`.
2. Any caller invokes `updateRSETHPrice()` → `isPriceDecreaseOffLimit = true` → all three contracts auto-pause.
3. Admin calls `unpause()` on `LRTOracle`, `LRTDepositPool`, `LRTWithdrawalManager`.
4. Any EOA immediately calls `updateRSETHPrice()` again. Price is still below the unchanged `highestRsethPrice` → auto-pause fires again.
5. Steps 3–4 repeat indefinitely.

The only admin escape is `setPricePercentageLimit(0)`, which disables downside protection entirely. [5](#0-4) 

## Impact Explanation

Every iteration of the loop freezes `LRTDepositPool` and `LRTWithdrawalManager`, preventing all deposits and pending withdrawal completions. This is **Medium: Temporary freezing of funds** — the freeze is not permanent (admin can disable the guard), but it is indefinitely repeatable by any unprivileged EOA at negligible cost, making recovery operationally infeasible without sacrificing the price-drop protection.

## Likelihood Explanation

EigenLayer slashing is a documented, realistic risk for any restaking protocol. A meaningful slash can drop rsETH price by several percent from its all-time high. Once that threshold is crossed, the griefing loop requires no special privilege — any EOA can call the public `updateRSETHPrice()` to re-trigger the pause after each admin unpause. Gas cost per re-pause is minimal. Likelihood is **Medium**.

## Recommendation

Add an admin-restricted function to reset `highestRsethPrice` to the current `rsETHPrice`, allowing the protocol to re-anchor its downside-protection baseline after a verified slashing event:

```solidity
function resetHighestRsethPrice() external onlyLRTAdmin {
    highestRsethPrice = rsETHPrice;
    emit HighestRsethPriceReset(rsETHPrice);
}
```

This allows the admin to acknowledge the new price floor after a slashing event without disabling the protection entirely.

## Proof of Concept

1. Deploy with `highestRsethPrice = 1.10e18`, `pricePercentageLimit = 5e16` (5%).
2. Simulate slashing: `_getTotalEthInProtocol()` now yields `newRsETHPrice = 1.04e18`.
3. `diff = 0.06e18`; `pricePercentageLimit.mulWad(1.10e18) = 0.055e18`; `0.06e18 > 0.055e18` → `isPriceDecreaseOffLimit = true`.
4. Call `updateRSETHPrice()` → all three contracts pause. `highestRsethPrice` remains `1.10e18`.
5. Admin calls `unpause()` on all three contracts.
6. Any EOA calls `updateRSETHPrice()`. Price is still `1.04e18`; `highestRsethPrice` still `1.10e18`. Same condition fires → all three contracts pause again.
7. Steps 5–6 repeat indefinitely.

**Foundry test sketch:**
```solidity
function testGriefingLoop() public {
    // setup: highestRsethPrice = 1.10e18, pricePercentageLimit = 5e16
    // mock _getTotalEthInProtocol to return value yielding price = 1.04e18
    oracle.updateRSETHPrice(); // triggers auto-pause
    assertTrue(oracle.paused());

    vm.prank(admin);
    oracle.unpause(); lrtDepositPool.unpause(); withdrawalManager.unpause();

    vm.prank(attacker);
    oracle.updateRSETHPrice(); // re-triggers auto-pause
    assertTrue(oracle.paused()); // funds frozen again
}
```
