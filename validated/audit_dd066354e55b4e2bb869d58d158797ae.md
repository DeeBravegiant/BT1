### Title
`respond_verify_foreign_tx` Accepts Attacker-Controlled `payload_hash` Without Binding It to the Pending Request - (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_verify_foreign_tx` verifies that `response.signature` is valid over `response.payload_hash`, but never checks that `response.payload_hash` is the hash of a payload derived from the actual pending `request`. A single Byzantine MPC participant (below the signing threshold) can replay any previously observed valid `(signature, payload_hash)` pair as the response to a different pending foreign-transaction verification request. The contract accepts it, and the caller receives a signed attestation that corresponds to a completely different foreign transaction than the one they requested.

---

### Finding Description

In `respond` (the standard sign path), the contract derives the payload hash directly from the immutable `request` object:

```rust
// crates/contract/src/lib.rs ~line 600
let payload_hash = request.payload.as_ecdsa().expect("Payload is not ECDSA");
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    payload_hash,          // ← taken from the request, not from the response
    &expected_public_key,
)
```

In `respond_verify_foreign_tx`, the analogous check instead trusts the caller-supplied `response.payload_hash`:

```rust
// crates/contract/src/lib.rs ~line 726-734
let payload_hash: [u8; 32] = response.payload_hash.0;   // ← attacker-controlled

near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,          // ← NOT derived from `request`
    &secp_pk,
)
.is_ok()
```

The contract only checks that the signature is mathematically valid over whatever hash the responding node chose to put in `response.payload_hash`. It never verifies that this hash encodes the correct `(request, extracted_values)` tuple for the pending request being resolved.

Because `respond_verify_foreign_tx` is callable by any single attested participant (`assert_caller_is_signer` + `assert_caller_is_attested_participant_and_protocol_active`), no threshold cooperation is required to exploit this.

---

### Impact Explanation

A malicious MPC node observes a previously settled `VerifyForeignTransactionResponse` on-chain — say, one that attests to Bitcoin transaction T1 with extracted block-hash B1, producing `(S1, H1)` where `H1 = SHA-256(borsh(ForeignTxSignPayload::V1 { request: T1, values: [B1] }))`.

When a new `verify_foreign_transaction` request for transaction T2 arrives, the malicious node calls:

```
respond_verify_foreign_tx(request = T2, response = { payload_hash: H1, signature: S1 })
```

The contract:
1. Looks up T2 in `pending_verify_foreign_tx_requests` — found.
2. Verifies `S1` over `H1` against the root public key — valid.
3. Resolves the T2 yield with `{ payload_hash: H1, signature: S1 }`.

The caller's contract receives a `VerifyForeignTransactionResponse` that carries a valid MPC signature, but the signed hash corresponds to T1, not T2. Any downstream bridge or NEAR contract that does not independently re-derive and compare the expected `payload_hash` will treat this as a valid attestation of T2, enabling forged foreign-chain verification and potential double-spend or unauthorized bridge execution.

This matches the allowed impact: **High — forged foreign-chain verification that causes invalid bridge execution or double-spend conditions**.

---

### Likelihood Explanation

- Requires only **one** Byzantine attested participant — well below the signing threshold.
- No cryptographic forgery is needed; the attacker replays a legitimately produced signature from any prior settled request.
- Prior responses are publicly visible on-chain (emitted as promise data / logs), so the attacker has a growing pool of reusable `(S, H)` pairs.
- The attack is silent: the contract emits no error, and the victim caller receives a structurally valid response.

---

### Recommendation

Bind `response.payload_hash` to the pending `request` inside `respond_verify_foreign_tx`. Because the contract does not know the extracted values, the minimal fix is to require the responding node to also supply the extracted values in the response, then recompute and compare the hash on-chain:

```rust
// Pseudocode
let expected_hash = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: request.request.clone(),
    values: response.extracted_values.clone(),   // add to VerifyForeignTransactionResponse
})
.compute_msg_hash()?;

if expected_hash != response.payload_hash {
    return Err(RespondError::PayloadHashMismatch.into());
}
```

Alternatively, mirror the pattern used in `respond`: derive the message to verify from the immutable request fields rather than from the caller-supplied response field.

### Proof of Concept

1. **Setup**: Two pending `verify_foreign_transaction` requests exist — one for Bitcoin tx T1 (already responded to, response `(S1, H1)` visible on-chain) and one for Bitcoin tx T2 (pending).

2. **Attack**: Malicious attested participant calls:
   ```
   respond_verify_foreign_tx(
       request  = VerifyForeignTransactionRequest { request: T2, domain_id: D, ... },
       response = VerifyForeignTransactionResponse { payload_hash: H1, signature: S1 }
   )
   ```

3. **Contract execution** (`crates/contract/src/lib.rs` lines 718–747):
   - `payload_hash = response.payload_hash.0` → `H1` (hash of T1's payload)
   - `verify_ecdsa_signature(S1, H1, root_pk)` → **valid** (S1 was honestly produced over H1)
   - `signature_is_valid = true`
   - `resolve_yields_for(&mut pending_verify_foreign_tx_requests, &T2_request, serialize(response))` → T2's yield resolved with `(S1, H1)`

4. **Result**: T2's caller receives `{ payload_hash: H1, signature: S1 }`. The MPC network has effectively attested to T1's data in response to a T2 query. Any bridge contract that gates fund release on this response without re-deriving the expected hash will be deceived. [1](#0-0) [2](#0-1) [3](#0-2)
