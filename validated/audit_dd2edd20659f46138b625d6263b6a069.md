### Title
Proposal Deposits Permanently Lost When `do_update` Clears All Pending Entries - (File: `crates/contract/src/update.rs`)

### Summary
When any update proposal reaches the signing threshold and `do_update` executes, it unconditionally clears **all** pending proposals via `self.entries.clear()`. The NEAR deposits paid by every other proposer are permanently absorbed into the contract balance with no refund path, because neither the proposer's account ID nor the deposit amount is stored in `UpdateEntry`.

### Finding Description

`propose_update` in `lib.rs` requires a deposit proportional to the size of the proposed update (storage cost). The deposit is accepted and only the excess above `required` is refunded to the caller: [1](#0-0) 

The deposit for the `required` portion is retained by the contract. When any proposal reaches threshold, `do_update` is called: [2](#0-1) 

`self.entries.clear()` silently discards every other pending proposal. The `UpdateEntry` struct stores only `update` and `bytes_used` — no proposer account ID, no deposit amount: [3](#0-2) 

Because the proposer identity is never recorded inside the entry, there is no mechanism to issue refunds even if the code attempted to. The deposits are permanently locked in the contract balance.

The `propose` function simply inserts the entry with no deposit tracking: [4](#0-3) 

### Impact Explanation

Every participant whose proposal is cleared by a competing `do_update` loses their full storage deposit permanently. For a maximum-size contract proposal (~1.5 MB), the required deposit is on the order of **~40 NEAR** (as evidenced by the sandbox test using `NearToken::from_near(40)`). Multiple concurrent proposals — a normal and expected workflow — multiply the loss. The deposits are irrecoverably absorbed into the contract balance with no withdrawal or recovery function. This breaks the production accounting invariant that deposited funds must either be consumed for their stated purpose or returned. [5](#0-4) 

### Likelihood Explanation

Multiple participants proposing updates concurrently is explicitly supported and tested behavior. [6](#0-5) 

Any time a threshold vote is reached on one proposal while others are pending — a routine operational scenario during governance — all non-winning proposers lose their deposits. No special attacker capability is required; the loss occurs as a natural consequence of normal governance activity. Any participant below the threshold can be a victim.

### Recommendation

Store the proposer's `AccountId` and the exact deposit amount inside `UpdateEntry`. In `do_update`, before calling `self.entries.clear()`, iterate over all remaining entries and issue `Promise::new(entry.proposer).transfer(entry.deposit)` for each one. This mirrors the pattern already used in `propose_update` for refunding excess deposits.

### Proof of Concept

1. Participant Alice calls `propose_update` with a 1 MB contract binary, attaching ~27 NEAR deposit. Contract stores entry ID=0, retains ~27 NEAR.
2. Participant Bob calls `propose_update` with a different 1 MB binary, attaching ~27 NEAR. Contract stores entry ID=1, retains another ~27 NEAR.
3. Threshold participants vote for Bob's proposal (ID=1). `vote_update` calls `do_update(&id=1, ...)`.
4. Inside `do_update`:
   - `self.entries.remove(&1)` — Bob's entry is removed and executed.
   - `self.entries.clear()` — Alice's entry (ID=0) is silently dropped.
   - No refund is issued to Alice.
5. Alice's ~27 NEAR is permanently locked in the contract. There is no `withdraw_proposal_deposit` or equivalent function anywhere in the contract. [7](#0-6)

        .build()

    const PROPOSAL_COUNT: usize = 2;
    let mut proposals = Vec::with_capacity(PROPOSAL_COUNT);
    // Try to propose multiple updates to check if they are being proposed correctly

            .unwrap();

            execution.is_success(),
            "failed to propose update [i={i}]; {execution:#?}"
        );
        let proposal_id = execution.json().expect("unable to convert into UpdateId");
        proposals.push(proposal_id);
    }

    // Vote for the last proposal
    vote_update_till_completion(&contract, &mpc_signer_accounts, proposals.last().unwrap()).await;
