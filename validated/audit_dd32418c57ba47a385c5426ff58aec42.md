Audit Report

## Title
Uninitialized `maxNumberOfWithdrawalsPerUser` Permanently Blocks All Withdrawals From Deployment - (File: contracts/KERNEL/KernelDepositPool.sol)

## Summary
`KernelDepositPool.initialize()` never sets `maxNumberOfWithdrawalsPerUser`, leaving it at the Solidity default of `0`. The guard in `initiateWithdrawal` evaluates `0 >= 0 == true` on every call and unconditionally reverts with `WithdrawalLimitReached`. Because `setMaxNumberOfWithdrawalsPerUser` explicitly rejects `0` as an argument, there is no self-healing path; all staked KERNEL tokens are frozen until an admin manually intervenes post-deployment.

## Finding Description
`KernelDepositPool.initialize()` sets only `kernelToken`, `rewardsToken`, and the admin role — `maxNumberOfWithdrawalsPerUser` is never assigned and remains `0`.

`initiateWithdrawal` contains the guard:
```solidity
if (userWithdrawalIds[msg.sender].length >= maxNumberOfWithdrawalsPerUser) revert WithdrawalLimitReached();
```
With `maxNumberOfWithdrawalsPerUser == 0`, a fresh user whose `userWithdrawalIds` array has length `0` satisfies `0 >= 0`, causing an unconditional revert. No user can ever pass this check until the variable is set to a non-zero value.

The only setter, `setMaxNumberOfWithdrawalsPerUser`, explicitly rejects `0`:
```solidity
if (_maxNumberOfWithdrawalsPerUser == 0 || _maxNumberOfWithdrawalsPerUser > MAX_WITHDRAWALS_PER_USER)
    revert InvalidMaxNumberOfWithdrawalsPerUser();
```
So the contract cannot self-heal; admin action is mandatory.

Meanwhile, `stake()` and `stakeFor()` carry no equivalent guard and succeed freely. `KernelTop100MerkleDistributor.claimAndStake()` calls `kernelDepositPool.stakeFor(user, amountToStake)`, depositing tokens into the pool on behalf of users who then cannot withdraw them.

## Impact Explanation
All KERNEL tokens deposited via `stake()` or `stakeFor()` (including those routed through `KernelTop100MerkleDistributor.claimAndStake()`) are inaccessible to their owners from block 0 of the contract's life. If the admin never calls `setMaxNumberOfWithdrawalsPerUser`, the freeze is permanent — matching **Critical: Permanent freezing of funds**. Even in the best case where the admin eventually calls the setter, every user who staked before that call experiences a **Medium: Temporary freezing of funds**.

## Likelihood Explanation
The condition exists from the moment the contract is deployed. Any user who calls `stake()` or `claimAndStake()` before the admin separately calls `setMaxNumberOfWithdrawalsPerUser` (which is not part of `initialize()` and has no deployment script enforcement) will have their funds frozen. This is reachable by any unprivileged external user through normal staking flows with no special preconditions.

## Recommendation
Set a safe non-zero default for `maxNumberOfWithdrawalsPerUser` inside `initialize()`:
```solidity
function initialize(address _admin, address _kernelToken, address _rewardToken) external initializer {
    // ... existing checks ...
    maxNumberOfWithdrawalsPerUser = MAX_WITHDRAWALS_PER_USER; // defaults to 100
}
```
This ensures the withdrawal path is open from deployment and is consistent with the existing constant `MAX_WITHDRAWALS_PER_USER = 100`.

## Proof of Concept
1. Deploy `KernelDepositPool` via `initialize(admin, kernelToken, rewardToken)`. `maxNumberOfWithdrawalsPerUser` is `0`.
2. Admin grants `STAKE_FOR_ROLE` to `KernelTop100MerkleDistributor`.
3. User calls `claimAndStake(amount, proof)` on the distributor → `stakeFor(user, amount)` is called → `balanceOf[user] += amount`. Succeeds.
4. User calls `initiateWithdrawal(amount)` on `KernelDepositPool`.
   - Check: `userWithdrawalIds[user].length (= 0) >= maxNumberOfWithdrawalsPerUser (= 0)` → `true`
   - Reverts with `WithdrawalLimitReached`.
5. User's KERNEL tokens are locked with no withdrawal path.

**Foundry test sketch:**
```solidity
function test_withdrawalBlockedOnFreshDeploy() public {
    // deploy pool (initialize does NOT set maxNumberOfWithdrawalsPerUser)
    pool.initialize(admin, address(kernelToken), address(rewardToken));
    assertEq(pool.maxNumberOfWithdrawalsPerUser(), 0);

    // stake succeeds
    vm.prank(user);
    kernelToken.approve(address(pool), 1e18);
    vm.prank(user);
    pool.stake(1e18);

    // withdrawal reverts
    vm.prank(user);
    vm.expectRevert(KernelDepositPool.WithdrawalLimitReached.selector);
    pool.initiateWithdrawal(1e18);
}
```
