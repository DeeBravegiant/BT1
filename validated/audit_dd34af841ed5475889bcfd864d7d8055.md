### Title
Unbounded Full-Map Iteration Over All Registered DReps Triggered on Every Governance Proposal Submission - (File: `eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Certs.hs`)

---

### Summary

In the Conway era LEDGER rule, every transaction that contains at least one governance proposal unconditionally invokes `updateDormantDRepExpiry`, which performs an O(n) `Map.map` traversal over the entire `vsDReps` map — the set of all registered DReps. Because there is no protocol-enforced cap on the number of registered DReps, an adversary who registers a large number of DReps (paying the deposit for each) and then allows dormant epochs to accumulate can cause the first governance-proposal-bearing transaction in the next active epoch to require an arbitrarily large amount of computation during block validation. This mirrors the `_beforeProviderOp` gas-exhaustion pattern from the external report: a state-altering operation that must iterate over an unbounded, attacker-growable collection is invoked as a mandatory prefix of every governance proposal submission.

---

### Finding Description

Any transaction carrying at least one `ProposalProcedure` unconditionally calls `updateDormantDRepExpiry`.

**The unbounded loop — `updateDormantDRepExpiry` (`Certs.hs` lines 308–328):**

```haskell
updateDormantDRepExpiry currentEpoch vState =
  if numDormantEpochs == EpochNo 0
    then vState
    else
      vState
        & vsNumDormantEpochsL .~ EpochNo 0
        & vsDRepsL %~ Map.map updateExpiry   -- O(|vsDReps|)
``` [4](#0-3) 

`Map.map updateExpiry` is a strict O(n) traversal of the entire `vsDReps` map. There is no protocol-level cap on the size of this map; the only barrier is the DRep registration deposit (`ppDRepDepositL`), a governance-adjustable parameter.

**The dormant-epoch accumulator — `updateNumDormantEpochs` (`Epoch.hs` lines 197–201):**

```haskell
updateNumDormantEpochs currentEpoch ps vState =
  if null $ OMap.filter ((currentEpoch <=) . gasExpiresAfter) $ ps ^. pPropsL
    then vState & vsNumDormantEpochsL %~ succ
    else vState
``` [5](#0-4) 

Every epoch boundary with no live proposals increments `vsNumDormantEpochs`. Once it is non-zero, the very next transaction that contains a governance proposal triggers the full `Map.map` sweep over all DReps.

**Attack sequence:**

1. Attacker registers N DReps (each paying `ppDRepDepositL`).
2. No governance proposals are submitted for several epochs; `vsNumDormantEpochs` grows.
3. Any honest user (or the attacker) submits a transaction containing a governance proposal.
4. The LEDGER rule calls `updateDormantDRepExpiry`, which performs `Map.map updateExpiry` over all N DRep entries before the transaction is accepted into a block.
5. Block validation time scales linearly with N. With a sufficiently large N, validation of any block containing such a transaction exceeds the slot-duration budget (currently 1 second), causing nodes to miss their slots.

---

### Impact Explanation

This matches the **Medium** allowed impact: *"Attacker-controlled transactions, blocks, certificates, votes, proposals, scripts, witnesses, or serialized inputs exceed intended validation limits."*

The attacker-controlled variable is the size of `vsDReps`. By inflating it, the attacker causes any governance-proposal-bearing transaction to consume unbounded CPU during block validation. In the extreme case — where the map is large enough that no block containing such a transaction can be validated within the slot budget — the governance subsystem is effectively frozen: no new proposals, parameter changes, hard-fork initiations, or treasury withdrawals can be enacted. Recovery from that state would require a hard fork to patch the iteration, escalating the impact to **High** (permanent freezing requiring a hard fork).

---

### Likelihood Explanation

**Low-to-Medium.** The DRep deposit (`ppDRepDepositL`) is an economic deterrent. However:

- The deposit is a governance-adjustable parameter; a future parameter-change proposal could lower it.
- A well-funded adversary (or a coalition) can register enough DReps to cause measurable slowdowns without reaching the full-freeze threshold.
- The condition requires dormant epochs (no live proposals), which is a realistic steady-state on mainnet between governance cycles.
- The `numDormantEpochs` counter resets to zero after the first triggering transaction, so the expensive sweep fires exactly once per dormant period — but that single firing is sufficient to stall or orphan the block containing the proposal.

---

### Recommendation

1. **Incremental / pulsed update**: Mirror the `DRepPulser` design already used for stake-distribution computation. Instead of updating all DRep expiries atomically inside the LEDGER rule, spread the work across multiple blocks or perform it lazily at the epoch boundary (where the full `vsDReps` map is already traversed for other purposes).

2. **Move the update to the epoch boundary**: `updateDormantDRepExpiry` is semantically an epoch-boundary operation (it adjusts expiry by the number of dormant epochs). Performing it inside `epochTransition` — where the full DRep map is already touched — eliminates the per-transaction O(n) cost entirely.

3. **Protocol-level cap on DRep registrations**: Introduce a `ppMaxDReps` parameter analogous to `ppPoolDeposit`/`ppMaxTxSize` to bound the size of `vsDReps`.

---

### Proof of Concept

```
1. Set ppDRepDepositL to a low value (or use existing mainnet value).
2. Register N DRep credentials (N ~ tens of thousands, each paying the deposit).
3. Allow K epochs to pass with no governance proposals
   → vsNumDormantEpochs = K > 0.
4. Craft a transaction T containing one ProposalProcedure.
5. Submit T to a block producer.
6. The LEDGER rule evaluates:
     updateDormantDRepExpiries T epoch
   which calls:
     vsDRepsL %~ Map.map updateExpiry   -- iterates all N entries
7. Measure block-validation wall-clock time as a function of N.
   At N ≈ 1,000,000 DReps the Map.map traversal alone
   (each entry requiring two EpochNo comparisons and an addition)
   will consume hundreds of milliseconds, approaching or exceeding
   the 1-second slot budget and causing the block to be orphaned.
8. Any subsequent governance proposal in the same epoch is cheap
   (numDormantEpochs resets to 0), but the attacker can repeat
   steps 3–7 every time a dormant period accumulates.
```

The root cause is the unconditional `Map.map updateExpiry` over the unbounded `vsDReps` map inside the mandatory LEDGER-rule path for every governance-proposal transaction. [6](#0-5) [7](#0-6) [8](#0-7) [5](#0-4)

updateDormantDRepExpiry ::
  -- | Current Epoch
  EpochNo ->
  VState era ->
  VState era
updateDormantDRepExpiry currentEpoch vState =
  if numDormantEpochs == EpochNo 0
    then vState
    else
      vState
        & vsNumDormantEpochsL .~ EpochNo 0
        & vsDRepsL %~ Map.map updateExpiry
  where
    numDormantEpochs = vState ^. vsNumDormantEpochsL
    updateExpiry =
      drepExpiryL
        %~ \currentExpiry ->
          let actualExpiry = binOpEpochNo (+) numDormantEpochs currentExpiry
           in if actualExpiry < currentEpoch
                then currentExpiry
                else actualExpiry
