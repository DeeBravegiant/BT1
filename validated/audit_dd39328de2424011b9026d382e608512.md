### Title
Attacker-Controlled Contract Panic Message Reflected Unsanitized into JSON-RPC `"error"` Field - (`chain/jsonrpc/src/lib.rs`)

### Summary
A deployed smart contract can call `panic_utf8` with an arbitrary attacker-controlled UTF-8 string. When any user queries that contract via the `query` RPC endpoint (`CallFunction`), the panic message is formatted and placed verbatim into the `"error"` field of a **successful** JSON-RPC response, with no sanitization or substitution. This is a direct analog of the original report: attacker-controlled data from contract execution is reflected unsanitized into the RPC result consumed by clients.

### Finding Description

The full injection path is:

**Step 1 — Attacker injects arbitrary string via `panic_utf8`:**

In `runtime/near-vm-runner/src/logic/logic.rs`, the `panic_utf8` host function reads an arbitrary UTF-8 string from contract memory and places it directly into `HostError::GuestPanic`:

```rust
pub fn panic_utf8(&mut self, len: u64, ptr: u64) -> Result<()> {
    self.result_state.gas_counter.pay_base(base)?;
    Err(HostError::GuestPanic { panic_msg: self.get_utf8_string(len, ptr)? }.into())
}
``` [1](#0-0) 

The `Display` impl for `HostError::GuestPanic` formats as `"Smart contract panicked: {panic_msg}"`, embedding the attacker's string verbatim. [2](#0-1) 

**Step 2 — Error message formatted and stored in `error_message`:**

In `runtime/runtime/src/state_viewer/mod.rs`, when a view call aborts, the error is formatted with `{:?}` (Debug) and stored as `error_message`:

```rust
let message = format!("wasm execution failed with error: {:?}", err);
Err(errors::CallFunctionError::VMError { error, error_message: message })
``` [3](#0-2) 

**Step 3 — `error_message` propagated to `QueryError::ContractExecutionError`:**

In `chain/chain/src/runtime/errors.rs`, the `error_message` (containing the attacker's panic string) is mapped directly into `QueryError::ContractExecutionError`:

```rust
node_runtime::state_viewer::errors::CallFunctionError::VMError {
    error,
    error_message,
} => Self::ContractExecutionError { error_message, error, block_height, block_hash },
``` [4](#0-3) 

**Step 4 — Propagated to `RpcQueryError::ContractExecutionError` as `vm_error`:**

In `chain/client/src/view_client_actor.rs`, the `error_message` becomes `vm_error`:

```rust
QueryError::ContractExecutionError {
    vm_error: error_message,
    ...
}
``` [5](#0-4) 

**Step 5 — `vm_error` injected directly into the JSON-RPC response `"error"` field:**

In `chain/jsonrpc/src/lib.rs`, `process_query_response` places `vm_error` verbatim into a **successful** JSON-RPC result:

```rust
RpcQueryError::ContractExecutionError {
    vm_error,
    error: _,
    block_height,
    block_hash,
} => Ok(json!({
    "error": vm_error,
    "logs": json!([]),
    "block_height": block_height,
    "block_hash": block_hash,
})),
``` [6](#0-5) 

### Impact Explanation

Any unprivileged user who deploys a contract and causes it to call `panic_utf8` with an arbitrary message controls the `"error"` field of the JSON-RPC response returned to any caller of the `query` endpoint for that contract. The corrupted value is the **RPC result** itself — specifically the `"error"` string field in the backward-compatible response body. Wallet applications, dApps, and tooling that parse this field will display the attacker's message verbatim. This enables phishing payloads (e.g., fake "please update your wallet at malicious-url.com" messages) to be injected into the RPC layer, exactly as described in the original report's exploit scenario.

### Likelihood Explanation

Likelihood is **medium**. Any account can deploy a contract on NEAR. The `panic_utf8` host function is a standard, documented, and widely-used API. The attacker only needs to deploy a contract that panics with a crafted message and wait for users or automated tooling to query it. No special privileges are required. The backward-compatible `"error"` field in the `process_query_response` path is specifically designed to be consumed by clients, making it a reliable injection point.

### Recommendation

**Short term:** In `process_query_response` (`chain/jsonrpc/src/lib.rs`), replace the raw `vm_error` string with a sanitized or pre-arranged error code/message. Do not inject the raw `vm_error` (which contains attacker-controlled contract panic text) directly into the `"error"` field. Instead, use a fixed message such as `"Contract execution error"` and log the full `vm_error` server-side only.

**Long term:** Implement a framework that distinguishes between internal diagnostic strings (which may contain attacker-controlled data) and user-facing RPC output. Attacker-controlled strings from contract execution (panic messages, log entries) should never be placed unsanitized into structured RPC response fields consumed by clients.

### Proof of Concept

1. Deploy a contract that calls `panic_utf8` with the payload: `"URGENT: Wallet vulnerability detected. Download the security patch at https://malicious.example.com/wallet-update"`.
2. Call the NEAR JSON-RPC `query` endpoint with `request_type: "call_function"` targeting that contract.
3. Observe the response:
```json
{
  "result": {
    "error": "wasm execution failed with error: HostError(GuestPanic { panic_msg: \"URGENT: Wallet vulnerability detected. Download the security patch at https://malicious.example.com/wallet-update\" })",
    "logs": [],
    "block_height": 12345,
    "block_hash": "..."
  }
}
```
The attacker's string appears verbatim in the `"error"` field of the successful RPC response, with no sanitization, exactly mirroring the original report's vulnerability class.
