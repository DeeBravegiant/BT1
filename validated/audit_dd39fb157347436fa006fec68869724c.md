Audit Report

## Title
Decimal Precision Oversight in Token-to-rsETH Swap Calculation Causes Massive Under-Minting for Non-18-Decimal Tokens - (File: contracts/pools/RSETHPoolV3.sol, RSETHPoolV3ExternalBridge.sol, RSETHPoolV3WithNativeChainBridge.sol, RSETHPoolNoWrapper.sol)

## Summary
The `viewSwapRsETHAmountAndFee(amount, token)` function in all four L2 pool variants computes rsETH output as `amountAfterFee * tokenToETHRate / rsETHToETHrate` without normalizing `amountAfterFee` to 18 decimals. The oracle always returns a 1e18-scaled price representing the ETH value of one whole token, making the formula dimensionally correct only for 18-decimal tokens. For a 6-decimal token like USDC, depositors receive `10^12` times fewer rsETH/wrsETH than owed while their full token balance is transferred and bridged to L1, constituting direct theft of user funds.

## Finding Description
In `RSETHPoolV3.sol`, `viewSwapRsETHAmountAndFee(uint256 amount, address token)` computes:

```solidity
// RSETHPoolV3.sol L324-334
fee = amount * feeBps / 10_000;
uint256 amountAfterFee = amount - fee;
uint256 rsETHToETHrate = getRate();
uint256 tokenToETHRate = IOracle(supportedTokenOracle[token]).getRate();
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
``` [1](#0-0) 

`ChainlinkOracleForRSETHPoolCollateral.getRate()` normalizes the Chainlink price to 1e18 scale, representing the ETH value of **one whole token**, regardless of the token's own decimal count:

```solidity
// ChainlinkOracleForRSETHPoolCollateral.sol L34
uint256 normalizedPrice = uint256(ethPrice) * 1e18 / 10 ** uint256(AggregatorV3Interface(oracle).decimals());
``` [2](#0-1) 

This means `tokenToETHRate` = `(ETH price of 1 whole token) × 1e18`. For the formula to be dimensionally correct, `amountAfterFee` must also be in 1e18 units. For an 18-decimal token (wstETH, WETH), depositing N tokens gives `amountAfterFee = N × 1e18`, and the result is correct. For a 6-decimal token (USDC), depositing N tokens gives `amountAfterFee = N × 1e6`, and the result is `10^12` times too small.

The `addSupportedToken` function enforces no decimal constraint:

```solidity
// RSETHPoolV3WithNativeChainBridge.sol L599-600
function addSupportedToken(address token, address oracle, address bridge) external onlyRole(TIMELOCK_ROLE) {
    _addSupportedToken(token, oracle, bridge);
``` [3](#0-2) 

The identical broken formula is present in all four pool variants: [4](#0-3) [5](#0-4) 

The `deposit` function transfers the full token amount from the user and mints the (severely undervalued) rsETH amount with no further validation:

```solidity
// RSETHPoolV3.sol L284-290
IERC20(token).safeTransferFrom(msg.sender, address(this), amount);
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount, token);
feeEarnedInToken[token] += fee;
wrsETH.mint(msg.sender, rsETHAmount);
``` [6](#0-5) 

No existing guard (reentrancy protection, pause, daily mint limit, `onlySupportedToken`) addresses the decimal mismatch.

## Impact Explanation
Once a non-18-decimal token is added as supported collateral, every depositor of that token loses virtually all deposited value. Their full token balance is transferred to the pool and bridged to L1 (capturing full ETH value), while they receive `10^(18 - tokenDecimals)` times fewer rsETH/wrsETH than owed. For USDC (6 decimals), the shortfall is exactly `10^12`. This is **direct theft of user funds in motion** — a Critical impact matching the allowed scope.

## Likelihood Explanation
The bug is latent with current supported tokens (wstETH, WETH, both 18 decimals). Activation requires a `TIMELOCK_ROLE` governance action to add a non-18-decimal token — a routine protocol expansion, not a compromise. Once such a token is added, the loss is automatic for every depositor of that token; no attacker action is required beyond making a normal deposit. The exploit is fully triggered by an unprivileged external user calling `deposit`. **Likelihood: Medium.**

## Recommendation
Normalize `amountAfterFee` to 18 decimals before applying the rate formula in all four pool variants:

```solidity
uint8 tokenDecimals = IERC20Metadata(token).decimals();
uint256 normalizedAmount = amountAfterFee * 10 ** (18 - tokenDecimals);
rsETHAmount = normalizedAmount * tokenToETHRate / rsETHToETHrate;
```

Apply consistently to `RSETHPoolV3.sol`, `RSETHPoolV3ExternalBridge.sol`, `RSETHPoolV3WithNativeChainBridge.sol`, and `RSETHPoolNoWrapper.sol`. Alternatively, enforce at `addSupportedToken` time that `IERC20Metadata(token).decimals() == 18`.

## Proof of Concept
1. Governance calls `addSupportedToken(USDC, usdcOracle, usdcBridge)` on `RSETHPoolV3`.
2. Alice calls `deposit(USDC, 1_000_000e6, "ref")` (1,000,000 USDC, worth ~300 ETH at a hypothetical rate of 0.0003 ETH/USDC).
3. `viewSwapRsETHAmountAndFee` computes:
   - `amountAfterFee` ≈ `1e12` (1,000,000 × 1e6)
   - `tokenToETHRate` = `3e14` (0.0003 × 1e18)
   - `rsETHToETHrate` ≈ `1e18`
   - `rsETHAmount` = `1e12 × 3e14 / 1e18` = **`3e8`** (≈ 0.0000003 wrsETH)
4. Correct result: `1_000_000 × 1e18 × 3e14 / 1e18` = **`3e20`** (≈ 300 wrsETH).
5. Alice receives `3e8` wrsETH instead of `3e20` — a `10^12` shortfall — while her 1,000,000 USDC is fully transferred and bridged to L1.

**Foundry fork test plan:** Deploy `RSETHPoolV3` on a mainnet fork; call `addSupportedToken` with a mock USDC oracle returning `3e14`; call `deposit(USDC, 1_000_000e6, "")` from an unprivileged address; assert `wrsETH.balanceOf(alice) == 3e8` (actual) vs. expected `3e20`; assert `IERC20(USDC).balanceOf(pool) == 1_000_000e6`.
