Audit Report

## Title
Unhandled Aave Withdrawal Revert in `_processWithdrawalCompletion` Causes Temporary Freezing of User ETH - (File: contracts/LRTWithdrawalManager.sol)

## Summary

`_processWithdrawalCompletion` calls `_withdrawFromAave` at line 724 without any try/catch, while the deposit path at lines 311–316 wraps the equivalent call in a try/catch. If Aave's `withdrawETH` reverts (e.g., pool paused), `completeWithdrawal` reverts entirely. Because rsETH was already burned in the prior `unlockQueue` call and ETH is held in Aave, users are left with no rsETH and no ETH until Aave becomes available again. Critically, no admin recovery path exists during an Aave pause: both `emergencyWithdrawFromAave` (line 560) and `setAaveIntegrationEnabled(false)` (line 495) also call `_withdrawFromAave` internally and would revert under the same condition.

## Finding Description

**Root cause — asymmetric error handling:**

The deposit path in `unlockQueue` wraps the Aave call in a try/catch: [1](#0-0) 

The withdrawal path in `_processWithdrawalCompletion` makes a bare, unguarded call: [2](#0-1) 

`_withdrawFromAave` calls `aaveWETHGateway.withdrawETH` with no error handling: [3](#0-2) 

**Exploit flow:**

1. User calls `initiateWithdrawal` — rsETH transferred to `LRTWithdrawalManager`.
2. Operator calls `unlockQueue` — rsETH is burned (`burnFrom` at line 305) and ETH is deposited into Aave.
3. Aave v3 pool is paused by Aave governance or emergency admin (a documented, exercised capability).
4. User calls `completeWithdrawal`:
   - `contractBalance < request.expectedAssetAmount` (ETH is in Aave).
   - `_withdrawFromAave(amountNeeded)` is called.
   - `aaveWETHGateway.withdrawETH(...)` reverts because Aave is paused.
   - The entire transaction reverts; the withdrawal request record is restored by the EVM rollback.
5. User has no rsETH (burned in step 2) and no ETH (locked in Aave). They cannot withdraw.

**No admin recovery path during Aave pause:**

- `emergencyWithdrawFromAave` (line 551) calls `_withdrawFromAave` internally — also reverts when Aave is paused. [4](#0-3) 
- `setAaveIntegrationEnabled(false)` (line 469) also calls `_withdrawFromAave` before disabling — also reverts when Aave is paused. [5](#0-4) 

There is no path to drain Aave funds back to the contract while Aave is paused, leaving all affected users frozen until Aave unpauses.

## Impact Explanation

**Medium — Temporary freezing of funds.** During any Aave v3 pause, all users whose ETH was deposited into Aave via `unlockQueue` cannot complete their withdrawals. Their rsETH is already burned and their ETH is inaccessible. The freeze persists for the full duration of the Aave pause. If Aave were permanently deprecated (worst case), the freeze becomes permanent, escalating to Critical.

## Likelihood Explanation

Aave v3 has a well-documented pause mechanism exercisable by Aave governance and emergency admins, and it has been exercised historically. Any Aave pause during the window between `unlockQueue` and `completeWithdrawal` affects all users with pending ETH withdrawals backed by Aave liquidity. No attacker action is required — a routine Aave governance pause is sufficient. The condition is realistic and non-hypothetical.

## Recommendation

Wrap the `_withdrawFromAave` call in `_processWithdrawalCompletion` with a try/catch, consistent with the deposit path. On failure, revert with a descriptive error so the user can retry later:

```solidity
try this.withdrawFromAaveExternal(amountNeeded) {
    // success
} catch (bytes memory reason) {
    revert AaveWithdrawalFailed(reason);
}
```

Additionally, fix `setAaveIntegrationEnabled(false)` and `emergencyWithdrawFromAave` to use a low-level call or a separate try/catch so that the manager can disable Aave integration and drain funds even when Aave is paused.

**Foundry fork test outline:**
