Audit Report

## Title
Push-ETH Pattern in `_processWithdrawalCompletion` Permanently Freezes ETH for Smart Contract Withdrawers — (File: `contracts/LRTWithdrawalManager.sol`)

## Summary

`LRTWithdrawalManager` uses a push-ETH pattern in `_transferAsset` to deliver ETH to withdrawing users. If the recipient is a smart contract without a `receive()` function, the transfer reverts and the entire `completeWithdrawal` transaction is rolled back. Because rsETH is already burned and ETH already redeemed into the contract during the earlier `unlockQueue` call, the ETH becomes permanently frozen with no admin recovery path.

## Finding Description

`initiateWithdrawal` (L150–178) accepts calls from any address including smart contracts, with no restriction on the caller type. During `unlockQueue` (L268–320), rsETH is irreversibly burned and ETH is pulled from `LRTUnstakingVault` into `LRTWithdrawalManager` at L305–307:

```solidity
if (rsETHBurned != 0) IRSETH(lrtConfig.rsETH()).burnFrom(address(this), rsETHBurned);
unstakingVault.redeem(asset, assetAmountUnlocked);
```

By the time `completeWithdrawal` is called, the ETH already sits inside `LRTWithdrawalManager`. Inside `_processWithdrawalCompletion` (L699–738), the function pops the nonce (L705), deletes the request (L712), decrements `unlockedWithdrawalsCount` (L717), and then calls `_transferAsset` at L734:

```solidity
_transferAsset(asset, user, request.expectedAssetAmount);
```

`_transferAsset` (L876–883) performs:

```solidity
(bool sent,) = payable(to).call{ value: amount }("");
if (!sent) revert EthTransferFailed();
```

If `to` has no `receive()` function, `sent` is `false`, `EthTransferFailed` is thrown, and the **entire transaction reverts** — undoing the `popFront`, `delete`, and `unlockedWithdrawalsCount--`. The withdrawal request is fully restored to its unlocked state, and `unlockedWithdrawalsCount[asset]` remains ≥ 1.

Both `completeWithdrawal` (L183–185) and `completeWithdrawalForUser` (L192–204) call the same `_processWithdrawalCompletion`, so both paths always revert for this user. The only potential recovery function, `sweepRemainingAssets` (L395–414), is gated at L403:

```solidity
if (hasUnlockedWithdrawals(asset)) revert PendingWithdrawalsExist();
```

Since the stuck request keeps `unlockedWithdrawalsCount[asset] > 0`, this gate is never cleared. There is no admin function to redirect ETH to an alternate address or force-delete the stuck request.

## Impact Explanation

**Critical — Permanent freezing of funds.** The user's rsETH is irreversibly burned during `unlockQueue`. The ETH equivalent is locked inside `LRTWithdrawalManager` indefinitely: `completeWithdrawal` always reverts, `completeWithdrawalForUser` always reverts, and `sweepRemainingAssets` is permanently blocked while any unlocked withdrawal exists for that asset. There is no on-chain escape hatch.

## Likelihood Explanation

**Low.** The affected user must be a smart contract without a working ETH `receive()` fallback. Realistic cases include multisig wallets lacking a `receive` fallback, DeFi vaults or aggregators routing withdrawals through a contract, or contracts upgraded post-initiation to remove ETH acceptance. The scenario is uncommon but entirely plausible in a protocol that imposes no restriction on smart-contract depositors in `initiateWithdrawal`.

## Recommendation

Replace the push-ETH pattern with a pull (claim) pattern for ETH withdrawals:

1. In `_processWithdrawalCompletion`, instead of calling `_transferAsset(asset, user, amount)` for ETH, credit `pendingETHClaims[user] += amount` and emit an event.
2. Add a separate `claimETH()` function that lets the user pull their credited balance at will.
3. Apply the same pattern to the `feeRecipient` transfer in `instantWithdrawal` and to `_collectInterestToTreasury`.

This eliminates the ability of any single recipient to block their own (or others') withdrawal execution.

## Proof of Concept

1. A smart contract wallet `VaultProxy` (no `receive()`) calls `initiateWithdrawal(ETH_TOKEN, rsETHAmount, "")`. rsETH is transferred to `LRTWithdrawalManager` at L166.
2. Operator calls `unlockQueue(ETH_TOKEN, ...)`. rsETH is burned at L305; ETH is redeemed from `LRTUnstakingVault` into `LRTWithdrawalManager` at L307.
3. `VaultProxy` calls `completeWithdrawal(ETH_TOKEN, "")`. `_processWithdrawalCompletion` reaches `_transferAsset(ETH_TOKEN, VaultProxy, amount)` at L734.
4. `payable(VaultProxy).call{value: amount}("")` returns `(false, "")` because `VaultProxy` has no `receive()`.
5. `EthTransferFailed` is thrown; the entire transaction reverts. The nonce is restored to the queue, the request is un-deleted, and `unlockedWithdrawalsCount` is restored.
6. Operator attempts `completeWithdrawalForUser(ETH_TOKEN, VaultProxy, "")` — same revert.
7. `unlockedWithdrawalsCount[ETH_TOKEN]` remains ≥ 1; `sweepRemainingAssets` reverts with `PendingWithdrawalsExist` at L403.
8. `VaultProxy`'s rsETH is permanently burned; the ETH equivalent is permanently locked in `LRTWithdrawalManager` with no recovery path.

**Foundry test plan:** Deploy `LRTWithdrawalManager` on a fork; deploy a `NoReceive` contract (no `receive()`); call `initiateWithdrawal` from `NoReceive`; call `unlockQueue` as operator; call `completeWithdrawal` from `NoReceive` and assert revert with `EthTransferFailed`; assert `unlockedWithdrawalsCount[ETH_TOKEN] > 0`; assert `sweepRemainingAssets` reverts with `PendingWithdrawalsExist`; assert ETH balance of `LRTWithdrawalManager` equals the redeemed amount.
