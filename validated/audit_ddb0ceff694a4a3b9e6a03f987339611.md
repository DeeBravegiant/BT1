Audit Report

## Title
ETH Deposit Limit Bypass Due to Missing Incoming Amount in Limit Check - (File: `contracts/LRTDepositPool.sol`)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` applies asymmetric logic: for ERC20 assets it correctly evaluates `totalAssetDeposits + amount > limit`, but for ETH it evaluates only `totalAssetDeposits > limit`, omitting the incoming deposit amount entirely. As a result, any ETH deposit made while `totalAssetDeposits <= limit` bypasses the cap check unconditionally, regardless of how large `msg.value` is, allowing the protocol's ETH TVL to exceed the configured limit.

## Finding Description
The root cause is confirmed at `contracts/LRTDepositPool.sol` lines 676–682:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));   // amount excluded
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // amount included
}
``` [1](#0-0) 

This function is the sole pre-deposit guard. It is called unconditionally from `_beforeDeposit` (lines 661–663), which is the only validation path for both `depositETH` and `depositAsset`: [2](#0-1) 

`depositETH` passes `msg.value` as `depositAmount` to `_beforeDeposit` (line 87), but because the ETH branch of the guard never adds `amount` to `totalAssetDeposits`, the check only fires when the cap has *already* been exceeded by prior deposits: [3](#0-2) 

**Exploit flow:**
1. Admin sets `depositLimitByAsset[ETH_TOKEN] = 100 ether`.
2. Running total reaches exactly `100 ether` (`totalAssetDeposits == limit`).
3. Attacker calls `depositETH{value: 50 ether}(0, "")`.
4. Guard evaluates `100 ether > 100 ether` → `false` → no revert.
5. `_mintRsETH` mints rsETH for the full 50 ETH.
6. Protocol holds 150 ETH against a 100 ETH cap; limit exceeded by 50 ETH.

No existing check compensates for this omission. The `minAmountToDeposit` check (line 657) only enforces a minimum, not a maximum. [4](#0-3) 

## Impact Explanation
The ETH deposit cap (`depositLimitByAsset[ETH_TOKEN]`) is not enforced for the incoming deposit amount. Any unprivileged depositor can push the protocol's ETH TVL above the configured limit in a single transaction, causing more rsETH to be minted than the protocol's risk parameters allow. This matches the allowed impact: **Low — contract fails to deliver promised returns** (deposit cap enforcement), with no direct loss of user funds.

## Likelihood Explanation
The condition is trivially reachable by any ETH depositor at any time `totalAssetDeposits <= limit`. No special role, timing window, front-running, or privileged access is required. The attacker simply calls `depositETH{value: largeAmount}()` when the running total is at or below the cap, which is the normal operating state of the protocol.

## Recommendation
Include the incoming ETH amount in the comparison, matching the ERC20 branch:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // add amount
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

## Proof of Concept
Foundry unit test plan:

```solidity
function test_ETHDepositLimitBypass() public {
    uint256 limit = 100 ether;
    // Admin sets ETH deposit limit
    vm.prank(admin);
    lrtConfig.setDepositLimitByAsset(LRTConstants.ETH_TOKEN, limit);

    // Fill deposits to exactly the limit
    vm.deal(depositor1, limit);
    vm.prank(depositor1);
    lrtDepositPool.depositETH{value: limit}(0, "");

    // Attacker deposits beyond the limit — should revert but does not
    uint256 excess = 50 ether;
    vm.deal(attacker, excess);
    vm.prank(attacker);
    lrtDepositPool.depositETH{value: excess}(0, ""); // expects MaximumDepositLimitReached, but passes

    // Assert TVL exceeds limit
    assertGt(lrtDepositPool.getTotalAssetDeposits(LRTConstants.ETH_TOKEN), limit);
}
```

The test demonstrates that after the running total equals the cap, a further deposit of any size is accepted without revert, confirming the bypass.
