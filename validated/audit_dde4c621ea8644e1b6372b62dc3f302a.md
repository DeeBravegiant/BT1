Audit Report

## Title
ETH Withdrawal Permanently Freezes Funds for Contract Recipients Unable to Receive ETH - (File: contracts/LRTWithdrawalManager.sol)

## Summary
The two-step ETH withdrawal process in `LRTWithdrawalManager` creates an irrecoverable state when the withdrawing address is a smart contract that cannot receive ETH. rsETH is burned in the `unlockQueue()` transaction (irreversible), and ETH is moved to the withdrawal manager. Any subsequent call to `completeWithdrawal()` or `completeWithdrawalForUser()` permanently reverts, leaving the ETH locked in the contract with no protocol-level recovery path.

## Finding Description
The withdrawal lifecycle spans two separate, independent transactions:

**Step 1 — `initiateWithdrawal()`** (L166): rsETH is transferred from `msg.sender` to the withdrawal manager, and `msg.sender` is permanently recorded as the beneficiary via `_addUserWithdrawalRequest`.

**Step 2 — `unlockQueue()`** (L305–307): rsETH held by the withdrawal manager is burned via `burnFrom`, and ETH is pulled from the unstaking vault via `unstakingVault.redeem()`. This transaction is finalized and irreversible.

**Step 3 — `completeWithdrawal()` / `completeWithdrawalForUser()`** (L183–184, L202): calls `_processWithdrawalCompletion`, which at L734 calls `_transferAsset(asset, user, request.expectedAssetAmount)`.

`_transferAsset` at L876–879 performs:
```solidity
(bool sent,) = payable(to).call{ value: amount }("");
if (!sent) revert EthTransferFailed();
```

When `to` is a contract without a `receive()` function, `sent` is `false` and `EthTransferFailed` is thrown. This revert propagates up through `_processWithdrawalCompletion`, rolling back all state changes in that transaction (the `popFront` at L705, `delete withdrawalRequests[requestId]` at L712, and `unlockedWithdrawalsCount[asset]--` at L717). The withdrawal request is restored to the queue in its unlocked state.

The result is a permanent loop: the request is unlocked, every completion attempt reverts, and the request is never consumed. Meanwhile:

- `sweepRemainingAssets()` at L403 checks `if (hasUnlockedWithdrawals(asset)) revert PendingWithdrawalsExist()` — blocked as long as the stuck request exists.
- `completeWithdrawalForUser()` at L202 calls the same `_processWithdrawalCompletion` with the same non-receiving `user` address — also always reverts.
- No admin-callable rescue or redirect function exists.

The rsETH burned in Step 2 is not rolled back — it was destroyed in a prior finalized transaction. The ETH sits in the withdrawal manager indefinitely.

## Impact Explanation
This constitutes **permanent freezing of funds** (Critical). The affected user loses both their rsETH (burned, unrecoverable) and their ETH (locked in the withdrawal manager with no extraction path). The ETH cannot be swept to treasury, redirected, or claimed by any other mechanism. The impact is total and irreversible for the affected withdrawal.

## Likelihood Explanation
Smart contracts regularly interact with DeFi protocols as first-class participants: DAO treasuries, protocol-owned liquidity managers, yield aggregators, and smart contract wallets. Not all implement a `receive()` function. Any such contract that calls `initiateWithdrawal()` for ETH will trigger this freeze once the operator runs `unlockQueue()`. The developer comment at L191 — `/// @dev Not expected to be used for ETH; potential gas grief scenarios are non-impactful for ETH` — acknowledges awareness of ETH-transfer edge cases on `completeWithdrawalForUser` but does not prevent the scenario. Likelihood: **Low-Medium**.

## Recommendation
1. **Preferred fix (pull-over-push)**: In `_processWithdrawalCompletion`, if the ETH transfer to `user` fails, do not revert. Instead, record the owed amount in a `claimableETH[user]` mapping and allow the user to pull their ETH via a separate `claimETH()` function. This ensures rsETH burn and ETH accounting are never left in an inconsistent state.

2. **Alternative — input validation**: At `initiateWithdrawal()` time, if `asset == ETH_TOKEN` and `msg.sender.code.length > 0`, require that the caller can receive ETH (e.g., via a test send or explicit interface check). This prevents the problematic state from ever being created.

3. **Minimum mitigation**: Add an admin-callable `rescueStuckWithdrawal(address asset, address user, address alternateRecipient)` function that can redirect a permanently stuck ETH withdrawal to an alternate address, bypassing the frozen state.

## Proof of Concept
1. Deploy a contract `NoReceive` with no `receive()` function that holds rsETH and calls `initiateWithdrawal(ETH_TOKEN, amount, "")`.
2. Operator calls `unlockQueue(ETH_TOKEN, ...)` — rsETH is burned at L305, ETH moves to withdrawal manager at L307.
3. `NoReceive` calls `completeWithdrawal(ETH_TOKEN, "")` — `_transferAsset` at L878 calls `payable(NoReceive).call{value: amount}("")`, which returns `sent = false`, triggering `revert EthTransferFailed()` at L879. All state changes in the transaction are rolled back.
4. Repeat step 3 indefinitely — always reverts with `EthTransferFailed`.
5. Observe: rsETH balance of `NoReceive` is 0 (burned in step 2, irreversible), ETH balance of withdrawal manager includes the stuck amount, `unlockedWithdrawalsCount[ETH_TOKEN] > 0` (blocking `sweepRemainingAssets`), and `userAssociatedNonces[ETH_TOKEN][NoReceive]` is non-empty (request perpetually re-queued on each revert). Funds are permanently frozen.
