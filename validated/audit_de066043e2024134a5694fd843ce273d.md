### Title
Identity-Point `AppPublicKeyPV` Bypasses CKD Encryption, Leaking Raw BLS Signature On-Chain — (`crates/contract/src/primitives/ckd.rs`, `crates/threshold-signatures/src/confidential_key_derivation/protocol_pv.rs`)

---

### Summary

`app_public_key_check` accepts `(pk1=G1::identity, pk2=G2::identity)` as a valid key pair. When such a pair is submitted in an `AppPublicKeyPV` CKD request, the ElGamal encryption term vanishes and the MPC nodes compute `big_c = msk * H(pk, app_id)` — the raw, unencrypted BLS signature — which is then posted on-chain and readable by anyone.

---

### Finding Description

**Step 1 — `app_public_key_check` passes for identity points.**

The contract-side check in `crates/contract/src/primitives/ckd.rs`:

```rust
pub(crate) fn app_public_key_check(app_public_key: &dtos::CKDAppPublicKeyPV) -> bool {
    let pk1 = env::bls12381_p1_decompress(&app_public_key.pk1);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pairing_input = [pk1, MINUS_G2_GENERATOR_UNCOMPRESSED, G1_GENERATOR_UNCOMPRESSED, pk2].concat();
    env::bls12381_pairing_check(&pairing_input)
}
``` [1](#0-0) 

The check evaluates `e(pk1, -G2) * e(G1, pk2) = 1`. With both points as identity: `e(identity, -G2) * e(G1, identity) = 1 * 1 = 1`. This passes. The NEAR host's `bls12381_pairing_check` does not reject identity points — this is explicitly confirmed by the test: [2](#0-1) 

The comment on `request_app_private_key` even documents this as intentional: [3](#0-2) 

The node-side `app_public_key_check` in `protocol_pv.rs` has the same behavior: `check_valid_point_g1(identity)` returns `true` because the identity point is on the curve and is torsion-free. [4](#0-3) 

**Step 2 — Encryption term vanishes in `compute_signature_share`.**

```rust
// C <- S + y . A
let big_c = big_s + app_pk.pk1 * y.0;
``` [5](#0-4) 

With `app_pk.pk1 = identity`: `identity * y = identity`, so `big_c = big_s = hash_point * private_share_i`. After Lagrange interpolation across all participants, the coordinator aggregates:

`big_c = Σ λᵢ · xᵢ · H(pk, app_id) = msk · H(pk, app_id)`

This is the raw BLS signature of `app_id` under the MPC master key — no encryption.

**Step 3 — `aggregated_output_check` also passes.**

```rust
multi_miller_loop(&[
    (output.big_c, -ElementG2::generator()),
    (output.big_y, app_pk.pk2),          // pk2 = identity → e(big_y, identity) = 1
    (*hash_point, public_key.to_element()),
])
``` [6](#0-5) 

With `pk2 = identity`, the middle term contributes 1. The check reduces to `e(big_c, -G2) * e(H, pk) = 1`, i.e. `e(msk·H, G2) = e(H, msk·G2)`, which holds by bilinearity. The check passes.

**Step 4 — Contract `ckd_output_check` also passes** for the same reason (same pairing equation, `pk2 = identity` contributes 1). [7](#0-6) 

**Step 5 — `respond_ckd` resolves the yield with `big_c` on-chain.** [8](#0-7) 

The attacker reads `big_c = msk * H(pk, app_id)` from the on-chain response.

---

### Impact Explanation

The CKD protocol's security guarantee is that `msk * H(pk, app_id)` is only revealed to the holder of `sk1` (where `pk1 = sk1 * G1`), enforced by the ElGamal encryption `big_c = msk*H + y*pk1`. With identity points, the encryption term is zero and the raw BLS signature is posted publicly on-chain.

The attacker learns `msk * H(pk, app_id)` for any `app_id` derived from their own NEAR account (any `derivation_path` they choose). This:

1. Bypasses the TEE attestation requirement — the key is obtained without any TEE, defeating the security model described in the CKD documentation.
2. Allows the attacker to use the derived key in an untrusted environment, impersonating a TEE app for their own `app_id`.
3. Exposes the raw BLS signature, which is the same value any legitimate TEE app for that `app_id` would decrypt — so if the attacker later registers a TEE app for the same account/path, the key is already compromised.

**Scope limitation:** The attacker can only extract BLS signatures for `app_ids` derived from their own NEAR account (`derive_app_id(attacker_account, path)`). They cannot extract signatures for other users' `app_ids`. The claim of "arbitrary app_ids" in the question is overstated.

---

### Likelihood Explanation

- `request_app_private_key` is a public, payable function requiring only 1 yoctoNEAR deposit.
- The identity-point encoding is standard and trivially constructable.
- No threshold collusion, TEE compromise, or privileged access is required.
- The exploit is fully local-testable.

---

### Recommendation

Add an explicit identity-point rejection to `app_public_key_check` in both the contract and the node-side protocol:

```rust
// Contract side (crates/contract/src/primitives/ckd.rs)
pub(crate) fn app_public_key_check(app_public_key: &dtos::CKDAppPublicKeyPV) -> bool {
    // Reject identity points — they would strip the ElGamal encryption term
    if app_public_key.pk1.is_identity() || app_public_key.pk2.is_identity() {
        return false;
    }
    // ... existing pairing check
}
```

Similarly, in `protocol_pv.rs`, `app_public_key_check` should call `p.is_identity()` after the `check_valid_point_g*` guards. Remove the comment in `request_app_private_key` that documents identity points as an accepted use case, as "intentionally public" derivation is not a safe design choice.

---

### Proof of Concept

1. Encode `G1::identity()` and `G2::identity()` in compressed BLS12-381 form.
2. Call `request_app_private_key` with `AppPublicKeyPV { pk1: identity_g1, pk2: identity_g2 }` and any `derivation_path`.
3. Observe that the contract accepts the request (no panic).
4. After MPC nodes process it, read `big_c` from the on-chain `CKDResponse`.
5. Verify: `e(big_c, G2) == e(H(pk, app_id), mpc_public_key)` — confirming `big_c = msk * H(pk, app_id)`.
