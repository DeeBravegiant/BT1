Audit Report

## Title
Stale `rsETHPrice` Cache Allows Depositors to Capture Accrued Yield at Existing Holders' Expense - (File: contracts/LRTOracle.sol)

## Summary
`LRTOracle` stores `rsETHPrice` as a cached state variable updated only when `updateRSETHPrice()` is called. Because `LRTDepositPool.getRsETHAmountToMint()` divides a live Chainlink asset price by this stale cached value, any depositor can mint more rsETH than fair value during the interval between price updates. When the price is subsequently updated to reflect accrued yield, the over-minted shares dilute existing holders, transferring their unclaimed yield to the attacker.

## Finding Description
`LRTOracle.updateRSETHPrice()` is unconditionally public:

```solidity
// LRTOracle.sol L87-89
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
```

Between calls, `rsETHPrice` grows stale while underlying LST assets (stETH, ETHx, etc.) continuously accrue yield and their Chainlink prices drift upward. The real TVL increases, but the stored price does not.

`LRTDepositPool.getRsETHAmountToMint()` computes the mint amount as:

```solidity
// LRTDepositPool.sol L520
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

`lrtOracle.getAssetPrice(asset)` reads the **live** Chainlink price; `lrtOracle.rsETHPrice()` returns the **stale** cached value. When yield has accrued since the last update, the denominator is artificially low, so the attacker receives more rsETH than their deposit is worth at the true current price.

The `pricePercentageLimit` guard inside `_updateRsETHPrice()` (L252–266) only fires when `newRsETHPrice > highestRsethPrice` **and** the increase exceeds the configured threshold. For normal daily yield accumulation this threshold is not breached, so the attacker can call `updateRSETHPrice()` freely. If `pricePercentageLimit == 0` (the zero-initialized default), the guard is entirely disabled. Furthermore, the attacker need not call the update themselves — they profit as soon as any caller (including the protocol's own keeper) triggers the update.

No existing check in `_beforeDeposit` or `getRsETHAmountToMint` enforces price freshness before minting.

## Impact Explanation
**High — Theft of unclaimed yield.**

Each update interval, the yield accrued by existing rsETH holders is partially captured by the attacker's over-minted shares. The attacker's rsETH is redeemable for more ETH than deposited, and the shortfall is borne by all pre-existing holders whose per-token price increase is diluted. The attack is repeatable every update cycle with no capital lockup requirement beyond the deposit itself.

## Likelihood Explanation
**Medium.**

The attacker requires no privileged role, no leaked key, and no off-chain data. All inputs — live Chainlink prices and the stored `rsETHPrice` — are readable on-chain. The only precondition is that yield has accrued since the last `updateRSETHPrice()` call, which is true for every normal update interval. The attack is fully self-contained and repeatable.

## Recommendation
1. **Atomic price refresh on deposit:** Call `_updateRsETHPrice()` at the start of `depositETH` / `depositAsset` (or enforce a staleness bound) before computing `rsethAmountToMint`, so the mint price always reflects current TVL.
2. **Compute mint amount from live TVL directly:** Replace the stale `rsETHPrice()` denominator in `getRsETHAmountToMint` with an on-the-fly calculation using `_getTotalEthInProtocol()` and `rsETH.totalSupply()`, bypassing the cache entirely.
3. **Deposit fee / TWAP:** Introduce a small deposit fee or time-weighted entry price to make the marginal yield-capture profit economically unattractive.

## Proof of Concept
**State:** `rsETHPrice = 1.010e18` (last updated 24 h ago); true current price = `1.012e18` (yield accrued).

1. Attacker calls `LRTDepositPool.depositETH{value: 100 ether}(0, "")`.
   - `getAssetPrice(ETH_TOKEN)` → `1e18` (live).
   - `rsETHPrice()` → `1.010e18` (stale).
   - Minted: `100e18 * 1e18 / 1.010e18 ≈ 99.0099 rsETH`.
   - Fair amount at true price: `100e18 / 1.012e18 ≈ 98.814 rsETH`.
   - Over-mint: `≈ 0.196 rsETH`.

2. Attacker (or any keeper) calls `LRTOracle.updateRSETHPrice()`.
   - New TVL includes attacker's 100 ETH; supply includes the over-minted rsETH.
   - `rsETHPrice` updates to ≈ `1.01009e18` (slightly below true `1.012e18` due to dilution).

3. Attacker redeems / sells `99.0099 rsETH` at the updated price.
   - Value recovered ≈ `99.0099 × 1.01009e18 / 1e18 ≈ 100.009 ETH`.
   - Net profit ≈ `0.009 ETH` per 100 ETH deposited per cycle, entirely at existing holders' expense.
