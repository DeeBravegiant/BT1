### Title
Cache-Slot Poisoning via Cross-Fork ChunkEndorsement Replay Silently Drops Legitimate Validator Endorsements - (`chain/client/src/stateless_validation/chunk_endorsement.rs`)

### Summary

The `process_chunk_endorsement` deduplication guard checks only `account_id` presence in the cache, not the stored `chunk_hash`. A network peer who relays a legitimately-signed `ChunkEndorsement` for `chunk_hash X` (from a fork) to a node whose canonical chunk is `chunk_hash Y` permanently occupies that validator's cache slot. Every subsequent legitimate endorsement for `chunk_hash Y` from the same validator is silently dropped. `collect_chunk_endorsements` then filters by `chunk_hash == chunk_header.chunk_hash()`, so the poisoned slot contributes zero stake toward endorsement of the canonical chunk.

### Finding Description

**Cache structure** (`chunk_endorsement.rs` lines 27-28):

```rust
SyncLruCache<ChunkProductionKey, HashMap<AccountId, (ChunkHash, Signature)>>
```

The outer key is `ChunkProductionKey = {shard_id, epoch_id, height_created}` — it does **not** include `chunk_hash`. [1](#0-0) 

**Deduplication guard** (`process_chunk_endorsement`, line 50):

```rust
if cache.peek(&key).is_some_and(|entry| entry.contains_key(account_id)) {
    return Ok(());
}
```

This returns early if `account_id` is already present for the `ChunkProductionKey`, regardless of which `chunk_hash` is stored. [2](#0-1) 

**Validation does not check `chunk_hash` against the canonical chunk.** `validate_chunk_endorsement` only verifies: (1) the account is a chunk validator for `{shard_id, epoch_id, height_created}`, and (2) the cryptographic signature is self-consistent (i.e., `signature` covers `inner` which contains the endorsement's own `chunk_hash`, and `metadata_signature` covers `{account_id, shard_id, epoch_id, height_created}`). Neither check compares the endorsement's `chunk_hash` to any canonical value. [3](#0-2) 

The `ChunkEndorsementV2::verify` confirms this: `metadata` does not include `chunk_hash`, so a valid endorsement for `chunk_hash X` has a fully valid `metadata_signature` regardless of what the canonical chunk is. [4](#0-3) 

**Cache write** (line 60-63):

```rust
cache.get_or_insert_mut(key, || HashMap::new()).insert(
    account_id.clone(),
    (endorsement.chunk_hash(), endorsement.signature()),
);
```

Stores `(chunk_hash_X, sig_for_X)` under `account_id`. [5](#0-4) 

**Collection filter** (`collect_chunk_endorsements`, line 109):

```rust
.filter(|(_, (chunk_hash, _))| chunk_hash == chunk_header.chunk_hash())
```

Only endorsements whose stored `chunk_hash` matches the canonical chunk are counted. The poisoned slot (`chunk_hash_X ≠ chunk_hash_Y`) is silently excluded. [6](#0-5) 

### Impact Explanation

A block producer calling `collect_chunk_endorsements` for the canonical chunk will see zero stake from every poisoned validator. If the attacker poisons enough validator slots before legitimate endorsements arrive, `compute_endorsement_state` returns `is_endorsed = false`, and the block producer cannot include the chunk with sufficient endorsement. This is a concrete stateless-validation consensus impact: the endorsement state for the canonical chunk is permanently wrong for the lifetime of the cache entry.

### Likelihood Explanation

**Attack vector — fork replay (unprivileged):** NEAR forks are routine. When two block producers produce competing blocks at the same height, the same `(shard_id, epoch_id, height_created)` maps to two different `chunk_hash` values. Chunk validators send endorsements for the chunk they validated. A network peer connected to nodes on both forks can observe a legitimate endorsement for `chunk_hash X` (fork A) and forward it to a block producer on fork B (canonical `chunk_hash Y`). The forwarded message carries a fully valid signature — the validator signed it — so it passes every check in `validate_chunk_endorsement`. The peer needs no private key and no special privilege beyond network connectivity.

**Timing:** The attacker must deliver the fork-A endorsement before the legitimate fork-B endorsement arrives. Given that endorsements are sent directly to block producers and network propagation is fast, this is a realistic race condition, especially if the attacker is topologically close to the target block producer.

### Recommendation

Change the deduplication guard to also compare the stored `chunk_hash`. If the incoming endorsement carries a different `chunk_hash` than what is cached, allow it to overwrite (or at minimum, do not drop it). Alternatively, key the inner `HashMap` on `(AccountId, ChunkHash)` instead of `AccountId` alone, and let `collect_chunk_endorsements` filter as it already does. The simplest fix:

```rust
if cache.peek(&key).is_some_and(|entry| {
    entry.get(account_id).is_some_and(|(cached_hash, _)| cached_hash == &endorsement.chunk_hash())
}) {
    return Ok(());
}
```

### Proof of Concept

1. Set up a local two-node NEAR network with a fork at height H: node A has chunk `chunk_hash X`, node B has chunk `chunk_hash Y`, same `(shard_id, epoch_id, height_created)`.
2. Observe the `ChunkEndorsement` for `chunk_hash X` sent by validator `V` on fork A.
3. Forward that endorsement to the block producer on fork B before validator `V`'s endorsement for `chunk_hash Y` arrives.
4. The block producer's `process_chunk_endorsement` accepts it (valid signature, valid validator, valid epoch) and stores `(chunk_hash_X, sig_for_X)` under `account_id = V`.
5. When validator `V`'s legitimate endorsement for `chunk_hash Y` arrives, the guard at line 50 fires (`entry.contains_key(V)` is true) and returns `Ok(())` without updating the cache.
6. Call `collect_chunk_endorsements` for the canonical chunk header (with `chunk_hash Y`): the filter at line 109 excludes `V`'s entry because `chunk_hash_X ≠ chunk_hash_Y`.
7. Assert: `V`'s stake is absent from `endorsed_chunk_validators`; `is_endorsed` may be false if enough validators are poisoned. [7](#0-6) [8](#0-7) [9](#0-8)

    pub fn process_chunk_endorsement(&self, endorsement: &ChunkEndorsement) -> Result<(), Error> {
        // Check if we have already received chunk endorsement from this validator.
        let key = endorsement.chunk_production_key();
        let account_id = endorsement.account_id();

        {
            let cache = self.chunk_endorsements.lock();
            if cache.peek(&key).is_some_and(|entry| entry.contains_key(account_id)) {
                tracing::debug!(target: "client", ?endorsement, "already received chunk endorsement");
                return Ok(());
            }
        }

        // Validate the chunk endorsement and store it in the cache.
        match validate_chunk_endorsement(self.epoch_manager.as_ref(), endorsement, &self.store)? {
            ChunkRelevance::Relevant => {
                let mut cache = self.chunk_endorsements.lock();
                cache.get_or_insert_mut(key, || HashMap::new()).insert(
                    account_id.clone(),
                    (endorsement.chunk_hash(), endorsement.signature()),
                );
                let shard_id = endorsement.shard_id();
                metrics::CHUNK_ENDORSEMENTS_ACCEPTED
                    .with_label_values(&[&shard_id.to_string()])
                    .inc();
            }
            irrelevant => {
                metrics::CHUNK_ENDORSEMENTS_REJECTED
                    .with_label_values::<&str>(&[
                        endorsement.shard_id().to_string().as_str(),
                        irrelevant.into(),
                    ])
                    .inc();
            }
        };
        Ok(())
    }
