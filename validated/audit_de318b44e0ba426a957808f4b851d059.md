### Title
User-Controlled Zero-Confirmation Threshold Bypasses Bitcoin Finality in `verify_foreign_transaction` - (File: `crates/foreign-chain-inspector/src/bitcoin/inspector.rs`)

### Summary

The `BitcoinRpcRequest.confirmations` field is fully user-controlled with no minimum bound enforced at the contract or node level. An unprivileged caller can submit `confirmations: 0`, causing MPC nodes to sign an attestation for an unconfirmed (or zero-confirmation) Bitcoin transaction. The resulting MPC signature is indistinguishable from one produced for a properly-finalized transaction, enabling a bridge double-spend.

### Finding Description

The `verify_foreign_transaction` flow accepts a `BitcoinRpcRequest` whose `confirmations: BlockConfirmations` field is a plain `u64` with no enforced minimum:

```rust
// crates/near-mpc-contract-interface/src/types/foreign_chain.rs:1282
pub struct BlockConfirmations(pub u64);
```

The ABI schema confirms `"minimum": 0.0` — zero is a valid on-wire value.

The contract's `verify_foreign_transaction` queues the request without inspecting the `confirmations` value:

```rust
// crates/contract/src/lib.rs:549-556
let request = args_into_verify_foreign_tx_request(request);
...
self.enqueue_yield_request(
    method_names::RETURN_VERIFY_FOREIGN_TX_AND_CLEAN_STATE_ON_SUCCESS,
    ...
```

MPC nodes pass the user-supplied value directly as the finality threshold to `BitcoinInspector::extract`:

```rust
// crates/foreign-chain-inspector/src/bitcoin/inspector.rs:50-58
let transaction_block_confirmation = rpc_response.confirmations.into();
let enough_block_confirmations =
    block_confirmations_threshold <= transaction_block_confirmation;

if !enough_block_confirmations {
    return Err(ForeignChainInspectionError::NotEnoughBlockConfirmations { ... });
}
```

When `block_confirmations_threshold = 0`, the comparison `0 <= actual_confirmations` is trivially true for any transaction the RPC returns, including transactions with 0 confirmations. The nodes then sign a canonical payload of `(request, extracted_values)` where `request` embeds `confirmations: 0`, and `respond_verify_foreign_tx` on the contract validates only the ECDSA signature over that payload — not the safety of the embedded parameters:

```rust
// crates/contract/src/lib.rs:718-747
let signature_is_valid = match (&response.signature, public_key) {
    ...
    near_mpc_signature_verifier::verify_ecdsa_signature(...).is_ok()
    ...
};
if !signature_is_valid {
    return Err(RespondError::InvalidSignature.into());
}
```

The contract accepts and resolves the yield, returning the MPC signature to the caller.

### Impact Explanation

This is a **High** impact finding matching the allowed category: *"forged foreign-chain verification... that causes invalid bridge execution or double-spend conditions."*

A bridge contract (e.g., Omnibridge inbound flow) that relies on `verify_foreign_transaction` to confirm a Bitcoin deposit before releasing NEAR-side funds would receive a valid MPC signature over an unconfirmed transaction. The attacker can:

1. Broadcast a Bitcoin transaction to the mempool.
2. Immediately call `verify_foreign_transaction` with `confirmations: 0`.
3. Receive a valid MPC-signed attestation.
4. Claim NEAR-side funds using the signature.
5. RBF or double-spend the Bitcoin transaction.

The MPC signature is cryptographically valid — the bridge contract has no way to distinguish it from a signature over a 6-confirmation transaction.

### Likelihood Explanation

**High.** Any unprivileged NEAR account can call `verify_foreign_transaction` with `confirmations: 0` and a deposit of 1 yoctoNEAR. No special access, key material, or threshold collusion is required. The attack is deterministic and repeatable.

### Recommendation

Enforce a protocol-level minimum on `BlockConfirmations` at the contract boundary before the request is enqueued. A reasonable minimum (e.g., 1 for any finality, 6 for economic finality) should be a named constant voted in alongside the chain's configuration, or at minimum hard-coded as a floor in `verify_foreign_transaction`:

```rust
// In verify_foreign_transaction, before enqueue_yield_request:
if let ForeignChainRpcRequest::Bitcoin(ref req) = request.request {
    if req.confirmations.0 < MIN_BITCOIN_CONFIRMATIONS {
        env::panic_str("confirmations below protocol minimum");
    }
}
```

Alternatively, make `BlockConfirmations` a newtype that is only constructible with a value ≥ 1, enforced in its `Deserialize` / `BorshDeserialize` implementations.

        );

                .to_string(),
