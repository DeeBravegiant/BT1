### Title
`initTransfer()` / `initTransfer1155()` read `currentOriginNonce` from storage after an attacker-controlled external call, enabling reentrancy to emit duplicate `InitTransfer` events with unbacked NEAR-side credit — (`File: evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.initTransfer()` and `initTransfer1155()` increment `currentOriginNonce` before any external call (partial CEI compliance), but then pass the **storage variable** — not a local snapshot — to `initTransferExtension()` and `emit InitTransfer()` after the external call. A malicious ERC-20 or ERC-1155 contract can reenter either function during the external call, increment `currentOriginNonce` a second time, and cause both the inner and outer execution frames to emit `InitTransfer` with the same (higher) nonce, while nonce N is silently skipped. Because the NEAR side treats every emitted `InitTransfer` event as proof that tokens are locked, the attacker obtains a NEAR-side credit without locking any tokens on EVM.

---

### Finding Description

**Root cause — storage variable read after external call**

In `initTransfer` the nonce is incremented at line 381, but `currentOriginNonce` is read from storage again at lines 418 and 430 — after the external token call:

```solidity
// OmniBridge.sol lines 373-436
function initTransfer(...) external payable whenNotPaused(PAUSED_INIT_TRANSFER) {
    currentOriginNonce += 1;                    // ← incremented to N (line 381)
    ...
    // ↓ external call to attacker-controlled token (lines 407-411)
    IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
    ...
    initTransferExtension(
        msg.sender, tokenAddress,
        currentOriginNonce,                     // ← reads storage, now N+1 after reentry (line 418)
        ...
    );
    emit BridgeTypes.InitTransfer(
        msg.sender, tokenAddress,
        currentOriginNonce,                     // ← reads storage, now N+1 after reentry (line 430)
        ...
    );
}
```

`initTransfer1155` has the identical pattern: nonce incremented at line 448, then `IERC1155(tokenAddress).safeTransferFrom(...)` at line 458 (attacker-controlled), then `currentOriginNonce` read from storage at lines 470 and 483.

There is **no `nonReentrant` modifier** anywhere in the EVM contracts (confirmed: zero matches for `nonReentrant` / `ReentrancyGuard` in `evm/src/**/*.sol`). `SelectivePausableUpgradable` provides only pause flags, not reentrancy protection.

**Attack path (ERC-20 variant)**

1. Attacker deploys `MaliciousERC20` whose `transferFrom` reenters `initTransfer` once, then returns `true` without transferring any tokens.
2. Attacker calls `OmniBridge.initTransfer(MaliciousERC20, amount, fee, 0, "near:attacker", "")`.
3. OmniBridge: `currentOriginNonce` → N.
4. OmniBridge calls `MaliciousERC20.transferFrom(attacker, OmniBridge, amount)`.
5. `MaliciousERC20` reenters `initTransfer(MaliciousERC20, amount, fee, 0, "near:attacker", "")`.
6. OmniBridge: `currentOriginNonce` → N+1.
7. Inner `MaliciousERC20.transferFrom` returns `true` (no tokens moved).
8. Inner frame: `initTransferExtension(..., N+1, ...)` and `emit InitTransfer(..., N+1, ...)` — **event A**.
9. Outer `MaliciousERC20.transferFrom` returns `true` (no tokens moved).
10. Outer frame: `initTransferExtension(..., N+1, ...)` and `emit InitTransfer(..., N+1, ...)` — **event B** (duplicate nonce).

**Result:** Nonce N is never emitted. Nonce N+1 is emitted twice. Zero tokens are locked on EVM. The NEAR side sees at least one valid `InitTransfer` for nonce N+1 and credits the attacker.

The ERC-1155 variant (`initTransfer1155`) is identical: the malicious ERC-1155 reenters during `safeTransferFrom` at line 458 before `onERC1155Received` is invoked, so OmniBridge's `onERC1155Received` operator check (line 530) is never reached.

---

### Impact Explanation

**Critical — unauthorized creation of wrapped bridge assets.**

The NEAR side's documented invariant is: *"The NEAR side will treat any emitted event as proof that tokens are held."* A duplicate `InitTransfer` with nonce N+1 and no locked collateral directly violates the backing guarantee. The attacker receives NEAR-side wrapped tokens (or native NEAR value) for free. Nonce N is permanently skipped, meaning any legitimate transfer that was assigned nonce N is silently lost.

This maps directly to the allowed impact: *"Unauthorized creation, release, withdrawal, or custody escape of native, locked, or wrapped bridge assets through settlement, deployment, or verification failure."*

---

### Likelihood Explanation

**High.** `initTransfer` is a fully public, permissionless entry point. Any caller can supply an arbitrary `tokenAddress`; the non-bridge, non-custom ERC-20 branch (lines 407–411) and the ERC-1155 branch (line 458) both accept any address. No admin key, leaked secret, or privileged role is required. Deploying a malicious ERC-20 costs only gas.

---

### Recommendation

Capture `currentOriginNonce` into a local variable immediately after incrementing it, before any external call, and use only the local variable in `initTransferExtension` and `emit`:

```solidity
function initTransfer(...) external payable whenNotPaused(PAUSED_INIT_TRANSFER) {
    currentOriginNonce += 1;
    uint64 originNonce = currentOriginNonce;   // ← local snapshot
    ...
    IERC20(tokenAddress).safeTransferFrom(...);
    ...
    initTransferExtension(..., originNonce, ...);
    emit BridgeTypes.InitTransfer(..., originNonce, ...);
}
```

Apply the same fix to `initTransfer1155`. Alternatively, add OpenZeppelin's `ReentrancyGuardUpgradeable` and apply `nonReentrant` to both functions.

```markdown
- **State before external calls**: Always mutate state (e.g. mark nonce used) before any external call (token transfer, ETH send, custom minter). This is the primary reentrancy defense
- **No token release without signature**: Never mint, transfer, or unlock tokens to a recipient without first verifying a valid MPC signature. No admin function, emergency path, or refactor may bypass this — it is the only authorization gate for finTransfer
- **Event–transfer atomicity**: `InitTransfer` must only be emitted in a code path where tokens have already been burned/locked in the same transaction. If the token transfer reverts or is skipped, the event must not emit — the NEAR side will treat any emitted event as proof that tokens are held
```
