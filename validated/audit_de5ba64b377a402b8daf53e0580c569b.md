## Title
SwapAllowlistExtension Bypassed via MetricOmmSimpleRouter â€” Any User Can Swap on Restricted Pools â€” (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which is set to `msg.sender` of the `pool.swap()` call. When users route through `MetricOmmSimpleRouter`, the router contract becomes that `sender`. If the router is allowlisted â€” a prerequisite for any legitimate user to use it on a restricted pool â€” every user who calls the router passes the check regardless of whether they are individually authorized.

## Finding Description

`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the first argument forwarded from the pool: [1](#0-0) 

`MetricOmmPool.swap` passes its own `msg.sender` as `sender` to `_beforeSwap`: [2](#0-1) 

`ExtensionCalling._beforeSwap` forwards that value unchanged into the extension call: [3](#0-2) 

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap(...)` directly, making the router the pool's `msg.sender`: [4](#0-3) 

The same pattern applies to `exactInput`, `exactOutputSingle`, and `exactOutput`: [5](#0-4) 

The result: the extension evaluates `allowedSwapper[pool][router]` for every user who routes through the router. Once the router is allowlisted (required for any legitimate user to use it), the per-user allowlist is completely inoperative for router-mediated swaps.

## Impact Explanation

A pool configured with `SwapAllowlistExtension` to restrict swaps to specific addresses (e.g., KYC'd counterparties, institutional LPs, whitelisted market makers) is fully bypassed for any user routing through `MetricOmmSimpleRouter`. The attacker executes swaps at oracle-anchored prices that the pool was designed to prohibit, breaking the core access-control invariant of the pool. This constitutes broken core pool functionality with potential loss of LP assets, meeting the contest's Medium/High threshold.

## Likelihood Explanation

The precondition â€” a pool with `SwapAllowlistExtension` active and the router allowlisted â€” is the normal operational state for any restricted pool that intends to support router-based access. Allowlisting the router is not an edge case; it is required for legitimate users to use the router at all. Any unprivileged user can exploit this by simply calling the public router.

## Recommendation

The extension must identify the original user, not the intermediate contract. Two approaches:

1. **Pass the original user through `extensionData`:** The router encodes `msg.sender` into `extensionData`; the extension decodes and checks it. This requires a trusted router or a signed payload.
2. **Transient-storage original caller:** The pool sets a transient-storage slot to the original caller at entry; the extension reads it instead of relying on `sender`.

The simplest safe fix: require routers to forward the original user identity in a verifiable way (e.g., via `extensionData`), and have the extension decode and check that identity rather than the intermediate `sender`.

## Proof of Concept

1. Pool admin deploys a pool with `SwapAllowlistExtension` configured.
2. Admin allowlists only a KYC'd user: `setAllowedToSwap(pool, kyc_user, true)`.
3. Admin allowlists the router so KYC'd users can use it: `setAllowedToSwap(pool, router, true)`.
4. Non-KYC'd attacker calls:
   ```solidity
   router.exactInputSingle(ExactInputSingleParams({
       pool: pool,
       recipient: attacker,
       zeroForOne: true,
       amountIn: X,
       ...
   }));
   ```
5. Router calls `pool.swap(attacker, true, X, ...)` with `msg.sender = router`.
6. Pool calls `_beforeSwap(router, attacker, ...)`.
7. Extension evaluates `allowedSwapper[pool][router]` â†’ `true` â†’ no revert.
8. Swap executes. Attacker receives output tokens at oracle price.
9. The allowlist is fully bypassed; the attacker was never individually authorized.
