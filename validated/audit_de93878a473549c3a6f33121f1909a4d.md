### Title
Gateway Skips `__validate__` for Invoke Transactions When Any Account Transaction Exists in Mempool, Allowing Invalid Signatures to Bypass Admission — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator is designed to improve UX for the deploy_account + invoke simultaneous submission pattern. However, it uses an overly broad condition — `account_tx_in_pool_or_recent_block` — that returns `true` for **any** pending transaction, not just a `deploy_account`. This allows an unprivileged attacker to submit an invoke transaction with an **invalid signature** (nonce 1, account on-chain nonce 0) and have it admitted to the mempool without running the `__validate__` entry point, as long as any valid nonce-0 transaction for that account is already in the pool.

---

### Finding Description

`skip_stateful_validations` in `crates/apollo_gateway/src/stateful_transaction_validator.rs` contains the following logic: [1](#0-0) 

The condition triggers when:
- The transaction is an `Invoke`
- `tx.nonce() == Nonce(Felt::ONE)`
- `account_nonce == Nonce(Felt::ZERO)` (account not yet used on-chain)

When all three hold, it calls `account_tx_in_pool_or_recent_block` and returns the result as the `skip_validate` flag. The implementation of that check is: [2](#0-1) 

This returns `true` if the account has **any** transaction in the pool — including a plain nonce-0 `Invoke` — not specifically a `DeployAccount`. The comment in `skip_stateful_validations` claims "It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations," but this reasoning is incorrect: a nonce-0 `Invoke` satisfies the check without any `DeployAccount` being present.

When `skip_validate = true` is returned, `run_validate_entry_point` sets `ExecutionFlags { validate: false }`: [3](#0-2) 

With `validate: false`, `blockifier_validator.validate(account_tx)` immediately returns `Ok(None)` without executing the `__validate__` entry point: [4](#0-3) 

The transaction then proceeds through `add_tx_inner` and is inserted into the mempool via `mempool_client.add_tx`: [5](#0-4) 

The `InternalRpcTransaction` stored in the mempool carries no `skip_validate` flag, so the batcher will attempt to execute it with `validate: true` — but the mempool has already accepted an invalid transaction.

---

### Impact Explanation

**High. Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

An attacker can submit an `Invoke` transaction with an **invalid or forged signature** (nonce 1) for any account that has a pending nonce-0 transaction in the mempool. The gateway skips `__validate__`, and the transaction is admitted to the mempool without signature verification. The mempool's invariant — that all admitted transactions have passed account validation — is broken. The batcher will later reject the transaction during execution (since it runs `__validate__` independently), but the mempool has already accepted and is holding an invalid transaction, wasting resources and potentially disrupting sequencing for the targeted account.

---

### Likelihood Explanation

**High.** The attack requires no privileges. Any observer of the public mempool can identify accounts with pending nonce-0 transactions (a common state for newly active accounts) and immediately submit a nonce-1 invoke with an arbitrary or invalid signature. The condition is trivially satisfiable on a live network.

---

### Recommendation

Replace the `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a `DeployAccount` transaction is pending for the account. The mempool should expose a dedicated `has_pending_deploy_account(address)` query, or the pool lookup should filter by transaction type. The current check conflates "account is known to the mempool" with "a deploy_account is pending," which is incorrect.

---

### Proof of Concept

1. Account `A` has on-chain nonce `0` (never executed a transaction).
2. The legitimate owner of `A` submits a valid nonce-0 `Invoke` transaction; it is accepted into the mempool.
3. Attacker calls `gateway.add_tx(invoke_nonce_1_invalid_sig_for_A, None)`.
4. Gateway stateful validator:
   - `account_nonce = 0` (from state)
   - `tx.nonce() == 1` ✓, `account_nonce == 0` ✓
   - `account_tx_in_pool_or_recent_block(A)` → `true` (because of the nonce-0 invoke in step 2)
   - `skip_validate = true`
   - `run_validate_entry_point(tx, skip_validate=true)` → `ExecutionFlags { validate: false }` → `Ok(None)` (no `__validate__` call)
5. `extract_state_nonce_and_run_validations` returns `Ok(0)`.
6. `mempool_client.add_tx(...)` is called; the invalid-signature transaction is inserted into the mempool.
7. The mempool now holds a transaction for account `A` with an invalid signature that bypassed all account-level validation at the gateway.

The existing test `test_skip_validate` with `#[case::should_skip_validation]` confirms this path is reachable: [6](#0-5)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
