### Title
Stale `VerifyForeignTransactionResponse` Replay by a Single Malicious Participant — (`File: crates/contract/src/lib.rs`)

### Summary

`respond_verify_foreign_tx` verifies only that the submitted signature is cryptographically valid over the caller-supplied `payload_hash`, but never binds that `payload_hash` to the *current* pending-request instance. A single malicious attested participant (strictly below the signing threshold) can observe a previously accepted on-chain response, then replay it against a newly queued pending entry for the same request key, delivering stale extracted values to every waiting caller.

### Finding Description

The vulnerability class from the reference report is: a persistent authorization (the ERC-20 `approve`) is written before an action (the `safeTransfer`), and the residual allowance can be exploited for a second unauthorized withdrawal. The analog here is that a valid MPC signature over a `payload_hash` is a persistent, reusable authorization — once produced and published on-chain, it can be replayed by any single attested participant against any future pending entry that shares the same request key.

`respond_verify_foreign_tx` performs two checks:

1. The caller is an attested participant.
2. `response.signature` is a valid ECDSA signature over `response.payload_hash` under the MPC root public key. [1](#0-0) 

It does **not** verify that `response.payload_hash` was freshly derived from the current pending request. The `payload_hash` is taken verbatim from the caller-supplied `response` struct: [2](#0-1) 

After a successful call, `resolve_yields_for` removes the request from the pending map: [3](#0-2) 

However, the `VerifyForeignTransactionRequest` key contains no nonce, no timestamp, and no per-instance identifier — only `domain_id`, `payload_version`, and the chain-specific RPC parameters: [4](#0-3) 

This means the same request key can be re-enqueued after the first response is consumed, and any previously accepted `(payload_hash, signature)` pair remains valid for the new pending entry.

### Impact Explanation

A bridge service that retries a timed-out or re-submitted `verify_foreign_transaction` call for the same foreign transaction (same `tx_id`, same extractors) creates a new pending entry under the identical request key. A single malicious attested participant who observed the old on-chain response can immediately call `respond_verify_foreign_tx` with the stale `(payload_hash_old, sig_old)`. The contract accepts it — the signature is still cryptographically valid, and the request key exists in pending state. Every caller waiting on that entry receives the old extracted values (e.g., a block hash from a since-reorganized chain). For an omnibridge inbound flow, this enables a double-spend: the bridge releases funds based on a stale confirmation that no longer reflects the canonical chain state.

This matches the allowed High impact: *"Cross-chain replay, forged foreign-chain verification, light-client-style verification bypass, or participant/attestation authorization bypass that causes invalid bridge execution or double-spend conditions."*

### Likelihood Explanation

The attack requires only a single malicious attested participant — strictly below the signing threshold. The old response is already public on-chain (submitted by an honest participant in the first round), so the attacker needs no threshold cooperation and no key material beyond their own node credentials. The trigger condition (a retry of the same foreign-tx request) is a normal operational pattern for any bridge service.

### Recommendation

Bind the response to the specific pending-request instance by including a per-enqueue nonce (e.g., a monotonic counter or the NEAR yield `data_id`) in the `VerifyForeignTransactionRequest` key, or by having the contract recompute `payload_hash` from the `request` fields and the submitted `extracted_values` and reject any response whose `payload_hash` does not match. The node-side code already computes `payload_hash` deterministically from `(request, extracted_values)`: [5](#0-4) 

The contract should perform the same derivation and reject responses that deviate.

### Proof of Concept

1. Bridge submits `verify_foreign_transaction` for Bitcoin `tx_id = X` with 1 confirmation. MPC nodes query the chain, extract `block_hash = B1`, compute `payload_hash_1 = hash(request, {BlockHash: B1})`, and sign it. An honest participant calls `respond_verify_foreign_tx(request, {payload_hash: payload_hash_1, sig: sig1})`. The contract resolves the yield and removes the pending entry. The bridge receives the response.

2. The Bitcoin chain reorganizes; `tx_id = X` is now in block `B2` (or dropped). The bridge retries by calling `verify_foreign_transaction` again with the identical arguments, creating a new pending entry under the same request key.

3. A malicious attested participant (who observed `sig1` on-chain in step 1) immediately calls:
   ```
   respond_verify_foreign_tx(
     request = { domain_id, payload_version, request: { tx_id: X, confirmations: 1, extractors: [BlockHash] } },
     response = { payload_hash: payload_hash_1, signature: sig1 }
   )
   ```

4. The contract checks: (a) caller is attested ✓, (b) `sig1` is a valid MPC signature over `payload_hash_1` ✓, (c) the request key exists in pending state ✓. `resolve_yields_for` drains the new pending entry with the stale response.

5. The bridge receives `{payload_hash: payload_hash_1, signature: sig1}` — a response attesting to block `B1`, which is no longer canonical — and incorrectly releases funds, enabling a double-spend. [6](#0-5) [7](#0-6)

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

/// Resume every yield queued for `request` with `response_bytes`, draining the
/// fan-out map in one pass. Returns `Err(RequestNotFound)` if the map held no entry.
///
/// Resuming a yield that has already timed out is a no-op at the SDK level.
pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
