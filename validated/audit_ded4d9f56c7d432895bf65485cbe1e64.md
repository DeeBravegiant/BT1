Audit Report

## Title
Nested Loop Gas Exhaustion in `updateRSETHPrice()` Can Permanently Stale rsETH Price - (File: contracts/LRTOracle.sol)

## Summary
`LRTOracle.updateRSETHPrice()` is a public function that triggers a nested call chain: for each supported asset, it calls `LRTDepositPool.getAssetDistributionData()`, which for each NDC calls `NodeDelegator.getAssetUnstaking()`, which in turn calls `DelegationManager.getQueuedWithdrawals()` and iterates over all queued withdrawal strategies. The total number of external calls scales as `N_assets × M_NDCs × K_withdrawals`, and the developers' own comment acknowledges the gas concern but the existing cap of 80 total withdrawals does not account for the N-asset multiplication factor. If the function reverts due to gas exhaustion, `rsETHPrice` becomes permanently stale, breaking deposits, withdrawals, and fee minting.

## Finding Description

`LRTOracle._getTotalEthInProtocol()` iterates over every supported asset and calls `getTotalAssetDeposits(asset)` for each: [1](#0-0) 

`getTotalAssetDeposits` routes to `getAssetDistributionData`, which for non-ETH assets loops over all NDCs and calls `getAssetUnstaking` on each: [2](#0-1) 

For ETH, `getETHDistributionData` performs the same NDC loop: [3](#0-2) 

`getAssetUnstaking` makes one `getQueuedWithdrawals` external call to EigenLayer per invocation, then iterates over all queued withdrawals and their strategies, calling `strategy.sharesToUnderlyingView()` for each non-ETH match: [4](#0-3) 

The public entry point has no access control beyond `whenNotPaused`: [5](#0-4) 

The critical flaw in the existing mitigation is that `maxUncompletedWithdrawalCount` (capped at 80) is a **global** counter across all NDCs, but `getAssetUnstaking` is called **once per asset per NDC**. With N assets and M NDCs, `getQueuedWithdrawals` is called `N × M` times (e.g., 5 assets × 10 NDCs = 50 external calls), and the withdrawal iteration count is `N × total_withdrawals` (e.g., 5 × 80 = 400 iterations), plus `sharesToUnderlyingView` calls for each matching non-ETH withdrawal. The developers' own comment acknowledges the gas concern but only reasons about the withdrawal count in isolation: [6](#0-5) 

`maxNodeDelegatorLimit` starts at 10 and can be raised by admin with no hard upper bound enforced in the loop path: [7](#0-6) [8](#0-7) 

## Impact Explanation

If `updateRSETHPrice()` reverts due to gas exhaustion, `rsETHPrice` becomes permanently stale. `updateRSETHPriceAsManager()` calls the same `_updateRsETHPrice()` and would also fail: [9](#0-8) 

A stale `rsETHPrice` directly breaks `getRsETHAmountToMint` used in `depositETH`/`depositAsset`, and all withdrawal computations in `LRTWithdrawalManager` that rely on the stored price. This constitutes **Medium — Unbounded gas consumption** escalating to **temporary or permanent freezing of the price update mechanism**.

## Likelihood Explanation

`updateRSETHPrice()` is callable by any unprivileged external user. The gas-exhaustion state is reached through normal protocol operation: as EigenLayer queued withdrawals accumulate (up to 80 globally, distributed across up to 10 NDCs, across 5+ supported assets), the multiplicative call count grows to `5 × 10 = 50` `getQueuedWithdrawals` external calls plus up to `5 × 80 = 400` withdrawal iterations with additional `sharesToUnderlyingView` calls. The developers' own comment ("120 is the max number of uncompleted withdrawals that allows us to still perform update rsETH price") confirms awareness of the gas budget, but the analysis omits the N-asset multiplier, making the actual safe limit significantly lower than 80.

## Recommendation

1. Restructure `_getTotalEthInProtocol` to call `getQueuedWithdrawals` once per NDC (not once per asset per NDC), caching the result and summing across all assets in a single pass.
2. Alternatively, store a periodically-updated snapshot of unstaking amounts (updated by a privileged keeper) and use the snapshot in `_getTotalEthInProtocol` instead of live EigenLayer state.
3. Revise the `maxUncompletedWithdrawalCount` cap to account for the `N_assets` multiplier: the effective safe limit is `block_gas_budget / (gas_per_getQueuedWithdrawals × N_assets × M_NDCs)`.

## Proof of Concept

Call path for a single `updateRSETHPrice()` with 5 assets, 10 NDCs, 8 withdrawals per NDC (80 total):

```
updateRSETHPrice()                                          // LRTOracle.sol:87
  └─ _updateRsETHPrice()
       └─ _getTotalEthInProtocol()                          // LRTOracle.sol:331
            └─ for assetIdx in supportedAssets (5 iters)   // LRTOracle.sol:336
                 └─ getTotalAssetDeposits(asset)            // LRTDepositPool.sol:385
                      └─ getAssetDistributionData(asset)
                           └─ for i in nodeDelegatorQueue (10 iters)  // LRTDepositPool.sol:447
                                └─ getAssetUnstaking(asset)           // NodeDelegator.sol:405
                                     └─ getQueuedWithdrawals()        // 1 external call → EigenLayer
                                     └─ for withdrawal in results (8 iters)
                                          └─ strategy.sharesToUnderlyingView()  // external call

Total getQueuedWithdrawals calls: 5 × 10 = 50
Total withdrawal iterations:      5 × 80 = 400
Total sharesToUnderlyingView:     up to 400 (for non-ETH assets)
```

A Foundry fork test can demonstrate this by deploying the protocol with 5 supported assets, 10 NDCs, and queuing 8 withdrawals per NDC via `initiateUnstaking`, then calling `updateRSETHPrice()` with a gas limit matching mainnet's block gas limit (~30M gas) and asserting the call reverts with out-of-gas.

    function getAssetUnstaking(address asset) external view returns (uint256 amount) {
        (IDelegationManager.Withdrawal[] memory queuedWithdrawals, uint256[][] memory withdrawalShares) =
            _getDelegationManager().getQueuedWithdrawals(address(this));

        for (uint256 withdrawalIndex = 0; withdrawalIndex < queuedWithdrawals.length; withdrawalIndex++) {
            IDelegationManager.Withdrawal memory withdrawal = queuedWithdrawals[withdrawalIndex];

            for (uint256 strategyIndex = 0; strategyIndex < withdrawal.strategies.length; strategyIndex++) {
                IStrategy strategy = withdrawal.strategies[strategyIndex];

                address strategyAsset = address(strategy) == address(lrtConfig.beaconChainETHStrategy())
                    ? LRTConstants.ETH_TOKEN
                    : address(strategy.underlyingToken());

                if (strategyAsset != asset) continue;

                uint256 sharesToUnstake = withdrawalShares[withdrawalIndex][strategyIndex];
                amount += strategyAsset == LRTConstants.ETH_TOKEN
                    ? sharesToUnstake
                    : strategy.sharesToUnderlyingView(sharesToUnstake);
            }
        }
    }
