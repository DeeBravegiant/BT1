### Title
Raw Internal Error Details Leaked in Transaction RPC Responses via Unguarded `debug_info` Field - (File: chain/jsonrpc/src/api/transactions.rs, chain/jsonrpc-primitives/src/types/transactions.rs)

### Summary
The `RpcTransactionError::InternalError { debug_info }` variant serializes raw internal error strings — including actor messaging errors and chain error debug representations — directly into the JSON-RPC response body. Unlike every other sensitive error field in the codebase, `debug_info` is not marked `#[serde(skip_serializing)]`, so it is returned verbatim to any unprivileged caller of the transaction RPC endpoints.

### Finding Description

The `RpcTransactionError` enum defines an `InternalError` variant with a `debug_info: String` field: [1](#0-0) 

Unlike the `InvalidTransaction.context` field in the same enum, which is explicitly suppressed with `#[serde(skip_serializing)]`: [2](#0-1) 

`debug_info` carries no such annotation and is therefore serialized into the response. Three distinct code paths populate this field with raw internal data and return it to callers:

**Path 1 — `AsyncSendError` (actor mailbox failure):** [3](#0-2) 

**Path 2 — `TxStatusError::ChainError` (raw chain error debug representation):** [4](#0-3) 

**Path 3 — `ProcessTxResponse` catch-all (any unrecognized response variant):** [5](#0-4) 

The `From<RpcTransactionError> for RpcError` conversion then places `error.to_string()` — which interpolates `debug_info` verbatim — into the `data` field of the outgoing `RpcError`: [6](#0-5) 

Inside `new_internal_or_handler_error`, the function looks for `error_struct["info"]["error_message"]` but the serialized key is `debug_info`, so the lookup fails and falls to the generic fallback. However, `error_data` — which already contains the full string with `debug_info` — is passed unchanged as the `data` field of the final `RpcError`: [7](#0-6) 

The resulting JSON-RPC response sent to the client contains:
```json
{
  "code": -32000,
  "message": "Server error",
  "data": "The node reached its limits. Try again later. More details: <raw internal error>"
}
```

The contrast with other error types in the codebase is explicit. For example, `RpcChunkError::UnknownBlock` and `RpcGasPriceError::UnknownBlock` both mark their `error_message` fields with `#[serde(skip_serializing)]`: [8](#0-7) 

### Impact Explanation

Any unprivileged user calling `tx_status`, `broadcast_tx_commit`, or `EXPERIMENTAL_tx_status` who triggers an internal error receives the raw debug representation of internal node state in the `data` field of the JSON-RPC response. This includes:

- Actor mailbox error strings (revealing internal actor topology)
- `format!("{:?}", ChainError)` output (revealing internal chain state, file paths, and Rust module paths embedded in `Debug` impls)
- Arbitrary `TxStatusError::InternalError` strings propagated from lower layers

The exact corrupted RPC result is the `data` field of the `RpcError` response, which contains internal debug information that should never be client-visible.

### Likelihood Explanation

Any user can submit a transaction to a node under load (causing `AsyncSendError` from a full actor mailbox) or query the status of a transaction that triggers a chain error. Both are reachable without any special privileges. The node's public RPC port (default 3030) is the entry point.

### Recommendation

Mark `debug_info` with `#[serde(skip_serializing)]` in `RpcTransactionError::InternalError`, matching the pattern already used for `InvalidTransaction.context` and the `error_message` fields in `RpcChunkError` and `RpcGasPriceError`. Log the raw debug string server-side only. Return a generic message to the client.

### Proof of Concept

1. Submit a transaction via `broadcast_tx_commit` to a node whose client actor mailbox is saturated (or craft a transaction that triggers a `TxStatusError::ChainError`).
2. The `RpcFrom<AsyncSendError>` or `RpcFrom<TxStatusError>` impl at `chain/jsonrpc/src/api/transactions.rs:54-72` constructs `RpcTransactionError::InternalError { debug_info: <raw error> }`.
3. `From<RpcTransactionError> for RpcError` at `chain/jsonrpc-primitives/src/types/transactions.rs:127` sets `error_data = Value::String(error.to_string())`, embedding `debug_info` in the string.
4. `new_internal_error` at `chain/jsonrpc-primitives/src/errors.rs:116-126` places this value in `RpcError.data`.
5. The client receives `"data": "The node reached its limits. Try again later. More details: <internal actor/chain error>"` in the JSON-RPC response body.
