### Title
Missing lower-bound staleness check in `is_proposal_init_valid` allows proposals with arbitrarily old timestamps to be accepted — (`crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

---

### Summary

`is_proposal_init_valid` enforces that a proposed block timestamp is not too far **in the future** (`timestamp <= now + block_timestamp_window_seconds`) but imposes no symmetric lower bound relative to `now`. The only lower bound is `timestamp >= last_block_timestamp` (the previous committed block's timestamp), which can be arbitrarily far in the past. A proposal carrying a stale timestamp — either from a malicious proposer or from a legitimately queued future-round proposal that sat in `queued_proposals` across many timed-out rounds — passes all validation checks and is committed with the wrong timestamp. This corrupts the L1 gas prices embedded in the block (which are looked up at `init_proposed.timestamp`) and the `get_block_timestamp()` syscall value seen by every contract executing in that block.

---

### Finding Description

**Invariant stated in config documentation:**

> `block_timestamp_window_seconds`: "Maximum allowed deviation (seconds) of a proposed block's timestamp from the current time."

The word "deviation" implies a symmetric window: `|now - timestamp| <= window`. The implementation enforces only one half:

```rust
// crates/apollo_consensus_orchestrator/src/validate_proposal.rs  lines 260-284
let now: u64 = clock.unix_now();
let last_block_timestamp =
    proposal_init_validation.previous_proposal_init.as_ref().map_or(0, |info| info.timestamp);
if init_proposed.timestamp < last_block_timestamp {          // lower bound = previous block
    return Err(...);
}
if init_proposed.timestamp > now + proposal_init_validation.block_timestamp_window_seconds {
    return Err(...);                                          // upper bound = now + window
}
// ← no check: init_proposed.timestamp >= now - block_timestamp_window_seconds
``` [1](#0-0) 

The production value of `block_timestamp_window_seconds` is **1 second**: [2](#0-1) 

**Attack path 1 — malicious proposer (direct):**

The legitimate proposer for round R constructs a `ProposalInit` with `timestamp = last_block_timestamp + 1` (one second after the previous block, potentially hours before `now`). The proposal is sent to validators. `is_proposal_init_valid` checks:
- `last_block_timestamp + 1 >= last_block_timestamp` → **passes**
- `last_block_timestamp + 1 <= now + 1` → **passes** (past timestamps are always ≤ now + 1)

The block is committed with the stale timestamp.

**Attack path 2 — queued future-round proposal (no malice required):**

When `validate_proposal` is called for a proposal whose `init.round > current_round`, the proposal is stored in `queued_proposals` **without any timestamp validation**:

```rust
// crates/apollo_consensus_orchestrator/src/sequencer_consensus_context.rs  lines 873-877
std::cmp::Ordering::Greater => {
    trace!("Queueing proposal for future round.");
    self.queued_proposals
        .insert(init.round, ((init, timeout, content_receiver), fin_sender));
    fin_receiver
}
``` [3](#0-2) 

When `set_height_and_round` later advances to that round (potentially after many proposal timeouts), the queued proposal is dequeued and validated: [4](#0-3) 

At dequeue time, `is_proposal_init_valid` runs with the **current** `now`, but the proposal's timestamp is from when it was originally received. If N rounds timed out (each round timeout is tens of seconds), the timestamp can be minutes stale. The missing lower-bound check means this stale timestamp passes validation.

**Corrupted values downstream:**

The stale timestamp is passed directly to `get_l1_prices_in_fri_and_wei`, which looks up historical L1 gas prices at that timestamp:

```rust
// validate_proposal.rs  lines 322-328
let (l1_gas_prices_fri, l1_gas_prices_wei) = get_l1_prices_in_fri_and_wei(
    l1_gas_price_provider,
    init_proposed.timestamp,          // ← stale timestamp used for price lookup
    proposal_init_validation.previous_proposal_init.as_ref(),
    gas_price_params,
).await;
``` [5](#0-4) 

The stale timestamp is also forwarded to `initiate_validation` → `convert_to_sn_api_block_info` → `ValidateBlockInput.block_info`, which becomes the `BlockContext` for all transaction execution in that block. Every `get_block_timestamp()` syscall in that block returns the stale value. [6](#0-5) 

---

### Impact Explanation

1. **Wrong L1 gas prices committed to the block.** L1 gas prices are looked up at the stale timestamp. If the timestamp is hours old, the embedded `l1_gas_price_fri`, `l1_data_gas_price_fri`, `l1_gas_price_wei`, and `l1_data_gas_price_wei` in the block header reflect historical prices, not current ones. Every transaction in the block is charged fees based on wrong L1 gas prices — matching the "Incorrect fee, gas … or L1 gas price effect with economic impact" Critical impact category.

2. **Wrong `get_block_timestamp()` syscall result.** Contracts that branch on block time (e.g., time-locked vaults, oracle freshness checks, expiry logic) receive a stale timestamp, potentially causing incorrect execution results — matching the "Wrong state, receipt, event … or revert result from blockifier/syscall/execution logic" Critical impact category.

---

### Likelihood Explanation

- **Attack path 1** requires a malicious validator who is the designated proposer for a round. In a decentralized validator set this is a realistic threat.
- **Attack path 2** requires no malice: it occurs whenever a proposal for a future round is received and multiple subsequent rounds time out before that round is reached. Each round timeout is on the order of tens of seconds; five timed-out rounds produce ~5 minutes of staleness, which is significant relative to the 1-second window.

---

### Recommendation

Add a symmetric lower-bound check in `is_proposal_init_valid`:

```rust
// After the existing upper-bound check:
if init_proposed.timestamp + proposal_init_validation.block_timestamp_window_seconds < now {
    return Err(ValidateProposalError::InvalidProposalInit(
        init_proposed.clone(),
        proposal_init_validation.clone(),
        format!(
            "Timestamp is too old: now={}, block_timestamp_window_seconds={}, proposed={}",
            now,
            proposal_init_validation.block_timestamp_window_seconds,
            init_proposed.timestamp
        ),
    ));
}
```

This enforces `timestamp >= now - block_timestamp_window_seconds`, making the window symmetric as the config description implies. For queued proposals, the check should be applied at dequeue time (which it already is, once the missing branch is added), ensuring that a proposal that sat in the queue too long is rejected rather than committed with a stale timestamp.

---

### Proof of Concept

```
1. Node is at height H, round 0. Previous block timestamp = T_prev.
2. The designated proposer for round 1 sends a ProposalInit with:
       timestamp = T_prev + 1   (e.g., T_prev = now - 3600, so timestamp is 1 hour stale)
       round = 1
3. validate_proposal() is called; init.round (1) > current_round (0), so the proposal
   is inserted into queued_proposals without any timestamp check.
4. Round 0 times out. set_height_and_round(H, 1) is called.
5. The queued proposal is dequeued. ProposalInitValidation is constructed with now = T_prev + 3600.
6. is_proposal_init_valid checks:
       T_prev + 1 >= T_prev          → PASS (lower bound = previous block timestamp)
       T_prev + 1 <= T_prev+3600 + 1 → PASS (upper bound = now + window)
   No lower bound relative to now is checked.
7. initiate_validation() calls get_l1_prices_in_fri_and_wei(timestamp = T_prev + 1).
   L1 gas prices from 1 hour ago are embedded in the block.
8. Block is committed. All transactions in the block are charged fees based on
   1-hour-old L1 gas prices. All get_block_timestamp() calls return T_prev + 1.
```
