### Title
Stale Non-Participant Votes Counted in TEE Governance Thresholds Without Filtering — (File: `crates/contract/src/lib.rs`)

---

### Summary

The TEE governance voting functions (`vote_code_hash`, `vote_add_launcher_hash`, `vote_remove_launcher_hash`, `vote_add_os_measurement`, `vote_remove_os_measurement`) compare raw accumulated vote counts against the current threshold or participant count without filtering out stale votes cast by former participants. This is structurally identical to the reported vault-interest omission: a quantity that should reflect only the current valid state is used without first "accruing" (filtering) the accumulated stale component.

---

### Finding Description

After a successful resharing, `vote_reshared` spawns several **detached** cleanup promises, including `clean_tee_status`: [1](#0-0) 

`clean_tee_status` removes non-participant votes from the TEE state vote maps. Because it is detached, it can fail silently. The `vote_update` function explicitly acknowledges this risk and compensates by re-filtering votes against the current participant set before comparing to the threshold: [2](#0-1) 

However, the TEE governance voting functions do **not** apply this same filtering. They use the raw count returned by `tee_state.vote()` / `tee_state.vote_launcher()` / `tee_state.vote_measurement()` directly:

**`vote_code_hash`** — raw count compared to threshold: [3](#0-2) 

**`vote_add_launcher_hash`** — raw count compared to threshold: [4](#0-3) 

**`vote_remove_launcher_hash`** — raw count compared to `total_participants`: [5](#0-4) 

**`vote_add_os_measurement`** — raw count compared to threshold: [6](#0-5) 

**`vote_remove_os_measurement`** — raw count compared to `total_participants`: [7](#0-6) 

The `AuthenticatedParticipantId::new(threshold_parameters.participants())?` guard only verifies the **caller** is a current participant; it does not retroactively invalidate votes already stored from former participants. [8](#0-7) 

---

### Impact Explanation

**For add-type votes (`vote_code_hash`, `vote_add_launcher_hash`, `vote_add_os_measurement`):**

If a former participant P₀ voted for a TEE image hash H before being removed via resharing, and the `clean_tee_status` promise fails, P₀'s vote persists. If only `threshold − 1` current participants subsequently vote for H, the raw count reaches `threshold`, and H is whitelisted. A malicious TEE image can thus be approved without the required threshold of current participants consenting. Nodes running the malicious image would pass `assert_caller_is_attested_participant_and_protocol_active`, enabling them to call `respond`, `respond_ckd`, and `respond_verify_foreign_tx` — producing unauthorized threshold signatures.

**For remove-type votes (`vote_remove_launcher_hash`, `vote_remove_os_measurement`):**

Stale votes from former participants count toward the "all participants must vote" requirement, making it possible to remove a valid launcher or OS measurement with fewer than all current participants agreeing. This breaks the safety invariant that removal of security-critical configurations requires unanimous current-participant consent, and can cause honest nodes to fail TEE validation and halt signing.

Impact classification: **Critical** (add-type path) — unauthorized threshold signature issuance enabled by whitelisting a malicious TEE image below the required participant authorization threshold.

---

### Likelihood Explanation

The `clean_tee_status` promise is detached and can fail due to gas exhaustion or other runtime errors. The developers explicitly acknowledge this failure mode in `vote_update`'s comment and compensate there, but not in the TEE governance functions. A resharing event that removes a participant who had previously voted for a specific hash is a realistic operational scenario (e.g., a node is rotated out). The attacker-controlled entry path is: (1) vote for a target hash while a participant, (2) get removed via resharing while cleanup fails, (3) observe that stale vote persists and contributes to threshold.

---

### Recommendation

Apply the same participant-filtering pattern used in `vote_update` to all TEE governance voting functions. Before comparing the vote count to the threshold, recount only votes cast by accounts present in the current `threshold_parameters.participants()` set. Alternatively, on each vote call, explicitly evict any stored votes belonging to accounts no longer in the participant set before tallying. [9](#0-8) 

---

### Proof of Concept

1. Network has participants `{P1, P2, P3}` with threshold `t = 2`.
2. P1 calls `vote_code_hash(malicious_hash)` — vote stored, count = 1 < 2, no whitelist.
3. Resharing removes P1; new set is `{P2, P3}`, threshold still 2.
4. `clean_tee_status` detached promise fails (e.g., gas exhaustion). P1's vote for `malicious_hash` remains in `tee_state`.
5. P2 calls `vote_code_hash(malicious_hash)` — `AuthenticatedParticipantId::new` passes (P2 is current). `tee_state.vote()` returns count = 2 (P1 stale + P2 current).
6. `2 >= self.threshold()?.value()` (= 2) → `tee_state.whitelist_tee_proposal(malicious_hash, ...)` executes.
7. A node running the malicious TEE image now passes `assert_caller_is_attested_participant_and_protocol_active` and can call `respond()` to submit forged signatures. [10](#0-9) [1](#0-0)
