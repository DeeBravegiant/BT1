Audit Report

## Title
Permissionless `claim()` Enables Any Caller to Force Fee-Bearing Claims on Behalf of Any Account — (`contracts/utils/MerkleDistributor/MerkleDistributor.sol`)

## Summary
The `claim()` function in `MerkleDistributor` accepts an arbitrary `account` parameter with no restriction that `msg.sender == account`. Any caller with a valid Merkle proof (which is public by design) can trigger a claim for any victim at the current `feeInBPS`, which can be as high as 10%. The victim's `userClaims` state is permanently updated, preventing any future claim for the same index, and the fee portion of their yield is irrecoverably sent to `protocolTreasury`.

## Finding Description
`claim()` is declared `external` with no caller restriction: [1](#0-0) 

The only guards are an index-bounds check, an `isClaimed` check, and Merkle proof verification — none of which require `msg.sender == account`: [2](#0-1) 

After passing these checks, the contract permanently marks the account as claimed and deducts the fee: [3](#0-2) 

`isClaimed` returns `true` once `userClaims[account].lastClaimedIndex >= index`, so any subsequent call by the victim reverts with `AlreadyClaimed`: [4](#0-3) 

Merkle tree leaf data (index, account, cumulativeAmount, proof) must be published for the distributor to function, making all proofs trivially available to any attacker.

## Impact Explanation
`feeInBPS` can be set up to `MAX_FEE_IN_BPS = 1000` (10%): [5](#0-4) 

A victim who intends to defer their claim until `feeInBPS` decreases (or reaches 0) is forced to claim at the attacker-chosen moment. The fee is permanently extracted from the victim's unclaimed yield and sent to `protocolTreasury` — the victim cannot reclaim it. The loss is up to 10% of every user's cumulative entitlement. This concretely matches **High — Theft of unclaimed yield**.

## Likelihood Explanation
The attack requires no special role, no privileged key, and no governance action. The attacker only needs to call one public function (`claim()`) with publicly available inputs (index, account address, cumulativeAmount, Merkle proof). It is repeatable across all accounts and all Merkle root updates. Likelihood is **High**.

## Recommendation
Add a caller restriction to `claim()`:

```solidity
function claim(
    uint256 index,
    address account,
    uint256 cumulativeAmount,
    bytes32[] calldata merkleProof
) external override whenNotPaused {
    if (msg.sender != account) revert Unauthorized();
    // ...
}
```

If third-party claiming is intentional (e.g., for gas relayers), introduce an explicit opt-in delegation mapping so users can authorize specific callers per account.

## Proof of Concept

```solidity
// State: feeInBPS = 1000 (10%), currentIndex = 2, currentMerkleRoot = R2
// R2 leaf: (index=2, victim, cumulativeAmount=200)
// victim.userClaims = {lastClaimedIndex: 0, cumulativeAmount: 0}

// Attacker obtains victim's Merkle proof from public tree data.
merkleDistributor.claim(2, victim, 200, proofR2);
// fee = 200 * 1000 / 10000 = 20 tokens → protocolTreasury
// amountToSend = 180 tokens → victim
// userClaims[victim] = {lastClaimedIndex: 2, cumulativeAmount: 200}

// Victim attempts to claim when feeInBPS later drops to 0:
merkleDistributor.claim(2, victim, 200, proofR2);
// Reverts: AlreadyClaimed
// Victim permanently lost 20 tokens (10% of entitlement).
```

**Foundry test plan:** Deploy `MerkleDistributor`, initialize with `feeInBPS = 1000`, construct a two-leaf Merkle tree, call `setMerkleRoot`, then call `claim(index, victim, amount, proof)` from an attacker address. Assert victim received `amount * 0.9`, `protocolTreasury` received `amount * 0.1`, and a subsequent call from `victim` reverts with `AlreadyClaimed`.
