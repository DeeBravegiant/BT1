### Title
Proof Verification Tasks for Truncated (Non-Committed) Transactions Cause Valid Proposals to Be Rejected — (`crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

---

### Summary

When a validator processes a proposal, it spawns proof-verification tasks for **every** received transaction. When `ProposalFin` arrives with `executed_transaction_count < n_received_txs`, the validator correctly truncates `content` to the committed subset, but it **does not** truncate `verify_and_store_proof_tasks`. All tasks — including those for non-committed, truncated transactions — are then awaited. A failure in any of those extra tasks causes the entire proposal to be rejected, even though the committed portion is valid.

---

### Finding Description

**Step 1 — Tasks are collected for all received transactions.**

In `handle_proposal_part`, for each `ProposalPart::Transactions` batch, every transaction is converted and its optional proof task is appended to the shared `verify_and_store_proof_tasks` vector: [1](#0-0) 

**Step 2 — Content is truncated to `executed_txs_count`, but tasks are not.**

When `ProposalFin` arrives, `content` is truncated to the proposer-declared count: [2](#0-1) 

`truncate_to_executed_txs` operates only on `content`; it has no knowledge of `verify_and_store_proof_tasks`: [3](#0-2) 

**Step 3 — ALL tasks are awaited, including those for truncated transactions.**

Immediately after truncation, the code drains and awaits every task in `verify_and_store_proof_tasks`: [4](#0-3) 

A failure in any task — even one belonging to a transaction that was excluded from the block — returns `HandledProposalPart::Failed`, which propagates to `ValidateProposalError::ProposalPartFailed` and causes the proposal to be rejected: [5](#0-4) 

**Root cause — proposer streams transactions before execution results are known.**

The proposer sends transactions to validators immediately after submitting them to the executor, before knowing which ones will be committed: [6](#0-5) 

The proposer later sets `executed_transaction_count` (via `final_n_executed_txs`) to reflect only the successfully executed subset: [7](#0-6) 

This means validators routinely receive more transactions than are committed, and the proof tasks for the non-committed tail are never supposed to gate proposal acceptance.

---

### Impact Explanation

A malicious proposer can craft a proposal where:
- The first K transactions are valid and form a legitimate block.
- The remaining M transactions (beyond `executed_transaction_count = K`) carry deliberately invalid proofs.

Validators will verify proofs for all K+M transactions. The M invalid-proof tasks fail, causing `HandledProposalPart::Failed` to be returned. The valid K-transaction block is rejected by every validator. Consensus must advance to the next round, delaying block production. Repeated across rounds, this constitutes a sustained DoS on the consensus layer.

This matches the allowed impact: **High — valid transactions/proposals are rejected before sequencing** due to broken admission logic in the proposal-validation path.

---

### Likelihood Explanation

The proposer controls the proposal stream and can freely include transactions with invalid proofs in the non-committed tail. No existing guard prevents this: the only check on `executed_transaction_count` is that it does not exceed `n_received_txs`: [8](#0-7) 

There is no check that the extra transactions carry valid proofs, and no mechanism to skip their proof tasks.

---

### Recommendation

Track proof tasks per-transaction position so they can be selectively awaited. Concretely:

1. Change `verify_and_store_proof_tasks` from a flat `Vec<VerifyAndStoreProofTask>` to a `Vec<(usize, VerifyAndStoreProofTask)>` where the `usize` is the transaction's absolute index in the received stream.
2. After `truncate_to_executed_txs` sets the committed count, drain only tasks whose index is `< executed_txs_count`.
3. Drop (or let run in the background without awaiting) tasks for indices `>= executed_txs_count`.

This mirrors the existing truncation logic applied to `content` and closes the mismatch.

---

### Proof of Concept

```
Proposer (malicious):
  1. Obtain K valid transactions from mempool (all with valid proofs).
  2. Craft M additional transactions whose proof field is invalid/empty.
  3. Stream all K+M transactions as ProposalPart::Transactions batches.
  4. Send ProposalFin { executed_transaction_count: K, proposal_commitment: <valid hash> }.

Validator (honest):
  1. Receives K+M transactions; convert_consensus_tx_to_internal_consensus_tx spawns
     proof tasks for all K+M.
  2. Receives Fin; executed_txs_count = K ≤ n_received_txs = K+M → passes guard.
  3. truncate_to_executed_txs reduces content to K transactions.
  4. verify_and_store_proof_tasks still holds K+M tasks.
  5. Awaits all K+M tasks; tasks for the M invalid-proof transactions fail.
  6. Returns HandledProposalPart::Failed → ValidateProposalError::ProposalPartFailed.
  7. Valid K-transaction block is rejected; consensus advances to next round.
``` [9](#0-8)

pub(crate) fn truncate_to_executed_txs(
    content: &mut Vec<Vec<InternalConsensusTransaction>>,
    final_n_executed_txs: usize,
) -> Vec<Vec<InternalConsensusTransaction>> {
    let content = std::mem::take(content);
    // Truncate `content` to keep only the first `final_n_executed_txs`, preserving batch
    // structure.
    let mut executed_content: Vec<Vec<InternalConsensusTransaction>> = Vec::new();
    let mut remaining_tx_count = final_n_executed_txs;

    for batch in content {
        if remaining_tx_count < batch.len() {
            executed_content.push(batch.into_iter().take(remaining_tx_count).collect());
            break;
        } else {
            remaining_tx_count -= batch.len();
            executed_content.push(batch);
        }
    }

    executed_content
}
