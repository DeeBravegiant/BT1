Audit Report

## Title
`maxNumberOfWithdrawalsPerUser` Uninitialized in `initialize()` Permanently Blocks `initiateWithdrawal()` Until Admin Intervenes - (File: contracts/KERNEL/KernelDepositPool.sol)

## Summary
`KernelDepositPool.initialize()` never assigns `maxNumberOfWithdrawalsPerUser`, leaving it at its Solidity default of `0`. Because `initiateWithdrawal()` enforces `userWithdrawalIds[msg.sender].length >= maxNumberOfWithdrawalsPerUser` and `0 >= 0` is always `true`, every call to `initiateWithdrawal()` reverts with `WithdrawalLimitReached` from the moment the contract is deployed. All staked KERNEL tokens are frozen until an admin manually calls `setMaxNumberOfWithdrawalsPerUser()`.

## Finding Description
`maxNumberOfWithdrawalsPerUser` is declared as a plain storage variable with no default assignment:

```solidity
// L107-108
uint256 public maxNumberOfWithdrawalsPerUser;
```

`initialize()` sets `kernelToken` and `rewardsToken` but omits `maxNumberOfWithdrawalsPerUser`:

```solidity
// L259-271
function initialize(address _admin, address _kernelToken, address _rewardToken) external initializer {
    ...
    kernelToken = IERC20(_kernelToken);
    rewardsToken = IERC20(_rewardToken);
    // maxNumberOfWithdrawalsPerUser never set → remains 0
}
```

`initiateWithdrawal()` enforces the limit before any state changes:

```solidity
// L323
if (userWithdrawalIds[msg.sender].length >= maxNumberOfWithdrawalsPerUser) revert WithdrawalLimitReached();
```

With `maxNumberOfWithdrawalsPerUser == 0`, the condition evaluates to `0 >= 0 == true` for every caller, causing an unconditional revert. The admin setter explicitly rejects `0` as invalid (`L614`), confirming `0` is not a valid operational state. `stake()` and `stakeFor()` have no such guard and succeed normally, so users can deposit tokens into a contract from which they cannot withdraw.

## Impact Explanation
Any user who stakes KERNEL tokens via `stake()` or `stakeFor()` cannot initiate a withdrawal until an admin calls `setMaxNumberOfWithdrawalsPerUser()`. Staked KERNEL tokens are inaccessible — users cannot begin the unlock process at all. This constitutes **Medium — Temporary freezing of funds**.

## Likelihood Explanation
The broken state is the deterministic default on every fresh deployment. No attacker action, special conditions, or external dependencies are required. Any user who stakes and then attempts to withdraw will hit the revert. Likelihood is **High**.

## Recommendation
Set `maxNumberOfWithdrawalsPerUser` to a valid non-zero value inside `initialize()`:

```solidity
function initialize(address _admin, address _kernelToken, address _rewardToken) external initializer {
    UtilLib.checkNonZeroAddress(_admin);
    UtilLib.checkNonZeroAddress(_kernelToken);
    UtilLib.checkNonZeroAddress(_rewardToken);

    __AccessControl_init();
    __ReentrancyGuard_init();

    _setupRole(DEFAULT_ADMIN_ROLE, _admin);

    kernelToken = IERC20(_kernelToken);
    rewardsToken = IERC20(_rewardToken);
    maxNumberOfWithdrawalsPerUser = MAX_WITHDRAWALS_PER_USER; // add this line
}
```

## Proof of Concept
1. Deploy `KernelDepositPool` proxy and call `initialize(admin, kernelToken, rewardToken)`.
2. `maxNumberOfWithdrawalsPerUser` is `0` (never set in `initialize()`).
3. User calls `stake(100e18)` — succeeds; KERNEL tokens transferred in.
4. User calls `initiateWithdrawal(100e18)`:
   - Check at L323: `userWithdrawalIds[user].length >= maxNumberOfWithdrawalsPerUser` → `0 >= 0` → `true`
   - Reverts with `WithdrawalLimitReached`.
5. User's KERNEL tokens are locked with no withdrawal path until admin calls `setMaxNumberOfWithdrawalsPerUser(N)`.
