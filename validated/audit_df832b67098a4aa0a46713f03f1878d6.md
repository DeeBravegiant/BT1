Audit Report

## Title
Stale Cross-Chain rsETH Rate Enables Over-Issuance of wrsETH on Deposit — (`contracts/cross-chain/CrossChainRateReceiver.sol`, `contracts/pools/RSETHPoolV3.sol`)

## Summary
`CrossChainRateReceiver.getRate()` returns the stored `rate` with no staleness check against `lastUpdated`. `RSETHPoolV3.viewSwapRsETHAmountAndFee()` divides a fresh token/ETH Chainlink rate by this potentially stale rsETH/ETH rate, causing the pool to mint more wrsETH than the deposited collateral is worth at current prices. An unprivileged depositor can exploit any natural staleness gap to extract excess wrsETH, which can be bridged back to L1 as rsETH, draining pool reserves.

## Finding Description
**Root cause — unconditional rate return with no freshness guard**

`CrossChainRateReceiver.lzReceive()` stores the received value and records `lastUpdated = block.timestamp` at the time of the LayerZero message delivery. `CrossChainRateReceiver.getRate()` returns `rate` unconditionally with no comparison of `lastUpdated` against any maximum-staleness threshold:

```solidity
// CrossChainRateReceiver.sol L103-105
function getRate() external view returns (uint256) {
    return rate;
}
```

`CrossChainRateProvider.updateRate()` is a public payable function with only a `nonReentrant` guard — no keeper enforces a maximum update interval, and LayerZero delivery introduces additional latency.

**Deposit math — asymmetric oracle freshness**

`RSETHPoolV3.viewSwapRsETHAmountAndFee(amount, token)` computes:

```solidity
// RSETHPoolV3.sol L328-334
uint256 rsETHToETHrate = getRate();                                    // stale, no freshness check
uint256 tokenToETHRate = IOracle(supportedTokenOracle[token]).getRate(); // fresh Chainlink
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
```

When `rsETHToETHrate` is stale-low (rsETH has accrued staking yield since the last LayerZero update), the denominator is smaller than the true current rate, so `rsETHAmount` is inflated. The result is passed directly to `wrsETH.mint(msg.sender, rsETHAmount)` at line 290 with no additional validation.

**Exploit flow**

1. Observe that `CrossChainRateReceiver.lastUpdated` is old (e.g., 2+ days) and that rsETH has appreciated since then (stale rate = 1.050e18, true rate = 1.100e18).
2. Call `RSETHPoolV3.deposit(wstETH, 1e18, "")`.
3. Pool computes: `1e18 * 1.080e18 / 1.050e18 ≈ 1.0286e18 wrsETH` minted.
4. Fair value: `1e18 * 1.080e18 / 1.100e18 ≈ 0.9818e18 wrsETH`.
5. Excess minted: ~4.8% per deposit.
6. Bridge wrsETH back to L1 to redeem rsETH worth more than the deposited wstETH.

**Existing checks are insufficient**

The `dailyMintLimit` modifier only caps total rsETH-equivalent minted per day; it does not bound the per-unit over-issuance ratio. An attacker extracts the maximum daily allowance at the favorable stale rate, then repeats the next day. The `nonReentrant` guard on `updateRate()` does not prevent staleness.

## Impact Explanation
**Critical — Direct theft of pool reserves.** The pool accepts real wstETH and mints excess wrsETH. When the excess wrsETH is bridged to L1 and redeemed as rsETH, the attacker receives more rsETH than the deposited wstETH was worth. The pool's collateral reserves are progressively undercollateralized relative to outstanding wrsETH, constituting direct theft of pool reserves. This matches the allowed impact: *"Direct theft of any user funds, whether at-rest or in-motion."*

## Likelihood Explanation
- Requires no special role, no front-running, and no oracle manipulation.
- rsETH accrues staking yield continuously, so any gap between LayerZero updates creates a stale-low condition naturally.
- `updateRate()` is manually triggered with no on-chain automation enforcing a maximum update interval.
- The attacker only needs to observe `CrossChainRateReceiver.lastUpdated` on-chain and confirm rsETH has appreciated since then — both are publicly readable.
- Repeatable daily up to `dailyMintLimit`, making extraction systematic.

## Recommendation
1. **Add a staleness check in `CrossChainRateReceiver.getRate()`**: revert if `block.timestamp - lastUpdated > MAX_STALENESS` (e.g., 24 hours).
2. **Add a staleness check in `RSETHPoolV3.getRate()`**: validate the oracle's `lastUpdated` before using the rate in deposit math.
3. Consider a pull-based oracle (e.g., a Chainlink rsETH/ETH feed on L2) instead of a push-based cross-chain rate that can go stale between manual updates.

## Proof of Concept
```solidity
// Foundry fork test (L2, e.g., Arbitrum)
// Setup:
//   - RSETHRateReceiver.rate = 1.05e18, lastUpdated = block.timestamp - 2 days
//   - True current rsETH/ETH = 1.10e18 (not yet pushed via LayerZero)
//   - Chainlink wstETH/ETH oracle returns 1.08e18 (fresh)
// Attack:
//   attacker.approve(wstETH, pool, 1e18);
//   pool.deposit(wstETH, 1e18, "");
//   uint256 received = wrsETH.balanceOf(attacker);
//   uint256 fair     = 1e18 * 1.08e18 / 1.10e18; // ≈ 0.9818e18
//   assert(received > fair);  // ~4.8% excess
// Repeat daily up to dailyMintLimit to maximize extraction.
// Bridge wrsETH to L1 and redeem rsETH to realize profit.
```

The invariant `wrsETH_received ≤ deposited_token_value / true_rsETH_rate` is violated whenever the cross-chain rate is stale-low, which occurs naturally without any attacker action due to rsETH's monotonically increasing ETH price.
