### Title
Attested Participant Can Submit Arbitrary CKD Response for `AppPublicKey` Requests, Bypassing Threshold Authorization - (File: crates/contract/src/lib.rs)

### Summary
The `respond_ckd` function in the MPC contract performs cryptographic output verification only for the `AppPublicKeyPV` variant of CKD requests. For the `AppPublicKey` variant, any single attested participant (in practice, the coordinator/leader) can submit an arbitrary `CKDResponse` — with any `big_y` and `big_c` values — and the contract will accept it unconditionally. This allows a single malicious participant to deliver a fraudulent derived key to the user, one that the attacker controls, enabling them to later decrypt any data the user encrypts with that key.

### Finding Description
In `respond_ckd` at lines 675–682 of `crates/contract/src/lib.rs`, the contract branches on the request's `app_public_key` variant:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}   // ← no verification at all
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
```

For `AppPublicKeyPV`, the contract calls `ckd_output_check` (defined in `crates/contract/src/primitives/ckd.rs` lines 80–102), which verifies the pairing equation `e(big_c, g2) = e(big_y, app_pk2) · e(hash_point, public_key)`. For `AppPublicKey`, the arm is a no-op. The contract then unconditionally resolves the pending yield with whatever `response` was supplied.

The `AppPublicKey` CKD flow is used in production (confirmed by the e2e test at `crates/e2e-tests/tests/ckd_verification.rs` lines 40–91). In this flow the user supplies their BLS G1 public key `A = a·G`, and the MPC network is supposed to compute `big_c = S + y·A` and `big_y = y·G` where `S = x·H(app_id)` is derived from the threshold secret. The user recovers the derived secret as `S = big_c − big_y·a`.

Because the contract imposes no on-chain check for the `AppPublicKey` variant, a malicious coordinator can instead choose arbitrary scalars `y′` and `S′`, compute `big_c′ = S′ + y′·A` and `big_y′ = y′·G`, and call `respond_ckd` with `(big_y′, big_c′)`. The user computes `S′ = big_c′ − big_y′·a`, which equals the attacker-chosen `S′`. The attacker knows `S′` and can decrypt anything the user encrypts under it.

The asymmetry is stark: `AppPublicKeyPV` has a full on-chain pairing check; `AppPublicKey` has none.

### Impact Explanation
A single malicious attested participant acting as the CKD coordinator can:
1. Substitute a self-chosen `(big_y′, big_c′)` for the legitimately computed output.
2. The contract accepts the fraudulent response with no verification.
3. The user derives `S′` — a key the attacker knows — and uses it to protect sensitive data.
4. The attacker decrypts that data, fully compromising the confidentiality guarantee of the CKD protocol.

This is a **Critical** impact: unauthorized confidential key derivation output delivered without the required threshold-participant authorization, matching the allowed scope item *"Critical. Unauthorized … confidential key derivation output without the required participant authorization."*

### Likelihood Explanation
The MPC node architecture designates a single leader per request (`MpcLeaderCentricComputation` in `crates/node/src/providers/ckd/sign.rs` lines 133–185). Any participant that is selected as leader for a CKD request can execute this attack unilaterally. In a network of N nodes where the adversary controls k < threshold nodes, the probability of controlling the leader for any given request is k/N. With typical deployments (e.g., 3-of-10), a single compromised node has a 30 % per-request chance of being leader. The attack requires no collusion and no threshold breach.

### Recommendation
Add an on-chain cryptographic verification for the `AppPublicKey` variant analogous to `ckd_output_check` for `AppPublicKeyPV`. Specifically, verify that the submitted `(big_y, big_c)` satisfies the expected algebraic relation with respect to the network's BLS public key and the user-supplied app public key before resolving the yield. Until such a check exists, the `AppPublicKey` variant's security guarantee is entirely dependent on the honesty of whichever node happens to be the leader.

### Proof of Concept
1. User calls `request_app_private_key` with `CKDAppPublicKey::AppPublicKey(A)` where `A = a·G` is their BLS G1 public key.
2. The MPC network runs the threshold CKD protocol; the coordinator aggregates shares and obtains the correct `(big_y, big_c)`.
3. The malicious coordinator instead picks arbitrary `y′, S′ ∈ Zq`, computes `big_y′ = y′·G` and `big_c′ = S′ + y′·A`.
4. The coordinator calls `respond_ckd(ckd_request, CKDResponse { big_y: big_y′, big_c: big_c′ })`.
5. The contract's `AppPublicKey` arm is a no-op; `resolve_yields_for` delivers the fraudulent response to the user.
6. The user computes `S′ = big_c′ − big_y′·a = S′ + y′·A − y′·a·G = S′` (since `A = a·G`).
7. The user encrypts data under `S′`; the attacker, knowing `S′`, decrypts it.

pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}
