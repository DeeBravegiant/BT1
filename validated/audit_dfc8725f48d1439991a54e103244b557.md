### Title
Unbound `payload_hash` in `respond_verify_foreign_tx` Enables Cross-Request Response Replay by a Single Malicious Attested Participant — (`File: crates/contract/src/lib.rs`)

---

### Summary

`respond_verify_foreign_tx` verifies that `response.signature` is cryptographically valid over `response.payload_hash` using the root public key, but **never verifies that `response.payload_hash` is the correct hash for the `request` being resolved**. A single malicious attested participant can replay any previously observed valid `(payload_hash, sig)` pair to resolve an unrelated pending `verify_foreign_transaction` request, delivering a forged attestation to its caller.

---

### Finding Description

In `respond_verify_foreign_tx` (`crates/contract/src/lib.rs`, lines 691–754), the contract performs the following check:

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;

// Check the signature is correct against the root public key
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,
)
.is_ok()
``` [1](#0-0) 

The contract only checks: *is `sig` a valid ECDSA signature over `payload_hash` under the root key?* It does **not** check: *does `payload_hash` equal `SHA-256(borsh(ForeignTxSignPayloadV1 { request, values }))` for the specific `request` being resolved?*

The `payload_hash` field is entirely attacker-supplied in the response struct:

```rust
pub struct VerifyForeignTransactionResponse {
    pub payload_hash: Hash256,
    pub signature: SignatureResponse,
}
```

The `request` argument to `respond_verify_foreign_tx` is used only as a map key to look up and drain the pending yield queue:

```rust
pending_requests::resolve_yields_for(
    &mut self.pending_verify_foreign_tx_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
``` [2](#0-1) 

There is no binding between `request` and `response.payload_hash` enforced by the contract. The `args_into_verify_foreign_tx_request` conversion also confirms no commitment to the request is embedded in the stored key:

```rust
pub fn args_into_verify_foreign_tx_request(
    args: dtos::VerifyForeignTransactionRequestArgs,
) -> dtos::VerifyForeignTransactionRequest {
    dtos::VerifyForeignTransactionRequest {
        domain_id: args.domain_id,
        request: args.request,
        payload_version: args.payload_version,
    }
}
``` [3](#0-2) 

The `ForeignTxSignPayloadV1` that nodes actually sign includes both the `request` and the `extracted_values`:

```rust
pub struct ForeignTxSignPayloadV1 {
    pub request: ForeignChainRpcRequest,
    pub values: Vec<ExtractedValue>,
}
``` [4](#0-3) 

The contract cannot independently recompute `payload_hash` because it does not know `extracted_values`. However, it also does not require the responder to prove that `payload_hash` commits to the pending `request`. This gap is the root cause.

The SDK's `ForeignChainSignatureVerifier::verify_signature` does perform this check client-side:

```rust
let payload_is_correct = expected_payload_hash == response.payload_hash;
if !payload_is_correct {
    return Err(VerifyForeignChainError::IncorrectPayloadSigned { ... });
}
``` [5](#0-4) 

But this is an off-chain SDK helper, not an on-chain invariant. Bridge contracts that do not use the SDK, or that trust the on-chain response directly, receive no protection.

---

### Impact Explanation

A single malicious attested participant (strictly below the signing threshold) can:

1. Observe any previously accepted `(payload_hash_A, sig_A)` pair from a legitimate `respond_verify_foreign_tx` call for request A. Because foreign-tx signatures use the **root key with zero tweak** (confirmed by the test comment "simulate signature with the root key (no tweak for foreign tx)"), the same key signs all foreign-tx payloads.
2. Submit or wait for a pending `verify_foreign_transaction(B)` for a different transaction B.
3. Call `respond_verify_foreign_tx(request=B, response=(payload_hash_A, sig_A))`.
4. The contract accepts: `sig_A` is valid over `payload_hash_A` under the root key ✓, and B is in the pending map ✓.
5. B's yield is resolved with `(payload_hash_A, sig_A)` — a response for A, not B.

The caller of `verify_foreign_transaction(B)` receives a `VerifyForeignTransactionResponse` whose `payload_hash` does not correspond to B. Any bridge contract that does not independently recompute and compare the expected hash will treat this as a valid attestation of B, enabling **invalid bridge execution** (e.g., releasing funds for a foreign transaction that was never actually verified by the MPC network).

This matches the allowed High impact: *"Cross-chain replay, forged foreign-chain verification, or participant/attestation authorization bypass that causes invalid bridge execution or double-spend conditions."*

---

### Likelihood Explanation

- **Attacker role**: A single attested MPC participant — below the signing threshold. This is a realistic adversary model (one compromised or malicious node operator).
- **Prerequisite**: One previously observed valid `(payload_hash, sig)` pair. Since `respond_verify_foreign_tx` is a public on-chain call, all past responses are visible in chain history.
- **Target**: Any currently pending `verify_foreign_transaction` request. The pending map is publicly readable via `get_pending_verify_foreign_tx_request`.
- **No threshold collusion required**: The attacker reuses a legitimately produced threshold signature; they do not need to forge one.

Likelihood is **Medium**: the attack requires a malicious attested participant, but once that condition is met, execution is straightforward and fully on-chain.

---

### Recommendation

The contract must enforce that `response.payload_hash` commits to the specific `request` being resolved. Two approaches:

1. **Include a request commitment in the signed payload**: Require MPC nodes to sign `SHA-256(borsh(request) || borsh(extracted_values))` and have the contract verify that the first 32 bytes of the preimage match `borsh(request)`. This requires the response to include `extracted_values` so the contract can recompute and compare.

2. **Bind `payload_hash` to the request key at respond time**: Require the responder to also provide the `extracted_values`, compute `expected_hash = SHA-256(borsh(ForeignTxSignPayloadV1 { request, values }))` on-chain, and assert `expected_hash == response.payload_hash` before accepting the response.

Either approach closes the gap by making the contract an active verifier of the request–response binding rather than delegating that check entirely to off-chain SDK consumers.

### Proof of Concept

```
// Setup: legitimate request A has already been responded to on-chain.
// payload_hash_A and sig_A are visible in chain history.

// Step 1: Attacker (single attested participant) submits a new request B
// for a different (or non-existent) foreign transaction.
contract.verify_foreign_transaction(request_args_B);  // 1 yoctoNEAR deposit

// Step 2: Attacker calls respond_verify_foreign_tx with B's request key
// but A's (payload_hash, sig) — replayed from chain history.
contract.respond_verify_foreign_tx(
    request = VerifyForeignTransactionRequest { request: B, domain_id, payload_version },
    response = VerifyForeignTransactionResponse {
        payload_hash: payload_hash_A,   // hash of A's data, not B's
        signature: sig_A,               // valid sig over payload_hash_A under root key
    }
);

// Contract check (lib.rs:729-734):
//   verify_ecdsa_signature(sig_A, payload_hash_A, root_pk) → OK  ✓
//   pending_verify_foreign_tx_requests.get(B) → Some(yield_B)    ✓
// → yield_B is resolved with (payload_hash_A, sig_A)

// Caller of verify_foreign_transaction(B) receives:
//   VerifyForeignTransactionResponse { payload_hash: payload_hash_A, signature: sig_A }
// If caller does not call ForeignChainSignatureVerifier::verify_signature,
// they accept this as a valid attestation of B.
``` [6](#0-5) [7](#0-6)

impl ForeignChainSignatureVerifier {
    pub fn verify_signature(
        self,
        response: &VerifyForeignTransactionResponse,
        // TODO(#2232): don't use interface API types for public keys
        public_key: &PublicKey,
    ) -> Result<(), VerifyForeignChainError> {
        let expected_payload = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
            request: self.request,
            values: self.expected_extracted_values,
        });

        let expected_payload_hash = expected_payload
            .compute_msg_hash()
            .map_err(|_| VerifyForeignChainError::FailedToComputeMsgHash)?;

        let payload_is_correct = expected_payload_hash == response.payload_hash;

        if !payload_is_correct {
            return Err(VerifyForeignChainError::IncorrectPayloadSigned {
                got: response.payload_hash.clone(),
                expected: expected_payload_hash,
            });
        }
        let verification_result = match (public_key, &response.signature) {
            (
                PublicKey::Secp256k1(secp256k1_public_key),
                SignatureResponse::Secp256k1(k256_signature),
            ) => near_mpc_signature_verifier::verify_ecdsa_signature(
                k256_signature,
                &expected_payload_hash,
                secp256k1_public_key,
            ),
            (PublicKey::Ed25519(ed25519_public_key), SignatureResponse::Ed25519 { signature }) => {
                near_mpc_signature_verifier::verify_eddsa_signature(
                    signature,
                    expected_payload_hash.as_slice(),
                    ed25519_public_key,
                )
            }
            // TODO(#2234): improve types so these errors can't happen
            (PublicKey::Bls12381(_bls12381_g2_public_key), _) => {
                return Err(VerifyForeignChainError::UnexpectedSignatureScheme);
            }
            _ => return Err(VerifyForeignChainError::UnexpectedSignatureScheme),
        };

        verification_result.map_err(|_| VerifyForeignChainError::SignatureVerificationFailed)
    }
