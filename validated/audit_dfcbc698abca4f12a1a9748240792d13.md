Audit Report

## Title
Insufficient `msg.value` Validation in Batch LayerZero Send Loop Causes Rate Update Failure - (File: contracts/cross-chain/MultiChainRateProvider.sol)

## Summary
`MultiChainRateProvider.updateRate()` iterates over all registered `rateReceivers` and calls `ILayerZeroEndpoint.send{ value: estimatedFee }` per chain, deducting from the contract's ETH balance each iteration. The `refundAddress` is `payable(msg.sender)`, so excess ETH from each `send` is returned to the caller rather than retained by the contract. There is no pre-loop check that `msg.value` covers the sum of all per-chain fees, so an underpaying caller causes the contract to exhaust its ETH balance mid-loop and revert, failing to propagate the rate to any chain.

## Finding Description
In `updateRate()` at lines 119–134 of `contracts/cross-chain/MultiChainRateProvider.sol`, the loop estimates each chain's fee and immediately spends it:

```solidity
(uint256 estimatedFee,) = ILayerZeroEndpoint(layerZeroEndpoint)
    .estimateFees(dstChainId, address(this), _payload, false, bytes(""));

ILayerZeroEndpoint(layerZeroEndpoint).send{ value: estimatedFee }(
    dstChainId, remoteAndLocalAddresses, _payload, payable(msg.sender), address(0x0), bytes("")
);
``` [1](#0-0) 

The `refundAddress` is `payable(msg.sender)`, meaning any excess ETH from each `send` call is returned to the caller, not back to the contract, making it unavailable for subsequent iterations. [2](#0-1) 

The helper `estimateTotalFee()` (lines 154–173) computes the aggregate fee but is never called or enforced inside `updateRate()`. [3](#0-2) 

There is no `require(msg.value >= totalFee)` guard anywhere in `updateRate()`. [4](#0-3) 

Both `RSETHMultiChainRateProvider` and `AGETHMultiChainRateProvider` inherit this unfixed base without adding any such guard. [5](#0-4) 

## Impact Explanation
When the transaction reverts mid-loop, all state changes — including the updates to `rate` and `lastUpdated` — are rolled back. Destination-chain receiver contracts retain their previously stored stale rate. The contract fails to deliver its core promised function (cross-chain rate propagation) without any permanent loss of ETH. This matches the allowed impact: **Low — Contract fails to deliver promised returns, but doesn't lose value.**

## Likelihood Explanation
`updateRate()` is `external` with no access control, callable by any address. The function provides no on-chain guidance about the required ETH amount. A caller who does not first invoke `estimateTotalFee()` off-chain, or whose estimate is stale due to fee fluctuation, will trigger the revert. The probability increases as more `rateReceivers` are registered, since the total required ETH grows while the per-call estimation gap widens.

## Recommendation
Add a pre-loop aggregation pass and enforce `msg.value >= totalFee` before any `send` call:

```solidity
uint256 totalFee;
for (uint256 i; i < rateReceiversLength; ++i) {
    (uint256 fee,) = ILayerZeroEndpoint(layerZeroEndpoint)
        .estimateFees(uint16(rateReceivers[i]._chainId), address(this), _payload, false, bytes(""));
    totalFee += fee;
}
require(msg.value >= totalFee, "Insufficient fee");
```

Alternatively, change `refundAddress` from `payable(msg.sender)` to `payable(address(this))` so excess ETH from each `send` is returned to the contract and can fund subsequent iterations, with a final sweep returning any remainder to `msg.sender`.

## Proof of Concept

1. Deploy `RSETHMultiChainRateProvider` with two `rateReceivers` on different chains (e.g., Arbitrum and Optimism).
2. Call `estimateTotalFee()` — suppose it returns `0.002 ETH` (0.001 ETH per chain).
3. Call `updateRate{ value: 0.001 ETH }()` (sufficient for only one chain).
4. Iteration 0: `estimateFees` returns `0.001 ETH`. `send{ value: 0.001 ETH }` succeeds. Contract balance: `0 ETH`.
5. Iteration 1: `estimateFees` returns `0.001 ETH`. `send{ value: 0.001 ETH }` reverts — contract has `0 ETH`.
6. Entire transaction reverts. `rate` and `lastUpdated` are rolled back. No chain receives the updated rate.
