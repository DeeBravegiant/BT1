Audit Report

## Title
`_checkAndUpdateDailyFeeMintLimit` Reverts When `maxFeeMintAmountPerDay` Is Zero and Protocol Fee Is Non-Zero — (File: `contracts/LRTOracle.sol`)

## Summary
`maxFeeMintAmountPerDay` is uninitialized (defaults to `0`) and `reinitialize` never sets it. When `protocolFeeInBPS` is non-zero and TVL grows, `_updateRsETHPrice` computes a positive `rsethAmountToMintAsProtocolFee` and passes it to `_checkAndUpdateDailyFeeMintLimit`, which unconditionally reverts because `feeAmount > 0 == maxFeeMintAmountPerDay`. Every public call to `updateRSETHPrice()` reverts, the rsETH price is never updated, and the protocol fee for the frozen period is permanently unrecoverable.

## Finding Description
`maxFeeMintAmountPerDay` is declared as a plain `uint256` storage variable (line 35) and defaults to `0`. The `reinitialize` function (lines 72–79) only sets `feePeriodStartTime`; it never initialises `maxFeeMintAmountPerDay`. `setMaxFeeMintAmountPerDay` (lines 132–135) accepts zero with no guard.

Inside `_checkAndUpdateDailyFeeMintLimit` (lines 197–210):

```solidity
if (currentPeriodMintedFeeAmount + feeAmount > maxFeeMintAmountPerDay) {
    revert DailyFeeMintLimitExceeded(...);
}
```

When `maxFeeMintAmountPerDay == 0` and `feeAmount > 0`, the condition `0 + feeAmount > 0` is always `true` → always reverts.

`_updateRsETHPrice` (lines 299–303) reaches this path whenever `protocolFeeInETH > 0`:

```solidity
if (protocolFeeInETH > 0) {
    uint256 rsethAmountToMintAsProtocolFee = protocolFeeInETH.divWad(newRsETHPrice);
    _checkAndUpdateDailyFeeMintLimit(rsethAmountToMintAsProtocolFee); // ← reverts
```

`remainingDailyFeeMintLimit()` (line 171) already contains `if (maxFeeMintAmountPerDay == 0) return 0;`, confirming the developers anticipated this state, but the internal function was never given the same treatment. `updateRSETHPrice()` is `public` (line 87), so any external caller observes the revert.

## Impact Explanation
**Medium — Permanent freezing of unclaimed yield.**

The protocol fee (`protocolFeeInETH`) that should be minted as rsETH to the treasury is never minted for any period during which `maxFeeMintAmountPerDay == 0`. Because `rsETHPrice` is never written (line 313 is never reached), the fee calculation on the next successful call uses the stale stored price, making the accumulated fee for the frozen periods permanently unrecoverable. This matches the allowed impact: *Permanent freezing of unclaimed yield*.

## Likelihood Explanation
**Medium.** The default post-deployment/upgrade state has `maxFeeMintAmountPerDay == 0`. If the manager does not call `setMaxFeeMintAmountPerDay` before staking rewards cause TVL to grow, the oracle breaks on the very first price update. `protocolFeeInBPS` is a live config parameter; as long as it is non-zero and TVL grows (normal operating conditions), the revert path is hit on every call. No attacker capability is required — any public caller triggers the revert.

## Recommendation
Add a zero-limit early-return inside `_checkAndUpdateDailyFeeMintLimit`, mirroring the guard already present in `remainingDailyFeeMintLimit`:

```solidity
function _checkAndUpdateDailyFeeMintLimit(uint256 feeAmount) internal {
    if (maxFeeMintAmountPerDay == 0) return; // ← add this guard
    ...
}
```

Alternatively, add a non-zero check in `setMaxFeeMintAmountPerDay` and ensure `reinitialize` initialises `maxFeeMintAmountPerDay` to a safe non-zero default.

## Proof of Concept
1. Deploy `LRTOracle`. `maxFeeMintAmountPerDay` is `0` (uninitialized). `protocolFeeInBPS` in `LRTConfig` is non-zero (e.g., 500 bps).
2. Time passes; staking rewards cause `totalETHInProtocol > previousTVL`.
3. Any account calls `updateRSETHPrice()` (public, no access control).
4. Inside `_updateRsETHPrice`: `protocolFeeInETH > 0` → `rsethAmountToMintAsProtocolFee > 0`.
5. `_checkAndUpdateDailyFeeMintLimit(rsethAmountToMintAsProtocolFee)`: `0 + rsethAmountToMintAsProtocolFee > 0` → `revert DailyFeeMintLimitExceeded(...)`.
6. `rsETHPrice` is never updated (line 313 unreachable). Protocol fee is never minted. All subsequent calls revert identically. The fee accrued during the frozen period is permanently lost.

```text
    function _checkAndUpdateDailyFeeMintLimit(uint256 feeAmount) internal {
        // Reset the period if it's unset or a day has passed
        if (block.timestamp >= feePeriodStartTime + 1 days) {
            currentPeriodMintedFeeAmount = 0;
            feePeriodStartTime = getCurrentPeriodStartTime();
        }

        // Check if minting would exceed the daily limit
        if (currentPeriodMintedFeeAmount + feeAmount > maxFeeMintAmountPerDay) {
            revert DailyFeeMintLimitExceeded(currentPeriodMintedFeeAmount + feeAmount, maxFeeMintAmountPerDay);
        }

        currentPeriodMintedFeeAmount += feeAmount;
    }
```
