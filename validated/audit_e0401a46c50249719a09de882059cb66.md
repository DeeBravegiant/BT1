Audit Report

## Title
Unrestricted `sendFunds()` in `FeeReceiver` Enables Stale-Price Deposit to Steal Accumulated MEV Yield - (File: contracts/FeeReceiver.sol)

## Summary
`FeeReceiver.sendFunds()` has no access-control modifier, allowing any external caller to flush accumulated MEV/execution-layer rewards into `LRTDepositPool` at will. Because `LRTOracle.updateRSETHPrice()` reverts when the resulting price increase exceeds `pricePercentageLimit`, an attacker can force the reward dump, cause the oracle update to revert, and then deposit at the stale (under-valued) `rsETHPrice` — receiving more rsETH than the true exchange rate warrants and diluting existing holders' claim on those rewards.

## Finding Description
`FeeReceiver.sendFunds()` is declared `external` with no role check:

```solidity
// contracts/FeeReceiver.sol L53-58
function sendFunds() external {
    uint256 balance = address(this).balance;
    ILRTDepositPool(depositPool).receiveFromRewardReceiver{ value: balance }();
    emit MevRewardsAddedToTVL(balance);
}
``` [1](#0-0) 

The receiving function in `LRTDepositPool` is also unrestricted:

```solidity
// contracts/LRTDepositPool.sol L61
function receiveFromRewardReceiver() external payable { }
``` [2](#0-1) 

Once ETH lands in the deposit pool, `getETHDistributionData()` counts it immediately via `address(this).balance`: [3](#0-2) 

`LRTOracle.updateRSETHPrice()` is publicly callable and computes a new price from this inflated TVL. If the increase exceeds `pricePercentageLimit`, it reverts for non-managers, leaving `rsETHPrice` at its old, lower value: [4](#0-3) 

A subsequent deposit then mints rsETH using the stale denominator: [5](#0-4) 

**Exploit sequence:**
1. FeeReceiver accumulates a large ETH balance over days/weeks.
2. Attacker calls `FeeReceiver.sendFunds()` — all ETH moves to `LRTDepositPool`.
3. Attacker calls `LRTOracle.updateRSETHPrice()` — TVL spike pushes `newRsETHPrice` above `highestRsethPrice * (1 + pricePercentageLimit)`, causing revert. `rsETHPrice` remains stale.
4. Attacker calls `LRTDepositPool.depositETH{value: X}(minRSETH, "")` — rsETH minted at the stale lower price, giving the attacker more rsETH than the true rate warrants.
5. Manager eventually calls `updateRSETHPriceAsManager()` — price corrects upward. Attacker's rsETH is now over-valued at the expense of all pre-existing holders.

The `pricePercentageLimit` guard is explicitly designed to throttle large single-block price jumps, but it inadvertently creates a window where the TVL has increased (rewards are in the pool) while the price has not yet reflected it.

## Impact Explanation
**High — Theft of unclaimed yield.**

Existing rsETH holders are entitled to the appreciation that accumulated MEV rewards represent. By forcing the reward dump and depositing at the stale price before the oracle can update, the attacker captures a disproportionate share of that appreciation. Every pre-existing holder's rsETH is diluted relative to the true underlying ETH value. The magnitude scales with the size of the accumulated FeeReceiver balance and the rsETH supply.

## Likelihood Explanation
**Medium.**

- `pricePercentageLimit` is an explicitly supported admin parameter with a dedicated setter (`setPricePercentageLimit`) and enforcement logic, making it likely to be non-zero in production.
- MEV/execution-layer rewards accumulate passively; over days or weeks the balance can be large enough to exceed a 1–2% daily threshold in a single call.
- The attack requires only two public calls (`sendFunds()` + `updateRSETHPrice()`) and a deposit — no special privileges, no flash loan, no front-running of a specific transaction.
- It is repeatable each time the FeeReceiver re-accumulates a sufficient balance.

## Recommendation
Restrict `sendFunds()` to an authorized role:

```solidity
function sendFunds() external onlyRole(LRTConstants.MANAGER) {
    uint256 balance = address(this).balance;
    ILRTDepositPool(depositPool).receiveFromRewardReceiver{ value: balance }();
    emit MevRewardsAddedToTVL(balance);
}
```

Additionally, restrict `receiveFromRewardReceiver()` in `LRTDepositPool` to only accept calls from the registered `FeeReceiver` address, preventing direct ETH injection from arbitrary callers that could achieve the same TVL inflation independently.

## Proof of Concept
```
Setup:
  - FeeReceiver holds 500 ETH (accumulated MEV rewards)
  - rsETHPrice = highestRsethPrice = 1.05e18
  - pricePercentageLimit = 1e16 (1%)
  - rsETH totalSupply = 50,000e18

Step 1: attacker.call(FeeReceiver.sendFunds())
  → 500 ETH transferred to LRTDepositPool
  → LRTDepositPool.balance increases by 500 ETH
  → getETHDistributionData() now returns 500 ETH more

Step 2: attacker.call(LRTOracle.updateRSETHPrice())
  → _getTotalEthInProtocol() returns ~500 ETH higher value
  → newRsETHPrice ≈ 1.06e18 (>1% above highestRsethPrice of 1.05e18)
  → isPriceIncreaseOffLimit = true
  → msg.sender is not MANAGER → revert PriceAboveDailyThreshold()
  → rsETHPrice remains 1.05e18

Step 3: attacker.call(LRTDepositPool.depositETH{value: 100 ETH}(minRSETH, ""))
  → getRsETHAmountToMint: (100e18 * 1e18) / 1.05e18 ≈ 95.24 rsETH
  → True fair price would be ~1.06e18, giving only ~94.34 rsETH
  → Attacker receives ~0.9 rsETH excess (~0.95 ETH of value)

Step 4: manager.call(LRTOracle.updateRSETHPriceAsManager())
  → rsETHPrice updates to ~1.06e18
  → Attacker's 95.24 rsETH is now worth ~101 ETH instead of 100 ETH
  → Existing holders' rsETH is diluted by the attacker's over-minted share

Foundry fork test plan:
  1. Fork mainnet at a block where FeeReceiver has significant balance
  2. Record rsETHPrice and highestRsethPrice
  3. Prank as attacker, call sendFunds() then updateRSETHPrice() (expect revert)
  4. Prank as attacker, depositETH with X ETH, record rsethReceived
  5. Prank as manager, call updateRSETHPriceAsManager()
  6. Assert attacker's rsETH * newRsETHPrice > X ETH (profit confirmed)
  7. Assert existing holder's ETH-equivalent value decreased
```
