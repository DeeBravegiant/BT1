Audit Report

## Title
Stale Cross-Chain Rate Used for wrsETH Minting With No Staleness Guard - (File: contracts/cross-chain/CrossChainRateReceiver.sol)

## Summary
`CrossChainRateReceiver.getRate()` returns the stored `rate` unconditionally, never consulting `lastUpdated`. All L2 pool contracts (`RSETHPoolV3ExternalBridge`, `RSETHPoolV3WithNativeChainBridge`, `RSETHPoolV2ExternalBridge`, `RSETHPoolNoWrapper`) route every deposit through this function to compute the wrsETH mint amount. Because rsETH/ETH is monotonically increasing, any stale snapshot is always lower than the true rate, causing depositors to receive excess wrsETH proportional to the staleness gap, which they can redeem at the true L1 rate for a profit at the protocol's expense.

## Finding Description
`CrossChainRateReceiver` stores the rsETH/ETH exchange rate in `rate` and records the update time in `lastUpdated` on every `lzReceive()` call:

```solidity
// contracts/cross-chain/CrossChainRateReceiver.sol L93-97
uint256 _rate = abi.decode(_payload, (uint256));
rate = _rate;
lastUpdated = block.timestamp;
```

However, `getRate()` returns `rate` without any reference to `lastUpdated`:

```solidity
// contracts/cross-chain/CrossChainRateReceiver.sol L102-105
function getRate() external view returns (uint256) {
    return rate;
}
```

Every L2 pool contract calls `IOracle(rsETHOracle).getRate()` to compute the mint amount:

```solidity
// contracts/pools/RSETHPoolV3ExternalBridge.sol L422-426
uint256 rsETHToETHrate = getRate();
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
```

The same pattern appears in `RSETHPoolV3WithNativeChainBridge` (L340-343), `RSETHPoolV2ExternalBridge` (L312-315), and `RSETHPoolNoWrapper` (L282-285). The rate is only refreshed when an off-chain keeper pays LayerZero fees to call `updateRate()` on `RSETHMultiChainRateProvider`. There is no on-chain maximum-age threshold, no revert on staleness, and no fallback. By contrast, `ChainlinkOracleForRSETHPoolCollateral.getRate()` enforces `answeredInRound < roundID` and `timestamp == 0` guards, demonstrating the protocol is aware of staleness risks for other oracle types but omitted the guard here.

**Exploit flow:**
1. Keeper stops updating (high gas, inactivity, or deliberate delay). `lastUpdated` falls behind.
2. True rsETH/ETH rate rises from `1.05e18` to `1.06e18` while on-chain `rate` remains `1.05e18`.
3. Attacker calls `deposit{value: 1 ether}()` on any L2 pool.
4. `viewSwapRsETHAmountAndFee(1e18)` calls `getRate()` → returns stale `1.05e18`.
5. `rsETHAmount = 1e18 * 1e18 / 1.05e18 ≈ 0.9524 wrsETH` (correct at true rate would be `≈ 0.9434 wrsETH`).
6. Attacker bridges wrsETH to L1 and unwraps at the true rate `1.06e18`, recovering `0.9524 * 1.06e18 / 1e18 ≈ 1.0096 ETH`.
7. Net profit: ~0.96% of deposit, sourced from the protocol's pre-minted rsETH reserve or diluting other holders.

## Impact Explanation
A normal depositor can extract more ETH than they deposited by timing a deposit during a stale oracle window and redeeming wrsETH at the true L1 rate. The surplus is drawn from the protocol's rsETH reserve, constituting direct theft of unclaimed yield. Repeated over many deposits during a prolonged stale window, the cumulative drain can reach protocol insolvency. This matches the allowed impact: **High — Theft of unclaimed yield**.

## Likelihood Explanation
`updateRate()` on `RSETHMultiChainRateProvider` is permissionless but requires the caller to supply native ETH for LayerZero messaging fees. During periods of high gas prices, network congestion, or keeper inactivity, updates can be delayed for hours or longer. The existence of `lastUpdated` in the contract confirms the design anticipates gaps between updates, yet no on-chain guard enforces a maximum gap. Any depositor can read `lastUpdated` from the public state variable and time a deposit to exploit the stale window. No special privileges are required. **Likelihood: Medium.**

## Recommendation
Add a configurable `maxStaleness` parameter to `CrossChainRateReceiver` and revert in `getRate()` if the rate is too old:

```solidity
uint256 public maxStaleness; // e.g. 24 hours, set by owner

function getRate() external view returns (uint256) {
    require(block.timestamp - lastUpdated <= maxStaleness, "Rate is stale");
    return rate;
}
```

This mirrors the staleness-check pattern already present in `ChainlinkOracleForRSETHPoolCollateral` and should be applied consistently across all oracle implementations used by the L2 pools.

## Proof of Concept
**Minimal Foundry fork test plan:**

1. Fork an L2 (e.g., Optimism) at a block where `CrossChainRateReceiver.lastUpdated` is recent.
2. Record `rate = CrossChainRateReceiver.rate` (e.g., `1.05e18`).
3. `vm.warp(block.timestamp + 24 hours)` to simulate keeper inactivity; do not call `lzReceive`.
4. Confirm `CrossChainRateReceiver.getRate()` still returns `1.05e18`.
5. Call `RSETHPoolV3ExternalBridge.deposit{value: 1 ether}("")` as attacker.
6. Assert attacker received `≈ 0.9524 wrsETH` (stale rate) vs. expected `≈ 0.9434 wrsETH` (true rate).
7. Simulate bridging wrsETH to L1 and unwrapping at the true rate `1.06e18`.
8. Assert attacker recovers `> 1 ETH`, confirming extraction of unclaimed yield at the protocol's expense.
