Audit Report

## Title
Deposit output amount silently truncates to zero with no output guard, permanently losing depositor funds — (`contracts/pools/RSETHPool.sol`, `contracts/pools/RSETHPoolNoWrapper.sol`, `contracts/pools/RSETHPoolV3.sol`, `contracts/agETH/AGETHPoolV3.sol`)

## Summary
All four pool contracts compute the output token amount (`rsETHAmount` / `agETHAmount`) via integer division that can silently produce zero for dust-sized inputs. No guard exists on the output value. When this occurs, the depositor's ETH or ERC-20 tokens are accepted and permanently absorbed into the pool's bridgeable balance, while the depositor receives nothing and has no recovery path.

## Finding Description
Every deposit path shares the same truncation-prone formula. For ETH:

```solidity
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;   // RSETHPool.sol L319
```

For tokens:

```solidity
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;  // RSETHPool.sol L346
```

Because `rsETHToETHrate` is always `>= 1e18` (rsETH/agETH accrue value over time), any ETH deposit where `amountAfterFee < rsETHToETHrate / 1e18` (a few wei) produces `rsETHAmount = 0`. The token path truncates when `amountAfterFee * tokenToETHRate < rsETHToETHrate`.

The only existing guard checks the *input*:

```solidity
if (amount == 0) revert InvalidAmount();   // RSETHPool.sol L269, RSETHPoolNoWrapper.sol L235, RSETHPoolV3.sol L256, AGETHPoolV3.sol L119
```

There is no corresponding guard on the *output*. Execution then proceeds unconditionally:

- `RSETHPool.sol` L275: `IERC20(address(wrsETH)).safeTransfer(msg.sender, rsETHAmount)` — transfers 0 silently.
- `RSETHPoolNoWrapper.sol` L241: `rsETH.safeTransfer(msg.sender, rsETHAmount)` — transfers 0 silently.
- `RSETHPoolV3.sol` L262: `wrsETH.mint(msg.sender, rsETHAmount)` — mints 0 silently.
- `AGETHPoolV3.sol` L125: `agETH.mint(msg.sender, agETHAmount)` — mints 0 silently.

The deposited ETH or tokens are now held by the pool, counted as part of the bridgeable balance, and will be bridged to L1 with no depositor claim attached. The `addSupportedToken` function (L637–655) imposes no restriction on token decimals, meaning a 6-decimal token (e.g., USDC) would raise the truncation threshold to potentially thousands of base units — making accidental zero-output deposits realistic for non-dust amounts.

## Impact Explanation
**Low — Contract fails to deliver promised returns.**

A depositor who sends a dust ETH amount (e.g., 1 wei) or a small token amount receives 0 rsETH/agETH. Their deposited assets are permanently absorbed into the pool's bridgeable balance with no recovery mechanism. The protocol does not lose funds, but the depositor does. The front-running sub-scenario is excluded per program rules.

## Likelihood Explanation
**Low.** For 18-decimal ETH-correlated LSTs, the truncation threshold is a few wei — amounts no rational user would intentionally deposit. However, the `addSupportedToken` function places no restriction on token decimals; if a 6-decimal token is added by the TIMELOCK_ROLE, the truncation threshold rises to potentially several thousand base units, making accidental zero-output deposits realistic for ordinary users.

## Recommendation
Add an output-amount guard immediately after computing `rsETHAmount` / `agETHAmount` in every deposit function and in the corresponding `view` helpers:

```solidity
if (rsETHAmount == 0) revert InvalidAmount();
```

Apply the same guard in `viewSwapRsETHAmountAndFee` / `viewSwapAgETHAmountAndFee` so callers can detect the condition off-chain before submitting a transaction.

## Proof of Concept
**ETH deposit path (`RSETHPool.sol`):**

1. `rsETHToETHrate` = `1.05e18` (rsETH has accrued 5% above peg).
2. Depositor calls `deposit{value: 1}("")` (1 wei ETH).
3. `viewSwapRsETHAmountAndFee(1)`:
   - `fee = 1 * feeBps / 10_000 = 0`
   - `amountAfterFee = 1`
   - `rsETHAmount = 1 * 1e18 / 1.05e18 = 0` (integer truncation)
4. `feeEarnedInETH += 0`.
5. `IERC20(address(wrsETH)).safeTransfer(msg.sender, 0)` — succeeds silently.
6. Depositor's 1 wei is now in the pool's ETH balance, counted as bridgeable, and will be sent to L1. Depositor holds no rsETH and has no claim.

**Foundry fuzz test plan:**

```solidity
function testFuzz_depositZeroOutput(uint256 amount) public {
    vm.assume(amount > 0 && amount < pool.getRate() / 1e18);
    uint256 balBefore = rsETH.balanceOf(address(this));
    pool.deposit{value: amount}("");
    uint256 balAfter = rsETH.balanceOf(address(this));
    assertEq(balAfter - balBefore, 0); // depositor received nothing
    // ETH is now permanently in pool with no claim
}
```
