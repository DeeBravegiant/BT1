Audit Report

## Title
Uninitialized `maxNumberOfWithdrawalsPerUser` Causes DoS on `initiateWithdrawal()` Until Admin Intervention - (File: contracts/KERNEL/KernelDepositPool.sol)

## Summary
`KernelDepositPool.initialize()` never assigns `maxNumberOfWithdrawalsPerUser`, leaving it at the Solidity default of `0`. The guard in `initiateWithdrawal()` evaluates `userWithdrawalIds[msg.sender].length (0) >= maxNumberOfWithdrawalsPerUser (0)` as `true` for every caller immediately after deployment, reverting with `WithdrawalLimitReached()`. All staked funds are frozen until an admin manually calls `setMaxNumberOfWithdrawalsPerUser()` with a non-zero value.

## Finding Description
`initialize()` configures roles and token addresses but omits any assignment to `maxNumberOfWithdrawalsPerUser`:

```solidity
// contracts/KERNEL/KernelDepositPool.sol lines 259-271
function initialize(address _admin, address _kernelToken, address _rewardToken) external initializer {
    ...
    kernelToken = IERC20(_kernelToken);
    rewardsToken = IERC20(_rewardToken);
    // maxNumberOfWithdrawalsPerUser is never set → defaults to 0
}
```

`initiateWithdrawal()` enforces the limit before creating a withdrawal record:

```solidity
// line 323
if (userWithdrawalIds[msg.sender].length >= maxNumberOfWithdrawalsPerUser) revert WithdrawalLimitReached();
```

With `maxNumberOfWithdrawalsPerUser == 0`, the condition `0 >= 0` is always `true` for any user whose `userWithdrawalIds` array is empty (i.e., every user on first attempt). The revert fires before any state change, so no withdrawal record is ever created and `claimWithdrawal()` is unreachable. The only recovery path is an admin calling `setMaxNumberOfWithdrawalsPerUser()` (lines 610–620), which is gated by `DEFAULT_ADMIN_ROLE` and rejects `0` as input, confirming `0` is never a valid operational value.

## Impact Explanation
Every user who has staked KERNEL tokens via `stake()` or `stakeFor()` is unable to initiate a withdrawal from the moment of deployment. Their principal remains locked in the contract. This is a concrete **temporary freezing of funds** (Medium per the allowed scope), recoverable only through admin intervention.

## Likelihood Explanation
The condition is triggered unconditionally on the very first call to `initiateWithdrawal()` by any user after deployment. No attacker setup, special conditions, or economic assumptions are required. Any ordinary staker attempting to exit will hit the revert. The likelihood is high.

## Recommendation
Set a safe default in `initialize()`:

```solidity
maxNumberOfWithdrawalsPerUser = MAX_WITHDRAWALS_PER_USER; // 100
```

This mirrors the intent of the constant defined at line 38 and eliminates the uninitialized-state window entirely.

## Proof of Concept
1. Deploy `KernelDepositPool` behind a proxy and call `initialize(admin, kernelToken, rewardToken)`.
2. Read `maxNumberOfWithdrawalsPerUser` — returns `0`.
3. User calls `stake(1e18)` — succeeds; `balanceOf[user] = 1e18`.
4. User calls `initiateWithdrawal(1e18)`.
5. Execution reaches line 323: `userWithdrawalIds[user].length (0) >= maxNumberOfWithdrawalsPerUser (0)` → `true` → reverts `WithdrawalLimitReached()`.
6. No withdrawal record is created; `claimWithdrawal()` is unreachable.
7. User's `1e18` KERNEL is locked until admin calls `setMaxNumberOfWithdrawalsPerUser(N)` where `0 < N ≤ 100`.

**Foundry test sketch:**
```solidity
function test_withdrawalDosOnDeploy() public {
    // deploy + initialize (no setMaxNumberOfWithdrawalsPerUser call)
    kernelToken.approve(address(pool), 1e18);
    pool.stake(1e18);
    vm.expectRevert(KernelDepositPool.WithdrawalLimitReached.selector);
    pool.initiateWithdrawal(1e18);
}
```
