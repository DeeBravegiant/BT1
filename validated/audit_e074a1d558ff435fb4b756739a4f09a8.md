### Title
Delegate Authority Allows Attacker to Redirect Victim's Tokens to Attacker-Controlled NEAR Account — (`solana/programs/bridge_token_factory/src/instructions/user/init_transfer.rs`)

---

### Summary

The `InitTransfer` account struct imposes no ownership constraint binding `from` to `user`. Because SPL Token's `transfer_checked` accepts either the account owner **or** a delegate as the authority, any attacker who holds delegate authority over a victim's token account can call `init_transfer` with `from=victim_account, user=attacker`. The transfer succeeds, the victim's tokens are locked in the vault, and the Wormhole message is emitted with `sender=attacker_pubkey` — causing NEAR to credit the attacker's account.

---

### Finding Description

In `InitTransfer`, the `from` account is declared with only mint and token-program constraints:

```rust
#[account(
    mut,
    token::mint = mint,
    token::token_program = token_program,
)]
pub from: Box<InterfaceAccount<'info, TokenAccount>>,
``` [1](#0-0) 

There is no `token::authority = user` constraint. The `user` field is only required to be a `Signer` owned by the system program:

```rust
#[account(
    mut,
    owner = common.system_program.key(),
)]
pub user: Signer<'info>,
``` [2](#0-1) 

In `process()`, the `transfer_checked` CPI passes `self.user` as the authority:

```rust
TransferChecked {
    from: self.from.to_account_info(),
    to: vault.to_account_info(),
    authority: self.user.to_account_info(),
    mint: self.mint.to_account_info(),
},
``` [3](#0-2) 

SPL Token's `transfer_checked` accepts **either the owner or a delegate** as the authority. If `user` (attacker) holds delegate authority over `from` (victim's account), the CPI succeeds without any program-level rejection.

After the transfer, the Wormhole message is posted with `sender = self.user.key()`:

```rust
self.common.post_message(payload.serialize_for_near((
    self.common.sequence.sequence,
    self.user.key(),
    self.mint.key(),
))?)?;
``` [4](#0-3) 

On the NEAR side, `serialize_for_near` encodes `params.1` (the `user` pubkey) as the `sender` field:

```rust
// 1. sender
writer.write_all(&[SOLANA_OMNI_BRIDGE_CHAIN_ID])?;
params.1.serialize(&mut writer)?;
``` [5](#0-4) 

NEAR's `fin_transfer_callback` uses this `sender` field directly from the verified Wormhole message to credit the cross-chain transfer — so the attacker's NEAR account receives the credit for the victim's locked tokens. [6](#0-5) 

The same missing constraint applies to the `burn` path for bridged tokens:

```rust
Burn {
    mint: self.mint.to_account_info(),
    from: self.from.to_account_info(),
    authority: self.user.to_account_info(),
},
``` [7](#0-6) 

---

### Impact Explanation

The victim's tokens are irreversibly locked in the vault (native path) or burned (bridged path). The cross-chain credit is minted/released to the attacker's NEAR account. The victim has no recourse: the Wormhole VAA is valid, the nonce is consumed, and the NEAR-side transfer is finalized. This breaks the core backing invariant that the cross-chain sender corresponds to the economic owner of the locked tokens.

---

### Likelihood Explanation

Delegate authority is a standard SPL Token feature routinely granted by users to DeFi protocols (DEXes, lending markets, aggregators). Any protocol that holds delegate authority over user token accounts — or any attacker who can socially engineer a small approval — can exploit this. The attack requires no privileged bridge role, no compromised key, and no chain-level assumption. It is locally reproducible with a standard Anchor/Mollusk test.

---

### Recommendation

Add `token::authority = user` to the `from` account constraint in `InitTransfer`:

```rust
#[account(
    mut,
    token::mint = mint,
    token::authority = user,   // <-- add this
    token::token_program = token_program,
)]
pub from: Box<InterfaceAccount<'info, TokenAccount>>,
```

This Anchor constraint checks `from.owner == user.key()` at deserialization time, ensuring only the token account owner can initiate a bridge transfer. Delegates are explicitly excluded, which is the correct semantics for a bridge that encodes `user` as the cross-chain sender.

### Proof of Concept

```rust
// Setup:
// - victim owns victim_token_account with 1_000_000 tokens
// - victim calls spl_token::approve(victim_token_account, attacker, 1_000_000)
// - attacker now holds delegate authority

// Attack:
// attacker constructs and signs the following instruction:
InitTransfer {
    authority: authority_pda,
    mint: token_mint,
    from: victim_token_account,   // victim's account, NOT attacker's
    vault: Some(vault_pda),
    sol_vault: sol_vault_pda,
    user: attacker,               // attacker signs
    ...
}
payload = InitTransferPayload {
    amount: 1_000_000,
    recipient: "attacker.near",   // attacker's NEAR account
    fee: 0,
    native_fee: 0,
    message: "",
}

// Result:
// 1. transfer_checked(from=victim_account, authority=attacker) succeeds
//    because attacker is a valid delegate
// 2. Wormhole message emitted: sender=attacker_pubkey, recipient="attacker.near"
// 3. NEAR fin_transfer_callback credits attacker.near with 1_000_000 tokens
// 4. victim_token_account is drained; victim has no recourse
```
