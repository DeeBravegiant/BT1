### Title
Missing Response Validation in `respond_ckd` for `AppPublicKey` Variant Allows Single Byzantine Participant to Deliver Forged Key Derivation Output - (File: crates/contract/src/lib.rs)

### Summary

In `respond_ckd`, when a CKD request uses the `CKDAppPublicKey::AppPublicKey` variant, the contract performs **zero validation** of the submitted response. Any single attested MPC participant can call `respond_ckd` with an arbitrary forged `CKDResponse`, and the contract will unconditionally resolve all pending yields for that request with the forged data. This is the direct analog of the reported Chakra bug: a handler that always returns success regardless of whether the response is correct.

### Finding Description

In `crates/contract/src/lib.rs`, the `respond_ckd` function contains the following validation block:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}   // ← empty: no validation
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For the `AppPublicKeyPV` variant, `ckd_output_check` is called to verify the CKD output against the app public key and the domain's root public key. For the `AppPublicKey` variant, the match arm is completely empty — no check is performed. The function then unconditionally proceeds to resolve all pending yields:

```rust
pending_requests::resolve_yields_for(
    &mut self.pending_ckd_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
``` [2](#0-1) 

The only access control is that the caller must be an attested participant: [3](#0-2) 

Any single attested MPC participant — not requiring threshold consensus — can call `respond_ckd` with a forged `CKDResponse` for any pending `AppPublicKey` request, and the contract will accept it and deliver the forged key to all waiting callers.

Compare this to `respond` (for signatures) and `respond_verify_foreign_tx`, both of which perform cryptographic signature verification before resolving yields: [4](#0-3) [5](#0-4) 

The `AppPublicKey` path in `respond_ckd` has no equivalent protection.

### Impact Explanation

A single Byzantine attested MPC participant (strictly below the signing threshold) can:

1. Observe a pending `request_app_private_key` call that uses `CKDAppPublicKey::AppPublicKey`.
2. Call `respond_ckd` with an arbitrary forged `CKDResponse` (e.g., a key they control).
3. The contract resolves all queued yields for that request with the forged key material.
4. The user receives a wrong confidential key, believing it to be their legitimately derived app private key.
5. The attacker, knowing the correct derived key (from their participation in the off-chain MPC computation), can use it to control any assets the user subsequently locks under that key.

This constitutes **confidential key derivation output delivered without the required threshold participant authorization**, bypassing the threshold requirement for the CKD computation. Impact: **Critical** — theft of funds or permanent loss of key material for users relying on `AppPublicKey`-variant CKD requests.

### Likelihood Explanation

The attacker must be an attested MPC participant (a Byzantine node below threshold). The `respond_ckd` method is callable by any single attested participant — there is no on-chain check that the response was produced by a threshold quorum. The `AppPublicKey` variant is a first-class enum variant in the public contract interface, reachable by any user calling `request_app_private_key`. The attack requires no special timing, no network-level interference, and no collusion beyond a single compromised node.

### Recommendation

Apply the same cryptographic output validation to `AppPublicKey` requests as is applied to `AppPublicKeyPV` requests. If the `AppPublicKey` variant genuinely cannot be verified on-chain (e.g., because it lacks the public verifiability data), the contract should either:

1. Reject `AppPublicKey` requests entirely and require callers to use `AppPublicKeyPV`, or
2. Implement an equivalent on-chain check using the domain's root public key and the app ID, analogous to `ckd_output_check`.

The empty match arm `dtos::CKDAppPublicKey::AppPublicKey(_) => {}` should never silently skip validation for a function that resolves confidential key material.

### Proof of Concept

1. User calls `request_app_private_key` with `CKDAppPublicKey::AppPublicKey(some_bls_pk)`.
2. Contract stores a pending yield under the resulting `CKDRequest` key.
3. Malicious attested participant constructs a `CKDResponse` containing an arbitrary key they control.
4. Malicious participant calls `respond_ckd(request, forged_response)`.
5. Contract enters the `AppPublicKey(_) => {}` branch — no validation occurs.
6. `resolve_yields_for` is called, resuming the user's yield with the forged key.
7. User's transaction completes successfully, returning the forged key as if it were the legitimate MPC-derived output.
8. Attacker uses the real derived key (known from off-chain MPC participation) to control any assets the user places under it. [6](#0-5)

    #[handle_result]
    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }
