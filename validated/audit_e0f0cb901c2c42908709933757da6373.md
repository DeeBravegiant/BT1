Audit Report

## Title
Rounding to Zero in Token Deposit Calculation Causes Depositor to Lose Tokens Without Receiving wrsETH - (File: contracts/pools/RSETHPoolV3.sol)

## Summary
The `deposit` function in `RSETHPoolV3` (and sibling pool contracts) transfers the depositor's tokens into the pool before computing the wrsETH mint amount. When the deposit amount is small relative to the rsETH/token rate ratio, integer division in `viewSwapRsETHAmountAndFee` truncates `rsETHAmount` to zero. No zero-amount guard exists on the result, so the depositor permanently loses their tokens while receiving nothing.

## Finding Description
In `RSETHPoolV3.deposit` (L271–293), tokens are transferred from the user before the mint amount is computed:

```solidity
IERC20(token).safeTransferFrom(msg.sender, address(this), amount); // tokens taken first
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount, token);
feeEarnedInToken[token] += fee;
wrsETH.mint(msg.sender, rsETHAmount); // mints 0 — no zero-check
```

In `viewSwapRsETHAmountAndFee` (L315–335):

```solidity
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
```

When `amountAfterFee * tokenToETHRate < rsETHToETHrate`, Solidity integer division yields `rsETHAmount = 0`.

Oracle constraints enforce `tokenToETHRate ∈ [1e16, 1e19]` (via `LRTOracle.updatePriceOracleForValidated` L104) and `rsETHToETHrate >= 1e18` (via `InterimRSETHOracle._setRate` L42). At the worst-case ratio (`tokenToETHRate = 1e16`, `rsETHToETHrate = 1e18`), any deposit where `amountAfterFee < 100` yields `rsETHAmount = 0`.

The only input guard is `if (amount == 0) revert InvalidAmount()` (L282), which does not prevent a non-zero amount from producing a zero output. The `limitDailyMint` modifier (L96–125) also does not block this: when `rsETHAmount = 0`, the check `dailyMintAmount + 0 > dailyMintLimit` passes normally, and `dailyMintAmount += 0` is a no-op.

The identical pattern exists in `RSETHPool`, `RSETHPoolNoWrapper`, `RSETHPoolV3ExternalBridge`, `RSETHPoolV3WithNativeChainBridge`, `RSETHPoolV2`, `RSETHPoolV2ExternalBridge`, and `RSETHPoolV2NBA`.

## Impact Explanation
A depositor who sends a small token amount (e.g., 1–99 wei of a token whose oracle rate is at the minimum 1e16) has their tokens permanently transferred into the pool contract while receiving zero wrsETH. The tokens are not recoverable by the user. This matches the allowed impact: **Low — Contract fails to deliver promised returns; depositor loses deposited token value without receiving the corresponding wrsETH.**

## Likelihood Explanation
Any external caller can invoke `deposit(address token, uint256 amount, string referralId)` with a non-zero but small `amount`. No role or whitelist is required. The condition is triggered whenever `amountAfterFee * tokenToETHRate < rsETHToETHrate`, which is achievable with small amounts of any supported low-rate token. The likelihood is low in practice because the affected amounts are tiny (sub-100 wei), but the path is fully permissionless and requires no special conditions beyond a small deposit.

## Recommendation
Add a zero-amount guard on the computed `rsETHAmount` before minting (or before transferring tokens):

```solidity
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount, token);
if (rsETHAmount == 0) revert InvalidAmount();
```

Alternatively, enforce a minimum deposit amount per token that guarantees `rsETHAmount >= 1` given the current rate, similar to how `LRTDepositPool` uses `minAmountToDeposit` and `minRSETHAmountExpected`.

## Proof of Concept
Assume:
- `feeBps = 0`
- `tokenToETHRate = 1e16` (minimum allowed by `LRTOracle.updatePriceOracleForValidated`)
- `rsETHToETHrate = 1e18` (minimum enforced by `InterimRSETHOracle`)
- User calls `deposit(token, 50, "")`

Execution trace:
1. `fee = 50 * 0 / 10_000 = 0`
2. `amountAfterFee = 50`
3. `rsETHAmount = 50 * 1e16 / 1e18 = 5e17 / 1e18 = 0` (integer truncation)
4. `IERC20(token).safeTransferFrom(msg.sender, address(this), 50)` — 50 wei transferred out of user
5. `wrsETH.mint(msg.sender, 0)` — user receives nothing

The user has permanently lost 50 wei of the token with no recourse.

**Foundry fuzz test plan:**
```solidity

    amount = bound(amount, 1, 99); // sub-threshold amounts
