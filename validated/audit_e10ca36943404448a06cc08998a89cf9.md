### Title
Premature Yield Resumption via Arbitrary `account_id` in `storage_deposit` Permanently Locks Victim Storage Deposits and Cancels Pending Transfers - (File: near/omni-bridge/src/storage.rs)

### Summary

`storage_deposit` is a public, permissionless function that accepts an arbitrary `account_id`. When a user's `init_transfer` yields (due to insufficient storage), the contract stores a yield-resume handle keyed by a deterministic virtual `message_storage_account_id`. An unprivileged attacker can call `storage_deposit` for that virtual account with only the minimum account balance, prematurely consuming the yield handle. The `init_transfer_resume` callback then fires with insufficient balance, cancels the transfer (refunding the user's tokens), and leaves the yield handle permanently removed. Any storage the victim subsequently deposits to the same virtual account is irreversibly locked with no withdrawal path.

### Finding Description

**Step 1 – Yield creation.**
When `init_transfer` cannot find sufficient storage for the signer, it creates a NEAR yield promise and stores the yield ID in `init_transfer_promises[message_storage_account_id]`: [1](#0-0) 

The `message_storage_account_id` is a deterministic hash of `(token, amount, recipient, fee, sender, msg)` — **nonces are explicitly excluded** — so it is fully predictable from the public `ft_transfer_call` transaction.

**Step 2 – Public entry point with no access control.**
`storage_deposit` accepts an arbitrary `account_id`, updates `accounts_balances` for it, and unconditionally calls `resume_promise`: [2](#0-1) 

**Step 3 – Premature yield consumption.**
`resume_promise` calls `env::promise_yield_resume(promise_id, [])`. A single successful call permanently consumes the yield handle: [3](#0-2) 

**Step 4 – Callback fires with insufficient balance.**
`init_transfer_resume` removes the promise entry and calls `try_to_transfer_balance_from_message_account`. Because the attacker deposited only `min_required_storage_balance` (far below the transfer storage cost), the check fails and the transfer is cancelled: [4](#0-3) 

**Step 5 – Victim's subsequent deposit is permanently locked.**
After the yield is consumed, the victim calls `storage_deposit(message_storage_account_id, sufficient_amount)`. The balance is added to `accounts_balances` for the virtual account, but `resume_promise` finds no live yield (it was already consumed) and does nothing. The virtual account is not a real NEAR account, so `storage_withdraw` (which uses `env::predecessor_account_id()`) cannot be called for it. The victim's deposit is permanently unrecoverable: [5](#0-4) 

### Impact Explanation

- **Frozen redemption path:** The victim's `ft_transfer_call` is cancelled on every attempt via the yield path. The attacker can repeat the attack for each new transfer attempt since the `message_storage_account_id` is predictable.
- **Irreversible fund lock:** Any NEAR the victim deposits to the virtual `message_storage_account_id` after the attack is permanently locked. There is no privileged recovery function and no way to call `storage_withdraw` as a virtual account.
- The victim's bridged tokens are refunded (not lost), but the NEAR storage deposit is permanently unclaimable.

### Likelihood Explanation

- `storage_deposit` is fully public with no role restriction.
- The `message_storage_account_id` is deterministic and computable from the public `ft_transfer_call` transaction before the victim's `storage_deposit` is submitted.
- The attacker must front-run the victim's `storage_deposit` call. On NEAR, transaction ordering within a block is observable, making front-running feasible.
- The attacker's cost is `required_balance_for_account()` (a small fixed NEAR amount) per attack, which is also permanently locked — but the attacker can sustain a griefing campaign at low cost.

### Recommendation

1. **Do not call `resume_promise` from `storage_deposit` for arbitrary accounts.** Only resume the yield when the depositor is the `storage_owner` recorded in the yield's parameters, or when the deposited amount is sufficient to satisfy the pending transfer's storage requirement.
2. Alternatively, validate inside `resume_promise` (before calling `promise_yield_resume`) that the current balance of the virtual account meets the required threshold for the pending transfer. Only resume if the balance is sufficient.
3. Provide a recovery path (e.g., a DAO-callable function) to drain orphaned virtual message accounts whose yield handles have been consumed.

### Proof of Concept

```
1. Victim calls ft_transfer_call(token, amount, msg=InitTransfer{...})
   → init_transfer yields; init_transfer_promises[V] = yield_id
   (V = deterministic hash of transfer params, visible on-chain)

2. Attacker observes V from the transaction log.
   Attacker calls storage_deposit(account_id=V, deposit=min_required_storage_balance)
   → accounts_balances[V] created with tiny balance
   → resume_promise(V) called
   → promise_yield_resume(yield_id) returns true (yield consumed)

3. init_transfer_resume fires:
   → remove_promise(V)  [yield handle gone]
   → try_to_transfer_balance_from_message_account fails (balance < required)
   → returns transfer_message.amount  [victim's tokens refunded]

4. Victim, unaware, calls storage_deposit(account_id=V, deposit=sufficient_amount)
   → balance added to accounts_balances[V]
   → resume_promise(V): init_transfer_promises.get(V) = None → no-op
   → victim's NEAR deposit is now permanently locked in accounts_balances[V]
   → no withdrawal path exists for virtual account V
```

    #[payable]
    pub fn storage_deposit(&mut self, account_id: Option<AccountId>) -> StorageBalance {
        let account_id = account_id.unwrap_or_else(env::predecessor_account_id);
        let amount = env::attached_deposit();
        let storage = self.accounts_balances.get(&account_id).map_or_else(
            || {
                let min_required_storage_balance = self.required_balance_for_account();
                let available = amount
                    .checked_sub(min_required_storage_balance)
                    .near_expect(StorageError::NotEnoughStorageBalanceAttached {
                        required: min_required_storage_balance,
                        attached: amount,
                    });
                StorageBalance {
                    total: amount,
                    available,
                }
            },
            |mut storage| {
                storage.total = storage.total.saturating_add(amount);
                storage.available = storage.available.saturating_add(amount);
                storage
            },
        );
        self.accounts_balances.insert(&account_id, &storage);

        self.resume_promise(&account_id).detach();

        storage
    }

    pub fn resume_promise(&self, account_id: &AccountId) -> PromiseOrValue<()> {
        if let Some(promise_id) = &self.init_transfer_promises.get(account_id) {
            let result = env::promise_yield_resume(promise_id, []);
            env::log_str(&format!("Resume promise. Result: {result}"));

            if !result {
                return Self::ext(env::current_account_id())
                    .resume_promise(account_id)
                    .into();
            }
        }
        PromiseOrValue::Value(())
    }

    #[payable]
    pub fn storage_withdraw(&mut self, amount: Option<NearToken>) -> StorageBalance {
        assert_one_yocto();
        let account_id = env::predecessor_account_id();
        let mut storage = self
            .storage_balance_of(&account_id)
            .near_expect(StorageError::AccountNotRegistered(account_id.clone()));
        let to_withdraw = amount.unwrap_or(storage.available);
        storage.total = storage.total.checked_sub(to_withdraw).near_expect(
            StorageError::NotEnoughStorageBalance {
                requested: to_withdraw,
                available: storage.total,
            },
        );
        storage.available = storage.available.checked_sub(to_withdraw).near_expect(
            StorageError::NotEnoughStorageBalance {
                requested: to_withdraw,
                available: storage.available,
            },
        );

        self.accounts_balances.insert(&account_id, &storage);

        Promise::new(account_id).transfer(to_withdraw).detach();

        storage
    }
