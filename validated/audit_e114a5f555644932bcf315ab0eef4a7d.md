### Title
`respond_verify_foreign_tx` Accepts Any Valid Root-Key Signature Regardless of Request Binding — Cross-Request Replay Delivers Forged Foreign-Chain Verification - (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_verify_foreign_tx` verifies only that the submitted signature is cryptographically valid over the submitted `payload_hash` using the MPC root public key. It does **not** verify that `payload_hash` is the correct hash of `ForeignTxSignPayload { request, values }` for the specific pending `request`. A single malicious attested participant (strictly below the signing threshold) can replay any previously-produced root-key signature — obtained from a completely different `verify_foreign_transaction` request — to resolve an unrelated pending request with a fabricated response.

---

### Finding Description

In `respond_verify_foreign_tx`, the signature check is:

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;
// Check the signature is correct against the root public key
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,   // ← ROOT key, no tweak
)
.is_ok()
``` [1](#0-0) 

The contract then immediately resolves all pending yields keyed on `request`:

```rust
pending_requests::resolve_yields_for(
    &mut self.pending_verify_foreign_tx_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
``` [2](#0-1) 

There is **no check** that `response.payload_hash` equals `SHA-256(borsh(ForeignTxSignPayload { request, values }))` for the supplied `request`. The contract accepts any `(payload_hash, signature)` pair where the signature is a valid root-key ECDSA signature over `payload_hash`, regardless of which foreign-chain transaction that hash actually describes.

This is structurally different from `respond` (sign requests), where the signature is verified against a **derived** key that is cryptographically bound to the specific `(predecessor_id, path)` pair:

```rust
let expected_public_key =
    derive_key_secp256k1(&affine, &request.tweak).map_err(RespondError::from)?;
``` [3](#0-2) 

For sign requests the tweak binds the signature to the exact request; for foreign-tx requests no such binding exists.

The `VerifyForeignTransactionRequest` struct contains no caller identity field:

```rust
pub struct VerifyForeignTransactionRequest {
    pub request: ForeignChainRpcRequest,
    pub domain_id: DomainId,
    pub payload_version: ForeignTxPayloadVersion,
}
``` [4](#0-3) 

And `args_into_verify_foreign_tx_request` confirms that `predecessor_id` is silently dropped when converting user arguments into the map key:

```rust
pub fn args_into_verify_foreign_tx_request(
    args: dtos::VerifyForeignTransactionRequestArgs,
) -> dtos::VerifyForeignTransactionRequest {
    dtos::VerifyForeignTransactionRequest {
        domain_id: args.domain_id,
        request: args.request,
        payload_version: args.payload_version,
    }
}
``` [5](#0-4) 

---

### Impact Explanation

A single malicious attested participant can resolve any pending `verify_foreign_transaction` request with a fabricated `(payload_hash, signature)` pair sourced from a completely different foreign-chain transaction. The MPC contract delivers this forged response to every caller queued under that request key. Downstream bridge contracts that do not independently re-derive and compare `payload_hash` against their expected `(request, values)` tuple will accept the forged attestation as proof that a specific foreign-chain transaction finalized, enabling invalid bridge execution (e.g., releasing bridged assets for a transaction that never occurred or occurred on a different chain/with different parameters).

This matches the allowed impact: **High — forged foreign-chain verification that causes invalid bridge execution**.

---

### Likelihood Explanation

The attacker must be a single attested MPC participant (below the signing threshold). No threshold cooperation is required for the attack itself. The attacker only needs one legitimately-produced root-key signature, which is publicly observable on-chain from any prior `respond_verify_foreign_tx` call. The attack is executable by any one of the `n` participants at any time a target request is pending, with no special tooling beyond a NEAR transaction submission.

---

### Recommendation

Bind the signature to the specific request by having the MPC nodes sign a payload that commits to the `request` fields the contract already knows. Two concrete options:
