Audit Report

## Title
Donation-Based rsETH Price Inflation Enables Direct Theft of Depositor Funds — (`contracts/LRTDepositPool.sol` / `contracts/LRTOracle.sol`)

## Summary
An attacker who is the first depositor can send ETH directly to `LRTDepositPool` via its unrestricted `receive()` function, then call the public `updateRSETHPrice()` to inflate `rsETHPrice` to an arbitrarily large value. Because `pricePercentageLimit` is never initialized and defaults to `0`, the price-cap check is a no-op. Subsequent depositors who pass `minRSETHAmountExpected = 0` receive 0 rsETH while their ETH is absorbed into the protocol, which the attacker — holding the only rsETH in existence — can then fully redeem.

## Finding Description

**Root cause chain:**

1. `LRTDepositPool.receive()` accepts ETH from any caller with no accounting: [1](#0-0) 

2. `getETHDistributionData()` reports the raw contract balance as protocol-owned ETH, so donated ETH is indistinguishable from deposited ETH: [2](#0-1) 

3. `_getTotalEthInProtocol()` in `LRTOracle` calls `getTotalAssetDeposits(ETH_TOKEN)`, which routes through `getAssetDistributionData` → `getETHDistributionData()`, pulling in the inflated balance: [3](#0-2) [4](#0-3) [5](#0-4) 

4. `updateRSETHPrice()` is publicly callable with no access restriction: [6](#0-5) 

5. The price-cap guard is a no-op because `pricePercentageLimit` is never set in `initialize` and defaults to `0`: [7](#0-6) [8](#0-7) 

6. `_beforeDeposit` only reverts if `rsethAmountToMint < minRSETHAmountExpected`; when the victim passes `0`, minting 0 rsETH satisfies the check silently: [9](#0-8) 

**Exploit flow:**

- Attacker deposits 1 wei ETH → receives 1 wei rsETH (price bootstraps to `1e18` when supply is 0).
- Attacker sends X ETH directly to `LRTDepositPool` via `receive()`.
- Attacker calls `lrtOracle.updateRSETHPrice()`. With `rsethSupply = 1` and `totalETHInProtocol = 1 + X`, the new price is `(1 + X) * 1e18`. The `pricePercentageLimit == 0` guard does not fire.
- Victim calls `depositETH{value: Y}(0, "")`. `rsethAmountToMint = Y * 1e18 / ((1+X)*1e18) = 0` (integer truncation). 0 rsETH is minted; Y ETH is accepted.
- Attacker redeems their 1 wei rsETH — the only rsETH in existence — and recovers the entire protocol balance `1 + X + Y`.

## Impact Explanation
**Critical — direct theft of user funds.** Any depositor calling `depositETH` or `depositAsset` with `minRSETHAmountExpected = 0` after the price inflation loses 100% of their deposit. The attacker recovers the full protocol ETH balance including all stolen deposits. This matches the "Direct theft of any user funds" critical impact class.

## Likelihood Explanation
**Medium.** The attack requires: (1) being the first depositor — achievable at launch or after a full withdrawal period; (2) `pricePercentageLimit == 0` — the **default** state since `initialize` never sets it; (3) a victim passing `minRSETHAmountExpected = 0` — common in direct contract calls and integrations that omit slippage protection. No front-running is required; the attacker inflates the price and waits. The attacker must commit capital approximately equal to the victim's deposit, which is standard inflation-attack cost.

## Recommendation

1. **Revert on zero rsETH output.** Add a check in `_beforeDeposit` or `_mintRsETH`:
   ```solidity
   if (rsethAmountToMint == 0) revert ZeroRsETHMinted();
   ```

2. **Initialize `pricePercentageLimit` to a non-zero value** (e.g., `1e16` for 1%) in `LRTOracle.initialize` so that a single large donation cannot inflate the price in one transaction.

3. **Replace `address(this).balance` with internal accounting.** In `getETHDistributionData`, track deposited ETH in a storage variable incremented only on legitimate deposit paths (`depositETH`, `receiveFromNodeDelegator`, etc.), preventing untracked donations from affecting the price oracle.

## Proof of Concept

```
1. Deploy protocol; rsETHPrice = 1e18, rsethSupply = 0.
2. Attacker calls depositETH{value: 1}(0, "") → receives 1 wei rsETH.
3. Attacker sends 15e18 ETH directly to LRTDepositPool (triggers receive()).
4. Attacker calls lrtOracle.updateRSETHPrice().
   → totalETHInProtocol = 1 + 15e18 (via address(this).balance)
   → rsETHPrice = (1 + 15e18) * 1e18 / 1 ≈ 15e36
   → pricePercentageLimit == 0, so no revert.
5. Victim calls depositETH{value: 10e18}(0, "").
   → rsethAmountToMint = 10e18 * 1e18 / 15e36 = 0
   → 0 rsETH minted; 10e18 ETH absorbed.
6. Attacker redeems 1 wei rsETH through withdrawal mechanism.
   → Recovers ≈ 25e18 + 1 ETH (donation + victim deposit).
   → Net profit ≈ 10e18 ETH.
```

Foundry test plan: deploy `LRTDepositPool` + `LRTOracle` with `pricePercentageLimit = 0` (default), execute steps 2–6 above, assert victim's rsETH balance is 0 and attacker's recovered ETH equals `1 + 15e18 + 10e18`.
