Audit Report

## Title
Missing Staleness Check in `CrossChainRateReceiver.getRate()` Allows Over-Minting of wrsETH — (`contracts/cross-chain/CrossChainRateReceiver.sol`)

## Summary

`CrossChainRateReceiver` stores `lastUpdated` on every LayerZero rate delivery but `getRate()` returns `rate` unconditionally without consulting it. All L2 pool contracts (`RSETHPoolV2NBA`, `RSETHPoolV2`, `RSETHPoolV3`, etc.) use this rate directly to compute wrsETH mint amounts. If the off-chain bot or LayerZero delivery stalls, a depositor can exploit the stale (lower) rate to receive more wrsETH per ETH than the current backing justifies, permanently diluting existing holders' yield.

## Finding Description

`CrossChainRateReceiver` records both `rate` and `lastUpdated` on every `lzReceive` call: [1](#0-0) 

However, `getRate()` returns `rate` with no reference to `lastUpdated`: [2](#0-1) 

Every L2 pool delegates pricing entirely to this call. `RSETHPoolV2NBA.getRate()`: [3](#0-2) 

The returned rate is used directly to compute the wrsETH mint amount: [4](#0-3) 

The same pattern is present in `RSETHPoolV3.viewSwapRsETHAmountAndFee()`: [5](#0-4) 

The rate is pushed from mainnet by an off-chain bot calling `CrossChainRateProvider.updateRate()`, which sends a LayerZero message: [6](#0-5) 

`RSETHRateReceiver` is the concrete production implementation deployed across Arbitrum, Optimism, Polygon zkEVM, Blast, Mode, and Scroll: [7](#0-6) 

The protocol already applies a staleness guard in `ChainlinkOracleForRSETHPoolCollateral.getRate()` (reverting on `answeredInRound < roundID`), confirming awareness of the pattern — but it was not applied to `CrossChainRateReceiver`: [8](#0-7) 

## Impact Explanation

rsETH accrues staking yield continuously, so its ETH value increases over time. The L2 pool rate only reflects this when the off-chain bot successfully delivers a LayerZero message. When the stored rate is stale and lower than the true current value, the formula `rsETHAmount = amountAfterFee * 1e18 / staleRate` mints more wrsETH per ETH than the actual backing justifies. This over-minting permanently dilutes the wrsETH/ETH backing ratio for all existing holders. The attacker retains the excess wrsETH once the rate is updated, having extracted accrued yield from prior depositors.

**Impact: High — Theft of unclaimed yield from existing wrsETH holders.**

## Likelihood Explanation

The rate update path has two off-chain dependencies: (1) a bot must call `updateRate()` on the mainnet provider, and (2) LayerZero must deliver the message to the L2 receiver. Either can fail due to bot bugs, gas exhaustion, network congestion, or LayerZero infrastructure issues. rsETH yield accrues every Ethereum epoch (~6.4 minutes), so even a few hours of staleness creates a measurable exploitable gap. An attacker can monitor `CrossChainRateReceiver.lastUpdated` on-chain and act whenever the gap between the stale rate and the live mainnet `LRTOracle.rsETHPrice` is profitable. The attack requires no special privileges — only a standard `deposit()` call.

## Recommendation

Add a configurable `maxStaleness` threshold to `CrossChainRateReceiver` and revert in `getRate()` if `block.timestamp - lastUpdated > maxStaleness`:

```solidity
uint256 public maxStaleness; // e.g. 24 hours

error StaleRate();

function getRate() external view returns (uint256) {
    if (block.timestamp - lastUpdated > maxStaleness) revert StaleRate();
    return rate;
}
```

This mirrors the staleness guard already present in `ChainlinkOracleForRSETHPoolCollateral.getRate()`.

## Proof of Concept

1. The current mainnet `LRTOracle.rsETHPrice` is `1.05e18` (updated regularly on L1).
2. The off-chain bot fails to call `CrossChainRateProvider.updateRate()` for 12 hours. `CrossChainRateReceiver.rate` on Arbitrum remains at `1.03e18` (stale).
3. Attacker calls `RSETHPoolV2NBA.deposit{value: 100 ether}("")`.
4. `viewSwapRsETHAmountAndFee(100e18)` computes: `rsETHAmount = 100e18 * 1e18 / 1.03e18 ≈ 97.09 wrsETH`.
5. Correct amount at live rate: `100e18 * 1e18 / 1.05e18 ≈ 95.24 wrsETH`.
6. Attacker receives `≈1.85 excess wrsETH` (≈1.9% over-mint) at the expense of existing holders.
7. When the bot resumes and the rate updates to `1.05e18`, the attacker's wrsETH is fully backed at the correct rate, having extracted yield from prior depositors.

**Foundry fork test plan:** Fork an L2 (e.g., Arbitrum) at a block where `RSETHRateReceiver.lastUpdated` is recent. Warp `block.timestamp` forward by 24+ hours without triggering `lzReceive`. Call `RSETHPoolV2NBA.deposit{value: 100 ether}("")` and assert that `wrsETH.balanceOf(attacker)` exceeds the amount computed using the current mainnet `LRTOracle.rsETHPrice`, confirming over-minting.
