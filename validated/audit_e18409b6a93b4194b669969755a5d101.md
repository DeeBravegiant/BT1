### Title
Missing Early-Return on `spent_height=None` Causes `AssertionError` Crash in `coins_of_interest_farmed` — (`chia/wallet/trade_manager.py`)

---

### Summary

`TradeManager.coins_of_interest_farmed()` logs an error when `coin_state.spent_height is None` but does **not** return or continue. If the wallet has no coins in its local coin store matching the offer's removals (e.g., the triggered coin is a counterparty coin in a `PENDING_CONFIRM` trade), the function reaches `assert height is not None` with `height = None`, raising an `AssertionError` and crashing the wallet's trade-processing coroutine.

---

### Finding Description

The guard at line 156–157 is incomplete:

```python
if coin_state.spent_height is None:
    self.log.error(f"Coin: {coin_state.coin}, has not been spent so trade can remain valid")
# execution falls through — no return/continue
``` [1](#0-0) 

After the log, the function continues to build `our_primary_coins` from the wallet's coin store. For a `PENDING_CONFIRM` trade (the wallet accepted someone else's offer), the counterparty's offer coins are tracked in `coins_of_interest` but are **not** in the wallet's own coin store. So:

- `our_primary_coins = []` (no matching records)
- `our_additions = []` (filter against empty set)
- `our_addition_ids = []` [2](#0-1) 

`get_coin_state([])` returns `[]`, so `coin_state_names = []`. The equality check:

```python
if set(our_addition_ids) == set(coin_state_names):  # set([]) == set([]) → True
    height = coin_state.spent_height               # None
    assert height is not None                      # AssertionError
``` [3](#0-2) 

---

### Impact Explanation

An `AssertionError` is raised inside the trade-processing coroutine. This crashes the coroutine handling that trade update, preventing the wallet from correctly processing any subsequent coin-state updates for that trade. A malicious peer can repeatedly trigger this for every coin of interest the wallet subscribes to, permanently disrupting trade settlement for the victim wallet. This maps to **High: Corruption of offer/trade settlement state with direct security impact** and **High: Long-lived inability for honest wallets to process valid trade updates**.

---

### Likelihood Explanation

The wallet subscribes to all coins of interest (including counterparty coins) via `get_coins_of_interest()`, which covers `PENDING_ACCEPT`, `PENDING_CONFIRM`, and `PENDING_CANCEL` trades. A connected full-node peer can send a `CoinState` update for any subscribed coin with `spent_height=None` — this is a normal protocol message. No special privileges are required; any peer the wallet connects to can send this message. [4](#0-3) 

---

### Recommendation

Add a `continue` (or `return`) immediately after the error log when `spent_height is None`:

```python
if coin_state.spent_height is None:
    self.log.error(f"Coin: {coin_state.coin}, has not been spent so trade can remain valid")
    continue  # <-- add this
```

This matches the documented intent of the function: a coin that has not been spent cannot indicate trade completion or failure.

---

### Proof of Concept

**Preconditions:**
- Wallet has a `PENDING_CONFIRM` trade (accepted a counterparty's offer). The counterparty's offer coin `C` is in `trade.coins_of_interest` but not in the wallet's coin store.

**Steps:**
1. Malicious peer sends `CoinState(coin=C, created_height=H, spent_height=None)` to the wallet node for coin `C`.
2. Wallet node calls `coins_of_interest_farmed(coin_state, fork_height, peer)`.
3. `get_trades_by_coin(C)` returns the `PENDING_CONFIRM` trade.
4. `coin_state.spent_height is None` → error logged, execution continues.
5. `our_primary_coins = []` (C not in wallet coin store).
6. `our_addition_ids = []`.
7. `get_coin_state([]) → []`; `coin_state_names = []`.
8. `set([]) == set([])` → `True` → enters the "success" branch.
9. `height = None` → `assert height is not None` → **`AssertionError`**.
