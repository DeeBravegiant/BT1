### Title
Threshold-Lowering Resharing Retroactively Promotes Sub-Threshold TEE Verifier Votes to a Win — (`crates/contract/src/tee/verifier_votes.rs`)

### Summary

`TeeVerifierVotes::vote` evaluates ALL pending votes against the threshold in effect at the moment of the triggering call. There is no record of which threshold was in effect when each vote was cast. A resharing that lowers the governance threshold can therefore retroactively cause a previously sub-threshold vote bucket to cross the new threshold on the very next `vote_tee_verifier_change` call, installing a malicious TEE verifier with fewer votes than the old threshold required.

### Finding Description

`TeeVerifierVotes::vote` reads the current threshold and participant set from the `ThresholdParameters` argument at call time: [1](#0-0) 

```rust
let protocol_threshold = threshold_parameters.threshold().value();
let participants = threshold_parameters.participants();
...
let count_usize = {
    let voter_set = self.pending.vote(participant, proposal_hash);
    voter_set.count_for(|p| participants.is_participant_given_participant_id(&p.get()))
};
...
if count >= protocol_threshold {
    self.pending.clear();
    Ok(Some(proposal.candidate_account_id))
}
```

The `count_for` predicate correctly filters out dropped participants, but it does **not** filter out votes that were cast under a higher threshold. All votes from participants who remain in the new set are counted against the new (lower) threshold.

The caller (`vote_tee_verifier_change`) always passes the **current** `ThresholdParameters` from `protocol_state`: [2](#0-1) 

After a resharing completes, `protocol_state` holds the new (lower) threshold. The post-resharing cleanup (`remove_non_participant_tee_verifier_votes`) only removes votes from accounts that are **no longer participants**; it does not clear votes from participants who remain: [3](#0-2) 

The governance threshold has a 60%-rounded-up floor: [4](#0-3) 

This floor still permits concrete threshold drops (e.g., n=5, T=4 → n=4, T=3).

### Impact Explanation

A malicious TEE verifier, once installed, is trusted by `mpc-contract` for all subsequent `submit_participant_info` calls. It can forge attestations, admit unauthorized nodes, and ultimately enable unauthorized threshold signing or key-share exposure. This is a Critical impact: unauthorized TEE verifier installation enables forged attestations and unauthorized signing capability.

### Likelihood Explanation

The attack requires:
1. **T-1 Byzantine participants** (strictly below the old threshold T, so they cannot sign alone).
2. **A threshold-lowering resharing** that keeps all T-1 Byzantine participants in the new set. The resharing itself must be approved by the old threshold T, so at least one honest participant must vote for it — but they may do so legitimately (e.g., removing a non-responsive participant), unaware of the pre-staged verifier votes.
3. **One more `vote_tee_verifier_change` call** after resharing (by any participant, including one of the Byzantine ones re-voting idempotently).

This is a realistic multi-step attack requiring sub-threshold Byzantine collusion plus an opportunistic resharing. Likelihood is Low-Medium but the impact is Critical.

### Recommendation

Enforce threshold consistency at vote-evaluation time. Two concrete options:

1. **Clear all pending verifier votes on every resharing**, regardless of whether participants were dropped. This is the simplest fix: add `self.tee_verifier_votes.pending.clear()` (or expose a `clear` method) in the resharing completion path, alongside the existing `retain` call.

2. **Stamp each vote with the threshold epoch** (e.g., a resharing counter or the threshold value at cast time) and reject votes cast under a different epoch when evaluating the count. This is more complex but preserves votes across resharing when the threshold does not change.

Option 1 is strongly preferred for simplicity and correctness.

### Proof of Concept

Concrete scenario with n=5, T=4:

1. 5 participants, governance threshold T=4. Three Byzantine participants (P1, P2, P3) call `vote_tee_verifier_change(malicious_verifier, hash)`. Count = 3 < 4 → no winner yet. Votes are stored in `tee_verifier_votes.pending`.

2. An honest participant (P4) proposes and votes for a resharing that removes P5 (non-responsive) and lowers the threshold to 3. The Byzantine participants also vote for this resharing (they need 4 votes total; P1+P2+P3+P4 = 4 ✓). Resharing completes: new set = {P1,P2,P3,P4}, new threshold = 3.

3. `remove_non_participant_tee_verifier_votes` runs. P1, P2, P3 are all still participants → no votes removed. The 3 pre-staged votes remain in `pending`.

4. P1 calls `vote_tee_verifier_change(malicious_verifier, hash)` again (re-vote). Inside `TeeVerifierVotes::vote`:
   - `protocol_threshold` = 3 (new threshold)
   - `count_for` counts P1, P2, P3 → 3 votes
   - 3 >= 3 → `self.pending.clear()`, returns `Some(malicious_verifier)`

5. `tee_verifier_account_id` is set to `malicious_verifier`. The malicious verifier is now trusted for all future attestation verification, with only 3 votes that were originally cast under a threshold of 4. [5](#0-4) [6](#0-5) [4](#0-3)

    pub fn vote(
        &mut self,
        proposal: VerifierChangeProposal,
        participant: AuthenticatedParticipantId,
        threshold_parameters: &ThresholdParameters,
    ) -> Result<Option<AccountId>, Error> {
        let protocol_threshold = threshold_parameters.threshold().value();
        let participants = threshold_parameters.participants();
        let proposal_hash: ProposalHash = proposal.clone().into();

        let count_usize = {
            let voter_set = self.pending.vote(participant, proposal_hash);
            voter_set.count_for(|p| participants.is_participant_given_participant_id(&p.get()))
        };
        let count = u64::try_from(count_usize).map_err(|e| ConversionError::DataConversion {
            reason: format!("vote count {count_usize} does not fit in u64: {e}"),
        })?;

        if count >= protocol_threshold {
            self.pending.clear();
            Ok(Some(proposal.candidate_account_id))
        } else {
            Ok(None)
        }
    }
