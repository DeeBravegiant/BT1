Audit Report

## Title
Permissionless `sendFunds()` with No Balance Threshold Enables Block Stuffing to Temporarily Delay MEV Reward Forwarding — (`contracts/FeeReceiver.sol`)

## Summary
`FeeReceiver.sendFunds()` carries no access control and no minimum balance guard, making it blockable by any attacker willing to fill blocks with high-gas-price transactions. Because `FeeReceiver` balance is explicitly excluded from TVL accounting until forwarded, a sustained block-stuffing window causes accumulated MEV rewards to sit idle and the rsETH price to be temporarily understated relative to actual protocol holdings.

## Finding Description
`FeeReceiver.sendFunds()` is fully permissionless with no modifier and no balance check: [1](#0-0) 

ETH accumulates passively via the `receive()` fallback: [2](#0-1) 

The downstream `receiveFromRewardReceiver()` in `LRTDepositPool` is equally permissionless: [3](#0-2) 

Critically, `getETHDistributionData()` — the function that feeds TVL accounting — explicitly excludes the `FeeReceiver` balance by design, only accounting for it after it has been forwarded to the deposit pool: [4](#0-3) 

`_getTotalEthInProtocol()` in `LRTOracle` calls `getTotalAssetDeposits()` → `getETHDistributionData()`, so any ETH sitting in `FeeReceiver` is invisible to the oracle: [5](#0-4) 

Exploit path: attacker submits high-gas-price transactions filling the block gas limit for N consecutive blocks → `sendFunds()` cannot land → MEV rewards accumulate in `FeeReceiver` → `_getTotalEthInProtocol()` understates TVL → `rsETHPrice` is computed lower than actual → `updateRSETHPrice()` reflects a depressed price for the duration of the stuffing window.

No existing guard prevents this: there is no access control on `sendFunds()`, no minimum balance threshold, no time-lock, and no on-chain fallback path to forward rewards if the keeper transaction is crowded out.

## Impact Explanation
**Low — Block stuffing.** This matches the explicitly allowed impact class. The rsETH price is temporarily understated during the stuffing window, but no user funds are permanently lost and the state self-corrects once `sendFunds()` is called after the attack ends. The impact is bounded to the duration of the stuffing window and the size of accumulated MEV rewards.

## Likelihood Explanation
Block stuffing on Ethereum mainnet requires the attacker to pay `base_fee × block_gas_limit` per block, which is economically expensive with no direct financial return. The attacker gains nothing beyond delaying reward forwarding. However, the preconditions are permanently present in the deployed contract (permissionless function, no balance floor, no time enforcement), and the attack requires no privileged access — any externally-owned account can execute it. Likelihood is low but non-zero, particularly during high-MEV events where a well-funded adversary might benefit from a temporarily depressed rsETH price in secondary markets.

## Recommendation
1. **Add access control** to `sendFunds()` (e.g., `onlyRole(LRTConstants.MANAGER)` or a dedicated keeper role) to eliminate the permissionless attack surface. This requires an active keeper but removes the block-stuffing vector entirely.
2. **Add a minimum balance threshold**: revert or no-op in `sendFunds()` when `address(this).balance` is below a configurable dust threshold, reducing zero-value griefing surface.
3. Alternatively, include `FeeReceiver.balance` directly in `getETHDistributionData()` so rewards are always reflected in TVL regardless of whether `sendFunds()` has been called, eliminating the oracle impact.

## Proof of Concept
```solidity
// Foundry invariant/fork test sketch
function testBlockStuffingInvariant() public {
    // 1. Simulate MEV rewards accumulating in FeeReceiver
    vm.deal(address(feeReceiver), 10 ether);

    // 2. Simulate N blocks passing without sendFunds() being called
    //    (block stuffing prevents keeper tx from landing)
    vm.roll(block.number + 50);

    // 3. FeeReceiver still holds the balance — rewards not forwarded
    assertEq(address(feeReceiver).balance, 10 ether);

    // 4. depositPool received nothing — TVL understated for 50 blocks
    //    getETHDistributionData() excludes feeReceiver.balance by design
    assertEq(address(depositPool).balance, 0);

    // 5. After stuffing ends, anyone can call sendFunds() to restore state
    feeReceiver.sendFunds();
    assertEq(address(feeReceiver).balance, 0);
}
```

The test confirms that for any N-block window where `sendFunds()` is not called, rewards remain in `FeeReceiver` and are excluded from `_getTotalEthInProtocol()`, causing a temporarily understated rsETH price. The `getETHDistributionData()` comment at line 465–466 explicitly documents this accounting gap.

        address lrtDepositPoolAddr = lrtConfig.getContract(LRTConstants.LRT_DEPOSIT_POOL);
        address[] memory supportedAssets = lrtConfig.getSupportedAssetList();
        uint256 supportedAssetCount = supportedAssets.length;

            // assetER is in 1e18 precision (1.0 = 1e18)

            // totalAssetAmt is in 1e18 precision (standard token decimals)

            unchecked {
                ++assetIdx;
            }
        }
    }
