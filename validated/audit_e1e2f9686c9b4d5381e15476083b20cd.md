### Title
Attached Deposit Silently Lost on Early Return in `rlp_execute` — (File: runtime/near-wallet-contract/implementation/wallet-contract/src/lib.rs)

### Summary

The `rlp_execute` entry point of the wallet contract is `#[payable]` and accepts an `attached_deposit` from external callers. When the function returns early with an error value (rather than panicking), the NEAR runtime does **not** auto-refund the deposit — it stays in the wallet contract's balance. Multiple early-return paths in `rlp_execute` and its callbacks fail to refund the caller's deposit, causing permanent loss of the attached NEAR tokens.

### Finding Description

The `rlp_execute` function has two early-return paths that silently absorb the caller's deposit:

**Path 1 — `has_in_flight_tx` guard:** [1](#0-0) 

When `has_in_flight_tx` is `true`, the function returns `PromiseOrValue::Value(...)` immediately. No refund is issued. Any `attached_deposit` is absorbed into the wallet contract's balance.

**Path 2 — User/AccountId error from `inner_rlp_execute`:** [2](#0-1) 

When `inner_rlp_execute` returns `Err(e)` for user errors (`ValueTooLarge`, `ExcessYoctoNear`, `UnknownPublicKeyKind`, `EvmDeployDisallowed`, etc.) or `AccountId` errors, the function returns `PromiseOrValue::Value(e.into())` without refunding the deposit.

The `CallerDeposit` mechanism is constructed inside `inner_rlp_execute` and is only passed to the callback promise: [3](#0-2) 

The callback `rlp_execute_callback` correctly refunds the deposit **only** when the downstream cross-contract call fails: [4](#0-3) 

But this path is never reached when `rlp_execute` itself returns early. The deposit is already in the contract's balance and is never returned.

**Additional early-return paths in callbacks that also drop `caller_deposit`:**

In `address_check_callback`, when the address registrar lookup finds a registered account and the signer is not the current account: [5](#0-4) 

In `nep_141_storage_balance_callback`, when the `storage_balance_of` call fails or returns unparseable data: [6](#0-5) 

And when the action is not a `FunctionCall` in the `None` branch: [7](#0-6) 

In all these callback cases, `caller_deposit` is silently dropped and the caller's NEAR is not returned.

### Impact Explanation

In NEAR, when a `#[payable]` function returns normally (not via panic), the attached deposit is permanently credited to the contract's balance. The caller loses the full `attached_deposit`. Since the wallet contract is an ETH-implicit account, the absorbed deposit accrues to the wallet owner's balance. A wallet owner can exploit the `has_in_flight_tx` path to systematically drain deposits from any external caller who attaches NEAR to `rlp_execute` calls: keep a transaction in flight, wait for callers to attach deposits, and collect the silently absorbed funds. The corrupted value is the caller's NEAR balance and the wallet contract's NEAR balance (a concrete on-chain balance entry).

### Likelihood Explanation

- The `has_in_flight_tx` path is reachable by any unprivileged external caller who calls `rlp_execute` while a prior transaction is pending — a normal operational condition.
- User-error paths (`ValueTooLarge`, `ExcessYoctoNear`, etc.) are reachable by any caller who submits a malformed Ethereum transaction payload, which is fully attacker-controlled input.
- The callback failure paths are reachable whenever the NEP-141 token contract or address registrar returns an error or unexpected response.

### Recommendation

1. In `rlp_execute`, before any early `return PromiseOrValue::Value(...)`, check `env::attached_deposit()` and issue a refund promise to `env::predecessor_account_id()` if the deposit is non-zero.
2. In `address_check_callback` and `nep_141_storage_balance_callback`, before any early `return PromiseOrValue::Value(...)`, check `caller_deposit` and issue a refund transfer if it is `Some`.
3. Alternatively, restructure the early-error paths to `panic!` (which triggers the NEAR runtime's automatic deposit refund), or use `#[handle_result]` with a panic on error.

### Proof of Concept

1. Deploy the wallet contract on an ETH-implicit account.
2. Initiate a long-running transaction so that `has_in_flight_tx` is set to `true`.
3. From a separate account, call `rlp_execute` with a non-zero `attached_deposit` (e.g., 1 NEAR).
4. The call returns `ExecuteResponse { success: false, error: "transaction already in progress" }`.
5. Observe that the caller's balance decreased by the full deposit amount, and the wallet contract's balance increased by the same amount — the deposit was silently absorbed with no refund.
