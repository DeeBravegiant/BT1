### Title
Missing Cryptographic Output Verification in `respond_ckd` for `AppPublicKey` Variant Allows Single Byzantine Participant to Forge CKD Response - (File: crates/contract/src/lib.rs)

### Summary
The `respond_ckd` contract method applies a cryptographic pairing check only for the `AppPublicKeyPV` variant of CKD requests. For the `AppPublicKey` variant, **no cryptographic verification** of the `CKDResponse` is performed. A single attested participant (Byzantine node below the signing threshold) can call `respond_ckd` with an arbitrary forged `(big_y, big_c)` response, and the contract will accept it and deliver the forged output to the user, constituting unauthorized confidential key derivation output without threshold authorization.

### Finding Description
In `respond_ckd`, after verifying the caller is an attested participant, the contract branches on the request's `app_public_key` variant:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}   // ← NO CHECK
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV`, `ckd_output_check` verifies the pairing equation `e(big_c, g2) = e(big_y, app_pk2) · e(H(pk, app_id), mpc_pk)`, which cryptographically proves the response encodes the correct BLS signature under the network's master secret key. [2](#0-1) 

For `AppPublicKey`, the contract performs **zero** cryptographic verification. The `pending_requests::resolve_yields_for` call immediately resolves the pending yield with whatever `(big_y, big_c)` the caller supplied. [3](#0-2) 

The `respond_ckd` function only requires the caller to be an attested participant — it does not require the caller to be the designated leader for the request, nor does it require any threshold of participants to have agreed on the response. [4](#0-3) 

The CKD protocol specification confirms that the coordinator is supposed to verify the aggregated output before sending it on-chain. In the `AppPublicKeyPV` off-chain protocol, `aggregated_output_check` is called before returning the result: [5](#0-4) 

However, this off-chain check is not enforced by the contract for `AppPublicKey` requests. The contract is the only trust anchor for on-chain state; if the off-chain check is bypassed (e.g., by a Byzantine leader node), the contract provides no backstop.

### Impact Explanation
A single Byzantine attested participant can call `respond_ckd` for any pending `AppPublicKey` CKD request with arbitrary `big_y` and `big_c` values. The contract resolves the pending yield with the forged response. The user receives a `CKDResponse` that does not encode `msk · H(pk, app_id)` — the correct BLS signature under the MPC network's master secret key. The user cannot unmask a valid app private key from this forged output. This constitutes **unauthorized confidential key derivation output without the required threshold participant authorization**: the threshold of honest nodes never agreed on this output, yet it is delivered as if they did.

### Likelihood Explanation
The attacker must be a single attested participant (a TEE node that has passed attestation but is acting maliciously — a Byzantine participant strictly below the signing threshold). This is within the stated threat model. The `AppPublicKey` variant is the default/simpler CKD variant and is expected to be widely used. The attack requires only one malicious call to `respond_ckd` with a matching `CKDRequest` key (which is observable on-chain) and arbitrary response values.

### Recommendation
For the `AppPublicKey` variant, the contract cannot perform the same pairing check as `AppPublicKeyPV` because `pk2` (the G2 component of the app key) is not available. However, the contract should enforce that only the designated leader for a given request can submit the response, or require a threshold of participants to co-sign the response before it is accepted. At minimum, the contract should document that `AppPublicKey` requests rely entirely on the TEE model for output integrity, and the threat model should be updated accordingly. The preferred fix is to require callers to use `AppPublicKeyPV` for any use case where output integrity must be verifiable on-chain.

### Proof of Concept
1. User calls `request_app_private_key` with `CKDAppPublicKey::AppPublicKey(pk1)` and a derivation path. A `CKDRequest` is stored in `pending_ckd_requests`.
2. Attacker (a single attested participant, e.g., `participant_0`) observes the pending request on-chain.
3. Attacker calls `respond_ckd(ckd_request, CKDResponse { big_y: [1u8; 48], big_c: [2u8; 48] })` — arbitrary forged values.
4. The contract passes the `AppPublicKey(_) => {}` branch with no check, then calls `resolve_yields_for` with the forged response.
5. The user's yield is resolved with `big_y = [1u8; 48], big_c = [2u8; 48]` — values that do not encode any valid BLS signature under the MPC network's key.
6. The user receives a `CKDResponse` from which no valid app private key can be derived, and has no on-chain mechanism to detect the forgery.

This is directly analogous to the external report's "Timing" class: a critical check (cryptographic output verification) is absent at the point where the CKD response is accepted, allowing broken/forged outputs to be committed to the contract state without detection.

/// Check that `e(big_c, g2) = e(big_y, app_pk2) . e(hash_point, public_key)`.
///
/// Point validation is fully delegated to the host, as in
/// [`app_public_key_check`].
pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}
