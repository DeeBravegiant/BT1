### Title
Cross-Network Replay of `DelegateAction` Signatures Due to Missing Chain/Network Identifier in Signed Payload - (File: `core/primitives/src/action/delegate.rs`, `core/primitives/src/signable_message.rs`)

### Summary
`DelegateAction` (NEP-366 meta transactions) and `DelegateActionV2` (NEP-611) sign a payload that contains no chain or network identifier. Unlike regular `SignedTransaction`, which binds to a specific chain via `block_hash`, a `SignedDelegateAction` can be replayed by a malicious relayer on a different NEAR network (e.g., mainnet vs. testnet) if the same account and key exist on both networks and the nonce is valid.

### Finding Description

`DelegateAction::get_nep461_hash()` computes the signed hash as:

```
SHA-256( borsh( SignableMessage { discriminant: u32(NEP_366), msg: &DelegateAction } ) )
``` [1](#0-0) 

`SignableMessage` contains only a `MessageDiscriminant` (a 4-byte NEP number) and the message body: [2](#0-1) 

The `DelegateAction` struct itself contains `sender_id`, `receiver_id`, `actions`, `nonce`, `max_block_height`, and `public_key` — **no chain ID, no network ID, no block hash**: [3](#0-2) 

The same omission applies to `VersionedDelegateActionPayload::get_nep461_hash()` used by `DelegateActionV2`: [4](#0-3) 

By contrast, a regular `TransactionV0` includes `block_hash`, which is chain-specific (a block hash from mainnet cannot exist on testnet), providing implicit cross-network replay protection: [5](#0-4) 

`DelegateAction` has only `max_block_height` (a plain integer, not chain-specific) for time-bounding.

At runtime, `apply_delegate_action` calls `signed_delegate_action.verify()`, which only checks the signature against the chain-ID-free hash: [6](#0-5) 

No chain or network identifier is checked anywhere in the delegate action processing path.

### Impact Explanation

A malicious relayer who receives a `SignedDelegateAction` intended for NEAR testnet can submit it verbatim to NEAR mainnet (or any other NEAR network). If the sender's account exists on both networks with the same public key and the nonce is valid on mainnet, the runtime will accept the signature and execute the inner actions — e.g., token transfers, `AddKey`, `DeleteKey`, `FunctionCall` — on the unintended network. The corrupted protocol values are the account state (balance, access keys, contract storage) on the target network.

### Likelihood Explanation

Medium. The preconditions are: (1) the same account ID exists on both networks, (2) the same access key is registered on both, (3) the nonce is valid on the target network, and (4) `max_block_height` has not been exceeded. Conditions (1)–(3) are common for developers and power users who operate on both mainnet and testnet with the same key material. The relayer is the only party who needs to act maliciously or be compromised — an unprivileged external actor who obtains the `SignedDelegateAction` off-chain (e.g., from a public relayer API) can perform the replay.

### Recommendation

Include the NEAR chain/network identifier in the `SignableMessage` payload used to compute the NEP-461 hash for `DelegateAction` and `DelegateActionV2`. The chain ID is available at runtime via `ApplyState` and is already exposed as a host function (`chain_id`). The `MessageDiscriminant` or a new wrapper struct should incorporate the chain ID so that a signature produced for one NEAR network is cryptographically invalid on any other.

### Proof of Concept

1. Alice signs a `DelegateAction` on NEAR testnet: `AddKey` to her account, `nonce=N`, `max_block_height=H`.
2. The signed payload is `SHA-256(borsh(discriminant=0x4000016E || sender_id || receiver_id || actions || N || H || pubkey))` — no network identifier.
3. A malicious relayer intercepts the `SignedDelegateAction` and submits it inside a transaction on NEAR mainnet.
4. `apply_delegate_action` calls `signed_delegate_action.verify()`, which recomputes the identical hash (same borsh bytes, same discriminant) and confirms the signature is valid.
5. If Alice's account exists on mainnet with the same key and nonce `N-1`, the `AddKey` action executes on mainnet — an action Alice never authorized for mainnet. [7](#0-6) [8](#0-7)

impl SignedDelegateAction {
    pub fn verify(&self) -> bool {
        let delegate_action = &self.delegate_action;
        let hash = delegate_action.get_nep461_hash();
        let public_key = &delegate_action.public_key;

        self.signature.verify(hash.as_ref(), public_key)
    }
