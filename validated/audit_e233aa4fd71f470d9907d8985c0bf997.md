### Title
User-Controlled Finality Parameters in `verify_foreign_transaction` Enable Signing of Unfinalized Foreign-Chain Transactions — (`crates/node/src/providers/verify_foreign_tx/sign.rs`)

### Summary

The `verify_foreign_transaction` flow accepts user-supplied finality parameters (`BlockConfirmations` for Bitcoin, `EvmFinality` for EVM chains) with no contract-level or node-level minimum floor. An unprivileged caller can set `confirmations: 0` (Bitcoin) or `finality: Latest` (EVM) and cause the MPC network to produce a threshold signature attesting to a transaction that has not finalized on the foreign chain, enabling double-spend or bridge-drain attacks.

### Finding Description

The `verify_foreign_transaction` contract method accepts a `VerifyForeignTransactionRequestArgs` that includes a `ForeignChainRpcRequest`. For Bitcoin, this contains a `confirmations: BlockConfirmations` field described as "required confirmations before considering final." For EVM chains, it contains `finality: EvmFinality` which can be `Latest`, `Safe`, or `Finalized`.

Neither the contract nor the node enforces any minimum on these values. The contract's `verify_foreign_transaction` only checks that the chain is whitelisted and the domain exists: [1](#0-0) 

The node's `execute_foreign_chain_request` passes the user-supplied confirmation count directly to the Bitcoin inspector without any floor check: [2](#0-1) 

For EVM chains, the user-supplied `finality` is passed directly as well: [3](#0-2) 

The `BitcoinRpcRequest` struct exposes `confirmations` as a plain user-controlled field: [4](#0-3) 

The analog to the original report is exact: just as `buyFlanAndBurn` executes swaps with no minimum accepted price (allowing the attacker to control effective slippage to zero), `verify_foreign_transaction` executes foreign-chain verification with no minimum finality floor (allowing the attacker to control effective confirmation depth to zero).

### Impact Explanation

An attacker who submits `verify_foreign_transaction` with `confirmations: 0` (Bitcoin) or `finality: Latest` (EVM) receives a valid MPC threshold signature over an unfinalized transaction. In a bridge context (the stated primary use case — Omnibridge inbound flow), the attacker can:

1. Broadcast a Bitcoin/EVM transaction sending funds to the bridge deposit address.
2. Immediately call `verify_foreign_transaction` with `confirmations: 0`.
3. Receive a valid MPC signature attesting to the transaction.
4. Claim bridge funds on NEAR using the signature.
5. Replace or reorganize the original foreign-chain transaction (via RBF on Bitcoin, or chain reorg on EVM `Latest`).

This constitutes a **forged foreign-chain verification** causing **invalid bridge execution and double-spend conditions**, matching the allowed High impact scope.

### Likelihood Explanation

- The entry point is fully unprivileged: any NEAR account can call `verify_foreign_transaction` with 1 yoctoNEAR deposit.
- Setting `confirmations: 0` or `finality: Latest` requires no special knowledge or resources.
- Bitcoin RBF (Replace-By-Fee) is a standard mempool feature; `Latest` EVM blocks are routinely reorganized on chains like Polygon.
- The Omnibridge use case is explicitly documented as the primary consumer of this API, making exploitation directly profitable.

### Recommendation

1. **Enforce a minimum confirmation floor in the contract**: Add a governance-controlled `min_confirmations` per chain to `ForeignChainsMetadata` and reject requests below the floor in `verify_foreign_transaction`.
2. **Enforce a minimum finality level for EVM chains**: Reject `EvmFinality::Latest` for bridge-critical operations; require at least `Safe` or `Finalized`.
3. **Node-side defense-in-depth**: Even if the contract enforces minimums, nodes should independently refuse to sign for confirmation counts or finality levels below a locally configured floor.

### Proof of Concept

```
// Attacker submits with zero confirmations
verify_foreign_transaction({
    domain_id: <foreign_tx_domain>,
    payload_version: V1,
    request: Bitcoin({
        tx_id: <attacker_deposit_tx>,
        confirmations: BlockConfirmations(0),  // no floor enforced
        extractors: [BlockHash],
    })
})
// → MPC nodes call inspector.extract(tx_id, 0, ...) 
// → Transaction passes (0 >= 0 confirmations required)
// → MPC produces threshold signature over unfinalized tx
// → Attacker claims NEAR bridge funds
// → Attacker RBF-replaces the Bitcoin deposit tx
```

    pub fn verify_foreign_transaction(&mut self, request: VerifyForeignTransactionRequestArgs) {
        log!(
            "verify_foreign_transaction: predecessor={:?}, request={:?}",
            env::predecessor_account_id(),
            request
        );

        self.check_request_preconditions(
            request.domain_id,
            DomainPurpose::ForeignTx,
            Gas::from_tgas(self.config.sign_call_gas_attachment_requirement_tera_gas),
            MINIMUM_SIGN_REQUEST_DEPOSIT,
        );

        let requested_chain = request.request.chain();
        let supported_chains = self.get_supported_foreign_chains();
        if !supported_chains.contains(&requested_chain) {
            env::panic_str(
                &InvalidParameters::ForeignChainNotSupported {
                    requested: requested_chain,
                }
                .to_string(),
            );
        }

        let callback_gas = Gas::from_tgas(
            self.config
                .return_signature_and_clean_state_on_success_call_tera_gas,
        );

        let request = args_into_verify_foreign_tx_request(request);
        let callback_args = serde_json::to_vec(&(&request,)).unwrap();
        self.enqueue_yield_request(
            method_names::RETURN_VERIFY_FOREIGN_TX_AND_CLEAN_STATE_ON_SUCCESS,
            callback_args,
            callback_gas,
            move |this, id| this.add_verify_foreign_tx_request(request, id),
        );
    }
