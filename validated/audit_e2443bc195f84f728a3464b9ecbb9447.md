Audit Report

## Title
`_checkAndUpdateDailyFeeMintLimit` Lacks Zero-Guard for `maxFeeMintAmountPerDay`, Causing `updateRSETHPrice()` to Revert and Freeze Protocol Fee Yield — (File: `contracts/LRTOracle.sol`)

## Summary

`maxFeeMintAmountPerDay` defaults to `0` in Solidity and is never initialized in either `initialize` or `reinitialize`. When TVL increases and `protocolFeeInBPS > 0`, `_checkAndUpdateDailyFeeMintLimit` is called with a non-zero `feeAmount`, and the strict `>` comparison against `maxFeeMintAmountPerDay == 0` always reverts. Every call to the public, permissionless `updateRSETHPrice()` fails, the rsETH price goes stale, and protocol fee yield accrued during the broken window is not minted to the treasury. The yield is not fully recoverable because new depositors receive inflated rsETH at the stale lower price, diluting the supply and reducing the fee that would be captured in the next successful update.

## Finding Description

**Root cause — missing zero-guard in the write path:**

`remainingDailyFeeMintLimit()` correctly short-circuits when `maxFeeMintAmountPerDay == 0`:

```solidity
// LRTOracle.sol L171
if (maxFeeMintAmountPerDay == 0) return 0;
```

`_checkAndUpdateDailyFeeMintLimit` has no equivalent guard:

```solidity
// LRTOracle.sol L205-207
if (currentPeriodMintedFeeAmount + feeAmount > maxFeeMintAmountPerDay) {
    revert DailyFeeMintLimitExceeded(currentPeriodMintedFeeAmount + feeAmount, maxFeeMintAmountPerDay);
}
```

When `maxFeeMintAmountPerDay == 0` and `feeAmount > 0`, the condition reduces to `feeAmount > 0`, which is always true, so the function always reverts.

**How `feeAmount > 0` is reached:**

```solidity
// LRTOracle.sol L244-247
if (!protocolPaused && totalETHInProtocol > previousTVL) {
    uint256 rewardAmount = totalETHInProtocol - previousTVL;
    protocolFeeInETH = (rewardAmount * lrtConfig.protocolFeeInBPS()) / 10_000;
}
```

Any TVL increase while the protocol is unpaused produces `protocolFeeInETH > 0`, which flows into:

```solidity
// LRTOracle.sol L299-303
if (protocolFeeInETH > 0) {
    uint256 rsethAmountToMintAsProtocolFee = protocolFeeInETH.divWad(newRsETHPrice);
    _checkAndUpdateDailyFeeMintLimit(rsethAmountToMintAsProtocolFee);  // ← reverts
```

**`maxFeeMintAmountPerDay` is never initialized:**

```solidity
// LRTOracle.sol L64-68
function initialize(address lrtConfigAddr) external initializer {
    UtilLib.checkNonZeroAddress(lrtConfigAddr);
    lrtConfig = ILRTConfig(lrtConfigAddr);
    emit UpdatedLRTConfig(lrtConfigAddr);
}
```

```solidity
// LRTOracle.sol L72-79
function reinitialize(uint256 _feePeriodStartTime) external reinitializer(2) onlyLRTManager {
    ...
    feePeriodStartTime = _feePeriodStartTime;
    // maxFeeMintAmountPerDay is NOT set
}
```

It can only be set via the separate `setMaxFeeMintAmountPerDay` call (L132-135), which is not enforced at deployment or upgrade time.

**Why existing checks are insufficient:**

The only guard against `maxFeeMintAmountPerDay == 0` exists in the view function `remainingDailyFeeMintLimit` (L171), which is never called from the write path. The write path `_checkAndUpdateDailyFeeMintLimit` has no such guard.

## Impact Explanation

**Medium — Permanent freezing of unclaimed yield.**

Protocol fee yield (a percentage of all staking rewards, set by `protocolFeeInBPS`) is not minted to the treasury during the broken window. The yield is not fully recoverable: while `rsETHPrice` is stale, new depositors receive rsETH at the artificially low price, inflating `rsethSupply`. When `maxFeeMintAmountPerDay` is eventually set and `updateRSETHPrice()` succeeds, `previousTVL = rsethSupply.mulWad(rsETHPrice)` is computed with the inflated supply, reducing the measured reward amount and the fee minted. The portion of protocol fee yield attributable to the diluted supply is permanently unrecoverable. This matches the allowed impact: **Medium — Permanent freezing of unclaimed yield**.

## Likelihood Explanation

`maxFeeMintAmountPerDay` is absent from both `initialize` and `reinitialize`, so every fresh deployment and every upgrade via `reinitialize` starts in the broken state. The trigger requires no privilege: any external caller invoking the public `updateRSETHPrice()` (L87) after TVL has grown will hit the revert. No malicious intent is needed — a legitimate keeper bot or any user performing a routine price update is sufficient. The condition is met as soon as any staking rewards accrue, which is the normal operating state of the protocol.

## Recommendation

Add a zero-guard in `_checkAndUpdateDailyFeeMintLimit` consistent with the existing guard in `remainingDailyFeeMintLimit`:

```solidity
function _checkAndUpdateDailyFeeMintLimit(uint256 feeAmount) internal {
+   if (maxFeeMintAmountPerDay == 0) return; // limit not configured; skip enforcement
    if (block.timestamp >= feePeriodStartTime + 1 days) { ... }
    ...
}
```

Alternatively, require `maxFeeMintAmountPerDay` to be set to a non-zero value inside `reinitialize` so the invariant is enforced at upgrade time.

## Proof of Concept

1. Deploy `LRTOracle` and call `reinitialize` without subsequently calling `setMaxFeeMintAmountPerDay`. `maxFeeMintAmountPerDay` remains `0`.
2. Users deposit assets into `LRTDepositPool`; staking rewards accrue, increasing `totalETHInProtocol` above `rsethSupply.mulWad(rsETHPrice)`.
3. Any account calls `updateRSETHPrice()` (public, no access control).
4. Inside `_updateRsETHPrice`: `totalETHInProtocol > previousTVL` → `protocolFeeInETH > 0` → `rsethAmountToMintAsProtocolFee > 0`.
5. `_checkAndUpdateDailyFeeMintLimit(rsethAmountToMintAsProtocolFee)` evaluates `0 + rsethAmountToMintAsProtocolFee > 0` → reverts with `DailyFeeMintLimitExceeded`.
6. `updateRSETHPrice()` reverts. `rsETHPrice` is not updated. Protocol fee is not minted. Every subsequent call repeats steps 3–6.
7. New depositors mint rsETH at the stale lower price, inflating supply. When `setMaxFeeMintAmountPerDay` is eventually called and the price update succeeds, the fee is computed against the inflated `previousTVL`, permanently reducing the recoverable protocol fee yield.
