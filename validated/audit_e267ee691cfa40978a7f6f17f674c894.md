Audit Report

## Title
Withdrawal Queue Liveness Failure: `getAvailableAssetAmount` Counts Illiquid EigenLayer Assets While `unlockQueue` Can Only Draw From Liquid Vault Balance - (File: `contracts/LRTWithdrawalManager.sol`)

## Summary
`getAvailableAssetAmount` computes available withdrawal capacity using `lrtDepositPool.getTotalAssetDeposits(asset)`, which includes assets locked in EigenLayer strategies and the EigenLayer withdrawal queue. However, `unlockQueue` sources funds exclusively from `unstakingVault.balanceOf(asset)`. When the vault balance is insufficient to cover a queued request, the user's rsETH is frozen in the contract with no cancellation path until an operator completes a full EigenLayer unwind cycle (≥7 days). Additionally, `_calculatePayoutAmount` caps the payout at `min(expectedAssetAmount, currentReturn)`, so any adverse price movement during the extended wait reduces the user's payout below the originally quoted amount with no recourse.

## Finding Description

**Root cause — liquidity accounting mismatch:**

`getAvailableAssetAmount` (L599–603) calls `getTotalAssetDeposits`, which aggregates all protocol locations including illiquid EigenLayer positions:

```solidity
// LRTWithdrawalManager.sol L599-603
function getAvailableAssetAmount(address asset) public view override returns (uint256 availableAssetAmount) {
    ILRTDepositPool lrtDepositPool = ILRTDepositPool(lrtConfig.getContract(LRTConstants.LRT_DEPOSIT_POOL));
    uint256 totalAssets = lrtDepositPool.getTotalAssetDeposits(asset);
    availableAssetAmount = totalAssets > assetsCommitted[asset] ? totalAssets - assetsCommitted[asset] : 0;
}
```

`getTotalAssetDeposits` (L385–397) includes `assetStakedInEigenLayer + assetUnstakingFromEigenLayer`, both of which are subject to EigenLayer's ~7-day withdrawal delay and are not redeemable on demand.

`initiateWithdrawal` (L170) gates the request against this inflated figure:
```solidity
if (expectedAssetAmount > getAvailableAssetAmount(asset)) revert ExceedAmountToWithdraw();
```
The user's rsETH is transferred to the contract at L166 before this check, and `assetsCommitted[asset]` is incremented at L173.

**`unlockQueue` is constrained to vault balance only:**

`_createUnlockParams` (L846–850) sets `totalAvailableAssets` to `unstakingVault.balanceOf(asset)` — only the liquid vault balance:
```solidity
totalAvailableAssets: unstakingVault.balanceOf(asset)
```
`unlockQueue` immediately reverts if this is zero (L297), and `_unlockWithdrawalRequests` (L800) breaks out of the loop if `availableAssetAmount < payoutAmount`.

**No cancellation path:**

There is no `cancelWithdrawal` function anywhere in `LRTWithdrawalManager`. Once rsETH is transferred in at L166, the user cannot reclaim it. They must wait for an operator to: (a) queue an EigenLayer withdrawal, (b) wait the EigenLayer delay (~7 days), (c) call `completeUnstaking` on `NodeDelegator`, (d) route funds to the vault, and (e) call `unlockQueue`.

**Stale-price execution:**

`_calculatePayoutAmount` (L833–834) computes:
```solidity
uint256 currentReturn = (request.rsETHUnstaked * rsETHPrice) / assetPrice;
return (request.expectedAssetAmount < currentReturn) ? request.expectedAssetAmount : currentReturn;
```
If the rsETH/asset exchange rate declines during the multi-day unwind, `currentReturn < expectedAssetAmount` and the user receives less than the amount quoted at request time, with no slippage protection or cancellation right.

## Impact Explanation

**Medium — Temporary freezing of funds:** A user who calls `initiateWithdrawal` when the vault is underfunded but total protocol TVL (including EigenLayer) is large will have their rsETH locked in `LRTWithdrawalManager` for the full EigenLayer unwind period (≥7 days on top of `withdrawalDelayBlocks` ≈ 8 days), with no ability to cancel. This is a concrete, bounded freeze of user funds caused directly by the liquidity accounting mismatch in this repository.

**Low — Contract fails to deliver promised returns:** The `expectedAssetAmount` stored at request time is the amount the user was quoted. If the oracle price moves adversely during the extended wait, `_calculatePayoutAmount` delivers a lower payout than quoted, with no slippage protection or cancellation right.

## Likelihood Explanation

This is a normal operating condition. The protocol actively moves assets from `LRTDepositPool` to `NodeDelegator` and into EigenLayer strategies. The `LRTUnstakingVault` balance is routinely near zero between unwind cycles. Any user who initiates a withdrawal during such a period — which is the common state — will experience the freeze. No attacker capability is required; any unprivileged user calling `initiateWithdrawal` triggers this. The EigenLayer withdrawal delay is a protocol-level constant (~7 days), making the window for price movement substantial and the freeze duration material.

## Recommendation

1. **Fix the availability check:** `getAvailableAssetAmount` should count only liquid assets (vault balance + deposit pool balance), not EigenLayer-locked assets. This prevents users from queuing withdrawals that cannot be serviced without an EigenLayer unwind.

2. **Add a cancellation path:** Allow users to cancel a pending (not yet unlocked) withdrawal request and reclaim their rsETH, so they are not indefinitely frozen while waiting for operator action.

3. **Positive-slippage protection:** In `_calculatePayoutAmount`, if the price has moved in the user's favor (`currentReturn > expectedAssetAmount`), pay `currentReturn` rather than capping at the original quote.

## Proof of Concept

**Setup:** Protocol has 1000 stETH total: 990 stETH staked in EigenLayer, 10 stETH in `LRTUnstakingVault`, `assetsCommitted[stETH] = 0`.

1. User calls `initiateWithdrawal(stETH, rsETHAmount)` where `expectedAssetAmount = 40 stETH`.
2. `getAvailableAssetAmount(stETH)` calls `getTotalAssetDeposits` → returns 1000 stETH (includes 990 EigenLayer-locked). `1000 - 0 = 1000 ≥ 40` → check passes. rsETH transferred to contract. `assetsCommitted[stETH] = 40`.
3. Operator calls `unlockQueue(stETH, ...)`. `_createUnlockParams` sets `totalAvailableAssets = unstakingVault.balanceOf(stETH) = 10`. `10 < 40` → loop at L800 breaks immediately. `rsETHBurned = 0`, `assetAmountUnlocked = 0`. Nothing is unlocked; user's rsETH remains frozen.
4. Operator must queue EigenLayer withdrawal → wait ≥7 days → `completeUnstaking` → route to vault → call `unlockQueue` again.
5. During those 7 days, stETH/ETH rate drops 2%. At unlock time, `currentReturn = 39.2 stETH < 40 stETH = expectedAssetAmount`. User receives 39.2 stETH instead of the quoted 40 stETH, with no recourse.
