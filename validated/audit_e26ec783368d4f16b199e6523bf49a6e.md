The claim requires careful tracing through the actual call chain. Here is the analysis:

**Call chain:**

1. `WalletStateManager._add_coin_states` receives a `CoinState` from a peer saying the singleton is spent. [1](#0-0) 

2. It calls `fetch_coin_spend_for_coin_state(curr_coin_state, peer)` to retrieve the `CoinSpend`. [2](#0-1) 

3. `fetch_coin_spend` validates that `puzzle.get_tree_hash() == coin.puzzle_hash` and `coin_name` matches — but **does not validate the solution in any way**. [3](#0-2) 

4. The resulting `CoinSpend` (correct puzzle, arbitrary solution) is passed to `apply_state_transition`. [4](#0-3) 

5. `apply_state_transition` only checks coin name continuity (`spent_coin_name == new_state.coin.name()`). It performs **no CLVM execution, no AGG_SIG_ME verification, no puzzle-solution consistency check**. [5](#0-4) 

6. It calls `solution_to_pool_state`, which purely parses the solution bytes to extract `PoolState` (including `pool_url` and `target_puzzle_hash`) with no cryptographic validation. [6](#0-5) 

7. `update_pool_config` then writes whatever `pool_url` and `target_puzzle_hash` were parsed from the solution directly to the `PoolingShareState` config file on disk. [7](#0-6) 

**The puzzle hash constraint does NOT save you here.** The coin's `puzzle_hash` is fixed by the coin_id (which the wallet tracks), so the peer must return the correct puzzle. But the **solution is completely unconstrained** — the peer can return any solution bytes, and the wallet will parse attacker-controlled `pool_url`/`target_puzzle_hash` from it without running the CLVM or verifying the owner's BLS signature.

The on-chain pool member puzzle enforces `AGG_SIG_ME` signed by the owner pubkey for state transitions: [8](#0-7) 

But the wallet never executes the puzzle against the solution — it only parses the solution structure.

### Title
Malicious Peer Can Redirect Pool Farming Rewards via Unvalidated CoinSpend Solution in `apply_state_transition` — (`chia/pools/pool_wallet.py`)

### Summary
A malicious full-node peer can supply a `CoinSpend` with the correct puzzle (matching the on-chain singleton puzzle hash) but a crafted solution encoding attacker-controlled `pool_url` and `target_puzzle_hash`. Because `apply_state_transition` never executes the CLVM puzzle against the solution and never verifies the owner's BLS signature, `update_pool_config` writes the attacker's values to the `PoolingShareState` config file, causing the farmer to submit partials to the attacker's pool server and direct future block rewards to the attacker's address.

### Finding Description
`apply_state_transition` (`chia/pools/pool_wallet.py:261–303`) is documented as the place where "all validation should be done." In practice, it only checks that the incoming `CoinSpend.coin.name()` matches the current singleton tip coin name (line 276). It does not:
- Execute the CLVM puzzle against the provided solution
- Verify the `AGG_SIG_ME` condition signed by the owner pubkey that the on-chain pool puzzles require
- Verify that the solution is consistent with the puzzle reveal

`fetch_coin_spend` (`wallet_sync_utils.py:336–352`) validates the puzzle hash against the on-chain coin record but places no constraint on the solution. A malicious peer returns the correct puzzle and an arbitrary solution. `solution_to_pool_state` (`pool_puzzles.py:399–433`) parses the solution's inner key-value list for the `"p"` key and deserializes it as a `PoolState` — accepting any `pool_url` and `target_puzzle_hash` the attacker encodes. `update_pool_config` (`pool_wallet.py:231–259`) then persists these values to disk unconditionally.

### Impact Explanation
The farmer process reads the `PoolingShareState` config to determine which pool server to submit partials to and which address to direct block rewards to. After a successful attack, the farmer submits partials to the attacker's pool URL and the singleton's `target_puzzle_hash` is set to the attacker's address, redirecting all future farming rewards. This is unauthorized payout redirection and pool membership state corruption — both High-severity impacts per the scope rules.

### Likelihood Explanation
Any peer the wallet connects to (including peers on the public network) can execute this attack. The attacker needs only to: (1) know the singleton's coin ID (public on-chain), (2) send a fake `CoinState` claiming the singleton is spent, and (3) return the correct puzzle with a crafted solution. No key material is required.

### Recommendation
In `apply_state_transition`, before accepting a `CoinSpend`, execute the CLVM puzzle against the provided solution using `run_with_cost` and verify that the resulting conditions include a valid `AGG_SIG_ME` condition signed by the known `owner_pubkey` stored in the wallet's existing pool state. Alternatively, verify the solution against the puzzle hash by running the puzzle and checking that the output conditions are consistent with the claimed state transition. The owner pubkey is already available from the current `PoolWalletInfo`.

        self, new_state: CoinSpend, block_height: uint32, action_scope: WalletActionScope
    ) -> bool:
        """
        Updates the Pool state (including DB) with new singleton spends.
        The DB must be committed after calling this method. All validation should be done here. Returns True iff
        the spend is a valid transition spend for the singleton, False otherwise.
        """
        tip: tuple[uint32, CoinSpend] = await self.get_tip()
        tip_spend = tip[1]

        tip_coin: Coin | None = get_most_recent_singleton_coin_from_coin_spend(tip_spend)
        assert tip_coin is not None
        spent_coin_name: bytes32 = tip_coin.name()

        if spent_coin_name != new_state.coin.name():
            history: list[tuple[uint32, CoinSpend]] = await self.get_spend_history()
            if new_state.coin.name() in [sp.coin.name() for _, sp in history]:
                self.log.info(f"Already have state transition: {new_state.coin.name().hex()}")
            else:
                self.log.warning(
                    f"Failed to apply state transition. tip: {tip_coin} new_state: {new_state} height {block_height}"
                )
            return False

        await self.wallet_state_manager.pool_store.add_spend(self.wallet_id, new_state, block_height)
        tip_spend = (await self.get_tip())[1]
        self.log.info(f"New PoolWallet singleton tip_coin: {tip_spend} farmed at height {block_height}")

        # If we have reached the target state, resets it to None. Loops back to get current state
        for _, added_spend in reversed(
            await self.wallet_state_manager.pool_store.get_spends_for_wallet(self.wallet_id)
        ):
            latest_state: PoolState | None = solution_to_pool_state(added_spend)
            if latest_state is not None:
                if self.target_state == latest_state:
                    self.target_state = None
                    self.next_transaction_fee = uint64(0)
                    self.next_tx_config = DEFAULT_TX_CONFIG
                break

        await self.update_pool_config(action_scope)
        return True

def solution_to_pool_state(full_spend: CoinSpend) -> PoolState | None:
    full_solution_ser: SerializedProgram = full_spend.solution
    full_solution: Program = Program.from_bytes(bytes(full_solution_ser))

    if full_spend.coin.puzzle_hash == SINGLETON_LAUNCHER_HASH:
        # Launcher spend
        extra_data: Program = full_solution.rest().rest().first()
        return pool_state_from_extra_data(extra_data)

    # Not launcher spend
    inner_solution: Program = full_solution.rest().rest().first()

    # Spend which is not absorb, and is not the launcher
    num_args = len(inner_solution.as_python())
    assert num_args in {2, 3}

    if num_args == 2:
        # pool member
        if inner_solution.rest().first().as_int() != 0:
            return None

        # This is referred to as p1 in the chialisp code
        # spend_type is absorbing money if p1 is a cons box, spend_type is escape if p1 is an atom
        # TODO: The comment above, and in the CLVM, seems wrong
        extra_data = inner_solution.first()
        if isinstance(extra_data.as_python(), bytes):
            # Absorbing
            return None
        return pool_state_from_extra_data(extra_data)
    else:
        # pool waitingroom
        if inner_solution.first().as_int() == 0:
            return None
        extra_data = inner_solution.rest().first()
        return pool_state_from_extra_data(extra_data)
