### Title
`accept_requests = false` Blocks MPC Nodes from Delivering Already-Computed Signatures, Permanently Stranding In-Flight Requests Until Timeout - (File: `crates/contract/src/lib.rs`)

---

### Summary

When `verify_tee()` sets `accept_requests = false` due to insufficient valid TEE attestations, the contract blocks not only new incoming requests but also the `respond`, `respond_ckd`, and `respond_verify_foreign_tx` node-facing methods. Any request that was already queued (yield created, MPC computation completed) cannot be fulfilled â€” the node's `respond*` call is rejected, the yield times out after ~200 blocks, and the user's transaction fails. This is the direct analog of the `sDaiStrategy` pause blocking withdrawals: a safety flag intended to stop new intake also freezes the delivery path for already-in-flight work.

---

### Finding Description

The `MpcContract` struct holds a boolean field `accept_requests` that is set to `false` by `verify_tee()` when fewer than `threshold` participants have a valid TEE attestation. [1](#0-0) 

This flag is checked in `check_request_preconditions`, which gates all user-facing submission methods (`sign`, `request_app_private_key`, `verify_foreign_transaction`): [2](#0-1) 

However, the **same flag is also checked inside the three node-facing `respond*` methods**:

```
respond          â†’ lines 579-581
respond_ckd      â†’ lines 662-664
respond_verify_foreign_tx â†’ lines 711-713
``` [3](#0-2) [4](#0-3) [5](#0-4) 

The yield-resume lifecycle is:

1. User calls `sign` â†’ `enqueue_yield_request` parks the user's call and stores a `YieldIndex` in `pending_signature_requests`.
2. MPC nodes compute the signature and call `respond`.
3. `respond` calls `pending_requests::resolve_yields_for`, which calls `env::promise_yield_resume` to unpark the user's call. [6](#0-5) 

If `accept_requests` is `false` at step 2, `respond` returns `Err(TeeError::TeeValidationFailed)` before reaching `resolve_yields_for`. The yield is never resumed. After ~200 blocks the NEAR runtime fires the yield-callback with `Err(PromiseError::Failed)`, which pops the queue entry and calls `fail_on_timeout`: [7](#0-6) 

The user's transaction fails even though the MPC network successfully computed a valid signature.

---

### Impact Explanation

**Medium.** This is a request-lifecycle invariant break. Any request submitted before `accept_requests` was set to `false` â€” including bridge `verify_foreign_transaction` requests that may gate downstream fund releases â€” is permanently stranded until the NEAR yield timeout fires and the transaction fails. The MPC nodes hold a valid, computed signature that they cannot deliver. The user must resubmit after the TEE situation is resolved, but the original transaction is already failed and any time-sensitive cross-chain operation (e.g., a bridge unlock with a deadline) is broken. The deposit is consumed and not refunded on timeout.

---

### Likelihood Explanation

**Medium.** TEE attestation expiry is a documented, expected operational event. The `verify_tee()` function is callable by any current participant (via `voter_or_panic()`), and the `accept_requests = false` branch is explicitly tested and exercised in the sandbox test suite: [8](#0-7) 

The window during which `accept_requests` is `false` is intended to be transient (until participants renew attestations), but during that window any in-flight request is silently stranded. On a busy network with many concurrent requests, multiple users are affected simultaneously.

---

### Recommendation

Remove the `accept_requests` guard from the three `respond*` methods. The flag's purpose is to stop **new** requests from entering the queue when the TEE posture is degraded. Requests already in the queue were accepted under a valid TEE posture and should be fulfilled. The analogous fix to the `sDaiStrategy` report is to delete these three blocks:

```rust
// In respond, respond_ckd, respond_verify_foreign_tx â€” REMOVE:
if !self.accept_requests {
    return Err(TeeError::TeeValidationFailed.into());
}
``` [3](#0-2) 

The `check_request_preconditions` guard on the submission side is sufficient to stop new intake. Already-queued yields must remain deliverable regardless of the `accept_requests` state.

---

### Proof of Concept

1. Contract is `Running` with 3 participants (threshold 2). Two participants have valid TEE attestations; one has an expiring attestation.
2. User calls `sign(...)` â€” yield is created, `pending_signature_requests` entry is stored, user's call is parked.
3. MPC nodes complete the threshold signing protocol and the leader prepares a valid `respond(request, signature)` call.
4. Before the `respond` transaction lands, a participant calls `verify_tee()`. The expiring attestation has now lapsed, leaving only 1 valid attestation (below threshold=2). `accept_requests` is set to `false`. [1](#0-0) 

5. The leader's `respond` transaction executes. It hits:

```rust
if !self.accept_requests {
    return Err(TeeError::TeeValidationFailed.into());
}
``` [3](#0-2) 

`resolve_yields_for` is never called. The yield remains parked.

6. After ~200 blocks the NEAR runtime fires `return_signature_and_clean_state_on_success` with `Err(PromiseError::Failed)`. The queue entry is popped and `fail_on_timeout` is scheduled. [9](#0-8) 

7. The user's `sign` transaction fails. The valid signature computed by the MPC network is discarded. The user must resubmit after the TEE situation is resolved, losing the gas and deposit from the original call.
