### Title
Stale Votes from Removed Participants Inflate TEE Code-Hash and OS-Measurement Vote Counts, Enabling Sub-Threshold Allowlist Manipulation - (File: `crates/contract/src/tee/proposal.rs`, `crates/contract/src/tee/measurements.rs`)

---

### Summary

After a resharing that removes participants, their previously cast votes for TEE node-image hashes and OS measurements persist in contract storage until the asynchronous `clean_tee_status` cleanup promise executes. During this window, `CodeHashesVotes::count_votes()` and `MeasurementVotes::count_votes()` count **all** stored votes without filtering by the current participant set, unlike `vote_update` which explicitly filters stale votes. A Byzantine participant who survives the resharing (or a new participant admitted by it) can cast one vote for the same malicious proposal and, combined with the stale votes, cross the governance threshold — adding an unauthorized TEE code hash or OS measurement to the allowlist with fewer current-participant votes than the threshold requires.

---

### Finding Description

**Root cause — unfiltered vote counts in TEE governance paths**

`CodeHashesVotes::count_votes()` counts every stored vote for a proposal with no participant-membership filter: [1](#0-0) 

```rust
fn count_votes(&self, proposal: &NodeImageHash) -> u64 {
    self.proposal_by_account
        .values()
        .filter(|&prop| prop == proposal)
        .count() as u64
}
```

`MeasurementVotes::count_votes()` has the identical defect: [2](#0-1) 

Both `vote()` methods return this raw count to the contract, which compares it directly to the governance threshold to decide whether to apply the proposal.

**Contrast with `vote_update`**

`vote_update` was explicitly hardened against exactly this class of bug. It re-filters the stored vote map against the live participant set before comparing to the threshold: [3](#0-2) 

```rust
let valid_votes_count = running_state
    .parameters
    .participants()
    .participants()
    .iter()
    .filter(|(account_id, _, _)| {
        self.proposed_updates
            .vote_by_participant
            .get(account_id)
            .is_some_and(|voted_id| *voted_id == id)
    })
    .count();
```

The inline comment even acknowledges the risk: *"Note: MpcContract::vote_update uses filtering to ensure correctness even if this cleanup fails."* No equivalent guard exists for `vote_add_code_hash`, `vote_add_launcher_hash`, or `vote_add_os_measurement`.

**The exploitable window**

When resharing completes, `vote_reshared` spawns `CLEAN_TEE_STATUS` as a **detached** promise: [4](#0-3) 

Because the promise is detached, it executes in a later block as an independent transaction. Any `vote_add_code_hash` or `vote_add_os_measurement` call that lands between the resharing-completion block and the cleanup block operates on the stale vote map.

**Attack scenario (N=5, threshold=3)**

1. Byzantine old participants P1 and P2 vote for malicious node-image hash `H` before resharing.
2. Honest participants vote to remove P1 and P2; resharing completes, new set = {P3, P4, P5}.
3. Before `clean_tee_status` executes, Byzantine new participant P3 calls `vote_add_code_hash(H)`.
4. `count_votes(H)` returns 3 (P1 + P2 + P3). Threshold = 3 → `H` is added to the allowlist.
5. Only 1 current-participant vote was cast; the threshold was never genuinely reached.

The codebase's own test documents this exact scenario as a known risk that `clean_non_participant_votes` is meant to prevent: [5](#0-4) 

---

### Impact Explanation

Adding an unauthorized TEE node-image hash to the allowlist permits nodes running that image to pass `assert_caller_is_attested_participant_and_protocol_active()` and participate in threshold signing rounds. If enough such nodes are admitted, they can collectively produce unauthorized threshold signatures or expose key-share material. This breaks the production safety invariant that only governance-approved TEE images may participate in the MPC network.

**Impact class:** Medium — participant-state and contract execution-flow manipulation that breaks production safety invariants (unauthorized TEE image allowlisting) without requiring network-level DoS.

---

### Likelihood Explanation

- Requires Byzantine participants to have voted for a malicious hash before being removed by resharing — a realistic adversarial strategy for a participant who anticipates removal.
- The exploitable window is narrow (one or a few blocks between resharing completion and `clean_tee_status` execution), but NEAR block times are ~1 second and the attacker controls when they submit their transaction.
- The attacker needs only one surviving or newly admitted Byzantine participant to cast the final vote.
- No privileged access, leaked keys, or TEE physical attacks are required — only a standard contract call from a current participant account.

---

### Recommendation

Apply the same inline filtering used in `vote_update` to all TEE governance vote-count paths. Replace the raw `count_votes()` call with a filtered count that only tallies votes from accounts that are members of the current participant set:

```rust
// In CodeHashesVotes::vote() / MeasurementVotes::vote():
fn count_votes_for_participants(
    &self,
    proposal: &NodeImageHash,
    participants: &Participants,
) -> u64 {
    self.proposal_by_account
        .iter()
        .filter(|(participant_id, prop)| {
            *prop == proposal
                && participants.is_participant_given_participant_id(&participant_id.get())
        })
        .count() as u64
}
```

Pass the current `ThresholdParameters` into the vote methods (as `ProviderVotes::vote` and `TeeVerifierVotes::vote` already do) so the filtered count is used for the threshold comparison. This eliminates the dependency on the asynchronous cleanup promise for correctness, matching the design already applied to `vote_update`. [6](#0-5)

### Proof of Concept

```
State: N=5 participants {P1,P2,P3,P4,P5}, governance threshold T=3.

Block B:
  P1 calls vote_add_code_hash(H)  → count_votes(H) = 1, threshold not reached.
  P2 calls vote_add_code_hash(H)  → count_votes(H) = 2, threshold not reached.

Block B+1:
  Resharing completes removing P1, P2. New set = {P3,P4,P5}.
  CLEAN_TEE_STATUS detached promise scheduled for block B+2.

Block B+1 (same block, later receipt, or block B+2 before cleanup):
  P3 calls vote_add_code_hash(H).
  CodeHashesVotes::count_votes(H) iterates proposal_by_account:
    → P1→H (stale, not filtered), P2→H (stale, not filtered), P3→H (fresh)
    → returns 3.
  3 >= threshold(3) → H added to allowed_mpc_image_hashes.

Block B+2:
  CLEAN_TEE_STATUS runs, removes P1 and P2 votes — but H is already allowlisted.
```

Nodes running image `H` can now call `submit_participant_info` with a valid attestation quoting `H`, pass `assert_caller_is_attested_participant_and_protocol_active`, and participate in threshold signing rounds.

```rust
    pub fn vote(
        &mut self,
        chain: ForeignChain,
        hash: ProposalHash,
        participant: AuthenticatedParticipantId,
        threshold_parameters: &ThresholdParameters,
    ) -> Result<bool, Error> {
        let protocol_threshold = threshold_parameters.threshold().value();
        let participants = threshold_parameters.participants();
        // Scope the borrow on `self.pending.vote` so we can mutate `self.pending`
        // after `count_for`.
        let count_usize = {
            let voter_set = self.pending.vote((participant, chain), hash);
            voter_set.count_for(|(p, c)| {
                *c == chain && participants.is_participant_given_participant_id(&p.get())
            })
        };
        let count = u64::try_from(count_usize).map_err(|e| ConversionError::DataConversion {
            reason: format!("vote count {count_usize} does not fit in u64: {e}"),
        })?;
        if count >= protocol_threshold {
            // Drop ALL pending rows for this chain regardless of which proposal
            // they held — matches the previous `clear_chain` semantics.
            self.pending.retain_votes(|(_, c)| *c != chain);
            Ok(true)
        } else {
            Ok(false)
        }
    }
```
