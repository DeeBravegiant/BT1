### Title
Single Participant Can Freeze All `verify_foreign_transaction` Requests via Empty Chain Registration — (`crates/contract/src/lib.rs`)

---

### Summary

`verify_foreign_transaction` gates on `get_supported_foreign_chains()`, which applies a **strict all-participant intersection rule**. A single registered participant that calls `register_foreign_chain_support` with an empty set — or that has never registered — causes the intersection to collapse to the empty set, permanently rejecting every `verify_foreign_transaction` call until the participant re-registers. This is a direct analog to the external report's "pair not marked as delegated → lookup defaults to wrong address → operation reverts" pattern.

---

### Finding Description

**Root cause — three cooperating code paths:**

**1. `register_foreign_chain_support` accepts an empty set with no validation** [1](#0-0) 

Any registered participant can call this with `foreign_chain_support = {}`. There is no non-empty check. The call succeeds and overwrites the participant's prior registration with an empty set.

**2. `get_supported_foreign_chains()` uses a strict all-participant intersection** [2](#0-1) 

The filter `nodes_supporting_chain.is_superset(&active_participant_account_ids)` requires **every** active participant to appear in the supporter set for a chain. If participant P registered an empty set, P is in `active_participant_account_ids` but absent from `nodes_supporting_chain` for every chain. The superset check fails for all chains, so the function returns `{}`.

The design document explicitly acknowledges this: *"A single node that registers an empty list (or hasn't registered yet) drops every chain — one operator can take the whole feature down."*



**3. `verify_foreign_transaction` hard-panics on the empty result** [3](#0-2) 

Every `verify_foreign_transaction` call reads `get_supported_foreign_chains()` and panics with `ForeignChainNotSupported` if the requested chain is absent. When the supported set is empty, **all** foreign-chain verification requests are rejected regardless of which chain they target.

Note: the newer `get_available_foreign_chains()` / `register_foreign_chains_config` path (threshold-based, stored in `foreign_chains`) is not used here — `verify_foreign_transaction` still reads the old `node_foreign_chain_support` store. [4](#0-3) 

---

### Impact Explanation

A single participant (strictly below the signing threshold) can freeze the entire `verify_foreign_transaction` request lifecycle for all foreign chains simultaneously. Every pending and future `verify_foreign_transaction` call will revert with `ForeignChainNotSupported` until the attacker re-registers a non-empty set. This breaks the production request-lifecycle invariant that foreign-chain verification is available whenever the network is running and chains are whitelisted. Deposited funds attached to in-flight `verify_foreign_transaction` yield-resume calls will time out and be lost to the caller.

**Allowed impact matched:** Medium — request-lifecycle and contract execution-flow manipulation that breaks production safety/accounting invariants without relying on network-level DoS or operator misconfiguration.

---

### Likelihood Explanation

The attack requires only one registered participant to call `register_foreign_chain_support({})` — a single NEAR transaction from their own account. No threshold collusion, no key material, and no privileged access beyond being a participant is needed. The MPC system is explicitly designed to tolerate Byzantine participants **below** the threshold; this path violates that design guarantee with a one-participant action. The design document confirms the team is aware of the class of issue and is working toward a fix via the threshold-based `get_available_foreign_chains()` path, but `verify_foreign_transaction` has not yet been migrated.

---

### Recommendation

Replace the `get_supported_foreign_chains()` call inside `verify_foreign_transaction` with `get_available_foreign_chains()`:

```rust
// Before (strict intersection — one empty registration collapses to {})
let supported_chains = self.get_supported_foreign_chains();

// After (threshold-based — tolerates up to n−threshold non-registering nodes)
let supported_chains = self.get_available_foreign_chains();
```

Additionally, add a non-empty validation guard in `register_foreign_chain_support` to reject empty registrations, preventing accidental or deliberate collapse of the supported-chain set.

---

### Proof of Concept

1. Network is Running with participants `[P0, P1, P2]`, threshold = 2. All three have previously registered `{Bitcoin, Ethereum}` via `register_foreign_chain_support`.
2. Participant `P0` sends: `register_foreign_chain_support({})` — a valid NEAR transaction from their own account, accepted with no error.
3. `node_foreign_chain_support.foreign_chain_support_by_node` now maps `P0 → {}`, `P1 → {Bitcoin, Ethereum}`, `P2 → {Bitcoin, Ethereum}`.
4. Any user calls `verify_foreign_transaction` for Bitcoin.
5. `get_supported_foreign_chains()` builds `foreign_chain_to_node_mapping = {Bitcoin: {P1,P2}, Ethereum: {P1,P2}}`. For Bitcoin: `{P1,P2}.is_superset({P0,P1,P2})` → **false**. Returns `{}`.
6. `verify_foreign_transaction` panics: `ForeignChainNotSupported { requested: Bitcoin }`.
7. All foreign-chain verification requests are frozen until `P0` re-registers a non-empty set. [1](#0-0) [2](#0-1) [3](#0-2)

    pub fn get_supported_foreign_chains(&self) -> dtos::SupportedForeignChains {
        let active_participant_account_ids = self
            .protocol_state
            .active_participants()
            .participants()
            .iter()
            .map(|(account_id, _, _)| account_id.clone())
            .collect::<BTreeSet<_>>();

        let mut foreign_chain_to_node_mapping: BTreeMap<
            &dtos::ForeignChain,
            BTreeSet<dtos::AccountId>,
        > = BTreeMap::new();

        for (account_id, chains) in self
            .node_foreign_chain_support
            .foreign_chain_support_by_node
            .iter()
        {
            for chain in chains.iter() {
                foreign_chain_to_node_mapping
                    .entry(chain)
                    .or_default()
                    .insert(account_id.clone());
            }
        }

        foreign_chain_to_node_mapping
            .into_iter()
            .filter_map(|(foreign_chain, nodes_supporting_chain)| {
                let all_active_nodes_supports_chain =
                    nodes_supporting_chain.is_superset(&active_participant_account_ids);

                if all_active_nodes_supports_chain {
                    Some(foreign_chain)
                } else {
                    None
                }
            })
            .cloned()
            .collect::<BTreeSet<dtos::ForeignChain>>()
            .into()
    }
