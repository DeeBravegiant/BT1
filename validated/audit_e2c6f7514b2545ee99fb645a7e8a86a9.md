Audit Report

## Title
Missing Minimum-Output Slippage Protection on `removeLiquidity` with No Periphery Alternative — (`metric-core/contracts/MetricOmmPool.sol`)

## Summary
`MetricOmmPool.removeLiquidity` accepts only bin indices and share counts; it provides no `minAmount0Out` / `minAmount1Out` parameters and no periphery wrapper enforces them. Because the `msg.sender == owner` constraint prevents EOA LPs from routing through a self-built slippage-checking wrapper, and because output amounts are computed from live bin balances at execution time, an LP can receive materially less than expected with no on-chain revert path.

## Finding Description
`LiquidityDelta` carries only `binIdxs` and `shares`: [1](#0-0) 

`removeLiquidity` accepts no minimum-output parameters: [2](#0-1) 

`LiquidityLib.removeLiquidity` computes output amounts from the live `token0BalanceScaled` / `token1BalanceScaled` at the moment of execution, burns shares, and immediately transfers tokens — all inside a single atomic path with no post-computation revert hook available to the caller: [3](#0-2) [4](#0-3) [5](#0-4) 

The `msg.sender == owner` guard is the critical aggravating factor: [6](#0-5) 

Because `msg.sender` must equal `owner`, an EOA LP cannot route through any intermediary contract that would check output amounts after the call returns and revert if they are insufficient. The LP must call the pool directly and accept whatever the bins hold at execution time.

The periphery provides slippage protection only for the add path (`maxAmountToken0` / `maxAmountToken1` caps enforced in the callback): [7](#0-6) 

No `MetricOmmPoolLiquidityRemover` contract exists in the periphery:


## Impact Explanation
An LP submitting `removeLiquidity` expecting to redeem shares at the current bin composition can receive a materially different token mix if swaps cross their bins between submission and mining. Because shares are burned and tokens transferred atomically with no revert path, the loss is permanent. This is a direct loss of LP principal with no existing on-chain guard, satisfying the Medium impact threshold under Sherlock rules.

## Likelihood Explanation
Any swap that crosses the LP's bins between tx submission and mining changes the payout ratio. In active markets or under MEV sandwich conditions this window is routinely exploited. The `msg.sender == owner` constraint means EOA LPs — the common case — have no defensive option whatsoever. Likelihood is Medium: requires concurrent swap activity, which is normal for an active pool.

## Recommendation
1. **Core**: Add `uint256 minAmount0Out` and `uint256 minAmount1Out` to `removeLiquidity`; revert if computed outputs fall below them.
2. **Periphery**: Create a `MetricOmmPoolLiquidityRemover` that wraps `removeLiquidity`, checks the returned `amount0Removed` / `amount1Removed` against caller-supplied minimums, and reverts if either is below threshold — mirroring how `MetricOmmPoolLiquidityAdder.addLiquidityExactShares` enforces `maxAmountToken0` / `maxAmountToken1`.

## Proof of Concept
1. LP holds 50% of shares in bin 0, which currently holds 1 000 token0 and 0 token1.
2. LP (EOA) submits `removeLiquidity` for their shares, expecting ≈500 token0.
3. Before the LP's tx is mined, a swap sells token1 into the pool, buying all 1 000 token0 from bin 0; bin 0 now holds 0 token0 and ~1 000 token1.
4. LP's tx executes: `LiquidityLib.removeLiquidity` reads `binState.token0BalanceScaled = 0` and `binState.token1BalanceScaled ≈ 1 000 scaled`, computes `amount0Scaled = 0`, `amount1Scaled ≈ 500 scaled`, burns shares, and transfers ≈500 token1 to the LP.
5. If token1 is worth half of token0, the LP has lost ~$250 with no revert path.
6. Because `msg.sender == owner` is enforced, the LP could not have routed through a slippage-checking wrapper contract.
