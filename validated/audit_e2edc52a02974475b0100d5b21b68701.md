### Title
Caller-Agnostic Queue Key in `verify_foreign_transaction` Enables Queue Saturation to Permanently Block Legitimate Bridge Verifications ‚Äî (File: `crates/contract/src/dto_mapping.rs`, `crates/contract/src/pending_requests.rs`)

---

### Summary

The `VerifyForeignTransactionRequest` map key does not include the caller's account ID. Any unprivileged account can fill the 128-slot fan-out queue for any specific foreign transaction at the cost of gas fees only (deposits are refunded on response). This permanently blocks legitimate bridge users from getting their foreign transaction verified through the contract, breaking the request-lifecycle invariant.

---

### Finding Description

**Asymmetry between `sign()` and `verify_foreign_transaction()` request keys:**

For `sign()`, the stored key is `SignatureRequest`, which includes a `tweak` derived from `(predecessor_id, path)`:

```rust
pub fn new(domain: DomainId, payload: Payload, predecessor_id: &AccountId, path: &str) -> Self {
    let tweak = crate::kdf::derive_tweak(predecessor_id, path);
    SignatureRequest { domain_id: domain, tweak, payload }
}
```

This makes the queue per-caller ‚Äî an attacker cannot fill another user's `sign()` queue. [1](#0-0) 

For `verify_foreign_transaction()`, the conversion to the stored key strips the caller entirely:

```rust
pub fn args_into_verify_foreign_tx_request(
    args: dtos::VerifyForeignTransactionRequestArgs,
) -> dtos::VerifyForeignTransactionRequest {
    dtos::VerifyForeignTransactionRequest {
        domain_id: args.domain_id,
        request: args.request,
        payload_version: args.payload_version,
    }
}
``` [2](#0-1) 

The `VerifyForeignTransactionRequest` struct contains only `{request, domain_id, payload_version}` ‚Äî no caller identity: [3](#0-2) 

This is confirmed by the unit test comment: *"duplicate foreign-tx requests from different callers should fan out"* under the same key: [4](#0-3) 

**The fan-out queue cap is 128:**

```rust
pub const MAX_PENDING_REQUEST_FAN_OUT: u8 = 128;
```

When the queue is full, new submissions panic with `PendingRequestQueueFull`: [5](#0-4) 

**Attack path:**

1. Attacker observes (on-chain or in mempool) that Alice needs to verify Bitcoin `tx_id = [0xAB; 32]` for a bridge operation.
2. Attacker front-runs Alice by submitting 128 `verify_foreign_transaction` calls for the same `tx_id` (from one or many accounts). Each call requires only 1 yoctoNEAR deposit + ~7 Tgas.
3. Alice's submission is rejected with `PendingRequestQueueFull`.
4. MPC nodes eventually respond; all 128 attacker yields are drained and deposits refunded.
5. Attacker immediately re-submits 128 requests, blocking Alice again.
6. Repeat indefinitely. Cost to attacker: gas fees only (~7 Tgas √ó 128 per cycle on NEAR, which is negligible).

The `verify_foreign_transaction` entry point has no per-caller rate limit, no non-refundable penalty, and no minimum collateral beyond 1 yoctoNEAR: [6](#0-5) 

---

### Impact Explanation

A bridge user who has sent funds on a foreign chain (e.g., Bitcoin, Ethereum) and needs the MPC network to verify and sign the transaction can be permanently blocked from receiving the verification response. The user's NEAR yield promise times out, their transaction fails, and they cannot resubmit as long as the attacker maintains the block. This breaks the request-lifecycle safety invariant for the `verify_foreign_transaction` flow and can cause bridge funds to be permanently inaccessible.

This maps to: **Medium ‚Äî request-lifecycle and contract execution-flow manipulation that breaks production safety/accounting invariants without relying on network-level DoS or operator misconfiguration.**

---

### Likelihood Explanation

- The target foreign transaction is publicly observable on-chain (the `tx_id` is in the `ForeignChainRpcRequest`).
- NEAR gas fees are extremely cheap; 128 submissions per cycle is economically trivial.
- No privileged access is required ‚Äî any NEAR account can call `verify_foreign_transaction`.
- The attacker's deposits are fully refunded on each MPC response, so the sustained cost is gas only.
- Front-running is straightforward on NEAR since transactions are visible before finalization.

---

### Recommendation

Include the caller's account ID in the `VerifyForeignTransactionRequest` key, analogous to how `SignatureRequest` includes the tweak derived from `predecessor_id`. This makes each caller's queue slot independent:

```rust
pub struct VerifyForeignTransactionRequest {
    pub caller: AccountId,   // add this
    pub request: ForeignChainRpcRequest,
    pub domain_id: DomainId,
    pub payload_version: ForeignTxPayloadVersion,
}
```

Alternatively, require a non-trivial non-refundable deposit (slashed if the queue is full or the request is abandoned), so the attacker bears a real economic cost per attack cycle.

pub const MAX_PENDING_REQUEST_FAN_OUT: u8 = 128;

/// Append a yield index to the pending-request fan-out queue for `request`.
///
/// Panics with `RequestError::PendingRequestQueueFull` if the resulting queue would
/// exceed `MAX_PENDING_REQUEST_FAN_OUT`.
pub(crate) fn push_pending_yield<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: K,
    data_id: CryptoHash,
) where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let queue = requests.entry(request).or_default();
    if queue.len() >= usize::from(MAX_PENDING_REQUEST_FAN_OUT) {
        env::panic_str(
            &RequestError::PendingRequestQueueFull {
                limit: MAX_PENDING_REQUEST_FAN_OUT,
            }
            .to_string(),
        );
    }
