Audit Report

## Title
LP Principal Permanently Locked When USDC/USDT Blacklists Position Owner in `removeLiquidity` — (`metric-core/contracts/libraries/LiquidityLib.sol`)

## Summary

`LiquidityLib.removeLiquidity` unconditionally transfers pool tokens to `owner` with no alternative recipient path. If `owner` is blacklisted by USDC or USDT — both of which are explicitly deployed as pool tokens across the protocol's supported chains — every withdrawal attempt reverts and the LP's principal is permanently locked with no on-chain recovery mechanism.

## Finding Description

In `LiquidityLib.removeLiquidity`, after all bin accounting and share burns are computed, the function calls `safeTransfer` directly to `owner` with no fallback: [1](#0-0) 

If either `ctx.token0` or `ctx.token1` is USDC or USDT and `owner` is on the blacklist, `safeTransfer` reverts. Because the entire transaction reverts atomically, the share-burn and bin-balance updates also roll back — the LP's shares are preserved in storage but can never be redeemed.

The `removeLiquidity` entry point in `MetricOmmPool` enforces `msg.sender == owner`: [2](#0-1) 

There is no `recipient` parameter, no position-transfer mechanism, and no admin rescue path. A blacklisted `owner` has no on-chain route to recover their capital. The interface NatSpec confirms this design: "Only `owner` may burn." [3](#0-2) 

USDC and USDT are explicitly deployed as pool tokens across Base, Optimism, Linea, Ethereum, BSC, and Arbitrum in the protocol's own deployment configs: [4](#0-3) [5](#0-4) 

## Impact Explanation

An LP whose address is USDC/USDT-blacklisted loses their entire deposited principal permanently. The pool's bin accounting correctly records the shares, but no withdrawal can ever succeed. This is a direct, irrecoverable loss of user principal — a broken core pool withdrawal flow — with no admin mitigation path. The impact meets the Critical/High threshold for direct loss of LP principal.

## Likelihood Explanation

USDC and USDT blacklisting is a documented, real-world event (Circle and Tether have blacklisted hundreds of addresses for OFAC compliance). The protocol explicitly targets USDC/USDT pools across multiple chains. Any LP using an EOA that is later blacklisted — or any LP whose address is sanctioned after deposit — triggers this condition without any privileged or malicious setup. No attacker capability is required; the condition arises from external regulatory action on a real-world token the protocol explicitly integrates.

## Recommendation

Add an optional `recipient` parameter to `removeLiquidity` (separate from `owner`) so a blacklisted owner can redirect token output to a non-blacklisted address they control. Alternatively, implement a two-step withdrawal pattern that separates share burning from token claiming, allowing the owner to specify a recipient at claim time. The `msg.sender == owner` guard should remain on the burn authorization step; only the transfer destination needs to be made flexible.

## Proof of Concept

1. Alice (`0xAlice`) provides liquidity to a USDC/WETH pool on Optimism. Her position is keyed `(0xAlice, salt)` in `_positionBinShares`.
2. Circle blacklists `0xAlice` (e.g., OFAC compliance action).
3. Alice calls `MetricOmmPool.removeLiquidity(0xAlice, salt, deltas, "")`.
4. `msg.sender == owner` check passes (Alice is calling herself).
5. `LiquidityLib.removeLiquidity` burns her shares and updates bin balances in storage.
6. `IERC20(USDC).safeTransfer(0xAlice, amount0Removed)` reverts — USDC blacklist check fails.
7. The entire transaction reverts; all storage changes roll back, shares are restored.
8. Alice cannot call `removeLiquidity` from any other address — `msg.sender != owner` guard reverts.
9. Alice's USDC principal is permanently locked in the pool.

Foundry test plan: deploy a mock USDC with a blacklist, create a pool with it as `token0`, mint LP shares for `alice`, blacklist `alice` on the mock USDC, then assert that `pool.removeLiquidity(alice, salt, deltas, "")` reverts and that `alice`'s `positionBinShares` remain unchanged after the revert.
