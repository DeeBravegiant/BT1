### Title
Zero `BlockConfirmations` Accepted in `verify_foreign_transaction` Bypasses Bitcoin Finality Guarantee — (File: `crates/contract/src/lib.rs`)

### Summary

The `verify_foreign_transaction` contract method accepts a `BitcoinRpcRequest` with `confirmations: BlockConfirmations(0)` without any minimum-bound enforcement. Because the node-side inspector's confirmation check is `threshold <= actual`, a threshold of zero is trivially satisfied by any transaction — including unconfirmed mempool transactions. The MPC network therefore produces a valid threshold signature attesting to a Bitcoin transaction that has never been mined, breaking the core finality invariant the feature is designed to enforce.

### Finding Description

`BlockConfirmations` is a plain `u64` newtype with no minimum constraint:

```rust
pub struct BlockConfirmations(pub u64);
```

The ABI schema explicitly allows `"minimum": 0.0`. The `verify_foreign_transaction` entry point performs no validation on the `confirmations` field of a `BitcoinRpcRequest` before enqueuing the yield:

```rust
pub fn verify_foreign_transaction(&mut self, request: VerifyForeignTransactionRequestArgs) {
    self.check_request_preconditions(...);          // checks domain, gas, deposit only
    let requested_chain = request.request.chain();
    if !supported_chains.contains(&requested_chain) { ... }
    // ← no validation of BitcoinRpcRequest::confirmations
    self.enqueue_yield_request(...);
}
```

On the node side, `BitcoinInspector::extract` enforces the threshold with:

```rust
let enough_block_confirmations =
    block_confirmations_threshold <= transaction_block_confirmation;
if !enough_block_confirmations {
    return Err(ForeignChainInspectionError::NotEnoughBlockConfirmations { ... });
}
```

When `block_confirmations_threshold = 0`, the condition `0 <= any_u64` is always `true` (since `u64` is unsigned), so the check never fires. The inspector proceeds to extract values and the MPC network signs the payload.

The signed payload is `ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 { request, values })`, where `request` includes `confirmations: 0`. The `compute_msg_hash` function Borsh-serialises the full request, so the signature commits to `confirmations: 0` — but it still constitutes a valid MPC threshold signature over a transaction that may not exist in any mined block.

### Impact Explanation

The stated purpose of `verify_foreign_transaction` is:

> "This feature lets the MPC network sign payloads **only after verifying** a specific foreign-chain transaction."

With `confirmations: 0`, the MPC network signs for a transaction that has received zero block confirmations — i.e., a mempool transaction that may be double-spent or never confirmed. This breaks the production safety invariant that the feature is designed to enforce. In the primary use case (Omnibridge inbound flow: foreign chain → NEAR), a bridge contract that releases NEAR-side assets upon receiving a valid MPC signature over a Bitcoin transaction ID can be exploited: the attacker obtains the signature for an unconfirmed transaction, claims the NEAR-side assets, and then double-spends the Bitcoin transaction. This constitutes an invalid bridge execution / double-spend condition.

**Allowed impact match:** Medium — balance and contract execution-flow manipulation that breaks production safety/accounting invariants.

### Likelihood Explanation

The entry path is fully unprivileged. Any NEAR account can call `verify_foreign_transaction` with `confirmations: 0` and a 1 yoctoNEAR deposit. No special role, key, or collusion is required. The contract accepts the request, the node-side check trivially passes, and the MPC network produces a signature. Likelihood is high.

### Recommendation

Enforce a minimum of `1` on `BlockConfirmations` at the contract boundary inside `verify_foreign_transaction`, before the request is enqueued:

```rust
if let ForeignChainRpcRequest::Bitcoin(ref btc) = request.request {
    if btc.confirmations == BlockConfirmations(0) {
        env::panic_str("Bitcoin confirmations must be at least 1");
    }
}
```

Alternatively, enforce the minimum inside the `BlockConfirmations` constructor or via a validated newtype, mirroring the pattern already used for `ChainEntry::quorum` (which rejects `quorum == 0` at the DTO boundary). A protocol-level minimum (e.g., 6 confirmations for Bitcoin) should be considered for production deployments.

### Proof of Concept

1. Any NEAR account calls:
   ```json
   {
     "request": {
       "request": {
         "Bitcoin": {
           "tx_id": "<mempool-tx-id-hex>",
           "confirmations": 0,
           "extractors": ["BlockHash"]
         }
       },
       "domain_id": <foreign_tx_domain_id>,
       "payload_version": 1
     }
   }
   ```
   with 1 yoctoNEAR attached to `verify_foreign_transaction`.

2. The contract passes `check_request_preconditions` (domain, gas, deposit all valid) and enqueues the yield. No validation of `confirmations` occurs.

3. MPC nodes receive the request. `BitcoinInspector::extract` evaluates `0 <= rpc_response.confirmations` — always `true` for any `u64`. The inspector proceeds to the canonical-chain check and extracts the block hash.

4. The MPC network produces a threshold signature over `SHA-256(borsh(ForeignTxSignPayload::V1 { request: Bitcoin { confirmations: 0, ... }, values: [BlockHash(...)] }))`.

5. The attacker receives a valid MPC signature attesting to an unconfirmed Bitcoin transaction. A bridge contract that does not independently re-validate `confirmations` from the signed payload will accept this as proof of a finalised inbound transfer and release NEAR-side assets. The attacker then double-spends the Bitcoin transaction.
