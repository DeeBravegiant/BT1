### Title
Cross-Chain Signature Replay in `DelegateAction` Meta-Transactions — (`core/primitives/src/action/delegate.rs`)

### Summary

`DelegateAction` (NEP-366 meta-transactions) signed payloads do not include a chain identifier or genesis hash. Unlike regular `SignedTransaction` which binds to a specific chain via `block_hash`, a `DelegateAction` signature is valid on any NEAR-compatible chain (mainnet, testnet, localnet) where the same account ID, public key, and a sufficiently low nonce exist. A malicious relayer can take a `SignedDelegateAction` intended for one chain and submit it on another, causing unauthorized execution of the inner actions.

### Finding Description

Regular NEAR transactions include a `block_hash` field that anchors the transaction to a specific chain and provides a freshness window: [1](#0-0) 

This `block_hash` is chain-specific; a block hash from mainnet does not exist on testnet, so cross-chain replay of regular transactions is impossible.

`DelegateAction` (and `DelegateActionV2`) have no equivalent chain-binding field. The struct contains only `sender_id`, `receiver_id`, `actions`, `nonce`, `max_block_height`, and `public_key`: [2](#0-1) 

The signing hash is computed via `get_nep461_hash`, which wraps the action in a `SignableMessage` containing only a NEP-number discriminant and the serialized action body — no chain ID, no genesis hash: [3](#0-2) [4](#0-3) 

The `MessageDiscriminant` is a fixed constant derived from the NEP number (366 or 611), identical across all NEAR chains: [5](#0-4) 

At execution time, `apply_delegate_action` verifies the signature, checks `max_block_height`, validates the nonce, and executes — but never checks which chain the action was intended for: [6](#0-5) 

### Impact Explanation

A malicious relayer who receives a `SignedDelegateAction` intended for mainnet can submit it on testnet (or vice versa) if:
1. The sender's account ID exists on both chains (common for named accounts like `alice.near`).
2. The same public key is registered on both chains (common when users share a seed phrase).
3. The access key nonce on the target chain is lower than the nonce in the signed action.
4. The `max_block_height` is still valid on the target chain.

The inner actions can include `Transfer`, `AddKey`, `DeleteKey`, `FunctionCall`, `DeleteAccount`, etc. Unauthorized execution of any of these on the wrong chain corrupts the victim's account state and can cause direct loss of funds. The corrupted protocol value is the access key nonce (advanced on the wrong chain), account balance (drained by Transfer), and access key set (modified by AddKey/DeleteKey).

### Likelihood Explanation

The scenario is realistic:
- NEAR developers routinely test on testnet with the same account names and keys they later use on mainnet.
- A testnet-signed `DelegateAction` with a high `max_block_height` (e.g., set to a large value for long-lived relayer queues) and a nonce that exceeds the mainnet access key nonce can be replayed on mainnet.
- The relayer is an untrusted third party by design; NEP-366 explicitly acknowledges that relayers are application-layer entities that cannot be fully trusted. [7](#0-6) 

### Recommendation

Include the chain's genesis ID (chain name + genesis block hash) in the `DelegateAction` signed payload, mirroring how `SignedTransaction` uses `block_hash` for chain binding. Alternatively, add a `chain_id: String` field to `DelegateAction` and `DelegateActionV2` and verify it against `apply_state.config.chain_id` (or equivalent) inside `apply_delegate_action`.

### Proof of Concept

1. Alice has account `alice.near` on testnet (access key nonce = 5) and mainnet (access key nonce = 3), using the same ED25519 key pair.
2. Alice signs a `DelegateAction` on testnet with `nonce = 6`, `max_block_height = 999_999_999`, and `actions = [Transfer { deposit: 100 NEAR }]`.
3. The relayer, instead of submitting to testnet, wraps the identical `SignedDelegateAction` in a mainnet transaction and submits it to a mainnet RPC node.
4. `apply_delegate_action` on mainnet: `signed_delegate_action.verify()` passes (same key, same payload hash), `block_height (e.g. 120_000_000) <= max_block_height (999_999_999)` passes, `sender_id` matches, nonce check `6 > 3` passes.
5. The Transfer executes on mainnet, draining Alice's mainnet balance.

The root cause is at: [3](#0-2) 

where `get_nep461_hash` produces an identical hash on every NEAR chain for the same `DelegateAction` content, and at: [6](#0-5) 

where no chain identity check is performed before accepting the signature as valid.

impl<'a, T: BorshSerialize> SignableMessage<'a, T> {
    pub fn new(msg: &'a T, ty: SignableMessageType) -> Self {
        let discriminant = ty.into();
        Self { discriminant, msg }
    }

    pub fn sign(&self, signer: &Signer) -> Signature {
        let bytes = borsh::to_vec(&self).expect("Failed to deserialize");
        let hash = hash(&bytes);
        signer.sign(hash.as_bytes())
    }
}

## Relayer

Meta transactions only work with a relayer. This is an application layer
concept, implemented off-chain. Think of it as a server that accepts a
`SignedDelegateAction`, does some checks on them and eventually forwards it
inside a transaction to the blockchain network.

A relayer may choose to offer their service for free but that's not going to be
financially viable long-term. But they could easily have the user pay using
other means, outside of Near blockchain. And with some tricks, it can even be
paid using fungible tokens on Near.

In the example visualized above, the payment is done using \$FT. Together with
the transfer to John, Alice also adds an action to pay 0.1 \$FT to the relayer.
The relayer checks the content of the `SignedDelegateAction` and only processes
it if this payment is included as the first action. In this way, the relayer
will be paid in the same transaction as John.

Note that the payment to the relayer is still not guaranteed. It could be that
Alice does not have sufficient $FT and the transfer fails. To mitigate, the
relayer should check the $FT balance of Alice first.

Unfortunately, this still does not guarantee that the balance will be high
enough once the meta transaction executes. The relayer could waste NEAR gas
without compensation if Alice somehow reduces her \$FT balance in just the right
moment. Some level of trust between the relayer and its user is therefore
required.

The vision here is that there will be mostly application-specific relayers. A
general-purpose relayer is difficult to implement with just the MVP. See
limitations below.
