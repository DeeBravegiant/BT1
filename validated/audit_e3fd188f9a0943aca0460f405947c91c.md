### Title
Key Blob Deserialization Silently Accepts Trailing Bytes via `coinbase::convert<T>()` — (`include-internal/cbmpc/internal/core/convert.h`)

### Summary

The `coinbase::convert<T>(dst, src)` template function used throughout the public API for key blob deserialization does not check whether all input bytes were consumed. The strict sibling `coinbase::deser()` explicitly rejects trailing bytes. Every key blob deserialization call in the public API uses `convert()`, not `deser()`, so a caller can supply a key blob with arbitrary trailing bytes appended and the library accepts it without error, silently ignoring the excess. This is the direct structural analog of the Solidity `>=` fee check: the input is accepted as long as it contains at least the expected content; the excess is silently discarded.

### Finding Description

`coinbase::deser()` enforces strict deserialization:

```cpp
// include-internal/cbmpc/internal/core/convert.h:254-266
template <typename... ARGS>
error_t deser(mem_t bin, ARGS&... args) {
  converter_t converter(bin);
  converter.convert(args...);
  error_t rv = converter.get_rv();
  if (rv != SUCCESS) return rv;
  // Strict deserialization: reject trailing bytes
  if (converter.get_offset() != converter.get_size()) {
    return coinbase::error(E_BADARG);
  }
  return SUCCESS;
}

### Impact Explanation

**Message malleability in key blobs.** Because `convert()` silently ignores trailing bytes, two distinct byte sequences — the canonical blob and any blob with arbitrary bytes appended — deserialize to identical internal key state and pass all subsequent cryptographic checks (`x_share * G == Qi`, Paillier verification, curve-point validation, etc.). Any application layer that:

1. stores a hash or MAC of the key blob for integrity verification, or
2. compares two blobs byte-for-byte to detect tampering,

will produce a different result for the padded blob even though the library accepts it as fully valid. An adversary who can write to key blob storage (or intercept a blob in transit before it reaches the API) can append arbitrary bytes to produce a blob that the library accepts but that the application's integrity layer rejects — or vice versa — breaking the invariant that a valid blob has a unique canonical encoding. This falls under the "public API reachable parsing invariant break causes unsafe state acceptance" category.

### Likelihood Explanation

The affected code path is the primary deserialization entry point for every key blob type (ECDSA-2PC, ECDSA-MP, EdDSA-2PC, EdDSA-MP, Schnorr-2PC, Schnorr-MP, HD-MPC, TDH2, PVE). Any caller that stores, transmits, or compares key blobs by their byte content is exposed. The attacker surface is any system where key blobs transit an untrusted channel or are stored in attacker-reachable storage before being passed to the API.

### Recommendation

Replace `coinbase::convert(blob, in)` with `coinbase::deser(in, blob)` at every key blob deserialization site, or add the trailing-byte check directly to `convert<T>()`:

```cpp
template <typename T>
error_t convert(T& dst, mem_t src) {
  if (src.size < 0 || (src.size && !src.data)) return coinbase::error(E_BADARG);
  converter_t converter(src);
  converter.convert(dst);
  if (converter.get_rv() != SUCCESS) return converter.get_rv();
  // Reject trailing bytes — same invariant as deser()
  if (converter.get_offset() != converter.get_size())
    return coinbase::error(E_BADARG);
  return SUCCESS;
}
```

### Proof of Concept

```cpp
#include <cbmpc/internal/core/convert.h>

// 1. Obtain a valid key blob (e.g., from coinbase::api::ecdsa_2p::dkg()).
buf_t valid_blob = /* ... */;

// 2. Append arbitrary trailing bytes.
byte_t garbage[] = {0xDE, 0xAD, 0xBE, 0xEF};
buf_t padded_blob = valid_blob + mem_t(garbage, sizeof(garbage));

// 3. Both blobs deserialize successfully via the public API.
buf_t pub1, pub2;
assert(coinbase::api::ecdsa_2p::get_public_key_compressed(valid_blob, pub1) == SUCCESS);
assert(coinbase::api::ecdsa_2p::get_public_key_compressed(padded_blob, pub2) == SUCCESS);
assert(pub1 == pub2);  // same key — but valid_blob != padded_blob byte-for-byte

// 4. deser() correctly rejects the padded blob; convert() does not.
key_blob_v1_t dst;
assert(coinbase::deser(padded_blob, dst) != SUCCESS);   // strict: fails
assert(coinbase::convert(dst, padded_blob) == SUCCESS); // lenient: passes — the bug
```
