Audit Report

## Title
ETH Deposit Limit Check Omits Deposit Amount, Allowing Limit Bypass - (File: contracts/LRTDepositPool.sol)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` applies an asymmetric comparison for ETH vs ERC20 assets. The ETH branch checks only whether the current total already exceeds the cap (`totalAssetDeposits > depositLimit`), while the ERC20 branch correctly includes the incoming amount (`totalAssetDeposits + amount > depositLimit`). Any unprivileged caller can therefore execute a single `depositETH` call that pushes the protocol's ETH holdings above the admin-configured cap.

## Finding Description
In `contracts/LRTDepositPool.sol` at lines 676–682, `_checkIfDepositAmountExceedesCurrentLimit` branches on asset type:

```solidity
// L678-681
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));   // amount omitted
}
return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // correct
```

The ETH branch returns `false` (no limit breach) whenever `totalAssetDeposits ≤ depositLimit`, regardless of how large `amount` is. The result is consumed at lines 661–663 in `_beforeDeposit`:

```solidity
if (_checkIfDepositAmountExceedesCurrentLimit(asset, depositAmount)) {
    revert MaximumDepositLimitReached();
}
```

Because the function returns `false` when it should return `true`, the revert is never triggered for a boundary-crossing ETH deposit. The public entry point `depositETH` (lines 76–93) calls `_beforeDeposit` directly with `msg.value` and has no additional cap enforcement. No other guard exists between the caller and `_mintRsETH`.

## Impact Explanation
The deposit limit is the protocol's sole mechanism for capping ETH exposure. Because the ETH branch never accounts for the incoming amount, a single deposit can push total ETH holdings arbitrarily above the configured cap. This constitutes **Low — Contract fails to deliver promised returns**: the deposit ceiling guarantee is broken for ETH while no funds are directly lost. If the cap was set to bound EigenLayer strategy capacity, excess deposits could cause downstream operational failures; however, that causal chain depends on external protocol behavior and is not concretely demonstrated here, so the impact is assessed at Low.

## Likelihood Explanation
`depositETH` is fully permissionless (`external payable`, no role check). The vulnerable condition (`totalAssetDeposits ≤ depositLimit` and `totalAssetDeposits + amount > depositLimit`) is the normal boundary state the limit is designed to enforce. No front-running, special role, or external dependency is required. Likelihood is **High**.

## Recommendation
Add `amount` to the ETH branch to match the ERC20 branch:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

## Proof of Concept
1. Admin sets `depositLimitByAsset[ETH_TOKEN] = 100 ether`.
2. `getTotalAssetDeposits(ETH_TOKEN)` returns `99 ether`.
3. Attacker calls `depositETH{value: 10 ether}(0, "")`.
4. `_checkIfDepositAmountExceedesCurrentLimit`: `99 ether > 100 ether` → `false` → no revert.
5. `_mintRsETH` executes; on-chain ETH total becomes `109 ether`, exceeding the cap by `9 ether`.
6. An equivalent ERC20 deposit at the same state correctly reverts: `99 ether + 10 ether > 100 ether` → `true` → `MaximumDepositLimitReached`.

Foundry test plan: deploy `LRTDepositPool` with a mock `LRTConfig` returning `depositLimit = 100 ether`; seed `getTotalAssetDeposits` to `99 ether`; call `depositETH{value: 10 ether}`; assert the call succeeds and total deposits exceed `100 ether`; then assert an equivalent `depositAsset` call with the same parameters reverts.
