Audit Report

## Title
`rsETHPrice` Baseline Advanced Without Fee Collection During Partial Pause, Permanently Excluding Rewards From Protocol Fee — (`contracts/LRTOracle.sol`)

## Summary

`_updateRsETHPrice()` unconditionally writes `rsETHPrice = newRsETHPrice` at line 313 even when the protocol-fee calculation was skipped because `protocolPaused == true`. Since `previousTVL` on the next call is derived from the already-advanced `rsETHPrice`, any rewards that accrued during the pause window are permanently excluded from fee collection. The oracle's own `paused` flag is independent of `lrtDepositPool.paused()` and `withdrawalManager.paused()`, so `updateRSETHPrice()` remains callable by any unprivileged user during routine operational pauses of those contracts.

## Finding Description

`updateRSETHPrice()` is a public function gated only by the oracle's own `whenNotPaused` modifier ( [1](#0-0) ). Inside `_updateRsETHPrice()`, `protocolPaused` is the logical OR of three independent flags:

```solidity
bool protocolPaused = lrtDepositPool.paused() || withdrawalManager.paused() || paused;
``` [2](#0-1) 

When `protocolPaused` is `true`, the fee is zeroed out entirely:

```solidity
uint256 protocolFeeInETH = 0;
if (!protocolPaused && totalETHInProtocol > previousTVL) {
    uint256 rewardAmount = totalETHInProtocol - previousTVL;
    protocolFeeInETH = (rewardAmount * lrtConfig.protocolFeeInBPS()) / 10_000;
}
``` [3](#0-2) 

However, `rsETHPrice` — the variable that feeds `previousTVL = rsethSupply.mulWad(rsETHPrice)` on the next invocation — is written unconditionally at the end of the function regardless of whether a fee was collected:

```solidity
rsETHPrice = newRsETHPrice;
``` [4](#0-3) 

The fee-accounting baseline (`previousTVL`) is therefore advanced to the full new TVL/supply ratio even when no fee was taken. On the subsequent call after the pause is lifted, `previousTVL` already incorporates the rewards that accrued during the pause, so the reward delta is zero and no fee is collected on those rewards. The loss is irreversible.

The oracle's `whenNotPaused` modifier checks only `paused` (the oracle's own flag), not the deposit pool or withdrawal manager flags. [5](#0-4)  Consequently, when either of the latter two is paused (a routine operational event) but the oracle itself is not, any external caller can invoke `updateRSETHPrice()`, trigger the fee-skip path, and advance the baseline.

## Impact Explanation

**High — Theft of unclaimed yield.**

The protocol treasury is the intended recipient of `protocolFeeInBPS` of all staking rewards. Every call to `updateRSETHPrice()` while the deposit pool or withdrawal manager is paused (but the oracle is not) permanently excludes the entire reward delta for that period from fee collection. The treasury receives zero fee on those rewards; rsETH holders retain the full price appreciation. The loss scales with the reward delta and the fee rate and is irreversible once `rsETHPrice` has been advanced.

## Likelihood Explanation

**Medium.** The deposit pool and withdrawal manager are paused regularly for operational reasons (upgrades, EigenLayer queue management, security incidents). The oracle is not co-paused in these scenarios — the downside-protection auto-pause only fires when the oracle itself detects an out-of-limit price drop and explicitly calls `_pause()`. [6](#0-5)  During any ordinary operational pause of the deposit pool or withdrawal manager, `updateRSETHPrice()` remains open to the public. Any rsETH holder is economically incentivized to call it: doing so advances the price baseline without the fee dilution that would otherwise reduce their per-share value. The attack requires no capital, no flash loan, and no privileged access.

## Recommendation

Do not advance `rsETHPrice` when the fee calculation was skipped. Two options:

1. **Guard the price write** — only commit `rsETHPrice = newRsETHPrice` when `!protocolPaused` (i.e., when the fee was actually computed).
2. **Defer the baseline** — when `protocolPaused`, compute and display `newRsETHPrice` but do not update the stored `rsETHPrice`. This also prevents the `highestRsethPrice` ratchet from advancing during a pause, which has its own implications for the downside-protection threshold. [7](#0-6) 

Option 2 is safer and more complete.

## Proof of Concept

1. Protocol is live: `rsETHPrice = 1.05e18`, `rsethSupply = 1_000_000e18`, `protocolFeeInBPS = 1000` (10%).
2. Admin pauses `LRTDepositPool` for an upgrade. Oracle is **not** paused.
3. EigenLayer staking rewards accrue; `totalETHInProtocol` grows by `1_000e18`.
4. An rsETH holder calls `updateRSETHPrice()`:
   - `protocolPaused = true` (deposit pool is paused, oracle is not).
   - `protocolFeeInETH = 0` (fee skipped).
   - `newRsETHPrice = (1_050_000e18 + 1_000e18) / 1_000_000e18 = 1.051e18`.
   - `rsETHPrice = 1.051e18` ← baseline advanced without fee. [4](#0-3) 
5. Admin unpauses `LRTDepositPool`.
6. Next `updateRSETHPrice()` call: `previousTVL = 1_000_000 * 1.051e18 = 1_051_000e18`. The 1_000 ETH reward from step 3 is already baked in; `totalETHInProtocol - previousTVL = 0`; no fee is collected.
7. Treasury permanently loses `1_000 * 10% = 100 ETH` worth of protocol fee.

**Foundry test plan:** Deploy `LRTOracle` with a mock deposit pool, mock withdrawal manager, and mock rsETH. Set `rsETHPrice`, pause the deposit pool, advance the mock TVL, call `updateRSETHPrice()` as an unprivileged address, unpause, call `updateRSETHPrice()` again, and assert that `IRSETH.mint` was never called for the treasury during either call despite a positive reward delta existing at step 4.
