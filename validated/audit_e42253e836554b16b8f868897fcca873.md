### Title
Stale Votes from Removed Participants Counted in TEE Code Hash Threshold — (`crates/contract/src/tee/proposal.rs`)

### Summary

`CodeHashesVotes::count_votes()` tallies **all** stored votes without filtering by the current participant set. After a resharing event, removed participants' votes are cleaned up only via an asynchronous detached promise (`CLEAN_TEE_STATUS`). During the window between resharing completion and that cleanup, a removed participant's pre-cast vote for a malicious TEE image hash is still counted toward the whitelisting threshold, allowing the threshold to be satisfied with fewer current-participant votes than required.

### Finding Description

`CodeHashesVotes` in `crates/contract/src/tee/proposal.rs` stores one vote per `AuthenticatedParticipantId`:

```rust
pub struct CodeHashesVotes {
    pub proposal_by_account: BTreeMap<AuthenticatedParticipantId, NodeImageHash>,
}
```

Its `count_votes` method counts every entry in the map unconditionally: [1](#0-0) 

```rust
fn count_votes(&self, proposal: &NodeImageHash) -> u64 {
    self.proposal_by_account
        .values()
        .filter(|&prop| prop == proposal)
        .count() as u64
}
```

This is the return value used directly in `vote_code_hash` to decide whether to whitelist a TEE image: [2](#0-1) 

```rust
let votes = self.tee_state.vote(code_hash, &participant);
if votes >= self.threshold()?.value() {
    self.tee_state.whitelist_tee_proposal(code_hash, tee_upgrade_deadline_duration);
}
```

After resharing completes in `vote_reshared`, the stale-vote cleanup is dispatched as a **detached** promise: [3](#0-2) 

```rust
Promise::new(env::current_account_id())
    .function_call(
        method_names::CLEAN_TEE_STATUS.to_string(),
        ...
    )
    .detach();
```

`CLEAN_TEE_STATUS` calls `clean_non_participant_votes`, which calls `get_remaining_votes` to filter out removed participants: [4](#0-3) 

Because this cleanup is asynchronous, any `vote_code_hash` call that lands in the same block **after** the resharing receipt but **before** the `CLEAN_TEE_STATUS` receipt will see the stale votes.

Contrast this with `ThresholdParametersVotes::n_votes`, which **correctly** filters by current participants before counting: [5](#0-4) 

```rust
pub fn n_votes(&self, proposal: &ProposedThresholdParameters, participants: &Participants) -> u64 {
    self.proposal_by_account
        .iter()
        .filter(|&(acc, prop)| {
            participants.participants().iter().any(|(acc_id, _, _)| acc.get() == acc_id)
                && prop == proposal
        })
        .count() as u64
}
```

The same unfiltered counting flaw exists in `LauncherHashVotes::count_votes` and `MeasurementVotes`: [6](#0-5) 

### Impact Explanation

A removed participant's pre-cast vote for a malicious TEE image hash persists in `CodeHashesVotes` until the async cleanup runs. If a colluding current participant calls `vote_code_hash` for the same hash in the same block as resharing completion (before cleanup), `count_votes` returns `stale_votes + new_votes`. If this sum meets `threshold`, `whitelist_tee_proposal` is called and the malicious image hash is permanently added to the allowlist.

Once whitelisted, any node running that malicious image can submit a passing attestation via `submit_participant_info` and be proposed as a participant. If enough such nodes are admitted, they can participate in threshold signing or key generation, enabling unauthorized signature issuance or key share disclosure — a participant/attestation authorization bypass.

This maps to: **High — participant/attestation authorization bypass that enables invalid MPC execution**.

### Likelihood Explanation

The attack requires:
1. A participant who knows they will be removed to pre-vote for a malicious code hash before resharing.
2. A colluding current participant to call `vote_code_hash` for the same hash in the same block as resharing completion, before the `CLEAN_TEE_STATUS` receipt executes.

The window is narrow (within one block), but NEAR's receipt ordering does not guarantee `CLEAN_TEE_STATUS` runs before other transactions' receipts in the same block. The coordination requirement is below the signing threshold (only `threshold - 1` current participants needed alongside one stale vote), making this reachable by a Byzantine participant strictly below the signing threshold.

### Recommendation

1. **Filter by current participants in `count_votes`**, mirroring `ThresholdParametersVotes::n_votes`:

```rust
fn count_votes_for_participants(&self, proposal: &NodeImageHash, participants: &Participants) -> u64 {
    self.proposal_by_account
        .iter()
        .filter(|(p, v)| {
            *v == proposal
                && participants.is_participant_given_participant_id(&p.get())
        })
        .count() as u64
}
```

2. Apply the same fix to `LauncherHashVotes` and `MeasurementVotes`.

3. Alternatively, call `clean_non_participant_votes` **synchronously** at the start of `vote_code_hash`, `vote_add_launcher_hash`, and `vote_add_os_measurement` before counting votes.

### Proof of Concept

1. Running state: 5 participants `{A, B, C, D, E}`, governance threshold = 3.
2. Participant `E` (who will be removed) calls `vote_code_hash(malicious_hash)`. `CodeHashesVotes` now holds `{E → malicious_hash}`.
3. Resharing completes removing `E`; new participant set is `{A, B, C, D, F}`, threshold = 3. `CLEAN_TEE_STATUS` is scheduled as a detached promise.
4. Before `CLEAN_TEE_STATUS` executes, participants `A` and `B` each call `vote_code_hash(malicious_hash)`.
5. After `A`'s vote: `count_votes` returns 2 (`E`'s stale + `A`'s). Threshold not met.
6. After `B`'s vote: `count_votes` returns 3 (`E`'s stale + `A`'s + `B`'s). Threshold met → `whitelist_tee_proposal(malicious_hash)` executes.
7. `CLEAN_TEE_STATUS` runs and removes `E`'s stale vote — but `malicious_hash` is already permanently whitelisted.
8. Only 2 current participants (`A`, `B`) voted, below the required threshold of 3. [7](#0-6) [8](#0-7) [2](#0-1)

impl CodeHashesVotes {
    /// Casts a vote for the proposal and returns the total number of participants who have voted
    /// for the same code hash. If the participant already voted, their previous vote is replaced.
    pub fn vote(
        &mut self,
        proposal: NodeImageHash,
        participant: &AuthenticatedParticipantId,
    ) -> u64 {
        if self
            .proposal_by_account
            .insert(participant.clone(), proposal)
            .is_some()
        {
            log!("removed old vote for signer");
        }
        let total = self.count_votes(&proposal);
        log!("total votes for proposal: {}", total);
        total
    }

    /// Counts the total number of participants who have voted for the given code hash.
    fn count_votes(&self, proposal: &NodeImageHash) -> u64 {
        self.proposal_by_account
            .values()
            .filter(|&prop| prop == proposal)
            .count() as u64
    }

    /// Clears all proposals.
    pub fn clear_votes(&mut self) {
        self.proposal_by_account.clear();
    }

    /// Returns a new `CodeHashesVotes` containing only votes from current participants.
    pub fn get_remaining_votes(&self, participants: &Participants) -> Self {
        let remaining = self
            .proposal_by_account
            .iter()
            .filter(|(participant_id, _)| {
                participants.is_participant_given_participant_id(&participant_id.get())
            })
            .map(|(participant_id, vote)| (participant_id.clone(), *vote))
            .collect();
        CodeHashesVotes {
            proposal_by_account: remaining,
        }
    }

    pub fn vote_code_hash(&mut self, code_hash: NodeImageHash) -> Result<(), Error> {
        log!(
            "vote_code_hash: signer={}, code_hash={:?}",
            env::signer_account_id(),
            code_hash,
        );
        self.voter_or_panic();

        let threshold_parameters = self.protocol_state.threshold_parameters_or_panic();

        let participant = AuthenticatedParticipantId::new(threshold_parameters.participants())?;
        let votes = self.tee_state.vote(code_hash, &participant);

        let tee_upgrade_deadline_duration =
            Duration::from_secs(self.config.tee_upgrade_deadline_duration_seconds);

        // If the vote threshold is met and the new Docker hash is allowed by the TEE's RTMR3,
        // update the state
        if votes >= self.threshold()?.value() {
            self.tee_state
                .whitelist_tee_proposal(code_hash, tee_upgrade_deadline_duration);
        }

        Ok(())
    }
