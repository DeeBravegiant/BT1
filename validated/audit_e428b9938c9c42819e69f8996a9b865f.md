### Title
Reentrancy in `initTransfer1155()` via Malicious ERC1155 Enables Double-Minting on NEAR — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.initTransfer1155()` makes an external call to a user-supplied ERC1155 contract before completing its own state finalization, with no reentrancy guard. An attacker who controls the ERC1155 contract can reenter `initTransfer1155()` during the `safeTransferFrom` callback, causing two distinct `InitTransfer` events to be emitted while only locking tokens once on EVM. The NEAR bridge processes both events and mints wrapped tokens twice, producing unbacked supply.

---

### Finding Description

`initTransfer1155()` increments `currentOriginNonce` and then immediately calls into the attacker-controlled ERC1155 contract: [1](#0-0) 

```solidity
function initTransfer1155(...) external payable whenNotPaused(PAUSED_INIT_TRANSFER) {
    currentOriginNonce += 1;          // ← only state change before external call
    ...
    IERC1155(tokenAddress).safeTransferFrom(   // ← external call to attacker contract
        msg.sender,
        address(this),
        tokenId,
        amount,
        ""
    );
    ...
    initTransferExtension(...);        // ← emits InitTransfer after returning
    emit BridgeTypes.InitTransfer(...);
}
```

There is no `nonReentrant` modifier anywhere on this function, and no import of `ReentrancyGuardUpgradeable`. [2](#0-1) 

The `onERC1155Received` guard checks `operator == address(this)`, but a malicious ERC1155 contract controls what value it passes as `operator` when it calls `onERC1155Received` on OmniBridge — it can pass `address(OmniBridge)` regardless of who actually initiated the transfer: [3](#0-2) 

This means the guard does not protect against a malicious ERC1155 that fabricates the `operator` argument.

---

### Impact Explanation

**Attack flow:**

1. Attacker deploys a malicious ERC1155 contract `MalERC1155`.
2. Attacker calls `initTransfer1155(MalERC1155, tokenId, amount, ...)` — **outer call**, nonce becomes `N+1`.
3. OmniBridge calls `MalERC1155.safeTransferFrom(attacker, OmniBridge, tokenId, amount, "")`.
4. Inside `MalERC1155.safeTransferFrom`, before returning, the attacker reenters `initTransfer1155(MalERC1155, tokenId, amount, ...)` — **inner call**, nonce becomes `N+2`.
5. Inner call's `safeTransferFrom` actually transfers tokens and calls `OmniBridge.onERC1155Received(OmniBridge, ...)` — passes the operator check. Inner call completes, emitting `InitTransfer` with nonce `N+2`.
6. Control returns to the outer `safeTransferFrom`. It calls `OmniBridge.onERC1155Received(OmniBridge, ...)` with `operator = address(OmniBridge)` (fabricated) — passes the check. No tokens are actually transferred this time.
7. Outer call completes, emitting `InitTransfer` with nonce `N+1`.

**Result:** Two valid, distinct `InitTransfer` events (nonces `N+1` and `N+2`) are emitted. The NEAR bridge processes both and mints wrapped tokens twice. The attacker locked tokens only once on EVM, producing unbacked wrapped supply on NEAR.

This falls squarely under: **High — duplicate cross-chain settlement that produces double-credit or unbacked supply.**

---

### Likelihood Explanation

Any unprivileged user can supply an arbitrary `tokenAddress` to `initTransfer1155()`. The ERC1155 interface is permissionless — no admin approval is required to register a token before calling this function. The attack requires only deploying a malicious ERC1155 contract and calling a public function. Likelihood is **high**.

---

### Recommendation

1. Add `ReentrancyGuardUpgradeable` to `OmniBridge` and apply `nonReentrant` to `initTransfer1155()`, `initTransfer()`, and `finTransfer()`.
2. Follow strict Checks-Effects-Interactions: emit the `InitTransfer` event and call `initTransferExtension` **before** the external `safeTransferFrom` call, or use a reentrancy lock.
3. Do not rely on the `operator` field in `onERC1155Received` as a security gate against malicious ERC1155 contracts — it is attacker-controlled.

---

### Proof of Concept

```solidity
contract MalERC1155 is ERC1155 {
    OmniBridge bridge;
    bool reentered;

    function safeTransferFrom(
        address from, address to, uint256 id, uint256 amount, bytes memory
    ) public override {
        if (!reentered) {
            reentered = true;
            // Inner call: actually transfer tokens, get nonce N+2
            _mint(from, id, amount, "");
            bridge.initTransfer1155(address(this), id, uint128(amount), 0, 0, "attacker.near", "");
        }
        // Outer call: fabricate operator = bridge to pass onERC1155Received check
        // No real token transfer — just call the hook with bridge as operator
        IERC1155Receiver(to).onERC1155Received(address(bridge), from, id, amount, "");
    }
}
```

- Inner call transfers tokens once and emits `InitTransfer` (nonce `N+2`).
- Outer call emits `InitTransfer` (nonce `N+1`) with no token transfer.
- NEAR mints for both nonces → attacker receives 2× wrapped tokens for 1× locked EVM tokens. [4](#0-3)
