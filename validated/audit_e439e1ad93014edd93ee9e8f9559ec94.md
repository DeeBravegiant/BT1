### Title
Missing Predicate Failure for Duplicate `UnRegDepositTxCert` Across Sub-Transactions in a Batch - (File: `eras/dijkstra/impl/testlib/Test/Cardano/Ledger/Dijkstra/Imp/CertSpec.hs`, `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Utxo.hs`)

---

### Summary

The Dijkstra era introduces nested/batch transactions (`sub_transactions`). Multiple sub-transactions within a single top-level transaction can each include an `UnRegDepositTxCert` certificate for the **same** staking credential. No predicate failure has been implemented to prevent this. The developers explicitly acknowledge this gap with a disabled (`xit`) test and a `TODO` comment. This is the direct analog of the Squads batch-ephemeral-key-reuse bug: a shared parent resource (the staking credential's deposit) is referenced by multiple children in a batch without a uniqueness guard.

---

### Finding Description

The Dijkstra era's `UTXO` rule in `dijkstraUtxoTransition` calls `validateBatchWithdrawals` to guard against multiple sub-transactions over-withdrawing from the same account. However, **no equivalent guard exists for deposit refunds** issued via `UnRegDepositTxCert` across sub-transactions. [1](#0-0) 

The `validateBatchWithdrawals` function aggregates all withdrawals across the top-level transaction and all sub-transactions using `Map.unionsWith (<>)`, then checks the total against the **original** account balance. This correctly prevents double-withdrawal. But deposit refunds from `UnRegDepositTxCert` are processed sequentially per sub-transaction inside the `SUBENTITIES` → `SUBCERTS` → `DELEG` rule chain, with no analogous batch-level guard. [2](#0-1) 

The developers themselves document this gap with a disabled test: [3](#0-2) 

The test is marked `xit` (skipped entirely) and the expected predicate failure is `error "TODO: predicate failure not yet implemented"`. This means the check has been identified as necessary but has not been implemented. The transaction may currently **succeed** when two sub-transactions both carry `UnRegDepositTxCert stakingCred keyDeposit` for the same credential, because:

1. The `validateValueNotConservedUTxO` check at line 381 uses the **original** `certState` (before any sub-transactions are applied) and the **top-level** `txBody` only. [4](#0-3) 

2. The `getTotalRefundsTxCerts` for the Dijkstra era delegates to `dijkstraTotalRefundsTxCerts`: [5](#0-4) 

If `dijkstraTotalRefundsTxCerts` counts refunds from sub-transaction certificates against the original cert state, it would count the same deposit refund twice (once per sub-transaction), while the actual deposit pot is only debited once. If it does not count sub-transaction refunds at all, the double refund escapes the value conservation check entirely.

The sub-transactions are stored as `OMap TxId (Tx SubTx era)` keyed by `TxId` (body hash), so two sub-transactions with **different** inputs but the same `UnRegDepositTxCert` certificate have different body hashes and are both admitted into the batch: [6](#0-5) 

---

### Impact Explanation

If both sub-transactions successfully execute `UnRegDepositTxCert` for the same credential, the staking deposit (a `Coin` value locked in the deposit pot) is refunded twice while only being deducted once. This constitutes **direct creation of ADA** — a Critical ledger state transition that violates the global value-conservation invariant. Even in the lower-impact scenario where the second sub-transaction fails at the `DELEG` rule level (credential already unregistered), the transaction fails with an opaque, unintended error rather than the designed predicate failure, causing the entire batch to be rejected and all sub-transaction effects to be rolled back — permanently blocking the batch's execution without a clear recovery path.

---

### Likelihood Explanation

Any unprivileged transaction author can craft a top-level Dijkstra transaction containing two sub-transactions that both carry `UnRegDepositTxCert` for the same staking credential with different spend inputs (to ensure distinct `TxId`s). No special privilege, governance majority, or key compromise is required. The attacker controls the transaction body entirely.

---

### Recommendation

**Short Term**: Implement a batch-level validation analogous to `validateBatchWithdrawals` that collects all `UnRegDepositTxCert` (and `UnRegDRepTxCert`) credentials across all sub-transactions and the top-level transaction, checks for duplicates, and raises a predicate failure if the same credential appears more than once. Enable and complete the disabled test in `CertSpec.hs`.

**Long Term**: Extend the `validateValueNotConservedUTxO` check (or introduce a dedicated batch-level accounting check) to correctly account for the net effect of all sub-transaction certificate refunds and deposits against the original cert state, ensuring the global value-conservation invariant holds across the entire batch atomically.

---

### Proof of Concept

The following scenario, already sketched by the developers in the disabled test, demonstrates the issue:

```haskell
-- stakingCred is registered with deposit = keyDeposit
-- subTx1 and subTx2 have different spend inputs (different TxIds)
-- but both carry UnRegDepositTxCert for the same stakingCred

subTx1 = mkBasicTx mkBasicTxBody
  & bodyTxL . inputsTxBodyL  .~ Set.singleton input1
  & bodyTxL . certsTxBodyL   .~ SSeq.singleton (UnRegDepositTxCert stakingCred keyDeposit)

subTx2 = mkBasicTx mkBasicTxBody
  & bodyTxL . inputsTxBodyL  .~ Set.singleton input2   -- different input → different TxId
  & bodyTxL . certsTxBodyL   .~ SSeq.singleton (UnRegDepositTxCert stakingCred keyDeposit)

tx = mkBasicTx mkBasicTxBody
  & bodyTxL . subTransactionsTxBodyL .~ OMap.fromFoldable [subTx1, subTx2]
``` [3](#0-2) 

Because the two sub-transactions have distinct body hashes (different `input1`/`input2`), the `OMap` deduplication check (keyed on `TxId`) does not reject them: [7](#0-6) 

The `validateBatchWithdrawals` guard does not cover deposit refunds: [8](#0-7) 

No other production rule in `dijkstraUtxoTransition` rejects this batch. The predicate failure that should catch it is explicitly marked as not yet implemented, leaving the batch either accepted with a double refund (Critical: direct ADA creation) or rejected with an opaque internal error (High: permanent batch freeze with no informative recovery path).

-- | For each account, the total withdrawals across the entire batch should not exceed the original account balance.
-- Unregistered accounts are treated as having 0 balance.
validateBatchWithdrawals ::
  ( EraTx era
  , EraAccounts era
  , DijkstraEraTxBody era
  ) =>
  Accounts era ->
  Tx TopTx era ->
  Test (DijkstraUtxoPredFailure era)
validateBatchWithdrawals accounts tx =
  let allWithdrawals =
        Map.unionsWith (<>) $
          unWithdrawals (tx ^. bodyTxL . withdrawalsTxBodyL)
            : [ unWithdrawals $ subTx ^. bodyTxL . withdrawalsTxBodyL
              | subTx <- OMap.elems $ tx ^. bodyTxL . subTransactionsTxBodyL
              ]
      badWithdrawals =
        Map.mapMaybeWithKey
          ( \acctAddr withdrawn ->
              let balance = getAccountBalance acctAddr
               in if withdrawn > balance
                    then Just Mismatch {mismatchSupplied = withdrawn, mismatchExpected = balance}
                    else Nothing
          )
          allWithdrawals
   in failureOnNonEmptyMap badWithdrawals WithdrawalsExceedAccountBalance

dijkstraSubEntitiesTransition = do
  TRC (subCertsEnv, certState, certificates) <- judgmentContext
  let tx = certsTx subCertsEnv
      pp = certsPParams subCertsEnv
      curEpoch = certsCurrentEpoch subCertsEnv
      withdrawals = tx ^. bodyTxL . withdrawalsTxBodyL
      accounts = certState ^. certDStateL . accountsL

  runTest $ Conway.validateWithdrawalsDelegated accounts tx

  network <- liftSTS $ asks networkId
  let (missingWithdrawals, exceededWithdrawals) =
        case withdrawalsThatExceedAccountBalance withdrawals network accounts of
          Nothing -> (Map.empty, Map.empty)
          Just (missing, exceeded) -> (unWithdrawals missing, exceeded)
  failOnNonEmptyMap missingWithdrawals $
    injectFailure . SubWithdrawalsMissingAccounts . Withdrawals . NEM.toMap
  failOnNonEmptyMap exceededWithdrawals $ injectFailure . SubWithdrawalAmountsExceedAccountBalances

  let certStateBeforeSubCerts =
        certState
          & Conway.updateDormantDRepExpiries tx curEpoch
          & Conway.updateVotingDRepExpiries tx curEpoch (pp ^. ppDRepActivityL)
          & certDStateL . accountsL %~ applyWithdrawals withdrawals
  certStateAfterSubCerts <-
    trans @(EraRule "SUBCERTS" era) $ TRC (subCertsEnv, certStateBeforeSubCerts, certificates)

  let directDeposits = tx ^. bodyTxL . directDepositsTxBodyL
      accountsAfterSubCerts = certStateAfterSubCerts ^. certDStateL . accountsL
  failOnJust (directDepositsMissingAccounts directDeposits accountsAfterSubCerts) $
    injectFailure . SubDirectDepositsToMissingAccounts

  pure $ certStateAfterSubCerts & certDStateL . accountsL %~ applyDirectDeposits directDeposits
