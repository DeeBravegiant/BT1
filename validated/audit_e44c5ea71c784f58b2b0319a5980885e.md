Audit Report

## Title
No Staleness Check on Chainlink Price Feed Allows Stale LST Prices to Drive rsETH Over-Minting — (File: `contracts/oracles/ChainlinkPriceOracle.sol`)

## Summary
`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards every return value except `answer`, performing no validation on `updatedAt`, `answeredInRound`, `roundId`, or `price > 0`. If a supported LST's Chainlink feed goes stale while the real market price has fallen, any depositor can call `LRTDepositPool.depositAsset()` and receive rsETH priced at the stale (inflated) rate, diluting all existing rsETH holders' redemption value.

## Finding Description
`ChainlinkPriceOracle.getAssetPrice()` (lines 49–55) fetches only the `answer` field:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

All five return values — `roundId`, `answer`, `startedAt`, `updatedAt`, `answeredInRound` — are available but only `answer` is consumed. No checks are performed for:
- `answeredInRound >= roundId` — incomplete/in-progress round
- `updatedAt > 0` — uninitialized round
- `block.timestamp - updatedAt <= heartbeat` — time-stale price
- `price > 0` — zero or negative price

This price flows directly into `LRTDepositPool.getRsETHAmountToMint()` (line 520):

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

The sibling contract `ChainlinkOracleForRSETHPoolCollateral.getRate()` (lines 30–32) already implements partial guards (`answeredInRound < roundID`, `timestamp == 0`, `ethPrice <= 0`) but still lacks the time-based staleness check. `ChainlinkPriceOracle` has none of these guards at all, and it is the oracle used in the deposit path.

## Impact Explanation
**High — Theft of unclaimed yield.**

When a Chainlink LST/ETH feed goes stale while the real market price has dropped (e.g., an LST depeg), the stale inflated price is used to compute rsETH minted per deposited LST. An attacker depositing `N` units of a depegged LST (real value: `0.95 ETH` each, stale oracle: `1.00 ETH` each) receives rsETH representing `1.00 ETH` per unit of claims against the pool. The excess rsETH minted is a direct dilution claim against all existing rsETH holders' redemption value — equivalent to theft of their accumulated yield.

## Likelihood Explanation
**Medium.** Chainlink feeds have documented heartbeat intervals (e.g., 24 hours for stETH/ETH on mainnet). During network congestion, node failures, or rapid LST depeg events, feeds can remain stale for hours. The attack requires no privileged access — any unprivileged depositor can call `depositAsset()` the moment the feed is stale and the real price has diverged. Historical LST depeg events (stETH in 2022, cbETH volatility) confirm this is a realistic scenario.

## Recommendation
Add staleness and validity checks in `ChainlinkPriceOracle.getAssetPrice()`:

```solidity
(uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
    priceFeed.latestRoundData();

if (answeredInRound < roundId) revert StalePrice();
if (updatedAt == 0) revert IncompleteRound();
if (price <= 0) revert InvalidPrice();
if (block.timestamp - updatedAt > STALENESS_THRESHOLD) revert StalePrice();
```

`STALENESS_THRESHOLD` should be set per-feed based on its documented heartbeat (e.g., 25 hours for a 24-hour heartbeat feed). Additionally, `ChainlinkOracleForRSETHPoolCollateral.getRate()` should be updated to add the time-based staleness check it currently lacks, bringing both contracts to full parity.

## Proof of Concept
1. Chainlink's stETH/ETH feed last updated at `T=0` with price `1.00 ETH`. Feed heartbeat is 24 hours.
2. At `T=12h`, stETH depegs to `0.90 ETH` on the open market. Chainlink nodes fail to push an update due to network congestion.
3. At `T=13h`, attacker calls `LRTDepositPool.depositAsset(stETH, 1000e18, minRsETH, "")`.
4. `_beforeDeposit` → `getRsETHAmountToMint` → `lrtOracle.getAssetPrice(stETH)` → `ChainlinkPriceOracle.getAssetPrice(stETH)` returns `1.00e18` (stale).
5. `rsethAmountToMint = (1000e18 * 1.00e18) / rsETHPrice` — attacker receives rsETH representing `1000 ETH` of claims.
6. Real value deposited: `900 ETH`. The `100 ETH` difference is a dilution claim against all existing rsETH holders' redemption value.
7. Attacker holds rsETH representing more underlying value than deposited, at the expense of existing holders.

**Foundry fork test plan:** Fork mainnet, manipulate `block.timestamp` past the stETH/ETH feed's last `updatedAt` by >24 hours without triggering a feed update (mock the aggregator to return a fixed stale answer), then call `depositAsset` and assert that `rsethAmountToMint` exceeds the fair value computed using the real market price.
