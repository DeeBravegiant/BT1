### Title
Malicious Third-Party Harvester Can Unconditionally Redirect Farmer Block Rewards to Arbitrary Address — (File: chia/farmer/farmer_api.py)

---

### Summary

Under CHIP-22, a third-party harvester connected to a farmer can set `farmer_reward_address_override` in a `RespondSignatures` message to any arbitrary `bytes32` puzzle hash. The farmer unconditionally substitutes this value for its own configured `farmer_target`, causing the farmer block reward (1/8 XCH + transaction fees) to be paid to the harvester's chosen address. The fee-threshold check introduced by CHIP-22 is advisory-only (logging) and does not block or reject the override under any condition.

---

### Finding Description

**Protocol message definitions:**

`RespondSignatures` carries an optional `farmer_reward_address_override` field: [1](#0-0) 

`NewProofOfSpace` carries the same field: [2](#0-1) 

**Step 1 — Harvester sends `NewProofOfSpace` with override set.**

In `FarmerAPI.new_proof_of_space()`, when `farmer_reward_address_override is not None`, the farmer calls `notify_farmer_reward_taken_by_harvester_as_fee()`: [3](#0-2) 

**Step 2 — Fee-threshold check is advisory only.**

`notify_farmer_reward_taken_by_harvester_as_fee()` computes a `fee_quality` and compares it to the harvester-supplied `applied_fee_threshold`. When the threshold is violated it emits a `log.warning`. It does **not** return an error, raise an exception, or abort the block-making flow: [4](#0-3) 

A malicious harvester trivially passes this check by setting `applied_fee_threshold = 0xFFFFFFFF` (maximum uint32), guaranteeing `fee_quality <= fee_threshold` for every proof.

**Step 3 — Farmer unconditionally adopts the override in `_process_respond_signatures()`.**

After the farmer sends `RequestSignatures` and receives `RespondSignatures`, it replaces its own `farmer_target` with whatever `farmer_reward_address_override` the harvester supplies — with no threshold re-check: [5](#0-4) 

**Step 4 — Overridden address flows into `DeclareProofOfSpace` and then into the block.**

The overridden address becomes `farmer_puzzle_hash` in `DeclareProofOfSpace`: [6](#0-5) 

The full node's `declare_proof_of_space` handler uses `request.farmer_puzzle_hash` directly as `farmer_ph` when building the unfinished block: [7](#0-6) 

`farmer_ph` is passed to `create_unfinished_block` → `create_foliage`, which embeds it in `FoliageBlockData.farmer_reward_puzzle_hash`: [8](#0-7) 

This hash is recorded in the `BlockRecord` and used by `validate_block_body` to compute the expected farmer reward coin: [9](#0-8) 

Consensus validates only that the foliage signature matches the plot public key — it does **not** validate that `farmer_reward_puzzle_hash` belongs to the farmer: [10](#0-9) 

**Additional bypass vector:** The harvester can set `farmer_reward_address_override = None` in `NewProofOfSpace` (skipping even the advisory log) and then set it to an attacker address only in `RespondSignatures`. The farmer never calls `notify_farmer_reward_taken_by_harvester_as_fee()` in this case, so there is zero logging of the diversion.

---

### Impact Explanation

Every block won by a plot managed by the malicious harvester pays the 1/8 XCH farmer reward (plus all transaction fees in that block) to the attacker's puzzle hash instead of the farmer's configured address. The farmer receives nothing for those blocks. This is a direct, on-chain, irreversible redirection of XCH. It matches the allowed High impact: *"Bypass of … authorization that enables … payout redirection … affecting XCH."*

---

### Likelihood Explanation

CHIP-22 was explicitly designed to support third-party harvesters, and farmers are expected to connect to them in production. Any operator of a third-party harvester service can perform this attack against every farmer that connects to them. No key compromise, cryptographic break, or host compromise is required — only a network connection that the farmer initiates voluntarily.

---

### Recommendation

1. **Enforce the fee threshold, not just log it.** In `FarmerAPI.new_proof_of_space()`, if `fee_quality > fee_threshold` (or if `fee_info` is absent), drop the proof and do not proceed with block creation.
2. **Re-validate the override in `_process_respond_signatures()`.** Before substituting `farmer_reward_address_override`, confirm that the same override was declared in the corresponding `NewProofOfSpace` and that the fee threshold was satisfied. Reject the `RespondSignatures` message if the override appears for the first time here.
3. **Optionally, allow farmers to configure a maximum acceptable fee rate** and refuse connections from harvesters that exceed it.

---

### Proof of Concept

```
Attacker operates a third-party harvester service.

1. Farmer connects to attacker's harvester (normal CHIP-22 usage).

2. Harvester finds a valid proof of space for a signage point.

3. Harvester sends NewProofOfSpace to farmer:
     farmer_reward_address_override = None   # skip advisory log
     fee_info = None

4. Farmer sends RequestSignatures back to harvester.

5. Harvester responds with RespondSignatures:
     farmer_reward_address_override = <attacker_puzzle_hash>
     # No fee_info check is performed here

6. FarmerAPI._process_respond_signatures() executes:
     farmer_reward_address = self.farmer.farmer_target          # farmer's address
     if response.farmer_reward_address_override is not None:
         farmer_reward_address = response.farmer_reward_address_override  # attacker's address
     # No threshold check, no rejection

7. DeclareProofOfSpace is sent to full node with
     farmer_puzzle_hash = <attacker_puzzle_hash>

8. Full node creates block; farmer reward coin (1/8 XCH + fees)
   is created at <attacker_puzzle_hash>.

9. Farmer receives 0 XCH for the block they won.
   Attacker receives the full farmer reward.
```
