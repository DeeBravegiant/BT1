### Title
Missing Invariant: Multiple Sub-Transactions Can Claim the Same Deposit Refund in Dijkstra Era — (`File: eras/dijkstra/impl/testlib/Test/Cardano/Ledger/Dijkstra/Imp/CertSpec.hs`)

---

### Summary

In the Dijkstra era, the ledger lacks an explicit invariant preventing multiple sub-transactions within a single top-level transaction from each claiming the deposit refund for the same stake credential deregistration. The developers have explicitly acknowledged this gap with a disabled (`xit`) test and a `TODO: predicate failure not yet implemented` comment. The `dijkstraTotalRefundsTxCerts` function reads refund amounts directly from certificate fields without cross-checking uniqueness of credential deregistrations across sub-transactions.

---

### Finding Description

The Dijkstra era introduces nested ("sub") transactions. A top-level transaction may contain an `OMap` of sub-transactions, each of which is processed sequentially through the `SUBLEDGERS` → `SUBLEDGER` → `SUBENTITIES` → `SUBCERTS` → `SUBCERT` → `SUBDELEG` pipeline.

**Root cause 1 — `dijkstraTotalRefundsTxCerts` reads refunds from certificate fields, not ledger state:** [1](#0-0) 

The comment on line 228 explicitly states: *"Unlike previous eras, we no longer need to lookup refunds from the ledger state, since all of the certificates specify the actual refund."* This means the UTxO balance check for a sub-transaction credits the refund amount stated in the `UnRegDepositTxCert` certificate field, regardless of whether the credential was already deregistered by a prior sub-transaction.

**Root cause 2 — `getTotalRefundsTxBody` for Dijkstra only looks at the top-level tx body's certs, not sub-transaction certs:** [2](#0-1) 

The `getTotalRefundsTxBody` implementation only folds over `txBody ^. certsTxBodyL` (the top-level body's certs), not the certs of each sub-transaction. Sub-transaction refunds are accounted for individually inside each `SUBLEDGER` invocation via `updateUTxOStateNoFees`: [3](#0-2) 

**Root cause 3 — No cross-sub-transaction deduplication check exists; the test for it is explicitly disabled:** [4](#0-3) 

The test constructs two sub-transactions (`subTx1`, `subTx2`) that each carry `UnRegDepositTxCert stakingCred keyDeposit` for the **same** `stakingCred`. It expects the top-level transaction to be rejected, but the expected predicate failure is `error "TODO: predicate failure not yet implemented"` — meaning the rejection logic has not been written. The test is disabled with `xit` precisely because the protection does not exist.

**Processing order in `dijkstraLedgerTransition`:** [5](#0-4) 

All sub-transactions are processed first via `SUBLEDGERS`. Within each `SUBLEDGER` invocation, `SUBENTITIES` (certificates) runs before `SUBUTXOW` (UTxO balance). However, the `SUBUTXOW` call passes `certState` — the state **before** the current sub-transaction's certificates — to the UTxO environment: [6](#0-5) 

Because `dijkstraTotalRefundsTxCerts` reads the refund from the certificate field (not from `certState`), the UTxO balance check for the second sub-transaction credits `keyDeposit` as a consumed refund even when the credential is no longer present in `certState`. The `SUBDELEG` rule may reject the second deregistration attempt with a generic "credential not registered" failure, but no dedicated invariant check exists to detect and reject the double-refund pattern as a distinct protocol violation — which is exactly what the disabled test acknowledges.

---

### Impact Explanation

If the `SUBDELEG` rule for sub-transactions does not properly reject deregistration of an already-deregistered credential (which the disabled test implies is the case), an attacker can submit a single top-level transaction containing N sub-transactions each carrying `UnRegDepositTxCert cred deposit` for the same credential. Each sub-transaction's UTxO balance check credits `deposit` as consumed value, allowing the attacker to extract `N × deposit` lovelace from the deposit pot while only one deposit was ever paid. This constitutes **direct loss of ADA through an invalid ledger state transition** — the deposit pot (`utxosDeposited`) would be decremented by more than was ever deposited for that credential.

---

### Likelihood Explanation

The Dijkstra era is the current development frontier. The disabled test and explicit TODO comment confirm the developers are aware of the scenario but have not implemented the guard. Any user who can submit a Dijkstra-era transaction (an unprivileged transaction sender) can construct this pattern. No special keys, governance majority, or privileged access is required.

---

### Recommendation

1. **Short term:** Re-enable the test `"Multiple subtransactions cannot get the same refund"` and implement the corresponding predicate failure. Add a cross-sub-transaction deduplication check in the `SUBLEDGERS` rule that tracks which credentials have been deregistered across all sub-transactions and rejects any second deregistration of the same credential within the same top-level transaction.

2. **Long term:** Extend the Dijkstra-era simulation/property tests to assert the invariant: `total deposit refunds claimed across all sub-transactions ≤ total deposits held for those credentials before the transaction`. This mirrors the Umee recommendation to assert `tokenSupply != 0` when `uTokenSupply != 0`.

---

### Proof of Concept

The existing (disabled) test in the repository directly demonstrates the attack vector:

```haskell
-- eras/dijkstra/impl/testlib/Test/Cardano/Ledger/Dijkstra/Imp/CertSpec.hs, lines 53-75
xit "Multiple subtransactions cannot get the same refund" $ do
  stakingCred <- KeyHashObj <$> freshKeyHash
  _ <- registerStakeCredential stakingCred   -- pays keyDeposit once
  keyDeposit <- getsPParams ppKeyDepositL
  ...
  let
    subTx1 = mkBasicTx mkBasicTxBody
      & bodyTxL . inputsTxBodyL .~ Set.singleton input1
      & bodyTxL . certsTxBodyL  .~ SSeq.singleton (UnRegDepositTxCert stakingCred keyDeposit)
    subTx2 = mkBasicTx mkBasicTxBody
      & bodyTxL . inputsTxBodyL .~ Set.singleton input2
      & bodyTxL . certsTxBodyL  .~ SSeq.singleton (UnRegDepositTxCert stakingCred keyDeposit)
    tx = mkBasicTx mkBasicTxBody
      & bodyTxL . subTransactionsTxBodyL .~ OMap.fromFoldable [subTx1, subTx2]
  submitFailingTx tx . NE.singleton $ error "TODO: predicate failure not yet implemented"
  -- ^^^ disabled because the rejection predicate does not exist
``` [4](#0-3) 

Both `subTx1` and `subTx2` carry `UnRegDepositTxCert stakingCred keyDeposit`. Only one deposit was ever paid. The test expects rejection but the predicate failure is explicitly marked as not yet implemented, confirming the invariant is unenforced.
