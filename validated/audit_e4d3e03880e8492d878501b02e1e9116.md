Audit Report

## Title
Swap Conservation Failure in `buyToken0InBinSpecifiedIn`: Forced `totalIn1Scaled` Assignment Without Updating `out0Scaled` Causes Trader to Overpay — (`metric-core/contracts/libraries/SwapMath.sol`)

## Summary

In `SwapMath.buyToken0InBinSpecifiedIn`, when two consecutive refinement passes both undershoot (`totalIn1Scaled < amountSpecifiedRemainingScaled && targetPos < maxFinalBinPos`), a fallback at line 621 forces `totalIn1Scaled = amountSpecifiedRemainingScaled` without updating `out0Scaled`. The subsequent rescaling guard at line 624 requires a strict `>` comparison, which is now false (equality), so it is skipped. Settlement then debits the full input from the trader but credits only the stale, smaller `out0Scaled`, with the excess token1 silently absorbed by the bin.

## Finding Description

The function operates in three stages after the initial affordability check:

**Stage 2 — analytical + proportional refinement (lines 581–618):** An analytical `targetPos` is computed, `out0Scaled` and `totalIn1Scaled` are evaluated, and if still undershooting, a proportional scale-up is applied and both values are recomputed. [1](#0-0) 

**Stage 3 — forced assignment (lines 620–622):** If the second evaluation still undershoots, only `totalIn1Scaled` is overwritten; `out0Scaled` retains the value computed for the smaller `targetPos`. [2](#0-1) 

**Stage 4 — rescaling guard (lines 624–634):** This block, which proportionally adjusts `out0Scaled` to match the capped input, fires only on strict `>`. After the forced assignment the two values are equal, so the block is entirely skipped and `out0Scaled` remains stale. [3](#0-2) 

**Settlement (lines 639–644):** `binState.token0BalanceScaled` is reduced by the stale (smaller) `out0Scaled`, while `binState.token1BalanceScaled` is increased by the full `totalIn1Scaled`. The trader's `amountSpecifiedRemainingScaled` is fully consumed, but `amountCalculatedScaled` records only the smaller output. [4](#0-3) 

The excess token1 (`amountSpecifiedRemainingScaled − actual_cost_for_targetPos`) is permanently credited to the bin, accruing to LPs at the trader's expense.

## Impact Explanation

This is a direct swap conservation failure: the trader pays the full exact-input amount but receives less token0 than the bin curve permits for that payment. The shortfall is not returned; it is credited to the bin's token1 balance, permanently benefiting LPs. The swap callback receives the correct `amountIn` debit but a reduced `amountOut` credit. This constitutes a direct loss of user principal meeting Sherlock High/Medium thresholds whenever the discrepancy is non-trivial.

## Likelihood Explanation

The condition requires two consecutive undershoots of the refinement loop. This occurs when the price curve is sufficiently non-linear relative to the linear proportional scaling — e.g., wide bins, large `currBinPos` offsets, or extreme price ratios. No privileged role is required. The reachable call path is `swap` → `_swapToken1ForToken0SpecifiedInput` → `buyToken0InBinSpecifiedIn`. [2](#0-1) 

## Recommendation

After the forced assignment at line 621, recompute `out0Scaled` consistently with `totalIn1Scaled`. The simplest fix is to treat the forced case identically to the overshoot case — apply the proportional rescaling of `out0Scaled` before settling:

```solidity
if (totalIn1Scaled < state.amountSpecifiedRemainingScaled && targetPos < maxFinalBinPos) {
    uint256 delta = targetPos - currBinPos;
    uint256 scaledDelta = Math.ceilDiv(delta * state.amountSpecifiedRemainingScaled, totalIn1Scaled);
    if (scaledDelta == 0) scaledDelta = 1;
    uint256 scaledTarget = currBinPos + scaledDelta;
    targetPos = scaledTarget > maxFinalBinPos ? maxFinalBinPos : scaledTarget;
    out0Scaled = calculateOutputToken0FromBinPosition(binState.token0BalanceScaled, currBinPos, targetPos);
    totalIn1Scaled = state.amountSpecifiedRemainingScaled;
}
```

Alternatively, add a third refinement pass or clamp `targetPos = maxFinalBinPos` and let the existing Stage 4 overshoot rescaling handle it correctly.
