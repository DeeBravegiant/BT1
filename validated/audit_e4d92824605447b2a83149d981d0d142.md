Audit Report

## Title
Deposit-Before-Price-Update Arbitrage via Public `updateRSETHPrice()` and `instantWithdrawal()` - (File: contracts/LRTOracle.sol, contracts/LRTWithdrawalManager.sol)

## Summary
`LRTOracle.updateRSETHPrice()` is a permissionless public function that any caller can invoke. An attacker can atomically deposit ETH at a stale (lower) `rsETHPrice`, trigger the price update to reflect accrued rewards, and immediately call `instantWithdrawal()` at the elevated price — extracting more assets than deposited and stealing yield from all existing rsETH holders. The profit is mathematically provable and positive whenever rewards have accrued since the last price update.

## Finding Description

**Root cause:** `rsETHPrice` is a stored value updated only on explicit calls to `updateRSETHPrice()`. Between updates, EigenLayer rewards cause the true backing per rsETH to exceed the stored price. Because `updateRSETHPrice()` is unrestricted (`public whenNotPaused`), any caller can trigger it at will.

**Deposit path** mints rsETH using the current stored `rsETHPrice`:

```solidity
// LRTDepositPool.sol:520
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

A lower stale `rsETHPrice` yields more rsETH for the same deposit.

**Instant withdrawal path** redeems assets using the current stored `rsETHPrice` at withdrawal time:

```solidity
// LRTWithdrawalManager.sol:593
underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset);
```

A higher updated `rsETHPrice` yields more assets for the same rsETH.

**Mathematical proof of profit:**

Let `T` = true total ETH in protocol (with accrued rewards), `S` = current rsETH supply, `P_stale` = stale stored price, where `P_true = T/S > P_stale`.

- Attacker deposits `D` ETH → mints `D/P_stale` rsETH (inflated)
- After deposit: `totalETH = T + D`, `rsethSupply = S + D/P_stale`
- After `updateRSETHPrice()`: `newPrice = (T + D) · P_stale / (S · P_stale + D)`
- Since `T > S · P_stale`, `newPrice > P_stale` ✓
- Attacker withdraws `D/P_stale` rsETH → receives `D · (T + D) / (S · P_stale + D)` ETH
- **Profit = `D · (T − S · P_stale) / (S · P_stale + D)` > 0** ✓

**Why the `pricePercentageLimit` guard does not prevent this:**

The guard at `LRTOracle.sol:252-266` only triggers when `newRsETHPrice > highestRsethPrice`. The attacker's deposit actually *dilutes* the price increase (inflated rsETH supply reduces the per-token price), so `newRsETHPrice` is always below `P_true`. If `highestRsethPrice` was previously set to `P_true`, then `newRsETHPrice < highestRsethPrice` and the guard never fires. If `highestRsethPrice = P_stale`, the increase is smaller than it would have been without the attack, making it less likely to exceed the threshold. In either case, the guard is bypassed.

**Why the burn does not prevent this:**

`instantWithdrawal()` correctly burns the attacker's rsETH at line 229. However, the profit comes from the price delta between deposit and withdrawal, not from unburned tokens. The burn is irrelevant to the arbitrage.

**All three steps execute atomically in one transaction** across three separate contracts (`LRTDepositPool`, `LRTOracle`, `LRTWithdrawalManager`), each with independent `nonReentrant` guards that do not interfere with each other.

## Impact Explanation

**High — Theft of unclaimed yield.**

The attacker extracts yield that should have been distributed pro-rata to all existing rsETH holders. Concretely: with `T = 101 ETH`, `S = 100 rsETH`, `P_stale = 1.00e18`, and `D = 100 ETH`, the attacker mints 100 rsETH, the price updates to 1.005e18, and the attacker withdraws 100.5 ETH — a profit of 0.5 ETH minus the `instantWithdrawalFee`. This profit is extracted directly from the yield that should have increased the price for all existing holders. The attack is repeatable across every oracle update cycle.

The `instantWithdrawalFee` (capped at 10% by `setInstantWithdrawalFee`) reduces but does not eliminate the profit when the fee is set below the reward accrual rate, which is the normal operational state.

## Likelihood Explanation

**Medium.** Prerequisites that must hold simultaneously:
- `isInstantWithdrawalEnabled[asset]` is `true` (operator-controlled flag, normal when instant withdrawals are active)
- `LRTUnstakingVault` holds sufficient assets for instant withdrawal (normal operational state)
- `rsETHPrice` is stale — rewards have accrued since the last `updateRSETHPrice()` call (the normal state between oracle updates)
- `instantWithdrawalFee` is less than the pending reward accrual percentage

All conditions are routinely satisfied during normal protocol operation. The attack requires no privileged access and is fully atomic. It can be repeated every oracle update cycle.

## Recommendation

1. **Update `rsETHPrice` atomically inside `depositETH`/`depositAsset`** before computing the rsETH mint amount, so deposits always use the freshest price.
2. **Alternatively**, apply the same min-of-two-prices logic used in `_calculatePayoutAmount` for queued withdrawals to `instantWithdrawal` — record the `rsETHPrice` at deposit time and use the minimum of the deposit-time price and the withdrawal-time price when computing the instant withdrawal payout.
3. **Ensure `instantWithdrawalFee` is set high enough** to exceed the maximum possible single-update reward accrual rate as a defense-in-depth measure.

## Proof of Concept

```solidity
// Attacker contract — executes atomically in one transaction
function attack(
    ILRTDepositPool depositPool,
    ILRTOracle oracle,
    ILRTWithdrawalManager withdrawalManager,
    address rsETHToken,
    address asset
) external payable {
    uint256 depositAmount = msg.value;

    // Step 1: Deposit at stale (low) rsETHPrice — receive inflated rsETH
    depositPool.depositETH{value: depositAmount}(0, "arb");

    // Step 2: Trigger price update — rsETHPrice rises to reflect accrued rewards
    oracle.updateRSETHPrice();

    // Step 3: Instantly withdraw at the new (higher) rsETHPrice
    uint256 rsETHBalance = IERC20(rsETHToken).balanceOf(address(this));
    IERC20(rsETHToken).approve(address(withdrawalManager), rsETHBalance);
    withdrawalManager.instantWithdrawal(asset, rsETHBalance, "arb");

    // address(this).balance > depositAmount — profit extracted from other holders
    require(address(this).balance > depositAmount, "no profit");
}
```

**Foundry fork test plan:**
1. Fork mainnet at a block where `rsETHPrice` is stale (rewards have accrued).
2. Deploy the attacker contract.
3. Call `attack()` with a meaningful ETH amount.
4. Assert `address(attacker).balance > depositAmount` after the call.
5. Assert that existing rsETH holders' effective price per token is lower than it would have been without the attack (compare `rsETHPrice` after attack vs. `P_true = totalETH/rsethSupply` computed before the attack).
