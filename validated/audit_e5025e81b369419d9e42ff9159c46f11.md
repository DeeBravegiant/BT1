Audit Report

## Title
`LRTOracle.updateRSETHPrice()` DoS via shared `RSETH.checkDailyMintLimit` exhausted by user deposits - (File: contracts/LRTOracle.sol)

## Summary
`LRTOracle._updateRsETHPrice()` has its own oracle-level fee mint limit (`_checkAndUpdateDailyFeeMintLimit` using `currentPeriodMintedFeeAmount`/`maxFeeMintAmountPerDay`), but still calls `IRSETH.mint()` which enforces a second, shared `checkDailyMintLimit` counter in `RSETH.sol`. User deposits via `LRTDepositPool._mintRsETH()` consume the same shared RSETH-level counter. Once ordinary depositors exhaust `maxMintAmountPerDay` in `RSETH`, the oracle's fee-mint call reverts, making `updateRSETHPrice()` uncallable for the remainder of the 24-hour window and leaving `rsETHPrice` stale.

## Finding Description
`LRTOracle` declares its own fee-period variables (`currentPeriodMintedFeeAmount`, `feePeriodStartTime`, `maxFeeMintAmountPerDay` at lines 33–35) and calls `_checkAndUpdateDailyFeeMintLimit` (line 303) before minting. This oracle-level guard is independent and does not prevent the revert. The actual mint at line 306 routes through `IRSETH(rsETHTokenAddress).mint(treasury, rsethAmountToMintAsProtocolFee)`, which is `RSETH.mint()` (lines 229–240). That function carries the `checkDailyMintLimit(amount)` modifier (line 236), which reads and writes the shared state variables `currentPeriodMintedAmount` and `maxMintAmountPerDay` (lines 50–54). `LRTDepositPool._mintRsETH()` (lines 686–690) calls the identical `IRSETH(rsethToken).mint(msg.sender, rsethAmountToMint)`, incrementing the same `currentPeriodMintedAmount`. When depositors collectively push `currentPeriodMintedAmount` to `maxMintAmountPerDay`, any subsequent call to `RSETH.mint()` — including the oracle's fee mint — hits the `DailyMintLimitExceeded` revert (line 51), which propagates through `_updateRsETHPrice()` and out of the public `updateRSETHPrice()`. The oracle's own fee-limit check at line 303 passes fine; it is the RSETH-level shared counter that causes the revert. The existing guard is therefore insufficient because it operates on a different counter than the one that actually blocks execution.

## Impact Explanation
`updateRSETHPrice()` is the sole public mechanism for refreshing `rsETHPrice`. While the DoS lasts (up to 24 hours), the stored exchange rate is stale. All subsequent depositors receive rsETH amounts computed from the outdated price via `getRsETHAmountToMint`, and any withdrawal valuation dependent on the oracle is similarly affected. Funds are not permanently locked, but the contract fails to deliver its core promised return — accurate, up-to-date rsETH/ETH pricing — for the duration of the window. This matches **Low: Contract fails to deliver promised returns, but doesn't lose value**.

## Likelihood Explanation
`updateRSETHPrice()` is a permissionless public function. Any combination of ordinary depositors reaching `maxMintAmountPerDay` triggers the condition; no special privilege is required. The attacker receives rsETH in exchange for their deposit, so the direct cost is only opportunity cost of capital. On a chain with high deposit volume or a conservatively set cap, this condition can be reached organically without coordination. The condition is repeatable every 24-hour window.

## Recommendation
Decouple the oracle's fee-minting path from the shared user-deposit counter. Options include: (1) adding a separate `mintFee` function on `RSETH` that carries its own independent daily limit and is callable only by `LRTOracle`; (2) catching the `DailyMintLimitExceeded` revert inside `_updateRsETHPrice()` with a try/catch and continuing with the price update while deferring the fee mint; or (3) moving fee accrual to a lazy-claim model that does not require an immediate mint during the price update call.

## Proof of Concept
1. `RSETH.maxMintAmountPerDay` is set to some finite value `M`.
2. Users (or a single attacker) call `LRTDepositPool.depositETH{value: X}()` repeatedly until `currentPeriodMintedAmount == M`. Each call routes through `_mintRsETH` → `RSETH.mint` → `checkDailyMintLimit`, incrementing the shared counter.
3. TVL has grown since the last price update, so `protocolFeeInETH > 0` when `updateRSETHPrice()` is next called.
4. `_updateRsETHPrice()` passes the oracle-level `_checkAndUpdateDailyFeeMintLimit` check (its own counter has room), then reaches `IRSETH(rsETHTokenAddress).mint(treasury, rsethAmountToMintAsProtocolFee)` at line 306.
5. `RSETH.mint` executes `checkDailyMintLimit(rsethAmountToMintAsProtocolFee)`: `currentPeriodMintedAmount + fee > M` → `revert DailyMintLimitExceeded(...)`.
6. The revert propagates; `updateRSETHPrice()` reverts. `rsETHPrice` is not updated.
7. All subsequent deposits for the remainder of the 24-hour window use the stale price, minting incorrect rsETH amounts.
