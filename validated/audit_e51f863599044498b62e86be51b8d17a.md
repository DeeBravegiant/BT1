Audit Report

## Title
Stale `rsETHPrice` Used for rsETH Minting While Oracle Is Paused and After Unpause — (`contracts/LRTOracle.sol`)

## Summary

`LRTOracle` stores `rsETHPrice` as a cached state variable updated only via `updateRSETHPrice()`, which is blocked by `whenNotPaused`. The `PAUSER_ROLE` can pause the oracle without pausing `LRTDepositPool`, leaving deposits active against a frozen price. After unpause, `unpause()` does not refresh the price. Any depositor who deposits while the price is stale-low (real price has risen) receives excess rsETH, directly diluting the redemption value of all existing rsETH holders — constituting theft of unclaimed yield.

## Finding Description

`LRTOracle` holds a cached price: [1](#0-0) 

The only public update path is blocked during a pause: [2](#0-1) 

`PAUSER_ROLE` pauses only the oracle — the deposit pool is untouched: [3](#0-2) 

`unpause()` clears the flag but never refreshes the price: [4](#0-3) 

`LRTDepositPool.getRsETHAmountToMint()` divides by the raw cached value with no staleness check: [5](#0-4) 

`getAssetPrice()` carries no `whenNotPaused` guard either, so neither leg of the division is protected: [6](#0-5) 

The deposit pool's `depositETH` / `depositAsset` check only their own `PausableUpgradeable` state, which is independent of the oracle's `paused` flag: [7](#0-6) 

There are therefore **two** stale-price windows:

1. **During the oracle pause** — the deposit pool is fully active; every deposit uses the frozen `rsETHPrice`.
2. **After oracle unpause, before `updateRSETHPrice()` is called** — the price remains frozen until someone explicitly calls the update function.

A secondary aggravating factor: if the price moved significantly during the pause, a non-manager caller's `updateRSETHPrice()` will revert due to the `PriceAboveDailyThreshold` check, extending the stale window until a manager acts: [8](#0-7) 

## Impact Explanation

`getRsETHAmountToMint` computes `rsethAmountToMint = (amount × assetPrice) / rsETHPrice`. When `rsETHPrice` is stale-low (the real price has risen while the oracle was paused), every depositor receives more rsETH than the current TVL justifies. The excess rsETH inflates total supply without a corresponding increase in backing assets, reducing the redemption value of every existing rsETH holder's position. This is a direct, on-chain transfer of accrued yield from existing holders to the depositor.

**Impact: High — Theft of unclaimed yield.**

## Likelihood Explanation

- `PAUSER_ROLE` is a distinct role that can pause the oracle without touching the deposit pool; no admin error or collusion is required.
- Oracle pauses are an expected operational event (emergency response, anomalous price feed).
- The stale-price window during the pause is open for every block the oracle remains paused.
- After unpause, the window persists until `updateRSETHPrice()` succeeds; if the price moved beyond `pricePercentageLimit`, only a manager can close it, potentially extending the window by hours.
- Any depositor can exploit the window by simply calling `depositETH` or `depositAsset` — no special role required.

**Likelihood: Medium.**

## Recommendation

1. In `LRTOracle.unpause()`, call `_updateRsETHPrice()` before clearing the paused flag so the price is always fresh the moment deposits can resume:

```solidity
function unpause() external whenPaused onlyLRTAdmin {
    _updateRsETHPrice();   // refresh before re-opening
    paused = false;
    emit Unpaused(msg.sender);
}
```

2. To close the during-pause window, add a `whenNotPaused` guard to `getRsETHAmountToMint()` in `LRTDepositPool`, or enforce that pausing the oracle always co-pauses the deposit pool (mirror the auto-pause logic already present in `_updateRsETHPrice()`).

3. Consider adding a staleness timestamp to `rsETHPrice` and reverting in `getRsETHAmountToMint()` if the price is older than a configurable threshold.

## Proof of Concept

**Minimal call sequence (no fork required):**

1. Deploy protocol; `rsETHPrice = 1.1e18` at block N.
2. `PAUSER_ROLE` calls `LRTOracle.pause()`. `LRTDepositPool` is **not** paused.
3. Blocks N+1 … N+1000 pass; underlying LST rewards accrue. Real rsETH/ETH rate is now `1.2e18`, but `rsETHPrice` is frozen at `1.1e18`.
4. *(Optional — demonstrates the during-pause window)* Attacker calls `LRTDepositPool.depositETH{value: 1 ether}(0, "")`. `getRsETHAmountToMint` returns `1e18 × 1e18 / 1.1e18 ≈ 0.909e18` rsETH. Correct amount: `≈ 0.833e18`. Attacker receives ~9% excess.
5. `LRTAdmin` calls `LRTOracle.unpause()`. `rsETHPrice` is still `1.1e18`.
6. Attacker (or any depositor) immediately calls `depositETH{value: 1 ether}(0, "")` again — same excess minting.
7. Only after a separate `updateRSETHPrice()` call (which may revert for non-managers if price moved beyond `pricePercentageLimit`) does `rsETHPrice` correct to `1.2e18`.

**Foundry invariant test sketch:**

```solidity
function invariant_rsETHPriceNotStaleOnDeposit() public {
    // After any oracle unpause, assert rsETHPrice == freshly computed price
    // before any deposit is processed.
    uint256 freshPrice = oracle.computeCurrentPrice(); // off-chain equivalent
    assertApproxEqRel(oracle.rsETHPrice(), freshPrice, 1e15); // 0.1% tolerance
}
```
