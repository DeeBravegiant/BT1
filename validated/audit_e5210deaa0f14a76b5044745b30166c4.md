### Title
Unbounded Follower Task Fan-Out in `monitor_passive_channels_inner` Enables Byzantine Participant to Starve Signing Requests — (File: crates/node/src/mpc_client.rs)

### Summary
A Byzantine registered MPC participant strictly below the signing threshold can flood honest nodes with arbitrary P2P `Start` messages for fake signing task channels. Because `monitor_passive_channels_inner` spawns one tokio task per incoming channel with no admission cap, the honest node's main MPC runtime is exhausted with spurious follower tasks, starving legitimate signing requests and breaking the request lifecycle.

### Finding Description
In `crates/node/src/mpc_client.rs`, the function `monitor_passive_channels_inner` receives incoming `NetworkTaskChannel` objects from the P2P network and unconditionally spawns a new tokio task for each one:

```rust
// crates/node/src/mpc_client.rs lines 652–667
async fn monitor_passive_channels_inner(
    mut channel_receiver: mpsc::UnboundedReceiver<NetworkTaskChannel>,
    mpc_client: Arc<MpcClient<ForeignChainPolicyReader>>,
) -> anyhow::Result<()> {
    let mut tasks = AutoAbortTaskCollection::new();
    while let Some(channel) = channel_receiver.recv().await {
        let task = async move { mpc_clone.process_channel_task(channel).await };
        if is_heavy_generation_task(&task_id) {
            tasks.spawn_checked_on(&mpc_client.gen_runtime_handle, &description, task);
        } else {
            tasks.spawn_checked(&description, task);  // ← no cap
        }
    }
    ...
}
```

Non-heavy tasks — signing (`EcdsaTaskId::Signature`, `EddsaTaskId::Signature`), CKD, and foreign-tx verification — are dispatched to the main MPC runtime with **no per-peer or global admission limit**.

New channels are created in `run_receive_message` → `sender_for` whenever a `MpcMessageKind::Start` message arrives for an unknown `ChannelId`:

```rust
// crates/node/src/network.rs lines 384–398
PeerMessage::Mpc(message) => {
    match client.sender_for(channel_id, start_msg, message.from) {
        SenderOrNewChannel::NewChannel(channel) => {
            new_channel_sender.send(channel)?;   // feeds monitor_passive_channels_inner
        }
        ...
    }
}
```

The `channels_waiting_for_start` LRU cache (capacity 10 000) only buffers channels that have not yet received their Start message. When the attacker sends a Start message for a fresh `ChannelId`, the incomplete channel is immediately popped from the LRU and a live `NetworkTaskChannel` is emitted — bypassing the LRU bound entirely. The `senders` `HashMap` inside `NetworkTaskChannelManager` is also unbounded and grows with each new channel until the spawned task drops it.

The project's own design document explicitly acknowledges this root cause:

> **Unbounded follower fan-out.** `mpc_client.rs::monitor_passive_channels_inner` spawns one task per incoming peer channel with no cap, so a node has no way to bound how much follower work peers can push onto it.
> — `docs/design/signing-starvation-solution.md`, root cause item 3

The partial mitigation applied (routing heavy generation tasks to a lower-priority `gen_runtime`) only separates CPU priority between generation and signing. It does **not** cap the number of tasks spawned, leaving the main runtime fully exposed to flooding with fake signing follower channels.

### Impact Explanation
A Byzantine participant below threshold can sustain an unbounded stream of fake `EcdsaTaskId::Signature` / `EddsaTaskId::Signature` Start messages. Each message causes the honest node to:
1. Allocate a new `NetworkTaskChannel` (unbounded sender + receiver pair in the `senders` HashMap).
2. Spawn a new tokio task on the main MPC runtime that calls `process_channel_task` → `ecdsa_signature_provider.process_channel(channel)`, consuming CPU and memory until the task times out.

With enough concurrent fake tasks, the main runtime's thread pool is saturated. Legitimate signing requests — which also run on the main runtime — are starved and time out. Users whose cross-chain transactions depend on MPC signatures receive no response; funds awaiting a signature to complete a bridge operation are effectively frozen for the duration of the attack. This breaks the request-lifecycle safety invariant of the MPC network without requiring threshold-level collusion or network-level DoS.

### Likelihood Explanation
Any single registered MPC participant can execute this attack. The TLS P2P mesh authenticates participants but does not rate-limit or validate the semantic content of Start messages. The attacker needs only to generate fresh `ChannelId` values (trivially unique 256-bit identifiers) and send Start messages at the rate the network allows. No special key material, threshold collusion, or operator misconfiguration is required. The attack can be sustained indefinitely as long as the Byzantine participant remains registered.

### Recommendation
Implement per-peer admission control for follower tasks in `monitor_passive_channels_inner`, as described in the design document (Option B):

- Use a non-blocking `try_admit(leader_id) -> Option<Permit>` mechanism that caps concurrent follower tasks per peer.
- On unavailable permit, drop the incoming channel immediately and let the leader time out and retry with different participants.
- The cap must be **per-peer** (not a single global semaphore) to avoid the threshold-N circular-wait deadlock described in the design document.

Additionally, bound the `senders` HashMap in `NetworkTaskChannelManager` or enforce a per-peer channel creation rate limit in `run_receive_message` to prevent unbounded memory growth from the `senders` map itself.

### Proof of Concept
1. Byzantine participant B is a registered MPC participant (below signing threshold).
2. B connects to honest node A via the authenticated TLS P2P mesh.
3. B generates a large number of fresh `ChannelId` values and sends `MpcMessage { channel_id, kind: MpcMessageKind::Start(MpcStartMessage { task_id: EcdsaTaskId::Signature { id: <fresh_hash>, presignature_id: <uid> }, participants: [...] }) }` to A in rapid succession.
4. For each message, `run_receive_message` on A calls `sender_for(channel_id, Some(start_msg), B)`. Since each `channel_id` is new, a fresh sender/receiver pair is created, the incomplete channel is inserted into the LRU cache, and immediately popped (because the Start message is present), producing a live `NetworkTaskChannel`.
5. Each `NetworkTaskChannel` is sent to `monitor_passive_channels_inner` via `new_channel_sender`.
6. `monitor_passive_channels_inner` spawns a new task on the main runtime for each channel (non-heavy task path).
7. The main runtime is flooded with fake signing follower tasks. Each task calls `ecdsa_signature_provider.process_channel(channel)`, consuming CPU and memory until it times out.
8. Legitimate signing requests submitted by real users are starved; they time out and return no signature. Cross-chain transactions depending on those signatures fail.
