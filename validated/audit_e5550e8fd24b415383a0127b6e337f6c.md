Audit Report

## Title
Sequential Unbounded External Calls in `updateRate()` Can Permanently Block Cross-Chain rsETH Rate Broadcasts - (File: `contracts/cross-chain/MultiChainRateProvider.sol`)

## Summary
`MultiChainRateProvider.updateRate()` iterates over an unbounded `rateReceivers` array, executing two external LayerZero calls per entry with no cap on array length. As the protocol adds destination chains via normal owner operation, gas consumption grows linearly until the function exceeds the block gas limit and becomes permanently uncallable, freezing rsETH rate updates to all destination chains.

## Finding Description
`updateRate()` at L108–137 of `contracts/cross-chain/MultiChainRateProvider.sol` is the sole mechanism for broadcasting the rsETH/ETH exchange rate cross-chain. For each entry in `rateReceivers`, it executes two sequential external calls to the LayerZero endpoint: a `staticcall` to `estimateFees` (L124–125) and a state-changing `send` with ETH transfer (L127–129). `addRateReceiver()` at L72–76 unconditionally pushes to the array with no length cap. There is no batching, no index-range parameter, and no maximum enforced anywhere. `RSETHMultiChainRateProvider` inherits this logic directly and is the live contract providing rsETH rates cross-chain. Because `updateRate()` has no access control (`external payable`, no role check), any caller will hit the same gas wall once the array is large enough. Once the per-transaction gas cost exceeds the block gas limit, every invocation reverts and no receiver on any destination chain can receive an updated rate.

## Impact Explanation
This matches **Medium — Unbounded gas consumption**. `updateRate()` is the only push mechanism for rsETH rates to all registered `RSETHRateReceiver` contracts. If it becomes permanently uncallable, all destination-chain receivers hold a stale rate indefinitely, causing mispriced swaps, incorrect collateral valuations, and loss of yield for rsETH holders and integrating DeFi protocols. The impact is concrete and directly caused by this repository's logic, not an external dependency alone.

## Likelihood Explanation
rsETH is already deployed across multiple chains and the protocol's stated goal is continued multi-chain expansion. Each new chain deployment requires a call to `addRateReceiver()`, which is normal, expected owner operation — not an attack. The gas cost per receiver is high (two external LayerZero calls per iteration). The function is callable by any address, so no privileged trigger is needed to hit the gas wall; any caller encounters the same revert once the array is sufficiently large. The threshold is reached through ordinary protocol growth, not adversarial action.

## Recommendation
1. Enforce a maximum cap on `rateReceivers.length` inside `addRateReceiver()`.
2. Add an indexed overload `updateRate(uint256 fromIndex, uint256 toIndex)` so callers can broadcast to a subset of receivers per transaction, enabling batched updates across multiple transactions.
3. Alternatively, replace the push model with a pull model: emit an on-chain event and let each destination chain relay the rate independently, eliminating the loop entirely.

## Proof of Concept
1. Owner calls `addRateReceiver()` for each new destination chain as rsETH expands (e.g., 50+ chains over time).
2. Any caller invokes `updateRate{ value: totalFee }()`.
3. The loop executes `estimateFees` + `send` for each of the N receivers sequentially.
4. Gas consumed per iteration ≈ 50,000–100,000 gas (LayerZero `send` with cross-chain messaging overhead); at N receivers this is `N × [50k–100k]` gas, scaling linearly with no bound.
5. Beyond the block gas limit threshold, every invocation reverts with out-of-gas.
6. No receiver on any destination chain can receive an updated rsETH rate; `RSETHRateReceiver.rate` on all chains is permanently stale.

**Foundry test plan:** Deploy a mock `ILayerZeroEndpoint` that consumes a fixed amount of gas per call. Call `addRateReceiver()` in a loop to populate the array to a large count. Call `updateRate{value: ...}()` with a gas limit matching Ethereum's block gas limit (30M). Assert the transaction reverts with out-of-gas, confirming the function is uncallable at that array size.
