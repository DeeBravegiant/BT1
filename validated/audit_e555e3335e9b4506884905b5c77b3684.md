Audit Report

## Title
Cross-Token Withdrawal Enables Theft of Yield Differential Between Allowed Tokens — (`contracts/L2/RsETHTokenWrapper.sol`)

## Summary
`RsETHTokenWrapper` mints and burns wrsETH strictly 1:1 against any allowed token, with no binding between the deposited token and the withdrawn token and no exchange-rate check. Once `TIMELOCK_ROLE` adds a second allowed token — a normal, intended protocol operation — any wrsETH holder can deposit the lower-value token and withdraw the higher-value token, extracting the yield differential at the expense of other depositors.

## Finding Description
`_deposit` (lines 134–141) accepts any `allowedTokens[_asset]` token, pulls it in via `safeTransferFrom`, and mints an equal amount of wrsETH. `_withdraw` (lines 120–128) burns wrsETH and transfers whichever allowed token the caller names; the sole guard is `if (!allowedTokens[_asset]) revert TokenNotAllowed()`. There is no per-user deposit record, no per-token share accounting, and no oracle or rate comparison. wrsETH is fully fungible across all allowed tokens.

`addAllowedToken` (lines 174–176) is a normal, intended operation gated only on `TIMELOCK_ROLE`. Once a second token exists in `allowedTokens`, the wrapper holds balances of both tokens and any wrsETH holder — regardless of which token they deposited — can freely choose which allowed token to receive on withdrawal.

rsETH is a yield-bearing token whose exchange rate against ETH increases over time. Different L2 bridge implementations of rsETH (e.g., a static-balance bridge token vs. a rate-accumulating one) will diverge in value as yield accrues. Even a 0.1% divergence is immediately exploitable with no risk to the attacker.

**Attack path:**
1. `TIMELOCK_ROLE` calls `addAllowedToken(altRsETH_B)` — normal protocol operation.
2. The wrapper accumulates a balance of `altRsETH_B` (via `depositBridgerAssets` or other depositors).
3. Attacker calls `deposit(altRsETH_A, N)` — mints `N` wrsETH against the cheaper token.
4. Attacker calls `withdraw(altRsETH_B, N)` — burns `N` wrsETH, receives `N` units of the more-valuable token.
5. Profit = `N × (P_B − P_A)` ETH-equivalent, stolen from other depositors.

## Impact Explanation
**High — Theft of unclaimed yield.** The attacker extracts the yield differential between two allowed tokens. The loss falls directly on depositors of the higher-value token: their collateral is replaced by the cheaper token, leaving their wrsETH under-collateralized. The wrapper's total ETH-value backing decreases permanently for remaining holders.

## Likelihood Explanation
All three preconditions are plausible in normal protocol operation: (1) `TIMELOCK_ROLE` adding a second allowed token is an explicitly designed feature; (2) the wrapper accumulating a balance of that token is the normal operating state; (3) yield-bearing rsETH tokens diverge in value over time as a matter of course. The attack itself requires no special role, no front-running, and no external protocol compromise — only standard `deposit` + `withdraw` calls available to any address.

## Recommendation
1. **Per-token share accounting:** Replace the single wrsETH supply with per-token share tracking so that `withdraw(tokenB, amount)` is bounded by shares minted from `tokenB` deposits, not the global wrsETH balance.
2. **Enforce single canonical token:** If all allowed tokens are intended to remain permanently 1:1, enforce this with an on-chain rate oracle check in `_withdraw` that reverts if the rate deviates beyond a tight bound (e.g., 0.1%).
3. **Restrict cross-token withdrawals:** Require that `withdraw(asset, amount)` can only be called with the same `asset` that was deposited, tracked per user or per wrsETH mint event.

## Proof of Concept
The submitted Foundry test is minimal and correct. It deploys `RsETHTokenWrapper`, initializes with `tokenA`, adds `tokenB` via `TIMELOCK_ROLE`, has a victim deposit 100 `tokenB`, then has an attacker deposit 100 `tokenA` and withdraw 100 `tokenB`. The final assertions — `tokenB.balanceOf(attacker) == 100e18` and `tokenA.balanceOf(wrapper) == 100e18` — demonstrate the complete theft on unmodified production code with no privileged access by the attacker.
