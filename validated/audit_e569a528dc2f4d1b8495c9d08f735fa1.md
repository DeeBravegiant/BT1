After thorough analysis of the codebase, I found a valid analog vulnerability.

### Title
Stale Votes from Dropped Participants Counted Toward TEE Governance Thresholds Without Participant Filtering — (`crates/contract/src/tee/measurements.rs`)

### Summary

`MeasurementVotes::count_votes()` tallies all stored votes without filtering by the current participant set, while other voting subsystems (`ProviderVotes`, `TeeVerifierVotes`) correctly filter by current participants at count time. After a resharing that drops participants, if the post-resharing cleanup promise fails, stale votes from dropped participants remain in `vote_by_account` and are counted toward the threshold for `vote_add_os_measurement` and `vote_remove_os_measurement`. This allows TEE governance actions to be applied with fewer current-participant votes than the protocol requires.

### Finding Description

**Root cause — inconsistent vote-source treatment:**

`MeasurementVotes::count_votes()` counts every entry in `vote_by_account` that matches the action, with no participant-membership filter:

```rust
fn count_votes(&self, action: &MeasurementVoteAction) -> u64 {
    u64::try_from(
        self.vote_by_account
            .values()
            .filter(|a| *a == action)
            .count(),
    )
    .expect("participant count should not overflow u64")
}
``` [1](#0-0) 

This is called directly from `vote_add_os_measurement` and `vote_remove_os_measurement`:

```rust
let votes = self.tee_state.vote_measurement(action, &participant);
if votes >= self.threshold()?.value() {
    let added = self.tee_state.add_measurement(measurement);
``` [2](#0-1) 

**Contrast with correctly-implemented voting subsystems:**

`ProviderVotes::vote()` filters by current participants at count time:
```rust
voter_set.count_for(|(p, c)| {
    *c == chain && participants.is_participant_given_participant_id(&p.get())
})
``` [3](#0-2) 

`TeeVerifierVotes::vote()` does the same:
```rust
voter_set.count_for(|p| participants.is_participant_given_participant_id(&p.get()))
``` [4](#0-3) 

`MeasurementVotes` has no such filter. The same structural gap exists in `CodeHashesVotes::count_votes()` used by `vote_code_hash` and `vote_add_launcher_hash`: [5](#0-4) 

**Cleanup path and its failure mode:**

The intended cleanup is `clean_non_participant_votes`, called after resharing via the private `clean_tee_status` promise:

```rust
pub fn clean_non_participant_votes(&mut self, participants: &Participants) {
    self.votes = self.votes.get_remaining_votes(participants);
    self.launcher_votes = self.launcher_votes.get_remaining_votes(participants);
    self.measurement_votes = self.measurement_votes.get_remaining_votes(participants);
}
``` [6](#0-5) 

`get_remaining_votes` correctly filters by participant membership: [7](#0-6) 

However, `clean_tee_status` is dispatched as a fire-and-forget promise. If it fails (e.g., gas exhaustion, promise error), stale votes are never purged. There is no retry mechanism and no on-chain invariant that prevents `vote_add_os_measurement` from being called while stale votes are present.

**Exploit scenario:**

1. Network has 5 participants, governance threshold = 3.
2. Participants A and B vote for OS measurement `M_evil` (2 votes stored).
3. A resharing drops A and B; new participant set is {C, D, E}.
4. The `clean_tee_status` promise fails; A's and B's votes remain in `vote_by_account`.
5. Participant C calls `vote_add_os_measurement(M_evil)`.
6. `count_votes` returns 3 (A_stale + B_stale + C_current) ≥ threshold(3).
7. `M_evil` is added to `allowed_measurements` with only **1** current-participant vote.

### Impact Explanation

`allowed_measurements` gates TEE attestation verification. A measurement added to this list allows any node whose TDX quote matches it to pass `submit_participant_info` and be accepted as an attested participant. If a malicious or compromised OS measurement is whitelisted with sub-threshold current-participant support, a node running in that environment can obtain a valid `TeeQuoteStatus::Valid`, enabling it to call `respond` (sign) and `respond_ckd` on behalf of the MPC network. This breaks the production safety invariant that TEE governance changes require a genuine threshold of current participants, and can escalate to unauthorized signature issuance if the malicious node acquires key shares.

This maps to: **Medium — participant-state and contract execution-flow manipulation that breaks production safety/accounting invariants.**

### Likelihood Explanation

- Resharing is a normal, expected protocol operation.
- Promise failures in NEAR are a documented failure mode (gas, callback errors).
- The comment in `lib.rs` for `vote_update` explicitly acknowledges this class of problem: *"This ensures correctness even if the cleanup promise in `MpcContract::vote_reshared()` fails."* — but the analogous guard is absent for measurement votes. [8](#0-7) 
- No privileged access is required beyond having been a participant before the resharing.

### Recommendation

Apply the same participant-membership filter used by `ProviderVotes` and `TeeVerifierVotes` inside `MeasurementVotes::vote()` (and `CodeHashesVotes::vote()`). Pass the current `ThresholdParameters` into the count call and filter `vote_by_account` to only entries whose key is a current participant before comparing against the threshold. This mirrors the pattern already established in the codebase and eliminates the dependency on the cleanup promise succeeding.

### Proof of Concept

```
// Setup: 5 participants, threshold = 3
// Step 1: Participants A and B vote for M_evil (2 stale votes stored)
// Step 2: Resharing drops A and B → new set {C, D, E}
// Step 3: clean_tee_status promise fails → A, B votes remain in vote_by_account
// Step 4: C calls vote_add_os_measurement(M_evil)
//   → count_votes returns 3 (A_stale + B_stale + C_current)
//   → 3 >= threshold(3) → M_evil added to allowed_measurements
// Result: malicious OS measurement whitelisted with 1/3 required current votes
```

The structural proof is the direct comparison between `MeasurementVotes::count_votes` (no participant filter, lines 58–66 of `measurements.rs`) and `TeeVerifierVotes::vote` (explicit `is_participant_given_participant_id` filter, lines 75–77 of `verifier_votes.rs`): two vote-counting paths in the same contract, one correctly scoped to current participants, one not.
