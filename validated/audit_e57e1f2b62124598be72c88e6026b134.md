### Title
Missing Emergency Pause Mechanism — No Participant-Callable Method to Halt Signing (`crates/contract/src/lib.rs`)

---

### Summary

The MPC contract contains an `accept_requests` circuit-breaker field that, when `false`, blocks all user-facing signing operations (`sign`, `request_app_private_key`, `verify_foreign_transaction`). However, no participant-callable governance method exists to set this flag to `false` deliberately. The only automatic path that sets it is `verify_tee()` failing TEE attestation validation. In a security incident — a discovered signing-protocol bug, a compromised domain key, or a critical contract logic flaw — participants have no on-chain mechanism to immediately halt new signature requests while a contract upgrade is prepared and voted through.

---

### Finding Description

`check_request_preconditions` in `crates/contract/src/lib.rs` enforces the circuit breaker on every user-facing request:

```rust
// 4. Refuse the request if the contract is not currently accepting requests
//    (e.g. because TEE validation has failed).
if !self.accept_requests {
    env::panic_str(&TeeError::TeeValidationFailed.to_string())
}
``` [1](#0-0) 

This guard is reached by all three public signing entry points — `sign`, `request_app_private_key`, and `verify_foreign_transaction` — via the shared `check_request_preconditions` helper: [2](#0-1) 

The `accept_requests` field can only be set to `false` through `verify_tee()`, which does so automatically when fewer than threshold participants hold valid TEE attestations. This is an attestation-lifecycle mechanism, not an emergency stop. The complete participant governance API — `vote_new_parameters`, `vote_add_domains`, `vote_cancel_keygen`, `vote_cancel_resharing`, `propose_update`, `vote_update`, etc. — contains no `vote_pause`, `pause`, or equivalent method: [3](#0-2) 

The contract state machine (`NotInitialized → Running → Resharing → Running`) has no `Paused` state and no transition into one: [4](#0-3) 

The only alternative — a contract upgrade via `propose_update` / `vote_update` — requires threshold votes to be collected and a new WASM to be deployed, which takes meaningful wall-clock time during which the vulnerable code continues to execute.

---

### Impact Explanation

If a critical vulnerability is discovered in the signing protocol (e.g., a flaw that allows forged signatures, a replay of a signing request, or a bug in the `respond` path that bypasses signature verification), participants have no on-chain lever to immediately stop new requests from being enqueued and processed. Every new `sign` call accepted during the window between discovery and a successful contract upgrade is a potential unauthorized signature issuance. Because MPC-produced signatures authorize transactions on foreign chains (Bitcoin, Ethereum, Solana, etc.), each unauthorized signature can directly drain funds controlled by the MPC network's derived keys.

This maps to the allowed Medium impact: **contract execution-flow manipulation that breaks production safety/accounting invariants** — specifically, the invariant that the protocol can be halted by its governance body in response to a security event.

---

### Likelihood Explanation

The gap is a design omission, not a probabilistic event. Every security incident that warrants an emergency stop — a zero-day in the signing library, a discovered key-derivation flaw, a compromised node that is actively submitting malicious `respond` calls — is blocked from being mitigated quickly. The Tapioca project suffered exactly this class of incident in production. The NEAR MPC contract controls keys that sign transactions on multiple high-value chains, making the blast radius of any signing vulnerability proportionally large.

---

### Recommendation

Add a threshold-gated `vote_pause` / `vote_unpause` governance method that sets `accept_requests` to `false` or `true` when a threshold of participants agree:

```rust
pub fn vote_pause(&mut self) -> Result<(), Error> {
    let caller = Self::assert_caller_is_signer();
    // collect votes; if threshold reached, set accept_requests = false
    self.pause_votes.insert(caller);
    if self.pause_votes.len() >= self.threshold()?.value() as usize {
        self.accept_requests = false;
        self.pause_votes.clear();
    }
    Ok(())
}

pub fn vote_unpause(&mut self) -> Result<(), Error> {
    let caller = Self::assert_caller_is_signer();
    self.unpause_votes.insert(caller);
    if self.unpause_votes.len() >= self.threshold()?.value() as usize {
        self.accept_requests = true;
        self.unpause_votes.clear();
    }
    Ok(())
}
```

This mirrors the pattern already used for `vote_cancel_resharing` and `vote_cancel_keygen`, and ensures the circuit breaker can be triggered deliberately without requiring a full contract upgrade.

---

### Proof of Concept

1. A critical bug is discovered in the `respond` path that allows a malicious MPC node to submit a forged `SignatureResponse` for an arbitrary payload.
2. Participants attempt to halt the contract. They have no `vote_pause` method. Their only options are:
   - Call `verify_tee()` — does nothing unless TEE attestations are already invalid.
   - Collect threshold votes for `propose_update` + `vote_update` with a patched WASM — takes multiple NEAR block rounds and coordination time.
3. During the upgrade window, any user (or the attacker themselves) calls `sign(payload)` with the target payload. The malicious node calls `respond` with the forged signature. The contract's `accept_requests` is still `true`, so `check_request_preconditions` passes and the request is enqueued.
4. The forged signature is delivered to the caller, who uses it to authorize a foreign-chain transaction draining funds from the MPC-controlled address.

The root cause — `accept_requests` is checked but cannot be set to `false` by participants — is confirmed by the test at: [5](#0-4) 

which directly mutates the field (`contract.accept_requests = false`) because no public method exists to do so.

    fn check_request_preconditions(
        &self,
        domain_id: DomainId,
        expected_purpose: DomainPurpose,
        minimum_gas: Gas,
        minimum_deposit: NearToken,
    ) -> (DomainConfig, AccountId) {
        // 1. Look up the domain and check its purpose.
        let domains = match self.protocol_state.domain_registry() {
            Ok(domains) => domains,
            Err(err) => env::panic_str(&err.to_string()),
        };
        let Some(domain_config) = domains.get_domain_by_domain_id(domain_id) else {
            env::panic_str(
                &InvalidParameters::DomainNotFound {
                    provided: domain_id,
                }
                .to_string(),
            );
        };
        if domain_config.purpose != expected_purpose {
            env::panic_str(
                &InvalidParameters::WrongDomainPurpose {
                    domain_id: domain_config.id,
                    expected: expected_purpose,
                    actual: domain_config.purpose,
                }
                .to_string(),
            );
        }
        let domain_config = domain_config.clone();

        // 2. Make sure the call will not run out of gas doing yield/resume logic.
        let prepaid_gas = env::prepaid_gas();
        if prepaid_gas < minimum_gas {
            env::panic_str(
                &InvalidParameters::InsufficientGas {
                    provided: prepaid_gas.as_gas(),
                    required: minimum_gas.as_gas(),
                }
                .to_string(),
            );
        }

        // 3. Require the minimum deposit and refund any excess.
        let predecessor = env::predecessor_account_id();
        require_deposit(minimum_deposit, &predecessor);

        // 4. Refuse the request if the contract is not currently accepting requests
        //    (e.g. because TEE validation has failed).
        if !self.accept_requests {
            env::panic_str(&TeeError::TeeValidationFailed.to_string())
        }

        (domain_config, predecessor)
    }
