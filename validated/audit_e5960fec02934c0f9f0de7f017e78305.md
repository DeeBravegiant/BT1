### Title
IEEE 754 Floating-Point Used in Consensus-Critical `NonMyopic` Ledger State Computation Instead of the Project's Own Fixed-Precision Library — (`eras/shelley/impl/src/Cardano/Ledger/Shelley/PoolRank.hs`)

---

### Summary

The `leaderProbability` and `likelihood` functions compute `Likelihood` values that are stored in the `NonMyopic` field of the consensus-critical `EpochState` using IEEE 754 `Double`/`Float` arithmetic. The project's own formal specification explicitly rejects IEEE 754 as unsuitable for Cardano due to non-determinism across architectures and implementations, and a purpose-built fixed-precision library (`non-integral`) already exists and is used for the analogous consensus-critical `checkLeaderNatValue` path. The `NonMyopic` path bypasses it entirely, and the code itself contains a `TODO: worry more about loss of floating point precision` comment acknowledging the gap.

---

### Finding Description

**The fixed-precision mandate and the existing library**

The project's own specification document (`docs/non-integer-calculations/non-integer-calculations.tex`) explicitly evaluates IEEE 754 and rejects it:

> *"Moreover, the elementary functions that are required for Cardano are not standardized and there can be subtle differences in different implementations, with results even varying when different compiler settings are used on the same implementation… A final problem is that IEEE 754 supports different rounding modes… For the Cardano cryptocurrency, the fixed precision arithmetic approach seems to be most suitable."*

In response, the `libs/non-integral` library was built, providing `taylorExpCmp`, `exp'`, `ln'`, and `(***)` over a `FixedPoint` type. This library **is** used for the consensus-critical slot-leader check:

```
-- libs/cardano-protocol-tpraos/src/Cardano/Protocol/TPraos/BHeader.hs
checkLeaderNatValue bn σ f =
  ...
  case taylorExpCmp 3 recip_q x of   -- uses FixedPoint via non-integral
    ABOVE _ _ -> False
    BELOW _ _ -> True
    MaxReached _ -> False
``` [1](#0-0) 

**The vulnerable path: `leaderProbability` and `likelihood` use raw `Double`/`Float`**

`leaderProbability` computes the same `1 - (1-f)^σ` formula using Haskell's built-in `(**)` on `Double`:

```haskell
leaderProbability :: ActiveSlotCoeff -> Rational -> UnitInterval -> Double
leaderProbability activeSlotCoeff relativeStake decentralizationParameter =
  (1 - (1 - asc) ** s) * (1 - d')
``` [2](#0-1) 

The result is passed to `likelihood`, which performs further `Double` arithmetic (`log`, `*`, `+`) and then **narrows to `Float`** (32-bit single precision) via `realToFrac`:

```haskell
newtype LogWeight = LogWeight {unLogWeight :: Float}   -- 32-bit!

l :: Double -> Double
l x = n * log x + m * log (1 - t * x)
-- TODO: worry more about loss of floating point precision
sample position = LogWeight (realToFrac $ l position)  -- Double -> Float truncation
``` [3](#0-2) [4](#0-3) 

**`NonMyopic` is part of the consensus-critical `EpochState`**

The `Likelihood` values (sequences of `LogWeight` / `Float`) are accumulated into `NonMyopic`, which has full `EncCBOR`/`DecCBOR` instances and is the fourth field of `EpochState`:

```haskell
startStep ... es@(EpochState acnt ls ss nm) ... =
  ...
  newLikelihoods = VMap.map makeLikelihoods allPoolInfo
  -- makeLikelihoods calls leaderProbability (Double) then likelihood (Double→Float)
``` [5](#0-4) 

`NonMyopic` is serialized with `EncCBOR`/`DecCBOR`: [6](#0-5) 

Because `EpochState` (including `NonMyopic`) is part of the ledger state that all nodes must agree on, any divergence in the floating-point computation of `Likelihood` values produces a different serialized `EpochState` on different nodes.

---

### Impact Explanation

**Impact: High — Deterministic disagreement between honest nodes from ledger rule evaluation.**

If two honest nodes compute different `Likelihood` / `LogWeight` values for the same epoch (due to architecture-level floating-point differences, compiler flags, or a second independent implementation), their `NonMyopic` fields diverge, their `EpochState` CBOR serializations diverge, and they disagree on the ledger state. This is precisely the ledger-fork scenario the `non-integral` library was designed to prevent, and which the spec document explicitly warns about.

The `Double → Float` narrowing in `sample` (line 172) compounds the issue: a 64-bit intermediate result is silently truncated to 32-bit before storage, introducing an additional, architecture-sensitive rounding step that is absent from the `FixedPoint`-based path.

---

### Likelihood Explanation

**Likelihood: Medium.**

In the current deployment, all Cardano nodes run the same GHC-compiled binary on x86-64 with SSE2 floating-point, so results are deterministic in practice. However:

1. Cardano explicitly targets multiple independent implementations (the `non-integral` spec was written precisely for this reason).
2. ARM-based nodes (e.g., Raspberry Pi stake pools, which are common in the Cardano ecosystem) use a different FPU and may produce different `Double` results for transcendental functions (`log`, `**`).
3. The `TODO: worry more about loss of floating point precision` comment in the production code is an acknowledged, unresolved concern.
4. Any future Rust/Java/Go implementation that faithfully implements the spec's fixed-precision mandate for `checkLeaderNatValue` but uses native floating-point for `leaderProbability` would immediately diverge on `NonMyopic`.

---

### Recommendation

Replace the `Double`/`Float` arithmetic in `leaderProbability` and `likelihood` with the `FixedPoint`-based `non-integral` library, exactly as is done in `checkLeaderNatValue`. Specifically:

- Replace `(1 - asc) ** s` in `leaderProbability` with `(1 - asc) *** s` from `Cardano.Ledger.NonIntegral` operating on `FixedPoint`.
- Replace `log x` and `log (1 - t * x)` in `likelihood` with `ln'` from the same library.
- Change `LogWeight` from `Float` to `FixedPoint` (or at minimum `Rational`) to eliminate the `Double → Float` truncation.
- Remove the `-- TODO: worry more about loss of floating point precision` comment by resolving it.

      k = toIntegerNonZero secparam
      -- We expect approximately 10k-many blocks to be produced each epoch.
      -- The reward calculation begins (4k/f)-many slots into the epoch,
      -- and we guarantee that it ends (2k/f)-many slots before the end
      -- of the epoch (to allow tools such as db-sync to see the reward
      -- values in advance of them being applied to the ledger state).
      --
      -- Therefore to evenly space out the reward calculation, we divide
      -- the number of stake credentials by 4k in order to determine how many
      -- stake credential rewards we should calculate each block.
      -- If it does not finish in this amount of time, the calculation is
      -- forced to completion.
      pulseSize = max 1 (ceiling (numStakeCreds %. (knownNonZero @4 `mulNonZero` k)))
      -- We now compute the amount of total rewards that can potentially be given
      -- out this epoch, and the adjustments to the reserves and the treasury.
      Coin reserves = acnt ^. casReservesL
      accounts = ls ^. lsCertStateL . certDStateL . accountsL
      -- reserves and rewards change
      pr = es ^. prevPParamsEpochStateL

        rationalToCoinViaFloor $
          min 1 eta
            * unboundRational (pr ^. ppRhoL)
            * fromIntegral reserves
      d = unboundRational (pr ^. ppDG)
      expectedBlocks =
        floor $
          (1 - d) * unboundRational (activeSlotVal asc) * fromIntegral (unEpochSize slotsPerEpoch)

      -- it would be nice to not have to compute expectedBlocks every epoch
      blocksMade = fromIntegral $ Map.foldr (+) 0 b' :: Integer
      eta

        | otherwise =

            -- should never have expectedBlocks anywhere close to zero
            blocksMade % expectedBlocks

      deltaT1 = floor $ unboundRational (pr ^. ppTauL) * fromIntegral rPot

      -- member and leader rewards.

          (fromIntegral blocksMade)

\subsection{IEEE~754 Floating Point}
\label{sec:ieee-754-floating}

A straightforward approach is to use IEEE~754 floating-point arithmetic.  This is
widely supported by common programming languages and CPU/GPU architectures.
%
The basic arithmetic operations of IEEE~754 floating-point are well specified,
and they are usually very efficient due to direct HW implementation.
%
Double-precision IEEE~754 floating-point numbers (\emph{binary64}) provide 53 bits (~15.99 decimal digits) of
precision.  This is slightly less than is needed to represent the
required fraction of 1 lovelace: there are $4.5\cdot10^{16}$ possible fractions.
Moreover, the  elementary functions that are required for Cardano are not standardized and there can be
subtle differences in different implementations, with results even varying when different compiler settings are used
on the same implementation\footnote{e.g. maintaining intermediate representations within internal registers \emph{vs.} writing to memory.}. It is also worth noting that some architectures
provide excess precision~\footnote{The original x87 provided 80 bit
  floating-point}, which can result in greater accuracy, but can also give slight differences in results
between implementations. Some programming languages, e.g., D, exploit this by default. Many
languages support excess precision, but default to IEEE~754 64 bits. In
particular on the AMD64 architecture most language implementations use SSE floating-point arithmetic.
A final problem is that IEEE~754 supports different rounding modes\footnote{round to nearest (with even digits breaking ties or ties rounding away from zero), round down, round up, round to zero},  which will obviously give
different results in some cases.  Some programming languages allow the rounding mode to be changed, e.g. C99 onwards, but others do
not, e.g., Haskell.
