Audit Report

## Title
Uninitialized `dailyMintLimit` Blocks All Deposits After `initialize()` Without Subsequent `reinitialize()` — (`contracts/pools/RSETHPoolV3WithNativeChainBridge.sol`, `contracts/pools/RSETHPoolV3ExternalBridge.sol`)

## Summary
Both `RSETHPoolV3WithNativeChainBridge` and `RSETHPoolV3ExternalBridge` leave `dailyMintLimit` and `startTimestamp` at their zero defaults after `initialize()`. These values are only populated via separate `reinitialize()` calls (`reinitializer(2)` and `reinitializer(4)` respectively). If a reinitializer is omitted on a new chain deployment, every call to `deposit()` reverts with `DailyMintLimitExceeded`, completely blocking the deposit path until an admin intervenes. No user funds are lost since deposits revert, but the contract fails to deliver its core promised functionality.

## Finding Description
`RSETHPoolV3WithNativeChainBridge.initialize()` (L245–268) sets `wrsETH`, `feeBps`, `rsETHOracle`, and roles, but never assigns `dailyMintLimit` or `startTimestamp`. These are only set in `reinitialize(uint256,uint256)` tagged `reinitializer(2)` (L218–237). The `limitDailyMint` modifier (L108–137) is applied to both `deposit(string)` (L282) and `deposit(address,uint256,string)` (L316). With `startTimestamp = 0`, the guard `block.timestamp < startTimestamp` is always false (any real timestamp exceeds 0), so it passes silently. With `dailyMintLimit = 0`, the guard `dailyMintAmount + rsETHAmount > dailyMintLimit` reduces to `rsETHAmount > 0`, which is true for every non-zero deposit, causing an unconditional revert. The identical pattern exists in `RSETHPoolV3ExternalBridge`: `initialize()` (L330–352) omits both fields, which are only set in `reinitialize(uint256,uint256)` tagged `reinitializer(4)` (L276–295), while `deposit(string)` (L366) and `deposit(address,uint256,string)` (L399) both apply `limitDailyMint`.

## Impact Explanation
The deposit path is completely non-functional from the moment `initialize()` is called until `reinitialize()` is executed. All deposit transactions revert; no ETH or tokens enter the contract. Since deposits revert and user funds are returned, no value is lost or frozen. The correct impact classification is **Low — Contract fails to deliver promised returns, but doesn't lose value**: the contract is deployed and callable but cannot accept any deposits, failing its core purpose until admin remediation.

## Likelihood Explanation
The vulnerability requires only that a deployer calls `initialize()` without subsequently calling the appropriate `reinitialize()`. For `RSETHPoolV3ExternalBridge`, the version history documents six sequential reinitializers, establishing a multi-step deployment practice where omitting one step is realistic. For `RSETHPoolV3WithNativeChainBridge`, a fresh deployment requires `initialize()` + `reinitialize(2)` + `reinitialize(3)`. No attacker action is required — any user attempting a deposit triggers the revert. The protocol's multi-chain expansion makes new deployments likely, increasing the probability of a missed step.

## Recommendation
Add `_dailyMintLimit` and `_startTimestamp` as parameters to `initialize()` in both contracts, validate them with non-zero checks (mirroring the existing `UtilLib.checkNonZeroAddress` pattern), and assign them directly. This eliminates the dependency on a separate post-deployment call and ensures the contract is fully operational immediately after initialization.

## Proof of Concept
1. Deploy `RSETHPoolV3WithNativeChainBridge` behind a proxy and call `initialize(admin, bridger, wrsETH, feeBps, rsETHOracle)`.
2. Do **not** call `reinitialize(dailyMintLimit, startTimestamp)`.
3. As any unprivileged user, call `deposit{value: 1 ether}("ref")`.
4. The `limitDailyMint` modifier executes: `startTimestamp = 0` so `block.timestamp < 0` is false (passes); `dailyMintLimit = 0` so `rsETHAmount > 0` is true → revert `DailyMintLimitExceeded`.
5. All deposits from all users revert until the admin calls `reinitialize(2)`.

Foundry test sketch:
```solidity
function test_depositRevertsWithoutReinitialize() public {
    // Deploy proxy, call initialize only
    pool.initialize(admin, bridger, address(wrsETH), 10, address(oracle));
    // Do NOT call reinitialize(dailyMintLimit, startTimestamp)
    vm.deal(user, 1 ether);
    vm.prank(user);
    vm.expectRevert(RSETHPoolV3WithNativeChainBridge.DailyMintLimitExceeded.selector);
    pool.deposit{value: 1 ether}("ref");
}
```
