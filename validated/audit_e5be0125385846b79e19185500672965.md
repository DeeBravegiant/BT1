### Title
Generic Request Queue Retries Non-Retryable Sub-Quorum Foreign-TX Failures, Consuming a Presignature on Every Attempt — (File: `crates/node/src/requests/queue.rs`, `crates/node/src/providers/verify_foreign_tx/sign.rs`)

---

### Summary

The `PendingRequests` generic queue retries every failed `verify_foreign_tx` attempt up to `MAX_ATTEMPTS_PER_REQUEST_AS_LEADER = 10` times, without distinguishing between transient failures (which should retry) and non-retryable sub-quorum RPC disagreements (which must not retry). Because `make_verify_foreign_tx_leader` consumes a presignature from the store **before** executing the foreign-chain RPC check, each retry on a persistently-failing sub-quorum request silently destroys a presignature. An unprivileged caller who submits a `verify_foreign_transaction` request for a foreign-chain transaction that causes persistent RPC provider disagreement can force the leader to waste up to 10 presignatures per request, breaking the production accounting invariant that presignatures are consumed only when a signature is actually produced.

---

### Finding Description

The design specification is unambiguous. `docs/design/calculating-supported-foreign-chains.md` states:

> **This sub-quorum outcome must be terminal — the leader must not re-attempt the request.** Implementation requirement, not current behavior: the generic queue retries every request, so the foreign-tx path must special-case a sub-quorum result as non-retryable. [1](#0-0) 

The same requirement appears in `docs/design/allowing-per-node-foreign-chain-rpc-configuration.md`:

> This sub-quorum outcome must be **terminal** for the request — the leader does not re-attempt it. This is an implementation requirement, not current behavior. [2](#0-1) 

**Root cause — step 1: presignature consumed before foreign-chain check.**

In `make_verify_foreign_tx_leader`, the presignature is taken from the store unconditionally before `execute_foreign_chain_request` is called:

```rust
let (presignature_id, presignature) = domain_data.presignature_store.take_owned().await;
// ...
let response_payload = self
    .execute_foreign_chain_request(...)
    .await?;   // ← if sub-quorum, presignature already gone
``` [3](#0-2) 

If `execute_foreign_chain_request` returns `Err` (e.g., `ForeignChainInspectionError::InspectorResponseMismatch` from the `FanOut` inspector when providers disagree), the `?` operator returns early and the presignature is dropped without producing a signature.

**Root cause — step 2: the generic queue retries unconditionally.**

`QueuedRequest::process` in the generic queue issues a new `GenerationAttempt` whenever the previous attempt's `Arc` has been dropped and `attempts < MAX_ATTEMPTS_PER_REQUEST_AS_LEADER`:

```rust
match progress.update_computation_progress(now) {
    ComputationProgressStatus::MaxAttemptsExceeded => {
        RequestStatus::Drop(DropReason::MaxAttemptsExceeded)
    }
    ComputationProgressStatus::Pending => RequestStatus::Wait("pending computation"),
    ComputationProgressStatus::NewAttempt => {
        // issues another attempt
    }
}
``` [4](#0-3) 

`MAX_ATTEMPTS_PER_REQUEST_AS_LEADER` is 10. [5](#0-4) 

**Root cause — step 3: the `computed_response` cache does not protect against sub-quorum failures.**

`mpc_client.rs` skips recomputation only if `computed_response` is already set. A sub-quorum failure never sets `computed_response`, so every retry re-enters `make_verify_foreign_tx_leader` and consumes a fresh presignature. [6](#0-5) 

**Root cause — step 4: `FanOut` correctly classifies sub-quorum as non-transient, but the queue ignores the distinction.**

`ForeignChainInspectionError::InspectorResponseMismatch` is non-transient (not listed in `is_transient()`), so the `FanOut` inspector correctly surfaces it as a substantive verdict. However, the queue has no hook to receive this classification — it only sees `anyhow::Err` from the failed task. [7](#0-6) 

---

### Impact Explanation

Each `verify_foreign_tx` request that persistently fails with a sub-quorum RPC disagreement causes the leader to consume up to 10 presignatures — one per retry — without producing any signature. Presignatures are expensive multi-party cryptographic artifacts generated by all threshold participants. Consuming them without producing a signature breaks the production accounting invariant that presignature consumption is 1:1 with signature production. Depleting the presignature pool delays all legitimate signing requests (signatures, CKD, and other foreign-tx verifications) until new presignatures are generated.

This matches the **Medium** allowed impact: *"request-lifecycle manipulation that breaks production safety/accounting invariants without relying on network-level DoS or operator misconfiguration."*

---

### Likelihood Explanation

Any unprivileged caller can submit a `verify_foreign_transaction` request. A foreign-chain transaction that causes RPC provider disagreement (e.g., a transaction in a block that some providers consider canonical and others do not, or a transaction whose extraction results differ across providers) will trigger `InspectorResponseMismatch` on every attempt. The attacker needs no special privilege — only the ability to call `verify_foreign_transaction` with a 1 yoctoNEAR deposit. The effect is deterministic and repeatable for the lifetime of the request (up to 200 blocks / ~200 seconds).

---

### Recommendation

1. **Propagate the non-retryable signal from the task to the queue.** Introduce a typed error wrapper (e.g., `NonRetryableError`) that the `verify_foreign_tx` task returns on sub-quorum failure. The queue's task-completion handler should mark the request as `Drop(DropReason::NonRetryableFailure)` instead of allowing the next `process()` cycle to issue a new attempt.

2. **Consume the presignature after the foreign-chain check.** Reorder `make_verify_foreign_tx_leader` so that `execute_foreign_chain_request` is called first, and the presignature is taken from the store only after the foreign-chain result is confirmed. This eliminates the presignature waste even if the non-retryable signal is not yet propagated.

3. **Track issue #3477** (already open) to completion, ensuring the sub-quorum terminal behavior is implemented before the foreign-chain verification feature is used in production at scale.

---

### Proof of Concept

1. Operator deploys the MPC network with a `ForeignTx` domain and registers a foreign chain (e.g., Bitcoin) with `rpc_quorum = 2` and two providers that return different block hashes for the same transaction.

2. Attacker calls `verify_foreign_transaction` with a Bitcoin transaction ID that causes the two providers to disagree on the block hash (e.g., a transaction in a recently-reorganized block).

3. The leader node's `make_verify_foreign_tx_leader` executes:
   - Takes presignature #1 from the store.
   - Calls `execute_foreign_chain_request` → `FanOut` returns `InspectorResponseMismatch`.
   - Returns `Err`, dropping presignature #1.

4. The `GenerationAttempt` `Arc` is dropped. On the next `CHECK_EACH_REQUEST_INTERVAL` (1 second), `QueuedRequest::process` issues attempt #2 → presignature #2 is consumed and dropped.

5. This repeats 10 times (`MAX_ATTEMPTS_PER_REQUEST_AS_LEADER`). The request is then dropped with `DropReason::MaxAttemptsExceeded`.

6. Result: 10 presignatures consumed, 0 signatures produced. The attacker can repeat this for every new `verify_foreign_transaction` request they submit, continuously draining the presignature pool. [8](#0-7) [9](#0-8) [10](#0-9)
