### Title
Fabricated `weight` in `NewPeak` causes honest peers to be banned during weight proof validation — (`chia/full_node/sync_store.py`, `chia/full_node/full_node.py`)

### Summary

An unprivileged remote peer can send a `NewPeak` message with an arbitrarily large fabricated `weight` for a real, known block hash. The victim node stores this weight without any validation, selects it as the sync target, then bans honest peers whose valid weight proofs do not match the fabricated value.

### Finding Description

**Step 1 — Unauthenticated weight storage.**

`new_peak()` in `full_node.py` calls `peer_has_block` with the peer-supplied `request.weight` directly, before any block validation: [1](#0-0) 

`peer_has_block` stores it verbatim with no cross-check against the actual block: [2](#0-1) 

**Step 2 — Fabricated weight wins the heaviest-peak election.**

`get_heaviest_peak()` selects purely by the stored `weight` field: [3](#0-2) 

A fabricated `weight = 2**64` beats every honest peer's entry.

**Step 3 — Sync target is set to the fabricated peak.**

`_sync()` uses the result of `get_heaviest_peak()` as `target_peak` and passes its weight directly to `request_validate_wp`: [4](#0-3) 

**Step 4 — Honest peers are banned.**

`request_validate_wp` checks that the weight proof's actual weight equals `peak_weight` (the fabricated `2**64`). An honest peer's valid proof carries the real weight, so the check fails and the peer is banned for `CONSENSUS_ERROR_BAN_SECONDS = 600` seconds: [5](#0-4) [6](#0-5) 

### Impact Explanation

Every honest peer that is randomly selected to serve the weight proof gets banned for 10 minutes. The attacker can reconnect (or use multiple sybil peers) and repeat the attack continuously, keeping all honest peers banned and preventing the victim node from ever completing a long sync. This is a **High** impact: long-lived inability for an honest node to process sync updates under normal network assumptions.

### Likelihood Explanation

The `NewPeak` message requires no authentication, no stake, and no proof of work. Any peer that can establish a TCP connection to port 8444 can send it. The real block hash needed for the attack is trivially obtained from any block explorer or peer. The attack is repeatable with no cost to the attacker.

### Recommendation

Validate the claimed weight before storing it in `SyncStore`. At minimum, after the block at `target_peak.height` is fetched from peers during `_sync()`, cross-check `target_peak_response.block.weight == target_peak.weight` before accepting the peer's claimed weight as the sync target. Alternatively, use the weight from the fetched block itself (not the peer's `NewPeak` claim) when calling `request_validate_wp`.
