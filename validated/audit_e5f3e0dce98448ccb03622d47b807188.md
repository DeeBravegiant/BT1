Audit Report

## Title
ETH Deposit Limit Check Omits Deposit Amount, Allowing Limit Bypass - (File: contracts/LRTDepositPool.sol)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` applies an asymmetric comparison for ETH vs. ERC20 assets. The ETH branch tests only whether the current total already exceeds the cap, not whether the incoming deposit would push it over. Any unprivileged caller can therefore deposit ETH past the admin-configured limit in a single transaction.

## Finding Description
In `_checkIfDepositAmountExceedesCurrentLimit` (L676–682), the ETH branch returns `totalAssetDeposits > depositLimit` while the ERC20 branch returns `totalAssetDeposits + amount > depositLimit`: [1](#0-0) 

The `amount` parameter is silently dropped for ETH. The function is the sole deposit-cap gate, called from `_beforeDeposit` at L661–663: [2](#0-1) 

which is invoked by the public, permissionless `depositETH` entry point at L76–93: [3](#0-2) 

Exploit flow:
1. `depositLimitByAsset[ETH_TOKEN] = 100 ether`, `getTotalAssetDeposits(ETH_TOKEN) = 99 ether`.
2. Attacker calls `depositETH{value: 10 ether}(0, "")`.
3. Check: `99 ether > 100 ether` → `false` → no revert.
4. `_mintRsETH` executes; on-chain ETH total becomes 109 ether, 9 ether above the cap.
5. An equivalent ERC20 deposit would correctly revert: `99 + 10 > 100` → `true` → `MaximumDepositLimitReached`.

No existing guard compensates for the missing `amount` term.

## Impact Explanation
The deposit limit is the protocol's primary mechanism for capping ETH exposure. Because the ETH branch never includes the incoming amount, a single deposit can push total ETH holdings arbitrarily above the configured cap. This directly maps to **Low — Contract fails to deliver promised returns**: the deposit ceiling guarantee is broken for ETH. If the limit was set to bound EigenLayer strategy capacity, excess deposits could also cause downstream operational failures, elevating to **Medium — Temporary freezing of funds**.

## Likelihood Explanation
The entry path is fully permissionless — any address can call `depositETH` with `msg.value > 0`. No special role, front-running, or external dependency is required. The vulnerable condition (`totalAssetDeposits ≤ depositLimit` and `totalAssetDeposits + depositAmount > depositLimit`) is the normal boundary state the limit is designed to enforce, making this trivially reachable. Likelihood is **High**.

## Recommendation
Add `amount` to the ETH branch to match the ERC20 branch:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

## Proof of Concept
```solidity
// Foundry test sketch
function test_ethDepositBypassesLimit() public {
    // Admin sets limit to 100 ether
    vm.prank(admin);
    lrtConfig.setDepositLimitByAsset(LRTConstants.ETH_TOKEN, 100 ether);

    // Seed pool with 99 ether via prior deposits
    vm.deal(depositor1, 99 ether);
    vm.prank(depositor1);
    lrtDepositPool.depositETH{value: 99 ether}(0, "");

    // Attacker deposits 10 ether — should revert, but does not
    vm.deal(attacker, 10 ether);
    vm.prank(attacker);
    lrtDepositPool.depositETH{value: 10 ether}(0, ""); // no revert

    // Total ETH now 109 ether, exceeding the 100 ether cap
    assertGt(lrtDepositPool.getTotalAssetDeposits(LRTConstants.ETH_TOKEN), 100 ether);
}
```

    function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
        uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
        if (asset == LRTConstants.ETH_TOKEN) {
            return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));
        }
        return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
    }
