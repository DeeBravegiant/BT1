### Title
`respond_verify_foreign_tx` Does Not Bind `response.payload_hash` to the `request` Parameter — Cross-Request Payload Substitution Attack - (File: `crates/contract/src/lib.rs`)

### Summary

`respond_verify_foreign_tx` verifies that `response.signature` is a valid MPC signature over `response.payload_hash`, but it never checks that `response.payload_hash` was actually computed from the `request` parameter supplied in the same call. A single malicious attested participant can front-run the legitimate response for pending request A by substituting a valid MPC-signed response that was produced for a completely different request B, causing the original caller to receive fabricated foreign-chain verification data.

### Finding Description

`respond_verify_foreign_tx` in `crates/contract/src/lib.rs` accepts two independent arguments: `request: VerifyForeignTransactionRequest` (used only as a lookup key into `pending_verify_foreign_tx_requests`) and `response: VerifyForeignTransactionResponse` (containing `payload_hash` and `signature`). [1](#0-0) 

The signature check performed is:

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,
).is_ok()
```

This only proves that `response.signature` is a valid ECDSA signature over `response.payload_hash` under the MPC root public key. It does **not** prove that `response.payload_hash` was derived from the `request` that is being resolved. [2](#0-1) 

The `payload_hash` that the MPC network is supposed to sign is `SHA-256(borsh(ForeignTxSignPayload::V1 { request: <ForeignChainRpcRequest>, values: <ExtractedValue[]> }))`. [3](#0-2) 

The `request` field inside `ForeignTxSignPayloadV1` must match the `request.request` field of the `VerifyForeignTransactionRequest` being resolved. The contract never enforces this binding.

After the signature check passes, `resolve_yields_for` is called with `&request` as the lookup key and `serde_json::to_vec(&response)` as the payload delivered to every waiting yield: [4](#0-3) 

The entire `response` — including the attacker-chosen `payload_hash` — is serialized and delivered verbatim to the original caller.

### Impact Explanation

A bridge or application contract that calls `verify_foreign_transaction(A)` to verify that foreign-chain transaction A was finalized will receive a `VerifyForeignTransactionResponse` whose `payload_hash` and `signature` correspond to a completely different transaction B. The response carries a genuine MPC threshold signature, so any downstream check that only verifies the signature over `payload_hash` (without also verifying that `payload_hash` encodes the correct request) will accept it as authentic.

Concretely:
- A bridge contract verifying Bitcoin tx A (to release funds) receives a response signed over Bitcoin tx B's block hash.
- The bridge cannot distinguish this from a legitimate response unless it independently knows the expected block hash in advance — which defeats the purpose of the verification service.
- This enables invalid bridge execution and potential double-spend conditions, matching the **High** impact category: *"forged foreign-chain verification … that causes invalid bridge execution or double-spend conditions."*

### Likelihood Explanation

The attacker must be a single attested participant — no threshold collusion is required. The attack steps are:

1. Observe a pending `VerifyForeignTransactionRequest` for victim request A on-chain.
2. Submit a separate `verify_foreign_transaction(request=B)` for a different transaction on the same domain.
3. Receive the MPC-signed `response_B = { payload_hash: H_B, signature: sig_B }` via the normal flow.
4. Call `respond_verify_foreign_tx(request=A, response=response_B)` before the honest node submits the legitimate response for A.

Step 4 passes all contract checks: `sig_B` is a valid MPC signature over `H_B`, and request A exists in the pending map. The contract resolves all yields queued under A with `response_B`.

Any single Byzantine attested participant below the signing threshold can execute this attack against any pending foreign-tx verification request.

### Recommendation

The contract must verify that `response.payload_hash` was computed from a `ForeignTxSignPayload` whose `request` field matches `request.request`. Since the contract does not know the extracted `values`, the response should be extended to include them, and the contract should recompute and compare the hash:

```rust
// In respond_verify_foreign_tx, after signature verification:
let expected_hash = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: request.request.clone(),
    values: response.extracted_values.clone(), // new field in response
})
.compute_msg_hash()
.map_err(|_| RespondError::InvalidPayloadHash)?;

if expected_hash != response.payload_hash {
    return Err(RespondError::PayloadHashMismatch.into());
}
```

Alternatively, include the `ForeignChainRpcRequest` hash as a separate field in the signed payload so the contract can verify the request binding without knowing the extracted values.

### Proof of Concept

**Setup**: Two pending requests exist for the same ForeignTx domain:
- Request A: `Bitcoin { tx_id: [0x01; 32], confirmations: 1, extractors: [BlockHash] }`
- Request B: `Bitcoin { tx_id: [0x02; 32], confirmations: 1, extractors: [BlockHash] }`

**Attack**:

```rust
// Attacker (attested participant) obtains a valid MPC response for request B:
let payload_b = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: ForeignChainRpcRequest::Bitcoin(BitcoinRpcRequest {
        tx_id: [0x02; 32].into(), ..
    }),
    values: vec![ExtractedValue::BitcoinExtractedValue(
        BitcoinExtractedValue::BlockHash([0xBB; 32].into()), // block hash of tx B
    )],
});
let hash_b = payload_b.compute_msg_hash().unwrap();
// sig_b is a valid MPC threshold signature over hash_b
let response_b = VerifyForeignTransactionResponse {
    payload_hash: hash_b,
    signature: sig_b,
};

// Attacker front-runs the legitimate respond for request A:
contract.respond_verify_foreign_tx(
    request_a,    // lookup key: resolves the yield for tx A's caller
    response_b,   // payload: contains tx B's block hash with valid MPC sig
);
// ✓ Signature check passes (sig_b is valid over hash_b)
// ✓ Request A found in pending map
// ✗ Caller of verify_foreign_transaction(A) receives tx B's block hash
```

The caller of `verify_foreign_transaction(A)` receives `response_b` — a genuine MPC-signed response — but one that attests to the block hash of transaction B, not A. Any bridge contract that uses this response to release funds locked against transaction A is operating on forged verification data. [5](#0-4) [3](#0-2) [6](#0-5)

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

pub struct ForeignTxSignPayloadV1 {
    pub request: ForeignChainRpcRequest,
    pub values: Vec<ExtractedValue>,

impl ForeignTxSignPayload {
    pub fn compute_msg_hash(&self) -> std::io::Result<Hash256> {
        let mut hasher = sha2::Sha256::new();
        borsh::BorshSerialize::serialize(self, &mut hasher)?;
        Ok(Hash256(hasher.finalize().into()))
    }

pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
