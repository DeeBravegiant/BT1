### Title
Stale `cancel_votes` Across Domain Transitions Allows Single Participant to Unilaterally Cancel Keygen — (`crates/contract/src/state/initializing.rs`)

---

### Summary

`InitializingContractState.cancel_votes` is a `BTreeSet` that accumulates votes across domain transitions but is **never cleared** when `vote_pk` advances the state to the next domain. Combined with a `next_domain_id` guard that is fixed for the entire `InitializingContractState` lifetime, T-1 stale votes from a prior domain's phase can be combined with a single fresh vote in the next domain to fire the cancellation threshold check — allowing one participant acting alone to unilaterally cancel an in-progress keygen session.

---

### Finding Description

**Root cause — `cancel_votes` is never cleared on domain advancement:**

In `vote_pk`, when a domain completes, the code creates a new `KeyEvent` for the next domain but does **not** reset `cancel_votes`:

```rust
// crates/contract/src/state/initializing.rs, lines 86-91
if let Some(next_domain) = self.domains.get_domain_by_index(self.generated_keys.len()) {
    self.generating_key = KeyEvent::new(
        self.epoch_id,
        next_domain.clone(),
        self.generating_key.proposed_parameters().clone(),
    );
    // ← cancel_votes is NOT cleared here
}
``` [1](#0-0) 

**The `next_domain_id` guard does not distinguish between domains:**

The guard at line 121 compares against `self.domains.next_domain_id()`, which is the ID *after all new domains* and is a fixed constant for the entire `InitializingContractState` lifetime — it does not change as individual domains complete:

```rust
// line 121
if next_domain_id != self.domains.next_domain_id() {
    return Err(InvalidParameters::NextDomainIdMismatch.into());
}
``` [2](#0-1) 

`DomainRegistry::next_domain_id()` simply returns the stored field, which is set once at construction and never mutated during domain-by-domain key generation: [3](#0-2) 

**The threshold check counts all accumulated votes:**

```rust
// line 132
if self.cancel_votes.insert(participant) && self.cancel_votes.len() >= required_threshold {
``` [4](#0-3) 

`AuthenticatedParticipantId` is just a `ParticipantId` (a numeric ID) with no domain or round binding, so votes cast during domain 0's phase are indistinguishable from votes cast during domain 1's phase: [5](#0-4) 

---

### Impact Explanation

**Attack sequence (threshold T, N domains being added):**

1. T-1 participants call `vote_cancel(next_domain_id)` during domain 0's generation phase → T-1 entries accumulate in `cancel_votes`; threshold not reached, no transition.
2. Domain 0 completes via `vote_pk` → `generating_key` advances to domain 1; `cancel_votes` is **not cleared**, still holds T-1 entries.
3. A single (T-th) participant calls `vote_cancel(next_domain_id)` during domain 1's phase → `cancel_votes.insert()` returns `true` (new entry), `cancel_votes.len()` becomes T ≥ `required_threshold` → cancellation fires.

**Consequence:** The contract transitions to `RunningContractState` with only the keys generated so far. All remaining domain IDs are permanently deleted via `domains.retain_domains(self.generated_keys.len())`: [6](#0-5) 

As confirmed by the code comment and `retain_domains` tests, deleted domain IDs are **never reused**: [7](#0-6) 

This breaks the invariant that cancellation requires T independent votes within a single coherent decision round. A single participant below threshold can permanently destroy pending domain IDs and prevent those keys from ever being generated under the same IDs.

---

### Likelihood Explanation

The attack requires:
- T-1 participants to have voted to cancel during a prior domain's phase (they may have done so legitimately, then changed their minds after domain 0 succeeded).
- One additional participant to call `vote_cancel` in the next domain.

This is reachable by any single participant who is aware of the stale vote accumulation. No collusion above threshold is required; the T-1 prior voters need not be coordinating with the attacker.

---

### Recommendation

Clear `cancel_votes` whenever `vote_pk` advances to the next domain:

```rust
// In vote_pk, after advancing generating_key:
self.generating_key = KeyEvent::new(...);
self.cancel_votes.clear(); // ← add this
``` [8](#0-7)

#[near(serializers=[borsh, json])]
#[derive(Debug, Clone, PartialEq, Eq, PartialOrd, Ord)]
pub struct AuthenticatedParticipantId(ParticipantId);
impl AuthenticatedParticipantId {
    pub fn get(&self) -> ParticipantId {
        self.0
    }

    pub fn new(participants: &Participants) -> Result<Self, Error> {
        let signer = env::signer_account_id();
        participants
            .participants()
            .iter()
            .find(|(a_id, _, _)| *a_id == signer)
            .map(|(_, p_id, _)| AuthenticatedParticipantId(*p_id))
            .ok_or_else(|| InvalidState::NotParticipant { account_id: signer }.into())
    }
