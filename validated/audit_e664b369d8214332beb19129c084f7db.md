Audit Report

## Title
`ChainlinkPriceOracle.getAssetPrice()` Accepts Stale and Non-Positive Chainlink Prices Without Validation - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary
`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards all validity fields, accepting stale, unfinalized, or non-positive prices without any checks. The same contract suite already implements all three required checks in `ChainlinkOracleForRSETHPoolCollateral.getRate()`, confirming the team is aware of the requirement. A stale high price flows directly into rsETH minting, allowing a depositor to receive more rsETH than the deposited LST is worth, diluting the yield of all existing rsETH holders.

## Finding Description
`contracts/oracles/ChainlinkPriceOracle.sol` L49–55:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);
    (, int256 price,,,) = priceFeed.latestRoundData();
    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

All five return values `(roundId, answer, startedAt, updatedAt, answeredInRound)` are available; only `answer` is used. No check is performed for:

- **Staleness**: `updatedAt` is never compared to `block.timestamp`.
- **Incomplete round**: `updatedAt == 0` is not rejected.
- **Stale round**: `answeredInRound < roundId` is not rejected.
- **Non-positive price**: `price <= 0` is not rejected; `uint256(price)` silently wraps a negative `int256` to an astronomically large value.

By contrast, `contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol` L26–37 performs all three checks:

```solidity
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
```

The stale price propagates through two critical paths:

**Path 1 – Deposit minting** (`contracts/LRTDepositPool.sol`):
`depositAsset()` (L99) → `_beforeDeposit()` (L648) → `getRsETHAmountToMint()` (L506) → `lrtOracle.getAssetPrice(asset)` (L520) → `ChainlinkPriceOracle.getAssetPrice()`.

**Path 2 – rsETH price update** (`contracts/LRTOracle.sol`):
`updateRSETHPrice()` (L87) → `_updateRsETHPrice()` (L214) → `_getTotalEthInProtocol()` (L331) → `getAssetPrice(asset)` (L339) for every supported LST.

## Impact Explanation
**High – Theft of unclaimed yield.**

If a Chainlink LST/ETH feed (e.g., stETH/ETH) goes stale at a price above the current market price, any depositor calling `depositAsset()` receives more rsETH than the deposited LST is worth. The excess rsETH represents a claim on ETH that was not contributed, directly diluting the backing of all existing rsETH holders and constituting theft of their accrued yield. The `minRSETHAmountExpected` slippage parameter in `depositAsset()` provides no protection because the inflated price benefits the depositor.

## Likelihood Explanation
**Low.** Chainlink LST/ETH feeds on Ethereum mainnet operate on a 24-hour heartbeat and a 0.5% deviation threshold. However, feeds can be deprecated without notice (leaving the last answer permanently stale), and during extreme network congestion keeper transactions may fail to land within the heartbeat window. No attacker action is required beyond waiting for a stale feed and then calling `depositAsset()`. A negative `int256` answer (possible on a malfunctioning feed) would wrap to a huge `uint256`, immediately corrupting the rsETH price and potentially triggering protocol-wide insolvency.

## Recommendation
Apply the same validation already present in `ChainlinkOracleForRSETHPoolCollateral`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);
    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
        priceFeed.latestRoundData();

    if (answeredInRound < roundId) revert StalePrice();
    if (updatedAt == 0) revert IncompleteRound();
    if (price <= 0) revert InvalidPrice();
    // Optional heartbeat check:
    // if (block.timestamp - updatedAt > HEARTBEAT) revert StalePrice();

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

## Proof of Concept
1. Chainlink stETH/ETH feed last updated at `T − 25h`; `updatedAt` is stale but `latestRoundData()` still returns the old answer (e.g., `0.9990e18`).
2. Real stETH/ETH market price has dropped to `0.9900e18` (a 0.9% drop, within normal range).
3. Attacker calls `LRTDepositPool.depositAsset(stETH, 1000e18, minRsETH, "")`.
4. `getRsETHAmountToMint()` calls `lrtOracle.getAssetPrice(stETH)` → `ChainlinkPriceOracle.getAssetPrice(stETH)` → returns stale `0.9990e18`.
5. `rsethAmountToMint = (1000e18 × 0.9990e18) / rsETHPrice` — attacker receives ~0.9% more rsETH than the deposited stETH is worth.
6. Attacker requests withdrawal, redeeming the excess rsETH for ETH contributed by other depositors, stealing their unclaimed yield.

**Foundry fork test plan:**
- Fork Ethereum mainnet at a block where a Chainlink LST/ETH feed's `updatedAt` is older than the heartbeat.
- Call `ChainlinkPriceOracle.getAssetPrice(stETH)` and assert it returns the stale value without reverting.
- Call `LRTDepositPool.depositAsset(stETH, 1000e18, 0, "")` and record `rsethAmountToMint`.
