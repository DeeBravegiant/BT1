### Title
Fast-Transfer-to-Other-Chain Permanently Deflates `locked_tokens` Counter, Freezing Redemption Path - (File: `near/omni-bridge/src/lib.rs`)

### Summary

In `fast_fin_transfer_to_other_chain`, the destination-chain `locked_tokens` counter is incremented by `amount_without_fee`. However, when the relayer later calls `claim_fee`, `claim_fee_callback` decrements that same counter by `fee`. The net counter after a complete fast-transfer + fee-claim cycle is `amount_without_fee − fee`, but the actual token supply on the destination chain is `amount_without_fee`. This permanent per-transfer deficit accumulates and eventually causes all subsequent destination→NEAR redemptions to panic with `InsufficientLockedTokens`, irreversibly locking user funds.

### Finding Description

**Step 1 – Lock in `fast_fin_transfer_to_other_chain`**

When a relayer executes a fast transfer whose recipient is on a non-NEAR chain, the bridge locks only `amount_without_fee` (the net amount the relayer actually provides) on the destination chain:

```rust
self.lock_tokens_if_needed(
    fast_transfer.get_destination_chain(),
    &fast_transfer.token_id,
    amount_without_fee,          // ← only amount_without_fee, NOT full amount
);
```

A `TransferMessage` is then stored with `amount = fast_transfer.amount` (the **full** amount including fee) and `origin_transfer_id = Some(fast_transfer.transfer_id)`. [1](#0-0) [2](#0-1) 

**Step 2 – Unlock in `claim_fee_callback`**

After the destination chain finalizes the transfer, the relayer calls `claim_fee`. Inside `claim_fee_callback`, the fee is computed as `transfer_message.amount.0 − denormalized_amount` (≈ `fee_amount`), and `send_fee_internal` is called:

```rust
let fee = transfer_message.amount.0 - denormalized_amount;
self.send_fee_internal(&transfer_message, fee_recipient, fee)
```

`send_fee_internal` then calls:

```rust
self.unlock_tokens_if_needed(transfer_message.get_destination_chain(), &token, token_fee);
// token_fee ≈ fee_amount
``` [3](#0-2) [4](#0-3) 

**Net accounting after one complete cycle:**

| Operation | Counter delta |
|---|---|
| `fast_fin_transfer_to_other_chain` | `+amount_without_fee` |
| `claim_fee_callback` | `−fee` |
| **Net** | `amount_without_fee − fee` |

Actual tokens on destination chain = `amount_without_fee`. The counter is permanently short by `fee`.

**Contrast with the normal (non-fast) path**

In `process_fin_transfer_to_other_chain`, the destination counter is incremented by the **full** amount (`fee + amount_without_fee`), so after `claim_fee` decrements by `fee`, the counter correctly equals `amount_without_fee`. [5](#0-4) 

**Step 3 – Redemption blocked**

When a destination-chain recipient later bridges tokens back to NEAR, `process_fin_transfer_to_near` calls:

```rust
self.unlock_tokens_if_needed(
    transfer_message.get_origin_chain(),   // destination chain
    &token,
    transfer_message.amount.0,             // amount_without_fee
);
```

`unlock_tokens` enforces:

```rust
require!(available >= amount, TokenLockError::InsufficientLockedTokens.as_ref());
```

Because the counter is `amount_without_fee − fee < amount_without_fee`, this check fails and the transaction panics. The user's tokens are permanently locked on the destination chain. [6](#0-5) [7](#0-6) 

### Impact Explanation

Every fast transfer to a non-NEAR chain followed by a fee claim permanently deflates `locked_tokens[destination][token]` by `fee`. After N such transfers with fee F each, the counter is short by `N × F` tokens. Any destination→NEAR redemption that would bring the counter below zero panics. This is an **irreversible fund lock / frozen redemption path** — the affected tokens on the destination chain can never be bridged back to NEAR.

### Likelihood Explanation

The comment inside `fast_fin_transfer_to_other_chain` explicitly states:

> "With transfers to other chain the fee will be claimed after finalization on the destination chain"

Relayers have a direct financial incentive to call `claim_fee`. This is the normal, expected operational flow. Any fast transfer to a non-NEAR chain with a non-zero fee triggers the bug. No special attacker action is required — a regular user initiating a fast transfer to SOL/EVM/Starknet/BTC is sufficient. [8](#0-7) 

### Recommendation

In `fast_fin_transfer_to_other_chain`, lock the **full** amount (not just `amount_without_fee`) on the destination chain, consistent with the normal transfer path:

```rust
// Before (incorrect):
self.lock_tokens_if_needed(
    fast_transfer.get_destination_chain(),
    &fast_transfer.token_id,
    amount_without_fee,
);

// After (correct):
self.lock_tokens_if_needed(
    fast_transfer.get_destination_chain(),
    &fast_transfer.token_id,
    fast_transfer.amount.0,   // full amount, fee will be unlocked by claim_fee
);
```

This mirrors the normal path in `process_fin_transfer_to_other_chain` where `fee + amount_without_fee` are both locked, and `claim_fee` later unlocks only the fee portion, leaving the counter at `amount_without_fee`.

### Proof of Concept

1. Token `T` is registered for both EVM (origin) and SOL chains. `locked_tokens[(SOL, T)] = 0`.
2. EVM user initiates a transfer of 100 T to SOL with fee = 10.
3. Relayer calls `fast_fin_transfer` (via `ft_transfer_call`) providing 90 T (`amount_without_fee`).
   - `lock_tokens_if_needed(SOL, T, 90)` → counter = 90.
4. Relayer calls `sign_transfer`; SOL mints 90 T to recipient.
5. Relayer calls `claim_fee` with proof of SOL finalization.
   - `fee = 100 − 90 = 10`; `unlock_tokens_if_needed(SOL, T, 10)` → counter = **80**.
6. SOL recipient tries to bridge 90 T back to NEAR.
   - `unlock_tokens_if_needed(SOL, T, 90)`: requires counter ≥ 90, but counter = 80 → **panic: `ERR_INSUFFICIENT_LOCKED_TOKENS`**.
7. The 90 T on SOL are permanently unredeemable.
