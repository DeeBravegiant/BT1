### Title
Weight Proof Sub-Epoch Sampling Uses Attacker-Controlled Seed, Enabling Crafted Proof to Bypass PoSpace Validation — (File: chia/full_node/weight_proof.py)

### Summary
The weight proof system derives its RNG seed from `summaries[-2].get_hash()`, where `summaries` is reconstructed entirely from attacker-supplied `weight_proof.sub_epochs` data. Because the proof creator knows this seed before finalizing which segments to include, they can precompute which segment index will be sampled per sub-epoch and craft a weight proof where only the sampled segment contains a valid proof of space. All other segments — which are never PoSpace-validated — can contain fabricated data. A syncing node that accepts such a proof is deceived into trusting a fake chain.

### Finding Description

**Seed derivation is circular with proof content.**

In `_create_proof_of_weight`, the seed is derived from the second-to-last sub-epoch summary:

```python
# use second to last ses as seed
seed = self.get_seed_for_proof(summary_heights, tip_rec.height)
rng = random.Random(seed)
weight_to_check = _get_weights_for_sampling(rng, tip_rec.weight, recent_chain)
``` [1](#0-0) 

`get_seed_for_proof` returns `ses.get_hash()` — the hash of the second-to-last `SubEpochSummary`: [2](#0-1) 

On the validation side, both `validate_weight_proof_single_proc` and `validate_weight_proof_inner` reconstruct the same seed from the summaries derived from the attacker-supplied proof:

```python
seed = summaries[-2].get_hash()
rng = random.Random(seed)
``` [3](#0-2) [4](#0-3) 

The `summaries` list is built from `weight_proof.sub_epochs` — data the attacker provides — via `_validate_sub_epoch_summaries` → `_map_sub_epoch_summaries`: [5](#0-4) 

**The sampled segment index is derived from this seed.**

Inside `_validate_sub_epoch_segments`, for each sub-epoch, exactly one segment is randomly selected for PoSpace validation:

```python
sampled_seg_index = rng.choice(range(len(segments)))
``` [6](#0-5) 

Only the segment at `sampled_seg_index` has its proof of space verified. All other segments skip PoSpace validation entirely:

```python
if sampled and sub_slot_data.is_challenge():
    ...
    required_iters = __validate_pospace(...)
    if required_iters is None:
        return False, ...
``` [7](#0-6) 

Non-sampled segments return `True` without any PoSpace check: [8](#0-7) 

### Impact Explanation

A malicious peer serving a weight proof can:

1. Construct a fake chain with inflated weight (fabricated sub-epoch summaries with plausible difficulty/weight values).
2. Compute `seed = summaries[-2].get_hash()` from their own proof data before finalizing it.
3. Compute `rng.choice(range(len(segments)))` for each sub-epoch to determine which segment index will be sampled.
4. Provide valid PoSpace only for the sampled segment in each sub-epoch; all other segments contain fabricated or invalid proofs of space.
5. Serve this proof to a syncing node.

The syncing node calls `validate_weight_proof` → `validate_weight_proof_inner`, reconstructs the same seed from the attacker-supplied summaries, samples the same segment indices, and validates only the segments the attacker prepared. The proof passes. The node accepts the fake chain as canonical.

This constitutes **forged weight proof trust** and **consensus divergence** — a syncing node is redirected to a fake chain, enabling reward diversion, double-spend facilitation, and long-range attack acceptance. Impact is **Critical**.

### Likelihood Explanation

Any unprivileged peer on the network can serve weight proofs. A syncing node selects a peer at random (`random.choice(peers_with_peak)`) and requests a weight proof from it: [9](#0-8) 

No authentication or trust is required to serve a weight proof. The attacker only needs to be reachable by the victim and claim a higher peak weight. The computation required to craft the proof is modest — the attacker precomputes the seed and selects which segments to populate with real PoSpace data.

### Recommendation

The RNG seed must not be derivable from data the proof creator controls. Options:

1. **Commit-reveal**: The verifier supplies a random challenge (nonce) that the prover must incorporate into the proof. The seed becomes `H(verifier_nonce || summaries[-2].get_hash())`, where the verifier_nonce is chosen after the proof is received.
2. **VDF-based randomness**: Derive the seed from a VDF output that cannot be predicted or ground by the proof creator.
3. **Peer-supplied entropy**: The syncing node sends a random 32-byte challenge with its `RequestProofOfWeight` message; the proof creator must include it in the seed, and the verifier checks this.

The core requirement is that the sampling seed must be unpredictable to the proof creator at the time they select which segments to include.

The attacker constructs `weight_proof.sub_epoch_segments` such that only `segments[sampled_seg_index]` for each sub-epoch contains a real proof of space. The validator calls `_validate_segment(..., sampled=True)` only for that index and `_validate_segment(..., sampled=False)` for all others — the latter returns `True` without any PoSpace check, so the fabricated segments pass silently.     def get_seed_for_proof(self, summary_heights: list[uint32], tip_height: uint32) -> bytes32:
        count = 0
        ses = None
        for sub_epoch_n, ses_height in enumerate(reversed(summary_heights)):
            if ses_height <= tip_height:
                count += 1
            if count == 2:
                ses = self.blockchain.get_ses(ses_height)
                break
        assert ses is not None
        seed = ses.get_hash()
        return seed
