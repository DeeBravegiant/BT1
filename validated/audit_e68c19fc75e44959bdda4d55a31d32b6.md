Audit Report

## Title
Chainlink Price Feed Staleness Not Validated in `ChainlinkPriceOracle.getAssetPrice()`, Enabling Stale-Price-Based rsETH Over-Minting - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary
`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards `updatedAt` and `answeredInRound`, accepting stale Chainlink prices without any heartbeat or round-completeness check. When a feed freezes at a price above the true market value, any depositor calling `depositAsset()` receives more rsETH than their assets are worth, diluting all existing rsETH holders. The same repository's `ChainlinkOracleForRSETHPoolCollateral.sol` already implements all three required guards, confirming the protocol is aware of the pattern.

## Finding Description

`ChainlinkPriceOracle.getAssetPrice()` silently discards every validation field:

```solidity
// contracts/oracles/ChainlinkPriceOracle.sol  L49-55
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);
    (, int256 price,,,) = priceFeed.latestRoundData();
    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
``` [1](#0-0) 

Neither `updatedAt` (heartbeat/staleness) nor `answeredInRound` vs `roundId` (incomplete-round) is checked. The sibling oracle in the same repository performs all three checks:

```solidity
// contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol  L27-32
(uint80 roundID, int256 ethPrice,, uint256 timestamp, uint80 answeredInRound) =
    AggregatorV3Interface(oracle).latestRoundData();
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
``` [2](#0-1) 

`ChainlinkPriceOracle` is registered as the price oracle for supported LST assets in `LRTOracle.assetPriceOracle`. It is consumed in two critical paths:

**Path 1 — `LRTDepositPool.getRsETHAmountToMint()`** calls `lrtOracle.getAssetPrice(asset)` directly to compute rsETH to mint:

```solidity
// contracts/LRTDepositPool.sol  L519-520
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [3](#0-2) 

The denominator `lrtOracle.rsETHPrice()` is the *stored* price, not freshly recomputed. A stale asset price inflates only the numerator, directly increasing rsETH minted per unit deposited.

**Path 2 — `LRTOracle._getTotalEthInProtocol()`** iterates every supported asset and calls `getAssetPrice(asset)`, feeding the result into `_updateRsETHPrice()`: [4](#0-3) 

The `pricePercentageLimit` guard inside `_updateRsETHPrice()` only applies to the `updateRSETHPrice()` call path; it does **not** protect `getRsETHAmountToMint()`, which is the direct deposit-time exploit path.

**Note on the negative-price sub-claim:** In Solidity 0.8.x, `uint256(negativeInt256) * 1e18` overflows and reverts due to checked arithmetic. The "near-`type(uint256).max`" scenario described in the submission does not materialise; the negative-price path causes a revert (DoS), not fund theft. The Critical-severity claim is therefore not supported. The valid impact is High.

## Impact Explanation

When a Chainlink feed freezes at a price above the true market value, every call to `depositAsset()` mints more rsETH than the deposited assets are worth. The excess rsETH is backed by nothing, diluting the ETH-per-rsETH ratio for all existing holders. This is a continuous, repeatable extraction of yield from existing rsETH holders for as long as the feed remains stale — matching the allowed impact **High: Theft of unclaimed yield**.

## Likelihood Explanation

Chainlink feeds have documented historical staleness events (e.g., during the March 2023 USDC depeg, feeds lagged significantly). The protocol supports multiple LST assets (stETH, ETHx, rETH, sfrxETH, swETH), each with its own feed; the probability that at least one feed experiences a staleness event over the protocol's lifetime is non-trivial. No heartbeat duration is enforced, so even a feed that is technically "live" but hours old is accepted. The entry path requires no privilege — any user calling `depositAsset()` triggers the vulnerable code path.

## Recommendation

Apply the same validation pattern already present in `ChainlinkOracleForRSETHPoolCollateral.sol`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);
    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
        priceFeed.latestRoundData();

    if (answeredInRound < roundId) revert StalePrice();
    if (updatedAt == 0) revert IncompleteRound();
    if (price <= 0) revert InvalidPrice();
    // Optional: per-feed heartbeat enforcement
    // if (block.timestamp - updatedAt > MAX_STALENESS) revert StalePrice();

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

## Proof of Concept

**Foundry fork test outline:**

1. Fork mainnet at a block where the stETH/ETH Chainlink feed is live.
2. Warp `block.timestamp` forward by 4 hours (past the 1-hour heartbeat) without advancing the feed's `updatedAt`.
3. Record the stale price returned by `ChainlinkPriceOracle.getAssetPrice(stETH)` — it reflects the pre-warp value (e.g., 1.05e18).
4. Call `LRTDepositPool.depositAsset(stETH, 1000e18, 0, "")` as an unprivileged attacker.
5. Observe `getRsETHAmountToMint` returns `(1000e18 * 1.05e18) / rsETHPrice` — ~5% more rsETH than the true 1.00e18 price would yield.
6. Verify `ChainlinkOracleForRSETHPoolCollateral.getRate()` reverts with `StalePrice` under the same conditions, confirming the disparity.
7. Repeat step 4 in a loop; each iteration extracts yield from existing rsETH holders with no additional privilege required.
