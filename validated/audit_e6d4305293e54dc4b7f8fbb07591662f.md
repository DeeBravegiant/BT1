Audit Report

## Title
Missing L2 Sequencer Uptime Feed Check Enables Stale-Price Overminting — (`contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol`)

## Summary

`ChainlinkOracleForRSETHPoolCollateral.getRate()` calls Chainlink's `latestRoundData()` on L2 (Arbitrum, Unichain) without verifying the Arbitrum Sequencer Uptime Feed. When the sequencer recovers from downtime, buffered price updates land simultaneously, leaving a window where the on-chain price is stale. Any unprivileged depositor can exploit this window to receive more rsETH than the real value of their deposit, diluting all existing rsETH holders.

## Finding Description

`ChainlinkOracleForRSETHPoolCollateral.getRate()` performs three checks — `answeredInRound < roundID`, `timestamp == 0`, and `ethPrice <= 0` — but performs no check against the Chainlink L2 Sequencer Uptime Feed:

```solidity
// contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol L26-37
function getRate() public view returns (uint256) {
    (uint80 roundID, int256 ethPrice,, uint256 timestamp, uint80 answeredInRound) =
        AggregatorV3Interface(oracle).latestRoundData();

    if (answeredInRound < roundID) revert StalePrice();
    if (timestamp == 0) revert IncompleteRound();
    if (ethPrice <= 0) revert InvalidPrice();
    ...
}
```

During sequencer downtime, Chainlink cannot push new rounds. When the sequencer recovers, the first new round that lands will have `answeredInRound == roundID` and a non-zero timestamp, so all three existing guards pass — even though the price reflects the pre-downtime state.

This oracle is wired into both pool contracts as `supportedTokenOracle[token]`:

- **`RSETHPoolV3.viewSwapRsETHAmountAndFee`** (L331–334): `rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate` — mints new `wrsETH` to the depositor.
- **`RSETHPoolNoWrapper.viewSwapRsETHAmountAndFee`** (L308–311): same formula — transfers pre-minted rsETH from the pool's balance.

`RSETHPriceFeed.latestRoundData()` (L68) has the same gap, exposing a stale rsETH/USD composite price to external consumers (e.g., lending protocols on Arbitrum).

## Impact Explanation

**High — Theft of unclaimed yield.**

The minting formula `rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate` means an inflated `tokenToETHRate` directly inflates the rsETH minted per unit of deposited token. The attacker receives rsETH backed by less real ETH value than the protocol expects, diluting the backing ratio for all existing rsETH holders. Their proportional claim on the underlying ETH pool is reduced — this is a concrete, quantifiable theft of yield/value from existing holders. `RSETHPoolNoWrapper` (deployed on Arbitrum per its NatSpec) has no daily mint cap, so the attack is bounded only by the pool's rsETH balance. `RSETHPoolV3` has a daily mint limit that partially caps damage but does not eliminate it.

## Likelihood Explanation

**Medium.** Arbitrum sequencer outages are documented and recurring (multiple incidents in 2022–2023, some lasting 30+ minutes). No special privilege is required — any address can call `deposit(token, amount, referralId)` on either pool. The attacker only needs to monitor sequencer status (publicly observable via the Sequencer Uptime Feed or block explorers) and submit a deposit immediately after recovery, before Chainlink prices update on-chain.

## Recommendation

Follow the [Chainlink L2 Sequencer Uptime Feed documentation](https://docs.chain.link/data-feeds/l2-sequencer-feeds). Add a sequencer check and a grace period inside `ChainlinkOracleForRSETHPoolCollateral.getRate()` and `RSETHPriceFeed.latestRoundData()`:

```solidity
(, int256 answer, uint256 startedAt,,) = sequencerUptimeFeed.latestRoundData();
if (answer != 0) revert SequencerDown();
if (block.timestamp - startedAt < GRACE_PERIOD) revert GracePeriodNotOver();
```

`GRACE_PERIOD` should be at least 1 hour (Chainlink's recommendation) to allow prices to fully catch up after sequencer recovery. The `sequencerUptimeFeed` address should be set at construction time and be immutable.

## Proof of Concept

1. Arbitrum sequencer goes offline. wstETH/ETH price on mainnet drops from 1.15 ETH to 1.05 ETH. Chainlink on Arbitrum cannot push new rounds.
2. Sequencer recovers. The first new Chainlink round lands with `answeredInRound == roundID` and a valid timestamp, but the `answer` is still 1.15 ETH (the last pre-downtime value). All three guards in `getRate()` pass.
3. Attacker calls `RSETHPoolNoWrapper.deposit(wstETH, 100e18, "")` on Arbitrum.
4. `viewSwapRsETHAmountAndFee` computes `tokenToETHRate = 1.15e18` (stale) and `rsETHToETHrate = 1.08e18` (correct cross-chain rate).
5. Attacker receives `100 * 1.15 / 1.08 ≈ 106.48` rsETH instead of the correct `100 * 1.05 / 1.08 ≈ 97.22` rsETH — a ~9.5% overmint at the expense of existing rsETH holders.
6. Attacker bridges or sells the excess rsETH before the price corrects.

**Foundry fork test plan:**
- Fork Arbitrum mainnet at a block just after a historical sequencer recovery event.
- Deploy or point to the live `RSETHPoolNoWrapper` and `ChainlinkOracleForRSETHPoolCollateral`.
- Mock the Chainlink aggregator to return the pre-downtime price (simulating the stale window).
- Call `deposit(wstETH, 100e18, "")` as an unprivileged address.
- Assert that `rsETHAmount` received exceeds the fair value computed using the correct post-recovery price.
- Confirm the three existing guards (`answeredInRound < roundID`, `timestamp == 0`, `ethPrice <= 0`) all pass despite the stale price.
