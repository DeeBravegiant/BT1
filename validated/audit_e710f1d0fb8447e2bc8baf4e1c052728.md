### Title
Stale Foreign-Chain Attestation Replay via Unbound `payload_hash` in `respond_verify_foreign_tx` - (File: crates/contract/src/lib.rs)

### Summary

`respond_verify_foreign_tx` verifies that the submitted signature is valid over the caller-supplied `response.payload_hash`, but never verifies that `payload_hash` is freshly derived from the current foreign-chain state for the pending request. Any single attested MPC participant can replay a previously-observed, on-chain `(payload_hash, signature)` pair against a newly-pending request for the same foreign transaction, delivering a stale attestation to the caller.

### Finding Description

When `respond_verify_foreign_tx` is called, the contract performs the following signature check: [1](#0-0) 

The `payload_hash` used for verification is taken directly from `response.payload_hash` — a caller-supplied field — not recomputed from the pending request's content: [2](#0-1) 

The contract only checks:
1. The caller is an attested participant.
2. The ECDSA signature is valid over the supplied `payload_hash` under the root MPC public key.
3. The `request` exists in `pending_verify_foreign_tx_requests`.

It does **not** check that `payload_hash` equals `SHA-256(borsh(ForeignTxSignPayload{ request, current_values }))` for the current foreign-chain state. The `values` (extracted foreign-chain observations) are embedded in the hash but are never re-verified by the contract. [3](#0-2) 

Because the contract has no "completed requests" set and no per-invocation nonce, the same `request` can be submitted again after it has been resolved: [4](#0-3) 

The `VerifyForeignTransactionRequest` key is caller-agnostic (confirmed by the fan-out test), so any account can re-submit the same request: [5](#0-4) 

The old `(payload_hash, signature)` pair is publicly visible on-chain from the first `respond_verify_foreign_tx` call. Any attested participant can extract it and replay it.

### Impact Explanation

A single malicious attested MPC node can replay a previously-computed `VerifyForeignTransactionResponse` against a newly-pending request for the same foreign transaction. The caller receives a stale attestation — for example, a Bitcoin block hash from before a chain reorganization. A bridge contract consuming this response would believe a transaction is confirmed in a block that no longer contains it, enabling double-spend or invalid bridge execution.

This maps to: **High — Cross-chain replay / forged foreign-chain verification causing invalid bridge execution or double-spend conditions.**

### Likelihood Explanation

- Requires only **one** malicious attested MPC participant (below threshold).
- The old response is publicly available on-chain; no secret material is needed.
- The attack is only profitable when the foreign chain reorganizes after the first attestation, which is realistic for chains with low confirmation counts (e.g., Bitcoin with `confirmations=1`).
- The malicious node simply races honest nodes on the second submission.

### Recommendation

Bind the response to a specific invocation of the request by including a per-request nonce or `data_id` (the NEAR yield ID) inside the signed payload. Concretely:

1. Include the `data_id` (yield resume ID) in `ForeignTxSignPayloadV1` so each signing session produces a unique hash even for identical requests.
2. In `respond_verify_foreign_tx`, recompute the expected `payload_hash` from the request and the submitted `values`, and reject any response whose `payload_hash` does not match.

Alternatively, maintain a `completed_foreign_tx_requests` set (keyed by `(request, data_id)`) and reject responses for already-resolved invocations.

### Proof of Concept

**Setup**: MPC network is running. Bitcoin tx `X` is confirmed in block `A`.

1. Alice calls `verify_foreign_transaction({ tx_id: X, confirmations: 1, extractors: [BlockHash] })`.
2. MPC nodes query Bitcoin RPC, observe block hash `A`, compute `payload_hash_1 = SHA-256(borsh({ request, [BlockHash(A)] }))`, produce threshold signature `sig_1`. One node calls `respond_verify_foreign_tx(request, { payload_hash: payload_hash_1, signature: sig_1 })`. Alice receives the attestation. `payload_hash_1` and `sig_1` are now publicly visible on-chain.
3. Bitcoin reorganizes: tx `X` is now in block `B` (or removed entirely).
4. Bob calls `verify_foreign_transaction({ tx_id: X, confirmations: 1, extractors: [BlockHash] })` — identical request, newly pending.
5. The malicious attested node calls `respond_verify_foreign_tx(request, { payload_hash: payload_hash_1, signature: sig_1 })` before honest nodes respond.
6. The contract checks: (a) caller is attested ✓, (b) `sig_1` is valid over `payload_hash_1` under root key ✓, (c) request is pending ✓. It resolves Bob's yield with the stale response. [6](#0-5) 

Bob's bridge contract receives `{ payload_hash: payload_hash_1, signature: sig_1 }` — an attestation that tx `X` is in block `A` — and acts on it, even though `X` is no longer in block `A`. The bridge executes an invalid inbound transfer, enabling a double-spend.

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
