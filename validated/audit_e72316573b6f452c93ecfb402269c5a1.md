All cited code paths are confirmed in the repository. The vulnerability is real and fully substantiated.

Audit Report

## Title
Push-payment in `collectFees` to `adminFeeDestination` permanently blocks fee collection and fee-configuration updates when pool token is USDC — (`metric-core/contracts/MetricOmmPool.sol`, `metric-core/contracts/MetricOmmPoolFactory.sol`)

## Summary
`collectFees` in `MetricOmmPool` pushes accrued fees directly to `adminFeeDestination_` via `IERC20.safeTransfer` before clearing accumulators. Both `setPoolProtocolFee` and `setPoolAdminFees` in the factory call `collectFees` atomically as their first step; if `adminFeeDestination` is USDC-blacklisted, all three entry-points revert permanently, locking protocol fees in the pool and freezing fee-rate governance for the affected pool.

## Finding Description
`transferToken0`/`transferToken1` are bare `safeTransfer` wrappers: [1](#0-0) 

`collectFees` pushes to `adminFeeDestination_` before zeroing accumulators: [2](#0-1) 

`setPoolProtocolFee` (owner-only) calls `collectFees` as its first action, before writing any new state: [3](#0-2) 

`setPoolAdminFees` (pool-admin) does the same: [4](#0-3) 

The permissionless `collectPoolFees` also calls `collectFees` with the stored destination: [5](#0-4) 

`setPoolAdminFeeDestination` — the only recovery path — is gated to `onlyPoolAdmin` and does not call `collectFees`, so the factory owner cannot unilaterally fix the destination: [6](#0-5) 

If USDC blacklists `adminFeeDestination` (or the pool admin sets it to a blacklisted address), every `safeTransfer` to that address reverts. Because the accumulator reset at lines 429–430 is only reached after all transfers succeed, `notionalFeeToken0Scaled`/`notionalFeeToken1Scaled` are never cleared, fees remain in the pool balance, and both fee-collection and fee-configuration calls revert indefinitely.

## Impact Explanation
Protocol fees accrued in the pool are permanently inaccessible — a direct loss of protocol fee revenue meeting the "direct loss of protocol fees" allowed impact. Additionally, the factory owner loses the ability to adjust protocol fee rates for the affected pool via `setPoolProtocolFee`, constituting broken core fee-governance functionality. USDC is explicitly in scope as a non-standard ERC20 exception.

## Likelihood Explanation
USDC's blacklist capability is documented and actively exercised. Any pool whose token0 or token1 is USDC and whose `adminFeeDestination` is ever blacklisted — whether by USDC acting on that address for unrelated reasons or by the pool admin deliberately calling `setPoolAdminFeeDestination` with a known-blacklisted address — triggers the condition automatically. No special privilege beyond pool-admin role is required for the deliberate path; the external USDC-blacklisting path requires no on-chain action at all.

## Recommendation
1. **Decouple fee collection from fee configuration**: Remove the `collectFees` call from `setPoolProtocolFee` and `setPoolAdminFees`; let fee collection happen in independent transactions.
2. **Pull-over-push**: Accrue fees to a claimable-balance mapping keyed by destination; let recipients pull their own share so a single bad destination cannot block the entire system.
3. **Wrap transfers in try/catch**: If atomic collection must be preserved, catch transfer failures, emit an event, and continue so that state updates and fee-rate changes are not blocked.

## Proof of Concept
1. Deploy a pool with USDC as token0; set `adminFeeDestination` to address `A`.
2. Execute several swaps to accumulate `notionalFeeToken0Scaled` / spread fees.
3. USDC blacklists address `A` (or pool admin calls `setPoolAdminFeeDestination(pool, blacklistedAddress)`).
4. Call `factory.collectPoolFees(pool)` → reverts at `IERC20(USDC).safeTransfer(A, amount)`.
5. Call `factory.setPoolProtocolFee(pool, newFee, newNotional)` as owner → reverts at the internal `collectFees` call; no state is written.
6. Call `factory.setPoolAdminFees(pool, newFee, newNotional)` as pool admin → same revert.
7. Confirm `notionalFeeToken0Scaled` / `notionalFeeToken1Scaled` remain non-zero and grow with each subsequent swap; tokens are permanently locked in the pool with no factory-owner recovery path.
