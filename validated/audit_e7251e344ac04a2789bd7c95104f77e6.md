### Title
Unbound `payload_hash` in `respond_verify_foreign_tx` Enables Cross-Request Replay by a Single Byzantine MPC Node - (File: `crates/contract/src/lib.rs`)

### Summary
`respond_verify_foreign_tx` verifies only that the ECDSA signature in the response is valid over `response.payload_hash` using the root public key. It does **not** verify that `response.payload_hash` is cryptographically derived from the `request` parameters. A single Byzantine MPC node (below signing threshold) can replay a `(payload_hash, signature)` pair obtained from any prior legitimate `verify_foreign_transaction` response to resolve a completely different pending request, delivering a forged foreign-chain verification result to the waiting caller.

### Finding Description

In `respond_verify_foreign_tx` the contract performs the following check:

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;
// Check the signature is correct against the root public key
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,
)
.is_ok()
``` [1](#0-0) 

The `payload_hash` is taken entirely from the **response** supplied by the calling MPC node. The contract never checks that this hash is a commitment to the `request` that is being resolved (i.e., the transaction ID, chain, extractors, and extracted values). After the signature check passes, the full `response` — including the attacker-chosen `payload_hash` — is serialised and delivered to every waiting yield:

```rust
pending_requests::resolve_yields_for(
    &mut self.pending_verify_foreign_tx_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
``` [2](#0-1) 

Contrast this with the regular `respond` path, where the payload hash is taken **from the request itself** (not from the response), making it impossible for the responder to substitute a different hash:

```rust
let payload_hash = request.payload.as_ecdsa().expect("Payload is not ECDSA");
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    payload_hash,
    &expected_public_key,
)
``` [3](#0-2) 

The design intent documented in `docs/foreign-chain-transactions.md` is that MPC nodes sign a canonical encoding of `(request, observed_values, observed_at)`:

> "If extraction succeeds, MPC signs a canonical encoding of `(request, observed_values, observed_at)` and returns the signature on-chain." [4](#0-3) 

The contract enforces none of this binding. The `payload_hash` is a free parameter chosen by the responding node.

### Impact Explanation

A single Byzantine MPC node (attested participant, below signing threshold) can:

1. Observe any prior on-chain `respond_verify_foreign_tx` call that produced a valid `(payload_hash_A, sig_A)` for request A (e.g., Bitcoin TX_A). Both values are public.
2. Wait for a different request B (e.g., Bitcoin TX_B) to appear in the pending map.
3. Call `respond_verify_foreign_tx(request_B, { payload_hash: payload_hash_A, sig: sig_A })`.
4. The contract accepts: `verify_ecdsa_signature(sig_A, payload_hash_A, root_pk)` is valid.
5. All yields queued under request B are drained and each waiting caller receives `{ payload_hash: payload_hash_A, sig: sig_A }` — a response that attests to TX_A, not TX_B.

The legitimate response for request B is permanently blocked because `resolve_yields_for` removes the map entry on first call. Any bridge contract that trusts the MPC contract to bind the returned `payload_hash` to the submitted transaction will accept the forged attestation, enabling invalid bridge execution or double-spend conditions.

### Likelihood Explanation

**Medium-High.** The attacker is a single Byzantine MPC node — explicitly within the "Byzantine participant strictly below the signing threshold" scope. No collusion is required. The only prerequisite is that at least one prior `verify_foreign_transaction` response exists on-chain (trivially satisfied in any live deployment). The attacker can observe pending requests from the NEAR blockchain and race to respond before honest nodes do.

### Recommendation

Bind `payload_hash` to the request on-chain. The simplest fix is to compute the expected payload prefix from the `VerifyForeignTransactionRequest` fields that are available to the contract (chain, transaction ID, extractors, payload version) and assert that `response.payload_hash` matches. If the full `ForeignTxSignPayload` cannot be reconstructed without the extracted values, the contract should at minimum verify that `payload_hash` is a hash of a structure that commits to the `request` fields, rejecting any response whose hash does not include the correct request commitment.

### Proof of Concept

```
// Step 1 – Attacker collects a prior valid response from on-chain history
(payload_hash_A, sig_A) = on_chain_response_for_request_A   // public data

// Step 2 – Victim submits a new request
user.call("verify_foreign_transaction", { request: request_B, tx_id: TX_B })
// → pending_verify_foreign_tx_requests[request_B] = [yield_B]

// Step 3 – Byzantine node replays the old (payload_hash_A, sig_A)
byzantine_node.call("respond_verify_foreign_tx", {
    request: request_B,                          // correct pending key
    response: { payload_hash: payload_hash_A,    // wrong hash (TX_A, not TX_B)
                signature: sig_A }               // valid sig over payload_hash_A
})

// Step 4 – Contract check (lib.rs:726-734):
//   verify_ecdsa_signature(sig_A, payload_hash_A, root_pk) → OK  ✓
//   pending_verify_foreign_tx_requests[request_B] removed
//   yield_B resumed with { payload_hash_A, sig_A }

// Step 5 – User's bridge contract receives attestation for TX_A, not TX_B
//          Legitimate response for TX_B is permanently blocked
``` [5](#0-4)

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }
