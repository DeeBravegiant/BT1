Audit Report

## Title
`depositAsset` Mints rsETH Against Stated `depositAmount` Rather Than Actual Received stETH Balance — (`contracts/LRTDepositPool.sol`)

## Summary
`depositAsset` computes the rsETH mint amount from the caller-supplied `depositAmount` before executing `safeTransferFrom`, and mints that pre-computed amount without ever measuring the actual balance change. For stETH — a shares-based rebasing token that is a core supported asset — `transferFrom` internally rounds down via integer share arithmetic, crediting 1–2 wei less than requested. The protocol therefore mints rsETH against a stated amount that exceeds the real backing, creating a persistent, monotonically accumulating under-collateralisation.

## Finding Description
In `depositAsset` (L99–118), the execution order is:

```solidity
uint256 rsethAmountToMint = _beforeDeposit(asset, depositAmount, minRSETHAmountExpected);
IERC20(asset).safeTransferFrom(msg.sender, address(this), depositAmount);
_mintRsETH(rsethAmountToMint);
```

`_beforeDeposit` (L648–670) calls `getRsETHAmountToMint(asset, depositAmount)` (L665), which computes:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

No balance snapshot is taken before or after `safeTransferFrom`. The actual tokens credited to the contract are never measured.

stETH is added as a supported asset in `LRTConfig.initialize` (L56) and is referenced throughout the protocol (`LRTConstants.ST_ETH_TOKEN`, `stakeEthForStETH`, `UnstakeStETH` adapter). stETH's `transferFrom` converts the requested token amount to shares via integer division (`getSharesByPooledEth`), then converts back, yielding `depositAmount − δ` (δ ∈ {1, 2}) in the contract. The contract holds less stETH than the amount used to compute the rsETH mint.

When `_updateRsETHPrice()` is subsequently called in `LRTOracle`, it uses `getTotalAssetDeposits(asset)` → `IERC20(asset).balanceOf(address(this))` (L444), which reflects the actual (lower) balance. The rsETH price is therefore computed against a TVL that is slightly smaller than the rsETH supply implies, diluting all holders by the accumulated shortfall.

No existing check compensates for this: `minRSETHAmountExpected` guards only against oracle slippage, not transfer shortfalls; `nonReentrant` and `whenNotPaused` are irrelevant to the accounting error.

## Impact Explanation
Every stETH deposit mints a fractionally larger rsETH amount than is backed by real assets. The discrepancy is 1–2 wei per call but accumulates monotonically with deposit volume. The rsETH price, recalculated from actual balances, will be infinitesimally lower than it would be with correct accounting, meaning the last redeemers in a full-withdrawal scenario cannot redeem their full proportional entitlement.

**Impact: Low — Contract fails to deliver promised returns (persistent, accumulating under-collateralisation).**

## Likelihood Explanation
stETH is a core, always-supported asset. Every ordinary user deposit of stETH through `depositAsset` triggers the path. No attacker privileges, special conditions, or external protocol compromise are required. The rounding shortfall is deterministic and occurs on virtually every stETH `transferFrom` call due to Lido's share arithmetic.

## Recommendation
Capture the contract's stETH balance before and after `safeTransferFrom` and use the measured delta as the basis for the rsETH mint:

```solidity
uint256 balanceBefore = IERC20(asset).balanceOf(address(this));
IERC20(asset).safeTransferFrom(msg.sender, address(this), depositAmount);
uint256 actualReceived = IERC20(asset).balanceOf(address(this)) - balanceBefore;

uint256 rsethAmountToMint = getRsETHAmountToMint(asset, actualReceived);
if (rsethAmountToMint < minRSETHAmountExpected) revert MinimumAmountToReceiveNotMet();
_mintRsETH(rsethAmountToMint);
```

## Proof of Concept
1. Alice calls `depositAsset(stETH, 1e18, 0, "")`.
2. `_beforeDeposit` calls `getRsETHAmountToMint(stETH, 1e18)` → returns `X` rsETH (L665).
3. `safeTransferFrom` credits `1e18 − 1` stETH to the pool (stETH share rounding).
4. `_mintRsETH(X)` mints rsETH computed from `1e18`, not `1e18 − 1` (L115).
5. Pool holds `1e18 − 1` stETH but has minted rsETH backed by `1e18` stETH — 1 wei short.
6. On the next `updateRSETHPrice()` call, `_getTotalEthInProtocol()` reads `balanceOf` (actual: `1e18 − 1`), computing a slightly lower rsETH price than the minted supply warrants.
7. Repeated across all stETH deposits, the cumulative shortfall grows proportionally, leaving the last withdrawers unable to redeem their full entitlement.

Foundry fork test plan: fork mainnet, deploy/configure the protocol with real stETH, call `depositAsset(stETH, 1e18, 0, "")`, assert `IERC20(stETH).balanceOf(depositPool) < 1e18` while `rsETH.totalSupply()` increased by `X` (computed from `1e18`), then call `updateRSETHPrice()` and assert `rsETHPrice` is lower than `(1e18 * stETHPrice) / X`.
