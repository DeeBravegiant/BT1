### Title
Missing witness-count guard in `updateUTxOTxWitness` allows spending unowned UTxO entries via `zip` truncation — (`eras/byron/ledger/impl/src/Cardano/Chain/UTxO/Validation.hs`)

### Summary

`updateUTxOTxWitness` zips the per-input address list against the attacker-supplied witness vector without first asserting the two lists have equal length. Haskell's `zip` silently truncates to the shorter list, so any inputs beyond the witness vector are consumed by `updateUTxOTx` with zero cryptographic authorization.

### Finding Description

In `updateUTxOTxWitness` the witness validation loop is:

```haskell
mapM_
  (uncurry $ validateWitness pmi sigData)
  (zip addresses (V.toList witness))
``` [1](#0-0) 

`addresses` has exactly `N = length (txInputs tx)` elements. `witness` is the attacker-controlled `Vector TxInWitness` from `taWitness ta`. If the attacker supplies `M < N` witnesses, `zip` produces only `M` pairs; `mapM_ (uncurry $ validateWitness …)` checks only those `M` pairs; the remaining `N − M` inputs are never checked. [2](#0-1) 

After the `whenTxValidation` block, `updateUTxOTx` unconditionally removes **all N inputs** from the UTxO and adds the transaction outputs:

```haskell
UTxO.union (S.fromList (NE.toList (txInputs tx)) </| utxo) (txOutputUTxO tx)
``` [3](#0-2) 

The `TxValidationError` type has no variant for a witness-count mismatch, confirming the check was never implemented: [4](#0-3) 

By contrast, the formal executable spec explicitly enforces `length wits == length ins` as a precondition:

```haskell
witnessed (Tx tx wits) utxo =
  length wits == length ins && all (isWitness tx utxo) (zip ins wits)
``` [5](#0-4) 

The production implementation diverges from the spec at exactly this point.

### Impact Explanation

An attacker who constructs an `ATxAux` with `N` inputs (at least one belonging to a key they do not control) and a witness vector of length `M < N` can spend the unwitnessed inputs. The resulting UTxO state transition is invalid — funds are transferred without authorization — but every honest node running this code accepts it, causing permanent divergence from the correct UTxO set. This matches the **Critical** impact: "Honest nodes accept an invalid block or transaction causing permanent ledger divergence requiring a hard fork."

### Likelihood Explanation

The exploit requires only the ability to submit a Byron-era transaction. The construction is trivial: build a 2-input `Tx`, sign only the first input, set `aTaWitness` to a length-1 vector. No privileged access, no key leakage, no governance majority is needed. The only practical constraint is that Byron is a historical era on mainnet; the vulnerability is fully exploitable on any Byron-capable network (private testnets, chain-replay environments, or any deployment that has not yet transitioned past Byron).

### Recommendation

Before the `zip`, assert that the witness vector length equals the input count:

```haskell
let nInputs  = length (NE.toList (txInputs tx))
    nWitness = V.length witness
nInputs == nWitness
  `orThrowError` UTxOValidationTxValidationError
    (TxValidationWitnessCountMismatch nInputs nWitness)
```

Add `TxValidationWitnessCountMismatch Int Int` to `TxValidationError` and its `EncCBOR`/`DecCBOR` instances. This mirrors the invariant already present in the executable spec.
