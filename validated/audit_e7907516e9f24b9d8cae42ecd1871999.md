## Attack Feasibility

A Byzantine party can:
1. Send `h_consistency` = SHA256("secp256k1") — matches the agreed curve name ✓
2. Commit to a P256 point `Qi_j` in round 1 (commitment covers P256 curve tag + point bytes)
3. In round 2, reveal the same P256 point + a valid P256 ZK proof for it

At the honest verifier:
- `h_consistency` check: SHA256("secp256k1") == SHA256("secp256k1") → **passes** ✓
- Commitment open with P256 `Qi._j`: HMAC matches (same bytes committed) → **passes** ✓
- `pi._j.verify(Qi._j, sid, j)`: derives P256 from `Qi._j.get_curve()`, checks P256 point on P256, verifies valid P256 ZK proof → **passes** ✓

There is **no check** that `Qi._j.get_curve() == curve` (the agreed curve). The comment at line 157 is incorrect — the ZK verify only checks the point is on its own claimed curve, not on the agreed curve.

## Impact

After the loop, `key.Qis` contains a mix of secp256k1 points (honest parties) and a P256 point (Byzantine party). Then:

This performs cross-curve point addition. The `ecc_point_t` union stores either an `EC_POINT*` (OpenSSL) or a `secp256k1::point_ptr_t`. ### Title
Missing agreed-curve validation on received `Qi` in `key_share_mp_t::dkg` allows Byzantine peer to inject a point on a different curve — (`src/cbmpc/protocol/ec_dkg.cpp`)

### Summary
`key_share_mp_t::dkg` relies on `h_consistency` to enforce curve agreement and delegates curve validation of received `Qi._j` to `uc_dl_t::verify`. However, `uc_dl_t::verify` derives the curve from `Qi._j.get_curve()` rather than from the agreed curve, so a Byzantine peer can send a point on a different curve with a valid ZK proof for that different curve, bypassing all guards.

The `h_consistency` check only verifies the curve name hash. After the commitment open at line 155, the code relies on the comment at line 157 ("curve check of Qi._j is done inside the zk verify function") to justify not explicitly checking `Qi._j.get_curve() == curve`. [2](#0-1) 

In `uc_dl_t::verify`, the curve is derived from the input point itself: [3](#0-2)

There is no check that `Qi._j.get_curve()` equals the agreed `curve`. A Byzantine party can commit to a P256 point, reveal it with a valid P256 ZK proof, pass the `h_consistency` check (by sending the correct secp256k1 hash), pass the commitment open (HMAC covers the P256 curve tag), and pass the ZK verify (valid P256 proof on a P256 point).

### Impact Explanation
Honest parties store the Byzantine party's P256 `Qi` in `key.Qis` and then compute `key.Q = SUM(key.Qis)`: [5](#0-4) 

This performs cross-curve point addition. The `ecc_point_t` union stores either an OpenSSL `EC_POINT*` or a secp256k1 internal pointer: [6](#0-5) 

Adding a P256 point using secp256k1 arithmetic causes type confusion, producing a garbage joint key `Q` or a crash. Honest parties accept an invalid DKG output, causing all downstream signing operations to fail or produce invalid results. This satisfies the "High" impact criterion: attacker-controlled points are accepted under the wrong curve.

### Likelihood Explanation
Any single Byzantine participant in a multi-party DKG can trigger this. No threshold collusion is required. The attack requires only the ability to craft a serialized `ecc_point_t` with a different curve tag, which is trivially achievable by any party controlling their own message bytes.

### Recommendation
Add an explicit curve check immediately after the commitment open, before the ZK verify:

```cpp
if (rv = crypto::commitment_t(sid_i._j, job.get_pid(j)).set(rho._j, c._j).open(Qi._j)) return rv;
if (Qi._j.get_curve() != curve) return coinbase::error(E_CRYPTO, "Qi curve mismatch");
if (rv = pi._j.verify(Qi._j, sid, j)) return rv;
```

The comment at line 157 should be removed or corrected — the ZK verify does not and cannot enforce agreement with the protocol-level curve.

### Proof of Concept
In a 3-party DKG with agreed curve secp256k1:
1. Byzantine party P2 generates a P256 key share: `x2 = rand()`, `Qi2 = x2 * G_P256` (P256 point).
2. P2 commits to `Qi2` (commitment HMAC covers P256 curve tag + compressed bytes).
3. P2 sends `h_consistency` = SHA256("secp256k1") — matches honest parties.
4. P2 generates a valid `uc_dl_t` proof for `Qi2` using P256 parameters.
5. P2 broadcasts `Qi2` (P256), `rho2`, `pi2` in round 2.
6. Honest parties P0, P1 verify:
   - `h_consistency` check: SHA256("secp256k1") == SHA256("secp256k1") → passes.
   - Commitment open with P256 `Qi2` → passes (HMAC matches).
   - `pi2.verify(Qi2, sid, 2)`: derives P256 from `Qi2.get_curve()`, verifies valid P256 proof → passes.
7. `key.Qis` = {P0: secp256k1 point, P1: secp256k1 point, P2: P256 point}.
8. `key.Q = SUM(key.Qis)` → cross-curve addition → garbage/crash.
9. Assert: DKG completes without error on honest parties but `key.Q` is invalid.
