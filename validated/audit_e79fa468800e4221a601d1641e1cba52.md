Audit Report

## Title
Missing Staleness Validation on Chainlink `latestRoundData()` Enables Stale-Price Minting Abuse - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary
`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards every return value except `answer`, performing no check on `updatedAt`, `answeredInRound`, or price sign/zero. If a supported LST's Chainlink feed goes stale, the last reported price is silently accepted and used to compute rsETH minting amounts, allowing an attacker to deposit at a favourable stale price and receive more rsETH than the deposited collateral is worth, diluting all existing holders.

## Finding Description
`ChainlinkPriceOracle.getAssetPrice()` at line 52 reads:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
``` [1](#0-0) 

All five return values of `latestRoundData()` — `roundId`, `answer`, `startedAt`, `updatedAt`, `answeredInRound` — are available via the interface declared at lines 14–17, but only `answer` is used. There is no check that `updatedAt != 0`, no `block.timestamp - updatedAt <= threshold` staleness guard, no `answeredInRound >= roundId` round-completeness check, and no `price > 0` sign check. [2](#0-1) 

This stale price flows directly into `LRTDepositPool.getRsETHAmountToMint()`:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [3](#0-2) 

`rsETHPrice` is a stored value updated by a separate privileged call to `updateRSETHPrice()` / `_updateRsETHPrice()`. [4](#0-3) [5](#0-4) 

This creates a timing gap: if a Chainlink feed goes stale at a price higher than the true market price (e.g., an LST depegs during L2 sequencer downtime), `getAssetPrice(asset)` returns the inflated stale price while `rsETHPrice` still reflects the correct historical value, causing the minting ratio to be wrong in the attacker's favour.

The stale price also flows into `_getTotalEthInProtocol()` via `LRTOracle._updateRsETHPrice()`, but the `PriceAboveDailyThreshold` guard only triggers when `updateRSETHPrice()` is called — it does not prevent minting abuse in the window between the feed going stale and the next price update. [6](#0-5) 

The `_beforeDeposit` function's `minRSETHAmountExpected` parameter is a depositor-controlled slippage guard that an attacker sets to 0 or a favourable value, providing no protection. [7](#0-6) 

## Impact Explanation
**Critical — Direct theft of user funds.**

An attacker who deposits a stale-priced LST receives more rsETH than the deposited collateral is worth at true market price. The excess rsETH represents a diluted claim on the protocol's TVL. When the attacker redeems or sells this rsETH, they extract real ETH value from the pool at the expense of honest existing holders. The attack is repeatable up to the per-asset deposit limit and requires no special privileges.

## Likelihood Explanation
Any unprivileged user can call `depositAsset()` or `depositETH()` during a staleness window. [8](#0-7) 

Realistic staleness triggers include: L2 sequencer downtime on Optimism, Arbitrum, or Base (where the protocol's `RSETHPoolV2`/`RSETHPoolV3` contracts are deployed), Chainlink feed deprecation (last price remains readable indefinitely), and network congestion delaying heartbeat updates. No oracle compromise or privileged access is required — the attacker only needs to observe that a feed has stopped updating and submit a deposit transaction. [9](#0-8) 

## Recommendation
Add staleness and validity checks in `ChainlinkPriceOracle.getAssetPrice()`:

```solidity
(uint80 roundId, int256 price, , uint256 updatedAt, uint80 answeredInRound)
    = priceFeed.latestRoundData();

require(price > 0, "Invalid price");
require(answeredInRound >= roundId, "Stale round");
require(updatedAt != 0, "Round not complete");
require(block.timestamp - updatedAt <= stalenessThreshold[asset], "Stale price");

return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

Store a per-feed configurable `stalenessThreshold` (heartbeat + buffer) alongside each feed address in `assetPriceFeed`. For L2 deployments, additionally integrate a Chainlink L2 Sequencer Uptime Feed check before consuming any price data. [10](#0-9) 

## Proof of Concept
1. Assume `cbETH/ETH` Chainlink feed last reported `1.05 ETH` per cbETH and then goes stale (e.g., L2 sequencer downtime). True market price drops to `0.95 ETH`.
2. `rsETHPrice` was last updated when cbETH was `1.05 ETH`, so `rsETHPrice ≈ 1.05e18`.
3. Attacker calls `depositAsset(cbETH, 100e18, 0, "")`.
4. `getRsETHAmountToMint` computes: `(100e18 * 1.05e18) / 1.05e18 = 100e18 rsETH`.
5. At true market price, 100 cbETH is worth only 95 ETH. At `rsETHPrice = 1.05`, the attacker should receive `95e18 / 1.05e18 ≈ 90.5 rsETH`. Instead they receive `100 rsETH` — approximately `9.5 rsETH` of excess (~10 ETH of value) extracted from existing holders.
6. Attacker repeats until the per-asset deposit limit is exhausted.

**Foundry fork test plan:**
- Fork mainnet/L2 at a block where a supported LST feed is live.
- Mock `latestRoundData()` to return a stale `updatedAt` (e.g., `block.timestamp - 2 hours`) with an inflated price.
- Call `depositAsset` as an unprivileged address with `minRSETHAmountExpected = 0`.
- Assert that `rsethAmountToMint` exceeds the fair value computed using the true price, confirming the dilution. [1](#0-0) [11](#0-10)
