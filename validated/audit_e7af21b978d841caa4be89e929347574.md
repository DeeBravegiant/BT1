### Title
Unprivileged User Can Drain Relayer's Deposited NEAR via Intentionally Failing Meta-Transaction Inner Receipt — (File: runtime/runtime/src/actions.rs)

---

### Summary

NEAR's meta-transaction (`DelegateAction`) implementation contains a structural imbalance: the relayer pays for any NEAR deposit attached to inner actions, but when the inner receipt fails on the receiver's shard, the deposit refund is sent to the **sender** (`Alice`), not to the relayer who actually funded it. An unprivileged user can exploit this by crafting a `SignedDelegateAction` with a large attached deposit targeting a call that is engineered to fail, causing the relayer to permanently lose the deposit while the user gains it.

---

### Finding Description

When `apply_delegate_action` processes a `DelegateAction`, it constructs a new inner receipt with `predecessor_id` set to `sender_id` (Alice):

```rust
let new_receipt = Receipt::V0(ReceiptV0 {
    predecessor_id: sender_id.clone(),   // ← Alice, not the relayer
    receiver_id: delegate_action.receiver_id().clone(),
    ...
});
``` [1](#0-0) 

The code comment immediately below this block acknowledges the asymmetry:

> "Relayer prepaid all fees and all things required by actions: attached deposits and attached gas. If something goes wrong, deposit is refunded to the predecessor, this is sender_id/Sender in DelegateAction. Gas is refunded to the signer, this is Relayer." [2](#0-1) 

The deposit refund logic in `refund_unspent_gas_and_deposits` computes `deposit_refund = total_deposit` when `result.result.is_err()`, and the refund receipt is addressed to `receipt.predecessor_id()` — which is Alice, not the relayer: [3](#0-2) 

The protocol specification confirms this: deposit refunds go to `predecessor_id` because "only the predecessor can attach deposits." [4](#0-3) 

The architecture documentation explicitly names the financial incentive this creates:

> "there is a financial incentive for Alice to submit meta transactions that have high balances attached but will fail on Bob's shard." [5](#0-4) 

---

### Impact Explanation

**Corrupted protocol value**: Alice's on-chain NEAR balance is increased by the full deposit amount; the relayer's balance is decreased by the same amount. This is a direct, irreversible balance transfer from the relayer to Alice that the relayer did not authorize.

**Concrete scenario**:
1. Alice deploys a contract with a method that always panics (or controls a contract whose state she can manipulate).
2. Alice signs a `DelegateAction` targeting that method with a large attached deposit (e.g., 1000 NEAR).
3. The relayer inspects the action, sees a plausible call, and submits the wrapping transaction — paying the 1000 NEAR deposit.
4. The inner receipt executes on the receiver's shard and fails (panic / missing method / Alice front-ran a state change).
5. `refund_unspent_gas_and_deposits` emits a `new_balance_refund` to `predecessor_id` = Alice.
6. Alice receives 1000 NEAR; the relayer is out 1000 NEAR.

The relayer cannot prevent this atomically: it must submit the transaction before knowing the execution outcome, and Alice can race or pre-arrange the failure.

---

### Likelihood Explanation

Any account can create a `SignedDelegateAction` and submit it to a public relayer. The attack requires no special privileges — only the ability to sign a `DelegateAction` (any NEAR account) and access to a relayer that will forward it. Relayers that serve open or semi-open user bases (e.g., application-specific relayers that check FT balances off-chain) are directly exposed to the front-running variant of this attack. The protocol provides no on-chain mechanism for the relayer to recover the deposit.

---

### Recommendation

1. **Protocol-level fix**: When processing a `DelegateAction`, set the inner receipt's `predecessor_id` to the relayer (the outer transaction signer) rather than Alice, so that deposit refunds flow back to the party that funded them.
2. **Alternatively**: Introduce a `refund_to` override field in `DelegateAction` that the relayer can set to itself, directing deposit refunds appropriately.
3. **Short-term mitigation for relayer operators**: Never attach NEAR deposits to meta-transaction inner actions unless the relayer can guarantee execution success (e.g., via simulation), or require Alice to pre-fund the deposit separately.

---

### Proof of Concept

**Entry path** (unprivileged, via public RPC):

```
Alice signs DelegateAction {
    sender_id: alice,
    receiver_id: alice_contract,   // contract Alice controls
    actions: [FunctionCall {
        method_name: "always_panic",
        deposit: 1_000_000_000_000_000_000_000_000,  // 1000 NEAR
        gas: 30 TGas,
    }],
    nonce: ...,
    max_block_height: ...,
}
```

Relayer wraps it in a `SignedTransaction` (paying the 1000 NEAR deposit) and submits via RPC.

**On-chain execution**:
- `apply_delegate_action` creates inner receipt with `predecessor_id = alice`.
- `always_panic` fails on Alice's contract shard.
- `refund_unspent_gas_and_deposits` emits `Receipt::new_balance_refund(&alice, 1000_NEAR)`.
- Alice's balance increases by 1000 NEAR; relayer's balance decreases by 1000 NEAR.

**Corrupted DB entry**: `Account { account_id: alice, amount: +1000 NEAR }` — a balance that Alice did not earn and the relayer did not intend to transfer. [6](#0-5) [7](#0-6) [8](#0-7)

```markdown
## Deposit Refunds

Deposit refunds are generated when an action receipt fails to execute. All attached deposit amounts are summed together and
sent as a refund to a `predecessor_id` (because only the predecessor can attach deposits).

Deposit refunds have the following fields in the `ActionReceipt`:

- `signer_id` is `system`
- `signer_public_key` is ED25519 key with data equal to 32 bytes of `0`.

Deposit refunds are free for the user and incur no refund fee.
```
