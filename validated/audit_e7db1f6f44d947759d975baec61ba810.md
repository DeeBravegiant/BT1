### Title
Out-of-Bounds Index Access via Attacker-Controlled `sub_epoch_n` in Weight Proof Segment Validation — (`File: chia/full_node/weight_proof.py`)

### Summary
In `_validate_sub_epoch_segments`, the field `sub_epoch_n` is read directly from attacker-supplied `SubEpochChallengeSegment` data and used to index into the `summaries` list without a bounds check. An adversary can craft a weight proof containing a segment whose `sub_epoch_n` exceeds `len(summaries) - 1`, triggering an unhandled `IndexError` that crashes the syncing node's weight-proof validation path.

### Finding Description

`_validate_sub_epoch_segments` builds `segments_by_sub_epoch` from the weight proof's `challenge_segments` field via `map_segments_by_sub_epoch`. The resulting dictionary keys are the raw `sub_epoch_n` values embedded in each `SubEpochChallengeSegment` — fully attacker-controlled. [1](#0-0) 

The loop then uses those keys to index directly into `summaries` (a list whose length equals `len(weight_proof.sub_epochs)`): [2](#0-1) 

```python
prev_ses = summaries[sub_epoch_n - 1]          # line 983
...
if not summaries[sub_epoch_n].reward_chain_hash == rc_sub_slot_hash:   # line 985
```

`_get_curr_diff_ssi` at line 974 also receives `sub_epoch_n` and internally indexes `summaries`: [3](#0-2) 

No bounds check exists anywhere between reading `sub_epoch_n` from the wire and using it as a list index.

The upstream guard `validate_sub_epoch_sampling` only verifies that every *required* (sampled) sub-epoch has at least one segment present. It does **not** reject extra segments whose `sub_epoch_n` is out of range: [4](#0-3) 

An attacker can therefore include all required segments (so sampling validation passes) **plus** one extra segment with `sub_epoch_n >= len(summaries)`. The sampling check returns `True`, and `_validate_sub_epoch_segments` then raises `IndexError`.

### Impact Explanation

`_validate_sub_epoch_segments` is called from both `validate_weight_proof_single_proc` and `validate_weight_proof_inner`: [5](#0-4) [6](#0-5) 

Neither call site wraps the call in a `try/except`. An `IndexError` propagates as an unhandled exception through the weight-proof validation coroutine, aborting the sync handshake. A node that cannot complete weight-proof validation cannot advance its peak or sync to the chain tip. Repeated delivery of such proofs by any reachable peer constitutes a **High** impact: long-lived inability for honest nodes to sync.

### Likelihood Explanation

Weight proofs are exchanged over the peer-to-peer protocol. Any peer (no keys, no stake, no privilege required) can send a `RespondProofOfWeight` message. The crafted proof needs only to pass `_validate_sub_epoch_summaries` (which validates `sub_epochs` data, not `sub_epoch_segments`) and `validate_sub_epoch_sampling` (which does not bound-check `sub_epoch_n`). Both checks are bypassable as described above.

### Recommendation

Add an explicit bounds check at the top of the `for sub_epoch_n, segments in segments_by_sub_epoch.items()` loop in `_validate_sub_epoch_segments`:

```python
if sub_epoch_n >= len(summaries):
    log.error(f"sub_epoch_n {sub_epoch_n} out of range for summaries (len={len(summaries)})")
    return None
```

Apply the same guard inside `_get_curr_diff_ssi` before it indexes `summaries`.

### Proof of Concept

1. Obtain or generate a legitimate weight proof `wp` for a chain with `N` sub-epoch summaries (`len(wp.sub_epochs) == N`).
2. Append one extra `SubEpochChallengeSegment` to `wp.sub_epoch_segments` with `sub_epoch_n = N` (one past the end).
3. Send this crafted `RespondProofOfWeight` to a syncing full node or light wallet.
4. The node calls `validate_weight_proof_inner` → `validate_sub_epoch_sampling` returns `True` (the extra segment's `sub_epoch_n` is simply not in `sampled_sub_epochs`, so no deletion occurs and the dict empties from the legitimate segments) → `_validate_sub_epoch_segments` iterates to `sub_epoch_n = N` → `summaries[N]` raises `IndexError` → sync aborts. [7](#0-6)
