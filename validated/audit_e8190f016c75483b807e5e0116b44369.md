### Title
Single Byzantine Participant Can Forge Foreign-Chain Verification Responses via Cross-Request Signature Replay — (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_verify_foreign_tx` verifies that the submitted signature is valid over the submitted `payload_hash`, but never verifies that `payload_hash` was derived from the actual `request` data. A single Byzantine attested participant who holds a valid threshold signature from any prior foreign-tx signing can replay it against any other pending `VerifyForeignTransactionRequest`, causing the contract to resolve that request with a fabricated attestation.

---

### Finding Description

In `respond_verify_foreign_tx` the contract performs two independent operations that are never bound together:

**Operation 1 — signature check** (lines 718–743):
```rust
let payload_hash: [u8; 32] = response.payload_hash.0;
// Check the signature is correct against the root public key
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,
).is_ok()
``` [1](#0-0) 

**Operation 2 — request resolution** (lines 749–753):
```rust
pending_requests::resolve_yields_for(
    &mut self.pending_verify_foreign_tx_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
``` [2](#0-1) 

The contract never checks that `response.payload_hash` was computed from `request.request`. The `payload_hash` is defined by the node side as `ForeignTxSignPayload::V1 { request, values }.compute_msg_hash()`: [3](#0-2) 

Because the contract accepts any `payload_hash` for which a valid root-key signature exists, a Byzantine node can supply `payload_hash = H1` (from a previously completed signing for request R1) while naming `request = R2` (a different pending request). The signature check passes (S1 is valid over H1), and R2 is resolved with R1's attestation data.

Contrast with `respond` for regular signing, which correctly derives the expected public key from `request.tweak` before verifying: [4](#0-3) 

The `VerifyForeignTransactionRequest` struct has no `tweak` field and no caller-binding, so there is no equivalent binding mechanism: [5](#0-4) 

---

### Impact Explanation

A single Byzantine attested participant (one node, strictly below the signing threshold) can execute the following:

1. Participate honestly in the MPC signing protocol for request **R1** (e.g., Bitcoin tx B1), receiving the completed threshold signature **S1** over **H1 = hash(R1, extracted\_values\_B1)**.
2. Observe any other pending `VerifyForeignTransactionRequest` **R2** (e.g., Bitcoin tx B2) from on-chain state — this is public.
3. Call `respond_verify_foreign_tx(R2, { payload_hash: H1, signature: S1 })`.
4. The contract verifies S1 over H1 against the root key — **passes**.
5. R2 is resolved and its yield-resume fires, delivering `{ payload_hash: H1, signature: S1 }` to the user who submitted R2.
6. The user's contract verifies S1 over H1 — **passes** — but H1 encodes B1's extracted values, not B2's.

The user receives a cryptographically valid but semantically false attestation. Any bridge or NEAR contract that gates fund release on a `verify_foreign_transaction` response (the primary stated use case) can be tricked into releasing funds based on a different transaction's data. Additionally, the legitimate response for R2 is permanently blocked: `resolve_yields_for` drains the queue, so the correct response will return `RequestNotFound`.

This matches the allowed impact: **"Cross-chain replay, forged foreign-chain verification … that causes invalid bridge execution or double-spend conditions."**

---

### Likelihood Explanation

**Medium.** The attacker must be an attested participant — a legitimate node that has passed TEE attestation. However, only a single such node is required; no threshold collusion is needed. The attacker needs only one completed signing (to obtain a valid signature) and knowledge of any pending request (observable from chain state). The window is open for as long as R2 remains in the pending queue (up to the yield-resume timeout). In a live bridge deployment with continuous traffic, both preconditions are routinely satisfied.

---

### Recommendation

Bind `payload_hash` to `request` inside `respond_verify_foreign_tx`. The simplest fix is to compute the expected `payload_hash` prefix from `request.request` on-chain and assert that `response.payload_hash` matches, or to require the node to include the full `ForeignTxSignPayload` (with extracted values) in the response so the contract can recompute and compare the hash. At minimum, the signed message should commit to the `VerifyForeignTransactionRequest` identity so that a signature produced for R1 is cryptographically invalid for R2.

---

### Proof of Concept

```
Setup:
  - Contract in Running state, ForeignTx domain active.
  - Attacker is attested participant P_evil.
  - R1 = VerifyForeignTransactionRequest { request: Bitcoin(tx_id=B1), ... }
  - R2 = VerifyForeignTransactionRequest { request: Bitcoin(tx_id=B2), ... }

Step 1: User_A calls verify_foreign_transaction(R1). Pending queue: {R1: [yield_A]}.
Step 2: User_B calls verify_foreign_transaction(R2). Pending queue: {R1: [yield_A], R2: [yield_B]}.
Step 3: MPC protocol runs for R1. P_evil receives S1 = ECDSA_sign(root_sk, H1)
        where H1 = ForeignTxSignPayload::V1{request:R1, values:V1}.compute_msg_hash().
Step 4: P_evil calls respond_verify_foreign_tx(R2, { payload_hash: H1, signature: S1 }).
        Contract: verify_ecdsa_signature(S1, H1, root_pk) → OK.
        Contract: resolve_yields_for(R2, serialize({payload_hash:H1, sig:S1})) → drains yield_B.
Step 5: User_B's contract receives { payload_hash: H1, signature: S1 }.
        User_B verifies S1 over H1 against root_pk → PASS.
        User_B's bridge logic interprets H1 as attestation of B2 → incorrect funds released.
Step 6: Legitimate respond_verify_foreign_tx(R2, correct_response) → RequestNotFound.
```

fn build_signature_request(
    request: &VerifyForeignTxRequest,
    foreign_tx_payload: &dtos::ForeignTxSignPayload,
) -> anyhow::Result<SignatureRequest> {
    let payload_hash: [u8; ECDSA_PAYLOAD_SIZE_BYTES] =
        foreign_tx_payload.compute_msg_hash()?.into();
    let payload_bytes: BoundedVec<u8, ECDSA_PAYLOAD_SIZE_BYTES, ECDSA_PAYLOAD_SIZE_BYTES> =
        payload_hash.into();

    Ok(SignatureRequest {
        id: request.id,
        receipt_id: request.receipt_id,
        payload: Payload::Ecdsa(payload_bytes),
        tweak: Tweak::new([0u8; 32]),
        entropy: request.entropy,
        timestamp_nanosec: request.timestamp_nanosec,
        domain: request.domain_id,
    })
