### Title
`cancel_votes` Not Reset Between Domain Key-Generation Rounds Allows Sub-Threshold Cancellation - (File: `crates/contract/src/state/initializing.rs`)

### Summary

`InitializingContractState.cancel_votes` accumulates votes across successive domain key-generation rounds within the same initializing session. When one domain's key generation succeeds and the contract advances to the next domain, the cancel-vote set is never cleared. A sub-threshold coalition that voted to cancel during an earlier domain can have those votes silently carry forward, so a single additional participant vote in a later domain crosses the cancellation threshold — aborting key generation that a full threshold never agreed to cancel.

### Finding Description

When `vote_add_domains` succeeds in `RunningContractState`, the contract transitions to `InitializingContractState` with `cancel_votes: BTreeSet::new()`. [1](#0-0) 

As each domain's key generation completes (via `vote_pk`), the contract replaces `self.generating_key` with a fresh `KeyEvent` for the next domain, but **never touches `self.cancel_votes`**: [2](#0-1) 

The `vote_cancel` function inserts the authenticated caller into `cancel_votes` and fires if `len() >= required_threshold`. The only replay guard is a `next_domain_id` check, which is the registry-level next ID — constant throughout the entire initializing session, not per-domain: [3](#0-2) 

Because `cancel_votes` is a `BTreeSet` keyed on `AuthenticatedParticipantId` (no domain or attempt binding), votes cast during domain 0's generation silently persist into domain 1's generation. [4](#0-3) 

### Impact Explanation

Consider 5 participants with governance threshold = 3 and two new domains being added:

1. Domain 0 key generation begins.
2. Participants A and B call `vote_cancel` (2 votes; threshold = 3, so cancellation does not fire).
3. Participants C, D, E call `vote_pk` with the correct public key — domain 0 succeeds; the contract advances to domain 1.
4. `cancel_votes` still contains `{A, B}`.
5. Participant C calls `vote_cancel` for domain 1 (1 new vote).
6. `cancel_votes` is now `{A, B, C}` — length 3 = threshold — cancellation fires.

Domain 1's key generation is aborted even though only **one** participant (C) expressed intent to cancel it. The other two votes (A, B) were cast against domain 0 and should have been invalidated when domain 0 succeeded. The MPC network loses the ability to sign with the domain 1 key without any further governance action.

This maps to the allowed Medium impact: *"participant-state or contract execution-flow manipulation that breaks production safety/accounting invariants."*

### Likelihood Explanation

The preconditions are realistic in production:

- Multi-domain `vote_add_domains` calls are the normal path for adding new signing schemes (e.g., adding FROST + CKD domains together).
- Participants routinely vote to cancel when they detect a problem during key generation (e.g., a node is offline). If domain 0 recovers and succeeds before the cancel threshold is reached, those votes are not cleaned up.
- No privileged access is required; any current participant can call `vote_cancel`.

### Recommendation

Reset `cancel_votes` to `BTreeSet::new()` whenever the contract advances to the next domain inside `vote_pk`:

```rust
// In vote_pk, after pushing to generated_keys:
if let Some(next_domain) = self.domains.get_domain_by_index(self.generated_keys.len()) {
    self.generating_key = KeyEvent::new(...);
    self.cancel_votes = BTreeSet::new();   // ← add this
}
```

This mirrors the pattern already used when `InitializingContractState` is first constructed. [1](#0-0) 

Add a multi-domain test that verifies `cancel_votes` is empty after each domain transition.

### Proof of Concept

```
Setup: 5 participants (A–E), governance threshold = 3, 2 new domains.

1. All 5 call vote_add_domains([domain0, domain1]) → Initializing state,
   cancel_votes = {}.

2. Leader starts domain 0 keygen instance.

3. A calls vote_cancel(next_domain_id=2) → cancel_votes = {A}, len=1 < 3, no cancel.
4. B calls vote_cancel(next_domain_id=2) → cancel_votes = {A,B}, len=2 < 3, no cancel.

5. C, D, E call vote_pk(domain0, pk) → threshold reached, domain 0 key stored.
   generating_key advances to domain 1.
   cancel_votes is still {A, B}.   ← BUG

6. Leader starts domain 1 keygen instance.

7. C calls vote_cancel(next_domain_id=2) → cancel_votes = {A,B,C}, len=3 >= 3.
   → Cancellation fires. Domain 1 key generation is aborted.
   → Contract reverts to Running with only domain 0's key.

Result: domain 1 was cancelled with only 1 participant (C) expressing intent
to cancel it during its own generation round.
``` [3](#0-2) [2](#0-1)

    pub fn vote_cancel(
        &mut self,
        next_domain_id: u64,
    ) -> Result<Option<RunningContractState>, Error> {
        if next_domain_id != self.domains.next_domain_id() {
            return Err(InvalidParameters::NextDomainIdMismatch.into());
        }
        let participant = AuthenticatedParticipantId::new(
            self.generating_key.proposed_parameters().participants(),
        )?;
        let required_threshold = self
            .generating_key
            .proposed_parameters()
            .threshold()
            .value() as usize;
        if self.cancel_votes.insert(participant) && self.cancel_votes.len() >= required_threshold {
            let mut domains = self.domains.clone();
            domains.retain_domains(self.generated_keys.len());
            return Ok(Some(RunningContractState::new(
                domains,
                Keyset::new(self.epoch_id, self.generated_keys.clone()),
                self.generating_key.proposed_parameters().clone(),
                AddDomainsVotes::default(),
            )));
        }
        Ok(None)
    }
