### Title
`combine_ac` Accepts Crafted Blob Where `batch_count` Disagrees With Inner `Q.size()`, Returning `SUCCESS` With Truncated `out_xs` — (`src/cbmpc/api/pve_batch_ac.cpp`)

---

### Summary

`combine_ac` (and its C wrapper `cbmpc_pve_ac_combine`) never validates that the outer blob's `batch_count` field equals the inner `ec_pve_ac_t`'s `Q.size()`. An attacker who supplies the ciphertext blob can set `batch_count=N` while serializing an inner `ec_pve_ac_t` with only `M < N` points in `Q`. The function returns `SUCCESS` and an `out_xs` vector of size `M`, while `cbmpc_pve_ac_get_count` reports `N`. Any caller that trusts `batch_count` to index into `out_xs` encounters out-of-bounds access or silently accepts a truncated decryption as the full batch.

---

### Finding Description

**Blob structure.** The outer wrapper stores `batch_count` as an independent `uint32_t` field, separate from the inner serialized `ec_pve_ac_t`: [1](#0-0) 

**`parse_ac_ciphertext` only range-checks `batch_count`.** It validates `batch_count != 0` and `batch_count <= MAX_BATCH_COUNT`, but never compares it against the `Q` vector that will be deserialized from `blob.ct`: [2](#0-1) 

**`combine_ac` never reconciles the two counts.** After deserializing `pve_ct` from `blob.ct`, the function proceeds directly to `aggregate_to_restore_row` with `skip_verify=true`. There is no guard of the form `blob.batch_count == pve_ct.get_Q().size()`: [3](#0-2) 

**`aggregate_to_restore_row` uses `Q.size()` as the authoritative batch size.** With `skip_verify=true` the full ZK proof is bypassed. `batch_size` is derived exclusively from the attacker-controlled `Q` vector: [4](#0-3) 

The only per-element guard is `x[j] * G != Q[j]`, which verifies recovered scalars against the attacker-supplied `Q` points — it does not enforce that the count matches `batch_count`: [5](#0-4) 

**`out_xs` is sized from `xs_bn.size()` (= `Q.size()`), not from `blob.batch_count`.** The function returns `SUCCESS` with a vector whose length is determined by the inner `Q`, not the outer `batch_count`: [6](#0-5) 

---

### Impact Explanation

A caller following the documented two-step pattern — call `cbmpc_pve_ac_get_count` to learn `N`, then call `cbmpc_pve_ac_combine` and iterate over `N` outputs — will receive `SUCCESS` from `combine_ac` but only `M` elements in `out_xs`. Accessing indices `M..N-1` is undefined behavior (out-of-bounds on the returned buffer). Alternatively, a caller that uses `out_xs.count` (C API `cmems_t.count`) to determine how many secrets were recovered will silently accept `M` secrets as the full batch of `N`, substituting attacker-chosen values for the missing positions. This constitutes accepted invalid cryptographic output from a public PVE API path.

---

### Likelihood Explanation

The ciphertext blob is an opaque byte buffer passed across an API or protocol boundary. Any peer or caller that supplies the ciphertext can craft the mismatch with no special privilege. The path is reachable via both the C++ API (`combine_ac`) and the C API (`cbmpc_pve_ac_combine`). The `skip_verify=true` flag in `combine_ac` is hardcoded, so the ZK proof that would otherwise bind `Q` to the encrypted content is never checked on this path.

---

### Recommendation

Add a consistency check in `combine_ac` immediately after deserializing `pve_ct`, before calling `aggregate_to_restore_row`:

```cpp
if (pve_ct.get_Q().size() != static_cast<size_t>(blob.batch_count))
    return coinbase::error(E_FORMAT, "batch_count mismatch");
```

Apply the same guard in `partial_decrypt_ac_attempt` and `verify_ac` for defense-in-depth, since those functions also parse the blob and use `Q.size()` independently of `batch_count`.

### Proof of Concept

1. Generate a legitimate 1-element PVE-AC ciphertext for a known scalar `x0` under the target access structure. Serialize the inner `ec_pve_ac_t` to bytes `ct_inner`.
2. Craft a raw `pve_ac_ciphertext_blob_v1_t` with `version=1`, `batch_count=8`, `ct=ct_inner`.
3. Serialize this blob and pass it to `cbmpc_pve_ac_get_count` → returns `8`.
4. Collect a valid quorum of partial decryptions (using the legitimate 1-element ciphertext).
5. Call `cbmpc_pve_ac_combine` with the crafted blob → returns `CBMPC_SUCCESS` and `out_xs.count=1`.
6. Assert `out_xs.count != batch_count` (1 ≠ 8): the API returned success with a truncated batch.
7. Accessing `out_xs.data[out_xs.sizes[0]..out_xs.sizes[0]+31]` (i.e., index 1) is out-of-bounds UB.

static error_t parse_ac_ciphertext(mem_t ciphertext, pve_ac_ciphertext_blob_v1_t& out_blob) {
  error_t rv = coinbase::convert(out_blob, ciphertext);
  if (rv) return rv;
  if (out_blob.version != pve_ac_ciphertext_version_v1)
    return coinbase::error(E_FORMAT, "unsupported ciphertext version");
  if (out_blob.batch_count == 0) return coinbase::error(E_FORMAT, "invalid batch count");
  if (out_blob.batch_count > static_cast<uint32_t>(MAX_BATCH_COUNT)) return coinbase::error(E_RANGE, "batch too large");
  return SUCCESS;
}
