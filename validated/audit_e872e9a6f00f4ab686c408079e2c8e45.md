### Title
Unverified CKD Response for `AppPublicKey` Variant Allows Single Byzantine Participant to Forge Derived Key Output - (File: `crates/contract/src/lib.rs`)

### Summary

`respond_ckd` enforces that the caller is an attested participant but performs **no cryptographic verification** of the submitted `CKDResponse` when the request uses the `AppPublicKey` variant. A single Byzantine attested participant (below the signing threshold) can race to call `respond_ckd` with an arbitrary forged response, delivering an attacker-controlled derived key to the user.

### Finding Description

In `respond_ckd`, the contract validates the response only for the `AppPublicKeyPV` variant via `ckd_output_check`. The `AppPublicKey` branch is an empty no-op:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← no verification
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

After this match, `resolve_yields_for` is called unconditionally, resuming every queued yield with whatever bytes the caller supplied: [2](#0-1) 

The only gate is `assert_caller_is_attested_participant_and_protocol_active()`, which confirms the caller is *some* attested participant — analogous to the zNS `authorizedOwner(parentId)` check that confirmed the caller owned *some* parent domain without verifying authority over the specific resource being acted upon. [3](#0-2) 

The pending CKD request map is keyed by the full `CKDRequest` struct (app_public_key, domain_id, predecessor, derivation_path), all of which are observable on-chain once the user calls `request_app_private_key`. [4](#0-3) 

`resolve_yields_for` drains the entire queue on the first successful call, so the legitimate leader's subsequent call returns `RequestNotFound` and is silently discarded: [5](#0-4) 

### Impact Explanation

A Byzantine attested participant (a single node, strictly below the signing threshold) can:

1. Observe a pending `AppPublicKey` CKD request on-chain.
2. Call `respond_ckd` with the correct `CKDRequest` key but a `CKDResponse` containing a derived key the attacker controls.
3. The contract accepts the forged response with no cryptographic check and resumes the user's yield.
4. The user receives an attacker-known derived key and uses it to control assets on a foreign chain.
5. The attacker drains those assets.

This is **Critical**: unauthorized confidential key derivation output delivered without the required threshold participant authorization, enabling direct theft of funds.

### Likelihood Explanation

Any single attested participant in the MPC network can execute this attack. Attested participants are a small, known set, but the attacker model explicitly includes a Byzantine participant below the signing threshold. The attack requires only observing a pending request (public on-chain state) and submitting a transaction before the legitimate leader — a straightforward race condition with no additional prerequisites.

### Recommendation

Apply the same cryptographic output verification to `AppPublicKey` requests that is already applied to `AppPublicKeyPV` requests. If on-chain verification of the `AppPublicKey` variant is not feasible with the current cryptographic primitives, require a threshold-weighted vote among participants before `resolve_yields_for` is called, so no single participant can unilaterally determine the CKD output.

### Proof of Concept

1. User calls `request_app_private_key` with `CKDRequestArgs { app_public_key: AppPublicKey(pk), domain_id, derivation_path }`.
2. Contract stores the request in `pending_ckd_requests` and suspends the user's call via yield-resume.
3. Attacker (a single attested participant) reads the `CKDRequest` key from chain state.
4. Attacker calls `respond_ckd(request, forged_response)` where `forged_response` contains a derived key the attacker generated locally.
5. `assert_caller_is_attested_participant_and_protocol_active()` passes (attacker is a valid participant).
6. The `AppPublicKey` match arm executes the empty body — no check.
7. `resolve_yields_for` resumes the user's yield with `serde_json::to_vec(&forged_response)`.
8. The legitimate leader's `respond_ckd` call returns `RequestNotFound` because the queue was already drained.
9. User's `request_app_private_key` call returns the attacker-controlled derived key.

```rust
pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
