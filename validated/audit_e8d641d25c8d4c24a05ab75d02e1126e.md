Audit Report

## Title
ETH Deposit Limit Check Omits Deposit Amount, Allowing Any Single Deposit to Bypass the Cap ŌĆö (File: `contracts/LRTDepositPool.sol`)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` applies an asymmetric comparison for ETH versus ERC20 tokens: the ETH branch evaluates `totalAssetDeposits > limit` (omitting `amount`), while the ERC20 branch correctly evaluates `totalAssetDeposits + amount > limit`. As a result, any single ETH deposit can push total protocol ETH deposits arbitrarily above the configured `depositLimitByAsset` cap without triggering a revert.

## Finding Description
In `contracts/LRTDepositPool.sol` at lines 676ŌĆō682:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));   // ŌåÉ amount absent
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // ŌåÉ correct
}
```

The ETH branch at line 679 returns `false` whenever `totalAssetDeposits <= limit`, regardless of `amount`. This means a depositor can send an arbitrarily large `msg.value` and the guard will not revert as long as the running total has not already exceeded the limit.

The call chain is:
- `depositETH` (line 87) ŌåÆ `_beforeDeposit(LRTConstants.ETH_TOKEN, msg.value, ...)` (line 87)
- `_beforeDeposit` (lines 661ŌĆō663) ŌåÆ `_checkIfDepositAmountExceedesCurrentLimit(asset, depositAmount)`

The guard at lines 661ŌĆō663 only reverts on `true`, so the omission of `amount` in the ETH branch means the check is effectively bypassed for any deposit where the pre-deposit total is at or below the limit.

Additionally, `getAssetCurrentLimit` (lines 402ŌĆō409) returns `0` when `totalAssetDeposits > limit` (strictly greater), while the deposit guard also uses strictly greater. This creates a one-deposit window where the view function reports zero remaining capacity but the guard still permits a deposit that would push the total exactly to the limit ŌĆö a secondary inconsistency stemming from the same root cause.

## Impact Explanation
**Low ŌĆö Contract fails to deliver promised returns, but doesn't lose value.**

The `depositLimitByAsset` cap is the protocol's primary mechanism for controlling ETH intake during phased rollouts or risk-management windows. Because the ETH branch ignores `amount`, a single depositor can push `getTotalAssetDeposits(ETH)` to any value above the cap in one transaction, violating the invariant `getTotalAssetDeposits(ETH) Ōēż depositLimitByAsset(ETH)`. No funds are stolen and no user loses principal, but the protocol accepts more ETH than its configured risk limit permits.

## Likelihood Explanation
**High.** The entry path is the standard, permissionless `depositETH()` function. Any depositor who observes that `totalAssetDeposits` is at or near the limit can send a large `msg.value` in a single call. No special role, flash loan, or multi-step setup is required. The condition is trivially observable on-chain via `getTotalAssetDeposits` and `getAssetCurrentLimit`.

## Recommendation
Add `amount` to the ETH branch to match the ERC20 branch:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

## Proof of Concept
1. Admin sets `depositLimitByAsset(ETH_TOKEN) = 1000 ether`.
2. Current `getTotalAssetDeposits(ETH_TOKEN) = 999 ether`.
3. Depositor calls `depositETH{value: 500 ether}(0, "")`.
4. `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 500 ether)`:
   - ETH branch: `999 ether > 1000 ether` ŌåÆ `false` ŌåÆ guard does **not** revert.
5. Deposit succeeds; `getTotalAssetDeposits(ETH_TOKEN)` becomes `1499 ether`, 49.9% above the cap.
6. Identical call with an ERC20 token at the same state: `999 ether + 500 ether > 1000 ether` ŌåÆ `true` ŌåÆ `MaximumDepositLimitReached` revert.
