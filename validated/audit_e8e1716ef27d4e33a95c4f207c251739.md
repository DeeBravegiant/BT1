### Title
Wrong SID-Generation Function in ECDSA-2PC HD-Keyset Derivation Weakens ZK-DH Session Binding - (File: src/cbmpc/protocol/hd_keyset_ecdsa_2p.cpp)

### Summary
`key_share_ecdsa_hdmpc_2p_t::derive_keys` calls `generate_sid_fixed_mp` (the multi-party variant) instead of `generate_sid_fixed_2p` (the two-party variant) when generating the session identifier used in the ZK-DH proofs that authenticate each party's VRF share. The EdDSA counterpart in `hd_keyset_eddsa_2p.cpp` correctly calls `generate_sid_fixed_2p(job, party_t::p2, sid)`. Both files carry the comment "The beginning of this function is the same as its counterpart," making the divergence clearly unintentional.

### Finding Description
In `hd_keyset_ecdsa_2p.cpp`, `derive_keys` generates the session ID as follows:

```cpp
// hd_keyset_ecdsa_2p.cpp  line 101-103
if (sid.empty()) {
    if (rv = generate_sid_fixed_mp(job, sid)) return rv;   // ← wrong: MP variant
}
```

The EdDSA twin does:

```cpp
// hd_keyset_eddsa_2p.cpp  line 97-99
if (sid.empty()) {
    if (rv = generate_sid_fixed_2p(job, party_t::p2, sid)) return rv;  // ← correct: 2P variant, p2 is the fixed party
}
```

`generate_sid_fixed_2p(job, party_t::p2, sid)` designates p2 as the "fixed" party: p2 generates the SID unilaterally and transmits it to p1, so the SID is cryptographically bound to p2's session state. `generate_sid_fixed_mp` uses a different multi-party SID-generation algorithm that does not enforce this asymmetric binding.

The SID produced by whichever function is called is then passed directly into the ZK-DH proofs that authenticate each party's VRF share:

```cpp
// hd_keyset_ecdsa_2p.cpp  lines 127, 130, 135, 141
zk_dh1.prove(P, K_share, Z1, k_share, sid, 1);
...
if (rv = zk_dh1.verify(P, other_K_share, Z1, sid, 1)) return rv;
zk_dh2.prove(P, K_share, Z2, k_share, sid, 2);
...
if (rv = zk_dh2.verify(P, other_K_share, Z2, sid, 2)) return rv;
```

The VRF output `Z = Z1 + Z2` is then used to derive `delta` and `chain_code`, which determine all derived ECDSA-2PC key shares:

```cpp
// hd_keyset_ecdsa_2p.cpp  lines 143-169
ecc_point_t Z = Z1 + Z2;
buf_t y = crypto::ro::hash_string(Z).bitlen(...);
bn_t delta = bn_t::from_bin(y.take(delta_size)) % q;
...
derived_keys[i].Q = Q_derived + non_hard_delta_G;
if (job.get_party() == party_t::p2)
    MODULO(q) derived_keys[i].x_share = x_share + delta + non_hard_delta[i];
```

### Impact Explanation
`generate_sid_fixed_2p` with `party_t::p2` enforces that p2 is the sole contributor to the SID, preventing p1 from influencing it. Substituting `generate_sid_fixed_mp` changes the SID-generation algorithm to one that may incorporate contributions from all parties, including p1. A malicious p1 that can steer the SID can pre-compute a ZK-DH proof `(e, z)` satisfying `z·G − e·A = X` and `z·Q − e·B = Y` for a chosen `B = k*P` (i.e., a chosen VRF share `Z1`). If the proof verifies under the attacker-controlled SID, p2 accepts a forged `Z1`, causing `Z = Z1_forged + Z2` to be wrong. Because `delta` and all derived key shares are deterministic functions of `Z`, the attacker controls the derived ECDSA-2PC key material for every child path under the hardened derivation.

### Likelihood Explanation
The function is reachable through the public API: `key_share_ecdsa_hdmpc_2p_t::derive_keys` is the production HD-keyset derivation entry point for ECDSA-2PC. A malicious p1 (one of the two protocol participants) triggers the path by initiating a normal `derive_keys` call. No threshold collusion is required; a single corrupt party suffices.

### Recommendation
Replace `generate_sid_fixed_mp(job, sid)` with `generate_sid_fixed_2p(job, party_t::p2, sid)` in `key_share_ecdsa_hdmpc_2p_t::derive_keys`, matching the EdDSA counterpart. Add a cross-check test that verifies the two `derive_keys` implementations use identical SID-generation logic.

### Proof of Concept
1. Honest p2 calls `derive_keys`; both parties reach the SID-generation branch.
2. Because `generate_sid_fixed_mp` is used, p1 can influence the SID value (e.g., by manipulating its contribution to the multi-party SID protocol).
3. Malicious p1 pre-computes a ZK-DH proof for a chosen `Z1*` under the steered SID.
4. p1 sends `Z1*` and the forged proof; p2 calls `zk_dh1.verify(P, other_K_share, Z1*, sid, 1)` — the proof verifies because the SID matches the pre-computation.
5. p2 computes `Z = Z1* + Z2`, derives `delta` from `hash(Z)`, and outputs `x_share_p2 + delta + non_hard_delta[i]` as the derived key share.
6. p1 knows `Z1*` and `Z2` (received from p2), so it knows `Z` and therefore `delta`, giving it full knowledge of the derived key. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
