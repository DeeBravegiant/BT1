### Title
Unverified CKD Response for `AppPublicKey` Variant Allows Single Byzantine Participant to Forge Confidential Key Derivation Output — (File: `crates/contract/src/lib.rs`)

---

### Summary

The `respond_ckd` function in the MPC smart contract performs **no cryptographic verification** of the `CKDResponse` when the request uses the `AppPublicKey` (privately verifiable) variant. A single attested participant — strictly below the signing threshold — can race the honest MPC nodes and submit an arbitrary forged `CKDResponse` that the contract unconditionally accepts and delivers to the requesting TEE application. This allows one Byzantine participant to substitute an attacker-controlled secret for the legitimate threshold-derived secret, defeating the confidentiality guarantee of the CKD feature.

---

### Finding Description

The `respond_ckd` entry point in `crates/contract/src/lib.rs` handles two variants of the app public key:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← NO CHECK
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV` (publicly verifiable), the contract calls `ckd_output_check`, which performs a BLS pairing check to confirm the response was computed from the real MPC master secret. For `AppPublicKey` (privately verifiable), the arm is **empty** — any `CKDResponse` with arbitrary `big_y` and `big_c` values is accepted and forwarded to the caller via `resolve_yields_for`. [2](#0-1) 

The CKD protocol design explicitly states that the secret `s` must never be revealed to any entity other than the requesting application: [3](#0-2) 

The `AppPublicKeyPV` variant was introduced precisely to add on-chain verifiability. The `AppPublicKey` variant was left without on-chain verification, relying entirely on the app to verify after decryption. The contract does not enforce this.

---

### Impact Explanation

**Impact: Critical** — Confidential key derivation output produced without required threshold participant authorization.

An attacker who is a single attested participant can:

1. Observe a pending `AppPublicKey` CKD request from the NEAR indexer.
2. Immediately call `respond_ckd` with a forged `CKDResponse` — e.g., `big_y = G1` (generator) and `big_c = attacker_chosen_point` — before the honest MPC nodes complete the multi-round CKD protocol.
3. The contract accepts the forged response (no check for `AppPublicKey` variant) and resolves the yield.
4. The requesting TEE application receives the forged `(big_y, big_c)` and decrypts: `big_s = big_c − a·big_y`.
5. If the app does not independently verify `e(big_s, G2) = e(H(app_id, mpc_pk), mpc_pk)`, it derives a final secret from an attacker-controlled value. The attacker knows this secret.

Setting `big_y = identity` and `big_c = attacker_chosen_G1_point` makes `big_s = attacker_chosen_G1_point` regardless of the app's ephemeral private key `a`, giving the attacker full control over the derived secret.

The honest nodes' subsequent `respond_ckd` call fails because the request has already been resolved — the legitimate secret is permanently lost for this request.

---

### Likelihood Explanation

**Likelihood: Medium.**

- The attacker must be a registered, attested MPC participant — a role that is not open to the public but is achievable by a malicious node operator.
- The attack requires winning a race against honest nodes. The attacker has a structural timing advantage: honest nodes must complete multiple rounds of P2P communication before submitting, while the attacker can submit immediately upon seeing the on-chain request.
- The `AppPublicKey` variant is the legacy/default variant used by the `ckd-example-cli` and is the simpler path for developers, making it likely to be used in production. [4](#0-3) 

---

### Recommendation

Apply the same `ckd_output_check` pairing verification to the `AppPublicKey` variant that is already applied to `AppPublicKeyPV`. The check `e(big_c, G2) = e(big_y, app_pk2) · e(H(app_id, pk), mpc_pk)` requires `pk2 = a·G2`, which is not present in the `AppPublicKey` variant. Therefore, the recommended fix is to **deprecate the unverifiable `AppPublicKey` variant** and require all callers to use `AppPublicKeyPV`, or to add a separate on-chain verification path for `AppPublicKey` using the available public parameters.

---

### Proof of Concept

1. Attacker is an attested participant in the running MPC network.
2. A TEE app calls `request_app_private_key` with `AppPublicKey(pk1)` where `pk1 = a·G1`.
3. Before honest nodes complete the CKD protocol, attacker calls:
   ```
   respond_ckd(
     request = <the pending CKDRequest>,
     response = CKDResponse {
       big_y: G1_identity,   // identity point
       big_c: attacker_G1,   // any G1 point the attacker knows the discrete log of
     }
   )
   ```
4. Contract executes lines 675–688: `AppPublicKey` arm is empty, no check runs. `resolve_yields_for` delivers the forged response. [5](#0-4) 

5. App decrypts: `big_s = attacker_G1 − a · identity = attacker_G1`. Attacker knows `big_s`.
6. App derives final secret via `derive_strong_key(big_s, b"")` — attacker knows the result. [6](#0-5) 

The test at line 3424–3427 confirms the contract accepts `CKDResponse { big_y: [1u8;48], big_c: [2u8;48] }` — arbitrary byte arrays — for the `AppPublicKey` variant without any cryptographic check: [7](#0-6)
