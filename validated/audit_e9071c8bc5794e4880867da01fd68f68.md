### Title
Cache Poisoning in `request_and_validate_additions` Allows Wallet to Accept Coin States with Unvalidated Inclusion Proofs — (`chia/wallet/util/wallet_sync_utils.py`)

### Summary

`request_and_validate_additions` unconditionally writes to the additions cache at line 223 regardless of whether `validate_additions` returned `True` or `False`. Because `_collect_valid_states` continues iterating over coin states after one fails, a second coin state sharing the same `(header_hash, puzzle_hash)` pair will hit the poisoned cache entry and be accepted as valid without any Merkle proof verification.

### Finding Description

In `chia/wallet/util/wallet_sync_utils.py`, `request_and_validate_additions`:

```python
result: bool = validate_additions(
    additions_res.coins,
    additions_res.proofs,
    additions_root,
)
peer_request_cache.add_to_additions_in_block(header_hash, puzzle_hash, height)  # ← always runs
return result
``` [1](#0-0) 

`add_to_additions_in_block` is called at line 223 before the caller ever inspects `result`. If `validate_additions` returns `False` (e.g., because `len(coins) != len(proofs)`), the `(header_hash, puzzle_hash)` key is still written into `_additions_in_block`. [2](#0-1) 

The cache guard at lines 205–206 then returns `True` unconditionally for any subsequent call with the same key:

```python
if peer_request_cache.in_additions_in_block(header_hash, puzzle_hash):
    return True
``` [3](#0-2) 

The cache is per-peer, stored in `WalletNode.peer_caches` keyed by `peer_node_id`, and is **not cleared** when a peer is closed: [4](#0-3) [5](#0-4) 

`_collect_valid_states` iterates over all coin states in a batch and **continues the loop** after any single state fails validation: [6](#0-5) 

### Impact Explanation

An unprivileged remote peer can make the wallet accept coin states whose Merkle inclusion proofs were never verified. The wallet's coin store is updated with fabricated coin records, corrupting its sync state and displayed balances. This enables a payment-fraud attack: the attacker convinces the wallet it received XCH/CATs/NFTs that do not exist on-chain, potentially causing the victim to release goods or services.

This matches: **High — Corruption of coin records / wallet sync state with direct security impact.**

### Likelihood Explanation

Any peer the wallet connects to in untrusted mode can trigger this. No keys, admin access, or cryptographic break is required. The attacker only needs to:
1. Serve as a peer to the target wallet.
2. Send a batch of two coin states sharing the same `puzzle_hash` and `created_height`.
3. Respond to the first `RequestAdditions` with a malformed `RespondAdditions` (`len(coins) != len(proofs)`).

### Proof of Concept

**Setup:** Attacker operates a node that speaks the Chia wallet protocol. Wallet connects to it in untrusted mode.

**Step 1:** Attacker sends `CoinStateUpdate` containing two `CoinState` objects:
- `cs1`: `coin.puzzle_hash = PH_A`, `created_height = H`
- `cs2`: `coin.puzzle_hash = PH_A`, `created_height = H` (different coin name, same puzzle hash and height)

**Step 2:** Wallet calls `_collect_valid_states([cs1, cs2], peer, cache, ...)`.

**Step 3 (cs1 processing):** `validate_received_state_from_peer` calls `request_and_validate_additions(peer, cache, H, header_hash_H, PH_A, additions_root)`.
- No cache hit.
- Wallet sends `RequestAdditions`.
- Attacker responds with `RespondAdditions(coins=[(PH_A, [coin1])], proofs=[(PH_A, b"", None), (PH_A, b"", None)])` — `len(coins)=1 != len(proofs)=2`.
- `validate_additions` returns `False` at the `len(coins) != len(proofs)` check.
- **Line 223 executes:** `add_to_additions_in_block(header_hash_H, PH_A, H)` — cache poisoned.
- Returns `False`.
- `validate_received_state_from_peer` calls `peer.close(9999)`, returns `False`.
- cs1 is rejected. Loop continues.

**Step 4 (cs2 processing):** `validate_received_state_from_peer` calls `request_and_validate_additions(peer, cache, H, header_hash_H, PH_A, additions_root)`.
- **Cache hit:** `in_additions_in_block(header_hash_H, PH_A)` returns `True`.
- Returns `True` immediately — no network call, no proof verification.
- `validate_received_state_from_peer` proceeds past the additions check and accepts cs2.
- cs2 is added to `valid_states` and committed to the wallet's coin store.

**Result:** The wallet's coin store now contains a coin record for a coin that was never proven to exist in any block.

### Recommendation

Move `add_to_additions_in_block` inside a conditional so it is only called when validation succeeds:

```python
result: bool = validate_additions(
    additions_res.coins,
    additions_res.proofs,
    additions_root,
)
if result:
    peer_request_cache.add_to_additions_in_block(header_hash, puzzle_hash, height)
return result
``` [1](#0-0)

def validate_additions(
    coins: list[tuple[bytes32, list[Coin]]],
    proofs: list[tuple[bytes32, bytes, bytes | None]] | None,
    root: bytes32,
) -> bool:
    if proofs is None:
        # Verify root
        additions_merkle_items: list[bytes32] = []

        # Addition Merkle set contains puzzlehash and hash of all coins with that puzzlehash
        for puzzle_hash, coins_l in coins:
            additions_merkle_items.append(puzzle_hash)
            additions_merkle_items.append(hash_coin_ids([c.name() for c in coins_l]))

        additions_root = bytes32(compute_merkle_set_root(additions_merkle_items))
        if root != additions_root:
            return False
    else:
        if len(coins) != len(proofs):
            return False
