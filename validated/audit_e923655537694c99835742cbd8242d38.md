Audit Report

## Title
`swap()` Silently Succeeds with Zero Output When No Liquidity Is Available or Price Limit Is Pre-Satisfied — (`metric-core/contracts/MetricOmmPool.sol`)

## Summary

`MetricOmmPool.swap()` does not revert when the swap produces zero output due to exhausted liquidity or a price limit already being satisfied. All four internal swap helpers contain early-exit paths that return `(0, 0, 0)` or `(0, 0, 0, 0)` without reverting. These zero values propagate through `_executeSwap` back to `swap()`, which then calls the swap callback with `(0, 0)`, emits a `Swap` event with zero amounts, and returns `(0, 0)` — a silent no-op that is indistinguishable from a successful swap.

## Finding Description

**Root cause:** The only guard against a zero-amount swap is `require(amountSpecified != 0, InvalidAmount())` at entry ( [1](#0-0) ), which checks the *requested* amount, not the *executed* amount. There is no post-execution check that the deltas are non-zero.

**Four early-exit paths confirmed in production code:**

1. `_swapToken0ForToken1SpecifiedInput`: returns `(0, 0, 0)` when `params.priceLimitX64 >= initialPriceX64` [2](#0-1) 

2. `_swapToken1ForToken0SpecifiedInput`: returns `(0, 0, 0)` when `params.priceLimitX64 <= initialPriceX64` [3](#0-2) 

3. `_swapToken0ForToken1SpecifiedOutput`: silently caps `amountOutScaled` to zero when `binTotals.scaledToken1 == 0`, then returns `(0, 0, 0, 0)` on the price-limit check [4](#0-3) 

4. `_swapToken1ForToken0SpecifiedOutput`: same cap for token0 [5](#0-4) 

**Propagation through `_executeSwap`:** When any helper returns `(0, 0, 0)`, `_executeSwap` sets `amount0DeltaScaled = 0` and `amount1DeltaScaled = 0` and returns them to `swap()`. [6](#0-5) 

**Consequence in `swap()`:** With both deltas at zero:
- `amount1Delta < 0` is false → no token transfer to recipient [7](#0-6) 
- `metricOmmSwapCallback(0, 0, callbackData)` is called — caller pays nothing [8](#0-7) 
- `amount0Delta > 0` is false → `IncorrectDelta` guard is skipped entirely [9](#0-8) 
- `emit Swap(...)` fires with zero amounts [10](#0-9) 
- `(0, 0)` is returned — transaction succeeds [11](#0-10) 

## Impact Explanation

Any router, aggregator, or user calling `swap()` with a non-zero `amountSpecified` receives a successful transaction and a `(0, 0)` return value when the pool has no liquidity in the requested direction or when the price limit is already satisfied. There is no on-chain signal distinguishing "swap executed" from "swap silently did nothing." Downstream logic in routers that assumes a successful `swap()` call delivered tokens will proceed incorrectly — constituting broken core pool functionality causing unusable swap flows, which is an allowed impact under the contest rules.

## Likelihood Explanation

Both triggering conditions are reachable by any unprivileged caller under normal operating conditions:

- **No liquidity:** A pool fully drained of one token (all token1 removed by LPs, `binTotals.scaledToken1 == 0`) will silently accept any `zeroForOne = true` swap. The `_swapToken0ForToken1SpecifiedInput` loop breaks immediately at `if (totalAvailableToken1Scaled == 0) { break; }` and returns `(0, 0, 0)`. [12](#0-11) 
- **Price limit pre-satisfied:** A caller passing a `priceLimitX64` already satisfied by the current oracle price triggers the early-exit on every swap call. This requires no malicious setup and is a normal user error that should revert rather than silently succeed.

## Recommendation

After `_executeSwap` returns, add a check that reverts if both deltas are zero and `amountSpecified != 0`:

```solidity
if (amount0Delta == 0 && amount1Delta == 0) revert SwapResultedInZeroOutput();
```

This should be placed immediately after line 248 (after `_executeSwap` returns) and before the transfer/callback logic, ensuring callers always receive a descriptive revert rather than a silent no-op.

## Proof of Concept

1. Deploy a pool with token0/token1 and add liquidity only to token0-side bins (`binTotals.scaledToken1 == 0`).
2. Call `swap(recipient, true, 1e18, 0, callbackData, "")` — exact-input, selling token0 for token1.
3. Inside `_swapToken0ForToken1SpecifiedInput`, `totalAvailableToken1Scaled == 0` causes the loop to break immediately after the price-limit check passes.
4. `_executeSwap` returns `(0, 0, 0)`.
5. `swap()` calls `metricOmmSwapCallback(0, 0, callbackData)` — caller pays nothing.
6. `emit Swap(msg.sender, recipient, true, 0, 0, curBinIdx, curPosInBin, 0)` fires.
7. `swap()` returns `(0, 0)` — transaction succeeds with no state change.
8. Recipient received 0 token1; caller paid 0 token0. The swap silently did nothing.
