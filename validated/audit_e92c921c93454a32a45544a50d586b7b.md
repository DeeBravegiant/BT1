### Title
Pool Operator Can Silently Redirect Farmer to Malicious Pool via HTTP 301/308 Without Timelock — (File: chia/farmer/farmer.py)

### Summary

The Chia farmer's `update_pool_state` loop automatically follows HTTP 301/308 permanent redirects from the pool server and immediately overwrites the locally stored `pool_url` with no farmer consent, no on-chain transaction, and no timelock. A pool operator can silently redirect all connected farmers to an arbitrary new pool endpoint. Because the farmer is simultaneously locked into the singleton for `relative_lock_height` blocks, it cannot exit before the redirect takes effect, mirroring the ERC1538 "no timelock on privileged parameter change" class exactly.

Any HTTP 301 or 308 redirect served by the pool's `/pool_info` endpoint causes `new_pool_url` to be set to the redirect destination. [1](#0-0) 

**Immediate, unconditional write — `update_pool_state`:**

```python

The new URL is written to the persistent `PoolingShareState` config file immediately, with no delay, no farmer confirmation, no on-chain record, and no notification. [2](#0-1) 

From this point forward every partial submission (`POST /partial`) and every farmer registration (`POST /farmer`, `PUT /farmer`) is sent to the attacker-controlled endpoint. [3](#0-2) 

The farmer's `payout_instructions` (the address where pool rewards are sent) are stored locally and pushed to whichever pool URL is currently active. [4](#0-3)  The malicious pool receives those instructions but is under no obligation to honour them.

**Missing guard:** There is no timelock, no farmer-side confirmation prompt, no on-chain verification of the new URL, and no comparison against the URL embedded in the on-chain singleton state before the config is overwritten.

### Impact Explanation

**High — pool authorization bypass enabling payout redirection.**

After the redirect:
1. All partials are submitted to the attacker's pool. The attacker's pool can silently discard them, crediting zero points to the farmer.
2. The farmer's pool rewards (the 7/8 block reward paid to `p2_singleton_puzzle_hash`) accumulate at the attacker's pool address and are never distributed to the farmer's `payout_instructions`.
3. The farmer's 1/8 farmer reward is unaffected (it goes directly to `farmer_target`), but the pooled portion — the primary reason for joining a pool — is fully diverted.

This matches the allowed High impact: *"Bypass of … pool … authorization that enables … payout redirection."*

### Likelihood Explanation

The pool operator controls the HTTP server that the farmer polls every `UPDATE_POOL_INFO_INTERVAL` seconds. Issuing a 301/308 redirect requires no Chia-protocol privilege, no key material, and no on-chain action — only control of the pool's web server. On mainnet the redirect destination must also be HTTPS (enforced by the `enforce_https` check), but any pool operator can obtain a valid TLS certificate trivially.

The farmer cannot exit the singleton immediately; it must wait `relative_lock_height` blocks (set at join time, e.g. 32 blocks ≈ 5 minutes, but pools commonly require much longer values). During that window the attacker collects all partials without crediting them. [5](#0-4) 

### Recommendation

Mirror the ERC1538 report's own recommendation: enforce a timelock before the new pool URL becomes active.

1. **Staged URL update**: store `pending_pool_url` and `pending_pool_url_since` in `PoolingShareState`. Only promote `pending_pool_url` to `pool_url` after `relative_lock_height` blocks (or a fixed minimum, e.g. 32 blocks) have elapsed, giving the farmer time to detect and abort.
2. **Farmer notification**: log a prominent warning and emit a state-change event when a redirect is detected, so GUI/monitoring tools can alert the operator.
3. **On-chain anchor**: optionally require that the new URL match the `pool_url` field recorded in the on-chain singleton state before accepting it.

### Proof of Concept

```
1. Farmer joins pool at https://legitimate-pool.example.com
   (relative_lock_height = 32, singleton confirmed on-chain)

2. Pool operator configures nginx on legitimate-pool.example.com:
      location /pool_info {
          return 301 https://attacker-pool.example.com/pool_info;
      }

3. Farmer's update_pool_state() fires (every UPDATE_POOL_INFO_INTERVAL seconds):
   - GET https://legitimate-pool.example.com/pool_info
   - aiohttp follows the 301 → lands on https://attacker-pool.example.com/pool_info
   - resp.history = [301], resp.url = "https://attacker-pool.example.com/pool_info"
   - new_pool_url = "https://attacker-pool.example.com"
   - PoolingShareState.pool_url overwritten immediately (no timelock)

4. All subsequent POST /partial requests go to attacker-pool.example.com.
   Attacker discards them → farmer earns 0 points.

5. Farmer attempts to leave pool:
   - Must wait relative_lock_height blocks before singleton can transition.
   - During this window attacker continues to receive (and discard) partials.

6. Pool rewards (7/8 block reward) sent to p2_singleton_puzzle_hash are
   never distributed by the attacker pool → farmer loses all pooled rewards.
``` [6](#0-5) [7](#0-6) [8](#0-7)

    async def _pool_get_pool_info(self, pool_config: PoolingShareState) -> GetPoolInfoResult | None:
        try:
            async with aiohttp.ClientSession(trust_env=True) as session:
                url = f"{pool_config.pool_url}/pool_info"
                async with session.get(url, ssl=ssl_context_for_root(get_mozilla_ca_crt(), log=self.log)) as resp:
                    if resp.ok:
                        response: dict[str, Any] = json.loads(await resp.text())
                        self.log.info(f"GET /pool_info response: {response}")
                        if "error_code" in response:
                            self.handle_failed_pool_response(
                                pool_config.p2_singleton_puzzle_hash,
                                f"Error in GET /pool_info {pool_config.pool_url}, {response}",
                            )
                            return None
                        try:
                            pool_info = GetPoolInfoResponse.from_json_dict(response)
                        except Exception as e:
                            self.handle_failed_pool_response(
                                pool_config.p2_singleton_puzzle_hash,
                                f"Invalid GET /pool_info response {pool_config.pool_url}, {e}",
                            )
                            return None
                        new_pool_url: str | None = None
                        response_url_str = f"{resp.url}"
                        if (
                            response_url_str != url
                            and len(resp.history) > 0
                            and all(r.status in {301, 308} for r in resp.history)
                        ):
                            new_pool_url = response_url_str.replace("/pool_info", "")

                        return GetPoolInfoResult(pool_info=pool_info, new_pool_url=new_pool_url)
                    else:
                        self.handle_failed_pool_response(
                            pool_config.p2_singleton_puzzle_hash,
                            f"Error in GET /pool_info {pool_config.pool_url}, {resp.status}",
                        )

        except Exception as e:
            self.handle_failed_pool_response(
                pool_config.p2_singleton_puzzle_hash, f"Exception in GET /pool_info {pool_config.pool_url}, {e}"
            )

        return None
