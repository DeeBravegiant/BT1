Audit Report

## Title
Attacker Can Exhaust Daily Mint Limit to Temporarily Block All Deposits - (File: contracts/pools/RSETHPoolV3.sol)

## Summary
The `limitDailyMint` modifier in `RSETHPoolV3.sol`, `RSETHPoolV3ExternalBridge.sol`, and `RSETHPoolV3WithNativeChainBridge.sol` tracks `dailyMintAmount` as a monotonically increasing counter with no unprivileged decrement path. Any depositor with sufficient capital can exhaust the entire `dailyMintLimit` in a single transaction, blocking all subsequent deposits for up to 24 hours. Because the attacker receives `wrsETH` at fair value in return, the attack costs only L2 gas and is repeatable every day.

## Finding Description
The `limitDailyMint` modifier in all three pool contracts enforces a per-day cap:

```solidity
// RSETHPoolV3.sol lines 119–124
if (dailyMintAmount + rsETHAmount > dailyMintLimit) {
    revert DailyMintLimitExceeded();
}
dailyMintAmount += rsETHAmount;
```

`dailyMintAmount` is incremented on every successful deposit and is only reset when `currentDay > lastMintDay` (i.e., after a full 24-hour period elapses relative to `startTimestamp`). There is no decrement path available to unprivileged users — `swapAssetToPremintedRsETH` (the reverse-swap) is gated behind `OPERATOR_ROLE`.

An attacker calls `deposit{value: X}("")` where `X` is chosen so that `viewSwapRsETHAmountAndFee(X).rsETHAmount == dailyMintLimit - dailyMintAmount`. The modifier increments `dailyMintAmount` to exactly `dailyMintLimit`. Every subsequent `deposit()` call from any user then hits the revert condition `dailyMintAmount + rsETHAmount > dailyMintLimit` (true for any `rsETHAmount > 0`) until the next day boundary. The attacker receives `wrsETH` via `wrsETH.mint(msg.sender, rsETHAmount)` — a liquid token representing fair value of the deposited ETH — so no capital is destroyed.

The identical root cause exists in:
- `RSETHPoolV3ExternalBridge.sol` lines 130–158
- `RSETHPoolV3WithNativeChainBridge.sol` lines 108–137

Existing mitigations are insufficient: `pause()` (callable by `PAUSER_ROLE`) would also block legitimate users; `setDailyMintLimit()` (callable by `DEFAULT_ADMIN_ROLE`) can raise the limit but the attacker can simply re-exhaust it at the start of the next day; there is no emergency reset of `dailyMintAmount` to zero.

## Impact Explanation
**Medium. Temporary freezing of funds.**

All users are blocked from depositing ETH or supported LSTs into any affected pool for up to 24 hours per attack cycle. The deposit path is the sole entry point for minting `wrsETH`/`rsETH` on L2. A sustained attacker repeating the attack each day can indefinitely deny the deposit service, preventing users from entering the protocol and locking out time-sensitive depositors.

## Likelihood Explanation
**Medium.**

The attacker must hold capital equal to the remaining `dailyMintLimit` in ETH or a supported LST, but receives `wrsETH` at fair value in return — no capital is destroyed. The only real cost is L2 gas, which is negligible on Optimism, Base, or Arbitrum. If `feeBps == 0` (a valid configuration since `setFeeBps` permits 0 in `RSETHPoolV3.sol` and `RSETHPoolV3WithNativeChainBridge.sol`), the attack is entirely free except for gas. The attack is repeatable every 24 hours with no additional capital requirement.

## Recommendation
1. **Emergency reset**: Add a function callable by `PAUSER_ROLE` or `DEFAULT_ADMIN_ROLE` (without timelock) to reset `dailyMintAmount` to zero, enabling rapid recovery from a DoS attempt.
2. **Shorter time windows**: Consider a rolling window (e.g., per-hour) rather than a fixed 24-hour period to reduce the maximum DoS duration.
3. **Minimum deposit fee**: Enforce a non-zero minimum `feeBps` to impose a real capital cost on exhausting the daily limit.
4. **Per-depositor sub-limits**: Cap the maximum single-transaction deposit to a fraction of `dailyMintLimit` to prevent a single actor from exhausting the full limit in one call.

## Proof of Concept
**Setup**: `dailyMintLimit = 1000 ether` (in rsETH terms), `feeBps = 0`, pool is live on an L2, `block.timestamp >= startTimestamp`.

**Steps**:
1. Attacker calls `deposit{value: X}("")` where `X` is chosen so that `viewSwapRsETHAmountAndFee(X).rsETHAmount == dailyMintLimit`.
2. Inside `limitDailyMint`: `dailyMintAmount += 1000e18` → `dailyMintAmount == dailyMintLimit`.
3. Attacker receives `1000e18` `wrsETH` tokens (full fair-value equivalent of their ETH).
4. Any subsequent call by any user to `deposit()` hits:
   ```solidity
   if (dailyMintAmount + rsETHAmount > dailyMintLimit) {
       revert DailyMintLimitExceeded(); // always true for any rsETHAmount > 0
   }
   ```
5. All deposits are blocked for up to 24 hours.
6. Next day: attacker repeats. Cost per day: gas only.

**Foundry test plan**:
```solidity
function test_exhaustDailyMintLimit() public {
    // Setup: deploy pool, set dailyMintLimit = 1000 ether, feeBps = 0
    // Warp to startTimestamp
    vm.warp(pool.startTimestamp());
    uint256 rate = pool.getRate(); // e.g. 1.05e18
    uint256 depositAmount = dailyMintLimit * rate / 1e18; // ETH needed to mint exactly dailyMintLimit rsETH
    
    // Attacker exhausts the limit
    vm.deal(attacker, depositAmount);
    vm.prank(attacker);
    pool.deposit{value: depositAmount}("");
    
    // Victim cannot deposit
    vm.deal(victim, 1 ether);
    vm.prank(victim);
    vm.expectRevert(RSETHPoolV3.DailyMintLimitExceeded.selector);
    pool.deposit{value: 1 ether}("");
}
```
