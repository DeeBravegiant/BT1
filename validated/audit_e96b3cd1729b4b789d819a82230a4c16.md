### Title
Unbounded Loop Over Attacker-Controlled Weight Proof Data Blocks Sync Event Loop - (File: chia/full_node/weight_proof.py)

### Summary
The weight proof validation path in `chia/full_node/weight_proof.py` contains multiple loops that iterate over attacker-controlled lists (`weight_proof.sub_epochs` and `weight_proof.recent_chain_data`) with no enforced upper-bound on their sizes. A malicious peer can craft a `RespondProofOfWeight` message containing an arbitrarily large `WeightProof`, causing the receiving node's validation to consume unbounded CPU time. The most critical path runs synchronously in the main async event loop, blocking the node from processing any valid blocks or sync updates for the duration.

### Finding Description

**Unbounded loop 1 — `_validate_sub_epoch_summaries` (synchronous, main event loop)**

`_validate_sub_epoch_summaries` is called directly (not in an executor) from both `validate_weight_proof` and `validate_weight_proof_single_proc`. It delegates to `_map_sub_epoch_summaries`, which iterates over the entire `weight_proof.sub_epochs` list with no size check: [1](#0-0) 

The `WeightProof.sub_epochs` field is a plain `list[SubEpochData]` with no enforced maximum: [2](#0-1) 

In the async path, this call happens before the process-pool executor is entered, directly on the event loop: [3](#0-2) 

**Unbounded loop 2 — `validate_recent_blocks` (process pool, but unbounded CPU)**

`validate_recent_blocks` iterates over every entry in `recent_chain.recent_chain_data` with no size guard: [4](#0-3) 

No check on `len(recent_chain_data)` exists before this loop. The honest node caps its own proof at two sub-epochs of blocks, but nothing enforces this on a received proof.

**Contrast with the per-sub-epoch segment check (partial mitigation)**

`_validate_sub_epoch_segments` does enforce a per-sub-epoch segment cap: [5](#0-4) 

However, the total number of sub-epochs iterated is itself unbounded, and the `sub_epochs` list is not checked before the synchronous `_validate_sub_epoch_summaries` call.

**Entry point**

The full node calls `validate_weight_proof` on any `RespondProofOfWeight` received from a peer during sync: [6](#0-5) 

The only pre-validation checks are on the last element's height and weight, not on the list sizes: [7](#0-6) 

### Impact Explanation

When `_validate_sub_epoch_summaries` iterates over a crafted `sub_epochs` list with millions of entries, it runs synchronously on the asyncio event loop. During this time the node cannot process any incoming messages — including valid blocks, peer announcements, or other sync messages. This constitutes a **long-lived inability for honest nodes to process valid blocks and sync updates**, matching the High impact tier. The wallet's `WalletWeightProofHandler.validate_weight_proof` calls the same inner function, so wallets are equally affected during their sync path. [8](#0-7) 

### Likelihood Explanation

Any unprivileged actor who can act as a Chia full node peer can trigger this. During initial sync or a reorg, the victim node requests a weight proof from a randomly chosen peer with the claimed peak. An attacker who advertises a high-weight peak will be selected and can respond with a crafted `WeightProof`. No keys, admin access, or cryptographic break are required — only the ability to run a Chia node and respond to `RequestProofOfWeight`.

### Recommendation

1. **Enforce a maximum on `weight_proof.sub_epochs` length** before calling `_validate_sub_epoch_summaries`. The expected maximum is bounded by `tip_height // SUB_EPOCH_BLOCKS`; reject proofs that exceed a safe constant (e.g., `MAX_SAMPLES * 2`).
2. **Enforce a maximum on `weight_proof.recent_chain_data` length** before calling `validate_recent_blocks`. The honest bound is approximately `WEIGHT_PROOF_RECENT_BLOCKS`; reject proofs that exceed it.
3. **Move `_validate_sub_epoch_summaries` into the process-pool executor** so that even an oversized proof cannot block the main event loop.
4. **Add a total sub-epoch count check** in `_validate_sub_epoch_segments` in addition to the existing per-sub-epoch segment cap.

### Proof of Concept

```python
# Attacker constructs a WeightProof with an enormous sub_epochs list
from chia.types.weight_proof import WeightProof
from chia_rs import SubEpochData
from chia_rs.sized_ints import uint8, uint32, uint64, uint128

# SubEpochData is small; millions of entries are serializable within
# typical network message size limits
huge_sub_epochs = [
    SubEpochData(
        reward_chain_hash=bytes32(b"\x00" * 32),
        num_blocks_overflow=uint8(0),
        new_sub_slot_iters=None,
        new_difficulty=None,
        challenge_merkle_root=None,
    )
    for _ in range(5_000_000)  # 5 million entries
]

malicious_wp = WeightProof(
    sub_epochs=huge_sub_epochs,
    sub_epoch_segments=[],
    recent_chain_data=[],  # will fail later, but _validate_sub_epoch_summaries runs first
)

# When the victim calls:
#   summaries, _ = _validate_sub_epoch_summaries(constants, malicious_wp)
# _map_sub_epoch_summaries iterates all 5M entries synchronously on the event loop,
# blocking the node for tens of seconds to minutes.
```

The victim node's event loop is blocked for the duration of the iteration, preventing it from processing any valid blocks or peer messages received during that window.
