### Title
Multiple Sub-Transactions Can Claim the Same Stake Credential Deposit Refund — (`eras/dijkstra/impl/testlib/Test/Cardano/Ledger/Dijkstra/Imp/CertSpec.hs`)

---

### Summary

In the Dijkstra era, a top-level transaction may contain multiple sub-transactions. No batch-level guard prevents two or more sub-transactions from each including an `UnRegDepositTxCert` for the **same** staking credential. Because the predicate failure that would block this has not been implemented, the transaction currently succeeds, allowing an attacker to extract N × `keyDeposit` ADA from the deposit pot while having paid only 1 × `keyDeposit`.

---

### Finding Description

The Dijkstra era introduces nested sub-transactions via `subTransactionsTxBodyL`. Each sub-transaction is processed sequentially by the `SUBLEDGERS` rule. The `validateBatchWithdrawals` function in `dijkstraUtxoTransition` correctly aggregates reward withdrawals across the entire batch and checks them against the original account balance: [1](#0-0) 

No equivalent batch-level guard exists for **deposit refunds** arising from `UnRegDepositTxCert` certificates. The `SUBENTITIES` rule processes each sub-transaction's certificates independently, applying `applyWithdrawals` to the threaded `certState`: [2](#0-1) 

The developers have explicitly acknowledged this gap with a disabled (`xit`) test whose expected predicate failure is `error "TODO: predicate failure not yet implemented"`: [3](#0-2) 

The `xit` marker means the test body is never executed; the `error "TODO"` placeholder confirms no blocking check has been added. The test comment — *"Multiple subtransactions cannot get the same refund"* — directly names the vulnerability.

The attack vector: craft a top-level transaction whose `subTransactionsTxBodyL` contains N sub-transactions, each carrying `UnRegDepositTxCert stakingCred keyDeposit` for the same credential. Because no cross-sub-transaction deduplication of deposit refunds is enforced, the value-conservation check for each sub-transaction independently accounts for the full `keyDeposit` refund, and the transaction is accepted, draining N × `keyDeposit` from the deposit pot.

The analogous batch-withdrawal guard that **does** exist shows the intended pattern: [4](#0-3) 

No such aggregation is performed for deposit refunds across sub-transactions.

---

### Impact Explanation

**Critical — Direct creation of ADA through an invalid ledger state transition.**

An attacker who registered a staking credential (paying `keyDeposit` once) can construct a Dijkstra-era transaction with N sub-transactions each claiming the same `keyDeposit` refund. The deposit pot is debited N times while only one deposit was ever paid, creating N − 1 units of `keyDeposit` ADA out of thin air. This violates the preservation-of-value invariant that is the foundational correctness property of the Cardano ledger. [5](#0-4) 

---

### Likelihood Explanation

The Dijkstra era is the next scheduled era after Conway. The vulnerability is in production rule code, not in tests or documentation. The attack requires only:
1. Registering a staking credential (permissionless, costs `keyDeposit`).
2. Constructing a valid Dijkstra-era transaction with duplicate `UnRegDepositTxCert` certificates spread across sub-transactions.

No privileged access, governance majority, or external dependency is required. The attacker controls the transaction body entirely.

---

### Recommendation

Add a batch-level check for deposit refunds across all sub-transactions, mirroring `validateBatchWithdrawals`. Before accepting the batch, aggregate all `UnRegDepositTxCert` (and `UnRegDRepTxCert`) certificates across the top-level transaction and every sub-transaction, deduplicate by credential, and verify that no credential appears more than once across the batch. Alternatively, thread the post-certificate `certState` into each subsequent sub-transaction's value-conservation check so that a credential unregistered by `subTx1` is seen as absent (refund = 0) by `subTx2`.

The existing `validateBatchWithdrawals` in `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Utxo.hs` is the correct structural template for this fix.

---

### Proof of Concept

The developers' own pending test is the proof of concept: [3](#0-2) 

```
stakingCred  ← register once, paying keyDeposit

subTx1 = { certs: [UnRegDepositTxCert stakingCred keyDeposit] }
subTx2 = { certs: [UnRegDepositTxCert stakingCred keyDeposit] }

topTx = { subTransactions: [subTx1, subTx2] }
```

`subTx1` claims `keyDeposit` in its value-conservation balance.  
`subTx2` claims `keyDeposit` again in its value-conservation balance.  
No cross-sub-transaction deduplication check exists; the transaction is accepted.  
Net effect: 2 × `keyDeposit` extracted from the deposit pot; 1 × `keyDeposit` was deposited.

The `xit` marker and `error "TODO: predicate failure not yet implemented"` confirm the blocking check is absent from the production rule code in `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/SubEntities.hs` and `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Utxo.hs`.

-- | For each account, the total withdrawals across the entire batch should not exceed the original account balance.
-- Unregistered accounts are treated as having 0 balance.
validateBatchWithdrawals ::
  ( EraTx era
  , EraAccounts era
  , DijkstraEraTxBody era
  ) =>
  Accounts era ->
  Tx TopTx era ->
  Test (DijkstraUtxoPredFailure era)
validateBatchWithdrawals accounts tx =
  let allWithdrawals =
        Map.unionsWith (<>) $
          unWithdrawals (tx ^. bodyTxL . withdrawalsTxBodyL)
            : [ unWithdrawals $ subTx ^. bodyTxL . withdrawalsTxBodyL
              | subTx <- OMap.elems $ tx ^. bodyTxL . subTransactionsTxBodyL
              ]
      badWithdrawals =
        Map.mapMaybeWithKey
          ( \acctAddr withdrawn ->
              let balance = getAccountBalance acctAddr
               in if withdrawn > balance
                    then Just Mismatch {mismatchSupplied = withdrawn, mismatchExpected = balance}
                    else Nothing
          )
          allWithdrawals
   in failureOnNonEmptyMap badWithdrawals WithdrawalsExceedAccountBalance
  where
    getAccountBalance (AccountAddress _ (AccountId cred)) =
      case lookupAccountState cred accounts of
        Nothing -> mempty -- unregistered account, 0 balance
        Just accountState -> fromCompact $ accountState ^. balanceAccountStateL

dijkstraSubEntitiesTransition = do
  TRC (subCertsEnv, certState, certificates) <- judgmentContext
  let tx = certsTx subCertsEnv
      pp = certsPParams subCertsEnv
      curEpoch = certsCurrentEpoch subCertsEnv
      withdrawals = tx ^. bodyTxL . withdrawalsTxBodyL
      accounts = certState ^. certDStateL . accountsL

  runTest $ Conway.validateWithdrawalsDelegated accounts tx

  network <- liftSTS $ asks networkId
  let (missingWithdrawals, exceededWithdrawals) =
        case withdrawalsThatExceedAccountBalance withdrawals network accounts of
          Nothing -> (Map.empty, Map.empty)
          Just (missing, exceeded) -> (unWithdrawals missing, exceeded)
  failOnNonEmptyMap missingWithdrawals $
    injectFailure . SubWithdrawalsMissingAccounts . Withdrawals . NEM.toMap
  failOnNonEmptyMap exceededWithdrawals $ injectFailure . SubWithdrawalAmountsExceedAccountBalances

  let certStateBeforeSubCerts =
        certState
          & Conway.updateDormantDRepExpiries tx curEpoch
          & Conway.updateVotingDRepExpiries tx curEpoch (pp ^. ppDRepActivityL)
          & certDStateL . accountsL %~ applyWithdrawals withdrawals
  certStateAfterSubCerts <-
    trans @(EraRule "SUBCERTS" era) $ TRC (subCertsEnv, certStateBeforeSubCerts, certificates)

  let directDeposits = tx ^. bodyTxL . directDepositsTxBodyL
      accountsAfterSubCerts = certStateAfterSubCerts ^. certDStateL . accountsL
  failOnJust (directDepositsMissingAccounts directDeposits accountsAfterSubCerts) $
    injectFailure . SubDirectDepositsToMissingAccounts

  pure $ certStateAfterSubCerts & certDStateL . accountsL %~ applyDirectDeposits directDeposits
