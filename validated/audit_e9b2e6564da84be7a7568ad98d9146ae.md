### Title
`POOLREAP` Uses `Map.withoutKeys` for Dangling VRF Hashes Instead of Decrement, Allowing Post-v11 Duplicate VRF Registration — (`eras/shelley/impl/src/Cardano/Ledger/Shelley/Rules/PoolReap.hs`)

---

### Summary

In `poolReapTransition`, when a pool re-registers with a new VRF key, the old VRF key is collected into `danglingVRFKeyHashes` and then removed from `psVRFKeyHashes` via `Map.withoutKeys`. This unconditionally deletes the entire map entry regardless of its reference count. Because `psVRFKeyHashes` stores a `NonZero Word64` count to handle pre-v11 pools that legitimately share the same VRF key, this deletion is incorrect when the count is greater than one. The result is that `psVRFKeyHashes` no longer reflects the true set of VRF keys in use, and a subsequent pool registration certificate can bypass the `VRFKeyHashAlreadyRegistered` guard, re-introducing a duplicate VRF key that the v11 hard fork was designed to prevent.

---

### Finding Description

`poolReapTransition` in `PoolReap.hs` performs two distinct VRF-key cleanup operations at the epoch boundary:

**Step 1 — compute dangling VRF hashes** (lines 138–148): pools that have a pending re-registration (`psFutureStakePoolParams`) with a *different* VRF key contribute their *current* (old) VRF key to `danglingVRFKeyHashes`. [1](#0-0) 

**Step 2 — update `psVRFKeyHashes`** (lines 224–227): retired pools' VRF keys are handled with `removeVRFKeyHashOccurrences`, which correctly *decrements* the count and removes the entry only when it reaches zero. But dangling VRF keys are handled with `Map.withoutKeys`, which *unconditionally deletes* the entire entry. [2](#0-1) 

The `removeVRFKeyHashOccurrence` helper that is used for retired pools is: [3](#0-2) 

`psVRFKeyHashes` is a `Map (VRFVerKeyHash StakePoolVRF) (NonZero Word64)` — a reference-counted map: [4](#0-3) 

The count can exceed 1 because the v11 migration (`populateVRFKeyHashes`) scans all currently registered pools and counts every occurrence of each VRF key, including pre-v11 duplicates: [5](#0-4) 

The `VRFKeyHashAlreadyRegistered` guard in `poolTransition` relies exclusively on `psVRFKeyHashes` membership: [6](#0-5) 

**Concrete attack path:**

1. Before v11: Pool A and Pool B are both registered with VRF key `v` (allowed pre-v11).
2. v11 hard fork: `populateVRFKeyHashes` runs → `psVRFKeyHashes[v] = 2`.
3. After v11: Pool A re-registers with a fresh VRF `v2` → `psFutureStakePoolParams[A] = {vrf: v2}`, `psVRFKeyHashes[v2] = 1`.
4. Epoch boundary (`POOLREAP`):
   - `danglingVRFKeyHashes = {v}` (Pool A's old VRF, since `sps.vrf = v ≠ v2 = sppF.vrf`).
   - `Map.withoutKeys {v}` removes `v` entirely from `psVRFKeyHashes`.
   - Pool B still uses VRF `v`; the entry should have been decremented to 1, not deleted.
5. `psVRFKeyHashes` now contains only `{v2: 1}` — `v` is absent.
6. An unprivileged attacker submits `RegPool` with VRF `v`. `Map.notMember v psVRFKeyHashes` evaluates to `True`, so `VRFKeyHashAlreadyRegistered` is not raised.
7. Pool B and the attacker's new pool now both hold VRF `v` post-v11, violating the uniqueness invariant.

The test suite explicitly acknowledges that pre-v11 duplicate VRFs can survive into the post-v11 era and must be handled correctly (the retirement path is handled correctly; the re-registration path is not): [7](#0-6) 

---

### Impact Explanation

An unprivileged pool operator can submit a `RegPool` certificate that bypasses the `VRFKeyHashAlreadyRegistered` validation limit introduced at v11. Two pools sharing the same VRF key post-v11 means both are elected as slot leaders for identical slots (their VRF outputs are identical), allowing the attacker to predict the victim pool's entire slot-leadership schedule and to produce competing blocks for every slot the victim wins. This exceeds the intended validation limits of the pool registration rule.

**Applicable impact category:** Medium — attacker-controlled certificate exceeds intended validation limits.

---

### Likelihood Explanation

The precondition (pre-v11 duplicate VRFs surviving into the post-v11 era) is explicitly acknowledged and tested by the development team. Any pool that registered before v11 with a VRF key shared by another pool satisfies the precondition. The trigger (that pool re-registering with a new VRF after v11) is a routine operational action. Once the epoch boundary processes the re-registration, the window is permanent — `psVRFKeyHashes` is not self-healing. The attacker needs only to observe the on-chain state and submit a single `RegPool` transaction.

---

### Recommendation

Replace the `Map.withoutKeys danglingVRFKeyHashes` call with the same `removeVRFKeyHashOccurrences` helper used for retired pools. Because `danglingVRFKeyHashes` is a `Set` (deduplicated), the underlying `Map.elems` list (before `Set.fromList`) should be passed directly to `removeVRFKeyHashOccurrences` so that each re-registering pool decrements the count once:

```haskell
-- Before (incorrect):
& certPStateL . psVRFKeyHashesL
  %~ ( removeVRFKeyHashOccurrences retiredVRFKeyHashes
         . (`Map.withoutKeys` danglingVRFKeyHashes)
     )

-- After (correct):
& certPStateL . psVRFKeyHashesL
  %~ ( removeVRFKeyHashOccurrences retiredVRFKeyHashes
         . removeVRFKeyHashOccurrences (Set.toList danglingVRFKeyHashes)
     )
```

This mirrors the correct decrement-and-remove-if-zero semantics already applied to retired pools' VRF keys. [8](#0-7)

-- Map.withoutKeys {vrf} removes vrf entirely
-- psVRFKeyHashes = {vrfNew: 1}   ← BUG: should be {vrf: 1, vrfNew: 1}

```haskell
populateVRFKeyHashes :: PState era -> PState era
populateVRFKeyHashes pState =
  pState
    & psVRFKeyHashesL
      %~ accumulateVRFKeyHashes (pState ^. psStakePoolsL) (^. spsVrfL)
        . accumulateVRFKeyHashes (pState ^. psFutureStakePoolParamsL) (^. sppVrfL)
  where
    accumulateVRFKeyHashes ::
      Map (KeyHash StakePool) a ->
      (a -> VRFVerKeyHash StakePoolVRF) ->
      Map (VRFVerKeyHash StakePoolVRF) (NonZero Word64) ->
      Map (VRFVerKeyHash StakePoolVRF) (NonZero Word64)
    accumulateVRFKeyHashes spMap getVrf acc =
      Map.foldr' (addVRFKeyHashOccurrence . getVrf) acc spMap
    addVRFKeyHashOccurrence vrfKeyHash =
      Map.insertWith combine vrfKeyHash (knownNonZeroBounded @1)
      where
        -- Saturates at maxBound: if (+1) would overflow to 0, keep existing value
        combine _ oldVal = fromMaybe oldVal $ mapNonZero (+ 1) oldVal
```
