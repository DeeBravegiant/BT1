### Title
Constructor `replace_class` Bypasses `__validate_deploy__` Signature Verification in DeployAccount — (`crates/blockifier/src/execution/syscalls/syscall_base.rs`, `crates/blockifier/src/transaction/account_transaction.rs`)

### Summary
A `DeployAccount` transaction's constructor runs before `validate_tx`. Because `replace_class` has no execution-mode guard, a constructor executing in validate mode can call `replace_class(B)` to overwrite the class hash at the newly deployed address. When `validate_tx` subsequently reads the class hash from state, it dispatches `__validate_deploy__` to class B instead of the originally declared class A. An attacker who controls class B can make its `__validate_deploy__` unconditionally return `VALID`, bypassing all signature verification.

### Finding Description
The execution order in `run_non_revertible` for `DeployAccount` is: constructor (`run_execute`) → `validate_tx`. The constructor runs in `ExecutionMode::Validate`. The `replace_class` syscall in `syscall_base.rs::replace_class` performs only a Cairo1 type check — it has no `ExecutionMode::Validate` rejection guard, unlike `get_class_hash_at`, `meta_tx_v0`, and (conditionally) `deploy`. After the constructor mutates the class hash via `replace_class`, `validate_tx` calls `state.get_class_hash_at(storage_address)` and gets the replacement class hash B. The OS code confirms this is the dispatch target: it explicitly reads `state_entry.class_hash` after the constructor and uses it as the `class_hash` field of the `validate_deploy_execution_context`. Class B's `__validate_deploy__` is called instead of class A's.

### Impact Explanation
**Critical.** An attacker can deploy a contract without a valid signature for the declared class. The transaction hash commits to class A, but `__validate_deploy__` runs against class B (attacker-controlled). If class B's `__validate_deploy__` always returns `VALID`, the transaction is accepted with an arbitrary or empty signature. This is a direct bypass of account validation and signature verification for `DeployAccount` transactions — an invalid/unauthorized transaction is accepted through the account validation path.

### Likelihood Explanation
Fully unprivileged. The attacker only needs to: (1) pre-fund the deterministic contract address, (2) declare class A (constructor calls `replace_class(B)`) and class B (trivially-passing `__validate_deploy__`), (3) submit the `DeployAccount` transaction with any signature. No operator, validator, or privileged access is required.

### Recommendation
Add an execution-mode guard to `replace_class` in `syscall_base.rs`, consistent with `get_class_hash_at`:

```rust
pub fn replace_class(&mut self, class_hash: ClassHash) -> SyscallResult<()> {
    if self.context.execution_mode == ExecutionMode::Validate {
        self.reject_syscall_in_validate_mode("replace_class")?;
    }
    // ...
}
```

Alternatively, `validate_tx` for `DeployAccount` could pin the class hash to the value declared in the transaction fields rather than reading it from post-constructor state.

### Proof of Concept
1. Declare class B: `__validate_deploy__` ignores all arguments and returns `VALID`.
2. Declare class A: constructor calls `replace_class(class_hash_of_B)`.
3. Pre-compute the `DeployAccount` contract address for class A, salt S, calldata C.
4. Fund that address.
5. Submit `DeployAccount` with class A, salt S, calldata C, and signature = `[0, 0]`.
6. Observe: constructor sets class hash at deployed address to B; `validate_tx` dispatches to B's `__validate_deploy__`; transaction is accepted.
7. Assert `validate_call_info.call.class_hash == class_hash_of_B` (not class A).
