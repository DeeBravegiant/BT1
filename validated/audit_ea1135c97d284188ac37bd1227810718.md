### Title
Malicious P2P Peer Can Commit a Truncated `ThinStateDiff` by Forging `state_diff_length` in `SignedBlockHeader` — (`crates/apollo_p2p_sync/src/client/state_diff.rs`)

---

### Summary

`HeaderStreamBuilder::parse_data_for_block` accepts a `SignedBlockHeader` from any p2p peer with no cryptographic verification of the block hash or signature. A malicious peer can supply a header whose `state_diff_length` is smaller than the canonical diff's `ThinStateDiff::len()`. `StateDiffStreamBuilder::parse_data_for_block` blindly uses that stored value as `target_state_diff_len`, so it stops collecting chunks early and commits a truncated `ThinStateDiff` to storage, permanently corrupting the node's state view for that block.

---

### Finding Description

**Step 1 — Header accepted without cryptographic verification.**

`HeaderStreamBuilder::parse_data_for_block` performs only two checks before storing the header:

- Block number matches the expected sequence. [1](#0-0) 
- `signatures.len() == ALLOWED_SIGNATURES_LENGTH` (== 1). [2](#0-1) 

It does **not**:
- Verify the `block_hash` against a hash computed from the header fields.
- Call `verify_block_signature` (that function exists in `starknet_api/src/block.rs` lines 722–734 but is **never called** anywhere in the p2p sync client path — confirmed by grep across all of `crates/apollo_p2p_sync/**`).
- Validate `state_diff_length` or `state_diff_commitment` against any trusted source.

The forged header is then written directly to storage. [3](#0-2) 

**Step 2 — State diff sync trusts the stored `state_diff_length` unconditionally.**

`StateDiffStreamBuilder::parse_data_for_block` reads `target_state_diff_len` from the stored header's `state_diff_length` field with no further validation: [4](#0-3) 

It then loops until `current_state_diff_len == target_state_diff_len`: [5](#0-4) 

After the loop exits, the only additional check is `validate_deprecated_declared_classes_non_conflicting` — there is **no** call to `calculate_state_diff_hash` and **no** comparison of the assembled diff against the `state_diff_commitment` stored in the header. `calculate_state_diff_hash` is never called anywhere in `crates/apollo_p2p_sync/src/client/**` (confirmed by grep). [6](#0-5) 

The truncated `ThinStateDiff` is then committed to storage: [7](#0-6) 

**What `ThinStateDiff::len()` counts** (the canonical measure the header should reflect): [8](#0-7) 

---

### Impact Explanation

A malicious peer that the syncing node connects to can:

1. Respond to a header query with a `SignedBlockHeader` carrying `state_diff_length = K` where `K < canonical_len`.
2. Respond to the subsequent state diff query with exactly `K` units of chunk data (e.g., one `DeclaredClass` chunk when the canonical diff has three entries).
3. The node accepts the truncated diff (`current_state_diff_len == target_state_diff_len == K`) and commits it.

Result: the node permanently stores an incomplete `ThinStateDiff` for that block — missing storage diffs, nonces, deployed contracts, or class declarations. All subsequent RPC state queries for that block return wrong values. The node's state root view diverges from the canonical chain.

This matches the Critical scope criterion: *"Unprivileged-user-triggered p2p sync path causes a node to anchor to the wrong state diff."*

---

### Likelihood Explanation

The p2p sync client connects to peers in the network without authenticating their identity or verifying their block data cryptographically. Any node that can respond to a `BlockHeadersRequest` can execute this attack. The only bar to clear is passing the two trivial checks (correct block number, `signatures.len() == 1`). No validator key, sequencer key, or operator privilege is required.

---

### Recommendation

1. **Verify the block signature** before storing the header: call `verify_block_signature` with the known sequencer public key. [9](#0-8) 
2. **Verify the block hash** is consistent with the header fields before storing.
3. **After assembling the full state diff**, call `calculate_state_diff_hash` on the result and compare it against the `state_diff_commitment` stored in the header. Reject the peer if they differ. [10](#0-9) 
4. **Cross-check `state_diff_length`**: assert that `result.len() == target_state_diff_len` after assembly, so a peer cannot satisfy the length counter with chunks that do not match the committed diff.

---

### Proof of Concept

```
1. Start a p2p sync client node.
2. Respond to its HeaderQuery for block N with a SignedBlockHeader where:
     state_diff_length = 1
     state_diff_commitment = <hash of the single-entry truncated diff>
     block_hash = <any value>
     signatures = [<any (r,s) pair>]   // passes the len==1 check
3. Respond to its StateDiffQuery for block N with:
     DataOrFin(Some(StateDiffChunk::DeclaredClass { class_hash: X, compiled_class_hash: Y }))
     DataOrFin(None)   // Fin
4. Assert: storage now contains a ThinStateDiff for block N with only one entry.
5. Assert: the two remaining canonical entries (e.g., a storage diff and a nonce update)
   are absent from storage — silently dropped.
```

pub fn calculate_state_diff_hash(state_diff: &ThinStateDiff) -> StateDiffCommitment {
    let mut hash_chain = HashChain::new();
    hash_chain = hash_chain.chain(&STARKNET_STATE_DIFF0);
    hash_chain = chain_deployed_contracts(&state_diff.deployed_contracts, hash_chain);
    hash_chain = chain_declared_classes(&state_diff.class_hash_to_compiled_class_hash, hash_chain);
    hash_chain =
        chain_deprecated_declared_classes(&state_diff.deprecated_declared_classes, hash_chain);
    hash_chain = hash_chain.chain(&Felt::ONE) // placeholder.
        .chain(&Felt::ZERO); // placeholder.
    hash_chain = chain_storage_diffs(&state_diff.storage_diffs, hash_chain);
    hash_chain = chain_nonces(&state_diff.nonces, hash_chain);
    StateDiffCommitment(PoseidonHash(hash_chain.get_poseidon_hash()))
}
