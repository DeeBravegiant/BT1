### Title
Unconditional Cache Write on Failed Validation in `request_and_validate_additions` Allows Cache Poisoning — (`chia/wallet/util/wallet_sync_utils.py`)

### Summary

`request_and_validate_additions` writes to `PeerRequestCache` at line 223 **unconditionally**, regardless of whether `validate_additions` returned `True` or `False`. A subsequent call for the same `(header_hash, puzzle_hash)` pair hits the cache at line 205–206 and returns `True` without re-validating, violating the invariant that a `True` return means the addition was cryptographically proven.

### Finding Description

In `request_and_validate_additions`: [1](#0-0) 

The early-return cache hit returns `True` unconditionally. [2](#0-1) 

`validate_additions` result is stored in `result`, but `add_to_additions_in_block` is called at line 223 **before** checking `result`. If `result` is `False` (bad proofs), the cache entry is still written. The next call for the same pair returns `True` at line 206 without issuing a new peer request or re-running validation.

`in_additions_in_block` and `add_to_additions_in_block` operate on a simple LRU keyed by `(header_hash, puzzle_hash)`: [3](#0-2) 

There is no stored validation result — presence in the cache is treated as proof of validity.

### Impact Explanation

In untrusted sync mode, the wallet relies on this function to confirm that a coin addition is cryptographically included in the block's additions Merkle root. If the cache is poisoned with a `False` result, the wallet will subsequently accept a fabricated coin addition as confirmed, corrupting coin records and wallet sync state (High impact per scope rules).

### Likelihood Explanation

Any peer the wallet connects to in untrusted mode can trigger this. The peer sends a `RespondAdditions` with invalid proofs. The first call returns `False` but poisons the cache. If the same peer (before disconnect) or any code path re-calls `request_and_validate_additions` for the same `(header_hash, puzzle_hash)`, it gets `True` back. Whether the peer is immediately disconnected on a `False` return depends on the caller in `wallet_node.py`, but the cache write is unconditional regardless.

### Recommendation

Condition the cache write on a successful validation result:

```python
result: bool = validate_additions(
    additions_res.coins,
    additions_res.proofs,
    additions_root,
)
if result:  # Only cache on success
    peer_request_cache.add_to_additions_in_block(header_hash, puzzle_hash, height)
return result
```

    def add_to_additions_in_block(self, header_hash: bytes32, addition_ph: bytes32, height: uint32) -> None:
        self._additions_in_block.put((header_hash, addition_ph), height)

    def in_additions_in_block(self, header_hash: bytes32, addition_ph: bytes32) -> bool:
        return self._additions_in_block.get((header_hash, addition_ph)) is not None
