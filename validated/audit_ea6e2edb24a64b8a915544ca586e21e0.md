Audit Report

## Title
Uninitialized `tokenFeeBps` Mapping Allows Zero-Fee Token Deposits - (File: contracts/pools/RSETHPool.sol)

## Summary

`addSupportedToken` never writes to `tokenFeeBps[token]`, so the mapping defaults to `0` for every newly added token. Any depositor calling `deposit(token, amount, referralId)` before an admin separately calls `setTokenFeeBps` pays zero protocol fees. Because there is no enforcement that `setTokenFeeBps` must precede the first deposit, the protocol collects no fee revenue on token deposits for an indefinite window — or permanently if the admin omits the call.

## Finding Description

`RSETHPool` maintains two fee variables: `feeBps` (ETH deposits, set atomically in `initialize()`) and `tokenFeeBps` (per-token ERC-20 deposits, a storage mapping). When a token is added via `addSupportedToken` (L637–656), only `supportedTokenOracle[token]` and `tokenBridge[token]` are written; `tokenFeeBps[token]` is never touched and remains `0`.

The fee calculation in `viewSwapRsETHAmountAndFee(uint256 amount, address token)` (L335–336) reads:

```solidity
uint256 feeBpsForToken = tokenFeeBps[token];   // always 0 for new tokens
fee = amount * feeBpsForToken / 10_000;         // always 0
```

`deposit(address token, ...)` (L298–300) then accumulates this zero fee:

```solidity
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount, token);
feeEarnedInToken[token] += fee;   // += 0
```

The depositor receives rsETH computed on the full `amount` with no deduction. `withdrawFees(receiver, token)` (L438–440) would transfer `feeEarnedInToken[token]` to the treasury, but that value is permanently `0`. The only remediation path — `setTokenFeeBps` (L583–594) — requires a separate `DEFAULT_ADMIN_ROLE` call with no on-chain enforcement that it precedes deposits.

Existing guards (`onlySupportedToken`, `nonReentrant`, `whenNotPaused`) do not check whether `tokenFeeBps[token]` has been initialized.

## Impact Explanation

**High — Theft of unclaimed yield.** The protocol's fee revenue on all ERC-20 token deposits is the yield at stake. Because `tokenFeeBps[token]` is `0` from the moment a token is added, every deposit in that window transfers the full token value to the depositor rather than deducting the intended fee. The `feeEarnedInToken[token]` accumulator that feeds `withdrawFees` never grows, so the treasury receives nothing. This matches the allowed impact "High. Theft of unclaimed yield" exactly: fee revenue the protocol is designed to earn is diverted to depositors due to a code defect.

## Likelihood Explanation

**High.** The entry point is the public `deposit(address token, uint256 amount, string referralId)` function, callable by any address holding a supported token. No privileges, front-running, or special setup are required beyond the normal state that exists immediately after `addSupportedToken` is called. The vulnerable window opens the moment `addSupportedToken` executes and closes only if and when an admin separately calls `setTokenFeeBps` — a call that has no on-chain enforcement. Any depositor who acts before that admin call pays zero fees.

## Recommendation

Initialize `tokenFeeBps[token]` atomically inside `addSupportedToken`, requiring an explicit fee parameter:

```diff
-function addSupportedToken(address token, address oracle, address bridge)
+function addSupportedToken(address token, address oracle, address bridge, uint256 _feeBps)
     external onlyRole(TIMELOCK_ROLE)
 {
     ...
+    if (_feeBps > 10_000) revert InvalidFeeAmount();
+    tokenFeeBps[token] = _feeBps;
     emit AddSupportedToken(token, oracle, bridge);
 }
```

Alternatively, add a guard in `viewSwapRsETHAmountAndFee` that reverts when `tokenFeeBps[token] == 0` if a zero fee is not a valid protocol state.

## Proof of Concept

1. Admin calls `addSupportedToken(wstETH, oracle, bridge)` — `tokenFeeBps[wstETH]` is `0` (never written).
2. User calls `deposit(wstETH, 10 ether, "ref")`.
3. `viewSwapRsETHAmountAndFee(10 ether, wstETH)` executes (L335–336):
   - `feeBpsForToken = tokenFeeBps[wstETH]` → `0`
   - `fee = 10 ether * 0 / 10_000` → `0`
4. `feeEarnedInToken[wstETH] += 0` — treasury accumulates nothing (L300).
5. User receives rsETH equivalent to the full `10 ether` with zero fee deducted (L302).
6. Admin never calls `setTokenFeeBps`; all subsequent deposits also pay zero fees.

**Foundry test sketch:**
```solidity
function test_zeroFeeBypass() public {
    vm.prank(timelock);
    pool.addSupportedToken(address(wstETH), oracle, bridge);
    // tokenFeeBps[wstETH] is 0 — no setTokenFeeBps called

    deal(address(wstETH), user, 10 ether);
    vm.startPrank(user);
    wstETH.approve(address(pool), 10 ether);
    pool.deposit(address(wstETH), 10 ether, "ref");
    vm.stopPrank();

    assertEq(pool.feeEarnedInToken(address(wstETH)), 0); // fee never collected
}
```
