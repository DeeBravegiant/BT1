### Title
Arbitrary `response.payload_hash` Accepted in `respond_verify_foreign_tx` Enables Cross-Request Signature Replay — (File: `crates/contract/src/lib.rs`)

### Summary

`respond_verify_foreign_tx` validates that `response.signature` is a valid threshold signature over `response.payload_hash`, but never validates that `response.payload_hash` is actually derived from the `request.request` field supplied in the same call. A single malicious attested MPC participant (strictly below the signing threshold) can replay a valid threshold signature produced for one foreign-chain request to resolve a completely different pending request, delivering a forged `VerifyForeignTransactionResponse` to the waiting caller.

### Finding Description

`respond_verify_foreign_tx` accepts two caller-supplied arguments:

- `request: VerifyForeignTransactionRequest` — used as the map key to locate and drain the pending yield.
- `response: VerifyForeignTransactionResponse` — contains `payload_hash` (a `Hash256`) and `signature`.

The only cryptographic check performed is:

```rust
// crates/contract/src/lib.rs  ~line 726-734
let payload_hash: [u8; 32] = response.payload_hash.0;
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,          // root public key, no tweak
)
.is_ok()
``` [1](#0-0) 

The contract confirms only that `response.signature` is a valid ECDSA signature over `response.payload_hash` under the domain's root public key. It never checks that `response.payload_hash` equals `SHA-256(borsh(ForeignTxSignPayload::V1 { request: request.request, values: … }))`.

The correct payload hash is defined as:

```
msg_hash = SHA-256(borsh(ForeignTxSignPayload))
```

where `ForeignTxSignPayload::V1` binds together `request.request` (the foreign-chain RPC query) and the `extracted_values` observed by the nodes. [2](#0-1) 

Because the contract never recomputes or checks this binding, the `request` and `response.payload_hash` fields are effectively decoupled. An attacker can supply any `request` that exists in `pending_verify_foreign_tx_requests` together with any `response` whose signature happens to be valid — including a signature produced for an entirely different foreign-chain request.

After the signature check passes, the contract unconditionally resolves the pending yield:

```rust
pending_requests::resolve_yields_for(
    &mut self.pending_verify_foreign_tx_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),   // response is never re-validated
)
``` [3](#0-2) 

The serialised `response` — including the mismatched `payload_hash` — is delivered verbatim to the waiting caller.

### Impact Explanation

A bridge or application contract that calls `verify_foreign_transaction` receives a `VerifyForeignTransactionResponse` whose `payload_hash` does not correspond to the transaction it submitted. If the downstream contract trusts the MPC contract's response without independently recomputing the expected hash (as the SDK helper `ForeignChainSignatureVerifier::verify_signature` does), it will act on a forged attestation — e.g., crediting a deposit that was never made, or releasing funds against a transaction that was never finalised. Even when the SDK verifier is used, the contract's own security invariant ("the signed hash always corresponds to the submitted request") is violated, breaking the trust model of the entire foreign-chain verification feature.

This maps to: **High — forged foreign-chain verification that causes invalid bridge execution**.

### Likelihood Explanation

The attack requires only a single malicious attested MPC participant (strictly below the signing threshold). No key-share forgery or threshold collusion is needed. The attacker:

1. Observes a valid threshold signature `sig_A` submitted on-chain for a previously completed request `req_A` (all on-chain data is public).
2. Waits for (or induces) a new pending request `req_B` in the contract.
3. Calls `respond_verify_foreign_tx(req_B, { payload_hash: hash_A, signature: sig_A })`.

Step 3 passes all contract checks: `sig_A` is a valid root-key signature over `hash_A`, and `req_B` exists in the pending map. The yield for `req_B` is resolved with `hash_A`.

Any of the `n` attested participants can call `respond_verify_foreign_tx`; the function does not restrict the caller to the designated leader for that request. [4](#0-3) 

### Recommendation

The contract must bind `response.payload_hash` to `request.request` before accepting the response. Two complementary approaches:

**Short-term:** Change `VerifyForeignTransactionResponse` to include the raw `extracted_values` alongside `payload_hash`. The contract then recomputes `SHA-256(borsh(ForeignTxSignPayload::V1 { request: request.request, values: response.extracted_values }))` and asserts it equals `response.payload_hash` before proceeding.

**Long-term:** Store a commitment to `request.request` in the pending-request map entry at submission time, and verify on response that the provided `payload_hash` is consistent with that commitment. Add a test that calls `respond_verify_foreign_tx` with a valid signature produced for a *different* request and asserts the call is rejected.

### Proof of Concept

```
1. Alice submits verify_foreign_transaction(req_A) → pending entry keyed by req_A.
2. MPC nodes process req_A; leader submits respond_verify_foreign_tx(req_A, resp_A)
   where resp_A.payload_hash = H(req_A.request, values_A).
   Contract accepts; Alice's yield is resolved. resp_A is now public on-chain.

3. Bob submits verify_foreign_transaction(req_B) → pending entry keyed by req_B.

4. Mallory (attested participant, single node) calls:
     respond_verify_foreign_tx(
         request  = req_B,          // valid pending key
         response = resp_A,         // payload_hash = H(req_A.request, values_A)
                                    // signature    = valid threshold sig over hash_A
     )

5. Contract checks:
   a. verify_ecdsa_signature(resp_A.signature, resp_A.payload_hash, root_pk) → OK
   b. resolve_yields_for(pending_map, req_B, serialize(resp_A))              → OK

6. Bob's callback receives resp_A.
   resp_A.payload_hash corresponds to req_A, not req_B.
   Bob's bridge contract, if it does not independently recompute the expected hash,
   processes the wrong foreign-chain attestation.

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }
