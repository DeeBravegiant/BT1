### Title
`verify_foreign_transaction` Fan-Out Queue Saturation Permanently Blocks Legitimate Requests — (File: `crates/contract/src/pending_requests.rs`, `crates/contract/src/lib.rs`)

### Summary

The `verify_foreign_transaction` request key (`VerifyForeignTransactionRequest`) contains no caller identity, unlike `sign()` whose key is bound to the caller via a derived tweak. Any unprivileged account can therefore submit the same foreign-tx request as a victim and saturate the 128-slot fan-out queue, causing the victim's subsequent submission to be permanently rejected. The attacker can re-saturate the queue immediately after each MPC drain, making the block indefinitely persistent at near-zero cost.

### Finding Description

`pending_requests::push_pending_yield` enforces a hard cap of `MAX_PENDING_REQUEST_FAN_OUT = 128` on the number of concurrent yield-resume promises queued per request key:

```rust
// crates/contract/src/pending_requests.rs
if queue.len() >= usize::from(MAX_PENDING_REQUEST_FAN_OUT) {
    env::panic_str(
        &RequestError::PendingRequestQueueFull { limit: MAX_PENDING_REQUEST_FAN_OUT }.to_string(),
    );
}
```

For `sign()`, the map key is `SignatureRequest = { domain_id, payload, tweak }` where `tweak = derive_tweak(predecessor_id, path)`. Because the tweak is derived from the caller's account ID, two different callers with the same payload and path produce **different** map keys and therefore **independent** queues — cross-user saturation is structurally impossible.

For `verify_foreign_transaction()`, the map key is `VerifyForeignTransactionRequest = { request: ForeignChainRpcRequest, domain_id, payload_version }` — **no caller identity is included**:

```rust
// crates/near-mpc-contract-interface/src/types/foreign_chain.rs
pub struct VerifyForeignTransactionRequest {
    pub request: ForeignChainRpcRequest,
    pub domain_id: DomainId,
    pub payload_version: ForeignTxPayloadVersion,
}
```

Any caller can submit an identical `VerifyForeignTransactionRequest` and share the same queue slot as the victim. The attack flow:

1. Attacker monitors the foreign chain (e.g., Ethereum mempool) for a bridge transaction the victim intends to verify.
2. Attacker submits 128 identical `verify_foreign_transaction` calls for that specific `{tx_id, domain_id, payload_version}` tuple, saturating the queue.
3. Victim's call panics with `PendingRequestQueueFull`.
4. MPC nodes respond and drain the queue; attacker immediately re-saturates it.
5. Victim is permanently blocked from obtaining the signature for that foreign transaction.

The attacker also receives the MPC-produced signature via their 128 resolved yield promises. Depending on the bridge contract's design, this signature could be used to claim the victim's bridged funds.

### Impact Explanation

The request-lifecycle invariant — that any caller who submits a valid `verify_foreign_transaction` request with sufficient deposit and gas will eventually receive the MPC signature — is broken. The victim cannot complete their cross-chain bridge operation. If the downstream bridge contract does not bind the signature to the original submitter's identity, the attacker who holds the signature can additionally claim the victim's bridged funds, escalating the impact to Critical (theft of funds). At minimum this is a Medium-severity request-lifecycle invariant violation.

### Likelihood Explanation

The attacker needs only to observe the target foreign-chain transaction (publicly visible on-chain) and submit 128 calls at 1 yoctoNEAR each — a total cost of 128 yoctoNEAR plus gas, which is negligible. The queue refill can be automated and repeated indefinitely. No privileged access, key material, or threshold collusion is required. The attack is fully reachable by any unprivileged NEAR account.

### Recommendation

Bind the `VerifyForeignTransactionRequest` map key to the caller's identity, mirroring the `sign()` design. The simplest fix is to include a `tweak` derived from `(predecessor_id, derivation_path)` in `VerifyForeignTransactionRequest`, so that different callers produce different map keys and cannot interfere with each other's queues. Alternatively, add a `predecessor_id: AccountId` field directly to the request struct and populate it from `env::predecessor_account_id()` inside `verify_foreign_transaction()`.

### Proof of Concept

```
// Victim submits first (or attacker pre-fills before victim):
Alice calls verify_foreign_transaction({ Ethereum(tx_id=[0xAB;32]), domain_id:0, V1 })
  → queue["(Ethereum(0xAB..),0,V1)"] = [alice_yield]

// Attacker fills remaining 127 slots:
Attacker calls verify_foreign_transaction({ Ethereum(tx_id=[0xAB;32]), domain_id:0, V1 }) × 127
  → queue["(Ethereum(0xAB..),0,V1)"] = [alice_yield, atk_yield×127]  // full

// MPC nodes respond → queue drained, attacker receives 127 copies of the signature
// Attacker immediately re-fills:
Attacker calls verify_foreign_transaction({ Ethereum(tx_id=[0xAB;32]), domain_id:0, V1 }) × 128
  → queue full again

// Alice retries:
Alice calls verify_foreign_transaction({ Ethereum(tx_id=[0xAB;32]), domain_id:0, V1 })
  → PANIC: PendingRequestQueueFull { limit: 128 }
```

```rust
/// Maximum number of concurrent yield-resume promises that can be queued for a single
/// request key (i.e. the number of duplicate submissions whose responses fan out from
/// one MPC reply).
///
/// The ceiling is needed because `respond*` drains the entire queue in one call: every
/// queued yield triggers a host-side `promise_yield_resume`, paid for out of the
/// responder's 300 TGas budget. Without a cap, an attacker could enqueue enough
/// duplicates to make `respond*` run out of gas and strand every queued caller.
///
/// 128 is validated empirically by the sandbox test
/// `test_contract_request_duplicate_requests_fan_out`, which fills the queue to this
/// cap across all four signature schemes and confirms `respond*` drains it inside its
/// 300 TGas budget.
pub const MAX_PENDING_REQUEST_FAN_OUT: u8 = 128;
```

```rust
pub(crate) fn push_pending_yield<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: K,
    data_id: CryptoHash,
) where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let queue = requests.entry(request).or_default();
    if queue.len() >= usize::from(MAX_PENDING_REQUEST_FAN_OUT) {
        env::panic_str(
            &RequestError::PendingRequestQueueFull {
                limit: MAX_PENDING_REQUEST_FAN_OUT,
            }
            .to_string(),
        );
    }
    queue.push(YieldIndex { data_id });
```

```rust
        log!(
            "verify_foreign_transaction: predecessor={:?}, request={:?}",

            request
        );

        self.check_request_preconditions(
            request.domain_id,
            DomainPurpose::ForeignTx,

            MINIMUM_SIGN_REQUEST_DEPOSIT,
        );

        let requested_chain = request.request.chain();
        let supported_chains = self.get_supported_foreign_chains();
        if !supported_chains.contains(&requested_chain) {
            env::panic_str(
                &InvalidParameters::ForeignChainNotSupported {
                    requested: requested_chain,
                }
                .to_string(),
            );
        }

        let callback_gas = Gas::from_tgas(
            self.config
                .return_signature_and_clean_state_on_success_call_tera_gas,
        );

        let request = args_into_verify_foreign_tx_request(request);
        let callback_args = serde_json::to_vec(&(&request,)).unwrap();
        self.enqueue_yield_request(
            method_names::RETURN_VERIFY_FOREIGN_TX_AND_CLEAN_STATE_ON_SUCCESS,
            callback_args,
            callback_gas,
            move |this, id| this.add_verify_foreign_tx_request(request, id),
        );
    }
```
