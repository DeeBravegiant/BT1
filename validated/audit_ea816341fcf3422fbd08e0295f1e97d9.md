### Title
`msg.value` Fully Forwarded to Wormhole After Native-ETH Already Sent to Recipient — (`OmniBridgeWormhole.sol`)

---

### Summary

`OmniBridgeWormhole.finTransferExtension` forwards the raw `msg.value` to `_wormhole.publishMessage`. When the bridged token is native ETH (`tokenAddress == address(0)`), the parent `finTransfer` has already spent `payload.amount` of that same `msg.value` sending ETH to the recipient. The contract then has only `msg.value − payload.amount` remaining, but tries to forward the full `msg.value` to Wormhole. The call always reverts, permanently blocking finalization of every native-ETH cross-chain transfer.

---

### Finding Description

`OmniBridge.finTransfer` is `payable`. For native-ETH payloads it first sends `payload.amount` ETH to the recipient:

```solidity
// OmniBridge.sol L317-L322
if (payload.tokenAddress == address(0)) {
    (bool success, ) = payload.recipient.call{value: payload.amount}("");
    if (!success) revert FailedToSendEther();
}
``` [1](#0-0) 

It then calls `finTransferExtension(payload)`:

```solidity
// OmniBridge.sol L357
finTransferExtension(payload);
``` [2](#0-1) 

`OmniBridgeWormhole.finTransferExtension` blindly forwards `msg.value` — the **original** call value, not the remaining balance — to Wormhole:

```solidity
// OmniBridgeWormhole.sol L109
_wormhole.publishMessage{value: msg.value}(
    wormholeNonce,
    messagePayload,
    _consistencyLevel
);
``` [3](#0-2) 

At the point `finTransferExtension` executes, the contract's ETH balance from this call is `msg.value − payload.amount`. Attempting to forward `msg.value` exceeds the available balance and the EVM reverts.

No value of `msg.value` supplied by the relayer can satisfy both constraints simultaneously:
- `msg.value ≥ payload.amount` (to pay the recipient)
- `msg.value` available for Wormhole fee (already reduced by `payload.amount`)

The `TransferMessagePayload` struct contains no separate Wormhole-fee field, so there is no in-band way to communicate the split:

```solidity
// BridgeTypes.sol L5-L14
struct TransferMessagePayload {
    uint64 destinationNonce;
    uint8 originChain;
    uint64 originNonce;
    address tokenAddress;
    uint128 amount;
    address recipient;
    string feeRecipient;
    bytes message;
}
``` [4](#0-3) 

Contrast with `initTransferExtension`, which correctly receives a pre-computed `value` parameter (already stripped of `amount` and `nativeFee`) and forwards only that:

```solidity
// OmniBridgeWormhole.sol L143
_wormhole.publishMessage{value: value}(...)
``` [5](#0-4) 

`finTransferExtension` receives no such pre-computed remainder; it only has access to `msg.value`.

---

### Impact Explanation

Every attempt to finalize a native-ETH transfer through `OmniBridgeWormhole` reverts. Because the nonce is set before the revert (`completedTransfers[payload.destinationNonce] = true` at line 287 is also reverted), the nonce is not consumed — but the finalization can **never** succeed regardless of how many times it is retried or how much ETH the relayer sends. Users who initiated a native-ETH bridge from NEAR to EVM via the Wormhole path receive nothing on the EVM side; their locked value on NEAR is permanently unclaimable. This matches the **Critical / Irreversible fund lock, frozen redemption path** impact category. [6](#0-5) 

---

### Likelihood Explanation

The failure is deterministic and 100% reproducible for every native-ETH `finTransfer` call on the Wormhole deployment. No special attacker action is required; the bug triggers on any legitimate relayer attempt to settle a native-ETH transfer. The `initTransfer` path for native ETH (`tokenAddress == address(0)`) is fully functional, so users can initiate transfers that can never be completed. [7](#0-6) 

---

### Recommendation

Pass the Wormhole fee as a separate parameter through `finTransferExtension`, analogous to how `initTransferExtension` receives `value`. Compute the remainder before the ETH transfer:

```solidity
// In OmniBridge.finTransfer:
uint256 wormholeFee = msg.value;
if (payload.tokenAddress == address(0)) {
    wormholeFee = msg.value - payload.amount; // subtract what goes to recipient
    (bool success, ) = payload.recipient.call{value: payload.amount}("");
    if (!success) revert FailedToSendEther();
}
// ... other token branches ...
finTransferExtension(payload, wormholeFee);
```

Then in `OmniBridgeWormhole.finTransferExtension`, forward only `wormholeFee` to `publishMessage`.

---

### Proof of Concept

1. User bridges 1 ETH from NEAR → EVM via `OmniBridgeWormhole`. NEAR side locks funds and emits a signed `TransferMessagePayload` with `tokenAddress = address(0)`, `amount = 1 ETH`.
2. Relayer calls `finTransfer` with `msg.value = 1 ETH + wormhole_fee` (e.g., `1.001 ETH`).
3. `OmniBridge.finTransfer` sends `1 ETH` to `payload.recipient`. Contract now holds `0.001 ETH`.
4. `OmniBridgeWormhole.finTransferExtension` calls `_wormhole.publishMessage{value: 1.001 ETH}(...)`. Contract only has `0.001 ETH`. EVM reverts with insufficient balance.
5. Entire transaction reverts. Recipient receives nothing. Relayer loses gas. Retry with any `msg.value` produces the same outcome. User's ETH is permanently unclaimable on EVM. [8](#0-7)
