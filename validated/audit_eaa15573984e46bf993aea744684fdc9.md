### Title
Attached NEAR Deposit Permanently Locked in `submit_participant_info` When Existing Participant Re-submits Attestation - (File: `crates/contract/src/lib.rs`)

### Summary

`submit_participant_info` is marked `#[payable]` and accepts an attached NEAR deposit, but the deposit-refund logic is gated behind a conditional that is `false` for existing participants performing a re-submission. Any NEAR attached in that code path is silently consumed by the contract and permanently locked.

### Finding Description

In `submit_participant_info`, after calling `tee_state.add_participant(...)`, the contract computes:

```rust
let attestation_storage_must_be_paid_by_caller =
    is_new_attestation || caller_is_not_participant;
``` [1](#0-0) 

The deposit-check-and-refund block is only entered when this flag is `true`:

```rust
if attestation_storage_must_be_paid_by_caller {
    // ... check attached >= cost, refund excess
}
``` [2](#0-1) 

When the flag is `false` — i.e., when the caller **is** an existing participant **and** the insertion result is `UpdatedExistingParticipant` — the function returns `Ok(())` without ever reading `env::attached_deposit()` or scheduling a refund. Because the function is `#[payable]`, the NEAR runtime accepts any deposit unconditionally; the contract simply never returns it.

`add_participant` returns `UpdatedExistingParticipant` whenever the TLS key already exists in `stored_attestations` and the submitting account matches the stored owner:

```rust
Ok(match insertion {
    Some(_previous_attestation) => ParticipantInsertion::UpdatedExistingParticipant,
    None => ParticipantInsertion::NewlyInsertedParticipant,
})
``` [3](#0-2) 

This is the normal path for periodic attestation renewal (TEE upgrade, key rotation, quote refresh). The `ParticipantInsertion` enum explicitly models this case: [4](#0-3) 

### Impact Explanation

Any NEAR tokens attached to a `submit_participant_info` call by an existing participant performing a re-submission are permanently locked inside the MPC contract. There is no withdrawal mechanism, no admin sweep, and no subsequent code path that would return them. This directly breaks the balance/accounting invariant: the contract silently absorbs funds it has no right to retain.

This maps to the **Medium** allowed impact: *"Balance, request-lifecycle, participant-state, or contract execution-flow manipulation that breaks production safety/accounting invariants."*

### Likelihood Explanation

The documentation explicitly instructs node operators to call `submit_participant_info` manually with `--deposit` for first-time joins:

> *"Attached deposit is lower than required. Attached: X, required: Y — first-time joiners must attach enough yoctoNEAR for storage; the node attaches 0, so call `submit_participant_info` manually with `--deposit` once."* [5](#0-4) 

An operator who follows this guidance and later re-runs the same manual command (e.g., after a TEE image upgrade or key rotation) will be in the `UpdatedExistingParticipant` branch. Their deposit is locked. Additionally, any tooling or script that unconditionally attaches a deposit on every submission will silently drain funds on every renewal cycle.

### Recommendation

Add an explicit zero-deposit assertion in the `else` branch (when `attestation_storage_must_be_paid_by_caller` is `false`), mirroring the NestedFactory fix pattern:

```rust
if attestation_storage_must_be_paid_by_caller {
    // existing deposit check + refund logic
} else {
    let attached = env::attached_deposit();
    if attached > NearToken::from_yoctonear(0) {
        // Either panic or silently refund:
        return Err(InvalidParameters::InsufficientDeposit {
            attached: attached.as_yoctonear(),
            required: 0,
        }.into());
        // Or: Promise::new(account_id).transfer(attached).detach();
    }
}
```

This ensures no deposit is silently consumed when storage payment is not required.

### Proof of Concept

1. Participant `alice.near` calls `submit_participant_info` with a valid attestation and `--deposit 1 NEAR` for the first time → `NewlyInsertedParticipant`, deposit is checked and excess refunded correctly.
2. After a TEE image upgrade, `alice.near` calls `submit_participant_info` again with the same `--deposit 1 NEAR` flag (following the documented manual procedure).
3. `add_participant` returns `UpdatedExistingParticipant` because the TLS key already exists and the account matches. [6](#0-5) 
4. `is_new_attestation` is `false`; `caller_is_not_participant` is `false` (alice is a participant); therefore `attestation_storage_must_be_paid_by_caller` is `false`. [7](#0-6) 
5. The deposit-handling block is skipped entirely. The function returns `Ok(())`. [8](#0-7) 
6. Alice's 1 NEAR is permanently locked in the MPC contract with no recovery path.
