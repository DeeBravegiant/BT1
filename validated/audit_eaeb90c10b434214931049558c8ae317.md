### Title
PTQ `add_chunk_transactions` Incorrectly Attributes `WithdrawFromGasKey` Amounts to Target Gas Key, Enabling Temporary RPC DoS of Victim Gas Key — (`chain/client/src/pending_transaction_queue.rs`)

### Summary

In `add_chunk_transactions`, when scanning a transaction's actions for `WithdrawFromGasKey`, the PTQ indexes the withdrawal amount under `(signer_id, withdraw.public_key)`. Because `signer_id` is the account that owns both gas keys, a transaction signed by gas key A containing `WithdrawFromGasKey { public_key: B_pubkey, amount: large }` causes the PTQ to record a large pending cost against gas key B. Subsequent RPC submissions of valid gas key B transactions are then rejected with `NotEnoughGasKeyBalance` until the block is certified and the entry is removed — even though the `WithdrawFromGasKey` action will fail at execution time (insufficient balance).

### Finding Description

In `add_chunk_transactions`, the PTQ scans every included transaction's actions:

```rust
// Scan actions for WithdrawFromGasKey (affects gas key balance).
for action in tx.actions() {
    if let Action::WithdrawFromGasKey(withdraw) = action {
        let gas_key_entry = chunk_data
            .gas_key_costs
            .entry((signer_id.clone(), withdraw.public_key.clone()))
            .or_insert(Balance::ZERO);
        *gas_key_entry = gas_key_entry.saturating_add(withdraw.amount);
    }
}
``` [1](#0-0) 

The key used is `(signer_id, withdraw.public_key)`. When `query_pending_state` is later called for a gas key B transaction, it looks up `(account_id, B_pubkey)`:

```rust
let gas_key = (signer_id.clone(), public_key.clone());
let pending_gas_key_cost =
    self.pending_gas_key_costs.get(&gas_key).copied().unwrap_or(Balance::ZERO);
``` [2](#0-1) 

If the attacker signs a transaction with gas key A (same account) and includes `WithdrawFromGasKey { public_key: B_pubkey, amount: u128::MAX }`, the PTQ stores `u128::MAX` under `(account_id, B_pubkey)`. This is the exact key that gas key B's transactions look up. There is **no validation** at inclusion time that the withdrawal amount is within gas key B's actual balance:

```rust
fn validate_withdraw_from_gas_key_action(
    current_protocol_version: ProtocolVersion,
) -> Result<(), ActionsValidationError> {
    require_protocol_feature(ProtocolFeature::GasKeys, "GasKeys", current_protocol_version)?;
    Ok(())
}
``` [3](#0-2) 

The balance check only happens at execution time in `action_withdraw_from_gas_key`:

```rust
let Some(updated_balance) = gas_key_info.balance.checked_sub(action.amount) else {
    result.result = Err(ActionErrorKind::InsufficientGasKeyBalance { ... });
    return Ok(());
};
``` [4](#0-3) 

So the action fails at execution time, but the PTQ has already inflated gas key B's pending cost. `verify_and_charge_gas_key_tx_ephemeral` then rejects gas key B's RPC submissions:

```rust
let Some(available_gas_key_balance) =
    gas_key_info.balance.checked_sub(pending.paid_from_gas_key)
else {
    ...
    return TxVerdict::Failed(InvalidTxError::NotEnoughGasKeyBalance { ... });
};
if available_gas_key_balance < gas_cost {
    return TxVerdict::Failed(InvalidTxError::NotEnoughGasKeyBalance { ... });
}
``` [5](#0-4) 

### Impact Explanation

Any holder of a key on an account (access key or gas key A) can temporarily prevent all RPC-submitted gas key B transactions on the same account from being accepted, for the duration between block inclusion and certification. The `WithdrawFromGasKey` action will fail at execution time (leaving gas key B's actual balance intact), but the PTQ-based RPC rejection is real and observable. This violates the invariant that PTQ pending gas key cost tracking must only account for costs attributable to transactions that will actually succeed.

### Likelihood Explanation

The attack requires the attacker to hold any key on the same account as gas key B. This is realistic in multi-party account setups (e.g., a user account with a service-controlled gas key). The attack is cheap (one transaction), repeatable (re-submit before certification), and requires no validator privileges.

### Recommendation

In `add_chunk_transactions`, `WithdrawFromGasKey` amounts should be tracked under `(receiver_id, withdraw.public_key)` rather than `(signer_id, withdraw.public_key)`, since `WithdrawFromGasKey` operates on the **receiver's** gas keys. Additionally, the PTQ should only record a `WithdrawFromGasKey` amount if the withdrawal is plausibly within the gas key's known balance (or accept that this is a best-effort check and document the limitation).

### Proof of Concept

1. Account `alice.near` has gas key A (attacker-controlled) and gas key B (service-controlled), each funded with 1 milliNEAR.
2. Attacker submits a tx signed by gas key A with `WithdrawFromGasKey { public_key: B_pubkey, amount: u128::MAX }`.
3. Block producer includes the tx (no balance check at inclusion).
4. `add_chunk_transactions` stores `u128::MAX` in `pending_gas_key_costs[(alice.near, B_pubkey)]`.
5. Service submits a valid gas key B tx via RPC.
6. `query_pending_state` returns `pending_gas_key_cost = u128::MAX` for gas key B.
7. `verify_and_charge_gas_key_tx_ephemeral` computes `available = B_balance - u128::MAX` → underflow → `NotEnoughGasKeyBalance`.
8. RPC rejects the valid gas key B transaction.
9. When the block is certified, `action_withdraw_from_gas_key` fails (`InsufficientGasKeyBalance`), gas key B's balance is unchanged, and the PTQ entry is removed — but the DoS window was real. [6](#0-5)
