### Title
Gateway L2 Gas Price Admission Check Uses Stale Previous-Block Price Instead of Next-Block EIP-1559 Price — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `validate_resource_bounds` function in the gateway's stateful transaction validator checks a transaction's `max_l2_gas_price` against the **previous committed block's** L2 gas price. However, the transaction will be executed in the **next** block, whose L2 gas price is deterministically computed via EIP-1559 (`calculate_next_base_gas_price`) from the previous block's gas usage. This creates a systematic mismatch: during high load the gateway under-rejects (admits transactions that will fail at execution), and during low load it over-rejects (rejects transactions that would succeed). The code itself acknowledges the gap with a `TODO` comment at the exact site.

---

### Finding Description

**Root cause — `validate_resource_bounds`:**

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs, lines 228-240
if self.config.validate_resource_bounds {
    // TODO(Arni): getnext_l2_gas_price from the block header.
    let previous_block_l2_gas_price = self
        .gateway_fixed_block_state_reader
        .get_block_info()
        .await?
        .gas_prices
        .strk_gas_prices
        .l2_gas_price;
    self.validate_tx_l2_gas_price_within_threshold(
        executable_tx.resource_bounds(),
        previous_block_l2_gas_price,
    )?;
}
``` [1](#0-0) 

The `GatewayFixedBlockSyncStateClient` is constructed once per validator instantiation, pinned to the latest committed block number, and its `block_info_cache` is a `OnceCell` that never refreshes. [2](#0-1) 

The next block's L2 gas price is **not** the same as the previous block's. It is computed by `calculate_next_base_gas_price` using the previous block's actual gas usage:

```rust
// crates/apollo_consensus_orchestrator/src/fee_market/mod.rs, lines 86-139
pub fn calculate_next_base_gas_price(
    price: GasPrice,
    gas_used: GasAmount,
    gas_target: GasAmount,
    min_gas_price: GasPrice,
) -> GasPrice { ... }
``` [3](#0-2) 

This value is computed and stored in `SequencerConsensusContext::l2_gas_price` after every `finalize_decision` call via `update_l2_gas_price`: [4](#0-3) 

The next-block L2 gas price is therefore **already known** at gateway admission time (it was computed when the previous block was finalized), but the gateway reads the previous block's price from storage instead.

**The threshold check:**

```rust
// lines 366-383
let tx_l2_gas_price = tx_resource_bounds.l2_gas.max_price_per_unit;
let threshold = (gas_price_threshold_multiplier
    * previous_block_l2_gas_price.get().0)
    .to_integer();
if tx_l2_gas_price.0 < threshold {
    return Err(...GAS_PRICE_TOO_LOW...);
}
``` [5](#0-4) 

The threshold is `min_gas_price_percentage% × prev_block_price`. The correct threshold is `min_gas_price_percentage% × next_block_price`.

**Concrete price divergence (from test data):**

| Scenario | prev price | next price | delta |
|---|---|---|---|
| 75% full block | 30,000,000,000 | 30,312,500,000 | +1.04% |
| 25% full block | 30,000,000,000 | 29,687,500,000 | −1.04% | [6](#0-5) 

During sustained high load the price drifts further each block, widening the gap.

---

### Impact Explanation

**High — Mempool/gateway admission accepts invalid transactions or rejects valid transactions before sequencing.**

**Path A — High load (previous block above gas target):**
`next_block_price > prev_block_price`. The gateway threshold is anchored to `prev_block_price`, so it is too low. A transaction with `max_l2_gas_price` in the range `[threshold(prev), threshold(next))` passes gateway admission, enters the mempool, gets sequenced, and then **fails at execution** because its gas price is below the next block's actual price. The sequencer wastes execution resources on a transaction that was never viable.

**Path B — Low load (previous block below gas target):**
`next_block_price < prev_block_price`. The gateway threshold is too high. A transaction with `max_l2_gas_price` in the range `[threshold(next), threshold(prev))` is **rejected at gateway** even though it would succeed at execution. Valid user transactions are silently dropped.

---

### Likelihood Explanation

The condition is triggered on every block where gas usage deviates from the target — which is the normal operating state of any active network. No special privileges or adversarial setup are required; any user submitting a transaction with a gas price in the gap between the two thresholds triggers the mismatch. The `TODO` comment at the exact site confirms the developers are aware the wrong value is being used.

---

### Recommendation

Replace `previous_block_l2_gas_price` with the next block's computed L2 gas price. The `SequencerConsensusContext` already maintains this value in `self.l2_gas_price` after each `finalize_decision`. It should be exposed (e.g., via the state-sync block header field `next_l2_gas_price` that already exists in `BlockHeaderWithoutHash`) so the gateway can read it from the latest committed block header instead of recomputing the previous block's price. [7](#0-6) 

---

### Proof of Concept

1. Observe that the previous block was above the gas target (e.g., 75% full). From the fee-market test, `prev_price = 30_000_000_000`, `next_price = 30_312_500_000`.
2. Set `min_gas_price_percentage = 100` (threshold = 100% of reference price).
3. Submit a transaction with `max_l2_gas_price = 30_100_000_000` (between `prev_price` and `next_price`).
4. Gateway computes `threshold = 100% × 30_000_000_000 = 30_000_000_000`. Transaction passes (`30_100_000_000 ≥ 30_000_000_000`).
5. Transaction enters mempool and is sequenced into the next block.
6. At execution time the block's L2 gas price is `30_312_500_000`. The transaction's `max_l2_gas_price = 30_100_000_000 < 30_312_500_000` → execution fails.

The gateway admitted a transaction that was guaranteed to fail at execution, because it checked against the stale previous-block price rather than the already-computed next-block price.

```rust
pub fn calculate_next_base_gas_price(
    price: GasPrice,
    gas_used: GasAmount,
    gas_target: GasAmount,
    min_gas_price: GasPrice,
) -> GasPrice {
    let versioned_constants = VersionedConstants::latest_constants();
    assert!(
        gas_target < versioned_constants.max_block_size,
        "Gas target must be lower than max block size."
    );
    assert!(gas_target.0 > 0, "Gas target must be greater than zero.");
    assert!(
        versioned_constants.gas_price_max_change_denominator > 0,
        "Denominator constant must be greater than zero."
    );

    // If the current price is below the minimum, apply a gradual adjustment and return early.
    // This allows the price to increase by at most 1/MIN_GAS_PRICE_INCREASE_DENOMINATOR per block.
    if price < min_gas_price {
        let max_increase = price.0 / MIN_GAS_PRICE_INCREASE_DENOMINATOR;
        let adjusted = price.0 + max_increase;
        // Cap at min_gas_price to avoid overshooting
        let adjusted_price = adjusted.min(min_gas_price.0);
        info!(
            "Fee Market: Price {} below minimum gas price {}, adjusted price: {} )",
            price.0, min_gas_price.0, adjusted_price
        );
        return GasPrice(adjusted_price);
    }

    // Use U256 to avoid overflow, as multiplying a u128 by a u64 remains within U256 bounds.
    let gas_delta = U256::from(gas_used.0.abs_diff(gas_target.0));
    let gas_target_u256 = U256::from(gas_target.0);
    let price_u256 = U256::from(price.0);

    // Calculate price change by multiplying first, then dividing. This avoids the precision loss
    // that occurs when dividing before multiplying.
    let denominator =
        gas_target_u256 * U256::from(versioned_constants.gas_price_max_change_denominator);
    let price_change = (price_u256 * gas_delta) / denominator;

    let adjusted_price_u256 =
        if gas_used > gas_target { price_u256 + price_change } else { price_u256 - price_change };

    // Sanity check: ensure direction of change is correct
    assert!(
        gas_used > gas_target && adjusted_price_u256 >= price_u256
            || gas_used <= gas_target && adjusted_price_u256 <= price_u256
    );

    // Price should not realistically exceed u128::MAX, bound to avoid theoretical overflow.
    let adjusted_price = u128::try_from(adjusted_price_u256).unwrap_or(u128::MAX);
    GasPrice(max(adjusted_price, min_gas_price.0))
```
