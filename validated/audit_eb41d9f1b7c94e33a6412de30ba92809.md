Audit Report

## Title
JIT Liquidity Sandwich Steals LP Fees From Existing Bin Providers ‚Äî (`metric-core/contracts/libraries/LiquidityLib.sol`, `metric-core/contracts/libraries/SwapMath.sol`)

## Summary
Every swap deposits the LP fee portion directly into the bin's token balance via `SwapMath` (e.g., `buyToken0InBinSpecifiedOut` L414‚Äì415). `LiquidityLib.removeLiquidity` distributes the full current bin balance pro-rata to all current shareholders (L205‚Äì206). Because `addLiquidity` and `removeLiquidity` impose no lockup, cooldown, or withdrawal fee, an MEV actor can sandwich any large swap: add shares before the swap, let the swap inflate the bin balance with LP fees, then immediately remove shares and extract a proportional cut of fees that should have accrued entirely to pre-existing LPs.

## Finding Description
**Fee accumulation path:** In all four `SwapMath` swap-in-bin functions, the LP fee (total fee minus protocol cut) is added directly to the bin's token balance. For example in `buyToken0InBinSpecifiedOut`:

```solidity
// SwapMath.sol L409-415
uint256 feeAmountScaled = Math.ceilDiv(amountInScaled * currBinBuyFeeX64, ONE_X64);
amountInScaled += feeAmountScaled;
uint256 protocolFeeAmountScaled = (feeAmountScaled * spreadFeeE6) / 1e6;
binState.token0BalanceScaled -= amountOutScaled.toUint104();
binState.token1BalanceScaled =
  (uint256(binState.token1BalanceScaled) + amountInScaled - protocolFeeAmountScaled).toUint104();
```

The same pattern holds in `buyToken1InBinSpecifiedOut` (L495‚Äì501), `buyToken0InBinSpecifiedIn` (L639‚Äì641), and `buyToken1InBinSpecifiedIn` (L779‚Äì781). The LP fee is permanently merged into the bin balance with no separate accumulator.

**Pro-rata removal:** `LiquidityLib.removeLiquidity` computes each token's payout as a simple fraction of the current bin balance:

```solidity
// LiquidityLib.sol L205-206
uint256 amount0Scaled = _checkedMul(binState.token0BalanceScaled, sharesToRemove) / binTotalSharesVal;
uint256 amount1Scaled = _checkedMul(binState.token1BalanceScaled, sharesToRemove) / binTotalSharesVal;
```

Any LP fee that accumulated after a new LP joined is therefore shared with that new LP.

**No lockup or removal fee:** `MetricOmmPool.addLiquidity` and `removeLiquidity` (L182‚Äì212) impose only: (a) `nonReentrant` (prevents same-transaction reentrancy, not cross-transaction sandwiching), (b) `minimalMintableLiquidity` dust floor (a share count floor, not a timing restriction), and (c) optional extension hooks (`_beforeAddLiquidity`, `_afterAddLiquidity`, `_beforeRemoveLiquidity`, `_afterRemoveLiquidity`) that are no-ops when no extensions are configured. The base pool deploys with no extensions.

**Proportional add pricing:** When adding to an existing bin, `LiquidityLib.addLiquidity` (L108‚Äì110) charges the attacker proportional to the *current* (pre-fee) bin balance:

```solidity
amount0Scaled = Math.ceilDiv(_checkedMul(binState.token0BalanceScaled, sharesToAdd), binTotalSharesVal);
amount1Scaled = Math.ceilDiv(_checkedMul(binState.token1BalanceScaled, sharesToAdd), binTotalSharesVal);
```

After the swap inflates the bin balance with LP fees, the attacker's shares represent a claim on the inflated balance, yielding a risk-free profit equal to `lpFee √ó attackerShares / totalSharesAfterAdd`.

**Exploit flow:**
1. Attacker observes a large pending swap targeting bin B.
2. Attacker front-runs: calls `addLiquidity` for bin B, paying tokens proportional to the current (pre-fee) bin balance.
3. Swap executes: LP fee is deposited into bin B's balance.
4. Attacker back-runs: calls `removeLiquidity`, receiving `shares / totalShares √ó (originalBalance + lpFee)`.

The attacker paid for a share of `originalBalance` but receives a share of `originalBalance + lpFee`.

## Impact Explanation
Existing LPs suffer a direct, quantifiable loss of earned LP fees on every sandwiched swap. The attacker's profit equals `lpFee √ó attackerShares / totalSharesAfterAdd`. For a bin with 1,000 shares and an attacker who adds 1,000 shares, the attacker captures exactly half the LP fee generated by the swap. The original LP's fee income is halved with no recourse. Repeated across many swaps or bins, this constitutes a sustained drain of LP revenue ‚Äî a direct loss of owed LP assets above Sherlock thresholds.

## Likelihood Explanation
The attack is viable for any MEV searcher with mempool visibility. The oracle-mediated pricing eliminates price risk: the swap executes at the oracle mid-price regardless of bin state, so the attacker's round-trip (add ‚Üí swap ‚Üí remove) carries no directional exposure. The only cost is gas. For swaps generating LP fees above the gas threshold the attack is unconditionally profitable and can be automated. No special privileges are required ‚Äî `addLiquidity` is open to any caller and `removeLiquidity` only requires `msg.sender == owner`.

## Recommendation
Implement at least one of the following mitigations:

1. **Fee-checkpoint accounting (most robust):** Track a per-bin `feeGrowthGlobal` accumulator (similar to Uniswap v3's `feeGrowthInside`). Credit LP fees only to shares that existed *before* the swap, so newly minted shares start with a zero fee checkpoint and cannot claim historical fees. This eliminates the attack entirely without restricting liquidity timing.

2. **Minimum holding period:** Record the block number at which each position's shares were last increased; disallow `removeLiquidity` within N blocks of the most recent `addLiquidity` for that position.

3. **Withdrawal fee on short-term liquidity:** Charge a fee proportional to the LP fee earned if liquidity is removed within a configurable window.

## Proof of Concept
**Setup:** Bin 4 (above current price, token0-only): `token0BalanceScaled = 1,000`, `binTotalShares = 1,000` (Alice holds all shares). Oracle price: 1 token0 = 1 token1. LP fee rate: 2% total; protocol takes 50%, so LP net fee = 1%.

**Step 1 ‚Äî Attacker front-runs:** Bob calls `addLiquidity` for bin 4 with 1,000 shares. Payment (proportional per `LiquidityLib.sol` L109): `ceil(1,000 √ó 1,000 / 1,000) = 1,000 token0`. Bin state: `token0 = 2,000`, `shares = 2,000`.

**Step 2 ‚Äî Swap executes:** A trader buys 200 token0 from bin 4, paying token1. Per `SwapMath.sol` L409‚Äì415: net token1 in = 200; total fee = 4 token1; protocol fee = 2 token1; LP fee to bin = 2 token1. Bin state: `token0 = 1,800`, `token1 = 202`, `shares = 2,000`.

**Step 3 ‚Äî Attacker back-runs:** Bob calls `removeLiquidity` for 1,000 shares. Per `LiquidityLib.sol` L205‚Äì206: token0 received = `1,800 √ó 1,000 / 2,000 = 900`; token1 received = `202 √ó 1,000 / 2,000 = 101`. Bob paid 1,000 token0, received 900 token0 + 101 token1. At oracle price 1:1: **net profit = 1 token1**.

**Alice's loss:** Without Bob, Alice would receive `1,800 token0 + 202 token1` (net gain of 2 token1 LP fee). With Bob, Alice receives `900 token0 + 101 token1` (net gain of 1 token1 LP fee). **Alice loses 1 token1** ‚Äî exactly the amount Bob extracted.

A Foundry integration test can reproduce this by: (1) seeding bin 4 with Alice's liquidity, (2) having Bob call `addLiquidity` in the same block before a swap, (3) executing the swap, (4) having Bob call `removeLiquidity`, and (5) asserting Bob's net token gain equals the LP fee portion and Alice's net gain is reduced by the same amount.
