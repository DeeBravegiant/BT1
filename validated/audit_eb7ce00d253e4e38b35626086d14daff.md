### Title
Caller-Controlled Finality Parameters Allow Signing of Unconfirmed/Unreorged Foreign-Chain Transactions - (`crates/foreign-chain-inspector/src/bitcoin/inspector.rs`, `crates/near-mpc-contract-interface/src/types/foreign_chain.rs`)

---

### Summary

The `verify_foreign_transaction` flow accepts finality/confirmation requirements directly from the unprivileged caller (`BitcoinRpcRequest.confirmations`, `EvmRpcRequest.finality`, `SolanaRpcRequest.finality`). No minimum is enforced by the contract or the MPC nodes. An attacker can set `confirmations: 0` (Bitcoin) or `finality: Latest` (EVM) and obtain a valid MPC threshold signature over a transaction that has zero on-chain finality, enabling double-spend or bridge-drain attacks.

---

### Finding Description

`BitcoinRpcRequest` carries a `confirmations: BlockConfirmations` field that the caller populates freely: [1](#0-0) 

The contract's `verify_foreign_transaction` stores this request verbatim with no validation of the `confirmations` value: [2](#0-1) 

Each MPC node's `BitcoinInspector::extract` then uses the caller-supplied value directly as the threshold: [3](#0-2) 

The check is simply `block_confirmations_threshold <= transaction_block_confirmation`. With `confirmations: 0`, this is always satisfied — even for a mempool transaction with zero confirmations. The same structural problem applies to EVM chains, where the caller freely chooses `EvmFinality::Latest`, `Safe`, or `Finalized`: [4](#0-3) 

The on-chain `ChainEntry` / `foreign_chain_rpc_whitelist` stores an `rpc_quorum` (how many RPC providers must agree), but no per-chain **minimum confirmation count** or **minimum finality level** is stored or enforced anywhere in the contract or node pipeline.

This is the direct analog of the external report: just as a maker manually entering a wrong token decimal causes the swap to transfer the wrong amount, a caller manually entering `confirmations: 0` causes the MPC network to sign for a transaction with no finality guarantee.

---

### Impact Explanation

An attacker who controls a bridge or omnibridge inbound flow can:

1. Broadcast a Bitcoin (or EVM) transaction `T` that sends funds to the bridge.
2. Immediately call `verify_foreign_transaction` with `confirmations: 0` (or `finality: Latest`).
3. Receive a valid MPC threshold signature attesting that `T` was observed.
4. Use that signature to claim bridge funds on NEAR.
5. Double-spend `T` on Bitcoin (or wait for an EVM reorg) to recover the original funds.

The MPC signature is irrevocable once issued. The bridge contract on NEAR has no way to distinguish a "0-confirmation" attestation from a "6-confirmation" one because the `confirmations` value is embedded in the signed payload hash — the verifier only checks the signature, not the security level of the finality parameter used.

Impact: **High** — forged foreign-chain verification / double-spend condition causing invalid bridge execution.

---

### Likelihood Explanation

The entry path requires only a standard NEAR account and a 1 yoctoNEAR deposit. No privileged role, no threshold collusion, and no TEE attack is needed. Any user of the `verify_foreign_transaction` API can exploit this by simply setting `confirmations: 0` in their request. The vulnerability is reachable on every supported chain that exposes a caller-controlled finality parameter (Bitcoin, all EVM variants, Solana).

---

### Recommendation

Enforce a **per-chain minimum finality requirement** in the on-chain `ChainEntry` (already stored in `foreign_chain_rpc_whitelist`) and validate the caller's requested finality against it inside `verify_foreign_transaction` before enqueuing the request. For Bitcoin, add a `min_confirmations: BlockConfirmations` field to `ChainEntry` and reject any request where `request.confirmations < chain_entry.min_confirmations`. For EVM/Solana, store a `min_finality` level and reject weaker caller-supplied values. The MPC nodes should also independently enforce the minimum as a defense-in-depth measure, rather than blindly trusting the caller-supplied threshold.

---

### Proof of Concept

```rust
// Attacker submits with confirmations: 0 — no minimum is enforced
let request_args = VerifyForeignTransactionRequestArgs {
    domain_id: foreign_tx_domain_id,
    payload_version: ForeignTxPayloadVersion::V1,
    request: ForeignChainRpcRequest::Bitcoin(BitcoinRpcRequest {
        tx_id: attacker_tx_id,
        confirmations: BlockConfirmations(0), // ← caller sets this freely
        extractors: vec![BitcoinExtractor::BlockHash],
    }),
};
// contract.verify_foreign_transaction(request_args) — accepted without validation
// BitcoinInspector::extract: 0 <= actual_confirmations → always passes
// MPC nodes sign → attacker receives valid threshold signature for unconfirmed tx
```

The `BitcoinInspector` test suite itself confirms the check is purely `threshold <= actual`: [5](#0-4) 

With `threshold = 0`, any transaction — including one with zero confirmations — passes the check.

    pub fn verify_foreign_transaction(&mut self, request: VerifyForeignTransactionRequestArgs) {
        log!(
            "verify_foreign_transaction: predecessor={:?}, request={:?}",
            env::predecessor_account_id(),
            request
        );

        self.check_request_preconditions(
            request.domain_id,
            DomainPurpose::ForeignTx,
            Gas::from_tgas(self.config.sign_call_gas_attachment_requirement_tera_gas),
            MINIMUM_SIGN_REQUEST_DEPOSIT,
        );

        let requested_chain = request.request.chain();
        let supported_chains = self.get_supported_foreign_chains();
        if !supported_chains.contains(&requested_chain) {
            env::panic_str(
                &InvalidParameters::ForeignChainNotSupported {
                    requested: requested_chain,
                }
                .to_string(),
            );
        }

        let callback_gas = Gas::from_tgas(
            self.config
                .return_signature_and_clean_state_on_success_call_tera_gas,
        );

        let request = args_into_verify_foreign_tx_request(request);
        let callback_args = serde_json::to_vec(&(&request,)).unwrap();
        self.enqueue_yield_request(
            method_names::RETURN_VERIFY_FOREIGN_TX_AND_CLEAN_STATE_ON_SUCCESS,
            callback_args,
            callback_gas,
            move |this, id| this.add_verify_foreign_tx_request(request, id),
        );
