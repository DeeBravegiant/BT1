### Title
Timelord `_check_for_new_ip` uses hardcoded `uint32(0)` for `prev_transaction_block_height`, permanently blocking infusion of valid v2 plot blocks — (`chia/timelord/timelord.py`, `chia/timelord/iters_from_block.py`)

### Summary
After `HARD_FORK2_HEIGHT`, the timelord's `_can_infuse_unfinished_block` correctly accepts v2 plot blocks (using the real previous transaction block height), schedules VDF computation, and completes it. However, `_check_for_new_ip` re-evaluates the same block using `uint32(0)` as `prev_transaction_block_height`, causing `verify_and_get_quality_string` to return `None` for all v2 proofs, which causes `iters_from_block` to raise an `AssertionError`, which is silently caught. No block is ever matched, and infusion never occurs.

### Finding Description
Two call sites of `iters_from_block` within the timelord use inconsistent `prev_transaction_block_height` arguments for the same block:

- `_can_infuse_unfinished_block` (timelord.py:285): `self.last_state.get_last_tx_height()` — correct real value
- `_check_for_new_ip` (timelord.py:594): `uint32(0)` — hardcoded zero

`verify_and_get_quality_string` gates v2 proof validity on `prev_transaction_block_height >= HARD_FORK2_HEIGHT`. With `uint32(0)`, every v2 proof fails this check and returns `None`. `iters_from_block` then hits `assert required_iters is not None`, raising `AssertionError`. The `except Exception` block in `_check_for_new_ip` silently swallows this, leaving `block = None`. The `if block is not None:` guard at line 603 prevents infusion.

### Impact Explanation
After `HARD_FORK2_HEIGHT`, every v2 plot block submitted to the timelord via `new_unfinished_block_timelord` is scheduled (VDF computation runs to completion) but never infused. This is a permanent, systematic failure: **no v2 farmer can ever have their block finalized by a timelord running this code**. This constitutes a long-lived inability for honest farmers and timelords to process valid blocks under normal network assumptions.

### Likelihood Explanation
This triggers deterministically for every v2 plot block after `HARD_FORK2_HEIGHT`. No special conditions, attacker, or race are required — any farmer using v2 plots submitting a valid block via the standard protocol path hits this.

### Recommendation
Replace `uint32(0)` in `_check_for_new_ip` with the same `self.last_state.get_last_tx_height()` call used in `_can_infuse_unfinished_block`. Both call sites must use identical parameters for the same block to maintain the invariant that the scheduled VDF iteration count matches what infusion lookup expects.

### Proof of Concept
1. Configure a timelord with `HARD_FORK2_HEIGHT = N` for some reachable height.
2. Advance the chain past height `N` so `last_state.get_last_tx_height() >= N`.
3. Submit a `NewUnfinishedBlockTimelord` message containing a v2 proof-of-space.
4. Observe: `_can_infuse_unfinished_block` returns a non-None `new_block_iters` (block is scheduled).
5. Wait for VDF to complete at the scheduled iteration count.
6. Observe: `_check_for_new_ip` iterates `unfinished_blocks`, calls `iters_from_block(..., uint32(0))`, which raises `AssertionError` (caught silently), `block` remains `None`, infusion is skipped.
7. Assert: no `NewInfusionPointVDF` message is ever sent for this block.
