### Title
`normalize_amount` floor-division silently truncates small NEAR-to-destination transfers to zero, permanently locking user funds — (File: near/omni-bridge/src/lib.rs)

---

### Summary

When a user initiates a NEAR-outbound transfer with an amount small enough that `normalize_amount` (floor division) produces zero on the destination chain, the subsequent `sign_transfer` call permanently reverts with `InvalidAmountToTransfer`. Because the user's tokens are already burned/locked at `init_transfer` time and no cancel path exists, the funds are irreversibly locked.

---

### Finding Description

**Step 1 — Tokens are burned/locked before any normalization check.**

`init_transfer` (called via `ft_transfer_call`) burns or locks the user's NEAR-side tokens and stores the `TransferMessage` in `pending_transfers`. The only guard at this stage is `fee < amount`; there is no check that `amount - fee` will survive decimal normalization. [1](#0-0) [2](#0-1) 

**Step 2 — `normalize_amount` uses floor division.**

```rust
fn normalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    amount / (10_u128.pow(diff_decimals))
}
```

Any `amount` strictly less than `10^(origin_decimals − decimals)` produces **zero**. [3](#0-2) 

**Step 3 — `sign_transfer` hard-reverts on zero.**

```rust
let amount_to_transfer = Self::normalize_amount(
    transfer_message.amount_without_fee()...,
    decimals,
);
require!(
    amount_to_transfer > 0,
    BridgeError::InvalidAmountToTransfer.as_ref()
);
```

The `require!` macro panics (NEAR transaction revert). The transfer message is **not** removed from `pending_transfers` — that only happens inside `sign_transfer_callback`, which is never reached. [4](#0-3) 

**Step 4 — No cancel/refund path exists.**

`remove_transfer_message` is called only from `sign_transfer_callback` (post-MPC success) and `claim_fee_callback`. There is no public `cancel_transfer` entrypoint. The transfer sits in `pending_transfers` forever, and the burned/locked tokens are unrecoverable. [5](#0-4) 

---

### Impact Explanation

**Critical — Irreversible fund lock.**

A user who sends any amount below the normalization threshold (e.g., any value `< 10^(origin_decimals − dest_decimals)`) will have their NEAR-side tokens permanently destroyed with no corresponding credit on the destination chain and no recovery mechanism. This matches the allowed impact: *"Irreversible fund lock … permanently unclaimable user … value in bridge … flows."*

---

### Likelihood Explanation

**Low-to-medium.** For the NEAR native token bridged to EVM chains (`origin_decimals = 24`, `dest_decimals = 18`), the threshold is `10^6` yoctoNEAR — negligible in practice. However, for tokens bridged to Solana (`dest_decimals = 9`) the threshold rises to `10^15` yoctoNEAR (≈ 0.000001 NEAR), and for tokens with even larger decimal gaps the threshold grows further. Any user who sends an amount below the threshold — whether by accident or because they are testing with dust — permanently loses those funds. The protocol provides no warning and no guard at deposit time.

---

### Recommendation

Add a normalization check inside `init_transfer` (before burning/locking) and reject the call early if the normalized `amount_without_fee` would be zero:

```rust
// In init_transfer, after building transfer_message:
let token_address = self.get_token_address(
    transfer_message.get_destination_chain(),
    self.get_token_id(&transfer_message.token),
);
if let Some(decimals) = token_address.and_then(|a| self.token_decimals.get(&a)) {
    let normalized = Self::normalize_amount(
        transfer_message.amount_without_fee().expect("fee < amount"),
        decimals,
    );
    require!(normalized > 0, BridgeError::InvalidAmountToTransfer.as_ref());
}
```

Alternatively, add a public `cancel_transfer` function that allows the original sender to reclaim their tokens when a transfer has been pending for longer than a timeout period.

---

### Proof of Concept

1. Token `T` is registered with `origin_decimals = 24` (NEAR) and `decimals = 9` (Solana destination). Normalization threshold = `10^15`.
2. User calls `ft_transfer_call` on the NEAR bridge with `amount = 10^14` (below threshold), `fee = 0`, `recipient = "sol:<address>"`.
3. `init_transfer_internal` burns `10^14` units of `T` from the user and stores the `TransferMessage`.
4. Relayer calls `sign_transfer(transfer_id, ...)`.
5. `normalize_amount(10^14, Decimals { origin_decimals: 24, decimals: 9 })` → `10^14 / 10^15 = 0`.
6. `require!(0 > 0, ...)` panics → transaction reverts.
7. `TransferMessage` remains in `pending_transfers`; user's `10^14` units of `T` are permanently burned with no destination credit and no refund path. [4](#0-3) [6](#0-5)
