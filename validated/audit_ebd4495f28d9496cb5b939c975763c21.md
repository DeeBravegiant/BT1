Audit Report

## Title
Stale `rsETHPrice` in `instantWithdrawal()` Allows Over-Redemption of Vault Assets — (File: contracts/LRTWithdrawalManager.sol)

## Summary
`LRTWithdrawalManager.instantWithdrawal()` computes the asset payout using the cached `LRTOracle.rsETHPrice` state variable without forcing a price refresh or applying a minimum-of-current-vs-stored guard. If the true rsETH price has decreased since the last `updateRSETHPrice()` call — for example due to an EigenLayer slashing event within the `pricePercentageLimit` threshold — any rsETH holder can call `instantWithdrawal()` with the stale, inflated price and drain more assets from `LRTUnstakingVault` than the current exchange rate entitles them to, at the direct expense of other protocol participants.

## Finding Description

`LRTOracle.rsETHPrice` is a stored state variable written only inside `_updateRsETHPrice()`: [1](#0-0) [2](#0-1) 

`updateRSETHPrice()` is a separate public transaction that must be called explicitly; it is not invoked atomically with withdrawals: [3](#0-2) 

`instantWithdrawal()` calls `getExpectedAssetAmount()` at line 228, which reads `lrtOracle.rsETHPrice()` — the cached value — with no freshness check: [4](#0-3) [5](#0-4) 

The normal `initiateWithdrawal()` + `unlockQueue()` path applies `_calculatePayoutAmount()`, which takes the **minimum** of the stored expected amount and the current-price-derived amount, protecting users and the vault from price drops: [6](#0-5) 

`instantWithdrawal()` has no equivalent protection. The payout is computed once from the stale `rsETHPrice` and transferred directly.

The `_updateRsETHPrice()` downside-protection logic pauses the protocol **only** when the price drop exceeds `pricePercentageLimit`: [7](#0-6) 

For drops **within** the configured threshold (e.g., ≤1%), the protocol remains unpaused, `instantWithdrawal()` remains callable, and the stale higher price is used to compute the payout.

**Exploit path:**
1. A slashing event reduces the true rsETH price by an amount within `pricePercentageLimit` (no automatic pause triggered).
2. `updateRSETHPrice()` has not yet been called; `rsETHPrice` still reflects the pre-slash value.
3. Attacker (any rsETH holder) calls `instantWithdrawal(asset, rsETHUnstaked, "")`.
4. `getExpectedAssetAmount` returns `rsETHUnstaked * stalePriceHigh / assetPrice`, which is larger than `rsETHUnstaked * truePriceLow / assetPrice`.
5. The excess assets are redeemed from `LRTUnstakingVault` and transferred to the attacker.
6. The attacker can repeat this until `updateRSETHPrice()` is called or vault liquidity is exhausted.

## Impact Explanation

Direct theft of funds at rest in `LRTUnstakingVault`. The attacker receives more underlying assets than the current rsETH/asset exchange rate entitles them to; the shortfall is borne by other protocol participants whose queued withdrawals or vault balances are reduced. This matches the **Critical** impact class: direct theft of user funds at rest.

## Likelihood Explanation

- EigenLayer slashing is a documented, live risk for the protocol's node delegators.
- A small slashing event within `pricePercentageLimit` does not trigger an automatic pause, leaving the window open.
- `updateRSETHPrice()` is not called atomically with withdrawals; there is always a non-zero window between a slashing event and the next keeper call.
- An attacker can monitor on-chain EigenLayer slashing events and front-run the price update by calling `instantWithdrawal()` in the same or next block.
- `isInstantWithdrawalEnabled` must be set for the target asset, but this is a live operational feature with a dedicated setter, making it a realistic precondition.

**Likelihood: Medium** — realistic trigger condition combined with a straightforward front-running opportunity.

## Recommendation

Before computing `assetAmountUnlocked` in `instantWithdrawal()`, force a price refresh:

```solidity
function instantWithdrawal(address asset, uint256 rsETHUnstaked, string calldata referralId)
    external nonReentrant whenNotPaused ...
{
    ILRTOracle(lrtConfig.getContract(LRTConstants.LRT_ORACLE)).updateRSETHPrice();
    uint256 assetAmountUnlocked = getExpectedAssetAmount(asset, rsETHUnstaked);
    ...
}
```

Alternatively, apply the same `min(expectedAmount, currentReturn)` guard used in `_calculatePayoutAmount()` inside `instantWithdrawal()`, computing the current-price-derived amount on-the-fly and capping the payout at the lower of the two values. This mirrors the protection already present in the queued withdrawal path.

## Proof of Concept

**Setup:**
- `rsETHPrice = 1.05e18` (last updated 12 hours ago).
- A small EigenLayer slashing event occurs; true current price is `1.04e18`.
- Price drop = ~0.95%, within `pricePercentageLimit = 1%` → no pause triggered.
- `isInstantWithdrawalEnabled[ETH_TOKEN] = true`.

**Steps:**
1. Attacker holds `10_000e18` rsETH.
2. Attacker calls `instantWithdrawal(ETH_TOKEN, 10_000e18, "")` before any keeper calls `updateRSETHPrice()`.
3. `getExpectedAssetAmount` computes: `10_000e18 * 1.05e18 / 1e18 = 10_500 ETH`.
4. True fair payout at current price: `10_000e18 * 1.04e18 / 1e18 = 10_400 ETH`.
5. Attacker receives `10_500 ETH` (minus fee), extracting `~100 ETH` of excess value from `LRTUnstakingVault`.

**Foundry fork test plan:**
- Fork mainnet at a block after a known slashing event.
- Manually set `rsETHPrice` to the pre-slash value via `vm.store`.
- Call `instantWithdrawal` as an unprivileged address.
- Assert that the transferred amount exceeds `rsETHUnstaked * trueCurrentPrice / assetPrice`.
- Assert that calling `updateRSETHPrice()` first and then `instantWithdrawal()` yields the lower, correct amount.
