Audit Report

## Title
Multiplicative Gas Consumption in `updateRSETHPrice()` via Per-Asset `getQueuedWithdrawals` Calls Across NDCs — (File: contracts/NodeDelegator.sol)

## Summary
`LRTOracle.updateRSETHPrice()` is publicly callable with no access control beyond `whenNotPaused`. Its internal call chain invokes `NodeDelegator.getAssetUnstaking(asset)` once per supported asset per NDC, causing `getQueuedWithdrawals` to be called `assets × NDCs` times rather than once per NDC. The protocol's own comment in `LRTUnstakingVault.sol` acknowledges a gas ceiling of 120 total queued withdrawals, but the buffer calculation assumes a small NDC and asset count that does not hold at the current admin-settable maximums. EigenLayer-side forced undelegations can push the actual queued-withdrawal count above the protocol-tracked ceiling without triggering the `maxUncompletedWithdrawalCount` guard, causing `updateRSETHPrice()` to revert with out-of-gas.

## Finding Description
The public entry point `updateRSETHPrice()` calls `_updateRsETHPrice()` → `_getTotalEthInProtocol()`.

`_getTotalEthInProtocol()` iterates over every supported asset and calls `ILRTDepositPool.getTotalAssetDeposits(asset)` for each:

```solidity
// LRTOracle.sol L336-341
for (uint16 assetIdx; assetIdx < supportedAssetCount;) {
    address asset = supportedAssets[assetIdx];
    uint256 assetER = getAssetPrice(asset);
    uint256 totalAssetAmt = ILRTDepositPool(lrtDepositPoolAddr).getTotalAssetDeposits(asset);
    ...
}
```

`getTotalAssetDeposits` dispatches to `getAssetDistributionData` (LSTs) or `getETHDistributionData` (ETH). Both loop over every NDC and call `INodeDelegator.getAssetUnstaking(asset)` on each:

```solidity
// LRTDepositPool.sol L446-456
for (uint256 i; i < ndcsCount;) {
    assetUnstakingFromEigenLayer += INodeDelegator(nodeDelegatorQueue[i]).getAssetUnstaking(asset);
    ...
}
```

`getAssetUnstaking` calls EigenLayer's `getQueuedWithdrawals(address(this))` and iterates over every returned withdrawal and every strategy within each withdrawal:

```solidity
// NodeDelegator.sol L405-427
function getAssetUnstaking(address asset) external view returns (uint256 amount) {
    (IDelegationManager.Withdrawal[] memory queuedWithdrawals, uint256[][] memory withdrawalShares) =
        _getDelegationManager().getQueuedWithdrawals(address(this));
    for (uint256 withdrawalIndex = 0; ...) {
        for (uint256 strategyIndex = 0; ...) {
            ...
            amount += strategy.sharesToUnderlyingView(sharesToUnstake);
        }
    }
}
```

The total work scales as **O(assets × NDCs × withdrawals_per_NDC × strategies_per_withdrawal)**. `getQueuedWithdrawals` is invoked `assets × NDCs` times — once per asset per NDC — rather than once per NDC. This means each NDC's full withdrawal list is fetched and iterated redundantly for every supported asset.

The `maxUncompletedWithdrawalCount` guard (capped at 80) tracks only protocol-initiated withdrawals via `initiateUnstaking` and `undelegate`. EigenLayer operator-initiated forced undelegations create queued withdrawals on-chain that are not reflected in `uncompletedWithdrawalCount`, bypassing this guard entirely. The protocol's own comment acknowledges this:

```solidity
// LRTUnstakingVault.sol L151-152
// 120 is the max number of uncompleted withdrawals that allows us to still perform update rsETH price
// Need buffer for theoretical operator forced undelegations (ndc count * asset count = 15)
```

The comment's buffer calculation assumes `ndc count * asset count = 15` (e.g., 3 NDCs × 5 assets). However, `maxNodeDelegatorLimit` is currently 10 and admin-settable upward, and the protocol supports more than 5 assets. At 10 NDCs and 10 assets, forced undelegations can add up to 100 additional withdrawals (one per NDC per asset strategy), pushing the actual on-chain count to 80 + 100 = 180 — well above the 120 gas-safe ceiling the protocol itself identifies.

## Impact Explanation
When `updateRSETHPrice()` reverts due to gas exhaustion, `rsETHPrice` is never written. The stored price becomes stale. Because rsETH is yield-bearing and its price monotonically increases, a stale price is lower than the true price. Subsequent depositors receive more rsETH than they are entitled to at the stale rate, diluting all existing rsETH holders. The protocol's fee-minting mechanism also stops functioning. The primary allowed impact is **Unbounded Gas Consumption (Medium)**, with a secondary effect of temporary freezing of the price update mechanism.

## Likelihood Explanation
`updateRSETHPrice()` is callable by any unprivileged external account (`public whenNotPaused`). The gas exhaustion condition is reached through normal protocol growth: (a) admin increases `maxNodeDelegatorLimit` to 10 (already the default) and adds 10 NDCs, (b) protocol supports 10+ assets (already approaching this), and (c) an EigenLayer operator executes a forced undelegation on one or more NDCs, adding queued withdrawals outside the protocol-tracked count. None of these steps require attacker privileges — the attacker merely calls `updateRSETHPrice()` after the state is reached. The protocol's own comment at `LRTUnstakingVault.sol` L151–152 confirms awareness of this ceiling and that the buffer assumption (`ndc count * asset count = 15`) is already stale relative to current parameter maximums.

## Recommendation
Refactor `getAssetUnstaking` so that `getQueuedWithdrawals` is called **once per NDC** and the result is reused across all assets. Introduce a batch view function (e.g., in `NodeDelegatorHelper`) that returns all asset unstaking amounts in a single pass over the queued withdrawals, and call it once per NDC inside `_getTotalEthInProtocol`. This reduces the call complexity from O(assets × NDCs × withdrawals) to O(NDCs × withdrawals). Additionally, update the forced-undelegation buffer calculation in `setMaxUncompletedWithdrawalCount` to reflect the actual current `maxNodeDelegatorLimit` and supported asset count rather than hardcoding `ndc count * asset count = 15`.

## Proof of Concept
1. Protocol reaches: 10 supported assets, 10 NDCs (within `maxNodeDelegatorLimit = 10`), 80 protocol-tracked queued withdrawals (at `maxUncompletedWithdrawalCount = 80`).
2. An EigenLayer operator executes forced undelegation on all 10 NDCs. Each undelegation creates one queued withdrawal per strategy (up to 10 assets × 10 NDCs = 100 additional withdrawals). These are not reflected in `uncompletedWithdrawalCount`.
3. Any caller invokes `updateRSETHPrice()`.
4. `_getTotalEthInProtocol()` calls `getTotalAssetDeposits` 10 times (once per asset).
5. Each call iterates 10 NDCs and calls `getAssetUnstaking` per NDC → 100 calls to `getQueuedWithdrawals`, each returning up to 18 withdrawals (80 tracked / 10 NDCs + 10 forced).
6. Total withdrawal iterations: 100 calls × 18 withdrawals × strategies per withdrawal, plus `sharesToUnderlyingView` per strategy.
7. Aggregate gas exceeds the 30 M block gas limit; the transaction reverts.
8. `rsETHPrice` is not updated; the price update mechanism is frozen until the queued withdrawal count is reduced below the gas-safe threshold.

**Foundry fork test plan**: Fork mainnet, deploy protocol with 10 NDCs and 10 supported assets, fill `uncompletedWithdrawalCount` to 80 via `initiateUnstaking`, then simulate forced undelegations by directly calling EigenLayer's `DelegationManager.undelegate` from the operator address for each NDC. Measure gas of `updateRSETHPrice()` before and after forced undelegations to demonstrate the revert.
