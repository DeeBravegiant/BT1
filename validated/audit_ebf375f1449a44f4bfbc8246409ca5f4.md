Audit Report

## Title
Zero `rsETHPrice` at `initiateWithdrawal` Permanently Burns User rsETH With Zero Payout — (File: `contracts/LRTWithdrawalManager.sol`)

## Summary

When `rsETHPrice` is 0 in `LRTOracle` (either before `updateRSETHPrice()` is ever called, or after a full-slashing event with the default `pricePercentageLimit = 0`), a user calling `initiateWithdrawal` has their rsETH transferred to the contract while `expectedAssetAmount = 0` is stored in their withdrawal request. Because `_calculatePayoutAmount` returns `min(expectedAssetAmount, currentReturn)`, the stored zero is permanent — even after price recovery, the operator's `unlockQueue` call burns the user's full rsETH while unlocking zero assets, and `completeWithdrawal` transfers nothing to the user.

## Finding Description

**Root cause — `rsETHPrice` can be 0 and the downside guard is off by default.**

`rsETHPrice` is a storage variable in `LRTOracle` that initializes to 0 and is only updated by `updateRSETHPrice()`. [1](#0-0) 

The downside-protection guard that would pause the protocol on a price crash is gated on `pricePercentageLimit > 0`, which also defaults to 0, leaving it permanently disabled unless an admin explicitly sets it: [2](#0-1) 

If `totalETHInProtocol` reaches 0 (complete slashing) while `pricePercentageLimit = 0`, `newRsETHPrice = 0` is written to storage without triggering a pause: [3](#0-2) 

**Step 1 — `initiateWithdrawal` stores `expectedAssetAmount = 0` without reverting.**

`getExpectedAssetAmount` computes `amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset)`. When `rsETHPrice = 0` this returns 0: [4](#0-3) 

The user's rsETH is transferred first, then `expectedAssetAmount = 0` is stored with no zero-check: [5](#0-4) 

The only guard present (`expectedAssetAmount > getAvailableAssetAmount`) evaluates to `0 > x`, which is always false for uint256, so it never reverts.

**Step 2 — `_calculatePayoutAmount` permanently locks the payout at 0.**

Even after `rsETHPrice` recovers, `_calculatePayoutAmount` returns `min(request.expectedAssetAmount, currentReturn)`. Because `request.expectedAssetAmount` was stored as 0, the result is always 0 regardless of the current price: [6](#0-5) 

**Step 3 — `_unlockWithdrawalRequests` burns rsETH unconditionally for zero-payout requests.**

The loop guard `if (availableAssetAmount < payoutAmount) break` evaluates to `if (x < 0)` for uint256, which is always false, so the loop never exits on a zero-payout request. The full `rsETHUnstaked` is added to the burn total while `assetAmountToUnlock += 0`: [7](#0-6) 

Back in `unlockQueue`, the rsETH is burned and zero assets are redeemed from the vault: [8](#0-7) 

**Step 4 — `completeWithdrawal` silently transfers 0.**

`_transferAsset` with `amount = 0` succeeds for both ETH (`call{value:0}`) and ERC20 (`safeTransfer(..., 0)`): [9](#0-8) [10](#0-9) 

**Why `_validatePrices` in `unlockQueue` does not save the user.**

`_validatePrices` checks the price at the time the operator calls `unlockQueue`, not at the time the withdrawal was initiated: [11](#0-10) 

If the operator calls `unlockQueue` after price recovery with a valid non-zero price range, `_validatePrices` passes, but `_calculatePayoutAmount` still returns 0 because `request.expectedAssetAmount` was permanently stored as 0 at initiation time. The operator does not need to act negligently — normal operation after price recovery still burns the user's rsETH for nothing.

## Impact Explanation

**Critical — Permanent burning of user funds.**

A user who calls `initiateWithdrawal` while `rsETHPrice = 0` has their rsETH irreversibly transferred to the contract and later burned by `unlockQueue`. The withdrawal request is deleted after `completeWithdrawal`, the rsETH is gone, and the user receives 0 assets. There is no recovery path. This matches the allowed impact: *Permanent freezing of funds* (the rsETH is consumed by the protocol with no corresponding output).

## Likelihood Explanation

**Low.**

Two conditions must coincide:
1. `rsETHPrice = 0` in `LRTOracle` storage — reachable either (a) during the initial deployment window before `updateRSETHPrice()` is first called, or (b) after a catastrophic full-slashing event while `pricePercentageLimit = 0` (its default), which disables the automatic pause guard.
2. A user calls `initiateWithdrawal` while the price is in this state and the contract is not paused.

The default `pricePercentageLimit = 0` makes condition (b) more likely than it appears, as the downside protection is off-by-default. The deployment window (condition a) is a realistic and repeatable scenario.

## Recommendation

1. **In `initiateWithdrawal`**: revert if `expectedAssetAmount == 0` after computing it, before transferring rsETH (or after, with a revert that returns the rsETH):
   ```solidity
   if (expectedAssetAmount == 0) revert InvalidAmountToWithdraw();
   ```

2. **In `_unlockWithdrawalRequests`**: skip (do not burn rsETH for) requests where `payoutAmount == 0`:
   ```solidity
   if (payoutAmount == 0) break; // or continue to skip and process later requests
   ```

3. **Set a non-zero `pricePercentageLimit`** during deployment initialization to ensure the automatic pause guard is active from the start.

4. **Consider adding a `rsETHPrice > 0` check** in `initiateWithdrawal` directly against the oracle before accepting the withdrawal request.

## Proof of Concept

```
Scenario A — Deployment window:
1. Deploy protocol. Do NOT call updateRSETHPrice(). rsETHPrice = 0 in storage.
2. Victim calls initiateWithdrawal(stETH, 1e18, ""):
   - 1e18 rsETH transferred from victim to LRTWithdrawalManager.
   - getExpectedAssetAmount returns 0 (1e18 * 0 / assetPrice = 0).
   - WithdrawalRequest stored: { rsETHUnstaked: 1e18, expectedAssetAmount: 0 }.
3. Price recovers. Operator calls unlockQueue(stETH, N, 0.9e18, type(uint256).max, ...):
   - _validatePrices passes (recovered price is in range).
   - _calculatePayoutAmount: min(0, currentReturn) = 0.
   - payoutAmount = 0; guard `availableAssetAmount < 0` is false → loop continues.
   - rsETHAmountToBurn += 1e18; assetAmountToUnlock += 0.
   - burnFrom(withdrawalManager, 1e18) → victim's rsETH permanently destroyed.
   - unstakingVault.redeem(stETH, 0) → no assets moved.
4. Victim calls completeWithdrawal(stETH, ""):
   - _transferAsset(stETH, victim, 0) → victim receives 0 stETH.
   - Victim has lost 1e18 rsETH with no recovery path.

Scenario B — Full slashing with default pricePercentageLimit = 0:
1. All EigenLayer assets slashed → totalETHInProtocol = 0.
2. updateRSETHPrice() called → newRsETHPrice = 0, no pause triggered (pricePercentageLimit = 0).
3. Same steps 2–4 as above.
```
