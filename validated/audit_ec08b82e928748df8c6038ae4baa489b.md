### Title
`respond_ckd` Skips Cryptographic Output Validation for `AppPublicKey` Variant, Enabling Single-Participant Threshold Bypass - (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_ckd` performs a cryptographic pairing check (`ckd_output_check`) on the CKD response **only** when the request carries the `AppPublicKeyPV` variant. When the `AppPublicKey` variant is used, the contract accepts any arbitrary `CKDResponse` from a single attested participant with zero cryptographic validation, bypassing the threshold security model entirely.

---

### Finding Description

In `respond_ckd`, the contract branches on the request's `app_public_key` variant:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}   // ← no validation whatsoever
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For the `AppPublicKeyPV` arm, `ckd_output_check` enforces the pairing equation `e(big_c, g2) = e(big_y, app_pk2) · e(hash_point, public_key)`, which cryptographically binds the response to the master secret key and the specific app identity — meaning only a threshold-coordinated computation can produce a passing value. [2](#0-1) 

For the `AppPublicKey` arm, **no equivalent check exists**. The `CKDResponse` fields `big_c` and `big_y` are accepted as-is and immediately forwarded to the waiting yield: [3](#0-2) 

The `AppPublicKey` variant is a legitimate, documented production path accepted by `request_app_private_key`: [4](#0-3) 

The analog to the ERC721 report is exact: just as `safeTransferFrom` called `onERC721Received` but never checked the returned magic bytes (allowing any contract with a fallback to pass), `respond_ckd` calls into the CKD response path but never checks the cryptographic "magic bytes" (the pairing equation) for `AppPublicKey` requests — allowing any single attested participant to pass an arbitrary response.

---

### Impact Explanation

A single malicious attested MPC participant (strictly below the signing threshold) can:

1. Watch the chain for a pending `CKDRequest` using the `AppPublicKey` variant.
2. Call `respond_ckd` with a fully fabricated `CKDResponse` — arbitrary `big_c` and `big_y` — before honest nodes respond.
3. The contract resolves the yield immediately, delivering the forged output to the user.
4. All subsequent honest-node responses are rejected because the yield is already consumed.

This is a **threshold bypass**: the protocol requires `t+1` participants to cooperate to produce a cryptographically valid CKD output, but for `AppPublicKey` requests a single participant can unilaterally forge one. The user receives a wrong encrypted key material, breaking the confidentiality and integrity guarantees of the CKD protocol.

---

### Likelihood Explanation

Any single currently-attested MPC participant can trigger this. The `AppPublicKey` variant is a supported, non-deprecated production path. The attacker only needs to race honest nodes to call `respond_ckd` first, which is straightforward given that all participants observe the same on-chain pending requests via the chain gateway.

---

### Recommendation

Apply `ckd_output_check` (or an equivalent binding check) to `AppPublicKey` responses as well. If the `AppPublicKey` variant lacks the G2 component (`pk2`) required for the full pairing equation, one option is to require callers to always use `AppPublicKeyPV`. Alternatively, derive a binding commitment from the G1 component alone and verify it against the response before resolving the yield.

---

### Proof of Concept

1. User calls `request_app_private_key` with `app_public_key = AppPublicKey(some_g1_point)`.
2. Contract enqueues a `CKDRequest` and yields.
3. Malicious attested participant calls `respond_ckd(request, CKDResponse { big_c: arbitrary_g1, big_y: arbitrary_g1 })`.
4. Contract enters the `AppPublicKey(_) => {}` branch — no check — and calls `resolve_yields_for`, resolving the yield with the forged response.
5. User's callback receives the forged `big_c`/`big_y` as if it were a legitimate threshold-computed CKD output. [5](#0-4) [2](#0-1)

    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}
