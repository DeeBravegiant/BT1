Audit Report

## Title
Payable Swap and Liquidity Functions Strand ETH on Non-WETH Paths, Claimable by Any Caller via `refundETH()` — (`metric-periphery/contracts/MetricOmmSimpleRouter.sol`, `metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol`)

## Summary
All four swap functions (`exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`) and both liquidity entry points (`addLiquidityExactShares`, `addLiquidityWeighted`) carry the `payable` modifier with no enforcement that `msg.value == 0` when the input token is not WETH. When `token != WETH`, `PeripheryPayments.pay()` executes a plain ERC-20 `safeTransferFrom` and never reads `address(this).balance`, leaving any attached ETH stranded in the contract. `refundETH()` has no access control and forwards the entire contract balance to whoever calls it next, enabling any third party to steal the stranded ETH.

## Finding Description
`PeripheryPayments.pay()` branches on the token address:

```solidity
// PeripheryPayments.sol L69-88
if (payer == address(this)) {
    IERC20(token).safeTransfer(recipient, value);
} else if (token == WETH) {
    // wraps ETH from address(this).balance
} else {
    IERC20(token).safeTransferFrom(payer, recipient, value); // ETH ignored
}
```

The `else` branch is reached for every non-WETH token. `address(this).balance` is never decremented. Any `msg.value` attached to the call accumulates silently.

`refundETH()` then drains the full balance to the caller:

```solidity
// PeripheryPayments.sol L58-63
function refundETH() external payable override {
    uint256 balance = address(this).balance;
    if (balance > 0) {
        _transferETH(msg.sender, balance);
    }
}
```

There is no per-user accounting, no reentrancy guard relevant here, and no automatic refund at the end of any swap or liquidity function. The `receive()` guard (`if (msg.sender != WETH) revert NotWETH()`) only blocks bare ETH transfers with no calldata; it does not prevent ETH from entering via `msg.value` on `payable` functions. The `multicall` wrapper is also `payable`, so ETH can arrive through batched calls where one sub-call is non-WETH.

For WETH swaps where the user over-sends, `pay()` wraps exactly `value` (or `nativeBalance` if less), leaving any surplus in the contract, also claimable by anyone via `refundETH()`.

## Impact Explanation
Direct, irreversible loss of user ETH principal. A user who calls any of the six entry points with `tokenIn/token != WETH` and `msg.value > 0` loses that ETH with no recovery path. Any address can call `refundETH()` in a subsequent transaction and receive the full stranded balance. This is a direct loss of user funds meeting Critical/High impact thresholds under the allowed impact gate.

## Likelihood Explanation
User error (sending ETH with a non-WETH swap) is a well-documented and frequently occurring mistake. Wallets, aggregators, and scripts that forward ETH will silently succeed because all six functions are `payable`. MEV bots routinely monitor contract ETH balances and can back-run any transaction that strands ETH by immediately calling `refundETH()`. The `multicall` wrapper being `payable` further widens the attack surface. No special privileges or setup are required for the thief.

## Recommendation
Add a `msg.value == 0` guard in each function where ETH is not consumed:

```solidity
// In exactInputSingle / exactInput / exactOutputSingle / exactOutput
// and addLiquidityExactShares / addLiquidityWeighted, before any state changes:
if (params.tokenIn != WETH && msg.value != 0) revert UnexpectedETH();
```

Alternatively, remove the `payable` modifier from all six functions (and `multicall` if ETH is not needed there), or add an automatic `refundETH()` call at the end of each function so any surplus is returned to `msg.sender` atomically within the same transaction.

## Proof of Concept
1. Pool exists for `USDC/DAI` (neither token is WETH).
2. Alice calls `exactInputSingle({tokenIn: USDC, ...})` with `msg.value = 1 ether` (user error or wallet bug).
3. Inside `_justPayCallback`, `pay(USDC, Alice, pool, amount)` executes the `else` branch — `safeTransferFrom` — and never reads `address(this).balance`. The 1 ETH remains in `MetricOmmSimpleRouter`.
4. Bob observes the contract balance on-chain and calls `refundETH()`.
5. `refundETH()` transfers `address(this).balance` (1 ETH) to Bob. Alice's ETH is gone with no recourse.

Foundry test outline:
```solidity
function test_strandedEthStolenViaRefund() public {
    // Alice swaps USDC->DAI with 1 ETH accidentally attached
    vm.prank(alice);
    router.exactInputSingle{value: 1 ether}(usdcDaiParams);
    assertEq(address(router).balance, 1 ether);

    // Bob steals it
    vm.prank(bob);
    router.refundETH();
    assertEq(bob.balance, 1 ether);
    assertEq(alice.balance, initialAliceBalance - 1 ether); // permanent loss
}
```
