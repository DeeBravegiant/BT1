Audit Report

## Title
Missing Non-Positive Price Validation Before `uint256` Cast Causes Revert-Based Freeze of Deposits and Price Updates - (File: `contracts/oracles/ChainlinkPriceOracle.sol`)

## Summary
`ChainlinkPriceOracle.getAssetPrice()` casts the raw `int256 price` from Chainlink's `latestRoundData()` directly to `uint256` without validating that `price > 0`. In Solidity 0.8, an explicit cast of a negative `int256` to `uint256` wraps silently to a value near `type(uint256).max`; the subsequent `* 1e18` multiplication then triggers a checked-arithmetic overflow revert. This revert propagates through `LRTOracle._getTotalEthInProtocol()` → `_updateRsETHPrice()` → `updateRSETHPrice()` and through `LRTDepositPool.getRsETHAmountToMint()` → `depositAsset()`, freezing both price updates and all deposits until the oracle recovers.

## Finding Description
In `contracts/oracles/ChainlinkPriceOracle.sol` at lines 52–54:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

There is no guard against `price <= 0`. In Solidity 0.8, `uint256(negative_int256)` does **not** revert — it wraps to a value near `type(uint256).max`. The subsequent `uint256(price) * 1e18` then triggers an overflow revert under Solidity 0.8's checked arithmetic.

The sibling contract `contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol` at line 32 correctly validates:

```solidity
if (ethPrice <= 0) revert InvalidPrice();
```

The exploit path is:
1. A Chainlink feed for any supported LST asset (e.g., stETH/ETH) returns `price <= 0` (circuit breaker, feed misconfiguration, or minAnswer hit).
2. `ChainlinkPriceOracle.getAssetPrice(asset)` executes `uint256(-1)` → `2^256 - 1`, then `(2^256 - 1) * 1e18` → overflow revert.
3. `LRTOracle.getAssetPrice(asset)` (line 157) reverts.
4. `LRTOracle._getTotalEthInProtocol()` (line 339) reverts.
5. `LRTOracle._updateRsETHPrice()` (line 231) reverts.
6. `LRTOracle.updateRSETHPrice()` (line 87–89, `public whenNotPaused`) reverts for all callers.
7. `LRTDepositPool.getRsETHAmountToMint()` (line 520) also calls `lrtOracle.getAssetPrice(asset)` → reverts.
8. `LRTDepositPool.depositAsset()` → `_beforeDeposit()` → `getRsETHAmountToMint()` → revert, blocking all deposits.

No existing guard in `ChainlinkPriceOracle` prevents this. The `onlySupportedAsset` modifier only checks that a price feed is configured, not that the returned value is valid.

## Impact Explanation
**Medium — Temporary freezing of funds.** While the Chainlink feed returns a non-positive value, every call to `updateRSETHPrice()` and `depositAsset()` reverts. Users cannot deposit any LST asset. The rsETH price becomes stale. The freeze persists until the Chainlink feed recovers or an admin replaces the oracle. Funds already in the protocol are not lost, making this temporary rather than permanent.

## Likelihood Explanation
Chainlink feeds can return non-positive values during circuit breaker events when the underlying asset price hits the feed's configured `minAnswer` bound (a known, documented Chainlink behavior). `ChainlinkPriceOracle` is the primary oracle for all supported LST assets in the protocol. Any single feed returning a non-positive value is sufficient to trigger the freeze. No privileged access or oracle operator compromise is required — this is a passive failure mode of the external feed.

## Recommendation
Add a non-positive price check before casting, mirroring the pattern already used in `ChainlinkOracleForRSETHPoolCollateral`:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
if (price <= 0) revert InvalidPrice();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

## Proof of Concept
1. Deploy a mock Chainlink aggregator for a supported asset (e.g., stETH) that returns `answer = -1` from `latestRoundData()`.
2. Configure `ChainlinkPriceOracle` to use this mock feed via `updatePriceFeedFor`.
3. Call `LRTOracle.updateRSETHPrice()` — observe revert due to overflow at `uint256(-1) * 1e18`.
4. Call `LRTDepositPool.depositAsset(stETH, amount, 0, "")` — observe revert propagating from `getRsETHAmountToMint()` → `lrtOracle.getAssetPrice(stETH)`.
5. Restore the mock to return `answer = 1e18` — both calls succeed, confirming the freeze is temporary and oracle-dependent.

Foundry test sketch:
```solidity
function test_negativePrice_freezesDeposits() public {
    mockFeed.setAnswer(-1);
    vm.expectRevert(); // arithmetic overflow
    lrtOracle.updateRSETHPrice();
    vm.expectRevert();
    lrtDepositPool.depositAsset(stETH, 1 ether, 0, "");
}
```
