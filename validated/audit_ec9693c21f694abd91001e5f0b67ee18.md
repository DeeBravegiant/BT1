### Title
Missing CKD Response Integrity Check for `AppPublicKey` Variant Allows Single Byzantine Participant to Inject Fake Key Derivation Output - (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_ckd()` performs a cryptographic output check only for the `AppPublicKeyPV` (publicly verifiable) variant of a CKD request. For the `AppPublicKey` (legacy, privately verifiable) variant, the check arm is completely empty. A single Byzantine attested participant can call `respond_ckd()` with a fabricated `CKDResponse` for any pending `AppPublicKey` CKD request, and the contract will accept and deliver it to the user without any integrity verification.

---

### Finding Description

In `respond_ckd()`, after caller authentication, the contract branches on the `app_public_key` variant of the pending request:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}   // ← empty: no check
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV`, `ckd_output_check()` verifies that the encrypted response is a valid BLS12-381 encryption of the derived key under the user's ephemeral public key pair. For `AppPublicKey`, the match arm is a no-op. The `response` is then passed directly to `resolve_yields_for`, which resolves the user's yield promise with whatever bytes were supplied. [2](#0-1) 

The caller only needs to pass `assert_caller_is_attested_participant_and_protocol_active()` — a single-participant check, not a threshold check. [3](#0-2) 

The analog to the original report is exact: just as `liquidatorNFTClaim()` accepted a caller-supplied `params.endTime` without validating it against the stored Seaport order hash, `respond_ckd()` accepts a caller-supplied `CKDResponse` without validating it against any cryptographic commitment for the `AppPublicKey` variant.

---

### Impact Explanation

**Critical.** A single Byzantine attested participant can deliver an arbitrary `CKDResponse` to any user who submitted a CKD request using the `AppPublicKey` (legacy) variant. The attacker constructs a response that encrypts a known plaintext under the user's `app_public_key`. The user decrypts it and obtains what they believe is their MPC-derived private key — but it is a value the attacker already knows. The attacker can then independently derive the same key and sign arbitrary transactions on behalf of the user's derived address on any foreign chain. This constitutes unauthorized confidential key derivation output and enables theft of funds controlled by the derived key.

---

### Likelihood Explanation

**Medium-High.** The `AppPublicKey` variant is explicitly documented as the legacy format and is still accepted by the contract. Any single attested participant who turns Byzantine can exploit this without threshold cooperation. The attacker only needs to:
1. Monitor the NEAR chain for pending `request_app_private_key` calls using the `AppPublicKey` variant.
2. Race the honest MPC nodes to call `respond_ckd` first with a crafted response.

The race is realistic because the attacker is an MPC node with direct chain access and can submit the transaction immediately upon observing the pending request.

---

### Recommendation

Add the same `ckd_output_check` call for the `AppPublicKey` variant, or reject `AppPublicKey` requests in `respond_ckd` if no verifiable check can be performed. The fix mirrors the recommendation in the original report: validate the caller-supplied response against a cryptographic commitment stored at request time (analogous to storing the Seaport order hash and verifying `params` against it).

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(app_pk) => {
        // Perform equivalent integrity check for legacy variant,
        // or reject if no on-chain verifiable check is possible.
        if !ckd_output_check_legacy(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed (AppPublicKey)");
        }
    }
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
```

### Proof of Concept

1. User calls `request_app_private_key` with `app_public_key: AppPublicKey(user_g1_pk)` and `domain_id` pointing to a BLS12-381 CKD domain. The request is stored in `pending_ckd_requests`.

2. Byzantine attested participant observes the pending request on-chain.

3. Attacker constructs a `CKDResponse` that encrypts a known scalar `k_fake` under `user_g1_pk` (a standard BLS12-381 ElGamal-style encryption the attacker computes locally).

4. Attacker calls `respond_ckd(request, fake_response)` from their attested account. The contract:
   - Passes `assert_caller_is_attested_participant_and_protocol_active()` ✓
   - Enters the `AppPublicKey(_) => {}` arm — **no check** ✓
   - Calls `resolve_yields_for(&mut self.pending_ckd_requests, &request, fake_response_bytes)` ✓

5. The user's yield promise resolves with `fake_response`. The user decrypts it and obtains `k_fake`.

6. The attacker already knows `k_fake` and can derive the same foreign-chain address and sign transactions, stealing any funds sent to that address. [4](#0-3)

```rust
    #[handle_result]
    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }
```
