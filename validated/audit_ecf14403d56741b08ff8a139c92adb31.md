### Title
Plaintext Storage of `PersistentSecrets` (Including Keyshare AES Encryption Key) in `backup-cli` Enables Full Keyshare Disclosure - (File: `crates/backup-cli/src/adapters/secrets_storage.rs`)

---

### Summary

The `backup-cli` tool writes its `PersistentSecrets` struct — which contains the AES-128 key used to encrypt locally stored MPC keyshares, plus two Ed25519 private keys — to a plain JSON file (`secrets.json`) with no encryption at rest. Because the keyshare encryption key and the encrypted keyshare files reside in the same directory, any party with read access to the backup-cli home directory can trivially decrypt the stored MPC keyshares.

---

### Finding Description

`PersistentSecrets` is defined in `crates/backup-cli/src/types.rs` and contains three sensitive fields:

```rust
pub struct PersistentSecrets {
    pub p2p_private_key: SigningKey,       // mTLS identity for MPC node auth
    pub near_signer_key: SigningKey,       // NEAR transaction signing key
    pub local_storage_aes_key: [u8; 16],  // AES-128 key for local keyshare encryption
}
``` [1](#0-0) 

The `store_secrets` method in `JsonSecretsStorage` serializes this struct directly to JSON and writes it to `secrets.json` with no encryption layer:

```rust
pub async fn store_secrets(&mut self, secrets: &types::PersistentSecrets) -> Result<(), Error> {
    let encoded_secrets = serde_json::to_vec(&secrets).map_err(Error::JsonSerialization)?;
    self.destination.seek(std::io::SeekFrom::Start(0)).await...;
    self.destination.write_all(&encoded_secrets).await...;
    Ok(())
}
``` [2](#0-1) 

The `local_storage_aes_key` extracted from `secrets.json` is then passed directly to `KeyshareStorageAdapter::new`, which uses it as the AES-128-GCM key for `LocalPermanentKeyStorageBackend`: [3](#0-2) 

`LocalPermanentKeyStorageBackend` uses this exact 16-byte key to encrypt/decrypt keyshare files stored in the same home directory: [4](#0-3) 

The encrypted keyshare files are written to `$HOME_DIR/permanent_keys/` and a symlink `key` points to the active one: [5](#0-4) 

Both `secrets.json` (containing the AES key in plaintext) and the encrypted keyshare files live under the same `--home-dir`. The encryption is therefore self-defeating: the key and the ciphertext are co-located with no additional protection.

---

### Impact Explanation

The backup-cli stores one MPC node's full set of permanent keyshares — the secret shares for all signing domains (Secp256k1, Ed25519, BLS12-381) for that participant. These are the secret shares used in threshold signing protocols.

An attacker who reads `secrets.json` obtains `local_storage_aes_key` and can immediately decrypt all keyshare files in the same directory. This constitutes **unauthorized access to MPC key shares and secret material**. Additionally, the exposed `p2p_private_key` is the identity registered on-chain for the backup service; an attacker holding it can impersonate the backup service in mTLS connections to MPC nodes and request fresh keyshares from the live node.

**Impact class**: Critical — unauthorized access to MPC key shares / secret material that materially enables secret recovery when combined with shares from other participants (e.g., other backup-cli instances with the same vulnerability, or a low threshold configuration).

---

### Likelihood Explanation

The backup-cli is an operator-managed tool running on a separate machine from the MPC node. The `secrets.json` file is created with standard file permissions and no application-enforced access controls. Any co-located process, backup agent, log collector, or user with read access to the backup-cli home directory can read `secrets.json`. The migration guide explicitly instructs operators to store the backup in a local directory: [6](#0-5) 

The threat model document acknowledges the operator environment as trusted but notes the host OS is **untrusted** (full root access assumed): [7](#0-6) 

Any host-level compromise of the backup-cli machine — which is explicitly outside the TEE boundary — immediately yields all three secrets in `secrets.json` plus the decrypted keyshares.

---

### Recommendation

1. **Short term**: Encrypt `secrets.json` at rest. Derive a file-encryption key from an operator-supplied passphrase (e.g., via Argon2/scrypt) or from a hardware-backed secret (TPM, OS keychain). Never store the AES key that protects keyshares in the same unprotected file as the keyshares' encryption key.

2. **Long term**: Separate the keyshare encryption key from the backup-cli identity keys. Store the keyshare AES key in a hardware-backed store (OS keychain, TPM sealed blob) rather than a plaintext JSON file. The hard-launch design (backup service running inside its own TEE) already addresses this for production; ensure the soft-launch path does not leave a permanent plaintext credential on disk.

### Proof of Concept

```
# 1. Attacker reads the plaintext secrets file
cat $BACKUP_HOME_DIR/secrets.json
# Output: {"p2p_private_key":"<hex>","near_signer_key":"<hex>","local_storage_aes_key":"<16 bytes hex>"}

# 2. Extract local_storage_aes_key
AES_KEY=$(jq -r '.local_storage_aes_key' $BACKUP_HOME_DIR/secrets.json)

# 3. Locate the encrypted keyshare file (symlinked as 'key')
KEYSHARE_FILE=$BACKUP_HOME_DIR/permanent_keys/key

# 4. Decrypt using AES-128-GCM (nonce prepended per db::encrypt convention)
# The LocalPermanentKeyStorageBackend uses Aes128Gcm::new(&aes_key.into())
# and db::decrypt strips the prepended nonce before decrypting.
python3 -c "
from cryptography.hazmat.primitives.ciphers.aead import AESGCM
import sys, json
key = bytes.fromhex('$AES_KEY')
data = open('$KEYSHARE_FILE','rb').read()
nonce, ct = data[:12], data[12:]
pt = AESGCM(key).decrypt(nonce, ct, None)
print(json.loads(pt))   # Full keyshare JSON for this MPC node
"
```

The decrypted output is the full `PermanentKeyshareData` JSON containing the node's secret shares for all signing domains, as defined in: [8](#0-7)

    async fn store(&self, data: &[u8], identifier: &str) -> anyhow::Result<()> {
        let cipher = Aes128Gcm::new(&self.encryption_key.into());
        let encrypted = db::encrypt(&cipher, data);
        // Write the new permanent keyshare to a separate file, and then create a link to it.
        // That way, we don't lose any keys if we somehow mess up.
        let keyfile_for_epoch = self.permanent_key_dir.join(identifier);
        let mut file = tokio::fs::File::create_new(&keyfile_for_epoch)
            .await
            .context("Failed to create PermanentKeyshareData file")?;
        file.write_all(&encrypted)
            .await
            .context("Failed to write PermanentKeyshareData file")?;
        file.sync_all()
            .await
            .context("Failed to sync PermanentKeyshareData file")?;
        drop(file);

        let keyfile = permanent_key_link(&self.home_dir);
        tokio::fs::remove_file(&keyfile).await.ok();
        tokio::fs::hard_link(&keyfile_for_epoch, &keyfile)
            .await
            .context("Failed to link PermanentKeyshareData file")?;
        Ok(())
    }
