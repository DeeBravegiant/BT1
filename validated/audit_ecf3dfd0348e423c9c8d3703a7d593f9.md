Audit Report

## Title
Nested Loops in `getAssetUnstaking` Called Per-Asset Per-NDC in `updateRSETHPrice` Cause Unbounded Gas Consumption - (File: contracts/NodeDelegator.sol)

## Summary
`LRTOracle.updateRSETHPrice()` is a public function that triggers `_getTotalEthInProtocol()`, which iterates over all supported assets and calls `LRTDepositPool.getAssetDistributionData()` per asset. That function iterates over all NDCs and calls `NodeDelegator.getAssetUnstaking()` per NDC, which itself contains two nested loops over EigenLayer queued withdrawals and their strategies. The resulting O(N_assets × M_NDCs × P_withdrawals × Q_strategies) complexity of external calls can cause `updateRSETHPrice()` to exceed the block gas limit, permanently staling the rsETH price.

## Finding Description
The full call chain is confirmed in code:

1. `LRTOracle.updateRSETHPrice()` is `public` and calls `_updateRsETHPrice()` → `_getTotalEthInProtocol()`. [1](#0-0) 

2. `_getTotalEthInProtocol()` loops over every supported asset and calls `ILRTDepositPool.getTotalAssetDeposits(asset)` → `getAssetDistributionData(asset)`. [2](#0-1) 

3. `getAssetDistributionData()` loops over every NDC and calls `INodeDelegator(ndc).getAssetUnstaking(asset)` per NDC. [3](#0-2) 

4. `getETHDistributionData()` independently repeats the same NDC loop calling `getAssetUnstaking(ETH_TOKEN)` per NDC. [4](#0-3) 

5. `getAssetUnstaking()` fetches all queued withdrawals from EigenLayer's `DelegationManager` and runs two nested loops — outer over `queuedWithdrawals.length`, inner over `withdrawal.strategies.length` — making external calls to `strategy.sharesToUnderlyingView()` inside. [5](#0-4) 

**Why the existing cap is insufficient:** `LRTUnstakingVault.setMaxUncompletedWithdrawalCount()` enforces a soft cap of ≤80 total withdrawals across the protocol. [6](#0-5) 

However, `getAssetUnstaking` reads directly from EigenLayer's `DelegationManager.getQueuedWithdrawals(address(this))` with no protocol-level cap applied. The protocol's own comment acknowledges that forced operator undelegations (estimated at `ndc count × asset count = 15`) can add withdrawals outside the cap. Furthermore, the cap is global across all NDCs, not per-NDC, so each NDC can still hold many withdrawals. Each queued withdrawal can also contain multiple strategies, multiplying the inner loop count independently of the cap.

## Impact Explanation
If `updateRSETHPrice()` exceeds the block gas limit, `rsETHPrice` in `LRTOracle` becomes permanently stale. This blocks protocol fee minting, causes deposits and withdrawals to use incorrect exchange rates (wrong rsETH minting and wrong withdrawal amounts), and disables the price-deviation circuit breaker that pauses the protocol on price drops. This matches **Medium — Unbounded gas consumption** from the allowed impact scope.

## Likelihood Explanation
The condition is reachable through normal protocol operation: users initiating unstaking via the public withdrawal flow accumulate queued withdrawals in EigenLayer. With 3 assets, 5 NDCs, 16 withdrawals per NDC, and 3 strategies per withdrawal, a single `updateRSETHPrice()` call already involves 720 external `sharesToUnderlyingView` calls plus 15 `getQueuedWithdrawals` calls, each allocating full withdrawal arrays in memory. Memory expansion costs grow quadratically with array size. No privileged access or attacker action is required — the gas cost grows passively as the protocol scales and withdrawals accumulate. Forced undelegations by EigenLayer operators (an acknowledged scenario in the codebase) can push counts beyond the soft cap without any protocol action.

## Recommendation
1. **Cache `getQueuedWithdrawals` once per NDC** and reuse the result across all asset queries, eliminating per-asset re-fetches of the same withdrawal array.
2. **Maintain a running TVL accumulator** updated on each deposit/withdrawal event ("push" pattern), so `updateRSETHPrice()` reads a single cached value per asset rather than iterating on-chain.
3. **Enforce a per-NDC withdrawal count cap** at the `NodeDelegator` level (not just a global total in `LRTUnstakingVault`) to bound the inner loop regardless of forced undelegations.
4. Alternatively, supply TVL via an off-chain oracle, removing on-chain iteration entirely.

## Proof of Concept
**Minimal call sequence demonstrating the gas explosion (N=3 assets, M=5 NDCs, P=16 withdrawals/NDC, Q=3 strategies/withdrawal):**

```
updateRSETHPrice()                                    // public, no auth
└── _getTotalEthInProtocol()
    └── [loop: 3 assets]
        └── getTotalAssetDeposits(asset)
            └── getAssetDistributionData(asset)
                └── [loop: 5 NDCs]
                    └── getAssetUnstaking(asset)      // external call per NDC per asset
                        └── DelegationManager.getQueuedWithdrawals(ndc)
                            └── [loop: 16 withdrawals]
                                └── [loop: 3 strategies]
                                    └── strategy.sharesToUnderlyingView(shares)  // external call
```

Total `sharesToUnderlyingView` calls: 3 × 5 × 16 × 3 = **720**
Total `getQueuedWithdrawals` calls: 3 × 5 = **15** (each allocating full withdrawal arrays)

A Foundry fork test can reproduce this by: (1) deploying the protocol against a mainnet fork, (2) queuing 16 withdrawals per NDC across 5 NDCs via the public unstaking flow, (3) calling `updateRSETHPrice()` and measuring gas with `vm.expectRevert` or `gasleft()` assertions to confirm the call approaches or exceeds the 30M block gas limit.

**File:** contracts/NodeDelegator.sol (L405-427)

```text
    function getAssetUnstaking(address asset) external view returns (uint256 amount) {
        (IDelegationManager.Withdrawal[] memory queuedWithdrawals, uint256[][] memory withdrawalShares) =
            _getDelegationManager().getQueuedWithdrawals(address(this));

        for (uint256 withdrawalIndex = 0; withdrawalIndex < queuedWithdrawals.length; withdrawalIndex++) {
            IDelegationManager.Withdrawal memory withdrawal = queuedWithdrawals[withdrawalIndex];

            for (uint256 strategyIndex = 0; strategyIndex < withdrawal.strategies.length; strategyIndex++) {
                IStrategy strategy = withdrawal.strategies[strategyIndex];

                address strategyAsset = address(strategy) == address(lrtConfig.beaconChainETHStrategy())
                    ? LRTConstants.ETH_TOKEN
                    : address(strategy.underlyingToken());

                if (strategyAsset != asset) continue;

                uint256 sharesToUnstake = withdrawalShares[withdrawalIndex][strategyIndex];
                amount += strategyAsset == LRTConstants.ETH_TOKEN
                    ? sharesToUnstake
                    : strategy.sharesToUnderlyingView(sharesToUnstake);
            }
        }
    }
```
