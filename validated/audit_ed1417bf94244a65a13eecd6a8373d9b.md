Audit Report

## Title
ETH Deposit Limit Check Omits Incoming Amount, Allowing One-Deposit Bypass — (File: contracts/LRTDepositPool.sol)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` uses `totalAssetDeposits > limit` for ETH but `totalAssetDeposits + amount > limit` for ERC20 assets. The missing `+ amount` in the ETH branch means the guard only fires when the cap is already exceeded, not when the incoming deposit would push it over. Any unprivileged depositor can breach the ETH deposit cap by exactly one deposit.

## Finding Description
In `_checkIfDepositAmountExceedesCurrentLimit` ( [1](#0-0) ):

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)); // ← no `+ amount`
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // ← correct
}
```

`getTotalAssetDeposits` reflects the pre-deposit state. For ETH, when `totalAssetDeposits == limit`, the expression `totalAssetDeposits > limit` evaluates to `false`, so no revert is triggered. The deposit proceeds through `_beforeDeposit` ( [2](#0-1) ) and `_mintRsETH` is called ( [3](#0-2) ), pushing the total to `limit + msg.value`. No other guard exists to prevent this.

## Impact Explanation
The deposit limit is the protocol's sole mechanism for capping ETH exposure. The bypass breaks the invariant `getTotalAssetDeposits(ETH_TOKEN) ≤ depositLimitByAsset(ETH_TOKEN)`. No funds are stolen or frozen, but the contract fails to enforce its advertised deposit cap. This matches the allowed impact: **Low — Contract fails to deliver promised returns, but doesn't lose value.**

## Likelihood Explanation
The condition is trivially reachable by any unprivileged caller of the public `depositETH` function. No special timing, front-running, privileged access, or victim mistake is required. The attacker only needs to call `depositETH` when cumulative ETH deposits equal the configured limit, which is a naturally occurring state as the pool fills.

## Recommendation
Apply the same `+ amount` inclusion to the ETH branch:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

## Proof of Concept
1. Admin sets ETH deposit limit to `100 ether` via `updateAssetDepositLimit`.
2. Cumulative ETH deposits reach exactly `100 ether` (`getTotalAssetDeposits(ETH_TOKEN) == 100 ether`).
3. Alice calls `depositETH{value: 1 ether}(minRSETH, "")`.
4. `_checkIfDepositAmountExceedesCurrentLimit` evaluates `100 ether > 100 ether` → `false` → no revert.
5. `_mintRsETH` mints rsETH for Alice; total ETH deposits become `101 ether`.
6. `MaximumDepositLimitReached` is never triggered despite the cap being breached.

**Foundry test sketch:**
```solidity
function test_ethDepositLimitBypass() public {
    uint256 limit = 100 ether;
    lrtConfig.updateAssetDepositLimit(LRTConstants.ETH_TOKEN, limit);
    // fill pool to exactly the limit
    vm.deal(address(this), limit + 1 ether);
    depositPool.depositETH{value: limit}(0, "");
    // this should revert but does not
    depositPool.depositETH{value: 1 ether}(0, "");
    assertGt(depositPool.getTotalAssetDeposits(LRTConstants.ETH_TOKEN), limit);
}
```
