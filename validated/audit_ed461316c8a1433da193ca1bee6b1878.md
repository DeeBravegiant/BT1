### Title
Missing Contract-Identity Binding in `ForeignTxSignPayload` Enables Cross-Deployment Signature Replay - (File: `crates/near-mpc-contract-interface/src/types/foreign_chain.rs`)

---

### Summary

`ForeignTxSignPayload::compute_msg_hash()` hashes only `(ForeignChainRpcRequest, Vec<ExtractedValue>)`. It omits the NEAR contract account ID, NEAR network/chain ID, and domain ID. Because the MPC network signs this hash with the root key (zero tweak), a valid `respond_verify_foreign_tx` signature produced for one contract deployment is cryptographically identical to a valid signature for any other deployment that shares the same key material. A single Byzantine MPC node (below the signing threshold) can replay a legitimately-produced signature from one contract instance onto another, causing the target contract to accept a forged foreign-chain verification attestation.

---

### Finding Description

`ForeignTxSignPayloadV1` is defined as:

```rust
pub struct ForeignTxSignPayloadV1 {
    pub request: ForeignChainRpcRequest,
    pub values: Vec<ExtractedValue>,
}
```

The hash that nodes sign is:

```
msg_hash = SHA-256(borsh(ForeignTxSignPayload))
``` [1](#0-0) 

The signed payload contains no binding to:
- The NEAR contract account ID (`env::current_account_id()`)
- The NEAR network/chain ID
- The `domain_id` of the request
- Any per-request nonce or receipt ID

The node-side `build_signature_request` explicitly sets a **zero tweak**, meaning the signature is produced under the raw root key with no key derivation:

```rust
Ok(SignatureRequest {
    ...
    tweak: Tweak::new([0u8; 32]),
    ...
})
``` [2](#0-1) 

On the contract side, `respond_verify_foreign_tx` verifies the signature against the root public key for the domain and then resolves the pending yield: [3](#0-2) 

The pending-request map key is `VerifyForeignTransactionRequest`, which contains `(ForeignChainRpcRequest, domain_id, payload_version)` — also no contract-identity binding: [4](#0-3) 

---

### Impact Explanation

**Scenario — cross-deployment replay during contract migration:**

NEAR MPC contracts are upgraded by deploying a new contract account while preserving key material through resharing (resharing does not change the underlying key). If `v1.mpc.near` and `v2.mpc.near` share the same ForeignTx domain key:

1. Attacker submits `verify_foreign_transaction(bitcoin_tx_id=X)` to `v2.mpc.near` (target bridge contract).
2. The same request is processed on `v1.mpc.near` (attacker submits it there, or it was processed legitimately).
3. The MPC network produces `sig = ECDSA_sign(root_key, SHA256(borsh({request: Bitcoin(X), values: [block_hash_Y]})))`.
4. A single Byzantine MPC node (below signing threshold) calls `respond_verify_foreign_tx` on `v2.mpc.near` with the same `(request, response)` pair.
5. `v2.mpc.near` verifies `sig` against its root public key — the signature is valid because the payload hash is identical (no contract-address binding). The pending yield is resolved.
6. The bridge contract on `v2.mpc.near` receives a valid attestation for Bitcoin tx X and releases funds — a double-spend.

This matches the allowed impact: **High — forged foreign-chain verification causing invalid bridge execution or double-spend conditions.**

---

### Likelihood Explanation

- **Byzantine node below threshold**: Only one attested MPC participant needs to be Byzantine to call `respond_verify_foreign_tx`. The signing threshold (e.g., t-of-n) applies to the off-chain MPC protocol, not to the on-chain `respond` call. A single compromised node is sufficient.
- **Shared key across deployments**: Key resharing (which preserves the key) is a documented, expected operation in this codebase. The `ResharingContractState` is a first-class protocol state.
- **Attacker-controlled submission**: Submitting `verify_foreign_transaction` to the target contract costs only 1 yoctoNEAR and is open to any caller.
- **No on-chain deduplication across contracts**: The pending-request map is per-contract; there is no cross-contract replay guard.

---

### Recommendation

Include the NEAR contract account ID and NEAR network identifier in the signed payload. Extend `ForeignTxSignPayloadV1` with a domain-separation prefix:

```rust
pub struct ForeignTxSignPayloadV1 {
    pub contract_id: AccountId,   // env::current_account_id() at request time
    pub near_chain_id: String,    // e.g. "mainnet" / "testnet"
    pub domain_id: DomainId,      // the signing domain
    pub request: ForeignChainRpcRequest,
    pub values: Vec<ExtractedValue>,
}
```

Alternatively, include the NEAR receipt ID (already available in the node-side `VerifyForeignTxRequest.receipt_id`) in the signed payload, since receipt IDs are globally unique and contract-scoped.

### Proof of Concept

**Step 1.** Deploy two contracts `v1.mpc.near` and `v2.mpc.near` sharing the same ForeignTx domain key (as occurs during a migration resharing).

**Step 2.** Submit to `v1.mpc.near`:
```json
{ "request": { "Bitcoin": { "tx_id": "AAAA...AA", "confirmations": 1, "extractors": ["BlockHash"] } }, "domain_id": 0, "payload_version": 1 }
```

**Step 3.** MPC nodes process the request and produce:
```
payload = ForeignTxSignPayload::V1 { request: Bitcoin(AAAA..AA), values: [block_hash_BBBB] }
msg_hash = SHA256(borsh(payload))
sig = ECDSA(root_key, msg_hash)
```

**Step 4.** Submit the identical `verify_foreign_transaction` request to `v2.mpc.near` (costs 1 yoctoNEAR).

**Step 5.** Byzantine node calls on `v2.mpc.near`:
```json
respond_verify_foreign_tx(
  request = { "Bitcoin": { "tx_id": "AAAA...AA", ... }, "domain_id": 0, "payload_version": 1 },
  response = { "payload_hash": "<msg_hash from step 3>", "signature": "<sig from step 3>" }
)
```

**Step 6.** `v2.mpc.near` verifies `sig` against its root public key — passes, because `ForeignTxSignPayload` contains no contract-identity binding. The pending yield on `v2.mpc.near` is resolved with a forged attestation. [5](#0-4) [6](#0-5) [7](#0-6)

    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
