Audit Report

## Title
`checkDailyMintLimit` Modifier Blocks All Minting When `maxMintAmountPerDay` Is Zero — (File: `contracts/RSETH.sol`)

## Summary
The `checkDailyMintLimit` modifier in `RSETH.sol` lacks a guard for `maxMintAmountPerDay == 0`. When this variable holds its default post-initialization value of zero, the condition `currentPeriodMintedAmount + amount > maxMintAmountPerDay` reduces to `amount > 0`, causing every non-zero mint to revert with `DailyMintLimitExceeded`. Since `reinitialize()` does not set `maxMintAmountPerDay`, the protocol's deposit functionality is non-operational until the manager explicitly calls `setMaxMintAmountPerDay()` with a non-zero value. Transactions revert atomically, so no user funds are transferred to or frozen in the protocol — the protocol simply fails to accept deposits.

## Finding Description
After `RSETH.initialize()` is called, `maxMintAmountPerDay` is `0` (Solidity default). The `reinitialize()` function sets only `periodStartTime` and `custodyAddress`, leaving `maxMintAmountPerDay` at zero. [1](#0-0) 

The `checkDailyMintLimit` modifier performs the check without any zero-value guard: [2](#0-1) 

When `maxMintAmountPerDay == 0` and `amount > 0`, the expression `currentPeriodMintedAmount + amount > 0` evaluates to `true`, unconditionally reverting. The `mint()` function applies this modifier, so every call to `RSETH.mint()` reverts in this state: [3](#0-2) 

The full deposit path is: `depositETH`/`depositAsset` → `_mintRsETH` → `IRSETH(rsethToken).mint(msg.sender, rsethAmountToMint)` → `checkDailyMintLimit` → revert. [4](#0-3) 

The inconsistency is confirmed by `remainingDailyMintLimit()`, which explicitly guards against zero with `if (maxMintAmountPerDay == 0) return 0;`, demonstrating the developers treat `0` as a distinct meaningful state — but the modifier does not mirror this guard: [5](#0-4) 

`setMaxMintAmountPerDay()` accepts `0` without validation, allowing the manager to inadvertently re-enable the blocked state: [6](#0-5) 

The identical pattern exists in `_checkAndUpdateDailyFeeMintLimit` in `LRTOracle.sol`, which would cause `updateRSETHPrice()` to revert when fee minting is attempted with `maxFeeMintAmountPerDay == 0`: [7](#0-6) 

## Impact Explanation
The deposit functionality is completely non-operational when `maxMintAmountPerDay == 0`. Users calling `depositETH` or `depositAsset` receive a revert. Because Solidity reverts all state changes atomically — including the `safeTransferFrom` in `depositAsset` that precedes `_mintRsETH` — no user funds are transferred to or retained by the protocol. The user's ETH/LST remains in their wallet. The protocol fails to deliver its core promised service (accepting deposits and minting rsETH), but no funds are lost or frozen in the contract.

**Impact: Low — Contract fails to deliver promised returns, but doesn't lose value.**

The submitted claim of "Medium — Temporary freezing of funds" is not supported: funds are never transferred into the protocol before the revert, so nothing is frozen. The correct classification is Low.

## Likelihood Explanation
- `maxMintAmountPerDay` is `0` by default after `initialize()` and remains `0` after `reinitialize()`.
- There is a deployment window between `reinitialize()` and a subsequent `setMaxMintAmountPerDay()` call during which all deposits revert.
- `setMaxMintAmountPerDay()` accepts `0` with no validation, so a manager can inadvertently re-enable the broken state at any time.
- No attacker action is required; the condition arises from the default contract state.

## Recommendation
Add a zero-value guard in `checkDailyMintLimit` to treat `maxMintAmountPerDay == 0` as "no limit" (consistent with `remainingDailyMintLimit()`'s intent), or add input validation in `setMaxMintAmountPerDay` to reject `0`. Apply the same fix to `_checkAndUpdateDailyFeeMintLimit` in `LRTOracle.sol` for `maxFeeMintAmountPerDay`.

```solidity
modifier checkDailyMintLimit(uint256 amount) {
    if (maxMintAmountPerDay != 0) {
        if (block.timestamp >= periodStartTime + 1 days) {
            currentPeriodMintedAmount = 0;
            periodStartTime = getCurrentPeriodStartTime();
        }
        if (currentPeriodMintedAmount + amount > maxMintAmountPerDay) {
            revert DailyMintLimitExceeded(currentPeriodMintedAmount + amount, maxMintAmountPerDay);
        }
        currentPeriodMintedAmount += amount;
    }
    _;
}
```

## Proof of Concept
1. Deploy `RSETH` and call `initialize(admin, lrtConfig)`. `maxMintAmountPerDay` is `0`.
2. Call `reinitialize(periodStartTime, custodyAddress)`. `maxMintAmountPerDay` remains `0`.
3. As a user, call `LRTDepositPool.depositETH{value: 1 ether}(0, "")`.
4. Execution reaches `IRSETH(rsethToken).mint(msg.sender, rsethAmountToMint)`.
5. Inside `checkDailyMintLimit`: `0 + rsethAmountToMint > 0` → `true` → reverts with `DailyMintLimitExceeded(rsethAmountToMint, 0)`.
6. The entire transaction reverts atomically; the user's ETH is returned. The `_mint(to, amount)` call is never reached.
7. Repeat for `depositAsset`: the `safeTransferFrom` at line 114 also reverts, returning the LST to the user.
