Audit Report

## Title
Stale Cached `rsETHPrice` Used in Deposit Minting Allows Dilution of Accrued Yield — (`contracts/LRTOracle.sol` / `contracts/LRTDepositPool.sol`)

## Summary

`LRTOracle` stores `rsETHPrice` as a persistent cached state variable updated only when `updateRSETHPrice()` is explicitly called. `LRTDepositPool.depositETH()` and `depositAsset()` compute the rsETH mint amount using this stale cached price without refreshing it first. Because `updateRSETHPrice()` is a permissionless `public` function, any depositor can deposit at the stale (lower) price and then trigger or wait for the price update, receiving more rsETH than their deposit warrants at the true rate — extracting accrued yield from existing rsETH holders.

## Finding Description

`LRTOracle` stores `rsETHPrice` as a public state variable: [1](#0-0) 

This value is only refreshed inside `_updateRsETHPrice()`, which is triggered by the permissionless `public` function `updateRSETHPrice()`: [2](#0-1) 

`LRTDepositPool.getRsETHAmountToMint()` divides by the cached `lrtOracle.rsETHPrice()` to determine how many rsETH tokens to mint: [3](#0-2) 

Both `depositETH()` and `depositAsset()` route through `_beforeDeposit()` → `getRsETHAmountToMint()` without ever calling `updateRSETHPrice()` first: [4](#0-3) [5](#0-4) [6](#0-5) 

Between oracle updates, EigenLayer staking rewards and LST rebasing cause the true ETH value of the protocol to grow. `_updateRsETHPrice()` computes the new price from live balances via `_getTotalEthInProtocol()`: [7](#0-6) 

Until `updateRSETHPrice()` is called, `rsETHPrice` remains below its true value. Any deposit made during this window uses the artificially low denominator, minting more rsETH than the deposit is worth at the true rate.

**Regarding the `pricePercentageLimit` guard:** `_updateRsETHPrice()` contains a check that reverts for non-managers if the price increase from `highestRsethPrice` exceeds `pricePercentageLimit`: [8](#0-7) 

This guard does **not** neutralize the vulnerability for two independent reasons:
1. If `pricePercentageLimit == 0` (the check is `pricePercentageLimit > 0 && ...`), the guard is entirely bypassed and the attacker can atomically deposit + update in one transaction.
2. Even when the guard is active and the price gap exceeds the limit, the attacker can still deposit at the stale price and simply wait for the authorized keeper to call `updateRSETHPrice()`. The over-minted rsETH is already in the attacker's wallet; the profit is realized at the next keeper update regardless of who triggers it.

## Impact Explanation

**High — Theft of unclaimed yield.**

When `rsETHPrice` is stale and lower than the true value, a depositor receives more rsETH than their deposit warrants. After the price is updated (by anyone, including the attacker when `pricePercentageLimit == 0`, or by the keeper otherwise), the attacker's rsETH is redeemable for more ETH than they deposited. The surplus comes directly from the accrued staking rewards that belong to existing rsETH holders, whose proportional share of the pool is permanently diluted. This matches the allowed impact: **High — Theft of unclaimed yield**.

## Likelihood Explanation

The `rsETHPrice` is updated by off-chain keepers on a periodic schedule, not on every block. EigenLayer staking rewards and LST rebasing accrue continuously, creating a predictable, observable staleness window. An attacker requires no privileged access. The two attack variants are:

- **Atomic (when `pricePercentageLimit == 0` or price gap is within limit):** Monitor `rsETHPrice` on-chain vs. live TVL from `_getTotalEthInProtocol()`. When a meaningful gap exists, take a flash loan, call `depositETH()` at the stale price, then call `updateRSETHPrice()` in the same transaction.
- **Non-atomic (always available):** Deposit at the stale price and hold rsETH until the keeper updates the price. No flash loan required; capital cost is the deposit amount held for one keeper cycle.

Both variants are executable by any EOA or contract with no privileged access.

## Recommendation

Call `updateRSETHPrice()` (or an internal equivalent) at the start of `depositETH()` and `depositAsset()`, before computing `rsethAmountToMint`, so that the mint calculation always uses the current true price:

```solidity
// In depositETH() and depositAsset(), before _beforeDeposit():
ILRTOracle(lrtConfig.getContract(LRTConstants.LRT_ORACLE)).updateRSETHPrice();
uint256 rsethAmountToMint = _beforeDeposit(asset, depositAmount, minRSETHAmountExpected);
```

This ensures every deposit mints rsETH at the live price, eliminating the staleness window entirely.

## Proof of Concept

**Setup:** `rsETHPrice` = 1.00 ETH (stale; true value = 1.05 ETH due to accrued rewards). Total rsETH supply = 100,000. Total ETH in protocol (true) = 105,000 ETH. `pricePercentageLimit == 0` (guard bypassed).

**Step 1 — Attacker deposits using stale price (flash loan of 100,000 ETH):**
```
depositETH{value: 100,000 ETH}(minRSETHAmountExpected, "")
→ getRsETHAmountToMint: (100,000e18 * 1e18) / 1.00e18 = 100,000 rsETH minted
Correct at true price: 100,000 / 1.05 ≈ 95,238 rsETH
Excess minted: ~4,762 rsETH
```

**Step 2 — Attacker triggers price update (same tx or next keeper cycle):**
```
LRTOracle.updateRSETHPrice()
→ totalETHInProtocol = 105,000 + 100,000 = 205,000 ETH
→ rsethSupply = 100,000 + 100,000 = 200,000
→ newRsETHPrice = 205,000 / 200,000 = 1.025 ETH
```

**Step 3 — Attacker's position:**
- Attacker holds 100,000 rsETH × 1.025 = **102,500 ETH value**
- Attacker deposited 100,000 ETH → **profit ≈ 2,500 ETH**
- Original 100,000 rsETH holders: their rsETH is now worth 102,500 ETH instead of 105,000 ETH — **2,500 ETH of accrued yield stolen**

**Foundry fork test plan:** Fork mainnet, set `rsETHPrice` to a value below the live TVL-derived price, call `depositETH` with a large value, call `updateRSETHPrice`, assert that the attacker's rsETH value exceeds their deposit and that existing holders' rsETH value decreased relative to the pre-deposit true price.
