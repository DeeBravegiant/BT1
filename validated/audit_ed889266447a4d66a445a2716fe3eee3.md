Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Checks `owner` Instead of Actual Caller, Allowing Allowlist Bypass — (File: metric-periphery/contracts/extensions/DepositAllowlistExtension.sol)

## Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards its first parameter (the actual caller) and gates access solely on the `owner` parameter. Because `MetricOmmPool.addLiquidity` imposes no `msg.sender == owner` constraint, any unprivileged address can supply an allowlisted address as `owner`, pass the extension check, pay tokens themselves via the callback, and credit LP shares to the allowlisted address — fully bypassing the pool admin's deposit restriction.

## Finding Description

`MetricOmmPool.addLiquidity` passes `msg.sender` as the first argument and the caller-supplied `owner` as the second argument to the extension hook: [1](#0-0) 

Unlike `removeLiquidity`, which enforces `msg.sender == owner` at line 206, `addLiquidity` has no such guard: [2](#0-1) 

Inside `LiquidityLib.addLiquidity`, the token-pull callback is invoked on `msg.sender` (the actual payer): [3](#0-2) 

While LP shares are credited to `owner` via `positionBinShares[posKey]`, where `posKey = keccak256(owner, salt, bin)`: [4](#0-3) 

`DepositAllowlistExtension.beforeAddLiquidity` receives the actual caller as its first (unnamed, discarded) parameter and checks only `owner` against the allowlist: [5](#0-4) 

The extension's NatSpec states it "Gates `addLiquidity` by depositor address" — but the depositor is `msg.sender`, not `owner`. The first parameter is unnamed and unused, so the gate is evaluated against the wrong identity. An attacker not on the allowlist calls `pool.addLiquidity(allowlisted_victim, salt, deltas, ...)`, the extension checks `allowedDepositor[pool][alice]` → `true` → passes, and the attacker pays tokens while the victim receives LP shares.

## Impact Explanation

The `DepositAllowlistExtension` is the pool admin's mechanism to restrict which addresses may deposit into a permissioned pool. The bypass allows any unprivileged address to inject tokens and credit LP shares to an allowlisted address, directly violating the admin-configured access boundary. This matches the allowed impact: **admin-boundary break — factory/pool-admin role checks bypassed by an unprivileged path**. Secondary effects include the allowlisted victim receiving unwanted LP shares in positions they did not initiate, and potential manipulation of per-bin `token0BalanceScaled`/`token1BalanceScaled` values affecting proportional redemption for other LPs in the same bin. [6](#0-5) 

## Likelihood Explanation

Medium. The precondition is that the pool has `DepositAllowlistExtension` registered as a `beforeAddLiquidity` hook and at least one allowlisted address is publicly known (observable from on-chain `AllowedToDepositSet` events). The attacker must hold enough tokens to fund the callback but gains no financial benefit — the attack is a griefing/compliance bypass. Both preconditions are observable on-chain and the call path is straightforward.

## Recommendation

`DepositAllowlistExtension.beforeAddLiquidity` should check the actual caller (first parameter) rather than `owner`:

```solidity
function beforeAddLiquidity(address caller, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][caller]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

Alternatively, `MetricOmmPool.addLiquidity` should enforce `msg.sender == owner` (mirroring `removeLiquidity`) when a restrictive extension is active, ensuring the payer and position owner are always the same address.

## Proof of Concept

1. Pool is deployed with `DepositAllowlistExtension` as a `beforeAddLiquidity` hook.
2. Pool admin calls `setAllowedToDeposit(pool, alice, true)`. Bob is not allowlisted.
3. Bob (not allowlisted) calls:
   ```solidity
   pool.addLiquidity(alice, salt, deltas, callbackData, extensionData);
   ```
4. Pool calls `_beforeAddLiquidity(bob, alice, salt, deltas, extensionData)` → extension receives `(bob, alice, ...)`.
5. Extension discards `bob` (unnamed first param) and checks `allowedDepositor[pool][alice]` → `true` → passes.
6. `LiquidityLib.addLiquidity` calls `IMetricOmmModifyLiquidityCallback(bob).metricOmmModifyLiquidityCallback(...)` — Bob pays tokens.
7. `positionBinShares[keccak256(alice, salt, bin)]` is incremented — Alice receives LP shares.
8. Bob, not on the allowlist, has successfully deposited into the restricted pool, violating the admin's access control invariant. [7](#0-6) [5](#0-4)
