### Title
Integer Division in `compute_royalty_amount` Allows NFT Offer Settlement with Zero Royalty Payment - (File: `chia/wallet/nft_wallet/nft_wallet.py`)

### Summary

The `compute_royalty_amount` function uses sequential integer (floor) division to compute NFT royalty payments. When the offered fungible amount is small relative to the royalty percentage denominator (10,000 basis points), the computed royalty truncates to zero. The wallet's offer-construction code in `make_nft1_offer` then silently skips the royalty coin spend entirely, allowing an NFT to be acquired through the offer protocol with zero royalty paid to the creator.

### Finding Description

`compute_royalty_amount` in `chia/wallet/nft_wallet/nft_wallet.py` computes the royalty as:

```python
amount = abs(offered_amount) // royalty_split * percentage // MAX_ROYALTY_BASIS_POINTS
``` [1](#0-0) 

`MAX_ROYALTY_BASIS_POINTS` is 10,000. When `abs(offered_amount) // royalty_split * percentage < 10,000`, the result is 0. The test suite explicitly acknowledges this truncation: [2](#0-1) 

In `make_nft1_offer`, this function is called to populate `royalty_payments`: [3](#0-2) 

Later, when constructing the offer bundle, if the total royalty payment sum is zero, the royalty coin spend is **skipped entirely** with `continue`: [4](#0-3) 

Additionally, no royalty coin is created in the first place when `payment_sum == 0`: [5](#0-4) 

**Concrete example:**
- NFT has 1% royalty (100 basis points)
- Attacker creates offer: offer 99 mojos XCH, request 1 NFT
- `compute_royalty_amount(-99, 1, 100)` = `99 * 100 // 10000` = `9900 // 10000` = **0**
- Royalty payment = 0 mojos; offer bundle contains no royalty coin spend
- NFT is transferred; creator receives 0 royalty

The same double-floor-division pattern also appears in the `royalty_calculation` static method used for display: [6](#0-5) 

### Impact Explanation

An unprivileged attacker can systematically acquire royalty-bearing NFTs through the standard offer protocol without paying any royalty to the NFT creator. The royalty payout — which is part of the contractual offer/trade settlement — is silently diverted away from the creator. This constitutes both a payout redirection and corruption of offer/trade settlement state. The NFT creator suffers direct financial loss with no on-chain recourse, since the offer bundle accepted by the network contains no royalty output.

### Likelihood Explanation

Any user can craft an offer with a small fungible amount. The threshold below which royalty rounds to zero is `offered_amount < 10000 / percentage` mojos. For a 1% royalty NFT, any offer below 9,999 mojos (less than 0.00001 XCH) pays zero royalty. For a 0.01% royalty NFT (1 basis point), any offer below 999,999 mojos pays zero royalty. These thresholds are easily reachable in normal trading. The `make_nft1_offer` path is triggered for all NFT1 offers via `check_for_special_offer_making` → `make_nft1_offer`. [7](#0-6) 

### Recommendation

Replace the sequential floor-division with a ceiling-division or a minimum-of-1 guard when the royalty percentage is non-zero:

```python
# Option A: ceiling division
import math
amount = math.ceil(abs(offered_amount) * percentage / (royalty_split * MAX_ROYALTY_BASIS_POINTS))

# Option B: enforce minimum 1 mojo when percentage > 0
amount = abs(offered_amount) // royalty_split * percentage // MAX_ROYALTY_BASIS_POINTS
if percentage > 0 and amount == 0:
    raise ValueError("Offered amount too small to satisfy non-zero royalty")
```

Apply the same fix to `royalty_calculation` at line 719.

### Proof of Concept

1. Mint an NFT with 1% royalty (100 basis points) via `generate_new_nft`.
2. As attacker, call `make_nft1_offer` with `offer_dict = {None: -99, nft_asset_id: 1}` (offer 99 mojos XCH, request the NFT).
3. Observe: `compute_royalty_amount(-99, 1, 100)` returns `uint64(0)`.
4. Observe: `payment_sum == 0` → `continue` at line 911 skips royalty coin spend.
5. The resulting offer bundle contains no royalty output coin.
6. NFT seller accepts the offer; NFT is transferred on-chain.
7. NFT creator's royalty address receives 0 mojos despite the 1% royalty setting. [1](#0-0) [8](#0-7)

def compute_royalty_amount(offered_amount: int, royalty_split: int, percentage: int) -> uint64:
    """Compute royalty using integer arithmetic, validating against overflow and excessive percentage."""
    if percentage > MAX_ROYALTY_BASIS_POINTS:
        raise ValueError(f"NFT royalty percentage {percentage} exceeds 100% ({MAX_ROYALTY_BASIS_POINTS} basis points)")
    amount = abs(offered_amount) // royalty_split * percentage // MAX_ROYALTY_BASIS_POINTS
    royalty = uint64(amount)
    if royalty >= abs(offered_amount):
        raise ValueError("Royalty amount meets or exceeds the offered amount")
    return royalty
```
