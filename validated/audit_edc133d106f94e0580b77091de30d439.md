Audit Report

## Title
Zero `rsETHAmount` Deposit Proceeds Without Revert, Consuming User Funds for Zero wrsETH — (`contracts/pools/RSETHPoolV3.sol`)

## Summary

All L2 pool deposit functions guard only against a zero input `amount` but never check whether the computed `rsETHAmount` is zero after rate conversion. Because `viewSwapRsETHAmountAndFee` uses integer division, any dust deposit where `amountAfterFee * 1e18 < rsETHToETHrate` (ETH path) or `amountAfterFee * tokenToETHRate < rsETHToETHrate` (token path) silently truncates `rsETHAmount` to zero. The contract then accepts the user's ETH or tokens, mints or transfers zero wrsETH/rsETH to the user, and emits a misleading `SwapOccurred` event — the depositor permanently loses their assets with no recovery path.

## Finding Description

Every pool deposit function follows the same pattern. In `RSETHPoolV3.sol`:

```solidity
// L256
if (amount == 0) revert InvalidAmount();   // only guards zero input

// L258
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
// No guard: if (rsETHAmount == 0) revert InvalidAmount();

// L262
wrsETH.mint(msg.sender, rsETHAmount);      // mints 0 — no-op
```

The rate conversion truncates to zero via integer division:

```solidity
// L307
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;   // truncates when amountAfterFee * 1e18 < rsETHToETHrate
// L334
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;  // same for token path
```

Since `rsETHToETHrate` is always slightly above `1e18` (rsETH appreciates over time), even a 1-wei ETH deposit satisfies the truncation condition. The same root cause is confirmed in all five pool contracts:

- `RSETHPoolV2.sol` L210, L233
- `RSETHPoolV3.sol` L256, L307, L334
- `RSETHPoolV3ExternalBridge.sol` L375, L426, L452
- `RSETHPoolV3WithNativeChainBridge.sol` L292, L343, L370
- `RSETHPoolNoWrapper.sol` L235, L285, L311 (uses `rsETH.safeTransfer(msg.sender, 0)` — also a no-op)

No subsequent guard exists in any of these paths to revert when `rsETHAmount == 0`.

## Impact Explanation

**Low — Contract fails to deliver promised returns.**

When `rsETHAmount` truncates to zero, the deposit function completes successfully: ETH is absorbed into the pool balance (or tokens are pulled via `safeTransferFrom`), fee accounting is updated, zero wrsETH/rsETH is issued to the depositor, and a `SwapOccurred` event is emitted with `rsETHAmount = 0`. There is no depositor-facing withdrawal or refund path. The depositor receives nothing for their deposit — the contract fails to deliver its core promised return (wrsETH/rsETH in exchange for ETH/tokens).

## Likelihood Explanation

No special permissions, oracle manipulation, or privileged access are required. Any unprivileged external caller can trigger this by sending a dust ETH deposit (e.g., 1 wei) directly to `deposit()`. Since `rsETHToETHrate > 1e18` always holds (rsETH only appreciates), the condition `1 * 1e18 / rsETHToETHrate = 0` is always satisfied for a 1-wei deposit. For token deposits, the zero-output threshold scales with `rsETHToETHrate / tokenToETHRate`; tokens with low ETH value have a proportionally larger threshold. UI rounding errors or programmatic integrations sending small amounts can reach this path without any adversarial intent.

## Recommendation

Add a zero-output guard immediately after computing `rsETHAmount` in every pool deposit function, before any state change:

```solidity
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
if (rsETHAmount == 0) revert InvalidAmount();
```

Apply this fix to all five pool contracts: `RSETHPoolV2.sol`, `RSETHPoolV3.sol`, `RSETHPoolV3ExternalBridge.sol`, `RSETHPoolV3WithNativeChainBridge.sol`, and `RSETHPoolNoWrapper.sol`.

## Proof of Concept

Deploy `RSETHPoolV3` with `feeBps = 0` and mock oracle returning `rsETHToETHrate = 1.05e18`:

1. Call `deposit{value: 1}("")` (1 wei ETH).
2. `viewSwapRsETHAmountAndFee(1)`:
   - `fee = 1 * 0 / 10_000 = 0`
   - `amountAfterFee = 1`
   - `rsETHAmount = 1 * 1e18 / 1.05e18 = 0` (integer truncation)
3. `feeEarnedInETH += 0`; pool balance increases by 1 wei.
4. `wrsETH.mint(msg.sender, 0)` — no-op; user holds 0 wrsETH.
5. `SwapOccurred` emitted with `rsETHAmount = 0`.
6. User's 1 wei is permanently in the pool; no depositor recovery path exists.

**Foundry fuzz test sketch:**

```solidity
function testFuzz_zeroRsETHAmountDeposit(uint256 amount) public {
    vm.assume(amount > 0 && amount < rsETHToETHrate / 1e18); // dust range
    uint256 balanceBefore = wrsETH.balanceOf(user);
    vm.deal(user, amount);
    vm.prank(user);
    pool.deposit{value: amount}("");
    assertEq(wrsETH.balanceOf(user), balanceBefore); // user received 0 wrsETH
    assertGt(address(pool).balance, 0);              // ETH is in pool
}
```
