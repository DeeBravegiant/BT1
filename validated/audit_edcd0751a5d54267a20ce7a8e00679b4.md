Audit Report

## Title
Missing Chainlink Price Feed Validation Allows Stale or Zero Prices to Corrupt Core Deposit and Withdrawal Accounting - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary
`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards all return values except `price`, performing no staleness, round-completeness, or positivity checks. A zero price causes a division-by-zero revert in `LRTWithdrawalManager.getExpectedAssetAmount()`, temporarily freezing withdrawals for the affected asset. A stale price silently corrupts the rsETH mint and asset-payout calculations used by every depositor and withdrawer. The same repository's `ChainlinkOracleForRSETHPoolCollateral` already implements all three guards, confirming the team is aware of the risk.

## Finding Description
`contracts/oracles/ChainlinkPriceOracle.sol` line 52 reads only the `price` field:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

No checks are performed on `updatedAt` (staleness), `answeredInRound >= roundId` (round completeness), or `price > 0` (validity).

By contrast, `contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol` lines 30–32 performs all three:

```solidity
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
```

The unvalidated return value propagates into two public user-facing paths:

**Deposit path** — `LRTDepositPool.getRsETHAmountToMint()` line 520:
```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

**Withdrawal path** — `LRTWithdrawalManager.getExpectedAssetAmount()` line 593:
```solidity
underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset);
```

**Zero-price DoS**: If `price == 0`, `getAssetPrice` returns 0. `getExpectedAssetAmount` then performs `amount * rsETHPrice / 0`, which reverts with a division-by-zero panic. `initiateWithdrawal` calls `getExpectedAssetAmount` unconditionally at line 168, so every withdrawal attempt for the affected asset reverts for the duration of the oracle failure. No existing guard in `initiateWithdrawal` or `_beforeDeposit` checks the oracle return value before use.

**Stale-price accounting error**: If the Chainlink heartbeat is missed (e.g., 24-hour LST/ETH feeds during network congestion), `getAssetPrice` silently returns a stale value. A stale price that is higher than the true price (pre-depeg snapshot) causes `getRsETHAmountToMint` to mint more rsETH than the deposited assets are worth, diluting existing rsETH holders. A stale price that is lower than the true price causes `getExpectedAssetAmount` to over-allocate assets to withdrawers.

The `unlockQueue` operator path does apply `_validatePrices` bounds checks, but `initiateWithdrawal` and `depositAsset` are fully public and have no equivalent guard.

## Impact Explanation
**Medium — Temporary freezing of funds**: A Chainlink circuit-breaker event or feed misconfiguration returning `price = 0` causes every call to `initiateWithdrawal` for the affected asset to revert with a division-by-zero panic. Users holding rsETH cannot queue withdrawals until the oracle recovers. The freeze is bounded by the oracle outage duration but is triggered by any unprivileged user attempting a withdrawal.

**Low — Contract fails to deliver promised returns**: A stale price (heartbeat miss) causes `getRsETHAmountToMint` and `getExpectedAssetAmount` to compute incorrect exchange amounts, resulting in depositors receiving more rsETH than their assets are worth or withdrawers receiving incorrect asset amounts. This constitutes incorrect accounting without necessarily causing direct theft at the severity of a critical/high finding.

## Likelihood Explanation
Chainlink LST/ETH feeds (stETH/ETH, ETHx/ETH, rETH/ETH) use 24-hour heartbeat updates. During periods of network congestion or Chainlink node downtime, `updatedAt` can lag significantly beyond the heartbeat. Chainlink circuit-breaker activation returning `price = 0` is a documented edge case. Both failure modes are known to the protocol team — evidenced by `ChainlinkOracleForRSETHPoolCollateral` guarding against all three. Any unprivileged user can trigger the DoS path by calling `initiateWithdrawal` during an oracle failure; no special privileges or victim cooperation is required.

## Recommendation
Add staleness and validity checks to `ChainlinkPriceOracle.getAssetPrice()`, mirroring the pattern already used in `ChainlinkOracleForRSETHPoolCollateral`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);

    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
        priceFeed.latestRoundData();

    if (answeredInRound < roundId) revert StalePrice();
    if (updatedAt == 0) revert IncompleteRound();
    if (price <= 0) revert InvalidPrice();
    // Optionally: if (block.timestamp - updatedAt > HEARTBEAT_THRESHOLD) revert StalePrice();

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

## Proof of Concept
**Zero-price DoS path (Medium — Temporary freezing of funds):**

1. Chainlink stETH/ETH feed activates circuit breaker; `latestRoundData()` returns `price = 0`.
2. Any user calls `LRTWithdrawalManager.initiateWithdrawal(stETH, rsETHAmount, "")`.
3. `initiateWithdrawal` → `getExpectedAssetAmount(stETH, rsETHAmount)` → `lrtOracle.getAssetPrice(stETH)` → `ChainlinkPriceOracle.getAssetPrice(stETH)` returns `0`.
4. `underlyingToReceive = rsETHAmount * rsETHPrice / 0` → EVM division-by-zero panic → revert.
5. All stETH withdrawal initiations are blocked for the duration of the oracle failure. No admin action is required to trigger this; any user attempting a withdrawal reproduces it.

**Stale-price accounting error path (Low — Incorrect returns):**

1. Chainlink stETH/ETH 24-hour heartbeat is missed during network congestion; `updatedAt` is 25+ hours old and `price` reflects a pre-depeg value (e.g., 0.9999 ETH when true value is 0.97 ETH).
2. Attacker calls `LRTDepositPool.depositAsset(stETH, largeAmount, 0, "")`.
3. `_beforeDeposit` → `getRsETHAmountToMint` → `lrtOracle.getAssetPrice(stETH)` returns stale high price.
4. `rsethAmountToMint = largeAmount * staleHighPrice / rsETHPrice` — attacker receives more rsETH than the current fair value of their stETH deposit, diluting existing rsETH holders.
