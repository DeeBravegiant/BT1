Audit Report

## Title
Missing Chainlink `latestRoundData()` Return Value Validation Enables Stale-Price rsETH Over-Minting - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary

`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but silently discards `roundId`, `startedAt`, `updatedAt`, and `answeredInRound`, performing no staleness, round-completeness, or price-sign check. Because this oracle is the registered price source for L1 LST assets fed into `LRTDepositPool.getRsETHAmountToMint()`, a stale Chainlink answer is accepted without rejection, allowing any depositor to mint rsETH at an inflated rate and extract value from existing holders. The same protocol already applies all three required checks in `ChainlinkOracleForRSETHPoolCollateral`, confirming the omission is a code defect, not a design choice.

## Finding Description

`contracts/oracles/ChainlinkPriceOracle.sol` line 52 reads:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

All five return values of `latestRoundData()` — `(roundId, answer, startedAt, updatedAt, answeredInRound)` — are available, but only `answer` is used. No check is made that:
- `updatedAt + heartbeat >= block.timestamp` (staleness)
- `answeredInRound >= roundId` (round completeness)
- `price > 0` (price validity)

Contrast with `contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol` lines 30–32, which is used for L2 pool collateral and performs all three checks:

```solidity
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
```

The unvalidated `ChainlinkPriceOracle` is the oracle registered for L1 LST assets via `LRTOracle.updatePriceOracleFor()`. The deposit call chain is:

1. `LRTDepositPool.depositAsset()` → `_beforeDeposit()` → `getRsETHAmountToMint(asset, amount)`
2. `getRsETHAmountToMint` → `lrtOracle.getAssetPrice(asset)` → `ChainlinkPriceOracle.getAssetPrice(asset)` (unvalidated)
3. `rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice()`

`rsETHPrice` is a stored value updated by `updateRSETHPrice()`, not recalculated on-the-fly during deposit. If the Chainlink feed for an LST is stale and its last reported price is above the current market price, `getAssetPrice()` returns the inflated stale value while `rsETHPrice` reflects the last correctly-updated state, causing the depositor to receive more rsETH than the deposited collateral is worth.

Additionally, if `price` is zero or negative (a documented Chainlink edge case), `uint256(price)` for a negative `int256` wraps to a near-maximum value in Solidity 0.8.x (explicit casts do not revert), causing the subsequent multiplication to overflow and revert, permanently bricking deposits for that asset until governance replaces the oracle. ## Impact Explanation

**High — Theft of unclaimed yield.** A stale LST/ETH price that is above the current market rate allows any depositor to mint more rsETH than the deposited collateral is worth. The excess rsETH represents accrued yield extracted from all existing rsETH holders, because the total collateral backing is now diluted relative to the outstanding rsETH supply. The magnitude scales linearly with the price deviation and deposit size; a 1% stale premium on a large deposit extracts proportional value from the pool.

**Medium — Temporary freezing of funds (secondary).** A zero or negative Chainlink answer causes `uint256(price)` to wrap to `type(uint256).max`, making the multiplication overflow and revert under Solidity 0.8.x checked arithmetic. This freezes deposits for the affected asset until governance replaces the oracle. [5](#0-4) 

## Likelihood Explanation

Chainlink LST/ETH feeds (stETH/ETH, rETH/ETH) operate with 24-hour heartbeats and 0.5% deviation thresholds. Staleness is uncommon under normal conditions but is a documented failure mode during network congestion, node operator failures, feed deprecation, or L1 block production anomalies. The attacker does not need to cause the staleness — they only need to observe it and call `depositAsset` before the feed updates. The entry point is fully permissionless, requires no capital beyond the deposit itself, and is repeatable for as long as the feed remains stale. The SECURITY.md exclusion for "incorrect data supplied by third-party oracles" does not apply here because the defect is in the protocol's own oracle wrapper, which already applies the required checks in `ChainlinkOracleForRSETHPoolCollateral` for L2 collateral, confirming the omission is a code bug rather than an accepted oracle data risk. [6](#0-5) 

## Recommendation

Replace the bare `latestRoundData()` call in `ChainlinkPriceOracle.getAssetPrice()` with the same validation pattern already present in `ChainlinkOracleForRSETHPoolCollateral`, adding a configurable per-asset `MAX_STALENESS` constant matching each feed's documented heartbeat:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);
    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
        priceFeed.latestRoundData();

    if (answeredInRound < roundId) revert StalePrice();
    if (updatedAt == 0) revert IncompleteRound();
    if (price <= 0) revert InvalidPrice();
    if (block.timestamp - updatedAt > maxStaleness[asset]) revert StalePrice();

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

`maxStaleness[asset]` should be set per-asset to match the Chainlink feed's documented heartbeat (e.g., 86 400 seconds for 24-hour feeds) and updated via a restricted setter. [1](#0-0) 

## Proof of Concept

**Setup (Foundry fork test against mainnet):**

```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.27;

import "forge-std/Test.sol";

interface IChainlinkPriceOracle {
    function getAssetPrice(address asset) external view returns (uint256);
}

interface IDepositPool {
    function depositAsset(address, uint256, uint256, string calldata) external;
    function getRsETHAmountToMint(address, uint256) external view returns (uint256);
}

interface IERC20 {
    function approve(address, uint256) external;
    function balanceOf(address) external view returns (uint256);
}

contract StaleOraclePoC is Test {
    // Replace with actual deployed addresses
    address constant DEPOSIT_POOL = 0x...;
    address constant STETH = 0x...;
    address constant STETH_FEED = 0x...;  // Chainlink stETH/ETH feed

    function testStalePrice() public {
        // 1. Warp time forward 25 hours so the feed is stale (heartbeat = 24h)
        vm.warp(block.timestamp + 25 hours);

        // 2. The feed's last reported price (e.g. 1.05e18) is now stale but
        //    ChainlinkPriceOracle accepts it without any staleness check.
        uint256 stalePrice = IChainlinkPriceOracle(/* oracle address */).getAssetPrice(STETH);
        // stalePrice == 1.05e18 (stale, above current market 1.00e18)

        // 3. Attacker deposits 100 stETH
        deal(STETH, address(this), 100e18);
        IERC20(STETH).approve(DEPOSIT_POOL, 100e18);

        uint256 rsethBefore = IERC20(/* rsETH address */).balanceOf(address(this));
        IDepositPool(DEPOSIT_POOL).depositAsset(STETH, 100e18, 0, "");
        uint256 rsethAfter = IERC20(/* rsETH address */).balanceOf(address(this));

        // 4. Attacker received 105 rsETH for 100 stETH worth of collateral
        assertGt(rsethAfter - rsethBefore, 100e18, "Excess rsETH minted from stale price");
    }
}
```

**Steps:**
1. Fork mainnet at a block where the stETH/ETH Chainlink feed last updated more than 24 hours ago (or `vm.warp` past the heartbeat).
2. Confirm `ChainlinkPriceOracle.getAssetPrice(stETH)` returns the stale price without reverting.
3. Call `LRTDepositPool.depositAsset(stETH, 100e18, 0, "")`.
4. Observe that `rsethAmountToMint = (100e18 * stalePrice) / rsETHPrice` yields more rsETH than the deposited collateral is worth at the current market price.
5. When the oracle updates to the correct price, the attacker's rsETH is redeemable at the correct rate, having captured the premium at the expense of existing holders. [7](#0-6) [5](#0-4)
