Audit Report

## Title
Reward Tokens Permanently Locked When `totalKernelStaked` Drops to Zero During Active Reward Period - (File: contracts/KERNEL/KernelDepositPool.sol)

## Summary
`KernelDepositPool` uses a Synthetix-style staking rewards model where the `updateReward` modifier unconditionally advances `updatedAt` to `lastTimeRewardApplicable()` even when `totalKernelStaked == 0`. Because `rewardPerToken()` short-circuits and returns the unchanged `rewardPerTokenStored` when there are no stakers, any `rewardRate * Δt` rewards accrued during a zero-staked interval are silently discarded and permanently locked in the contract. No admin rescue or sweep function exists.

## Finding Description
The `updateReward` modifier always advances `updatedAt`:

```solidity
modifier updateReward(address _account) {
    rewardPerTokenStored = rewardPerToken();   // unchanged when totalKernelStaked == 0
    updatedAt = lastTimeRewardApplicable();    // ALWAYS advances
    ...
}
```

`rewardPerToken()` short-circuits when `totalKernelStaked == 0`, returning the stored value without accumulating any rewards for the elapsed interval:

```solidity
function rewardPerToken() public view returns (uint256) {
    if (totalKernelStaked == 0) {
        return rewardPerTokenStored;   // time consumed, no rewards distributed
    }
    ...
}
```

`initiateWithdrawal` subtracts from `totalKernelStaked` with no floor, allowing it to reach zero during an active reward window:

```solidity
balanceOf[msg.sender] -= _amount;
totalKernelStaked -= _amount;   // can reach 0
```

`notifyRewardAmount` guards against starting a reward period with zero stakers (`if (totalKernelStaked == 0) revert NoStakedTokens()`), but provides no protection against stakers exiting after a reward period has begun. The admin functions (`setRewardsDuration`, `notifyRewardAmount`, `setWithdrawalDelay`, `setMaxNumberOfWithdrawalsPerUser`) include no sweep or recovery path. The contract's own NatSpec at L14–22 explicitly acknowledges this behavior and relies solely on an operational (off-chain) mitigation.

## Impact Explanation
Reward tokens (`rewardsToken`) deposited by the admin via `notifyRewardAmount` are permanently locked in `KernelDepositPool` for any time interval during which `totalKernelStaked == 0` and `block.timestamp < finishAt`. The locked amount equals `rewardRate * Δt` for the zero-staked interval. There is no on-chain mechanism to recover these tokens. This constitutes **permanent freezing of unclaimed yield** (Medium).

## Likelihood Explanation
No privileged role needs to be compromised. The admin legitimately starts a reward period (enforced to require stakers); users legitimately call `initiateWithdrawal`, which is a standard protocol action. Because `totalKernelStaked` drops at `initiateWithdrawal` time rather than `claimWithdrawal` time, the zero-staked window can span the full `withdrawalDelay` (up to `MAX_WITHDRAWAL_DELAY = 30 days`). A single large staker or coordinated exit during a reward window is a realistic, non-adversarial scenario. The NatSpec itself acknowledges the risk, confirming the developers consider it a real operational concern.

## Recommendation
In the `updateReward` modifier, only advance `updatedAt` when there are stakers to absorb rewards:

```solidity
modifier updateReward(address _account) {
    rewardPerTokenStored = rewardPerToken();
    if (totalKernelStaked > 0) {
        updatedAt = lastTimeRewardApplicable();
    }
    if (_account != address(0)) {
        rewards[_account] = earned(_account);
        userRewardPerTokenPaid[_account] = rewardPerTokenStored;
    }
    _;
}
```

This freezes the time window when no stakers are present, preserving unallocated rewards for future stakers rather than silently discarding them.

## Proof of Concept
1. Admin calls `notifyRewardAmount(1_000e18)` while Alice has 100 KERNEL staked → `rewardRate = 1_000e18 / duration`, `finishAt = now + duration`, `updatedAt = now`.
2. Alice calls `initiateWithdrawal(100)` → `totalKernelStaked = 0`, `updatedAt` advances to `now` (Alice's accrued rewards up to this point are checkpointed correctly).
3. 30 days pass (`withdrawalDelay`). During this window `totalKernelStaked == 0` and `block.timestamp < finishAt`. No `updateReward` calls occur.
4. Bob calls `stake(1)` → `updateReward` fires: `rewardPerToken()` returns `rewardPerTokenStored` (unchanged), but `updatedAt` jumps to `now`. The 30-day reward window is consumed with zero distribution.
5. The reward tokens for those 30 days (`rewardRate * 30 days`) remain in the contract balance permanently with no recovery mechanism.

**Foundry test plan**: Deploy `KernelDepositPool`, call `notifyRewardAmount`, have the sole staker call `initiateWithdrawal`, `warp` 30 days, have a new user `stake`, then assert that `rewardsToken.balanceOf(address(pool))` exceeds the sum of all claimable rewards by `rewardRate * 30 days`.
