### Title
Missing On-Chain Output Verification for `AppPublicKey` CKD Responses Allows Single Malicious Participant to Inject Forged Confidential Key - (`File: crates/contract/src/lib.rs`)

---

### Summary

`respond_ckd` in the MPC smart contract performs cryptographic output verification **only** for the `AppPublicKeyPV` variant of CKD requests. For the `AppPublicKey` (privately-verifiable) variant, the response is accepted and delivered to the user with **zero on-chain verification**. A single malicious attested participant — strictly below the signing threshold — can call `respond_ckd` with an arbitrary forged `CKDResponse`, bypassing the threshold requirement for confidential key derivation output.

---

### Finding Description

The `respond_ckd` method in `crates/contract/src/lib.rs` contains the following branch:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← no verification
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV`, the contract calls `ckd_output_check`, which enforces the pairing equation `e(big_c, G2) = e(big_y, app_pk2) · e(H(pk, app_id), mpc_pk)`, cryptographically binding the response to the MPC master key and the user's app identity. [2](#0-1) 

For `AppPublicKey`, the branch is a no-op. Any `(big_y, big_c)` pair is accepted and immediately delivered to the user via `resolve_yields_for`. [3](#0-2) 

The CKD protocol for `AppPublicKey` works as follows: the user provides `A = a·G1` (their ephemeral public key). The MPC network computes `Y = y·G1` and `C = msk·H(pk, app_id) + y·A`. The user decrypts `sig = C − a·Y = msk·H(pk, app_id)` and verifies it against the MPC public key. [4](#0-3) 

Because the contract does not enforce this verification on-chain for `AppPublicKey`, a single attested participant can submit a crafted response. Specifically, the attacker can:

1. Pick a random scalar `y'`.
2. Compute `big_y = G1` (the generator) and `big_c = s' + A` for any chosen point `s'`, where `A` is the user's public key visible in the pending request.
3. The user decrypts: `sig = big_c − a·big_y = (s' + A) − a·G1 = s' + a·G1 − a·G1 = s'`.

The attacker has delivered a key `s'` of their choosing. The user's client-side verification (`e(H(pk, app_id), pk) = e(s', G2)`) will fail unless `s' = msk·H(pk, app_id)`, which the attacker cannot compute. However, the yield is already resolved and the forged response is already delivered on-chain. [5](#0-4) 

---

### Impact Explanation

**Critical — Permanent freezing of funds / unauthorized confidential key derivation output without required threshold authorization.**

The CKD feature is explicitly used in production to derive wrapping keys for TEE application key backups (e.g., the HOT Archive Signer uses CKD to encrypt its private key backup). A forged response causes the TEE app to derive and use a wrong key. [6](#0-5) 

If the TEE app does not independently verify the CKD output (which is the app's responsibility per the "privately verifiable" design, not enforced by the contract), it will:
- Encrypt its key backup with the wrong wrapping key.
- Be permanently unable to recover its private keys from backup.
- Permanently freeze all funds controlled by those keys.

Even if the app does verify and rejects the forged response, the yield is already consumed. The user's pending request is resolved and removed from the contract state. The user must pay gas and deposit again for a new request, and the attacker can repeat the attack indefinitely, constituting a sustained denial of the CKD service.

This is a **threshold bypass**: the threshold requirement for CKD output is that `t` participants must cooperate to compute `msk·H(pk, app_id)`. A single malicious participant can inject an arbitrary output, bypassing this requirement entirely.

---

### Likelihood Explanation

**Medium.** The attacker must be an attested participant in the current epoch — a non-trivial bar requiring a valid TEE attestation accepted by the contract. However, this is a single-participant attack, strictly below the signing threshold, and is the exact attacker profile the threshold model is designed to resist. Once a participant is attested, the attack is trivially executable: the attacker simply monitors the chain for pending `AppPublicKey` CKD requests and races to call `respond_ckd` before the honest nodes do. The `AppPublicKey` variant is the default/legacy variant and is the one used by the example CLI and documented as the primary interface. [7](#0-6) 

---

### Recommendation

Apply the same cryptographic output verification to `AppPublicKey` responses that is already applied to `AppPublicKeyPV`. For `AppPublicKey`, the verifiable relation is:

```
e(big_c − big_y * a, G2) = e(H(pk, app_id), mpc_pk)
```

However, since `a` (the user's private scalar) is not known on-chain, the equivalent on-chain check uses the user's public key `A = a·G1`:

```
e(big_c, G2) = e(H(pk, app_id), mpc_pk) · e(big_y, A_as_G2)
```

This requires `A` to be provided in G2 form (or the check restructured). Alternatively, require all CKD requests to use the `AppPublicKeyPV` variant, which already has a sound on-chain verification path, and deprecate the unverifiable `AppPublicKey` variant.

### Proof of Concept

**Setup**: Contract is in Running state with 3 attested participants (threshold = 2). User submits a CKD request with `AppPublicKey(A)` where `A = a·G1`.

**Attack** (single malicious attested participant, account `attacker.near`):

```rust
// Attacker reads the pending request from chain: CKDRequest { app_public_key: AppPublicKey(A), app_id, domain_id }
// A = user's public key (visible on-chain in the request map)

// Attacker crafts a forged response:
// big_y = G1 (the generator)
// big_c = s' + A  (for any chosen point s', e.g., s' = G1)
let forged_response = CKDResponse {
    big_y: G1_GENERATOR,
    big_c: s_prime + A,  // attacker knows s' = G1, so big_c = G1 + A
};

// Attacker calls respond_ckd before honest nodes:
contract.respond_ckd(pending_request, forged_response);
// → passes assert_caller_is_signer() ✓
// → passes assert_caller_is_attested_participant_and_protocol_active() ✓
// → AppPublicKey branch: no verification ✓
// → resolve_yields_for delivers forged response to user ✓
```

**User receives**: `big_y = G1`, `big_c = G1 + A`.

**User decrypts**: `sig = big_c − a·big_y = (G1 + A) − a·G1 = G1 + a·G1 − a·G1 = G1 = s'`.

**Attacker knows**: `s' = G1` (the generator point), so the attacker knows the user's derived key is `G1`.

**User verification fails** (since `G1 ≠ msk·H(pk, app_id)`), but the yield is already consumed. The user's TEE app, if it skips verification and uses the key, will encrypt its backup with a key the attacker knows — permanently compromising or freezing the protected assets. [8](#0-7) [9](#0-8)

    #[handle_result]
    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

///
/// Point validation is fully delegated to the host, as in
/// [`app_public_key_check`].
pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,

    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);

    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)

fn decrypt_secret_and_verify(
    big_y: &Bls12381G1PublicKey,
    big_c: &Bls12381G1PublicKey,
    private_key: Scalar,
    app_id: CkdAppId,
    mpc_public_key: &Bls12381G2PublicKey,
) -> Result<[u8; BLS12381G1_PUBLIC_KEY_SIZE]> {
    let big_y: G1Projective = big_y.try_into().context("invalid G1 point")?;
    let big_c: G1Projective = big_c.try_into().context("invalid G1 point")?;
    let mpc_public_key: G2Projective = mpc_public_key.try_into().context("invalid G2 point")?;

    // decrypt the secret
    let secret = big_c - big_y * private_key;

    // verify the secret
    if !verify(&mpc_public_key, app_id.as_ref(), &secret) {
        anyhow::bail!("Verification failed!");
    }

    // return the secret as bytes
    Ok(secret.to_compressed())
}

### Recovery via Confidential Key Derivation (CKD)

Encrypted disk persistence protects against restarts, but not against total loss of the CVM instance — for example, if the TEE image is updated (changing RTMR measurements and invalidating the disk encryption key) or all CVM instances are destroyed. Without an additional recovery mechanism, this would be unrecoverable after the HOT MPC network is retired, since keyshares can no longer be re-imported.

To address this, we use [Confidential Key Derivation][ckd] (CKD) via the NEAR MPC network to create a hardware-independent encrypted backup:

1. **Derive a wrapping key.** After key reconstruction, the Archive Signer calls the NEAR MPC network's CKD endpoint with its `app_id`. CKD returns a deterministic, confidential key that only this application can obtain.
2. **Encrypt and store a backup.** The reconstructed HOT private keys are encrypted with the CKD-derived wrapping key. The resulting ciphertext is stored in external storage.
3. **Recover from any new instance.** A new instance re-derives the same CKD key — since CKD is deterministic on `app_id`, not on hardware measurements — and decrypts the backup.

This decouples recovery from any specific CVM instance or disk encryption key, requiring only that the NEAR MPC network remains operational.
```
