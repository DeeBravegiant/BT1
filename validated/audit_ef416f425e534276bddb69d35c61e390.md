Audit Report

## Title
Uninitialized `dailyMintLimit` Causes All Deposits to Revert Until Admin Intervention - (File: `contracts/pools/RSETHPoolV3.sol`)

## Summary
`RSETHPoolV3.initialize()` does not set `dailyMintLimit`, leaving it at the Solidity default of `0`. The `limitDailyMint` modifier applied to both `deposit` functions unconditionally reverts with `DailyMintLimitExceeded` for any non-zero deposit when `dailyMintLimit == 0`. The contract is non-functional for its primary purpose from deployment until an admin calls `reinitialize()` or `setDailyMintLimit()`.

## Finding Description
`initialize()` (L207–232) sets roles, `wrsETH`, `feeBps`, `rsETHOracle`, and `isEthDepositEnabled`, but never assigns `dailyMintLimit` or `startTimestamp`. Both remain `0`.

The `limitDailyMint` modifier (L96–125) first checks `block.timestamp < startTimestamp`. With `startTimestamp == 0`, this check always passes (current block timestamps are far above 0). It then computes `rsETHAmount` via `viewSwapRsETHAmountAndFee`, resets `dailyMintAmount` to `0` (since `getCurrentDay()` returns a large value > `lastMintDay == 0`), and then evaluates:

```solidity
if (dailyMintAmount + rsETHAmount > dailyMintLimit) {
    revert DailyMintLimitExceeded();
}
```

With `dailyMintLimit == 0` and any non-zero deposit producing `rsETHAmount > 0`, the condition `0 + rsETHAmount > 0` is always true. Both `deposit(string)` (L246, uses `limitDailyMint(msg.value, ETH_IDENTIFIER)`) and `deposit(address, uint256, string)` (L271, uses `limitDailyMint(amount, token)`) revert before any token transfer occurs.

`setDailyMintLimit()` (L605–611) explicitly rejects `0` as a value, confirming `0` is an invalid operational state. The admin can unblock deposits by calling either `reinitialize()` (L179–198, `reinitializer(2)`) or `setDailyMintLimit()` (L605–611), but neither is called atomically with `initialize()`, creating a window of complete non-functionality.

## Impact Explanation
**Low — Contract fails to deliver promised returns, but doesn't lose value.** Every deposit attempt reverts with `DailyMintLimitExceeded` from deployment until admin action. No user funds are at risk because the revert occurs before any `safeTransferFrom` or `mint` call. The contract's core function — minting wrsETH in exchange for deposited assets — is entirely unavailable during this window.

## Likelihood Explanation
This affects every fresh deployment of `RSETHPoolV3`. The window exists between `initialize()` and the admin's subsequent call to `reinitialize()` or `setDailyMintLimit()`. Because `reinitialize()` uses the `reinitializer(2)` upgrade idiom, operators may not recognize it as a mandatory post-deployment step. Any unprivileged depositor triggers the failure with a standard deposit call; no special conditions or attacker capability are required.

## Recommendation
Set a non-zero `dailyMintLimit` directly in `initialize()` by accepting it as a parameter, or add an explicit `require(dailyMintLimit > 0, "DailyMintLimitNotSet")` guard at the top of `limitDailyMint` to surface a clear error distinct from a genuine limit-exceeded condition. Alternatively, document `reinitialize()` or `setDailyMintLimit()` as a mandatory post-deployment step and enforce it via a deployment script.

## Proof of Concept
1. Deploy `RSETHPoolV3` and call `initialize(admin, bridger, wrsETH, feeBps, oracle, true)`. `dailyMintLimit` is `0`.
2. Do **not** call `reinitialize()` or `setDailyMintLimit()`.
3. Call `deposit{value: 1 ether}("")` as any user.
4. Transaction reverts with `DailyMintLimitExceeded` because `0 + rsETHAmount > 0`.
5. As admin, call `setDailyMintLimit(1000 ether)`.
6. Repeat step 3 — deposit succeeds.
