### Title
Missing Ed25519 Handler in `respond_verify_foreign_tx` Permanently Breaks Foreign-Tx Verification for Ed25519 Domains - (File: crates/contract/src/lib.rs)

### Summary
`respond_verify_foreign_tx` only implements a `Secp256k1` signature-verification arm. The `Ed25519` arm present in `respond` is entirely absent. If a `ForeignTx`-purpose domain is ever configured with an Ed25519 (FROST) key via governance, every `verify_foreign_transaction` request on that domain will be permanently unresolvable: the contract accepts the request, nodes refuse to respond (node-side guard), and even if a node were patched to respond, the contract would reject the response with `SignatureSchemeMismatch`. All queued yields time out, breaking the request lifecycle for that domain class.

### Finding Description

`respond` handles both signature schemes:

```rust
// crates/contract/src/lib.rs:586-640
match (&response, public_key) {
    (SignatureResponse::Secp256k1(...), PublicKeyExtended::Secp256k1 { ... }) => { ... }
    (SignatureResponse::Ed25519 { signature }, PublicKeyExtended::Ed25519 { edwards_point, .. }) => { ... }
    _ => Err(RespondError::SignatureSchemeMismatch { ... })
}
```

`respond_verify_foreign_tx` handles only one:

```rust
// crates/contract/src/lib.rs:718-743
match (&response.signature, public_key) {
    (SignatureResponse::Secp256k1(...), PublicKeyExtended::Secp256k1 { ... }) => { ... }
    // Ed25519 arm is absent
    (signature_response, public_key_requested) => {
        return Err(RespondError::SignatureSchemeMismatch { ... }.into());
    }
}
``` [1](#0-0) [2](#0-1) 

The node side independently enforces the same gap — it explicitly rejects `Protocol::Frost` for foreign-tx attempts:

```rust
// crates/node/src/mpc_client.rs:605-614
Some(Protocol::ConfidentialKeyDerivation)
| Some(Protocol::DamgardEtAl)

    "Signature scheme is not allowed for domain: {:?}",
    verify_foreign_tx_attempt.request.domain_id.clone()

``` [3](#0-2) 

Both layers are missing the Ed25519 handler, making the gap structural rather than incidental.

The contract accepts `verify_foreign_transaction` for any domain whose `DomainPurpose` is `ForeignTx`, regardless of the underlying curve. There is no guard at request-submission time that rejects Ed25519 ForeignTx domains: [4](#0-3) 

New domains — including Ed25519 ForeignTx domains — can be added via `vote_add_domains`, a standard governance path requiring only a threshold of participant votes: [5](#0-4) 

### Impact Explanation

Once an Ed25519 ForeignTx domain exists:

1. Any user can call `verify_foreign_transaction` targeting that domain. The contract queues the yield and deducts the deposit.
2. Nodes see the request but return an error internally and never call `respond_verify_foreign_tx`.
3. Even if a node were patched to respond, the contract would return `SignatureSchemeMismatch` and reject the response.
4. After ~200 blocks the yield times out, `return_verify_foreign_tx_and_clean_state_on_success` fires with `Err(PromiseError::Failed)`, pops the queue entry, and calls `fail_on_timeout` — the caller receives a timeout error. [6](#0-5) 

The entire request lifecycle for Ed25519 ForeignTx domains is permanently broken. This matches the Medium allowed impact: *request-lifecycle manipulation that breaks production safety/accounting invariants* — the invariant being that a properly configured domain must be able to service requests.

### Likelihood Explanation

The vulnerable state requires participants to vote in an Ed25519 ForeignTx domain. This is a legitimate governance action (not collusion), and the design explicitly supports multiple domain purposes and curves. The contract code itself is the root cause; governance merely activates it. Once activated, every user request to that domain fails deterministically — no attacker action is needed beyond submitting a normal `verify_foreign_transaction` call.

### Recommendation

Add the missing Ed25519 arm to `respond_verify_foreign_tx`, mirroring the pattern in `respond`:

```rust
(
    dtos::SignatureResponse::Ed25519 { signature },
    PublicKeyExtended::Ed25519 { edwards_point: public_key_edwards_point, .. },
) => {
    let derived_public_key_edwards_point =
        derive_public_key_edwards_point_ed25519(&public_key_edwards_point, &request.tweak);
    let derived_public_key_32_bytes =
        dtos::Ed25519PublicKey::from(derived_public_key_edwards_point.compress());
    let message = response.payload_hash.as_slice();
    near_mpc_signature_verifier::verify_eddsa_signature(
        signature,
        message,
        &derived_public_key_32_bytes,
    )
    .is_ok()
}
```

Remove the corresponding node-side rejection of `Protocol::Frost` for foreign-tx domains, and add a test covering an Ed25519 ForeignTx domain end-to-end.

### Proof of Concept

1. Participants call `vote_add_domains` with a domain `{ id: N, protocol: Frost, purpose: ForeignTx }`.
2. After threshold votes, the domain is live.
3. A user calls `verify_foreign_transaction` targeting domain `N` with a valid Bitcoin request. The contract queues the yield.
4. Nodes observe the request, hit the `Protocol::Frost` guard in `mpc_client.rs`, and return an error — no `respond_verify_foreign_tx` call is ever submitted.
5. After ~200 blocks, the yield times out. `return_verify_foreign_tx_and_clean_state_on_success` fires with `Err(PromiseError::Failed)`, pops the queue, and calls `fail_on_timeout`.
6. The user receives `RequestError::Timeout`. Every subsequent request to domain `N` follows the same path — the domain is permanently non-functional. [7](#0-6)

    pub fn return_verify_foreign_tx_and_clean_state_on_success(
        &mut self,
        request: VerifyForeignTransactionRequest,
        #[callback_result] response: Result<VerifyForeignTransactionResponse, PromiseError>,
    ) -> PromiseOrValue<VerifyForeignTransactionResponse> {
        match response {
            Ok(response) => PromiseOrValue::Value(response),
            Err(_) => {
                pending_requests::pop_oldest_pending_yield(
                    &mut self.pending_verify_foreign_tx_requests,
                    &request,
                );
                let fail_on_timeout_gas = Gas::from_tgas(self.config.fail_on_timeout_tera_gas);
                let promise = Promise::new(env::current_account_id()).function_call(
                    method_names::FAIL_ON_TIMEOUT.to_string(),
                    vec![],
                    NearToken::from_near(0),
                    fail_on_timeout_gas,
                );
                near_sdk::PromiseOrValue::Promise(promise.as_return())
            }
        }
    }
