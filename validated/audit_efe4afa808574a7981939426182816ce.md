Audit Report

## Title
ETH Deposit Cap Not Enforced Due to Missing `amount` in Limit Check - (File: contracts/LRTDepositPool.sol)

## Summary
`LRTDepositPool._checkIfDepositAmountExceedesCurrentLimit` omits the incoming `amount` from the ETH branch of the cap check, evaluating only `totalAssetDeposits > depositLimit` instead of `totalAssetDeposits + amount > depositLimit`. As a result, any depositor can bypass the ETH deposit cap in a single `depositETH` call as long as the current total has not yet exceeded the limit, causing rsETH to be minted far beyond the configured cap.

## Finding Description
In `_checkIfDepositAmountExceedesCurrentLimit` (L676–682), the ETH branch at L679 checks `totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)` — it asks only whether the cap was already exceeded *before* this deposit. The LST branch at L681 correctly checks `totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)`. Because `_beforeDeposit` (L661–663) reverts only when this function returns `true`, and the ETH branch never includes `amount`, the revert never fires while `totalAssetDeposits ≤ depositLimit`, regardless of how large `amount` is. The public, payable `depositETH` (L76–93) calls `_beforeDeposit` with `msg.value` as the deposit amount, so any caller can send an arbitrarily large ETH value and bypass the cap entirely.

## Impact Explanation
The ETH deposit cap (`depositLimitByAsset[ETH_TOKEN]`) is a TVL risk-management invariant. Because it is never checked against the incoming amount, a single depositor can push total ETH deposits arbitrarily above the configured limit in one transaction, causing `_mintRsETH` to mint rsETH far in excess of what the cap was designed to allow. This matches **Low — Contract fails to deliver promised returns**: the cap guarantee is broken, though direct fund loss requires additional assumptions about the protocol's backing model.

## Likelihood Explanation
`depositETH` is public and permissionless — no role, whitelist, or special condition is required. Any depositor who observes `getTotalAssetDeposits(ETH_TOKEN) ≤ depositLimit` can exploit this in a single transaction. Likelihood is **High**.

## Recommendation
Add `amount` to the ETH branch, mirroring the LST branch:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

## Proof of Concept
1. Admin sets `depositLimitByAsset[ETH_TOKEN] = 100 ether`.
2. `getTotalAssetDeposits(ETH_TOKEN)` returns `50 ether`.
3. Attacker calls `depositETH{value: 10_000 ether}(0, "")`.
4. `_checkIfDepositAmountExceedesCurrentLimit` evaluates `50 ether > 100 ether` → `false` → no revert.
5. `_mintRsETH` mints rsETH for 10 000 ETH; `totalAssetDeposits` becomes `10 050 ether`.
6. All future `depositETH` calls revert with `MaximumDepositLimitReached`, but the cap has already been violated by 9 950 ETH.

**Foundry test plan:** Deploy `LRTDepositPool` with `depositLimitByAsset[ETH_TOKEN] = 100 ether`, seed `50 ether` of prior deposits, call `depositETH{value: 10_000 ether}`, assert `getTotalAssetDeposits(ETH_TOKEN) > 100 ether` and that rsETH was minted for the full 10 000 ETH.

### Citations

**File:** contracts/LRTDepositPool.sol (L676-682)
```text
    function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
        uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
        if (asset == LRTConstants.ETH_TOKEN) {
            return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));
        }
        return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
    }
```
