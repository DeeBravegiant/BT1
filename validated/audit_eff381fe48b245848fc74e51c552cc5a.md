### Title
Farmer Silently Accepts Harvester Reward Diversion When Fee Threshold Is Invalid or Absent — (`File: chia/farmer/farmer_api.py`, `chia/farmer/farmer.py`)

### Summary
The CHIP-22 fee convention allows a third-party harvester to redirect the farmer's block reward to itself as a fee, but only when the proof's "fee quality" meets a threshold the harvester declares. The farmer's fee-threshold check in `notify_farmer_reward_taken_by_harvester_as_fee()` only emits log warnings on failure — it never blocks the reward diversion. A malicious harvester can omit `fee_info` entirely or supply an invalid threshold and the farmer will still redirect its block reward to the harvester's address, silently.

### Finding Description

**Root cause — two decoupled code paths:**

**Path 1 — validation (advisory only):**
In `farmer_api.py`, when a `NewProofOfSpace` arrives with `farmer_reward_address_override` set, the farmer calls `notify_farmer_reward_taken_by_harvester_as_fee()`:

```python
# farmer_api.py line 128-129
if new_proof_of_space.farmer_reward_address_override is not None:
    self.farmer.notify_farmer_reward_taken_by_harvester_as_fee(sp, new_proof_of_space)
``` [1](#0-0) 

Inside `notify_farmer_reward_taken_by_harvester_as_fee()`, when `fee_info is None` or the fee quality fails the threshold, the function only logs a warning and returns normally — it does **not** raise an exception or return a sentinel that would abort the block-making flow:

```python
# farmer.py lines 911-934
if proof_of_space.fee_info is not None:
    ...
    if fee_quality <= fee_threshold:
        self.log.info(...)   # OK
    else:
        self.log.warning(...)  # ONLY a warning — no abort
else:
    self.log.warning(...)      # ONLY a warning — no abort
``` [2](#0-1) 

**Path 2 — actual diversion (unconditional):**
After the warning-only check, the farmer unconditionally proceeds to request signatures from the harvester. When the harvester responds with `RespondSignatures`, `_process_respond_signatures()` blindly uses whatever `farmer_reward_address_override` the harvester supplies:

```python
# farmer_api.py lines 916-919
farmer_reward_address = self.farmer.farmer_target
if response.farmer_reward_address_override is not None:
    farmer_reward_address = response.farmer_reward_address_override
    include_source_signature_data = True
``` [3](#0-2) 

There is no re-check of the fee threshold here. The `DeclareProofOfSpace` message is then built with the harvester's address as `farmer_reward_address`, causing the full-node to assign the block reward to the harvester. [4](#0-3) 

The `ProofOfSpaceFeeInfo` struct and `fee_info` field are optional by design: [5](#0-4) 

### Impact Explanation
A malicious third-party harvester that connects to a farmer can redirect every block reward the farmer wins to the harvester's own address. The farmer receives only a log warning; the XCH block reward (currently 0.25 XCH per block for the farmer portion) is permanently diverted. This is unauthorized payout redirection of XCH block rewards — a High-severity impact under the allowed scope ("Bypass of … authorization that enables … payout redirection").

### Likelihood Explanation
Any third-party harvester that has a legitimate network connection to the farmer can exploit this. No keys, no admin access, and no cryptographic break are required. The harvester simply sets `farmer_reward_address_override` to its own address and omits `fee_info` (or supplies a threshold higher than the actual fee quality). The farmer's only response is a log line that most operators will never see in production.

### Recommendation
The fee-threshold validation must be enforced before the block-making flow continues. Specifically:

1. In `farmer_api.py`, after calling `notify_farmer_reward_taken_by_harvester_as_fee()`, check whether the fee threshold was valid. If `fee_info is None` or `fee_quality > fee_threshold`, **abort** the signature request for that proof (return `None` and do not call `peer.send_message`).
2. In `_process_respond_signatures()`, re-validate that the `farmer_reward_address_override` in the `RespondSignatures` message matches the one that was validated in the earlier `NewProofOfSpace` step, and that the fee threshold was confirmed valid at that time.
3. Consider storing the per-proof fee-validation result (pass/fail) in `self.farmer.proofs_of_space` alongside the proof, so `_process_respond_signatures()` can gate on it.

### Proof of Concept

A malicious harvester sends a `NewProofOfSpace` with a valid proof but with `farmer_reward_address_override` set to its own address and `fee_info = None`:

```python
pos = harvester_protocol.NewProofOfSpace(
    challenge_hash=challenge,
    sp_hash=sp_hash,
    plot_identifier="attacker.plot",
    proof=valid_proof,
    signage_point_index=uint8(0),
    include_source_signature_data=False,
    farmer_reward_address_override=attacker_puzzle_hash,  # attacker's address
    fee_info=None,  # no fee threshold provided — should be rejected
)
```

The farmer receives this, calls `notify_farmer_reward_taken_by_harvester_as_fee()` which logs:
> "Harvester illegitimately took reward by failing to provide its fee rate…"

But then **proceeds** to send `RequestSignatures` to the harvester. The harvester signs and returns `RespondSignatures` with `farmer_reward_address_override = attacker_puzzle_hash`. The farmer's `_process_respond_signatures()` builds `DeclareProofOfSpace` with `farmer_reward_address = attacker_puzzle_hash`. The full node creates the block and the farmer reward coin goes to the attacker. The farmer loses their block reward with no on-chain signal and only a log warning as evidence.
