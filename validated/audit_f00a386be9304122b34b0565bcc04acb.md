### Title
Single-connection `is_reliable()` bypass allows DNS seeder `good_peers` poisoning — (`chia/seeder/peer_record.py`, `chia/seeder/crawl_store.py`)

### Summary
The `is_reliable()` bootstrap branch unconditionally trusts a peer after a single successful crawl interaction (tries=1, successes=1), allowing an attacker to immediately inject their IP into the DNS seeder's `good_peers` table and have it served to all bootstrapping nodes.

### Finding Description
`PeerReliability.is_reliable()` contains a bootstrap branch:

```python
if self.tries > 0 and self.tries <= 3 and self.successes * 2 >= self.tries:
    return True
``` [1](#0-0) 

With tries=1 and successes=1 (one successful crawl), this branch returns `True` immediately, bypassing all statistical reliability windows (2h, 8h, 1d, 1w, 1m). The full exploit path:

1. Attacker runs a node; its IP is gossiped into the crawler's peer DB via any `RespondPeers` message from an existing peer.
2. Crawler's `connect_task` initiates an outbound connection to the attacker's IP.
3. Attacker completes the TLS handshake and sends one `NewPeak` message with `height >= minimum_height` (no proof required — the crawler does not validate the claimed height against actual chain state).
4. `new_peak` → `peer_connected_hostname(host, True)` → `peer_connected` → `reliability.update(True, age)` → tries=1, successes=1.
5. Next `save_to_db()` call (every ~15 seconds) invokes `load_reliable_peers_to_db`, which iterates all peers, finds `is_reliable() == True`, and inserts the attacker's IP into `good_peers`.
6. `get_good_peers()` returns the attacker's IP to the DNS server, which serves it to bootstrapping nodes.

The `NewPeak` message carries only `height`, `weight`, `header_hash`, `prev_header_hash`, and `reward_chain_block_hash` — none of which are validated by the crawler against actual chain data.

### Impact Explanation
The attacker's IP is served via DNS to every new node bootstrapping onto the network. An attacker operating many such nodes (each requiring only one outbound connection from the crawler) can dominate DNS responses. New nodes that connect primarily or exclusively to attacker-controlled peers cannot sync valid chain data, resulting in long-lived inability to process blocks. This meets the High impact criterion: "Permanent or long-lived inability for honest nodes... to process valid blocks... under normal network assumptions."

### Likelihood Explanation
The preconditions are minimal: run a node, wait for the crawler to connect (which it will, once the IP is gossiped), complete TLS, send one `NewPeak`. No keys, no admin access, no cryptographic breaks required. The attack scales linearly with the number of IPs the attacker controls.

### Recommendation
Remove or harden the bootstrap branch. Options:
- Require a minimum of N>3 tries before the bootstrap branch applies, or eliminate it entirely and rely solely on the statistical windows.
- Require `stat_2h.count > 2` (at least 3 observations in the 2-hour window) before any branch returns `True`.
- Validate that the claimed `NewPeak` height is plausible against a known checkpoint or the crawler's own best-known height before crediting a successful connection.
