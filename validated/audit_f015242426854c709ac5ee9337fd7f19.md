Audit Report

## Title
Token Deposit Path Charges Zero Fee Due to Uninitialized `tokenFeeBps` - (File: contracts/pools/RSETHPool.sol)

## Summary
`RSETHPool.sol` exposes two `deposit` overloads that both yield `wrsETH` but apply fees from different sources: the ETH path reads the global `feeBps`, while the token path reads `tokenFeeBps[token]`. Because `addSupportedToken` never initializes `tokenFeeBps[token]`, it defaults to `0` in Solidity, making every token deposit permanently fee-free until an admin separately calls `setTokenFeeBps`. Any depositor can observe this on-chain and route through the token path to avoid paying the protocol fee entirely.

## Finding Description
`viewSwapRsETHAmountAndFee(uint256 amount)` at line 312 computes `fee = amount * feeBps / 10_000` using the global rate. `viewSwapRsETHAmountAndFee(uint256 amount, address token)` at lines 335–336 computes `fee = amount * tokenFeeBps[token] / 10_000`. The `addSupportedToken` function at lines 637–656 writes `supportedTokenOracle[token]`, `tokenBridge[token]`, and pushes to `supportedTokenList`, but never writes `tokenFeeBps[token]`. Solidity mapping defaults leave it at `0`. The only setter is `setTokenFeeBps` (lines 583–594), which requires `DEFAULT_ADMIN_ROLE` and is a completely separate, optional transaction. Until that call is made, `deposit(token, amount, referralId)` at line 298 calls the token overload and collects zero fee, while `deposit{value: ...}(referralId)` at line 271 calls the ETH overload and collects `feeBps`. Both paths transfer `wrsETH` to the caller; the economic outcome is equivalent, but the fee burden is not.

## Impact Explanation
This is **Theft of unclaimed yield (High)**. Protocol fee revenue that should accrue on token deposits is permanently lost for every deposit made before `setTokenFeeBps` is called. A depositor who would pay 50 bps on an ETH deposit can instead acquire the equivalent token on a DEX (paying ~5 bps swap fee) and deposit it for 0 bps, netting a material advantage at the protocol's expense. The loss is not recoverable — fees not collected at deposit time are gone.

## Likelihood Explanation
The condition is always true for any token registered via `addSupportedToken` unless `setTokenFeeBps` is called in the same or a prior transaction. `tokenFeeBps[token]` is a public mapping; any depositor can read it and confirm the fee-free path before acting. No special privileges, timing windows, or front-running are required. The exploit is repeatable for every token deposit until the admin intervenes.

## Recommendation
Initialize `tokenFeeBps[token]` inside `addSupportedToken` to the current global `feeBps` (or to an explicit parameter), ensuring newly added tokens inherit a non-zero fee rate by default:

```solidity
tokenFeeBps[token] = feeBps;
```

Alternatively, fall back to `feeBps` in `viewSwapRsETHAmountAndFee(amount, token)` when `tokenFeeBps[token]` is zero:

```solidity
uint256 feeBpsForToken = tokenFeeBps[token] != 0 ? tokenFeeBps[token] : feeBps;
```

Emit the effective fee rate in `addSupportedToken` to make the initial value auditable.

## Proof of Concept
1. Deploy or fork `RSETHPool` with `feeBps = 50` and call `addSupportedToken(wstETH, oracle, bridge)` without subsequently calling `setTokenFeeBps(wstETH, 50)`.
2. Confirm `tokenFeeBps[wstETH] == 0` via a public read.
3. User A calls `deposit{value: 1 ether}("")` → `viewSwapRsETHAmountAndFee(1e18)` returns `fee = 0.005 ETH`; user receives `wrsETH` for `0.995 ETH`.
4. User B acquires equivalent wstETH on Uniswap (~0.05% fee), then calls `deposit(wstETH, amount, "")` → `viewSwapRsETHAmountAndFee(amount, wstETH)` returns `fee = 0`; user receives `wrsETH` for the full wstETH value.
5. Assert `feeEarnedInToken[wstETH] == 0` after step 4 and `feeEarnedInETH > 0` after step 3.

Foundry fork test: fork Arbitrum mainnet, impersonate the TIMELOCK_ROLE to call `addSupportedToken`, skip `setTokenFeeBps`, then call `deposit(token, amount, "")` and assert `feeEarnedInToken[token] == 0` while `feeBps > 0`.
