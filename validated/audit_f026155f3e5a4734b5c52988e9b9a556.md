Audit Report

## Title
Zero-Minimum Withdrawal Queue Stuffing Delays Legitimate User Withdrawals - (File: contracts/LRTWithdrawalManager.sol)

## Summary
`LRTWithdrawalManager.initiateWithdrawal` accepts 1-wei rsETH withdrawal requests when `minRsEthAmountToWithdraw[asset]` has not been explicitly set by an admin, because the mapping defaults to `0` and the only effective guard becomes `rsETHUnstaked == 0`. An unprivileged rsETH holder can flood the global FIFO queue with dust entries, forcing legitimate users' requests to wait behind every dust entry before `nextLockedNonce` advances to their nonce, temporarily locking their rsETH in the contract.

## Finding Description
`minRsEthAmountToWithdraw` is declared as a plain mapping at line 35 and is never initialized in `initialize()`, so it defaults to `0` for every asset. The guard at line 162 evaluates as:

```solidity
if (rsETHUnstaked == 0 || rsETHUnstaked < minRsEthAmountToWithdraw[asset])
```

When the mapping value is `0`, the second condition `rsETHUnstaked < 0` is always false, leaving only the `== 0` check. Any call with `rsETHUnstaked = 1` passes. Each such call transfers 1 wei rsETH to the contract (line 166), increments `assetsCommitted[asset]` by `expectedAssetAmount` (line 173), and pushes a new entry into the sequential nonce queue via `_addUserWithdrawalRequest` (line 175), advancing `nextUnusedNonce[asset]`.

The unlock loop in `_unlockWithdrawalRequests` (lines 790–815) iterates strictly from `nextLockedNonce_` upward and cannot skip entries. A legitimate user's request assigned nonce `N` cannot be completed because `completeWithdrawal` enforces `usersFirstWithdrawalRequestNonce >= nextLockedNonce[asset]` reverts with `WithdrawalLocked` (line 707). The operator must advance `nextLockedNonce` through every preceding dust entry before the victim's nonce becomes claimable. With a large protocol TVL the `assetsCommitted` ceiling (line 170) permits an enormous number of dust entries before it is hit.

## Impact Explanation
Legitimate users who call `initiateWithdrawal` after the attacker's dust requests have their rsETH transferred to and locked in the contract, and their expected asset amount committed. They cannot call `completeWithdrawal` until every preceding dust nonce has been processed by the operator. This is a concrete **temporary freezing of funds** (Medium), matching the allowed impact scope. The freeze duration scales with the number of dust entries and the operator's batch-processing throughput.

## Likelihood Explanation
The default state of every newly deployed or upgraded contract is immediately vulnerable — no admin action is required to enable the attack. An attacker needs only a trivial rsETH balance (obtainable on the open market) and pays only per-transaction gas. The attack is repeatable and can be sustained as long as the attacker holds rsETH and the minimum remains unset. Any party motivated to delay competitors' withdrawals or grief the protocol can execute this at low cost.

## Recommendation
Set a non-zero `minRsEthAmountToWithdraw` for every supported asset inside `initialize()` (or in a dedicated initializer step) rather than relying on a post-deployment admin call. Additionally, consider adding a per-user cap on the number of pending withdrawal requests (e.g., checking `userAssociatedNonces[asset][msg.sender].length()` before enqueuing) to bound queue depth independently of the minimum amount.

## Proof of Concept
1. Deploy or upgrade `LRTWithdrawalManager`; `minRsEthAmountToWithdraw[ETH_TOKEN]` is `0` (default — admin has not called `setMinRsEthAmountToWithdraw`).
2. Attacker acquires `N` wei of rsETH (e.g., `N = 50_000`).
3. Attacker calls `initiateWithdrawal(ETH_TOKEN, 1, "")` in a loop `N` times. Each call succeeds: `1 != 0` and `1 < 0` is false, so no revert. Each call transfers 1 wei rsETH to the contract and pushes a new nonce.
4. `nextUnusedNonce[ETH_TOKEN]` is now `N`; `assetsCommitted[ETH_TOKEN]` has increased by `N * expectedAssetAmount` (≈ N wei).
5. Victim calls `initiateWithdrawal(ETH_TOKEN, 1 ether, "")` — assigned nonce `N`.
6. Operator calls `unlockQueue(ETH_TOKEN, N+1, ...)`. The while-loop must iterate through all `N` dust entries before reaching nonce `N`, consuming gas proportional to `N` and requiring multiple batched calls.
7. Victim's `completeWithdrawal` reverts with `WithdrawalLocked` until `nextLockedNonce[ETH_TOKEN]` reaches `N+1`, i.e., until all `N` dust entries are processed. Their 1 ETH worth of rsETH remains locked in the contract throughout.
