Audit Report

## Title
Unbounded Gas Consumption in `getAssetUnstaking()` Blocks `updateRSETHPrice()` and User Deposits - (File: contracts/NodeDelegator.sol)

## Summary
`NodeDelegator.getAssetUnstaking()` fetches all queued EigenLayer withdrawals via an external call to `getQueuedWithdrawals()` and iterates over them in a nested loop on every invocation. This function is called once per NDC inside `LRTDepositPool.getAssetDistributionData()` and `getETHDistributionData()`, which are in turn called by every `updateRSETHPrice()` and every user deposit. At protocol-configured limits (10 NDCs, 80 queued withdrawals each, 5 supported assets), the cumulative gas cost exceeds Ethereum's 30M block gas limit, permanently blocking price updates and user deposits.

## Finding Description
`NodeDelegator.getAssetUnstaking()` makes one external call to `_getDelegationManager().getQueuedWithdrawals(address(this))` and then iterates over all returned withdrawals and their strategies in a nested loop. [1](#0-0) 

This function is called inside `LRTDepositPool.getAssetDistributionData()` once per NDC in the `nodeDelegatorQueue`: [2](#0-1) 

And again inside `getETHDistributionData()` once per NDC: [3](#0-2) 

`getTotalAssetDeposits()` calls `getAssetDistributionData()` (or `getETHDistributionData()` for ETH): [4](#0-3) 

`LRTOracle._getTotalEthInProtocol()` calls `getTotalAssetDeposits()` once per supported asset, and is invoked by the public, permissionless `updateRSETHPrice()`: [5](#0-4) [6](#0-5) 

Every user deposit calls `_checkIfDepositAmountExceedesCurrentLimit()`, which also calls `getTotalAssetDeposits()`: [7](#0-6) 

There is no caching, snapshotting, or short-circuit guard anywhere in this call chain. The developers themselves acknowledge the gas budget is tight in a comment that caps `maxUncompletedWithdrawalCount` at 80: [8](#0-7) 

The comment states "120 is the max number of uncompleted withdrawals that allows us to still perform update rsETH price," yet the cap is set to 80 per NDC. With 10 NDCs and 5 supported assets, a single `updateRSETHPrice()` call triggers **50 external calls** to `getQueuedWithdrawals()` (each returning up to 80 large withdrawal structs) and up to **8,000 inner loop iterations** — well beyond the 30M gas block limit.

## Impact Explanation
**Medium — Unbounded gas consumption.** When the protocol operates near its configured limits during normal high-withdrawal-activity periods, both `updateRSETHPrice()` and `depositAsset()`/`depositETH()` revert with out-of-gas. This causes rsETH price staleness (fee minting halted, downside-protection auto-pause cannot trigger) and temporary freezing of the user deposit path — matching the allowed Medium impact of "Unbounded gas consumption" and "Temporary freezing of funds."

## Likelihood Explanation
No attacker action is required. The condition is triggered by ordinary protocol usage: users initiating unstaking across multiple NDCs accumulates queued withdrawals. The protocol's own configured limits (10 NDCs, 80 withdrawals/NDC, 5 assets) are the direct cause. The developers' own comment confirms awareness that the gas budget is tight at these numbers. This is a realistic operational scenario, not a theoretical edge case.

## Recommendation
- Maintain an on-chain running total of `assetUnstaking` per NDC per asset, updated atomically on `initiateUnstaking` and `completeUnstaking`, instead of recomputing by iterating EigenLayer's full withdrawal queue on every read.
- Decouple the deposit limit check from the full TVL computation; use a cached/snapshotted value for deposit gating rather than recomputing live on every deposit.
- If live iteration is retained, add a per-block cache (e.g., store the result keyed by `block.number`) so repeated calls within the same block pay the iteration cost only once.

## Proof of Concept
Deploy with 5 supported assets, 10 NDCs, each NDC having 80 queued EigenLayer withdrawals with 2 strategies each. Call `updateRSETHPrice()`:

```
updateRSETHPrice()
  └─ _updateRsETHPrice()
       └─ _getTotalEthInProtocol()              // loops 5 assets
            └─ getTotalAssetDeposits(asset)     // per asset
                 └─ getAssetDistributionData(asset)
                      └─ [loop: 10 NDCs]
                           └─ getAssetUnstaking(asset)   // 50 total calls
                                └─ getQueuedWithdrawals() // external, returns 80 withdrawals
                                     └─ [loop: 80 × 2]  // 8,000 inner iterations
```

A Foundry fork test against mainnet with 10 NDCs each having 80 queued withdrawals will demonstrate the transaction exceeding 30M gas and reverting. The same revert occurs on any `depositAsset()` or `depositETH()` call via `_checkIfDepositAmountExceedesCurrentLimit()` → `getTotalAssetDeposits()`.

```text
    function getAssetUnstaking(address asset) external view returns (uint256 amount) {
        (IDelegationManager.Withdrawal[] memory queuedWithdrawals, uint256[][] memory withdrawalShares) =
            _getDelegationManager().getQueuedWithdrawals(address(this));

        for (uint256 withdrawalIndex = 0; withdrawalIndex < queuedWithdrawals.length; withdrawalIndex++) {
            IDelegationManager.Withdrawal memory withdrawal = queuedWithdrawals[withdrawalIndex];

            for (uint256 strategyIndex = 0; strategyIndex < withdrawal.strategies.length; strategyIndex++) {
                IStrategy strategy = withdrawal.strategies[strategyIndex];

                address strategyAsset = address(strategy) == address(lrtConfig.beaconChainETHStrategy())
                    ? LRTConstants.ETH_TOKEN
                    : address(strategy.underlyingToken());

                if (strategyAsset != asset) continue;

                uint256 sharesToUnstake = withdrawalShares[withdrawalIndex][strategyIndex];
                amount += strategyAsset == LRTConstants.ETH_TOKEN
                    ? sharesToUnstake
                    : strategy.sharesToUnderlyingView(sharesToUnstake);
            }
        }
    }
```

```text
    function _getTotalEthInProtocol() private view returns (uint256 totalETHInProtocol) {
        address lrtDepositPoolAddr = lrtConfig.getContract(LRTConstants.LRT_DEPOSIT_POOL);
```
