Audit Report

## Title
ETH Permanently Frozen When Withdrawal Recipient Is a Contract That Rejects ETH — (`contracts/LRTWithdrawalManager.sol`)

## Summary
A smart contract that calls `initiateWithdrawal` has its rsETH burned irreversibly during `unlockQueue`, but every subsequent call to `completeWithdrawal` or `completeWithdrawalForUser` permanently reverts because `_transferAsset` unconditionally pushes ETH to the original `user` address with no alternative recipient. The ETH is permanently frozen inside `LRTWithdrawalManager` with no admin recovery mechanism, and `sweepRemainingAssets` is permanently blocked by the non-zero `unlockedWithdrawalsCount`.

## Finding Description

**Root cause:** `_transferAsset` performs a bare low-level call and hard-reverts on failure with no fallback path. [1](#0-0) 

**Exploit flow:**

1. A smart contract (e.g., a multisig or DAO treasury) calls `initiateWithdrawal`. There is no EOA check. rsETH is transferred from the caller to the withdrawal manager. [2](#0-1) 

2. An operator calls `unlockQueue`. rsETH held by the contract is burned via `burnFrom(address(this), rsETHBurned)` — irreversible. ETH is redeemed from the unstaking vault into the withdrawal manager. [3](#0-2) 

3. The contract calls `completeWithdrawal`, which routes to `_processWithdrawalCompletion(asset, msg.sender, ...)`. The nonce is popped, the request is deleted, `unlockedWithdrawalsCount[asset]--` is executed, and then `_transferAsset(asset, user, ...)` is called. [4](#0-3) 

4. `_transferAsset` calls `payable(to).call{value: amount}("")`. If `to` is a contract whose `receive()` reverts, `sent` is `false`, and the function reverts with `EthTransferFailed`. The entire transaction reverts, restoring `unlockedWithdrawalsCount` and the nonce queue to their pre-call state. [5](#0-4) 

5. `completeWithdrawalForUser` is equally blocked: the `user` parameter must match the original requester (withdrawal requests are stored under `userAssociatedNonces[asset][user]`), so ETH cannot be redirected to a different address. The NatSpec even acknowledges this path is "Not expected to be used for ETH." [6](#0-5) 

6. `sweepRemainingAssets` is gated by `hasUnlockedWithdrawals(asset)`, which returns `unlockedWithdrawalsCount[asset] > 0`. Since every `completeWithdrawal` call atomically reverts (restoring the count), this check never passes, permanently blocking the sweep. [7](#0-6) [8](#0-7) 

There is no admin function to cancel a stuck withdrawal request, redirect ETH, or decrement `unlockedWithdrawalsCount` manually.

## Impact Explanation
**Critical. Permanent freezing of funds.** The user loses both their rsETH (burned in `unlockQueue`, unrecoverable) and their ETH (moved from the unstaking vault into `LRTWithdrawalManager`, permanently stuck). No recovery path exists in the contract.

## Likelihood Explanation
**Medium.** Smart contract wallets (multisigs, DAO treasuries, ERC-4337 smart accounts) are common DeFi participants. A contract without a `receive()` function, or with a reverting fallback — whether by design, upgrade, or bug — triggers this. The protocol imposes no EOA restriction on `initiateWithdrawal`. No privileged access or external compromise is required; the scenario is triggered entirely through the normal user withdrawal flow.

## Recommendation
1. **Add a `recipient` parameter** to `completeWithdrawal` (or a separate `claimTo(address recipient)` function) so the caller can redirect ETH to an address capable of receiving it.
2. **Alternatively**, implement a pull-payment pattern: credit the user's claimable ETH balance on completion and let them pull it separately, avoiding push-to-contract failures.
3. **Add an admin escape hatch** — a manager-callable function to cancel a stuck unlocked withdrawal, return the ETH to the treasury, and decrement `unlockedWithdrawalsCount` — so `sweepRemainingAssets` can eventually clear the contract.

    function _processWithdrawalCompletion(address asset, address user, string calldata referralId) internal {
        if (userAssociatedNonces[asset][user].empty()) {
            revert NoWithdrawalRequests(user, asset);
        }

        // Retrieve and remove the oldest withdrawal request for the user.
        uint256 usersFirstWithdrawalRequestNonce = userAssociatedNonces[asset][user].popFront();
        // Ensure the request is already unlocked.
        if (usersFirstWithdrawalRequestNonce >= nextLockedNonce[asset]) revert WithdrawalLocked();

        bytes32 requestId = getRequestId(asset, usersFirstWithdrawalRequestNonce);
        WithdrawalRequest memory request = withdrawalRequests[requestId];

        delete withdrawalRequests[requestId];

        // Check that the withdrawal delay has passed since the request's initiation.
        if (block.number < request.withdrawalStartBlock + withdrawalDelayBlocks) revert WithdrawalDelayNotPassed();

        unlockedWithdrawalsCount[asset]--;

        // If Aave integration is enabled and asset is ETH, withdraw from Aave if needed
        if (isAaveIntegrationEnabled && asset == LRTConstants.ETH_TOKEN) {

            if (contractBalance < request.expectedAssetAmount) {
                uint256 amountNeeded = request.expectedAssetAmount - contractBalance;
                _withdrawFromAave(amountNeeded);

                uint256 balanceAfter = address(this).balance;
                if (balanceAfter < request.expectedAssetAmount) {
                    revert InsufficientLiquidityForWithdrawal();
                }
            }
        }

        _transferAsset(asset, user, request.expectedAssetAmount);
