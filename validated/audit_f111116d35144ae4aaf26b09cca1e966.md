### Title
Missing `beneficiary_id != account_id` Sanity Check in `DeleteAccountAction` Allows Permanent Fund Loss - (File: runtime/runtime/src/action_validation.rs)

### Summary
`validate_delete_action` only checks that `beneficiary_id` is a syntactically valid account ID. It does not check that `beneficiary_id` differs from the account being deleted. When a user (or a contract via `promise_batch_action_delete_account`) sets `beneficiary_id` equal to the account being deleted, the balance refund receipt is dispatched to the now-deleted account. The transfer fails, and because the receipt carries `predecessor_id = "system"`, the protocol burns the amount with no recovery path.

### Finding Description
In `validate_delete_action`, the only check performed on `DeleteAccountAction` is a format-validity check on `beneficiary_id`:

```rust
fn validate_delete_action(action: &DeleteAccountAction) -> Result<(), ActionsValidationError> {
    validate_action_account_id(&action.beneficiary_id)?;
    Ok(())
}
``` [1](#0-0) 

There is no check that `beneficiary_id != account_id`. In `action_delete_account`, the account is removed from state and a `Receipt::new_balance_refund` is pushed to `result.new_receipts` targeting `beneficiary_id`:

```rust
if account_balance > Balance::ZERO {
    result
        .new_receipts
        .push(Receipt::new_balance_refund(&delete_account.beneficiary_id, account_balance));
}
let remove_result = remove_account(state_update, account_id)?;
``` [2](#0-1) 

`Receipt::new_balance_refund` sets `predecessor_id = "system"`: [3](#0-2) 

When `beneficiary_id == account_id`, the refund receipt targets the account that was just deleted. In the subsequent block, the runtime attempts a `Transfer` to the non-existent account. The action fails with `AccountDoesNotExist`. Because the receipt's `predecessor_id` is `"system"`, the protocol's refund rule applies: **"If the execution of a refund fails, the refund amount is burnt."** No further recovery receipt is generated.

The same path is reachable from a contract via the host function `promise_batch_action_delete_account`: [4](#0-3) 

which appends the action without any check that `beneficiary_id != current_account_id`.

### Impact Explanation
The entire liquid balance of the deleted account is permanently burnt. The exact corrupted protocol value is the `amount` field of the account's state entry — it is debited at deletion time and the corresponding credit (the refund receipt) is silently destroyed. The state root after the refund receipt's failure reflects a net loss of the full account balance with no on-chain record of where it went.

### Likelihood Explanation
An unprivileged user can trigger this by submitting a signed `DeleteAccount` transaction (via public RPC) with `beneficiary_id` set to their own `receiver_id`. A deployed contract can trigger it programmatically via `promise_batch_action_delete_account` — a realistic mistake for any contract that constructs delete actions dynamically (e.g., self-destructing contracts that pass `current_account_id()` as the beneficiary). Neither path requires any elevated privilege.

### Recommendation
Add a check in `validate_delete_action` (or at the top of `action_delete_account`) analogous to the original report's `recovery != 0` / `agent != 0` recommendations:

```rust
fn validate_delete_action(
    action: &DeleteAccountAction,
    receiver: &AccountId,          // the account being deleted
) -> Result<(), ActionsValidationError> {
    validate_action_account_id(&action.beneficiary_id)?;
    if &action.beneficiary_id == receiver {
        return Err(ActionsValidationError::DeleteAccountBeneficiaryIsSelf);
    }
    Ok(())
}
```

The `receiver` is already available at the call site in `validate_action_with_mode`.
