### Title
`tierRefScriptFee` Superlinear Pricing Bypassed by Splitting Reference Script Load Across Transactions - (`eras/conway/impl/src/Cardano/Ledger/Conway/Tx.hs`)

---

### Summary

The Conway era introduced a tiered, exponentially-growing fee formula (`tierRefScriptFee`) for reference scripts to deter DDoS attacks. Because this fee is computed **per transaction** and resets to the base tier for each new transaction, an attacker can split a large reference script load across multiple transactions to stay permanently in the cheapest tier, paying a fraction of the fee that a single large transaction would incur. This directly undermines the superlinear pricing deterrent and modifies fees outside design parameters.

---

### Finding Description

The `tierRefScriptFee` function in `eras/conway/impl/src/Cardano/Ledger/Conway/Tx.hs` implements an exponentially increasing fee for reference scripts:

```haskell
tierRefScriptFee multiplier sizeIncrement
  | multiplier <= 0 || sizeIncrement <= 0 = error "..."
  | otherwise = go 0
  where
    go !acc !curTierPrice !n
      | n < sizeIncrement =
          Coin $ floor (acc + toRational n * curTierPrice)
      | otherwise =
          go (acc + sizeIncrementRational * curTierPrice) (multiplier * curTierPrice) (n - sizeIncrement)
``` [1](#0-0) 

For Conway, the hardcoded parameters are: multiplier = 1.2, stride = 25,600 bytes (25 KiB), base cost = 15 lovelace/byte: [2](#0-1) 

This fee is applied **per transaction** via `getConwayMinFeeTx`: [3](#0-2) 

which calls `txNonDistinctRefScriptsSize` to measure only the current transaction's reference scripts: [4](#0-3) 

The per-transaction limit (`maxRefScriptSizePerTx` = 200 KiB) and per-block limit (`maxRefScriptSizePerBlock` = 1 MiB) are enforced separately: [5](#0-4) [6](#0-5) 

**The fee counter resets to zero for every new transaction.** There is no cumulative fee accounting across transactions in a block.

**Numerical demonstration** (Conway mainnet parameters: multiplier=1.2, stride=25,600, baseFee=15 lovelace/byte):

| Strategy | Total ref-script bytes | Total fee paid |
|---|---|---|
| 1 tx × 200 KiB | 204,800 | ~6,335,648 lovelace |
| 8 txs × 25 KiB | 204,800 | 8 × 384,000 = **3,072,000 lovelace** |
| 40 txs × 25 KiB (1 MiB block) | 1,048,576 | 40 × 384,000 = **15,360,000 lovelace** |
| 5 txs × 200 KiB (1 MiB block) | 1,048,576 | 5 × 6,335,648 = **31,678,240 lovelace** |

Splitting 200 KiB across 8 transactions costs **~51% less** than a single 200 KiB transaction, for identical total reference script bytes processed. For a full 1 MiB block, splitting saves ~51% of the intended fee.

The design rationale explicitly states the goal is to make large reference script usage expensive to deter DDoS: [7](#0-6) 

---

### Impact Explanation

This falls under: **Medium. Attacker-controlled transactions modify fees outside design parameters.**

The superlinear fee formula is the protocol's primary economic deterrent against reference-script DDoS attacks (as documented in ADR-009 and motivated by the June 2024 mainnet attack). By splitting reference script usage across multiple transactions, an attacker reduces the effective fee to the base linear rate, paying approximately half the intended fee for the same total reference script load per block. The `maxRefScriptSizePerBlock` limit still caps total bytes, but the fee deterrent — the core security mechanism — is undermined. The protocol collects materially less fee revenue than the formula intends for a given attack load.

---

### Likelihood Explanation

The attack requires no special privileges: any transaction submitter can craft multiple transactions each referencing 25 KiB of scripts. The optimal split size (one stride = 25,600 bytes per transaction) is trivially computable from the public protocol parameters. The block limit (1 MiB) still bounds the total attack surface per block, but the cost to fill a block with reference scripts is ~51% lower than the protocol intends. This is directly reachable by an unprivileged transaction sender.

---

### Recommendation

Apply the tiered fee to the **cumulative** reference script size across all transactions in a block, not per transaction. Alternatively, enforce that the fee multiplier applies to the aggregate per-block reference script bytes, so that splitting provides no fee advantage. In the Dijkstra era, where `refScriptCostStride` and `refScriptCostMultiplier` become proper protocol parameters, the block-level aggregation should be introduced alongside them. [8](#0-7) 

---

### Proof of Concept

Using the Conway hardcoded parameters (`multiplier = 1.2`, `stride = 25600`, `baseFee = 15`):

**Single transaction with 200 KiB (maximum per-tx):**
```
tierRefScriptFee 1.2 25600 15 204800
= tier1: 25600*15       = 384,000
+ tier2: 25600*18       = 460,800
+ tier3: 25600*21.6     = 552,960
+ tier4: 25600*25.92    = 663,552
+ tier5: 25600*31.104   = 796,262
+ tier6: 25600*37.3248  = 955,515
+ tier7: 25600*44.78976 = 1,146,618
+ tier8: 25600*53.74771 = 1,375,941
= 6,335,648 lovelace
```

**Eight transactions each with 25 KiB (same total bytes):**
```
8 × tierRefScriptFee 1.2 25600 15 25600
= 8 × (25600 * 15)
= 8 × 384,000
= 3,072,000 lovelace
```

**Savings: 3,263,648 lovelace (~51.5%)** for identical total reference script bytes processed. The existing test at line 41–47 of `eras/conway/impl/test/Main.hs` confirms the tiered values but does not test the cross-transaction splitting scenario: [9](#0-8)
