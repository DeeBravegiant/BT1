### Title
Single Byzantine MPC Participant Can Replay Foreign-Chain Verification Signatures Across Requests - (File: `crates/contract/src/lib.rs`)

### Summary
The `respond_verify_foreign_tx` function verifies that a supplied `response.payload_hash` is signed by the MPC root key, but never verifies that `payload_hash` was actually derived from the pending `request`. A single malicious attested MPC participant (below signing threshold) can replay any previously observed `(payload_hash, signature)` pair from an old on-chain `respond_verify_foreign_tx` call to satisfy a completely different pending `verify_foreign_transaction` request. The contract accepts the fraudulent response, and the caller receives a signed attestation for a foreign-chain state that was never verified for their transaction.

### Finding Description

**Root cause — `respond_verify_foreign_tx` does not bind `payload_hash` to the pending `request`.**

In `respond_verify_foreign_tx` (lines 692–754 of `crates/contract/src/lib.rs`), the contract performs two checks:

1. The caller is an attested participant.
2. `response.signature` is a valid ECDSA signature over `response.payload_hash` under the root public key.

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;
// Check the signature is correct against the root public key
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,
)
.is_ok()
```

`payload_hash` is taken verbatim from the caller-supplied `response` — it is never verified to be the hash of a `ForeignTxSignPayload` whose embedded `request` field matches the pending `VerifyForeignTransactionRequest` that is being resolved.

Compare this with the regular `respond` function (lines 600–608), where the payload hash is derived directly from the stored request, not from the response:

```rust
let payload_hash = request.payload.as_ecdsa().expect("Payload is not ECDSA");
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    payload_hash,
    &expected_public_key,
)
```

The `ForeignTxSignPayload` that the MPC nodes are supposed to sign encodes both the original request and the extracted foreign-chain values:

```rust
pub struct ForeignTxSignPayloadV1 {
    pub request: ForeignChainRpcRequest,
    pub values: Vec<ExtractedValue>,
}
```

Because the contract never reconstructs or checks this payload, a caller can supply any `payload_hash` that has a valid root-key signature — including one harvested from a completely different, already-settled `respond_verify_foreign_tx` transaction visible on-chain.

**Exploit path:**

1. Attacker is a single attested MPC participant (below threshold).
2. Attacker observes any past `respond_verify_foreign_tx` transaction on-chain and extracts `(payload_hash_old, sig_old)` from the `VerifyForeignTransactionResponse` argument.
3. A victim submits `verify_foreign_transaction(request=bitcoin_tx_456, confirmations=6)` — a high-value bridge inbound transfer.
4. Attacker immediately calls `respond_verify_foreign_tx(request=bitcoin_tx_456, response={payload_hash=payload_hash_old, signature=sig_old})`.
5. The contract checks: `sig_old` is a valid root-key signature over `payload_hash_old` ✓; `bitcoin_tx_456` is a pending request ✓.
6. `pending_requests::resolve_yields_for` resolves the yield, delivering `{payload_hash_old, sig_old}` to the victim's callback.
7. The victim (or bridge contract) receives a signed attestation whose `payload_hash` encodes a completely different foreign-chain observation (e.g., `bitcoin_tx_123` with 1 confirmation), not the 6-confirmation confirmation of `bitcoin_tx_456` they requested.

### Impact Explanation

A bridge contract that calls `verify_foreign_transaction` and trusts the returned `VerifyForeignTransactionResponse` to gate an inbound mint or unlock will be presented with a cryptographically valid root-key signature over a `payload_hash` that encodes the wrong transaction and/or wrong confirmation depth. Because the bridge contract cannot reconstruct the full `ForeignTxSignPayload` from the hash alone (it does not know the extracted values), it cannot detect the substitution. This enables:

- **Invalid bridge execution**: an inbound transfer for `bitcoin_tx_456` is processed using an attestation that was actually produced for `bitcoin_tx_123`.
- **Double-spend**: the attacker can reuse the same `(payload_hash_old, sig_old)` pair to satisfy multiple independent pending requests for different transactions, causing the bridge to mint/unlock funds for transactions that were never independently verified.

This matches the allowed impact: *"High. Cross-chain replay, forged foreign-chain verification, light-client-style verification bypass, or participant/attestation authorization bypass that causes invalid bridge execution or double-spend conditions."*

### Likelihood Explanation

- Requires only a **single** Byzantine attested MPC participant — strictly below the signing threshold.
- All inputs needed for the attack (`payload_hash_old`, `sig_old`) are **publicly visible on-chain** in past `respond_verify_foreign_tx` call arguments; no off-chain secret is needed.
- The attacker does not need to forge any cryptographic material; they only replay existing, valid data.
- The window of opportunity is any time a `verify_foreign_transaction` request is pending, which is the normal operating state of any active bridge.

### Recommendation

The contract must verify that `response.payload_hash` was derived from the pending `request`. The simplest fix is to require the responder to supply the full `ForeignTxSignPayload` in the response, then:

1. Verify `payload_hash == payload.compute_msg_hash()`.
2. Verify `payload.request() == request.request` (the embedded request matches the pending one).
3. Verify the signature over `payload_hash` as today.

This binds the signature to the specific pending request and makes cross-request replay impossible, mirroring the pattern already used in `respond` where the payload hash is derived from the stored request rather than accepted from the caller.

```rust
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }
```
