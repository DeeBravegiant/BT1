Audit Report

## Title
Reward Tokens Permanently Locked When `totalKernelStaked` Drops to Zero During Active Reward Period - (File: contracts/KERNEL/KernelDepositPool.sol)

## Summary
`KernelDepositPool` implements a Synthetix-style staking rewards model where `rewardPerToken()` short-circuits to return the stored value unchanged when `totalKernelStaked == 0`. If all stakers call `initiateWithdrawal` during an active reward window, the remaining undistributed reward tokens — computed as `rewardRate × (finishAt − updatedAt)` — become permanently locked in the contract with no on-chain recovery path. The contract's own NatSpec explicitly acknowledges this risk and relies solely on off-chain coordination that is not enforced by any contract logic.

## Finding Description
The root cause is in `rewardPerToken()` ( [1](#0-0) ): when `totalKernelStaked == 0`, it returns `rewardPerTokenStored` unchanged, meaning no rewards accrue to any address for the zero-staking interval.

The exploit path:
1. Admin calls `notifyRewardAmount(N)` while at least one staker is present (the `totalKernelStaked == 0` guard at L570 is satisfied). `rewardRate` and `finishAt` are set. [2](#0-1) 
2. The last staker calls `initiateWithdrawal`. The `updateReward(msg.sender)` modifier fires first, snapshotting `rewardPerTokenStored` and `updatedAt` at the current timestamp. Then `totalKernelStaked` is decremented to zero. [3](#0-2) 
3. For the remainder of the reward window (`finishAt − updatedAt`), every call to `rewardPerToken()` returns the frozen `rewardPerTokenStored`. No staker earns the remaining `rewardRate × (finishAt − updatedAt)` tokens. [4](#0-3) 
4. No admin function (`setRewardsDuration`, `notifyRewardAmount`, `setWithdrawalDelay`, `setMaxNumberOfWithdrawalsPerUser`) provides a mechanism to recover these tokens. The contract has no `recoverTokens` or equivalent function.

The contract's own NatSpec at L18–23 explicitly acknowledges this: *"If `totalKernelStaked` ever hits zero during a reward distribution window, any remaining rewards for that period will stay locked in the contract."* The stated mitigation — ensuring tokens are always staked — is entirely off-chain and unenforceable by the contract. [5](#0-4) 

The guard in `notifyRewardAmount` (`if (totalKernelStaked == 0) revert NoStakedTokens()`) only prevents starting a new reward period with zero stakers; it does not prevent stakers from withdrawing to zero after a period has already started. [6](#0-5) 

## Impact Explanation
Reward tokens deposited via `notifyRewardAmount` are permanently frozen inside `KernelDepositPool` with no recovery path. This constitutes **Medium — Permanent freezing of unclaimed yield**. The magnitude is `rewardRate × (finishAt − block.timestamp)` tokens at the moment `totalKernelStaked` reaches zero, which can represent a substantial portion of the deposited reward allocation.

## Likelihood Explanation
The trigger requires all stakers to call `initiateWithdrawal` during an active reward period. This is realistic when: (a) the staker set is small (e.g., early deployment); (b) a single dominant staker exits; or (c) market conditions incentivize mass unstaking. `initiateWithdrawal` is fully permissionless and reduces `totalKernelStaked` immediately upon call — before the `withdrawalDelay` elapses — so the lock-in occurs at the moment of the last withdrawal initiation, not at claim time. [7](#0-6) 

## Recommendation
Add a permissioned `recoverLockedRewards()` function that transfers stranded reward tokens to the treasury or admin when `totalKernelStaked == 0` and the reward period has ended (or is active with zero stakers). Alternatively, track elapsed zero-staking time and exclude it from the reward window so `rewardRate` is effectively paused, resuming when staking returns — this is the standard fix used in updated Synthetix-style implementations.

## Proof of Concept
1. Deploy `KernelDepositPool`. Alice stakes 100 KERNEL.
2. Admin calls `notifyRewardAmount(1_000e18)` with `duration = 100 days`. `rewardRate = 1_000e18 / (100 days)`, `finishAt = block.timestamp + 100 days`.
3. Warp 50 days. Alice calls `initiateWithdrawal(100)`.
   - `updateReward(alice)` snapshots `rewardPerTokenStored` and `updatedAt = block.timestamp`.
   - `totalKernelStaked` becomes 0.
4. Warp to `finishAt`. Assert: `rewardPerToken()` still returns the value snapshotted in step 3 (unchanged). Assert: `earned(alice) ≈ 500e18` (her 50-day share). Assert: `rewardsToken.balanceOf(address(pool)) ≈ 500e18` (the remaining 50-day allocation, permanently locked). Assert: no function call can extract these tokens.

```text
    function notifyRewardAmount(uint256 _amount) external onlyRole(DEFAULT_ADMIN_ROLE) updateReward(address(0)) {
        if (_amount == 0) revert AmountZero();

        // Prevent starting a reward period when no tokens are staked to avoid unallocated rewards
        if (totalKernelStaked == 0) revert NoStakedTokens();

        // Transfer reward tokens into the contract
        uint256 balanceBefore = rewardsToken.balanceOf(address(this));
        rewardsToken.safeTransferFrom(msg.sender, address(this), _amount);
        uint256 balanceAfter = rewardsToken.balanceOf(address(this));
        // Calculate the actual amount of tokens received in case of a transfer fee (tax)
        uint256 receivedAmount = balanceAfter - balanceBefore;

        if (block.timestamp >= finishAt) {
            rewardRate = receivedAmount / duration;
        } else {
            uint256 remaining = (finishAt - block.timestamp) * rewardRate;
            rewardRate = (receivedAmount + remaining) / duration;
        }

        if (rewardRate == 0) revert RewardRateZero();

        finishAt = block.timestamp + duration;
        updatedAt = block.timestamp;

        emit NotifyRewardAmount(receivedAmount, finishAt);
```
