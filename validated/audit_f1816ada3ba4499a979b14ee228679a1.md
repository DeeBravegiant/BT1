### Title
Stale Attestation Accepted in Per-Call Authorization Check — (`crates/contract/src/tee/tee_state.rs`)

### Summary

`is_caller_an_attested_participant` verifies that a stored attestation *exists* and matches the caller's identity, but never checks whether the attestation has expired. This is the direct analog of the Chainlink `latestAnswer()` pattern: the code confirms the data is present (`price > 0` / attestation found) but omits the freshness check (`answeredInRound >= roundId` / `expiry_timestamp_seconds > now`). As a result, a participant node whose TEE attestation has lapsed — or whose image hash has been removed from the governance allowlist — can continue to call `respond_ckd` and `respond_verify_foreign_tx` (and the equivalent signing `respond`) until `verify_tee` is separately invoked to sweep the stale entry.

### Finding Description

**Root cause — missing expiry check in `is_caller_an_attested_participant`**

`is_caller_an_attested_participant` in `crates/contract/src/tee/tee_state.rs` performs four checks: [1](#0-0) 

1. Caller is in the active participant set.
2. A `NodeAttestation` entry exists for the caller's TLS key.
3. The stored `account_id` matches the signer.
4. The stored `account_public_key` matches the signer's Ed25519 key.

It does **not** check `verified_attestation.expiry_timestamp_seconds` against the current block timestamp. The expiry field is populated at submission time: [2](#0-1) 

and is enforced only inside `re_verify`, which is called exclusively from `reverify_and_cleanup_participants` → `verify_tee`: [3](#0-2) 

**Call path that reaches the vulnerable check**

`assert_caller_is_attested_participant_and_protocol_active` delegates directly to `is_caller_an_attested_participant`: [4](#0-3) 

This guard is invoked on every signing-response path:

- `respond_ckd` — line 666 [5](#0-4) 
- `respond_verify_foreign_tx` — line 705 [6](#0-5) 

**Contrast with the periodic sweep**

`verify_tee` → `reverify_and_cleanup_participants` does perform a full re-verification including expiry and allowlist membership, and removes stale entries: [7](#0-6) 

But `verify_tee` is a separate, manually-triggered call. Between the moment an attestation expires (or an image hash is revoked) and the next `verify_tee` invocation, the stale entry remains in `stored_attestations` and `is_caller_an_attested_participant` returns `Ok(())` for it.

**Concrete exploit path**

1. Governance votes to remove image hash `H` from the allowlist (e.g., a vulnerability is discovered in the running MPC binary).
2. Node N, currently running image `H`, can no longer submit a valid `submit_participant_info` — the `add_participant` call would fail the image-hash check.
3. Node N's existing `VerifiedAttestation::Dstack` entry remains in `stored_attestations` with its original `expiry_timestamp_seconds = submission_time + DEFAULT_EXPIRATION_DURATION_SECONDS` (up to 1 day in the future).
4. `verify_tee` has not yet been called (or was last called before the revocation).
5. Node N calls `respond_verify_foreign_tx` with a crafted response. `is_caller_an_attested_participant` finds the entry, matches account ID and public key, and returns `Ok(())`. The contract accepts the response.
6. Node N continues to contribute signature shares to threshold operations for up to `DEFAULT_EXPIRATION_DURATION_SECONDS` (currently 1 day) after its image hash was revoked. [8](#0-7) 

### Impact Explanation

A participant node running a revoked or expired TEE image can continue to submit signing responses (`respond`, `respond_ckd`, `respond_verify_foreign_tx`) for up to one day after governance removes its image hash, because the per-call authorization gate never checks `expiry_timestamp_seconds`. This breaks the production safety invariant that only currently-attested (TEE-verified) nodes may contribute to threshold operations. It is a **participant-state manipulation that breaks production safety/accounting invariants** — matching the Medium allowed impact tier. A single node below the signing threshold cannot forge a complete signature unilaterally, which prevents this from reaching Critical.

### Likelihood Explanation

The window is bounded by `DEFAULT_EXPIRATION_DURATION_SECONDS` (1 day) and closes as soon as any participant calls `verify_tee`. However, `verify_tee` is not called automatically; it requires an explicit transaction from a participant. In a scenario where governance revokes an image hash due to a discovered vulnerability, the revoked node has a concrete incentive to delay or prevent `verify_tee` from being called (e.g., by not calling it itself and hoping other participants are slow). The precondition — being an existing participant — is reachable without any privileged access beyond normal MPC participation.

### Recommendation

Add an expiry check inside `is_caller_an_attested_participant`, mirroring the check already present in `re_verify`:

```rust
// After retrieving `attestation` from stored_attestations:
let current_time_seconds = Self::current_time_seconds();
let is_expired = match &attestation.verified_attestation {
    VerifiedAttestation::Dstack(d) => d.expiry_timestamp_seconds < current_time_seconds,
    VerifiedAttestation::Mock(MockAttestation::WithConstraints {
        expiry_timestamp_seconds: Some(exp), ..
    }) => *exp < current_time_seconds,
    _ => false,
};
if is_expired {
    return Err(AttestationCheckError::AttestationExpired);
}
```

This makes the per-call authorization gate self-contained and independent of whether `verify_tee` has been called recently — the same fix the Chainlink report recommended (use `latestRoundData` + check `updatedAt > 0` rather than relying on a separate staleness-sweep).

### Proof of Concept

1. Deploy the contract with 3 participants (threshold 2), all with `MockAttestation::WithConstraints { expiry_timestamp_seconds: Some(T) }` where `T` is 5 seconds in the future.
2. Fast-forward 100 blocks (past `T`) without calling `verify_tee`.
3. Have participant 1 (expired attestation) call `respond_verify_foreign_tx` with a valid-format response.
4. Observe that `is_caller_an_attested_participant` returns `Ok(())` and the contract accepts the response — despite the attestation having expired — because `expiry_timestamp_seconds` is never read in that function.

The existing sandbox test `clean_invalid_attestations__should_remove_expired_entries` already demonstrates that expired entries survive in `stored_attestations` until an explicit sweep is triggered, confirming the window exists. [9](#0-8)

    pub(crate) fn is_caller_an_attested_participant(
        &self,
        participants: &Participants,
    ) -> Result<(), AttestationCheckError> {
        let signer_account_pk = env::signer_account_pk();
        let signer_id = env::signer_account_id();

        let info = participants
            .info(&signer_id)
            .ok_or(AttestationCheckError::CallerNotParticipant)?;

        let attestation = self
            .stored_attestations
            .get(&info.tls_public_key)
            .ok_or(AttestationCheckError::AttestationNotFound)?;

        if attestation.node_id.account_id != signer_id {
            return Err(AttestationCheckError::AttestationOwnerMismatch);
        }

        // Stored account keys are Ed25519 by construction; a non-Ed25519
        // signer necessarily mismatches.
        let signer_ed25519 = Ed25519PublicKey::try_from(&signer_account_pk)
            .map_err(|_| AttestationCheckError::AttestationKeyMismatch)?;
        if attestation.node_id.account_public_key != signer_ed25519 {
            return Err(AttestationCheckError::AttestationKeyMismatch);
        }

        Ok(())
    }

/// **`clean_invalid_attestations` end-to-end** — an attestation whose expiry has passed
/// is evicted from `stored_attestations` when the endpoint is invoked. Restores the
/// functional cleanup-path coverage previously asserted via `clean_tee_status`.
#[tokio::test]
async fn clean_invalid_attestations__should_remove_expired_entries() -> Result<()> {
    // `verify()` at insert time rejects attestations that are already expired, so the
    // expiring attestation is submitted with an expiry a few seconds in the future and
    // the test then fast-forwards past it. 100 blocks is enough that the block
    // timestamp reliably advances past a 5-second expiry window.
    const ATTESTATION_EXPIRY_SECONDS: u64 = 5;
    const BLOCKS_TO_FAST_FORWARD: u64 = 100;

    // Given
    let SandboxTestSetup {
        worker,
        contract,
        mut mpc_signer_accounts,
        ..
    } = SandboxTestSetup::builder()
        .with_protocols(ALL_PROTOCOLS)
        .build()
        .await;

    // Submit a structurally-valid attestation for every current participant so those
    // entries survive the sweep.
    let participants: Participants =
        (&assert_running_return_participants(&contract).await?).into_contract_type();
    let participant_uids = build_sandbox_node_ids(&participants, &mpc_signer_accounts);
    submit_tee_attestations(&contract, &mut mpc_signer_accounts, &participant_uids).await?;

    // Submit an attestation from a non-participant that will expire shortly.
    let (stale_accounts, _stale_participants) = gen_accounts(&worker, 1).await;
    let stale_account = &stale_accounts[0];
    let stale_tls_key: dtos::Ed25519PublicKey = p2p_tls_key().into();
    let block_info = worker.view_block().await?;
    let expiry_timestamp_seconds =
        block_info.timestamp() / 1_000_000_000 + ATTESTATION_EXPIRY_SECONDS;
    let expiring_attestation = Attestation::Mock(MockAttestation::WithConstraints {
        mpc_docker_image_hash: None,
        launcher_docker_compose_hash: None,
        expiry_timestamp_seconds: Some(expiry_timestamp_seconds),
        expected_measurements: None,
    });
    let submit_result = submit_participant_info(
        stale_account,
        &contract,
        &expiring_attestation,
        &stale_tls_key,
    )
    .await?;
    assert!(submit_result.is_success());

    let before_cleanup = get_tee_accounts(&contract).await?;
    assert_eq!(before_cleanup.len(), participant_uids.len() + 1);

    // Advance past the expiry.
    worker.fast_forward(BLOCKS_TO_FAST_FORWARD).await?;

    // When: any account calls `clean_invalid_attestations` with a scan budget large enough
    // to cover every stored entry.
    let scan_budget: u32 = (before_cleanup.len() as u32) + 1;
    let result = contract
        .as_account()
        .call(contract.id(), method_names::CLEAN_INVALID_ATTESTATIONS)
        .args_json(serde_json::json!({ "max_scan": scan_budget }))
        .transact()
        .await?;
    assert!(result.is_success());

    // Then: the expired entry is evicted while the valid participant entries remain.
    let after_cleanup = get_tee_accounts(&contract).await?;
    assert_eq!(after_cleanup, participant_uids);

    Ok(())
}
