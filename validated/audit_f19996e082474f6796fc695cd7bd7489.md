Audit Report

## Title
`NodeDelegator.getAssetUnstaking()` Nested Loop Multiplied by Asset Count Causes `LRTOracle.updateRSETHPrice()` Gas Exhaustion — (File: `contracts/NodeDelegator.sol`)

## Summary

`NodeDelegator.getAssetUnstaking()` issues an external `getQueuedWithdrawals()` call and iterates over all queued withdrawals every time it is invoked. Because `LRTOracle.updateRSETHPrice()` (public, no auth) ultimately calls this function `numSupportedAssets × numNDCs` times, the total gas cost scales as `numAssets × totalQueuedWithdrawals`, not just `totalQueuedWithdrawals`. The protocol's own cap of 80 withdrawals was calibrated without this multiplier, so at realistic asset and NDC counts the transaction can exhaust the block gas limit, permanently staling `rsETHPrice` and blocking fee minting.

## Finding Description

**Call chain:**

`updateRSETHPrice()` (public) → `_updateRsETHPrice()` → `_getTotalEthInProtocol()` loops over every supported asset and calls `ILRTDepositPool.getTotalAssetDeposits(asset)` per asset. `getTotalAssetDeposits()` calls `getAssetDistributionData()` (or `getETHDistributionData()`), both of which loop over every NDC in `nodeDelegatorQueue` and call `INodeDelegator(ndc).getAssetUnstaking(asset)` per NDC.

`getAssetUnstaking()` unconditionally calls `delegationManager.getQueuedWithdrawals(address(this))` — fetching **all** queued withdrawals for that NDC regardless of which asset is being queried — and then runs a nested loop over every withdrawal and every strategy inside it:

```solidity
// contracts/NodeDelegator.sol L405-427
function getAssetUnstaking(address asset) external view returns (uint256 amount) {
    (IDelegationManager.Withdrawal[] memory queuedWithdrawals, uint256[][] memory withdrawalShares) =
        _getDelegationManager().getQueuedWithdrawals(address(this));   // external call every invocation

    for (uint256 withdrawalIndex = 0; ...) {
        for (uint256 strategyIndex = 0; ...) {
            ...
            strategy.sharesToUnderlyingView(sharesToUnstake);           // external call per match
        }
    }
}
```

The same full withdrawal list is re-fetched for every asset. With 5 supported assets and 10 NDCs this produces **50 external `getQueuedWithdrawals` calls** per `updateRSETHPrice()` invocation. At 80 total queued withdrawals (8 per NDC) that is 400 withdrawal-level iterations plus up to 800 `sharesToUnderlyingView` calls — all inside a single view transaction.

**Why the existing cap is insufficient:**

`LRTUnstakingVault.setMaxUncompletedWithdrawalCount()` enforces a hard cap of 80 and its own comment states *"120 is the max number of uncompleted withdrawals that allows us to still perform update rsETH price"*. This calibration treats the withdrawal count as the sole cost driver. It does not account for the `numAssets` multiplier: with 5 assets the effective iteration count at 80 withdrawals equals what 400 withdrawals would cost in a single-asset model — well above the acknowledged 120-withdrawal ceiling.

Additionally, `setUncompletedWithdrawalCount()` can resync `uncompletedWithdrawalCount` to the live EigenLayer state (e.g., after forced undelegations), potentially allowing the real on-chain count to exceed the soft cap before operators can react and reduce it.

## Impact Explanation

If `updateRSETHPrice()` reverts with out-of-gas:
- `rsETHPrice` in `LRTOracle` becomes permanently stale.
- Protocol fee minting inside `_updateRsETHPrice()` is blocked — **permanent freezing of unclaimed yield** (allowed Medium impact).
- `updateRSETHPriceAsManager()` calls the same `_updateRsETHPrice()` path and is equally blocked.
- New deposits and `unlockQueue()` in `LRTWithdrawalManager` consume the stale price, causing incorrect rsETH minting and withdrawal payouts.

**Severity: Medium — Permanent freezing of unclaimed yield / Unbounded gas consumption.**

## Likelihood Explanation

`updateRSETHPrice()` is public with no access control (only `whenNotPaused`). No attacker action is required: normal operator activity (`initiateUnstaking()`, `undelegate()`) fills the withdrawal queue toward the 80-withdrawal cap. The risk grows monotonically as the protocol adds supported assets or NDCs, and is reachable by any external caller once the queue is sufficiently populated.

## Recommendation

1. **Cache per-NDC withdrawals once per price update**: compute `getAssetUnstaking()` for all assets in a single pass over `getQueuedWithdrawals()` per NDC, eliminating the `numAssets` multiplier on external calls.
2. **Decouple unstaking accounting from the price path**: store per-asset unstaking amounts in storage and update them lazily when withdrawals are initiated or completed, so `updateRSETHPrice()` reads a cached value instead of recomputing it.
3. **Correct the cap formula**: if the on-chain approach is kept, enforce `maxUncompletedWithdrawalCount ≤ safeLimit / numSupportedAssets` rather than a fixed absolute ceiling that ignores the asset multiplier.

## Proof of Concept

**Minimal call sequence (no privileged access required):**

1. Protocol reaches normal operating state: 5 supported assets, 10 NDCs, operators have called `initiateUnstaking()` until `uncompletedWithdrawalCount` approaches 80 (permitted by the current cap).
2. Any EOA calls `LRTOracle.updateRSETHPrice()`.
3. Execution path:
   ```
   updateRSETHPrice()                          // public, whenNotPaused
     _updateRsETHPrice()
       _getTotalEthInProtocol()
         for each asset (×5):
           LRTDepositPool.getTotalAssetDeposits(asset)
             getAssetDistributionData(asset)
               for each NDC (×10):
                 NodeDelegator.getAssetUnstaking(asset)
                   delegationManager.getQueuedWithdrawals(ndc)  // ×50 external calls
                   for each withdrawal (×8):
                     for each strategy (×2):
                       strategy.sharesToUnderlyingView(...)      // ×800 external calls
   ```
4. Total: 50 `getQueuedWithdrawals` + 800 `sharesToUnderlyingView` external calls in one transaction → out-of-gas revert → `rsETHPrice` permanently stale, fee minting permanently blocked.

**Foundry fork test plan:** Fork mainnet, configure 5 assets and 10 NDCs, call `initiateUnstaking()` from operator accounts until `uncompletedWithdrawalCount == 80`, then call `updateRSETHPrice()` with a gas limit matching the current Ethereum block gas limit (~30M) and assert the transaction reverts with out-of-gas.
