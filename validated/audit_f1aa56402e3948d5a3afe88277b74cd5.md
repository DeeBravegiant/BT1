### Title
Attestation Stored Before Deposit Check in `submit_participant_info` Allows Free Storage Without Payment — (File: `crates/contract/src/lib.rs`)

### Summary

In `submit_participant_info`, the attestation is written to `tee_state.stored_attestations` **before** the deposit sufficiency check. When the deposit is insufficient, the function returns `Err(InsufficientDeposit)`. In NEAR SDK's `#[handle_result]` model, returning `Err` does **not** revert state changes — only `env::panic_str` does. The attestation is therefore committed to contract storage without the caller ever paying for it, breaking the accounting invariant that storage costs are always covered by the caller's deposit.

### Finding Description

`submit_participant_info` executes in this order:

1. **State mutated**: `self.tee_state.add_participant(...)` is called and, on success, writes the attestation into `tee_state.stored_attestations`. [1](#0-0) 

2. **Storage cost computed** (after the mutation): `env::storage_usage().saturating_sub(initial_storage)` measures how many bytes were just written. [2](#0-1) 

3. **Deposit check**: if `attached < cost`, the function returns `Err(InsufficientDeposit)`. [3](#0-2) 

Because `#[handle_result]` serialises the `Err` value and returns normally (no panic), the NEAR runtime treats the receipt as **successful** and commits every state change made before the `return Err(...)`. The attestation written in step 1 survives.

Contrast this with `propose_update`, which correctly checks the deposit **before** mutating state: [4](#0-3) 

And with `require_deposit`, which uses `env::panic_str` (state-reverting) when the deposit is too low: [5](#0-4) 

The `submit_participant_info` path is the only place in the contract that (a) mutates state, (b) then checks a deposit, and (c) uses `return Err(...)` rather than `env::panic_str` on failure.

### Impact Explanation

An unprivileged caller who can produce a valid attestation (any operator running a TDX-capable machine in production, or any account in mock/test mode) can call `submit_participant_info` with 0 attached deposit. `add_participant` succeeds and writes the attestation; the deposit check fires and returns `Err`; the attestation remains in `tee_state.stored_attestations` without the caller having paid the storage

fn require_deposit(minimum_deposit: NearToken, predecessor: &AccountId) {
    let deposit = env::attached_deposit();
    match deposit.checked_sub(minimum_deposit) {
        None => {
            env::panic_str(
                &InvalidParameters::InsufficientDeposit {
                    attached: deposit.as_yoctonear(),
                    required: minimum_deposit.as_yoctonear(),
                }
                .to_string(),
            );
        }
        Some(diff) => {
            if diff > NearToken::from_yoctonear(0) {
                log!("refund excess deposit {diff} to {predecessor}");
                Promise::new(predecessor.clone()).transfer(diff).detach();
            }
        }
    }
}
