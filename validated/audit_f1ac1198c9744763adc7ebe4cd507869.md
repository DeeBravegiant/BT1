Audit Report

## Title
`getRoundData()` Applies Current rsETH/ETH Rate to Historical ETH/USD Round Data, Producing Incorrect rsETH/USD Price - (File: contracts/oracles/RSETHPriceFeed.sol)

## Summary

`RSETHPriceFeed.getRoundData(_roundId)` fetches a historical ETH/USD price from Chainlink but unconditionally multiplies it by the **current** `rsETHPrice()` from `LRTOracle`. Because `LRTOracle.rsETHPrice` is a state variable that only reflects the most recently updated rate, the returned `answer` for any non-latest round is a synthetic price that never existed on-chain. The contract implements `AggregatorV3Interface` and is deployed as a Chainlink-compatible feed, so any downstream consumer calling `getRoundData` for historical validation receives fabricated data.

## Finding Description

In `contracts/oracles/RSETHPriceFeed.sol` at lines 53–61:

```solidity
function getRoundData(uint80 _roundId)
    external
    view
    returns (uint80 roundId, int256 answer, uint256 startedAt, uint256 updatedAt, uint80 answeredInRound)
{
    (roundId, answer, startedAt, updatedAt, answeredInRound) = ETH_TO_USD.getRoundData(_roundId);
    answer = int256(RS_ETH_ORACLE.rsETHPrice()) * answer / 1e18;
}
```

`ETH_TO_USD.getRoundData(_roundId)` returns the ETH/USD price recorded at the time of that historical Chainlink round. `RS_ETH_ORACLE.rsETHPrice()` calls `LRTOracle.rsETHPrice`, which is a storage variable updated only when `updateRSETHPrice()` is called — it always reflects the current rate, not the rate at `_roundId`. The rsETH/ETH rate monotonically increases over time as staking rewards accrue, so for any past round the current rate is higher than the rate that existed at that round, causing the returned `answer` to be inflated. The `updatedAt` and `answeredInRound` fields carry the historical round's timestamps, creating a metadata/price mismatch. No historical mapping of rsETH/ETH rates exists anywhere in the contract or in `LRTOracle`, so there is no way to reconstruct the correct historical rsETH/USD price.

The `latestRoundData()` path (lines 63–70) is unaffected because both the ETH/USD price and the rsETH/ETH rate are contemporaneous.

Exploit path: any unprivileged caller invokes `RSETHPriceFeed.getRoundData(R_historical)` → contract returns `currentRsETHRate × historicalETHUSD` → caller receives a fabricated rsETH/USD price that overstates the true historical value.

## Impact Explanation

**Low — Contract fails to deliver promised returns, but doesn't lose value.**

`RSETHPriceFeed` is explicitly deployed as a Chainlink `AggregatorV3Interface` drop-in (confirmed by the README deployment table listing `RSETHPriceFeed (Morph)` at `0x4B9C66c2C0d3706AabC6d00D2a6ffD2B68A4E383`). Protocols integrating this feed and calling `getRoundData` for staleness checks, TWAP calculations, or historical price assertions receive an incorrect answer. The LRT-rsETH core contracts do not call `getRoundData` internally, so no direct fund loss occurs within the protocol. The impact is that the contract fails to fulfill its stated interface contract for historical round queries.

## Likelihood Explanation

`getRoundData` is a standard `AggregatorV3Interface` method. Any protocol integrating `RSETHPriceFeed` as a Chainlink-compatible oracle — a common pattern for lending markets, derivatives platforms, and aggregators — may call it with a non-latest round ID for staleness validation or historical price checks. No special privileges, timing, or attacker setup are required; the function is public and always returns the wrong answer for any `_roundId` that is not the current latest round.

## Recommendation

Since `LRTOracle` does not store a history of rsETH/ETH rates, `getRoundData` cannot reconstruct the correct historical rsETH/USD price. The correct fix is to have `getRoundData` revert with an explicit `NotSupported()` error rather than silently returning a fabricated answer. If historical rsETH/ETH rates are desired in the future, a mapping keyed by Chainlink round ID or block timestamp should be maintained and populated on each `updateRSETHPrice()` call. The `latestRoundData()` implementation is correct and requires no changes.

## Proof of Concept

1. Deploy `RSETHPriceFeed` pointing to the live ETH/USD Chainlink feed and `LRTOracle`.
2. Record the current latest round ID `R_latest` and current `rsETHPrice` = 1.10e18.
3. Advance time (or use a fork at a later block) so that `rsETHPrice` has grown to 1.10e18 from a prior value of 1.05e18 at round `R_0`.
4. Call `RSETHPriceFeed.getRoundData(R_0)`.
5. Observe: `answer = 1.10e18 * ETH_USD_at_R0 / 1e18` instead of the correct `1.05e18 * ETH_USD_at_R0 / 1e18`.
6. The returned price overstates the true historical rsETH/USD by ~4.8%, while `updatedAt` carries the timestamp of `R_0`, creating a misleading historical data point.
