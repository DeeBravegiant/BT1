Audit Report

## Title
Missing Daily Mint Limit in `RSETHPoolNoWrapper` Enables Unlimited rsETH Extraction Against Stale Cross-Chain Oracle Rate - (File: contracts/pools/RSETHPoolNoWrapper.sol)

## Summary
`RSETHPoolNoWrapper` distributes pre-minted rsETH to depositors using a cross-chain oracle rate sourced from `RSETHRateReceiver`, which stores the last rate pushed from L1 via LayerZero and has no staleness check. Unlike `RSETHPoolV3`, which enforces a `dailyMintLimit` cap via the `limitDailyMint` modifier, `RSETHPoolNoWrapper` imposes no per-day ceiling on rsETH outflow. Because rsETH continuously appreciates and the L2 oracle rate is structurally stale between updates, any depositor can extract rsETH at a discount — bounded only by the pool's total rsETH balance — with no rate-limiting mechanism to contain the damage.

## Finding Description
`RSETHPoolV3.deposit()` is guarded by the `limitDailyMint` modifier at lines 251 and 280, which computes the rsETH amount that would be distributed and reverts if it would exceed `dailyMintLimit` for the current 24-hour window.

`RSETHPoolNoWrapper.deposit(string)` (L231–244) and `deposit(address,uint256,string)` (L250–271) carry no equivalent guard. The rsETH amount given to the depositor is computed as:

```
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate
```

where `rsETHToETHrate` comes from `IOracle(rsETHOracle).getRate()` (L282), which calls `CrossChainRateReceiver.getRate()` (L103–105 of `CrossChainRateReceiver.sol`), returning the last stored `rate` with no staleness check. This `rate` is only updated when `lzReceive()` is called by the LayerZero endpoint (L82–100 of `CrossChainRateReceiver.sol`). The L1 source is `RSETHRateProvider.getLatestRate()` (L27–29 of `RSETHRateProvider.sol`), which reads `ILRTOracle(rsETHPriceOracle).rsETHPrice()` — a stored value in `LRTOracle` that is only refreshed when `updateRSETHPrice()` is called (L87–89 of `LRTOracle.sol`).

Between L1 oracle updates and LayerZero relay, the L2 rate is stale. Because rsETH continuously appreciates (staking rewards accrue), the L2 rate is structurally lower than the true L1 rate during every inter-update window. A depositor sending `A` ETH receives `A / R_stale` rsETH, which exceeds the fair amount `A / R_true`. The excess rsETH is transferred directly from the pool's pre-minted rsETH reserves (`rsETH.safeTransfer(msg.sender, rsETHAmount)` at L241), not freshly minted — meaning the pool's finite rsETH balance can be fully drained in a single call with no cap.

## Impact Explanation
The excess rsETH extracted per deposit represents the yield that has accrued to the pool's pre-minted rsETH reserves since the last oracle update. Because there is no daily cap, a single depositor can drain the entire pool balance in one transaction, stealing all accrued-but-not-yet-reflected yield from every existing rsETH holder. This maps directly to **High: Theft of unclaimed yield**.

## Likelihood Explanation
The cross-chain oracle is structurally stale by design: every period between a LayerZero push from L1 and the next one creates an exploitable window. rsETH appreciates continuously (~4% APY), so even a 24-hour staleness window produces a measurable discount per unit. With no daily cap, an attacker can amplify this by depositing an arbitrarily large amount of ETH in a single call, extracting the full accrued-yield surplus from the pool in one transaction. The attack requires no special permissions, no oracle manipulation, and no front-running — only a deposit call during any normal inter-update window.

## Recommendation
Add a `dailyMintLimit` state variable and a `limitDailyMint` modifier to `RSETHPoolNoWrapper`, mirroring the implementation already present in `RSETHPoolV3` (L50–57 and L96–125 of `RSETHPoolV3.sol`). Apply the modifier to both `deposit(string)` and `deposit(address,uint256,string)` in `RSETHPoolNoWrapper`. Additionally, consider adding a staleness check on the cross-chain oracle rate (e.g., revert if `block.timestamp - lastUpdated` exceeds a configurable threshold in `CrossChainRateReceiver`).

## Proof of Concept
1. Assume the L2 oracle last received `R_stale = 1.040 ETH/rsETH` 20 hours ago.
2. On L1, staking rewards have pushed the true rate to `R_true = 1.041 ETH/rsETH`.
3. Attacker calls `RSETHPoolNoWrapper.deposit{value: 10_000 ETH}("")`.
4. Contract computes `rsETHAmount = 10_000e18 * 1e18 / 1.040e18 = 9_615.38 rsETH`.
5. Fair amount at true rate: `10_000e18 / 1.041e18 = 9_606.15 rsETH`.
6. Attacker receives **9.23 rsETH extra** (~9.61 ETH of value) in a single call, with no daily cap to prevent it.
7. Attacker repeats or scales the deposit to drain the pool's entire rsETH balance.

**Foundry fork test plan:** Deploy `RSETHPoolNoWrapper` with a mock `CrossChainRateReceiver` returning a fixed stale rate. Fund the pool with pre-minted rsETH. Call `deposit{value: largeAmount}("")` from an unprivileged address. Assert that the attacker receives more rsETH than `largeAmount / trueRate` and that the pool's rsETH balance is fully drained, with no revert.
