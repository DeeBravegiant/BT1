Audit Report

## Title
Minimum Deposit Floor and Per-Asset Deposit Ceiling Create Permanent Deadlock Near Capacity - (File: contracts/LRTDepositPool.sol)

## Summary
In `LRTDepositPool._beforeDeposit`, a `minAmountToDeposit` floor check and a `depositLimitByAsset` ceiling check interact such that when remaining LST deposit capacity falls below `minAmountToDeposit`, no deposit amount can satisfy both guards simultaneously. The remaining capacity becomes permanently inaccessible, and the protocol can never reach its configured deposit limit for that asset.

## Finding Description
`_beforeDeposit` enforces two sequential guards at lines 657–663:

```solidity
if (depositAmount == 0 || depositAmount < minAmountToDeposit) {
    revert InvalidAmountToDeposit();
}
if (_checkIfDepositAmountExceedesCurrentLimit(asset, depositAmount)) {
    revert MaximumDepositLimitReached();
}
``` [1](#0-0) 

For LST assets, `_checkIfDepositAmountExceedesCurrentLimit` returns `true` when `totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)`: [2](#0-1) 

`minAmountToDeposit` is a protocol-wide state variable settable by the LRT admin: [3](#0-2) [4](#0-3) 

`depositLimitByAsset` is a per-asset ceiling stored in `LRTConfig`: [5](#0-4) 

**Deadlock:** Let `R = depositLimitByAsset(asset) - totalAssetDeposits`. When `0 < R < minAmountToDeposit`:
- Any `d < minAmountToDeposit` → reverts `InvalidAmountToDeposit` (floor check).
- Any `d >= minAmountToDeposit` → `totalAssetDeposits + d > depositLimitByAsset` → reverts `MaximumDepositLimitReached` (ceiling check).

No value of `d` satisfies both guards. The remaining capacity `R` is permanently inaccessible until an admin intervenes.

## Impact Explanation
The protocol can never reach its configured `depositLimitByAsset` for any LST asset once remaining capacity drops below `minAmountToDeposit`. No user funds are lost, but the protocol permanently under-collects relative to its configured ceiling. This matches the allowed impact: **Low — Contract fails to deliver promised returns, but doesn't lose value.**

## Likelihood Explanation
No attacker action is required. As organic deposits accumulate and `totalAssetDeposits` approaches `depositLimitByAsset`, the remaining capacity naturally shrinks. The moment it falls below `minAmountToDeposit` (a non-trivial value such as 0.01 ETH), the deadlock is permanent. The entry path is the public `depositAsset` function callable by any user. [6](#0-5) 

## Recommendation
In `_beforeDeposit`, relax the minimum deposit check when the deposit amount exactly fills the remaining capacity:

```solidity
uint256 remainingCapacity = lrtConfig.depositLimitByAsset(asset) - getTotalAssetDeposits(asset);
if (depositAmount == 0 || (depositAmount < minAmountToDeposit && depositAmount != remainingCapacity)) {
    revert InvalidAmountToDeposit();
}
```

This allows a depositor to fill the last slot even if it is smaller than `minAmountToDeposit`.

## Proof of Concept
1. Admin sets `depositLimitByAsset(stETH) = 100_000e18` and `minAmountToDeposit = 0.01e18`.
2. Organic depositors bring `totalAssetDeposits(stETH)` to `99_999.995e18`. Remaining capacity `R = 0.005e18`.
3. Alice calls `depositAsset(stETH, 0.005e18, ...)`:
   - `0.005e18 < 0.01e18` → reverts `InvalidAmountToDeposit`.
4. Alice calls `depositAsset(stETH, 0.01e18, ...)`:
   - `99_999.995e18 + 0.01e18 = 100_000.005e18 > 100_000e18` → reverts `MaximumDepositLimitReached`.
5. No deposit amount can succeed. The remaining `0.005e18` capacity is permanently locked and the protocol never reaches its 100,000 stETH target.

**Foundry test plan:** Deploy `LRTDepositPool` with a mock `LRTConfig` setting `depositLimitByAsset = 100_000e18` and `minAmountToDeposit = 0.01e18`. Simulate deposits totaling `99_999.995e18`. Assert that both `depositAsset(stETH, 0.005e18, ...)` and `depositAsset(stETH, 0.01e18, ...)` revert, confirming the deadlock.
