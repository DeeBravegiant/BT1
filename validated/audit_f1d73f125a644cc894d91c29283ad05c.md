Audit Report

## Title
`burnFrom` Does Not Decrement `currentPeriodMintedAmount`, Causing Daily Mint Limit Accounting Mismatch — (File: contracts/RSETH.sol)

## Summary
`RSETH.mint` increments `currentPeriodMintedAmount` via the `checkDailyMintLimit` modifier on every mint, but `RSETH.burnFrom` only calls `_burn` and never decrements `currentPeriodMintedAmount`. This causes the tracked daily-minted amount to diverge from actual circulating supply, which can block legitimate new deposits for the remainder of a 24-hour window even when real circulating supply is well below `maxMintAmountPerDay`.

## Finding Description
The `checkDailyMintLimit` modifier in `contracts/RSETH.sol` (L42–56) resets and increments `currentPeriodMintedAmount` on every `mint` call. The `burnFrom` function (L245–248) only calls `_enforceNotBlocked` and `_burn`, with no corresponding decrement to `currentPeriodMintedAmount`. As a result, burns do not free up daily mint capacity.

`burnFrom` is invoked by `LRTWithdrawalManager.instantWithdrawal` (L229), which is callable by any unprivileged rsETH holder with no role restriction beyond `whenNotPaused`. A sequence of:
1. Mint N rsETH → `currentPeriodMintedAmount += N`
2. Burn M rsETH via `instantWithdrawal` → `currentPeriodMintedAmount` unchanged
3. Attempt to mint K rsETH where `N - M + K ≤ maxMintAmountPerDay` but `N + K > maxMintAmountPerDay` → `DailyMintLimitExceeded` revert

…blocks all new deposits for the rest of the day despite the actual circulating supply being well within the configured limit. The period resets only when `block.timestamp >= periodStartTime + 1 days`.

## Impact Explanation
**Low — Contract fails to deliver promised returns, but doesn't lose value.**

No user funds are lost or frozen. However, the protocol's core deposit service becomes unavailable for the remainder of the 24-hour window. Depositors receive a `DailyMintLimitExceeded` revert even though the actual circulating supply is below `maxMintAmountPerDay`. This matches the allowed impact: "Contract fails to deliver promised returns, but doesn't lose value."

## Likelihood Explanation
Medium. `instantWithdrawal` is permissionless — any rsETH holder can call it. During periods of elevated withdrawal activity (e.g., market downturns), multiple users burning rsETH within a single day will silently exhaust the daily mint capacity. The effect is amplified when `maxMintAmountPerDay` is set conservatively close to expected daily volume. The condition is naturally self-triggering and requires no coordination or privileged access.

## Recommendation
Decrement `currentPeriodMintedAmount` inside `burnFrom`, applying the same period-reset logic used in `checkDailyMintLimit`, capped at zero to prevent underflow:

```solidity
function burnFrom(address account, uint256 amount) external onlyRole(LRTConstants.BURNER_ROLE) whenNotPaused {
    _enforceNotBlocked(account);

    if (block.timestamp >= periodStartTime + 1 days) {
        currentPeriodMintedAmount = 0;
        periodStartTime = getCurrentPeriodStartTime();
    }

    if (currentPeriodMintedAmount >= amount) {
        currentPeriodMintedAmount -= amount;
    } else {
        currentPeriodMintedAmount = 0;
    }

    _burn(account, amount);
}
```

## Proof of Concept
```
// Setup
maxMintAmountPerDay = 1000e18

Step 1: LRTDepositPool mints 800e18 rsETH
        → RSETH.mint(depositor, 800e18)
        → checkDailyMintLimit: currentPeriodMintedAmount = 800e18

Step 2: User calls LRTWithdrawalManager.instantWithdrawal(ETH, 500e18, "")
        → RSETH.burnFrom(user, 500e18)
        → _burn reduces totalSupply by 500e18  ✓
        → currentPeriodMintedAmount remains 800e18  ✗ (BUG)

Step 3: New depositor calls LRTDepositPool.depositETH{value: 300e18}(...)
        → RSETH.mint(depositor, 300e18)
        → checkDailyMintLimit: 800e18 + 300e18 = 1100e18 > 1000e18
        → REVERT: DailyMintLimitExceeded

Actual circulating supply: 300e18 (well below 1000e18 limit)
Tracked currentPeriodMintedAmount: 800e18 (stale, inflated)
Deposits blocked until periodStartTime + 1 days
```

Foundry invariant test plan: assert that after any sequence of `mint` and `burnFrom` calls, `currentPeriodMintedAmount` never exceeds `totalSupply()` within the same period window.
