Audit Report

## Title
Uninitialized `unlockedWithdrawalsCount` Causes Underflow Revert in `_processWithdrawalCompletion` During Upgrade Window — (File: `contracts/LRTWithdrawalManager.sol`)

## Summary
After the proxy is upgraded to the new `LRTWithdrawalManager` implementation but before `initialize2` is called, `unlockedWithdrawalsCount[asset]` is `0` for all assets. Any user with a pre-existing unlocked withdrawal who calls `completeWithdrawal` during this window will trigger an arithmetic underflow at `unlockedWithdrawalsCount[asset]--`, causing a revert and temporarily freezing their funds. The freeze persists for the entire duration between the upgrade and the `initialize2` call, which is operationally unbounded.

## Finding Description
`initialize2` is gated by the `UNLOCKED_WITHDRAWAL_INITIALIZER` role — a distinct constant from the upgrade admin role — meaning it is not required to execute atomically with the proxy upgrade. Between the upgrade and the `initialize2` call, `unlockedWithdrawalsCount[asset]` holds its Solidity default value of `0`.

`_processWithdrawalCompletion` (called by both `completeWithdrawal` and `completeWithdrawalForUser`) performs all eligibility checks first — verifying the user has a request, that the request nonce is below `nextLockedNonce[asset]` (i.e., already unlocked), and that the withdrawal delay has elapsed — and then unconditionally decrements the counter at L717:

```solidity
unlockedWithdrawalsCount[asset]--;
```

In Solidity 0.8.27, unsigned integer underflow reverts. Because the counter is `0` and no guard exists, every `completeWithdrawal` call for a pre-existing unlocked withdrawal reverts during the window. The user's rsETH has already been transferred to the contract at `initiateWithdrawal` time and cannot be recovered until `initialize2` executes.

The `UNLOCKED_WITHDRAWAL_INITIALIZER` role is confirmed as a separate constant in `LRTConstants.sol` (L23), making the two-transaction upgrade pattern architecturally expected.

Note on the "stale seed / permanent block" sub-claim: this sub-claim is self-contradicted. During the upgrade window, Problem 1 causes every `completeWithdrawal` to revert, so no completions can occur between the admin's read and `initialize2` mining. The seed therefore cannot become stale through the described path, and the permanent `sweepRemainingAssets` block is not achievable via this route.

## Impact Explanation
**Medium — Temporary freezing of funds.** Users with pre-existing unlocked withdrawals cannot claim their assets for the entire duration of the upgrade window. Their rsETH is held by the contract and is inaccessible until `initialize2` is called. The duration is operationally unbounded and depends entirely on the privileged operator's process.

## Likelihood Explanation
The `UNLOCKED_WITHDRAWAL_INITIALIZER` role is architecturally separate from the upgrade admin, making a multi-transaction upgrade the expected operational pattern. The protocol has active users with pending unlocked withdrawals — the very reason `initialize2` exists. Any such user who attempts to claim during the window (even a single block gap) will be frozen. The trigger requires no special capability: any unprivileged user calling `completeWithdrawal` on an eligible request is sufficient.

## Recommendation
**Short term:** Pause `LRTWithdrawalManager` via `PAUSER_ROLE` before deploying the new implementation, and unpause only after `initialize2` has been successfully called. This eliminates the window entirely.

**Long term:** Encode `initialize2` as the `data` argument of `upgradeToAndCall` so the seeding executes atomically with the implementation swap. Alternatively, add a guard in `_processWithdrawalCompletion` that skips the decrement when `unlockedWithdrawalsCount[asset] == 0` (or uses saturating subtraction) to prevent the underflow revert while the counter is being bootstrapped.

## Proof of Concept
```
1. Admin deploys new LRTWithdrawalManager implementation (adds unlockedWithdrawalsCount).
2. Proxy is upgraded to the new implementation via upgradeTo().
   → unlockedWithdrawalsCount[ETH] == 0 (Solidity default).
   → N unlocked withdrawal requests exist (nonce < nextLockedNonce[ETH]).
3. Alice (unprivileged user) calls completeWithdrawal(ETH_TOKEN, "ref").
4. _processWithdrawalCompletion executes:
   - userAssociatedNonces check passes (Alice has a request).
   - usersFirstWithdrawalRequestNonce < nextLockedNonce check passes (request is unlocked).
   - withdrawalDelayBlocks check passes (delay has elapsed).
   - unlockedWithdrawalsCount[ETH]-- → 0 - 1 → REVERT (Solidity 0.8.27 underflow).
5. Alice's withdrawal is frozen. Her rsETH remains locked in the contract.
6. All other users with unlocked withdrawals face the same revert.
7. Freeze resolves only when admin calls initialize2(), duration unbounded.

Foundry fork test plan:
- Fork mainnet at a block after the upgrade.
- Confirm unlockedWithdrawalsCount[ETH] == 0.
- Impersonate a user with a known unlocked withdrawal request.
- Call completeWithdrawal and assert the transaction reverts with arithmetic underflow (Panic(0x11)).
- Call initialize2 with the correct seed.
- Repeat completeWithdrawal and assert it succeeds.
```
