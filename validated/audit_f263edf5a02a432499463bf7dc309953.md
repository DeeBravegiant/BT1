### Title
`respond_verify_foreign_tx` Does Not Bind `payload_hash` to the Pending Request, Enabling Cross-Request Signature Replay by a Single Malicious Attested Participant - (File: `crates/contract/src/lib.rs`)

---

### Summary

The `respond_verify_foreign_tx` function in the MPC contract verifies that the submitted signature is valid over `response.payload_hash`, but never checks that `response.payload_hash` is the correct hash for the submitted `request`. A single malicious attested participant (below the signing threshold) can observe any legitimately produced MPC signature for one foreign-tx verification request and replay it as the response to a completely different pending request. The contract accepts the forged response and delivers it to all callers waiting on the second request, breaking the core invariant that a `VerifyForeignTransactionResponse` attests to the actual content of the requested foreign-chain transaction.

---

### Finding Description

In `respond_verify_foreign_tx` the contract performs the following signature check:

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;

// Check the signature is correct against the root public key
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,
)
.is_ok()
``` [1](#0-0) 

The `payload_hash` is taken entirely from the caller-supplied `response`, not recomputed from the `request`. The contract only checks that the signature is valid over whatever hash the caller provides. It never verifies that `response.payload_hash` equals `SHA-256(borsh(ForeignTxSignPayload { request, observed_values }))` for the actual pending `request`.

Contrast this with the regular `respond` function, where the payload hash is taken directly from the stored request — the caller cannot substitute a different hash:

```rust
let payload_hash = request.payload.as_ecdsa().expect("Payload is not ECDSA");
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    payload_hash,
    &expected_public_key,
)
.is_ok()
``` [2](#0-1) 

On the node side, the MPC network signs `payload_hash = SHA-256(borsh(ForeignTxSignPayload { request, observed_values }))`, which binds the signature to the specific request content:

```rust
fn build_signature_request(
    request: &VerifyForeignTxRequest,
    foreign_tx_payload: &dtos::ForeignTxSignPayload,
) -> anyhow::Result<SignatureRequest> {
    let payload_hash: [u8; ECDSA_PAYLOAD_SIZE_BYTES] =
        foreign_tx_payload.compute_msg_hash()?.into();
    ...
    tweak: Tweak::new([0u8; 32]),
``` [3](#0-2) 

The node-side binding is never enforced on-chain. The contract's `respond_verify_foreign_tx` and the MPC signing process are two independent paths that do not coordinate the invariant that `payload_hash` must correspond to the `request`. This is the structural analog to the external report's double-distribution bug: two paths that should share a state invariant but do not.

After the signature check passes, the contract resolves all queued yields for the submitted `request` with the (potentially forged) `response`:

```rust
pending_requests::resolve_yields_for(
    &mut self.pending_verify_foreign_tx_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
``` [4](#0-3)

### Impact Explanation

A single malicious attested participant can:

1. Observe any on-chain `respond_verify_foreign_tx` call for **Request A** (the call is public; the signature `sig_A` and `payload_hash_A` are visible in the transaction).
2. While **Request B** (a different `ForeignChainRpcRequest`) is pending in `pending_verify_foreign_tx_requests`, submit:
   ```
   respond_verify_foreign_tx(request = B, response = { payload_hash = payload_hash_A, signature = sig_A })
   ```
3. The contract checks: `sig_A` is valid over `payload_hash_A` ✓; Request B exists in the pending map ✓. Both checks pass.
4. The contract resolves all yields queued under Request B, delivering `{ payload_hash_A, sig_A }` to every caller waiting on Request B.

Every caller of Request B now holds a `VerifyForeignTransactionResponse` bearing a valid MPC root-key signature, but the `payload_hash` attests to the content of Request A's transaction, not Request B's. Any downstream bridge or NEAR contract that trusts the MPC signature without independently recomputing the expected hash from the transaction data it requested will accept this forged attestation and may execute an invalid cross-chain action (e.g., releasing funds for a transaction that was never actually verified).

This matches the allowed High impact: **forged foreign-chain verification that causes invalid bridge execution**.

---

### Likelihood Explanation

- The attacker must be a single attested MPC participant — below the signing threshold. No collusion is required.
- The attack requires only observing a legitimate on-chain `respond_verify_foreign_tx` transaction, which is public.
- The attacker needs a second pending request of a different type to be in flight simultaneously, which is a normal operational condition for any active bridge.
- The attack is fully reachable via a standard NEAR contract call with no special privileges beyond being an attested participant.

---

### Recommendation

Recompute the expected `payload_hash` from the `request` inside `respond_verify_foreign_tx` and verify that `response.payload_hash` matches, before checking the signature. The contract already has all the information needed: the `request` field contains the `ForeignChainRpcRequest`. However, because the contract does not know the `observed_values` (those are produced by the nodes), the simplest fix is to change the signing protocol so that the signature is computed over `(request_key || payload_hash)` — i.e., include the request identity in the signed message — so that a signature produced for Request A cannot be valid for Request B. Alternatively, the contract can require the responder to also submit the `observed_values`, recompute the hash on-chain, and verify it matches `response.payload_hash` before accepting the response.

### Proof of Concept

**Setup:**
- Two pending requests exist: Request A (`Bitcoin tx_id = [1u8;32]`) and Request B (`Bitcoin tx_id = [2u8;32]`), both in `pending_verify_foreign_tx_requests`.
- The MPC network legitimately processes Request A and a node submits `respond_verify_foreign_tx(request=A, response={payload_hash_A, sig_A})`. This resolves Request A.

**Attack:**
```rust
// Malicious attested participant observes payload_hash_A and sig_A from the on-chain tx for Request A.
// Request B is still pending.
contract.respond_verify_foreign_tx(
    request_B,  // different ForeignChainRpcRequest
    VerifyForeignTransactionResponse {
        payload_hash: payload_hash_A,  // hash from Request A
        signature: sig_A,              // signature from Request A
    }
)
// Returns Ok(()) — contract accepts the forged response.
// All callers waiting on Request B receive { payload_hash_A, sig_A }.
```

The contract's signature check at line 729 passes because `sig_A` is a valid ECDSA signature over `payload_hash_A` under the root key. The pending-map lookup at line 749 passes because Request B is a different key and is still present. The contract delivers the forged response to all callers of Request B. [5](#0-4) [6](#0-5)

pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
