### Title
Unbound `payload_hash` in `respond_verify_foreign_tx` Enables Cross-Request Signature Replay by a Single Byzantine Participant - (File: crates/contract/src/lib.rs)

### Summary

`respond_verify_foreign_tx` verifies that `response.signature` is a valid ECDSA signature over `response.payload_hash` under the MPC root key, but never verifies that `response.payload_hash` is the canonical hash of `ForeignTxSignPayload` derived from `request.request`. A single Byzantine attested MPC participant (below threshold) can replay a valid MPC signature produced for foreign-chain request A to forge a verification response for any pending request B, causing the contract to attest to a foreign-chain verification that never occurred for the requested transaction.

### Finding Description

In `respond_verify_foreign_tx`, the signature check is:

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;
// Check the signature is correct against the root public key
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,
)
.is_ok()
``` [1](#0-0) 

The contract verifies only that `response.signature` is a valid ECDSA signature over `response.payload_hash` under the domain's root public key. It does **not** verify that `response.payload_hash` equals `ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 { request: request.request, values: <actual_values> }).compute_msg_hash()`.

This is structurally different from `respond` (for regular signatures), where the payload hash is taken directly from `request.payload` — bound to the request at submission time:

```rust
let payload_hash = request.payload.as_ecdsa().expect("Payload is not ECDSA");
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    payload_hash,
    &expected_public_key,
)
``` [2](#0-1) 

The asymmetry exists because for foreign-tx verification, the `values` (extracted from the foreign chain by MPC nodes) are not known at request time, so the contract cannot recompute the expected hash. The `VerifyForeignTransactionRequest` stored in `pending_verify_foreign_tx_requests` contains only `{ request, domain_id, payload_version }` — no `payload_hash` field: [3](#0-2) 

The `payload_hash` is entirely caller-supplied in the response, and the contract uses it only to verify the signature — never to bind it back to the original request content.

The `ForeignTxSignPayload` canonical hash is computed as `SHA-256(borsh(ForeignTxSignPayload))`, which commits to both the `ForeignChainRpcRequest` (including `tx_id`) and the `values` extracted from the chain: [4](#0-3) 

Because the contract never checks that `response.payload_hash` encodes the correct `request.request`, any previously-observed valid `(payload_hash, signature)` pair — produced by the honest MPC network for any request on the same domain — can be submitted as the response to a different pending request.

### Impact Explanation

A single Byzantine attested MPC participant (below threshold) can forge a foreign-chain verification attestation for any pending `verify_foreign_transaction` request:

1. The attacker submits `verify_foreign_transaction(Bitcoin tx_id=A)` and waits for the honest MPC network to produce `(payload_hash_A, signature_A)`. This response is public on-chain once the leader calls `respond_verify_foreign_tx`.
2. A victim submits `verify_foreign_transaction(Bitcoin tx_id=B)`, creating a pending entry in `pending_verify_foreign_tx_requests`.
3. The attacker (as the elected leader for request B, or by front-running the honest leader) calls `respond_verify_foreign_tx(request=Bitcoin(tx_id=B), response={payload_hash=payload_hash_A, signature=signature_A})`.
4. The contract accepts because:
   - Caller is an attested participant ✓
   - `verify_ecdsa_signature(signature_A, payload_hash_A, root_key)` passes ✓
   - `pending_verify_foreign_tx_requests.get(&request_B)` exists ✓
5. All yields queued under request B are resolved with `{payload_hash=payload_hash_A, signature=signature_A}`. [5](#0-4) 

The victim's application receives a `VerifyForeignTransactionResponse` where `payload_hash` corresponds to transaction A's data, not transaction B's. Any downstream bridge contract or application that does not independently recompute and verify `payload_hash` against the expected hash for tx_id=B will accept a forged attestation. This enables invalid bridge execution: funds could be released based on a verification that was never performed for the requested transaction.

### Likelihood Explanation

The attacker must be a single attested MPC participant — a Byzantine node strictly below the signing threshold. No threshold collusion is required. The attacker does not need to forge a new MPC signature; they only need to replay one already produced by the honest network. The attack is most straightforward when the Byzantine node is elected leader for request B (probability 1/n), in which case there is no race condition. Even without being the leader, the attacker can attempt to front-run the honest leader's `respond_verify_foreign_tx` call. The required `(payload_hash, signature)` pair is freely obtainable by submitting any valid `verify_foreign_transaction` request.

### Recommendation

1. **Bind `payload_hash` to the request at submission time**: Have MPC nodes compute and commit to the expected `payload_hash` before the threshold signing round, and store it in the pending request map. `respond_verify_foreign_tx` can then verify `response.payload_hash == stored_payload_hash`.
2. **Include extracted values in the response and verify on-chain**: Require the responder to submit the full `ForeignTxSignPayload` (including `values`), recompute `compute_msg_hash()` on-chain, and verify `response.payload_hash == recomputed_hash`. This is gas-intensive but fully trustless.
3. **Require multi-participant response submission**: Require at least a threshold of participants to independently submit the same `(request, payload_hash)` pair before resolving yields, eliminating the single-leader forgery path.

### Proof of Concept

```
// Step 1: Attacker submits their own request and obtains a valid (payload_hash_A, signature_A)
attacker.call(contract, "verify_foreign_transaction", {
    request: { Bitcoin: { tx_id: A, confirmations: 6, extractors: [BlockHash] } },
    domain_id: foreign_tx_domain_id,
    payload_version: V1,
})
// MPC network produces (payload_hash_A, signature_A) — visible on-chain

// Step 2: Victim submits a different request
victim.call(contract, "verify_foreign_transaction", {
    request: { Bitcoin: { tx_id: B, confirmations: 6, extractors: [BlockHash] } },
    domain_id: foreign_tx_domain_id,
    payload_version: V1,
})
// pending_verify_foreign_tx_requests now contains request_B

// Step 3: Attacker (as elected leader or front-runner) calls respond with replayed signature
attacker.call(contract, "respond_verify_foreign_tx", {
    request: { request: Bitcoin(tx_id=B), domain_id: ..., payload_version: V1 },
    response: { payload_hash: payload_hash_A, signature: signature_A },
})
// Contract accepts: signature_A is valid over payload_hash_A under root key ✓
// request_B exists in pending map ✓
// Victim receives VerifyForeignTransactionResponse{payload_hash=payload_hash_A, signature=signature_A}
// payload_hash_A encodes tx_id=A's data — NOT tx_id=B's data
// Any bridge contract that skips payload_hash verification accepts the forged attestation
```

The root cause is directly analogous to the reference report: user-controlled data (`response.payload_hash`) is accepted by the contract without a binding check against the original request content, enabling state corruption (forged on-chain attestation) by a single unprivileged-but-attested participant.

    pub fn compute_msg_hash(&self) -> std::io::Result<Hash256> {
