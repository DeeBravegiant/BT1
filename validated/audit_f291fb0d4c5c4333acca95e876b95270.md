Audit Report

## Title
`maxNumberOfWithdrawalsPerUser` Uninitialized in `initialize()` Blocks All Withdrawal Initiations — (File: contracts/KERNEL/KernelDepositPool.sol)

## Summary
`KernelDepositPool.initialize()` never assigns `maxNumberOfWithdrawalsPerUser`, leaving it at its default value of `0`. The guard in `initiateWithdrawal()` evaluates `userWithdrawalIds[msg.sender].length >= maxNumberOfWithdrawalsPerUser` as `0 >= 0 = true` for every caller, causing every invocation to revert with `WithdrawalLimitReached`. All staked KERNEL tokens are inaccessible until an admin separately calls `setMaxNumberOfWithdrawalsPerUser()`.

## Finding Description
`maxNumberOfWithdrawalsPerUser` is declared as a plain storage `uint256` at line 108, defaulting to `0` in Solidity. [1](#0-0) 

The `initialize()` function sets `kernelToken`, `rewardsToken`, and the admin role, but never assigns this variable. [2](#0-1) 

The constant `MAX_WITHDRAWALS_PER_USER = 100` exists but is never used to seed the state variable during initialization. [3](#0-2) 

In `initiateWithdrawal()`, the guard `userWithdrawalIds[msg.sender].length >= maxNumberOfWithdrawalsPerUser` evaluates to `0 >= 0 = true` for any fresh caller, triggering an immediate revert. [4](#0-3) 

A setter `setMaxNumberOfWithdrawalsPerUser()` exists and can remediate the issue, but it is an admin-only function that is not called as part of the deployment flow. [5](#0-4) 

## Impact Explanation
**Medium — Temporary freezing of funds.** Users who have staked KERNEL tokens via `stake()` or `stakeFor()` cannot call `initiateWithdrawal()` to begin the unstaking process. Their balances are recorded in `balanceOf` and `totalKernelStaked`, but the only exit path is permanently gated behind the broken limit check until an admin intervenes. Funds are not permanently lost (the admin setter exists), but they are inaccessible for an indefinite period post-deployment, matching the "Temporary freezing of funds" impact class.

## Likelihood Explanation
**High.** The condition fires for every user on every call from the moment the contract is deployed. No special attacker capability, front-running, or external dependency is required. Any ordinary staker calling `initiateWithdrawal()` will hit the revert. The only mitigation is an out-of-band admin action that is not part of the deployment script.

## Recommendation
Initialize `maxNumberOfWithdrawalsPerUser` to the intended cap inside `initialize()`:

```solidity
maxNumberOfWithdrawalsPerUser = MAX_WITHDRAWALS_PER_USER; // 100
```

If the value must remain configurable post-deployment, ensure `setMaxNumberOfWithdrawalsPerUser()` is called atomically in the deployment script immediately after `initialize()`, and add a non-zero invariant check in `initialize()` to prevent future regressions.

## Proof of Concept
1. Deploy `KernelDepositPool` behind a proxy and call `initialize(admin, kernelToken, rewardToken)`. `maxNumberOfWithdrawalsPerUser` is now `0`.
2. As any user, approve and call `stake(1e18)`. Succeeds; `balanceOf[user] = 1e18`.
3. Call `initiateWithdrawal(1e18)`.
4. Execution reaches line 323: `userWithdrawalIds[msg.sender].length` is `0`, `maxNumberOfWithdrawalsPerUser` is `0`; condition `0 >= 0` is `true`.
5. Transaction reverts with `WithdrawalLimitReached`.
6. User's KERNEL tokens remain locked. Repeat for any user — all are blocked identically.

**Foundry test sketch:**
```solidity
function test_initiateWithdrawal_blockedByUninitializedLimit() public {
    // deploy + initialize
    KernelDepositPool pool = new KernelDepositPool();
    pool.initialize(admin, address(kernelToken), address(rewardToken));

    // stake
    vm.startPrank(user);
    kernelToken.approve(address(pool), 1e18);
    pool.stake(1e18);

    // initiateWithdrawal must revert
    vm.expectRevert(KernelDepositPool.WithdrawalLimitReached.selector);
    pool.initiateWithdrawal(1e18);
    vm.stopPrank();
}
```

        external
