### Title
Staking Credentials Delegated to Expired DReps Can Still Withdraw Rewards — (`File: eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Ledger.hs`)

---

### Summary

The `validateWithdrawalsDelegated` function in the Conway `LEDGER` rule checks only whether a staking credential has *any* DRep delegation recorded, but never checks whether the delegated DRep has expired. As a result, a staking credential whose delegated DRep has since expired can still submit a valid withdrawal transaction and drain its reward account, bypassing the governance-participation requirement that is the design intent of the Conway era.

---

### Finding Description

In Conway era, reward withdrawals from key-hash staking credentials require that the credential be delegated to an active DRep (`ConwayWdrlNotDelegatedToDRep` predicate failure). The check is implemented in `validateWithdrawalsDelegated`:

```haskell
isNotDRepDelegated keyHash = isNothing $ do
  accountState <- lookupAccountState (KeyHashObj keyHash) accounts
  accountState ^. dRepDelegationAccountStateL
``` [1](#0-0) 

This predicate returns `True` (i.e., "not delegated") only when the `dRepDelegationAccountStateL` field is `Nothing`. It does **not** consult `VState.vsDReps` to verify that the referenced DRep's `drepExpiry` epoch has not been passed. A DRep that has expired remains in `vsDReps` with a stale `drepExpiry` value; the delegation field in the account state still points to it; so the check passes.

The DRep expiry is tracked as:

```haskell
data DRepState = DRepState
  { drepExpiry :: !EpochNo
  ...
  }
``` [2](#0-1) 

Expiry is evaluated as `binOpEpochNo (+) numDormantEpochs drepExpiry < currentEpoch`, but this computation is never performed inside `validateWithdrawalsDelegated`. [3](#0-2) 

The formal ledger specification explicitly disagrees with this behavior. The conformance test "Withdraw from a key delegated to a DRep that expired after delegation" is disabled with a direct reference to the open specification issue:

```haskell
-- https://github.com/IntersectMBO/formal-ledger-specifications/issues/635
-- TODO: Re-enable after issue is resolved, by removing this override
disableInConformanceIt "Withdraw from a key delegated to a DRep that expired after delegation" $ do
    ...
    isDRepExpired drep `shouldReturn` True
    submitTx_ $   -- succeeds in the Haskell implementation
      mkBasicTx $
        mkBasicTxBody
          & withdrawalsTxBodyL .~ Withdrawals [(ra, balance)]
``` [4](#0-3) 

The `disableInConformanceIt` wrapper suppresses conformance comparison, meaning the Haskell implementation and the Agda formal spec produce different ledger states for the same transaction. [5](#0-4) 

---

### Impact Explanation

The Conway governance design requires staking credentials to be delegated to an *active* DRep as a precondition for reward withdrawal. This is the mechanism that incentivises governance participation: holders who do not maintain an active DRep delegation cannot access accumulated rewards. The missing expiry check allows any holder whose DRep has expired to withdraw rewards without re-delegating, modifying reward withdrawals outside the design parameters of the protocol. This matches the **Medium** allowed impact: *attacker-controlled transactions modify rewards outside design parameters*.

---

### Likelihood Explanation

DRep expiry is a normal, expected event governed by the `ppDRepActivity` protocol parameter (default: 20 epochs). Any staking credential delegated to a DRep that subsequently becomes inactive and expires is automatically in the vulnerable state. No special privilege, key compromise, or coordination is required. The attacker-controlled entry path is a standard withdrawal transaction submitted by the credential owner.

---

### Recommendation

`validateWithdrawalsDelegated` should additionally look up the DRep credential in `VState.vsDReps` and verify that the DRep's actual expiry (accounting for `vsNumDormantEpochs`) has not been reached before the current epoch. Concretely, the `isNotDRepDelegated` predicate should be extended to also return `True` when the referenced DRep is expired, mirroring the logic already present in `isDRepExpired` / `vsActualDRepExpiry`. [6](#0-5) 

---

### Proof of Concept

The existing (disabled) conformance test at lines 177–198 of `LedgerSpec.hs` is a direct proof of concept:

1. Set `ppDRepActivityL = EpochInterval 1` and `ppGovActionLifetimeL = EpochInterval 4`.
2. Register a staking credential and accumulate a reward balance.
3. Set up a DRep and delegate the staking credential to it.
4. Submit a governance action and advance 4 epochs so the DRep expires (`isDRepExpired drep` returns `True`).
5. Submit a withdrawal transaction for the full balance.
6. The transaction is accepted by the Haskell ledger implementation (`submitTx_` succeeds), even though the formal specification requires it to be rejected. [7](#0-6)
