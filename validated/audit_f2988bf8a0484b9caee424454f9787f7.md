### Title
Sub-transaction ExUnits not aggregated in `validateExUnitsTooBigUTxO`, allowing unbounded batch Plutus computation — (`eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Utxo.hs`)

### Summary
In the Dijkstra era, `validateExUnitsTooBigUTxO` checks only the top-level transaction's ExUnits via `totExUnits tx`, which reads only `tx ^. witsTxL . rdmrsTxWitsL`. Sub-transactions carry their own witnesses and redeemers, which are never aggregated into this check. Neither `SUBUTXO` nor `SUBUTXOW` impose any per-sub-tx or aggregate ExUnits limit. An attacker can submit a top-level transaction with zero redeemers and N sub-transactions each declaring ExUnits just below `maxTxExUnits`, causing nodes to execute N × `maxTxExUnits` of Plutus computation while the top-level check passes trivially (0 ≤ `maxTxExUnits`).

### Finding Description
`totExUnits` is defined as:
```haskell
totExUnits tx = foldMap snd $ tx ^. witsTxL . rdmrsTxWitsL . unRedeemersL
```
It accesses only the top-level transaction's witness redeemers. Sub-transactions are embedded in the tx body under `subTransactionsTxBodyL` and each has its own `witsTxL`. The Dijkstra `UTXO` rule calls `Alonzo.validateExUnitsTooBigUTxO pp tx` at line 415, which calls `totExUnits tx` — returning 0 for a top-level tx with no redeemers. The `SUBUTXO` and `SUBUTXOW` rules contain no ExUnits limit check at all. The block-level `validateExUnits` in `dijkstraBbodyTransition` also uses `foldMap totExUnits txs`, missing sub-tx ExUnits entirely. The minimum fee calculation (`alonzoMinFeeTx`) similarly uses `totExUnits tx`, so sub-tx script execution costs are not reflected in the required fee.

### Impact Explanation
A single accepted batch transaction can force every validating node to execute up to N × `maxTxExUnits` of Plutus computation, where N is the number of sub-transactions (bounded only by `maxTxSize`). The `maxTxExUnits` protocol parameter — the primary per-transaction resource cap for Plutus execution — is rendered ineffective for batch transactions. Additionally, the attacker pays script fees only for top-level ExUnits (zero), not for sub-tx script execution, violating the fee-covers-computation invariant. This matches the Medium impact scope: attacker-controlled transactions exceed intended validation limits.

### Likelihood Explanation
The path is direct and requires no privilege: any user can construct a valid top-level transaction with zero redeemers and multiple sub-transactions each carrying redeemers with ExUnits just below `maxTxExUnits`. The transaction passes all existing checks. The only practical constraint is `maxTxSize`, which limits how many sub-transactions can be packed.

### Recommendation
Aggregate ExUnits across the top-level transaction and all sub-transactions before applying the `maxTxExUnits` check. In `dijkstraUtxoTransition`, replace the call to `Alonzo.validateExUnitsTooBigUTxO pp tx` with a check that sums `totExUnits` over the top-level tx and each sub-tx:
```haskell
let batchExUnits = totExUnits tx
      <> foldMap totExUnits (OMap.elems $ txBody ^. subTransactionsTxBodyL)
```
Apply the same aggregation in `alonzoMinFeeTx` (or a Dijkstra-specific override) so that the minimum fee correctly accounts for sub-tx script execution costs. Apply the same fix to the block-level `validateExUnits` in `dijkstraBbodyTransition`.

### Proof of Concept
1. Set `maxTxExUnits = ExUnits { exUnitsMem = M, exUnitsSteps = S }`.
2. Construct a top-level tx with zero redeemers in `witsTxL`.
3. Attach K sub-transactions, each with one Plutus redeemer declaring `ExUnits { exUnitsMem = M-1, exUnitsSteps = S-1 }`.
4. Submit the batch. `validateExUnitsTooBigUTxO pp tx` computes `totExUnits tx = ExUnits 0 0 ≤ maxTxExUnits` — passes. Each sub-tx passes `SUBUTXOW`/`SUBUTXO` (no ExUnits check). Total Plutus budget consumed by nodes: K × (M-1, S-1), which grows without bound as K increases (limited only by `maxTxSize`).
