Audit Report

## Title
Single Asset Oracle Revert Permanently Blocks Protocol Fee Minting — (`contracts/LRTOracle.sol`)

## Summary
`LRTOracle._getTotalEthInProtocol()` iterates over all supported assets and calls `getAssetPrice(asset)` for each one with no error handling. If any single asset's price oracle permanently reverts (e.g., a Chainlink feed goes offline), `updateRSETHPrice()` permanently reverts, freezing `rsETHPrice` and permanently blocking all protocol fee minting. The insolvency sub-claim is not valid at the severity claimed because `unlockQueue` is restricted to the operator role.

## Finding Description
`_getTotalEthInProtocol()` iterates over every supported asset and calls `getAssetPrice(asset)` at each iteration with no `try/catch`:

```solidity
// LRTOracle.sol L336-348
for (uint16 assetIdx; assetIdx < supportedAssetCount;) {
    address asset = supportedAssets[assetIdx];
    uint256 assetER = getAssetPrice(asset);   // ← no error handling
    uint256 totalAssetAmt = ILRTDepositPool(lrtDepositPoolAddr).getTotalAssetDeposits(asset);
    totalETHInProtocol += totalAssetAmt.mulWad(assetER);
    unchecked { ++assetIdx; }
}
```

`getAssetPrice` delegates to the registered `IPriceFetcher`, e.g. `ChainlinkPriceOracle.getAssetPrice()`, which makes a bare external call to `priceFeed.latestRoundData()` with no error handling:

```solidity
// ChainlinkPriceOracle.sol L49-55
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

If `latestRoundData()` reverts (feed offline, deprecated, or paused), the revert propagates through `getAssetPrice` → `_getTotalEthInProtocol` → `_updateRsETHPrice` → `updateRSETHPrice`. The stored `rsETHPrice` is never written (line 313), and the fee-minting block at lines 299–311 is never reached. Both `updateRSETHPrice()` (public) and `updateRSETHPriceAsManager()` (manager-only) share the same internal path and are equally blocked.

The downside-protection auto-pause at lines 270–282 is also unreachable, but this is a safety-mechanism failure, not a direct fund-loss path.

The insolvency sub-claim relies on an operator calling `unlockQueue`, which carries `onlyAssetTransferOrOperatorRole`. Per the exclusion rules, operator-dependent exploit paths are out of scope. `_createUnlockParams` does read the frozen `lrtOracle.rsETHPrice()` (line 847), but triggering the payout at an inflated price requires the operator to act.

## Impact Explanation
**Medium — Permanent freezing of unclaimed yield.**

Protocol fees are minted inside `_updateRsETHPrice()` at lines 299–311 via `_checkAndUpdateDailyFeeMintLimit`. Because `_getTotalEthInProtocol()` reverts before that code is ever reached, no protocol fee rsETH can be minted to the treasury for as long as the failing oracle remains unresolved. This matches the allowed impact "Medium. Permanent freezing of unclaimed yield." No privileged action is required to trigger or sustain this condition.

## Likelihood Explanation
The protocol integrates multiple external price feeds (Chainlink for stETH/ETHx, native exchange-rate calls for rETH/sfrxETH/swETH). Any one of these feeds going offline — a realistic and documented Chainlink risk — is sufficient to trigger the condition. With `n` supported assets there are `n` independent failure points. No attacker capability is required; the condition is triggered passively by an external feed failure.

## Recommendation
Wrap each `getAssetPrice(asset)` call inside `_getTotalEthInProtocol()` in a `try/catch` block. On revert, either skip the asset and emit a warning event, or revert only when a configurable minimum number of oracles fail. Additionally, `ChainlinkPriceOracle.getAssetPrice()` should validate that `price > 0` and check `updatedAt` staleness before returning, to guard against feeds that return without reverting but supply stale or zero data.

## Proof of Concept
1. Deploy protocol with three supported assets: stETH, rETH, ETHx, each with a registered price oracle.
2. Simulate the rETH Chainlink feed going offline by making `latestRoundData()` revert (e.g., deploy a mock feed that always reverts, and point `ChainlinkPriceOracle.assetPriceFeed[rETH]` to it via `updatePriceFeedFor`).
3. Call `updateRSETHPrice()`. Observe it reverts at the `getAssetPrice(rETH)` call inside `_getTotalEthInProtocol()`.
4. Advance time by 24 hours. Call `updateRSETHPrice()` again. It still reverts.
5. Observe that `rsETHPrice` remains at its last stored value and no fee rsETH has been minted to the treasury.
6. Confirm that `feePeriodStartTime` and `currentPeriodMintedFeeAmount` are never updated, proving fee minting is permanently frozen for the duration of the oracle outage.
