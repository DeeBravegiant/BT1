Audit Report

## Title
Missing `tokenFeeBps` Initialization in `addSupportedToken()` Creates Guaranteed Zero-Fee Deposit Window - (File: contracts/pools/RSETHPool.sol)

## Summary
`addSupportedToken()` (restricted to `TIMELOCK_ROLE`) never initializes `tokenFeeBps[token]`, leaving it at the Solidity default of zero. Because `setTokenFeeBps()` is restricted to a separate `DEFAULT_ADMIN_ROLE`, the two calls cannot be made atomically, guaranteeing a window during which any depositor pays zero fees. All deposits made in this window accrue no fee revenue to the protocol, constituting theft of unclaimed yield.

## Finding Description
`RSETHPool.sol` declares a per-token fee mapping at line 88:

```solidity
mapping(address token => uint256 feeBps) public tokenFeeBps;
```

`addSupportedToken()` (lines 637–656) sets `supportedTokenOracle[token]`, `tokenBridge[token]`, and pushes to `supportedTokenList`, but never writes `tokenFeeBps[token]`. It immediately emits `AddSupportedToken`, making the token live for deposits.

`viewSwapRsETHAmountAndFee` (lines 335–336) reads this mapping directly:

```solidity
uint256 feeBpsForToken = tokenFeeBps[token];   // 0 until setTokenFeeBps is called
fee = amount * feeBpsForToken / 10_000;         // fee == 0
```

`setTokenFeeBps()` (lines 583–594) is the only way to write a non-zero fee, but it requires `DEFAULT_ADMIN_ROLE`, while `addSupportedToken()` requires `TIMELOCK_ROLE`. These are distinct roles; no single caller can invoke both in one transaction, and no multicall path is provided. The zero-fee window is therefore structurally guaranteed across at least two separate transactions.

The public `deposit(address token, uint256 amount, string referralId)` function (lines 284–305) is callable by any address once the token is supported:

```solidity
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount, token);
feeEarnedInToken[token] += fee;   // += 0; protocol earns nothing
IERC20(address(wrsETH)).safeTransfer(msg.sender, rsETHAmount); // full amount, no deduction
```

No existing guard prevents deposits during the zero-fee window. `onlySupportedToken` only checks that `supportedTokenOracle[token] != address(0)`, which is satisfied immediately after `addSupportedToken()`.

## Impact Explanation
**High — Theft of unclaimed yield.** The protocol fee (`feeEarnedInToken[token]`) is the yield earned by the pool on token deposits. During the zero-fee window, every deposit results in `feeEarnedInToken[token] += 0`, and the depositor receives rsETH computed on the full `amount` rather than `amount - fee`. The protocol loses all fee revenue it was designed to collect on those deposits. This is a direct, quantifiable, and irreversible loss of yield for every deposit made before `setTokenFeeBps()` is called.

## Likelihood Explanation
The precondition — a new token being listed — is a routine operational action. The gap is structurally guaranteed because `TIMELOCK_ROLE` and `DEFAULT_ADMIN_ROLE` are separate roles with no atomic execution path. Any user (or mempool-watching bot) can observe the `AddSupportedToken` event on-chain and immediately call `deposit()` with a large amount at 0 bps. No special privileges, flash loans, or oracle manipulation are required. The exploit is repeatable for every new token listing.

## Recommendation
Accept an `initialFeeBps` parameter in `addSupportedToken()` and write it atomically:

```diff
-function addSupportedToken(address token, address oracle, address bridge) external onlyRole(TIMELOCK_ROLE) {
+function addSupportedToken(address token, address oracle, address bridge, uint256 initialFeeBps) external onlyRole(TIMELOCK_ROLE) {
     UtilLib.checkNonZeroAddress(token);
     UtilLib.checkNonZeroAddress(oracle);
     UtilLib.checkNonZeroAddress(bridge);
+    if (initialFeeBps == 0 || initialFeeBps > 10_000) revert InvalidFeeAmount();
     ...
     supportedTokenList.push(token);
     supportedTokenOracle[token] = oracle;
     tokenBridge[token] = bridge;
+    tokenFeeBps[token] = initialFeeBps;
     emit AddSupportedToken(token, oracle, bridge);
 }
```

This eliminates the zero-fee window entirely. Alternatively, if a zero-fee token is intentional, the `deposit` function should gate on an explicit "fee initialized" flag rather than relying on a zero-default mapping value.

## Proof of Concept
1. `TIMELOCK_ROLE` calls `addSupportedToken(tokenX, oracleX, bridgeX)`. `tokenFeeBps[tokenX]` remains `0`.
2. Attacker observes the `AddSupportedToken(tokenX, ...)` event and immediately calls `deposit(tokenX, 1_000_000e18, "")`.
3. Inside `viewSwapRsETHAmountAndFee(1_000_000e18, tokenX)`:
   - `feeBpsForToken = tokenFeeBps[tokenX]` → `0`
   - `fee = 1_000_000e18 * 0 / 10_000` → `0`
   - `rsETHAmount = 1_000_000e18 * tokenToETHRate / rsETHToETHrate` (full amount, no deduction)
4. `feeEarnedInToken[tokenX] += 0` — protocol earns nothing.
5. Attacker receives rsETH equivalent to the full deposit with zero fee paid.
6. `DEFAULT_ADMIN_ROLE` later calls `setTokenFeeBps(tokenX, 100)` — all prior deposits already settled at 0 bps.

Foundry test plan: deploy `RSETHPool` with a mock oracle and mock token; call `addSupportedToken`; assert `tokenFeeBps[token] == 0`; call `deposit` as an unprivileged address; assert `feeEarnedInToken[token] == 0` and attacker received rsETH based on full `amount`; call `setTokenFeeBps(token, 100)`; assert subsequent deposits correctly accrue fees.
