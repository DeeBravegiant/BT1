### Title
Excess Deposit Silently Retained in `submit_participant_info` for Existing-Participant Re-submissions — (File: `crates/contract/src/lib.rs`)

---

### Summary

`submit_participant_info` is a `#[payable]` function that accepts any attached NEAR deposit. Its deposit-handling logic is gated behind a boolean flag `attestation_storage_must_be_paid_by_caller`. When that flag is `false` — which occurs whenever the caller is an **existing participant** re-submitting an updated attestation — the entire deposit-check-and-refund block is skipped. Any NEAR attached to such a call is silently retained by the contract with no refund issued.

---

### Finding Description

In `crates/contract/src/lib.rs`, `submit_participant_info` computes:

```rust
let caller_is_not_participant = self.voter_account().is_err();
let is_new_attestation = matches!(
    attestation_insertion_result,
    ParticipantInsertion::NewlyInsertedParticipant
);

let attestation_storage_must_be_paid_by_caller =
    is_new_attestation || caller_is_not_participant;

if attestation_storage_must_be_paid_by_caller {
    // ... storage cost check ...
    // Refund the difference if the proposer attached more than required
    if let Some(diff) = attached.checked_sub(cost)
        && diff > NearToken::from_yoctonear(0)
    {
        Promise::new(account_id).transfer(diff).detach();
    }
}
``` [1](#0-0) 

When `is_new_attestation == false` AND `caller_is_not_participant == false` (i.e., an existing participant re-submits their attestation — the normal hourly renewal path), `attestation_storage_must_be_paid_by_caller` evaluates to `false`. The `if` block is entirely skipped. No deposit is checked, and no refund is issued. Any NEAR attached to the call is absorbed by the contract.

The function is marked `#[payable]`, so the NEAR runtime accepts any deposit amount without restriction. [2](#0-1) 

The contrast with every other payable entry point in the contract is stark: `require_deposit` (used by `sign`, `request_app_private_key`, `verify_foreign_transaction`) always refunds excess, and `propose_update` explicitly refunds the difference after computing the required amount. [3](#0-2) [4](#0-3) 

---

### Impact Explanation

**Medium. Balance/accounting invariant violation.**

Any NEAR attached to a re-submission call by an existing participant is permanently locked in the contract. The contract's own README states "Any excess deposit is automatically refunded" as a system-wide invariant for payable methods. This invariant is broken for the re-submission path of `submit_participant_info`.

The docs explicitly document a manual-deposit workflow for first-time joiners:

> "first-time joiners must attach enough yoctoNEAR for storage; the node attaches `0`, so call `submit_participant_info` manually with `--deposit` once." [5](#0-4) 

An operator who follows this guidance, then later re-submits (e.g., after attestation expiry) using the same manual command with `--deposit`, will silently lose the attached NEAR. The contract gives no error, no log, and no refund.

---

### Likelihood Explanation

- `submit_participant_info` is called on a **1-hour cadence** by every MPC node for attestation renewal.
- The automated node path attaches `0` deposit, so automated re-submissions are unaffected.
- However, the documented operator recovery path explicitly instructs manual calls with `--deposit`. An operator who is already a participant and follows this path for a re-submission (e.g., after an attestation rejection or expiry) will trigger the silent-retention path.
- The `#[payable]` attribute provides no guard; the NEAR runtime will accept any deposit silently.

---

### Recommendation

Add an unconditional refund for any excess deposit when `attestation_storage_must_be_paid_by_caller` is `false`. Since no storage cost is charged in that branch, the entire attached deposit should be refunded:

```rust
} else {
    // Existing participant re-submission: no storage cost, refund any deposit.
    let attached = env::attached_deposit();
    if attached > NearToken::from_yoctonear(0) {
        Promise::new(account_id).transfer(attached).detach();
    }
}
```

Alternatively, align with the `require_deposit` helper pattern used by all other payable methods.

---

### Proof of Concept

1. Deploy the contract and initialize it with participant `alice.near`.
2. `alice.near` calls `submit_participant_info` with a valid attestation and `attached_deposit = 1_000_000 yoctoNEAR`.
3. The call succeeds (`Ok(())`).
4. `alice.near`'s balance decreases by `1_000_000 yoctoNEAR` (plus gas).
5. The contract balance increases by `1_000_000 yoctoNEAR`.
6. No refund promise is scheduled; the deposit is permanently retained.

The condition is confirmed by the `add_participant` test that explicitly returns `Ok(ParticipantInsertion::UpdatedExistingParticipant)` for re-submissions: [6](#0-5) 

With `attestation_insertion_result == UpdatedExistingParticipant`, `is_new_attestation` is `false`; if the caller is a current voter, `caller_is_not_participant` is also `false`; the deposit block is skipped entirely. [7](#0-6)
