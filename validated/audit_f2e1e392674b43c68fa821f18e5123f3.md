### Title
Unverified Binding Between `respond_verify_foreign_tx` Response and Request Allows Cross-Request Response Injection - (File: crates/contract/src/lib.rs)

### Summary
`respond_verify_foreign_tx` verifies that the submitted signature is valid over `response.payload_hash` using the root MPC key, but never verifies that `response.payload_hash` is actually derived from the `request` being resolved. A single malicious attested participant (below threshold) can replay any previously-seen valid `(payload_hash, signature)` pair to resolve a different pending request, delivering a forged foreign-chain verification result to the waiting caller.

### Finding Description
In `respond_verify_foreign_tx`, the contract performs two checks before resolving the pending yield:

1. The caller is an attested participant.
2. The signature in `response` is valid over `response.payload_hash` under the root MPC public key. [1](#0-0) 

What is **never checked** is whether `response.payload_hash` equals `SHA-256(borsh(ForeignTxSignPayload { request, extracted_values }))` — i.e., that the hash was actually computed from the `request` argument passed to the same call.

The `ForeignTxSignPayload` structure binds the request data to the extracted values: [2](#0-1) 

The `near-mpc-sdk` client-side verifier **does** perform this check (`expected_payload_hash == response.payload_hash`): [3](#0-2) 

But the on-chain contract enforces no such binding. The `resolve_yields_for` helper simply resumes all yields queued under the given `request` key with whatever `response_bytes` it is given: [4](#0-3) 

### Impact Explanation
A single malicious attested participant (strictly below the signing threshold) can:

1. Observe any previously-settled `respond_verify_foreign_tx` call on-chain and extract its `{payload_hash: H1, signature: sig_H1}` — both are public.
2. Wait for a victim's `verify_foreign_transaction` request R2 (for transaction T2, e.g., a fraudulent or unconfirmed transfer) to appear in `pending_verify_foreign_tx_requests`.
3. Call `respond_verify_foreign_tx(R2, {payload_hash: H1, signature: sig_H1})`.
4. The contract accepts the call: `sig_H1` is a valid threshold signature over `H1` under the root key ✓. It resolves R2's yield with the mismatched response.
5. The bridge contract waiting on R2 receives a `VerifyForeignTransactionResponse` carrying a valid MPC root-key signature — but over H1 (T1's verification data), not H2 (T2's).

Any bridge contract that checks only signature validity without independently recomputing the expected `payload_hash` from the transaction it submitted will accept this as proof that T2 was verified, enabling invalid bridge execution or double-spend conditions.

This matches the **High** impact category: *forged foreign-chain verification that causes invalid bridge execution*.

### Likelihood Explanation
- **Entry path is unprivileged within the participant set**: requires only a single attested participant account, well below the signing threshold.
- **No new cryptographic material needed**: the attacker reuses a legitimately-produced threshold signature already published on-chain.
- **Victim requests are public**: all pending `verify_foreign_transaction` requests are visible in contract state.
- **Bridge contracts are the primary consumers** of this flow (Omnibridge inbound, chain abstraction), and many will rely on the contract's response without re-deriving the payload hash.

### Recommendation
Inside `respond_verify_foreign_tx`, after retrieving the domain public key and before calling `resolve_yields_for`, recompute the minimum expected payload hash from `request` and verify it matches `response.payload_hash`. Because the contract does not have the extracted values at response time, the simplest enforceable invariant is to verify that the signature was produced over a hash that encodes the `request` field — or, alternatively, store the expected `payload_hash` alongside the pending yield at submission time (if the hash can be pre-committed) and compare on response.

A lighter mitigation is to document clearly that `respond_verify_foreign_tx` does **not** bind the response to the request, and require all consuming bridge contracts to call `ForeignChainSignatureVerifier::verify_signature` from the `near-mpc-sdk` before acting on the result. [5](#0-4) 

### Proof of Concept

```
Setup:
  - Contract has two pending requests:
      R1 = VerifyForeignTransactionRequest { tx_id: [0xAA; 32], ... }  (legitimate)
      R2 = VerifyForeignTransactionRequest { tx_id: [0xBB; 32], ... }  (fraudulent tx)

Step 1: Honest nodes process R1 normally.
  Leader calls: respond_verify_foreign_tx(R1, { payload_hash: H1, signature: sig_H1 })
  H1 = SHA-256(borsh(ForeignTxSignPayload { request: R1.request, values: [...] }))
  Contract accepts. R1 resolved. sig_H1 is now public on-chain.

Step 2: Malicious attested participant (single node) calls:
  respond_verify_foreign_tx(R2, { payload_hash: H1, signature: sig_H1 })

Step 3: Contract checks (lib.rs:718-734):
  - assert_caller_is_attested_participant_and_protocol_active() → PASS
  - verify_ecdsa_signature(sig_H1, H1, root_pk) → PASS (sig_H1 is a valid threshold sig)
  - resolve_yields_for(&pending_verify_foreign_tx_requests, &R2, response_bytes) → PASS

Step 4: Bridge contract waiting on R2 receives:
  VerifyForeignTransactionResponse { payload_hash: H1, signature: sig_H1 }
  If it only checks `verify_ecdsa_signature(sig_H1, H1, root_pk)` → accepts as valid proof
  that fraudulent tx [0xBB; 32] was verified by the MPC network.
``` [6](#0-5)

    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

        let payload_is_correct = expected_payload_hash == response.payload_hash;

                got: response.payload_hash.clone(),
                expected: expected_payload_hash,

        };

        verification_result.map_err(|_| VerifyForeignChainError::SignatureVerificationFailed)

pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
