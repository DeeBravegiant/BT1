### Title
Silent Sync Stall via Empty `RespondBlocks` — (`chia/full_node/full_node.py`, `chia/util/task_pipeline.py`)

### Summary

An unprivileged remote peer can respond to every `RequestBlocks` message with a `RespondBlocks` containing an empty `blocks` list. The sync pipeline accepts this as a valid response, `skip_blocks` returns `[]` for an empty input, `validate_batch` returns `None` (filter), the `TaskPipeline` silently discards all filtered items, and `sync_from_fork_point` returns normally — without having ingested a single block and without raising any error. The victim node exits sync mode with its peak unchanged and is never alerted to the failure.

---

### Finding Description

**Entrypoint — `fetch_blocks` yields empty block lists without validation**

Inside `sync_from_fork_point`, the inner generator `fetch_blocks` iterates over every batch range and sends `RequestBlocks` to a peer. The only guard on the response is a type check:

```python
elif isinstance(response, RespondBlocks):
    ...
    yield (peer, response.blocks)   # ← yields even when response.blocks == []
    fetched = True
    break
``` [1](#0-0) 

There is no check that `response.blocks` is non-empty. A peer returning `RespondBlocks(start, end, [])` is treated identically to one returning a full batch. The peer is never closed or penalised.

---

**`skip_blocks` returns `[]` for an empty input**

`validate_batch` passes the received block list directly to `skip_blocks`:

```python
blocks_to_validate = await self.skip_blocks(blockchain, blocks, fork_info, vs)
``` [2](#0-1) 

`skip_blocks` initialises `blocks_to_validate = []` and iterates over `all_blocks`. When `all_blocks` is `[]`, the loop body never executes and `[]` is returned immediately:

```python
blocks_to_validate: list[FullBlock] = []
for i, block in enumerate(all_blocks):   # never entered
    ...
return blocks_to_validate                # returns []
``` [3](#0-2) 

---

**`validate_batch` returns `None` — the pipeline filter path**

```python
if len(blocks_to_validate) == 0:
    return None
``` [4](#0-3) 

The assertion on line 1370 does **not** catch this:

```python
assert first_batch or len(blocks_to_validate) == len(blocks)
```

When `blocks = []` and `blocks_to_validate = []`, the condition `0 == 0` is trivially true. Furthermore, `first_batch` is never set to `False` (line 1376 is unreachable when `blocks_to_validate` is always empty), so the `first_batch` branch of the `or` is always taken. [5](#0-4) 

---

**`TaskPipeline` silently discards `None` results**

In `_run_stage`, a `None` return from any stage is treated as a filter signal — the item is dropped and the loop continues:

```python
result = await self._stages[idx](item)
...
if output_queue is None or result is None:
    continue
``` [6](#0-5) 

All batches produce `None` from `validate_batch`. The pipeline drains normally, no exception is set, and `pipeline.run()` returns without error.

---

**`sync_from_fork_point` returns silently**

```python
try:
    await pipeline.run()
except Exception:
    self.log.exception("sync from fork point failed")
``` [7](#0-6) 

No exception is raised, so the `except` branch is never taken. The function returns normally. The node's blockchain peak is unchanged.

---

### Impact Explanation

The victim node:
- Exits sync mode believing it has completed a sync.
- Its peak has not advanced.
- No error is logged at a level that would trigger operator intervention.
- If the attacker controls all peers advertising the target peak, the node can never advance its chain — a permanent sync stall.
- Even with honest peers present, the attacker can win the per-batch peer race (peers are sorted by timestamp; a fast-responding malicious peer will consistently have the lowest timestamp and be tried first), blocking every batch.

This satisfies the **High** impact criterion: *"Permanent or long-lived inability for honest nodes to process valid blocks during sync."*

---

### Likelihood Explanation

- Requires only a standard peer connection — no keys, no admin access, no cryptographic break.
- The malicious peer is never banned (only `None`/timeout responses trigger `peer.close()`).
- A single malicious peer that is the sole advertiser of the target peak is sufficient to stall the victim indefinitely.
- With multiple honest peers, the attacker must win the timestamp sort for each batch, which is achievable by responding quickly.

---

### Recommendation

1. **In `fetch_blocks`**: reject a `RespondBlocks` with `len(response.blocks) == 0`, close the peer, and fall through to try the next peer — just as a `None` response is handled.
2. **In `validate_batch`**: add an assertion or error when `blocks` (the raw fetched list) is empty, since `fetch_blocks` should never yield an empty batch.
3. **Post-sync invariant**: after `pipeline.run()` completes, verify that the node's peak has advanced to at least `target_peak_sb_height`; raise an error if it has not.

---

### Proof of Concept

```python
# Monkeypatch all peers to return RespondBlocks with empty block lists.
# Assert that sync_from_fork_point returns without error but the peak
# has not advanced.

class EmptyBlocksPeer:
    closed = False
    peer_info = PeerInfo("127.0.0.1", uint16(8444))

    async def call_api(self, *args, **kwargs):
        req = args[1]
        if isinstance(req, full_node_protocol.RequestBlocks):
            # Valid RespondBlocks message, but with no blocks
            return full_node_protocol.RespondBlocks(
                req.start_height, req.end_height, []
            )
        return None

    async def close(self, *args, **kwargs):
        self.closed = True

peer = EmptyBlocksPeer()
monkeypatch.setattr(full_node, "get_peers_with_peak", lambda _: [peer])

pre_sync_peak = full_node.blockchain.get_peak()
# Sync over a non-zero height range
await asyncio.wait_for(
    full_node.sync_from_fork_point(uint32(0), uint32(100), std_hash(b"peak"), []),
    timeout=30,
)
post_sync_peak = full_node.blockchain.get_peak()

# BUG: returns without error, peak unchanged
assert pre_sync_peak == post_sync_peak   # passes — no blocks were ingested
assert not peer.closed                   # peer was never penalised
```
