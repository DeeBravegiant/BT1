Audit Report

## Title
Excess ETH Sent to `MultiChainRateProvider.updateRate()` Is Permanently Trapped With No Recovery Path - (File: contracts/cross-chain/MultiChainRateProvider.sol)

## Summary
`MultiChainRateProvider.updateRate()` is a permissionless `payable` function that forwards exactly `estimatedFee` per chain to the LayerZero endpoint but imposes no constraint on `msg.value` and performs no post-loop refund. The contract inherits only `Ownable` and `ReentrancyGuard` and defines no `withdraw`, `receive`, `fallback`, or sweep function, so any ETH sent above the sum of per-chain fees is permanently unrecoverable.

## Finding Description
`updateRate()` (lines 108–137) iterates over `rateReceivers`, queries `estimateFees()` per chain, and forwards exactly that amount:

```solidity
(uint256 estimatedFee,) = ILayerZeroEndpoint(layerZeroEndpoint)
    .estimateFees(dstChainId, address(this), _payload, false, bytes(""));

ILayerZeroEndpoint(layerZeroEndpoint).send{ value: estimatedFee }(
    dstChainId, remoteAndLocalAddresses, _payload, payable(msg.sender), address(0x0), bytes("")
);
```

There is no pre-loop `require(msg.value >= totalFee)` guard and no post-loop `msg.value - totalConsumed` refund. A full `grep` over `contracts/cross-chain/` for `withdraw`, `receive()`, `fallback`, and `sweep` returns zero matches, confirming there is no recovery path. Any ETH left in the contract after the loop is permanently frozen.

By contrast, `CrossChainRateProvider.updateRate()` (line 96) passes `{ value: msg.value }` directly to the LZ endpoint with `payable(msg.sender)` as the refund address, so the endpoint itself returns any excess — that contract is unaffected.

The root cause is the architectural mismatch: the multi-chain variant must split `msg.value` across N sends, so it cannot pass the full `msg.value` to a single endpoint call, yet it never accounts for the remainder.

## Impact Explanation
Any ETH sent above the sum of per-chain `estimatedFee` values is permanently frozen inside `MultiChainRateProvider`. Because the contract has no withdrawal or sweep mechanism, the loss is irreversible. This matches **Critical — Permanent freezing of funds**: the caller's ETH is locked in the contract with no on-chain path to recover it.

## Likelihood Explanation
`updateRate()` has no access control — any external account or keeper bot can call it. On-chain fee estimates can shift between the `estimateTotalFee()` view call and the actual transaction (gas price changes, LZ config updates), so callers who add a safety buffer — a standard defensive pattern — will silently lose that buffer on every call. The function is also callable by automated keeper infrastructure that routinely over-provisions ETH to guarantee execution. Likelihood is **Low-to-Medium**: the permissionless entry point and common buffering practice make repeated, small-but-permanent losses realistic over the contract's lifetime.

## Recommendation
After the loop, compute total consumed fees and refund any remainder:

```solidity
uint256 totalConsumed;
for (uint256 i; i < rateReceiversLength;) {
    // ... existing estimateFees + send logic ...
    totalConsumed += estimatedFee;
    unchecked { ++i; }
}
uint256 excess = msg.value - totalConsumed;
if (excess > 0) {
    (bool ok,) = payable(msg.sender).call{ value: excess }("");
    require(ok, "Refund failed");
}
```

Alternatively, add a pre-loop `require(msg.value == estimateTotalFee(), "Incorrect fee")` to reject any call that does not supply the exact fee, eliminating the possibility of trapped ETH entirely.

## Proof of Concept
1. Deploy `MultiChainRateProvider` with two configured `rateReceivers`.
2. Call `estimateTotalFee()` — suppose it returns `0.01 ETH`.
3. Call `updateRate{ value: 0.02 ETH }()` (caller adds a 2× buffer).
4. The loop consumes exactly `0.01 ETH` across the two LZ `send` calls.
5. The remaining `0.01 ETH` remains in the contract's balance.
6. No function exists to recover it; the ETH is permanently frozen.

Foundry fork test: deploy a mock LZ endpoint that returns fixed `estimatedFee` values and records `send` calls; call `updateRate{ value: 2 * totalFee }()`; assert `address(multiChainRateProvider).balance == totalFee` after the call and that no subsequent call can reduce that balance to zero.
