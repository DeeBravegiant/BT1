After thorough investigation of the codebase, I traced the analog of the "wrong identity used in a privileged operation" bug class through the proposal validation path.

### Title
Unvalidated `builder` Field in `ProposalInit` Allows Malicious Proposer to Redirect Block Fees to Arbitrary Address — (`File: crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

### Summary
`ProposalInit` carries two distinct identity fields: `proposer` (the consensus committee member) and `builder` (the address of the entity that builds/sequences the block, mapped to `sequencer_address` in block execution). Validators check `proposer` against the committee but never validate `builder`. A malicious staker elected as proposer can set `builder` to any address; validators accept it, execute the block with that address as `sequencer_address`, and all fee transfers for the block are routed to the attacker-controlled address.

### Finding Description

`ProposalInit` has two separate identity fields: [1](#0-0) 

The proposer populates `builder` from a static config value: [2](#0-1) 

which flows into `initiate_build`: [3](#0-2) 

On the validator side, `is_proposal_init_valid` checks `height`, `l1_da_mode`, `l2_gas_price_fri`, `starknet_version`, `version_constant_commitment`, and L1 gas prices — but **never checks `init_proposed.builder`**: [4](#0-3) 

Both proposer and validator then call `convert_to_sn_api_block_info(&init)` to derive the `block_info` passed to the batcher: [5](#0-4) 

`builder` maps to `sequencer_address` in the resulting `BlockInfo`. The blockifier's fee transfer uses `sequencer_address` as the recipient: [6](#0-5) 

Because validators compute the `ProposalCommitment` using the attacker-supplied `builder`, the `ProposalFin` comparison passes and the block is committed with the wrong sequencer address.

### Impact Explanation

A malicious staker elected as proposer sets `builder` to an attacker-controlled address. Validators accept the `ProposalInit` without checking `builder`, execute the block with `sequencer_address = attacker_address`, and all transaction fees for the block are transferred to the attacker. The committed state diff and block hash both embed the attacker's address as sequencer. This matches: **Critical — Incorrect fee/balance effect with economic impact** and **Wrong state/receipt value from blockifier execution**.

Additionally, if `builder` is set to match a specific transaction's `sender_address`, `is_sequencer_the_sender()` returns `true` for that transaction, switching it from the concurrent fee-transfer path to the sequential path: [7](#0-6) 

This produces a different state diff than the honest execution path.

### Likelihood Explanation

Any staker in the committee who is elected as proposer for a round can trigger this. No external attacker capability is required beyond holding a staking position. The attack is silent — validators log no warning about `builder` — and the committed block appears valid to all consensus participants.

### Recommendation

Add `builder` to `ProposalInitValidation` and enforce it in `is_proposal_init_valid`. The expected `builder` for a given proposer should be derived from the committee/staking registry (analogous to how `proposer` is checked against `get_proposer(height, round)`). At minimum, validators should reject any `ProposalInit` whose `builder` does not match the locally-configured or committee-registered builder address for that proposer. [8](#0-7) 

### Proof of Concept

1. Attacker stakes tokens and is elected proposer for block N.
2. Attacker modifies `builder_address` in their local config (or patches `initiate_build`) to set `builder = 0xATTACKER`.
3. Attacker broadcasts `ProposalInit { proposer: <legitimate>, builder: 0xATTACKER, ... }`.
4. Validators call `is_proposal_init_valid` — no check on `builder` — and proceed.
5. Both proposer and validators call `convert_to_sn_api_block_info(&init)` → `sequencer_address = 0xATTACKER`.
6. Blockifier executes all transactions; every fee transfer calls `transfer(recipient=0xATTACKER, amount=fee)`.
7. `ProposalFin` comparison passes (both sides computed the hash with `0xATTACKER`).
8. Block is committed; state diff shows all fees credited to `0xATTACKER`; legitimate sequencer receives nothing.

```rust
#[instrument(level = "warn", skip_all, fields(?proposal_init_validation, ?init_proposed))]
async fn is_proposal_init_valid(
    proposal_init_validation: &ProposalInitValidation,
    init_proposed: &ProposalInit,
    clock: &dyn Clock,
    l1_gas_price_provider: Arc<dyn L1GasPriceProviderClient>,
    gas_price_params: &GasPriceParams,
) -> ValidateProposalResult<()> {
    let now: u64 = clock.unix_now();
    let last_block_timestamp =
        proposal_init_validation.previous_proposal_init.as_ref().map_or(0, |info| info.timestamp);
    if init_proposed.timestamp < last_block_timestamp {
        return Err(ValidateProposalError::InvalidProposalInit(
            init_proposed.clone(),
            proposal_init_validation.clone(),
            format!(
                "Timestamp is too old: last_block_timestamp={}, proposed={}",
                last_block_timestamp, init_proposed.timestamp
            ),
        ));
    }
    if init_proposed.timestamp > now + proposal_init_validation.block_timestamp_window_seconds {
        return Err(ValidateProposalError::InvalidProposalInit(
            init_proposed.clone(),
            proposal_init_validation.clone(),
            format!(
                "Timestamp is in the future: now={}, block_timestamp_window_seconds={}, \
                 proposed={}",
                now,
                proposal_init_validation.block_timestamp_window_seconds,
                init_proposed.timestamp
            ),
        ));
    }
    if init_proposed.starknet_version != proposal_init_validation.starknet_version {
        return Err(ValidateProposalError::InvalidProposalInit(
            init_proposed.clone(),
            proposal_init_validation.clone(),
            format!(
                "starknet_version mismatch: expected={:?}, proposed={:?}",
                proposal_init_validation.starknet_version, init_proposed.starknet_version
            ),
        ));
    }
    // `version_constant_commitment` is proposer-supplied (network-derived). It is not yet a real
    // commitment (see `expected_version_constant_commitment`): the only valid value is the
    // sentinel, so reject anything else. Enforcing the same value the proposer emits keeps the two
    // sides in lockstep, so a real value cannot ship on one side without the other.
    let expected_commitment = expected_version_constant_commitment();
    if init_proposed.version_constant_commitment != expected_commitment {
        return Err(ValidateProposalError::InvalidProposalInit(
            init_proposed.clone(),
            proposal_init_validation.clone(),
            format!(
                "version_constant_commitment mismatch: expected={expected_commitment:?}, \
                 proposed={:?}",
                init_proposed.version_constant_commitment
            ),
        ));
    }
    if !(init_proposed.height == proposal_init_validation.height
        && init_proposed.l1_da_mode == proposal_init_validation.l1_da_mode
        && init_proposed.l2_gas_price_fri == proposal_init_validation.l2_gas_price_fri)
    {
        return Err(ValidateProposalError::InvalidProposalInit(
            init_proposed.clone(),
            proposal_init_validation.clone(),
            "ProposalInit validation failed".to_string(),
        ));
    }
```
