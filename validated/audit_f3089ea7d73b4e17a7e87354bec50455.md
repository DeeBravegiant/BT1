Audit Report

## Title
ETH Deposit Limit Check Omits Deposit Amount, Allowing Limit Bypass - (File: contracts/LRTDepositPool.sol)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` in `LRTDepositPool.sol` uses `totalAssetDeposits > limit` for ETH but `totalAssetDeposits + amount > limit` for ERC20 assets. This asymmetry means the ETH branch never tests whether the incoming deposit would push total deposits past the cap — only whether the cap is already exceeded. Any unprivileged ETH depositor can push total ETH deposits arbitrarily above the configured limit.

## Finding Description
The root cause is at line 679 of `contracts/LRTDepositPool.sol`:

```solidity
// L676-682
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));   // ← amount omitted
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // ← correct
}
```

The ETH branch returns `true` only when `totalAssetDeposits` **already** exceeds the limit. It never evaluates whether `totalAssetDeposits + amount` would cross the limit. The ERC20 branch correctly includes `amount`.

The call chain is:
- `depositETH` (L76–93) is the public payable entry point, calling `_beforeDeposit(LRTConstants.ETH_TOKEN, msg.value, ...)`.
- `_beforeDeposit` (L648–670) calls `_checkIfDepositAmountExceedesCurrentLimit` and reverts with `MaximumDepositLimitReached` only if it returns `true`.
- Because the ETH branch omits `amount`, the revert is never triggered for a deposit that would cross (but not already exceed) the limit.

No other guard exists between `depositETH` and `_mintRsETH` that enforces the deposit cap.

## Impact Explanation
The deposit limit is a protocol-stated invariant. With this bug, the invariant is silently violated for ETH: total ETH deposits can exceed the configured cap by an unbounded amount in a single transaction. This maps directly to **Low — Contract fails to deliver promised returns** (deposit limit invariant broken). If the cap is sized to match EigenLayer strategy capacity or operator delegation limits, exceeding it can cause the protocol to accept liabilities it cannot service, escalating toward **Critical — Protocol insolvency**, though that escalation depends on external configuration not proven here.

## Likelihood Explanation
No special role, privilege, or precondition beyond a normal operating state is required. The only precondition is that `totalAssetDeposits < depositLimitByAsset(ETH_TOKEN)`, which is the normal state of any live protocol that has not yet hit its cap. Any external ETH depositor can trigger this in a single `depositETH` call. The exploit is repeatable and requires no front-running, oracle manipulation, or governance action.

## Recommendation
Add `amount` to the ETH branch to mirror the ERC20 branch:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // fixed
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

## Proof of Concept
1. Admin sets `depositLimitByAsset(ETH_TOKEN) = 1000 ether`.
2. Protocol accumulates `999.9 ether` in ETH deposits (`totalAssetDeposits = 999.9 ether`).
3. Attacker calls `depositETH{value: 500 ether}(0, "")`.
4. `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 500 ether)` evaluates `999.9 ether > 1000 ether` → `false` → no revert.
5. `_mintRsETH` mints rsETH for 500 ETH; total ETH deposits become `1499.9 ether`, 50% above the cap.
6. The equivalent ERC20 call would evaluate `999.9 + 500 > 1000` → `true` → `MaximumDepositLimitReached` revert.

**Foundry test plan:** Deploy `LRTDepositPool` with a mock `LRTConfig` returning `depositLimitByAsset = 1000 ether`. Seed `getTotalAssetDeposits` to return `999.9 ether`. Call `depositETH{value: 500 ether}`. Assert the call succeeds (no revert) and that `getTotalAssetDeposits` would now reflect `1499.9 ether`. Confirm the same scenario with an ERC20 asset reverts with `MaximumDepositLimitReached`.
