### Title
Single Attested Participant Can Submit Arbitrary CKD Response for `AppPublicKey` Variant Without On-Chain Validation - (`crates/contract/src/lib.rs`)

---

### Summary

In `respond_ckd`, when the pending request uses the `CKDAppPublicKey::AppPublicKey` variant, the contract performs **zero cryptographic validation** of the submitted `CKDResponse`. Any single attested participant can call `respond_ckd` with completely fabricated `big_y` and `big_c` values, and the contract will accept and deliver them to the user. By contrast, the `AppPublicKeyPV` variant is protected by a BLS12-381 pairing check (`ckd_output_check`). This asymmetry means a single Byzantine participant below the signing threshold can corrupt or fully control the confidential key delivered to any user who submits a `request_app_private_key` with the `AppPublicKey` variant.

---

### Finding Description

In `crates/contract/src/lib.rs`, the `respond_ckd` function handles both CKD variants:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}   // ← NO VALIDATION
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV`, the contract enforces the pairing equation `e(big_c, g2) = e(big_y, app_pk2) · e(H(pk||app_id), mpc_pk)` via `ckd_output_check`: [2](#0-1) 

For `AppPublicKey`, the branch is an empty no-op. The `CKDResponse` struct contains only two BLS12-381 G1 points (`big_y`, `big_c`) with no attached proof: [3](#0-2) 

The only gate before the empty branch is `assert_caller_is_attested_participant_and_protocol_active`, which requires the caller to be a single TEE-attested participant — not a threshold of them: [4](#0-3) 

After the empty branch, `resolve_yields_for` immediately drains all queued yields for the request and delivers the fabricated response to every waiting caller: [5](#0-4) 

The existing unit test confirms this: it submits `big_y = [1u8; 48]` and `big_c = [2u8; 48]` (clearly invalid G1 points) and the contract accepts them without error: [6](#0-5) 

---

### Impact Explanation

The CKD protocol for `AppPublicKey` computes:

```
big_c = H(mpc_pk || app_id) * msk  +  app_pk * y
big_y = g * y
```

The user recovers their derived key as `big_c − app_sk · big_y = H(mpc_pk || app_id) * msk`. [7](#0-6) 

A Byzantine attacker who knows a scalar `k` can set:

```
big_y  = g * y'          (for any chosen y')
big_c  = app_pk * y' + g * k
       = g * (app_sk * y') + g * k
```

The user then decrypts: `big_c − app_sk · big_y = g * k`. The attacker has delivered a derived key whose discrete logarithm `k` they know, giving them full signing capability over the user's derived identity on the target application. This constitutes **unauthorized confidential key derivation output** — the attacker controls the key the user believes is their private MPC-derived secret, enabling theft of any funds or assets protected by that key.

---

### Likelihood Explanation

The attacker must be a single TEE-attested participant in the MPC network. This is a Byzantine participant strictly below the signing threshold — exactly the threat model the protocol is designed to resist. The `AppPublicKey` variant is the default/legacy variant used by existing integrations (e.g., `crates/contract/tests/sandbox/utils/sign_utils.rs` `create_response_ckd` uses it). The attack requires only one malicious node to race the honest nodes to call `respond_ckd` first; once `resolve_yields_for` drains the queue, the request is gone and honest nodes cannot overwrite it. [8](#0-7) 

---

### Recommendation

Apply the same on-chain cryptographic validation to `AppPublicKey` responses that is already applied to `AppPublicKeyPV`. For `AppPublicKey`, the contract holds `app_pk` (a G1 point) and the network public key. A verifier can check the analogous relation:

```
e(big_c, g2) = e(big_y, g2)^app_sk · e(H(mpc_pk || app_id), mpc_pk)
```

However, since `app_sk` is the user's private scalar (unknown to the contract), this pairing check cannot be done on-chain without the user's cooperation. The correct fix is one of:

1. **Deprecate `AppPublicKey` in favour of `AppPublicKeyPV`**, which is publicly verifiable on-chain and already protected.
2. **Require the user to submit `app_pk2 = g2 * app_sk`** alongside `app_pk`, converting every `AppPublicKey` request into an `AppPublicKeyPV` request so the existing `ckd_output_check` pairing gate applies.
3. **At minimum**, document that `AppPublicKey` responses carry no on-chain integrity guarantee and that callers must verify the response off-chain before using the derived key, and surface this prominently in the SDK. [9](#0-8) 

---

### Proof of Concept

1. User calls `request_app_private_key` with `CKDAppPublicKey::AppPublicKey(app_pk)` and `domain_id` pointing to a BLS12-381 CKD domain. The request is queued in `pending_ckd_requests`.

2. A single Byzantine attested participant calls `respond_ckd` with the matching `CKDRequest` and a fabricated response:

```rust
let fabricated_response = CKDResponse {
    big_y: Bls12381G1PublicKey::from(&(ElementG1::generator() * y_attacker)),
    big_c: Bls12381G1PublicKey::from(
        &(app_pk_g1 * y_attacker + ElementG1::generator() * k_attacker)
    ),
};
contract.respond_ckd(ckd_request, fabricated_response).unwrap();
// ← succeeds; no pairing check fires for AppPublicKey variant
```

3. The contract drains the yield queue and returns `fabricated_response` to the user.

4. The user decrypts: `big_c − app_sk · big_y = g * k_attacker`. The attacker knows `k_attacker` and can now sign on behalf of the user's derived identity.

The existing unit test at line 3424–3440 already demonstrates step 2 succeeds with byte-array garbage (`[1u8;48]`, `[2u8;48]`), confirming no validation occurs. [6](#0-5)

/// Check that `e(big_c, g2) = e(big_y, app_pk2) . e(hash_point, public_key)`.
///
/// Point validation is fully delegated to the host, as in
/// [`app_public_key_check`].
pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}
