Audit Report

## Title
Unprivileged Caller Can Atomically Pause Three Contracts via `updateRSETHPrice()` Downside-Protection Branch — (`contracts/LRTOracle.sol`)

## Summary

`LRTOracle.updateRSETHPrice()` is a public function with no access-control guard beyond `whenNotPaused`. Its internal downside-protection logic in `_updateRsETHPrice()` atomically pauses `LRTDepositPool`, `LRTWithdrawalManager`, and `LRTOracle` itself whenever the computed rsETH price falls more than `pricePercentageLimit` below `highestRsethPrice`. Any unprivileged EOA can invoke this path whenever the natural oracle-reported price satisfies the threshold condition, temporarily freezing all user-facing operations across three contracts until three separate admin `unpause()` calls are made.

## Finding Description

**No access control on the entry point:**

`updateRSETHPrice()` at `contracts/LRTOracle.sol:87` is declared `public` with only the `whenNotPaused` modifier:

```solidity
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
```

No role check exists. Any EOA or contract can call it at any time the oracle is not already paused.

**Atomic three-contract pause in the downside-protection branch:**

Inside `_updateRsETHPrice()` at lines 270–282, when `newRsETHPrice < highestRsethPrice` and the absolute difference exceeds `pricePercentageLimit.mulWad(highestRsethPrice)`:

```solidity
if (isPriceDecreaseOffLimit) {
    if (!lrtDepositPool.paused()) lrtDepositPool.pause();
    if (!withdrawalManager.paused()) withdrawalManager.pause();
    _pause();
    return;
}
```

All three contracts are paused in one transaction and the function returns immediately, skipping the price update.

**Prerequisite — `LRTOracle` must hold `PAUSER_ROLE` on the other contracts:**

`LRTDepositPool.pause()` at line 349 and `LRTWithdrawalManager.pause()` at line 347 both require `PAUSER_ROLE`. For the downside-protection feature to function at all in any correct deployment, `LRTOracle` must already hold this role on both contracts. This is a deployment prerequisite, not an additional attacker requirement.

**`highestRsethPrice` is the all-time high, never decremented:**

At lines 293–296, `highestRsethPrice` is only ever increased:

```solidity
if (newRsETHPrice > highestRsethPrice) {
    highestRsethPrice = newRsETHPrice;
}
```

This means a historical price spike sets a permanent ceiling. Any subsequent price that is more than `pricePercentageLimit` below that spike — even if the current price is stable and slowly appreciating — satisfies the pause condition. The attacker does not need to manipulate any oracle; they only need to observe when the condition is naturally met and call the public function.

**Downstream impact — all user operations blocked:**

- `LRTWithdrawalManager.completeWithdrawal()` — `whenNotPaused` (line 183)
- `LRTWithdrawalManager.initiateWithdrawal()` — `whenNotPaused` (line 158)
- `LRTWithdrawalManager.unlockQueue()` — `whenNotPaused` (line 279)
- `LRTDepositPool.depositETH()` / `depositAsset()` — `whenNotPaused` (lines 83, 107)

**Recovery requires three separate privileged transactions:**

- `LRTOracle.unpause()` — `onlyLRTAdmin` (line 143)
- `LRTWithdrawalManager.unpause()` — `onlyLRTAdmin` (line 352)
- `LRTDepositPool.unpause()` — `onlyLRTAdmin` (line 354)

## Impact Explanation

**Medium — Temporary freezing of funds.**

Users with rsETH already locked in `initiateWithdrawal` queues cannot call `completeWithdrawal`. Operators cannot call `unlockQueue`. New deposits are blocked. All three contracts remain frozen until three separate admin unpause transactions are confirmed on-chain. Funds are not lost, but access is denied for the duration of the freeze. This matches the allowed impact: *Medium. Temporary freezing of funds.*

## Likelihood Explanation

The trigger requires only that the oracle-reported rsETH price be more than `pricePercentageLimit` below `highestRsethPrice` at the moment the attacker calls `updateRSETHPrice()`. This condition arises naturally from:

- Any LST experiencing a temporary depeg or market dip
- A historical price spike that set `highestRsethPrice` high, followed by a modest correction
- A low `pricePercentageLimit` setting (e.g., 2%, i.e., `2e16`)

No oracle manipulation, private key compromise, governance capture, or victim mistake is required. The attacker monitors oracle prices off-chain and calls the public function at the opportune moment. The attack is repeatable: after each admin unpause cycle, if the price condition persists, the attacker can call `updateRSETHPrice()` again immediately.

## Recommendation

1. **Add access control to `updateRSETHPrice()`** — restrict it to `MANAGER` or a dedicated `PRICE_UPDATER` role, consistent with the already-existing `updateRSETHPriceAsManager()` pattern.
2. **Separate the pause trigger from the price-update path** — the downside-protection pause should be callable only by a privileged role, not as a side-effect of a public price update.
3. **Use a rolling window or moving average** for `highestRsethPrice` rather than an all-time high, to reduce the attack surface from historical price spikes.
4. **Implement a single `unpauseAll()` admin function** to reduce recovery friction (note: `pauseAll()` already exists in `LRTConfig` at line 262, but no equivalent `unpauseAll()` exists).
