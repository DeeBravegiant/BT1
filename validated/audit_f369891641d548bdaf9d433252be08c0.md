Audit Report

## Title
Unprotected `sendFunds()` in `FeeReceiver` Enables MEV Reward Sniping — (File: contracts/FeeReceiver.sol)

## Summary
`FeeReceiver.sendFunds()` carries no access-control modifier, allowing any external caller to flush the entire accumulated MEV/execution-layer reward balance into `LRTDepositPool` at will. Because `LRTOracle.updateRSETHPrice()` is also permissionless, an attacker can deposit ETH at the pre-reward rsETH price, trigger the reward flush, update the oracle, and immediately sell rsETH at the inflated price on a secondary market, extracting yield earned by long-term holders.

## Finding Description
`FeeReceiver.sendFunds()` has no access-control guard:

```solidity
// contracts/FeeReceiver.sol L53-58
function sendFunds() external {          // ← no onlyRole or equivalent
    uint256 balance = address(this).balance;
    ILRTDepositPool(depositPool).receiveFromRewardReceiver{ value: balance }();
    emit MevRewardsAddedToTVL(balance);
}
``` [1](#0-0) 

The receiving function is equally open:

```solidity
// contracts/LRTDepositPool.sol L61
function receiveFromRewardReceiver() external payable { }
``` [2](#0-1) 

Once ETH lands in `LRTDepositPool`, it is counted in TVL via `getETHDistributionData()`:

```solidity
// contracts/LRTDepositPool.sol L480
ethLyingInDepositPool = address(this).balance;
``` [3](#0-2) 

Critically, `LRTOracle.rsETHPrice` is a **stored/cached** state variable updated only when `updateRSETHPrice()` is called:

```solidity
// contracts/LRTOracle.sol L28
uint256 public override rsETHPrice;
``` [4](#0-3) 

And `updateRSETHPrice()` is **permissionless**:

```solidity
// contracts/LRTOracle.sol L87-89
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
``` [5](#0-4) 

The mint calculation uses the cached price:

```solidity
// contracts/LRTDepositPool.sol L520
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [6](#0-5) 

**Full exploit path:**
1. Attacker deposits ETH via `depositETH()` — rsETH is minted at the current **cached** (pre-reward) price.
2. Attacker calls `FeeReceiver.sendFunds()` — accumulated MEV rewards move into `LRTDepositPool`, increasing `address(this).balance`.
3. Attacker calls `LRTOracle.updateRSETHPrice()` — the cached price is recomputed from the now-larger TVL and written to storage.
4. Attacker sells rsETH on a secondary market at the new, higher price, capturing the yield increment.

The `pricePercentageLimit` guard in `_updateRsETHPrice()` is a partial mitigation: if the reward flush causes a price increase exceeding the configured threshold, the public `updateRSETHPrice()` call reverts for non-managers. However, this guard is ineffective when `pricePercentageLimit == 0` (unset), when reward accumulations are small enough to stay within the limit, or when the attacker batches multiple smaller flushes. [7](#0-6) 

## Impact Explanation
**High — Theft of unclaimed yield.** MEV rewards accumulate in `FeeReceiver` over time and represent yield proportionally earned by all rsETH holders during the accrual period. By depositing at the pre-reward price and then atomically triggering the reward flush and oracle update, an attacker receives a share of those rewards without having held rsETH during the period they were earned. Existing holders are diluted: the same TVL increase is now shared with the attacker's newly minted rsETH. This maps directly to the "Theft of unclaimed yield" impact class.

## Likelihood Explanation
**Medium.** The attack requires: (1) a meaningful ETH balance in `FeeReceiver` — routine after any validator proposal or MEV event; (2) a liquid secondary market for rsETH — exists on mainnet; (3) two permissionless calls (`sendFunds()` and `updateRSETHPrice()`). No privileged access, oracle manipulation, or governance capture is needed. The attack is repeatable after every reward accrual cycle.

## Recommendation
Add an access-control guard to `sendFunds()` so only an authorized role can trigger reward distribution:

```solidity
function sendFunds() external onlyRole(LRTConstants.MANAGER) {
    uint256 balance = address(this).balance;
    ILRTDepositPool(depositPool).receiveFromRewardReceiver{ value: balance }();
    emit MevRewardsAddedToTVL(balance);
}
```

Additionally, restrict `LRTDepositPool.receiveFromRewardReceiver()` to only accept calls from the registered `FeeReceiver` contract, and consider restricting `LRTOracle.updateRSETHPrice()` to authorized callers or coupling it with the reward distribution flow.

## Proof of Concept
Assume: `FeeReceiver` holds **100 ETH** in accumulated MEV rewards. `LRTDepositPool` TVL = **1 000 ETH**, rsETH supply = **1 000 rsETH**, cached `rsETHPrice` = **1.000 ETH/rsETH**.

**Step 1** — Attacker calls `LRTDepositPool.depositETH{value: 100 ETH}(minRSETH, "")`.
Receives ≈ **100 rsETH** at cached price 1.000 (TVL not yet updated in oracle).

**Step 2** — Attacker calls `FeeReceiver.sendFunds()` (no guard, succeeds).
100 ETH moves to `LRTDepositPool`. Protocol ETH = **1 200 ETH**, rsETH supply = **1 100 rsETH**.

**Step 3** — Attacker calls `LRTOracle.updateRSETHPrice()` (permissionless).
New cached price = 1 200 / 1 100 ≈ **1.0909 ETH/rsETH**.

**Step 4** — Attacker sells **100 rsETH** on a secondary market at ≈ 1.0909 ETH each → receives ≈ **109.09 ETH**.

**Net profit**: ≈ **9.09 ETH** extracted from MEV rewards earned by the 1 000 pre-existing rsETH holders.

Foundry fork test outline:
```solidity
function testSendFundsYieldSniping() public fork {
    // fund FeeReceiver with 100 ETH simulating MEV rewards
    vm.deal(address(feeReceiver), 100 ether);
    uint256 priceBefore = lrtOracle.rsETHPrice();

    // attacker deposits at old price
    vm.prank(attacker);
    lrtDepositPool.depositETH{value: 100 ether}(0, "");
    uint256 rsethReceived = rseth.balanceOf(attacker);

    // attacker flushes rewards and updates oracle
    feeReceiver.sendFunds();
    lrtOracle.updateRSETHPrice();

    uint256 priceAfter = lrtOracle.rsETHPrice();
    assertGt(priceAfter, priceBefore);

    // attacker's rsETH is now worth more than deposited
    uint256 attackerValueETH = rsethReceived * priceAfter / 1e18;
    assertGt(attackerValueETH, 100 ether); // profit confirmed
}
```
