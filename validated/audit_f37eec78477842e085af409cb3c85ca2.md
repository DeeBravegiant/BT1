### Title
Unvalidated `builder` Field in `ProposalInit` Allows Proposer to Redirect All Block Transaction Fees and Corrupt the Block Hash — (File: `crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

### Summary

`ProposalInit` carries two distinct address fields: `proposer` (the consensus identity, checked against the committee) and `builder` (the sequencer address embedded into `BlockInfo`). The `builder` field is **never validated** by `is_proposal_init_valid()`. Because `convert_to_sn_api_block_info()` maps `init.builder` directly to `BlockInfo.sequencer_address`, a malicious-but-legitimately-selected proposer can set `builder` to any address they control. Every transaction fee in the block is then transferred to that address, and the `sequencer_address` slot in the canonical block hash is permanently corrupted.

### Finding Description

`ProposalInit` has two address fields:

```
proposer: ContractAddress  // "Address of the one who proposed the block in consensus"
builder:  ContractAddress  // "Address of the one who builds/sequences the block"
``` [1](#0-0) 

`proposer` is checked in `single_height_consensus.rs` against the committee-derived expected proposer before validation begins: [2](#0-1) 

`builder` is **never checked**. `ProposalInitValidation` — the struct that carries all locally-trusted reference values into `is_proposal_init_valid()` — contains no `builder` field: [3](#0-2) 

`is_proposal_init_valid()` validates `height`, `l1_da_mode`, `l2_gas_price_fri`, `starknet_version`, `version_constant_commitment`, `timestamp`, all four L1 gas prices, and `fee_proposal_fri` — but never `builder`: [4](#0-3) 

After `is_proposal_init_valid()` passes, `initiate_validation()` calls `convert_to_sn_api_block_info(init)`, which maps the proposer-supplied `init.builder` directly to `BlockInfo.sequencer_address`: [5](#0-4) 

This `BlockInfo` is passed to the batcher for both the build path (`propose_block`) and the validate path (`validate_block`): [6](#0-5) [7](#0-6) 

Inside the blockifier, `execute_fee_transfer` uses `block_context.block_info.sequencer_address` as the fee recipient for every transaction: [8](#0-7) 

`PartialBlockHashComponents::new()` also reads `block_info.sequencer_address` into the `sequencer` slot that is hashed into the `PartialBlockHash` / `ProposalCommitment`: [9](#0-8) 

`calculate_block_hash()` chains `sequencer` into the canonical block hash: [10](#0-9) 

### Impact Explanation

A legitimately-selected proposer sets `builder` to an attacker-controlled address in the `ProposalInit` they broadcast. Every validator accepts the proposal (all checked fields pass), executes the block with the attacker's address as `sequencer_address`, and votes to commit it. Consequences:

1. **Fee theft (Critical economic impact):** All transaction fees in the block are transferred to the attacker's address instead of the legitimate sequencer.
2. **Wrong block hash committed to chain:** The canonical block hash permanently encodes the attacker's address in the `sequencer_address` slot, corrupting the authoritative state root chain.
3. **Wrong `PartialBlockHash` / `ProposalCommitment`:** The commitment that consensus votes on and that is anchored to L1 reflects the injected address.

### Likelihood Explanation

The attacker must be the designated proposer for a consensus round. In a decentralized validator set, any validator can be selected as proposer. The attack requires no special privilege beyond being the round's proposer — a normal, expected network role. The manipulation is a single-field change in the outbound `ProposalInit` message, requiring no cryptographic forgery.

### Recommendation

Add `builder` to `ProposalInitValidation` and enforce it in `is_proposal_init_valid()`. The locally-trusted `builder` value should be derived from the node's own configuration (the same `builder_address` used in `initiate_build()`), not accepted from the network without verification:

```rust
// In ProposalInitValidation:
pub builder: ContractAddress,

// In is_proposal_init_valid():
if init_proposed.builder != proposal_init_validation.builder {
    return Err(ValidateProposalError::InvalidProposalInit(
        init_proposed.clone(),
        proposal_init_validation.clone(),
        format!("builder mismatch: expected={:?}, proposed={:?}",
            proposal_init_validation.builder, init_proposed.builder),
    ));
}
```

### Proof of Concept

1. Attacker is selected as proposer for round R at height H.
2. Attacker constructs `ProposalInit` with all valid fields (`height`, `l1_da_mode`, `l2_gas_price_fri`, gas prices, `starknet_version`, `version_constant_commitment`, `fee_proposal_fri` all within bounds) but sets `builder = attacker_wallet_address`.
3. Attacker broadcasts this `ProposalInit` followed by a valid transaction batch and `ProposalFin`.
4. Each validator calls `is_proposal_init_valid()` — all checked fields pass; `builder` is never inspected.
5. Each validator calls `convert_to_sn_api_block_info(init)` → `BlockInfo { sequencer_address: attacker_wallet_address, … }`.
6. The batcher executes all transactions; `execute_fee_transfer` sends every fee to `attacker_wallet_address`.
7. `PartialBlockHashComponents { sequencer: attacker_wallet_address, … }` is hashed into `PartialBlockHash`.
8. Consensus reaches decision; the block is committed with `attacker_wallet_address` as the canonical sequencer address in the block hash.
9. All fees from block H are in the attacker's wallet; the block hash on L1 encodes the wrong sequencer.
