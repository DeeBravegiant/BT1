Audit Report

## Title
Missing Caller Identity Check Allows Forced Fee Deduction on Any User's Claim - (File: contracts/utils/MerkleDistributor/MerkleDistributor.sol)

## Summary

`MerkleDistributor.claim()` accepts an arbitrary `account` parameter with no check that `msg.sender == account`. Any unprivileged caller can supply a valid merkle proof for any victim and trigger their claim, causing a fee of up to 10% to be permanently deducted from the victim's entitled allocation and transferred to `protocolTreasury`. The victim's claim state is then marked as fully consumed, making the lost fee irrecoverable.

## Finding Description

`MerkleDistributor.claim()` is a public, permissionless function. It validates the merkle proof against the supplied `account` and `cumulativeAmount`, but never checks that `msg.sender == account`: [1](#0-0) 

All inputs required to construct a valid call â€” `index`, `account`, `cumulativeAmount`, and `merkleProof` â€” are either derivable from the publicly distributed merkle tree or emitted in on-chain `Claimed` events for other users. After the proof passes, the contract deducts a fee and updates the victim's claim record: [2](#0-1) 

Because `userClaims[account].cumulativeAmount` is set to the full `cumulativeAmount`, the victim can never re-claim the fee portion. The sister contract `KernelMerkleDistributor._processClaim()` explicitly guards against this exact pattern: [3](#0-2) 

The guard is absent in `MerkleDistributor`.

## Impact Explanation

**High â€” Theft of unclaimed yield.**

`feeInBPS` is configurable up to `MAX_FEE_IN_BPS = 1000` (10%): [4](#0-3) 

An attacker who forces a claim for a victim causes a permanent, irrecoverable transfer of up to 10% of the victim's entitled token allocation to `protocolTreasury`. This is a concrete, non-hypothetical loss of yield that the victim was entitled to receive in full. It falls squarely within the allowed impact: **Theft of unclaimed yield (High)**.

## Likelihood Explanation

**High.** The attack requires no special privileges, no capital, and no victim interaction. The attacker needs only gas and the victim's merkle proof data, which is publicly available from the merkle tree distribution or from on-chain events. The attack is repeatable for every user in the distribution and can be executed immediately after any new merkle root is set. A user who intentionally delays claiming to wait for a fee reduction via `setFeeInBPS` is particularly vulnerable.

## Recommendation

Add a caller identity check at the top of `claim()`, mirroring the pattern already used in `KernelMerkleDistributor._processClaim()`:

```solidity
if (account != msg.sender) {
    revert Unauthorized();
}
``` [3](#0-2) 

## Proof of Concept

1. Deploy `MerkleDistributor` with `feeInBPS = 1000` (10%).
2. Owner sets a merkle root covering Alice with `cumulativeAmount = 1000e18`.
3. Attacker obtains `(index, account=Alice, cumulativeAmount=1000e18, merkleProof)` from the public merkle tree.
4. Attacker calls `MerkleDistributor.claim(index, Alice, 1000e18, merkleProof)` from any EOA.
5. Contract computes `fee = 100e18`, transfers `900e18` to Alice and `100e18` to `protocolTreasury`.
6. `userClaims[Alice].cumulativeAmount` is set to `1000e18` â€” Alice can never reclaim the `100e18`.
7. Alice permanently loses 10% of her entitled allocation without consent.

**Foundry test sketch:**
```solidity
function test_forcedClaimFeeDeduction() public {
    // Setup: deploy distributor with feeInBPS=1000, set merkle root for Alice
    vm.prank(attacker);
    distributor.claim(aliceIndex, alice, aliceCumulativeAmount, aliceProof);
    // Assert: alice received 90%, treasury received 10%, alice cannot claim again
    assertEq(token.balanceOf(alice), aliceCumulativeAmount * 9000 / 10_000);
    assertEq(token.balanceOf(treasury), aliceCumulativeAmount * 1000 / 10_000);
    vm.expectRevert(IMerkleDistributor.AlreadyClaimed.selector);
    vm.prank(alice);
    distributor.claim(aliceIndex, alice, aliceCumulativeAmount, aliceProof);
}
```
