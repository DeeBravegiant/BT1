### Title
Gateway Admits Invoke Transactions with Invalid Signatures via Overly Broad `skip_stateful_validations` Check — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

`skip_stateful_validations` in the Apollo gateway is designed to skip `__validate__` (signature verification) for an invoke transaction with nonce=1 when a `deploy_account` is pending for the same account. However, the guard condition `account_tx_in_pool_or_recent_block` returns `true` for **any** account that has **any** transaction in the pool — not only accounts with a pending `deploy_account`. This allows an attacker to submit an invoke transaction with an arbitrary (wrong) signature for any account that (a) exists on-chain with nonce=0 and (b) already has a nonce=0 transaction in the mempool, bypassing the `__validate__` entry-point check at admission time.

---

### Finding Description

`skip_stateful_validations` decides whether to skip the blockifier's `__validate__` call during gateway admission:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs:437-456
if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
    return mempool_client
        .account_tx_in_pool_or_recent_block(tx.sender_address())
        .await
        ...
}
``` [1](#0-0) 

When this returns `true`, `run_validate_entry_point` sets `validate: !skip_validate = false`, so the account's `__validate__` entry point is never called:

```rust
// line 311-312
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
``` [2](#0-1) 

And in `StatefulValidator::perform_validations`, when `validate` is `false`, the function returns `Ok(())` immediately without calling `__validate__`:

```rust
// crates/blockifier/src/blockifier/stateful_validator.rs:79-81
if !tx.execution_flags.validate {
    return Ok(());
}
``` [3](#0-2) 

The guard `account_tx_in_pool_or_recent_block` is implemented as:

```rust
// crates/apollo_mempool/src/mempool.rs:697-700
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [4](#0-3) 

`MempoolState::contains_account` returns `true` if the address appears in `staged` or `committed` maps — i.e., for **any** account that has **any** transaction in the pool, regardless of transaction type:

```rust
// crates/apollo_mempool/src/mempool.rs:115-117
fn contains_account(&self, address: ContractAddress) -> bool {
    self.staged.contains_key(&address) || self.committed.contains_key(&address)
}
``` [5](#0-4) 

The code comment claims this is sufficient because the account "either has a deploy_account transaction or transactions with future nonces that passed validations." The second case is the flaw: a nonce=0 **invoke** transaction that passed validation means the account **already exists in state** — `__validate__` is callable and must not be skipped for the subsequent nonce=1 transaction.

---

### Impact Explanation

An attacker who observes that account `X` (on-chain nonce=0) has a nonce=0 invoke transaction in the mempool can submit a nonce=1 invoke transaction for `X` with an **arbitrary/wrong signature**. The gateway will:

1. Read `account_nonce = 0` from state.
2. See `tx.nonce() == 1` and `account_nonce == 0`.
3. Call `account_tx_in_pool_or_recent_block(X)` → `true` (because the nonce=0 tx is in the pool).
4. Set `skip_validate = true` → `validate: false`.
5. Admit the transaction to the mempool **without calling `__validate__`** (no signature check).

The admitted transaction carries an invalid signature. At execution time, `new_for_sequencing` sets `validate: true`, so the transaction will fail during block execution. However:

- **Mempool admission of invalid transactions**: The gateway accepts a transaction that violates the account's signature invariant. This matches the "High" impact: *Mempool/gateway/RPC admission accepts invalid transactions before sequencing.*
- **Transaction displacement via fee escalation**: If the attacker submits the invalid nonce=1 tx with a higher fee than the legitimate user's nonce=1 tx, the mempool may replace the legitimate tx. The attacker's tx then fails during execution, and the legitimate user's tx is permanently lost from the mempool.
- **Mempool spam/DoS**: An attacker can flood the mempool with signature-invalid transactions for any account with nonce=0 and an existing pool entry, consuming mempool capacity and block space.

---

### Likelihood Explanation

The preconditions are common in normal operation:
- Any account that was just deployed (on-chain nonce=0) and has submitted its first transaction (nonce=0 in pool) is vulnerable.
- The attacker only needs to observe the mempool (public information) to identify targets.
- No privileged access is required.

---

### Recommendation

Replace the overly broad `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a **`deploy_account` transaction** is pending for the address. The mempool should expose a dedicated query such as `deploy_account_in_pool(address) -> bool` that only returns `true` when the pending transaction is of type `DeployAccount`. This preserves the intended UX feature (deploy + invoke in one batch) while closing the bypass for accounts that already exist in state.

Additionally, the unused `max_nonce_for_validation_skip` field in `StatefulTransactionValidatorConfig` should either be wired into `skip_stateful_validations` (as it is in the legacy `PyValidator`) or removed to avoid confusion. [6](#0-5) [7](#0-6) 

---

### Proof of Concept

**Setup**:
- Account `A` is deployed on-chain; its on-chain nonce is `0`.
- User submits a valid invoke tx `T0` (nonce=0, correct signature) for `A`. `T0` is admitted to the mempool normally (signature verified).

**Attack**:
1. Attacker observes `T0` in the mempool → `account_tx_in_pool_or_recent_block(A)` will return `true`.
2. Attacker crafts invoke tx `T1` (nonce=1, **arbitrary/wrong signature**, higher tip than the user's legitimate nonce=1 tx).
3. Attacker submits `T1` to the gateway.
4. Gateway calls `skip_stateful_validations`:
   - `tx.nonce() == 1` ✓
   - `account_nonce == 0` ✓ (on-chain nonce is 0)
   - `account_tx_in_pool_or_recent_block(A)` → `true` ✓ (because `T0` is in pool)
   - Returns `true` → `skip_validate = true`
5. `run_validate_entry_point` sets `validate: false` → `__validate__` is **not called**.
6. `T1` is admitted to the mempool despite having an invalid signature.
7. If the user had also submitted a legitimate nonce=1 tx with lower fee, the mempool replaces it with `T1` (fee escalation).
8. During block execution, `T1` fails (wrong signature → revert). The user's legitimate nonce=1 tx is gone.

**Corrupted invariant**: The gateway's admission invariant — *every admitted invoke transaction has passed account signature verification* — is violated for accounts with on-chain nonce=0 that have any transaction in the mempool. [8](#0-7) [9](#0-8)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
