### Title
`Map.withoutKeys` in `poolReapTransition` Unconditionally Deletes Shared VRF Key Entries, Silently Dropping Surviving Pools from `psVRFKeyHashes` — (`eras/shelley/impl/src/Cardano/Ledger/Shelley/Rules/PoolReap.hs`)

---

### Summary

A real, locally-reproducible invariant-violation exists in `poolReapTransition`. When two pools share a VRF key V (permitted pre-v11), and one of them re-registers with a fresh VRF key post-v11, the POOLREAP epoch-boundary transition uses `Map.withoutKeys` to remove the "dangling" old VRF entry. `Map.withoutKeys` deletes the entire map entry regardless of its reference count, so a count of 2 drops to 0 instead of 1. The surviving pool's VRF key disappears from `psVRFKeyHashes`, and a new pool can immediately register with that key, giving two active pools the same VRF key and breaking the post-v11 uniqueness invariant.

---

### Finding Description

**`populateVRFKeyHashes` at v11 hardfork** counts every pool's VRF key: [1](#0-0) 

If pools A and B both carry VRF key V, the result is `psVRFKeyHashes = {V→2}`.

**Pool A re-registers with fresh VRF V' (post-v11).** The POOL rule inserts `V'→1` but does **not** decrement V, because A's current `psStakePools` entry still holds V until the next epoch boundary: [2](#0-1) 

State after re-registration: `psVRFKeyHashes = {V→2, V'→1}`, `psFutureStakePoolParams = {A→{vrf:V'}}`.

**POOLREAP at the next epoch boundary** computes `danglingVRFKeyHashes`: [3](#0-2) 

Because A's current VRF (V) ≠ A's future VRF (V'), `danglingVRFKeyHashes = {V}`.

Then the update is applied: [4](#0-3) 

`Map.withoutKeys psVRFKeyHashes {V}` deletes the entire entry for V, yielding `{V'→1}`. Pool B still has VRF V in `psStakePools`, but V is now absent from `psVRFKeyHashes`.

**Contrast with the correct helper** used for retired pools, which decrements by 1 and only removes the entry when the count reaches 0: [5](#0-4) 

The dangling-VRF path should use the same `removeVRFKeyHashOccurrence` logic, not `Map.withoutKeys`.

**New pool C registers with V.** The duplicate-VRF guard in the POOL rule checks `Map.notMember sppVrf psVRFKeyHashes`: [6](#0-5) 

Since V was deleted, `Map.notMember V {V'→1}` is `True`, the check passes, and C registers with V. Now both B and C are active with VRF key V, violating the post-v11 uniqueness invariant.

---

### Impact Explanation

The v11 hardfork (`hardforkConwayDisallowDuplicatedVRFKeys`, active when `pvMajor pv > 10`) was explicitly designed to enforce one-VRF-per-pool: [7](#0-6) 

After the bug fires, two active pools share the same VRF key. Because VRF keys drive slot-leader election and block-header proofs, both pools produce identical VRF outputs for every slot. They can both legitimately claim leadership for the same slots and produce competing blocks. Honest nodes that receive different competing blocks from B and C will diverge on chain selection, matching the **High — Deterministic disagreement between honest nodes from ledger rule evaluation** impact category.

---

### Likelihood Explanation

- **Precondition reachability**: Registering two pools with the same VRF key before v11 was explicitly permitted and costs only two pool deposits. A malicious operator can set this up deliberately.
- **Trigger**: A single ordinary `RegPool` certificate (pool re-registration with a fresh VRF) after v11 is all that is needed. No privileged access, no governance majority, no key leakage.
- **Automation**: The POOLREAP transition fires automatically at every epoch boundary; no further attacker action is required after the re-registration.
- **Detection difficulty**: The broken `psVRFKeyHashes` state is silent — no predicate failure is raised, and no existing post-condition assertion in POOLREAP checks VRF-key consistency.

---

### Recommendation

Replace `Map.withoutKeys` for the dangling-VRF case with the same reference-count decrement used for retired pools:

```haskell
-- current (wrong): deletes entire entry regardless of count
. (`Map.withoutKeys` danglingVRFKeyHashes)

-- correct: decrement by 1, remove only when count reaches 0
. removeVRFKeyHashOccurrences (Set.toList danglingVRFKeyHashes)
```

Additionally, add a post-condition assertion to POOLREAP verifying that every VRF key present in `psStakePools` after the transition is tracked in `psVRFKeyHashes` with a count ≥ 1.

### Proof of Concept

```
1. (pre-v11) Submit RegPool cert for pool A with VRF key V.
2. (pre-v11) Submit RegPool cert for pool B with VRF key V.
3. Enact v11 hardfork.
   → populateVRFKeyHashes: psVRFKeyHashes = {V→2}
4. Submit RegPool cert for pool A with fresh VRF key V'.
   → psVRFKeyHashes = {V→2, V'→1}
   → psFutureStakePoolParams = {A→{vrf:V'}}
5. passEpoch (POOLREAP fires).
   → danglingVRFKeyHashes = {V}
   → Map.withoutKeys removes V entirely: psVRFKeyHashes = {V'→1}
   → psStakePools = {A→{vrf:V'}, B→{vrf:V}}
6. Assert: Map.notMember V psVRFKeyHashes  -- True (BUG)
7. Submit RegPool cert for new pool C with VRF key V.
   → Succeeds (duplicate check passes)
   → psStakePools = {A→{vrf:V'}, B→{vrf:V}, C→{vrf:V}}
8. Assert: both B and C are active with VRF V  -- invariant violated
```

populateVRFKeyHashes :: PState era -> PState era
populateVRFKeyHashes pState =
  pState
    & psVRFKeyHashesL
      %~ accumulateVRFKeyHashes (pState ^. psStakePoolsL) (^. spsVrfL)
        . accumulateVRFKeyHashes (pState ^. psFutureStakePoolParamsL) (^. sppVrfL)
  where
    accumulateVRFKeyHashes ::
      Map (KeyHash StakePool) a ->
      (a -> VRFVerKeyHash StakePoolVRF) ->
      Map (VRFVerKeyHash StakePoolVRF) (NonZero Word64) ->
      Map (VRFVerKeyHash StakePoolVRF) (NonZero Word64)
    accumulateVRFKeyHashes spMap getVrf acc =
      Map.foldr' (addVRFKeyHashOccurrence . getVrf) acc spMap
    addVRFKeyHashOccurrence vrfKeyHash =
      Map.insertWith combine vrfKeyHash (knownNonZeroBounded @1)
      where
        -- Saturates at maxBound: if (+1) would overflow to 0, keep existing value
        combine _ oldVal = fromMaybe oldVal $ mapNonZero (+ 1) oldVal
