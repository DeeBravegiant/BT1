Audit Report

## Title
Unbounded Gas Loop in `_unlockWithdrawalRequests` Enables Temporary Freezing of Legitimate Withdrawals via Spam - (File: contracts/LRTWithdrawalManager.sol)

## Summary
Any rsETH holder can call `initiateWithdrawal()` with 1 wei of rsETH (since `minRsEthAmountToWithdraw[asset]` defaults to 0) thousands of times, flooding the global FIFO queue. The `_unlockWithdrawalRequests` while-loop iterates every queued entry in O(N) fashion with no gas cap. Legitimate users whose nonces fall behind the attacker's entries cannot call `completeWithdrawal()` until `nextLockedNonce` advances past all attacker entries, temporarily freezing their funds.

## Finding Description
`minRsEthAmountToWithdraw` is declared as a plain mapping with no initialization in `initialize()`, so it defaults to 0 for every asset. [1](#0-0) 

The only guard in `initiateWithdrawal()` rejects `rsETHUnstaked == 0` or amounts below the minimum; when the minimum is 0, any amount ≥ 1 wei passes: [2](#0-1) 

Each call pushes a new entry into the global sequential queue and increments `nextUnusedNonce[asset]`: [3](#0-2) 

`_unlockWithdrawalRequests` iterates from `nextLockedNonce` up to `firstExcludedIndex` (capped at `nextUnusedNonce[asset]`): [4](#0-3) 

The two early-exit conditions are: (1) withdrawal delay not yet passed, and (2) available assets exhausted. Because each spam entry commits only ~1 wei of `expectedAssetAmount`, condition (2) never triggers early — the loop must visit every attacker entry before reaching legitimate users' requests. [5](#0-4) 

`completeWithdrawal()` reverts with `WithdrawalLocked` for any user whose nonce has not yet been passed by `nextLockedNonce`: [6](#0-5) 

Until the operator processes all attacker entries (requiring many batched `unlockQueue` calls), legitimate users behind those entries cannot complete their withdrawals.

## Impact Explanation
**Temporary freezing of funds (Medium).** Legitimate users' rsETH is locked in the contract and their `completeWithdrawal()` calls revert until `nextLockedNonce` advances past all attacker-inserted entries. With tens of thousands of spam entries, this delay is extended by the time required for the operator to drain the queue in small batches. This matches the allowed impact "Temporary freezing of funds."

**Unbounded gas consumption (Medium).** If the operator passes a large `firstExcludedIndex`, the loop exhausts the block gas limit and reverts, blocking the unlock path entirely until the operator discovers the need for smaller batches.

## Likelihood Explanation
- `initiateWithdrawal()` is fully permissionless for any rsETH holder.
- `minRsEthAmountToWithdraw[asset]` defaults to 0 on all assets unless explicitly set by an admin.
- The attacker recovers their rsETH value as the underlying asset after the 8-day delay; the only net cost is gas fees.
- On L2 deployments or during low-fee periods, queuing tens of thousands of entries is economically feasible.
- The attack is repeatable across every supported asset independently.

## Recommendation
1. **Enforce a non-zero minimum withdrawal amount** for every supported asset via `setMinRsEthAmountToWithdraw` at initialization, making spam economically prohibitive.
2. **Cap the number of pending requests per user** or globally per asset to bound queue growth.
3. **Document and enforce a safe upper bound for `firstExcludedIndex`** in `unlockQueue()` so operators always process in bounded batches, preventing accidental out-of-gas reverts.
4. Consider a per-user nonce scheme or priority queue so one user's spam cannot block others in the global FIFO queue.

## Proof of Concept
1. Attacker deposits ETH into `LRTDepositPool.depositETH()` to receive rsETH.
2. Attacker calls `LRTWithdrawalManager.initiateWithdrawal(ETH_TOKEN, 1, "")` 50,000 times (valid because `minRsEthAmountToWithdraw[ETH_TOKEN] == 0`). Each call increments `nextUnusedNonce[ETH_TOKEN]`.
3. Legitimate users submit their own withdrawal requests (nonces 50,000+).
4. After the 8-day delay, the operator calls `unlockQueue(ETH_TOKEN, type(uint256).max, ...)`.
5. `firstExcludedIndex` is capped at `nextUnusedNonce[ETH_TOKEN]` (~50,000+); the while-loop attempts to iterate all 50,000 attacker entries, each consuming ~60k gas (3 SSTOREs + 1 SLOAD). At ~30M block gas limit, the transaction reverts after ~500 iterations.
6. Legitimate users' `completeWithdrawal()` calls revert with `WithdrawalLocked` because `nextLockedNonce[ETH_TOKEN]` has not advanced past their nonces.
7. The operator must issue ~100 batched `unlockQueue` calls to drain the attacker's entries before legitimate users can withdraw.
