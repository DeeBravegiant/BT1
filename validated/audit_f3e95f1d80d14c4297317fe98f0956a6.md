Audit Report

## Title
Auto-Pause Locks Initiated Withdrawals With No Cancel Path, Enabling Forced Settlement at Depressed Price — (File: `contracts/LRTWithdrawalManager.sol`, `contracts/LRTOracle.sol`)

## Summary

When `LRTOracle._updateRsETHPrice()` detects a price drop beyond `pricePercentageLimit`, it permissionlessly pauses `LRTWithdrawalManager`. Users who have already called `initiateWithdrawal` — transferring their rsETH into the contract — have no `cancelWithdrawal` path and cannot recover their tokens for the duration of the pause. The pause has no on-chain time limit. Upon unpause, if the oracle price has been updated to a lower value before `unlockQueue` is called, `_calculatePayoutAmount` settles users at `min(expectedAssetAmount, currentReturn)`, burning their full rsETH for fewer underlying assets than recorded at initiation.

## Finding Description

**Step 1 — User initiates withdrawal, rsETH is transferred to the contract.**

`initiateWithdrawal` pulls the user's rsETH and records `expectedAssetAmount` at the current oracle price:

```solidity
// LRTWithdrawalManager.sol L166-175
IERC20(lrtConfig.rsETH()).safeTransferFrom(msg.sender, address(this), rsETHUnstaked);
uint256 expectedAssetAmount = getExpectedAssetAmount(asset, rsETHUnstaked);
assetsCommitted[asset] += expectedAssetAmount;
_addUserWithdrawalRequest(asset, rsETHUnstaked, expectedAssetAmount);
```

**Step 2 — Any caller triggers the auto-pause via the public `updateRSETHPrice()`.**

`updateRSETHPrice()` is `public` with no access control beyond `whenNotPaused`:

```solidity
// LRTOracle.sol L87-89
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
```

Inside `_updateRsETHPrice()`, if the computed price falls more than `pricePercentageLimit` below `highestRsethPrice`, the function pauses both `LRTDepositPool` and `LRTWithdrawalManager` and returns **without updating `rsETHPrice`**:

```solidity
// LRTOracle.sol L277-282
if (isPriceDecreaseOffLimit) {
    if (!lrtDepositPool.paused()) lrtDepositPool.pause();
    if (!withdrawalManager.paused()) withdrawalManager.pause();
    _pause();
    return;
}
```

**Step 3 — All user-facing withdrawal functions are blocked; no cancel exists.**

Both `completeWithdrawal` and `unlockQueue` carry `whenNotPaused`:

```solidity
// LRTWithdrawalManager.sol L183
function completeWithdrawal(address asset, string calldata referralId) external nonReentrant whenNotPaused { ... }

// LRTWithdrawalManager.sol L279
whenNotPaused
```

There is no `cancelWithdrawal` function anywhere in `LRTWithdrawalManager`. The user's rsETH is irrecoverably locked for the duration of the pause, which has no on-chain expiry.

**Step 4 — Upon unpause, if `rsETHPrice` has been updated to a lower value, `unlockQueue` settles at the depressed price.**

Because `_updateRsETHPrice()` returns early without updating `rsETHPrice` when the pause triggers, the stored price remains at the pre-drop value immediately after the pause. However, once the admin unpauses `LRTOracle` and the price partially recovers (enough to clear the threshold), a subsequent call to `updateRSETHPrice()` will update `rsETHPrice` to the new (still lower than initiation) value. An operator then calling `unlockQueue` reads this updated price:

```solidity
// LRTWithdrawalManager.sol L847
rsETHPrice: lrtOracle.rsETHPrice(),
```

`_calculatePayoutAmount` then computes:

```solidity
// LRTWithdrawalManager.sol L833-834
uint256 currentReturn = (request.rsETHUnstaked * rsETHPrice) / assetPrice;
return (request.expectedAssetAmount < currentReturn) ? request.expectedAssetAmount : currentReturn;
```

If `currentReturn < expectedAssetAmount`, the user's `expectedAssetAmount` is overwritten to the lower value while their full rsETH is burned:

```solidity
// LRTWithdrawalManager.sol L802-805
assetsCommitted[asset] -= request.expectedAssetAmount;
request.expectedAssetAmount = payoutAmount;   // overwritten to lower value
rsETHAmountToBurn += request.rsETHUnstaked;   // full rsETH burned regardless
```

**Why existing guards are insufficient:**

- `unlockQueue` has price-range guards (`minimumRsEthPrice` / `maximumRsEthPrice`), but these are set by the operator calling the function — the same party executing the settlement. They do not protect users.
- The `withdrawalDelayBlocks` check only ensures a minimum delay has passed; it does not give users any ability to exit.
- There is no on-chain mechanism to bound the pause duration or force a cancel path.

## Impact Explanation

**Medium — Temporary freezing of funds:** Users who have called `initiateWithdrawal` have their rsETH locked in `LRTWithdrawalManager` with no cancel path for the entire duration of the pause. The pause is triggered permissionlessly and has no on-chain time limit, persisting until an admin manually calls `unpause()`. This directly matches the "Temporary freezing of funds" impact class.

**Low — Contract fails to deliver promised returns (secondary):** If the oracle price is updated to a lower value before `unlockQueue` is called post-unpause, users receive fewer underlying assets than `expectedAssetAmount` while burning their full rsETH. This is a direct, irreversible value shortfall caused by the forced settlement window created by the pause.

## Likelihood Explanation

The auto-pause trigger is permissionless — any address can call `updateRSETHPrice()` during a market downturn. EigenLayer slashing events, LST de-pegs, or coordinated selling are realistic triggers. The pause duration is unbounded (admin-controlled unpause only). During volatile markets — exactly when users most want to exit — the mechanism locks them in. The combination of a public trigger function, no cancel path, and indefinite pause duration makes this a realistic and repeatable scenario.

## Recommendation

1. **Add a `cancelWithdrawal` function** that allows users to reclaim their rsETH while a request is still in the locked (pre-`unlockQueue`) state. This eliminates the fund-freeze impact entirely.
2. **Introduce a post-unpause grace period** before `unlockQueue` can be called, giving users time to cancel pending requests if they no longer wish to withdraw at the current price.
3. **Alternatively, snapshot the rsETH price at `initiateWithdrawal` time** and guarantee payout is calculated at that price (not the price at `unlockQueue` time), so the pause window cannot worsen the user's settlement rate.

## Proof of Concept

1. rsETH price is 1.05 ETH/rsETH. Alice calls `initiateWithdrawal(ETH, 100e18)`. Her 100 rsETH is transferred to `LRTWithdrawalManager`; `expectedAssetAmount` is recorded as 105 ETH.
2. A market event causes the computed rsETH price to drop to 0.90 ETH/rsETH (> `pricePercentageLimit` below `highestRsethPrice`). Any caller invokes `updateRSETHPrice()`. `LRTOracle._updateRsETHPrice()` fires `withdrawalManager.pause()` and returns without updating `rsETHPrice` (stored price remains 1.05).
3. Alice attempts `completeWithdrawal` → reverts (`whenNotPaused`). She attempts to cancel → no such function exists. Her 100 rsETH is frozen.
4. After an extended pause, the admin calls `unpause()` on both `LRTOracle` and `LRTWithdrawalManager`. The market has partially recovered to 0.97 ETH/rsETH (above the threshold). A caller invokes `updateRSETHPrice()`, which now updates `rsETHPrice` to 0.97.
5. An operator immediately calls `unlockQueue`. `_calculatePayoutAmount` computes `currentReturn = 100e18 * 0.97e18 / 1e18 = 97 ETH`. Since `97 < 105`, Alice's `expectedAssetAmount` is overwritten to 97 ETH and all 100 rsETH is burned.
6. Alice calls `completeWithdrawal` and receives 97 ETH — 8 ETH less than she was promised, with no ability to have cancelled during the pause.
