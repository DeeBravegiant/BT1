Audit Report

## Title
`getMinAmount` Integer Truncation Causes Unexpected Revert for Small Non-Zero Amounts — (`contracts/bridges/TACWETHBridge.sol`)

## Summary

`getMinAmount` computes `minAmount` via integer division and reverts with `InvalidMinAmount` if the result truncates to zero. Because `bridgeTokenToL1` and `getNativeFee` only guard against `amount == 0`, any caller supplying a small but non-zero amount (e.g., 1 wei with any non-zero `slippageTolerance`) receives an unexpected revert. No funds are lost, but the contract fails to deliver its promised bridging service for these amounts.

## Finding Description

`getMinAmount` at lines 174–178 computes:

```solidity
uint256 minAmount = amount * (BASIS_POINTS_DIVISOR - slippageTolerance) / BASIS_POINTS_DIVISOR;
if (minAmount == 0) {
    revert InvalidMinAmount();
}
``` [1](#0-0) 

For any non-zero `slippageTolerance`, an `amount` small enough that `amount * (BASIS_POINTS_DIVISOR - slippageTolerance) < BASIS_POINTS_DIVISOR` truncates to zero and reverts. With `slippageTolerance = 500` (5%), `amount = 1` yields `1 * 9500 / 10000 = 0` → revert.

`getNativeFee` calls `getMinAmount` at line 153, propagating the revert to the view function: [2](#0-1) 

`bridgeTokenToL1` calls `getNativeFee` at line 108, propagating the revert to the state-changing entry point: [3](#0-2) 

The only explicit guard in `bridgeTokenToL1` and `getNativeFee` is `amount == 0` (lines 103–105 and 145–147), implying any positive amount is accepted. The `InvalidMinAmount` revert is an undocumented secondary rejection that surprises callers and integrators. [4](#0-3) 

The `slippageTolerance = 10_000` scenario (which bricks all amounts) requires admin action and is excluded per SECURITY.md. The small-amount truncation case requires no privileged access and is reachable by any external caller.

## Impact Explanation

**Low — Contract fails to deliver promised returns, but doesn't lose value.** Any user calling `bridgeTokenToL1` with a small non-zero amount (e.g., 1 wei) receives an `InvalidMinAmount` revert instead of a successful bridge. No tokens are transferred or locked; the call simply fails. UI/integrator layers calling `getNativeFee` as a pre-check also receive an unexpected revert rather than a fee quote, silently breaking quoting flows.

## Likelihood Explanation

With any non-zero `slippageTolerance` (the normal operational state), `amount = 1 wei` always triggers the truncation revert. This is reachable by any unprivileged external caller with no special preconditions. The condition is deterministic and repeatable.

## Recommendation

Add a minimum-amount pre-check before calling `getMinAmount`, or floor the computed value to 1 to avoid spurious reverts for dust amounts. Additionally, change the `slippageTolerance` guard from `>` to `>=` to reject the degenerate 100% case:

```solidity
// In constructor and setSlippageTolerance:
if (newSlippageTolerance >= BASIS_POINTS_DIVISOR) revert InvalidSlippageTolerance();

// In getMinAmount:
uint256 minAmount = amount * (BASIS_POINTS_DIVISOR - slippageTolerance) / BASIS_POINTS_DIVISOR;
if (minAmount == 0) minAmount = 1; // floor to prevent spurious revert on dust amounts
```

## Proof of Concept

```solidity
// Foundry test — no fork required
function testGetMinAmountRevertsForSmallAmount() public {
    // Deploy with slippageTolerance = 500 (5%)
    // amount = 1: 1 * 9500 / 10000 = 0 → InvalidMinAmount
    vm.expectRevert(TACWETHBridge.InvalidMinAmount.selector);
    bridge.getMinAmount(1);
}

function testGetNativeFeeRevertsForSmallAmount() public {
    // getNativeFee calls getMinAmount internally → same revert
    vm.expectRevert(TACWETHBridge.InvalidMinAmount.selector);
    bridge.getNativeFee(1, address(0xBEEF));
}

function testBridgeTokenToL1RevertsForSmallAmount() public {
    // bridgeTokenToL1 calls getNativeFee → getMinAmount → revert
    vm.expectRevert(TACWETHBridge.InvalidMinAmount.selector);
    bridge.bridgeTokenToL1{value: 0}(address(0xBEEF), 1);
}
```
