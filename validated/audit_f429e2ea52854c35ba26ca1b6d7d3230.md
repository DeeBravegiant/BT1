### Title
`DelegateAction` Expiry Boundary Condition Allows Execution at `max_block_height` - (File: `runtime/runtime/src/actions.rs`)

### Summary

The `DelegateAction` (meta-transaction) expiry check uses a strict `>` comparison instead of `>=`, allowing a relayer to execute a signed meta-transaction at exactly `block_height == max_block_height`. The protocol specification and documentation explicitly state the action must expire when the current block height is **equal to or greater than** `max_block_height`, but the implementation only rejects it when strictly greater.

### Finding Description

`DelegateAction` carries a `max_block_height` field described as:

> "The maximal height of the block in the blockchain **below which** the given DelegateAction is valid."

The protocol documentation at `docs/RuntimeSpec/Actions.md` line 402 states:

> "If the current block is **equal or greater than** `max_block_height`" → `DelegateActionExpired` [1](#0-0) 

The `DelegateAction` struct confirms the semantic: the action is valid only for blocks **below** `max_block_height`. [2](#0-1) 

However, the test in `runtime/runtime/src/actions.rs` reveals the implementation's actual boundary behavior. The test at line 1358–1373 only verifies that `max_block_height + 1` triggers `DelegateActionExpired`: [3](#0-2) 

Meanwhile, the test at line 1384–1386 uses `max_block_height` as the current block height and expects a **different** error (`DelegateActionSenderDoesNotMatchTxReceiver`), proving the expiry check passes (does not fire) when `block_height == max_block_height`: [4](#0-3) 

This confirms the implementation uses `apply_state.block_height > delegate_action.max_block_height` (strict) rather than `>=` (non-strict) in `apply_delegate_action` inside `runtime/runtime/src/actions.rs`. [5](#0-4) 

### Impact Explanation

A relayer can execute a `DelegateAction` at exactly `block_height == max_block_height`, one block beyond the signer's intended authorization window. The signer chose `max_block_height = H` to mean "valid only for blocks strictly below H," but the runtime accepts and executes the action at height H. This results in incorrect state changes — balance transfers, contract calls, key additions/deletions — that the signer did not authorize for that block height. The corrupted protocol value is the state root produced by the incorrectly accepted receipt.

### Likelihood Explanation

A relayer (unprivileged user) controls when to submit the outer transaction carrying the `DelegateAction`. By timing submission to land at exactly `max_block_height`, the relayer can exploit this off-by-one. This is realistic: a relayer who holds a signed `DelegateAction` and observes the chain can deliberately delay submission to the boundary block. No special privileges are required.

### Recommendation

Change the expiry comparison in `apply_delegate_action` from strict greater-than to greater-than-or-equal:

```rust
// Current (incorrect):
if apply_state.block_height > delegate_action.max_block_height {
    // DelegateActionExpired
}

// Correct:
if apply_state.block_height >= delegate_action.max_block_height {
    // DelegateActionExpired
}
```

Add a dedicated test asserting that `block_height == max_block_height` returns `DelegateActionExpired`.

### Proof of Concept

1. Signer creates and signs a `DelegateAction` with `max_block_height = 1000`, intending it to be valid only for blocks 0–999.
2. Relayer holds the signed action and waits until the chain reaches height 1000.
3. Relayer submits the outer transaction at height 1000.
4. `apply_delegate_action` checks `1000 > 1000` → `false` → expiry check passes.
5. The action executes at height 1000, which the signer intended to be the expiry boundary.
6. State changes (e.g., token transfer, contract call) are applied at a block height the signer did not authorize.

The existing test `test_delegate_action_validate_sender_account` (line 1384) inadvertently demonstrates this: it sets `block_height = max_block_height` and the code proceeds past the expiry check to a later validation, confirming the boundary is not enforced. [6](#0-5)

///
/// - Checks whether the access key is present fo given public_key and sender_id.
/// - Validates nonce and updates it if it's ok.
/// - Validates access key permissions.
fn validate_delegate_action_key(
    state_update: &mut TrieUpdate,
    apply_state: &ApplyState,
    delegate_action: VersionedDelegateActionRef<'_>,
    result: &mut ActionResult,
) -> Result<(), RuntimeError> {
    let sender_id = delegate_action.sender_id();
    let public_key = delegate_action.public_key();
    // 'sender_id' account existence must be checked by a caller
    let mut access_key = match get_access_key(state_update, sender_id, public_key)? {
        Some(access_key) => access_key,
        None => {
            result.result = Err(ActionErrorKind::DelegateActionAccessKeyError(
                InvalidAccessKeyError::AccessKeyNotFound {
                    account_id: sender_id.clone(),
                    public_key: public_key.clone().into(),
                },
            )
            .into());
            return Ok(());
        }
    };
