Audit Report

## Title
Uninitialized `tokenFeeBps` Allows Zero-Fee Token Deposits While ETH Deposits Pay Full Fee - (File: contracts/pools/RSETHPool.sol)

## Summary
`RSETHPool.sol` maintains two separate fee rates: `feeBps` for ETH deposits and `tokenFeeBps[token]` for token deposits. The `addSupportedToken` function never initializes `tokenFeeBps[token]`, so it defaults to `0` in Solidity. Any depositor can bypass the ETH deposit fee entirely by converting ETH to a supported token externally and depositing via the token path, paying zero fee while an equivalent ETH depositor pays `feeBps`.

## Finding Description
Two distinct fee storage variables exist:

- `feeBps` (line 43): applied to all ETH deposits
- `tokenFeeBps` (line 88): per-token mapping applied to token deposits

ETH deposit fee (line 312):
```solidity
fee = amount * feeBps / 10_000;
```

Token deposit fee (lines 335–336):
```solidity
uint256 feeBpsForToken = tokenFeeBps[token];
fee = amount * feeBpsForToken / 10_000;
```

The `addSupportedToken` function (lines 637–656) sets `supportedTokenOracle[token]`, `tokenBridge[token]`, and pushes to `supportedTokenList`, but **never sets `tokenFeeBps[token]`**. Solidity's default for unset `uint256` mapping entries is `0`, so every newly added token has a zero fee rate until the admin explicitly calls `setTokenFeeBps` (lines 583–594).

During this window — and permanently if the admin never calls `setTokenFeeBps` — the two deposit paths for economically equivalent value diverge:

| Path | Fee paid |
|---|---|
| ETH deposit of 1 ETH | `1e18 * feeBps / 10_000` |
| Token deposit of ~1 ETH worth of stETH | `0` |

Both paths produce the same wrsETH output (adjusted by oracle rate), but the token path pays zero fee. The `tokenFeeBps` mapping is public, so any depositor can verify the zero fee on-chain before acting.

## Impact Explanation
The protocol fails to collect fees consistently across deposit paths. ETH depositors are overcharged relative to token depositors for economically equivalent deposits. Protocol fee revenue is reduced or eliminated for any token whose `tokenFeeBps` has not been explicitly set. This matches **Low — Contract fails to deliver promised returns, but doesn't lose value**: no user funds are stolen, but the protocol does not collect the fees it is designed to charge.

## Likelihood Explanation
Any unprivileged depositor can observe that a supported token has `tokenFeeBps = 0` (public mapping) and immediately route their deposit through the token path. The window exists from the moment `addSupportedToken` is called until `setTokenFeeBps` is called. If the admin never calls `setTokenFeeBps`, the bypass is permanent. No special privileges, front-running, or external protocol compromise is required — only a standard DEX swap to obtain the supported token.

## Recommendation
Initialize `tokenFeeBps[token]` inside `addSupportedToken` to the current `feeBps` value (or require an explicit fee parameter), so newly added tokens are never silently fee-free:

```solidity
function addSupportedToken(address token, address oracle, address bridge, uint256 _feeBps)
    external onlyRole(TIMELOCK_ROLE)
{
    // ... existing checks ...
    supportedTokenList.push(token);
    supportedTokenOracle[token] = oracle;
    tokenBridge[token] = bridge;
    tokenFeeBps[token] = _feeBps;  // initialize fee at addition time
    emit AddSupportedToken(token, oracle, bridge);
}
```

Alternatively, fall back to `feeBps` when `tokenFeeBps[token]` is zero:
```solidity
uint256 feeBpsForToken = tokenFeeBps[token] > 0 ? tokenFeeBps[token] : feeBps;
```

## Proof of Concept
1. Admin calls `addSupportedToken(stETH, stETHOracle, bridge)` on `RSETHPool.sol`. `tokenFeeBps[stETH]` is now `0`.
2. Alice checks `tokenFeeBps[stETH]` on-chain — it is `0`.
3. Alice swaps 1 ETH → ~1 stETH on an external DEX (e.g., Curve).
4. Alice calls `deposit(stETH, 1e18, "ref")`. Fee computed: `1e18 * 0 / 10_000 = 0`. She receives the full wrsETH amount with zero fee deducted.
5. Bob deposits 1 ETH directly via `deposit("ref")`. Fee computed: `1e18 * feeBps / 10_000 > 0`. Bob receives fewer wrsETH than Alice for the same economic input.

Alice pays zero protocol fee; Bob pays the full fee. The protocol collects no fee from Alice's deposit.
