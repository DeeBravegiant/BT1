Audit Report

## Title
Missing Slippage Protection on `instantWithdrawal()` - (File: `contracts/LRTWithdrawalManager.sol`)

## Summary
`LRTWithdrawalManager.instantWithdrawal()` accepts no `minAssetAmountExpected` parameter, so a user who previews their expected return via `getExpectedAssetAmount()` has no on-chain mechanism to revert if oracle prices shift before their transaction executes. The deposit-side functions `depositETH()` and `depositAsset()` both enforce a `minRSETHAmountExpected` guard, creating a direct asymmetry. Once `burnFrom` executes at L229, the rsETH is permanently destroyed regardless of the asset amount ultimately received.

## Finding Description
`getExpectedAssetAmount()` (L580–594) computes `amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset)` using live oracle prices. `instantWithdrawal()` (L212–253) calls this function at L228 to determine `assetAmountUnlocked`, then immediately burns the caller's rsETH at L229 with no slippage check between the two operations. The function signature accepts only `(address asset, uint256 rsETHUnstaked, string calldata referralId)` — there is no `minAssetAmountExpected` parameter. If `rsETHPrice` decreases or `getAssetPrice(asset)` increases between the user's off-chain preview and on-chain execution, `assetAmountUnlocked` is lower, and the user receives fewer assets with no ability to revert. By contrast, `_beforeDeposit()` (L667–669) enforces `if (rsethAmountToMint < minRSETHAmountExpected) revert MinimumAmountToReceiveNotMet()`. The NatSpec at L210–211 itself acknowledges a related execution-time risk for fees, confirming the design is aware of but does not guard against this class of timing exposure.

## Impact Explanation
The impact is **Low — Contract fails to deliver promised returns, but doesn't lose value**. The user receives the oracle-fair value at execution time, not the previewed amount. The rsETH burn is irreversible; the user cannot recover tokens if the received asset amount is below their expectation. This matches the allowed Low impact: the contract fails to deliver the return implied by its own preview function, with no recourse for the user.

## Likelihood Explanation
Any unprivileged user calling `instantWithdrawal()` after previewing via `getExpectedAssetAmount()` is exposed. The LRT oracle is updated by protocol operators and can change between blocks. No attacker capability is required — normal oracle update cadence is sufficient. The gap between preview and execution is inherent to standard user behavior (off-chain preview → on-chain submit), making this a routine operational condition rather than an edge case.

## Recommendation
Add a `minAssetAmountExpected` parameter to `instantWithdrawal()` and revert if `userAmount` falls below it, mirroring the existing deposit-side pattern:

```solidity
function instantWithdrawal(
    address asset,
    uint256 rsETHUnstaked,
    uint256 minAssetAmountExpected,
    string calldata referralId
) external ... {
    ...
    uint256 userAmount = assetAmountUnlocked - fee;
    if (userAmount < minAssetAmountExpected) revert SlippageExceeded();
    ...
}
```

This check should be placed after `userAmount` is computed (post-fee) and before `_transferAsset` is called, consistent with the deposit-side guard at L667–669.

## Proof of Concept
1. User calls `getExpectedAssetAmount(ETH_TOKEN, 1 ether)` → returns `1.05 ETH`.
2. LRT oracle updates: `rsETHPrice` decreases 3% (routine operator update).
3. User submits `instantWithdrawal(ETH_TOKEN, 1 ether, referralId)`.
4. `getExpectedAssetAmount` now returns `~1.019 ETH`; after `instantWithdrawalFee` (e.g. 50 bps), `userAmount ≈ 1.014 ETH`.
5. `burnFrom` at L229 destroys the user's 1 rsETH irreversibly.
6. User receives `~1.014 ETH` — ~3.4% less than previewed — with no revert path.

Foundry test plan: deploy with a mock `LRTOracle`; call `getExpectedAssetAmount` to record preview; update mock oracle price downward; call `instantWithdrawal`; assert `userAmount < previewAmount` and that no revert occurred.
