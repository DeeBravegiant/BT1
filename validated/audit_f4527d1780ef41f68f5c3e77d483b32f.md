### Title
Raw MPC Key Share Material and Decryption Key Exposed via Positional CLI Arguments - (File: `crates/node/src/cli.rs`)

---

### Summary

The `ImportKeyshareCmd` and `ExportKeyshareCmd` CLI subcommands accept the raw threshold key share JSON (containing `private_share`) and the AES-128 local storage encryption key as **positional command-line arguments** with no environment-variable alternative. Every invocation is recorded verbatim in the operator's shell history file, persistently exposing MPC key share material in cleartext.

---

### Finding Description

In `crates/node/src/cli.rs`, `ImportKeyshareCmd` is defined as:

```rust
pub struct ImportKeyshareCmd {
    #[arg(long, env("MPC_HOME_DIR"))]
    pub home_dir: String,

    #[arg(
        help = "JSON string with the keyshare in format: {\"epoch\":1,\"private_share\":\"...\",\"public_key\":\"...\"}"
    )]
    pub keyshare_json: String,          // positional — no env(), no long

    #[arg(help = "Hex-encoded 16 byte AES key for local storage encryption")]
    pub local_encryption_key_hex: String,  // positional — no env(), no long
}
``` [1](#0-0) 

Neither `keyshare_json` nor `local_encryption_key_hex` carries an `env()` annotation or a `--long` flag. Clap therefore requires them to be supplied as positional arguments on the command line. A typical invocation looks like:

```
mpc-node import-keyshare \
  '{"epoch":1,"private_share":"<secret_scalar>","public_key":"..."}' \
  0123456789ABCDEF0123456789ABCDEF
```

The shell records this entire string in `~/.bash_history` / `~/.zsh_history` / audit logs.

`ExportKeyshareCmd` has the same problem for `local_encryption_key_hex`:

```rust
pub struct ExportKeyshareCmd {
    #[arg(long, env("MPC_HOME_DIR"))]
    pub home_dir: String,

    #[arg(help = "Hex-encoded 16 byte AES key for local storage encryption")]
    pub local_encryption_key_hex: String,  // positional — no env()
}
``` [2](#0-1) 

The `keyshare_json` field is deserialized directly into `PermanentKeyshareData`, which holds the node's threshold key share (`private_share`): [3](#0-2) 

The `local_encryption_key_hex` is the AES-128 key that encrypts the keyshare at rest. Whoever holds both the history entry and the encrypted keyshare file on disk can decrypt the share without running the node.

By contrast, the `StartCmd`'s `secret_store_key_hex` is correctly annotated with `#[arg(env("MPC_SECRET_STORE_KEY"))]`, giving operators a safe, history-free path: [4](#0-3) 

`ImportKeyshareCmd` and `ExportKeyshareCmd` have no such escape hatch.

---

### Impact Explanation

`private_share` is the node's individual threshold key share — the most sensitive cryptographic material in the system. Its exposure in shell history constitutes unauthorized access to MPC key share material. An attacker who reads the shell history of `t` (threshold) distinct nodes obtains `t` raw shares and can reconstruct the full distributed private key, enabling them to forge threshold signatures for arbitrary foreign-chain transactions without any participant's knowledge or consent. Even a single share reduces the remaining security margin from `t` to `t−1` additional compromises. This matches the Critical impact: *unauthorized access to MPC key shares or secret material that materially enables forgery or secret recovery*.

---

### Likelihood Explanation

**Low.** The `ImportKeyshare` command is a maintenance/migration operation, not a routine one. However, shell history files are readable by the local user and by root, and in cloud/container environments they may be captured by audit daemons, container log drivers, or process-level monitoring (e.g., `auditd`, CloudTrail command logging). An insider threat, a compromised SSH key, or a misconfigured log aggregator is sufficient to extract the material. Because the command must be run on each node during migration, the exposure surface spans every participating operator.

---

### Recommendation

1. **Accept a file path instead of the raw JSON string** for `keyshare_json` — read the file contents inside the command handler. This keeps the secret off the command line entirely.
2. **Add `env()` annotations** to `local_encryption_key_hex` in both `ImportKeyshareCmd` and `ExportKeyshareCmd` so operators can supply the key via an environment variable (which does not appear in shell history).
3. **Optionally**, support reading the encryption key from stdin with no echo (analogous to the `x/term.ReadPassword` recommendation in the original report) for interactive use.

---

### Proof of Concept

1. Operator migrates a node and runs:
   ```
   mpc-node import-keyshare \
     '{"epoch":3,"private_share":"<hex_scalar>","public_key":"<hex_point>"}' \
     deadbeefdeadbeefdeadbeefdeadbeef
   ```
2. Shell appends the full command to `~/.bash_history`.
3. Attacker with read access to that file (e.g., via a compromised SSH key or audit log) extracts `private_share` and `local_encryption_key_hex`.
4. With `local_encryption_key_hex` and the encrypted keyshare file at `$MPC_HOME_DIR/permanent_keys/`, the attacker decrypts the stored keyshare independently.
5. Repeating across `t` nodes' history files yields `t` raw shares; Lagrange interpolation reconstructs the full private key.
6. The attacker can now produce valid threshold signatures for arbitrary foreign-chain payloads, bypassing the MPC network entirely.
