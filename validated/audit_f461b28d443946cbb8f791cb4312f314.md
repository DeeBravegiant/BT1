Audit Report

## Title
Permissionless `FeeReceiver.sendFunds()` Enables Atomic Yield Front-Running via Stale `rsETHPrice` — (`contracts/FeeReceiver.sol` / `contracts/LRTDepositPool.sol`)

## Summary

`FeeReceiver.sendFunds()` carries no access control and can be called by any external account. It pushes accumulated MEV/execution-layer ETH rewards into `LRTDepositPool`, immediately increasing the protocol's on-chain ETH balance and therefore its TVL as measured by `_getTotalEthInProtocol()`. Because `depositETH()` / `depositAsset()` price rsETH using the **stored** `rsETHPrice` — which is never refreshed during the deposit flow — an attacker can atomically call `sendFunds()` and then deposit in the same transaction, minting rsETH at the pre-reward price and capturing yield that belongs to existing rsETH holders.

## Finding Description

**Root cause 1 — no access control on `sendFunds()`:**

`FeeReceiver.sendFunds()` is declared `external` with no role modifier:

```solidity
// contracts/FeeReceiver.sol:53-58
function sendFunds() external {
    uint256 balance = address(this).balance;
    ILRTDepositPool(depositPool).receiveFromRewardReceiver{ value: balance }();
    emit MevRewardsAddedToTVL(balance);
}
```

`receiveFromRewardReceiver()` in `LRTDepositPool` is likewise unrestricted (`external payable`, no modifier), so the ETH lands in the deposit pool immediately.

**Root cause 2 — deposit pricing uses stale stored price:**

`getRsETHAmountToMint()` reads `lrtOracle.rsETHPrice()`, which is a stored state variable last written by `_updateRsETHPrice()`:

```solidity
// contracts/LRTDepositPool.sol:519-520
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

Neither `depositETH()` nor `depositAsset()` calls `updateRSETHPrice()` before computing the mint amount. The price is only refreshed when `updateRSETHPrice()` is explicitly invoked externally.

**Exploit flow (single atomic transaction):**

1. Attacker deploys a proxy contract.
2. Proxy calls `FeeReceiver.sendFunds()` — accumulated MEV rewards (e.g. 10 ETH) move into `LRTDepositPool`, raising `address(depositPool).balance` and therefore `getETHDistributionData().ethLyingInDepositPool` and `_getTotalEthInProtocol()`.
3. Proxy immediately calls `LRTDepositPool.depositETH{value: 100 ether}(0, "")` — rsETH is minted using the **pre-reward** `rsETHPrice` (e.g. 1 ETH/rsETH), so attacker receives 100 rsETH.
4. When `updateRSETHPrice()` is later called, `previousTVL = rsethSupply × rsETHPrice = 1100 × 1 = 1100 ETH`, `totalETHInProtocol = 1110 ETH`, `rewardAmount = 10 ETH`. The new price is set to `(1110 − fee) / 1100 ≈ 1.00818 ETH/rsETH`.
5. Attacker's 100 rsETH is now worth ≈ 100.82 ETH — a profit of ≈ 0.82 ETH extracted from existing holders' accrued yield.

**Why existing guards do not prevent this:**

- `nonReentrant` on `depositETH()` does not apply because `sendFunds()` is called *before* entering the deposit, not during it.
- The `pricePercentageLimit` check in `_updateRsETHPrice()` fires only when `updateRSETHPrice()` is called, not during the deposit.
- `minRSETHAmountExpected` is a slippage parameter controlled by the attacker; setting it to 0 disables it entirely.

## Impact Explanation

**High — Theft of unclaimed yield.**

Existing rsETH holders are entitled to the full reward `R` (minus protocol fee) accrued before the attacker's deposit. Because the attacker's rsETH is minted at the pre-reward price, their share of the post-update price appreciation is proportionally larger than their fair contribution. The surplus — approximately `R × (attackerDeposit / newSupply)` — is extracted directly from the yield that pre-existing holders would otherwise receive. The attack scales linearly with both deposit size and accumulated reward size, and leaves no on-chain trace distinguishable from a normal deposit.

## Likelihood Explanation

Any unprivileged EOA or contract can call `sendFunds()` — there is no role, whitelist, time-lock, or cooldown. MEV and execution-layer rewards accumulate continuously between `updateRSETHPrice()` calls (which are off-chain triggered). The attack requires only a proxy contract and ETH, both standard on-chain primitives. It is repeatable every time rewards build up before the next price update, making it a recurring, low-effort opportunity for any sophisticated actor.

## Recommendation

Apply at least one of the following mitigations:

1. **Restrict `sendFunds()`** to a trusted role (e.g. `LRTConstants.MANAGER`) so only the protocol can decide when to flush rewards into the deposit pool.
2. **Refresh the price before minting**: call `updateRSETHPrice()` (or an internal equivalent) at the start of `depositETH()` / `depositAsset()` so every deposit uses the current TVL.
3. **Same-block deposit guard**: record the block number of the last `sendFunds()` call and revert deposits in the same block, preventing atomic exploitation.

## Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

interface IFeeReceiver { function sendFunds() external; }
interface IDepositPool { function depositETH(uint256 minRSETH, string calldata ref) external payable; }

contract YieldFrontRunner {
    IFeeReceiver public feeReceiver;
    IDepositPool public depositPool;

    constructor(address _feeReceiver, address _depositPool) {
        feeReceiver = IFeeReceiver(_feeReceiver);
        depositPool = IDepositPool(_depositPool);
    }

    function attack() external payable {
        // Step 1: flush accumulated MEV rewards into deposit pool — TVL rises, rsETHPrice stays stale
        feeReceiver.sendFunds();
        // Step 2: deposit at the now-stale lower price — receive more rsETH than fair value
        depositPool.depositETH{value: msg.value}(0, "");
    }

    receive() external payable {}
}
```

**Foundry fork test outline:**

1. Fork mainnet (or a testnet deployment).
2. Seed `FeeReceiver` with 10 ETH (simulating accumulated MEV rewards).
3. Record `rsETHPrice` and existing holder balances.
4. Deploy `YieldFrontRunner`, call `attack{value: 100 ether}()`.
5. Call `LRTOracle.updateRSETHPrice()`.
6. Assert attacker's rsETH value > 100 ETH and existing holders' per-rsETH value < expected post-reward price, confirming yield dilution.
