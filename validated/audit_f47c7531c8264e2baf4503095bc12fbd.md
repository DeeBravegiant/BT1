### Title
`recompute_available_foreign_chains` uses raw `reconstruction_threshold` instead of protocol-adjusted active-signer count, silently under-counting required signers for `DamgardEtAl` ForeignTx domains — (`File: crates/contract/src/lib.rs`)

---

### Summary

`recompute_available_foreign_chains()` computes the minimum supporter count for a chain to be marked "available" by reading `reconstruction_threshold.inner()` directly from each `ForeignTx` domain. For the `DamgardEtAl` protocol the honest-majority requirement means the actual minimum active signers is `2t − 1`, not `t`. The function therefore marks a chain as available when only `t` participants cover it, while signing would require `2t − 1`. A `verify_foreign_transaction` request for that chain is accepted by the contract but cannot reach the signing threshold and times out, breaking the contract's core liveness invariant. The developers have already flagged this with a `TODO` comment referencing issue #3556.

---

### Finding Description

In `recompute_available_foreign_chains()` the threshold passed to `update_available_chains_config_cache` is derived as:

```rust
// TODO(#3556): replace this with a per-scheme
// `required_active_signers(protocol, reconstruction_threshold)`.
let Some(threshold) = self.protocol_state.domain_registry().ok().and_then(|r| {
    r.domains()
        .iter()
        .filter(|d| d.purpose == DomainPurpose::ForeignTx)
        .map(|d| d.reconstruction_threshold.inner())   // ← raw t, not 2t-1
        .max()
}) else { … };
``` [1](#0-0) 

The `DamgardEtAl` protocol enforces an honest-majority bound: signing requires at least `2t − 1` active participants, as validated everywhere else in the codebase:

```rust
if domain.protocol == Protocol::DamgardEtAl {
    let required = t.checked_mul(2).and_then(|x| x.checked_sub(1))…;
    if required > num_participants { return Err(…); }
}
``` [2](#0-1) 

The same `2t − 1` formula appears in the threshold-signatures layer:

```rust
Protocol::DamgardEtAl => 2 * t - 1,
``` [3](#0-2) 

`recompute_available_foreign_chains` is the only place that reads the threshold for the available-chain cache and it is the only place that does **not** apply the protocol-specific adjustment. The mismatch is structural: `validate_domain_threshold` and `min_active_participants` both know about `DamgardEtAl`'s `2t − 1` requirement, but the cache-recompute path ignores it.

The function is called on every `register_foreign_chains_config` invocation:

```rust
self.foreign_chains.get_mut().register(tls_key, foreign_chains_config);
self.recompute_available_foreign_chains();
``` [4](#0-3) 

`update_available_chains_config_cache` then marks a chain available whenever the supporter count reaches the (too-low) threshold `t`:

```rust
.filter_map(|(chain, count)| (count >= threshold).then_some(chain))
``` [5](#0-4) 

`verify_foreign_transaction` gates on `get_supported_foreign_chains()` today, but the new design gates on `get_available_foreign_chains()` (the cached set produced by `recompute_available_foreign_chains`):

```rust
let supported_chains = self.get_supported_foreign_chains();
if !supported_chains.contains(&requested_chain) { env::panic_str(…); }
``` [6](#0-5) 

Once the migration to `get_available_foreign_chains()` completes (the stated direction in `docs/design/calculating-supported-foreign-chains.md`), every `verify_foreign_transaction` call will gate on the incorrectly-computed cache.

---

### Impact Explanation

**Medium — request-lifecycle and contract execution-flow manipulation that breaks a production safety invariant.**

The contract's documented invariant is: *"a request is accepted only when `C` is available (≥ `signing_threshold` participants cover it), so an accepted request can reach the signing threshold."* [7](#0-6) 

With a `DamgardEtAl` ForeignTx domain at threshold `t`:

- The cache marks chain `C` available once `t` participants register coverage.
- Signing `C` requires `2t − 1` participants.
- If only `t` participants cover `C`, every accepted `verify_foreign_transaction(C)` request will time out: the MPC nodes cannot assemble `2t − 1` shares, the yield-resume never fires, and the caller's deposit is consumed by `fail_on_timeout`.

This is not a transient liveness blip — it is a permanent, deterministic failure for any request submitted while coverage sits between `t` and `2t − 2`.

---

### Likelihood Explanation

Adding a `DamgardEtAl` ForeignTx domain is a normal governance operation (`vote_add_domains`) requiring a threshold of existing participants. Once such a domain exists, the bug is triggered by any participant calling `register_foreign_chains_config` — an unprivileged, routine operation every node performs on startup and periodically thereafter. No adversarial coordination is required beyond the legitimate governance vote that created the domain. The `TODO(#3556)` comment confirms the developers have already identified the gap; the fix has not yet landed.

---

### Recommendation

Replace the raw `reconstruction_threshold.inner()` read with a call to the existing `min_active_participants` helper (or an equivalent `required_active_signers` function) so that `DamgardEtAl` domains contribute `2t − 1` to the threshold rather than `t`:

```rust
// Before (incorrect for DamgardEtAl):
.map(|d| d.reconstruction_threshold.inner())

// After:
.map(|d| min_active_participants(&d.protocol, &d.reconstruction_threshold))
```

This mirrors the logic already present in `validate_domain_threshold` and the `min_active_participants` helper referenced in the design documents.

---

### Proof of Concept

1. Governance votes add a `DamgardEtAl` ForeignTx domain with `reconstruction_threshold = 3` (requires `2×3−1 = 5` active signers to sign).
2. Exactly 3 of 7 participants call `register_foreign_chains_config` reporting coverage of chain `C` (which is whitelisted).
3. Each call triggers `recompute_available_foreign_chains()`; the cache sees `count(C) = 3 >= threshold(3)` and marks `C` available.
4. A user calls `verify_foreign_transaction(C)` — the contract accepts it (chain is "available") and enqueues a yield-resume promise.
5. MPC nodes attempt to sign: only 3 nodes cover `C`, but `DamgardEtAl` requires 5. Signing fails; no `respond_verify_foreign_tx` is ever submitted.
6. After ~200 blocks the yield times out; `fail_on_timeout` fires, the user's request is permanently failed, and the deposit is consumed — despite the contract having advertised `C` as serviceable.
