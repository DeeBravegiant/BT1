### Title
Single-Participant Unilateral Key-Event Abort Enables Indefinite Keygen/Resharing Freeze - (`crates/contract/src/state/key_event.rs`)

---

### Summary

The `vote_abort_key_event_instance` contract method allows any single attested participant to unilaterally abort an active key-generation or resharing instance with a single transaction. There is no threshold requirement, no cost, and no cooldown. A Byzantine participant can watch the chain for each new instance started by the leader and immediately abort it, permanently blocking key generation or resharing for as long as they remain in the participant set. The structural analog to the defiProtocol bonding DOS is exact: one actor locks a shared protocol phase, the lock is cheap to re-apply, and the only escape requires governance action that the attacker can also block.

---

### Finding Description

`vote_abort_key_event_instance` in `crates/contract/src/lib.rs` delegates to `KeyEvent::vote_abort` in `crates/contract/src/state/key_event.rs`:

```rust
pub fn vote_abort(&mut self, key_event_id: KeyEventId) -> Result<(), Error> {
    let candidate = self.verify_vote(&key_event_id)?;
    if self
        .instance
        .as_ref()
        .unwrap()
        .completed
        .contains(&candidate)
    {
        return Err(VoteError::VoteAlreadySubmitted.into());
    }
    self.instance = None;   // ← single-vote, unconditional abort
    Ok(())
}
``` [1](#0-0) 

The only guard is that the caller must not have already cast a *success* vote for this instance. A participant who never votes success can call `vote_abort` on every new instance the leader starts, setting `self.instance = None` each time.

The leader-start path (`KeyEvent::start`) checks `self.instance.is_some()` and returns `ActiveKeyEvent` if an instance is already running, but after an abort the instance is `None` and the leader can start again — only to be aborted again immediately. [2](#0-1) 

This applies to both the `Initializing` state (key generation) and the `Resharing` state (participant-set update), as confirmed by the fake-indexer test harness: [3](#0-2) 

The `vote_abort_key_event_instance` entry point in `lib.rs` is gated only by `assert_caller_is_attested_participant_and_protocol_active`, meaning any single attested participant in the current epoch can call it: [4](#0-3) 

---

### Impact Explanation

**Resharing freeze (most critical path):** When governance initiates resharing to remove a compromised participant (via `vote_new_parameters`), the target participant can abort every resharing instance the leader starts. The other participants can call `vote_cancel_resharing` to escape the `Resharing` state, but that merely reverts to the previous `Running` state — with the compromised participant still in the set. The governance cycle can be repeated indefinitely, but the attacker aborts each attempt at zero cost. The compromised participant is permanently irremovable, breaking the participant-state safety invariant that governance is supposed to enforce.

**Keygen freeze:** When the network is in `Initializing` state (triggered by `vote_add_domains`), the `respond` function rejects all signing requests: [5](#0-4) 

A malicious participant can keep the network in `Initializing` indefinitely by aborting every keygen instance, freezing all signature production for every domain. Users can escape via `vote_cancel_keygen`, but the attacker can trigger the cycle again if they have governance voting rights.

---

### Likelihood Explanation

The attack requires only that the adversary is an attested participant — a role that is granted by governance but, once granted, carries no ongoing cost to abuse. The abort transaction is a cheap NEAR function call. The attacker needs no special timing advantage: they simply watch the chain for `start_keygen_instance` / `start_reshare_instance` events and respond with `vote_abort_key_event_instance` in the next block. The default `key_event_timeout_blocks = 30` (~18 s on mainnet) gives ample time to observe and react. [6](#0-5) 

The most realistic trigger is a participant whose node is compromised or whose operator turns adversarial after joining the network. Because the abort is free and instant, the cost-benefit ratio strongly favors the attacker.

---

### Recommendation

Replace the single-vote abort with a threshold-vote abort, consistent with how `vote_cancel_resharing` works (requires a threshold of previous-epoch participants). Specifically:

1. Change `KeyEvent::vote_abort` to accumulate abort votes in a `BTreeSet` (analogous to `completed`) and only set `self.instance = None` once the abort-vote count reaches the signing threshold.
2. Alternatively, remove `vote_abort_key_event_instance` entirely and rely solely on the existing block-height timeout (`key_event_timeout_blocks`) to clear stale instances, since the leader already retries automatically after timeout.

The timeout-only approach is simpler and eliminates the single-actor abort surface entirely. The leader node already handles the retry loop: [7](#0-6) 

---

### Proof of Concept

1. Governance calls `vote_new_parameters` to initiate resharing that removes participant `P_evil`.
2. The leader (lowest-ID participant) calls `start_reshare_instance(key_event_id_0)`.
3. `P_evil` immediately calls `vote_abort_key_event_instance(key_event_id_0)`. `KeyEvent::vote_abort` sets `instance = None`.
4. The leader retries: calls `start_reshare_instance(key_event_id_1)` (next `AttemptId`).
5. `P_evil` calls `vote_abort_key_event_instance(key_event_id_1)`. Instance aborted again.
6. Steps 4–5 repeat indefinitely. The resharing never completes. `P_evil` remains in the participant set permanently.
7. Other participants call `vote_cancel_resharing` to escape `Resharing` state, reverting to `Running` with `P_evil` still present. Any future resharing attempt is blocked by the same mechanism.

/// The leader logic for an entire key generation (initializing) state.
/// Handles multiple domains and attempts. It does not return, except in case of catastrophic
/// failure (node shutting down). The coordinator is expected to interrupt this when the
/// contract state transitions out of the key generation state.
pub async fn keygen_leader(
    client: impl KeyEventLeaderClient,
    keyshare_storage: Arc<RwLock<KeyshareStorage>>,
    mut key_event_receiver: watch::Receiver<ContractKeyEventInstance>,
    chain_txn_sender: impl TransactionSender,
    threshold: ReconstructionThreshold,
) -> anyhow::Result<()> {
    loop {
        // Wait for all participants to be connected. Otherwise, computations are most likely going
        // to fail so don't waste the effort.
        client.wait_for_all_participants_connected().await?;

        // Wait for the contract to have no active key event instance.
        let key_event_id = key_event_receiver
            .wait_for(|contract_event| !contract_event.started)
            .await?
            .id;
        // Send txn to start the keygen instance. This may or may not end up in the chain; we'll
        // wait for it. If it doesn't happen after some time, we try again.
        chain_txn_sender
            .send(ChainSendTransactionRequest::StartKeygen(
                contract_args::StartKeygenArgs::new(key_event_id),
            ))
            .await?;

        match timeout(
            MAX_LATENCY_BEFORE_EXPECTING_TRANSACTION_TO_FINALIZE,
            key_event_receiver.wait_for(|contract_event| contract_event.started),
        )
        .await
        {
            Ok(res) => {
                let contract_key_event_id = res?.id;
                if contract_key_event_id != key_event_id {

                        "Activated key event {:?} does not match expected {:?}; retrying.",

                        key_event_id
                    );
                    continue;
                }
            }
            Err(_) => {
                tracing::warn!(
                    "Key event {:?} did not activate in time; retrying.",
