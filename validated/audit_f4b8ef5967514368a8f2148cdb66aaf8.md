Audit Report

## Title
Missing Slippage Protection When Removing Liquidity - (File: `metric-core/contracts/MetricOmmPool.sol`, `metric-core/contracts/libraries/LiquidityLib.sol`)

## Summary
`MetricOmmPool.removeLiquidity` accepts no `minAmount0`/`minAmount1` parameters, and the returned token amounts are computed entirely from live bin state at execution time. Because swaps move the price cursor across bins and convert their token composition, a front-running swap can cause an LP to receive a completely different token than deposited, with no on-chain mechanism to revert the transaction. No periphery wrapper adds this protection.

## Finding Description
`MetricOmmPool.removeLiquidity` (L199–212) delegates to `LiquidityLib.removeLiquidity` (L161–251), which computes each token amount as a proportional share of the bin's current scaled balances:

```solidity
uint256 amount0Scaled = _checkedMul(binState.token0BalanceScaled, sharesToRemove) / binTotalSharesVal;
uint256 amount1Scaled = _checkedMul(binState.token1BalanceScaled, sharesToRemove) / binTotalSharesVal;
```

These values depend entirely on `token0BalanceScaled` / `token1BalanceScaled` at execution time. In the Metric OMM bin model, swaps move the price cursor across bins and convert their token composition: a bin above the current price cursor holds only token0, but after a swap moves the cursor past it, it holds only token1. An LP who deposited token0 into such a bin will receive only token1 on withdrawal if a swap has moved the cursor past that bin between submission and execution.

The pool signature has no minimum-output parameters:
```solidity
function removeLiquidity(address owner, uint80 salt, LiquidityDelta calldata deltas, bytes calldata extensionData)
```

After computing amounts, tokens are transferred directly with no post-condition check:
```solidity
if (amount0Removed > 0) { IERC20(ctx.token0).safeTransfer(owner, amount0Removed); }
if (amount1Removed > 0) { IERC20(ctx.token1).safeTransfer(owner, amount1Removed); }
```

`MetricOmmPoolLiquidityAdder` exposes only `addLiquidity` paths with `maxAmountToken0`/`maxAmountToken1` caps enforced in the callback, but no `removeLiquidity` function at all. There is no periphery wrapper anywhere in the stack that adds minimum-output protection for withdrawals. The `_afterRemoveLiquidity` extension hook exists but is not a slippage guard — it is an optional extension callback, not a protocol-level check.

## Impact Explanation
An LP who deposited token0 into a bin above the current price cursor can receive exclusively token1 (zero token0) if a swap moves the cursor past that bin before their withdrawal executes. While the scaled value is approximately preserved at oracle prices, the LP:
- Receives a token they did not want (e.g., cannot repay a token0-denominated debt)
- Must execute an additional swap to convert back, incurring protocol fees and market slippage
- Has no on-chain mechanism to revert the transaction if the output composition falls below their acceptable threshold

This constitutes a medium direct loss of owed LP assets: the LP is owed their deposited token composition but receives a different one with no recourse, and the conversion cost is a real economic loss not recoverable from the protocol.

## Likelihood Explanation
The trigger is any swap that moves the price cursor across a bin in which the victim holds shares — a normal, permissionless pool operation requiring no privileged access. In a multi-bin pool with active trading, cursor movement across bins is routine. The victim's `removeLiquidity` transaction is visible in the mempool, making targeted front-running straightforward for any MEV bot. No special conditions or privileged roles are required.

## Recommendation
Add `minAmount0` and `minAmount1` parameters to `removeLiquidity` in both `IMetricOmmPoolActions` and `LiquidityLib`, and revert if computed amounts fall below the caller's minimums:

```solidity
function removeLiquidity(
    address owner, uint80 salt, LiquidityDelta calldata deltas,
    uint256 minAmount0, uint256 minAmount1,
    bytes calldata extensionData
) external nonReentrant(PoolActions.REMOVE_LIQUIDITY) returns (uint256 amount0Removed, uint256 amount1Removed) {
    ...
    (amount0Removed, amount1Removed) = LiquidityLib.removeLiquidity(...);
    if (amount0Removed < minAmount0 || amount1Removed < minAmount1) revert SlippageExceeded();
    ...
}
```

Additionally, expose a `removeLiquidity` wrapper in `MetricOmmPoolLiquidityAdder` (or a dedicated periphery contract) that enforces these bounds, mirroring the existing `maxAmountToken0`/`maxAmountToken1` pattern used for deposits.

## Proof of Concept
1. Pool has bins at indices -1, 0, 1. Current price cursor is at bin 0.
2. Alice calls `addLiquidity` for bin 1 (above current price), depositing token0.
3. Alice submits `removeLiquidity` for bin 1, expecting to receive token0.
4. Bob (or a MEV bot) front-runs with a large `swap(zeroForOne=false)` that moves the cursor from bin 0 to bin 1 and beyond, converting bin 1's `token0BalanceScaled` entirely to `token1BalanceScaled`.
5. Alice's `removeLiquidity` executes: `LiquidityLib.removeLiquidity` at L205 computes `amount0Scaled = _checkedMul(0, sharesToRemove) / binTotalSharesVal = 0`; at L206 computes `amount1Scaled > 0`. Alice receives only token1 and zero token0.
6. The transfer block at L242–247 skips the token0 transfer silently. The transaction succeeds with Alice receiving a composition she never consented to and no revert occurs because there is no minimum-output check anywhere in the call path.
