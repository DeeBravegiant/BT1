### Title
Deposit Loss for Non-Winning Update Proposers When `do_update` Clears All Entries Without Refund - (File: `crates/contract/src/update.rs`)

---

### Summary

When a contract update reaches threshold votes and `do_update` executes, it unconditionally clears **all** pending update proposals and their associated votes. However, every proposer paid a mandatory storage deposit via `propose_update` that is never refunded for the non-winning entries. Those deposits are permanently locked in the contract with no recovery path.

---

### Finding Description

`propose_update` in `lib.rs` requires each proposer to attach a deposit covering storage costs: [1](#0-0) 

The required deposit is calculated as `storage_byte_cost * bytes_used`, where `bytes_used` includes the full contract binary size plus overhead for up to 128 participant vote entries: [2](#0-1) 

Only the excess above `required` is refunded at proposal time; the required portion stays in the contract: [3](#0-2) 

When `vote_update` reaches threshold, it calls `do_update`: [4](#0-3) 

`do_update` removes the winning entry, then **clears all remaining entries and all votes** with no deposit refund: [5](#0-4) 

There is no `withdraw_proposal` endpoint, and `remove_update_vote` only removes a vote — it does not remove the proposal entry or refund the deposit: [6](#0-5) 

The `UpdateEntry` struct stores `bytes_used` — the exact information needed to compute the refund — but it is discarded along with the entry during `entries.clear()`: [7](#0-6) 

---

### Impact Explanation

Every participant who proposed a competing update loses their storage deposit permanently when any other update reaches threshold. For a 500 KB contract binary, the deposit is approximately 5 NEAR per proposal (at NEAR's ~10 NEAR/MB storage rate). With multiple concurrent proposals, the aggregate loss scales linearly. The funds are irrecoverable: the entries are gone, the depositor identity is not tracked, and no refund path exists.

This breaks the production accounting invariant that deposits paid for storage are returned when that storage is freed. Impact: **Medium** — permanent balance loss for participants, triggered by normal protocol operation, with no operator misconfiguration required.

---

### Likelihood Explanation

This fires every time a contract upgrade or config change reaches threshold while at least one other proposal is pending. In a governance-active network with multiple participants proposing competing upgrades (e.g., different code versions or config values), concurrent proposals are a normal operational condition, not an edge case.

---

### Recommendation

Before calling `self.entries.clear()` in `do_update`, iterate over all non-winning entries and schedule a `Promise::transfer` back to each proposer for their stored `bytes_used`-derived deposit. The proposer's account ID must be stored in `UpdateEntry` at proposal time (it is currently absent). Alternatively, track a `(proposer: AccountId, deposit: NearToken)` field in `UpdateEntry` so the refund loop has all necessary information.

---

### Proof of Concept

1. Participant A calls `propose_update` with a 500 KB contract binary. Required deposit ≈ 5 NEAR. Excess is refunded; 5 NEAR stays in the contract.
2. Participant B calls `propose_update` with a different 500 KB binary. Another 5 NEAR stays in the contract.
3. Threshold participants vote for B's update via `vote_update`. Threshold is reached; `do_update` is called.
4. `do_update` executes B's update, then calls `self.entries.clear()` and `self.vote_by_participant.clear()`.
5. A's entry — and its associated 5 NEAR deposit — is silently discarded. A has no mechanism to recover it.
6. Net result: A permanently loses 5 NEAR with no on-chain recourse.

    pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
        let entry = self.entries.remove(id)?;

        // Clear all entries as they might be no longer valid
        self.entries.clear();
        self.vote_by_participant.clear();

        let mut promise = Promise::new(env::current_account_id());
        match entry.update {
            Update::Contract(code) => {
                // deploy contract then do a `migrate` call to migrate state.
                promise = promise.deploy_contract(code).function_call(
                    method_names::MIGRATE,
                    Vec::new(),
                    NearToken::from_near(0),
                    gas,
                );
            }
            Update::Config(config) => {
                // If we vote for a new config, we should use
                // the value `contract_upgrade_deposit_tera_gas` from the config
                // as the new gas value
                let new_config_gas_value = Gas::from_tgas(config.contract_upgrade_deposit_tera_gas);
                promise = promise.function_call(
                    method_names::UPDATE_CONFIG,
                    serde_json::to_vec(&(&config,)).unwrap(),
                    NearToken::from_near(0),
                    new_config_gas_value,
                );
            }
        }
        Some(promise)
    }

fn bytes_used(update: &Update) -> u128 {
    let mut bytes_used = std::mem::size_of::<UpdateEntry>() as u128;

    // Assume a high max of 128 participant votes per update entry.
    bytes_used += 128 * std::mem::size_of::<AccountId>() as u128;

    match update {
        Update::Contract(code) => {
            bytes_used += code.len() as u128;
        }
        Update::Config(config) => {
            let bytes = serde_json::to_vec(&config).unwrap();
            bytes_used += bytes.len() as u128;
        }
    }

    bytes_used
}

fn required_deposit(bytes_used: u128) -> NearToken {
    env::storage_byte_cost().saturating_mul(bytes_used)
}
