Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Gates on `owner` Instead of `sender`, Allowing Complete Allowlist Bypass — (File: `metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` argument (the actual caller) and checks `owner` (the position beneficiary) against the allowlist instead. Because `MetricOmmPool.addLiquidity` separates the payer (`msg.sender`) from the position owner (`owner`), any unprivileged address can bypass the allowlist by calling `pool.addLiquidity(allowlisted_address, ...)` directly. The admin-set access-control boundary on liquidity ingress is completely defeated.

## Finding Description

`MetricOmmPool.addLiquidity` accepts an arbitrary `owner` address distinct from `msg.sender` and calls `_beforeAddLiquidity(msg.sender, owner, ...)`: [1](#0-0) 

`ExtensionCalling._beforeAddLiquidity` correctly forwards both `sender` and `owner` to every registered extension: [2](#0-1) 

`DepositAllowlistExtension.beforeAddLiquidity` receives `sender` as its first argument but the parameter is unnamed — it is silently discarded. The check is performed on `owner`: [3](#0-2) 

The mapping and setter are both named `allowedDepositor`, making the intent unambiguous — the allowlist is meant to gate the depositor (caller), not the position beneficiary: [4](#0-3) 

Because the check is on `owner`, any caller can pass the gate by supplying an allowlisted address as `owner`. The operator pattern (adding liquidity on behalf of another address) is explicitly supported and tested: [5](#0-4) 

Additionally, `removeLiquidity` enforces `msg.sender == owner`, so an attacker who adds dust shares to a victim's position can force the victim into a `MinimalLiquidity` revert on partial exits: [6](#0-5) [7](#0-6) 

## Impact Explanation

**Admin-boundary break:** The pool admin's intent to restrict which addresses may add liquidity is completely defeated. Any unprivileged address can call `pool.addLiquidity(allowlisted_victim, salt, deltas, ...)` directly, pay the tokens through the callback, and have shares credited to the victim's position while the allowlist check silently passes. Additionally, allowlisted addresses have their positions exposed to arbitrary third-party deposits without consent, and non-allowlisted addresses cannot add liquidity even to their own positions — the semantics are inverted. The griefing vector (dust deposit locking a victim's partial exit via `MinimalLiquidity`) constitutes a broken core pool functionality causing loss of usable withdraw flows.

## Likelihood Explanation

Exploitation requires only deploying a minimal contract implementing `IMetricOmmModifyLiquidityCallback` and calling `pool.addLiquidity` with any allowlisted address as `owner`. No privileged access, no special tokens, and no complex setup are needed. The operator pattern is a documented and tested flow in the codebase itself.

## Recommendation

Change the allowlist check to validate `sender` (the actual caller) rather than `owner` (the position beneficiary):

```solidity
// After (fixed):
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

## Proof of Concept

```
Setup:
  - Pool deployed with DepositAllowlistExtension as beforeAddLiquidity hook.
  - Admin calls setAllowedToDeposit(pool, alice, true).
  - Bob (attacker) is NOT on the allowlist.

Attack:
  1. Bob deploys AttackerContract implementing IMetricOmmModifyLiquidityCallback.
  2. AttackerContract calls pool.addLiquidity(alice, salt, deltas, callbackData, "").
  3. Pool calls _beforeAddLiquidity(address(AttackerContract), alice, …).
  4. DepositAllowlistExtension checks allowedDepositor[pool][alice] → true → passes.
  5. Pool mints shares to alice's position.
  6. Pool calls AttackerContract.metricOmmModifyLiquidityCallback(…).
  7. AttackerContract pays the required tokens.

Result:
  - Bob (not on allowlist) successfully added liquidity, bypassing the allowlist.
  - Alice's position is modified without her consent.
  - If alice had N shares and Bob added 1 share, alice cannot remove exactly N shares
    (leaving 1 < minimalMintableLiquidity), forcing a full exit or acceptance of the
    unwanted position mutation.
```
