Audit Report

## Title
Cross-Token Withdrawal Allows Draining Bridger Collateral and Permanently Freezing Depositor Funds — (`contracts/L2/RsETHTokenWrapper.sol`)

## Summary

`_withdraw` transfers any `allowedToken` to the caller upon burning wrsETH, with no check that the requested asset matches what the caller originally deposited. Because wrsETH is a single fungible token with no per-asset accounting, a user who deposited tokenA can burn their wrsETH to claim tokenB (deposited by the bridger as collateral), leaving tokenA permanently irrecoverable in the wrapper with zero wrsETH supply remaining to redeem it.

## Finding Description

**Root cause — `_withdraw` (L120–128):**

```solidity
function _withdraw(address _asset, address _to, uint256 _amount) internal {
    if (!allowedTokens[_asset]) revert TokenNotAllowed();
    _burn(msg.sender, _amount);
    ERC20Upgradeable(_asset).safeTransfer(_to, _amount);
    emit Withdraw(_asset, msg.sender, _to, _amount);
}
```

The only guards are (1) `_asset` is in `allowedTokens` and (2) the caller holds enough wrsETH to burn. There is no per-token deposit ledger and no check that `_asset` matches the token the caller deposited.

**Enabling condition — `maxAmountToDepositBridgerAsset` (L99–110):**

```solidity
return wrsETHSupply - balanceOfAssetInWrapper;
```

This formula computes available bridger capacity using only the balance of the single asset being deposited, ignoring all other assets already in the wrapper. When tokenA is already held (backing existing wrsETH supply), the formula returns `totalSupply() - 0 = totalSupply()` for tokenB, allowing the bridger to deposit N tokenB against N wrsETH that is already fully backed by tokenA.

**Exploit path:**

1. userA calls `deposit(tokenA, N)` → `tokenA.balanceOf(wrapper) = N`, `totalSupply() = N`
2. Bridger calls `depositBridgerAssets(tokenB, N)`:
   - `maxAmountToDepositBridgerAsset(tokenB)` = `N - 0 = N` → passes
   - `tokenB.balanceOf(wrapper) = N`; `totalSupply()` unchanged = N
3. userA calls `withdraw(tokenB, N)`:
   - `allowedTokens[tokenB]` = true ✓
   - `_burn(userA, N)` ✓
   - `tokenB.safeTransfer(userA, N)` ✓

**Final state:**
- `tokenA.balanceOf(wrapper) = N` — permanently frozen
- `tokenB.balanceOf(wrapper) = 0` — drained
- `totalSupply() = 0` — no wrsETH left to redeem tokenA

tokenA has no recovery path: `_withdraw` requires burning wrsETH, supply is zero, and there is no admin rescue or sweep function anywhere in the contract.

## Impact Explanation

**Critical — Permanent freezing of funds.**

tokenA deposited by userA is irrecoverable. The invariant `sum(asset.balanceOf(wrapper)) == totalSupply()` is broken: N tokens remain in the wrapper with 0 wrsETH supply. No function in the contract can recover them.

## Likelihood Explanation

All preconditions are within normal operational flow:

- A second allowed token is an explicitly supported upgrade path via `reinitialize` (L47–49) and `addAllowedToken` (L174–176).
- The bridger depositing via `depositBridgerAssets` is the documented legacy bridge flow (L157–170).
- The user calling `withdraw(tokenB, N)` requires no special role, no front-running, and no oracle manipulation — only knowledge that tokenB is present in the wrapper.

The bridger is not acting maliciously; their deposit is a normal operational action. The exploit trigger is an unprivileged `withdraw` call.

## Recommendation

1. **Per-token accounting in `_withdraw`**: Maintain a `depositedBalance[token]` mapping. Require `depositedBalance[_asset] >= _amount` before transferring and decrement on withdrawal.

2. **Fix `maxAmountToDepositBridgerAsset`**: Replace `totalSupply() - singleAsset.balanceOf(wrapper)` with `totalSupply() - sumOfAllAssetBalances()` to prevent the bridger from double-collateralizing wrsETH already backed by another asset.

Alternatively, if all allowed tokens are intended to be economically equivalent (all pegged 1:1 to canonical rsETH), enforce the invariant `sum(asset.balanceOf(wrapper)) == totalSupply()` at all entry points and gate `depositBridgerAssets` accordingly.
