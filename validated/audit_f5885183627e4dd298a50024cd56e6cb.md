Audit Report

## Title
Aave Interest Trapped in `_withdrawFromAave` Causes Temporary ETH Withdrawal Freeze - (File: contracts/LRTWithdrawalManager.sol)

## Summary
`_withdrawFromAave` caps withdrawals at `totalETHDepositedToAave` (principal only). Once all principal is consumed through sequential `completeWithdrawal` calls, `totalETHDepositedToAave` reaches zero while accrued aWETH interest remains in Aave. Any subsequent user withdrawal requiring Aave liquidity silently receives 0 ETH and reverts with `InsufficientLiquidityForWithdrawal`, temporarily freezing those users' funds even though ETH is available in Aave.

## Finding Description
In `_withdrawFromAave` ( [1](#0-0) ), the principal cap is computed as:

```solidity
uint256 withdrawablePrincipal = aaveBalance < totalETHDepositedToAave ? aaveBalance : totalETHDepositedToAave;
withdrawnAmount = amount > withdrawablePrincipal ? withdrawablePrincipal : amount;
if (withdrawnAmount == 0) return 0;
```

When `totalETHDepositedToAave == 0` but `aaveAWETH.balanceOf(address(this)) > 0` (interest only remains), `withdrawablePrincipal = min(aaveBalance, 0) = 0`, so `withdrawnAmount = 0` and the function returns silently without reverting.

The caller `_processWithdrawalCompletion` ( [2](#0-1) ) then checks:

```solidity
uint256 balanceAfter = address(this).balance;
if (balanceAfter < request.expectedAssetAmount) {
    revert InsufficientLiquidityForWithdrawal();
}
```

Since `_withdrawFromAave` returned 0, `balanceAfter` is unchanged and the revert fires, blocking the user's withdrawal.

The lifecycle producing this state:
1. `unlockQueue` deposits ETH to Aave via `_depositToAave`, incrementing `totalETHDepositedToAave`. [3](#0-2) 
2. Users call `completeWithdrawal`; each call decrements `totalETHDepositedToAave` via `_withdrawFromAave`. [4](#0-3) 
3. After all principal is consumed, `totalETHDepositedToAave == 0`. Aave still holds accrued interest (aWETH balance > 0), but `_withdrawFromAave` returns 0 for any amount.
4. The only recovery path is `_collectInterestToTreasury`, which sends interest to the protocol treasury — not to the withdrawal manager — so it cannot serve pending user withdrawals. [5](#0-4) 

## Impact Explanation
Users with unlocked ETH withdrawal requests whose `expectedAssetAmount` exceeds the contract's idle ETH balance cannot complete their withdrawals. Their rsETH has already been burned at `unlockQueue` time and the ETH they are owed is locked in Aave as interest, inaccessible to them. This constitutes a **temporary freeze of user funds** (a valid Medium impact) until the operator runs another `unlockQueue` cycle to replenish `totalETHDepositedToAave`.

## Likelihood Explanation
The Aave integration is an active, enabled feature. As users progressively complete withdrawals, `totalETHDepositedToAave` monotonically decreases toward zero while interest silently accumulates — this is normal operating behavior, not an edge case. Any deployment where the Aave integration is enabled and withdrawal volume is high will eventually reach this state. The trigger is a standard unprivileged `completeWithdrawal` call by any user.

## Recommendation
In `_withdrawFromAave`, when `totalETHDepositedToAave` is zero but `aaveBalance > 0`, allow withdrawal of the full `aaveBalance` (treating accrued interest as available for user withdrawals). Alternatively, modify `_processWithdrawalCompletion` to call `_collectInterestToTreasury` before `_withdrawFromAave` so that interest is converted to contract ETH and available for the user. A simpler fix is to remove the principal cap entirely in `_withdrawFromAave` and allow withdrawal up to the full `aaveBalance`, tracking interest accounting separately.

## Proof of Concept
1. Operator calls `unlockQueue(ETH_TOKEN, ...)` — 100 ETH moves from `LRTUnstakingVault` to `LRTWithdrawalManager`, then is deposited to Aave. `totalETHDepositedToAave = 100`.
2. Aave accrues 2 ETH interest. `aaveAWETH.balanceOf(withdrawalManager) = 102`.
3. Users call `completeWithdrawal` repeatedly. Each call triggers `_withdrawFromAave`, decrementing `totalETHDepositedToAave`. After 100 ETH of principal is consumed, `totalETHDepositedToAave = 0`, `aaveBalance = 2`.
4. Next user calls `completeWithdrawal(ETH_TOKEN, ...)` for 1 ETH. Contract ETH balance = 0.
5. `_processWithdrawalCompletion` calls `_withdrawFromAave(1)`.
6. `withdrawablePrincipal = min(2, 0) = 0` → `withdrawnAmount = 0` → returns 0.
7. `balanceAfter = 0 < 1` → reverts `InsufficientLiquidityForWithdrawal`.
8. The user's withdrawal is frozen. The 2 ETH of interest in Aave can only be sent to treasury via `_collectInterestToTreasury`, not to the user.
