### Title
Token-2022 MetadataPointer Allows Cross-Mint Metadata Spoofing in `log_metadata` — (`solana/programs/bridge_token_factory/src/instructions/user/log_metadata.rs`)

---

### Summary

`parse_metadata_account` in the Token-2022 third-party metadata path verifies only that the passed metadata account is owned by MetaplexID. It does **not** verify that the Metaplex `MetadataAccount.mint` field matches the mint being logged. An unprivileged attacker can point a Token-2022 mint's `MetadataPointer` at a legitimate Metaplex metadata account belonging to a *different* mint (one the attacker controls and named arbitrarily), causing `log_metadata` to post a Wormhole message to NEAR with attacker-chosen `name` and `symbol` for an arbitrary native token.

---

### Finding Description

The `process()` function in `log_metadata.rs` handles three Token-2022 cases: [1](#0-0) 

When the `MetadataPointer` points to an external address (not the mint itself, not `Pubkey::default()`), it calls `parse_metadata_account` with that address: [2](#0-1) 

`parse_metadata_account` performs exactly two checks:
1. The passed `metadata` account key equals the address from the `MetadataPointer` (line 78–82).
2. The account is owned by `MetaplexID` (line 83).

It then deserializes and returns `metadata.name` and `metadata.symbol` without ever checking that `metadata.mint == self.mint.key()`.

The Metaplex `MetadataAccount` struct contains a `mint` field (confirmed by the test helper's serialization layout): [3](#0-2) 

This field is the cryptographic binding between a Metaplex metadata account and its mint. The code ignores it entirely in the Token-2022 path.

By contrast, the classic SPL token path is safe because the metadata address is derived as a PDA seeded with the mint's pubkey, making it cryptographically bound: [4](#0-3) 

The `metadata` account in the `LogMetadata` accounts struct has no constraints at all: [5](#0-4) 

---

### Impact Explanation

The `LogMetadataPayload` posted to NEAR via Wormhole contains the attacker-controlled `name` and `symbol` alongside the real mint pubkey: [6](#0-5) 

NEAR receives a message asserting that a native Solana token (mint B's pubkey) has `name='USDC'`, `symbol='USDC'`. This is token-identity divergence: the on-chain token identity (mint B) is decoupled from the metadata NEAR registers for it. Downstream integrators, UIs, and NEAR-side token registries that rely on `name`/`symbol` for display or routing will treat mint B as USDC.

This falls under **High — Asset-identity, token-mapping divergence that breaks backing guarantees or sends value to the wrong party**.

---

### Likelihood Explanation

The attack requires no privileged access:
- Creating a Token-2022 mint with a `MetadataPointer` extension is a standard permissionless operation.
- Creating a Metaplex metadata account with arbitrary `name`/`symbol` for an attacker-owned mint is permissionless (the attacker is the update authority).
- Calling `log_metadata` is a public instruction with no access control beyond the `!mint.mint_authority.contains(authority.key)` check (which the attacker satisfies by using a non-bridge mint). [7](#0-6) 

The full attack is executable in a single session with no special tooling.

---

### Recommendation

After deserializing the `MplMetadata` in `parse_metadata_account`, add a check that the metadata account's `mint` field matches the mint being logged:

```rust
// In parse_metadata_account, after line 85:
let metadata = MplMetadata::try_deserialize(&mut data.as_ref())?;
require_keys_eq!(
    metadata.mint,
    self.mint.key(),
    ErrorCode::InvalidTokenMetadataAddress,
);
Ok((metadata.name.clone(), metadata.symbol.clone()))
``` [8](#0-7) 

---

### Proof of Concept

1. Attacker creates **mint A** (standard SPL token, attacker is mint authority).
2. Attacker calls Metaplex `create_metadata_accounts_v3` for mint A with `name = "USDC"`, `symbol = "USDC"`. This produces a valid Metaplex `MetadataAccount` owned by `MetaplexID`, with `mint = mint_A`.
3. Attacker creates **mint B** as a Token-2022 mint with a `MetadataPointer` extension whose `metadata_address` is set to the Metaplex metadata PDA of mint A.
4. Attacker calls `log_metadata` passing mint B as `mint` and mint A's Metaplex metadata account as `metadata`.
5. `parse_metadata_account` is called with `address = metadata_pointer.metadata_address.0` (mint A's metadata PDA):
   - `metadata.key() == address` → ✓ (attacker passed the correct account)
   - `metadata.owner == MetaplexID` → ✓ (it's a real Metaplex account)
   - `metadata.mint` is never checked
   - Returns `("USDC", "USDC")`
6. `LogMetadataPayload { token: mint_B, name: "USDC", symbol: "USDC", decimals: ... }` is posted via Wormhole to NEAR.

NEAR now registers mint B as a native token named "USDC" with symbol "USDC".
