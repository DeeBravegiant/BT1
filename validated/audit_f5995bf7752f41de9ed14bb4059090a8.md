Audit Report

## Title
Stale `rsETHPrice` Used in Withdrawal Amount Calculation Without Prior Oracle Update - (File: contracts/LRTWithdrawalManager.sol)

## Summary
`LRTWithdrawalManager.initiateWithdrawal` and `instantWithdrawal` compute the user's asset entitlement via `getExpectedAssetAmount`, which reads `LRTOracle.rsETHPrice` — a cached storage variable updated only when `updateRSETHPrice()` is explicitly called. Neither withdrawal function refreshes this value before use. When the cached price is stale (lower than the true current rate due to accrued staking rewards), users receive fewer underlying assets than the fair current value of their rsETH.

## Finding Description
`LRTOracle.rsETHPrice` is a `uint256` storage variable declared at line 28 of `contracts/LRTOracle.sol`. It is only updated when `_updateRsETHPrice()` is invoked, either via the public `updateRSETHPrice()` (L87–89) or the manager-gated `updateRSETHPriceAsManager()` (L94–96). No automatic invocation occurs inside the withdrawal path.

In `initiateWithdrawal` (L168, `contracts/LRTWithdrawalManager.sol`):
```solidity
uint256 expectedAssetAmount = getExpectedAssetAmount(asset, rsETHUnstaked);
```

In `instantWithdrawal` (L228, `contracts/LRTWithdrawalManager.sol`):
```solidity
uint256 assetAmountUnlocked = getExpectedAssetAmount(asset, rsETHUnstaked);
```

`getExpectedAssetAmount` (L593, `contracts/LRTWithdrawalManager.sol`) computes:
```solidity
underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset);
```

Because `rsETHPrice` reflects the state at the last `updateRSETHPrice()` call, any accrued EigenLayer staking rewards or LST appreciation since that call are not reflected. The `expectedAssetAmount` locked in at `initiateWithdrawal` time is what the user ultimately receives at `completeWithdrawal`; there is no re-computation at completion. For `instantWithdrawal`, the stale amount is transferred immediately. No existing guard in either function checks the freshness of `rsETHPrice` or forces an update before the calculation.

## Impact Explanation
**Low — Contract fails to deliver promised returns, but doesn't lose value.**

When `rsETHPrice` is stale and lower than the true current rate, users who call `initiateWithdrawal` or `instantWithdrawal` receive fewer underlying assets than the fair current value of their rsETH. The protocol retains the difference (accrued yield not yet reflected in the cached price). The protocol itself does not lose funds; only the withdrawing user is shortchanged relative to the true exchange rate at the time of their request.

## Likelihood Explanation
**Medium.** `updateRSETHPrice()` is a public function that must be called externally; it is not invoked atomically inside the withdrawal path. The protocol relies on off-chain keepers or manual calls to keep `rsETHPrice` fresh. Since staking rewards accrue continuously and oracle updates are periodic (not per-block), the stale-price condition is the norm between updates. Any unprivileged user can trigger `initiateWithdrawal` or `instantWithdrawal` at any time, and the condition requires no special setup beyond the natural passage of time between oracle refreshes.

## Recommendation
Call `updateRSETHPrice()` (or its internal equivalent `_updateRsETHPrice()`) at the beginning of both `initiateWithdrawal` and `instantWithdrawal`, before `getExpectedAssetAmount` is invoked:

```solidity
function initiateWithdrawal(address asset, uint256 rsETHUnstaked, string calldata referralId) external ... {
    ILRTOracle(lrtConfig.getContract(LRTConstants.LRT_ORACLE)).updateRSETHPrice();
    // ... existing logic ...
    uint256 expectedAssetAmount = getExpectedAssetAmount(asset, rsETHUnstaked);
    // ...
}
```

Apply the same fix to `instantWithdrawal`. This ensures the exchange rate used to compute the user's asset entitlement reflects all accrued yield up to the current block.

## Proof of Concept
1. At block N, `updateRSETHPrice()` is called. `rsETHPrice` is stored as `1.05e18`.
2. Staking rewards accrue over 24 hours. The true rsETH/ETH rate rises to `1.06e18`, but `updateRSETHPrice()` has not been called again.
3. A user holds 100 rsETH and calls `initiateWithdrawal(stETH, 100e18, "")`.
4. `getExpectedAssetAmount` computes: `100e18 * 1.05e18 / stETHPrice`. The stale `1.05e18` rate is used; `expectedAssetAmount` is locked in.
5. Had `updateRSETHPrice()` been called first, the rate would be `1.06e18` and the user would receive ~0.95% more stETH.
6. The user completes the withdrawal and receives the under-valued amount.

**Foundry fork test plan:** Fork mainnet at a block where `rsETHPrice` was last updated >1 hour ago. Call `getExpectedAssetAmount` with a known rsETH amount. Then call `updateRSETHPrice()` and call `getExpectedAssetAmount` again with the same inputs. Assert that the second result is strictly greater than the first, demonstrating the stale-price shortfall.
