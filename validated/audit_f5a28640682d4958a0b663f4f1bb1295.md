### Title
CR-CAT Coins Permanently Locked in Pending Approval State When Recipient's VC Is Revoked — (File: chia/wallet/vc_wallet/cr_cat_wallet.py)

### Summary
When CR-CATs (Credential-Restricted CATs) are sent to a recipient who does not own the sender's wallet, `CRCATWallet.generate_signed_transaction` wraps the output coin in a `construct_pending_approval_state` puzzle. This pending-state coin can only be claimed by spending it alongside a valid VC that makes the required announcement. If the recipient's VC is revoked by the authorized provider after the CR-CATs are sent but before they are claimed, the CR-CATs become permanently locked on-chain with no recovery path for either party.

### Finding Description
In `CRCATWallet.generate_signed_transaction`, every outgoing payment whose destination puzzle hash is not already in the sender's puzzle store is silently redirected to the `construct_pending_approval_state` wrapper:


The wrapper puzzle is:

```python
# chia/wallet/vc_wallet/cr_cat_drivers.py  lines 165-166
def construct_pending_approval_state(puzzle_hash: bytes32, amount: uint64) -> Program:
    return PENDING_VC_ANNOUNCEMENT.curry(Program.to([[51, puzzle_hash, amount, [puzzle_hash]]]))
```

`PENDING_VC_ANNOUNCEMENT` (`CONDITIONS_W_FEE_ANNOUNCE`) is the inner puzzle of a CR-CAT coin. Spending it requires the CR-CAT layer to receive a valid VC announcement — i.e., a live VC whose `authorized_providers` and `proofs_checker` flags match the CR-CAT's parameters. The only code path that spends these pending coins is `claim_pending_approval_balance`:

```python

    vc.wrap_inner_with_backdoor().get_tree_hash(),
```

There is no alternative spend path, no timeout, and no sender-reclaim branch in the pending-state puzzle. If the recipient's VC is revoked via `VerifiedCredential.activate_backdoor` (a legitimate provider action defined in `chia/wallet/vc_wallet/vc_drivers.py` lines 746-787), the required announcement can never be produced, and the CR-CAT coin is permanently unspendable. ### Impact Explanation
**High.** The sender's CR-CAT coins are irrevocably spent the moment the transaction is confirmed. The resulting pending-state coin encodes only the recipient's puzzle hash and the VC-announcement requirement; there is no sender-recovery branch. Any CR-CATs in flight at the time of VC revocation are permanently frozen on-chain, constituting a direct, irreversible accounting loss of CAT assets — matching the "permanent inability to access CAT assets" category in the allowed High impact scope. [5](#0-4) 

### Likelihood Explanation
**Medium.** VC revocation (`activate_backdoor`) is an intentional, documented feature exercised by authorized providers. The window between a CR-CAT send and the recipient's `crcat_approve_pending` call can be arbitrarily long (the recipient must manually trigger the claim). Any revocation event during that window permanently locks the coins. The scenario is directly analogous to the L2→L1 bridge delay in the reference report. [6](#0-5) 

### Recommendation
1. **Add a sender-reclaim branch** to `construct_pending_approval_state`: after a configurable timelock, allow the original sender's puzzle hash (embedded as a memo or curried argument) to reclaim the coin without a VC announcement.
2. **Alternatively**, before broadcasting the spend, check on-chain whether the recipient's VC is still valid and revert if it has been revoked.
3. **At minimum**, document the irrecoverable-lock risk prominently so callers can implement application-level safeguards. [2](#0-1) 

### Proof of Concept
1. Alice calls `CRCATWallet.generate_signed_transaction(amounts=[100], puzzle_hashes=[bob_ph], ...)`. Because `bob_ph` is not in Alice's puzzle store, the output is wrapped: `construct_pending_approval_state(bob_ph, 100)`. Alice's CR-CAT coin is spent and confirmed on-chain.
2. The authorized provider calls `VerifiedCredential.activate_backdoor(provider_innerpuzhash)` on Bob's VC, revoking it. The VC singleton is now in a revoked state and can no longer produce the required puzzle announcement.
3. Bob calls `crcat_approve_pending` → `claim_pending_approval_balance`. `get_vc_with_provider_in_and_proofs` returns `None` (no valid VC), raising `RuntimeError`. The claim fails.
4. Alice has no reclaim path: the pending-state puzzle contains only Bob's puzzle hash and the VC-announcement requirement. No spend bundle can satisfy it without a live VC.
5. The 100 CR-CATs remain permanently locked in the pending-state coin. [7](#0-6) [8](#0-7)

```python
    def activate_backdoor(
        self, provider_innerpuzhash: bytes32, announcement_nonce: bytes32 | None = None
    ) -> tuple[CreatePuzzleAnnouncement, CoinSpend]:
        """
        Activates the backdoor in the VC to revoke the credentials and remove the provider's DID.

        Returns the announcement we expect from the provider's DID authorizing this, and the spend of the VC.
        Sync attempts by this class on spends generated by this method are expected to fail. This could be improved in
        the future with a separate type/state of VC that is revoked, but perfectly useful as a singleton.
        """
        vc_solution: Program = solution_for_singleton(
            self.singleton_lineage_proof,
            uint64(self.coin.amount),
            Program.to(
                [  # solve EML
                    solve_revocation_layer(
                        self.hidden_puzzle(),
                        solve_std_vc_backdoor(
                            self.launcher_id,
                            Program.to((self.proof_provider, self.proof_hash)).get_tree_hash(),
                            self.construct_transfer_program().get_tree_hash(),
                            self.inner_puzzle_hash,
                            uint64(self.coin.amount),
                            self.eml_lineage_proof,
                            provider_innerpuzhash,
                            self.coin.name(),
                            announcement_nonce,
                        ),
                        hidden=True,
                    ),
                ]
            ),
        )

        expected_announcement: CreatePuzzleAnnouncement = CreatePuzzleAnnouncement(
            std_hash(self.coin.name() + Program.NIL.get_tree_hash() + ACS_TRANSFER_PROGRAM.get_tree_hash())
        )

        return (
            expected_announcement,
            make_spend(self.coin, self.construct_puzzle(), vc_solution),
        )
```
