### Title
Missing Domain Separator in `ForeignTxSignPayload::compute_msg_hash()` Enables Cross-Deployment Signature Replay — (`File: crates/near-mpc-contract-interface/src/types/foreign_chain.rs`)

---

### Summary

`ForeignTxSignPayload::compute_msg_hash()` computes `SHA-256(borsh(ForeignTxSignPayload))` with no domain separator — no NEAR contract address, no NEAR network/chain ID, and no domain ID are bound into the signed digest. A Byzantine MPC participant (strictly below the signing threshold) who possesses a valid `VerifyForeignTransactionResponse` obtained from any deployment sharing the same key can replay it verbatim against a different deployment of the MPC contract, causing the target contract to accept a foreign-chain attestation that its own nodes never produced.

---

### Finding Description

The signed payload for foreign-chain transaction verification is defined and hashed as follows:

```rust
// crates/near-mpc-contract-interface/src/types/foreign_chain.rs
pub struct ForeignTxSignPayloadV1 {
    pub request: ForeignChainRpcRequest,
    pub values: Vec<ExtractedValue>,
}

impl ForeignTxSignPayload {
    pub fn compute_msg_hash(&self) -> std::io::Result<Hash256> {
        let mut hasher = sha2::Sha256::new();
        borsh::BorshSerialize::serialize(self, &mut hasher)?;
        Ok(Hash256(hasher.finalize().into()))
    }
}
```

The hash is purely `SHA-256(borsh({request, values}))`. It contains no:
- NEAR contract account ID (e.g., `v1.signer.near`)
- NEAR network / chain ID (mainnet vs. testnet)
- Domain ID that selected the signing key
- Nonce or request-lifecycle identifier

The `respond_verify_foreign_tx` entry point in the contract verifies only that the ECDSA signature is valid over `response.payload_hash` under the domain's root public key. It does **not** recompute the expected hash from the pending request and compare:

```rust
// crates/contract/src/lib.rs  respond_verify_foreign_tx()
let payload_hash: [u8; 32] = response.payload_hash.0;
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,   // taken verbatim from the response, not recomputed
    &secp_pk,
)
.is_ok()
```

Additionally, `build_signature_request` in the node always uses a zero tweak for foreign-tx signing:

```rust
// crates/node/src/providers/verify_foreign_tx/sign.rs
Ok(SignatureRequest {
    ...
    tweak: Tweak::new([0u8; 32]),   // root key, no derivation
    ...
})
```

This means the signed digest is entirely determined by `(request, values)` — identical across every deployment that holds the same root key.

The existing "domain separation" described in `docs/foreign-chain-transactions.md` (lines 227–248) separates `Sign` domains from `ForeignTx` domains at the **contract routing level**, but it does not embed any deployment-specific context into the **signed bytes themselves**. A signature produced by one deployment is cryptographically indistinguishable from one produced by another deployment sharing the same key.

---

### Impact Explanation

**High — Forged foreign-chain verification / cross-deployment replay causing invalid bridge execution.**

A bridge contract that calls `verify_foreign_transaction()` and trusts the returned `VerifyForeignTransactionResponse` as proof that the MPC network on the *current* deployment attested to a foreign transaction is deceived. The replayed response was signed by nodes of a *different* deployment (e.g., testnet, a prior contract version during migration, or a parallel deployment). Those nodes may have observed entirely different on-chain state — a different block hash, a different confirmation count, or a transaction that does not exist on the target network at all. A bridge relying on this attestation to release funds or finalize a cross-chain transfer would execute based on fabricated evidence, enabling double-spend or theft of bridged assets.

---

### Likelihood Explanation

**Medium.** The attack requires two conditions:

1. **Same key reused across deployments.** This is realistic during: (a) testnet → mainnet migrations where the key is exported and re-imported; (b) contract upgrades where the key is reshared to a new account ID; (c) parallel deployments for load-balancing or disaster-recovery. The NEAR MPC system explicitly supports resharing and multi-deployment scenarios.

2. **Attacker is a Byzantine participant strictly below the signing threshold.** The `respond_verify_foreign_tx` function requires `assert_caller_is_attested_participant_and_protocol_active()`. A single malicious participant (1-of-n, below the reconstruction threshold t) can call this function unilaterally — no collusion is needed. The attacker does not forge the signature; they replay a legitimately produced one from the source deployment.

Neither condition requires threshold collusion, leaked keys, or physical TEE attacks.

---

### Recommendation

Bind the signed digest to the deployment context by including a domain separator in `compute_msg_hash()`. At minimum, include:

- The NEAR contract account ID (`env::current_account_id()` at request time, stored in the pending request)
- The NEAR network chain ID
- The `DomainId` used for signing

A concrete fix mirrors the EIP-712 approach: define a typed, versioned prefix that is prepended to the Borsh-serialized payload before hashing:

```rust
impl ForeignTxSignPayload {
    pub fn compute_msg_hash(
        &self,
        contract_id: &AccountId,
        domain_id: DomainId,
    ) -> std::io::Result<Hash256> {
        let mut hasher = sha2::Sha256::new();
        // Domain separator: "NEAR_MPC_FOREIGN_TX_V1" || contract_id || domain_id
        hasher.update(b"NEAR_MPC_FOREIGN_TX_V1:");
        hasher.update(contract_id.as_bytes());
        hasher.update(&domain_id.0.to_le_bytes());
        borsh::BorshSerialize::serialize(self, &mut hasher)?;
        Ok(Hash256(hasher.finalize().into()))
    }
}
```

The `domain_id` and `contract_id` must be stored in the pending request and passed through to both the node's `build_signature_request` and the contract's `respond_verify_foreign_tx` verification path.

### Proof of Concept

**Setup:** MPC contract deployed at `v1.signer.near` (mainnet) and `v1.signer.testnet` (testnet) sharing the same Secp256k1 root key `K` for their respective `ForeignTx` domains.

**Step 1 — Obtain a valid response on testnet:**
```
User calls: v1.signer.testnet::verify_foreign_transaction({
    domain_id: 0,
    payload_version: V1,
    request: BitcoinRpcRequest { tx_id: T, confirmations: 6, extractors: [BlockHash] }
})
```
Testnet MPC nodes query Bitcoin RPC, observe `block_hash = H_testnet`, compute:
```
msg_hash = SHA-256(borsh(ForeignTxSignPayload::V1 { request, values: [BlockHash(H_testnet)] }))
```
and return `VerifyForeignTransactionResponse { payload_hash: msg_hash, signature: σ }`.

**Step 2 — Replay on mainnet:**
Attacker (an attested participant on mainnet, below threshold) submits:
```
v1.signer.near::verify_foreign_transaction({ same request })
```
Then immediately calls:
```
v1.signer.near::respond_verify_foreign_tx(
    request = { domain_id: 0, ..., tx_id: T },
    response = { payload_hash: msg_hash, signature: σ }   // ← testnet response
)
```

**Step 3 — Contract accepts:**
`respond_verify_foreign_tx` fetches the mainnet domain-0 public key (same `K`), verifies `σ` over `msg_hash` — valid. The pending mainnet request is resolved and the response is returned to the bridge caller.

**Result:** The mainnet bridge contract receives a `VerifyForeignTransactionResponse` asserting `block_hash = H_testnet` for Bitcoin tx `T`, a value that mainnet MPC nodes never independently verified. If `H_testnet ≠ H_mainnet` (different block hashes for the same tx_id across networks), the bridge has accepted a forged attestation.
