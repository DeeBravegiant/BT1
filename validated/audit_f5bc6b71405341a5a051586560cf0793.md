### Title
Missing CKD Output Validation for `AppPublicKey` Variant Allows Single Byzantine Participant to Forge Confidential Key Material - (File: `crates/contract/src/lib.rs`)

### Summary
`respond_ckd()` performs no cryptographic output check when the request uses the `AppPublicKey` (legacy) variant. A single attested-but-Byzantine participant can call `respond_ckd` with arbitrary `big_y` / `big_c` BLS12-381 G1 points, and the contract will accept and deliver the forged confidential key to the user. This is the direct analog of the Omnipool sandwich attack: just as `swapForGem` blindly accepts any exchange rate via infinite Balancer limits, `respond_ckd` blindly accepts any key material for the `AppPublicKey` path.

### Finding Description

`respond_ckd` in `crates/contract/src/lib.rs` contains a match on the request's `app_public_key` variant:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← NO CHECK
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV`, the contract calls `ckd_output_check`, which verifies the pairing equation `e(big_c, g2) = e(big_y, app_pk2) · e(hash_point, public_key)` using BLS12-381 pairings: [2](#0-1) 

For `AppPublicKey` (the legacy, widely-used variant documented in the README), the arm is a no-op `{}`. Any `CKDResponse` with any `big_y` and `big_c` values is accepted unconditionally and forwarded to the user via `resolve_yields_for`. [3](#0-2) 

By contrast, `respond` (for ECDSA/EdDSA signatures) always verifies the signature cryptographically before resolving the yield: [4](#0-3) 

The `AppPublicKey` variant is the default/legacy format described in the contract README and is the format used in the E2E test `ckd_response__passes_cryptographic_verification`: [5](#0-4) 

### Impact Explanation

A single Byzantine attested participant (strictly below the signing threshold) can call `respond_ckd` with a forged `CKDResponse{big_y: attacker_point, big_c: attacker_point}` for any pending `AppPublicKey` CKD request. The contract resolves the yield and delivers the forged key material to the user. Because the `AppPublicKey` variant is "privately verifiable" (the user must hold the corresponding BLS scalar to check correctness), the user cannot detect the forgery on-chain. Any downstream use of the forged confidential key — encrypting secrets, deriving further keys, or unlocking bridge flows — is under the attacker's control.

This matches the allowed impact: **Critical — Unauthorized confidential key derivation output without the required participant authorization**, and **Critical — Bypass of threshold-signature requirements** (threshold > 1 is required, but a single participant suffices to forge the output).

### Likelihood Explanation

The `AppPublicKey` variant is the primary, documented, and default CKD format. Any single attested MPC participant who turns Byzantine can exploit this. The attacker does not need to collude with other participants, break TEE attestation, or perform any network-level attack. The only prerequisite is holding a valid TEE attestation, which every participant already has.

### Recommendation

Apply the same pairing-based output check to the `AppPublicKey` path. Since `AppPublicKey` provides only `pk1` (a G1 point), the check must be adapted: the contract should verify that `e(big_c, g2) = e(big_y, pk1_as_g2_equivalent) · e(hash_point, public_key)`. Alternatively, require all new CKD requests to use `AppPublicKeyPV` (which supplies both `pk1` and `pk2` and already has a check), and reject `AppPublicKey` requests at the contract level.

### Proof of Concept

1. User submits `request_app_private_key` with `AppPublicKey(some_g1_point)` and `domain_id` pointing to the BLS12-381 CKD domain.
2. The contract queues the request in `pending_ckd_requests`.
3. A single Byzantine attested participant constructs a forged response:
   ```rust
   let forged_response = CKDResponse {
       big_y: Bls12381G1PublicKey([1u8; 48]),  // attacker-chosen G1 point
       big_c: Bls12381G1PublicKey([2u8; 48]),  // attacker-chosen G1 point
   };
   ```
4. The participant calls `respond_ckd(request, forged_response)`. The contract hits the `AppPublicKey(_) => {}` arm — no check — and calls `resolve_yields_for`, delivering the forged `(big_y, big_c)` to the user.
5. The user receives a confidential key entirely under the attacker's control. The existing unit test `respond_ckd__should_succeed_when_response_is_valid_and_request_exists` already demonstrates this path accepts `big_y: [1u8; 48], big_c: [2u8; 48]` without any cryptographic validation: [6](#0-5)

/// Check that `e(big_c, g2) = e(big_y, app_pk2) . e(hash_point, public_key)`.
///
/// Point validation is fully delegated to the host, as in
/// [`app_public_key_check`].
pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}

/// Verify that a CKD response (AppPublicKey variant) is mathematically correct.
#[tokio::test]
#[expect(non_snake_case)]
async fn ckd_response__passes_cryptographic_verification() {
    // given
    let (cluster, running) =
        common::must_setup_cluster(common::CKD_VERIFICATION_PORT_SEED, |_| {}).await;

    let bls_domain = running
        .domains
        .domains
        .iter()
        .find(|d| {
            Curve::from(d.protocol) == Curve::Bls12381 && matches!(d.purpose, DomainPurpose::CKD)
        })
        .expect("no Bls12381 CKD domain found")
        .clone();

    let mpc_pk = common::must_get_bls_public_key(&running, bls_domain.id);
    let user = cluster.default_user_account().clone();

    let mut rng = rand::rngs::StdRng::seed_from_u64(1);
    let private_key = Scalar::random(&mut rng);
    let app_public_key = CKDAppPublicKey::AppPublicKey(Bls12381G1PublicKey::from(
        &(G1Projective::generator() * private_key),
    ));

    // when
    let outcome = cluster
        .send_ckd_request(bls_domain.id, app_public_key, &user)
        .await
        .expect("CKD request transaction failed");

    // then
    assert!(
        outcome.is_success(),
        "CKD request failed: {:?}",
        outcome.failure_message()
    );

    let response: serde_json::Value = outcome.json().expect("failed to deserialize CKD response");
    let big_y: Bls12381G1PublicKey =
        serde_json::from_value(response["big_y"].clone()).expect("failed to parse big_y");
    let big_c: Bls12381G1PublicKey =
        serde_json::from_value(response["big_c"].clone()).expect("failed to parse big_c");

    assert!(
        verify_ckd(&user, DERIVATION_PATH, &mpc_pk, private_key, &big_y, &big_c)
            .expect("verify_ckd failed"),
        "CKD response failed cryptographic verification"
    );
}
