Audit Report

## Title
Single Broken Extension Permanently Freezes `removeLiquidity`, Locking LP Funds — (`metric-core/contracts/ExtensionCalling.sol`, `metric-core/contracts/libraries/CallExtension.sol`)

## Summary
`CallExtension.callExtension` re-raises every revert from an extension contract with no isolation. Because all seven extension slots are `internal immutable` values set once at construction and never replaceable, any extension registered in `BEFORE_REMOVE_LIQUIDITY_ORDER` or `AFTER_REMOVE_LIQUIDITY_ORDER` that begins reverting permanently blocks every LP from calling `removeLiquidity`. The `pauseLevel` mechanism provides no relief because `removeLiquidity` carries no `whenNotPaused` guard.

## Finding Description
`_callExtensionsInOrder` iterates the packed order word and calls each extension through `CallExtension.callExtension`:

```solidity
// ExtensionCalling.sol L75-86
function _callExtensionsInOrder(uint256 order, bytes memory data) private {
    if (order == 0) return;
    while (true) {
        uint256 extensionIndex = order & 0x7;
        if (extensionIndex == 0) break;
        address extension = _extensionAddress(extensionIndex);
        if (extension == address(0)) revert PanicEmptyExtension();
        CallExtension.callExtension(extension, data);   // no try/catch
        order >>= 3;
    }
}
```

`callExtension` re-raises the revert verbatim:

```solidity
// CallExtension.sol L8-17
function callExtension(address extension, bytes memory data) internal {
    (bool success, bytes memory result) = extension.call(data);
    if (!success) {
        if (result.length > 0) {
            assembly ("memory-safe") { revert(add(result, 32), mload(result)) }
        }
        revert ExtensionCallFailed();
    }
    ...
}
```

`removeLiquidity` calls both hooks through this path and carries no `whenNotPaused` modifier:

```solidity
// MetricOmmPool.sol L199-212
function removeLiquidity(...) external nonReentrant(PoolActions.REMOVE_LIQUIDITY) ... {
    ...
    _beforeRemoveLiquidity(msg.sender, owner, salt, deltas, extensionData);
    (amount0Removed, amount1Removed) = LiquidityLib.removeLiquidity(...);
    _afterRemoveLiquidity(msg.sender, owner, salt, deltas, amount0Removed, amount1Removed, extensionData);
}
```

Compare with `swap`, which does carry `whenNotPaused`:

```solidity
// MetricOmmPool.sol L224
function swap(...) external whenNotPaused nonReentrant(PoolActions.SWAP) ...
```

The seven extension addresses are `internal immutable` and set once in the constructor with no upgrade path. Once an extension begins reverting — due to an external oracle dependency being paused, an upgradeable proxy receiving a bad upgrade, or a latent bug triggered by pool state — every subsequent `removeLiquidity` call reverts. LP shares remain in `_positionBinShares` and token balances remain in `binTotals`, but no withdrawal path exists. Pausing the pool at any level does not help because `_checkNotPaused` is never invoked inside `removeLiquidity`. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5) 

## Impact Explanation
LP principal is permanently inaccessible. `binTotals.scaledToken0` and `binTotals.scaledToken1` continue to account for the locked tokens, but no withdrawal function exists. The pool's real ERC-20 balances exceed what can ever be claimed, constituting pool insolvency from the LP perspective. This matches the allowed impact: "Broken core pool functionality causing loss of funds or unusable withdraw/swap/liquidity flows" and "Pool insolvency: balances fail to cover LP claims."

## Likelihood Explanation
Extensions are external contracts. Realistic, non-malicious failure modes include: an extension that consults an external oracle or allowlist that is later paused or decommissioned; an extension implemented as an upgradeable proxy that receives a bad upgrade; or a latent bug triggered only after a specific `curBinIdx` value or cumulative volume threshold is reached. None of these require the pool admin to act maliciously. Because extensions are immutable, there is zero recovery path once the failure begins.

## Recommendation
1. Wrap `_beforeRemoveLiquidity` and `_afterRemoveLiquidity` calls in try/catch and emit a failure event rather than propagating the revert, so LPs retain a withdrawal path even when an extension is broken.
2. Add a `whenNotPaused`-style bypass for `removeLiquidity` that skips extension calls when `pauseLevel >= 2`, giving the protocol owner an emergency escape hatch.
3. Allow extension replacement via a timelocked factory action, analogous to the existing `proposePoolPriceProvider` / `executePoolPriceProviderUpdate` pattern.

## Proof of Concept
1. Deploy a pool with extension `E` registered in `BEFORE_REMOVE_LIQUIDITY_ORDER`.
2. LP Alice calls `addLiquidity`; tokens are deposited and `_binStates` / `binTotals` are updated.
3. Extension `E` begins reverting (e.g., its external oracle dependency is paused).
4. Alice calls `removeLiquidity`. Execution reaches `_beforeRemoveLiquidity` → `_callExtensionsInOrder` → `CallExtension.callExtension(E, ...)` → `E` reverts → `callExtension` re-raises the revert → `removeLiquidity` reverts.
5. Alice's shares remain in `_positionBinShares`; her tokens remain in `binTotals`. No alternative withdrawal function exists. Pausing the pool at level 1 or 2 does not help because `removeLiquidity` has no `whenNotPaused` guard.
6. The pool's ERC-20 balances permanently exceed what LPs can ever withdraw.
