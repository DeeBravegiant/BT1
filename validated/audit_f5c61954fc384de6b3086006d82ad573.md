Audit Report

## Title
LP Funds Permanently Locked When Position Owner Is USDC/USDT-Blacklisted in `removeLiquidity` — (File: metric-core/contracts/libraries/LiquidityLib.sol)

## Summary
`LiquidityLib.removeLiquidity` uses a push pattern that transfers owed tokens directly to `owner`. If `owner` is added to USDC's (or USDT's) blacklist after providing liquidity, every call to `removeLiquidity` reverts unconditionally. Because `MetricOmmPool.removeLiquidity` enforces `msg.sender == owner` with no delegation or recipient override, and positions have no transfer mechanism, the LP's principal is permanently locked with no on-chain recovery path.

## Finding Description
After computing scaled amounts across all bins, `LiquidityLib.removeLiquidity` pushes both tokens to `owner`:

```solidity
// LiquidityLib.sol lines 242-247
if (amount0Removed > 0) {
    IERC20(ctx.token0).safeTransfer(owner, amount0Removed);
}
if (amount1Removed > 0) {
    IERC20(ctx.token1).safeTransfer(owner, amount1Removed);
}
```

USDC's `transfer` reverts when either the sender or receiver is blacklisted. If `owner` is blacklisted, `safeTransfer(owner, ...)` reverts, rolling back all bin-state mutations computed in the loop above (lines 210–214: `token0BalanceScaled`, `token1BalanceScaled`, `binTotalShares`, `positionBinShares` are all written before the transfer but reverted with it).

`MetricOmmPool.removeLiquidity` enforces:
```solidity
// MetricOmmPool.sol line 206
if (msg.sender != owner) revert NotPositionOwner();
```

This prevents the LP from delegating withdrawal to a clean address. Positions are keyed by `keccak256(abi.encode(owner, salt, bin))` with no transfer or reassignment mechanism. There are no rescue, emergency withdrawal, or claimable-balance functions anywhere in the production contracts. The extension hooks (`beforeRemoveLiquidity`, `afterRemoveLiquidity`) receive `owner` and `amount0Removed`/`amount1Removed` but cannot redirect the token transfer, which occurs inside `LiquidityLib` before the after-hook fires.

A secondary effect: if token0 is USDC and the LP is blacklisted, the token0 transfer fails first, which also prevents the token1 (e.g., WETH) transfer — the LP loses access to both tokens even if token1 is not a blacklisting token.

## Impact Explanation
The LP's shares remain permanently recorded in `_positionBinShares` and `_binTotalShares` but can never be redeemed. The underlying token balances are locked inside the pool's bins indefinitely. This is a direct, permanent loss of user principal (both token0 and token1) with no on-chain recovery path. USDC and USDT are explicitly within scope as allowed non-standard ERC20 exceptions per the contest rules.

## Likelihood Explanation
USDC blacklisting is a documented real-world event (OFAC-sanctioned addresses, exchange hacks, regulatory actions). An LP can be blacklisted at any time after depositing — the pool has no mechanism to detect this condition in advance or route tokens to an alternative address after the fact. Likelihood is low but impact is total loss of principal, placing this at **Medium** severity under Sherlock's framework.

## Recommendation
Replace the push pattern with a pull pattern: record owed amounts in a claimable mapping and let the LP (or any address they authorize) call a separate `claim(recipient)` function that transfers to an arbitrary address. Alternatively, add an explicit `recipient` parameter to `removeLiquidity` distinct from `owner`, so a blacklisted LP can direct funds to a clean address. The `msg.sender == owner` ownership check can remain to authenticate the caller without forcing the transfer destination to equal `owner`.

## Proof of Concept
1. Alice calls `addLiquidity` on a USDC/WETH pool, creating position `(alice, salt, bin)` with shares recorded in `_positionBinShares`.
2. Alice's address is later added to USDC's blacklist.
3. Alice calls `removeLiquidity`; `msg.sender == owner == alice` passes the guard at `MetricOmmPool.sol:206`.
4. `LiquidityLib.removeLiquidity` computes `amount0Removed` (USDC) and `amount1Removed` (WETH), mutates bin state.
5. `IERC20(ctx.token0).safeTransfer(alice, amount0Removed)` reverts — USDC rejects transfers to blacklisted receivers.
6. The entire transaction reverts; all bin-state mutations are rolled back; Alice's shares in `_positionBinShares` are unchanged.
7. Alice cannot call `removeLiquidity` from any other address due to the `msg.sender != owner` guard.
8. No rescue or claimable-balance function exists in the production contracts.
9. Alice's USDC **and** WETH are permanently locked in the pool.

Foundry fork test: deploy pool with MockERC20 that implements a blacklist, add Alice as LP, blacklist Alice, call `pool.removeLiquidity(alice, salt, deltas, "")` from Alice — assert revert; assert `_positionBinShares` unchanged; assert pool token balances unchanged.
