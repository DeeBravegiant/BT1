### Title
Unrefunded Deposits on Proposal Eviction in `do_update` — (`crates/contract/src/update.rs`)

---

### Summary

When `ProposedUpdates::do_update` executes a winning proposal it calls `self.entries.clear()` and `self.vote_by_participant.clear()`, silently destroying every competing proposal. The NEAR deposits paid by the proposers of those evicted entries are never returned. The `bytes_used` field stored in each `UpdateEntry` was clearly intended to support refunds, but no refund path was ever implemented.

---

### Finding Description

`propose_update` (lib.rs:1298–1334) requires each proposer to attach a deposit equal to `ProposedUpdates::required_deposit(&update)`, which is `env::storage_byte_cost() × bytes_used`. Only the excess above that exact amount is refunded at proposal time: [1](#0-0) 

The required portion stays in the contract's NEAR balance. Each `UpdateEntry` records `bytes_used` so that a future refund could be calculated: [2](#0-1) 

When threshold votes accumulate for proposal A, `vote_update` calls `do_update`: [3](#0-2) 

Inside `do_update`, after removing the winning entry, the function unconditionally clears every remaining entry and every vote record: [4](#0-3) 

There is no loop over the cleared entries to issue `Promise::new(proposer).transfer(deposit)` before discarding them. The storage bytes are freed (NEAR reclaims them), but the NEAR tokens that were paid to cover that storage remain in the contract balance with no mechanism to recover them.

---

### Impact Explanation

Every participant who proposed a competing update loses their deposit permanently. For a contract-code proposal the deposit scales with the binary size: [5](#0-4) 

A typical MPC contract binary is hundreds of kilobytes; at NEAR's storage byte cost (~10 yoctoNEAR/byte) each deposit is on the order of several NEAR. Across repeated upgrade cycles with N competing proposals per cycle, the contract balance grows by `(N-1) × deposit` each time, permanently. The invariant broken is: *every deposit paid into `propose_update` must either be refunded when the proposal is superseded or consumed by actual storage*. After `entries.clear()` the storage obligation is gone but the tokens are not returned.

---

### Likelihood Explanation

Only registered participants can call `propose_update` (`voter_or_panic()` gate), so the depositors are MPC nodes, not arbitrary users. In any upgrade cycle where more than one participant independently proposes a different update — a realistic scenario given that participants may disagree on which binary to deploy — all but one deposit is lost. No collusion, no special privilege, and no operator misconfiguration is required; it is the normal execution path of the contract.

---

### Recommendation

Before calling `self.entries.clear()` in `do_update`, iterate over the remaining entries and issue a transfer back to each proposer for `required_deposit(bytes_used)`. The `bytes_used` field already stored in `UpdateEntry` provides the exact amount needed. Alternatively, store the proposer's `AccountId` alongside `bytes_used` in `UpdateEntry` so the refund target is unambiguous.

---

### Proof of Concept

1. Deploy the contract with threshold T and T+2 participants.
2. Have participant P1 call `propose_update(A)` with deposit D1; have participant P2 call `propose_update(B)` with deposit D2.
3. Have T participants call `vote_update(A)` until `do_update` fires.
4. Assert: P2's account balance has not increased by D2; the contract balance has increased by D2 relative to its storage obligations; `proposed_updates` is empty (storage freed) yet D2 is still held.

The existing sandbox test `test_proposed_updates_do_update_clears_all_state` already confirms that `entries` and `vote_by_participant` are fully cleared after `do_update`: [6](#0-5) 

That test verifies the clearing behaviour but does not assert any deposit refund, confirming the refund path is absent.

fn bytes_used(update: &Update) -> u128 {
    let mut bytes_used = std::mem::size_of::<UpdateEntry>() as u128;

    // Assume a high max of 128 participant votes per update entry.
    bytes_used += 128 * std::mem::size_of::<AccountId>() as u128;

    match update {
        Update::Contract(code) => {
            bytes_used += code.len() as u128;
        }
        Update::Config(config) => {
            let bytes = serde_json::to_vec(&config).unwrap();
            bytes_used += bytes.len() as u128;
        }
    }

    bytes_used
}

fn required_deposit(bytes_used: u128) -> NearToken {
    env::storage_byte_cost().saturating_mul(bytes_used)
}
