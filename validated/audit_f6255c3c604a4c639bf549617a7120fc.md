### Title
Vote Signatures Are Never Verified, Allowing Any Peer to Forge Validator Votes and Bypass BFT Quorum — (File: `crates/apollo_consensus/src/single_height_consensus.rs`)

---

### Summary

The `handle_vote` function accepts consensus votes from the network without ever verifying the cryptographic `signature` field carried in each `Vote` message. Because the only admission check is that the `voter` address appears in the committee list — and that address is a plain, attacker-controlled field — any connected peer can emit votes that impersonate every committee member, manufacture a quorum for any `proposal_commitment`, and drive the local node to a `decision_reached` call without genuine agreement from the real validators.

---

### Finding Description

`Vote` carries a `signature: RawSignature` field intended to authenticate the sender:

```rust
pub struct Vote {
    pub vote_type: VoteType,
    pub height: BlockNumber,
    pub round: Round,
    pub proposal_commitment: Option<ProposalCommitment>,
    pub voter: ContractAddress,   // ← attacker-controlled
    pub signature: RawSignature,  // ← never checked
}
```

`handle_vote` in `single_height_consensus.rs` explicitly defers verification with a `TODO`:

```rust
pub(crate) fn handle_vote(&mut self, vote: Vote) -> Requests {
    // TODO(Asmaa): verify the signature          // ← gap
    ...
    if !self.committee.members().iter().any(|s| s.address == vote.voter) {
        debug!("Ignoring vote from non validator: vote={:?}", vote);
        return VecDeque::new();
    }
    // vote accepted and forwarded to state machine
```

The committee-membership guard checks only the `voter` field, which the attacker sets freely. The `received_vote` deduplication in `state_machine.rs` keys on `(round, voter)`:

```rust
let key = (vote.round, vote.voter);
```

So a single peer can send one vote per distinct validator address and each is counted as a fresh, independent vote toward quorum. The codebase itself acknowledges the gap in `manager.rs`:

> "Since vote signatures are not yet verified, a peer can forge votes with arbitrary voter addresses; without this cap that would grow `future_votes` without bound and exhaust memory."

The cap mentioned there is a memory-only guard; it does not prevent a peer from forging exactly `quorum_size` votes (well within the cap) to manufacture a decision.

---

### Impact Explanation

**Broken invariant — BFT quorum integrity.** The safety property of the Tendermint-style BFT protocol used here is that a block is committed only after ≥ 2/3 of the weighted committee has genuinely precommitted to it. Without signature verification, a single malicious network peer can:

1. **Forge prevotes and precommits** for any `proposal_commitment` value by sending one vote per committee-member address.
2. **Drive `decision_reached`** on the local node for a commitment the real validators never agreed to.
3. **Cause a safety violation / fork** when combined with a malicious proposer who sends different proposals to different validators: each validator locally validates its own proposal, stores it in `valid_proposals`, and the attacker then forges the complementary quorum for each, causing two honest validators to commit different blocks at the same height.

The corrupted value is the `proposal_commitment` that the local node commits — it can be any value the attacker chooses, as long as it matches a proposal the local node has already validated (which the attacker can arrange by being the proposer or by replaying a legitimate proposal stream). This maps to **wrong committed block / state root** at the sequencer level.

---

### Likelihood Explanation

- **Trigger is unprivileged**: any peer reachable over the P2P layer can send `Vote` messages; no validator key is required.
- **Validator addresses are public**: committee membership is broadcast as part of the consensus protocol.
- **No cryptographic barrier**: the `signature` field is present but ignored; the attacker does not need to know any private key.
- **Explicitly acknowledged in production code**: the `TODO(Asmaa)` comment and the `manager.rs` note confirm this is a known, unresolved gap in the current production codebase.

---

### Recommendation

1. **Sign every outgoing vote** in the `broadcast` path using the node's validator private key before the message is sent over the network.
2. **Verify the signature in `handle_vote`** against the public key associated with `vote.voter` before the committee-membership check or any state-machine processing.
3. **Reject votes with invalid or missing signatures** rather than deferring to a `TODO`.
4. Until (1)–(3) are implemented, restrict vote ingestion to authenticated P2P sessions so that the `voter` field cannot be spoofed by an unauthenticated peer.

### Proof of Concept

```
// Attacker knows the committee for height H: [V1, V2, V3, V4] (4 validators, quorum = 3)
// Attacker connects as a regular P2P peer (no validator key needed).

for voter in [V1_address, V2_address, V3_address]:
    send Vote {
        vote_type: Precommit,
        height: H,
        round: 0,
        proposal_commitment: Some(target_commitment),  // any commitment in victim's valid_proposals
        voter: voter,          // impersonated — no signature check
        signature: [0u8; 64],  // ignored
    }

// handle_vote passes the committee-membership check for each voter address.
// received_vote sees three distinct (round, voter) keys → VoteStatus::New each time.
// State machine accumulates 3 precommits → quorum reached.
// decision_reached is called with target_commitment.
// Block is committed without any real validator having cast a vote.
```
