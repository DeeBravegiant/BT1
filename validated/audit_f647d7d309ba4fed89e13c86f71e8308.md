Audit Report

## Title
Missing Zero-Return Guard Before rsETH Burn in `instantWithdrawal` Allows Permanent Token Loss - (File: contracts/LRTWithdrawalManager.sol)

## Summary
`LRTWithdrawalManager.instantWithdrawal` computes `assetAmountUnlocked` via integer division and then unconditionally burns the caller's rsETH before verifying the computed return is non-zero. When `rsETHUnstaked * rsETHPrice < assetPrice`, the division truncates to zero, the rsETH is permanently destroyed, and the user receives no assets in return.

## Finding Description
At L228–229, the function computes the expected asset return and immediately burns rsETH without any zero-check on the result:

```solidity
uint256 assetAmountUnlocked = getExpectedAssetAmount(asset, rsETHUnstaked);
IRSETH(lrtConfig.rsETH()).burnFrom(address(msg.sender), rsETHUnstaked);
```

`getExpectedAssetAmount` at L593 performs:

```solidity
underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset);
```

This truncates to zero whenever `amount * rsETHPrice < assetPrice`. The only pre-burn guard at L224 is:

```solidity
if (rsETHUnstaked == 0 || rsETHUnstaked < minRsEthAmountToWithdraw[asset]) {
    revert InvalidAmountToWithdraw();
}
```

`minRsEthAmountToWithdraw[asset]` is a mapping that defaults to `0`, so the only enforced lower bound is `rsETHUnstaked != 0`. A call with `rsETHUnstaked = 1` passes this check. If the target asset's oracle price exceeds `rsETHPrice` (e.g., a high-value LST priced at `2e18` while `rsETHPrice = 1.05e18`), then `assetAmountUnlocked = 0`, the burn executes, and `userAmount = 0`. The subsequent `_transferAsset(asset, msg.sender, 0)` at L250 is a no-op.

## Impact Explanation
The contract burns the caller's rsETH and delivers zero underlying assets, directly contradicting the function's documented purpose. This maps to **Low — Contract fails to deliver promised returns**. While the rsETH principal is technically destroyed (making the loss real), the exploitable amount per call is bounded by the dust threshold (1 wei), limiting practical financial damage. The contract nonetheless fails its core accounting invariant: rsETH burned must correspond to a non-zero asset return.

## Likelihood Explanation
Under current asset configuration (ETH, stETH, ETHx all priced below rsETH), the truncation condition is not reachable. The scenario becomes reachable if: (1) a new asset is added whose oracle price exceeds `rsETHPrice`, and (2) `minRsEthAmountToWithdraw[asset]` is left at its default of `0`. Both are plausible during protocol expansion — condition (1) is a normal admin operation, and condition (2) is an easy oversight. Once these conditions hold, any unprivileged user can trigger the bug with a single public call.

## Recommendation
Add a zero-return guard immediately after computing `assetAmountUnlocked`, before the burn:

```solidity
uint256 assetAmountUnlocked = getExpectedAssetAmount(asset, rsETHUnstaked);
if (assetAmountUnlocked == 0) revert InvalidAmountToWithdraw();
IRSETH(lrtConfig.rsETH()).burnFrom(address(msg.sender), rsETHUnstaked);
```

Additionally, enforce a non-zero `minRsEthAmountToWithdraw[asset]` for every asset before enabling instant withdrawal, so that integer-division truncation to zero is structurally impossible regardless of oracle price relationships.

## Proof of Concept
1. Admin adds a new LST with oracle price `2e18`. `rsETHPrice = 1.05e18`.
2. Admin enables instant withdrawal for the asset; `minRsEthAmountToWithdraw[asset]` remains `0`.
3. User calls `instantWithdrawal(asset, 1, "")` (1 wei rsETH).
4. L228: `assetAmountUnlocked = 1 * 1.05e18 / 2e18 = 0` (integer truncation).
5. L229: `burnFrom(user, 1)` executes — 1 wei rsETH permanently destroyed.
6. L231 check: `0 > getAssetsAvailableForInstantWithdrawal(asset)` is false — no revert.
7. L237–238: `fee = 0`, `userAmount = 0`.
8. L250: `_transferAsset(asset, user, 0)` — user receives nothing.
9. Net: user loses 1 wei rsETH, receives zero assets.

Foundry fuzz test plan: fuzz `rsETHUnstaked` over `[1, 1e6]` with mocked oracle returning `assetPrice > rsETHPrice`; assert that `burnFrom` is never called when `getExpectedAssetAmount` returns `0`.
