### Title
Missing CKD Output Verification for `AppPublicKey` Variant Allows Single Byzantine Leader to Inject Fake Key Material - (File: `crates/contract/src/lib.rs`)

---

### Summary

In `respond_ckd`, the contract performs a cryptographic output check (`ckd_output_check`) only for `AppPublicKeyPV` requests. For the primary `AppPublicKey` variant, **no output verification is performed at all**. A single Byzantine attested participant acting as the MPC leader can call `respond_ckd` with an arbitrary `CKDResponse` (`big_y`, `big_c`), and the contract will accept and return it to the caller — bypassing the threshold-authorization requirement for confidential key derivation outputs.

---

### Finding Description

`respond_ckd` in `crates/contract/src/lib.rs` handles responses to `request_app_private_key` calls. The function branches on the request's `app_public_key` variant:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}   // ← NO CHECK
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV`, `ckd_output_check` verifies the BLS12-381 pairing equation `e(big_c, g2) = e(big_y, app_pk2) · e(hash_point, public_key)`, which cryptographically binds the response to the MPC network's master key and the caller's app identity. [2](#0-1) 

For `AppPublicKey`, the arm is an empty block — **any `CKDResponse` is accepted unconditionally**. The unit test at line 3424 confirms this: it passes `[1u8; 48]` and `[2u8; 48]` as `big_y`/`big_c` (not valid BLS12-381 points) and the call succeeds. [3](#0-2) 

The `AppPublicKey` variant is the primary/legacy variant documented in the contract README and used in production (the `AppPublicKeyPV` variant is the newer, publicly verifiable extension). [4](#0-3) 

`respond_ckd` is callable by any single attested participant — it does not require threshold cooperation: [5](#0-4) 

---

### Impact Explanation

The CKD protocol computes `C = S + y·A` where `S = H(pk ‖ app_id) · msk` is the secret derived from the MPC master key and `A = a·G` is the caller's ephemeral public key. The caller recovers the derived key as `C − a·Y`.

A Byzantine leader can submit a crafted response:
- Set `big_y = 0·G` (identity) and `big_c = K·G` for any attacker-chosen scalar `K`.
- The caller computes `K·G − a·0 = K·G`.
- The attacker knows `K`, so they know the caller's derived key.

This is **unauthorized confidential key derivation output without threshold authorization** — a single Byzantine participant below the signing threshold can fully control the derived key material returned to any `AppPublicKey` CKD requester.

Impact: **Critical** — matches "confidential key derivation output without the required participant authorization."

---

### Likelihood Explanation

`AppPublicKey` is the primary production variant. Any MPC leader node that turns Byzantine (e.g., compromised TEE, malicious operator) can exploit this for every pending `AppPublicKey` CKD request it leads. No threshold collusion is required — only the leader calls `respond_ckd`. The attacker-controlled entry path is direct: call `respond_ckd` with an arbitrary `CKDResponse` while being an attested participant.

---

### Recommendation

Since `AppPublicKey` omits `pk2` (the G2 component), the existing pairing check cannot be applied directly. Mitigations:

1. **Deprecate `AppPublicKey`** and require callers to use `AppPublicKeyPV`, which supports on-chain output verification.
2. **Require multiple independent `respond_ckd` submissions** (threshold-many) for `AppPublicKey` requests, with the contract only resolving the yield when a quorum of identical responses is received.
3. At minimum, **document explicitly** that `AppPublicKey` requests provide no on-chain output integrity guarantee and rely entirely on trusting the leader node.

---

### Proof of Concept

```rust
// Step 1: User submits AppPublicKey CKD request (primary/legacy variant)
let app_public_key = CKDAppPublicKey::AppPublicKey(user_bls_g1_public_key); // A = a·G
contract.request_app_private_key(CKDRequestArgs { app_public_key, derivation_path, domain_id });

// Step 2: Byzantine leader calls respond_ckd with a crafted fake response.
// No ckd_output_check is called for AppPublicKey — any bytes are accepted.
let fake_response = CKDResponse {
    big_y: dtos::Bls12381G1PublicKey([0u8; 48]), // identity point (0·G)
    big_c: dtos::Bls12381G1PublicKey(attacker_known_point_bytes), // K·G, K known to attacker
};
// This call SUCCEEDS — the empty match arm performs no verification:
contract.respond_ckd(ckd_request, fake_response).unwrap();

// Step 3: User receives fake derived key.
// derived_key = big_c − a·big_y = K·G − a·0 = K·G
// Attacker knows K → attacker knows derived_key → attacker can decrypt caller's secrets.
```

/// Check that `e(big_c, g2) = e(big_y, app_pk2) . e(hash_point, public_key)`.
///
/// Point validation is fully delegated to the host, as in
/// [`app_public_key_check`].
pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}
