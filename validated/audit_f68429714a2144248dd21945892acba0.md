### Title
Per-Transaction Reference Script Size Limit Bypassed for Phase-2 Invalid Transactions - (File: `eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Ledger.hs`)

---

### Summary

In the Conway era, the per-transaction reference script size limit (`maxRefScriptSizePerTx`, hardcoded at 200 KiB) is only enforced for phase-2 **valid** transactions (`IsValid True`). Phase-2 **invalid** transactions (`IsValid False`) skip this check entirely, allowing a single attacker-controlled transaction to reference up to the full per-block budget (~1 MiB) of reference scripts in one transaction — 5× the intended per-transaction cap.

---

### Finding Description

The ADR `docs/adr/2024-08-14_009-refscripts-fee-change.md` explicitly documents that the absence of reference script size limits in Babbage was exploited in a real DDoS attack on June 25, 2024. Conway introduced two hard limits to prevent recurrence:

- **Per-transaction**: 200 KiB (`maxRefScriptSizePerTx`, hardcoded in Conway, promoted to a protocol parameter in Dijkstra)
- **Per-block**: 1 MiB (`maxRefScriptSizePerBlock`)

The per-transaction check lives in `validateRefScriptSize` in `eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Ledger.hs`:

```haskell
validateRefScriptSize pp utxo tx =
  let totalRefScriptSize = txNonDistinctRefScriptsSize utxo tx
      maxRefScriptSizePerTx = fromIntegral @Word32 @Int $ pp ^. ppMaxRefScriptSizePerTxG
   in failureUnless (totalRefScriptSize <= maxRefScriptSizePerTx) $
        ConwayTxRefScriptsSizeTooBig ...
``` [1](#0-0) 

However, this check is only invoked inside the `IsValid True` branch of `conwayLedgerTransitionTRC`:

```haskell
(utxoState', certStateAfterCERTS) <-
  if tx ^. isValidTxL == IsValid True
    then do
      ...
      runTest $ validateRefScriptSize pp (utxoState ^. utxoL) tx
``` [2](#0-1) 

For `IsValid False` (phase-2 invalid) transactions, the entire `then` branch is skipped, and `validateRefScriptSize` is never called.

The per-block check in `validateBodyRefScriptsSizeTooBig` **does** count reference scripts from all transactions regardless of `IsValid`:

```haskell
accum (!accUtxo, !accSum) tx =
  ...
   in (updatedUtxo, accSum + txNonDistinctRefScriptsSize (UTxO accUtxo) tx)
``` [3](#0-2) 

So the per-block limit is respected, but the per-transaction limit is not enforced for invalid transactions.

---

### Impact Explanation

The per-transaction limit was introduced specifically to bound the deserialization and validation work a node must perform per transaction. By submitting a phase-2 invalid transaction, an attacker can reference up to ~1 MiB of reference scripts in a single transaction (bounded only by the block limit), forcing every validating node to deserialize 5× more script bytes per transaction than the protocol parameter was designed to allow. This exceeds the intended per-transaction validation resource limit.

**Impact class**: Medium — attacker-controlled transactions exceed intended validation limits (per-transaction reference script size cap).

---

### Likelihood Explanation

- The attacker must have previously placed large Plutus scripts into UTxO entries as reference scripts (a normal, permissionless operation).
- The attacker must craft a transaction that references those UTxO entries and is phase-2 invalid (e.g., a script that always fails, or an intentionally wrong redeemer).
- The attacker pays collateral on each such transaction, so there is a financial cost, but the cost is bounded by the collateral percentage of the fee — not by the amount of deserialization work imposed on nodes.
- This is a direct analog to the June 2024 DDoS attack vector that the per-tx limit was designed to close.

---

### Recommendation

Move `validateRefScriptSize` (or an equivalent check) outside the `IsValid True` guard so it applies to all transactions, including phase-2 invalid ones. The per-transaction reference script size limit should be enforced unconditionally, since the deserialization overhead is incurred regardless of whether the transaction ultimately succeeds or fails phase-2 validation.

---

### Proof of Concept

1. Deploy a large Plutus script (e.g., ~500 KiB) as a reference script in a UTxO entry.
2. Craft a transaction with `IsValid False` that references that UTxO entry via `referenceInputsTxBodyL`.
3. Submit the transaction with sufficient collateral.
4. The LEDGER rule skips `validateRefScriptSize` because `isValidTxL == IsValid False`.
5. The node deserializes ~500 KiB of reference scripts for this single transaction — 2.5× the 200 KiB per-tx cap — while the per-block check only fires if the block total exceeds 1 MiB.
6. Repeat with multiple such transactions per block (each under the block limit) to sustain elevated per-transaction deserialization load. [2](#0-1) [4](#0-3) [5](#0-4)

It was identified a while ago that there is an overhead associated with deserializing scripts, so much so that it would be possible to create a very large script that was fairly expensive to deserialize, but very cheap to execute. This opened up an attack vector when such a Plutus script would be used as a reference script. This problem was exacerbated by the fact that there was no real limit on the total size of reference scripts that could be used in a transaction, thus being limited only by the size of the transaction itself. Therefore this opened up Cardano to a DDoS attack where an attacker could submit many such transactions that would cost very little, but would be expensive for a `cardano-node` to validate.

In order to prevent such an attack a `"minFeeRefScriptCostPerByte"` protocol parameter was introduced in the Conway era. The idea was fairly simple: we would calculate the total size of reference scripts used by a transaction and multiply it by the value specified by this protocol parameter. Result would be added to the transaction fee. However, in order for this approach to be a definite deterrent of such attacks this parameter would have to be set to a fairly high value. Community was not very keen on having this value set to a high enough value, since that could make reference scripts almost as expensive to use as regular scripts, thus significantly reducing their usability.
