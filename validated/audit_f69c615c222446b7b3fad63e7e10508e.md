### Title
Proposer Deposits Permanently Locked When `do_update` Clears Competing Proposals - (File: `crates/contract/src/update.rs`)

### Summary
`propose_update` collects a NEAR storage deposit from each proposer. When `do_update` executes a winning proposal it calls `self.entries.clear()`, silently wiping every competing proposal from storage. The freed storage bytes return to the contract's own balance, but no refund is ever issued to the competing proposers. There is no withdrawal function for the contract's accumulated balance, so those NEAR tokens are permanently locked.

### Finding Description
`propose_update` is `#[payable]` and requires a deposit calculated by `ProposedUpdates::required_deposit`, which is `env::storage_byte_cost() * bytes_used`. For a contract-code update the `bytes_used` estimate includes the full wasm blob plus 128 × `size_of::<AccountId>()` bytes for anticipated votes. [1](#0-0) 

When `vote_update` reaches threshold it calls `do_update`: [2](#0-1) 

`do_update` removes the winning entry, then unconditionally clears **all** remaining entries and all votes: [3](#0-2) 

The freed storage bytes are credited back to the contract's own account balance by the NEAR runtime, but the contract has no function to transfer that balance back to the original depositors. There is no `withdraw_update_deposit` or equivalent anywhere in the contract. [4](#0-3) 

### Impact Explanation
Every participant who called `propose_update` for a proposal that was not the winning one permanently loses their storage deposit. For a contract-code update the deposit is proportional to the wasm size (up to ~4 MB), which at NEAR's storage cost of ≈1 NEAR per 10 KB amounts to hundreds of NEAR per proposal. Because the contract exposes no withdrawal path for its own balance, those tokens are frozen for the lifetime of the contract. This breaks the production accounting invariant that deposited funds are either consumed for their stated purpose or returned to the depositor.

### Likelihood Explanation
The scenario is routine: multiple participants independently propose different contract upgrades (e.g., different code versions or config values) before a threshold is reached. Exactly one proposal wins; all others are silently cleared by `entries.clear()` with no refund. Any participant who has ever called `propose_update` for a losing proposal is affected. No special attacker capability is required — the loss is triggered by the normal governance flow.

### Recommendation
Before calling `entries.clear()`, iterate over all remaining entries and issue a `Promise::new(proposer).transfer(deposit)` for each one. The proposer's account ID must be stored in `UpdateEntry` at proposal time (analogous to how `submit_participant_info` stores the account for refund purposes). Alternatively, allow proposers to explicitly withdraw their deposit for any proposal that has not yet been executed, similar to the existing `remove_update_vote` pattern. [5](#0-4) 

### Proof of Concept

1. Participant A calls `propose_update` with a 1 MB wasm blob, attaching ≈100 NEAR deposit. The deposit is accepted and the entry is stored.
2. Participant B calls `propose_update` with a different wasm blob, also attaching ≈100 NEAR.
3. Threshold participants vote for proposal A via `vote_update`.
4. `do_update` is called: it removes proposal A's entry, then calls `self.entries.clear()` — proposal B's entry is deleted from storage.
5. The NEAR runtime credits the freed storage bytes back to the contract's balance.
6. Participant B has no way to recover their ≈100 NEAR; the contract has no withdrawal function.
7. The contract's balance is permanently inflated by the locked deposit. [6](#0-5)

    pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
        let entry = self.entries.remove(id)?;

        // Clear all entries as they might be no longer valid
        self.entries.clear();
        self.vote_by_participant.clear();

        let mut promise = Promise::new(env::current_account_id());
        match entry.update {
            Update::Contract(code) => {
                // deploy contract then do a `migrate` call to migrate state.
                promise = promise.deploy_contract(code).function_call(
                    method_names::MIGRATE,
                    Vec::new(),
                    NearToken::from_near(0),
                    gas,
                );
            }
            Update::Config(config) => {
                // If we vote for a new config, we should use
                // the value `contract_upgrade_deposit_tera_gas` from the config
                // as the new gas value
                let new_config_gas_value = Gas::from_tgas(config.contract_upgrade_deposit_tera_gas);
                promise = promise.function_call(
                    method_names::UPDATE_CONFIG,
                    serde_json::to_vec(&(&config,)).unwrap(),
                    NearToken::from_near(0),
                    new_config_gas_value,
                );
            }
        }
        Some(promise)
    }

    /// Propose update to either code or config, but not both of them at the same time.
    #[payable]
    #[handle_result]
    pub fn propose_update(
        &mut self,
        #[serializer(borsh)] args: ProposeUpdateArgs,
    ) -> Result<UpdateId, Error> {
        // Only voters can propose updates:
        let proposer = self.voter_or_panic();
        let update: Update = args.try_into()?;

        let attached = env::attached_deposit();
        let required = ProposedUpdates::required_deposit(&update);
        if attached < required {
            return Err(InvalidParameters::InsufficientDeposit {
                attached: attached.as_yoctonear(),
                required: required.as_yoctonear(),
            }
            .into());
        }

        let id = self.proposed_updates.propose(update);

        log!(
            "propose_update: signer={}, id={:?}",
            env::signer_account_id(),
            id,
        );

        // Refund the difference if the proposer attached more than required.
        if let Some(diff) = attached.checked_sub(required)
            && diff > NearToken::from_yoctonear(0)
        {
            Promise::new(proposer).transfer(diff).detach();
        }

        Ok(id)
    }
