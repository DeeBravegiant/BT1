### Title
Gateway Admission Uses `BouncerConfig::max()`, Accepting Transactions the Block Builder Will Always Reject as `TransactionTooLarge` — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The gateway's stateful validation runs the `__validate__` entry point inside a `BlockContext` constructed with `BouncerConfig::max()`. This means the gateway never checks whether a transaction's actual execution resources (events, state-diff size, sierra gas, proving gas, etc.) exceed the real per-block capacity limits. The block builder, however, calls `verify_tx_weights_within_max_capacity` with the production `BouncerConfig` after every execution. A transaction whose resource footprint exceeds any single `block_max_capacity` dimension will be accepted by the gateway, admitted to the mempool, and then permanently rejected by the block builder with `TransactionTooLarge` — it can never be sequenced.

---

### Finding Description

**Gateway path — `BouncerConfig::max()` is used unconditionally**

In `run_validate_entry_point`, the gateway builds a `BlockContext` with `BouncerConfig::max()`:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs  line 325-330
let block_context = BlockContext::new(
    block_info,
    self.chain_info.clone(),
    versioned_constants,
    BouncerConfig::max(),   // ← all capacity fields set to usize::MAX / GasAmount::MAX
);
```

`BouncerConfig::max()` sets every `BouncerWeights` field to its maximum representable value (except `proving_gas`, which is kept at the default). [1](#0-0) 

The gateway then calls `StatefulValidator::validate(account_tx)`, which only runs the `__validate__` entry point — it never calls `verify_tx_weights_within_max_capacity`. [2](#0-1) 

**Block-builder path — real `BouncerConfig` is enforced**

`Transaction::execute_raw` (the path used by the block builder) calls `verify_tx_weights_within_max_capacity` with the production `block_context.bouncer_config` after every execution:

```rust
// crates/blockifier/src/transaction/transaction_execution.rs  line 167-176
verify_tx_weights_within_max_capacity(
    state,
    &tx_execution_summary,
    &tx_builtin_counters,
    &tx_execution_info.receipt.resources,
    &tx_state_changes_keys,
    &block_context.bouncer_config,   // ← real limits, e.g. n_events=5000
    &block_context.versioned_constants,
    tx_execution_info.receipt.gas.l2_gas,
)?;
``` [3](#0-2) 

The production limits (from `batcher_config.json`) are, for example:
- `n_events = 5 000`
- `state_diff_size = 4 000`
- `sierra_gas = 5 000 000 000`
- `l1_gas = 4 400 000` [4](#0-3) 

`verify_tx_weights_within_max_capacity` computes the full `BouncerWeights` for the transaction (including events, state-diff, sierra gas, proving gas, etc.) and returns `TransactionTooLarge` if any dimension exceeds `block_max_capacity`. [5](#0-4) 

**The gap**

The gateway never performs this check. A transaction that emits, say, 6 000 events, or writes a state diff of 5 000 entries, or consumes 6 × 10⁹ sierra gas will:

1. Pass stateless validation (resource-bound checks only cover `l2_gas.max_amount`, not event count or state-diff size). [6](#0-5) 
2. Pass stateful validation — `BouncerConfig::max()` means `verify_tx_weights_within_max_capacity` can never fail. [7](#0-6) 
3. Enter the mempool and be returned to the block builder.
4. Be permanently rejected by the block builder with `TransactionTooLarge` on every attempt.

The transaction is stuck: it passes every admission gate but can never be included in any block.

---

### Impact Explanation

**High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

Any unprivileged user who controls a contract that emits more than `n_events` events (or exceeds any other `block_max_capacity` dimension) can submit a transaction that the gateway accepts and the block builder permanently rejects. The transaction occupies a mempool slot indefinitely, the sender's nonce is never advanced, and the user receives no actionable error from the gateway. At scale, an attacker can flood the mempool with such transactions, degrading throughput for legitimate users.

---

### Likelihood Explanation

The trigger requires only a deployed contract that emits many events or writes many storage slots — both are normal contract patterns. No privileged access is needed. The discrepancy is structural: the gateway intentionally uses `BouncerConfig::max()` and intentionally skips `__execute__`, so the gap exists on every transaction submission.

---

### Recommendation

The gateway should enforce the same per-transaction capacity ceiling that the block builder enforces. Concretely:

1. Expose the production `BouncerConfig` (or at minimum `block_max_capacity`) to the gateway configuration.
2. After running `__validate__`, also run `__execute__` in simulation mode (or use the resource bounds declared in the transaction) to estimate the transaction's `BouncerWeights`.
3. Call `bouncer_config.within_max_capacity_or_err(tx_weights)` and reject the transaction with a clear error if it would always exceed the block capacity.

Alternatively, align the stateless validator's `max_l2_gas_amount` and add analogous per-transaction caps for `n_events` and `state_diff_size` so that transactions exceeding any single block-capacity dimension are rejected at the stateless layer before any execution occurs.

---

### Proof of Concept

1. Deploy a contract with an entry point that emits 6 000 events (above the `n_events = 5 000` block cap).
2. Submit an invoke transaction calling that entry point via the gateway.
3. **Observed**: gateway returns `OK` and a transaction hash; the transaction enters the mempool.
4. **Expected**: gateway should return an error indicating the transaction exceeds block capacity.
5. The block builder will call `verify_tx_weights_within_max_capacity` after executing the transaction, compute `n_events = 6 000 > 5 000`, and return `TransactionTooLarge`. The transaction is dropped and never sequenced. The sender's nonce is never advanced.

The root cause is the unconditional `BouncerConfig::max()` at [7](#0-6)  versus the real-limit check at [8](#0-7)  — the exact same structural gap as `maxWithdraw`/`maxRedeem` ignoring `mintCap`.

pub fn verify_tx_weights_within_max_capacity<S: StateReader>(
    state_reader: &S,
    tx_execution_summary: &ExecutionSummary,
    tx_builtin_counters: &CairoPrimitiveCounterMap,
    tx_resources: &TransactionResources,
    tx_state_changes_keys: &StateChangesKeys,
    bouncer_config: &BouncerConfig,
    versioned_constants: &VersionedConstants,
    receipt_l2_gas: GasAmount,
) -> TransactionExecutionResult<()> {
    let tx_weights = get_tx_weights(
        state_reader,
        &tx_execution_summary.executed_class_hashes,
        tx_execution_summary.visited_storage_entries.len(),
        tx_resources,
        tx_state_changes_keys,
        versioned_constants,
        tx_builtin_counters,
        bouncer_config,
        receipt_l2_gas,
    )?
    .bouncer_weights;

    bouncer_config.within_max_capacity_or_err(tx_weights)
}
