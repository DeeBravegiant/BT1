Audit Report

## Title
`_updateRsETHPrice` hard-reverts via `_checkAndUpdateDailyFeeMintLimit` when daily fee exceeds cap, freezing oracle price updates — (`contracts/LRTOracle.sol`)

## Summary

`LRTOracle._updateRsETHPrice()` computes `rsethAmountToMintAsProtocolFee` from live protocol rewards and passes it unconditionally to `_checkAndUpdateDailyFeeMintLimit()`, which performs a hard `revert DailyFeeMintLimitExceeded` when the fee amount exceeds `maxFeeMintAmountPerDay`. Both `updateRSETHPrice()` (public) and `updateRSETHPriceAsManager()` (manager-only) route through the same internal function with no bypass path, so when daily rewards legitimately exceed the configured cap — a condition reachable through normal TVL growth — every oracle update attempt reverts and `rsETHPrice` remains stale until an admin raises the cap.

## Finding Description

In `_updateRsETHPrice()` at line 301–303, the fee is computed and passed directly:

```solidity
uint256 rsethAmountToMintAsProtocolFee = protocolFeeInETH.divWad(newRsETHPrice);
_checkAndUpdateDailyFeeMintLimit(rsethAmountToMintAsProtocolFee);
``` [1](#0-0) 

Inside `_checkAndUpdateDailyFeeMintLimit()` at lines 205–207, the check is a hard revert with no capping or fallback:

```solidity
if (currentPeriodMintedFeeAmount + feeAmount > maxFeeMintAmountPerDay) {
    revert DailyFeeMintLimitExceeded(currentPeriodMintedFeeAmount + feeAmount, maxFeeMintAmountPerDay);
}
``` [2](#0-1) 

Because `currentPeriodMintedFeeAmount` is reset to `0` at the start of each new period (line 200), the effective daily check is simply `rsethAmountToMintAsProtocolFee > maxFeeMintAmountPerDay`. There is no partial-mint path, no skip-and-continue path, and no emit-only path — the entire `_updateRsETHPrice()` call reverts, leaving `rsETHPrice` unchanged. [3](#0-2) 

Both public entry points call the same internal function: [4](#0-3) 

An additional edge case: `maxFeeMintAmountPerDay` is a storage variable initialized to `0` by default. If it is never set via `setMaxFeeMintAmountPerDay()`, then `0 + feeAmount > 0` is true for any non-zero fee, making the oracle permanently non-functional from first deployment. [5](#0-4) 

## Impact Explanation

**Medium — Temporary freezing of funds.** While `rsETHPrice` is stale:

1. `LRTDepositPool.getRsETHAmountToMint()` uses the frozen stored price, so new depositors receive rsETH at an outdated rate, diluting existing holders.
2. `LRTWithdrawalManager.unlockQueue()` reads `lrtOracle.rsETHPrice()` to compute ETH payout amounts; a stale price causes withdrawers in the queue to receive incorrect (understated) ETH amounts — a concrete, quantifiable loss of value for queued withdrawers.
3. The freeze persists every day until `setMaxFeeMintAmountPerDay()` is called by the manager, meaning it is not self-healing.

## Likelihood Explanation

No attacker is required. The condition is reachable through normal protocol operation: the admin sets `maxFeeMintAmountPerDay` to a reasonable value at deployment, but as TVL grows (through ordinary user deposits), the daily staking rewards — proportional to TVL — eventually exceed the fixed cap. At 100,000 ETH TVL, 3–4% APR, and 10% protocol fee, the daily fee is approximately 0.8 ETH worth of rsETH. A conservatively set cap of 0.5 ETH worth of rsETH would trigger the revert on every subsequent `updateRSETHPrice()` call. Any unprivileged user calling the public `updateRSETHPrice()` function will observe the revert; no special permissions, price manipulation, or attacker involvement is needed.

## Recommendation

Replace the hard revert with a graceful cap. If the computed fee exceeds the remaining daily allowance, clamp it to the remaining limit and continue with the price update:

```solidity
uint256 remainingLimit = maxFeeMintAmountPerDay > currentPeriodMintedFeeAmount
    ? maxFeeMintAmountPerDay - currentPeriodMintedFeeAmount
    : 0;
rsethAmountToMintAsProtocolFee = rsethAmountToMintAsProtocolFee > remainingLimit
    ? remainingLimit
    : rsethAmountToMintAsProtocolFee;
_checkAndUpdateDailyFeeMintLimit(rsethAmountToMintAsProtocolFee);
```

This ensures the price update always succeeds; excess fee is discarded (or optionally carried forward). Additionally, ensure `maxFeeMintAmountPerDay` is set to a non-zero value during initialization or `reinitialize()`.

## Proof of Concept

1. Deploy with `maxFeeMintAmountPerDay = 0.5e18 / rsETHPrice` (e.g., 0.5 ETH worth of rsETH).
2. Accumulate 100,000 ETH TVL through normal deposits; `protocolFeeInBPS = 1000` (10%).
3. After one day, staking rewards ≈ 8 ETH; `protocolFeeInETH ≈ 0.8 ETH`; `rsethAmountToMintAsProtocolFee ≈ 0.8e18 / rsETHPrice`.
4. Call `updateRSETHPrice()` from any EOA.
5. `_checkAndUpdateDailyFeeMintLimit(0.8e18/price)` evaluates `0 + 0.8e18/price > 0.5e18/price` → `true` → `revert DailyFeeMintLimitExceeded`.
6. `updateRSETHPriceAsManager()` also reverts (same code path, lines 94–96).
7. `rsETHPrice` remains at yesterday's value. Withdrawal queue payouts use the stale price; depositors mint rsETH at the stale rate.
8. Condition repeats every day until `setMaxFeeMintAmountPerDay()` is called.

**Foundry test sketch:**
```solidity
function test_oracleDoS_feeLimitExceeded() public {
    // set cap below expected daily fee
    oracle.setMaxFeeMintAmountPerDay(0.5 ether);
    // simulate TVL growth to 100_000 ETH and advance 1 day
    vm.warp(block.timestamp + 1 days);
    // expect revert
    vm.expectRevert(LRTOracle.DailyFeeMintLimitExceeded.selector);
    oracle.updateRSETHPrice();
    // manager path also reverts
    vm.prank(manager);
    vm.expectRevert(LRTOracle.DailyFeeMintLimitExceeded.selector);
    oracle.updateRSETHPriceAsManager();
}
```
