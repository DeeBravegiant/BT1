### Title
Storage Deposit Paid to `propose_update` Is Permanently Locked — No Refund on Execution or Competing-Proposal Win - (File: `crates/contract/src/update.rs`)

---

### Summary

`propose_update` requires a caller-paid storage deposit proportional to the update payload size. When any update reaches threshold and `do_update` clears **all** pending entries, the freed storage is never returned to the proposers. There is no withdrawal or refund path. Every participant who proposed a losing update — or even the winning proposer — permanently forfeits their deposit.

---

### Finding Description

`propose_update` computes a required deposit via `ProposedUpdates::required_deposit`, which calls `bytes_used`: [1](#0-0) 

For a contract-code update, `bytes_used` includes the full WASM binary length plus a fixed overhead for 128 participant-vote slots. A typical MPC contract binary of several hundred KB to ~1 MB translates to several NEAR tokens at the on-chain storage byte cost.

The deposit is collected and the excess refunded, but the exact amount required is kept in the contract: [2](#0-1) 

When threshold votes are reached, `vote_update` calls `do_update`, which removes the winning entry and then **clears every other entry and all votes** in one sweep: [3](#0-2) 

`do_update` issues no `Promise::transfer` back to any proposer. The storage is freed (the contract's balance implicitly rises), but the NEAR tokens that were deposited to cover that storage are never returned. There is no `withdraw_update_deposit`, `cancel_proposal`, or any other refund path in the contract.

This is structurally identical to the M01 pattern: a user pays funds into a contract to initiate a lifecycle (here: a governance proposal), the lifecycle ends (update executed or a competing proposal wins), and the funds are permanently locked with no recovery mechanism.

The contrast with `submit_participant_info` — which has explicit refund logic when storage is freed — confirms this is an oversight, not a deliberate design choice: [4](#0-3) 

---

### Impact Explanation

Every contract upgrade cycle causes permanent loss of NEAR tokens for all proposers whose entries are cleared by `do_update`. For a 1 MB WASM binary at NEAR's storage byte cost (~10 milliNEAR/KB), the required deposit is approximately **10 NEAR per proposal**. With multiple competing proposals (a realistic governance scenario), multiple participants each lose that amount permanently. The contract's balance grows by the sum of all forfeited deposits, with no on-chain mechanism to recover them.

This breaks the production accounting invariant that storage deposits are returned when the storage they cover is freed — an invariant the same contract upholds in `submit_participant_info`.

**Impact class:** Medium — balance/request-lifecycle invariant broken, real NEAR token loss, no recovery path.

---

### Likelihood Explanation

Every successful contract upgrade triggers this loss. `propose_update` is a normal governance operation exercised on every software release. The devnet tooling explicitly funds proposers with extra tokens to cover the deposit, confirming this is a routine, expected call: [5](#0-4) 

Any participant who proposes an update that is either outcompeted or that wins (all entries cleared either way) loses their deposit. No privileged access, collusion, or adversarial action is required — the loss is a deterministic consequence of normal governance.

---

### Recommendation

In `do_update`, before clearing entries, iterate over all stored `UpdateEntry` values and issue a `Promise::transfer` refund to each proposer for the deposit they paid. The `UpdateEntry` struct already stores `bytes_used`; the refund amount is `env::storage_byte_cost().saturating_mul(entry.bytes_used)`. The proposer's `AccountId` must be stored alongside the entry (currently it is not), so `UpdateEntry` needs a `proposer: AccountId` field added at proposal time.

Alternatively, track a `deposits: IterableMap<UpdateId, (AccountId, NearToken)>` map and drain it with refunds inside `do_update`.

---

### Proof of Concept

1. Participant A calls `propose_update` with a 500 KB WASM binary, paying ~5 NEAR deposit. Entry stored in `proposed_updates.entries`.
2. Participant B calls `propose_update` with a different binary, paying ~5 NEAR deposit.
3. Threshold participants vote for B's proposal. `vote_update` calls `do_update(&B_id, gas)`.
4. `do_update` removes B's entry, then calls `self.entries.clear()` (removing A's entry) and `self.vote_by_participant.clear()`.
5. No `Promise::transfer` is issued to A or B.
6. Both deposits (~10 NEAR total) remain in the contract balance permanently. Neither participant has any on-chain method to recover them. [6](#0-5)

fn bytes_used(update: &Update) -> u128 {
    let mut bytes_used = std::mem::size_of::<UpdateEntry>() as u128;

    // Assume a high max of 128 participant votes per update entry.
    bytes_used += 128 * std::mem::size_of::<AccountId>() as u128;

    match update {
        Update::Contract(code) => {
            bytes_used += code.len() as u128;
        }
        Update::Config(config) => {
            let bytes = serde_json::to_vec(&config).unwrap();
            bytes_used += bytes.len() as u128;
        }
    }

    bytes_used
}
