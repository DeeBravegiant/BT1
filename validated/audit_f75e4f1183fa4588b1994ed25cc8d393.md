### Title
Peer-Controlled Pool Wallet Injection via Unvalidated Launcher Spend Solution — (`chia/wallet/wallet_state_manager.py`)

---

### Summary

When a user's coin is spent on-chain and produces a child coin at `SINGLETON_LAUNCHER_HASH` (e.g., during DID or NFT creation), the wallet fetches the launcher's spend from the connected peer. The peer's returned **solution is never validated against the blockchain**. A malicious peer can serve a crafted solution that passes `solution_to_pool_state`, causing `PoolWallet.create` to be called unconditionally with attacker-controlled `pool_url` and `target_puzzle_hash`, corrupting pool membership state.

---

### Finding Description

**Code path:**

1. `_add_coin_states` processes a spent coin belonging to the user's wallet. [1](#0-0) 

2. `fetch_children` is called. For untrusted peers, each child `CoinState` is validated via `validate_received_state_from_peer`, which checks the child coin's inclusion in the block's additions Merkle tree. This means the child coin at `SINGLETON_LAUNCHER_HASH` **must actually exist on-chain** — the peer cannot fabricate it. [2](#0-1) 

3. For each child with `puzzle_hash == SINGLETON_LAUNCHER_HASH`, `fetch_coin_spend_for_coin_state` is called to retrieve the launcher spend from the peer. [3](#0-2) 

4. `fetch_coin_spend` validates only that `puzzle.get_tree_hash() == coin.puzzle_hash` (i.e., the puzzle is `SINGLETON_LAUNCHER`) and that `coin_name` matches. **The solution is never validated against the blockchain.** The peer can return any arbitrary solution bytes. [4](#0-3) 

5. `solution_to_pool_state` parses the peer-supplied solution to extract a `PoolState`. It performs no on-chain verification — it only parses the CLVM structure. [5](#0-4) 

6. If `solution_to_pool_state` returns a non-`None` value, `PoolWallet.create` is called unconditionally with the peer-controlled `launcher_spend`, registering a new pool wallet with attacker-controlled `pool_url` and `target_puzzle_hash`. [6](#0-5) 

7. `PoolWallet.create` writes the pool config to disk and registers the wallet in `self.wallets`. [7](#0-6) 

---

### Impact Explanation

- A spurious `PoolWallet` is registered in `self.wallets` with attacker-controlled `pool_url` and `target_puzzle_hash`.
- The pool config file is written to disk with the attacker's `pool_url`, which the farmer reads to determine which pool to connect to. This enables pool URL redirection and phishing of farmer authentication tokens.
- `compute_additions(launcher_spend)` at line 2207 runs the CLVM launcher puzzle against the crafted solution, producing an attacker-controlled singleton coin that the wallet begins tracking.
- The `p2_singleton_puzzle_hash` subscribed to by the wallet is derived from attacker-controlled delay parameters, causing the wallet to monitor an attacker-chosen puzzle hash for incoming rewards.

This is pool membership state corruption with direct security impact.

---

### Likelihood Explanation

The precondition — a user's coin creating a child at `SINGLETON_LAUNCHER_HASH` — is satisfied whenever the user creates a DID, NFT, or pool wallet. These are common operations. Any peer the wallet connects to (including a malicious full node on the public network) can serve the crafted solution. No key material is required from the attacker.

---

### Recommendation

Validate the launcher spend solution against the blockchain before trusting it. Specifically:

1. After fetching the launcher spend from the peer, re-execute the `SINGLETON_LAUNCHER` puzzle with the peer-supplied solution and verify that the computed additions match the on-chain additions (which are already Merkle-proven via `validate_received_state_from_peer` for the child coin's children).
2. Alternatively, fetch the actual on-chain solution via a trusted source or validate it against the additions root of the block at `child.spent_height`.
3. At minimum, verify that `compute_additions(launcher_spend)` produces a coin whose name matches a known on-chain child of the launcher coin (i.e., the singleton eve coin, which can be validated via `fetch_children` on the launcher).

---

### Proof of Concept

**Setup:** Wallet connected to a malicious peer. User creates a DID (which produces a real on-chain launcher coin at `SINGLETON_LAUNCHER_HASH`).

**Attack:**
1. Malicious peer responds to `request_children` with the real launcher `CoinState` (passes `validate_received_state_from_peer`).
2. Malicious peer responds to `request_puzzle_solution` for the launcher coin with:
   - `puzzle_reveal`: the real `SINGLETON_LAUNCHER` program (puzzle hash check passes)
   - `solution`: a crafted CLVM list `[attacker_singleton_puzzle_hash, 1, [["p", attacker_pool_state_bytes], ["t", delay_time], ["h", delay_ph]]]`
3. `solution_to_pool_state` parses the crafted solution and returns a valid `PoolState` with `pool_url = "https://attacker.pool"` and `target_puzzle_hash = attacker_ph`.
4. `PoolWallet.create` is called; pool config is written to disk with `pool_url = "https://attacker.pool"`.
5. Farmer reads the pool config and begins authenticating to the attacker's pool server.

    async def fetch_children(
        self, coin_name: bytes32, peer: WSChiaConnection, fork_height: uint32 | None = None
    ) -> list[CoinState]:
        response: RespondChildren | None = await peer.call_api(FullNodeAPI.request_children, RequestChildren(coin_name))
        if response is None or not isinstance(response, RespondChildren):
            raise PeerRequestException(f"Was not able to obtain children {response}")

        if not self.is_trusted(peer):
            request_cache = self.get_cache_for_peer(peer)
            validated = []
            for state in response.coin_states:
                valid = await self.validate_received_state_from_peer(state, peer, request_cache, fork_height)
                if valid:
                    validated.append(state)
            return validated
        return response.coin_states
```

async def fetch_coin_spend(height: uint32, coin: Coin, peer: WSChiaConnection) -> CoinSpend:
    solution_response = await peer.call_api(
        FullNodeAPI.request_puzzle_solution, RequestPuzzleSolution(coin.name(), height)
    )
    if solution_response is None or not isinstance(solution_response, RespondPuzzleSolution):
        raise PeerRequestException(f"Was not able to obtain solution {solution_response}")
    coin_id = coin.name()
    if solution_response.response.puzzle.get_tree_hash() != coin.puzzle_hash:
        raise PeerRequestException(f"Peer returned wrong puzzle hash for coin {coin_id}")
    if solution_response.response.coin_name != coin_id:
        raise PeerRequestException(f"Peer returned wrong coin name in puzzle solution for coin {coin_id}")

    return make_spend(
        coin,
        solution_response.response.puzzle,
        solution_response.response.solution,
    )
```
