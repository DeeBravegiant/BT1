Audit Report

## Title
FIFO Withdrawal Queue Permanently Stalled by Oversized Withdrawal Request — (`contracts/LRTWithdrawalManager.sol`)

## Summary
`initiateWithdrawal` validates the requested amount against `getAvailableAssetAmount`, which counts all protocol assets including illiquid EigenLayer-staked funds. However, `_unlockWithdrawalRequests` can only disburse liquid vault assets and exits with `break` on the first request it cannot cover. A single oversized withdrawal request at the head of the queue blocks all subsequent users until the vault accumulates enough liquid assets to satisfy it.

## Finding Description
**Root cause — mismatched liquidity accounting between submission and disbursement:**

At submission, `initiateWithdrawal` checks:
```solidity
// LRTWithdrawalManager.sol:170
if (expectedAssetAmount > getAvailableAssetAmount(asset)) revert ExceedAmountToWithdraw();
```
`getAvailableAssetAmount` (L599–603) calls `lrtDepositPool.getTotalAssetDeposits(asset)`, which aggregates all protocol assets including `assetStakedInEigenLayer` and `assetUnstakingFromEigenLayer` — assets that are illiquid and cannot be immediately disbursed.

At disbursement, `_unlockWithdrawalRequests` (L770–816) receives only the liquid vault balance as `availableAssetAmount` and processes requests in strict FIFO order, exiting immediately on the first request it cannot cover:
```solidity
// LRTWithdrawalManager.sol:795
if (block.number < request.withdrawalStartBlock + withdrawalDelayBlocks) break;
// LRTWithdrawalManager.sol:800
if (availableAssetAmount < payoutAmount) break;
```
There is no `continue`, skip mechanism, or operator-controlled start index. `nextLockedNonce[asset]` is the immovable lower bound of the loop.

**Exploit path:**
1. Protocol holds 1,000 ETH total (100 ETH liquid in vault, 900 ETH staked in EigenLayer).
2. Attacker calls `initiateWithdrawal` for rsETH equivalent to 1,000 ETH. Check passes: `getAvailableAssetAmount = 1,000 − 0 = 1,000 ETH`. `assetsCommitted[ETH]` becomes 1,000 ETH.
3. Subsequent legitimate users cannot submit requests because `getAvailableAssetAmount = 0`.
4. Operator calls `unlockQueue`. `availableAssetAmount = 100 ETH` (vault balance). Loop hits attacker's request: `payoutAmount = 1,000 ETH > 100 ETH` → `break`. Zero requests unlocked.
5. All queued users' rsETH remains locked in `LRTWithdrawalManager` with no recourse until the vault accumulates 1,000 ETH.
6. Once processed, attacker immediately re-submits to re-block the queue.

## Impact Explanation
Users whose rsETH is already transferred to `LRTWithdrawalManager` at `initiateWithdrawal` time cannot reclaim it and cannot complete withdrawals. The freeze persists for as long as the vault lacks sufficient liquid assets to cover the blocking request — potentially days to weeks given EigenLayer's unbonding delay. This constitutes **Medium — Temporary freezing of funds** for an unbounded set of users.

## Likelihood Explanation
The attack requires holding rsETH (non-trivial capital), but the attacker recovers their assets after the vault refills — net cost is only foregone yield. No privileged role is needed; `initiateWithdrawal` is open to any rsETH holder. The structural mismatch between `getAvailableAssetAmount` (total protocol assets) and the liquid-only disbursement path is a persistent protocol property, making the attack indefinitely repeatable. Likelihood: **Medium**.

## Recommendation
1. **Replace `break` with `continue`** at L800 in `_unlockWithdrawalRequests` so that requests exceeding current liquidity are skipped rather than blocking the queue. Requires careful bookkeeping so skipped requests are revisited when liquidity improves.
2. **Cap `initiateWithdrawal` against liquid vault assets** rather than `getTotalAssetDeposits`. The submission check should use only the vault's liquid balance (or a conservative fraction) so committed amounts are always coverable at unlock time.
3. **Allow the operator to specify a start index** in addition to `firstExcludedIndex`, enabling targeted processing of a sub-range of the queue to bypass a stuck head request.
