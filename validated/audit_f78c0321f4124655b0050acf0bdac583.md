Audit Report

## Title
Missing Caller Identity Check Allows Anyone to Force Fee-Deducted Claims on Behalf of Any User - (File: contracts/utils/MerkleDistributor/MerkleDistributor.sol)

## Summary
`MerkleDistributor.claim()` accepts a caller-supplied `account` parameter but never verifies that `msg.sender == account`. Any external caller can reconstruct a victim's valid merkle proof from public on-chain data and trigger a claim on their behalf, permanently deducting up to 10% of the victim's entitled token allocation as a protocol fee. The victim's claim state is then marked consumed, preventing any recovery.

## Finding Description
In `contracts/utils/MerkleDistributor/MerkleDistributor.sol`, the `claim` function at lines 97–147 accepts `account` as a caller-supplied parameter and executes the full claim lifecycle — merkle proof verification, `userClaims[account]` state update, and token transfer — without any check that `msg.sender == account`. [1](#0-0) 

The fee deduction at lines 138–144 permanently redirects up to `MAX_FEE_IN_BPS = 1000` (10%) of the victim's claimable amount to `protocolTreasury`: [2](#0-1) 

After the forced claim, `userClaims[account].lastClaimedIndex` is updated to `index`, so any subsequent call by the legitimate user reverts with `AlreadyClaimed`: [3](#0-2) 

The sister contract `KernelMerkleDistributor._processClaim()` correctly guards this path at lines 311–313: [4](#0-3) 

`MerkleDistributor` has no equivalent guard. All inputs required to call `claim` for a victim (`index`, `account`, `cumulativeAmount`, `merkleProof`) are deterministically derivable from the public `MerkleRootSet` event and the published merkle tree.

## Impact Explanation
**High — Theft of unclaimed yield.** A forced claim causes the victim to permanently receive `claimableAmount - fee` instead of `claimableAmount`. The fee (up to 10%) is irreversibly redirected to `protocolTreasury`. Because `userClaims[account]` is updated to reflect the forced claim, the victim cannot reclaim the deducted portion. The attacker pays only gas; the victim loses a concrete, bounded fraction of their entitled yield with no recourse.

## Likelihood Explanation
The attack requires no privileged access, no private keys, and no oracle manipulation. The merkle tree is public; `index`, `account`, `cumulativeAmount`, and `merkleProof` for any user are all derivable from on-chain events. Any external caller can execute this against any unclaimed account at any time while the contract is unpaused. The attack is repeatable across every new merkle root epoch.

## Recommendation
Add a caller identity check at the top of `claim()`, mirroring the pattern already used in `KernelMerkleDistributor._processClaim()`:

```solidity
if (account != msg.sender) {
    revert Unauthorized();
}
``` [4](#0-3) 

## Proof of Concept
1. Protocol owner calls `setMerkleRoot(root)`. The merkle tree is public; Alice's leaf is `keccak256(abi.encodePacked(index, alice, cumulativeAmount))`.
2. Attacker reconstructs Alice's valid `merkleProof` from the public tree.
3. Attacker calls `MerkleDistributor.claim(index, alice, cumulativeAmount, merkleProof)` before Alice does.
4. Contract verifies the proof (passes), computes `fee = claimableAmount * feeInBPS / 10_000`, transfers `claimableAmount - fee` to Alice, and `fee` to `protocolTreasury`.
5. `userClaims[alice].lastClaimedIndex` is set to `index`; `userClaims[alice].cumulativeAmount` is set to `cumulativeAmount`.
6. Alice calls `claim` herself — reverts with `AlreadyClaimed`.
7. Alice has permanently lost up to 10% of her entitled token distribution with no recovery path.

**Foundry test plan:** Deploy `MerkleDistributor` with `feeInBPS = 1000`. Build a two-leaf merkle tree for Alice and Bob. Call `setMerkleRoot`. From `address(attacker)`, call `claim(index, alice, cumulativeAmount, aliceProof)`. Assert Alice received `claimableAmount * 9000 / 10_000`. Assert `isClaimed(index, alice) == true`. Assert Alice's subsequent `claim` call reverts with `AlreadyClaimed`.
