### Title
`Map.withoutKeys` in `poolReapTransition` unconditionally deletes a VRF key from `psVRFKeyHashes` instead of decrementing its reference count, allowing VRF key uniqueness to be violated post-v11 — (`eras/shelley/impl/src/Cardano/Ledger/Shelley/Rules/PoolReap.hs`)

---

### Summary

`poolReapTransition` uses two different and asymmetric mechanisms to remove VRF key hashes from `psVRFKeyHashes`. For **retired** pools it calls `removeVRFKeyHashOccurrences`, which correctly decrements the reference count and only deletes the key when the count reaches zero. For **re-registered** pools (the `danglingVRFKeyHashes` path) it calls `Map.withoutKeys`, which unconditionally deletes the key regardless of its count. When two pools share a VRF key (permitted pre-v11) and one re-registers with a new VRF key post-v11, the shared key is deleted entirely even though the other pool still holds a live reference to it. The invariant `psVRFKeyHashes[VRF_A] ≥ 1` for any VRF key still referenced by an active pool is broken, and a third pool can subsequently register with that key, defeating the uniqueness guarantee introduced at v11.

---

### Finding Description

**v11 hardfork initialisation — `populateVRFKeyHashes`**

At the v11 hardfork `hardforkTransition` calls `populateVRFKeyHashes`, which counts every VRF key appearing in both `psStakePools` and `psFutureStakePoolParams`: [1](#0-0) 

If pools P and Q both registered with `VRF_A` before v11, `populateVRFKeyHashes` sets `psVRFKeyHashes[VRF_A] = 2`.

**Re-registration of P with `VRF_B` post-v11**

In `poolTransition`, the re-registration branch inserts `VRF_B` into `psVRFKeyHashes` with count 1 but does **not** decrement `VRF_A`; that decrement is deferred to POOLREAP: [2](#0-1) 

After re-registration: `psVRFKeyHashes = {VRF_A: 2, VRF_B: 1}`, `psFutureStakePoolParams[P].vrfKey = VRF_B`.

**`danglingVRFKeyHashes` computation at POOLREAP**

`danglingVRFKeyHashes` is the set of old VRF keys for pools present in **both** `psStakePools` and `psFutureStakePoolParams` whose VRF key changed. Pool Q is only in `psStakePools` (it has not re-registered), so Q's use of `VRF_A` is invisible here. The merge uses `Map.dropMissing` on both sides, meaning Q is silently excluded: [3](#0-2) 

Result: `danglingVRFKeyHashes = {VRF_A}`.

**The asymmetric deletion**

The final state update applies two different removal strategies back-to-back: [4](#0-3) 

- `removeVRFKeyHashOccurrences retiredVRFKeyHashes` — uses `Map.update (mapNonZero (\n -> n - 1))`, a proper reference-count decrement that only removes the key when the count reaches zero.
- `` (`Map.withoutKeys` danglingVRFKeyHashes) `` — unconditionally deletes every key in the set, regardless of its count.

`Map.withoutKeys` deletes `VRF_A` entirely. After POOLREAP: `psVRFKeyHashes = {VRF_B: 1}`. Pool Q still holds `VRF_A` in `psStakePools`, but `psVRFKeyHashes` no longer records it.

**Uniqueness check bypass**

Post-v11, new pool registration checks: [5](#0-4) 

Because `VRF_A` is absent from `psVRFKeyHashes`, `Map.notMember VRF_A psVRFKeyHashes` is `True`, and a new pool R is permitted to register with `VRF_A`. Both Q and R now hold `VRF_A` as their active VRF key, violating the uniqueness invariant that v11 was designed to enforce.

---

### Impact Explanation

Two active pools sharing a VRF key post-v11 undermines the VRF-based slot-leader election security model: both pools are elected leader for the same slots, one pool can shadow another's block production, and the uniqueness guarantee that prevents VRF-key-based impersonation is defeated. This constitutes an attacker-controlled certificate sequence exceeding the intended validation limit (VRF uniqueness check) introduced at v11.

---

### Likelihood Explanation

The precondition — two pools sharing a VRF key before v11 — was explicitly permitted by the pre-v11 ledger rules. On mainnet such pools exist from the period before the Conway/v11 hardfork. Any operator of one such pool can trigger the bug unilaterally by submitting a single re-registration certificate with a fresh VRF key and waiting one epoch boundary. No governance majority, leaked key, or privileged access is required.

---

### Recommendation

Replace `Map.withoutKeys danglingVRFKeyHashes` with the same reference-count decrement used for retired pools:

```haskell
& certPStateL . psVRFKeyHashesL
  %~ ( removeVRFKeyHashOccurrences retiredVRFKeyHashes
         . removeVRFKeyHashOccurrences (Set.toList danglingVRFKeyHashes)
     )
```

`removeVRFKeyHashOccurrences` already handles the "remove only when count reaches zero" semantics correctly via `Map.update (mapNonZero (\n -> n - 1))`. Applying it to `danglingVRFKeyHashes` ensures that a VRF key shared by N pools is decremented to N-1 rather than deleted outright.

### Proof of Concept

State-machine sequence (all steps executable in the existing imp-test framework):

1. Set protocol version to 10 (`whenMajorVersion @10`).
2. Register pool P with `VRF_A`; register pool Q with `VRF_A` (no uniqueness check pre-v11).
3. Enact v11 hardfork via governance action. Assert `psVRFKeyHashes = {VRF_A: 2}`.
4. Submit a re-registration certificate for P with `VRF_B`. Assert `psVRFKeyHashes = {VRF_A: 2, VRF_B: 1}`.
5. Advance one epoch (triggers POOLREAP). Assert — **this assertion fails** — `psVRFKeyHashes[VRF_A] == 1` (Q is still active). Actual result: `psVRFKeyHashes = {VRF_B: 1}` (`VRF_A` deleted).
6. Register a new pool R with `VRF_A`. This succeeds, even though Q still holds `VRF_A`. Assert `psVRFKeyHashes = {VRF_A: 1, VRF_B: 1}` — but Q's reference is unaccounted for, so the count is wrong and the uniqueness invariant is broken.

  pState

        . accumulateVRFKeyHashes (pState ^. psFutureStakePoolParamsL) (^. sppVrfL)
  where
    accumulateVRFKeyHashes ::
      Map (KeyHash StakePool) a ->
      (a -> VRFVerKeyHash StakePoolVRF) ->
      Map (VRFVerKeyHash StakePoolVRF) (NonZero Word64) ->
      Map (VRFVerKeyHash StakePoolVRF) (NonZero Word64)
    accumulateVRFKeyHashes spMap getVrf acc =
      Map.foldr' (addVRFKeyHashOccurrence . getVrf) acc spMap

      Map.insertWith combine vrfKeyHash (knownNonZeroBounded @1)
      where

        combine _ oldVal = fromMaybe oldVal $ mapNonZero (+ 1) oldVal
