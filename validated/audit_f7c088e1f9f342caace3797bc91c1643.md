### Title
Validator Inserts Proposal into `valid_proposals` Cache Before Commitment Integrity Check, Leaving Corrupt Entry Permanently Accessible - (File: `crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

### Summary

In `validate_proposal`, the validator unconditionally inserts the proposal into the shared `valid_proposals` cache **before** checking whether the batcher-computed `built_block` commitment matches the network-received `received_fin.proposal_commitment`. When the check fails (`ProposalFinMismatch`), the function returns an error, but the already-inserted entry is never removed. This orphaned entry — keyed by `(height, round)` with the **batcher's** commitment — persists in the cache for the lifetime of the height. If consensus subsequently reaches a decision on the same `(height, round)` using the **network** commitment (which is what peers voted on), `decision_reached` will `panic` because the stored commitment does not match the requested one.

### Finding Description

In `crates/apollo_consensus_orchestrator/src/validate_proposal.rs`, the `validate_proposal` function performs the following sequence:

1. Executes all transaction batches through the batcher via `send_txs_for_proposal` and `finish_proposal`.
2. Receives `finished_info` (containing `batcher_block_commitment`) and `received_fin` (containing the network-supplied `proposal_commitment`).
3. **Inserts the proposal into `valid_proposals`** at line 241, keyed by `(init.height, init.round)` with the batcher's commitment.
4. **Then** checks at line 244 whether `built_block != received_fin.proposal_commitment`.
5. If they differ, returns `Err(ValidateProposalError::ProposalFinMismatch)` — but the entry inserted in step 3 is **never removed**. [1](#0-0) 

The `insert_proposal` function in `BuiltProposals` stores the entry keyed by `(height, round)` with the batcher-derived `proposal_commitment`: [2](#0-1) 

When `decision_reached` is later called with the network commitment (the one peers voted on), it calls `get_proposal(&height, &round, &commitment)`, which asserts that the stored commitment equals the requested one: [3](#0-2) 

If the stored commitment (batcher's) differs from the requested commitment (network's), this `assert_eq!` panics, crashing the consensus orchestrator.

The `decision_reached` flow itself confirms the block is committed to storage immediately after the batcher call: [4](#0-3) 

### Impact Explanation

**Impact: High — RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value / Wrong state committed.**

The concrete impact depends on which commitment wins consensus:

- **Scenario A (panic/crash):** The validator node detects a `ProposalFinMismatch` (batcher and proposer disagree), returns an error to consensus, but the orphaned entry remains. If the network still reaches a quorum on the proposer's commitment and calls `decision_reached` with that commitment, the `assert_eq!` in `get_proposal` panics because the stored entry holds the batcher's (different) commitment. The node crashes and cannot finalize the block.

- **Scenario B (silent wrong commitment):** In a scenario where the batcher's commitment and the network commitment differ only due to a transient computation difference (e.g., fee rounding), and the validator's `fin_sender` is dropped (returning `Canceled` to consensus), the orphaned entry with the wrong commitment remains in `valid_proposals`. If consensus later calls `repropose` or `decision_reached` for the same `(height, round)`, it retrieves the wrong `finished_info` (wrong `l2_gas_used`, wrong `block_header_commitments`), which propagates incorrect values into `finalize_decision` → `update_state_sync_with_new_block`.

### Likelihood Explanation

The `ProposalFinMismatch` path is a known, tested code path (see `validate_proposal_test.rs::proposal_fin_mismatch`). The test confirms the error is returned but does **not** verify that the `valid_proposals` cache is clean afterward. The mismatch can be triggered by:

1. A proposer running a different version of the commitment computation.
2. A `fee_proposal_fri` value that differs between proposer and validator (the commitment includes `fee_proposal_fri` via `proposal_commitment_from`).
3. Any transient divergence in `partial_block_hash` computation. [5](#0-4) 

The test at line 503 confirms `ProposalFinMismatch` is returned, but there is no assertion that `valid_proposals` is empty after the failure.

### Recommendation

Move the `valid_proposals.insert_proposal(...)` call to **after** the commitment integrity check, so it only executes when `built_block == received_fin.proposal_commitment`:

```rust
// TODO(matan): Switch to signature validation.
if built_block != received_fin.proposal_commitment {
    CONSENSUS_PROPOSAL_FIN_MISMATCH.increment(1);
    return Err(ValidateProposalError::ProposalFinMismatch);
}

// Only insert after integrity check passes.
let mut valid_proposals = args.valid_proposals.lock().unwrap();
valid_proposals.insert_proposal(args.init, content, &args.proposal_id, finished_info);

Ok(built_block)
```

The comment at line 238–239 justifies the current ordering as avoiding a race condition with `repropose`. However, `repropose` is only called after consensus receives the `fin_sender` signal — which is only sent on the `Ok(built_block)` path in `validate_and_send`. Since `ProposalFinMismatch` causes an early return before `fin_sender.send(...)`, the race condition the comment describes cannot occur in the mismatch case. Reordering is safe. [6](#0-5) 

Additionally, add a test asserting that `valid_proposals` is empty after a `ProposalFinMismatch` error.

### Proof of Concept

1. A validator node receives a proposal for `(height=H, round=R)`.
2. The batcher executes all transactions and returns `finished_info` with `partial_block_hash = X`.
3. The proposer's `ProposalFin` carries `proposal_commitment = Y` where `Y ≠ X` (e.g., different `fee_proposal_fri` encoding).
4. `validate_proposal` inserts `(H, R) → commitment_from(X, fee)` into `valid_proposals` at line 241.
5. `validate_proposal` detects `built_block != received_fin.proposal_commitment` at line 244 and returns `Err(ProposalFinMismatch)`.
6. The `fin_sender` is dropped (never sent), so consensus receives `Canceled` on the `fin_receiver`.
7. Consensus advances to round `R+1`. The proposer re-proposes the same block with `valid_round=R`.
8. `repropose` calls `update_for_reproposal(&H, &commitment_Y, &build_param)`, which calls `get_proposal(&H, &R, &commitment_Y)`.
9. The stored entry has `commitment_from(X, fee) ≠ commitment_Y`, triggering the `assert_eq!` panic at line 168–172 of `sequencer_consensus_context.rs`.
10. The consensus orchestrator crashes; the node cannot finalize block `H`. [7](#0-6)

    pub(crate) fn insert_proposal(
        &mut self,
        init: ProposalInit,
        transactions: Vec<Vec<InternalConsensusTransaction>>,
        proposal_id: &ProposalId,
        finished_info: FinishedProposalInfo,
    ) {
        let proposal_commitment = proposal_commitment_from(
            finished_info.proposal_commitment.partial_block_hash,
            init.fee_proposal_fri,
        );

        let height = init.height;
        let round = init.round;
        let by_round = self.data.entry(height).or_default();
        let previous = by_round.insert(
            round,

        );
        assert!(
            previous.is_none(),
            "Overwriting existing proposal for height {height} round {round}; at most one \
             proposal per (height, round) is allowed"
        );

async fn proposal_fin_mismatch() {

        .returning(move |_| {

                    final_n_executed_txs: n_executed,

                parent_proposal_commitment: None,
