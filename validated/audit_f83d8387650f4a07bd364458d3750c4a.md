### Title
Unrefunded Deposit in `submit_participant_info` When Existing Participant Re-submits Attestation - (File: `crates/contract/src/lib.rs`)

### Summary
The `submit_participant_info` function is marked `#[payable]` and accepts NEAR deposits, but contains a conditional refund path that silently swallows any attached deposit when an existing participant re-submits their attestation. This is the direct analog of the `BridgeExecutorBase.sol` issue: a payable entry point with no guaranteed refund of excess assets, causing permanent locking of NEAR in the contract.

### Finding Description
`submit_participant_info` computes a boolean guard:

```rust
let attestation_storage_must_be_paid_by_caller =
    is_new_attestation || caller_is_not_participant;

if attestation_storage_must_be_paid_by_caller {
    // ... check deposit, refund excess
}

Ok(())
``` [1](#0-0) 

When an **existing participant** (a current voter) re-submits their TEE attestation — the normal periodic renewal path — both `is_new_attestation` and `caller_is_not_participant` are `false`, so `attestation_storage_must_be_paid_by_caller` evaluates to `false`. The `if` block is skipped entirely. Any NEAR attached to the call is accepted by the `#[payable]` attribute but never read, checked, or refunded. The function returns `Ok(())` and the deposit is permanently locked in the contract. [2](#0-1) 

The other payable entry points (`sign`, `request_app_private_key`, `verify_foreign_transaction`, `propose_update`) all correctly refund excess via `require_deposit` or explicit `checked_sub` + `Promise::transfer`: [3](#0-2) [4](#0-3) 

`submit_participant_info` is the only payable function that lacks an unconditional refund path.

### Impact Explanation
Any NEAR attached to a re-submission call by an existing participant is permanently frozen in the contract. There is no admin withdrawal function, no sweep mechanism, and no way to recover the locked tokens. This breaks the production accounting invariant that excess deposits must be returned to callers, and constitutes a permanent freezing of funds held by the contract.

**Impact: Medium** — balance/accounting invariant violation causing permanent loss of NEAR tokens with no recovery path.

### Likelihood Explanation
MPC nodes perform periodic attestation renewal (re-submission) as a normal operational requirement. The node software or an operator script could attach a non-zero deposit — e.g., to satisfy a generic "always attach 1 yoctoNEAR" policy used for other payable calls, or due to a misconfiguration. Because the function is `#[payable]` and silently accepts any amount, there is no on-chain guard to prevent this. The trigger requires no special privilege beyond being an existing participant.

### Recommendation
Add an unconditional refund of any excess deposit at the end of `submit_participant_info`, regardless of the `attestation_storage_must_be_paid_by_caller` branch:

```rust
// After the conditional storage-cost block:
let attached = env::attached_deposit();
let cost = if attestation_storage_must_be_paid_by_caller { cost } else { NearToken::from_yoctonear(0) };
if let Some(diff) = attached.checked_sub(cost)
    && diff > NearToken::from_yoctonear(0)
{
    Promise::new(account_id).transfer(diff).detach();
}
```

Alternatively, remove `#[payable]` from `submit_participant_info` entirely and add a separate `#[payable]` storage-deposit helper, so the function panics on any non-zero deposit when storage payment is not required.

### Proof of Concept

1. Alice is an existing MPC participant (voter).
2. Alice's node re-submits its TEE attestation via `submit_participant_info`, attaching `1_000_000_000_000_000_000_000_000` yoctoNEAR (1 NEAR) — e.g., due to a generic deposit policy in the node runner script.
3. Inside the function: `caller_is_not_participant = false` (Alice is a voter), `is_new_attestation = false` (Alice already has a stored attestation entry).
4. `attestation_storage_must_be_paid_by_caller = false || false = false`.
5. The `if` block is skipped. `env::attached_deposit()` is never read or refunded.
6. The function returns `Ok(())`. Alice's 1 NEAR is permanently locked in the contract.
7. No withdrawal function exists in the contract to recover it. [5](#0-4)
