### Title
Proposer Deposits Permanently Locked When `do_update` Clears All Pending Proposals - (`File: crates/contract/src/update.rs`)

---

### Summary

`propose_update` requires participants to attach a NEAR deposit proportional to the size of the proposed update (contract binary or config). When any update reaches threshold and `do_update` executes, it bulk-clears **all** pending proposals without refunding the deposits paid by the non-winning proposers. Because `UpdateEntry` does not record the proposer's account ID or deposit amount, there is no mechanism — direct or indirect — to recover those funds. They are permanently locked in the contract's balance.

---

### Finding Description

`propose_update` in `crates/contract/src/lib.rs` enforces a storage-staking deposit:

```rust
let attached = env::attached_deposit();
let required = ProposedUpdates::required_deposit(&update);
// ...only the excess is refunded immediately:
if let Some(diff) = attached.checked_sub(required) && diff > NearToken::from_yoctonear(0) {
    Promise::new(proposer).transfer(diff).detach();
}
```

The required portion stays in the contract. The `UpdateEntry` struct that is stored records only the update payload and its byte count — not the proposer's identity or the deposit amount:

```rust
pub(crate) struct UpdateEntry {
    pub(super) update: Update,
    pub(super) bytes_used: u128,
}
```

When `do_update` is triggered by a winning vote, it clears every pending proposal:

```rust
pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
    let entry = self.entries.remove(id)?;
    // Clear all entries as they might be no longer valid
    self.entries.clear();
    self.vote_by_participant.clear();
    // ...no refund issued anywhere
    Some(promise)
}
```

Because the proposer's account ID and deposit amount are never stored in `UpdateEntry`, the contract has no information needed to issue refunds when it frees the storage. The NEAR tokens paid by every non-winning proposer are permanently absorbed into the contract's balance.

---

### Impact Explanation

**Medium.** This breaks the production accounting invariant that storage-staking deposits must be returned when the storage they cover is freed. A contract binary upload deposit is proportional to the WASM binary size (NEAR charges ~1 NEAR per 10 KB of storage); a realistic contract binary of 500 KB requires ~50 NEAR. In a competitive governance round where multiple participants propose different upgrades, every non-winning proposer permanently loses their full deposit with no retrieval path. The funds are not stolen by an external party — they are frozen inside the contract account — but they are irrecoverable by the rightful owner.

---

### Likelihood Explanation

Competitive governance is a normal and expected scenario: participants may disagree on which contract version to deploy, leading to multiple concurrent proposals. The README explicitly describes the multi-proposal, multi-vote flow. Every such round where more than one proposal is submitted guarantees that at least one proposer's deposit is permanently locked. No adversarial action is required; the loss is a deterministic consequence of normal protocol operation.

---

### Recommendation

Store the proposer's `AccountId` and the exact deposit amount inside `UpdateEntry`:

```rust
pub(crate) struct UpdateEntry {
    pub(super) update: Update,
    pub(super) bytes_used: u128,
    pub(super) proposer: AccountId,   // add
    pub(super) deposit: NearToken,    // add
}
```

In `do_update`, before calling `self.entries.clear()`, iterate over all remaining entries and schedule a `Promise::new(entry.proposer).transfer(entry.deposit)` for each one. This mirrors the refund pattern already used in `propose_update` for excess deposits and in `submit_participant_info` for rejected attestations.

---

### Proof of Concept

1. Participant A calls `propose_update` with a 500 KB contract binary, attaching ~50 NEAR (the required deposit). `UpdateEntry { update: ..., bytes_used: ... }` is stored; A's account ID and deposit are discarded.
2. Participant B calls `propose_update` with a different binary, also paying ~50 NEAR.
3. Threshold participants vote for B's proposal. `vote_update` calls `do_update(&B_id, gas)`.
4. `do_update` removes B's entry, then calls `self.entries.clear()` — silently dropping A's entry.
5. No refund promise is scheduled for A. A's ~50 NEAR is permanently locked in the contract balance.
6. A has no method to retrieve the deposit: there is no `withdraw_proposal_deposit` endpoint, and the contract stores no record that A ever paid. [1](#0-0) [2](#0-1) [3](#0-2)
