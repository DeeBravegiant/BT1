### Title
Missing Chain ID in DelegateAction Signed Payload Enables Cross-Network Replay - (File: core/primitives/src/action/delegate.rs)

### Summary
`DelegateAction` (NEP-366 meta-transactions) and `DelegateActionV2` (NEP-611) compute their signing hash without binding to any chain ID or network identifier. A malicious relayer who receives a user-signed `DelegateAction` intended for NEAR mainnet can replay it verbatim on NEAR testnet (or any NEAR fork) if the same account and key exist on both networks, causing unauthorized execution of the inner actions on the unintended network.

### Finding Description
The `DelegateAction` struct contains `sender_id`, `receiver_id`, `actions`, `nonce`, `max_block_height`, and `public_key`, but **no chain ID or network identifier**. [1](#0-0) 

The signing hash is computed in `get_nep461_hash()` by wrapping the action in a `SignableMessage` that prepends only a fixed NEP discriminant (`(1 << 30) + 366` for V1, `(1 << 30) + 611` for V2): [2](#0-1) 

The `SignableMessage` struct contains only the discriminant and the message body — no chain ID: [3](#0-2) 

The discriminant is a fixed constant, not network-specific: [4](#0-3) 

Contrast this with regular `SignedTransaction`, which includes a `block_hash` field that is chain-specific and provides implicit network binding: [5](#0-4) 

The transaction hash used for signing is computed directly from the borsh-serialized transaction body (which includes `block_hash`): [6](#0-5) 

`DelegateAction` has no equivalent binding. Signature verification in `apply_delegate_action` calls `signed_delegate_action.verify()`, which only checks the signature against the NEP-461 hash — no chain ID check anywhere in the path: [7](#0-6) [8](#0-7) 

### Impact Explanation
A malicious relayer who receives a user-signed `DelegateAction` (e.g., a token transfer or `AddKey` action intended for NEAR mainnet) can submit it to NEAR testnet (or any NEAR fork) by wrapping it in a new outer `SignedTransaction` signed with the relayer's own testnet key. The inner `DelegateAction` signature verifies correctly on testnet because the signed payload is identical — the discriminant is the same fixed constant on every NEAR network.

Concrete corrupted values: the victim's access key nonce is advanced on the unintended network, and the inner actions (token transfers, key additions/deletions, function calls) execute against the victim's account on that network, corrupting their balance, access key set, or contract state.

### Likelihood Explanation
The attack requires a malicious relayer and the victim having the same account ID and public key registered on multiple NEAR networks. This is common: NEAR documentation and tooling encourage using the same key pair on mainnet and testnet. The `max_block_height` field provides only a time window (not a chain binding), and users routinely set large values for long-lived meta-transactions. Any relayer service that accepts signed `DelegateAction` objects over an API is a realistic attack surface.

### Recommendation
Include the chain ID in the `DelegateAction` signed payload, analogous to how EIP-712 requires the `chainId` in the domain separator. The chain ID is already accessible to contracts via the `chain_id` host function. It should be added as a field to `DelegateAction` and `DelegateActionV2` and incorporated into the `get_nep461_hash()` computation, or alternatively embedded in the `MessageDiscriminant` scheme so that the discriminant is network-specific.

### Proof of Concept
1. Alice signs a `DelegateAction` on NEAR mainnet: `sender_id=alice.near`, `receiver_id=token.near`, `actions=[Transfer{1000}]`, `nonce=5`, `max_block_height=200000000`.
2. Alice sends the `SignedDelegateAction` to a relayer service.
3. The malicious relayer, instead of submitting to mainnet, constructs a new outer `SignedTransaction` on NEAR testnet with the same `SignedDelegateAction` as the inner action.
4. The testnet node calls `apply_delegate_action` → `signed_delegate_action.verify()` → `get_nep461_hash()`. The hash is `SHA256(LE32(1<<30 + 366) || borsh(delegate_action))` — identical to the mainnet hash because no chain ID is included.
5. Signature verification passes. The transfer executes on testnet, draining Alice's testnet balance without her consent and advancing her testnet nonce, corrupting her testnet account state.

impl SignedDelegateAction {
    pub fn verify(&self) -> bool {
        let delegate_action = &self.delegate_action;
        let hash = delegate_action.get_nep461_hash();
        let public_key = &delegate_action.public_key;

        self.signature.verify(hash.as_ref(), public_key)
    }

    pub fn sign(singer: &Signer, delegate_action: DelegateAction) -> Self {
        let signature = singer.sign(delegate_action.get_nep461_hash().as_bytes());
        Self { delegate_action, signature }
    }

    pub fn get_nep461_hash(&self) -> CryptoHash {
        let signable = SignableMessage::new(&self, SignableMessageType::DelegateAction);
        let bytes = borsh::to_vec(&signable).expect("Failed to deserialize");
        hash(&bytes)
