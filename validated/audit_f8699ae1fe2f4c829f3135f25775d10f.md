### Title
Multiple Sub-Transactions Can Claim the Same Deposit Refund, Creating ADA from Nothing — (`eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/SubLedgers.hs`)

---

### Summary

In the Dijkstra era, the `SUBLEDGERS` rule processes sub-transactions sequentially via `foldM` but lacks a cross-sub-transaction batch-level check for deposit refunds. An attacker can craft a top-level transaction containing multiple sub-transactions that each include an `UnRegDepositTxCert` for the same staking credential, claiming the `keyDeposit` refund N times while having paid it only once. This creates `(N-1) * keyDeposit` ADA from nothing. The codebase itself acknowledges this gap with a disabled test marked `error "TODO: predicate failure not yet implemented"`.

---

### Finding Description

The Dijkstra era introduces nested/batch transactions (`subTransactionsTxBodyL`). The `SUBLEDGERS` rule iterates over sub-transactions sequentially:

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/SubLedgers.hs
dijkstraSubLedgersTransition = do
  TRC (env, ledgerState, subTxs) <- judgmentContext
  foldM
    ( \ls subTx ->
        trans @(EraRule "SUBLEDGER" era) $ TRC (env, ls, subTx)
    )
    ledgerState
    subTxs
``` [1](#0-0) 

Each sub-transaction is processed by `SUBLEDGER → SUBENTITIES`, which validates and applies withdrawals against the **current** (sequentially updated) `certState`. For **withdrawals**, a batch-level guard exists in `dijkstraUtxoTransition`:

```haskell
runTest $ validateBatchWithdrawals accounts tx
``` [2](#0-1) 

`validateBatchWithdrawals` sums all withdrawals across the entire batch and checks them against the **original** pre-batch account balances, preventing any sub-transaction from over-withdrawing even if a prior sub-transaction already reduced the balance:

```haskell
validateBatchWithdrawals accounts tx =
  let allWithdrawals =
        Map.unionsWith (<>) $
          unWithdrawals (tx ^. bodyTxL . withdrawalsTxBodyL)
            : [ unWithdrawals $ subTx ^. bodyTxL . withdrawalsTxBodyL
              | subTx <- OMap.elems $ tx ^. bodyTxL . subTransactionsTxBodyL
              ]
``` [3](#0-2) 

**No equivalent batch-level guard exists for deposit refunds.** There is no `validateBatchRefunds` that aggregates all `UnRegDepositTxCert` certificates across sub-transactions and checks that the same credential is not unregistered more than once. The `SUBENTITIES` rule for each sub-transaction only validates against the current (per-sub-tx) `certState`:

```haskell
dijkstraSubEntitiesTransition = do
  ...
  let (missingWithdrawals, exceededWithdrawals) =
        case withdrawalsThatExceedAccountBalance withdrawals network accounts of
          ...
  ...
  & certDStateL . accountsL %~ applyWithdrawals withdrawals
``` [4](#0-3) 

The codebase explicitly acknowledges this gap with a disabled test:

```haskell
xit "Multiple subtransactions cannot get the same refund" $ do
  ...
  subTx1 = ... & bodyTxL . certsTxBodyL .~ SSeq.singleton (UnRegDepositTxCert stakingCred keyDeposit)
  subTx2 = ... & bodyTxL . certsTxBodyL .~ SSeq.singleton (UnRegDepositTxCert stakingCred keyDeposit)
  tx = ... & bodyTxL . subTransactionsTxBodyL .~ OMap.fromFoldable [subTx1, subTx2]
  submitFailingTx tx . NE.singleton $ error "TODO: predicate failure not yet implemented"
``` [5](#0-4) 

The `xit` prefix disables the test, and `error "TODO: predicate failure not yet implemented"` confirms the rejection predicate does not yet exist. The test is disabled precisely because the code currently lacks the check — if run, the transaction would succeed (double-claiming the refund) rather than fail.

---

### Impact Explanation

**Critical — Direct creation of ADA through an invalid ledger state transition.**

An attacker registers a staking credential (paying `keyDeposit` once), then submits a top-level transaction containing N sub-transactions each bearing `UnRegDepositTxCert stakingCred keyDeposit`. If the per-sub-tx SUBCERTS/SUBDELEG rule does not check that the credential is still registered (i.e., was not already unregistered by a prior sub-transaction in the same batch), each sub-transaction independently claims the refund. The deposit pot is debited N times while only one deposit was ever paid, creating `(N-1) * keyDeposit` ADA from nothing. This violates the preservation-of-value invariant that is the foundational security property of the Cardano ledger. [6](#0-5) 

---

### Likelihood Explanation

Any unprivileged transaction submitter can exploit this. The attacker only needs to:
1. Register a staking credential (a normal, permissionless operation).
2. Craft a transaction with multiple sub-transactions each containing `UnRegDepositTxCert` for the same credential.
3. Submit the transaction.

No governance majority, leaked key, or privileged access is required. The attack is straightforward and repeatable.

---

### Recommendation

Introduce a batch-level refund guard analogous to `validateBatchWithdrawals`. Before processing sub-transactions, aggregate all `UnRegDepositTxCert` certificates across the entire batch (top-level tx + all sub-transactions) and verify that no credential appears more than once. Alternatively, ensure the `SUBDELEG`/`SUBCERTS` rule for sub-transactions explicitly checks that the credential is still registered in the **current** (post-prior-sub-tx) `certState` before granting the refund, and define the corresponding predicate failure type (resolving the `TODO` in the disabled test). [5](#0-4)

### Proof of Concept

1. Register staking credential `C`, paying `keyDeposit = D`.
2. Construct:
   - `subTx1`: inputs = `{utxo1}`, certs = `[UnRegDepositTxCert C D]`, outputs = `[addr1: value1 + D]`
   - `subTx2`: inputs = `{utxo2}`, certs = `[UnRegDepositTxCert C D]`, outputs = `[addr2: value2 + D]`
   - `topTx`: subTransactions = `[subTx1, subTx2]`
3. Submit `topTx`.
4. `SUBLEDGERS` processes `subTx1` first: `SUBENTITIES` validates `C` is registered ✓, applies `UnRegDepositTxCert`, refunds `D` to `addr1`, unregisters `C`.
5. `SUBLEDGERS` processes `subTx2` next: without a check that `C` is already unregistered, `SUBENTITIES` again processes `UnRegDepositTxCert C D`, refunds `D` to `addr2`.
6. Net result: `D` ADA paid once, `2D` ADA refunded — `D` ADA created from nothing.

The disabled test at `CertSpec.hs:53–75` directly encodes this exact scenario and confirms the predicate failure to prevent it has not been implemented. [1](#0-0) [5](#0-4)

dijkstraSubEntitiesTransition = do
  TRC (subCertsEnv, certState, certificates) <- judgmentContext
  let tx = certsTx subCertsEnv
      pp = certsPParams subCertsEnv
      curEpoch = certsCurrentEpoch subCertsEnv
      withdrawals = tx ^. bodyTxL . withdrawalsTxBodyL
      accounts = certState ^. certDStateL . accountsL

  runTest $ Conway.validateWithdrawalsDelegated accounts tx

  network <- liftSTS $ asks networkId
  let (missingWithdrawals, exceededWithdrawals) =
        case withdrawalsThatExceedAccountBalance withdrawals network accounts of
          Nothing -> (Map.empty, Map.empty)
          Just (missing, exceeded) -> (unWithdrawals missing, exceeded)
  failOnNonEmptyMap missingWithdrawals $
    injectFailure . SubWithdrawalsMissingAccounts . Withdrawals . NEM.toMap
  failOnNonEmptyMap exceededWithdrawals $ injectFailure . SubWithdrawalAmountsExceedAccountBalances

  let certStateBeforeSubCerts =
        certState
          & Conway.updateDormantDRepExpiries tx curEpoch
          & Conway.updateVotingDRepExpiries tx curEpoch (pp ^. ppDRepActivityL)
          & certDStateL . accountsL %~ applyWithdrawals withdrawals
