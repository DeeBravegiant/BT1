Audit Report

## Title
Missing Staleness, Round Completeness, and Validity Checks on Chainlink `latestRoundData()` Allows Stale Price to Corrupt rsETH Exchange Rate - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary

`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but silently discards `roundId`, `updatedAt`, and `answeredInRound`, and performs no `price > 0` check. A stale or zero price propagates directly into `LRTOracle._getTotalEthInProtocol()`, corrupting the on-chain `rsETHPrice`. Because `updateRSETHPrice()` is publicly callable with no access control, an attacker can opportunistically trigger a price update at the moment a stale Chainlink feed is most favorable, enabling theft of user funds or yield.

## Finding Description

`ChainlinkPriceOracle.getAssetPrice()` (L49–55) discards all return values from `latestRoundData()` except the raw `price`:

```solidity
// contracts/oracles/ChainlinkPriceOracle.sol L52
(, int256 price,,,) = priceFeed.latestRoundData();
```

No check is made on `updatedAt` (heartbeat staleness), `answeredInRound >= roundId` (incomplete round), or `price > 0`. The sibling contract `ChainlinkOracleForRSETHPoolCollateral` (L27–32) applies all three guards, demonstrating the protocol is aware of the requirement but failed to apply it here.

The stale price flows through `LRTOracle.getAssetPrice()` → `_getTotalEthInProtocol()` (L336–343) → `_updateRsETHPrice()` (L250), where it is used to compute and persist `rsETHPrice`:

```solidity
// contracts/LRTOracle.sol L250
uint256 newRsETHPrice = (totalETHInProtocol - protocolFeeInETH).divWad(rsethSupply);
```

`updateRSETHPrice()` (L87–89) is `public` with only a `whenNotPaused` guard — no role restriction — so any external caller can trigger the price update at will.

The `pricePercentageLimit` guard in `_updateRsETHPrice()` (L256–266) provides only partial mitigation: it is bypassed entirely when `pricePercentageLimit == 0` (the default until explicitly set by admin), and even when set, it only compares against `highestRsethPrice` — not the true current market price — so a stale price within the deviation band passes unchecked.

**Exploit path (Scenario B — inflated stale price → direct fund theft):**
1. Chainlink's LST/ETH feed has a 24-hour heartbeat. The true LST price drops 1.5% but the on-chain feed still reports the old (higher) price.
2. Attacker observes the discrepancy off-chain.
3. Attacker calls `LRTOracle.updateRSETHPrice()` (no access control). `ChainlinkPriceOracle.getAssetPrice()` returns the stale inflated price; `rsETHPrice` is set above its true value.
4. Attacker immediately initiates a withdrawal via `LRTWithdrawalManager`, receiving a claim priced at the inflated `rsETHPrice`.
5. When Chainlink updates, the attacker's withdrawal claim was locked in at the inflated rate, extracting excess ETH from the pool at the expense of remaining rsETH holders.

**Exploit path (Scenario A — deflated stale price → yield theft):**
1. True LST price rises but Chainlink hasn't updated yet; on-chain feed reports a lower price.
2. Attacker calls `updateRSETHPrice()` to lock in the deflated `rsETHPrice`.
3. Attacker deposits LST collateral and receives more rsETH shares than the true value warrants.
4. When Chainlink updates and price corrects, attacker redeems at the true rate, extracting value from existing rsETH holders.

## Impact Explanation

**Scenario B** constitutes **direct theft of user funds (Critical)**: the attacker extracts more underlying ETH from the pool than their rsETH shares are worth, at the direct expense of other depositors. **Scenario A** constitutes **theft of unclaimed yield (High)**: existing rsETH holders are diluted by an attacker who mints excess shares against a temporarily deflated price. Both impacts are within the allowed scope.

## Likelihood Explanation

Chainlink feeds operate on a heartbeat (up to 24 hours for some LST feeds) plus a deviation threshold. During the heartbeat window, the on-chain price can diverge from the true market price by the full deviation band without triggering an update — this is a known, recurring condition, not a hypothetical. `updateRSETHPrice()` is permissionlessly callable, requiring no special privileges. An attacker needs only to monitor off-chain prices and call the function at the optimal moment, making this attack repeatable and low-cost.

## Recommendation

1. **Short term:** Add staleness and validity guards to `ChainlinkPriceOracle.getAssetPrice()`, mirroring the pattern in `ChainlinkOracleForRSETHPoolCollateral`:
   ```solidity
   (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
       priceFeed.latestRoundData();
   require(answeredInRound >= roundId, "Stale price");
   require(updatedAt != 0, "Incomplete round");
   require(price > 0, "Invalid price");
   require(block.timestamp - updatedAt <= MAX_STALENESS, "Price too old");
   ```
2. **Long term:** Restrict `updateRSETHPrice()` to trusted callers (e.g., a keeper with `MANAGER` role) or add a circuit-breaker that refuses to update the stored price if any individual feed is stale, preventing opportunistic timing attacks.

**Foundry fork test outline:**

```solidity
// Fork mainnet at a block where a Chainlink LST/ETH feed is near its heartbeat boundary
// (i.e., updatedAt is ~23 hours ago and true price has moved 1.5%)

function testStalePriceExploit() public {
    // 1. Warp time so the Chainlink feed is stale but not yet updated
    vm.warp(block.timestamp + 23 hours);

    // 2. Record attacker's rsETH balance and pool ETH balance
    uint256 attackerRsETH = rsETH.balanceOf(attacker);
    uint256 poolETHBefore = address(lrtDepositPool).balance;

    // 3. Attacker calls updateRSETHPrice() — no access control
    vm.prank(attacker);
    lrtOracle.updateRSETHPrice();

    // 4. Assert rsETHPrice is inflated vs. true price
    uint256 storedPrice = lrtOracle.rsETHPrice();
    assertGt(storedPrice, trueMarketPrice);

    // 5. Attacker initiates withdrawal at inflated price
    vm.prank(attacker);
    lrtWithdrawalManager.initiateWithdrawal(asset, attackerRsETH);

    // 6. Warp past unlock period, complete withdrawal
    vm.warp(block.timestamp + unlockPeriod);
    vm.prank(attacker);
    lrtWithdrawalManager.completeWithdrawal(asset);

    // 7. Assert attacker received more ETH than fair share
    assertGt(asset.balanceOf(attacker), fairShareAmount);
}
```

The entire attack path is permissionless and executable by any unprivileged external caller through standard deposit/withdrawal contract interactions.
