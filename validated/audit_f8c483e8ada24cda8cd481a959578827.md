### Title
Unenforced Mutual-Exclusion Invariant in DRep Expiry Update Allows Attacker-Controlled Expiry Extension — (File: `eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Certs.hs`)

---

### Summary

The comment on `updateVotingDRepExpiries` explicitly states that it "will execute in mutual-exclusion to the updates to the dormant DRep expiry, because if there are no proposals to vote on, there will be no votes either." This invariant is never enforced in the code. A Conway-era transaction is permitted to contain both new governance proposals and votes on existing proposals simultaneously. When this occurs, `updateDormantDRepExpiries` resets `vsNumDormantEpochs` to zero before `updateVotingDRepExpiries` reads it, causing the voting DRep to receive a longer expiry than the protocol intends — by exactly the number of dormant epochs that should have been deducted.

---

### Finding Description

In `conwayCertsTransition` (pre-hardfork path) and `conwayLedgerTransitionTRC` (post-hardfork path), the two DRep expiry update functions are chained sequentially:

```haskell
certState
  & updateDormantDRepExpiries tx currentEpoch
  & updateVotingDRepExpiries tx currentEpoch (pp ^. ppDRepActivityL)
``` [1](#0-0) 

`updateDormantDRepExpiries` fires when the transaction body contains at least one new governance proposal (`proposalProceduresTxBodyL` is non-empty). It calls `updateDormantDRepExpiry`, which reads `vsNumDormantEpochs = N`, bumps every registered DRep's stored expiry by `N`, and **resets `vsNumDormantEpochs` to zero**. [2](#0-1) [3](#0-2) 

`updateVotingDRepExpiries` fires when the transaction body contains DRep votes (`votingProceduresTxBodyL` is non-empty). It reads `numDormantEpochs` from the **already-modified** `certState` (i.e., after the reset), so it reads `0` instead of the original `N`. It then calls `computeDRepExpiry drepActivity currentEpoch 0`, which yields `currentEpoch + drepActivity` — the maximum possible expiry with no dormant deduction. [4](#0-3) 

The comment's mutual-exclusion claim rests on the reasoning that "if there are no proposals to vote on, there will be no votes either." This refers to *existing* proposals in the governance state, but the code gates `updateDormantDRepExpiries` on `proposalProceduresTxBodyL` — proposals submitted in the *current* transaction. These are entirely different conditions. A DRep can vote on existing proposals while simultaneously submitting new proposals in the same transaction body. The Conway ledger explicitly supports this: `GovSignal` carries both `gsVotingProcedures` and `gsProposalProcedures`, and the example transaction in the test suite demonstrates both fields populated together. [5](#0-4) 

The same dual-path bug exists in the post-hardfork LEDGER rule: [6](#0-5) 

`computeDRepExpiry` is defined as `currentEpoch + drepActivity - numDormantEpochs`. When `numDormantEpochs` is incorrectly read as `0`, the deduction is lost: [7](#0-6) 

---

### Impact Explanation

**Intended behavior** when a DRep votes (no proposals in same tx, `N` dormant epochs accumulated):
- Voting DRep expiry = `currentEpoch + drepActivity − N`

**Actual behavior** when a DRep votes AND submits proposals in the same tx:
- `updateDormantDRepExpiries` bumps all DRep expiries by `N`, resets counter to `0`
- `updateVotingDRepExpiries` reads `numDormantEpochs = 0`, overwrites voting DRep expiry to `currentEpoch + drepActivity`
- Net result: voting DRep expiry = `currentEpoch + drepActivity` — **N epochs longer than intended**

A DRep with sufficient delegated stake can remain "active" (non-expired) for `N` extra epochs beyond what the protocol permits. During those extra epochs:
- Their stake continues to count in the denominator of `dRepAcceptedRatio`, making governance actions harder to pass
- They can cast votes on governance actions (treasury withdrawals, protocol-parameter changes, hard-fork initiations, committee updates, constitutional changes) that they should no longer be eligible to vote on

If the DRep's continued participation is the deciding factor in ratifying a governance action — for example, a treasury withdrawal or a hard-fork initiation — this constitutes an unauthorized governance action being enacted, matching the **Critical** impact category.

---

### Likelihood Explanation

The attack requires only that:
1. At least one existing governance proposal is live (so the DRep has something to vote on)
2. The DRep submits a transaction containing both a new governance proposal and a vote on the existing proposal
3. `vsNumDormantEpochs > 0` at the time of submission (i.e., at least one dormant epoch has elapsed)

All three conditions are routinely satisfied on mainnet. Any registered DRep can craft such a transaction unilaterally. No privileged access, key compromise, or majority collusion is required. The attack is repeatable every time dormant epochs accumulate.

---

### Recommendation

The invariant claimed in the comment must be explicitly enforced. The simplest fix is to read `numDormantEpochs` from the **original** `certState` (before `updateDormantDRepExpiries` modifies it) and pass it explicitly to `updateVotingDRepExpiries`, or to restructure the two updates so that `updateVotingDRepExpiries` uses the pre-reset dormant epoch count:

```haskell
let numDormantEpochs = certState ^. certVStateL . vsNumDormantEpochsL
in certState
     & updateDormantDRepExpiries tx currentEpoch
     & updateVotingDRepExpiriesWithCount tx currentEpoch (pp ^. ppDRepActivityL) numDormantEpochs
     & certDStateL . accountsL %~ drainAccounts withdrawals
```

Alternatively, add an explicit assertion (analogous to the external report's recommendation) that `hasProposals && hasVotes` is impossible, and fail the transaction if both are present — though this would be a breaking change to the protocol.

---

### Proof of Concept

Setup: `drepActivity = 10`, `currentEpoch = 20`, `vsNumDormantEpochs = 5`. One existing governance proposal `P_old` is live. DRep `D` has `drepExpiry = 25`.

**Step 1 — Craft the transaction:**
```
tx.proposalProcedures = [P_new]   -- new InfoAction proposal
tx.votingProcedures   = {D: {P_old: VoteYes}}
```

**Step 2 — LEDGER/CERTS processes the transaction:**

`updateDormantDRepExpiries` fires (proposals non-empty):
- `numDormantEpochs = 5`
- All DRep expiries bumped by 5: `D.drepExpiry = 25 + 5 = 30`
- `vsNumDormantEpochs` reset to `0`

`updateVotingDRepExpiries` fires (votes non-empty):
- Reads `numDormantEpochs = 0` (already reset)
- `computeDRepExpiry 10 20 0 = 20 + 10 − 0 = 30`
- `D.drepExpiry` overwritten to `30`

**Step 3 — Compare to intended behavior (votes-only tx):**
- `computeDRepExpiry 10 20 5 = 20 + 10 − 5 = 25`
- `D.drepExpiry` should be `25`

**Result:** DRep `D` obtains expiry `30` instead of `25` — **5 epochs of unauthorized extended governance participation** — by including a cheap `InfoAction` proposal alongside their vote. This is repeatable whenever dormant epochs accumulate, and the magnitude scales with `vsNumDormantEpochs`. [4](#0-3) [8](#0-7)

updateDormantDRepExpiry ::
  -- | Current Epoch
  EpochNo ->
  VState era ->
  VState era
updateDormantDRepExpiry currentEpoch vState =
  if numDormantEpochs == EpochNo 0
    then vState
    else
      vState
        & vsNumDormantEpochsL .~ EpochNo 0
        & vsDRepsL %~ Map.map updateExpiry
  where
    numDormantEpochs = vState ^. vsNumDormantEpochsL
    updateExpiry =
      drepExpiryL
        %~ \currentExpiry ->
          let actualExpiry = binOpEpochNo (+) numDormantEpochs currentExpiry
           in if actualExpiry < currentEpoch
                then currentExpiry
                else actualExpiry
