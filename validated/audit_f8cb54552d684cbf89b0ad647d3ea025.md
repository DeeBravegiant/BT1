### Title
Harvester Can Unconditionally Redirect Farmer Block Reward Without Meeting CHIP-22 Fee Quality Threshold — (`chia/farmer/farmer_api.py`, `chia/farmer/farmer.py`)

---

### Summary

The farmer's `_process_respond_signatures` function in `chia/farmer/farmer_api.py` accepts `farmer_reward_address_override` from a harvester's `RespondSignatures` message and uses it as the `farmer_reward_address` in `DeclareProofOfSpace` without ever enforcing the CHIP-22 fee quality threshold. The only check — `notify_farmer_reward_taken_by_harvester_as_fee` — is a pure logging function that emits warnings but never blocks the override or raises an exception. Any connected harvester can therefore redirect the farmer's 1/8 XCH block reward to an arbitrary address on every won block, without meeting the fee quality gate.

---

### Finding Description

**Vulnerability class (from external report):** Missing active-status guard at the point of action — an entity that should be gated by a condition can still perform a privileged action because the condition is checked only for logging, not for enforcement.

**Chia analog:** The CHIP-22 fee-override mechanism is supposed to allow a harvester to take the farmer reward only when a cryptographic fee-quality threshold is satisfied. The threshold check exists in code but is never enforced; the override is accepted unconditionally at the point where it is applied.

**Execution path:**

1. A connected harvester sends `NewProofOfSpace` with a valid block-quality proof and `farmer_reward_address_override` set to an attacker-controlled puzzle hash, with `fee_info` absent or carrying a threshold the proof does not satisfy.

2. In `new_proof_of_space` (line 128–129), the farmer calls `notify_farmer_reward_taken_by_harvester_as_fee`. That function computes `fee_quality`, compares it to `fee_threshold`, and **only logs a warning** — it returns `None` and does not raise, does not set a flag, and does not prevent the farmer from continuing.

3. The farmer unconditionally sends `RequestSignatures` to the harvester (line 187).

4. The harvester responds with `RespondSignatures` carrying `farmer_reward_address_override`.

5. In `_process_respond_signatures` (lines 916–918):
   ```python
   farmer_reward_address = self.farmer.farmer_target
   if response.farmer_reward_address_override is not None:
       farmer_reward_address = response.farmer_reward_address_override
   ```
   There is **no fee-threshold check here**. The override is accepted unconditionally.

6. `DeclareProofOfSpace` is returned with the attacker's address as `farmer_reward_address`, and the full node pays the 1/8 farmer reward to that address.

**Additional escalation:** Because `_process_respond_signatures` reads `farmer_reward_address_override` from `RespondSignatures` independently of `NewProofOfSpace`, a harvester can send `NewProofOfSpace` with `farmer_reward_address_override = None` (bypassing even the logging call) and then inject the override only in `RespondSignatures`, making the diversion completely silent.

---

### Impact Explanation

Every block won by a plot managed by a malicious or compromised harvester results in the farmer's 1/8 XCH coinbase reward being paid to an address the harvester controls. The farmer receives nothing. This is a direct, permanent, per-block loss of XCH with no on-chain recourse once the block is confirmed. It falls squarely within:

> **High: Bypass of … authorization that enables … payout redirection … with direct security impact**

---

### Likelihood Explanation

Any harvester that is TCP-connected to the farmer can exploit this. In the CHIP-22 third-party harvester model (the exact scenario the feature targets), the farmer deliberately connects to external harvesters it does not fully control. No key material, admin access, or cryptographic break is required — only a valid proof of space good enough to win a block, which the harvester already possesses by design.

---

### Recommendation

In `_process_respond_signatures`, before accepting `response.farmer_reward_address_override`, re-derive `fee_quality` from the stored proof and signage-point data and verify it satisfies the threshold declared in `fee_info`. If the threshold is not met, or if `fee_info` is absent, reject the override and fall back to `self.farmer.farmer_target`. The existing `notify_farmer_reward_taken_by_harvester_as_fee` logic should be converted from a pure logging function into an enforcement function that returns a boolean (or raises), and its result must gate whether the override is applied.

    def notify_farmer_reward_taken_by_harvester_as_fee(
        self, sp: farmer_protocol.NewSignagePoint, proof_of_space: harvester_protocol.NewProofOfSpace
    ) -> None:
        """
        Apply a fee quality convention (see CHIP-22: https://github.com/Chia-Network/chips/pull/88)
        given the proof and signage point. This will be tested against the fee threshold reported
        by the harvester (if any), and logged.
        """
        assert proof_of_space.farmer_reward_address_override is not None

        challenge_str = str(sp.challenge_hash)

        ph_prefix = self.config["network_overrides"]["config"][self.config["selected_network"]]["address_prefix"]
        farmer_reward_puzzle_hash = encode_puzzle_hash(proof_of_space.farmer_reward_address_override, ph_prefix)

        self.log.info(
            f"Farmer reward for challenge '{challenge_str}' "
            + f"taken by harvester for reward address '{farmer_reward_puzzle_hash}'"
        )

        fee_quality = calculate_harvester_fee_quality(proof_of_space.proof.proof, sp.challenge_hash)
        fee_quality_rate = float(fee_quality) / float(0xFFFFFFFF) * 100.0

        if proof_of_space.fee_info is not None:
            fee_threshold = proof_of_space.fee_info.applied_fee_threshold
            fee_threshold_rate = float(fee_threshold) / float(0xFFFFFFFF) * 100.0

            if fee_quality <= fee_threshold:
                self.log.info(
                    f"Fee threshold passed for challenge '{challenge_str}': "
                    + f"{fee_quality_rate:.3f}%/{fee_threshold_rate:.3f}% ({fee_quality}/{fee_threshold})"
                )
            else:
                self.log.warning(
                    f"Invalid fee threshold for challenge '{challenge_str}': "
                    + f"{fee_quality_rate:.3f}%/{fee_threshold_rate:.3f}% ({fee_quality}/{fee_threshold})"
                )
                self.log.warning(
                    "Harvester illegitimately took a fee reward that "
                    + "did not belong to it or it incorrectly applied the fee convention."
                )
        else:
            self.log.warning(
                "Harvester illegitimately took reward by failing to provide its fee rate "
                + f"for challenge '{challenge_str}'. "
                + f"Fee quality was {fee_quality_rate:.3f}% ({fee_quality} or 0x{fee_quality:08x})"
            )
