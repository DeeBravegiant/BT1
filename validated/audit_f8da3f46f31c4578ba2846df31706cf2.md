### Title
`respond_verify_foreign_tx` Does Not Bind `response.payload_hash` to the Pending Request - (File: `crates/contract/src/lib.rs`)

### Summary

The `respond_verify_foreign_tx` contract method verifies that the submitted ECDSA signature is valid over the caller-supplied `response.payload_hash`, but never checks that `response.payload_hash` is the canonical hash of the pending `request`. A single malicious attested participant acting as the signing leader can substitute a legitimately-produced threshold signature for one pending foreign-tx request to resolve a completely different pending request, delivering a forged verification result to the caller.

### Finding Description

In `respond_verify_foreign_tx` the signature check is:

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;
// Check the signature is correct against the root public key
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,   // ← taken from response, not derived from request
    &secp_pk,
)
``` [1](#0-0) 

The `payload_hash` is taken verbatim from the caller-supplied `response.payload_hash`. The contract never checks that this hash equals `SHA-256(borsh(ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 { request: request.request, values: <observed_values> })))`.

Contrast this with the `respond` function for regular signatures, where the payload is derived from the stored request itself:

```rust
let payload_hash = request.payload.as_ecdsa().expect("Payload is not ECDSA");
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    payload_hash,   // ← derived from the request, not from the response
    &expected_public_key,
)
``` [2](#0-1) 

The `ForeignTxSignPayload` is defined as:

```rust
pub struct ForeignTxSignPayloadV1 {
    pub request: ForeignChainRpcRequest,  // includes tx_id
    pub values: Vec<ExtractedValue>,
}
``` [3](#0-2) 

So `payload_hash_B = SHA-256(borsh(ForeignTxSignPayload::V1({ request_B, values_B })))` is a hash that encodes `tx_id_B` and the extracted values for transaction B. The contract has no mechanism to verify that the submitted `payload_hash` encodes the `tx_id` from the pending `request`.

**Attack path (single malicious leader, no threshold collusion required):**

1. Two different `verify_foreign_transaction` requests are pending: `request_A` (for `tx_id_A`) and `request_B` (for `tx_id_B`).
2. The MPC network legitimately runs the threshold signing protocol for `request_B`, producing `(payload_hash_B, sig_B)` where `sig_B` is a valid threshold signature over `payload_hash_B`.
3. The malicious leader node calls `respond_verify_foreign_tx(request_A, { payload_hash: payload_hash_B, signature: sig_B })`.
4. The contract checks: is `sig_B` a valid signature over `payload_hash_B` under the root public key? **Yes** — the signature is genuine.
5. `resolve_yields_for(&mut self.pending_verify_foreign_tx_requests, &request_A, ...)` permanently resolves request A with `response_B`. [4](#0-3) 

The caller of `request_A` receives `{ payload_hash: payload_hash_B, signature: sig_B }` — a valid MPC signature over a payload that encodes `tx_id_B`, not `tx_id_A`.

The off-chain SDK helper `ForeignChainSignatureVerifier::verify_signature` does check `expected_payload_hash == response.payload_hash`: [5](#0-4) 

However, this check is in the SDK, not enforced by the contract. Bridge contracts that only verify the ECDSA signature validity (without reconstructing the expected `payload_hash` from known `values`) will accept the forged response. Furthermore, even a correctly-implemented bridge contract is permanently denied a valid response for `request_A` because the pending entry is removed from the map.

### Impact Explanation

**Forged foreign-chain verification / invalid bridge execution (High).**

A bridge contract that calls `verify_foreign_transaction(tx_id_B)` for a fraudulent or unconfirmed transaction can receive a valid MPC signature that was actually produced for a different, legitimate transaction `tx_id_A`. If the bridge contract does not independently reconstruct and compare the `payload_hash` (which requires knowing the extracted `values` in advance — information not available to the contract at request time), it will accept the forged attestation and execute the bridge action (e.g., unlock funds) for the fraudulent transaction.

Even for bridge contracts that do verify the `payload_hash` correctly, the attack permanently resolves the victim's pending request with a wrong response, consuming the yield slot and the deposit, with no path to recovery.

### Likelihood Explanation

- Requires exactly **one** malicious attested participant acting as the signing leader — strictly below the signing threshold.
- Requires two concurrent pending `verify_foreign_transaction` requests, which is routine in any active bridge.
- The malicious leader simply waits for the network to complete a legitimate signing session for any request, then redirects that response to a different pending request.
- No key compromise, no threshold collusion, no physical access, and no network-level attack is needed.

### Recommendation

The contract must derive the expected `payload_hash` from the pending `request` and verify that `response.payload_hash` matches it. Since the `values` (extracted chain data) are not stored on-chain, the simplest fix is to change the response submission to include the raw `values` alongside the signature, and have the contract recompute and compare the hash:

```rust
// In respond_verify_foreign_tx, after retrieving the pending request:
let expected_payload = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: request.request.clone(),
    values: response.values.clone(),  // submitted by the node
});
let expected_hash = expected_payload.compute_msg_hash()?;
if expected_hash != response.payload_hash {
    return Err(RespondError::PayloadHashMismatch.into());
}
```

Alternatively, the `ForeignTxSignPayload` hash construction can be split so that the `request` portion is hashed separately and verifiable by the contract without knowing the `values` (e.g., `payload_hash = SHA-256(request_hash || values_hash)`), allowing the contract to at minimum verify that the `request` component of the hash matches the pending request.

### Proof of Concept

1. Alice submits `verify_foreign_transaction({ tx_id: 0xAAAA, extractors: [BlockHash] })` — a legitimate bridge deposit.
2. Bob submits `verify_foreign_transaction({ tx_id: 0xBBBB, extractors: [BlockHash] })` — a fraudulent or double-spend transaction.
3. The MPC network runs the threshold signing protocol for Alice's request and produces `(payload_hash_A, sig_A)` where `payload_hash_A = SHA-256(borsh({ request: {tx_id: 0xAAAA, ...}, values: [block_hash_A] }))`.
4. The malicious leader calls:
   ```
   respond_verify_foreign_tx(
     request = { tx_id: 0xBBBB, ... },   // Bob's request
     response = { payload_hash: payload_hash_A, signature: sig_A }  // Alice's response
   )
   ```
5. The contract verifies `sig_A` over `payload_hash_A` — valid. Bob's request is resolved with Alice's response.
6. Bob's bridge contract receives `{ payload_hash: payload_hash_A, signature: sig_A }`. If it only checks signature validity, it accepts the attestation and unlocks funds for the fraudulent `tx_id: 0xBBBB`.
7. Alice's request is permanently consumed; she receives no valid response.
