Audit Report

## Title
Rewards Permanently Locked When `totalKernelStaked` Drops to Zero During Active Reward Period - (File: contracts/KERNEL/KernelDepositPool.sol)

## Summary
When all stakers call `initiateWithdrawal()` during an active reward window, `totalKernelStaked` immediately drops to zero. The `updateReward` modifier advances `updatedAt` to the current time while `rewardPerToken()` freezes accumulation, causing all rewards scheduled for the zero-staking interval to be permanently locked in the contract. No on-chain recovery mechanism exists.

## Finding Description
`rewardPerToken()` at L408-414 short-circuits when `totalKernelStaked == 0`, returning `rewardPerTokenStored` unchanged. The `updateReward` modifier at L232-242 unconditionally sets `updatedAt = lastTimeRewardApplicable()` regardless of whether `totalKernelStaked` is zero. `initiateWithdrawal()` at L325-326 decrements `totalKernelStaked` immediately at initiation time, not at claim time.

When the last staker calls `initiateWithdrawal()` at time T1, `updateReward` runs with `totalKernelStaked` still nonzero (the decrement happens after the modifier), correctly snapshotting rewards up to T1. After the decrement, `totalKernelStaked == 0`. From T1 to T2 (next stake), `rewardPerToken()` returns `rewardPerTokenStored` unchanged. When a new staker calls `stake()` at T2, `updateReward` runs with `totalKernelStaked` still 0 (the increment happens after the modifier), so `rewardPerToken()` again returns `rewardPerTokenStored`, and `updatedAt` is advanced to T2. The rewards for the interval T1→T2 (`rewardRate * (T2 - T1)`) are silently discarded — `updatedAt` skips over that window without accumulating anything.

If `notifyRewardAmount` is called at T2 < `finishAt`, `remaining = (finishAt - T2) * rewardRate` covers only T2→finishAt; the T1→T2 slice is not rolled in. If T2 >= `finishAt`, the entire undistributed balance from the prior period is lost. `notifyRewardAmount` also reverts if `totalKernelStaked == 0` (L570), so the admin cannot even re-queue rewards until someone stakes, by which point `updatedAt` has already been advanced past the gap.

No `recoverERC20`, `rescueToken`, or equivalent function exists in the contract (confirmed by full code review of L1-621). The contract's own NatSpec at L14-22 acknowledges this limitation but relies solely on an off-chain operational assumption that cannot be enforced on-chain.

## Impact Explanation
Reward tokens scheduled for distribution during the zero-staking interval are permanently locked in the contract with no recovery path. This is a concrete instance of **Medium — Permanent freezing of unclaimed yield**: the tokens remain in the contract balance but can never be distributed to stakers or recovered by the admin.

## Likelihood Explanation
Any staker can call `initiateWithdrawal()` at any time without permission. A coordinated or uncoordinated mass exit during a reward window (e.g., in response to a market event, a better yield opportunity, or a depeg scare) is a realistic scenario. The `withdrawalDelay` only delays the token transfer, not the `totalKernelStaked` decrement, so the gap opens immediately upon initiation. The condition is repeatable across every reward period.

## Recommendation
Introduce an `unallocatedRewards` accumulator. In `rewardPerToken()` (or a dedicated hook called from `updateReward`), when `totalKernelStaked == 0`, accumulate `rewardRate * (lastTimeRewardApplicable() - updatedAt)` into `unallocatedRewards` instead of discarding it. In `notifyRewardAmount`, add `unallocatedRewards` to `receivedAmount` before computing the new `rewardRate`, then reset `unallocatedRewards = 0`. Alternatively, add an admin-callable `recoverUnallocatedRewards()` that re-queues any reward token balance in excess of what is owed to current stakers, providing a fallback recovery path.

       // remaining = (finishAt - T2) * rewardRate  ← covers only T2→finishAt
