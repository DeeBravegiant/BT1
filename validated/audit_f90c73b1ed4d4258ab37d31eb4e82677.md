Audit Report

## Title
Missing Chainlink Staleness Check Allows Stale Price to Corrupt rsETH Exchange Rate — (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary

`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards every return value except `price`, with no check on `updatedAt`, no heartbeat comparison, and no `answeredInRound` guard. A stale Chainlink feed is consumed as fresh, corrupting the `rsETHPrice` used for all deposits and withdrawals. The `pricePercentageLimit` circuit breaker in `LRTOracle` does not mitigate this because a frozen stale price produces no measurable delta from the previously stored `rsETHPrice`, so the circuit breaker never fires.

## Finding Description

In `contracts/oracles/ChainlinkPriceOracle.sol` at L52–54, `getAssetPrice()` reads:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

All four other return values — `roundId`, `startedAt`, `updatedAt`, `answeredInRound` — are silently discarded. No staleness window is enforced.

This oracle is the `IPriceFetcher` registered in `LRTOracle` for each supported LST asset. `LRTOracle.getAssetPrice()` (L156–158) delegates directly to it. `_updateRsETHPrice()` (L214–316) calls `_getTotalEthInProtocol()` (L331–349), which aggregates asset balances weighted by prices from this oracle, and stores the result as `rsETHPrice`. That stored price governs both minting and redemption.

The `pricePercentageLimit` circuit breaker (L252–291) compares `newRsETHPrice` against `highestRsethPrice`. When a feed goes stale, the frozen price produces a `newRsETHPrice` nearly identical to the previously stored `rsETHPrice`, so the delta is below any reasonable threshold and the circuit breaker does not trigger. The protocol continues accepting the stale price indefinitely.

By contrast, `contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol` (L27–32) at least checks `answeredInRound < roundID` and `timestamp == 0`, confirming the protocol is aware of the need for oracle validation — yet `ChainlinkPriceOracle` has no equivalent guard.

`updateRSETHPrice()` (L87–89) is a public function with no access control, so any external caller can trigger the price update path.

## Impact Explanation

**Critical — Direct theft of user funds / Protocol insolvency.**

If a supported LST feed goes stale while the real market price has fallen:
1. `getAssetPrice()` returns the old (inflated) stale price.
2. `_getTotalEthInProtocol()` computes a `totalETHInProtocol` higher than the actual on-chain value.
3. `rsETHPrice` is set above its true value.
4. Any user redeeming rsETH at this inflated rate withdraws more ETH than their fair share, draining funds from remaining holders and pushing the protocol toward insolvency.

The reverse (stale price above actual) causes depositors to receive excess rsETH, constituting theft of unclaimed yield from existing holders.

## Likelihood Explanation

Chainlink feeds can and do go stale: sequencer downtime on L2s, feed deprecations, and extreme network congestion are documented real-world events. The protocol supports multiple LST assets, each with its own feed and heartbeat; any single feed going stale is sufficient. No privileged access is required — `updateRSETHPrice()` is public and callable by any EOA or contract. The exploit is repeatable for as long as the feed remains stale.

## Recommendation

Add a configurable `maxStaleness` (heartbeat) per asset feed and revert if the price is stale:

```solidity
mapping(address asset => uint256 maxStaleness) public maxStalenessFor;

function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);
    (, int256 price,, uint256 updatedAt,) = priceFeed.latestRoundData();
    if (block.timestamp - updatedAt > maxStalenessFor[asset]) revert StalePrice();
    if (price <= 0) revert InvalidPrice();
    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

Set `maxStalenessFor[asset]` at feed registration time in `updatePriceFeedFor()`, matching each feed's documented heartbeat. Different feeds have different update frequencies (e.g., 1 hour vs. 24 hours), so per-feed configuration is necessary.

## Proof of Concept

1. Protocol supports stETH with a Chainlink stETH/ETH feed (heartbeat: 1 hour). `rsETHPrice` is currently set based on a stETH price of 0.9999 ETH.
2. The Chainlink feed stops updating (e.g., sequencer downtime). Last recorded price: 0.9999 ETH per stETH.
3. Real market price of stETH drops to 0.95 ETH (depeg event). The feed remains frozen at 0.9999.
4. Attacker calls `LRTOracle.updateRSETHPrice()` (public, no access control, L87).
5. `ChainlinkPriceOracle.getAssetPrice(stETH)` returns 0.9999 — no revert, no staleness check (L52).
6. `_getTotalEthInProtocol()` computes TVL using 0.9999 instead of 0.95, inflating it by ~5%.
7. `newRsETHPrice` is ~5% above true value. The delta from the previously stored `rsETHPrice` (also based on 0.9999) is near zero, so `pricePercentageLimit` does not trigger.
8. `rsETHPrice` is stored at the inflated value (L313).
9. Attacker redeems rsETH via `LRTWithdrawalManager`, receiving ~5% more ETH than their fair share.
10. Honest holders are left with a protocol that cannot cover their redemptions.

**Foundry fork test plan:**
- Fork mainnet/testnet with a live stETH/ETH Chainlink feed.
- Mock `latestRoundData()` to return a price with `updatedAt = block.timestamp - 2 hours` (past any reasonable heartbeat).
- Call `updateRSETHPrice()` as an unprivileged address.
- Assert `rsETHPrice` is set to the stale value without revert.
- Simulate a redemption and assert the redeemer receives more ETH than their proportional share of actual protocol assets.
