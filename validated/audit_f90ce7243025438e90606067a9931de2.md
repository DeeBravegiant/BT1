Audit Report

## Title
`rewardPerToken()` Increment Rounds to Zero with Low-Decimal Reward Tokens and Large Stake — (`contracts/KERNEL/KernelDepositPool.sol`)

## Summary

`KernelDepositPool` stores `rewardRate` in raw token-wei per second with no precision scaling. When `rewardRate` is small (e.g., USDC at 6 decimals) and `totalKernelStaked` is large (e.g., 1 M KERNEL = 1e24 wei), the per-update increment `rewardRate * timeDelta * 1e18 / totalKernelStaked` evaluates to zero in Solidity integer division for any `timeDelta` below ~2,597 seconds. Rewards that round to zero in each `updateReward` call are never added to `rewardPerTokenStored` and cannot be recovered, permanently freezing a portion of deposited yield.

## Finding Description

`notifyRewardAmount` assigns `rewardRate` without any precision multiplier:

```solidity
rewardRate = receivedAmount / duration;   // L580
``` [1](#0-0) 

`rewardPerToken()` then applies `DECIMAL_PRECISION = 1e18` only at accumulation time:

```solidity
return rewardPerTokenStored
    + (rewardRate * (lastTimeRewardApplicable() - updatedAt) * DECIMAL_PRECISION)
    / totalKernelStaked;
``` [2](#0-1) 

For USDC rewards (6 decimals), 1,000 USDC = 1e9 wei, 30-day duration:
- `rewardRate = 1e9 / 2_592_000 = 385`
- With `totalKernelStaked = 1e24` (1 M KERNEL): `385 * 1 * 1e18 / 1e24 = 0`

The increment is non-zero only when `timeDelta ≥ 1e24 / (385 * 1e18) ≈ 2,597 s`. Every `updateReward` call within that window silently discards the accrued rewards.

The `updateReward` modifier fires on every user-facing call — `stake()`, `stakeFor()`, `initiateWithdrawal()`, and `getReward()`: [3](#0-2) [4](#0-3) [5](#0-4) 

No admin sweep or recovery function exists for undistributed reward tokens. The `initialize()` function places no constraint on the decimals of `rewardsToken`: [6](#0-5) 

## Impact Explanation

**Medium — Permanent freezing of unclaimed yield.** Reward tokens transferred into the contract via `notifyRewardAmount` are irrecoverably stuck whenever per-update increments consistently round to zero. The fraction lost scales with interaction frequency: in the concrete scenario above, users interacting every minute over 30 days lose ≈ 99.96% of deposited USDC rewards. There is no sweep path for the stranded tokens.

## Likelihood Explanation

Medium. USDC is a natural reward token for a KERNEL staking pool. No privileged compromise is required — the loss occurs automatically as ordinary users call `stake()`, `getReward()`, etc. A protocol with meaningful TVL trivially reaches 1 M KERNEL staked (1e24 wei), the threshold at which per-second increments round to zero for USDC rewards. The `rewardsToken` is unconstrained at initialization, so the condition can arise in any deployment that uses a sub-18-decimal token.

## Recommendation

Scale `rewardRate` by `DECIMAL_PRECISION` at assignment time (the canonical Synthetix approach), and remove the multiplier from `rewardPerToken()`:

```solidity
// notifyRewardAmount:
rewardRate = (receivedAmount * DECIMAL_PRECISION) / duration;

// rewardPerToken():
return rewardPerTokenStored
    + (rewardRate * (lastTimeRewardApplicable() - updatedAt))
    / totalKernelStaked;

// earned() is unchanged — DECIMAL_PRECISION cancels correctly
```

This eliminates the rounding-to-zero condition for any realistic combination of reward token decimals and stake size.

## Proof of Concept

1. `setRewardsDuration(2_592_000)` (30 days).
2. 1,000,000 KERNEL staked → `totalKernelStaked = 1e24`.
3. `notifyRewardAmount(1_000_000_000)` (1,000 USDC). `rewardRate = 385`.
4. User A calls `stake(1)` at `t = 0`. `updateReward`: increment = `385 * 0 * 1e18 / 1e24 = 0`.
5. 60 s later, User B calls `getReward()`. `updateReward`: increment = `385 * 60 * 1e18 / 1e24 = 23_100 / 1e6 = 0`.
6. First non-zero increment occurs only after 2,597 s. All rewards that should have accrued during the first 2,596 s (`385 * 2596 ≈ 999,460` USDC wei) are permanently lost.
7. With users interacting every minute over 30 days, total frozen ≈ `(2596/2597) * 1000 USDC ≈ 999.6 USDC`.

**Foundry test plan:** Deploy `KernelDepositPool` with a mock 6-decimal ERC-20 as `rewardsToken`. Stake 1e24 KERNEL. Call `notifyRewardAmount(1e9)`. Warp forward 60 s and call `getReward()` in a loop. Assert `rewardPerTokenStored` remains 0 after each call until `timeDelta ≥ 2597 s`. Assert the contract's reward token balance exceeds the sum of all claimed rewards at period end.
