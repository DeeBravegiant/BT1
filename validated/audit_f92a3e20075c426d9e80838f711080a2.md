### Title
`respond_verify_foreign_tx` Accepts Replayed MPC Signatures Across Different Foreign-Chain Requests — (`File: crates/contract/src/lib.rs`)

### Summary

`respond_verify_foreign_tx` verifies that `response.signature` is a valid MPC signature over `response.payload_hash`, but never checks that `response.payload_hash` is the correct hash for the submitted `request`. A single Byzantine attested MPC participant (below the signing threshold) can replay a `(payload_hash, signature)` pair produced for a prior foreign-chain verification request to resolve a completely different pending request, delivering a forged attestation to the caller.

### Finding Description

`respond_verify_foreign_tx` in `crates/contract/src/lib.rs` performs the following signature check:

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;
// Check the signature is correct against the root public key
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,
)
.is_ok()
``` [1](#0-0) 

The contract only checks that `response.signature` is a valid MPC signature over the caller-supplied `response.payload_hash`. It does **not** recompute or verify that `response.payload_hash` equals `ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 { request, values }).compute_msg_hash()` for the specific `request` being resolved.

The `VerifyForeignTransactionRequest` struct contains only `(request, domain_id, payload_version)` — no binding to a specific `payload_hash`: [2](#0-1) 

The `VerifyForeignTransactionResponse` carries the `payload_hash` as a free field supplied by the caller of `respond_verify_foreign_tx`: [3](#0-2) 

The SDK's `ForeignChainSignatureVerifier::verify_signature` does perform the correct binding check (recomputing the expected hash from `request` + `expected_extracted_values`), but this check lives entirely off-chain in the SDK, not in the contract: [4](#0-3) 

The contract resolves the yield and removes the pending entry unconditionally once the signature check passes: [5](#0-4) 

### Impact Explanation

A single Byzantine attested MPC participant can:

1. Observe a legitimately completed `respond_verify_foreign_tx` call for request A (e.g., Bitcoin tx_id=0xAA), capturing `(payload_hash_A, sig_A)` from the on-chain transaction.
2. Wait for a new `verify_foreign_transaction` request B (e.g., Bitcoin tx_id=0xBB) to enter the pending map.
3. Call `respond_verify_foreign_tx(request_B, { payload_hash: payload_hash_A, signature: sig_A })`.
4. The contract accepts the call: `sig_A` is a valid MPC threshold signature over `payload_hash_A`, so `signature_is_valid = true`.
5. The pending yield for request B is resolved with `payload_hash_A` and `sig_A` — an attestation for the wrong transaction.
6. The legitimate MPC nodes' correct response for request B arrives later but finds no pending entry (already consumed), so it is silently discarded.

The caller of request B receives a `VerifyForeignTransactionResponse` containing a valid MPC signature that attests to a **different** foreign-chain transaction than the one they requested. Any bridge contract that does not independently re-verify the `payload_hash` against its own expected transaction data (e.g., an Omnibridge inbound flow) will accept this forged attestation and execute an unauthorized action (e.g., minting tokens for a transaction that was never verified).

This matches the allowed impact: **Cross-chain replay, forged foreign-chain verification, or participant/attestation authorization bypass that causes invalid bridge execution or double-spend conditions.**

### Likelihood Explanation

The attacker is a single attested MPC participant — below the signing threshold. No threshold collusion is required. The attacker only needs to observe a prior on-chain `respond_verify_foreign_tx` transaction (public on NEAR) and submit a new call before the legitimate nodes respond to the target request. The window is the entire duration the request is pending (bounded by the yield timeout). This is a realistic race condition for any participant monitoring the chain.

### Recommendation

In `respond_verify_foreign_tx`, the contract must verify that `response.payload_hash` is consistent with the submitted `request`. Since the contract does not know the extracted values, the fix is to include the `request` identifier (e.g., a hash of the `VerifyForeignTransactionRequest`) in the signed payload so that a signature produced for request A is cryptographically bound to A and cannot satisfy request B.

Concretely, the `ForeignTxSignPayload` should commit to the `VerifyForeignTransactionRequest` (chain, tx_id, extractors, domain_id, payload_version) as part of the signed message. The contract can then recompute the expected prefix of the payload hash from the `request` argument and reject any response whose `payload_hash` does not share that prefix — or, equivalently, require that the `payload_hash` be derived deterministically from `(request, extracted_values)` and verify the request portion on-chain.

### Proof of Concept

1. Alice submits `verify_foreign_transaction` for Bitcoin tx_id=0xAA. MPC nodes respond legitimately; `respond_verify_foreign_tx(req_AA, {payload_hash_AA, sig_AA})` is accepted and emitted on-chain.
2. Bob submits `verify_foreign_transaction` for Bitcoin tx_id=0xBB. The request enters `pending_verify_foreign_tx_requests`.
3. Mallory (a single attested MPC participant) observes `(payload_hash_AA, sig_AA)` from step 1 and calls `respond_verify_foreign_tx(req_BB, {payload_hash: payload_hash_AA, signature: sig_AA})`.
4. The contract checks: is `sig_AA` a valid MPC signature over `payload_hash_AA`? Yes. Does `req_BB` have a pending entry? Yes. The yield is resolved with `{payload_hash_AA, sig_AA}`.
5. Bob's NEAR contract receives `{payload_hash_AA, sig_AA}` — a valid MPC signature attesting to tx_id=0xAA, not 0xBB.
6. If Bob's bridge contract mints tokens based on this response without re-verifying the payload_hash, it mints tokens for a transaction it never verified.
7. The legitimate MPC nodes' correct response for req_BB arrives and is silently dropped (no pending entry).
