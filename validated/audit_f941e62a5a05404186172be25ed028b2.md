Audit Report

## Title
Rewards Permanently Locked When `totalKernelStaked` Drops to Zero Mid-Period - (File: contracts/KERNEL/KernelDepositPool.sol)

## Summary
`KernelDepositPool` guards `notifyRewardAmount` against a zero-staking start, but provides no protection against all stakers exiting during an active reward window. When `totalKernelStaked` reaches zero mid-period, `rewardPerToken()` stops advancing while `updatedAt` is still pushed forward by the `updateReward` modifier, permanently skipping the rewards that accrued during the zero-staking interval with no recovery path.

## Finding Description
`rewardPerToken()` at line 409 short-circuits when `totalKernelStaked == 0`, returning the unchanged `rewardPerTokenStored`. The `updateReward` modifier at line 233–234 always writes `updatedAt = lastTimeRewardApplicable()` regardless of whether `rewardPerTokenStored` advanced. When the next user interaction occurs after a zero-staking gap, `updatedAt` has already been pushed past the gap, so the `rewardRate * (lastTimeRewardApplicable() - updatedAt)` term in `rewardPerToken()` never accounts for the skipped interval. The rewards that should have been distributed during that interval remain as surplus `rewardsToken` balance in the contract with no sweep, rescue, or re-injection function.

The `notifyRewardAmount` guard at line 570 (`if (totalKernelStaked == 0) revert NoStakedTokens()`) only prevents starting a new reward period with zero stakers; it does not prevent stakers from exiting during an already-active period. `initiateWithdrawal` at line 320 decrements `totalKernelStaked` unconditionally with no floor check.

## Impact Explanation
Rewards distributed during the zero-staking interval are permanently unrecoverable — no function exists to sweep, redistribute, or refund them. This constitutes **permanent freezing of unclaimed yield**, a Medium-severity impact per the allowed scope.

## Likelihood Explanation
Any scenario where all stakers exit during an active reward window triggers the bug. A single large staker who is the sole depositor can call `initiateWithdrawal` for their full balance, wait out the `withdrawalDelay` (up to 30 days), and claim, leaving `totalKernelStaked == 0` for the remainder of the reward period. No admin action can prevent this once the withdrawal is initiated. The scenario is realistic in early-deployment phases with few stakers.

## Recommendation
1. In `initiateWithdrawal`, track the cumulative time during which `totalKernelStaked == 0` and extend `finishAt` by that duration when staking resumes, so no rewards are skipped.
2. Alternatively, add an admin-callable rescue function that can reclaim unallocated rewards (i.e., `rewardsToken.balanceOf(address(this))` minus the sum of all `rewards[user]` balances) after a reward period ends.
3. Enforce a non-zero minimum staked floor on-chain rather than relying solely on the operational convention documented in the NatSpec comment.

## Proof of Concept
1. Admin calls `setRewardsDuration(7 days)`, then stakes 1 wei to satisfy the guard, then calls `notifyRewardAmount(1000e18)`. `rewardRate = 1000e18 / 7 days`.
2. Alice stakes `100e18`; `totalKernelStaked = 100e18 + 1 wei`.
3. Admin's 1 wei and Alice's full balance are withdrawn via `initiateWithdrawal`. After `withdrawalDelay` passes, both call `claimWithdrawal`. `totalKernelStaked = 0`.
4. For the remaining ~4 days, every call to `rewardPerToken()` returns `rewardPerTokenStored` unchanged. Each `updateReward` invocation advances `updatedAt` without advancing `rewardPerTokenStored`.
5. `≈ 571e18` reward tokens (`rewardRate * 4 days`) remain permanently locked. Calling `getReward` for any address yields 0 for this interval; no admin function can recover the tokens. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

    function initiateWithdrawal(uint256 _amount) external nonReentrant updateReward(msg.sender) {
        if (_amount == 0) revert AmountZero();
        if (balanceOf[msg.sender] < _amount) revert InsufficientStakedBalance();
        if (userWithdrawalIds[msg.sender].length >= maxNumberOfWithdrawalsPerUser) revert WithdrawalLimitReached();

        balanceOf[msg.sender] -= _amount;
        totalKernelStaked -= _amount;

        // Create a withdrawal record
        uint256 withdrawalId = ++withdrawalCounter;
        uint256 unlockTime = block.timestamp + withdrawalDelay;

        withdrawals[withdrawalId] = Withdrawal({
            user: msg.sender, amount: _amount, unlockTime: unlockTime, claimed: false, withdrawalId: withdrawalId
        });
        userWithdrawalIds[msg.sender].push(withdrawalId);

        emit WithdrawalInitiated(msg.sender, _amount, withdrawalId, unlockTime);
    }
