Audit Report

## Title
Stale `rsETHPrice` Used Alongside Live `getAssetPrice()` in Mint Calculation Allows Depositors to Extract Excess rsETH — (`contracts/LRTDepositPool.sol`)

## Summary

`LRTDepositPool.getRsETHAmountToMint()` divides a live Chainlink-backed asset price by a cached `rsETHPrice` state variable that is only updated on explicit calls to `updateRSETHPrice()`. Because neither `depositAsset()` nor `depositETH()` refreshes `rsETHPrice` before computing the mint amount, any depositor who acts in the window between a Chainlink heartbeat update and the next `updateRSETHPrice()` call receives more rsETH than their deposit is worth, diluting the yield accrued by existing holders.

## Finding Description

`getRsETHAmountToMint()` computes:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [1](#0-0) 

`getAssetPrice()` is a live, stateless call that reads directly from the underlying Chainlink price feed on every invocation:

```solidity
function getAssetPrice(address asset) public view onlySupportedOracle(asset) returns (uint256) {
    return IPriceFetcher(assetPriceOracle[asset]).getAssetPrice(asset);
}
``` [2](#0-1) 

`rsETHPrice`, by contrast, is a stored state variable updated only inside `_updateRsETHPrice()`:

```solidity
uint256 public override rsETHPrice;
``` [3](#0-2) 

```solidity
rsETHPrice = newRsETHPrice;
``` [4](#0-3) 

`_updateRsETHPrice()` itself computes `totalETHInProtocol` using the same live `getAssetPrice()` calls, so the correct `rsETHPrice` at any moment is `totalETHInProtocol_live / rsethSupply`. The cached `rsETHPrice` lags behind whenever a Chainlink feed updates.

Neither `depositAsset()` nor `depositETH()` calls `updateRSETHPrice()` before computing the mint amount: [5](#0-4) [6](#0-5) 

`_beforeDeposit()` calls `getRsETHAmountToMint()` directly with no price refresh: [7](#0-6) 

**Root cause:** The numerator (`getAssetPrice`) reflects the current Chainlink price immediately, while the denominator (`rsETHPrice`) reflects the price at the last explicit `updateRSETHPrice()` call. When an asset's Chainlink price rises, the numerator increases while the denominator remains at the old lower value, inflating the mint amount.

**Why existing checks are insufficient:** The `minRSETHAmountExpected` slippage guard only protects the depositor from receiving *less* than expected — it does not prevent over-minting. The `pricePercentageLimit` guard inside `_updateRsETHPrice()` only applies when that function is actually called, which is precisely what the attacker avoids triggering.

## Impact Explanation

**High — Theft of unclaimed yield.**

When a supported asset's Chainlink price increases, the price appreciation represents yield that has accrued to existing rsETH holders (their rsETH is now backed by more ETH). A depositor who acts before `updateRSETHPrice()` is called receives rsETH at the stale (lower) price, effectively capturing a portion of that accrued yield. Concretely:

- Pre-update: 100 stETH in protocol, 95.24 rsETH outstanding → `rsETHPrice` = 1.05
- Chainlink updates stETH/ETH from 1.00 → 1.02; correct new `rsETHPrice` = 102/95.24 ≈ 1.071
- Attacker deposits 100 stETH; receives `100 × 1.02 / 1.05 ≈ 97.14 rsETH` (stale denominator)
- Correct mint at updated price: `100 × 1.02 / 1.071 ≈ 95.24 rsETH`
- Attacker receives ~1.9 excess rsETH, each backed by protocol TVL not contributed by the attacker
- Existing holders' per-rsETH TVL claim is permanently diluted

This is not a theoretical rounding error; the magnitude scales with deposit size and the price delta between Chainlink updates.

## Likelihood Explanation

Chainlink price feeds update on a heartbeat schedule (typically every 1 hour or on a 0.5% deviation threshold). The window between a Chainlink update and the next `updateRSETHPrice()` call is always non-zero and can span many blocks. `updateRSETHPrice()` is a public function but is not called atomically with deposits. Any unprivileged external user can:

1. Monitor the Chainlink feed on-chain for a price increase
2. Immediately call `depositAsset()` or `depositETH()` in the same or next block
3. Deliberately refrain from calling `updateRSETHPrice()` first

No front-running of other users, no flash loans, no privileged access, and no victim mistakes are required. The condition is repeatable on every Chainlink heartbeat update.

## Recommendation

Call `updateRSETHPrice()` (or an internal equivalent) at the start of `depositAsset()` and `depositETH()` before computing the mint amount, ensuring `rsETHPrice` is always consistent with the live asset prices used in the same calculation. Alternatively, compute `rsETHPrice` on-the-fly inside `getRsETHAmountToMint()` using the same live `getAssetPrice()` calls rather than reading the cached storage variable, eliminating the staleness window entirely.

## Proof of Concept

**Foundry fork test outline:**

```solidity
// 1. Fork mainnet at a block where rsETHPrice is set.
// 2. Record rsETHPrice = lrtOracle.rsETHPrice() (e.g., 1.05e18).
// 3. Warp/mock the Chainlink stETH/ETH feed to return a higher price (e.g., 1.02e18 → simulate heartbeat).
// 4. Assert lrtOracle.getAssetPrice(stETH) == 1.02e18 (live).
// 5. Assert lrtOracle.rsETHPrice() == 1.05e18 (stale, unchanged).
// 6. Compute expected correct rsETHPrice = lrtOracle._getTotalEthInProtocol() / rsETH.totalSupply().
// 7. Call depositPool.depositAsset(stETH, 100e18, 0, "") as attacker.
// 8. Assert attacker rsETH balance == 100e18 * 1.02e18 / 1.05e18 ≈ 97.14e18.
// 9. Call updateRSETHPrice(); compute new rsETHPrice (≈ 1.071e18).
// 10. Assert attacker's rsETH is worth more ETH than the 100 stETH deposited,
//     confirming extraction of existing holders' yield.
```

The test demonstrates that steps 8 and 10 together prove over-minting at the expense of pre-existing rsETH holders, satisfying the High / theft-of-unclaimed-yield impact criterion.
