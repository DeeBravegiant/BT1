### Title
Sequential Integer Division in `calculate_reward` Causes Systematic Validator Reward Underflow - (File: chain/epoch-manager/src/reward_calculator.rs)

### Summary
`RewardCalculator::calculate_reward` performs two sequential integer divisions when computing each validator's reward. This causes a compounding precision loss that systematically underpays every eligible validator every epoch, causing `epoch_actual_reward` (the `minted_amount` stored in `EpochInfo`) to be less than `epoch_total_reward`. The discrepancy accumulates across epochs and across validators, causing a permanent, growing imbalance between the `total_supply` tracked in block headers and the actual sum of all account balances.

### Finding Description

In `calculate_reward`, the per-validator reward is computed as:

```rust
Balance::from_yoctonear(
    (U512::from(epoch_validator_reward.as_yoctonear())
        * U512::from(uptime_numer)
        * U512::from(stake.as_yoctonear())
        / U512::from(uptime_denum)
        / U512::from(total_stake.as_yoctonear()))
    .as_u128(),
)
``` [1](#0-0) 

The two divisions are performed sequentially: first `/ uptime_denum`, then `/ total_stake`. Each division independently truncates (floors) the intermediate result. The correct formula requires a single combined division:

```
reward = epoch_validator_reward * uptime_numer * stake / (uptime_denum * total_stake)
```

but the code computes:

```
reward = floor(floor(epoch_validator_reward * uptime_numer * stake / uptime_denum) / total_stake)
```

The inner `floor` introduces a remainder `r₁ < uptime_denum` that is discarded before the outer division. The outer division then discards a second remainder `r₂ < total_stake`. The total loss per validator per epoch is up to `uptime_denum + total_stake - 2` yoctoNEAR, multiplied by the number of eligible validators.

This is the same class of bug as the reported `MetaManager::unstake` issue: integer division is applied in stages rather than deferring all divisions to the end, causing each intermediate truncation to silently discard value.

The `epoch_actual_reward` accumulator sums the individually-truncated rewards:

```rust
epoch_actual_reward = epoch_actual_reward.checked_add(reward).unwrap();
``` [2](#0-1) 

This sum is returned as `minted_amount` and stored in `EpochInfo`: [3](#0-2) 

The `minted_amount` is then used by `verify_total_supply_checked` to validate the `total_supply` field in every epoch-boundary block header: [4](#0-3) 

called from block acceptance in `chain.rs`: [5](#0-4) 

Because `minted_amount` is itself computed with the truncated sum, the `total_supply` in block headers is accepted as valid even though it is systematically lower than the true inflation the protocol intends to mint. The discarded yoctoNEAR is neither credited to any account nor burned — it simply disappears, causing a permanent divergence between the protocol's intended token supply and the actual supply.

The existing test `test_reward_validator_different_online` explicitly observes this: with `epoch_total_reward = 10_000_000` and two eligible validators, `epoch_actual_reward` is `4_999_999` — one yoctoNEAR short of the correct `5_000_000`: [6](#0-5) 

### Impact Explanation

Every epoch, for every eligible validator, up to `(uptime_denum + total_stake_yoctonear - 2)` yoctoNEAR is silently destroyed. With mainnet-scale values (`total_stake` in the hundreds of millions of NEAR, `uptime_denum` a product of small integers), the per-validator loss per epoch is bounded but the aggregate across all validators and all epochs is unbounded and grows monotonically. The corrupted value is `minted_amount` in `EpochInfo` and `total_supply` in every epoch-boundary block header — both consensus-critical fields. The `total_supply` divergence is accepted by all nodes because they all run the same truncating code, so there is no fork, but the protocol's economic invariant (total tokens = genesis supply + minted - burned) is permanently violated.

### Likelihood Explanation

This triggers on every epoch finalization whenever at least one validator is eligible for a reward and the division does not divide evenly — which is the normal case for any realistic stake distribution. An unprivileged user cannot prevent or amplify this; it fires automatically as part of normal protocol operation. The only entry path is the standard block production and epoch finalization flow, reachable by any participant submitting a staking transaction.

### Recommendation

Combine the two divisions into a single operation to defer truncation:

```rust
Balance::from_yoctonear(
    (U512::from(epoch_validator_reward.as_yoctonear())
        * U512::from(uptime_numer)
        * U512::from(stake.as_yoctonear())
        / (U512::from(uptime_denum)
            * U512::from(total_stake.as_yoctonear())))
    .as_u128(),
)
```

This matches the spec formula `floor(uptime * stake * total_reward / total_stake)` exactly and eliminates the intermediate truncation.

### Proof of Concept

Using the values from `test_reward_validator_different_online`:
- `epoch_total_reward = 10_000_000`, `protocol_reward = 0`, `epoch_validator_reward = 10_000_000`
- Two validators, each with `stake = 500_000`, `total_stake = 1_000_000`
- test2: fully online, `uptime_numer = uptime_denum` (capped at 1)

```
Step 1: floor(10_000_000 * uptime_denum * 500_000 / uptime_denum) = floor(5_000_000_000_000) = 5_000_000_000_000
Step 2: floor(5_000_000_000_000 / 1_000_000) = 5_000_000  ✓ (no loss here)
```

- test1: 94.5% online, `uptime_numer/uptime_denum = 5/10` (50%)

```
Step 1: floor(10_000_000 * 5 * 500_000 / 10) = floor(25_000_000_000_000 / 10) = 2_500_000_000_000
Step 2: floor(2_500_000_000_000 / 1_000_000) = 2_500_000  ✓
```

But with `uptime_numer = 1_666_666` (non-round uptime):

```
Step 1: floor(10_000_000 * 1_666_666 * 500_000 / uptime_denum) → truncates remainder r₁
Step 2: floor(result / 1_000_000) → truncates remainder r₂
```

The test asserts `result.1 == 4_999_999` (not `5_000_000`), confirming one yoctoNEAR is lost per epoch even in this minimal example. [6](#0-5)
