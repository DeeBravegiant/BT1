### Title
Missing Completed-Request Registry in `verify_foreign_transaction` Enables Cross-Chain Replay Signing — (`crates/contract/src/lib.rs`)

---

### Summary

The `verify_foreign_transaction` / `respond_verify_foreign_tx` flow in the MPC contract has no persistent "already-processed" registry. After a foreign-transaction verification request is resolved and its pending entry is removed, the identical request can be re-submitted, causing MPC nodes to produce a second (and third, etc.) valid threshold signature over the same foreign-chain payload. This is the direct structural analog of the TrufVesting `claimable()` bug: just as that function returned `initialRelease` without subtracting `userVesting.claimed`, the MPC contract resolves a foreign-tx request without recording it in any durable "completed" set, so the same value can be extracted repeatedly.

---

### Finding Description

**Root cause — no durable completed-request set**

When a user calls `verify_foreign_transaction`, a `VerifyForeignTransactionRequest` key is inserted into `pending_verify_foreign_tx_requests` together with a `YieldIndex`: [1](#0-0) 

When an attested MPC node calls `respond_verify_foreign_tx`, `resolve_yields_for` is invoked, which **removes** the entry from the map and resumes all queued yields: [2](#0-1) 

After `remove` returns, the map contains no trace of the request. There is no secondary "completed foreign-tx" set, no nonce registry, and no blacklist. The contract state after a successful response is identical to the state before the first submission.

A user can therefore immediately re-call `verify_foreign_transaction` with the same arguments. The contract inserts a fresh `YieldIndex` for the same key: [3](#0-2) 

MPC nodes observe the new pending entry, re-verify the foreign-chain transaction (which is still present on the foreign chain), and produce a second threshold signature. The user receives a second valid signature over the same payload.

**Structural parallel to TrufVesting**

| TrufVesting | NEAR MPC |
|---|---|
| `claimable()` returns `initialRelease` without subtracting `userVesting.claimed` | `verify_foreign_transaction` inserts a new pending entry without checking a "completed" set |
| `claim()` updates `userVesting.claimed` but `claimable()` ignores it on the next call | `respond_verify_foreign_tx` removes the pending entry but leaves no durable record |
| Any user can call `claim` in a loop to drain the contract | Any user can re-submit the same foreign tx in a loop to obtain unlimited valid signatures |

**Additional observation — signature verified against root key, not derived key**

In `respond` (standard signing), the signature is verified against the **derived** public key: [4](#0-3) 

In `respond_verify_foreign_tx`, the signature is verified against the **root** public key with no key derivation applied: [5](#0-4) 

This asymmetry means the on-chain verification check in the foreign-tx path is weaker than in the standard signing path, compounding the replay risk.

---

### Impact Explanation

**Critical / High — Cross-chain replay enabling double-spend conditions.**

A user who has legitimately verified one foreign-chain deposit can re-submit the same `VerifyForeignTransactionRequest` an arbitrary number of times. Each round trip through the MPC network produces a fresh, cryptographically valid threshold signature over the same payload. If the NEAR-side application (or any downstream contract) uses these signatures to release NEAR-side funds or authorize bridge withdrawals, the attacker can drain those funds by replaying the same foreign-chain event repeatedly — matching the "drain the contract" impact of the TrufVesting bug.

---

### Likelihood Explanation

The entry path is fully unprivileged: `verify_foreign_transaction` is a public payable method requiring only a 1-yoctoNEAR deposit. No threshold collusion, TEE compromise, or operator access is needed. The foreign-chain transaction remains on-chain indefinitely, so MPC nodes will re-verify it successfully on every re-submission. The attack is repeatable until the NEAR-side application's funds are exhausted. [6](#0-5) 

---

### Recommendation

Introduce a persistent `LookupSet<VerifyForeignTransactionRequest>` (or a `LookupSet` keyed on a canonical tx identifier) that records every request that has been successfully resolved. In `respond_verify_foreign_tx`, before calling `resolve_yields_for`, check membership and return an error if the request is already present; after resolving, insert it into the set. This mirrors the fix recommended for TrufVesting: track what has already been processed and subtract (or reject) accordingly.

```rust
// pseudocode
if self.completed_foreign_tx_requests.contains(&request) {
    return Err(RequestError::AlreadyProcessed.into());
}
// ... existing verification ...
pending_requests::resolve_yields_for(...)?;
self.completed_foreign_tx_requests.insert(request);
```

---

### Proof of Concept

```
1. Attacker bridges 100 USDC from Ethereum → NEAR, producing eth_tx_id = 0xABC.
2. Attacker calls verify_foreign_transaction({ tx_id: 0xABC, chain: Ethereum, ... }).
   → Contract inserts pending entry; MPC nodes verify tx on Ethereum and call
     respond_verify_foreign_tx → entry removed, signature S1 delivered.
3. Attacker calls verify_foreign_transaction({ tx_id: 0xABC, chain: Ethereum, ... }) again.
   → Contract inserts a fresh pending entry (no completed-set check).
   → MPC nodes re-verify (tx still on Ethereum) and call respond_verify_foreign_tx again
     → signature S2 delivered.
4. Attacker presents S1 and S2 to the NEAR bridge contract, claiming the 100 USDC deposit twice.
5. Steps 3–4 repeat until the bridge's NEAR-side liquidity is drained.
```

pub(crate) fn push_pending_yield<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: K,
    data_id: CryptoHash,
) where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let queue = requests.entry(request).or_default();
    if queue.len() >= usize::from(MAX_PENDING_REQUEST_FAN_OUT) {
        env::panic_str(
            &RequestError::PendingRequestQueueFull {
                limit: MAX_PENDING_REQUEST_FAN_OUT,
            }
            .to_string(),
        );
    }
    queue.push(YieldIndex { data_id });
}
