### Title
Malicious Harvester Redirects Farmer Block Rewards via Unvalidated `farmer_reward_address_override` in `RespondSignatures` — (File: `chia/farmer/farmer_api.py`)

### Summary

The `_process_respond_signatures` function in `chia/farmer/farmer_api.py` unconditionally accepts `farmer_reward_address_override` from a harvester's `RespondSignatures` message and uses it as the farmer reward address in `DeclareProofOfSpace`, without verifying that the override was legitimately declared in the corresponding `NewProofOfSpace` or that the CHIP-22 fee quality threshold was met. A malicious harvester connected to a victim farmer can redirect the farmer's block reward (1/8 of the block reward in XCH) to an arbitrary attacker-controlled address on every block won.

### Finding Description

**Root cause — two independent fields, one missing cross-check:**

`RespondSignatures` carries its own `farmer_reward_address_override` field (harvester_protocol.py line 139), independent of the same-named field in `NewProofOfSpace` (harvester_protocol.py line 75). The farmer treats these as separate, unlinked values.

**Step 1 — Fee check is only advisory and only covers `NewProofOfSpace`:**

In `new_proof_of_space` (farmer_api.py lines 128–129), when a harvester sends `NewProofOfSpace` with `farmer_reward_address_override` set, the farmer calls `notify_farmer_reward_taken_by_harvester_as_fee`. That function (farmer.py lines 888–934) computes the CHIP-22 fee quality and logs a warning if the threshold is invalid — but it **never returns a value that gates further processing** and **never blocks the proof from proceeding**. [1](#0-0) [2](#0-1) 

**Step 2 — `NewProofOfSpace.farmer_reward_address_override` is never stored:**

After the (non-enforcing) fee check, the farmer stores only `plot_identifier` and `proof` in `proofs_of_space` (farmer_api.py lines 170–177). The `farmer_reward_address_override` from `NewProofOfSpace` is discarded. The farmer therefore has no record of what override (if any) was declared at proof-submission time. [3](#0-2) 

**Step 3 — `RespondSignatures.farmer_reward_address_override` is accepted unconditionally:**

In `_process_respond_signatures` (farmer_api.py lines 916–919), the farmer blindly substitutes the harvester-supplied address for its own `farmer_target` with no fee-quality check, no comparison against the `NewProofOfSpace` value, and no other guard:

```python
farmer_reward_address = self.farmer.farmer_target
if response.farmer_reward_address_override is not None:
    farmer_reward_address = response.farmer_reward_address_override
    include_source_signature_data = True
``` [4](#0-3) 

This address is then placed directly into `DeclareProofOfSpace` (farmer_api.py lines 921–933) and broadcast to the full node, which pays the farmer reward to it. [5](#0-4) 

**Complete exploit path (no privileged access required):**

1. Attacker runs a harvester connected to the victim farmer (normal, unprivileged role).
2. Harvester finds a valid proof and sends `NewProofOfSpace` with `farmer_reward_address_override = None` and `fee_info = None` — the fee check in `notify_farmer_reward_taken_by_harvester_as_fee` is never triggered.
3. Farmer sends `RequestSignatures` to the harvester.
4. Harvester responds with `RespondSignatures` containing `farmer_reward_address_override = <attacker_puzzle_hash>`.
5. `_process_respond_signatures` sets `farmer_reward_address = <attacker_puzzle_hash>` with no validation.
6. Farmer broadcasts `DeclareProofOfSpace` with the attacker's address as the farmer reward destination.
7. The full node pays the farmer's 1/8 block reward to the attacker's address.

The attacker can also send `NewProofOfSpace` with `farmer_reward_address_override` set to trigger the fee-check path, but since that check is only advisory (log-only), the outcome is identical — the override in `RespondSignatures` is still accepted unconditionally. [6](#0-5) [7](#0-6) 

### Impact Explanation

**High — unauthorized payout redirection of XCH block rewards.**

Every block the victim farmer wins while the malicious harvester is connected results in the farmer's 1/8 block reward being paid to the attacker's address instead of the farmer's configured `xch_target_address`. This is a direct, permanent loss of XCH with no recovery path once the block is confirmed. This falls squarely within the allowed High impact category: "Bypass of … pool … authorization that enables … payout redirection."

### Likelihood Explanation

**High.** The attacker only needs to be a connected harvester — a normal, unprivileged role in the Chia farming architecture. Third-party harvesters (e.g., DrPlotter, GPU harvesters) are explicitly supported and commonly used. The exploit requires no keys, no admin access, and no cryptographic break. It fires on every block won, silently, with only a log warning (if `NewProofOfSpace.farmer_reward_address_override` was also set) or no warning at all (if only `RespondSignatures.farmer_reward_address_override` is set).

### Recommendation

1. **Store the override at proof time.** In `new_proof_of_space`, persist `farmer_reward_address_override` from `NewProofOfSpace` alongside `plot_identifier` and `proof` in `self.farmer.proofs_of_space`.

2. **Enforce consistency in `_process_respond_signatures`.** Before applying `response.farmer_reward_address_override`, verify it exactly matches the value stored from the corresponding `NewProofOfSpace`. If they differ, reject the response.

3. **Enforce the fee quality check, not just log it.** `notify_farmer_reward_taken_by_harvester_as_fee` should return a boolean indicating whether the threshold was legitimately met. If it returns `False`, the farmer should not apply the override and should drop the proof.
