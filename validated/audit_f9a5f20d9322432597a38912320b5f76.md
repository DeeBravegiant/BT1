Audit Report

## Title
Reward Tokens Permanently Locked When `totalKernelStaked` Drops to Zero During Active Reward Period - (File: `contracts/KERNEL/KernelDepositPool.sol`)

## Summary
`KernelDepositPool` implements a Synthetix-style staking rewards mechanism where `rewardPerToken()` returns `rewardPerTokenStored` unchanged when `totalKernelStaked == 0`, silently halting reward accumulation. Meanwhile, the `updateReward` modifier continues advancing `updatedAt`, permanently skipping the zero-staking window. Because `notifyRewardAmount` uses a transfer-in delta (`balanceAfter - balanceBefore`) to compute `receivedAmount`, previously stranded reward tokens are never rolled forward, and no admin recovery function exists, leaving those tokens permanently locked.

## Finding Description
**Root cause:** `rewardPerToken()` (L408-414) guards against division by zero by returning `rewardPerTokenStored` when `totalKernelStaked == 0`. This is correct for preventing a panic, but it also freezes reward accumulation. Simultaneously, the `updateReward` modifier (L232-234) unconditionally sets `updatedAt = lastTimeRewardApplicable()` on every user action. When staking resumes, the reward calculation begins from the new `updatedAt`, permanently skipping the zero-staking window.

**Exploit path:**
1. Admin calls `notifyRewardAmount(X)` — `rewardRate = X / duration`, `finishAt = T0 + duration`.
2. A staker calls `initiateWithdrawal(totalBalance)` at time `T1` — `totalKernelStaked` drops to 0, `updatedAt` is set to `T1`.
3. From `T1` to `T2` (end of period), every call to `rewardPerToken()` returns the frozen `rewardPerTokenStored`. Reward tokens worth `rewardRate × (T2 − T1)` accumulate in the contract but are never attributed to anyone.
4. After the period ends, a user stakes — `updateReward` fires, `updatedAt = T2`.
5. Admin calls `notifyRewardAmount(Y)` — `receivedAmount = balanceAfter - balanceBefore = Y` only; the already-present stranded tokens are excluded from the new `rewardRate` calculation (L572-577).
6. The `rewardRate × (T2 − T1)` tokens remain in the contract with no distribution or recovery path.

**Why existing checks fail:** `notifyRewardAmount` (L570) reverts if `totalKernelStaked == 0` at the time of the call, but this does not prevent `totalKernelStaked` from reaching zero *during* an active period. `initiateWithdrawal` (L320-338) is permissionless for any staker and has no restriction tied to active reward periods. The contract's own NatSpec (L17-22) acknowledges this limitation and relies on an off-chain operational assumption ("ensuring there are always some tokens staked"), which is not enforced on-chain. There is no `recoverTokens`, `sweepRewards`, or equivalent admin function in the contract.

## Impact Explanation
**Medium — Permanent freezing of unclaimed yield.** Reward tokens equal to `rewardRate × (zero-staking duration)` are real ERC-20 tokens transferred into the contract by the admin. Once stranded, they cannot be distributed to stakers (no one was staked during that window) and cannot be recovered by the admin (no recovery function exists). This matches the allowed impact: *Medium. Permanent freezing of unclaimed yield.*

## Likelihood Explanation
The scenario requires `totalKernelStaked` to reach zero during an active reward period. This is achievable by a single staker withdrawing their full balance — a normal, permissionless action available to any user at any time. No malicious intent is required; ordinary unstaking behavior (e.g., in response to market conditions) is sufficient. The entry point `initiateWithdrawal` imposes no restriction on timing relative to active reward periods.

## Recommendation
Modify `notifyRewardAmount` to account for the contract's existing excess reward token balance when computing the new `rewardRate`. Specifically, compute `excessRewards = rewardsToken.balanceOf(address(this)) - (totalOwedRewards)` before the transfer and include it in `receivedAmount`, so previously unallocated rewards are automatically rolled forward into the new period. Alternatively, add an admin `recoverExcessRewards(address treasury)` function that can sweep unallocated reward tokens when `block.timestamp >= finishAt` and `totalKernelStaked == 0`, preventing permanent lock-up.

## Proof of Concept
```solidity
// Foundry test sketch
function test_rewardsLockedOnZeroStake() public {
    // 1. Admin starts reward period
    vm.prank(admin);
    rewardsToken.approve(address(pool), 1000e18);
    vm.prank(admin);
    pool.notifyRewardAmount(1000e18);

    // 2. Single staker withdraws entire balance mid-period
    vm.warp(block.timestamp + duration / 2);
    vm.prank(staker);
    pool.initiateWithdrawal(pool.balanceOf(staker));
    assertEq(pool.totalKernelStaked(), 0);

    // 3. Advance to end of period — rewards for this window are frozen
    vm.warp(pool.finishAt());

    // 4. New staker enters, admin starts new period with fresh tokens only
    vm.prank(newStaker);
    pool.stake(1e18);
    uint256 contractBalanceBefore = rewardsToken.balanceOf(address(pool));
    vm.prank(admin);
    rewardsToken.approve(address(pool), 100e18);
    vm.prank(admin);
    pool.notifyRewardAmount(100e18);
    uint256 contractBalanceAfter = rewardsToken.balanceOf(address(pool));

    // 5. Stranded tokens remain: balance grew by only 100e18, but contract holds more
    // rewardRate only reflects 100e18, not the stranded ~500e18 from the zero-staking window
    assertGt(contractBalanceBefore, 0); // stranded tokens still present
    // No admin function can recover them
}
```
