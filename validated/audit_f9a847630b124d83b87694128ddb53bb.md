### Title
Cross-Request Signature Replay in `respond_verify_foreign_tx()` Enables Forged Foreign-Chain Verification - (File: `crates/contract/src/lib.rs`)

---

### Summary

The `respond_verify_foreign_tx()` function in the MPC contract verifies that a submitted ECDSA signature is valid for a given `payload_hash`, but performs **no binding check** between `payload_hash` and the pending `VerifyForeignTransactionRequest`. A single Byzantine participant (below the signing threshold) can replay a previously observed, legitimately-produced `VerifyForeignTransactionResponse` against a different pending request, causing callers to receive a forged foreign-chain verification attestation.

---

### Finding Description

**Root cause — missing payload-to-request binding in `respond_verify_foreign_tx()`**

In `crates/contract/src/lib.rs` lines 718–753, the contract performs two independent checks:

1. **Cryptographic check**: Is `response.signature` a valid ECDSA signature over `response.payload_hash` under the domain's root public key?
2. **State check**: Does `request` exist in `pending_verify_foreign_tx_requests`? [1](#0-0) 

These two checks are entirely decoupled. The contract never verifies that `response.payload_hash` is the SHA-256 hash of `borsh(ForeignTxSignPayload{ request, extracted_values })` for the specific `request` being resolved. The `payload_hash` is accepted at face value from the caller.

**The signed payload structure**

The hash that nodes sign is:

```
msg_hash = SHA-256(borsh(ForeignTxSignPayload))
```

where `ForeignTxSignPayload::V1` contains `{ request: ForeignChainRpcRequest, values: Vec<ExtractedValue> }`. [2](#0-1) 

This hash is computed by nodes off-chain and submitted as `response.payload_hash`. The contract has no way to independently recompute it (it does not know `extracted_values`), so it trusts the submitter.

**The zero-tweak signing path**

For `verify_foreign_transaction`, nodes build the internal `SignatureRequest` with `tweak: Tweak::new([0u8; 32])`, meaning the MPC network signs with the **root key** (no derivation). [3](#0-2) 

The contract's `respond_verify_foreign_tx()` correspondingly verifies against the root public key (`secp_pk`, no tweak applied). [4](#0-3) 

Because the root key is shared across all `ForeignTx` domain requests, any valid `(payload_hash, signature)` pair produced for **any** prior foreign-tx request is cryptographically valid under the same root key and will pass the signature check for **any other** pending request.

**The caller-agnostic pending map**

`VerifyForeignTransactionRequest` (the map key) contains no caller identity:

```rust
pub struct VerifyForeignTransactionRequest {
    pub request: ForeignChainRpcRequest,
    pub domain_id: DomainId,
    pub payload_version: ForeignTxPayloadVersion,
}
``` [5](#0-4) 

Any participant can call `respond_verify_foreign_tx(request_B, old_response_A)` and the contract will resolve all yields queued under `request_B` with the forged response.

---

### Impact Explanation

A single Byzantine participant (one attested node, strictly below the signing threshold) can:

1. Observe a legitimately produced `VerifyForeignTransactionResponse` for `request_A` on-chain — the `(payload_hash_A, sig_A)` pair is public.
2. Submit a new `verify_foreign_transaction(request_B)` for a different foreign transaction (costs 1 yoctonear).
3. Call `respond_verify_foreign_tx(request_B, { payload_hash_A, sig_A })`.

The contract accepts this because:
- `sig_A` is a valid ECDSA signature over `payload_hash_A` under the root key → cryptographic check passes.
- `request_B` exists in the pending map → state check passes.
- No binding between `payload_hash_A` and `request_B` is enforced.

All callers who submitted `request_B` receive `{ payload_hash_A, sig_A }` as the "verified" result of their foreign-chain query. A bridge contract that uses this response to release funds — without independently recomputing and comparing `payload_hash` — would release funds based on a forged attestation of a completely different transaction.

This matches the allowed impact: **"High. Cross-chain replay, forged foreign-chain verification … that causes invalid bridge execution or double-spend conditions."**

---

### Likelihood Explanation

- **Attacker capability**: A single Byzantine participant (one compromised or malicious MPC node). No threshold collusion required.
- **Precondition**: At least one prior legitimate `respond_verify_foreign_tx` call must exist on-chain (trivially satisfied in any live deployment).
- **Barrier**: The attacker must be an attested participant. TEE attestation is intended to prevent this, but the threat model explicitly acknowledges rollback/replay as an **open risk** (T6). [6](#0-5) 

The contract-level protection is absent regardless of TEE guarantees; the vulnerability exists in the on-chain logic itself.

---

### Recommendation

Bind `response.payload_hash` to the specific pending `request` inside `respond_verify_foreign_tx()`. Since the contract cannot recompute the full `ForeignTxSignPayload` hash (it lacks `extracted_values`), the fix should be applied at the payload construction level:

1. **Include a unique per-request identifier** (e.g., the NEAR receipt ID, already present as `request.receipt_id` in the node's `VerifyForeignTxRequest`) inside `ForeignTxSignPayload`. The contract can then verify that `response.payload_hash` encodes the correct receipt ID for the pending request.
2. **Alternatively**, store the expected `payload_hash` prefix (derived from the `ForeignChainRpcRequest` alone, without `extracted_values`) on-chain at request submission time, and verify that `response.payload_hash` is consistent with it.
3. **Track used `payload_hash` values** per domain to prevent the same `(payload_hash, sig)` pair from being submitted more than once, analogous to nonce/order-ID tracking in the reference report's fix.

### Proof of Concept

**Setup**: MPC contract is running with a `ForeignTx` domain. Bitcoin is a supported chain.

**Step 1 — Observe a legitimate response on-chain.**

Alice submits `verify_foreign_transaction(bitcoin_tx_A)`. The MPC network verifies it and a participant calls:
```
respond_verify_foreign_tx(
    request = { request: BitcoinRpcRequest{ tx_id: tx_A, ... }, domain_id: 0, ... },
    response = { payload_hash: hash_A, signature: sig_A }
)
```
`hash_A` and `sig_A` are now public on-chain.

**Step 2 — Attacker submits a new request.**

The Byzantine participant (or any account) submits:
```
verify_foreign_transaction(bitcoin_tx_B)   // different tx_id
```
This creates a pending entry for `request_B` in `pending_verify_foreign_tx_requests`.

**Step 3 — Replay the old response.**

The Byzantine participant calls:
```
respond_verify_foreign_tx(
    request = { request: BitcoinRpcRequest{ tx_id: tx_B, ... }, domain_id: 0, ... },
    response = { payload_hash: hash_A, signature: sig_A }   // ← replayed from Step 1
)
```

**Contract execution** (`crates/contract/src/lib.rs` lines 718–753):
- `verify_ecdsa_signature(sig_A, hash_A, root_pk)` → **PASSES** (valid signature).
- `resolve_yields_for(pending_verify_foreign_tx_requests, request_B, response)` → **PASSES** (request_B is pending).

**Result**: The caller of `verify_foreign_transaction(bitcoin_tx_B)` receives `{ payload_hash: hash_A, signature: sig_A }` — a forged attestation claiming the MPC network verified `bitcoin_tx_B`, when in fact `hash_A` encodes the data of `bitcoin_tx_A`. A bridge contract that gates fund release on this response without recomputing and comparing `payload_hash` locally would execute an invalid cross-chain transfer.
