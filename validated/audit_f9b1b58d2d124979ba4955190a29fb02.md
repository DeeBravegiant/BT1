### Title
Delegate-as-Sender Mismatch in `init_transfer` Encodes Wrong Sender in Wormhole Payload — (`solana/programs/bridge_token_factory/src/instructions/user/init_transfer.rs`)

### Summary

The `InitTransfer::process` function encodes `user.key()` as the sender in the cross-chain Wormhole payload, but the `from` token account has no constraint binding it to `user` as its owner. Because SPL token's `transfer_checked` accepts a delegate as a valid authority, a delegate (`user = Bob`) can drain a third-party token account (`from` owned by Alice) while the NEAR payload records Bob — not Alice — as the economic sender. Any refund or callback flow on NEAR that uses the sender field will credit Bob, not Alice.

### Finding Description

In `InitTransfer`, the `from` account is constrained only by mint and token program: [1](#0-0) 

There is no `token::authority = user` constraint. The `user` field is only required to be a signer: [2](#0-1) 

The `transfer_checked` CPI passes `user` as the authority: [3](#0-2) 

SPL Token accepts a delegate as a valid authority for `transfer_checked`, so this CPI succeeds even when `user != from.owner`. Immediately after, the Wormhole payload is serialized with `user.key()` as `params.1` (the sender field): [4](#0-3) 

In `serialize_for_near`, `params.1` is written directly as the sender identity in the cross-chain message: [5](#0-4) 

### Impact Explanation

Alice's tokens are locked/burned on Solana while NEAR records Bob as the sender. Any NEAR-side refund or callback that uses the sender field to route value back will send Alice's funds to Bob. This is an asset-identity and refund-routing divergence that breaks the backing guarantee: the economic owner of the transferred tokens (Alice) is permanently displaced from the sender identity used for all downstream settlement.

### Likelihood Explanation

SPL token delegation is a standard, widely-used primitive (DEX approvals, protocol integrations). Any user who has delegated their token account to another party — even for an unrelated purpose — is exposed. Bob only needs to be a signer and hold a valid delegation; no privileged access or key compromise is required.

### Recommendation

Add an explicit owner binding on `from` so that only the true owner can initiate a transfer in their name:

```rust
#[account(
    mut,
    token::mint = mint,
    token::token_program = token_program,
    token::authority = user,   // <-- add this
)]
pub from: Box<InterfaceAccount<'info, TokenAccount>>,
```

This prevents delegates from calling `init_transfer` on behalf of another account's owner, ensuring `user.key()` always equals `from.owner` when the payload is serialized.

### Proof of Concept

```rust
// 1. Alice creates a token account and delegates full amount to Bob
spl_token::instruction::approve(&alice_token_account, &bob_pubkey, &alice_pubkey, amount);

// 2. Bob calls init_transfer with alice's token account as `from`, himself as `user`
program.init_transfer(
    from: alice_token_account,   // owned by Alice
    user: bob_keypair,           // signer = Bob (delegate)
    payload: InitTransferPayload { amount, recipient: bob_near_address, fee, native_fee, message },
    ...
);

// 3. transfer_checked succeeds (Bob is delegate)
// 4. Wormhole payload encodes sender = Bob.pubkey, not Alice.pubkey
// 5. Assert: decoded payload.sender == bob_pubkey  (not alice_pubkey)
// 6. On NEAR: refund for failed transfer goes to Bob, Alice loses tokens
```
