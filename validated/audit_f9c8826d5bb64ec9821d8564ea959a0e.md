### Title
Stale `drepDelegs` Reverse-Delegation Index in `DRepState` Causes Silent Delegation Erasure on DRep Unregistration During Bootstrap Phase — (File: `eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Deleg.hs`)

---

### Summary

During the Conway bootstrap phase (protocol version 9), when a stake credential re-delegates its vote from DRep A to DRep B, the `drepDelegs` reverse-delegation set inside DRep A's `DRepState` is **not updated** to remove the stake credential. This stale entry causes the stake credential's active delegation to DRep B to be **silently cleared** when DRep A subsequently unregisters, because the `ConwayUnRegDRep` handler in `GovCert.hs` uses `drepDelegs` as the authoritative source for clearing delegations. The result is that an attacker-controlled DRep operator can erase a victim's vote delegation to a third-party DRep without any action from the victim, distorting governance ratification.

---

### Finding Description

**Two data structures track DRep delegation in parallel:**

1. **Canonical (primary):** `dRepDelegationAccountStateL` inside each `AccountState` in the `Accounts` map — updated correctly on every delegation change.
2. **Reverse index (secondary):** `drepDelegs :: Set (Credential Staking)` inside `DRepState` in `vsDReps` — a per-DRep set of all stake credentials currently delegated to it. [1](#0-0) 

The `processDelegationInternal` function in `Deleg.hs` accepts a `preserveIncorrectDelegation :: Bool` flag. When `True` (i.e., `pvMajor pv < natVersion @10`, the bootstrap phase), the `delegVote` branch **only inserts** the stake credential into the new DRep's `drepDelegs` and **never removes** it from the old DRep's `drepDelegs`: [2](#0-1) 

The critical conditional: [3](#0-2) 

When `isNothing mAccountState || preserveIncorrectDelegation` is `True` (bootstrap phase with an existing account), the code path at line 363–365 only calls `Set.insert stakeCred` on the **new** DRep's `drepDelegs`. The `unDelegReDelegDRep` path (lines 369–370), which correctly removes from the old DRep, is **skipped**.

This flag is set to `True` for `ConwayDelegCert` and `ConwayRegDelegCert` whenever `pvMajor pv < natVersion @10`: [4](#0-3) 

**The stale entry is then weaponized during DRep unregistration.** In `GovCert.hs`, `ConwayUnRegDRep` uses `drepDelegs` as the source of truth to clear delegations: [5](#0-4) 

`clearDRepDelegations` iterates over `drepDelegs` and sets `dRepDelegationAccountStateL .~ Nothing` for every credential in the set. If DRep A's `drepDelegs` still contains stake credential S (which has already re-delegated to DRep B), unregistering DRep A will clear S's delegation to DRep B — even though DRep A is no longer S's delegatee.

The correct cleanup function `unDelegReDelegDRep` in `VState.hs` does handle this properly (it reads the canonical `dRepDelegationAccountStateL` before deciding what to remove), but it is **not called** in the `ConwayUnRegDRep` path: [6](#0-5) 

The bootstrap phase is defined as: [7](#0-6) 

The planned fix (`updateDRepDelegations`) is applied only at the PV10 hard fork transition: [8](#0-7) 

Until that transition fires, the stale state is live and exploitable.

---

### Impact Explanation

An attacker who controls DRep A can execute the following sequence during bootstrap (PV9):

1. Register DRep A (permissionless, deposit recoverable on unregistration).
2. Attract stake credential S to delegate to DRep A.
3. Wait for S to re-delegate to DRep B (a legitimate DRep the attacker wants to weaken).
4. Submit `ConwayUnRegDRep` for DRep A.
5. `clearDRepDelegations` reads the stale `drepDelegs` of DRep A, finds S, and sets `dRepDelegationAccountStateL` for S to `Nothing`.
6. S now has **no DRep delegation** even though it re-delegated to DRep B. DRep B loses S's stake weight.

The `computeDRepDistr` function (used for ratification) reads `dRepDelegationAccountStateL` from `AccountState`: [9](#0-8) 

After the attack, S's stake is excluded from DRep B's distribution. If DRep B was the deciding vote for a governance action (treasury withdrawal, protocol parameter change, committee update, hard-fork initiation), that action may fail to reach its ratification threshold. Conversely, if the attacker's own DRep holds the opposing stake, the attacker can tip ratification outcomes.

This matches the **Medium** allowed impact: *"Attacker-controlled transactions… modify… withdrawals outside design parameters"* — specifically, governance voting-power accounting is manipulated outside design parameters. It also borders on **Critical** if the distortion causes an unauthorized governance action to be enacted or a legitimate one to be blocked.

---

### Likelihood Explanation

- **Entry path is fully permissionless.** Any party can register a DRep, attract delegators, and unregister. No privileged role, key leak, or supermajority is required.
- **Deposit is fully recoverable.** The attacker pays a DRep deposit on registration and recovers it on unregistration, making the attack nearly free.
- **Victim action is normal user behavior.** Re-delegating from one DRep to another is a standard governance participation action.
- **Active during bootstrap.** The bootstrap phase (PV9) is the initial Conway deployment period, during which governance participation is highest and DRep competition is most active — maximizing the opportunity for this attack.
- **Confirmed by existing test.** The test at lines 307–323 of `DelegSpec.hs` explicitly asserts the buggy behavior is preserved during bootstrap: [10](#0-9) 

---

### Recommendation

The `ConwayUnRegDRep` handler in `GovCert.hs` should not rely on `drepDelegs` as the authoritative source for clearing delegations, because `drepDelegs` can be stale during bootstrap. Instead, it should either:

1. **Iterate over all `AccountState` entries** and clear `dRepDelegationAccountStateL` only where it equals the unregistering DRep credential — this is always correct regardless of `drepDelegs` consistency.
2. **Apply `unDelegReDelegDRep`** (which reads the canonical `dRepDelegationAccountStateL` before acting) instead of `clearDRepDelegations`.

The `updateDRepDelegations` hard-fork migration at PV10 resets and repopulates `drepDelegs` from the canonical `AccountState`, which is the correct approach but only runs once at the era boundary.

---

### Proof of Concept

```
Bootstrap phase (pvMajor pv == 9):

1. Attacker registers DRep_A.
   vsDReps[DRep_A].drepDelegs = {}

2. Victim registers stake cred S and delegates to DRep_A.
   AccountState[S].dRepDelegationAccountStateL = Just DRep_A
   vsDReps[DRep_A].drepDelegs = {S}

3. Victim re-delegates S to DRep_B (processDelegationInternal, preserveIncorrectDelegation=True).
   AccountState[S].dRepDelegationAccountStateL = Just DRep_B   ← canonical updated
   vsDReps[DRep_A].drepDelegs = {S}                            ← STALE, not cleaned
   vsDReps[DRep_B].drepDelegs = {S}                            ← new entry added

4. Attacker submits ConwayUnRegDRep for DRep_A.
   clearDRepDelegations reads vsDReps[DRep_A].drepDelegs = {S}
   Sets AccountState[S].dRepDelegationAccountStateL = Nothing   ← WRONG: clears DRep_B delegation

5. Result:
   AccountState[S].dRepDelegationAccountStateL = Nothing        ← victim has no delegation
   vsDReps[DRep_B].drepDelegs = {S}                            ← stale forward entry remains
   computeDRepDistr sees S has no delegation → DRep_B loses S's stake weight
```

unDelegReDelegDRep ::
  ConwayEraAccounts era =>
  Credential Staking ->
  -- | Account that is losing its current delegation and/or acquiring a new one
  AccountState era ->
  -- | Potential new delegation. In case when stake credential unregisters this must be `Nothing`.
  Maybe DRep ->
  VState era ->
  VState era
unDelegReDelegDRep stakeCred accountState mNewDRep =
  fromMaybe (vsDRepsL %~ addNewDelegation) $ do
    dRep@(DRepCredential dRepCred) <- accountState ^. dRepDelegationAccountStateL
    pure $
      -- There is no need to update set of delegations if delegation is unchanged
      if Just dRep == mNewDRep
        then id
        else
          vsDRepsL %~ addNewDelegation . Map.adjust (drepDelegsL %~ Set.delete stakeCred) dRepCred
  where
    addNewDelegation =
      case mNewDRep of
        Just (DRepCredential dRepCred) ->
          Map.adjust (drepDelegsL %~ Set.insert stakeCred) dRepCred
        _ -> id
```

updateDRepDelegations :: ConwayEraCertState era => CertState era -> CertState era
updateDRepDelegations certState =
  let accountsMap = certState ^. certDStateL . accountsL . accountsMapL
      dReps =
        -- Reset all delegations in order to remove any inconsistencies
        -- Delegations will be reset accordingly below.
        Map.map (\dRepState -> dRepState {drepDelegs = Set.empty}) $
          certState ^. certVStateL . vsDRepsL
      (dRepsWithDelegations, accountsWithoutUnknownDRepDelegations) =
        Map.mapAccumWithKey adjustDelegations dReps accountsMap
      adjustDelegations ds stakeCred accountState =
        case accountState ^. dRepDelegationAccountStateL of
          Just (DRepCredential dRep) ->
            let addDelegation _ dRepState =
                  Just $ dRepState {drepDelegs = Set.insert stakeCred (drepDelegs dRepState)}
             in case Map.updateLookupWithKey addDelegation dRep ds of
                  (Nothing, _) -> (ds, accountState & dRepDelegationAccountStateL .~ Nothing)
                  (Just _, ds') -> (ds', accountState)
          _ -> (ds, accountState)
   in certState
        -- Remove dangling delegations to non-existent DReps:
        & certDStateL . accountsL . accountsMapL .~ accountsWithoutUnknownDRepDelegations
        -- Populate DRep delegations with delegatees
        & certVStateL . vsDRepsL .~ dRepsWithDelegations
```
