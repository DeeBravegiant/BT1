Audit Report

## Title
Stale Manually Updated rsETH/ETH Rate in `InterimRSETHOracle` Allows Depositors to Extract Excess rsETH from L2 Pools - (File: contracts/pools/oracle/InterimRSETHOracle.sol)

## Summary
`InterimRSETHOracle` stores a manually set rsETH/ETH exchange rate with no staleness tracking, no timestamp, and no monotonicity enforcement. Every L2 pool contract reads this rate to compute rsETH output per ETH deposited. Because rsETH appreciates continuously as L1 staking rewards accrue, any lag between the true rate and the stored rate creates a window where any unprivileged depositor receives more rsETH than their ETH is worth, extracting accrued yield from existing rsETH holders.

## Finding Description
`InterimRSETHOracle._setRate()` enforces only a floor of `1e18` and stores the rate with no `lastUpdated` timestamp:

```solidity
function _setRate(uint256 newRate) internal {
    if (newRate < 1e18) revert InvalidRate();
    rate = newRate;
    emit RateUpdated(newRate);
}
```

`getRate()` returns the stored value verbatim with no staleness check. All pool contracts call `IOracle(rsETHOracle).getRate()` inside `viewSwapRsETHAmountAndFee` and compute:

```solidity
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
```

A rate that is lower than the true current rate produces a larger `rsETHAmount`. Since rsETH accrues staking yield monotonically on L1, every period between manual `MANAGER_ROLE` updates is a window where the stored rate is below the true rate. Any depositor calling `deposit()` during this window receives excess rsETH:

- **`RSETHPoolV2` and `RSETHPoolV3`**: excess rsETH is freshly minted via `wrsETH.mint(msg.sender, rsETHAmount)`, directly diluting all existing rsETH holders.
- **`RSETHPool` and `RSETHPoolNoWrapper`**: excess rsETH is transferred from the pool's own reserves via `safeTransfer`, draining them.

The daily mint limit in `RSETHPoolV2`/`RSETHPoolV3` caps per-day damage but does not prevent the attack; it merely bounds the maximum extraction per cycle.

No privileged access is required. The attacker only needs to observe the on-chain `InterimRSETHOracle.rate` and compare it to the true rate (readable from `LRTOracle.rsETHPrice` on L1 or secondary markets), then call `deposit()` when a profitable gap exists.

## Impact Explanation
**High — Theft of unclaimed yield.**

The excess rsETH minted or transferred represents staking yield that has accrued to existing rsETH holders but has not yet been reflected in the oracle. An attacker who deposits ETH during a stale-rate window receives rsETH tokens whose on-chain redemption value (set by `LRTOracle.rsETHPrice` on L1) exceeds the ETH they paid. The difference is extracted from the collective value backing all outstanding rsETH. For minting pools (`RSETHPoolV2`, `RSETHPoolV3`), this is unbounded dilution of every rsETH holder proportional to deposit size and rate lag magnitude. This matches the allowed impact: **Theft of unclaimed yield**.

## Likelihood Explanation
**Medium.** rsETH appreciates continuously and predictably. Manual rate updates require a privileged `MANAGER_ROLE` transaction, introducing inherent latency (block confirmation, keeper uptime, gas conditions). No on-chain mechanism enforces a maximum update interval. An attacker needs only to observe the on-chain rate, compare it to the true rate, and call `deposit()` when a profitable gap exists. No special permissions, flash loans, or front-running are required. The attack is repeatable every update cycle and scales linearly with deposit size and rate lag.

## Recommendation
1. Replace `InterimRSETHOracle` with a pull-based oracle that reads `LRTOracle.rsETHPrice` directly (or via `RSETHPriceFeed`) so the rate is always current without manual intervention.
2. If a manual oracle must be retained, add a `lastUpdated` timestamp and revert in `getRate()` if `block.timestamp - lastUpdated > MAX_STALENESS` (e.g., 1 hour).
3. Add a monotonicity check in `_setRate` so the rate cannot be set below the previously recorded rate (rsETH should never depreciate under normal conditions), preventing accidental or malicious rate reductions.

## Proof of Concept
**Setup:**
- `InterimRSETHOracle.rate` = `1.005e18` (set 6 hours ago).
- True current rsETH/ETH rate (from `LRTOracle.rsETHPrice` on L1) = `1.010e18`.
- `RSETHPoolV2` uses `InterimRSETHOracle` as `rsETHOracle`; `feeBps = 0`.

**Attack steps:**
1. Attacker calls `RSETHPoolV2.deposit{value: 100 ether}("")`.
2. `viewSwapRsETHAmountAndFee(100e18)` computes: `rsETHAmount = 100e18 * 1e18 / 1.005e18 ≈ 99.502e18`.
3. `wrsETH.mint(attacker, 99.502e18)` executes.
4. At the true rate of `1.010e18`, those 99.502 rsETH are redeemable for `99.502 * 1.010 ≈ 100.497 ETH`.
5. Attacker deposited 100 ETH and holds rsETH worth ~100.497 ETH — a gain of ~0.497 ETH extracted from existing rsETH holders.
6. Repeatable every update cycle; scales with deposit size and rate lag.

**Foundry fork test plan:**
- Fork an L2 where `RSETHPoolV2` is deployed with `InterimRSETHOracle`.
- Record `InterimRSETHOracle.rate` and `LRTOracle.rsETHPrice` (via cross-chain read or mock).
- Simulate time passage without a `setRate` call.
- Call `deposit` with a large ETH value.
- Assert `wrsETH.balanceOf(attacker) * trueRate / 1e18 > depositAmount`, confirming profit extracted from existing holders.
