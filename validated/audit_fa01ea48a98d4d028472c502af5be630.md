### Title
`RequireGuard` Native Script Satisfied Without Credential Authorization — (`eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Scripts.hs`)

---

### Summary

The Dijkstra era introduces a new native script variant `DijkstraRequireGuard (Credential Guard)`, documented as requiring "a credential to authorize execution." However, the evaluation function checks only whether the credential appears in the transaction body's `guards` field — a field the transaction submitter populates freely with no witness or signature requirement. Any unprivileged attacker can spend a UTxO locked by `RequireGuard (KeyHashObj X)` by simply listing `X` in their own transaction's `guards` field, without possessing or providing a signature from `X`. This is the direct Cardano analog of the BathToken `onlyPair` bypass: the authorization check consults an attacker-controlled value rather than an independently verified one.

---

### Finding Description

**Root cause — `evalDijkstraNativeScript`**

The evaluation function for Dijkstra native scripts is:

```haskell
evalDijkstraNativeScript ::
  ...
  OSet (Credential Guard) ->   -- guards: taken directly from txBody
  NativeScript era ->
  Bool
evalDijkstraNativeScript keyHashes (ValidityInterval txStart txExp) guards = go
  where
    go = \case
      ...
      RequireGuard cred -> cred `OSet.member` guards   -- ← attacker-controlled set
``` [1](#0-0) 

The `guards` argument is sourced directly from `txBody ^. guardsTxBodyL`:

```haskell
guardsTxBodyL :: Lens' (TxBody l era) (OSet (Credential Guard))
``` [2](#0-1) 

This field is an ordinary transaction body field. Any transaction submitter can set it to any set of credentials with no restriction.

**No witness requirement for guard key-hash credentials**

The Dijkstra UTXOW transition validates native scripts via `Babbage.validateFailedBabbageScripts`, which calls `evalDijkstraNativeScript` with the transaction's `guards` set. The subsequent `Shelley.validateNeededWitnesses` call checks that key witnesses are present for UTxO inputs, certificates, and withdrawals — but **not** for credentials listed in `guardsTxBodyL`. [3](#0-2) 

**Test confirms the bypass**

The existing test suite explicitly demonstrates that a UTxO locked by `RequireGuard (KeyHashObj guardKeyHash)` is successfully spent by a transaction that merely lists `guardKeyHash` in `guardsTxBodyL` — no signature from `guardKeyHash` is added or required:

```haskell
submitTx_ $ tx & bodyTxL . guardsTxBodyL .~ [guardKeyHash]
-- ↑ no witsTxL modification; no key witness for guardKeyHash
``` [4](#0-3) 

A second test, "A redundant guard is ignored," confirms that arbitrary credentials can be placed in `guardsTxBodyL` without any validation: [5](#0-4) 

**Structural analog to BathToken M-18**

| BathToken | Dijkstra |
|---|---|
| `IBathHouse(bathHouse).isApprovedPair(msg.sender)` | `cred \`OSet.member\` guards` |
| `bathHouse` is an admin-controlled reference | `guards` is a transaction-submitter-controlled field |
| Admin replaces `bathHouse` with always-true contract | Attacker populates `guards` with victim's credential |
| `rebalance()` drains funds | Spending input drains locked UTxO |

In both cases the authorization check consults a value the attacker controls, making the check trivially bypassable.

---

### Impact Explanation

**Critical — Direct loss of ADA or native assets through an invalid ledger state transition.**

Any UTxO whose locking script is (or contains) `RequireGuard (KeyHashObj X)` can be unconditionally spent by any unprivileged transaction submitter. The attacker does not need the private key for `X`; they only need to know `X`'s key hash (which is public once the UTxO is on-chain). ADA and all native assets held at such addresses are fully exposed to theft.

---

### Likelihood Explanation

**High.** The attacker requires no privileged access, no leaked keys, and no governance majority. The only precondition is that a UTxO locked by a `RequireGuard (KeyHashObj X)` script exists on-chain. The exploit is a single well-formed transaction. The CDDL specification comment explicitly advertises `script_require_guard` as providing security ("requires a credential to authorize execution"), making it likely that users and tooling will deploy it expecting authorization semantics. [6](#0-5) 

---

### Recommendation

The UTXOW rule must enforce that every key-hash credential appearing in `guardsTxBodyL` has a corresponding valid key witness (signature over the transaction body). Concretely, the Dijkstra override of `witsVKeyNeeded` (or an equivalent predicate in `dijkstraUtxowTransition`) should add all `KeyHashObj kh` entries from `txBody ^. guardsTxBodyL` to the required witness set, mirroring how `reqSignerHashesTxBodyL` is handled in Alonzo/Babbage/Conway.

For `ScriptHashObj sh` entries in `guardsTxBodyL`, the existing script-witness and script-validation checks already enforce that the guard script is provided and evaluates to `True`, so no additional change is needed for that case.

---

### Proof of Concept

1. Alice creates a UTxO locked by `RequireGuard (KeyHashObj aliceKeyHash)` (e.g., to protect a deposit).
2. Bob (no keys, no privileges) observes `aliceKeyHash` from the on-chain script.
3. Bob constructs:
   ```haskell
   let tx = mkBasicTx (mkBasicTxBody
             & inputsTxBodyL  .~ [aliceLockedTxIn]
             & outputsTxBodyL .~ [bobOutputWithAliceFunds])
             & bodyTxL . guardsTxBodyL .~ [KeyHashObj aliceKeyHash]
   ```
4. Bob submits `tx`. `evalDijkstraNativeScript` evaluates `RequireGuard (KeyHashObj aliceKeyHash)` as `True` because `KeyHashObj aliceKeyHash \`OSet.member\` {KeyHashObj aliceKeyHash}`. No signature from Alice is checked.
5. The ledger accepts the transaction. Alice's funds are transferred to Bob. [7](#0-6) [8](#0-7)

```haskell
evalDijkstraNativeScript ::
  (DijkstraEraScript era, NativeScript era ~ DijkstraNativeScript era) =>
  Set.Set (KeyHash Witness) ->
  ValidityInterval ->
  OSet (Credential Guard) ->
  NativeScript era ->
  Bool
evalDijkstraNativeScript keyHashes (ValidityInterval txStart txExp) guards = go
  where
    -- the evaluation will stop as soon as it reaches the required number of valid scripts
    isValidMOf n SSeq.Empty = n <= 0
    isValidMOf n (ts SSeq.:<| tss) =
      n <= 0 || if go ts then isValidMOf (n - 1) tss else isValidMOf n tss
    go = \case
      RequireTimeStart lockStart -> lockStart `lteNegInfty` txStart
      RequireTimeExpire lockExp -> txExp `ltePosInfty` lockExp
      RequireSignature hash -> hash `Set.member` keyHashes
      RequireAllOf xs -> all go xs
      RequireAnyOf xs -> any go xs
      RequireMOf m xs -> isValidMOf m xs
      RequireGuard cred -> cred `OSet.member` guards
      _ -> error "Impossible: All NativeScripts should have been accounted for"
```
