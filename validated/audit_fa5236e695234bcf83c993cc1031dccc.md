### Title
Missing resharding parent-shard resolution in `compute_state_response_header` causes `InvalidShardId` for all child shards at epoch boundary — (`chain/chain/src/state_sync/adapter.rs`)

### Summary

`compute_state_response_header` validates `shard_id` against the **new** epoch's shard layout (child shards), then unconditionally calls `prev_shard_layout.get_shard_index(shard_id)` against the **old** layout. At a resharding epoch boundary the child `ShardId` does not exist in the old layout, so the call returns `ShardLayoutError::InvalidShardId`, making state sync header computation fail for every child shard.

### Finding Description

In `compute_state_response_header`:

```
shard_ids  = epoch_manager.shard_ids(sync_block_epoch_id)   // NEW layout child IDs
shard_ids.contains(&shard_id)                               // passes for child shard
prev_shard_layout = get_shard_layout(&prev_epoch_id)        // OLD layout
prev_shard_index  = prev_shard_layout.get_shard_index(shard_id)  // ← child ID not in old layout → Err
``` [1](#0-0) 

The guard at line 83 only checks membership in the **new** epoch's shard IDs. It does not detect that `shard_id` is a freshly-created child shard absent from `prev_shard_layout`. The `?` at line 96 propagates `ShardLayoutError::InvalidShardId` as `Error::InvalidShardId`. [2](#0-1) 

The correct pattern — used in `get_prev_shard_infos` and `get_prev_shard_id_from_prev_hash` — is to detect the resharding boundary, call `shard_layout.get_parent_shard_id(shard_id)` on the **new** layout, and only then index into `prev_shard_layout` with the parent ID: [3](#0-2) 

`compute_state_response_header` omits this step entirely.

### Impact Explanation

Every call to `get_state_response_header` (and the state-dump path in `nearcore/src/state_sync.rs`) delegates to `compute_state_response_header`. [4](#0-3) 

When `sync_hash` is the first block of a resharding epoch, the function returns an error for every child shard. No state sync header can be computed or cached. Nodes that need to catch up via state sync after a dynamic resharding event cannot obtain headers and therefore cannot complete state sync. The failure is deterministic and affects all child shards simultaneously.

### Likelihood Explanation

The condition is reached whenever:
1. Dynamic resharding (ShardLayoutV2/V3) is active and an epoch boundary with a shard split is crossed.
2. Any node attempts state sync for that epoch (new node joining, node recovering from lag, state dump).

No privileged access is required. The `sync_hash` is a valid, finalized block hash; the `shard_id` is a legitimately assigned child shard ID. Both values are determined by normal protocol operation, not by an adversary.

### Recommendation

Before calling `prev_shard_layout.get_shard_index`, detect whether `shard_layout != prev_shard_layout` (resharding boundary) and, if so, resolve the parent shard ID first:

```rust
let is_resharding = shard_layout != prev_shard_layout;
let prev_shard_id = if is_resharding {
    shard_layout.get_parent_shard_id(shard_id)?
} else {
    shard_id
};
let prev_shard_index = prev_shard_layout.get_shard_index(prev_shard_id)?;
```

This mirrors the pattern already used in `get_prev_shard_infos` and `get_prev_shard_id_from_prev_hash`. [5](#0-4) 

### Proof of Concept

1. Configure a test network with `ShardLayoutV3` dynamic resharding enabled.
2. Advance to the first resharding epoch boundary.
3. Obtain `sync_hash` = first block of the resharding epoch; obtain a child `shard_id` from `epoch_manager.shard_ids(sync_block_epoch_id)`.
4. Call `chain.state_sync_adapter.compute_state_response_header(child_shard_id, sync_hash)`.
5. Assert the result is `Err(Error::InvalidShardId(child_shard_id))` — not a valid `ShardStateSyncResponseHeader`. [6](#0-5) [7](#0-6)

    fn get_prev_shard_id_from_prev_hash(

            Ok((shard_layout, shard_id, shard_index))
