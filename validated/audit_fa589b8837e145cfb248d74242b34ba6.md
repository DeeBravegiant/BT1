### Title
Caller-Agnostic `verify_foreign_transaction` Queue Key Allows Unprivileged Queue Saturation, Blocking Legitimate Users and Enabling Response Theft - (File: `crates/contract/src/lib.rs`, `crates/contract/src/pending_requests.rs`)

---

### Summary

The `verify_foreign_transaction` function stores pending requests under a key that does **not** include the caller's account ID. Any unprivileged actor can observe a target foreign-chain transaction ID, pre-fill all 128 fan-out queue slots for that request key, and thereby prevent the legitimate user from ever entering the queue. The attacker simultaneously receives the MPC-signed response when nodes process the request, enabling downstream theft if the consuming bridge contract does not bind the response to the original submitter.

---

### Finding Description

`SignatureRequest` binds the queue key to the caller's account ID via `SignatureRequest::new(..., &predecessor, ...)`, so each user occupies a distinct map entry. `VerifyForeignTransactionRequest` carries no such binding:

```rust
// crates/near-mpc-contract-interface/src/types/foreign_chain.rs:124-128
pub struct VerifyForeignTransactionRequest {
    pub request: ForeignChainRpcRequest,
    pub domain_id: DomainId,
    pub payload_version: ForeignTxPayloadVersion,
}
```

The contract's own test acknowledges this: *"a different account would today be blocked from receiving a response by alice's submission."*

All callers submitting the same `(chain_rpc_request, domain_id, payload_version)` tuple share one `LookupMap` entry. The fan-out queue for that entry is hard-capped at `MAX_PENDING_REQUEST_FAN_OUT = 128`:

```rust
// crates/contract/src/pending_requests.rs:37,51-57
pub const MAX_PENDING_REQUEST_FAN_OUT: u8 = 128;

if queue.len() >= usize::from(MAX_PENDING_REQUEST_FAN_OUT) {
    env::panic_str(
        &RequestError::PendingRequestQueueFull { limit: MAX_PENDING_REQUEST_FAN_OUT }
            .to_string(),
    );
}
```

Once all 128 slots are occupied, every subsequent `verify_foreign_transaction` call for that key panics with `PendingRequestQueueFull`. The queue only drains when `respond_verify_foreign_tx` is called (draining all 128 at once) or after the ~200-block NEAR yield-resume timeout expires per slot.

The signed response carries no submitter identity:

```rust
// crates/near-mpc-contract-interface/src/types/foreign_chain.rs:147-150
pub struct VerifyForeignTransactionResponse {
    pub payload_hash: Hash256,
    pub signature: SignatureResponse,
}
```

An attacker who saturates the queue receives the same MPC-signed `(payload_hash, signature)` as any legitimate user would have received, and can present it to any downstream contract that does not enforce submitter identity.

---

### Impact Explanation

**Request-lifecycle invariant broken (Medium):** A legitimate user cannot submit a `verify_foreign_transaction` request for a specific foreign-chain transaction while an attacker holds all 128 queue slots. The user is locked out for up to ~200 NEAR blocks per saturation cycle. The attacker can repeat the saturation immediately after each drain, creating a sustained denial of service for that specific request key.

**Response theft enabling downstream fund loss:** The MPC-signed response is a transferable proof that the MPC network verified the foreign-chain transaction. Any bridge contract that uses this proof to release funds without binding it to the original submitter's account ID will release funds to the attacker. This breaks the production safety invariant that only the party who initiated the foreign-chain transaction can claim the corresponding NEAR-side funds.

---

### Likelihood Explanation

- The foreign-chain transaction ID is publicly observable on-chain before the victim calls `verify_foreign_transaction`.
- Submitting 128 requests costs 128 yoctonear in deposits — economically negligible.
- No privileged access, TEE attestation, or participant role is required; any NEAR account can call `verify_foreign_transaction`.
- The attack window is the time between the foreign-chain transaction being confirmed and the victim submitting their NEAR call — typically seconds to minutes, easily exploitable by a mempool-watching bot.

---

### Recommendation

Include the caller's account ID in the `VerifyForeignTransactionRequest` key, mirroring the pattern used by `SignatureRequest`:

```rust
// Analogous to SignatureRequest::new(..., &predecessor, ...)
pub struct VerifyForeignTransactionRequest {
    pub request: ForeignChainRpcRequest,
    pub domain_id: DomainId,
    pub payload_version: ForeignTxPayloadVersion,
    pub submitter: AccountId,   // add this field
}
```

This gives each submitter a distinct map entry, eliminating cross-user queue interference. Downstream bridge contracts should additionally verify that the response is consumed by the same account that submitted the request.

---

### Proof of Concept

1. Alice sends ETH to a bridge contract on Ethereum; the tx_id `T` is visible on-chain.
2. Eve observes `T` and immediately submits 128 calls to `verify_foreign_transaction` with `request = T`, `domain_id = D`, `payload_version = V1` — each costing 1 yoctonear.
3. The fan-out queue for key `(T, D, V1)` is now at capacity (128 slots, all owned by Eve's yields).
4. Alice calls `verify_foreign_transaction` with the same arguments; the contract panics with `PendingRequestQueueFull`.
5. MPC nodes observe the request, query the foreign chain, and call `respond_verify_foreign_tx`. `resolve_yields_for` drains all 128 of Eve's yields, delivering the signed `(payload_hash, signature)` to each.
6. Eve presents the signed response to the NEAR bridge contract and claims Alice's bridged funds.
7. Alice's call remains permanently rejected until she retries after the queue drains — by which point her funds are already claimed.

```rust
pub const MAX_PENDING_REQUEST_FAN_OUT: u8 = 128;

/// Append a yield index to the pending-request fan-out queue for `request`.
///
/// Panics with `RequestError::PendingRequestQueueFull` if the resulting queue would
/// exceed `MAX_PENDING_REQUEST_FAN_OUT`.
pub(crate) fn push_pending_yield<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: K,
    data_id: CryptoHash,
) where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let queue = requests.entry(request).or_default();
    if queue.len() >= usize::from(MAX_PENDING_REQUEST_FAN_OUT) {
        env::panic_str(
            &RequestError::PendingRequestQueueFull {
                limit: MAX_PENDING_REQUEST_FAN_OUT,
            }
            .to_string(),
        );
    }
    queue.push(YieldIndex { data_id });
```
