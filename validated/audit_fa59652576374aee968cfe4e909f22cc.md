### Title
Harvester Can Unconditionally Redirect Farmer Block Rewards to Arbitrary Address Without Fee Quality Enforcement - (File: chia/farmer/farmer_api.py)

### Summary
In `_process_respond_signatures()`, the farmer unconditionally accepts a `farmer_reward_address_override` supplied by a connected harvester in the `RespondSignatures` protocol message, with no cryptographic verification and no fee-quality threshold enforcement. Any harvester can redirect the farmer's block reward (1/8 of the block reward, ~0.25 XCH per block) to an arbitrary puzzle hash of its choosing.

### Finding Description
The CHIP-22 harvester fee convention (`farmer_reward_address_override`) allows a harvester to claim the farmer reward as a fee when a proof-of-space meets a declared quality threshold. The farmer is supposed to validate this threshold before accepting the override.

However, in `_process_respond_signatures()`, the `farmer_reward_address_override` field from the `RespondSignatures` message is accepted with no fee-quality check whatsoever:

```python
# chia/farmer/farmer_api.py lines 916-919
farmer_reward_address = self.farmer.farmer_target
if response.farmer_reward_address_override is not None:
    farmer_reward_address = response.farmer_reward_address_override
    include_source_signature_data = True
``` [1](#0-0) 

The fee-quality check (`notify_farmer_reward_taken_by_harvester_as_fee`) is only called in the `new_proof_of_space` path when the harvester first announces a proof, and even there it only **logs a warning** — it never rejects the override:

```python
# chia/farmer/farmer_api.py lines 128-129
if new_proof_of_space.farmer_reward_address_override is not None:
    self.farmer.notify_farmer_reward_taken_by_harvester_as_fee(sp, new_proof_of_space)
``` [2](#0-1) 

The `notify_farmer_reward_taken_by_harvester_as_fee` function only logs warnings even when the fee threshold is violated: [3](#0-2) 

The `RespondSignatures` protocol message itself carries the `farmer_reward_address_override` field as an optional `bytes32`: [4](#0-3) 

The resulting `DeclareProofOfSpace` is sent to the full node with the harvester-supplied address as `farmer_reward_address`: [5](#0-4) 

### Impact Explanation
A malicious harvester connected to a farmer can set `farmer_reward_address_override` to any arbitrary puzzle hash in its `RespondSignatures` response. The farmer will use that address as the farmer reward destination in `DeclareProofOfSpace`, causing the farmer's block reward (~0.25 XCH per block) to be paid to the harvester's chosen address. The farmer has no mechanism to detect or block this at the protocol level — only a log warning is emitted, which the farmer operator may never see. This constitutes unauthorized payout redirection of XCH.

### Likelihood Explanation
Any harvester that is connected to the farmer (including third-party harvesters explicitly supported by CHIP-22) can exploit this. The attack requires no leaked keys, no cryptographic break, and no host compromise — only a network connection to the farmer's harvester port. Third-party harvester setups (e.g., DrPlotter, compressed plot services) are a common and growing deployment pattern, making this reachable for a meaningful fraction of the network.

### Recommendation
In `_process_respond_signatures()`, before accepting `farmer_reward_address_override`, the farmer should:
1. Verify that the fee quality for this proof meets or exceeds the threshold declared by the harvester in the original `NewProofOfSpace.fee_info.applied_fee_threshold`.
2. Reject (return `None`) if the threshold is not met or if `fee_info` was not provided with the original proof.

The fee-quality check already exists in `notify_farmer_reward_taken_by_harvester_as_fee` but is non-enforcing. It should be converted into a hard gate that prevents the override from being used in `DeclareProofOfSpace` when the threshold is not satisfied.

### Proof of Concept
1. A harvester connects to a farmer's harvester port.
2. The harvester finds a valid proof of space and sends `NewProofOfSpace` with `farmer_reward_address_override=<attacker_address>` and `fee_info=None` (or any threshold).
3. The farmer calls `notify_farmer_reward_taken_by_harvester_as_fee` which logs a warning but does not block.
4. The farmer sends `RequestSignatures` to the harvester.
5. The harvester responds with `RespondSignatures` containing `farmer_reward_address_override=<attacker_address>`.
6. `_process_respond_signatures` at line 917–918 unconditionally sets `farmer_reward_address = response.farmer_reward_address_override`.
7. `DeclareProofOfSpace` is sent to the full node with `farmer_reward_address=<attacker_address>`.
8. The block is farmed and the farmer reward coin is created at `<attacker_address>`, not the farmer's configured `xch_target_address`.
