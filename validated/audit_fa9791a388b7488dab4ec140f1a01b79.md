Audit Report

## Title
ETH Deposit Limit Check Omits Incoming Amount, Allowing Limit Bypass - (File: contracts/LRTDepositPool.sol)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` applies an asymmetric comparison: the ERC20 branch correctly adds the incoming `amount` to `totalAssetDeposits` before comparing against the limit, but the ETH branch omits `amount` entirely. When total ETH deposits exactly equal the configured cap, the ETH check evaluates `limit > limit` → `false`, the deposit is not reverted, and ETH deposits exceed the protocol-configured cap.

## Finding Description
In `contracts/LRTDepositPool.sol` at lines 676–682:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)); // `amount` absent
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // correct
}
```

The ETH branch at line 679 uses `totalAssetDeposits > limit` instead of `totalAssetDeposits + amount > limit`. When `totalAssetDeposits == depositLimit`, the expression is `limit > limit` → `false`, so `_beforeDeposit` (lines 648–670) does not revert with `MaximumDepositLimitReached`, and `depositETH` (lines 76–93) proceeds to mint rsETH and accept the ETH. The call path `depositETH → _beforeDeposit → _checkIfDepositAmountExceedesCurrentLimit` is fully public and requires no privileges.

## Impact Explanation
The deposit limit is the protocol's primary on-chain cap for ETH inflows. The bug allows any depositor to push ETH deposits past the configured cap by exactly `msg.value` once the limit is reached. No funds are stolen or frozen, but the protocol fails to enforce its stated deposit-cap guarantee for ETH. This matches the allowed Low impact: **Contract fails to deliver promised returns, but doesn't lose value**.

## Likelihood Explanation
The boundary condition `totalAssetDeposits == depositLimit` is a natural state that occurs as deposits accumulate toward the cap. Any depositor monitoring on-chain state can observe this moment and call `depositETH` with any non-zero `msg.value`. No privileged access, special role, or victim mistake is required. The condition is repeatable each time the limit is exactly met.

## Recommendation
Mirror the ERC20 branch in the ETH branch by including `amount`:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

## Proof of Concept
1. Set `depositLimitByAsset(ETH_TOKEN)` to `100 ether`.
2. Accumulate prior deposits until `getTotalAssetDeposits(ETH_TOKEN) == 100 ether`.
3. Call `depositETH(0, "ref")` with `msg.value = 1 ether` from any EOA.
4. `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 1 ether)` evaluates `100 ether > 100 ether` → `false`.
5. `_beforeDeposit` does not revert; rsETH is minted to the caller.
6. `getTotalAssetDeposits(ETH_TOKEN)` is now `101 ether`, exceeding the configured limit.

**Foundry test sketch:**
```solidity
function test_ethDepositLimitBypass() public {
    // set limit to 100 ether, fill deposits to exactly 100 ether via prior calls
    // then assert the following call succeeds and total > limit
    vm.deal(attacker, 1 ether);
    vm.prank(attacker);
    depositPool.depositETH{value: 1 ether}(0, "ref");
    assertGt(depositPool.getTotalAssetDeposits(ETH_TOKEN), lrtConfig.depositLimitByAsset(ETH_TOKEN));
}
```
