### Title
`respond_ckd` Skips Output Verification for `AppPublicKey` Variant, Allowing a Byzantine Leader to Deliver Arbitrary CKD Output — (`crates/contract/src/lib.rs`)

---

### Summary

`respond_ckd` enforces a full BLS12-381 pairing-based output check for `AppPublicKeyPV` requests but performs **zero cryptographic verification** for `AppPublicKey` (privately-verifiable) requests. A single Byzantine attested participant acting as the signing leader can call `respond_ckd` with a fabricated `CKDResponse` for any pending `AppPublicKey` request, and the contract will accept and resolve the yield with that forged output — bypassing the threshold-computation requirement entirely.

---

### Finding Description

The `respond_ckd` function in `crates/contract/src/lib.rs` handles two variants of CKD requests:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← no check at all
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV`, `ckd_output_check` verifies the pairing equation `e(big_c, g2) = e(big_y, app_pk2) · e(H(pk, app_id), pk)`, which cryptographically proves the response was computed using the MPC master secret key. [2](#0-1) 

For `AppPublicKey`, the arm is an empty no-op. Any `CKDResponse` — with arbitrary `big_y` and `big_c` — is accepted unconditionally and immediately resolved into the pending yield. [3](#0-2) 

This is the direct analog of the ERC4626 bug: `previewMint` enforced the first-deposit protection while `previewDeposit` did not, allowing an attacker to use the unprotected path. Here, `AppPublicKeyPV` enforces the threshold-output invariant while `AppPublicKey` does not, allowing a Byzantine leader to use the unprotected path.

The `AppPublicKey` variant is still actively supported (it is the "legacy" privately-verifiable form documented in the contract README): [4](#0-3) 

The node-side computation correctly runs the full threshold CKD protocol for both variants: [5](#0-4) 

But the on-chain response handler only enforces the output for one of them.

---

### Impact Explanation

A single Byzantine attested participant who is elected leader for a CKD signing round can call `respond_ckd` with a fabricated `(big_y, big_c)` pair for any pending `AppPublicKey` request. The contract resolves the yield with the forged output. The requesting TEE application receives a derived key that was not computed by the threshold of MPC participants — the threshold-computation requirement is bypassed for this variant.

The TEE app computes `secret = big_c − a · big_y`. Because the attacker does not know the app's ephemeral private key `a`, they cannot choose a specific target secret. However, they can:

1. Deliver a deterministically wrong key, causing the app to derive a value unrelated to the MPC master secret.
2. Persistently corrupt every `AppPublicKey` CKD response if they are consistently elected leader, making the privately-verifiable CKD path permanently unreliable.

The app can detect the wrong key off-chain (by checking `e(secret, G2) = e(H(pk, app_id), pk)` against the public MPC key), but the yield has already been resolved — the correct key cannot be obtained from that request. The user must re-submit, and the attacker can repeat the attack.

This breaks the production safety invariant that CKD outputs are produced by the threshold of participants, matching the allowed Medium impact: **request-lifecycle manipulation that breaks production safety/accounting invariants**.

---

### Likelihood Explanation

- The attacker must be a single attested participant in the MPC network — no threshold collusion is required.
- Leader election is deterministic and observable, so a Byzantine node knows when it will be leader.
- The `AppPublicKey` variant is the legacy default and is still used in production (documented in the contract README and tested in the sandbox test suite).
- The attack requires no special knowledge beyond being an attested participant and knowing the pending request's `CKDRequest` fields (which are on-chain). [6](#0-5) 

---

### Recommendation

Apply the same `ckd_output_check` to the `AppPublicKey` variant in `respond_ckd`. For the privately-verifiable variant, an equivalent check exists: verify `e(big_c, g2) = e(big_y, g2) · e(H(pk, app_id), pk)` (using `g2` in place of `app_pk2`, since `AppPublicKey` only provides a G1 point). Alternatively, require all new CKD requests to use `AppPublicKeyPV` and deprecate the unverifiable `AppPublicKey` path at the contract level.

---

### Proof of Concept

```rust
// Attacker is an attested participant and is the leader for this CKD round.
// A legitimate user has submitted a request_app_private_key with AppPublicKey variant.

// The attacker fabricates an arbitrary response:
let forged_response = CKDResponse {
    big_y: dtos::Bls12381G1PublicKey([0xAA; 48]), // arbitrary G1 point
    big_c: dtos::Bls12381G1PublicKey([0xBB; 48]), // arbitrary G1 point
};

// The contract accepts this with no verification:
contract.respond_ckd(ckd_request, forged_response).expect("accepted");
// The yield is resolved. The user's TEE app receives (0xAA..., 0xBB...)
// and derives secret = 0xBB... - a * 0xAA..., which is not the MPC-derived key.
```

The existing unit test `respond_ckd__should_succeed_when_response_is_valid_and_request_exists` already demonstrates this: it passes `big_y = [1u8; 48]` and `big_c = [2u8; 48]` (not a valid CKD output) and the contract accepts them without complaint. [7](#0-6)

pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}
