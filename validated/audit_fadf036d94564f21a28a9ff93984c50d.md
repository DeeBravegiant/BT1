### Title
Blocklisted Recipient Causes Permanent Fund Lock in `finTransfer` - (File: evm/src/omni-bridge/contracts/OmniBridge.sol)

### Summary
`OmniBridge.finTransfer` uses a push pattern to deliver native ERC20 tokens (e.g., USDC, USDT) directly to `payload.recipient`. If that address is on the token's admin-controlled blocklist, every delivery attempt reverts. Because the user's funds were already locked or burned on the source chain (NEAR) when `initTransfer` was called, and the signed payload cannot be altered, the funds become permanently unclaimable.

### Finding Description
`finTransfer` in `OmniBridge.sol` handles the final leg of a cross-chain transfer. For native ERC20 tokens that are neither bridge-minted nor custom-minted, it executes:

```solidity
} else {
    IERC20(payload.tokenAddress).safeTransfer(
        payload.recipient,
        payload.amount
    );
}
``` [1](#0-0) 

The `payload.recipient` is embedded in the MPC-signed message and cannot be changed without a new signature. The nonce guard at the top of `finTransfer` marks the nonce consumed only within the same transaction:

```solidity
completedTransfers[payload.destinationNonce] = true;
``` [2](#0-1) 

Because the entire transaction reverts when `safeTransfer` fails, the nonce is never permanently consumed — but the transfer also never succeeds. The result is an infinite revert loop: every relayer call to `finTransfer` for that payload reverts, and there is no alternative delivery path or refund entrypoint in the contract.

Tokens such as USDC and USDT implement an admin-controlled blocklist; any `transfer` or `transferFrom` to a blocked address reverts at the token level. The bridge has no awareness of this and no fallback.

### Impact Explanation
**Critical — Irreversible fund lock / permanently unclaimable user value.**

The user's tokens are burned or locked on NEAR the moment `initTransfer` completes on the source side. If `finTransfer` can never succeed on the EVM side (because the recipient is blocklisted), those tokens are permanently lost with no recovery path available to the user or the protocol. There is no admin rescue function, no alternative recipient override, and no refund mechanism in `OmniBridge`. [3](#0-2) 

### Likelihood Explanation
**Low.** Three conditions must coincide: (1) the bridged token must be a native ERC20 with a blocklist (USDC, USDT), (2) the recipient address must be on that blocklist at the time of settlement, and (3) the blocklist status must persist (i.e., not be a transient error). All three are realistic but uncommon in practice.

### Recommendation
Adopt the Pull-over-Push pattern for native ERC20 delivery in `finTransfer`. Instead of pushing tokens to `payload.recipient` immediately, credit the amount to an internal `claimable[recipient][token]` mapping and emit an event. Provide a separate `claimTokens(address token)` function that the recipient calls to withdraw. This decouples settlement finality from token delivery and eliminates the revert-on-blocklist DoS.

### Proof of Concept

1. Alice holds USDC on NEAR and calls the NEAR bridge contract to initiate a transfer to her EVM address. Her USDC is locked/burned on NEAR.
2. The MPC signs a `TransferMessagePayload` with `tokenAddress = USDC_EVM`, `recipient = Alice_EVM`, `amount = X`.
3. Before the relayer settles, USDC's admin blocklists `Alice_EVM` (e.g., due to a compliance action).
4. A relayer calls `OmniBridge.finTransfer(signatureData, payload)`.
5. Execution reaches line 351: `IERC20(USDC).safeTransfer(Alice_EVM, X)` — USDC reverts because `Alice_EVM` is blocklisted.
6. The entire transaction reverts; `completedTransfers[nonce]` is not persisted.
7. Any subsequent relayer call with the same (only valid) signed payload hits the same revert.
8. Alice's USDC is permanently locked on NEAR with no recourse. [4](#0-3)
