Audit Report

## Title
`maxNumberOfWithdrawalsPerUser` Uninitialized in `KernelDepositPool.initialize()`, Blocking All Withdrawals Until Admin Intervenes - (File: contracts/KERNEL/KernelDepositPool.sol)

## Summary
`KernelDepositPool.initialize()` never assigns `maxNumberOfWithdrawalsPerUser`, leaving it at the Solidity default of `0`. Because `initiateWithdrawal()` reverts when `userWithdrawalIds[msg.sender].length >= maxNumberOfWithdrawalsPerUser`, and any `uint256` is always `>= 0`, every withdrawal attempt reverts with `WithdrawalLimitReached` from the moment of deployment. Staked KERNEL tokens are frozen until an admin manually calls `setMaxNumberOfWithdrawalsPerUser` with a non-zero value.

## Finding Description
`initialize()` sets roles and token addresses but never sets `maxNumberOfWithdrawalsPerUser`: [1](#0-0) 

The variable therefore holds `0`. The admin setter explicitly rejects `0`: [2](#0-1) 

`initiateWithdrawal()` enforces the limit before creating any withdrawal record: [3](#0-2) 

Since `userWithdrawalIds[msg.sender].length` is a `uint256` and always `>= 0`, the guard is unconditionally `true` when `maxNumberOfWithdrawalsPerUser == 0`, causing every call to revert with `WithdrawalLimitReached`. There is no bypass: the setter cannot be used to set `0`, and `initialize()` is callable only once.

## Impact Explanation
Any user who calls `stake()` or `stakeFor()` has their KERNEL tokens locked in the contract with no withdrawal path until an admin intervenes. This is a **temporary freezing of funds** — a Medium-severity impact under the allowed scope. The freeze is not permanent because an admin can call `setMaxNumberOfWithdrawalsPerUser` with a valid value, but until that happens all depositors are unable to retrieve their principal. [4](#0-3) 

## Likelihood Explanation
The condition is triggered on the very first call to `initiateWithdrawal()` by any user, with no special preconditions beyond having staked tokens. Any depositor who stakes before the admin sets the variable will be unable to withdraw. Likelihood is **High**: the window between deployment and the admin's corrective call is an inherent deployment gap, and the contract is fully functional for staking during that window. [5](#0-4) 

## Recommendation
Initialize `maxNumberOfWithdrawalsPerUser` to `MAX_WITHDRAWALS_PER_USER` inside `initialize()`, mirroring the pattern used elsewhere in the codebase (e.g., `LRTWithdrawalManager.initialize()` sets `withdrawalDelayBlocks` at construction):

```solidity
function initialize(address _admin, address _kernelToken, address _rewardToken) external initializer {
    // ... existing setup ...
    maxNumberOfWithdrawalsPerUser = MAX_WITHDRAWALS_PER_USER;
}
``` [6](#0-5) 

## Proof of Concept
1. Deploy `KernelDepositPool` and call `initialize(admin, kernelToken, rewardToken)`.
2. Confirm `maxNumberOfWithdrawalsPerUser == 0` (never set).
3. User calls `stake(1e18)` — succeeds; `balanceOf[user] = 1e18`.
4. User calls `initiateWithdrawal(1e18)`.
5. Check at line 323: `userWithdrawalIds[user].length (= 0) >= maxNumberOfWithdrawalsPerUser (= 0)` → `true`.
6. Transaction reverts with `WithdrawalLimitReached`.
7. User's `1e18` KERNEL tokens remain locked; `claimWithdrawal` is unreachable since no withdrawal record was ever created.

    function initiateWithdrawal(uint256 _amount) external nonReentrant updateReward(msg.sender) {
        if (_amount == 0) revert AmountZero();
        if (balanceOf[msg.sender] < _amount) revert InsufficientStakedBalance();
        if (userWithdrawalIds[msg.sender].length >= maxNumberOfWithdrawalsPerUser) revert WithdrawalLimitReached();

        balanceOf[msg.sender] -= _amount;
        totalKernelStaked -= _amount;

        // Create a withdrawal record
        uint256 withdrawalId = ++withdrawalCounter;
        uint256 unlockTime = block.timestamp + withdrawalDelay;

            user: msg.sender, amount: _amount, unlockTime: unlockTime, claimed: false, withdrawalId: withdrawalId
        });
        userWithdrawalIds[msg.sender].push(withdrawalId);

        emit WithdrawalInitiated(msg.sender, _amount, withdrawalId, unlockTime);
    }
