Audit Report

## Title
Cross-Token Arbitrage Drains Wrapper of Higher-Value rsETH Variants via Unchecked 1:1 Swap - (File: contracts/L2/RsETHTokenWrapper.sol)

## Summary
`RsETHTokenWrapper` supports multiple allowed tokens and mints/burns `wrsETH` at a strict 1:1 ratio for any of them, but performs no value-equivalence check between the deposited token and the withdrawn token. Any unprivileged user can deposit a discounted rsETH variant, receive `wrsETH`, and immediately redeem that `wrsETH` for a higher-value rsETH variant held in the wrapper, extracting the price spread at the direct expense of honest depositors.

## Finding Description
The contract maintains a mapping `allowedTokens` and registers multiple tokens via `initialize`, `reinitialize`, and `addAllowedToken`. Both `_deposit` and `_withdraw` only check `allowedTokens[_asset]` before performing a strict 1:1 mint or burn:

`_deposit` (L134ã141): transfers `_amount` of any allowed token in, mints `_amount` of `wrsETH`.  
`_withdraw` (L120ã128): burns `_amount` of `wrsETH`, transfers `_amount` of any allowed token out.

There is no constraint linking which token was deposited to which token may be withdrawn. The public entry points `deposit` and `withdraw` (L69ã71, L84ã86) carry no access control. An attacker therefore executes two sequential public calls ã `deposit(cheap_token, N)` followed by `withdraw(expensive_token, N)` ã to atomically swap N units of a discounted variant for N units of a premium variant, with the wrapper absorbing the loss.

The broken assumption is the contract comment at L19: *"the alt rsETH tokens can be swapped 1:1 for the canonical rsETH token."* This is a protocol-level invariant that the contract never enforces on-chain; it is trivially violated whenever secondary-market prices diverge.

## Impact Explanation
**Critical ã Direct theft of user funds.**  
Honest users who deposited the higher-value rsETH variant into the wrapper now hold `wrsETH` claims that are backed only by the lower-value variant left behind by the attacker. The loss is concrete, immediate, and proportional to the price spread times the attack volume. This maps exactly to the allowed impact: *"Direct theft of any user funds, whether at-rest or in-motion."*

## Likelihood Explanation
Multi-token support is the explicit, intended design: `reinitialize` adds a second token and `addAllowedToken` (TIMELOCK_ROLE) can add further tokens. Bridged rsETH variants routinely trade at small discounts (0.1ã2%) on secondary markets due to bridge-specific risk premiums, withdrawal delays, and liquidity depth differences. No privileged role, flash loan, or oracle manipulation is required. Any holder of the discounted variant can execute the attack in two transactions (or one, via a contract). The attack is repeatable until the wrapper's supply of the premium token is exhausted.

## Recommendation
1. **Track per-token balances against per-token minted supply**: Record how much `wrsETH` was minted against each allowed token and enforce that withdrawals of token X cannot exceed the amount of token X deposited, preventing cross-token redemption entirely.
2. **Enforce single-token semantics at a time**: If only one canonical token should be active, gate `deposit`/`withdraw` to a single designated token and require governance action (with a timelock) to rotate it.
3. **Add value-equivalence checks**: If multi-token support must remain, integrate an on-chain price feed (e.g., Chainlink or a TWAP) and revert if the deposited and withdrawn tokens differ in value by more than an acceptable tolerance.

## Proof of Concept
**Setup:** Deploy `RsETHTokenWrapper` with `rsETH_v1` and `rsETH_v2` both in `allowedTokens`. Seed the wrapper with 1000e18 `rsETH_v2` deposited by honest users (wrapper holds 1000e18 `rsETH_v2`, 1000e18 `wrsETH` in circulation). Secondary market: `rsETH_v1` = 0.97 ETH, `rsETH_v2` = 1.00 ETH.

**Attack sequence (Foundry fork test):**
```solidity
// 1. Attacker acquires 1000e18 rsETH_v1 for ~970 ETH on secondary market
rsETH_v1.approve(address(wrapper), 1000e18);

// 2. Deposit discounted token ã mint wrsETH 1:1
wrapper.deposit(address(rsETH_v1), 1000e18);
// wrapper now holds: 1000e18 rsETH_v1 + 1000e18 rsETH_v2; wrsETH supply = 2000e18

// 3. Withdraw premium token ã burn wrsETH 1:1
wrapper.withdraw(address(rsETH_v2), 1000e18);
// wrapper now holds: 1000e18 rsETH_v1 + 0 rsETH_v2; wrsETH supply = 1000e18

// 4. Attacker sells 1000e18 rsETH_v2 for ~1000 ETH; net profit ~30 ETH
// Honest depositors' 1000e18 wrsETH is now backed only by rsETH_v1 worth ~970 ETH
```

**Invariant to fuzz:** `sum over all allowed tokens of (wrapper.balanceOf(token_i) * price_i) >= wrapper.totalSupply() * price_canonical` ã this invariant is broken after the attack.
