### Title
`response.payload_hash` Not Bound to `request` in `respond_verify_foreign_tx` — Forged Foreign-Chain Attestation by a Single Malicious Participant - (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_verify_foreign_tx` verifies that `response.signature` is a valid MPC signature over `response.payload_hash`, but it **never checks that `payload_hash` is actually derived from the `request` parameter**. A single malicious attested participant (below threshold) can replay any previously-produced valid `(payload_hash, signature)` pair — including one encoding different `extracted_values` for the same foreign-chain request — and the contract will accept it, delivering forged attestation data to the waiting user.

---

### Finding Description

The `respond_verify_foreign_tx` entry point in `crates/contract/src/lib.rs` accepts two independent parameters:

- `request: VerifyForeignTransactionRequest` — the original user request (used only as a lookup key into `pending_verify_foreign_tx_requests`)
- `response: VerifyForeignTransactionResponse` — contains `payload_hash: Hash256` and `signature: SignatureResponse`

The only cryptographic check performed is:

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,
)
.is_ok()
``` [1](#0-0) 

The contract confirms the signature is valid over `payload_hash`, then immediately resolves the yield with the full `response`:

```rust
pending_requests::resolve_yields_for(
    &mut self.pending_verify_foreign_tx_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
``` [2](#0-1) 

**What is never checked**: that `payload_hash` equals `SHA-256(borsh(ForeignTxSignPayload { request, extracted_values }))` for the specific `request` being resolved. The `payload_hash` is supposed to commit to both the original `ForeignChainRpcRequest` and the `ExtractedValue`s the MPC nodes observed on the foreign chain:

```rust
pub struct ForeignTxSignPayloadV1 {
    pub request: ForeignChainRpcRequest,
    pub values: Vec<ExtractedValue>,
}
// msg_hash = SHA-256(borsh(ForeignTxSignPayload))
``` [3](#0-2) 

Because the contract never enforces this derivation, `request` and `response.payload_hash` are completely decoupled at the contract level.

The analog to LID-5 is exact: just as `depositCalldata` was passed alongside guardian signatures but not included in the signed message, here `request` is passed alongside the MPC signature but is never bound to `payload_hash`.

---

### Impact Explanation

A single malicious attested participant can:

1. Observe a pending `verify_foreign_transaction(request_A)` in the contract.
2. Locate any previously produced valid `(payload_hash_old, sig_old)` pair where `payload_hash_old` encodes `request_A` but **different `extracted_values`** (e.g., from an earlier signing round when the foreign chain was in a different state, or from a reorganized block).
3. Call `respond_verify_foreign_tx(request_A, { payload_hash_old, sig_old })`.
4. The contract accepts the call (signature is valid over `payload_hash_old`), resolves the yield, and delivers `{ payload_hash_old, sig_old }` to the user.

The user's application receives a threshold-signed attestation that claims specific `extracted_values` were observed for `request_A`, but those values are stale or fabricated. Any bridge or smart contract that trusts this attestation (e.g., to release funds, confirm a deposit, or verify a cross-chain event) acts on incorrect foreign-chain state.

This matches the **High** impact category: *forged foreign-chain verification that causes invalid bridge execution*.

---

### Likelihood Explanation

- **Attacker role**: A single attested MPC participant — no threshold collusion required.
- **Prerequisite**: The attacker must have been a participant in at least one prior signing session for the same `ForeignChainRpcRequest`, so they possess a valid `(payload_hash, signature)` pair encoding that request with different extracted values. This is realistic whenever the foreign chain state changes between two signing rounds (block reorganization, confirmation count change, log index shift).
- **Entry path**: The attacker calls `respond_verify_foreign_tx` directly on-chain. No special privileges beyond being an attested participant are needed.
- **Detection**: The user's SDK (`ForeignChainSignatureVerifier::verify_signature`) does check `expected_payload_hash == response.payload_hash`, but only if the caller knows the expected `extracted_values` in advance — which they typically do not, since those values are the *output* of the verification. [4](#0-3) 

---

### Recommendation

Inside `respond_verify_foreign_tx`, after verifying the signature, recompute the expected `payload_hash` from the `request` and the `extracted_values` embedded in the signed payload, and assert equality:

```rust
// Derive the canonical payload hash from the request and verify it matches
// what the node claims to have signed.
let expected_payload_hash = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: request.request.clone(),
    values: response.extracted_values.clone(), // extracted_values must be added to the response DTO
}).compute_msg_hash()?;

if expected_payload_hash != response.payload_hash {
    return Err(RespondError::PayloadHashMismatch.into());
}
```

Alternatively, include the `extracted_values` in the `VerifyForeignTransactionResponse` DTO and have the contract recompute and verify `payload_hash` from `(request, extracted_values)` before accepting the response. This binds the signed hash to the specific request being resolved, closing the substitution gap.

### Proof of Concept

**Setup**: Two-node cluster. Node A is honest; Node B is malicious.

1. User calls `verify_foreign_transaction({ request: BitcoinRpcRequest { tx_id: X, extractors: [BlockHash] }, domain_id: D, payload_version: V1 })`.
2. In round 1, the MPC network signs `payload_hash_1 = SHA-256(borsh({ request: BitcoinRpcRequest{X}, values: [BlockHash([0xAA;32])] }))`, producing `sig_1`. Node B saves `(payload_hash_1, sig_1)`.
3. The user submits the same request again (or a second user submits it). In round 2, the honest network would sign `payload_hash_2` with `values: [BlockHash([0xBB;32])]` (block reorganized).
4. Before the honest node can call `respond_verify_foreign_tx`, Node B front-runs with:
   ```
   respond_verify_foreign_tx(
     request = { BitcoinRpcRequest{X}, D, V1 },
     response = { payload_hash: payload_hash_1, signature: sig_1 }
   )
   ```
5. The contract checks: `verify_ecdsa_signature(sig_1, payload_hash_1, root_pk)` → **valid**. Resolves the yield.
6. The user receives `{ payload_hash_1, sig_1 }` — a valid MPC-signed attestation claiming block hash `[0xAA;32]`, when the actual current block hash is `[0xBB;32]`.

The contract never checks that `payload_hash_1` is derived from the `request` parameter passed in step 4. [5](#0-4) [6](#0-5)

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }
