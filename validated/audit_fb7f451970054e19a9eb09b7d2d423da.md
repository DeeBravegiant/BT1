Audit Report

## Title
Permissionless `updateRate` with No Staleness Guard Enables Block-Stuffing to Exploit Stale Cross-Chain rsETH Rate — (`contracts/cross-chain/RSETHMultiChainRateProvider.sol`)

## Summary

`MultiChainRateProvider.updateRate()` is a permissionless `external payable` function with no access control. `CrossChainRateReceiver.getRate()` returns the stored rate unconditionally with no maximum-age check. L2 pools (`RSETHPoolNoWrapper`, `RSETHPoolV3ExternalBridge`) price every deposit directly against this stored rate. An attacker who stuffs Ethereum mainnet blocks during an `rsETHPrice` increase window can prevent the rate from propagating to L2, then deposit ETH on L2 at the artificially favorable stale rate, extracting excess rsETH from the pool's inventory — theft of unclaimed yield belonging to existing rsETH holders.

## Finding Description

**Rate propagation path (L1 → L2):**

`RSETHMultiChainRateProvider.getLatestRate()` reads `ILRTOracle.rsETHPrice()` live from L1 at `contracts/cross-chain/RSETHMultiChainRateProvider.sol` lines 26–28.

`MultiChainRateProvider.updateRate()` at line 108 of `contracts/cross-chain/MultiChainRateProvider.sol` is declared `external payable nonReentrant` with no role check or caller whitelist — any address can call it, and equally, any address can be blocked from calling it by block stuffing.

**Rate storage on L2 (no expiry):**

`CrossChainRateReceiver.lzReceive()` writes the received rate to `rate` and `lastUpdated` (lines 95–97 of `contracts/cross-chain/CrossChainRateReceiver.sol`). `getRate()` at lines 102–105 returns `rate` unconditionally — `lastUpdated` is stored but never read by any pricing function. There is no maximum-age check anywhere in the receiver or the pools.

**Pool pricing (no staleness guard):**

`RSETHPoolNoWrapper.deposit(string)` at line 237 calls `viewSwapRsETHAmountAndFee(amount)`, which at line 282 calls `getRate()` and divides by it, then at line 241 executes `rsETH.safeTransfer(msg.sender, rsETHAmount)` — transferring pre-loaded rsETH inventory directly to the depositor.

`RSETHPoolV3ExternalBridge.deposit(string)` at line 377 follows the identical pattern, calling `viewSwapRsETHAmountAndFee(amount)` at line 423 via `getRate()`, then minting `wrsETH` at line 381.

**Exploit flow:**

1. Attacker monitors L1 mempool for the `LRTOracle` transaction that increases `rsETHPrice` from `R_old` to `R_new`.
2. Immediately after the oracle update lands, attacker begins stuffing consecutive Ethereum mainnet blocks with high-gas-price filler transactions, preventing any `updateRate()` call from being included.
3. L2 `CrossChainRateReceiver` continues to hold `R_old`.
4. Attacker deposits ETH on L2. Pool computes `rsETHAmount = amountAfterFee * 1e18 / R_old`, which is greater than the correct `amountAfterFee * 1e18 / R_new`.
5. Excess rsETH (`delta = amountAfterFee * 1e18 / R_old − amountAfterFee * 1e18 / R_new`) is transferred from the pool's rsETH inventory to the attacker.
6. Attacker stops stuffing; `updateRate()` is called; rate normalizes. The pool's inventory is permanently short by the extracted delta.

**Why existing checks fail:** `nonReentrant` on `updateRate()` prevents re-entrancy but not exclusion. The `lzReceive` source-address and chain-id checks on the receiver are correct but irrelevant — the attack prevents the message from ever being sent. No circuit-breaker, no staleness revert, and no rate-age validation exists anywhere in the pricing path.

## Impact Explanation

The excess rsETH extracted per deposit is funded by the pool's pre-loaded rsETH inventory. That inventory represents accrued yield belonging to existing rsETH holders that has not yet been claimed or redistributed. Draining it constitutes **High. Theft of unclaimed yield**. The attack vector is **Low. Block stuffing** — the attacker must prevent all `updateRate()` callers from landing on Ethereum mainnet, which is expensive but does not require any privileged access, governance capture, or key compromise.

## Likelihood Explanation

`updateRate()` is permissionless, so the attacker only needs to outbid all legitimate callers for block space during the attack window. The window is bounded: it begins when `rsETHPrice` increases on L1 and ends when any caller successfully lands `updateRate()`. Profit scales with pool liquidity and the magnitude of the price jump; a large EigenLayer reward accrual event could make the economics favorable. The attack is repeatable across every reward cycle. No privileged access is required — only ETH for gas and block stuffing.

## Recommendation

1. **Add a staleness guard in `CrossChainRateReceiver.getRate()`**: revert if `block.timestamp - lastUpdated > MAX_RATE_AGE`. This prevents pools from pricing swaps against an arbitrarily old rate regardless of the cause of the delay.

2. **Add the same guard in the L2 pools**: `RSETHPoolNoWrapper` and `RSETHPoolV3ExternalBridge` should independently validate rate freshness before executing a deposit, so a lagging oracle cannot silently drain the pool.

3. **Consider a keeper with EIP-1559 tip floor**: ensure `updateRate()` is called with a competitive `maxPriorityFeePerGas` so that block stuffing becomes prohibitively expensive relative to any realistic arbitrage profit.

## Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

// Foundry fork test (fork of Ethereum mainnet for L1 state; L2 state simulated locally)
function testBlockStuffingStaleRate() public {
    uint256 oldRate = 1.05e18;  // rate stored in L2 CrossChainRateReceiver before reward accrual
    uint256 newRate = 1.08e18;  // rate now live on L1 LRTOracle after reward accrual

    // Step 1: Simulate oracle update on L1 (reward accrual)
    vm.store(address(lrtOracle), rsETHPriceSlot, bytes32(newRate));

    // Step 2: Simulate block stuffing — advance time/blocks without calling updateRate()
    vm.warp(block.timestamp + 1 hours);
    vm.roll(block.number + 300);

    // Step 3: L2 receiver still holds oldRate (updateRate was never called)
    assertEq(rateReceiver.getRate(), oldRate);

    // Step 4: Deposit on L2 pool at stale rate
    (uint256 rsETHAmount,) = pool.viewSwapRsETHAmountAndFee(1 ether);

    // Step 5: Assert excess rsETH received vs. correct amount at new rate
    uint256 correctAmount = (1 ether - fee) * 1e18 / newRate;
    assertGt(rsETHAmount, correctAmount, "stale rate yields excess rsETH");
    // Excess per ETH ≈ (1/1.05 - 1/1.08) * 1e18 ≈ 0.026e18 rsETH
    // This excess is transferred from pool inventory — theft of unclaimed yield
}
```

The test requires no privileged access: it only simulates a normal reward accrual (oracle storage write) and withholds `updateRate()` (simulating block stuffing). The assertion passes on unmodified code whenever `newRate > oldRate` and the L2 receiver has not been updated.
