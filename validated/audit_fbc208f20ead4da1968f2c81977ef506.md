Audit Report

## Title
Cross-Token Drain via Global-Only Accounting Permanently Freezes Deposited Collateral — (`contracts/L2/RsETHTokenWrapper.sol`)

## Summary

`RsETHTokenWrapper` maintains a single global `totalSupply` counter for wrsETH but accepts deposits and withdrawals across multiple independent allowed tokens. Because `maxAmountToDepositBridgerAsset` computes available capacity as `totalSupply - thisToken.balance` without accounting for other tokens already held as collateral, the bridger can deposit tokenB against wrsETH already fully backed by tokenA. A user can then burn all outstanding wrsETH by withdrawing tokenB, permanently locking tokenA with no remaining wrsETH to redeem it.

## Finding Description

`_deposit` mints wrsETH 1:1 against any allowed token: [1](#0-0) 

`_withdraw` burns wrsETH 1:1 and transfers any allowed token, with no per-token accounting: [2](#0-1) 

`maxAmountToDepositBridgerAsset` computes capacity using only the queried token's balance against the global supply: [3](#0-2) 

`depositBridgerAssets` does **not** mint wrsETH — it only transfers tokens in, relying entirely on `maxAmountToDepositBridgerAsset` as its guard: [4](#0-3) 

**Exploit path:**

| Step | Call | State |
|------|------|-------|
| 1 | `deposit(tokenA, N)` | `totalSupply=N`, `tokenA.bal=N`, `tokenB.bal=0` |
| 2 | `depositBridgerAssets(tokenB, N)` | `maxAmount = N − 0 = N` → passes; `totalSupply=N`, `tokenA.bal=N`, `tokenB.bal=N` |
| 3 | `withdraw(tokenB, N)` | burns N wrsETH; `totalSupply=0`, `tokenA.bal=N`, `tokenB.bal=0` |
| 4 | **tokenA permanently locked** | no wrsETH exists to redeem it |

The guard in step 2 fails because it subtracts only `tokenB.balance` (0) from `totalSupply` (N), ignoring that tokenA already fully backs the outstanding supply. There is no admin rescue path, no emergency withdrawal, and no way to mint wrsETH without depositing new collateral. [5](#0-4) 

## Impact Explanation

**Critical — Permanent freezing of funds.** The N tokenA units deposited by the user in step 1 become irrecoverable: `totalSupply = 0` means no wrsETH exists to burn in exchange for tokenA, and the contract exposes no rescue mechanism. The loss is permanent and proportional to the amount deposited.

## Likelihood Explanation

Two conditions are required, both explicitly supported by the contract design:

1. More than one token is added via `addAllowedToken` — the `reinitialize` path and `addAllowedToken` function exist precisely for this purpose. [6](#0-5) 
2. The bridger performs its normal duty of depositing bridged tokens to collateralize outstanding wrsETH via `depositBridgerAssets`.

Neither actor is malicious. The bridger observes outstanding wrsETH and deposits the corresponding bridged token — correct behavior per its role. The user calls a public function with a valid allowed token. The flaw is purely in the accounting model. This scenario arises in normal multi-token operation without any adversarial coordination.

## Recommendation

Replace the global `totalSupply` accounting with per-token deposit tracking:

```solidity
mapping(address => uint256) public depositedPerToken;

function _deposit(address _asset, address _to, uint256 _amount) internal {
    if (!allowedTokens[_asset]) revert TokenNotAllowed();
    ERC20Upgradeable(_asset).safeTransferFrom(msg.sender, address(this), _amount);
    depositedPerToken[_asset] += _amount;
    _mint(_to, _amount);
    emit Deposit(_asset, msg.sender, _to, _amount);
}

function _withdraw(address _asset, address _to, uint256 _amount) internal {
    if (!allowedTokens[_asset]) revert TokenNotAllowed();
    require(depositedPerToken[_asset] >= _amount, "insufficient per-token balance");
    depositedPerToken[_asset] -= _amount;
    _burn(msg.sender, _amount);
    ERC20Upgradeable(_asset).safeTransfer(_to, _amount);
    emit Withdraw(_asset, msg.sender, _to, _amount);
}
```

`depositBridgerAssets` should credit `depositedPerToken[_asset]` and `maxAmountToDepositBridgerAsset` should compare against `depositedPerToken[_asset]` rather than the raw ERC-20 balance, ensuring each token's collateral is tracked independently.

## Proof of Concept

The following Foundry test reproduces the broken invariant:

```solidity
// SPDX-License-Identifier: UNLICENSED
pragma solidity ^0.8.0;

import "forge-std/Test.sol";
import "../contracts/L2/RsETHTokenWrapper.sol";
import "@openzeppelin/contracts/token/ERC20/ERC20.sol";

contract MockToken is ERC20 {
    constructor(string memory n) ERC20(n, n) { _mint(msg.sender, 1e24); }
}

contract CrossTokenDrainTest is Test {
    RsETHTokenWrapper wrapper;
    MockToken tokenA;
    MockToken tokenB;
    address admin   = address(1);
    address bridger = address(2);
    address user    = address(3);

    function setUp() public {
        tokenA = new MockToken("A");
        tokenB = new MockToken("B");
        wrapper = new RsETHTokenWrapper();
        wrapper.initialize(admin, bridger, address(tokenA));
        vm.prank(admin);
        wrapper.grantRole(wrapper.TIMELOCK_ROLE(), admin);
        vm.prank(admin);
        wrapper.addAllowedToken(address(tokenB));
        tokenA.transfer(user,    1000e18);
        tokenB.transfer(bridger, 1000e18);
    }

    function test_crossTokenDrain() public {
        uint256 N = 100e18;
        // Step 1: user deposits tokenA
        vm.startPrank(user);
        tokenA.approve(address(wrapper), N);
        wrapper.deposit(address(tokenA), N);
        vm.stopPrank();
        // Step 2: bridger deposits tokenB (maxAmount = N - 0 = N, passes)
        vm.startPrank(bridger);
        tokenB.approve(address(wrapper), N);
        wrapper.depositBridgerAssets(address(tokenB), N);
        vm.stopPrank();
        // Step 3: user withdraws tokenB, burning all wrsETH
        vm.prank(user);
        wrapper.withdraw(address(tokenB), N);
        // Invariant: total collateral must be <= totalSupply
        uint256 totalBal = tokenA.balanceOf(address(wrapper))
                         + tokenB.balanceOf(address(wrapper));
        uint256 supply   = wrapper.totalSupply();
        // FAILS: totalBal = N, supply = 0 — tokenA permanently locked
        assertLe(totalBal, supply, "INVARIANT BROKEN: tokens stranded with no wrsETH");
    }
}
```

Running `forge test` will fail at the final assertion, confirming N tokenA is permanently locked with `totalSupply = 0`.
