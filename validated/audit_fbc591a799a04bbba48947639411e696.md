Audit Report

## Title
Stale `rsETHPrice` Used in Deposit Mint Calculation Without Prior Oracle Update — (File: `contracts/LRTDepositPool.sol`)

## Summary
`LRTDepositPool.getRsETHAmountToMint()` divides by the cached `LRTOracle.rsETHPrice` state variable without first invoking `updateRSETHPrice()`. Because `rsETHPrice` only updates on explicit invocation, any depositor who transacts while the price lags behind the true TVL-derived value receives more rsETH than their deposit warrants, directly diluting the accrued yield of all existing rsETH holders.

## Finding Description
`LRTOracle` stores the rsETH/ETH exchange rate in the state variable `rsETHPrice` (LRTOracle.sol L28). This value is not computed on-the-fly; it is only refreshed when `_updateRsETHPrice()` is executed via the public `updateRSETHPrice()` (L87–89) or the manager-gated `updateRSETHPriceAsManager()` (L94–96).

`LRTDepositPool.getRsETHAmountToMint()` (LRTDepositPool.sol L506–521) reads this cached value directly:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

The call chain `depositAsset()` → `_beforeDeposit()` → `getRsETHAmountToMint()` (LRTDepositPool.sol L111, L665, L520) contains no call to `updateRSETHPrice()` and no staleness check on `rsETHPrice`. As EigenLayer staking rewards accrue to the protocol TVL, the true rsETH price rises while `rsETHPrice` remains at its last-written value. A lower denominator inflates `rsethAmountToMint`, issuing the depositor more rsETH than their deposit backs.

No existing guard prevents this: `_beforeDeposit()` only checks deposit amount, deposit cap, and the depositor's own `minRSETHAmountExpected` slippage floor (LRTDepositPool.sol L657–669) — none of which protect existing holders from dilution.

## Impact Explanation
`rsETHPrice` monotonically increases as staking rewards accrue. When a deposit is processed against a stale (lower) `rsETHPrice`, the minted rsETH exceeds the depositor's fair share. Because rsETH value equals `totalETH / totalSupply`, the excess supply reduces the ETH-per-rsETH ratio for all existing holders. The yield those holders had accrued since the last price update is effectively transferred to the new depositor.

**Impact: High — Theft of unclaimed yield from existing rsETH holders.** This matches the allowed impact scope exactly.

## Likelihood Explanation
`updateRSETHPrice()` is public and permissionless, so the protocol relies entirely on off-chain keepers for freshness. No on-chain staleness window is enforced before minting. An attacker requires no special role: they need only monitor the gap between the live TVL (computable off-chain from `getTotalAssetDeposits` and asset prices) and the stored `rsETHPrice`, then submit a large deposit during any window where the keeper has not yet updated the price. The attack is passive, repeatable, and scales linearly with deposit size and price lag.

**Likelihood: Medium** — no privilege required; only timing awareness and keeper downtime or deliberate delay.

## Recommendation
Call `updateRSETHPrice()` (or an internal equivalent that does not revert on the price-increase threshold when called by the deposit pool) at the start of `depositAsset()` and `depositETH()`, before `_beforeDeposit()` computes the mint amount. Alternatively, record a `rsETHPriceLastUpdated` timestamp in `LRTOracle` and revert in `getRsETHAmountToMint()` if `block.timestamp - rsETHPriceLastUpdated` exceeds a configurable maximum staleness window (e.g., 1 hour).

## Proof of Concept
1. At time T, `rsETHPrice = 1.01e18` (last updated by keeper).
2. Staking rewards accrue; true rsETH price rises to `1.02e18`, but `updateRSETHPrice()` has not been called (keeper lag or deliberate timing).
3. Attacker calls `LRTDepositPool.depositAsset(stETH, 100e18, 0, "")`.
4. `getRsETHAmountToMint` computes: `(100e18 × 1e18) / 1.01e18 ≈ 99.0099 rsETH` instead of the correct `(100e18 × 1e18) / 1.02e18 ≈ 98.0392 rsETH`.
5. Attacker receives ≈ 0.97 excess rsETH per 100 ETH deposited.
6. When `updateRSETHPrice()` is next called, the new price is computed over the inflated supply, permanently reducing the ETH-per-rsETH ratio for all pre-existing holders.
7. Attacker redeems rsETH through the withdrawal pipeline, extracting the stolen yield.

**Foundry fork test plan:**
- Fork mainnet; snapshot `rsETHPrice` and total supply.
- Advance time without calling `updateRSETHPrice()` to simulate reward accrual (or directly manipulate a mock oracle to set a higher TVL).
- Call `depositAsset` as an unprivileged address.
- Assert `rsethAmountToMint > (depositValue / truePrice)`.
- Call `updateRSETHPrice()`; assert the resulting `rsETHPrice` is lower than it would have been had the correct mint amount been issued.
