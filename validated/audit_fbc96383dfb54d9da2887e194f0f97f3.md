### Title
Unvalidated `builder` Field in `ProposalInit` Allows Malicious Proposer to Redirect Sequencer Fees and Corrupt Block Execution Context — (File: `crates/apollo_consensus_orchestrator/src/validate_proposal.rs`, `crates/apollo_consensus_orchestrator/src/utils.rs`)

### Summary

The `builder` field of `ProposalInit` is accepted from the network and passed directly into `BlockInfo.sequencer_address` without any validation in `is_proposal_init_valid`. A Byzantine proposer can set `builder` to an arbitrary address. Every honest validator will execute the block with that attacker-controlled address as the sequencer, redirecting all transaction fees to the attacker and exposing a wrong `sequencer_address` to contracts via the `get_block_info` syscall. The corrupted value is baked into the `partial_block_hash` and therefore into the final `ProposalCommitment` that consensus signs over.

### Finding Description

`ProposalInit` carries a `builder` field (the sequencer/block-builder address): [1](#0-0) 

During proposal building the proposer sets it from its own static config: [2](#0-1) 

During validation, `is_proposal_init_valid` checks `height`, `timestamp`, `starknet_version`, `version_constant_commitment`, `l1_da_mode`, `l2_gas_price_fri`, all four L1 gas prices, and `fee_proposal_fri`. It does **not** check `builder`: [3](#0-2) 

Immediately after `is_proposal_init_valid` returns `Ok`, `initiate_validation` calls `convert_to_sn_api_block_info(init)`, which maps `init.builder` directly to `BlockInfo.sequencer_address`: [4](#0-3) 

This `BlockInfo` is forwarded to the batcher for full block execution: [5](#0-4) 

The batcher computes `partial_block_hash` over the full block header (which includes `sequencer_address`). The orchestrator then combines it with `fee_proposal_fri` to produce the `ProposalCommitment`: [6](#0-5) 

Because every honest validator uses the proposer-supplied `builder` value, the `ProposalFinMismatch` check at line 244 passes, and the block is committed with the attacker-controlled `sequencer_address` baked into the block hash. [7](#0-6) 

### Impact Explanation

**Fee redirection (Critical — economic impact):** In Starknet, transaction fees are transferred from the user's account to `BlockInfo.sequencer_address` during execution. With an attacker-controlled `builder`, all fees for the block are credited to the attacker's address in the state diff rather than the legitimate sequencer.

**Wrong syscall result (Critical — wrong execution context):** Contracts calling `get_block_info()` receive the attacker-supplied `sequencer_address`. Any contract with sequencer-gated logic (e.g., fee-related contracts, governance) will observe the wrong value.

**Corrupted block commitment:** The `partial_block_hash` — and therefore the `ProposalCommitment` that consensus signs — encodes the attacker's address. The block is permanently committed to L1 with a wrong sequencer identity.

### Likelihood Explanation

Any Byzantine validator that wins a proposer slot can trigger this. No special privilege beyond being a consensus participant is required. The attack is silent: honest validators accept the proposal because `builder` is never checked.

### Recommendation

Add a check in `is_proposal_init_valid` that `init_proposed.builder` equals the expected builder address for the current proposer. At minimum, the validator should verify that `builder` matches the proposer's registered address from the committee, or that it matches the locally configured `builder_address`. This mirrors how `l2_gas_price_fri`, `l1_da_mode`, and `starknet_version` are validated against locally-derived reference values.

### Proof of Concept

1. Byzantine validator wins proposer slot for height H.
2. It constructs `ProposalInit { builder: attacker_address, ... }` with all other fields set legitimately.
3. It streams this to honest validators.
4. Honest validators call `is_proposal_init_valid` — no check on `builder`, returns `Ok`.
5. `initiate_validation` → `convert_to_sn_api_block_info` → `BlockInfo { sequencer_address: attacker_address, ... }`.
6. Batcher executes all transactions; fee transfers credit `attacker_address`.
7. `partial_block_hash` encodes `attacker_address`; `proposal_commitment_from(partial, fee_proposal)` produces a commitment over the attacker's address.
8. `ProposalFinMismatch` check passes (both proposer and validator computed the same commitment using the same `builder`).
9. Consensus reaches quorum; block is committed. All fees for height H are in `attacker_address`'s balance. `get_block_info().sequencer_address` returns `attacker_address` for every transaction in that block.
