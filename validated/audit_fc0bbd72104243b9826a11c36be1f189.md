### Title
Detached Refund Transfers Silently Fail, Permanently Locking Excess Deposits in the Contract - (File: `crates/contract/src/lib.rs`)

### Summary
The MPC contract schedules NEAR token refund transfers using `.detach()`, which fires the promise but discards its result. If a refund transfer fails (e.g., the recipient account no longer exists at promise execution time), the failure is silent: no error is surfaced, no retry is attempted, and the excess deposit is permanently retained by the contract. This is the structural NEAR analog to unchecked ERC20 `transfer`/`transferFrom` return values.

### Finding Description

Three distinct call sites in `crates/contract/src/lib.rs` schedule NEAR token refunds via `Promise::new(...).transfer(...).detach()`:

**1. `require_deposit` — excess deposit refund to any caller:** [1](#0-0) 

**2. `submit_participant_info` — excess deposit refund to the attestation submitter:** [2](#0-1) 

**3. `propose_update` — excess deposit refund to the update proposer:** [3](#0-2) 

In all three cases, `.detach()` explicitly drops the promise result. NEAR's runtime executes the transfer in a subsequent receipt; if that receipt fails (recipient account deleted, or any other runtime rejection), the failure is invisible to the contract. There is no callback, no error log, and no recovery path. The excess NEAR remains locked in the contract balance with no mechanism to reclaim it.

The vulnerability class is identical to M-01: a value-transfer operation whose failure mode is silently swallowed rather than handled.

### Impact Explanation

A smart-contract caller (e.g., a DeFi integration or a composable NEAR contract) can:
1. Call `sign()`, `submit_participant_info()`, or `propose_update()` with a deposit significantly larger than the minimum required.
2. Self-destruct (delete) its account before the refund promise executes in the next block.
3. The refund transfer fails silently. The excess deposit is permanently locked in the MPC contract.

The MPC contract's NEAR balance grows beyond what it legitimately holds, breaking the accounting invariant that the contract balance equals the sum of all pending request deposits plus operational reserves. Funds are permanently frozen with no administrative recovery path.

This maps to the **Medium** allowed impact: *"Balance, request-lifecycle, participant-state, or contract execution-flow manipulation that breaks production safety/accounting invariants without relying on network-level DoS or operator misconfiguration."*

### Likelihood Explanation

The attacker must be a smart contract (not a human EOA, since human accounts are rarely deleted). The window between the original call and the promise execution is one block (~1 second on NEAR). Account deletion in that window requires a deliberate, coordinated action. Likelihood is **low** but the entry path is fully attacker-controlled and requires no privileged access — any unprivileged contract caller can trigger it.

### Recommendation

Replace `.detach()` refund transfers with a checked callback pattern. Schedule the transfer as a returned promise with a `.then` callback that logs or handles failure, or verify the recipient account exists before scheduling the transfer. At minimum, replace:

```rust
Promise::new(predecessor.clone()).transfer(diff).detach();
```

with a pattern that chains a failure handler:

```rust
Promise::new(predecessor.clone())
    .transfer(diff)
    .then(Self::ext(env::current_account_id()).on_refund_complete(predecessor.clone(), diff));
```

where `on_refund_complete` logs the failure and, if needed, re-credits the amount to a recoverable escrow.

### Proof of Concept

```
1. Deploy a NEAR smart contract `attacker.near`.
2. From `attacker.near`, call `mpc_contract.sign(request)` attaching 10 NEAR
   (minimum is 1 yoctoNEAR; excess = ~10 NEAR).
3. In the same transaction batch (or immediately after), call
   `attacker.near.delete_account(beneficiary)`.
4. The MPC contract's `require_deposit` schedules:
     Promise::new("attacker.near").transfer(9_999_999_999_999_999_999_999_999).detach()
5. By the time the refund receipt executes, "attacker.near" no longer exists.
6. The transfer receipt fails. `.detach()` means no callback fires.
7. The MPC contract retains the ~10 NEAR permanently.
8. Repeat across multiple accounts to accumulate locked funds.
```

fn require_deposit(minimum_deposit: NearToken, predecessor: &AccountId) {
    let deposit = env::attached_deposit();
    match deposit.checked_sub(minimum_deposit) {
        None => {
            env::panic_str(
                &InvalidParameters::InsufficientDeposit {
                    attached: deposit.as_yoctonear(),
                    required: minimum_deposit.as_yoctonear(),
                }
                .to_string(),
            );
        }
        Some(diff) => {
            if diff > NearToken::from_yoctonear(0) {
                log!("refund excess deposit {diff} to {predecessor}");
                Promise::new(predecessor.clone()).transfer(diff).detach();
            }
        }
    }
}
