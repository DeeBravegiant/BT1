### Title
Peer-Supplied Zero-Addition Spend Causes Permanent Clawback Coin State Discard — (`chia/wallet/wallet_state_manager.py`)

### Summary

In the resync branch of `handle_clawback`, the wallet calls `fetch_coin_spend_for_coin_state` against the connected peer and then immediately indexes `clawback_spend_bundle.additions()[0]` without a bounds check. A malicious peer can return a spend whose solution produces zero `CREATE_COIN` conditions, causing an `IndexError`. The outer `_add_coin_states` exception handler classifies this as a non-retryable error and calls `retry_store.remove_state(coin_state)`, permanently discarding the clawback coin state.

### Finding Description

**Vulnerable line:** [1](#0-0) 

```python
clawback_coin_spend: CoinSpend = await fetch_coin_spend_for_coin_state(coin_state, peer)
clawback_spend_bundle = WalletSpendBundle([clawback_coin_spend], G2Element())
if await self.puzzle_store.puzzle_hash_exists(clawback_spend_bundle.additions()[0].puzzle_hash):
```

`fetch_coin_spend_for_coin_state` calls `fetch_coin_spend`, which validates only that the returned puzzle's tree hash matches the coin's `puzzle_hash` and that the coin name matches: [2](#0-1) 

The **solution** is entirely peer-controlled and unvalidated. A malicious peer returns the correct puzzle (satisfying the hash check) paired with a crafted solution that, when executed by the CLVM, produces no `CREATE_COIN` conditions. `WalletSpendBundle.additions()` (inherited from `SpendBundle` in `chia_rs`) runs the puzzle and filters for `CREATE_COIN` opcodes — returning an empty list. Indexing `[0]` on an empty list raises `IndexError`.

**Exception handler — the permanent discard:** [3](#0-2) 

```python
except Exception as e:
    ...
    if isinstance(e, (PeerRequestException, aiosqlite.Error)):
        await self.retry_store.add_state(coin_state, peer.peer_node_id, fork_height)
    else:
        await self.retry_store.remove_state(coin_state)   # ← IndexError lands here
    continue
```

`IndexError` is neither a `PeerRequestException` nor an `aiosqlite.Error`, so the handler takes the `else` branch and **permanently removes** the clawback coin state from the retry store. The coin state will never be reprocessed automatically.

### Impact Explanation

The clawback coin state is silently dropped. The wallet never records the `WalletCoinRecord` for the clawback coin, never creates the `OUTGOING_CLAWBACK` transaction record, and never retries. The user loses the ability to recover clawback funds through normal wallet sync — a permanent, wallet-level denial of service for that specific coin.

This falls squarely under: **High — Permanent or long-lived inability for honest wallets to process valid sync updates under normal network assumptions.**

### Likelihood Explanation

The attacker only needs to operate a full node peer that the victim wallet connects to (or perform a man-in-the-middle on the `request_puzzle_solution` protocol message). No keys, admin access, or cryptographic breaks are required. The trigger condition (`coin_state.spent_height is not None`) is the normal resync path hit on every wallet restart after a clawback has been confirmed on-chain.

### Recommendation

Add a bounds check before indexing `additions()`:

```python
additions = clawback_spend_bundle.additions()
if not additions:
    self.log.warning("Clawback spend has no additions, skipping tx record creation")
else:
    if await self.puzzle_store.puzzle_hash_exists(additions[0].puzzle_hash):
        ...
```

Additionally, consider whether `IndexError` (and other data-integrity exceptions from peer-supplied content) should be treated as retryable (`PeerRequestException`-like) rather than causing permanent state removal.
