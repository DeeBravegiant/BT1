Looking at the exact code for `MetadataPayload::to_borsh`, `deploy_token`, `_verify_borsh_signature`, and the contrast with `TransferMessagePayload::to_borsh`.

### Title
Missing `chain_id` in `MetadataPayload::to_borsh` Enables Cross-Chain Replay of `deploy_token` Signatures — (`starknet/src/bridge_types.cairo`)

---

### Summary

`MetadataPayload::to_borsh` omits any chain-identifying field from the signed payload, while the analogous `TransferMessagePayload::to_borsh` explicitly encodes `chain_id` twice. Because the NEAR omni-bridge MPC key (`omni_bridge_derived_address`) is shared across all chain deployments by design, a valid MPC-signed `deploy_token` signature produced for one chain is cryptographically identical to a valid signature for any other chain using the same borsh encoding. An unprivileged attacker can capture a `(signature, MetadataPayload)` pair from chain A's `deploy_token` event and submit it verbatim to chain B's `deploy_token`, causing an unauthorized token deployment on chain B.

---

### Finding Description

**`MetadataPayload::to_borsh` — no chain binding** [1](#0-0) 

The borsh encoding contains only: `PayloadType::Metadata` (1 byte) + `token` + `name` + `symbol` + `decimals`. No `chain_id`, no destination contract address, no nonce.

**`TransferMessagePayload::to_borsh` — chain binding present** [2](#0-1) 

`fin_transfer` passes `self.omni_bridge_chain_id.read()` into `to_borsh`, which encodes it at two positions in the payload. This correctly binds the signature to a specific destination chain.

**`deploy_token` — no chain_id passed to borsh** [3](#0-2) 

`deploy_token` calls `payload.to_borsh()` with no arguments. The resulting byte string is identical regardless of which chain the OmniBridge contract is deployed on.

**`_verify_borsh_signature` — verifies against the shared MPC key** [4](#0-3) 

The verifier checks the signature against `self.omni_bridge_derived_address`, which is the same Ethereum-style MPC key shared across all chain deployments. There is no additional domain separator or chain-specific context injected here.

**Asymmetry is explicit**: `fin_transfer` passes `chain_id` into `to_borsh`; `deploy_token` does not. The storage field `omni_bridge_chain_id` exists and is set in the constructor, but is never used in the `deploy_token` path. [5](#0-4) [6](#0-5) 

---

### Impact Explanation

An attacker who observes a valid `deploy_token(signature, payload)` transaction on chain A can submit the identical call to chain B. The signature verifies successfully because the borsh-encoded message is byte-for-byte identical on both chains. The result:

- `near_to_starknet_token[keccak(token)]` is populated on chain B without MPC authorization for chain B.
- `starknet_to_near_token[contract_address]` is populated on chain B.
- A bridge token contract is deployed on chain B, making `is_bridge_token` return `true` for it.

This bypasses the MPC authorization gate for token deployment on chain B. The claim in the question that this enables replayed `fin_transfer` minting is **not correct** — `fin_transfer` is chain-bound via `chain_id` in its borsh payload and also has nonce replay protection. However, the unauthorized deployment itself is a concrete bypass of an execution gate, fitting the "High — acceptance of insufficiently-bound signatures that bypass execution gates" impact category.

---

### Likelihood Explanation

The precondition — shared MPC key across chains — is the **intended design** of the NEAR omni-bridge, not a compromise or misconfiguration. The MPC key is a single threshold key that authorizes operations across all supported chains; chain binding is supposed to come from the payload. Any chain that uses the same borsh `MetadataPayload` encoding (e.g., two StarkNet deployments, or a StarkNet mainnet and testnet) is immediately vulnerable. The attacker only needs to observe a public on-chain transaction.

---

### Recommendation

Pass `omni_bridge_chain_id` into `MetadataPayload::to_borsh` and encode it in the payload, mirroring the pattern already used in `TransferMessagePayload::to_borsh`:

```cairo
// bridge_types.cairo
fn to_borsh(self: @MetadataPayload, chain_id: u8) -> ByteArray {
    let mut borsh_bytes: ByteArray = Default::default();
    borsh_bytes.append_byte(PayloadType::Metadata.into());
    borsh_bytes.append_byte(chain_id);          // ← add this
    borsh_bytes.append(@borsh::encode_byte_array(self.token));
    borsh_bytes.append(@borsh::encode_byte_array(self.name));
    borsh_bytes.append(@borsh::encode_byte_array(self.symbol));
    borsh_bytes.append_byte(*self.decimals);
    borsh_bytes
}
```

And in `deploy_token`:

```cairo
_verify_borsh_signature(ref self, @payload.to_borsh(self.omni_bridge_chain_id.read()), signature);
```

The MPC signing service must also include `chain_id` in the payload it signs for `deploy_token`.

---

### Proof of Concept

1. On chain A (StarkNet mainnet), observe a successful `deploy_token` transaction. Extract `(signature, MetadataPayload)` from calldata.
2. On chain B (StarkNet testnet, or any other deployment sharing the same `omni_bridge_derived_address`), call `deploy_token(signature_from_A, payload_from_A)`.
3. `_verify_borsh_signature` computes `keccak(payload.to_borsh())` — identical bytes, identical hash, identical signature — and passes.
4. Assert `near_to_starknet_token[keccak(payload.token)] != 0` on chain B: the token is deployed without MPC authorization for chain B.

    fn _verify_borsh_signature(
        ref self: ContractState, borsh_bytes: @ByteArray, signature: Signature,
    ) {
        let message_hash_le = compute_keccak_byte_array(borsh_bytes);
        let message_hash = reverse_u256_bytes(message_hash_le);

        let sig = signature_from_vrs(signature.v, signature.r, signature.s);
        verify_eth_signature(message_hash, sig, self.omni_bridge_derived_address.read());
    }
