### Title
`respond_verify_foreign_tx` Accepts Signatures Over Arbitrary Payload Hashes Without Binding Them to the Submitted Request — (`File: crates/contract/src/lib.rs`)

---

### Summary

The `respond_verify_foreign_tx` function in the MPC contract verifies that a submitted signature is cryptographically valid over `response.payload_hash`, but it never verifies that `response.payload_hash` was actually derived from the `request` parameter supplied in the same call. A single Byzantine attested participant can reuse a legitimately-produced MPC signature from one foreign-chain verification request to resolve a completely different pending request, delivering a forged response to the waiting caller.

---

### Finding Description

`respond_verify_foreign_tx` performs two independent operations:

1. **Signature check** — verifies `response.signature` against `response.payload_hash` using the domain root public key.
2. **Yield resolution** — calls `pending_requests::resolve_yields_for(&mut self.pending_verify_foreign_tx_requests, &request, serde_json::to_vec(&response).unwrap())` to deliver the response to the caller who submitted `request`. [1](#0-0) 

The critical gap: the contract never checks that `response.payload_hash` encodes `request`. The payload hash is defined as `SHA-256(borsh(ForeignTxSignPayload::V1 { request, values }))`, so a hash produced for `request_B` is entirely unrelated to `request_A`. Yet the contract will accept `payload_hash_B` as a valid resolution for `request_A` as long as the signature over `payload_hash_B` verifies against the root key.

Any attested participant can call `respond_verify_foreign_tx`; the function only asserts `assert_caller_is_signer()` and `assert_caller_is_attested_participant_and_protocol_active()`. It does not restrict which participant may respond to which request, nor does it bind the response to the request. [2](#0-1)

### Impact Explanation

A Byzantine attested participant (one node, below the signing threshold) can:

1. Observe a legitimately produced `(payload_hash_B, signature_B)` for `request_B` (e.g., "Bitcoin tx Y confirmed with block-hash H").
2. Wait for a victim to submit `verify_foreign_transaction(request_A)` (e.g., "Bitcoin tx X confirmed").
3. Race-call `respond_verify_foreign_tx(request_A, VerifyForeignTransactionResponse { payload_hash: payload_hash_B, signature: signature_B })`.
4. The contract verifies `signature_B` over `payload_hash_B` — valid — and resolves the yield for `request_A` with `response_B`.
5. The victim's NEAR contract receives a `VerifyForeignTransactionResponse` whose `payload_hash` encodes a completely different foreign transaction.

Any bridge or omnibridge contract that does not independently recompute and compare the expected payload hash (i.e., does not use `ForeignChainSignatureVerifier::verify_signature` from the SDK) will accept this forged response as proof that `request_A`'s transaction was verified, enabling invalid fund release or double-spend conditions.

**Impact category**: High — forged foreign-chain verification causing invalid bridge execution.

---

### Likelihood Explanation

- Requires exactly **one** Byzantine attested participant — well within the allowed threat model (Byzantine participant strictly below the signing threshold).
- No new cryptographic material needs to be produced; the attacker reuses a signature the honest network already generated for a different request.
- The race window is the time between a user submitting `verify_foreign_transaction` and the honest MPC nodes calling `respond_verify_foreign_tx`. Because the Byzantine node is already an active participant monitoring the contract, it can front-run honest responders.
- Bridge contracts that omit SDK-level payload-hash verification are the realistic deployment target; the SDK helper is optional and not enforced on-chain.

---

### Recommendation

Bind the response to the request inside `respond_verify_foreign_tx` by recomputing the expected payload hash prefix from `request` and asserting it matches `response.payload_hash`. Concretely, the contract should verify:

```rust
// Reconstruct the canonical prefix that any valid payload_hash for this request must encode.
// The payload is SHA-256(borsh(ForeignTxSignPayload::V1 { request, values })).
// At minimum, assert the hash was produced for this exact request by requiring the node
// to also submit the extracted `values`, then recomputing and comparing:
let expected_hash = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: request.request.clone(),
    values: response.extracted_values.clone(), // add this field to the response DTO
}).compute_msg_hash()?;
require!(expected_hash == response.payload_hash, "payload_hash does not match request");
```

Alternatively, include the full `request` inside the signed payload and have the contract verify the binding before accepting the response. This ensures no valid signature for `request_B` can ever satisfy `request_A`.

### Proof of Concept

```
1. Honest network produces (payload_hash_B, sig_B) for request_B
   (e.g., Bitcoin tx Y confirmed, block-hash [0xBB;32]).

2. Victim submits verify_foreign_transaction(request_A)
   (e.g., Bitcoin tx X confirmed — tx X does NOT exist on-chain).

3. Byzantine attested node calls:
   respond_verify_foreign_tx(
       request  = request_A,          // looks up victim's pending yield
       response = VerifyForeignTransactionResponse {
           payload_hash: payload_hash_B,   // hash of request_B
           signature:    sig_B,            // valid sig over payload_hash_B
       }
   )

4. Contract check:
   verify_ecdsa_signature(sig_B, payload_hash_B, root_pk) → Ok  ✓
   resolve_yields_for(pending_verify_foreign_tx_requests, request_A, response_B) → yield resolved

5. Victim's NEAR contract receives response_B.
   If it calls only `verify_ecdsa_signature(sig_B, payload_hash_B, root_pk)` without
   recomputing the expected hash for request_A, it concludes "Bitcoin tx X verified" and
   releases bridge funds — but tx X was never on-chain.
```
