Audit Report

## Title
Push-Transfer ETH to Non-Receivable Contract Permanently Freezes Funds After rsETH Burn - (File: contracts/LRTWithdrawalManager.sol)

## Summary

`LRTWithdrawalManager.unlockQueue` irreversibly burns rsETH and moves ETH into the withdrawal manager in one committed transaction. A subsequent call to `_processWithdrawalCompletion` pushes ETH to the user via a low-level `call`; if the user's address is a contract that reverts on ETH receipt, the push fails and the entire Step 2 transaction reverts — but Step 1 is already final. The user's rsETH is permanently destroyed and the ETH is permanently stranded in the contract with no on-chain recovery path.

## Finding Description

The withdrawal lifecycle is split across two independent, non-atomic transactions:

**Step 1 — `unlockQueue` (operator-only, irreversible):**
rsETH held by the contract is burned and ETH is pulled from `LRTUnstakingVault`:

```solidity
// contracts/LRTWithdrawalManager.sol:305-307
if (rsETHBurned != 0) IRSETH(lrtConfig.rsETH()).burnFrom(address(this), rsETHBurned);
unstakingVault.redeem(asset, assetAmountUnlocked);
```

After this transaction commits, rsETH is gone and ETH sits in `LRTWithdrawalManager`. This cannot be undone.

**Step 2 — `completeWithdrawal` / `completeWithdrawalForUser` (always reverts for non-receivable contracts):**
`_processWithdrawalCompletion` pops the nonce, deletes the request, decrements `unlockedWithdrawalsCount`, then calls `_transferAsset`:

```solidity
// contracts/LRTWithdrawalManager.sol:734
_transferAsset(asset, user, request.expectedAssetAmount);
```

which resolves to:

```solidity
// contracts/LRTWithdrawalManager.sol:877-879
(bool sent,) = payable(to).call{ value: amount }("");
if (!sent) revert EthTransferFailed();
```

If `to` is a contract without a `receive()` function or with one that reverts, `sent == false` and the entire Step 2 transaction reverts. Because Solidity reverts roll back all state changes within that transaction, the nonce pop, the `delete withdrawalRequests[requestId]`, and the `unlockedWithdrawalsCount[asset]--` are all undone. The request remains in the queue permanently.

**No recovery path exists:**

- `sweepRemainingAssets` checks `hasUnlockedWithdrawals(asset)` at line 403, which returns `true` as long as `unlockedWithdrawalsCount[asset] > 0`. The stuck request keeps this permanently non-zero, so `sweepRemainingAssets` is permanently blocked for that asset.
- `completeWithdrawalForUser` (operator path, lines 192–204) calls `_processWithdrawalCompletion(asset, user, referralId)` — it sends ETH to the same `user` address and therefore also always reverts.
- There is no `cancelWithdrawal`, no alternative recipient parameter, and no admin escape hatch for stranded ETH.

The developer comment at line 191 acknowledges ETH transfer risk but dismisses it as "non-impactful gas grief," which underestimates the severity: the scenario is not gas grief but a permanent, irrecoverable fund freeze.

## Impact Explanation

**Critical — Permanent freezing of funds.**

After `unlockQueue` burns the user's rsETH (Step 1, committed on-chain), the ETH allocated to that user is held in `LRTWithdrawalManager`. If the user's address cannot accept ETH, neither the user nor any operator can ever complete the withdrawal. The ETH is locked in the contract indefinitely with no on-chain recovery mechanism. The user loses their entire principal.

## Likelihood Explanation

**Medium.** Smart-contract wallets, multisigs, and DeFi protocol treasuries that interact with LRT-rsETH may lack a `receive()` function or may have one that conditionally reverts (e.g., access-controlled, paused, or upgraded away). A malicious actor can also deliberately deploy such a contract, initiate a withdrawal, wait for the operator to burn their rsETH, and then use the permanently stuck ETH as leverage. No privileged access is required; any depositor can trigger this path.

## Recommendation

Replace the push-payment pattern with a pull-payment (claimable balance) pattern for ETH withdrawals:

1. In `_processWithdrawalCompletion`, instead of calling `_transferAsset` directly, credit `claimableETH[user] += request.expectedAssetAmount`.
2. Add a separate `claimETH(address recipient)` function that lets the user pull their balance to any address they specify.

This decouples the accounting step (which must succeed atomically) from the ETH delivery step (which is user-controlled), eliminating the ability of a reverting `receive()` to freeze funds.

## Proof of Concept

1. Attacker deploys `MaliciousRecipient` — a contract with `receive() external payable { revert(); }`.
2. `MaliciousRecipient` calls `initiateWithdrawal(ETH_TOKEN, rsETHAmount, "")`. rsETH is transferred to `LRTWithdrawalManager`; a `WithdrawalRequest` is recorded.
3. Operator calls `unlockQueue(ETH_TOKEN, ...)`. At line 305, rsETH is burned. At line 307, ETH is moved from `LRTUnstakingVault` into `LRTWithdrawalManager`. The request is now "unlocked" and `unlockedWithdrawalsCount[ETH_TOKEN]` is incremented.
4. `MaliciousRecipient` calls `completeWithdrawal(ETH_TOKEN, "")`. `_processWithdrawalCompletion` reaches line 878: `payable(MaliciousRecipient).call{value: amount}("")` → `receive()` reverts → `sent = false` → `revert EthTransferFailed()`. Entire tx reverts; all state changes (nonce pop, request delete, count decrement) are rolled back. Request remains in queue.
5. Operator calls `completeWithdrawalForUser(ETH_TOKEN, MaliciousRecipient, "")`. Same revert at line 878.
6. Operator attempts `sweepRemainingAssets(ETH_TOKEN)`. Reverts at line 403: `hasUnlockedWithdrawals(ETH_TOKEN)` is `true`.
7. **Result:** rsETH is permanently burned (Step 1 committed); ETH is permanently locked in `LRTWithdrawalManager`. No on-chain path exists to recover it.
