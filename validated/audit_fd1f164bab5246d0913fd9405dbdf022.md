### Title
Missing Constraint Between `request` and `response.payload_hash` in `respond_verify_foreign_tx` — (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_verify_foreign_tx` in the MPC contract verifies that the submitted signature is cryptographically valid over `response.payload_hash`, but never verifies that `response.payload_hash` is actually derived from the `request` argument. A malicious attested participant acting as the signing leader can therefore get the MPC network to sign an arbitrary `payload_hash` — one that commits to fabricated extracted values — and submit it as a valid foreign-chain verification response. The contract accepts it, and the caller receives a forged attestation.

---

### Finding Description

The `respond_verify_foreign_tx` function accepts two arguments: a `VerifyForeignTransactionRequest` (the original user request, including the `ForeignChainRpcRequest` and `domain_id`) and a `VerifyForeignTransactionResponse` (containing `payload_hash: Hash256` and `signature`).

The contract's only cryptographic check is:

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;

near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,
)
.is_ok()
``` [1](#0-0) 

This only confirms that `signature` is a valid ECDSA signature over `response.payload_hash` under the root public key. It does **not** verify that `response.payload_hash` equals `SHA-256(borsh(ForeignTxSignPayload::V1 { request: <the_request>, values: <actual_extracted_values> }))`.

The canonical payload construction is:

```rust
let payload = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: request.clone(),
    values,
});
// msg_hash = SHA-256(borsh(payload))
``` [2](#0-1) 

The `values` field (the extracted foreign-chain observations) is never included in the on-chain response, and the contract never recomputes the expected hash from `request`. The `payload_hash` is taken entirely at the submitter's word.

The client-side SDK does perform this check:

```rust
let payload_is_correct = expected_payload_hash == response.payload_hash;
if !payload_is_correct {
    return Err(VerifyForeignChainError::IncorrectPayloadSigned { ... });
}
``` [3](#0-2) 

But this is a client-side library check, not an on-chain enforcement. The MPC contract itself imposes no such constraint.

---

### Impact Explanation

The `respond_verify_foreign_tx` endpoint is callable by any single attested participant. The signing leader in the Cait-Sith protocol provides the message hash to the followers; followers participate in the signing protocol for whatever hash the leader supplies. A malicious leader can therefore:

1. Construct a `ForeignTxSignPayload` with the correct `request` but **fabricated `values`** (e.g., a fake block hash, a fake log entry, a fake Solana program ID).
2. Compute `fake_payload_hash = SHA-256(borsh(ForeignTxSignPayload::V1 { request, fake_values }))`.
3. Drive the MPC signing protocol with honest followers signing `fake_payload_hash`.
4. Submit `respond_verify_foreign_tx(request, { payload_hash: fake_payload_hash, signature: valid_threshold_sig })`.
5. The contract verifies the signature is valid over `fake_payload_hash` ✓, and that `request` matches a pending entry ✓ — and resolves the yield.

The caller receives a valid MPC threshold signature over a hash that attests to foreign-chain state that never occurred. For bridge use cases (e.g., Omnibridge inbound flow), this enables minting NEAR-side assets without a corresponding foreign-chain deposit — a direct theft of funds. [4](#0-3) 

---

### Likelihood Explanation

The `verify_foreign_transaction` flow is the primary use case for the Omnibridge inbound path. The leader role is assigned per signing round. Any single malicious attested participant who becomes leader for a foreign-tx signing round can exploit this. Only one such participant is required — well below the signing threshold. The feature is actively used and the attack requires no special setup beyond being an attested participant.

---

### Recommendation

The contract should enforce that `response.payload_hash` is consistent with the `request`. Since the contract does not store the extracted `values`, the response should be extended to include them, and the contract should recompute and verify the hash:

```rust
// In respond_verify_foreign_tx, after receiving response.values:
let expected_payload = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: request.request.clone(),
    values: response.values.clone(), // values must be included in the response
});
let expected_hash = expected_payload.compute_msg_hash()?;
if expected_hash != response.payload_hash {
    return Err(RespondError::PayloadHashMismatch.into());
}
```

This mirrors the check already present in `ForeignChainSignatureVerifier::verify_signature` in the client SDK. [5](#0-4) 

---

### Proof of Concept

1. Attacker is an attested MPC participant and becomes the signing leader for a pending `verify_foreign_transaction` request for Bitcoin tx `X`.
2. Attacker constructs `fake_payload = ForeignTxSignPayload::V1 { request: Bitcoin(X), values: [BlockHash([0xff; 32])] }` (a fabricated block hash).
3. Attacker computes `fake_hash = SHA-256(borsh(fake_payload))` and drives the Cait-Sith signing protocol with honest followers, who sign `fake_hash`.
4. Attacker calls `respond_verify_foreign_tx(request=Bitcoin(X), response={ payload_hash: fake_hash, signature: threshold_sig_over_fake_hash })`.
5. The contract at line 729–734 verifies `verify_ecdsa_signature(sig, fake_hash, root_pk)` — passes. The contract at line 749–752 resolves the yield with the forged response.
6. The bridge contract receives `VerifyForeignTransactionResponse { payload_hash: fake_hash, signature: valid }` and, if it does not independently recompute the expected hash, mints assets for a deposit that never occurred on Bitcoin. [6](#0-5) [7](#0-6)
