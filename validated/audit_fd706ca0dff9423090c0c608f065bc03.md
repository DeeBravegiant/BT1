### Title
Excess Attached Deposit Not Refunded in `submit_participant_info` for Existing-Participant Re-submissions — (File: `crates/contract/src/lib.rs`)

### Summary
`submit_participant_info` is `#[payable]` but only executes deposit-refund logic when `attestation_storage_must_be_paid_by_caller` is `true`. When an existing participant re-submits their attestation (the common hourly cadence), that flag is `false` and any attached NEAR deposit is silently consumed by the contract with no refund path and no recovery mechanism.

### Finding Description
The deposit-handling block in `submit_participant_info` is gated on a boolean that is `false` whenever the caller is already a participant **and** the attestation is an update (not a first insertion):

```rust
let caller_is_not_participant = self.voter_account().is_err();
let is_new_attestation = matches!(
    attestation_insertion_result,
    ParticipantInsertion::NewlyInsertedParticipant
);

let attestation_storage_must_be_paid_by_caller =
    is_new_attestation || caller_is_not_participant;

if attestation_storage_must_be_paid_by_caller {
    // ... deposit check and refund ...
    if let Some(diff) = attached.checked_sub(cost)
        && diff > NearToken::from_yoctonear(0)
    {
        Promise::new(account_id).transfer(diff).detach();
    }
}
// ← no else branch: attached deposit is silently consumed
Ok(())
```

When the flag is `false` the function returns `Ok(())` without touching `env::attached_deposit()`. Because the function is `#[payable]`, the NEAR runtime does not auto-refund; the deposit accretes to the contract balance permanently. [1](#0-0) 

The contrast with every other payable entry-point in the contract is stark: `sign`, `request_app_private_key`, `verify_foreign_transaction`, and `propose_update` all call `require_deposit` or an explicit `checked_sub`/`transfer` path that unconditionally refunds any excess. [2](#0-1) 

### Impact Explanation
Any NEAR tokens attached to a `submit_participant_info` call by an existing participant performing a routine re-submission are permanently locked in the contract. There is no sweep function, no admin recovery path, and no way for the participant to reclaim the deposit. This breaks the production accounting invariant that `#[payable]` functions must not silently absorb funds beyond what they consume for storage.

**Impact: Medium** — balance/accounting invariant broken; funds permanently frozen in the chain-signature contract without any recovery mechanism.

### Likelihood Explanation
**Likelihood: Low.** The MPC node's `periodic_attestation_submission` task attaches `0` deposit for re-submissions in the normal automated path. However, the operator guide explicitly documents a manual fallback: operators are instructed to call `submit_participant_info` manually with `--deposit` when the node fails to submit on its own. [3](#0-2) 

An operator following that documented procedure for a re-submission (not a first-time join) would attach a deposit that is silently consumed. The entry path is unprivileged: any existing participant account can trigger it.

### Recommendation
Add an unconditional refund of any attached deposit in the `else` branch (or after the `if` block) so that callers who are existing participants and attach a non-zero deposit are always made whole:

```rust
} else {
    // Existing participant re-submission: no storage charge, but refund any deposit.
    let attached = env::attached_deposit();
    if attached > NearToken::from_yoctonear(0) {
        Promise::new(account_id).transfer(attached).detach();
    }
}
```

Alternatively, restructure the function to call a single refund helper at the end (mirroring the `require_deposit` pattern used by `sign` and `verify_foreign_transaction`).

### Proof of Concept
1. Participant `alice.near` is already in the active participant set (i.e., `self.voter_account()` returns `Ok`).
2. Alice's attestation has already been stored (i.e., `add_participant` returns `ParticipantInsertion::UpdatedParticipant`).
3. Alice (or her operator) calls `submit_participant_info` with `attached_deposit = 1_000_000_000_000_000_000_000_000` yoctoNEAR (1 NEAR) following the manual-submission guide.
4. `is_new_attestation = false`, `caller_is_not_participant = false` → `attestation_storage_must_be_paid_by_caller = false`.
5. The deposit-handling block is skipped entirely; `Ok(())` is returned.
6. 1 NEAR is permanently locked in the contract with no recovery path. [4](#0-3)
