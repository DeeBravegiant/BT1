Audit Report

## Title
Prior User's Stranded ETH Silently Consumed to Subsidize Subsequent WETH Swap — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
The `pay()` function's partial-native branch uses `address(this).balance` without distinguishing ETH sent in the current transaction from ETH left over by prior callers. When a prior user's excess `msg.value` remains in the router, the next WETH swap's partial-native branch wraps and spends that ETH on behalf of the new caller, permanently destroying the prior user's funds with no recourse.

## Finding Description
`PeripheryPayments.pay()` contains three branches for WETH payment: [1](#0-0) 

When `0 < address(this).balance < value`, the middle branch fires unconditionally:
1. Wraps **all** of the router's native ETH via `IWETH9(WETH).deposit{value: nativeBalance}()`
2. Transfers that WETH to the pool recipient
3. Pulls only the remainder (`value - nativeBalance`) from the actual payer via `safeTransferFrom`

ETH accumulates between transactions because `exactInputSingle` is `payable` and has no automatic refund: [2](#0-1) 

If User A calls `exactInputSingle{value: 1 ether}` with `amountIn = 0.5 ether` and `tokenIn = WETH`, the first branch (`nativeBalance >= value`) fires and wraps only `0.5 ether`, leaving `0.5 ether` stranded. `refundETH()` must be explicitly called and is not automatic: [3](#0-2) 

The `receive()` guard only blocks direct ETH transfers from non-WETH addresses; it does not prevent `msg.value` accumulation from payable swap functions: [4](#0-3) 

User B then calls `exactInputSingle{value: 0}` with `tokenIn = WETH` and `amountIn = 1 ether`. The router's balance is `0.5 ether` (User A's), so `nativeBalance > 0` but `nativeBalance < value`. The partial-native branch wraps User A's `0.5 ether` and pulls only `0.5 WETH` from User B. User A's ETH is permanently gone.

## Impact Explanation
Direct loss of user principal. User A's ETH is transferred to the pool to settle User B's swap. The pool receives the correct total so no pool invariant breaks, but User A's ETH is unrecoverable. This is a Critical/High direct loss of user funds by an unprivileged caller requiring no special access.

## Likelihood Explanation
The scenario requires no attacker coordination. Sending slightly more ETH than `amountIn` for a WETH swap is a common UX pattern (rounding up to avoid slippage-induced reverts). The test suite itself demonstrates this pattern and explicitly requires `refundETH()` to be included in a multicall to recover excess ETH: [5](#0-4) 

A stranded balance is visible on-chain; a griever or MEV bot can immediately follow with a WETH `exactInputSingle` to drain it. No privileged access, malicious pool, or non-standard token is required.

## Recommendation
In the partial-native branch, do not silently consume the router's entire native balance. Track `msg.value` in transient storage at the entry of each payable function and use only that recorded amount in `pay()`, or remove the partial-native hybrid path entirely and require callers to either send the full amount as ETH or approve the full amount as WETH.

## Proof of Concept
```solidity
function test_priorUserEthConsumed() public {
    // User A: exactInputSingle WETH amountIn=0.5 ETH, sends 1 ETH (excess stays in router)
    vm.prank(userA);
    router.exactInputSingle{value: 1 ether}(
        ExactInputSingleParams({tokenIn: WETH, amountIn: 0.5 ether, ...})
    );
    assertEq(address(router).balance, 0.5 ether); // stranded

    // User B: exactInputSingle WETH amountIn=1 ETH, sends 0 ETH, has WETH approval
    vm.prank(userB);
    router.exactInputSingle{value: 0}(
        ExactInputSingleParams({tokenIn: WETH, amountIn: 1 ether, ...})
    );

    // Router's 0.5 ETH (User A's) was wrapped and sent to pool
    // User B only paid 0.5 WETH from their own balance
    assertEq(address(router).balance, 0);
    // User A's 0.5 ETH is permanently gone
}
```

The exact site of the loss is the partial-native branch: [6](#0-5)
