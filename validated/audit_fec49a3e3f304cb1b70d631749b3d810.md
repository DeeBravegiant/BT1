Audit Report

## Title
Stale `InterimRSETHOracle` Rate Enables Over-Minting of wrsETH, Diluting Existing Holder Yield — (`contracts/pools/oracle/InterimRSETHOracle.sol`, `contracts/pools/RSETHPoolV3.sol`)

## Summary
`InterimRSETHOracle` stores a manually-set rsETH/ETH rate with no on-chain staleness enforcement. When the stored rate lags below the true rate, `RSETHPoolV3.deposit()` mints more wrsETH per unit of WETH than the true backing warrants, undercollateralizing the `RsETHTokenWrapper` and allowing an attacker who redeems before the shortfall is discovered to extract yield that belongs to existing wrsETH holders.

## Finding Description
`InterimRSETHOracle.getRate()` returns `rate`, a storage variable set exclusively by a `MANAGER_ROLE` address via `setRate()`. There is no timestamp recorded on update, no heartbeat check, and no maximum-age revert path; the value can remain stale for an arbitrary duration. [1](#0-0) 

`RSETHPoolV3.viewSwapRsETHAmountAndFee(amount, token)` computes:
```
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate
```
where `rsETHToETHrate` is sourced directly from `InterimRSETHOracle` and `tokenToETHRate` is `1e18` for WETH (a pure constant). [2](#0-1) [3](#0-2) 

When the stored rate is stale-low (e.g., `1.001e18` while the true rate is `1.005e18`), the denominator is smaller than it should be, so `deposit()` mints more wrsETH than the deposited WETH actually backs: [4](#0-3) 

The normal bridger flow subsequently moves WETH to L1, deposits it into the LRT pool at the **true** rate, and bridges the resulting rsETH back to L2 to collateralize the wrapper via `depositBridgerAssets()`. Because the true rate is higher than the stale rate, the rsETH returned is less than the wrsETH already minted, leaving `RsETHTokenWrapper` undercollateralized. [5](#0-4) 

An attacker who holds the over-minted wrsETH and calls `RsETHTokenWrapper.withdraw()` before the shortfall is publicly visible redeems altRsETH at a 1:1 ratio against wrsETH, receiving more altRsETH than their WETH deposit warranted. Later redeemers find the wrapper short of altRsETH. [6](#0-5) 

The `dailyMintLimit` check caps per-day exposure but does not prevent the attack; it can be repeated on successive days. [7](#0-6) 

## Impact Explanation
Every unit of wrsETH minted above the true backing is a claim against the same altRsETH pool that backs all existing wrsETH. When the attacker redeems first, they extract altRsETH (and ultimately ETH on L1 via `LRTWithdrawalManager.instantWithdrawal()`) that belongs to existing holders. This is a direct, repeatable theft of unclaimed yield, matching the **High — Theft of unclaimed yield** impact class. [8](#0-7) 

## Likelihood Explanation
`InterimRSETHOracle` is explicitly documented as an interim solution requiring off-chain keeper calls to stay current. [9](#0-8) 
Any block-level delay between the true rsETH/ETH rate increasing and the keeper calling `setRate()` opens the window. No special privileges are required by the attacker — only a public `deposit()` call followed by `RsETHTokenWrapper.withdraw()` once the bridger has collateralized the wrapper. This is a normal operational condition, not an oracle operator compromise.

## Recommendation
1. Record `block.timestamp` on every `setRate()` call and add a staleness bound to `getRate()` (e.g., revert if `block.timestamp - lastUpdated > MAX_STALENESS`).
2. Add a `minRsETHOut` slippage parameter to `RSETHPoolV3.deposit()` so users cannot silently receive more wrsETH than expected when the oracle is stale.
3. Transition to an on-chain, manipulation-resistant oracle (Chainlink or a TWAP) to eliminate the manual-update latency window entirely. [10](#0-9)
