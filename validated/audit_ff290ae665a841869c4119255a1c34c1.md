Audit Report

## Title
Permissionless `FeeReceiver.sendFunds()` + `LRTOracle.updateRSETHPrice()` + `LRTWithdrawalManager.instantWithdrawal()` Compose Into JIT Sandwich That Steals Yield From rsETH Holders — (`contracts/FeeReceiver.sol`, `contracts/LRTOracle.sol`, `contracts/LRTWithdrawalManager.sol`)

## Summary

`FeeReceiver.sendFunds()` carries no access control and can be called by any address to flush accumulated MEV/execution-layer ETH rewards into `LRTDepositPool`, immediately increasing the protocol's TVL. `LRTOracle.updateRSETHPrice()` is equally permissionless and recomputes the rsETH price from the live TVL. `LRTWithdrawalManager.instantWithdrawal()` redeems rsETH at the current oracle price. An attacker can atomically deposit ETH at the pre-reward price, trigger the reward flush and price update, then instantly redeem at the post-reward price, capturing a disproportionate share of yield that should accrue to existing long-term rsETH holders.

## Finding Description

**Root cause — three permissionless functions with no temporal separation:**

`FeeReceiver.sendFunds()` has no role modifier: [1](#0-0) 

`LRTOracle.updateRSETHPrice()` is gated only by `whenNotPaused`, not by any role: [2](#0-1) 

`LRTWithdrawalManager.instantWithdrawal()` is callable by any address when `isInstantWithdrawalEnabled[asset]` is true: [3](#0-2) 

**Exploit flow:**

1. Attacker calls `LRTDepositPool.depositETH{value: X}()` at current price `P1`, receiving `X / P1` rsETH. The oracle price is stale and has not yet reflected the pending rewards sitting in `FeeReceiver`.

2. Attacker calls `FeeReceiver.sendFunds()`. The entire `FeeReceiver` balance `R` is forwarded to `LRTDepositPool.receiveFromRewardReceiver()`, landing in the deposit pool's ETH balance: [1](#0-0) 

3. `LRTOracle._getTotalEthInProtocol()` reads `address(depositPool).balance` as part of `getETHDistributionData()`, so the newly arrived `R` is immediately visible to the oracle: [4](#0-3) 

4. Attacker calls `LRTOracle.updateRSETHPrice()`. `previousTVL = rsethSupply × P1`. After the deposit and reward flush, `totalETHInProtocol = previousTVL + R`. The new price `P2 = (totalETHInProtocol − protocolFee) / rsethSupply > P1`: [5](#0-4) 

5. Attacker calls `LRTWithdrawalManager.instantWithdrawal()`. `getExpectedAssetAmount` uses the live `rsETHPrice` (now `P2`), so the attacker redeems `X / P1` rsETH for `(X / P1) × P2` ETH from the `LRTUnstakingVault`, netting `X × (P2/P1 − 1)` minus the instant-withdrawal fee: [6](#0-5) 

**Why existing guards are insufficient:**

The `pricePercentageLimit` guard in `_updateRsETHPrice()` reverts non-manager callers only when the price increase exceeds the configured threshold: [7](#0-6) 

However, `pricePercentageLimit` is a `uint256` that defaults to `0`. When it is zero, the condition `pricePercentageLimit > 0` is false and the entire threshold check is skipped, leaving no protection for non-manager callers regardless of the price jump magnitude. Even when the limit is set, an attacker can still profit from reward batches small enough to stay within the threshold.

The `onlyInstantWithdrawalAllowed` modifier requires `isInstantWithdrawalEnabled[asset]` to be true, which is manager-controlled. However, this feature is designed to be enabled in normal protocol operation, and once enabled it imposes no per-user or per-block restriction.

## Impact Explanation

**High — Theft of unclaimed yield.** Existing rsETH holders accumulate yield as MEV and execution-layer rewards flow into `FeeReceiver`. The attacker intercepts a fraction of each reward batch proportional to their flash-minted rsETH position. The attacker provides no lasting liquidity, bears no protocol risk, and exits in the same transaction. The stolen yield is the difference between what existing holders would have received and what they actually receive after the attacker's dilution. With `pricePercentageLimit == 0` (the default), the attack is unconditionally profitable for any non-zero `FeeReceiver` balance whenever instant withdrawal is enabled and the `LRTUnstakingVault` holds sufficient ETH.

## Likelihood Explanation

**High.** All three entry points are callable by any EOA or contract. MEV rewards accumulate continuously and predictably in `FeeReceiver`. The `FeeReceiver` balance is publicly observable on-chain. The attack is atomic, requires no privileged access, and is repeatable every time rewards accumulate. Any MEV searcher monitoring `FeeReceiver.balance` can profitably execute this pattern. The only operational prerequisite — `isInstantWithdrawalEnabled` being true — is expected to hold during normal protocol operation.

## Recommendation

1. **Add access control to `FeeReceiver.sendFunds()`**: restrict it to `MANAGER` or `OPERATOR` role so that reward timing cannot be weaponized by arbitrary callers.
2. **Set `pricePercentageLimit` to a non-zero value at deployment/initialization**: the current default of `0` disables the only price-increase guard for non-manager callers.
3. **Enforce a minimum holding period before rsETH minted in block `N` can be used in `instantWithdrawal`**: analogous to the existing `withdrawalDelayBlocks` for queued withdrawals.
4. **Alternatively, stream rewards over time** (e.g., 24-hour linear release after `sendFunds()` is called) so no single block captures the full price jump.

## Proof of Concept

```solidity
// Minimal attacker contract — all steps in one transaction
contract JITAttack {
    ILRTDepositPool  depositPool;
    IFeeReceiver     feeReceiver;
    ILRTOracle       oracle;
    ILRTWithdrawalManager withdrawalMgr;
    IRSETH           rsETH;
    address constant ETH_TOKEN = 0xEeeeeEeeeEeEeeEeEeEeeEEEeeeeEeeeeeeeEEeE;

    function attack() external payable {
        // Step 1: mint rsETH at stale (pre-reward) price P1
        depositPool.depositETH{value: msg.value}(0, "");

        // Step 2: flush accumulated MEV rewards into deposit pool, raising TVL
        feeReceiver.sendFunds();

        // Step 3: reprice rsETH upward to P2
        oracle.updateRSETHPrice();

        // Step 4: redeem at P2 — profit = msg.value * (P2/P1 - 1) - fee
        uint256 bal = rsETH.balanceOf(address(this));
        rsETH.approve(address(withdrawalMgr), bal);
        withdrawalMgr.instantWithdrawal(ETH_TOKEN, bal, "");

        payable(msg.sender).transfer(address(this).balance);
    }
    receive() external payable {}
}
```

**Foundry fork test plan:**
1. Fork mainnet at a block where `FeeReceiver.balance > 0` and `isInstantWithdrawalEnabled[ETH_TOKEN] == true`.
2. Confirm `pricePercentageLimit == 0` (or set it to 0 in the test).
3. Deploy `JITAttack`, call `attack{value: 10 ether}()`.
4. Assert `address(attacker).balance > 10 ether` after the call.
5. Assert `LRTOracle.rsETHPrice()` increased between steps 2 and 3.
