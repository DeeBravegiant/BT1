Audit Report

## Title
ETH Deposit Limit Check Excludes Incoming Amount, Allowing Cap Bypass — (File: `contracts/LRTDepositPool.sol`)

## Summary
`LRTDepositPool._checkIfDepositAmountExceedesCurrentLimit()` applies an inconsistent limit check: for LST tokens it correctly evaluates `totalAssetDeposits + amount > limit`, but for ETH it evaluates only `totalAssetDeposits > limit`, omitting the incoming deposit amount entirely. This allows any unprivileged depositor to push total ETH deposits above the configured cap whenever the running total is at or near the limit.

## Finding Description
In `_checkIfDepositAmountExceedesCurrentLimit` (L676–682), the ETH branch uses a strict `>` comparison against `totalAssetDeposits` alone, without adding the incoming `amount`:

```solidity
// L678-679 — ETH branch: amount is NOT included
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));
}
// L681 — LST branch: amount IS included
return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
```

When `totalAssetDeposits == depositLimitByAsset(ETH_TOKEN)`, the ETH check evaluates `limit > limit` → `false`, so `_beforeDeposit` (L661–663) does not revert, and the deposit is accepted. The total then rises to `limit + depositAmount`. This is the sole enforcement point for the ETH deposit cap; there is no secondary guard. The entry path `depositETH` (L76–93) is `external payable` with no role restriction, making this reachable by any caller.

## Impact Explanation
The deposit limit is the protocol's primary risk-management cap on ETH exposure. Bypassing it silently violates the invariant the protocol promises to enforce. No funds are directly stolen, but the protocol fails to deliver its promised risk boundary. This maps to **Low — Contract fails to deliver promised returns** from the allowed impact scope.

## Likelihood Explanation
The condition `getTotalAssetDeposits(ETH_TOKEN) == depositLimitByAsset(ETH_TOKEN)` is a realistic steady-state: the limit is actively managed and deposits approach the cap regularly. Any depositor who observes on-chain state can exploit the window. No special privilege, front-running, or external dependency is required. The exploit is repeatable until the admin raises the limit.

## Recommendation
Add `amount` to the ETH branch to mirror the LST branch:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

## Proof of Concept
1. Admin sets `depositLimitByAsset(ETH_TOKEN) = 1000 ether`.
2. Organic deposits bring `getTotalAssetDeposits(ETH_TOKEN)` to exactly `1000 ether`.
3. Attacker calls `depositETH{value: 100 ether}(0, "")`.
4. `_checkIfDepositAmountExceedesCurrentLimit` evaluates `1000 ether > 1000 ether` → `false` → no revert.
5. `_mintRsETH` mints rsETH for the attacker; total ETH in protocol becomes `1100 ether`, 10% above the configured cap.
6. Subsequent legitimate depositors are blocked (`1100 > 1000 = true`), but the cap has already been permanently breached for this cycle.

**Foundry test sketch:**
```solidity
function test_ethDepositBypassesLimit() public {
    vm.prank(admin);
    lrtConfig.setDepositLimitByAsset(ETH_TOKEN, 1000 ether);
    // fill to exactly the limit via prior deposits
    _fillETHDepositsTo(1000 ether);
    // attacker deposits beyond limit — should revert but does not
    vm.prank(attacker);
    depositPool.depositETH{value: 100 ether}(0, "");
    assertGt(depositPool.getTotalAssetDeposits(ETH_TOKEN), 1000 ether);
}
```
