Audit Report

## Title
Unconditional zero-value `safeTransfer` to `protocolTreasury` in `MerkleDistributor.claim()` permanently freezes unclaimed yield for tokens that revert on zero transfers - (File: contracts/utils/MerkleDistributor/MerkleDistributor.sol)

## Summary

`MerkleDistributor.claim()` unconditionally calls `IERC20(token).safeTransfer(protocolTreasury, fee)` at line 144 even when `fee` computes to zero. For ERC20 tokens that revert on zero-value transfers, every affected user's `claim()` call will revert permanently, freezing their unclaimed token distributions in the contract. The sibling contracts `KernelMerkleDistributor` and `KernelTop100MerkleDistributor` both correctly guard this transfer with `if (fee > 0)`, making `MerkleDistributor` the sole contract missing this protection.

## Finding Description

In `MerkleDistributor.claim()`, the fee is computed as:

```solidity
uint256 fee = (claimableAmount * feeInBPS) / 10_000;
uint256 amountToSend = claimableAmount - fee;

IERC20(token).safeTransfer(account, amountToSend);
IERC20(token).safeTransfer(protocolTreasury, fee);  // unconditional, fee can be 0
``` [1](#0-0) 

`fee` evaluates to zero in two concrete scenarios:
1. `feeInBPS = 0` (fee disabled): `fee` is always `0` for every claim.
2. Small claim amounts: with `feeInBPS = 1`, any `claimableAmount < 10_000` produces `fee = 0`.

Both `KernelMerkleDistributor` and `KernelTop100MerkleDistributor` guard the treasury transfer:

```solidity
if (fee > 0) {
    kernel.safeTransfer(protocolTreasury, fee);
}
``` [2](#0-1) [3](#0-2) 

`MerkleDistributor` has no such guard. The `token` field is set via `setToken()` and can be any ERC20 — the contract is explicitly generic. The `initialize()` function permits `_feeInBPS = 0` (it only rejects values above `MAX_FEE_IN_BPS`), making zero-fee operation a valid and expected configuration. [4](#0-3) 

## Impact Explanation

**Medium — Permanent freezing of unclaimed yield.** When the distributed token reverts on zero-value transfers (a documented behavior in tokens such as LEND and cUSDCv3), every user whose computed `fee` rounds to zero will have their `claim()` permanently revert. The transaction reverts entirely (Solidity atomicity), so no state is committed and the user cannot retry under the same conditions. Their allocated tokens remain locked in the contract with no recovery path for affected users, as there is no emergency withdrawal function for individual user allocations.

## Likelihood Explanation

- `MerkleDistributor` is a generic distributor; `token` is configurable by the owner and can be any ERC20, including tokens with zero-transfer revert behavior.
- `feeInBPS = 0` is a natural operational choice (no fee taken), which causes `fee = 0` for **every** claim simultaneously, affecting all users at once.
- Even with non-zero `feeInBPS`, users with small cumulative allocations (common in broad airdrop distributions) hit the rounding-to-zero case.
- No privileged access is required; any user with a valid merkle proof triggers the path.

## Recommendation

Mirror the guard already present in `KernelMerkleDistributor` and `KernelTop100MerkleDistributor`:

```diff
- IERC20(token).safeTransfer(protocolTreasury, fee);
+ if (fee > 0) IERC20(token).safeTransfer(protocolTreasury, fee);
``` [5](#0-4) 

## Proof of Concept

**Setup:**
- Deploy `MerkleDistributor` with a token that reverts on zero-value transfers (e.g., LEND).
- Set `feeInBPS = 0` via `setFeeInBPS(0)` (valid — passes the `> MAX_FEE_IN_BPS` check).
- Set a merkle root with Alice's entry: `index=1, account=alice, cumulativeAmount=500`.

**Execution:**
1. Alice calls `claim(1, alice, 500, proof)`.
2. `claimableAmount = 500 - 0 = 500`.
3. `fee = (500 * 0) / 10_000 = 0`.
4. `IERC20(token).safeTransfer(alice, 500)` succeeds.
5. `IERC20(token).safeTransfer(protocolTreasury, 0)` **reverts** — token disallows zero-value transfers.
6. Entire transaction reverts. Alice's state is not committed.
7. Alice retries — identical result. Her 500 tokens are permanently frozen.
