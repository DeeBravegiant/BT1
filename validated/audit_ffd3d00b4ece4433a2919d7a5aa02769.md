Audit Report

## Title
Block Stuffing Accumulates Multi-Day Yield Into Single Fee Mint, Triggering `DailyFeeMintLimitExceeded` and Freezing Price Updates — (`contracts/LRTOracle.sol`)

## Summary

When `updateRSETHPrice()` is suppressed for multiple days (e.g., via block stuffing), the entire accumulated TVL increase is presented as a single-period reward on the next call. The resulting protocol fee can exceed `maxFeeMintAmountPerDay`, causing `_checkAndUpdateDailyFeeMintLimit` to revert with `DailyFeeMintLimitExceeded`. Because both `updateRSETHPrice()` and `updateRSETHPriceAsManager()` share the same `_updateRsETHPrice()` internal path, neither entry point can succeed until the manager separately raises the cap via `setMaxFeeMintAmountPerDay`.

## Finding Description

`updateRSETHPrice()` is a permissionless `public` function that calls `_updateRsETHPrice()`. [1](#0-0) 

`updateRSETHPriceAsManager()` calls the identical internal function with no additional bypass logic. [2](#0-1) 

Inside `_updateRsETHPrice()`, the reward is computed as the raw delta between current TVL (from live oracle rates including `SwETHPriceOracle` → `ISwETH.getRate()`) and the previous TVL stored implicitly via `rsETHPrice`:

```solidity
uint256 rewardAmount = totalETHInProtocol - previousTVL;
protocolFeeInETH = (rewardAmount * lrtConfig.protocolFeeInBPS()) / 10_000;
``` [3](#0-2) 

There is no amortization: if N days of yield have accrued, the full N-day reward is treated as a single period's fee.

`_checkAndUpdateDailyFeeMintLimit` resets `currentPeriodMintedFeeAmount` when a new day begins, but then immediately checks the full (N-day) fee amount against the single-day cap:

```solidity
if (currentPeriodMintedFeeAmount + feeAmount > maxFeeMintAmountPerDay) {
    revert DailyFeeMintLimitExceeded(...);
}
``` [4](#0-3) 

The period reset (line 199–202) does not help: it only zeroes `currentPeriodMintedFeeAmount`, but `feeAmount` itself is already N× the expected daily fee, so the check still reverts. [5](#0-4) 

The manager's `updateRSETHPriceAsManager()` does bypass the `PriceAboveDailyThreshold` check (line 263), but it does **not** bypass `_checkAndUpdateDailyFeeMintLimit`, which is called unconditionally at line 303: [6](#0-5) 

`SwETHPriceOracle.getAssetPrice` returns the live `ISwETH.getRate()`, which continuously accrues yield regardless of whether `updateRSETHPrice` is called: [7](#0-6) 

## Impact Explanation

**Low — Block stuffing** (explicitly listed in the allowed impact scope).

`rsETHPrice` is not updated; all downstream consumers (deposit pool pricing, withdrawal valuations) read a stale price. Both the public and manager-gated price update entry points are simultaneously bricked. Recovery requires out-of-band manager action (`setMaxFeeMintAmountPerDay` followed by a retry), introducing a window of stale pricing.

## Likelihood Explanation

Block stuffing on Ethereum mainnet is expensive but economically rational if the attacker profits from stale pricing (e.g., holds a large short position on rsETH or exploits a downstream protocol reading `rsETHPrice`). The attack requires no privileged access and is fully permissionless. The vulnerability is triggered by any `maxFeeMintAmountPerDay` that is conservative relative to the protocol's TVL and yield rate — a realistic configuration.

## Recommendation

In `_checkAndUpdateDailyFeeMintLimit`, instead of reverting when the fee exceeds the daily cap, cap the minted fee at the remaining daily allowance and carry forward the uncollected fee, or skip fee minting for the excess. Alternatively, give `updateRSETHPriceAsManager()` its own internal path that bypasses the daily fee limit check (minting the full fee regardless of the cap), so the manager can always recover from a stuffed-block scenario without needing a separate `setMaxFeeMintAmountPerDay` call.

## Proof of Concept

```solidity
// Foundry unit test — no mainnet required
function testBlockStuffingDailyFeeLimitDOS() public {
    // 1. Set a small daily fee cap
    lrtOracle.setMaxFeeMintAmountPerDay(1e15);

    // 2. Simulate 7 days of block stuffing — no updateRSETHPrice called
    vm.warp(block.timestamp + 7 days);

    // 3. swETH rate has accrued 7 days of ~4% APR yield
    mockSwETH.setRate(1e18 + 7 * 1.1e13);

    // 4. Public call reverts
    vm.expectRevert(ILRTOracle.DailyFeeMintLimitExceeded.selector);
    lrtOracle.updateRSETHPrice();

    // 5. Manager call also reverts — same internal path, no bypass
    vm.prank(manager);
    vm.expectRevert(ILRTOracle.DailyFeeMintLimitExceeded.selector);
    lrtOracle.updateRSETHPriceAsManager();
}
```

The key code path: `updateRSETHPriceAsManager()` → `_updateRsETHPrice()` → computes N-day fee → `_checkAndUpdateDailyFeeMintLimit(N_day_fee)` → resets period counter but still checks `0 + N_day_fee > maxFeeMintAmountPerDay` → reverts. [8](#0-7)

    }
