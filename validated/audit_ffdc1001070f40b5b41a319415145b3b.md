Audit Report

## Title
Reward Tokens Permanently Locked When `totalKernelStaked` Drops to Zero During Active Reward Period - (File: contracts/KERNEL/KernelDepositPool.sol)

## Summary
`KernelDepositPool` distributes rewards linearly via a Synthetix-style `rewardPerToken()` accumulator. If `totalKernelStaked` reaches zero during an active reward window, `rewardPerToken()` freezes at `rewardPerTokenStored` and stops accumulating. Reward tokens allocated for the zero-staked interval are never credited to any user and cannot be recovered, as the contract has no admin rescue function for `rewardsToken`.

## Finding Description
`rewardPerToken()` (L408–414) short-circuits to return the frozen `rewardPerTokenStored` whenever `totalKernelStaked == 0`:

```solidity
function rewardPerToken() public view returns (uint256) {
    if (totalKernelStaked == 0) {
        return rewardPerTokenStored;  // frozen
    }
    return rewardPerTokenStored + (rewardRate * (lastTimeRewardApplicable() - updatedAt) * DECIMAL_PRECISION)
        / totalKernelStaked;
}
```

`initiateWithdrawal()` (L320–326) decrements `totalKernelStaked` with no guard preventing it from reaching zero while `block.timestamp < finishAt`:

```solidity
balanceOf[msg.sender] -= _amount;
totalKernelStaked -= _amount;
```

`notifyRewardAmount()` (L569–570) checks `if (totalKernelStaked == 0) revert NoStakedTokens()` only at the moment a new reward period is *started*, not during an ongoing one. This guard is therefore insufficient to prevent the supply from dropping to zero mid-period.

The contract has no `recoverTokens`, `emergencyWithdraw`, or any other admin function that operates on `rewardsToken`. The four admin functions (`setRewardsDuration`, `notifyRewardAmount`, `setWithdrawalDelay`, `setMaxNumberOfWithdrawalsPerUser`) provide no recovery path. Calling `notifyRewardAmount` again after `finishAt` would roll the locked balance into a new `rewardRate`, compounding the loss rather than recovering it.

The NatSpec at L18–22 explicitly acknowledges this behavior and states the mitigation is purely operational ("ensuring there are always some tokens staked"), with no code-level enforcement.

## Impact Explanation
Reward tokens (`rewardsToken` ERC20) allocated for the zero-staked interval are permanently locked in `KernelDepositPool` with no on-chain recovery path. This matches the allowed impact: **Medium — Permanent freezing of unclaimed yield**.

## Likelihood Explanation
Low. Requires all stakers to call `initiateWithdrawal()` during an active reward period, reducing `totalKernelStaked` to zero. No attacker action is required; ordinary user behavior (withdrawing stake) is sufficient. Plausible scenarios include a protocol migration, a coordinated exit, or a market panic. The condition is not artificially constructed.

## Recommendation
Add an admin-callable rescue function for `rewardsToken` gated on `block.timestamp > finishAt && totalKernelStaked == 0`, allowing recovery of any unallocated reward balance. Alternatively, enforce at the code level that `totalKernelStaked` cannot reach zero while a reward period is active by reverting `initiateWithdrawal` if it would bring `totalKernelStaked` to zero before `finishAt`.

## Proof of Concept
1. Admin calls `notifyRewardAmount(1_000e18)` with `duration = 30 days`. `rewardRate = 1_000e18 / 30 days`. `finishAt = block.timestamp + 30 days`. `totalKernelStaked = 100e18`.
2. After 10 days, all stakers call `initiateWithdrawal(fullBalance)`. The `updateReward` modifier snapshots each user's earned rewards up to this point. `totalKernelStaked` becomes `0`.
3. For the remaining 20 days, `rewardPerToken()` returns the frozen `rewardPerTokenStored`. No user accrues any new rewards.
4. Unallocated rewards ≈ `rewardRate * 20 days ≈ 666e18` tokens remain in the contract.
5. No function in `KernelDepositPool` allows these tokens to be recovered.

**Foundry test plan:**
- Deploy `KernelDepositPool` with mock `kernelToken` and `rewardsToken`.
- Stake `100e18` from a single address; call `notifyRewardAmount(1_000e18)` with `duration = 30 days`.
- `vm.warp(block.timestamp + 10 days)`.
- Call `initiateWithdrawal(100e18)`; assert `totalKernelStaked == 0`.
- `vm.warp(block.timestamp + 20 days)` (past `finishAt`).
- Assert `rewardsToken.balanceOf(address(pool)) > 0` (locked balance remains).
- Assert no callable function can transfer this balance out.
