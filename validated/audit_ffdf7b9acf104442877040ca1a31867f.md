Audit Report

## Title
`RSETHPriceFeed.latestRoundData()` Returns ETH/USD `updatedAt` for a Composite Answer That Includes a Separately-Cached rsETH Price, Masking Staleness of the rsETH Component - (File: `contracts/oracles/RSETHPriceFeed.sol`)

## Summary
`RSETHPriceFeed.latestRoundData()` computes `answer` as the product of the ETH/USD Chainlink price and `LRTOracle.rsETHPrice`, but returns `updatedAt` exclusively from the ETH/USD Chainlink feed. Because `LRTOracle.rsETHPrice` is a stored state variable updated only on explicit calls to `updateRSETHPrice()`, the returned `updatedAt` does not reflect the freshness of the rsETH component. Any integrator applying a standard Chainlink staleness check will always pass, even when the rsETH price is arbitrarily stale.

## Finding Description
`RSETHPriceFeed.latestRoundData()` (L63â€“70) fetches all five return values from `ETH_TO_USD.latestRoundData()` and then overwrites only `answer`:

```solidity
(roundId, answer, startedAt, updatedAt, answeredInRound) = ETH_TO_USD.latestRoundData();
answer = int256(RS_ETH_ORACLE.rsETHPrice()) * answer / 1e18;
```

The `updatedAt` field is left as the ETH/USD Chainlink timestamp. `LRTOracle.rsETHPrice` (L28) is a plain `uint256` storage variable. It is only written inside `_updateRsETHPrice()` (L313), which is invoked by the public `updateRSETHPrice()` (L87) or the manager-gated `updateRSETHPriceAsManager()` (L94). No timestamp is stored alongside `rsETHPrice`, and no on-chain mechanism enforces a maximum interval between calls. The identical defect exists in `getRoundData()` (L53â€“61).

If the keeper stops calling `updateRSETHPrice()` â€” due to downtime, gas spikes, or while `LRTOracle` is paused â€” `rsETHPrice` remains at its last stored value indefinitely. Every call to `latestRoundData()` will return a recent `updatedAt` from the ETH/USD heartbeat, giving integrators a false signal of freshness for the composite rsETH/USD answer.

## Impact Explanation
**Low â€” Contract fails to deliver promised returns, but doesn't lose value.**

`RSETHPriceFeed` is a Chainlink-compatible wrapper explicitly designed for external integrators (the `description` field is set to `"RSETH / USD"`). Any integrator applying the standard staleness guard (`block.timestamp - updatedAt > MAX_STALENESS`) will always pass because `updatedAt` tracks the ETH/USD heartbeat (typically 1 hour), not the rsETH price update cadence. The integrator will consume a stale rsETH/USD price without any on-chain signal that the rsETH component is outdated. This breaks the contract's promise of being a correct Chainlink-compatible feed and can cause downstream mispricing in any protocol that integrates it.

## Likelihood Explanation
The ETH/USD Chainlink feed updates on its own heartbeat regardless of whether `updateRSETHPrice()` is called. Any period during which the keeper does not call `updateRSETHPrice()` â€” including during `LRTOracle` pause, keeper downtime, or gas price spikes â€” will silently produce a stale composite answer while `updatedAt` remains current. This is a realistic operational scenario requiring no attacker action; it occurs passively whenever the keeper lags.

## Recommendation
Add a `rsETHPriceUpdatedAt` storage variable to `LRTOracle` that is written alongside `rsETHPrice` in `_updateRsETHPrice()`. Expose it via the `IRSETHOracle` interface. In `RSETHPriceFeed.latestRoundData()` (and `getRoundData()`), return `min(ethToUsdUpdatedAt, rsETHPriceUpdatedAt)` as `updatedAt`. This ensures the returned `updatedAt` reflects the staleness of the least-fresh component of the composite answer.

## Proof of Concept
1. `updateRSETHPrice()` is called at time `T`, setting `LRTOracle.rsETHPrice = P`.
2. No further calls to `updateRSETHPrice()` occur for 24 hours (keeper downtime or oracle pause).
3. At time `T + 24h`, an integrator calls `RSETHPriceFeed.latestRoundData()`.
4. The ETH/USD Chainlink feed has updated normally; `updatedAt` returned is `T + 24h - 30min` (within the 1-hour heartbeat).
5. The integrator's staleness check (`block.timestamp - updatedAt < 1 hour`) passes.
6. The integrator uses `answer = P_stale Ă— ETH_USD_current / 1e18`, where `P_stale` is 24 hours old.
7. The integrator has no on-chain way to detect that the rsETH component is stale.

**Foundry fork test plan:** Deploy a fork, call `updateRSETHPrice()`, `vm.warp(block.timestamp + 24 hours)`, call `RSETHPriceFeed.latestRoundData()`, assert that `block.timestamp - updatedAt < 3600` (staleness check passes) while `rsETHPrice` has not been updated for 24 hours.
