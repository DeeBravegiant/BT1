Audit Report

## Title
ETH Deposit Limit Guard Omits Incoming Amount, Allowing Deposits to Exceed the Configured Cap — (File: `contracts/LRTDepositPool.sol`)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` applies an asymmetric check: for ERC-20 assets it correctly evaluates `totalAssetDeposits + amount > limit`, but for ETH it evaluates only `totalAssetDeposits > limit`, silently discarding the `amount` argument. As a result, any unprivileged depositor can push the protocol's ETH holdings arbitrarily beyond `depositLimitByAsset[ETH_TOKEN]` in a single transaction, bypassing the protocol's primary ETH exposure cap.

## Finding Description
The root cause is in `_checkIfDepositAmountExceedesCurrentLimit` at lines 676–682 of `contracts/LRTDepositPool.sol`:

```solidity
// L676-682
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)); // amount ignored
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // correct
}
``` [1](#0-0) 

The ETH branch (line 679) never incorporates `amount`, so the guard only fires if the cap is *already* exceeded before the deposit arrives. During normal operation (`totalAssetDeposits < limit`), the check always returns `false` regardless of how large `amount` is.

This guard is the sole pre-condition enforced by `_beforeDeposit` before minting rsETH: [2](#0-1) 

`depositETH` is a public payable function that passes `msg.value` as `depositAmount` into `_beforeDeposit`, but that value is silently dropped in the ETH branch: [3](#0-2) 

The deposit limit is stored in `LRTConfig.depositLimitByAsset`: [4](#0-3) 

The view helper `getAssetCurrentLimit` correctly computes remaining headroom as `limit - totalAssetDeposits`, confirming the intended semantics of the cap: [5](#0-4) 

## Impact Explanation
The `depositLimitByAsset` cap is the protocol's primary mechanism for bounding ETH exposure to EigenLayer strategies. With the ETH branch broken, the cap is unenforceable: any depositor can exceed it by an arbitrary amount in a single call, and rsETH is minted for every wei deposited. This causes the protocol to fail to deliver the risk-bounded returns it promises to existing rsETH holders, mapping to **Low — Contract fails to deliver promised returns**.

## Likelihood Explanation
`depositETH` is a public payable function with no role restriction. Any wallet can call it at any time. The only prerequisite is the normal operating state (`totalAssetDeposits < limit`). No special conditions, front-running, or privileged access are required. Likelihood is **High**.

## Recommendation
Apply the same `+ amount` pattern used for ERC-20 assets to the ETH branch, eliminating the special case entirely:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

## Proof of Concept

```
Setup:
  depositLimitByAsset[ETH_TOKEN] = 100 ether
  getTotalAssetDeposits(ETH_TOKEN) = 99 ether   // one wei below the cap

Attack:
  attacker calls depositETH{value: 50 ether}()

Execution of _checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 50 ether):
  totalAssetDeposits = 99 ether
  ETH branch: return (99 ether > 100 ether)  →  false   ← amount=50 ether never used
  → MaximumDepositLimitReached() is NOT triggered

Result:
  50 ether deposited, rsETH minted
  getTotalAssetDeposits(ETH_TOKEN) = 149 ether  (49 ether over the cap)

Contrast with ERC-20 under identical conditions:
  return (99 ether + 50 ether > 100 ether)  →  true  →  revert  ✓
```

Foundry test plan: deploy `LRTDepositPool` with `depositLimitByAsset[ETH_TOKEN] = 100 ether`, seed the pool to 99 ether via a prior deposit, then call `depositETH{value: 50 ether}()` from an unprivileged address and assert the call succeeds and `getTotalAssetDeposits(ETH_TOKEN)` returns 149 ether, confirming the cap is not enforced.
